# API Gateway, Service Discovery, Regions, Availability Zones, and DNS (HLD)

> Beginner-first study notes expanded from the supplied bullets: **an "intelligent" API gateway; handling one million requests; avoiding a single point of failure; API gateway versus load balancer and their order; API composition; authentication; rate limiting, burst control, throttling, IP blocking, and queues; service registration, deregistration, and heartbeats; request/response transformation; response caching; logging; Regions; Availability Zones; DNS-based load balancing; and local/recursive, root, TLD, and authoritative DNS.**
>
> The central mental model is: **DNS helps a client find a reachable entry point, a load-balancing layer spreads traffic across entry-point instances, and an API gateway applies API-aware policy before routing to services.** These responsibilities can be implemented by separate products or overlap in one product.

## 0. Scope, assumptions, corrections, and evidence labels

### Assumptions

- "Handle million request" is interpreted as the harder and interview-relevant target of **1,000,000 requests per second (RPS) at peak**, not one million requests over an unspecified period. The design process still applies at smaller rates.
- The main subject is an external **Layer 7 API gateway** for HTTP, REST, GraphQL, or gRPC-style APIs in a microservices system. Internal gateways and non-HTTP protocols are called out where they differ.
- Cloud names such as *Region* and *Availability Zone* are explained using their common AWS-style meaning. Other providers use similar but not perfectly identical terms.
- "Intelligent gateway" means **application-aware and policy-aware**, not necessarily AI-powered.

### Corrections to the rough bullets

1. **An API gateway is a logical service, not necessarily one process or VM.** A production gateway is normally a fleet spread across failure domains.
2. **A gateway and a load balancer are not mutually exclusive.** A gateway often load-balances, and an L7 load balancer may implement gateway-like policies. Their primary responsibilities differ.
3. **There is no universal answer to "which comes first?"** DNS logically precedes the network connection; a self-hosted gateway normally has a load balancer in front of its replicas, while another load balancer or service-discovery mechanism may sit behind it.
4. **IP blocking is access control, not a rate-limiting algorithm.** It can complement rate limiting but has serious NAT, proxy, mobile-network, and IPv6 limitations.
5. **Throttling is usually the enforcement action** after a rate or concurrency limit is exceeded; it is not a separate placement tier.
6. **A queue is not a universal substitute for throttling.** A durable queue changes an API from "complete now" to "accept now, process later." It fits asynchronous work, not every read or interactive operation.
7. **A heartbeat proves only that something is sending heartbeats.** It does not prove that the instance is ready, fast, correct, or able to reach its own dependencies.
8. **DNS load balancing does not normally balance each HTTP request.** It returns endpoint information that resolvers and clients cache; the chosen endpoint then handles one or many connections and requests.
9. **An Availability Zone is a failure-isolation boundary, not simply a server rack.** In AWS, an AZ can contain one or more discrete data centers with independent infrastructure.

### Evidence labels used in this chapter

- **Verified platform behavior:** supported by current first-party documentation linked in the chapter.
- **Historical public production example:** a company described the system at the publication date; it is not a claim about its private 2026 architecture.
- **Illustrative design:** a reasonable interview design using a named product; not a claim about its actual private implementation.
- **Candidate-reported interview experience:** a public self-report, not an official company question bank and not independently verified.
- **Public interview-question thread:** a discussion labeled as an interview question, but not necessarily tied to a verified company interview.
- **Illustrative number:** arithmetic for learning, not a benchmark or product promise.

Research and linked product behavior were checked on **17 July 2026**.

## 1. Plain-language foundation

### 1.1 The problem starts when one application becomes many services

Imagine a shopping mobile app backed by one server:

```text
Mobile app ----------------------> Shopping server
                                   - login
                                   - catalog
                                   - cart
                                   - orders
```

The client has one address and one API contract. As the product grows, the backend may be split into services:

```text
Auth service     Catalog service     Cart service     Order service
10.1.1.7:8080    10.1.4.9:9000       10.2.3.5:8080    10.3.8.2:7000
```

It would be a bad client contract to make every phone know:

- every internal hostname and port;
- which service instance is alive;
- which protocol each service uses;
- how authentication is implemented by every team;
- every service deployment, split, merge, and version; and
- how to combine several services into one screen.

Mobile applications also cannot be upgraded instantly. Old versions may remain in use for months. Exposing internal topology directly would tightly couple those old clients to a backend that changes daily.

### 1.2 An API gateway is the backend's front desk

An API gateway presents a stable front door:

```text
                                +--> Auth service
Client --> api.shop.com --> API +--> Catalog service
                          Gateway+--> Cart service
                                +--> Order service
```

The client asks for a public API such as:

```http
GET /v1/orders/8472
Authorization: Bearer <access-token>
```

The gateway can:

1. terminate or inspect the application protocol;
2. identify the API, version, caller, and policy;
3. authenticate the caller and apply coarse authorization;
4. enforce rate, burst, size, and concurrency limits;
5. route the request to an eligible backend;
6. transform the public request into the backend contract;
7. optionally compose or cache a response; and
8. record access metrics, logs, and trace context.

The backend can change from one order service to ten instances, or from HTTP/JSON to gRPC/Protobuf, without forcing the public client contract to change at the same time.

### 1.3 Why call it "intelligent"?

A basic transport load balancer might see:

```text
source IP, destination IP, port, TCP connection
```

An application-aware gateway can additionally see, when TLS is terminated there:

```text
host      = api.shop.com
method    = POST
path      = /v2/orders
tenant    = merchant-42
user      = user-891
API cost  = expensive-write
version   = mobile-7.4
```

That context enables policy such as:

```text
/v1/orders       -> legacy order API
/v2/orders       -> new order API
premium tenant   -> larger quota
anonymous caller -> public read routes only
old mobile app   -> response compatibility transform
```

"Intelligent" should not imply that the gateway owns business truth. A good gateway knows **how to expose and protect an API**; the order service still decides whether an order can be cancelled.

### 1.4 One gateway icon does not mean one machine

Architecture diagrams compress a fleet into one box:

```text
                       logical API Gateway
                     /         |          \
                 node A      node B      node C
                  AZ-1        AZ-2        AZ-3
```

Clients use one hostname, but traffic reaches many gateway nodes. Nodes can be added, removed, upgraded, or failed independently. This is the first answer to "how is the gateway not a single point of failure?"

For now, an **Availability Zone (AZ)** means an isolated infrastructure failure domain inside a geographic **Region**; both are explained fully in Section 18.

The harder answer is that **common dependencies and configuration can still be single failure causes**. Later sections cover those.

## 2. Required prerequisites, explained inline

These concepts are necessary to understand the request path. They are included here so no previous system-design knowledge is assumed.

### 2.1 Client, server, request, and response

- A **client** initiates an operation: a browser, mobile app, another service, or partner system.
- A **server** listens for operations and returns results.
- An HTTP **request** includes a method, hostname, path, headers, and optionally a body.
- An HTTP **response** includes a status, headers, and optionally a body.

```text
request:  POST /orders + token + order body  --->
response: 202 Accepted + order-attempt ID    <---
```

The network can fail after the server performs the action but before the response reaches the client. Therefore, retrying a write safely may require an **idempotency key**. See [Idempotency](./13.%20Idempotency.md) for the full treatment.

### 2.2 IP address, hostname, DNS, and port

- A **hostname** such as `api.shop.com` is a human- and configuration-friendly name.
- **DNS** resolves that name to endpoint information, commonly one or more IP addresses.
- An **IP address** lets networks route packets toward an endpoint.
- A **port** identifies a listening service at that address; HTTPS commonly uses TCP port 443.

DNS does not usually forward the API request. It helps the client discover where to connect.

### 2.3 TLS and HTTPS

**TLS** encrypts and authenticates a network connection. HTTPS is HTTP over TLS.

If a gateway must route by HTTP path, validate a JWT header, or transform a JSON body, it normally terminates the client-side TLS connection so it can read the HTTP message:

```text
Client == TLS connection 1 ==> Gateway == TLS connection 2 ==> Service
```

The second connection should also be authenticated and encrypted when the threat model requires it. "It is inside our network" is not automatically a sufficient trust policy.

See [Network Protocols](./01.%20Network%20Protocols.md) for a deeper treatment of HTTP, TCP, TLS, QUIC, and WebSocket after this inline foundation.

### 2.4 Reverse proxy and load balancer

A **reverse proxy** receives a request on behalf of backend servers. A **load balancer** selects among multiple eligible targets.

```text
Client --> reverse proxy --> chosen backend
```

One software product can be a reverse proxy, an L7 load balancer, and an API gateway at the same time. These words describe responsibilities, not necessarily separate boxes. See [Proxy and Reverse Proxy](./16.%20Proxy%20and%20Reverse%20Proxy.md) and [Load Balancers](./17.%20Load%20Balancers.md) for deeper chapters.

### 2.5 Microservices and service instances

A **service** is a logical capability, such as `catalog`. It may have many running **instances**:

```text
catalog service
  - 10.2.1.4:8080
  - 10.2.2.8:8080
  - 10.2.3.6:8080
```

Instances are replaced during deployments, scaling, and failures. **Service discovery** maps the logical service name to the currently eligible endpoints.

Microservices are not automatically better than a monolith. They trade independent deployment and scaling for network failures, discovery, observability, and distributed-data complexity. See [Microservices](./03.%20Microservices.md).

### 2.6 Synchronous versus asynchronous work

- **Synchronous:** the caller waits while the server completes enough work to return the result.
- **Asynchronous:** the server durably accepts work, returns an acknowledgement, and completes it later.

```text
Synchronous:  Client -> API -> work -> final response

Asynchronous: Client -> API -> durable queue -> 202 Accepted
                                      |
                                      v
                                  worker later
```

This distinction determines whether an "API queue" is a valid overload tool. See [Messaging Queues](./15.%20Messaging%20Queues.md) for delivery and consumer semantics.

## 3. Why an API gateway is needed in system design

### 3.1 Stable public contract over changing internals

The gateway decouples external clients from service addresses, instance count, deployment layout, and often backend protocols.

```text
public contract:  GET /v1/profile

today:  gateway -> monolith
later:  gateway -> profile service -> database
future: gateway -> profile BFF -> profile + preferences services
```

This supports migrations, versioning, canaries, and the Strangler Fig pattern without making clients track the internal transition.

### 3.2 One enforcement point for cross-cutting policy

Authentication, request limits, TLS policy, schema validation, access logging, and correlation IDs are needed by many APIs. A gateway can implement a consistent baseline once.

Services still need defense in depth. The gateway is not permission to trust every request that somehow reaches the internal network.

### 3.3 Reduce client chattiness

A mobile home screen might need profile, active trip, promotions, and payment-summary data. Four client-to-data-center round trips over a mobile network can be far slower and harder to evolve than one call to a backend-for-frontend or lightweight composition endpoint.

### 3.4 Protect downstream services

The gateway sees demand before backend work is consumed. It can reject malformed, unauthorized, abusive, oversized, too-expensive, or over-quota requests early.

It can also apply:

- timeouts and deadline propagation;
- circuit breakers;
- bounded retries for safe operations;
- load shedding and priorities; and
- request-body and connection limits.

### 3.5 Central visibility into the public API surface

A gateway can provide a consistent view of:

- which public routes exist;
- who called them;
- request rate, errors, and latency;
- which backend and version served them;
- policy denials and quota use; and
- which client versions still use an old contract.

This visibility is valuable, but centralization also creates privacy, data-retention, and blast-radius risk.

## 4. Anatomy of a production gateway

### 4.1 The request data plane

The **data plane** handles live requests:

```text
Client request
    |
    v
[ listener / TLS ]
    |
[ route match ]
    |
[ authentication + policy middleware ]
    |
[ rate / size / concurrency checks ]
    |
[ cache lookup or backend selection ]
    |
[ transform + proxy / compose / enqueue ]
    |
[ response policy + access telemetry ]
    |
    v
Client response
```

The exact order is deliberate. For example:

- a cheap global connection limit may run before expensive token verification;
- user-level rate limiting requires identity, so it runs after authentication;
- a cache lookup before authorization can leak private data unless the cached object is explicitly public;
- body transformation requires size limits before buffering untrusted input.

### 4.2 The control plane

The **control plane** manages what the data plane should do:

```text
API definitions / route owners
Authentication and rate policies
Service registry / endpoint metadata
Certificates and trusted keys
Deployment and canary configuration
                |
                v
       versioned gateway config
                |
          push / pull / watch
                |
                v
        gateway data-plane fleet
```

A key reliability rule is:

> An existing data-plane node should usually continue serving with its last known good configuration during a temporary control-plane outage.

Otherwise, a configuration database or management UI becomes part of every request's availability.

### 4.3 Configuration is executable production behavior

A route configuration can change security, traffic, and data exposure without changing application code. Treat it like code:

- schema validation;
- ownership and review;
- version history;
- compatibility checks;
- dry-run or shadow evaluation;
- canary rollout;
- fast rollback; and
- an audit trail.

One bad global rule can break every healthy gateway node simultaneously. Replicas protect against machine failure, not shared bad configuration.

### 4.4 Keep the hot path cheap

At high request rates, avoid synchronous network calls for every piece of policy when possible:

```text
Good hot path candidates:
  route table in memory
  locally cached public signing keys
  compiled schema/policy
  pooled backend connections
  bounded local counters with coordinated global budgets

Dangerous on every request:
  management database lookup
  certificate store lookup
  remote log write before responding
  unbounded auth-service call
  full service-registry scan
```

Not every remote lookup can be removed. The design must define cache freshness, revocation needs, failure policy, and bounded resource use.

## 5. The full request path: what actually comes first?

### 5.1 A common self-hosted gateway path

```text
                                 Global layer
Client
  |
  | 1. resolve api.example.com
  v
DNS / traffic steering
  |
  | 2. connect to a regional/edge address
  v
CDN / DDoS protection / WAF              optional by workload
  |
  v
Regional L4 or L7 load balancer
  |
  | 3. spread traffic across gateway nodes
  +--------------+---------------+
  v              v               v
Gateway A      Gateway B       Gateway C
AZ-1           AZ-2            AZ-3
  |              |               |
  +--------------+---------------+
                 |
                 | 4. route using registry or service endpoint
                 v
        internal LB / service mesh / direct discovery
                 |
          +------+------+
          v             v
       Service 1     Service 2
```

- A **CDN (content delivery network)** caches or serves suitable content from locations closer to users.
- **DDoS protection** detects/absorbs large distributed traffic attacks before they exhaust the application.
- A **WAF (web application firewall)** evaluates HTTP requests for configured web-attack and access-control rules.

They are optional and overlap in some managed edge products.

From the client-facing direction:

1. **DNS is logically first**, although a cached answer means no DNS network lookup occurs for every request.
2. An optional global edge layer terminates or forwards traffic and provides DDoS/WAF/CDN functions.
3. A load balancer commonly distributes connections or requests across the self-hosted gateway fleet.
4. The gateway applies API policy and chooses a logical backend.
5. Internal load balancing or client-side discovery chooses a backend instance.

### 5.2 A managed gateway path

With a managed gateway, the provider hides its own load-balancing fleet:

```text
Client -> DNS -> managed API Gateway endpoint -> backend integration
                                                |
                                                +-> function
                                                +-> regional LB
                                                +-> private service
                                                +-> message broker
```

It is still incorrect to imagine one gateway machine. The load balancing is part of the managed service implementation or endpoint.

### 5.3 Can an API gateway come before a load balancer?

Yes, when "the load balancer" means the **backend service's** balancer:

```text
Client -> managed gateway -> catalog-service load balancer -> catalog instances
```

Can a load balancer come before the gateway? Also yes:

```text
Client -> ingress load balancer -> self-hosted gateway replicas -> services
```

Can the gateway directly load-balance instances and remove the second LB? Yes:

```text
Gateway -> service registry -> choose instance -> direct connection
```

Therefore the interview-ready answer is:

> DNS resolves the entry endpoint. A self-hosted gateway usually needs an ingress load balancer in front of its replicas. After the gateway selects a service, it can use an internal load balancer, a service mesh, or registry-driven client-side balancing to select an instance. The exact boxes can overlap.

### 5.4 Do not draw every box automatically

For a small service, this may be enough:

```text
Client -> managed L7 load balancer -> monolith
```

Adding a separate CDN, WAF, API gateway, ingress, service mesh, and internal load balancer without a requirement adds latency, cost, configuration, and failure modes. Start with responsibilities; introduce a component only when a requirement needs it.

## 6. API gateway versus load balancer

### 6.1 Plain-language difference

- A **load balancer** primarily answers: *Which healthy instance should receive this unit of traffic?*
- An **API gateway** primarily answers: *Which API is this, may this caller use it, what policies apply, and which backend capability should implement it?*

### 6.2 Comparison

| Dimension | Load balancer | API gateway |
|---|---|---|
| Primary goal | Distribute traffic across eligible targets | Expose, secure, govern, and route APIs |
| Typical visibility | L4 flow fields or L7 HTTP metadata | L7 route, identity, API version, tenant, schema, sometimes body |
| Typical selection | One instance within a target pool | First a service/API/operation, then possibly an instance |
| Common functions | Health checks, balancing algorithm, draining, affinity, TLS | AuthN/AuthZ hooks, quotas, transformations, composition, API versioning, developer policy, logging |
| Traffic types | TCP, UDP, TLS, HTTP, gRPC depending on layer | Primarily application protocols such as HTTP, WebSocket, GraphQL, and gRPC |
| Public or internal | Both | Often public north-south; can also be internal/partner-facing |
| Client contract role | Usually hides target instances | Often presents the stable external API contract |
| Can it do the other's job? | L7 products can do some gateway policy | Gateways normally perform some load balancing or call an LB |

### 6.3 Why the boundary is blurry

An L7 load balancer can route by host/path, terminate TLS, validate headers, and run a WAF. A gateway can select among service endpoints. Some products deliberately cover both.

Use responsibility language in interviews:

> "I need an L7 entry tier that authenticates callers, applies per-tenant quotas, routes `/orders` and `/catalog`, and distributes each route across healthy instances. This could be one capable gateway product or a gateway plus regional load balancers."

That is more precise than arguing about product labels.

### 6.4 Four-cue answer spine: `ROLE`

1. **R - Responsibility:** LB distributes; gateway governs and exposes APIs.
2. **O - Overlap:** both may terminate TLS and route at L7.
3. **L - Layering:** ingress LB can precede gateway replicas; internal LB can follow route selection.
4. **E - Example:** `/orders` is selected by the gateway; `order-17` is selected by balancing/discovery.

## 7. How a gateway handles one million requests per second

### 7.1 First correction: a gateway does not "handle a million" by being powerful

It handles the target by becoming a **distributed, horizontally scaled, measured system**:

```text
global traffic
      |
      +--> Region A / gateway cells / many nodes
      +--> Region B / gateway cells / many nodes
      +--> Region C / gateway cells / many nodes
```

Capacity is the sum of safe capacity across nodes and cells, constrained by the weakest shared dependency.

### 7.2 Quantify what "one million RPS" means

Request count alone is incomplete. Ask:

- peak versus average RPS;
- request and response byte distribution;
- TLS handshakes versus reused connections;
- HTTP/1.1, HTTP/2, HTTP/3, WebSocket, or gRPC;
- percentage cached, rejected, proxied, transformed, or composed;
- authentication method and remote lookups;
- average and p99 downstream latency;
- number of open and in-flight requests;
- regional and tenant skew;
- acceptable latency/error SLO; and
- failure the fleet must survive at peak.

An **SLO (service-level objective)** is a measurable reliability target. For example, a p99 latency SLO says that 99% of eligible requests should finish at or below the stated latency threshold during the measurement window; the slowest 1% are beyond p99.

One million tiny cached `GET`s and one million JSON transformations that each fan out to five slow services are entirely different systems.

### 7.3 Illustrative bandwidth and concurrency math

Assume, only for learning:

```text
peak rate                   = 1,000,000 requests/s
average request+response    = 2 KiB of application data
average end-to-end wait     = 100 ms = 0.1 s
```

Approximate application-data throughput:

```text
1,000,000/s x 2 KiB = 2,048,000,000 bytes/s
                    ~= 2.05 GB/s ~= 16.4 Gbit/s ~= 1.91 GiB/s
```

This excludes packet, TLS, retransmission, and internal fan-out overhead.

Using Little's Law for in-flight requests:

```text
concurrency ~= arrival rate x average time in system
            = 1,000,000/s x 0.1 s
            = 100,000 in-flight requests
```

These are **illustrative numbers**, not a production forecast. Use distributions and peak measurements in a real design; averages hide tail latency and large payloads.

### 7.4 Measure safe per-node capacity

Benchmark a representative node with production-like:

- TLS and connection reuse;
- route count and policy chain;
- token validation;
- payload distributions;
- transformations and compression;
- cache hit ratio;
- logging and tracing level;
- slow and failing backends; and
- p95/p99 latency and error SLOs.

The important capacity is not "the process did not crash." It is the rate at which the node stays within CPU, memory, network, file-descriptor, connection-pool, queue, and tail-latency limits.

### 7.5 Illustrative fleet calculation

Suppose a load test finds that one chosen gateway node saturates at `25,000 RPS` for this exact traffic mix. This is a **fictional benchmark**, not a claim about any product.

Operate at 60% of that measured saturation point:

```text
safe normal capacity/node = 25,000 x 0.60 = 15,000 RPS
```

Require 20% demand margin and survival of one of three equal AZs:

```text
required after failure = 1,000,000 x 1.20 = 1,200,000 RPS
surviving fraction     = 2/3

total nodes >= 1,200,000 / (15,000 x 2/3)
            >= 120 nodes
```

An initial layout is `40 nodes/AZ` across three AZs. Then validate actual AZ loss, skew, startup time, quotas, and downstream capacity. Do not use this arithmetic as a replacement for load testing.

### 7.6 Keep gateway nodes horizontally scalable

Prefer gateway nodes that keep durable business state elsewhere:

```text
request 1 -> node A
request 2 -> node C
request 3 -> node B
```

Useful node-local state can still exist:

- connection tables and pools;
- bounded caches;
- compiled routes and policies;
- local rate-limit allocations;
- metrics buffers; and
- short-lived request context.

The distinction is that losing one node should not lose an order, revoke the only copy of a policy, or make a user permanently sticky to a dead node.

### 7.7 Remove remote work from the per-request path where safe

At one million RPS, even one mandatory remote lookup per request creates another one-million-RPS dependency.

Typical techniques:

- verify signed access tokens locally using cached, rotated public keys;
- compile and keep route/policy tables in memory;
- watch incremental service-discovery updates instead of querying the full registry per call;
- reuse backend connections and multiplex HTTP/2/gRPC streams;
- emit logs asynchronously to bounded buffers;
- use hierarchical/local rate limiting with coordinated budgets; and
- cache only data with explicit safe cache semantics.

Fresh revocation, strict global quotas, and opaque tokens may still require shared state. State the consistency-versus-availability trade-off rather than hiding it.

### 7.8 Partition into cells to reduce blast radius

A single giant fleet shares too much fate. Split by region, tenant group, API family, or consistent hash:

```text
                    api.example.com
                           |
                 traffic steering layer
              /             |             \
         gateway cell 1  gateway cell 2  gateway cell 3
         tenants A-H     tenants I-P     tenants Q-Z
```

A bad route, memory leak, abusive tenant, or overload in one cell should not consume every gateway resource. Cells need a fallback/reassignment policy and enough spare capacity.

### 7.9 Autoscaling is useful but not instantaneous protection

Autoscaling reacts after a signal, and new nodes take time to allocate, start, load config, warm JIT/caches, establish connections, and pass readiness checks.

For flash traffic:

- keep failure and burst headroom;
- pre-scale for known events;
- enforce quotas and concurrency;
- shed lower-priority work;
- use bounded asynchronous queues where semantics allow; and
- ramp new nodes with slow start rather than sending a full share immediately.

### 7.10 Downstreams must scale too

A gateway that accepts one million RPS but sends all of it to one overloaded identity service or rate-limit database has not solved the problem.

Trace the multiplication:

```text
1,000,000 client RPS
  x 3 backend calls/request
  x 1.05 attempts/call after bounded retries
= 3,150,000 backend attempts/s
```

Composition, retries, cache misses, logs, and authentication can amplify work. Capacity-plan every mandatory dependency and define overload behavior.

### 7.11 A useful public scale reference - with strict labeling

Uber's first-party 2020 write-up described its **historical second-generation** API gateway at **800,000 requests/second peak**, deployed as **40+ independent services**, and documented API-sharded deployments and the problems caused by too much business logic in the gateway. Its 2021 architecture article described work scaling the newer gateway to **500,000 QPS (queries per second)**. These are dated public production reports, not current 2026 claims and not transferable per-node benchmarks. They show that gateway scale comes from fleet architecture, partitioning, constrained responsibility, and operations - not one oversized box. See [Uber's gateway evolution](https://www.uber.com/us/en/blog/gatewayuberapi/) and [gateway architecture](https://www.uber.com/us/en/blog/architecture-api-gateway/).

## 8. Why the gateway is not a single point of failure

### 8.1 Redundancy across failure domains

```text
                    resilient entry endpoint
                              |
                +-------------+-------------+
                |             |             |
             AZ-1          AZ-2          AZ-3
          gateway fleet  gateway fleet  gateway fleet
                |             |             |
                +------ healthy services ---+
```

At minimum:

- multiple gateway processes;
- multiple hosts;
- multiple network/power failure domains such as AZs;
- a load-balancing/edge tier that is itself redundant;
- health/readiness checks and graceful draining; and
- survivor capacity for the stated failure.

### 8.2 The gateway node should not own durable business state

If a gateway node dies:

- in-flight requests may fail and need safe retry semantics;
- its client connections normally close and reconnect elsewhere;
- local cache and counters can be rebuilt or reconciled;
- durable accepted work must already be in a durable system; and
- service instances remain reachable through other gateway nodes.

Do not claim seamless migration of an established TCP/WebSocket connection to another process. The client normally reconnects and resumes using externalized session state or sequence information.

### 8.3 Separate data-plane survival from control-plane availability

Gateway nodes should hold a versioned last known good configuration. If the management system is briefly unavailable:

```text
existing APIs continue on config version 1842
new configuration publication pauses
operators are alerted
```

Security policy determines exceptions. For example, inability to fetch a never-before-seen signing key may fail closed, while a telemetry sink outage may cause bounded buffering or sampled loss rather than taking down all APIs.

### 8.4 Common-mode failure checklist

| Failure | Replicas alone enough? | Required design question |
|---|---:|---|
| One gateway process/host dies | Usually | Does health detection remove it and can clients retry safely? |
| One AZ fails | No | Is traffic multi-AZ and is survivor capacity sufficient? |
| Whole Region fails | No | Is multi-region required, and are data/dependencies ready there? |
| Bad global route/config | No | Are changes validated, canaried, scoped, and reversible? |
| Expired certificate or bad key rollout | No | Are expiry, rotation, staged rollout, and rollback monitored? |
| Identity provider unavailable | No | Can existing signed tokens be verified locally? What must fail closed? |
| Service registry unavailable | No | Can nodes use bounded-stale last known endpoints? |
| Shared rate-limit store fails | No | Fail open, fail closed, or use local emergency budgets for each API class? |
| Logging sink slows | No | Are writes asynchronous and bounded? Which audit events may not be dropped? |
| Backend latency spike | No | Are timeout budgets, circuit breakers, bulkheads, and load shedding configured? |
| One tenant sends abusive traffic | No | Are queues, limits, and capacity isolated by tenant/API/cell? |

### 8.5 Health checking must distinguish states

```text
starting -> live -> ready -> serving -> draining -> removed
```

- **Liveness:** should the process be restarted?
- **Readiness:** should it receive new traffic now?
- **Deep functional health:** can a representative dependency path work?

A deep health check that calls every dependency on every probe can cause a health-check storm and eject healthy gateways because one optional service is down. Use layered signals, thresholds, hysteresis, and route-specific circuit breakers.

### 8.6 Multi-region is not automatically required

Multi-AZ commonly handles routine infrastructure faults within a Region. Multi-region adds protection from regional failure and can improve locality or data residency, but adds:

- state replication and write ownership decisions;
- regional capacity and failover planning;
- configuration and certificate replication;
- dependency parity and drift risk;
- higher cost; and
- more complex testing and operations.

Choose it from business SLO, RTO/RPO, locality, and regulatory requirements - not because every diagram should have two Regions.

- **RTO (recovery time objective):** the target maximum time to restore the capability after a defined disaster.
- **RPO (recovery point objective):** the target maximum data gap measured back from the disruption; it describes how much recent data the recovery may lose.

See [High Availability Architecture](./14.%20High%20Availability%20Architecture.md) for failure models, RTO/RPO, fencing, and failover detail.

## 9. Gateway routing and API lifecycle

### 9.1 Route from a public operation to a backend capability

Routes can match combinations of:

- hostname: `partner-api.example.com`;
- method: `GET`, `POST`, and so on;
- path: `/v2/orders/{id}`;
- headers or media type;
- API/client version;
- authenticated tenant or claims; and
- controlled rollout attributes.

```text
api.example.com/v1/orders/*  -> legacy-order pool
api.example.com/v2/orders/*  -> order-v2 pool
partner.example.com/orders/* -> partner adapter -> order-v2
```

Use deterministic rule precedence and reject ambiguous configurations before deployment. A broad `/orders/*` rule should not accidentally shadow a security-sensitive `/orders/admin/*` rule.

### 9.2 Versioning is a contract problem, not just a path

The gateway can route `/v1` and `/v2`, but the organization still needs:

- compatibility rules;
- deprecation notice and measurement;
- old-client support windows;
- schema evolution policy;
- ownership; and
- a removal plan.

Hiding an incompatible backend change behind endless transformations makes the gateway a permanent compatibility monolith.

### 9.3 Canary and traffic shifting

The gateway can send a controlled percentage or selected callers to a new backend:

```text
/checkout
  99% -> checkout-v1
   1% -> checkout-v2 canary
```

Good canaries compare error rate, p95/p99 latency, business correctness, resource saturation, and downstream impact. Use stable assignment for stateful user flows so one user does not randomly alternate between incompatible versions.

### 9.4 Routing is not business authorization

Routing may use a verified tenant or role to choose an API surface, but the service that owns a resource should enforce resource-level rules:

```text
Gateway: "This token may call the order API."
Order service: "This user may view order 8472 because it belongs to them."
```

This avoids turning the gateway into a copy of every domain's authorization model.

## 10. API composition and the Backend-for-Frontend pattern

### 10.1 Plain-language purpose

One client screen may require several backend services:

```text
                   +--> Profile service
Mobile -> Gateway -+--> Active-trip service
                   +--> Promotion service
                   +--> Payment-summary service
```

Without composition, the mobile client makes several high-latency internet round trips and must understand partial failures and backend contracts. A composition endpoint accepts one client call, fans out internally, and builds one client-oriented response.

This is also called **gateway aggregation**. A **Backend for Frontend (BFF)** is a client-specific API layer, for example separate mobile and web BFFs.

### 10.2 Parallel fan-out versus sequential calls

Independent calls should usually run in parallel:

```text
Sequential latency ~= profile + trip + promo + payment
Parallel latency   ~= max(profile, trip, promo, payment) + merge overhead
```

Parallel fan-out reduces latency but creates concurrent downstream load. Bound the fan-out and connection pools.

If call B needs data from call A, the dependency is sequential:

```text
get user -> use home city -> get local promotions
```

Make this dependency graph and its deadlines explicit.

### 10.3 Availability multiplication

If three required dependencies each independently succeed 99.9% of the time, an illustrative all-required composition succeeds:

```text
0.999 x 0.999 x 0.999 ~= 0.997003 = 99.7003%
```

Real failures are not always independent, so this is a teaching calculation, not an SLO prediction. The lesson is that adding required dependencies can reduce end-to-end availability.

### 10.4 Design partial failure deliberately

For each field, define whether it is:

- **required:** fail the whole response if missing;
- **optional:** return partial data with an explicit absence/error marker;
- **stale-acceptable:** use a bounded-stale cache fallback; or
- **deferred:** return a token/status and complete later.

Example:

```json
{
  "order": { "id": "8472", "status": "shipped" },
  "recommendations": null,
  "partial": true,
  "errors": [{ "component": "recommendations", "code": "TIMEOUT" }]
}
```

Do not silently substitute wrong financial, inventory, or security data just to return `200 OK`.

### 10.5 Timeout budget and cancellation

If the client deadline is 500 ms, the gateway cannot give every dependency a separate 500 ms timeout.

```text
client budget             500 ms
network + gateway reserve  80 ms
composition budget        420 ms
dependency deadline       <= remaining shared deadline
```

Propagate deadlines and cancel downstream work when the result is no longer useful. Otherwise, timed-out client calls continue consuming backend capacity.

### 10.6 Keep domain workflows out of the generic gateway

Lightweight shaping and parallel reads can fit the gateway/BFF. Move complex business orchestration to an owned service when it includes:

- durable multi-step workflows;
- payment or inventory state transitions;
- compensating actions;
- long-running jobs;
- complex domain rules; or
- a separate scaling and ownership profile.

Microsoft's [Gateway Aggregation pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/gateway-aggregation) similarly warns that aggregation can create a bottleneck/SPOF and suggests a separate aggregation service for complex or resource-heavy logic.

### 10.7 Use and avoid

Use composition when it materially reduces high-latency client round trips and creates a stable client-specific view. Avoid it when the client and services are already colocated, one batch endpoint would suffice, or composition turns the shared gateway into an all-domain monolith.

## 11. Authentication and authorization at the gateway

### 11.1 Do not mix the two words

- **Authentication (AuthN):** Who is the caller?
- **Authorization (AuthZ):** May that caller perform this operation on this resource?

```text
AuthN: token belongs to user-891
AuthZ: user-891 may read order-8472
```

### 11.2 Typical signed-token path

```text
1. User authenticates with identity provider.
2. Client receives a short-lived access token.
3. Client sends token to gateway over TLS.
4. Gateway validates signature, issuer, audience, expiry, and required claims.
5. Gateway applies route-level policy.
6. Service applies resource/domain authorization.
```

For a JWT-like signed token, gateway nodes can often validate signatures locally using cached public keys. This removes a per-request identity-provider call, but requires safe key rotation, bounded cache freshness, correct issuer/audience checks, clock handling, and a revocation strategy. See [OAuth](./23.%20OAuth.md) and [JWT](./25.%20JWT.md).

### 11.3 Opaque tokens and online checks

An opaque token may require introspection at an authorization server. That gives central and potentially fresher state but adds latency and a mandatory dependency unless results are safely cached.

Ask:

- How quickly must revocation take effect?
- What is the acceptable cached decision lifetime?
- What happens if introspection is unavailable?
- Can read-only and money-changing APIs use different failure policies?

For security-sensitive writes, fail closed is normally safer. For already-issued, still-valid signed tokens, local verification may keep APIs available through a temporary identity control-plane outage.

### 11.4 Passing identity downstream

After validation, the gateway may forward verified identity context:

```text
X-Authenticated-User: user-891
X-Tenant-ID: merchant-42
```

Backends must accept these headers **only from a trusted, authenticated gateway path**. The gateway must delete any client-supplied spoofed version before setting them. Stronger designs use mTLS, workload identity, or a gateway-signed internal token.

### 11.5 API keys are not normally user authentication

An API key commonly identifies a calling application/project for metering and quota. If embedded in a public mobile or browser application, it is not a secret that proves a human user's identity.

AWS explicitly says not to use its API Gateway API keys for authentication or authorization; they are used with usage plans for client identification, quotas, and throttling. See [AWS API Gateway usage plans and API keys](https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-api-usage-plans.html).

### 11.6 Defense in depth

The gateway should enforce a consistent external baseline, while services enforce their own sensitive invariants:

- gateway: token validity, allowed route/scope, tenant boundary, coarse role;
- service: object ownership, transaction rules, field-level access;
- data layer: row/tenant isolation where appropriate;
- network/workload identity: only expected callers can reach internal endpoints.

### 11.7 Common mistakes

- Logging access tokens, passwords, cookies, or full sensitive bodies.
- Checking signature but not `iss`, `aud`, expiry, or algorithm policy.
- Trusting identity headers from the public client.
- Treating authentication as complete authorization.
- Calling the identity database on every request without capacity/failure planning.
- Failing open on a money-changing API because "availability is important."

## 12. Rate limiting, burst control, throttling, IP blocking, and API queues

### 12.1 These terms belong to different layers of the decision

```text
Policy:       100 requests/s per user, burst up to 200
Measurement: token bucket keyed by authenticated user ID
Action:       throttle excess with 429
Abuse rule:   temporarily block a malicious account/IP
Async option: durably enqueue eligible work and return 202
```

Do not list them as interchangeable algorithms.

### 12.2 What can be limited?

Useful keys and dimensions include:

- IP or network prefix before authentication;
- authenticated user/account;
- API key/client application;
- tenant/merchant;
- route and HTTP method;
- region or gateway cell;
- concurrent requests;
- bytes, tokens, or an operation-specific cost; and
- a hierarchy such as tenant total plus per-user sublimit.

One `GET /health` and one large video-processing submission should not necessarily cost one equal unit. Weighted cost is more accurate but easier for clients to misunderstand and for policy to misconfigure.

### 12.3 Burst limit versus sustained rate

A **token bucket** is a useful mental model:

```text
refill rate     = 100 tokens/s
bucket capacity = 200 tokens
request cost    = 1 token
```

After being idle, the caller may send a burst of up to 200 requests immediately. After the bucket empties, it can sustain roughly 100 requests/s. The capacity manages burst tolerance; the refill rate controls long-term rate.

AWS documents token-bucket throttling and distinguishes rate and burst targets for API Gateway; it also notes enforcement is best-effort rather than a mathematically hard ceiling. See [HTTP API throttling](https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-throttling.html).

### 12.4 Hierarchical protection

Use several layers because each knows different information:

```text
Internet edge  -> coarse connection/IP/DDoS protection
API gateway    -> user, tenant, API-key, route, and cost-aware limits
Service        -> last-mile concurrency and scarce-resource limits
Database/queue -> connection, admission, and storage backpressure
```

The gateway cannot know every service's current lock contention or database hot partition. Services must still protect themselves.

### 12.5 Distributed limiter choices

#### Central strongly coordinated counter

```text
all gateway nodes -> shared limiter state
```

Benefits: closer to one global quota and simpler reasoning.

Costs: remote latency, enormous counter throughput, hot keys, and a new critical dependency.

#### Independent local counters

```text
node A: limit 100
node B: limit 100
node C: limit 100
```

Benefits: fast and available.

Costs: a supposed global limit of 100 can be exceeded across nodes, especially when traffic shifts.

#### Partitioned or hierarchical budget

A coordinator allocates token leases/budgets to nodes or cells. Nodes decide locally until their allocation is nearly exhausted, then obtain more.

Benefits: hot path is local and global overshoot is bounded by outstanding allocations.

Costs: allocation complexity, fairness lag, unused stranded quota, and failover reconciliation.

The right choice depends on whether the limit protects approximate fairness, billing, security, or a hard scarce resource.

### 12.6 Throttling response

For a caller-specific rate limit, HTTP commonly uses:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 2
```

Clients should use exponential backoff with jitter and respect `Retry-After`. Immediate identical retries amplify overload.

Use `503 Service Unavailable` when the service as a whole is temporarily unable to accept work rather than blaming one caller's quota. Exact API semantics must be documented.

### 12.7 IP blocking

IP allow/block lists can stop known abusive sources early, but IP is a weak identity:

- thousands of legitimate users may share carrier-grade NAT or a corporate proxy;
- one mobile user changes addresses;
- an attacker rotates bot/proxy addresses;
- IPv6 provides large address ranges;
- the gateway may see a CDN/LB address unless trusted forwarding metadata is configured; and
- blindly trusting `X-Forwarded-For` lets clients spoof identity.

Use IP reputation as one signal alongside account, device, API-key, behavioral, and network evidence. Define the trusted proxy chain that is allowed to set client-address metadata.

Blocking commonly returns `403`, closes the connection, or silently drops at a lower layer; rate excess commonly returns `429`. The choice can be security-sensitive.

### 12.8 Temporary blocks and TTLs

For short abuse bans, store:

```text
key      = account / device / IP prefix
reason   = abuse-policy-17
expires  = timestamp
source   = fraud-system
```

Use expiration to avoid permanent accidental denial, audit changes, protect the block-list control plane, and decide how gateways behave if the block cache is unavailable.

Uber's historical gateway architecture publicly described configurable limits using user ID, user agent, IP, headers/path/body fields, plus temporary user bans stored with TTL; that is a concrete public example, not a universal design requirement. See [Uber's gateway architecture](https://www.uber.com/us/en/blog/architecture-api-gateway/).

### 12.9 API queues: when queueing is correct

For work that can complete later:

```text
Client --POST /video-jobs + idempotency key--> Gateway
Gateway --durable publish--> Job queue
Gateway <--publish confirmed--
Client  <--202 Accepted + job ID-- Gateway

Worker --> processes job --> result store
Client --> GET /video-jobs/{id} --> status/result
```

Required details:

- acknowledge only after durable acceptance at the promised durability level;
- use an idempotency key for ambiguous client retries;
- bound queue depth and oldest-message age;
- partition for ordering only where required;
- make consumers idempotent;
- use dead-letter/reconciliation policy;
- expose status and failure to the caller; and
- capacity-plan drain rate after a burst.

### 12.10 When queueing is wrong

Do not put an unbounded in-memory queue inside every gateway node and keep synchronous requests waiting indefinitely. That turns overload into high latency, memory exhaustion, timeouts, and retry storms.

Queueing is a semantic change and does not fit:

- a cacheable read that must return data now;
- a login whose result the user immediately needs;
- an operation whose caller cannot handle `202` and polling/callback;
- work that will miss its usefulness deadline before processing; or
- a system where arrival rate persistently exceeds drain capacity.

See [Rate Limiter](./12.%20Rate%20Limiter.md), [Messaging Queues](./15.%20Messaging%20Queues.md), and [Thundering Herd Problem](./26.%20Thundering%20Herd%20Problem.md) for the deeper algorithms and overload patterns.

## 13. Service discovery: how the gateway finds changing microservices

### 13.1 Plain-language problem

Hard-coding instance addresses fails because instances are ephemeral:

```text
09:00 catalog-7 at 10.1.2.7 starts
09:05 catalog-4 at 10.1.3.9 drains for deployment
09:07 catalog-9 at 10.1.4.2 crashes
```

The gateway needs a current-enough answer to:

> "Which endpoints are eligible for `catalog`, in this region/version/zone, right now?"

### 13.2 Registry record

A service registry may store:

```text
service:       catalog
instance ID:   catalog-7
address:       10.1.2.7:8080
protocol:      HTTP/2
region/AZ:     ap-south-1 / AZ-1
version:       v3.12
health:        ready
weight:        10
metadata:      canary=false, capabilities=[search,detail]
lease expiry:  10:31:25
```

Do not put secrets in public discovery metadata.

### 13.3 Registration patterns

#### Self-registration

The service instance registers and renews itself.

```text
instance -> registry: register me
instance -> registry: heartbeat / renew lease
```

Pros: simple ownership and direct metadata.

Cons: every service needs registry logic/credentials; a stalled process might keep renewing while unable to serve; deregistration can be missed on a crash.

#### Third-party registration

An orchestrator, scheduler, or sidecar observes desired/actual instances and updates discovery.

```text
orchestrator -> starts pod
orchestrator -> readiness succeeds
controller   -> publishes endpoint
```

Pros: application code stays simpler and lifecycle is centrally observed.

Cons: controller/control-plane correctness and freshness become important.

Kubernetes Services and EndpointSlices are one first-party example: the platform tracks changing pod endpoints behind a logical Service and can expose discovery through DNS or APIs. See the [Kubernetes Service documentation](https://kubernetes.io/docs/concepts/services-networking/service/).

### 13.4 Deregistration and lease expiry

#### Graceful removal

```text
mark draining -> stop new requests -> finish bounded in-flight work
-> deregister/remove endpoint -> terminate instance
```

#### Ungraceful failure

The process cannot deregister if it crashes or the network partitions. Use:

- TTL/lease expiry;
- active health checks;
- passive connection/error observations; or
- orchestrator state.

HashiCorp Consul, for example, documents TTL health checks and automatic deregistration after a configured critical period. This is a product example, not a rule that every registry implements identically. See [Consul health checks](https://developer.hashicorp.com/consul/docs/reference/service/health-check).

### 13.5 Heartbeat is not readiness

Consider a process with a heartbeat thread still running while its request pool is deadlocked:

```text
heartbeat: healthy
real requests: time out
```

Combine signals:

- lease/liveness: process or agent is present;
- readiness: can accept new traffic;
- active protocol check: listener and simple operation work;
- passive health: real requests are succeeding within latency bounds;
- outlier detection: this instance differs badly from peers; and
- resource saturation: connection/queue/CPU limits are near exhaustion.

No one signal proves complete health.

### 13.6 How gateways consume discovery

#### Query per request - usually avoid

```text
gateway -> registry -> endpoints -> choose target
```

This adds a mandatory lookup and makes the registry scale with request traffic.

#### Poll or watch, cache locally

```text
registry --incremental updates--> gateway endpoint cache
gateway --local selection-------> service instance
```

This is fast and keeps the registry off the hot path. It introduces bounded staleness and requires versioned/snapshot recovery if updates are missed.

#### DNS discovery

The gateway resolves an internal service name to a stable virtual IP or endpoint set. DNS is simple and widely compatible, but TTL and record semantics limit how quickly all consumers see change.

#### Call a stable internal load balancer

```text
gateway -> catalog.internal -> internal LB -> instance
```

The gateway does not consume individual endpoints; the internal LB does. This centralizes balancing but adds a hop and another tier.

### 13.7 Client-side versus server-side load balancing

- **Client-side:** gateway obtains endpoint set and selects an instance. Fewer proxy hops and rich locality choices; every client library must implement correct discovery, balancing, retries, and outlier behavior.
- **Server-side:** gateway calls a stable proxy/LB. Simpler clients and centralized policy; additional hop/capacity tier.

A service mesh can distribute these responsibilities between sidecar/node proxies and a control plane.

### 13.8 Stale discovery and partitions

If the registry is temporarily unavailable, a gateway can often use its last known endpoint set for a bounded period. But ask:

- Could an old IP now belong to a different workload?
- How are endpoints authenticated with mTLS/workload identity?
- How long may a removed instance receive traffic?
- Do active failures eject it before the registry recovers?
- What happens when the cached set is empty?

Failing closed prevents misrouting but can take down an otherwise working service. Bounded-stale discovery plus endpoint identity and passive health is a common compromise.

### 13.9 Discovery does not replace readiness, balancing, or authentication

The registry provides candidates. The request path still needs:

```text
candidates -> correct region/version -> ready -> not draining
-> under limits -> balancing algorithm -> authenticated connection
```

See [Load Balancers](./17.%20Load%20Balancers.md) for the full target-selection pipeline.

## 14. Request and response transformation

### 14.1 What transformation means

The gateway can adapt one contract to another:

```text
Public request                  Internal request
GET /v1/users/42?full=true  -> GetUser(user_id=42, include_details=true)
JSON over HTTP              -> Protobuf over gRPC
external field `zipCode`    -> internal field `postal_code`
```

It can also transform the response back into the public schema.

AWS documents mappings among request parameters, JSON body fields, integration requests, and method responses in [API Gateway parameter mapping](https://docs.aws.amazon.com/apigateway/latest/developerguide/request-response-data-mappings.html).

### 14.2 Useful transformations

- add a correlation/trace ID;
- remove untrusted internal headers;
- normalize paths, query parameters, or headers;
- map public fields to backend fields;
- translate protocol or media type;
- redact fields the public API must not expose;
- convert a backend error into a stable public error schema;
- add verified identity/tenant context; and
- maintain a short compatibility window during migration.

### 14.3 Costs and risks

- CPU and allocation cost at the most heavily used tier;
- buffering large bodies, which increases memory and destroys streaming;
- extra serialization/deserialization latency;
- semantic loss between protocols;
- hidden coupling to backend schemas;
- harder debugging because client and service see different messages;
- accidental exposure or deletion of fields; and
- a growing pile of permanent legacy rules.

### 14.4 Safe design rules

- validate maximum headers/body before buffering;
- compile mappings rather than interpreting expensive scripts per call;
- version, test, and own transformations;
- preserve a request/trace ID across both representations;
- never log secrets during debugging;
- define numeric, null, timestamp, enum, and error semantics explicitly;
- use streaming transforms only when they can actually stream; and
- move complex domain logic to a service/BFF.

## 15. Response caching at the gateway

### 15.1 Plain-language purpose

If many callers ask for the same safe response, the gateway may serve a stored copy:

```text
first GET /catalog/categories -> service -> cache response
next GET /catalog/categories  -> gateway cache hit
```

Benefits:

- lower client latency;
- fewer backend requests;
- lower compute/database load; and
- some resilience to a short backend slowdown when bounded staleness is acceptable.

### 15.2 Cache only when semantics allow it

Good candidates often include:

- public read-only catalog/reference data;
- configuration with a clear version/TTL;
- deterministic expensive reads; and
- responses whose staleness is acceptable.

Dangerous candidates include:

- personalized or tenant data without identity in the cache key;
- balances, permissions, rapidly changing inventory, or security decisions;
- writes and side effects;
- responses with cookies or secrets; and
- an error response cached longer than the underlying failure.

### 15.3 The cache key is a correctness boundary

A key may need:

```text
HTTP method
normalized host/path/query
API version
tenant/user or authorization scope
language/locale
content encoding/media type
selected request headers
```

Omitting `tenant_id` can leak one tenant's response to another. Including every irrelevant header can destroy the hit ratio. Define the key from response variation and authorization semantics.

### 15.4 Freshness and invalidation choices

- TTL expiration;
- explicit invalidation/event;
- versioned keys;
- conditional requests with `ETag` / `If-None-Match`;
- stale-while-revalidate where allowed; and
- manual purge for emergencies.

When a cache is flushed or cold, the backend receives a sudden miss wave. Warm gradually, collapse identical in-flight misses, and protect the origin. See [Caching](./18.%20Caching.md) and [Thundering Herd Problem](./26.%20Thundering%20Herd%20Problem.md).

### 15.5 Security and operational risks

- cache poisoning through ambiguous normalization or untrusted headers;
- cross-user data leakage;
- stale authorization;
- cache stampedes;
- eviction skew/hot keys;
- regional inconsistency; and
- treating a best-effort cache as durable storage.

AWS's API Gateway documentation describes response caching by TTL and cache key, labels caching best-effort, and advises encryption for sensitive cached data. Specific AWS defaults/limits can change, so use the [current caching documentation](https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-caching.html) rather than memorizing product numbers.

## 16. Logging, metrics, tracing, and audit

### 16.1 Four different observability products

- **Access log:** one structured record for a gateway request.
- **Application/debug log:** diagnostic events from code/policies.
- **Metric:** numeric time-series aggregate for dashboards and alerts.
- **Distributed trace:** causal path across gateway and downstream calls.
- **Audit log:** security-relevant record of access or configuration change, with stronger integrity/retention requirements.

Do not try to use verbose logs as the only metrics system.

### 16.2 Useful access-log fields

```text
timestamp
request_id and trace_id
route/API/version
authenticated principal or privacy-safe identifier
tenant/client application
source network metadata from a trusted chain
gateway region/AZ/cell/node
status and upstream status
request/response bytes
total, policy, queue, connection, and upstream latency
cache hit/miss
rate-limit/auth/circuit decision
backend service/version (not secret address if unnecessary)
```

Never log raw passwords, access tokens, session cookies, payment data, or entire sensitive bodies by default.

### 16.3 Trace context

The gateway should accept only valid trace context according to policy, create one when absent, propagate it to downstream calls, and record fan-out child spans.

```text
client span
  -> gateway span
       -> profile span
       -> order span
       -> recommendation span (timeout)
```

This separates "gateway was slow" from "gateway waited on recommendation service."

### 16.4 Keep telemetry off the synchronous success path where possible

```text
gateway -> bounded local buffer -> telemetry collector -> durable analytics store
```

If the analytics store slows, gateway request threads should not all block. Define:

- buffer bound;
- backpressure behavior;
- sampling for high-volume debug/access data;
- events that may be dropped;
- audit events that need stronger durability; and
- privacy/retention/redaction policy.

Security/audit requirements may deliberately fail closed for a narrow class of operations if a mandatory audit record cannot be persisted. That should be an explicit business rule, not an accidental dependency.

### 16.5 Metrics worth mentioning in interviews

- request rate by route/tenant/status;
- p50, p95, and p99 latency, split into gateway and upstream time;
- auth and policy denial rate;
- rate-limit decisions and token-budget exhaustion;
- cache hit ratio and origin miss load;
- open connections and in-flight requests;
- event-loop/thread/CPU/memory/network saturation;
- backend pool queue and connection-pool saturation;
- circuit state, timeout, retry, and cancellation rate;
- config version/skew across nodes;
- discovery age and endpoint count;
- ready capacity by AZ/region; and
- SLO/error-budget burn.

Avoid high-cardinality metric labels such as raw user ID or request ID. Those belong in logs/traces, with privacy controls.

## 17. Timeouts, retries, circuit breakers, bulkheads, and load shedding

### 17.1 Why the gateway can cause a cascading failure

If a backend slows, requests remain in flight longer:

```text
slower backend -> more gateway concurrency -> pool/queue fills
-> timeouts -> clients retry -> still more requests
-> unrelated APIs lose resources
```

The InMobi candidate report later in this chapter describes an interviewer explicitly probing this kind of gateway-to-downstream cascade.

### 17.2 Timeout hierarchy

Set a total client-facing deadline and smaller internal budgets:

```text
client timeout                 1,000 ms
gateway processing reserve       50 ms
backend attempt timeout          700 ms
optional retry only if safe      within remaining budget
```

Avoid a longer inner timeout than the caller's outer deadline; that creates work no caller can use.

### 17.3 Retry only under safe conditions

Potentially safe:

- a read with remaining deadline;
- a request rejected before reaching a backend;
- a write with a correctly implemented idempotency key and known retry semantics.

Dangerous:

- blindly retrying `POST /payments` after an ambiguous timeout;
- retrying at client, gateway, service, and database layers independently;
- retrying overload responses without backoff/jitter; and
- retrying across regions without shared operation identity.

Cap attempts, use exponential backoff with jitter, and budget retries. A timeout means **unknown outcome**, not necessarily "nothing happened."

### 17.4 Circuit breaker

```text
CLOSED    -> calls flow; observe failures/latency
OPEN      -> fail fast / fallback; protect backend
HALF-OPEN -> allow a small probe set to test recovery
```

Use per-service or per-route breakers so one failed service does not block unrelated traffic. A breaker is not a health oracle: thresholds, sampling windows, probe rate, and fallback correctness matter.

### 17.5 Bulkhead

Reserve independent resources:

```text
payment route pool        -> protected capacity
recommendations route pool-> may shed first
large upload pool         -> separate memory/connections
```

Without isolation, a flood of cheap-looking but slow requests can exhaust all gateway workers or outbound connections.

### 17.6 Load shedding and priority

Reject low-value work before every queue fills:

- reject new optional work;
- disable expensive transformations or recommendations;
- serve bounded-stale public cache data;
- lower per-tenant concurrency;
- preserve health/control and payment capacity; and
- return explicit degraded responses.

Failing fast can produce a smaller number of clear errors instead of making every request time out.

## 18. Region and Availability Zone (AZ)

### 18.1 Region in plain language

A **Region** is a provider-defined geographic area in which cloud services are deployed. Regions are separated for fault isolation, latency, administration, and often data-residency choices.

```text
India Region              Europe Region
  AZ-1                      AZ-1
  AZ-2                      AZ-2
  AZ-3                      AZ-3
```

Region separation is much larger than putting two servers in one building. Cross-region traffic usually has higher latency and cost, and state must be deliberately replicated or routed.

### 18.2 Availability Zone in plain language

An **Availability Zone** is an isolated infrastructure failure domain inside a Region. In AWS's definition, an AZ consists of one or more discrete data centers with separate and redundant power, networking, and connectivity. AZs in a Region are connected by high-bandwidth, low-latency links. See the AWS [Availability Zones fault-isolation description](https://docs.aws.amazon.com/whitepapers/latest/aws-fault-isolation-boundaries/availability-zones.html).

The important interview idea is not the marketing name. It is:

> Deploy redundant capacity so one local infrastructure failure does not remove every copy, while keeping communicating replicas close enough for the workload's latency needs.

### 18.3 What belongs in each AZ?

For a multi-AZ gateway path:

```text
Region
  |
  +-- AZ-1: gateway capacity + service capacity + local dependencies
  |
  +-- AZ-2: gateway capacity + service capacity + local dependencies
  |
  +-- AZ-3: gateway capacity + service capacity + local dependencies
```

Do not put gateways in three AZs if all of them synchronously depend on one instance in AZ-1. The critical dependency graph, not the number of gateway replicas, determines availability.

### 18.4 Cross-zone versus zonal routing

#### Cross-zone balancing

Any gateway/LB can send to any healthy service instance in the Region.

Pros:

- better pooling and utilization;
- handles uneven instance counts easily; and
- simpler when traffic is skewed among zones.

Cons:

- more cross-zone traffic, latency, and possible transfer cost;
- a zonal impairment can create complex cross-zone dependencies; and
- local failure isolation is weaker.

#### Zonal affinity / zone-independent cells

Traffic entering AZ-1 prefers an AZ-1 gateway and AZ-1 service capacity.

Pros:

- contains many failures and reduces cross-zone dependency;
- makes a zone easier to evacuate; and
- can reduce cross-zone traffic.

Cons:

- each zone needs enough independent capacity and dependencies;
- uneven traffic/capacity can strand resources; and
- stateful shared systems may still require cross-zone access.

State the choice rather than assuming "spread across zones" automatically means optimal routing.

### 18.5 Region versus AZ comparison

| Question | Availability Zone | Region |
|---|---|---|
| Scope | Failure domain within a geographic Region | Separate geographic deployment area |
| Typical reason | Survive data-center/power/network domain failure | Regional disaster, locality, residency, DR/business continuity |
| Network | Lower-latency links within Region | Higher-latency wide-area link |
| State replication | Often synchronous/quorum is feasible | Often async or specialized global database; synchronous writes cost more latency |
| Normal baseline | Multi-AZ is common for production HA | Multi-region only when requirements justify complexity/cost |
| Gateway impact | Replicate nodes and capacity per AZ | Replicate full gateway config, auth keys, dependencies, and data strategy |

### 18.6 Active-active and active-passive Regions

#### Active-active

```text
Users -> Region A (serving)
      -> Region B (serving)
```

Benefits: locality, continuous use of capacity, traffic can shift away from one Region.

Costs: both must have failure headroom; concurrent writes and session/data consistency become harder; shared config bugs can still hit both.

#### Active-passive

```text
Users -> Region A (primary)
         Region B (warm/hot standby)
```

Benefits: simpler single write ownership and often lower running complexity.

Costs: failover detection/routing time, standby drift, unused capacity, cold caches/connections, and explicit RTO/RPO.

The gateway is only one part. A failover is not successful if DNS reaches Region B but its database, secrets, queues, identity path, or capacity is not ready.

## 19. DNS from the beginning: local, recursive, root, TLD, authoritative

### 19.1 What DNS does

The Domain Name System is a distributed naming system. For this chapter, its most visible job is mapping:

```text
api.example.com -> one or more reachable endpoint addresses
```

DNS also carries mail-routing, service, verification, and delegation records. It is not merely one global phone book server.

### 19.2 The four "levels" in the bullet need one extra distinction

The common lookup path includes:

1. **local client/stub and caches** - browser, OS, or local network behavior;
2. **recursive resolver** - ISP, enterprise, or public resolver that performs/caches lookups for the client;
3. **root authoritative servers** - point toward the relevant top-level domain;
4. **TLD authoritative servers** - point toward the domain's authoritative servers; and
5. **authoritative servers for the domain** - provide the configured final record or another delegation/alias.

"Local DNS" is ambiguous. People often mean the recursive resolver configured on the device/router, but the browser/OS may also maintain local caches. In an interview, say **stub resolver/local cache** and **recursive resolver** separately.

### 19.3 Cold lookup walk-through

Suppose the client needs `api.shop.example.com` and no relevant cache exists:

```text
Browser / application
        |
        | asks operating-system stub resolver
        v
Recursive resolver
        |
        | 1. Where is .com?
        v
Root server
        |
        | referral: ask these .com TLD servers
        v
TLD server
        |
        | referral: ask authoritative servers for example.com
        v
Authoritative server for example.com
        |
        | answer: api.shop.example.com = 203.0.113.10
        v
Recursive resolver caches answer/referrals and returns it
        |
        v
Client connects to 203.0.113.10:443
```

For a familiar real name like `api.example.com`, the root refers the resolver to `.com` servers; the `.com` servers refer it to authoritative servers for `example.com`; those servers answer for `api.example.com` or return a further record such as a CNAME.

### 19.4 What each participant knows

#### Stub resolver / local cache

Knows how to ask a configured recursive resolver. It usually does not walk the full hierarchy itself.

#### Recursive resolver

Answers from cache if possible; otherwise follows referrals/queries until it obtains an answer or error. One cached lookup can serve many clients.

#### Root server

Normally does **not** store the address of every website. It knows delegations needed to reach TLD authoritative service.

#### TLD server

Normally does **not** hold every host record inside the domain. It provides delegation information for domains under the TLD.

#### Authoritative server

Answers from locally configured authoritative zone data for the domain. RFC 8499 defines an authoritative server as one that knows a zone's content from local knowledge. See [RFC 8499 - DNS Terminology](https://datatracker.ietf.org/doc/html/rfc8499).

### 19.5 Common record types

| Record | Purpose |
|---|---|
| `A` | Name to IPv4 address |
| `AAAA` | Name to IPv6 address |
| `CNAME` | Alias one name to another canonical name |
| `NS` | Authoritative name servers/delegation for a zone |
| `MX` | Mail exchangers for a domain |
| `TXT` | Text-based policy/verification data; exact use depends on protocol |
| `SRV` | Service location including target and port for supporting clients |
| `PTR` | Reverse mapping from address space to a name |

Provider-specific `ALIAS`/`ANAME`-style records are conveniences, not the same thing as a standard DNS record type everywhere.

### 19.6 Caching and TTL

A DNS record's **TTL (time to live)** tells caches how long the data may normally be reused before it should be refreshed.

```text
long TTL  -> fewer DNS queries, stable cache, slower planned/failure change
short TTL -> faster potential change, more DNS load, still not instant failover
```

Why not instant?

- existing cached answers remain until refresh behavior permits change;
- some resolvers/clients cap or alter cache behavior;
- clients may keep established TCP/TLS connections beyond the DNS TTL;
- applications may cache independently;
- health detection and authoritative update have their own delay; and
- after receiving several addresses, client selection/retry behavior varies.

TTL is not a command that disconnects every client at expiry.

### 19.7 Negative caching

DNS can cache non-existence or errors. Accidentally publishing `NXDOMAIN` during a migration can therefore remain visible after the record is restored. Plan record creation, TTL reduction, cutover, and cleanup in stages.

### 19.8 Why DNS itself can be highly available

A domain normally delegates to multiple authoritative servers. Large DNS services distribute server instances geographically, often using anycast so many sites advertise the same service address.

Still consider:

- registrar/account compromise;
- bad zone publication;
- expired domain/delegation mistakes;
- DNSSEC key/validation errors when used;
- DDoS and provider dependency; and
- whether all authoritative servers share one control-plane failure.

Multiple DNS servers do not protect against one bad record pushed to all of them.

## 20. DNS-based load balancing and global traffic steering

### 20.1 Basic idea

One name can have multiple answers:

```text
api.example.com -> 203.0.113.10  (Region A)
                -> 198.51.100.20 (Region B)
```

The authoritative DNS service can vary answers using a policy. The resolver caches the answer, and the client chooses/connects to an address.

### 20.2 Common policies

- **Simple / multi-value:** return one or several endpoints.
- **Weighted:** shift an approximate fraction of DNS answers between endpoints.
- **Latency-based:** choose the endpoint expected to provide better network latency from the resolver/client-location estimate.
- **Geolocation/geoproximity:** steer by geographic policy, often for locality or compliance.
- **Failover:** return primary while healthy, secondary when the health policy changes.
- **IP/network-based:** use source-network rules where supported.

These names and exact semantics are product-specific. AWS Route 53, for example, documents simple, failover, geolocation, geoproximity, latency, IP-based, multivalue, and weighted policies in its [routing-policy documentation](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/routing-policy.html).

### 20.3 DNS balances resolutions, not exact requests

Suppose a corporate resolver serves 50,000 employees:

```text
50,000 clients -> one recursive resolver -> one cached DNS answer
```

Those users may all receive the same cached endpoint set. Therefore, 50/50 weighted DNS does not promise exactly 50/50 HTTP requests. Traffic depends on resolver aggregation, cache duration, connection reuse, client address choice, request rates, and retries.

Use DNS for coarse global/region steering; use a proxy/load-balancer tier for finer connection/request distribution.

### 20.4 Health-based DNS failover limitations

```text
failure occurs
  -> health checker detects it
  -> policy marks endpoint unhealthy
  -> new DNS answers change
  -> caches refresh
  -> clients reconnect/retry
```

Each arrow takes time. Existing connections do not move. A health checker may also see a healthy gateway while a critical backend path is broken, or falsely mark a healthy endpoint down during a network partition.

Define the health signal and avoid automatic flapping. For a dangerous stateful failover, a controlled operation with fencing may be safer than immediate DNS reaction.

### 20.5 Resolver location is not always user location

The authoritative DNS service generally sees the recursive resolver's query source, not necessarily the end user's exact address. A resolver may be geographically far from its clients. EDNS Client Subnet can convey an address prefix in supporting setups, but has privacy, cache, and implementation trade-offs.

Measure real user performance rather than assuming geographic DNS always chooses the lowest-latency endpoint.

### 20.6 DNS load balancing versus a proxy/global accelerator

| Property | DNS steering | Proxy/anycast global entry |
|---|---|---|
| Decision unit | DNS resolution/cache lifetime | Connection, flow, or request depending on system |
| In data path | No, after resolution | Yes |
| Failover speed | Limited by detection, caches, and reconnect | Can redirect new traffic more directly, but existing connections still matter |
| Per-request visibility | None | Possible at L7 |
| Precise load feedback | Limited/coarse | Richer real-time endpoint feedback |
| Cost/complexity | Relatively simple | Additional serving fleet and data path |
| Client source visibility | Endpoint sees client normally | May require forwarding metadata/protocol support |

An **anycast** address is advertised from multiple network locations; routing selects a reachable path/site. It can provide a stable global entry, but Internet routing is not guaranteed to pick the application-optimal site. Twitter's historical public edge write-up described this exact lesson: globally anycasted API addresses were broadly available but could route users suboptimally, leading Twitter to use measured traffic mapping. See [Twitter/X's 2019 edge post](https://blog.x.com/engineering/en_us/topics/infrastructure/2019/expand-the-edge).

### 20.7 Multi-level load balancing

A common design uses both:

```text
DNS / global traffic manager
        | chooses region or global edge
        v
Regional load balancer / gateway fleet
        | chooses gateway then service/instance
        v
Backend service
```

This separates slow-changing global placement from fast, health-aware local balancing.

## 21. Worked multi-region gateway design

### 21.1 Requirements

Assume a shopping API needs:

- peak 1,000,000 RPS globally;
- reads served near users;
- order writes with one authoritative business outcome;
- survival of one gateway node and one AZ failure without manual action;
- regional recovery for a severe outage;
- user-level and tenant-level quotas; and
- p99 latency and error SLOs defined per route.

The one-million number is illustrative; no product capacity is implied.

### 21.2 High-level design

```text
                                      +----------------------+
                                      | Gateway control plane|
                                      | versioned config     |
                                      +----------+-----------+
                                                 |
                                      signed/versioned rollout
                                                 |
Client                                           v
  |                                     last-known-good local config
  v
Authoritative DNS / global traffic steering
  |
  +---------------------------+----------------------------+
  |                                                        |
  v                                                        v
Region A                                                Region B
3 AZs                                                   3 AZs
  |                                                        |
Edge/LB fleet                                           Edge/LB fleet
  |                                                        |
Gateway cells                                           Gateway cells
  |  - local route/auth keys                              |  - local route/auth keys
  |  - local quota budgets                                |  - local quota budgets
  |  - async telemetry                                    |  - async telemetry
  |                                                        |
  +--> read services / regional caches                     +--> read services / caches
  |
  +--> order API -> authoritative order write path <-------+
                    idempotency + replicated durability
```

### 21.3 Request flow

1. The client resolves `api.shop.example`; global steering chooses a healthy region/edge based on policy.
2. The regional entry tier balances across ready gateway nodes in multiple AZs.
3. The gateway validates TLS/application limits and matches a route.
4. It authenticates the token using cached rotated keys and applies route/tenant/user quotas.
5. For public reads, it checks a correctly keyed regional cache.
6. On a miss, it obtains eligible endpoints from a local watched discovery snapshot and balances a backend connection/request.
7. For an order write, it passes one idempotency key to the authoritative order workflow; cross-region retry uses the same key.
8. The gateway propagates deadlines and trace context, returns a stable public schema, and emits bounded asynchronous telemetry.

### 21.4 Availability design

- Each AZ has gateway and service capacity; the remaining zones can handle peak plus stated margin after one AZ loss.
- Node-local durable business state is avoided.
- Gateway data planes continue on last known good config during control-plane interruption.
- Discovery uses local snapshots, passive health, and bounded staleness.
- Shared quota/auth/cache systems have explicit fail-open/fail-closed behavior per route.
- Config, certificates, and signing keys roll out by cell/region with canaries and rollback.
- Region failover is tested with backend data, queues, secrets, identity, DNS, and capacity - not gateway reachability alone.

### 21.5 Regional write correctness

Two active gateway regions do not automatically mean two active order writers. Options include:

- route each order/account to one home write region;
- use a globally coordinated database/quorum with latency cost;
- accept concurrent writes only with domain-safe conflict rules; or
- run active-passive writes while reads remain active-active.

For payments/orders, do not let two regions independently create a side effect after an ambiguous timeout. Preserve a global logical attempt ID and reconcile with the authoritative outcome.

### 21.6 Failure walkthroughs

#### One gateway node dies

Connections/in-flight work on it may fail. The regional LB removes it. Clients retry safe/idempotent operations through other nodes. Capacity remains within SLO.

#### One AZ fails

New traffic stops entering the AZ or is rerouted. The other zones carry the load. Discovery removes zonal service endpoints. Cross-zone state dependencies are checked for hidden impact.

#### Rate-limit store fails

Public reads may use conservative node-local emergency budgets; account creation and expensive writes may fail closed or use preallocated leases. Alert on overshoot risk. The precise policy follows abuse and resource risk.

#### Region A fails

Health and an authorized recovery mechanism steer new clients to Region B. Existing connections reconnect. Region B already has gateway config, keys, dependencies, data access, and failure capacity. Write authority is fenced/transferred according to the database design.

#### A global config is bad

Automatic promotion halts on cell canary errors. Roll back to the last known version. If it escaped to several cells, independent version pinning and blast-radius isolation keep unaffected cells serving.

## 22. Trade-offs, alternatives, and when to use or avoid a gateway

### 22.1 Managed versus self-hosted gateway

| Choice | Benefits | Costs/risks |
|---|---|---|
| Managed gateway | Provider operates scaling, patching, basic HA, integrations, and control plane | Quotas, pricing, feature constraints, provider coupling, less implementation visibility/control |
| Self-hosted gateway | Full protocol/plugin/deployment control; portable architecture possible | You own capacity, multi-AZ, upgrades, CVEs, config safety, observability, and on-call burden |

Check current product quotas rather than memorizing them. For example, AWS documents default per-account, per-Region throttling quotas and says some are increaseable in the [current API Gateway quota page](https://docs.aws.amazon.com/apigateway/latest/developerguide/limits.html). Those are account defaults, not the architectural maximum of all gateway systems.

### 22.2 One shared gateway versus gateway cells/domain gateways

#### One shared gateway

Pros: consistent policy, one public surface, better capacity pooling.

Cons: organization and configuration bottleneck, huge blast radius, noisy neighbors, release coupling.

#### Domain/cell gateways

Pros: team autonomy, independent scaling, smaller blast radius, domain-specific policy.

Cons: policy drift, more fleets/cost, duplicate operations, harder cross-domain discovery.

A common compromise is a thin shared edge baseline plus domain-owned BFF/composition services behind it.

### 22.3 Gateway versus BFF

- Generic gateway: TLS, auth baseline, quotas, routing, simple transforms, telemetry.
- BFF: client-specific view composition and presentation logic.

Separating them keeps mobile/web change from destabilizing the shared security and traffic tier.

### 22.4 Gateway versus service mesh

- API gateway mainly handles **north-south** client/partner traffic and public API contracts.
- Service mesh mainly handles **east-west** service-to-service identity, routing, telemetry, and resilience.

They overlap technically but operate at different trust boundaries and ownership scopes. Do not expose every internal mesh service publicly through automatic discovery.

### 22.5 Gateway versus GraphQL federation

GraphQL can provide client-selected fields and federated composition. It does not remove the need for:

- authentication and authorization;
- query depth/complexity/cost limits;
- backend batching to avoid N+1 calls;
- caching semantics;
- timeouts/partial errors; and
- gateway fleet availability.

It may be the public API layer behind or within a gateway, not a universal replacement.

### 22.6 When a separate API gateway is useful

- many external clients need one stable API surface;
- many microservices need consistent policy;
- API versioning and contract translation are real needs;
- partner APIs need metering, keys, quotas, or developer lifecycle;
- mobile networks benefit from BFF/composition;
- several backend protocols must be hidden; or
- centralized access audit is required.

### 22.7 When to avoid or defer it

- one small monolith behind a capable managed L7 load balancer;
- a private service with few trusted callers and an existing mesh/ingress meeting requirements;
- ultra-low-latency traffic where extra proxy policy is not worth the hop;
- raw bulk media/data transfer better served directly from object storage/CDN using scoped URLs;
- a protocol the candidate gateway cannot stream or proxy efficiently; or
- an organization unable to operate the new critical tier safely.

### 22.8 Gateway anti-pattern: the new monolith

Symptoms:

- every product team edits one repository;
- gateway deploys contain business workflows;
- one slow composition starves all APIs;
- transformations never retire;
- releases require organization-wide coordination; and
- gateway owners become the approval bottleneck.

Fix with a thin edge contract, self-service validated configuration, domain/BFF ownership, cells, isolated resources, and movement of business logic to services.

## 23. Real-system examples

The labels matter: public write-ups are historical snapshots, and the illustrative sketches are not claims about a company's current private architecture.

### 23.1 WhatsApp: persistent messaging is not an ordinary request/response API

**Illustrative design, with one narrowly scoped public gateway example.** A messaging app has at least two different traffic shapes:

```text
                          +--> account/profile API services
Device -> DNS/edge -------+--> media authorization -> object/CDN transfer
                          +--> long-lived connection gateway -> message routing
```

Gateway lessons:

- login/profile/group-management APIs can use ordinary auth, quotas, and routing;
- the long-lived message connection needs connection-aware gateways, reconnection, sequence/resume state, and aggressive isolation from slow request APIs;
- encrypted message contents should not be casually transformed or logged;
- media bytes should normally use a specialized upload/download path rather than pass through JSON composition; and
- per-account/device limits are more useful than IP-only limits on carrier NAT.

Meta publicly described a **specific 2025 WhatsApp Private Processing feature** in which a device establishes an Oblivious HTTP (OHTTP) connection to a Meta gateway through a third-party relay, separating what the relay and gateway can observe. That proves a gateway role for that feature only; it does not document WhatsApp's whole messaging edge. See [Meta Engineering - Private Processing for WhatsApp AI tools](https://engineering.fb.com/2025/04/29/security/whatsapp-private-processing-ai-tools/).

### 23.2 YouTube: keep API control traffic separate from video data

**Illustrative YouTube-style design; Google network evidence is company-wide, not YouTube-specific.**

```text
Creator -> API gateway -> authenticate / create video metadata
                     |
                     +-> issue scoped upload session/URL

Creator ==================> upload/data-ingestion path -> object storage
                                                       -> transcode pipeline

Viewer -> API gateway -> metadata, search, entitlement, manifest
Viewer ================================================> CDN video segments
```

The gateway is valuable for metadata APIs, authorization, quotas, and job submission. Routing multi-gigabyte media through a composition-oriented gateway adds unnecessary bandwidth, buffering, and blast radius; use direct scoped upload/download or dedicated streaming ingress.

At Google's network layer, the public Maglev paper describes a distributed software network load balancer whose capacity is expanded by adding machines, with routers using ECMP to distribute traffic to Maglev nodes. This supports the general lesson that the frontend entry path itself is a fleet, but it does **not** prove YouTube's current API-gateway topology. See [Google Research - Maglev](https://research.google/pubs/maglev-a-fast-and-reliable-software-network-load-balancer/).

### 23.3 Uber: a first-party gateway evolution with scale and boundary lessons

**Historical public production example.** Uber described this broad evolution:

```text
early monolithic public API
        |
large all-encompassing gateway
  routing + auth + transforms + composition + business logic
        |
thin self-service Edge Gateway
  + product/presentation services behind it
```

The 2020 write-up reported that the second generation reached 800,000 peak requests/second, communicated with hundreds of downstream services, and was deployed as 40+ independent services. It also described the engineering and failure risks of allowing composition and business logic to grow in a shared gateway.

The later Edge Gateway separated a thinner edge layer, used generated configuration/artifacts, and supported authentication, request/response transformation, protocol conversion, rate limiting, circuit breaking, audit logging, and data-center affinity. This is a strong real example of both gateway power and the "gateway becomes a monolith" failure mode. See [Uber - Designing Edge Gateway](https://www.uber.com/us/en/blog/gatewayuberapi/) and [The Architecture of Uber's API Gateway](https://www.uber.com/us/en/blog/architecture-api-gateway/).

### 23.4 Netflix: Zuul as an edge filter and routing service

**Historical public production example (2013 write-up), not a current-architecture claim.** Netflix introduced Zuul as an edge service in front of its API:

```text
Client -> Zuul filters -> routing/load balancing -> origin/API clusters
             |
             +-> authentication/access rules
             +-> traffic shaping/debug routing
             +-> metrics and load shedding
```

The public post described pre/routing/post-style filters, dynamic routing and authorization rules, Ribbon for outbound load balancing, Eureka for discovery, and Hystrix for origin protection/load shedding. The reusable lesson is to make policy stages composable and observable while bounding the power and risk of dynamic filters. See [Netflix TechBlog - Announcing Zuul](https://netflixtechblog.com/announcing-zuul-edge-service-in-the-cloud-ab3af5be08ee).

### 23.5 Twitter/X: DNS/anycast entry is related to, but different from, the API proxy

**Historical public production example (2019).** Twitter described:

```text
api.twitter.com
      |
global anycast / measured traffic mapping
      |
edge PoP: TLS termination + request proxy
      |
data-center origins / API services
```

The post reported an edge handling millions of requests per second, with Twitter-operated and public-cloud PoPs. It said the initial globally anycasted API addresses improved broad availability but could route users suboptimally; a traffic-mapping system used real-user measurements to recommend better edge VIPs with fallback to known-good anycast.

Lessons:

- DNS/address steering chooses an edge, not the final microservice instance;
- an edge proxy terminates/proxies application traffic after that choice;
- network-nearest is not always application-fastest; and
- client fallback still matters when a recommended route fails.

See [Twitter Engineering - #ExpandTheEdge](https://blog.x.com/engineering/en_us/topics/infrastructure/2019/expand-the-edge).

### 23.6 Amazon/AWS: a managed gateway and an illustrative shopping request

**Verified platform behavior plus an illustrative retail flow; not a claim about Amazon.com's private retail architecture.** AWS API Gateway is a managed example that can expose APIs, apply authentication/authorization integrations, throttle, transform, cache, log, and route to functions or HTTP backends.

An illustrative shopping flow:

```text
Customer -> DNS -> regional managed API Gateway
                     |
                     +-- GET /catalog -> cache/catalog API
                     +-- POST /orders -> order API -> durable order workflow
                     +-- GET /summary -> BFF -> orders + shipment + profile
```

The gateway gives the app one contract and protects APIs. It should not own inventory reservation or payment truth. API composition can build an order-summary view; order placement uses an idempotent authoritative workflow.

AWS also publishes a multi-region API Gateway pattern using regional endpoints and Route 53/ARC-based failover, emphasizing that recovery controls should avoid dependence on a failed primary Region's control plane. See [AWS - Multi-Region failover for API Gateway](https://aws.amazon.com/blogs/compute/implementing-multi-region-failover-for-amazon-api-gateway/).

## 24. Interview angle

### 24.1 How this topic appears

An interviewer may:

- ask directly, "Design an API gateway";
- point to the single gateway box and ask how it handles one million RPS;
- ask whether it is a single point of failure;
- ask gateway versus load balancer and their order;

- ask how rate limits remain global across gateway nodes;
- ask how a gateway finds service instances during autoscaling/deployments;
- ask what happens during an AZ/Region failure;
- ask how `api.example.com` resolves; or

The interviewer is usually testing whether the candidate can open a convenient one-box abstraction and reason about scale, state, failure, contracts, and trade-offs.

### 24.4 General answer spine: `GATEWAY`

Use these cues to reconstruct an answer instead of memorizing a paragraph:

1. **G - Goals and traffic:** clients, protocols, API operations, RPS/bytes/concurrency, latency SLO, security, and failure model.
2. **A - Arrival path:** DNS/global steering -> edge/LB -> gateway fleet; explain what comes before and after.
3. **T - Traffic/API policy:** routing, AuthN/AuthZ boundary, limits, transformations, caching, composition, logging.
4. **E - Endpoint discovery:** registry/Service/LB, readiness, heartbeats/leases, local cached endpoint set, balancing.
5. **W - Workload protection:** timeouts, idempotent retries, circuit breakers, bulkheads, queues only for async semantics, load shedding.
6. **A - Availability and scale:** stateless data plane, per-node load test, horizontal scale, multi-AZ headroom, cells, last-known-good config, optional multi-region.
7. **Y - Yardsticks and trade-offs:** p99/error SLO, saturation, config version, downstream amplification, managed/self-hosted, gateway/BFF/mesh alternatives.

### 24.5 Answer spine: "How will it handle one million RPS?" - `SCALE`

1. **S - Specify the load:** per second, bytes, protocols, cache/composition mix, regional skew.
2. **C - Capacity-test one unit:** measure safe node capacity at p99 SLO with real policies and failures.
3. **A - Add horizontal capacity:** redundant nodes/cells/AZs/Regions plus burst and failure headroom.
4. **L - Localize hot-path work:** in-memory config/keys, connection pooling, watched discovery, hierarchical limiter, async telemetry.
5. **E - End-to-end bottlenecks:** auth, limiter, registry, cache, logs, fan-out, retries, and backend capacity; shed load before collapse.

Spoken skeleton:

> "I first need to define whether that is one million peak RPS and the payload, TLS, policy, and fan-out mix. I would load-test one gateway node to a p99-safe capacity, then provision a stateless multi-AZ fleet with burst and zone-loss headroom rather than extrapolating from CPU alone. Routes, signing keys, and discovery snapshots stay local; quota state is partitioned or leased; telemetry is asynchronous. I would cell the fleet, protect every downstream with deadlines and bulkheads, and verify the identity, limiter, registry, cache, and backend systems for amplified load."

### 24.6 Answer spine: "Why is the gateway not an SPOF?"

```text
logical service, not box
-> replicas across hosts/AZs
-> redundant ingress and DNS
-> stateless durable semantics
-> last-known-good data plane
-> survivor capacity
-> dependency failure policies
-> config/certificate common-mode protection
-> region strategy only if required
```

### 24.7 Answer spine: "How does service discovery work?" - `LEASE`

1. **L - Logical service name** maps to endpoint records and metadata.
2. **E - Enrollment:** self-registration or orchestrator-controlled publication after readiness.
3. **A - Alive versus ready:** leases/heartbeats plus active/passive health; heartbeat alone is insufficient.
4. **S - Snapshot consumption:** gateway watches/polls/DNS and chooses locally; avoid lookup per request.
5. **E - Expiry/removal:** drain on graceful shutdown, lease/health expiry on crash, bounded stale fallback and endpoint identity.

### 24.8 Answer spine: "Walk me through DNS"

```text
browser/OS cache and stub
-> recursive resolver cache
-> root referral
-> TLD referral
-> domain authoritative answer
-> resolver caches by TTL
-> client chooses address and opens TCP/QUIC + TLS
-> regional edge/LB/gateway handles API traffic
```

Add: DNS is not necessarily queried for each request, and its load-balancing/failover behavior is constrained by caches, connections, and client retry behavior.

## 25. Common mistakes and precise corrections

| Mistake | Precise correction |
|---|---|
| "The API gateway is one server." | It is one logical endpoint backed by a fleet; then inspect shared dependencies and configuration. |
| "Put a gateway in front to handle scale." | Quantify traffic, load-test safe per-node capacity, add failure headroom, and capacity-plan downstream amplification. |
| "API gateway replaces load balancer." | It may include L7 balancing, but self-hosted gateway replicas still need ingress distribution and backend instances still need selection. |
| "Load balancer always comes first." | DNS logically resolves first; ingress LB often precedes a self-hosted gateway; a backend LB can follow it. |
| "The gateway authenticates, so services can trust all internal traffic." | Secure the gateway-to-service path and keep resource/domain authorization in the owning service. |
| "JWT means no auth dependency." | Local validation still depends on key distribution, rotation, claims, expiry, and revocation policy. |
| "API key authenticates the user." | It commonly identifies/meters an application; use real user/workload authentication. |
| "Rate limit by IP." | IP is one coarse signal; NAT, proxies, mobile changes, spoofed forwarding headers, and bot rotation make it weak identity. |
| "Burst limit is the same as sustained RPS." | Bucket capacity permits a short burst; refill rate controls sustained traffic. |
| "Queue requests when overloaded." | Queue only when the API supports durable asynchronous acceptance; bound backlog and expose status. |
| "Heartbeat means service is healthy." | It indicates lease renewal; combine readiness, active/passive health, latency, and saturation. |
| "Ask the registry on every request." | Watch/poll/cache an endpoint snapshot or use a stable server-side LB; define bounded staleness. |
| "Transform everything at the gateway." | Keep mappings simple, owned, versioned, bounded, and move domain logic to a BFF/service. |
| "Cache all GET requests." | Cache only with explicit authorization, key, freshness, invalidation, and sensitivity semantics. |
| "Log request/response bodies for debugging." | Default to structured metadata; redact secrets/PII and use tightly controlled sampling. |
| "Three gateway replicas remove the SPOF." | They do not protect against one AZ, global config, certificate, identity, registry, limiter, or backend failure. |
| "Multi-AZ means multi-region." | AZs are failure domains inside one Region; multi-region is a separate deployment/data strategy. |
| "AZ equals one data center." | In AWS's definition, an AZ can contain one or more discrete data centers. |
| "Root DNS knows the API server IP." | Root normally refers to the TLD; the TLD refers to the domain's authoritative servers. |
| "DNS gives per-request 50/50 load." | DNS answers are cached by resolvers/clients; traffic proportions are approximate and can be skewed. |
| "Short TTL gives instant failover." | Detection, cache behavior, existing connections, application caches, reconnects, and backend readiness all add delay. |
| "Active-active gateway makes writes active-active." | Gateway locality does not define database write ownership or conflict semantics. |

## 26. Compact review sheet

### 26.1 One-screen architecture

```text
Client
  |
  v
DNS: resolver -> root -> TLD -> authoritative
  |
  v
Global edge / traffic steering
  |
  v
Regional LB (for self-hosted gateway replicas)
  |
  v
API Gateway fleet across AZs
  |-- route/version
  |-- authenticate + coarse authorize
  |-- rate/burst/concurrency/size controls
  |-- transform/cache/light compose
  |-- timeout/circuit/bulkhead/load shed
  |-- structured log/metric/trace
  |
  v
Service discovery / internal LB / mesh
  |
  v
Service instances -> data/queue/cache
```

### 26.2 Responsibility map

```text
DNS              = find/steer to an entry endpoint
Load balancer    = choose among eligible instances/flows/requests
API gateway      = expose and govern application APIs
Service registry = publish logical service -> changing endpoints
Region           = geographic deployment/failure area
AZ               = isolated infrastructure failure domain within Region
Queue            = durable asynchronous acceptance, not generic waiting room
```

### 26.3 Gateway policy checklist

```text
contract and route
TLS and identity
authorization boundary
rate + burst + concurrency + size
timeout + retry + idempotency
circuit + bulkhead + load shedding
discovery + health + drain
transformation and composition limits
cache key/freshness/security
logs + metrics + trace + audit
versioned config + canary + rollback
multi-AZ capacity + optional multi-region data plan
```

### 26.4 The ordering answer in one line

> **DNS -> optional global edge -> ingress LB for gateway replicas -> API gateway -> service discovery/internal LB -> service instance**, with layers combined or omitted when one product or a simpler requirement covers several responsibilities.

## 27. Say-out-loud 30-second narrative

> "An API gateway is an application-aware front door: it gives clients a stable API, authenticates and limits calls, and routes to changing services. It is a logical service, not one machine; a load balancer spreads traffic across stateless nodes in multiple AZs, while DNS steers clients to a Region. Gateways use discovery to find healthy backends. At scale, I load-test node capacity, keep policy local, add failure headroom, and protect downstreams with deadlines, circuit breakers, bulkheads, and load shedding."
