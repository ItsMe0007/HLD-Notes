# Caching, Distributed Caching, and Fault Tolerance

> Beginner-first HLD notes expanded from the supplied bullets: **caching, fault tolerance, why distributed caching is needed, cache-aside, read-through, write-around, write-through, write-back/write-behind, and cache eviction policies.**
>
> The central idea is not merely “put Redis in front of the database.” A sound cache design answers: **what is copied, who owns the truth, how a key is found, how stale data is bounded, what happens on every read and write, and whether the origin survives when the cache fails.**

## 0. Scope, assumptions, corrections, and evidence labels

### Scope

This chapter focuses on application and distributed caches used in high-level system design. It also introduces browser, CDN, and in-process caches so that “cache” is not mistaken for one specific product.

The durable store is called the **origin**, **backing store**, **database**, or **system of record** depending on context. Redis and Memcached are examples, not automatic answers.

### Terminology corrections that matter in interviews

1. **Expiration, eviction, and invalidation are different.**
   - **Expiration:** an item becomes unusable because its time to live (TTL) ended.
   - **Eviction:** the cache removes an item, usually to reclaim limited space.
   - **Invalidation:** the application or another system deliberately removes/marks an item stale because the underlying fact changed.
2. **Cache-aside and read-through have similar miss behavior but different ownership.** In cache-aside, application code loads the database and cache. In read-through, the cache/cache library invokes a configured loader.
3. **Write-back and write-behind normally mean the same family of strategy:** acknowledge after writing to the cache and persist to the backing store asynchronously.
4. **Write-through is used inconsistently across articles and products.** In this chapter it means an inline cache/facade accepts the write and synchronously propagates it to the durable store before reporting success. Some teams also use the term for application code that writes the database and then the cache. State the exact order instead of relying on the label.
5. **Write-around is not a complete consistency solution.** It says writes bypass cache allocation. If an older value is already cached, it must still be invalidated or versioned.
6. **A cache is not a backup.** A typical cache contains disposable copies and may evict them at any time. A system using write-back has temporarily made its caching tier part of the durable write path and must engineer it accordingly.
7. **A distributed cache is not automatically fault-tolerant.** Replication, failure detection, failover, client timeouts, origin protection, and cold-start behavior determine fault tolerance.

### Evidence labels

- **Verified product behavior:** current first-party documentation linked in the chapter.
- **Public production snapshot:** a company publicly described this behavior at the publication date; it is not a claim about its complete private architecture today.
- **Candidate-reported interview prompt:** a public self-report, often anonymous, not a verified company question bank.
- **Illustrative design:** a reasonable design for a named product, not a claim about its private implementation.
- **Worked estimate:** arithmetic under stated assumptions, not a production benchmark.
- **Unverified:** a claim that should be checked before being repeated as fact.

## 1. Plain-language foundation

### 1.1 Start with a slow, repeated trip

Imagine a restaurant in which the waiter walks to a distant warehouse every time any customer asks for ketchup.

```text
Customer -> Waiter ----------------------> Warehouse
                    long repeated trip
```

Keeping one bottle near the tables makes repeated requests faster:

```text
                      miss: fetch and keep a copy
Customer -> Waiter -> Nearby shelf ----------------> Warehouse
                         |
                         +-- hit: return immediately
```

The nearby shelf is a **cache**: a faster, usually smaller place holding copies of data that are expensive to fetch or compute again.

The metaphor also exposes the hard parts:

- The shelf has limited space. Which bottle is removed when it is full?
- A bottle may become old. How long may it remain?
- The warehouse may replace the recipe. How does the shelf learn?
- Several dining rooms may have different copies. Which is current?
- If the shelf disappears, can the warehouse handle every waiter at once?

Caching makes the normal path faster by creating a second copy. The cost is **copy coordination and a new failure mode**.

### 1.2 Source of truth versus disposable copy

In the safest common model:

```text
Database: authoritative, durable truth
Cache:    derived, disposable copy
```

Suppose a product price is ₹500 in the database and copied to the cache:

```text
Database: product:42 -> ₹500   (truth)
Cache:    product:42 -> ₹500   (copy)
```

If the price changes to ₹550 only in the database, the cached ₹500 is **stale**. The cache may be fast and still be wrong.

If a cache node restarts and loses the copy, the application should normally rebuild it from the database. If losing the cache loses the only copy of a payment, order, or message, that component is no longer “just a cache.”

### 1.3 Cache key, value, hit, and miss

A cache is commonly accessed like a key-value map:

```text
key                         value
product:v3:42        ->     {"name":"Headphones","price":550}
user-profile:v7:91   ->     serialized profile bytes
feed-page:v2:8:0     ->     first page of user 8's feed
```

- A **cache hit** occurs when the required usable value is present.
- A **cache miss** occurs when it is absent, expired, evicted, invalidated, or rejected as too old.
- **Hit ratio** is the fraction of lookups served as hits:

  ```text
  hit ratio = hits / (hits + misses)
  ```

- **Miss penalty** is the extra work on a miss: cache lookup, origin call/computation, optional cache fill, and serialization/network overhead.

A hit is useful only if the key identifies the correct result. Caching `GET /price` without including the product, currency, region, or customer-specific pricing inputs can quickly return the wrong answer.

### 1.4 Why repeated access makes caching work

Caching benefits from **locality**:

- **Temporal locality:** something used recently is likely to be used again soon. A viral tweet is read repeatedly within minutes.
- **Spatial or related locality:** requesting one item suggests nearby/related items will be needed. A video player may request the next chunks.
- **Popularity skew:** a small hot portion of a large data set often receives a large portion of reads.

The subset that needs to remain cached to achieve the target hit ratio is the **working set**.

Caching is weak when almost every request is for a new value. A cache check then adds work without avoiding the origin:

```text
unique request -> cache miss -> origin
unique request -> cache miss -> origin
unique request -> cache miss -> origin
```

### 1.5 A basic read path

```text
Application                  Cache                  Database
    |                          |                        |
    |-- GET product:42 ------->|                        |
    |<--------- value ---------|                        |  cache hit
    |
```

On a miss:

```text
Application                  Cache                  Database
    |-- GET product:42 ------->|                        |
    |<--------- MISS ----------|                        |
    |----------------------- SELECT product 42 -------->|
    |<---------------------- authoritative value -------|
    |-- SET product:42, TTL -->|                        |
    |-- return value to caller
```

The first request may not become faster. Later requests do.

### 1.6 A cache changes both performance and correctness

A cache can improve:

- response latency;
- origin/database load;
- read throughput;
- compute and network cost;
- absorption of brief read spikes; and
- limited resilience, when safe stale data can be served during an origin problem.

It also creates:

- stale results and invalidation logic;
- memory and network cost;
- cache-node and cluster failures;
- cold-start load;
- hot keys and uneven shards;
- security/privacy risks from copied data;
- serialization/schema-version problems; and
- cascading failure risk if every miss suddenly reaches the origin.

The mature question is therefore not “Can I add a cache?” but:

> **Will the expected reuse justify the added state, and can the whole system survive its cache being empty, slow, partitioned, or unavailable?**

## 2. Required prerequisites, explained inline

These concepts are necessary for the chapter and are introduced here rather than deferred.

### 2.1 Latency, throughput, and tail latency

- **Latency** is how long one operation takes.
- **Throughput** is how many operations finish per unit time, often requests per second (RPS/QPS).
- **Tail latency** is slow-end behavior such as p95 or p99, not merely the average.

A cache can reduce both origin calls and typical latency, but a slow cache timeout can make failure latency worse:

```text
no cache:       database call
bad fallback:   wait for cache timeout + database call
```

This is why cache timeouts are deliberately short and fallback traffic is bounded.

### 2.2 Memory, disk, and network access

RAM usually provides lower access latency than durable network storage, but it is more expensive per byte and normally volatile. A remote distributed cache also requires a network hop, serialization, and connection management. It is faster than the origin only if the full path is actually cheaper.

Avoid memorizing universal “RAM is X nanoseconds” tables as current production facts; hardware, topology, cloud virtualization, payload size, and software stacks change the result. Benchmark the deployed path.

### 2.3 Durability and system of record

- **Durability:** acknowledged data survives the failures covered by the design.
- **System of record:** the authoritative place from which the correct state can be reconstructed.

A cache-aside cache can lose all entries without losing business facts. A write-back cache can contain **dirty** values not yet stored elsewhere, so losing it can lose acknowledged writes.

### 2.4 Consistency and staleness

Consistency here asks when different readers see a write.

- **Strong/read-after-write expectation:** after a successful write, subsequent relevant reads must see it.
- **Eventual consistency:** replicas/caches may temporarily disagree but converge if updates stop.
- **Bounded staleness:** stale data is acceptable only within a declared age/version limit.

“Eventually consistent” is not permission to remain wrong forever. Define how convergence happens: TTL, invalidation, refresh, repair, or version comparison.

### 2.5 Availability, fault tolerance, and failure domains

- **Availability:** the service can provide an acceptable response when requested.
- **Fault tolerance:** it continues correct or explicitly degraded operation despite specified faults.
- **Failure domain:** components likely to fail together, such as one process, host, rack, availability zone, or region.

Two replicas on one machine do not tolerate machine failure. Cache replicas across zones can tolerate a zone loss only if routing, capacity, and failover also work.

See [High Availability Architecture](./14.%20High%20Availability%20Architecture.md) for the broader treatment after this inline foundation.

### 2.6 Partitioning, hashing, and replication

When one cache node cannot hold or serve everything:

- **partition/shard** keys across nodes for capacity and throughput;
- **replicate** a shard to other nodes for availability/read scale;
- use a deterministic routing rule so every client finds the same key.

Consistent hashing or fixed hash slots reduce remapping when membership changes. See [Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md) for the deeper mechanics.

## 3. Why caching is needed in system design

### 3.1 Reduce latency

If constructing a recommendation page requires several database queries and model/service calls, caching the computed result can avoid repeating that work for every refresh.

```text
without cache: request -> joins + ranking + remote calls -> response
with hit:      request -> cached result                -> response
```

The value may be a database row, API response, rendered page fragment, authorization metadata, query result, or expensive computation.

### 3.2 Protect expensive origins

A database has finite CPU, I/O, locks, connections, and query capacity. If 100,000 read requests per second have a 95% cache hit ratio, a simplified steady-state estimate is:

```text
origin read rate = total read rate x miss ratio
                 = 100,000 x (1 - 0.95)
                 = 5,000 reads/second
```

This is a **worked estimate**, assuming each miss causes one origin read and ignoring refreshes, writes, retries, and multi-key requests. It still shows the important effect: a small hit-ratio change can drastically change origin load.

### 3.3 Increase effective read throughput

The cache tier and the origin can serve work in parallel. Frequently read data consumes cache capacity while the origin retains capacity for misses and writes.

Caching does not increase write capacity unless writes are coalesced/buffered with a write-back design, which changes durability semantics.

### 3.4 Reduce cost and bandwidth

Repeated queries, cross-region service calls, object-store reads, computation, and long-distance media transfer cost resources. Caching a reusable result closer to its consumers can reduce those costs.

For media, **byte hit ratio** matters in addition to request hit ratio:

```text
request hit ratio = hit requests / all requests
byte hit ratio    = bytes served from cache / all requested bytes
```

Hitting on many thumbnails but missing every large video can produce a high request hit ratio and poor bandwidth savings.

### 3.5 Absorb read bursts—but only while hot data remains available

A viral item already in cache can prevent a sudden read spike from reaching the database. If that hot key expires during the spike, all callers may miss together and create a **cache stampede**. Burst protection therefore needs stampede controls, not only a TTL.

### 3.6 Improve limited resilience

If a configuration, catalog description, or feed page may safely be slightly old, a cache can serve the last known value while an origin is briefly unavailable.

```text
origin healthy -> serve fresh/cache normally
origin failing -> serve stale within hard safety limit + alert
```

This is not safe for every fact. Stale recommendations may be acceptable; stale authorization, account balance, inventory reservation, or fraud status may not be.

### 3.7 When caching is a poor fit

Avoid or narrow caching when:

- requests rarely repeat before expiration/eviction;
- data changes faster than useful reuse occurs;
- every read requires the newest committed value;
- the origin is already cheap enough and within the latency target;
- cache keys would have unbounded/high-cardinality growth without controls;
- sensitive data cannot safely be duplicated in that cache;
- queries have so many parameter combinations that the hit ratio will be negligible;
- invalidation correctness would be more expensive than the saved work; or
- the team cannot operate/test the cache-outage path.

“Read-heavy” alone is insufficient. Ask **whether the same results repeat** and **how stale they may be**.

### 3.8 Alternatives and complements to caching

A cache is only one way to meet a latency or capacity target:

| Option | What it improves | Trade-off versus a cache |
|---|---|---|
| Better database index/query | Makes authoritative reads cheaper | Adds index storage/write cost but avoids a second application copy |
| Read replica | Adds database read capacity | Still executes queries; replication can be stale and costs a full database copy |
| Materialized view/precomputation | Stores a prepared query result | Refresh/incremental-update logic; can be durable and queryable |
| Denormalized read model | Makes reads direct | More write/update complexity and duplicate durable data |
| CDN/HTTP caching | Moves cacheable responses/media near users | Best for HTTP/content semantics, not arbitrary internal transactions |
| Batching/pipelining | Amortizes round trips | Does not eliminate underlying work or provide reuse |
| Scale/partition the origin | Raises authoritative capacity | Often more expensive; avoids cache consistency but adds database scaling work |
| Asynchronous computation/queue | Moves non-urgent work off request path | Result is available later; status and failure handling are required |

Often the best design combines them: fix a missing index first, cache a truly hot result, and use replicas or partitioning so the origin still survives cache misses.

## 4. Where a cache can live

### 4.1 Cache layers

```text
User
 |
 +-- Browser/mobile cache             private, closest to one user
 |
 +-- CDN/edge cache                   shared near a geographic edge
 |
 +-- Reverse-proxy/API cache          shared HTTP responses
 |
 +-- Application in-process cache     private to one service instance
 |
 +-- Distributed cache                shared by many service instances
 |
 +-- Database internal buffer cache   managed inside the database
 |
 +-- Durable database/object store    system of record
```

Each layer has a different owner, key, TTL, consistency boundary, and failure behavior. The word “cache” does not reveal which one is meant.

### 4.2 Local/in-process cache

```text
App A [local cache A] ----\
App B [local cache B] -----+--> Database
App C [local cache C] ----/
```

Advantages:

- no remote network hop;
- very low lookup overhead;
- each process can continue using its existing values if the network cache is unavailable.

Costs:

- duplicated memory;
- App A, B, and C can hold different values;
- every deploy/restart is a cold cache;
- hit ratio is fragmented among instances;
- invalidation must reach many processes; and
- unbounded caches can create process memory pressure or garbage-collection pauses.

Use it for small, frequently reused, safely stale data such as parsed configuration or extremely hot immutable objects.

### 4.3 Shared distributed cache

```text
App A --\
App B ---+--> [distributed cache cluster] --> Database
App C --/
```

Advantages:

- applications share one logical cached data set;
- aggregate memory and throughput scale across nodes;
- restarts of one app do not erase the shared cache;
- centralized eviction, TTL, and metrics are easier to operate.

Costs:

- a network hop on every lookup;
- cluster partitioning, replication, failover, and hot-key concerns;
- cache availability can become a dependency for every application;
- serialization/versioning and connection pools are required.

**Redis** and **Memcached** are common distributed-cache building blocks, not strategies by themselves. Memcached centers on a relatively simple volatile key-value cache model. Redis provides richer data structures and optional replication, persistence, clustering, and eviction choices. The application must still define truth, TTL, write order, consistency, and failure behavior.

### 4.4 Two-level/near cache

```text
App instance -> L1 local cache -> L2 distributed cache -> Database
                   fastest             shared
```

This can reduce both origin load and distributed-cache hot-key load. It also adds another stale copy. L1 should usually be small, short-lived, version-aware, or connected to an invalidation channel.

### 4.5 CDN cache versus application data cache

A CDN normally caches HTTP responses or media chunks near users. A distributed application cache usually stores internal objects/query results near services.

```text
video bytes: User -> edge CDN -> origin object storage
video metadata: API -> distributed key-value cache -> database
```

A CDN can greatly reduce distance and bandwidth for cacheable content, but personalized/private responses require correct cache keys and response directives. A shared cache must never accidentally serve one user's private response to another.

## 5. The six questions to answer before choosing a strategy

1. **What is the authoritative source?** Can the cache be discarded and rebuilt?
2. **What exactly is the key/value?** Does the key include every input that changes the result?
3. **How stale may it be?** Define TTL, invalidation, versions, and behavior during origin failure.
4. **Who performs misses and writes?** Application, cache library, proxy, or cache service?
5. **What happens when a step partially fails?** Database succeeded/cache failed, cache succeeded/database failed, timeout with unknown outcome, or a late stale fill?
6. **Can the origin survive misses?** Consider normal misses, one shard loss, total cache outage, deploy/restart, and synchronized expiration.

If an interview answer cannot state these, naming a strategy is premature.

## 6. Cache keys, values, and freshness contracts

### 6.1 A key must identify the complete result

Bad key:

```text
product:42
```

Potentially safer key when the response varies:

```text
product:v5:42:currency=INR:region=IN:price-tier=retail
```

Possible inputs include:

- entity ID;
- tenant/user ID where data is private;
- locale, currency, or region;
- query filters and pagination cursor;
- authorization scope;
- representation/schema version; and
- experiment/feature variant when it changes the response.

Including too little causes wrong cross-context hits. Including irrelevant inputs fragments the cache and lowers hit ratio.

### 6.2 Namespacing prevents collisions

Two services using key `12345` might mean different things:

```text
profile-service:v3:account:12345
transaction-service:v2:account:12345:recent
```

Use service/domain, environment, entity type, schema version, and identifier as appropriate. Namespaces also allow a controlled bulk invalidation by changing a version prefix instead of scanning/deleting every old key.

### 6.3 Values need schema/version handling

During a rolling deployment, old and new application versions coexist. A new serializer may write bytes an old instance cannot read.

**Serialization** converts an in-memory object into bytes/text for storage or transfer; **deserialization** reconstructs the object. Both sides must agree on the format and version.

Defenses:

- version keys or envelopes;
- backward-compatible serialization;
- treat decode failure as a miss and delete/quarantine the poison value;
- cap value size;
- monitor deserialization errors; and
- do not deserialize untrusted objects with unsafe mechanisms.

### 6.4 TTL is a business correctness choice, not a magic constant

Choose TTL from:

- how quickly the fact changes;
- maximum tolerable staleness;
- origin capacity if items refresh;
- recomputation cost;
- access frequency;
- invalidation reliability; and
- incident behavior when the origin is unavailable.

Use different TTLs for different data classes. A country-code list, product inventory count, user presence signal, and profile photo URL do not share one freshness requirement.

## 7. Strategy map before the details

The strategies answer two separate questions:

```text
READ MISS:  who loads the origin and cache?
WRITE:      which component is written first, and when is origin persistence complete?
```

| Strategy | Read/write path owner | Normal data flow | Main benefit | Main danger |
|---|---|---|---|---|
| Cache-aside | Application | Read cache; app loads origin on miss | Simple, flexible, caches demand | Staleness and application complexity |
| Read-through | Cache/library | Cache loads origin on miss | Hides miss-loading logic | Cache becomes inline dependency; loader semantics matter |
| Write-around | Application/bypass path | Write origin, do not populate cache | Avoids pollution from write-cold data | Next read misses; stale old entry unless invalidated |
| Write-through | Inline cache/facade | Synchronously persist origin and update cache | Fresh cache after controlled writes | Extra write latency; partial failure is not magically atomic |
| Write-back/behind | Cache + async flusher | Write cache now, persist origin later | Low write latency, batching/coalescing | Data loss/ordering/backpressure unless engineered as durable path |

These can be combined. A common application design is **cache-aside reads + write-around database writes + cache invalidation**. Another is **read-through + write-through** behind one cache interface.

## 8. Cache-aside (lazy loading)

### 8.1 Plain-language idea

The application checks the nearby shelf. If the item is missing, the application fetches it from the warehouse, places a copy on the shelf, and returns it.

The cache does not know how to query the database. The application orchestrates everything.

### 8.2 Read path

```text
1. App -> Cache: GET K
2a. hit  -> return cached V
2b. miss -> App -> DB: read K
3. DB -> App: authoritative V
4. App -> Cache: SET K=V with TTL
5. App -> caller: V
```

Pseudocode:

```text
function getProduct(id):
    key = "product:v5:" + id

    cached = cache.get(key)
    if cached exists:
        return decode(cached)

    product = database.getProduct(id)
    if product does not exist:
        cache.set(key, NOT_FOUND, short_negative_ttl)
        return notFound

    cache.set(key, encode(product), normal_ttl)  // often best effort
    return product
```

If the cache fill fails but the database read succeeded, a disposable cache should normally still return the database value. Otherwise a performance optimization reduces availability.

### 8.3 Common write path: update origin, then invalidate

```text
1. App -> DB: commit new V
2. after commit -> Cache: DELETE K
3. next read misses and reloads new V
```

Why delete instead of directly overwriting?

- the cached value may be a derived object assembled from several tables;
- many keys may depend on one write;
- invalidation avoids duplicating transformation logic; and
- the next real reader loads only data that is actually needed.

Deleting is not automatically perfect; the race in Section 18 can still reinsert an older value.

### 8.4 Advantages

- simple and widely supported;
- cache failure can be treated as a miss;
- only requested items consume space;
- application controls keys, TTLs, serialization, and negative caching;
- cache and database need no direct integration.

### 8.5 Costs and edge cases

- the first request pays the miss penalty;
- cache logic is repeated unless wrapped in a shared library/data-access layer;
- different callers can implement inconsistent policies;
- concurrent misses can stampede the origin;
- data changed outside this application stays stale until TTL/invalidation;
- delete failure after a database commit can retain stale data;
- a late fill can overwrite a newer invalidation/version.

### 8.6 Good fit

- read-heavy entities with repeated reads;
- cache is an optional accelerator, not the truth;
- some bounded staleness is acceptable;
- application teams want explicit control.

### 8.7 Poor fit

- every successful write requires immediate globally fresh reads;
- many systems write the same data but invalidation cannot be coordinated;
- misses are too expensive to expose to concurrent callers without a loader/coalescing layer.

[Azure's Cache-Aside pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/cache-aside) and [AWS's database caching patterns](https://docs.aws.amazon.com/whitepapers/latest/database-caching-strategies-using-redis/caching-patterns.html) document the basic behavior and its staleness/miss trade-offs.

## 9. Read-through caching

### 9.1 Plain-language idea

The application asks one librarian—the cache—for the item. If absent, the librarian knows how to fetch the warehouse, store a copy, and answer. The application does not implement the miss sequence.

```text
Application -> Cache facade -> hit: return V
                         |
                         +-> miss: loader -> Database -> fill -> return V
```

### 9.2 How it differs from cache-aside

| Question | Cache-aside | Read-through |
|---|---|---|
| Who sees the miss? | Application | Cache/cache library |
| Who calls the database? | Application | Configured loader behind cache |
| Where is loading logic? | App/data-access code | Cache service/library integration |
| Can cache be bypassed easily? | Usually yes | Depends on abstraction/product |
| Basic hit/miss result | Similar | Similar |

Read-through does not inherently guarantee fresher data. It changes ownership and abstraction.

### 9.3 Advantages

- consistent loading behavior across callers;
- simpler application read path;
- cache product/library may provide request coalescing, refresh, retries, and metrics;
- a single interface can hide origin details.

### 9.4 Costs

- cache/loader becomes an inline dependency;
- loader must handle database failures, timeouts, permissions, and schema evolution;
- complex queries may not fit a key-based loader;
- a shared generic abstraction can hide expensive misses from developers;
- bulk operations and transactions may be awkward.

### 9.5 Good fit

- stable key-to-record loading rules;
- teams want one caching abstraction;
- product/library has reliable loader semantics;
- read-heavy data that can tolerate the cache's consistency model.

### 9.6 Example

Amazon DynamoDB Accelerator (DAX) is documented as a read-through cache: on a cacheable read miss it obtains the result from DynamoDB, stores it, and returns it without application-managed cache-aside logic. See [DAX: how it works](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DAX.concepts.html).

## 10. Write-around

### 10.1 Plain-language idea

Writes go around the cache directly to the database. The cache is populated only when something is later read.

```text
write: App ------------------------------> Database
              \---- invalidate old K ---> Cache

read:  App -> Cache -> miss -> Database -> fill Cache
```

### 10.2 Why it exists

Some data is written often but rarely read. Populating the cache on every write wastes memory and network bandwidth and can evict genuinely hot read data. Write-around lets real reads prove what deserves caching.

### 10.3 The crucial invalidation detail

If `K=old` is already cached, merely writing `K=new` to the database leaves the old cache hit alive. Therefore a practical application pattern is:

```text
commit database write -> delete/invalidate cached K
```

“Bypass cache” means **do not allocate/update it with the new value**; it does not mean ignore existing stale copies.

### 10.4 Advantages

- prevents cache pollution by write-only/read-cold data;
- keeps the write path independent of cache availability if invalidation is asynchronous/retriable;
- simpler than making the cache durable.

### 10.5 Costs

- the first read after a write misses;
- a read immediately after write may be slower;
- failed/delayed invalidation causes stale reads;
- a burst of first reads after many writes can load the origin.

### 10.6 Use/avoid

Use for audit records, bulk ingestion, or entities with weak write-to-read correlation. Avoid when almost every write is followed immediately by latency-sensitive reads and freshness is required; write-through or explicit update/invalidation may fit better.

## 11. Write-through

### 11.1 Plain-language idea

The application writes through a caching facade. The facade synchronously persists the authoritative store and makes the new value available in cache before declaring the write complete under its contract.

```text
Application -> Cache/facade -> Database
                   |              |
                   |<-- commit ---|
                   +-- cache V
Application <------ success
```

Exact ordering varies by product. Always describe the success boundary.

### 11.2 Advantages

- controlled writes leave a warm, current cache entry;
- read-after-write behavior can be easier for those writes;
- applications use one data-access interface;
- avoids a miss immediately after a frequently read write.

### 11.3 Costs

- writes wait for durable store work and cache coordination;
- read-cold data pollutes the cache;
- cache is in the write path;
- external writers can still bypass the facade and create stale data;
- multi-key derived views still need invalidation logic.

### 11.4 The atomicity trap

Two independent systems are not made atomic by drawing one arrow.

Failure example:

```text
1. Database commits V2
2. Cache update fails
3. What should the client hear?
```

- Return failure: the client may retry even though the database committed.
- Return success: the cache may still contain V1.
- Roll back database: may be unavailable or unsafe after external effects.

Production choices include:

- make database commit authoritative, then invalidate/update cache with retry;
- write a transactional outbox/change-log row in the same database transaction, then let a background relay retry cache repair asynchronously;
- use versions so a stale update cannot replace a newer one;
- return success after the database and accept a bounded cache-repair window; or
- use a product whose defined consistency contract matches the requirement.

Do not promise a distributed transaction unless one actually exists and its latency/availability trade-offs are accepted.

### 11.5 Good fit

- read-heavy data commonly read soon after a write;
- writes flow through one controlled facade;
- fresh post-write cache matters more than minimum write latency;
- the partial-failure contract is explicit.

### 11.6 Poor fit

- write-heavy, read-cold data;
- cache must never affect write availability;
- multiple uncontrolled writers bypass the cache integration.

[Hazelcast's external store documentation](https://docs.hazelcast.com/hazelcast/5.1/data-structures/working-with-external-data) gives a concrete product example of synchronous write-through and asynchronous write-behind semantics. [Azure's write-through architecture](https://learn.microsoft.com/en-us/azure/architecture/databases/architecture/write-through-caching-azure-sql-managed-redis) explicitly discusses partial failure and repair rather than assuming two stores update atomically.

## 12. Write-back / write-behind

### 12.1 Plain-language idea

The application writes to the fast cache and receives success before the slower database is updated. A background flusher persists the change later.

```text
Application -> Cache + pending-write queue -> success
                       |
                       +---- later/batched ----> Database
```

The cached value is **dirty** until the database confirms it.

### 12.2 Why use it

- lower apparent write latency;
- absorb bursty writes;
- batch many small writes;
- coalesce repeated updates to the same key;
- reduce backing-store write rate.

Example: 100 increments to a non-critical view counter may be coalesced into a later update. This is not safe for every counter or financial fact.

### 12.3 Why it is dangerous

Once the client receives success, the system owes that write. If the only dirty copy was volatile and the node dies, acknowledged data is lost.

Other problems:

- later writes can reach the database before earlier writes;
- retries can duplicate non-idempotent operations;
- the database can be down long enough for the pending queue to fill;
- evicting a dirty entry before flush loses it unless separately logged;
- readers outside the cache may see old database values;
- failover to a replica missing recent dirty writes can move state backward.

### 12.4 What a serious write-back design needs

- a durable or sufficiently replicated write-ahead/pending log;
- acknowledgement only after the required replicas/log accept the write;
- per-key ordering or version/sequence numbers;
- idempotent database writes/deduplication;
- retry with bounded backoff;
- queue-size limits and backpressure;
- a rule to flush or pin dirty entries before eviction;
- reconciliation and dead-letter/manual repair paths;
- lag, oldest-dirty-age, failure, and loss-risk monitoring;
- a clear **recovery point objective (RPO)**—how much recent data loss is tolerable—and **recovery time objective (RTO)**—how long recovery may take.

Coalesce repeated writes only when they represent replacement state, such as “the latest counter value.” Do not coalesce audit events or operations where every intermediate transition matters.

At that point the “cache” is functioning partly as a write buffer/database subsystem.

### 12.5 Good fit

- high-volume, latency-sensitive, coalescible writes;
- temporary staleness in the backing store is acceptable;
- the application can tolerate the chosen loss window or the buffer is durably replicated;
- examples may include telemetry aggregates, approximate counters, or buffered storage systems.

### 12.6 Avoid for

- payments, ledger entries, orders, inventory reservations, or messages when loss/ordering errors are unacceptable—unless the tier is engineered as a genuine durable system with the required guarantees;
- data read by systems that bypass the cache and require immediate freshness;
- teams without operational ownership of backlog and recovery.

## 13. Comparing and choosing the strategies

| Requirement | Often start with | Why | Watch closely |
|---|---|---|---|
| Simple read acceleration | Cache-aside | Explicit and cache can fail as a miss | Stampede, invalidation race |
| Centralized miss loading | Read-through | Hides loader logic | Inline dependency and loader failures |
| Many writes are never read | Write-around + invalidate | Avoids pollution | Cold read after write |
| Reads commonly follow writes | Write-through + read-through/aside | Warmer fresh cache | Write latency and partial failure |
| Extremely latency-sensitive bursty writes | Write-back | Async persistence/batching | Durability, ordering, backlog |
| Strict correctness, low reuse | No cache or narrow immutable cache | Less duplicated mutable state | Origin sizing and latency |

### 13.1 Partial-failure comparison

| Strategy/path | Failure | Result to reason about |
|---|---|---|
| Cache-aside read | Cache errors | Treat as a miss only through a bounded origin fallback; otherwise degrade/shed |
| Cache-aside read | Origin errors after miss | Possibly serve safe stale; otherwise fail explicitly |
| Cache-aside write | Database commits, invalidation fails | Old cached value may remain; retry/CDC/TTL/version repairs it |
| Read-through | Cache/loader unavailable | Inline read path may fail unless a deliberate bypass exists |
| Write-around | Database commits, delete fails | Same stale-entry risk as cache-aside invalidation |
| Write-through | Database commits, cache step fails | Client outcome/cache state may be ambiguous; define authority and repair |
| Write-back | Cache acknowledges, database stays down | Dirty backlog grows; durable buffer, retry, backpressure, and recovery are mandatory |

### 13.2 A practical decision spine

```text
1. Is there repeat reuse before data changes?
   no  -> do not cache by default
   yes -> continue

2. Can stale data be tolerated, and for how long?
   none -> narrow the cached data or use a consistency-aware inline design
   bounded -> define TTL + invalidation/version

3. Must writes be durable before success?
   yes -> database/system of record commits before success; cache is derived
   no / durable buffer exists -> write-back may be considered

4. Who should own miss/write logic?
   application -> cache-aside/write-around
   cache facade -> read-through/write-through

5. What happens on cache outage/cold start?
   design origin budget, stale/degraded response, shedding, and recovery
```

There is no universally best strategy. It is common to use different strategies for different entities in one system.

## 14. Expiration, invalidation, refresh, and eviction

These mechanisms are related but solve different problems.

### 14.1 Expiration with TTL

```text
SET K=V with TTL 10 minutes
```

After the TTL, the value is no longer considered fresh. Depending on the system, it may be deleted actively, removed lazily on access, or retained physically but treated as expired.

TTL provides a maximum ordinary staleness bound only under assumptions. If clients continue serving a local copy, clocks/replication lag differ, or `stale-if-error` is allowed, the effective bound changes.

TTL advantages:

- simple self-healing when invalidation is missed;
- bounds how long stale/negative data normally lives;
- eventually removes unused keys.

TTL costs:

- shorter TTL lowers hit ratio and increases origin work;
- longer TTL increases the stale window;
- identical expirations can create a synchronized load spike;
- TTL alone does not provide immediate invalidation after a write.

Use **TTL jitter** to spread refresh work:

```text
actual_ttl = base_ttl + random(0, jitter_window)
```

The exact distribution and window are workload choices. [AWS cache-validity guidance](https://docs.aws.amazon.com/whitepapers/latest/database-caching-strategies-using-redis/cache-validity.html) recommends jitter to avoid simultaneous expiry pressure.

### 14.2 Explicit invalidation

```text
database commit -> delete K / publish invalidate(K, version)
```

Invalidation reduces the stale window after known writes. It can be:

- synchronous/best-effort in the request path;
- published through a message/change stream;
- driven from database change data capture (CDC);
- expressed by incrementing a namespace/version; or
- a purge request to proxies/CDNs.

Invalidation must be retried or repaired when delivery fails. At-least-once invalidations should be idempotent. Versioned invalidations prevent an older delayed event from deleting/replacing newer state incorrectly.

### 14.3 Refresh-ahead

Refresh an item shortly before it expires when it is still hot:

```text
reader gets current V
       |
       +-- near soft expiry? one worker refreshes V asynchronously
```

This reduces user-visible misses, but may waste origin work refreshing values that are not read again. It requires a lock/lease or request coalescing so every reader does not refresh.

### 14.4 Stale-while-revalidate and stale-if-error

- **Stale-while-revalidate:** briefly return an older value while one worker refreshes it.
- **Stale-if-error:** if the origin fails, return an expired/older value within a separately defined hard safety window.

Think in two deadlines:

```text
fresh until soft TTL
serve stale + refresh until hard TTL
after hard TTL: fail/degrade; do not silently serve forever
```

The business rule decides the hard limit. For a country list it might be generous; for an authorization decision it may be zero.

### 14.5 Eviction

Eviction answers: **when the cache lacks space, which resident item should leave?**

An item can be perfectly fresh and still be evicted because another item is more valuable under the selected policy. A later read will miss and reconstruct it.

### 14.6 One comparison

| Mechanism | Trigger | Primary purpose | Can remove a fresh item? | Can leave stale data until trigger? |
|---|---|---|---|---|
| Expiration | Time/age | Bound freshness/lifetime | Yes, even if unchanged | Yes, until TTL |
| Invalidation | Known change/event | Remove known-obsolete data | Yes, deliberately | Yes, if event fails/delays |
| Refresh | Time/access signal | Keep hot data warm/fresh | No, normally replaces | Yes, if refresh fails |
| Eviction | Space/policy pressure | Reclaim capacity | Yes | Not aimed at freshness |

## 15. Cache eviction policies

### 15.1 What makes an item valuable?

A good policy tries to keep entries likely to save the most future cost. Possible signals are:

- how recently it was read;
- how frequently it was read;
- how long it has been resident;
- remaining TTL;
- value size;
- cost to recompute/refetch;
- business priority; and
- predicted future demand.

No online policy knows the future perfectly.

### 15.2 FIFO — First In, First Out

Evict the item inserted earliest.

```text
oldest insertion [A] [B] [C] newest
evict A
```

Advantages:

- simple and low bookkeeping;
- predictable residence order.

Weaknesses:

- ignores reads; a heavily used old item can be evicted;
- insertion age is a weak popularity signal.

Useful when simplicity matters and access patterns do not strongly favor recency/frequency.

### 15.3 LRU — Least Recently Used

Evict the item whose most recent access was longest ago.

```text
least recent [A] [D] [B] [C] most recent
evict A
```

Why it often works: temporal locality makes recent use a clue for near-future reuse.

Weaknesses:

- a large one-time scan can push genuinely hot items out;
- cyclic access larger than cache capacity can thrash;
- exact access-order tracking costs metadata and synchronization;
- recency does not know recomputation cost or object size.

#### Interview coding version of exact O(1) LRU

```text
hash map: key -> node             O(1) lookup
doubly linked list: LRU <-> MRU   O(1) remove/move/evict
```

On `get`, move the node to the most-recent end. On inserting beyond capacity, remove the least-recent end and its hash-map entry.

This coding structure is not the whole distributed-cache design. Production caches may use sampling/approximations to reduce global ordering overhead.

### 15.4 LFU — Least Frequently Used

Evict the item with the lowest access frequency.

```text
A: 2 uses
B: 91 uses
C: 7 uses
evict A
```

Advantages:

- retains repeatedly popular entries even if not accessed in the last instant;
- can outperform LRU for stable popularity distributions.

Weaknesses:

- an item popular long ago can remain forever unless counts decay/age;
- new hot entries begin with low counts;
- frequency metadata costs memory and updates;
- tie-breaking still needs a policy, often recency.

Practical LFU uses approximate counters and aging so old popularity fades.

### 15.5 Random eviction

Choose a victim randomly.

Advantages:

- tiny bookkeeping overhead;
- avoids maintaining global order;
- a useful baseline and can perform acceptably for uniform access.

Weaknesses:

- may evict a hot/expensive item;
- ignores available demand signals.

### 15.6 TTL/expiration-aware eviction

Evict entries closest to expiration. This can be sensible because they would soon become unusable anyway.

Weakness: a soon-to-expire entry may be extremely hot and cheap to refresh, while a long-lived entry may never be read. TTL expresses freshness, not necessarily future value.

### 15.7 MRU — Most Recently Used

Evict the most recently used entry. This is less common but can work for access patterns where the newest/recent item is unlikely to be revisited, such as some large sequential scans.

It is not a default for ordinary web-object reuse.

### 15.8 Size-aware and cost-aware policies

Counting hits treats a 2 KB object and a 200 MB object equally. A policy may instead optimize:

- object hit ratio;
- byte hit ratio;
- latency saved;
- origin compute saved; or
- value per byte of cache.

A small moderately popular object may deliver more saved requests per byte than one huge popular object. CDNs commonly care about byte misses and storage/network cost, not only request count.

### 15.9 No eviction

Reject new writes once the memory limit is reached. This can protect data that must not be silently discarded, but callers must handle errors and the component is behaving more like a constrained data store.

For a disposable cache, uncontrolled out-of-memory termination is worse than a deliberate limit and policy.

### 15.10 Policy comparison

| Policy | Signal | Strength | Failure pattern |
|---|---|---|---|
| FIFO | Insertion age | Simple | Evicts old-but-hot items |
| LRU | Recent access | Good for temporal locality | Scan pollution/thrashing |
| LFU | Access count | Good for stable popularity | Old popularity sticks without aging |
| Random | None | Very cheap | Uninformed hot-item eviction |
| Shortest TTL | Near expiry | Reclaims soon-useless entries | TTL is not popularity |
| Size/cost aware | Benefit per resource | Matches real objective better | More metadata/complexity |
| No eviction | Preserve residents | No silent replacement | New writes fail at capacity |

### 15.11 Eviction versus admission

Eviction chooses **which resident item leaves**. Admission chooses **whether a new candidate should enter at all**.

```text
one-time scan item arrives
  naive LRU: admit it and evict a reusable item
  admission-aware cache: decline/bypass it unless reuse evidence is strong enough
```

Admission can use item size, frequency estimates, origin cost, endpoint class, or explicit rules. This is often a better response to scan pollution than endlessly tuning the eviction victim.

### 15.12 Redis example, not a universal standard

Redis exposes `maxmemory` and multiple policies including all-key and expiration-limited variants of LRU/LFU/random, TTL-oriented eviction, and no-eviction behavior. Product versions evolve, so use the [current Redis key-eviction documentation](https://redis.io/docs/latest/develop/reference/eviction/) rather than memorizing one version's complete list.

The interview-quality answer is: **choose based on measured access distribution and the metric you want to optimize, then monitor evictions and miss cost.** “LRU because everyone uses it” is not enough.

## 16. Distributed caching: why it is needed

### 16.1 The single-node limit

One cache server has finite:

- RAM;
- CPU/command throughput;
- network bandwidth/connections; and
- availability.

```text
All apps -> [one cache node] -> one capacity ceiling + one failure point
```

Vertical scaling helps only until cost, hardware, or failure-recovery limits become unacceptable.

### 16.2 Distribute capacity and traffic

```text
                        +--> Cache shard A: keys 0..x
Applications -> Router -+--> Cache shard B: keys x..y
                        +--> Cache shard C: keys y..z
```

Partitioning allows aggregate memory and throughput to grow by adding nodes. Replicas allow a shard to continue when a node fails.

### 16.3 Share state across application instances

With only local caches, each application instance sees its own history and loses it on restart. A distributed cache provides one logical key space shared across horizontally scaled applications.

### 16.4 Centralize expensive reusable results

If 500 service instances independently compute the same report, local caches may perform the work hundreds of times. A shared cache can compute/fetch once and reuse across the fleet—subject to stampede protection.

### 16.5 What distributed caching does not solve

- It adds a network dependency.
- Sharding does not eliminate a single extremely hot key.
- Replication does not guarantee strong consistency.
- More RAM does not fix a bad cache key or near-zero reuse.
- A cache cluster cannot protect an undersized database during total cache loss without explicit controls.
- Distribution increases operational complexity and failure combinations.

## 17. Anatomy of a distributed cache

### 17.1 Key routing

For a key `product:42`, every request must find the node holding it.

#### Client-side routing

```text
App/cache client -> hash/routing table -> chosen cache node
```

Advantages: one less proxy hop; high throughput. Costs: clients need membership updates, compatible algorithms, retry/redirection logic, and rolling-upgrade discipline.

Here **membership** means the current set of cache nodes plus the versioned map/rule saying which node owns each key range/slot.

#### Proxy/router-based routing

```text
App -> cache proxy/service endpoint -> chosen cache node
```

Advantages: simple clients and centralized policy. Costs: proxy latency/capacity and its own high-availability design.

#### Server redirection

The client contacts a node, which redirects it to the shard owner. This simplifies discovery but adds redirection handling and can cost an extra round trip until the client's map updates.

### 17.2 Modulo hashing and membership changes

Naive routing:

```text
node = hash(key) mod number_of_nodes
```

Changing the node count changes the divisor and remaps most keys, causing mass misses. Consistent hashing, rendezvous hashing, or fixed hash-slot maps reduce this movement.

Redis Cluster, for example, documents a fixed hash-slot space mapped to nodes; this is a specific product design, not the definition of all distributed caching. See [Redis Cluster scaling](https://redis.io/docs/latest/operate/oss_and_stack/management/scaling/).

### 17.3 Partitioning and replication serve different goals

```text
partitioning: K1..K500 -> shard A, K501..K1000 -> shard B
replication:  shard A primary -> shard A replica in another failure domain
```

- Partitioning adds capacity.
- Replication adds redundant copies and possibly read capacity.
- Replication multiplies memory/network cost.
- Asynchronous replication can lose recent cache writes on failover.

For derived cache-aside values, losing recent cache writes usually causes misses/refills. For write-back, the same loss can violate acknowledged-write durability.

### 17.4 Replication topology

```text
Zone A                 Zone B
Shard 1 primary -----> Shard 1 replica
Shard 2 replica  <----- Shard 2 primary
```

Place replicas across independent failure domains. Plan enough remaining capacity to carry traffic after a node/zone fails; a replica that exists but overloads immediately is not useful fault tolerance.

### 17.5 Consistent hashing reduces movement, not all pain

When a node is added or removed:

- only part of the key space should remap;
- remapped keys are cold on their new owners;
- clients must converge on the same membership/version;
- existing connections and retries must be handled;
- traffic can still spike at the origin while the cache warms.

Consistent hashing is necessary routing machinery in many designs, not a complete cache architecture.

### 17.6 Hot keys

One viral key maps to one shard:

```text
normal keys -> spread across A, B, C
viral K     -> all requests to B
```

Adding unrelated shards may not help. Options include:

- replicate the hot value and read from replicas;
- add a tiny L1 local cache;
- request coalescing/single-flight per process or cluster;
- split a large logical value/chunk it when possible;
- use read replicas or explicit hot-key fan-out;
- cache immutable responses at CDN/proxy layers;
- rate-limit abusive requests;
- detect hot keys and dynamically promote them.

Writing `K#random` to spread reads works only if writes update every copy or readers can tolerate/version divergence. It trades hot-key load for consistency and invalidation complexity.

### 17.7 Multi-key operations

Keys in different shards cannot usually be updated atomically with a normal single-node cache operation. Options:

- choose hash tags/co-location for related keys;
- avoid cross-key atomic assumptions;
- store one aggregate value;
- use versions and application-level workflow;
- choose a datastore supporting the required transaction.

Co-location can create a large/hot shard, so it is a trade-off.

## 18. Cache consistency and invalidation races

### 18.1 The obvious unsafe order: delete before database commit

```text
1. delete cached K
2. reader misses, reads old database V1, caches V1
3. writer commits database V2
4. cache still contains V1
```

This is why cache-aside writes usually commit the database first and then invalidate.

### 18.2 The less obvious race: a late stale fill

Even “database then delete” can race:

```text
Reader R                         Writer W
--------                         --------
cache miss
DB read starts -> obtains V1
                                 DB commits V2
                                 DELETE cache K
SET cache K=V1   (late)

Final state: DB=V2, Cache=V1
```

TTL eventually repairs it, but until then readers see V1.

### 18.3 Stronger defenses

Choose according to correctness needs:

- **Versioned values/events:** store `version=12`; reject a fill/invalidation older than the known version.
- **Compare-and-set/fencing token:** a loader may populate only if no newer generation appeared.
- **Per-key coordination:** serialize a fill with writes where feasible. Request coalescing alone coordinates readers, not necessarily writers.
- **Write-through facade:** control reads/writes in one consistency mechanism.
- **CDC/outbox invalidation:** publish committed database changes reliably and retry/repair.
- **Versioned keys:** write `product:42:v12`; a small current-version pointer or request context chooses it.
- **Short TTL:** bounds but does not eliminate the inconsistency.
- **Delayed second delete:** sometimes used as a heuristic to catch late fills; timing assumptions mean it is not a proof of correctness.

A **fencing token** is a monotonically increasing generation/lease number. A resource rejects work carrying an older token, preventing a paused former lease holder from overwriting work performed by a newer holder.

Meta's engineering write-up [“Cache made consistent”](https://engineering.fb.com/2022/06/08/core-infra/cache-made-consistent/) gives a first-party example of fill/invalidation races, versions, and the difficulty of diagnosing distributed cache state.

### 18.4 Multiple writers and out-of-order events

```text
DB commits V7, then V8
invalidation/update V8 arrives first
delayed V7 arrives later
```

Without versions, the cache may move backward. Every update/invalidation event should carry enough ordering identity—database sequence, entity version, or fencing token—for the receiver to ignore stale events.

Wall-clock timestamps alone are fragile when clocks differ or two updates share resolution.

### 18.5 Cache-aside cannot observe hidden writers automatically

An admin script, batch job, or other service updates the database without invalidating this cache. Solutions include:

- route all writes through a domain service;
- database CDC/change streams;
- triggers/outbox where appropriate;
- conservative TTL as a repair bound;
- periodic reconciliation.

Documenting “all writers must invalidate” is not enforcement.

### 18.6 Consistency requirements vary by datum

| Data | Typical tolerance | Safer posture |
|---|---|---|
| Static image/versioned video chunk | Effectively immutable | Long TTL, content-addressed key |
| Product description | Bounded staleness often acceptable | Cache-aside + invalidation + TTL |
| Feed/recommendations | Short staleness often acceptable | Cache computed pages, async refresh |
| Online presence | Very short-lived and approximate | Short TTL/heartbeat-derived state |
| Inventory available-to-promise | Business dependent, often strict near checkout | Cache display hints; validate authoritative store on reserve |
| Authorization/entitlement | Security sensitive | Short/versioned cache; fail policy explicitly; authoritative validation for critical actions |
| Account balance/ledger | Strict | Do not use a disposable stale cache as authority |

These are design defaults, not universal business rules.

## 19. Fault tolerance: the cache and the whole system

### 19.1 The most important failure test

> **What happens if the cache serves zero hits right now?**

If every application immediately sends full traffic to an origin sized only for 5% misses, the cache outage becomes a database outage:

```text
cache fails
   -> all reads hit database
      -> database saturates
         -> timeouts and retries
            -> wider cascading failure
```

A fast normal path is not fault-tolerant unless the abnormal path is bounded.

### 19.2 Cache failure scenarios and responses

| Failure | Risk | Possible response |
|---|---|---|
| One cache node dies | Its shard misses/unavailable | Replica failover, remap, controlled refill |
| Replica promotion | Recent async writes missing | Treat derived data as miss; for write-back require durable acknowledgement/versioning |
| Whole cache cluster unavailable | Origin flood | Short timeout, circuit breaker, rate-limited fallback, stale L1, degraded response, load shedding |
| Cache becomes slow | Thread/connection exhaustion | Tight timeout, bulkhead pools, concurrency cap, shed work |
| Network partition | Split views/stale writes | Source-of-truth rule, quorum/product semantics, version conflict handling |
| Application deploy/restart | Empty L1s | Gradual rollout, warm-up, shared L2, traffic ramp |
| Cluster resize | Remapped cold keys | Consistent routing, staged rebalance, prewarm where justified |
| Many TTLs expire together | Origin spike | TTL jitter, refresh-ahead, request coalescing |
| Origin fails | Cache miss cannot refill | Serve safe stale data, negative/error policy, degrade/fail explicitly |
| Corrupt/incompatible cached bytes | Repeated decode failures | Version schema, delete poison key, treat as miss, alert |

### 19.3 Fail open, fail closed, and degraded service

The terms depend on the control:

- **Fail open/bypass cache:** attempt the authoritative origin or allow the operation when cache/control is unavailable. This favors availability but can overload the origin or weaken a safety control.
- **Fail closed:** reject/deny because correctness or security cannot be established. This favors safety but reduces availability.
- **Degraded:** serve a limited response, stale safe value, default, or partial page.

Examples:

- Recommendations: omit them or serve a bounded stale list.
- Product details: rate-limited database fallback; display may degrade.
- Authorization: often deny a critical action if permission cannot be verified; do not treat a stale “allow” as harmless.
- Inventory display: stale estimate may be shown, but checkout must revalidate/reserve authoritatively.

There is no universal “always bypass cache.” State the business risk.

### 19.4 Protecting the origin

Use several layers:

```text
cache timeout (short)
   -> fallback concurrency budget
      -> origin rate limit / admission control
         -> request coalescing
            -> stale/default/degraded response
               -> load shedding before collapse
```

Important mechanisms:

- **Circuit breaker:** stop sending calls temporarily when the origin/cache is consistently failing.
- **Bulkhead:** separate connection/thread/concurrency budgets so cache problems do not consume every application resource.
- **Load shedding:** reject low-priority work early instead of letting all work time out.
- **Retry budget:** do not let every layer multiply traffic with retries.
- **Stale cache:** retain expired data separately/physically for safe incident use.

The [Amazon Builders' Library caching guidance](https://aws.amazon.com/builders-library/caching-challenges-and-strategies/) explicitly recommends testing cache non-availability, protecting dependencies, considering stale data, and monitoring real behavior.

### 19.5 Replication is not enough

Ask:

- Are replicas in different failure domains?
- Is replication synchronous or asynchronous?
- What acknowledged data can be lost at failover?
- How is a failed primary detected without split brain?
- Do remaining nodes have traffic and memory headroom?
- Will clients find the new owner quickly?
- How is the promoted/cold shard warmed?

[Redis Cluster documentation](https://redis.io/docs/latest/operate/oss_and_stack/management/scaling/) states that its replication is asynchronous and that acknowledged writes can be lost under described failures. For ordinary disposable cached copies this may be acceptable; for write-back it is a major durability constraint.

### 19.6 Cache availability versus data durability

Keep these separate:

- A replicated cache may stay available after a node failure but serve an older value.
- A persistent cache may recover data after restart but be unavailable during recovery.
- A disposable cache may lose everything yet preserve business durability because the database remains authoritative.

For caching, fast rebuild and origin protection can matter more than persisting every cached byte. For a write-back tier, pending writes do require genuine durability.

### 19.7 Graceful recovery and warming

Do not send full production traffic to an empty replacement and hope:

- ramp traffic gradually;
- warm only known high-value keys, not the entire database blindly;
- restore from a snapshot only if it is fresh/useful and cheaper than refill;
- let consistent hashing limit key movement;
- cap concurrent origin fills;
- prefer peer/replica transfer where supported;
- watch origin load and hit-ratio recovery.

Warm-up work itself can cause an outage if it competes with live requests.

## 20. Production failure patterns unique to caching

### 20.1 Cache stampede / thundering herd / dogpile

Many callers miss the same key and all recompute/fetch it:

```text
1000 callers -> miss K -> 1000 identical database queries
```

Mitigations:

- single-flight/request coalescing: one leader loads; others wait/share result;
- per-key distributed lock/lease with timeout and fencing;
- stale-while-revalidate;
- early/probabilistic refresh;
- TTL jitter for many keys;
- prewarm a known event's hot keys;
- origin concurrency limits.

Lock failures must not block forever. A lease holder can pause and resume after its lease expires, so fencing/version checks may still be needed before it writes the fill.

Google Cloud CDN calls this [request collapsing/coalescing](https://docs.cloud.google.com/cdn/docs/caching): concurrent fills for the same cache key can be collapsed into one origin request per edge node.

### 20.2 Cache avalanche

This term is common but not a formal universal standard. It usually means many keys expire together or a broad cache outage makes a huge share of traffic fall through to the origin.

Mitigations:

- TTL jitter;
- staged invalidation/deployment;
- replicated cache across failure domains;
- rate-limited fallback and shedding;
- gradual warming;
- capacity test with the cache disabled.

### 20.3 Cache penetration

Repeated requests are made for data that does not exist, so every lookup misses and reaches the origin:

```text
GET nonexistent-user-999 -> MISS -> DB not found
repeat                    -> MISS -> DB not found
```

Mitigations:

- **negative caching:** cache “not found” for a short, carefully chosen TTL;
- validate malformed/out-of-range input before the database;
- rate-limit abusive identities/IPs;
- a Bloom filter for known membership when appropriate.

A **Bloom filter** is a compact probabilistic membership structure. In its ordinary form, “definitely not present” is reliable while “possibly present” can be a false positive. It can prevent many pointless origin lookups, but it must be updated/rebuilt as valid IDs change and does not replace authorization.

Negative-cache keys must include tenant/user/security context where the answer differs. Keep the TTL short enough that a newly created entity does not remain falsely absent too long.

### 20.4 Hot key/hot shard

A celebrity profile, viral post, global configuration key, or synchronized lock can saturate one shard despite a healthy overall cluster.

Measure per-key or heavy-hitter approximations, per-shard QPS/bytes/latency, and connection imbalance. Cluster-average CPU can hide one shard at 100%.

### 20.5 Cache pollution

One-time scans or large cold values evict reusable hot data.

Mitigations:

- do not cache scan/bulk-export results by default;
- admission policy: require repeated access before admission;
- separate caches/pools for workloads with different behavior;
- size limits and cost-aware eviction;
- write-around for read-cold writes.

### 20.6 Cache poisoning

An attacker or bug causes a malicious/incorrect value to be stored under a trusted key, amplifying it to many readers.

Defenses include strict origin validation, authenticated internal writers, tenant-safe keys, safe serialization, integrity/version checks where warranted, and narrowly scoped purge capability.

### 20.7 Eviction storm and thrashing

If the working set exceeds available memory, useful values are continuously evicted and refetched:

```text
insert A -> evict B -> request B -> evict C -> request C -> evict A -> ...
```

Symptoms: high evictions, low/unstable hit ratio, rising origin QPS, and wasted cache write bandwidth. Fix sizing, admission/TTL/policy, data size, or workload separation—not merely retries.

## 21. Capacity planning and back-of-the-envelope reasoning

### 21.1 Estimate the working-set memory

```text
raw cached bytes
  = number of resident entries x average(key bytes + value bytes)

provisioned bytes
  ≈ raw bytes
    + object/allocator/metadata overhead
    + replication copies
    + fragmentation and connection/replication buffers
    + failure and growth headroom
```

Do not size only from payload bytes. A million tiny keys can have substantial per-entry overhead; a write-heavy replicated cache needs network and buffer headroom.

#### Worked estimate

Assume:

- 20 million useful resident entries;
- average key + value payload = 1.2 KB;
- assume 30% overhead for this exercise;
- 2 total copies (one primary + one replica);
- 25% operational headroom after replication.

```text
payload             = 20,000,000 x 1.2 KB = 24,000,000 KB
with 30% overhead   = 24,000,000 x 1.30
two copies          = previous result x 2
with 25% headroom   = previous result x 1.25
```

This is deliberately a **worked estimate**, not a Redis/Memcached sizing fact. Measure real serialized sizes, allocator behavior, persistence/replication buffers, and node limits.

### 21.2 Estimate origin traffic from misses

```text
origin miss reads ≈ read QPS x (1 - hit ratio)
```

If one logical request performs five cache lookups and any miss causes a separate query, calculate per lookup/key class, not only per API request.

Background refreshes, writes, negative lookups, retries, and stampedes add traffic beyond this simple model.

### 21.3 Approximate average latency carefully

For synchronous cache-aside fill:

```text
E[latency]
  ≈ H x Lcache
    + (1-H) x (Lcache + Lorigin + Lfill)
```

Where `H` is hit ratio. The formula omits queueing, payload variation, parallel calls, and tail behavior. Averages can look healthy while p99 misses violate the service-level objective (SLO).

### 21.4 Capacity after a failure

If an `N`-node cluster loses one node and traffic redistributes, survivors need spare CPU, memory, and network. Cross-zone/region failover may also add latency and bandwidth cost.

Model at least:

- normal peak;
- one node/shard unavailable;
- one zone unavailable;
- rolling restart/resize;
- cold cache;
- full cache bypass capped to an origin-safe rate.

### 21.5 Choose key/value granularity

#### One large aggregate

```text
feed:user:42 -> 500 feed items in one blob
```

Advantages: one lookup, consistent snapshot. Costs: large transfers, rewrite entire blob, poor partial reuse, hot large key.

#### Many small items

```text
feed:user:42:page:0 -> IDs
post:1001 -> post object
post:1002 -> post object
```

Advantages: shared object reuse, smaller invalidations. Costs: many round trips, partial misses, assembly complexity. Use pipelining/batch gets where supported, but cap fan-out.

Choose based on access shape and invalidation unit, not a universal object size.

## 22. Observability and operation

### 22.1 Metrics that matter

#### Effectiveness

- hit and miss counts/ratios by cache, key class, endpoint, and status;
- byte hit ratio for large objects/media;
- negative-cache hit ratio;
- origin work saved and recomputation cost;
- item age/version/staleness where measurable.

#### Performance

- cache hit latency and miss end-to-end latency at p50/p95/p99;
- timeouts and connection-pool wait;
- fill/refresh latency;
- per-shard QPS, bandwidth, CPU, and memory;
- hot-key/heavy-hitter signals.

#### Capacity/policy

- resident entries/bytes;
- evictions versus expirations;
- rejected writes/no-memory errors;
- fragmentation/allocator overhead;
- key/value size distribution;
- write-back queue depth and oldest dirty age.

#### Reliability/consistency

- cache availability/error rate;
- replication lag and failovers;
- database fallback rate;
- invalidation event lag/failures/retries;
- stale-version detections;
- decode/schema errors;
- stampede coalescing leader/waiter counts;
- origin saturation correlated with misses.

### 22.2 Why a global hit ratio can mislead

```text
Endpoint A: 99.9% hits, cheap origin
Endpoint B: 60% hits, extremely expensive origin
global: looks acceptable
incident: B overloads database
```

Segment by endpoint, tenant, key family, shard, item size, and miss cost. Also distinguish compulsory cold misses, expiration, eviction, invalidation, and errors where the product allows.

### 22.3 Essential failure tests

- disable the cache in a load test;
- inject latency and timeouts, not only clean failures;
- kill one shard and one replica;
- restart/replace the cluster cold;
- expire many hot keys together;
- create one viral key;
- drop/delay/reorder invalidations;
- deploy incompatible value schemas safely;
- make the database fail during refresh/write-through/write-back;
- fill memory and observe eviction/rejection behavior;
- test that retries do not multiply load.

Run these with origin and application capacity limits that resemble production.

### 22.4 Operational runbook questions

- How do we bypass or disable the cache safely?
- What is the maximum origin fallback rate?
- Can we serve stale, and for which key classes/how long?
- How do we identify and purge one poison key/namespace?
- How do we roll out a cache-key/schema version?
- How do we add/remove nodes without mass remapping?
- How do we rebuild/warm after loss?
- Who owns invalidation and write-back backlog incidents?

## 23. Security, privacy, and multi-tenancy

### 23.1 Authorization must still be enforced

Possession of a predictable cache key must not grant access. Authenticate and authorize before returning private data, or design a trusted internal cache boundary with per-tenant namespaces and controls.

### 23.2 Cross-user leakage

This is dangerous:

```text
key = "/account/home"
value = Alice's personalized page
shared cache returns it to Bob
```

The key must vary on all security-relevant context, or the response must be marked private/non-cacheable at shared layers. It is often safer not to cache entire sensitive responses.

### 23.3 Sensitive data duplication

Caching creates more locations containing data. Consider:

- encryption in transit and at rest where supported/required;
- access control and network isolation;
- secret/credential rotation and short TTL;
- data residency and deletion requirements;
- logs/metrics that might expose keys/values;
- memory dumps/backups/snapshots;
- purge propagation for privacy deletion.

Hashing a user ID is not anonymization if it remains linkable or reversible by guessing.

### 23.4 Cache key injection and poisoning

Canonicalize inputs, limit key/value sizes, authenticate writers, and avoid allowing user-provided delimiters to escape namespaces. Do not let an untrusted caller choose arbitrary internal cache keys.

### 23.5 Timing side channels

A hit can be observably faster than a miss, revealing that some value/key was recently requested or exists. This matters for sensitive membership/account-recovery endpoints. Constant-work responses, rate limits, and not caching such decisions may be required.

## 24. Real-system examples

The examples are deliberately labeled so an illustrative interview design is not confused with a company's private current architecture.

### 24.1 WhatsApp-style chat: presence and connection routing

**Illustrative design; not a claim about WhatsApp's current private caching stack.**

A chat backend may keep fast, short-lived mappings such as:

```text
user:42:connection -> gateway-17, expires after heartbeat window
user:42:presence   -> online/last-seen approximation
```

```text
Sender -> Message Service -> Presence/route cache -> Recipient Gateway
                  |
                  +-------------------------------> Durable message store
```

Reasoning:

- routing/presence is read frequently and changes often;
- short TTL/heartbeats remove abandoned sessions;
- a stale mapping should cause retry/re-resolution, not message loss;
- the durable message record must not exist only in a disposable presence cache;
- a hot user can require local caching/replication and bounded fan-out;
- privacy settings and “last seen” authorization must be enforced before returning cached presence.

Interviewer follow-up: “If the cache says online but the gateway died?” Answer: delivery attempt fails, invalidate/re-resolve, and retain/retry from the durable message workflow. The cache is a routing hint, not delivery truth.

### 24.2 YouTube: hierarchical edge caching for video

**Verified public behavior.** Google documents that Google Global Cache (GGC) lets ISPs serve certain Google content inside their networks, and that most high-volume GGC traffic is cacheable YouTube content. GGC cache-fill is hierarchical and demand-driven.

```text
Viewer
  |
  v
Local ISP GGC node --miss--> peer/parent GGC --miss--> Google core/origin
  |
  +-- hit: nearby video bytes
```

Why cache video chunks near viewers:

- popular bytes are reused by many viewers;
- shorter network path can improve startup/buffering behavior;
- origin and transit bandwidth are reduced;
- unpopular content can fall back to another serving location.

Eviction needs to consider object size and byte miss cost, not just request count. Google's [HALP paper](https://research.google/pubs/halp-heuristic-aided-learned-preference-eviction-policy-for-youtube-content-delivery-network/) reports a learned-assisted DRAM eviction policy deployed in YouTube CDN production. Its reported numbers are a **verified 2023 publication snapshot**, not a general promise for all caches.

Sources: [Google GGC introduction](https://support.google.com/interconnect/answer/9058809?hl=en), [content served](https://support.google.com/interconnect/answer/7658599?hl=en), and [cache fill](https://support.google.com/interconnect/answer/7658593?hl=en).

### 24.3 Uber: integrated cache in an online storage path

**Public production snapshot.** Uber described an integrated cache for its Docstore online storage system because read traffic could greatly exceed writes and burden underlying MySQL shards. The design moved caching/invalidation responsibility closer to the storage platform instead of making every microservice operate its own Redis cache.

```text
Uber service -> Docstore/cache-aware access layer -> distributed cache
                            |
                            +-----------------------> storage/MySQL
```

Lessons:

- a shared platform can standardize invalidation and failover;
- region failover must consider whether the destination cache is warm;
- hot keys/partitions remain concerns even with a cache;
- integrated caching trades service-team simplicity for storage-platform complexity.

The “over 40 million reads per second” figure is from the title/content of Uber's 2023 post and should be repeated only as a **dated first-party snapshot**, not current capacity. Source: [Uber Engineering: integrated cache](https://www.uber.com/en-ES/blog/how-uber-serves-over-40-million-reads-per-second-using-an-integrated-cache/).

### 24.4 Netflix: EVCache across availability zones

**Verified public project behavior.** Netflix's EVCache documentation describes a distributed in-memory service using memcached servers, with server groups in zones. The documented model writes a key to all zones and normally reads locally to meet latency goals.

```text
Netflix service in Zone A -> EVCache group A (local read)
                 write ----> groups A, B, C
```

Lessons:

- zonal copies improve read locality and failure tolerance;
- cross-zone writes cost bandwidth and may partially fail;
- short TTLs bound some inconsistency;
- high-consistency needs require explicit retry/repair behavior;
- “cache” products can support uses with different durability expectations, so never infer guarantees from the name.

Sources: [EVCache introduction](https://netflix.github.io/EVCache/introduction/) and [features](https://netflix.github.io/EVCache/features/). These describe the public project, not every current Netflix data path.

### 24.5 Twitter/X: caching users, tweets, and timelines

**Historical public production snapshot.** X/Twitter's 2017 infrastructure post described Redis and Twemcache clusters caching users, timelines, tweets, and objects with expensive hydration cost. A 2012 Twemcache post described a large dedicated caching fleet at that time.

```text
Timeline service -> cache key(s) for timeline/tweet/user -> backing stores
                         |
                         +-- viral tweet becomes hot key
```

Lessons:

- cache fully assembled/hydrated objects when reconstruction is expensive;
- separate key families/tenants to reduce noisy-neighbor eviction;
- viral items need hot-key handling, not merely more shards;
- timeline invalidation/versioning is harder than caching an immutable tweet object.

Sources: [X/Twitter infrastructure scale](https://blog.x.com/engineering/en_us/topics/infrastructure/2017/the-infrastructure-behind-twitter-scale) and [Caching with Twemcache](https://blog.x.com/engineering/en_us/a/2012/caching-with-twemcache). Specific fleet sizes/rates in those posts are **historical, not current claims**.

### 24.6 Amazon/AWS: DAX and graceful caching failures

**Verified product behavior plus first-party engineering guidance.** DAX is a read-through/write-through cache for DynamoDB API paths. On cacheable misses it fetches from DynamoDB; strongly consistent reads are not served as ordinary cache hits under the documented behavior.

```text
Application -> DAX cluster -> hit
                    |
                    +-- miss/pass-through -> DynamoDB
```

Amazon's engineering guidance emphasizes that caching is not free: teams should choose TTL/size/eviction empirically, use negative caching where appropriate, handle thundering herds, and test the service when the cache is unavailable.

Sources: [AWS DAX concepts](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DAX.concepts.html), [DAX suitability](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/evaluate-dax-suitability.html), and [Amazon Builders' Library: caching challenges and strategies](https://aws.amazon.com/builders-library/caching-challenges-and-strategies/).

### 24.7 One cross-system comparison

| System example | Cached thing | Main objective | Main correctness concern |
|---|---|---|---|
| WhatsApp-style chat (illustrative) | Presence/connection route | Fast routing | Stale route must not lose durable message |
| YouTube | Video content/chunks near viewers | Latency and bandwidth | Version/content identity, eviction efficiency |
| Uber | Online storage reads | Offload storage, simplify clients | Invalidation and regional cold cache |
| Netflix EVCache | Application key-values across zones | Low-latency shared data | Partial writes and cross-zone inconsistency |
| X/Twitter historical | Users/tweets/timelines/hydrated objects | Read scale and compute saving | Hot keys, invalidation, multi-tenancy |
| Amazon DAX | DynamoDB item/query results | Read latency/load | Eventual versus strong read requirements |

## 25. Interview angle

### 25.1 How caching appears in interviews

Caching can be:

1. a subcomponent in URL shortener, feed, e-commerce, video, chat, search, or API design;
2. the full prompt: “Design a distributed cache”; or
3. a coding/object-design prompt: implement LRU/LFU/ranked cache.

The interviewer normally cares less that you know the word Redis and more that you can reason through:

- expected reuse and cache placement;
- cache key/value and size;
- read/write strategy and source of truth;
- TTL/invalidation/consistency;
- eviction and admission;
- sharding/replication/hot keys;
- cache outage and origin protection;
- observability and trade-offs.

### 25.2 Clarifying questions to ask

- Is this an application cache in front of a database, an HTTP/CDN cache, or a general cache service?
- What are get/set/delete APIs, key/value size limits, and total data volume?
- Read/write QPS, payload distribution, and latency SLO?
- Expected popularity skew and hot keys?
- Is the cache the only copy or disposable?
- Required consistency and maximum staleness?
- Must writes survive acknowledged node/zone failures?
- Single region or multi-region?
- Which failures must the design tolerate?
- Is eviction configurable per tenant/key class?

Do not interrogate mechanically. Ask the questions that change the design.

### 25.3 Answer spine for “Where would you add caching?”

Use these cues to reconstruct an answer live:

```text
REUSE -> VALUE -> KEY -> TRUTH -> FLOW -> FRESHNESS -> FAILURE -> MEASURE
```

1. **Reuse:** identify repeated expensive reads/computation and estimate hit potential.
2. **Value:** choose object/result granularity and size limits.
3. **Key:** include all result-changing and security context; namespace/version it.
4. **Truth:** name the authoritative durable source.
5. **Flow:** state cache-aside/read-through and exact write order.
6. **Freshness:** TTL + jitter, invalidation/version, acceptable stale behavior.
7. **Failure:** stampede, hot key, cache outage, origin fallback budget, replicas.
8. **Measure:** hit/byte-hit ratio, p99, evictions, hot shards, origin load, invalidation lag.

### 25.4 Answer spine for “Design a distributed cache”

```text
CONTRACT -> ROUTE -> STORE -> REPLACE -> REPLICATE -> FAIL -> OPERATE
```

1. **Contract:** APIs, item limits, TTL, consistency, durability, expected QPS/capacity.
2. **Route:** hash keys; client/proxy; consistent hashing/slots and membership versions.
3. **Store:** in-memory map, serialization, memory allocator/accounting, namespaces/quotas.
4. **Replace:** TTL processing, admission, LRU/LFU/random/size-aware policy.
5. **Replicate:** copies across failure domains; synchronous/async acknowledgement trade-off.
6. **Fail:** detection, failover, hot keys, cold start, stampede, origin protection.
7. **Operate:** metrics, rebalancing, rolling upgrades, security, backups only if durability demands them.

### 25.5 A small distributed-cache API discussion

```text
GET(namespace, key) -> value | miss | error
SET(namespace, key, value, ttl, [version], [condition])
DELETE(namespace, key, [version])
MGET(keys...) / MSET(...) with clearly limited atomicity
```

Consider:

- distinguish miss from service error;
- cap key/value/batch sizes;
- tenant quotas and authentication;
- conditional set (`if absent`, version compare) for fills/locks;
- idempotent delete;
- do not promise cross-shard atomicity;
- binary protocol/persistent connections may reduce overhead, but protocol choice follows performance and ecosystem requirements.

## 27. Synthesized interviewer follow-ups and answer cues

The questions in this section are **my plausible synthesis**, not claims from real interview posts.

### “Why not query the database directly?”

- Show repeated-read or computation cost.
- Estimate hit ratio and origin QPS reduction.
- Include network/cache overhead and operational cost.
- Say no cache if reuse/latency benefit is insufficient.

### “Why Redis instead of an in-process map?”

- Shared across instances, larger aggregate memory, independent app restarts.
- But adds network hop and cluster operations.
- Consider L1 + distributed L2 for extremely hot safely stale values.
- Product choice follows data structures, persistence, eviction, HA, and team operations—not brand familiarity.

### “How do you keep cache and database consistent?”

- Name system of record and allowed staleness.
- State exact read and write order.
- Database commit then invalidate/update.
- TTL is a repair bound; CDC/outbox retries missed invalidations.
- Versions prevent delayed fill/event from moving state backward.
- Admit two independent stores are not automatically atomic.

### “Why delete instead of update after a database write?”

- Derived value may depend on multiple records.
- Delete avoids duplicating transformation logic and caching unread data.
- Update can reduce next-read miss when freshness/latency matters.
- Both face partial failures; use version/retry/CDC as needed.

### “What if the cache is down?”

- Short timeout; distinguish miss from error.
- Do not send unbounded traffic to origin.
- Circuit breaker, fallback concurrency/rate limit, coalescing.
- Serve safe stale/default/partial results or shed load.
- Replicas and zone failure capacity.
- Test the disabled-cache path.

### “What if the database is down?”

- Hits may continue for cached data.
- Misses cannot refill.
- Serve stale only within per-data hard limit.
- Writes follow durability contract; do not accept in disposable cache.
- Degrade/fail explicitly and avoid retry storms.

### “How do you handle one viral key?”

- Detect per-key/shard heat.
- L1/edge cache, replicas, hot-key fan-out, coalescing.
- TTL jitter/refresh-ahead prevents expiry cliff.
- Rate limit abuse; ensure invalidation reaches copies.
- Adding ordinary shards alone does not split one key.

### “LRU or LFU?”

- LRU follows recent demand and adapts quickly; scan pollution risk.
- LFU retains stable popularity; needs aging and new-item admission.
- Random is cheap; size/cost-aware may match objective better.
- Measure hit/byte-hit ratio and miss cost on real traces.
- Production implementations can be approximate.

### “Can write-through guarantee consistency?”

- It improves controlled post-write cache freshness.
- Name exact ordering and acknowledgement boundary.
- Database success/cache failure remains possible without a transaction.
- Use authoritative commit, retry/repair/outbox, and versions.
- External writers still require invalidation integration.

### “When would you use write-back?”

- Only when lower write latency/batching is worth delayed origin persistence.
- Acknowledged dirty writes require durable log/replication.
- Per-key ordering, idempotency, backlog/backpressure, recovery.
- Avoid a disposable implementation for critical financial/order/message facts.

### “How do you invalidate every feed page affected by a new post?”

- Avoid enumerating an unbounded dependency graph if possible.
- Cache IDs/pages with short TTL or per-user generation/version.
- Fan-out updates for ordinary users; special handling for celebrity fan-out.
- Event-driven invalidation is at least once and versioned.
- Accept/restate feed freshness requirement.

### “How would you support multiple tenants?”

- Namespace/authenticate every request.
- Per-tenant quotas, memory/admission policy, and rate limits.
- Prevent one tenant's scan from evicting everyone else's hot data.
- Encrypt/isolate sensitive values; audit purge/access.
- Fair scheduling or separate pools for strong isolation.

## 28. Common mistakes and interviewer red flags

1. **“Use Redis” without proving reuse.** A cache with mostly misses adds latency and complexity.
2. **Calling the cache the source of truth accidentally.** State what survives cache loss.
3. **Confusing cache-aside with read-through.** Similar miss result; different owner.
4. **Treating write-through as a distributed transaction.** Explain partial failure.
5. **Saying write-around means no invalidation.** Existing cached data can remain stale.
6. **Using write-back for critical data without durable pending writes.** A node failure can lose acknowledged state.
7. **Deleting cache before database commit.** A reader can refill the old value.
8. **Missing the late-fill race even with database-then-delete.** Use TTL/version/coordination/repair appropriate to risk.
9. **One global TTL for everything.** Freshness is a per-data business contract.
10. **Synchronized TTLs.** They can create an expiration avalanche; add jitter/refresh control.
11. **No negative caching or input validation.** Nonexistent keys repeatedly hit the database.
12. **Ignoring hot keys.** More shards do not automatically distribute one key.
13. **Assuming replication means strong consistency/durability.** State sync/async and failover loss window.
14. **Letting every cache error fall through to the database.** This can cause a cascading outage.
15. **Using average hit ratio/latency alone.** Segment expensive key classes and inspect p99/byte hits.
16. **Caching personalized data under a shared key.** This is a privacy/security incident.
17. **Ignoring schema versions during rolling deploys.** Old/new clients can poison each other.
18. **Unbounded keys, values, batches, or local cache.** Memory exhaustion becomes the eviction policy.
19. **Assuming exact LRU is always best.** Workload, object size, cost, and implementation overhead matter.
20. **Calling a cache fault-tolerant because it has replicas.** Test node/zone/cluster loss, cold recovery, and origin survival.
21. **Prewarming everything.** It can waste space and overload the origin before real traffic begins.
22. **Retrying cache/database timeouts at every layer.** Retry multiplication can exceed the original traffic.

## 29. Compact revision sheet

### 29.1 One mental model

```text
Cache = fast, limited, potentially stale copy

Design:
reuse -> key/value -> source of truth -> read/write flow
      -> freshness -> eviction -> distribution -> failure -> metrics
```

### 29.2 Strategy memory cues

| Strategy | Cue |
|---|---|
| Cache-aside | “App checks, app loads, only demanded data enters.” |
| Read-through | “Cache facade loads on a miss.” |
| Write-around | “Write origin, bypass allocation, invalidate old copy.” |
| Write-through | “Synchronous origin persistence through controlled facade; partial failure still exists.” |
| Write-back | “Acknowledge cache/buffer now, persist later; dirty writes need durability.” |

### 29.3 Failure memory cues

| Failure | Cue |
|---|---|
| Stampede | “One missing hot key, many identical fills—coalesce.” |
| Avalanche | “Many keys/whole cache fall through together—jitter and bound fallback.” |
| Penetration | “Nonexistent keys always miss—negative cache/filter/validate.” |
| Hot key | “One key defeats ordinary sharding—replicate/L1/edge/coalesce.” |
| Pollution | “Cold scan evicts hot reuse—admission/separate pool.” |
| Cache outage | “Protect origin before bypassing.” |

### 29.4 Eviction cues

```text
FIFO   -> oldest insertion
LRU    -> oldest access
LFU    -> lowest frequency, needs aging
Random -> cheapest bookkeeping
TTL    -> expires soonest, freshness signal
Cost/size aware -> maximize saved work per resource
```

### 29.5 Design checklist

- [ ] What expensive work repeats, and what hit ratio is plausible?
- [ ] Where does the cache live: browser, CDN, proxy, L1, distributed L2?
- [ ] What is the key, value, namespace, version, and size limit?
- [ ] What is the durable authoritative source?
- [ ] What exact read-miss flow is used?
- [ ] What exact write order and success boundary are used?
- [ ] How stale may each data class be?
- [ ] How do TTL, jitter, invalidation, versions, and refresh interact?
- [ ] Which eviction/admission metric matches the workload?
- [ ] How are keys partitioned and replicas placed?
- [ ] How are hot keys, stampedes, penetration, and pollution handled?
- [ ] Can the origin survive a cache shard/cluster/cold-start event?
- [ ] Is fail-open, fail-closed, stale, degraded, or shedding behavior explicit?
- [ ] Are privacy, authorization, encryption, quotas, and schema evolution covered?
- [ ] Do metrics show hit quality, p99, evictions, hot shards, fallback, and invalidation lag?
- [ ] Have slow, partial, reordered, and cold failures been tested?

> “A cache is a fast, limited copy of expensive data, so I first prove reuse and name the durable source of truth. I define the key, TTL, acceptable staleness, and exact read/write order.
