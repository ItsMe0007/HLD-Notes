## 1. Plain-language foundation: what does a protocol do?

Imagine that two people want to exchange parcels. They need rules for:

- how to write the destination;
- how to divide a large shipment into boxes;
- how to identify the boxes and put them back in order;
- whether the receiver must confirm delivery;
- what to do when a box is lost;
- whether the parcel must be locked or signed;
- what the contents mean.

A **network protocol** is an agreed set of rules for communication between machines. Different protocols solve different parts of the problem. There is no single protocol called “the Internet protocol” that does everything.

For example, when a browser requests `https://shop.example/orders/42`, several mechanisms cooperate:

```text
Human intent:       Open order 42
Application:        HTTP request: GET /orders/42
Security:           TLS encrypts/authenticates the connection
Transport:          TCP carries an ordered byte stream
Addressing/routing: IP packets travel toward the server's IP address
Local delivery:     Wi-Fi/Ethernet moves frames to the next hop
Physical signal:    Radio/electrical/optical signals carry bits
```

An application designer usually chooses an **application communication style**—HTTP request/response, WebSocket, gRPC, WebRTC, messaging, and so on. The operating system and network then use lower-layer protocols such as TCP/UDP and IP.

### The four questions behind almost every protocol choice

Before naming a protocol, ask:

1. **Who initiates communication?** Only the client, or either side at any time?
2. **What matters when the network misbehaves?** Must every byte arrive, in order and without duplicates?
3. **What matters more: freshness or completeness?** Is a late item still useful?
4. **What operational constraints exist?** Browsers, firewalls, mobile networks, NAT, security, server connection count, bandwidth, battery, and implementation complexity.

Those questions lead to a reasoned choice. “Chat means WebSocket” and “video means UDP” are shortcuts, not designs.

## 2. Prerequisites needed for this topic

These are explained briefly here so the rest of the chapter is self-contained.

### 2.1 IP address, hostname, DNS, and port

- An **IP address** identifies a network interface/addressable endpoint for routing, such as `203.0.113.7` or an IPv6 address.
- A **hostname** is a human-friendly name such as `api.example.com`.
- **DNS** usually resolves a hostname to one or more IP addresses. DNS can also provide other records; email routing uses `MX` records.
- A **port** identifies a network service/process at a host. The combination of transport protocol, IP address, and port identifies a transport endpoint. TCP port 443 and UDP port 443 are distinct.
- A **socket** is the operating-system abstraction an application commonly uses to communicate through a transport protocol. It is not the same thing as WebSocket.

```text
https://api.example.com:443/orders
        |               |
        hostname        port
            |
          DNS
            v
       203.0.113.7
```

### 2.2 Client/server and peer-to-peer

- In **client/server**, clients initiate requests or connections to a service controlled by a server operator.
- In **peer-to-peer (P2P)**, participants can both consume and provide resources. P2P is an architectural model, not one wire protocol.
- A system can be hybrid. BitTorrent may use a tracker or DHT for discovery and peers for data. WebRTC needs signaling and often STUN/TURN servers even when media later travels directly between peers.

### 2.3 NAT and firewalls

Home and mobile devices commonly have private addresses hidden behind **Network Address Translation (NAT)**. An outbound connection is easy: the NAT records it and permits matching replies. An unsolicited inbound connection is often blocked because the NAT has no mapping and a firewall may reject it. This is why direct P2P connectivity needs discovery and NAT-traversal techniques and is not always possible.

### 2.4 TLS and HTTPS

**TLS** supplies encryption, integrity protection, and server authentication; client authentication is optional. **HTTPS is HTTP protected by TLS**, not a separate replacement for HTTP semantics. TLS 1.3 is standardized in [RFC 8446](https://www.rfc-editor.org/info/rfc8446/).

TLS does not automatically make an application authorized or safe. A server must still authenticate users, check permissions, validate input, prevent replay where relevant, and protect stored data.

## 3. “All seven layers?” — OSI versus the real Internet stack

### 3.1 Why layers exist

Networking is divided into layers so that each layer offers a service to the layer above and hides some lower-level detail. A web developer can use HTTP without implementing radio modulation; a router can forward IP packets without understanding an order JSON body.

This creates:

- **modularity:** Wi-Fi can replace Ethernet without redesigning HTTP;
- **interoperability:** independent implementations can communicate using a shared standard;
- **debugging boundaries:** “DNS resolved, TCP connected, TLS failed” narrows the problem;
- **independent evolution:** HTTP/3 can map HTTP semantics onto QUIC rather than TCP.

The OSI model is a **reference model**, not a claim that every production packet passes through seven separately implemented software boxes. ISO explicitly describes it as a common framework rather than an implementation specification ([ISO/IEC 7498-1](https://www.iso.org/standard/20269.html)).

### 3.2 The seven OSI layers

Read the table from bottom to top when asking “how do bits reach the application?” and top to bottom when asking “how does an application message get sent?”

| # | OSI layer | Plain-language job | Typical examples or concepts | Useful troubleshooting question |
|---:|---|---|---|---|
| 7 | Application | Rules meaningful to applications | HTTP, SMTP, IMAP, FTP, DNS, WebSocket | Is the request valid? Did the server return an application error? |
| 6 | Presentation | Representation, encoding, compression, encryption | UTF-8, JSON, Protobuf, JPEG; TLS is often placed here in teaching diagrams | Can both sides decode/decrypt the bytes? |
| 5 | Session | Establish/manage logical conversations and checkpoints | Session management, reconnection, dialogue state | Did the logical session expire or lose state? |
| 4 | Transport | End-to-end process communication | TCP, UDP; ports, reliability, flow control | Is the port reachable? Loss? retransmission? |
| 3 | Network | Addressing and routing across networks | IPv4/IPv6, ICMP; routers | Is there a route to the destination IP? |
| 2 | Data link | Delivery on one local link | Ethernet, Wi-Fi MAC frames, VLANs, switches | Can this device reach the next hop on the LAN? |
| 1 | Physical | Transmit raw bits as signals | Copper, fiber, radio, connectors | Is the link/radio physically working? |

Common mnemonics exist, but understanding the responsibility of each layer is more useful than memorizing the order.

### 3.3 The Internet/TCP-IP model used in practice

Production Internet engineering is more often described with four or five layers. The IETF host architecture identifies link, Internet, transport, and application layers; its application layer does not subdivide OSI session and presentation functions ([RFC 1122](https://www.rfc-editor.org/info/rfc1122/)).

```text
OSI reference model                 Practical Internet model

7 Application  ┐
6 Presentation ├──────────────────> Application   HTTP, TLS*, DNS, SMTP...
5 Session      ┘
4 Transport    ───────────────────> Transport     TCP, UDP, QUIC*
3 Network      ───────────────────> Internet      IP, ICMP
2 Data Link    ┐
1 Physical     ┴──────────────────> Link          Ethernet, Wi-Fi, fiber...

* Layer labels are teaching aids, not perfect truths:
  TLS sits above a transport and below an application protocol.
  QUIC runs over UDP but provides transport-like capabilities to HTTP/3.
```

### 3.4 Encapsulation: how a message moves through layers

Each layer adds control information needed by its peer. This wrapping is **encapsulation**.

```text
HTTP message
   ↓ add TLS records (for HTTPS)
TLS-protected bytes
   ↓ divide into transport data
TCP segments
   ↓ add source/destination IP addresses
IP packets
   ↓ add next-link addressing/check information
Wi-Fi/Ethernet frames
   ↓ encode as signals
Bits on radio/copper/fiber
```

At the receiver, the process is reversed. Intermediate routers mainly inspect link/network information; they do not normally need to understand the application body.

### 3.5 What HLD interviewers expect from layers

Usually not a seven-layer recital. They want you to avoid category errors and reason across boundaries:

- REST is an architectural style commonly expressed over HTTP; it is not an alternative to TCP.
- HTTP and TCP are not direct competitors; HTTP/1.1 and HTTP/2 normally use TCP, while HTTP/3 uses QUIC over UDP.
- WebSocket is an application protocol that usually runs over TCP after an HTTP opening handshake.
- P2P is a topology; it may use TCP, UDP, QUIC, WebRTC, or a custom protocol.
- Reliability at TCP does not guarantee the business operation happened exactly once.

## 4. Transport foundation: TCP, UDP, and QUIC

### 4.1 TCP in plain language

TCP makes an unreliable packet network look to an application like a **reliable, ordered stream of bytes** between two endpoints.

Suppose the sender writes `ABCDEFGHIJ`. IP packets could be lost, duplicated, or reordered. TCP numbers bytes/segments, acknowledges received data, retransmits missing data, removes duplicates, and releases bytes to the receiving application in order.

```text
Sender                  Network                   Receiver
  |--- SYN ---------------------------------------->|
  |<-- SYN-ACK -------------------------------------|
  |--- ACK ---------------------------------------->|
  |                                                 | connection established
  |--- bytes 1..100 ------------------------------->|
  |--- bytes 101..200 ----X (lost)                  |
  |--- bytes 201..300 ----------------------------->| buffered
  |<-- ACK / missing progress ----------------------|
  |--- retransmit 101..200 ------------------------>|
  |                                                 | app reads ordered stream
```

TCP provides:

- connection establishment;
- reliable delivery of the byte stream while the connection remains viable;
- in-order delivery to the application;
- duplicate suppression;
- **flow control**, so a fast sender does not overwhelm the receiver’s buffer;
- **congestion control**, so senders adapt to perceived network capacity/congestion.

The current core TCP specification is [RFC 9293](https://www.rfc-editor.org/info/rfc9293/).

#### What TCP does not provide

- Message boundaries: two `send()` calls can be read as one chunk or many; the application protocol needs framing.
- Encryption or authentication: use TLS or an application security mechanism.
- Exactly-once business execution: a response can be lost after the server committed a payment. Retrying may duplicate it unless the API uses idempotency/deduplication.
- Infinite retry: TCP eventually times out/fails and the application must reconnect and recover.
- Low latency under every kind of loss: missing bytes delay later bytes on that ordered stream.

#### TCP head-of-line blocking

Because the application receives one ordered stream, later bytes wait for earlier missing bytes. HTTP/2 multiplexes several logical streams on one TCP connection, but a lost TCP segment can still temporarily stall all of them at the transport boundary. This is one motivation for QUIC/HTTP/3’s independent streams.

### 4.2 UDP in plain language

UDP sends independent **datagrams**. It has ports and a checksum, but it does not itself establish a connection, acknowledge delivery, retransmit, reorder, control the sending rate, or prevent duplicates. It deliberately provides little mechanism ([RFC 768](https://www.rfc-editor.org/info/rfc768/)).

```text
send frame 41  ------------------------------> receive frame 41
send frame 42  --------X lost
send frame 43  ------------------------------> receive frame 43 now

Application choice: use fresh frame 43; do not wait for stale frame 42.
```

UDP is useful when the application wants control over timeliness and recovery, for example:

- real-time voice/video where a late packet may be worse than a missing packet;
- DNS queries (with important qualifications and TCP fallback/other transports);
- telemetry or game-state updates where newer state supersedes older state;
- building a modern transport such as QUIC in user space.

The application may add selected reliability: sequence numbers, acknowledgements, forward error correction, jitter buffers, rate control, or retransmission only when it would still arrive in time.

#### UDP risks and costs

- The application has more responsibility.
- Large datagrams may be fragmented or dropped; designs generally stay within a safe path MTU and perform application-level chunking.
- NAT mappings expire; keepalives and reconnection may be required.
- Some networks block or throttle UDP.
- Sending without congestion control can harm both the service and the network. “UDP allows me to blast packets” is not a valid production design.

### 4.3 Correcting “TCP slow, UDP fast”

That sentence is **misleading**.

- UDP has a smaller protocol mechanism and no mandatory handshake/retransmission ordering, so it can avoid some delay.
- But UDP is not automatically faster. If the application needs reliable ordered delivery, it must implement similar mechanisms or use a transport such as QUIC; poor implementations can be slower or unsafe.
- TCP can deliver extremely high throughput on long transfers. Its congestion control, loss recovery, connection reuse, and tuned operating-system implementation are major strengths.
- “Fast” can mean low startup latency, low per-message latency, high throughput, fast recovery during network changes, or fresh real-time data. A design must name the metric.

The better interview statement is:

> Use reliable ordered transport when missing or reordering data is unacceptable. Use a datagram-oriented or independently streamed transport when timeliness and application-controlled recovery matter. Then account for security, congestion control, NAT, and deployment support.

### 4.4 QUIC: why “HTTP means TCP” is no longer always true

QUIC is a secure, multiplexed transport built over UDP. It implements reliability, congestion control, encryption, and multiple independent streams in user space. HTTP/3 maps HTTP semantics onto QUIC. A lost packet affecting one QUIC stream does not force unrelated streams to wait for that stream’s missing data. HTTP/3 also has low-latency connection-establishment advantages and can better handle some address changes, such as moving between Wi-Fi and cellular.

[RFC 9114](https://www.rfc-editor.org/rfc/rfc9114.html) defines HTTP/3 over QUIC and explicitly recommends falling back to TCP-based HTTP when UDP connectivity fails.

Important correction: QUIC running over UDP does **not** mean HTTP/3 is unreliable. QUIC adds reliable, ordered delivery *per stream*.

#### TCP vs UDP vs QUIC summary

| Property | TCP | UDP | QUIC (typical reliable stream) |
|---|---|---|---|
| Abstraction | Ordered byte stream | Independent datagrams | Multiple independent streams plus optional datagrams |
| Built-in delivery recovery | Yes | No | Yes, per reliable stream |
| Built-in congestion control | Yes | No | Yes |
| Encryption built in | No | No | Yes (TLS 1.3 integrated) |
| Setup | TCP handshake; TLS separately for HTTPS | No transport handshake | Combined secure transport setup; resumption possible |
| Loss effect | Can block later bytes in same connection stream | Application decides | Affected stream waits; unrelated streams can progress |
| Typical use | DB connections, SSH, HTTP/1.1/2, email | Real-time media building blocks, DNS, telemetry | HTTP/3, transports needing multiplexed secure streams |

## 5. Application protocols and architectures from your bullets

### 5.1 HTTP and HTTPS

#### Foundation

HTTP is a request/response application protocol. A client sends a method, target, headers, and possibly a body; a server returns a status, headers, and possibly a body.

```http
GET /videos/123/metadata HTTP/1.1
Host: api.example.com
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=60

{"id":"123","title":"Networks"}
```

HTTP semantics include methods, status codes, fields, caching, conditional requests, content negotiation, authentication hooks, and range requests. They are standardized in [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html).

#### Versions in one mental model

- **HTTP/1.1:** textual message framing; multiple connections are often used for parallelism.
- **HTTP/2:** binary framing and multiplexed logical streams on one TCP connection; header compression.
- **HTTP/3:** similar HTTP semantics mapped onto QUIC rather than TCP.

The method and resource model can remain the same while the wire mapping changes.

#### Why HTTP is a default in HLD

- Universal browser, mobile, proxy, load balancer, gateway, CDN, and observability support.
- Stateless request/response works well with horizontally scaled services.
- Caching, conditional GETs, range requests, authentication fields, and standardized errors already exist.
- HTTPS crosses enterprise and mobile networks more reliably than unfamiliar custom protocols.

#### When to use

- CRUD APIs and resource access;
- browser/mobile to backend requests;
- file/media delivery through CDNs;
- service APIs where interoperability and debuggability matter;
- short-lived or naturally request-driven interactions.

#### Limitations and alternatives

- Ordinary request/response does not let a server spontaneously send an event without a pending request.
- Repeated polling wastes requests and adds freshness delay.
- For server-to-client updates consider **Server-Sent Events (SSE)**, long polling, webhooks, or a streaming response.
- For full-duplex interactive communication consider **WebSocket** or bidirectional gRPC streaming where supported.
- For hard real-time peer media consider **WebRTC**, not raw WebSocket media by default.

#### HTTP is not REST

HTTP is a protocol. REST is an architectural style with constraints such as resource-oriented interfaces, stateless interactions, and cacheability. An HTTP API may be REST-like, RPC-like, GraphQL, or something else.

**RPC (Remote Procedure Call)** models a network interaction like invoking a named operation—such as `CreateOrder`—rather than manipulating a resource with a generic HTTP method. **gRPC** is a popular RPC framework that commonly uses Protocol Buffers for schemas/code generation and HTTP/2 for transport. REST-style HTTP and gRPC are application-interface alternatives, but both ultimately use lower network layers; neither replaces TCP/IP.

#### Idempotency and retries

Network failure creates ambiguity:

```text
Client ---- POST charge ----> Server commits charge
Client <--- response --------X response lost
Client sees timeout. Did the charge happen?
```

TCP reliability does not solve this after a connection failure. The API needs an idempotency key, a stable operation ID, deduplication storage, and a defined retry policy for non-idempotent operations.

### 5.2 FTP, FTPS, and SFTP

#### FTP

FTP is an older application protocol dedicated to file transfer ([RFC 959](https://www.rfc-editor.org/info/rfc959/)). It typically uses:

- a long-lived **control connection** for login and commands;
- a separate **data connection** for directory listings or each transfer.

```text
FTP client == control connection ==> FTP server
FTP client <=== separate data connection ===> FTP server
```

In **active FTP**, the server opens the data connection back toward the client, which conflicts with many client-side NAT/firewall policies. In **passive FTP**, the server tells the client where to open the data connection, which is generally easier through NAT but still requires extra port handling.

Classic FTP does not protect credentials or file contents. Do not choose plain FTP for sensitive Internet transfers.

#### FTPS

FTPS means FTP protected with TLS. It retains FTP’s command/data-channel behavior. It is useful when an existing partner or enterprise workflow requires FTP semantics but security is mandatory. Firewall/NAT configuration can remain more awkward than a single HTTPS or SSH connection.

#### SFTP

SFTP means **SSH File Transfer Protocol**. It is not “FTP with an S added,” and it is not the same as FTPS. It runs as a subsystem over a secure SSH channel and supports remote file-system operations. The widely deployed protocol evolved through IETF drafts rather than a final classic RFC; the draft specification describes secure file transfer/file-system access over SSH ([IETF SFTP draft](https://datatracker.ietf.org/doc/html/draft-ietf-secsh-filexfer-12)).

#### Choosing among them

| Requirement | Better starting choice | Reason / caution |
|---|---|---|
| Public web/mobile download | HTTPS + object storage/CDN | Broad support, caching, range/resume, security |
| Admin transfers to Unix-like hosts | SFTP/SCP-style tooling | SSH authentication and one secure channel |
| Managed B2B file drop | SFTP or managed HTTPS | Strong auth, audit, automation; follow partner constraints |
| Legacy integration explicitly requiring FTP | FTPS if possible | Do not expose credentials/data in plain FTP |
| Browser upload/download | HTTPS | Browsers do not provide a general SFTP client API |

Common mistake: choosing FTP merely because the payload is a file. HTTP object delivery is usually the modern Internet default; SFTP is common for operational/B2B file exchange.

### 5.3 SMTP, IMAP, POP3, and MTA

#### First correction

- **SMTP** is primarily for submitting and relaying outgoing mail.
- **IMAP** and **POP3** are for a mail client to access received mail.
- **POP3** is version 3 of POP; saying “POP and POP3” usually does not name two current alternatives.
- **MTA** (Message Transfer Agent) is a software role/component, not a protocol.

#### End-to-end email path

```text
Alice's mail app
    MUA (Mail User Agent)
          |
          | message submission, usually SMTP submission + TLS/auth
          v
    MSA (Mail Submission Agent)
          |
          | SMTP relay; DNS MX helps locate Bob's mail domain
          v
    outbound MTA  --->  one or more MTAs  --->  final MTA
                                                |
                                                v
                                          MDA / message store
                                                |
                              IMAP or POP3       |
Bob's mail app (MUA) <---------------------------+
```

The Internet mail architecture and these roles are documented in [RFC 5598](https://www.rfc-editor.org/info/rfc5598/). SMTP is standardized in [RFC 5321](https://www.rfc-editor.org/info/rfc5321/).

#### Why email uses store-and-forward

Alice and Bob do not have to be online simultaneously. An MTA accepts responsibility, stores the message durably, attempts the next hop, retries temporary failures, and may later generate a delivery-status notification. This is asynchronous application-level reliability, not merely TCP retransmission.

#### IMAP versus POP3

| Property | IMAP | POP3 |
|---|---|---|
| Main model | Server mailbox is authoritative; client synchronizes | Traditionally download messages to a client, optionally delete/retain server copy |
| Multi-device use | Strong fit: folders, flags/read state, server-side search/sync | Simpler and less capable for rich multi-device state |
| Offline use | Client caches then resynchronizes | Downloaded mail naturally available locally |
| Server operations | Manipulate folders/mailboxes and message flags | Small command set centered on listing/retrieval/deletion |

IMAP4rev2 supports server mailbox manipulation and offline resynchronization ([RFC 9051](https://www.rfc-editor.org/info/rfc9051/)). POP3 is defined in [RFC 1939](https://www.rfc-editor.org/info/rfc1939/).

For modern deployments, use TLS-protected mail submission/access. [RFC 8314](https://www.rfc-editor.org/rfc/rfc8314.html) recommends TLS for email submission and access; blindly memorizing old cleartext port numbers is less useful than understanding explicit versus implicit TLS and provider requirements.

#### Where this appears in HLD

- Design an email service: queues, durable store-and-forward, retry scheduling, spam filtering, bounce handling, idempotency, mailbox storage, search, attachment object storage.
- Send product emails: application → durable job queue → mail provider/SMTP submission; do not block the user request while waiting for remote mail delivery.
- Receive user mail: understand that SMTP delivery and IMAP access are different flows.

### 5.4 WebSocket

#### Foundation

WebSocket provides a long-lived, full-duplex message channel between client and server. After an HTTP/1.1 opening handshake/upgrade, either side can send WebSocket frames without creating a new HTTP request for each message. The standard defines its relationship to HTTP and TCP in [RFC 6455](https://www.rfc-editor.org/info/rfc6455/).

```text
Client                           Server
  |--- HTTP Upgrade request ------>|
  |<-- 101 Switching Protocols ----|
  |                                 |
  |<====== WebSocket frames =======>|
  |   either side, long-lived       |
```

Use `wss://` in production so TLS protects the channel.

#### Good requirements for WebSocket

- low-latency bidirectional events;
- both client and server frequently initiate messages;
- browser support is required;
- examples: collaborative editing, multiplayer control events, live chat, presence, dashboards with interactive control.

#### Costs hidden by “just use WebSocket”

- Each active client consumes connection state somewhere.
- Load balancers, gateways, and servers need long idle timeouts/keepalive handling.
- A dropped connection needs reconnect with backoff and jitter.
- The client may have missed events; it needs a cursor/sequence number and catch-up API.
- A user can have several devices/connections; the service needs a connection registry and fan-out routing.
- Slow consumers need bounded buffers and a backpressure/drop/disconnect policy.
- Deployments need connection draining so live connections migrate safely.
- WebSocket delivery is not durable messaging. A frame reaching a gateway is not proof it was stored or processed.

```text
                         +--> WS gateway A -- clients
Client connections --->  |                    
                         +--> WS gateway B -- clients
                                  ^
                                  | connection registry / routing
                                  v
Publisher --> durable log or broker --> fan-out workers
                  |
                  +--> history store for reconnect/catch-up
```

#### WebSocket alternatives

| Requirement | Candidate | Why |
|---|---|---|
| Occasional updates; seconds of delay acceptable | Polling | Simplest operationally; cacheable |
| Server → browser stream only | SSE | HTTP-based, automatic reconnection/event IDs, text events |
| Server → browser with broad fallback | Long polling | One pending request; works through common HTTP infrastructure |
| Server → server notification | Webhook | Receiver exposes HTTP endpoint; secure, sign, retry, deduplicate |
| Bidirectional interactive browser channel | WebSocket | Full duplex and low per-message overhead |
| Internal typed bidirectional streaming | gRPC streaming | Schema/codegen and HTTP/2; browser/proxy constraints need evaluation |
| Real-time audio/video | WebRTC | Media-specific transport, congestion/jitter handling, NAT traversal |

### 5.5 P2P

#### Foundation

In a P2P system, a node can download from and upload to other nodes. This can distribute bandwidth/storage cost and avoid making one origin send every copy.

```text
                 discovery service
                / tracker or DHT \
               v                  v
           Peer A <------------> Peer B
             ^  \                /  ^
             |   \              /   |
             |    v            v    |
           Peer D <-----------> Peer C

Each peer may own different chunks and exchange them.
```

BitTorrent is a concrete P2P file-distribution protocol. Peers exchange pieces, verify piece hashes, and upload to one another, reducing origin load when many clients want the same content ([BitTorrent BEP 3](https://www.bittorrent.org/beps/bep_0003.html)).

#### Problems every P2P design must solve

- **Peer discovery:** tracker, DHT, local discovery, invite/signaling service.
- **NAT traversal:** direct connections may fail; relay may be required.
- **Content identity/integrity:** cryptographic hashes or signatures.
- **Availability:** peers leave; rare pieces need replication/seeding.
- **Incentives/fairness:** prevent free-riding or abusive resource use.
- **Security and trust:** malicious peers, poisoning, metadata privacy, identity.
- **Operations/legal constraints:** enterprise networks may block it; users may not consent to upload/battery use.

#### Use or avoid

Use P2P when many participants want the same large content, can contribute resources, and decentralized scaling is valuable. Avoid it as a default when central audit/control, predictable latency, strict privacy, mobile battery/data, or simple operations matter more.

### 5.6 WebRTC

#### Foundation

WebRTC is a collection of browser APIs and protocols for real-time audio, video, and data. It is not merely “a UDP protocol” and not automatically serverless. The W3C specification includes peer connections, media tracks, data channels, and NAT traversal through ICE/STUN/TURN ([W3C WebRTC](https://www.w3.org/TR/webrtc/)).

#### Connection setup vocabulary

- **Signaling:** the applications exchange session descriptions, capabilities, and network candidates. WebRTC does not mandate the signaling protocol; HTTP or WebSocket is common.
- **ICE:** gathers and tests possible network paths and selects a working candidate pair.
- **STUN:** helps a device discover the public address/port mapping seen outside its NAT.
- **TURN:** relays traffic when a direct path cannot be established. TURN improves reachability but adds server bandwidth cost and latency.
- **SRTP:** protects real-time media.
- **DataChannel:** carries arbitrary data, commonly over SCTP secured by DTLS in the WebRTC stack.

```text
1. Signaling (not the media path)

Browser A ---- HTTPS/WebSocket ---- Signaling service ---- Browser B
     |                                                     |
     +-- exchange offer/answer and ICE candidates ---------+

2. Selected media/data path

Best case:    Browser A <========== direct ==========> Browser B

Fallback:     Browser A ======> TURN relay ==========> Browser B
```

#### Why real-time media often prefers datagram behavior

For a live call, frame 101 arriving very late is often useless after frame 120 is displayed. Media systems use timestamps, sequence numbers, jitter buffers, congestion control, selective retransmission, keyframes, and sometimes forward error correction. They trade perfect completeness for timely playout.

#### Multiparty calls: mesh versus SFU

```text
Mesh (small call)                 SFU (larger call)

A <----> B                         A --upload--> +-----+ --forward--> B
| \    / |                         B --upload--> | SFU | --forward--> C
|  \  /  |                         C --upload--> +-----+ --forward--> A
C <----> D
```

- **Mesh:** peers send to every other peer. Simple infrastructure, but each client’s upload and CPU grow with participant count.
- **SFU (Selective Forwarding Unit):** each client sends one/few streams to a server that forwards appropriate streams. Better client scaling, but the service pays network/operational cost.
- **MCU:** server decodes/composes/re-encodes media. Simplifies clients/output but costs much more compute and may add latency/quality loss.

#### WebRTC versus WebSocket

Use WebSocket for ordered application events such as “user raised hand” or chat messages. Use WebRTC for live media because it has codecs, timestamps, congestion adaptation, jitter handling, NAT traversal, and media security. A call product may use both.

## 6. How to choose a protocol from requirements

### 6.1 Decision sequence

Use this sequence in an interview:

```text
1. Is this request/response, server push, or bidirectional?
   |
   +-- request/response ----------> HTTPS API
   +-- server-to-client only -----> SSE / long polling / stream
   +-- bidirectional events ------> WebSocket / bidirectional RPC
   +-- real-time media -----------> WebRTC / real-time media protocol suite

2. Must every item arrive and remain ordered?
   |
   +-- yes -----------------------> reliable stream + app acknowledgements
   +-- no; newest supersedes old -> datagrams/selective reliability

3. Is it a large object or continuous media?
   |
   +-- large immutable object ----> HTTPS + CDN + byte ranges/resume
   +-- adaptive playback ---------> segmented HTTP streaming
   +-- mass distribution where peers help -> consider P2P

4. What environment limits the choice?
   Browser? mobile NAT? firewalls? partner standard? security? cost?

5. Define failure recovery.
   Retry? idempotency? reconnect? replay cursor? resume offset? fallback?
```

### 6.2 Requirement-to-choice matrix

| Requirement | Strong candidate | Main trade-off / follow-up |
|---|---|---|
| Public CRUD API | HTTPS (REST-like or RPC-like) | Cacheability, idempotency, versioning, pagination |
| Internal low-latency typed service calls | gRPC over HTTP/2 or HTTP/3-capable RPC | Coupling, deadlines, retries, load balancing, browser support |
| Live chat/presence | WebSocket plus durable history/catch-up | Connection scaling, ordering scope, offline delivery |
| One-way live dashboard | SSE | Server-to-client only; text-oriented browser API |
| Infrequent notification to another backend | Webhook | Receiver availability, signing, retry and dedupe |
| Voice/video call | WebRTC; SFU for larger rooms | NAT/TURN cost, congestion, codecs, regional media routing |
| Large public file download | HTTPS + CDN + ranges | Integrity, authorization, expiration, range support |
| Secure admin/B2B file transfer | SFTP or managed HTTPS | Partner support, key rotation, audit, automation |
| Email submission/relay | SMTP with appropriate TLS/auth | Queues, retry, bounce/spam/reputation |
| Multi-device mailbox access | IMAP | Sync complexity, storage/search |
| Massive shared file distribution | CDN first; P2P if peers may contribute | Trust, seeding, NAT, client bandwidth/battery |
| Frequent replaceable telemetry | UDP/QUIC datagrams or broker protocol | Loss tolerance, rate/congestion control, authentication |
| Financial command | HTTPS/gRPC over reliable transport | App idempotency and audit are more important than protocol label |

### 6.3 The most important distinction: transport reliability vs business reliability

Consider a message sent over TCP/WebSocket:

1. TCP ACK means bytes reached the remote TCP stack, not that the database committed.
2. WebSocket send completion may mean bytes entered a local buffer, not that the user saw them.
3. An application ACK can mean the server accepted/stored the message.
4. A delivery ACK can mean the recipient device received it.
5. A read receipt can mean the UI displayed/marked it read.

Define the exact guarantee instead of saying “TCP guarantees delivery.”

## 7. Download modes and how IDM works

### 7.1 Normal resumable HTTP download

A server can support HTTP **range requests**, allowing a client to request part of a representation:

```http
GET /large.iso HTTP/1.1
Range: bytes=1000000-

HTTP/1.1 206 Partial Content
Content-Range: bytes 1000000-4999999/5000000
```

This enables:

- resuming after an interruption;
- fetching only a needed region;
- downloading different regions concurrently.

Range semantics are now part of [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html#name-range-requests); the earlier range-specific [RFC 7233](https://www.rfc-editor.org/info/rfc7233/) explains recovery from interrupted transfers particularly clearly.

The server may ignore a range request. Resume also becomes unsafe if the file changed between attempts. A robust client uses validators such as an ETag or Last-Modified value with conditional/range logic, and restarts if it cannot prove it is continuing the same representation.

### 7.2 Segmented/parallel downloading

Suppose a 1,000 MB file is divided into four byte ranges:

```text
Connection 1: bytes   0 MB .. 249 MB  ───────┐
Connection 2: bytes 250 MB .. 499 MB  ───────┤
Connection 3: bytes 500 MB .. 749 MB  ───────┼─> verify and assemble file
Connection 4: bytes 750 MB .. 999 MB  ───────┘
```

Parallel connections may improve throughput when one connection does not fill the available path—because of latency, per-connection limits, slow-start behavior, or different paths/endpoints. They also let the client retry only a failed segment.

They do **not create bandwidth**. If one modern connection already saturates a 100 Mbps access link, four connections cannot make the physical link exceed 100 Mbps. They may instead compete with browsing/voice traffic, add server load, trigger rate limits, or reduce fairness to other users.

### 7.3 IDM’s working model

Internet Download Manager’s first-party documentation says it uses **dynamic file segmentation**, reuses completed connections, and splits the largest remaining segment when another connection becomes available. It periodically saves positions so interrupted work can resume, provided the remote server supports resumption ([IDM dynamic segmentation](https://www.internetdownloadmanager.com/support/segmentation.html), [IDM overview](https://www.internetdownloadmanager.com/support/about.html)).

Conceptually:

```text
Initially:
[------------------------- one large remaining region -------------------------]

Two connections:
[=========== C1 ===========][=========== C2 ===========]

C1 finishes early; split C2's largest remaining region:
[C1 done][--- C1 helps here ---][--------- C2 continues ---------]

Finally:
[segment A][segment B][segment C][segment D] -> assembled in byte order
```

The download manager typically needs:

- a URL and authorization/cookies if protected;
- content length and range support;
- a map of completed and pending byte ranges;
- per-connection retries/timeouts;
- representation validation so pieces are not mixed across file versions;
- final size/hash/integrity verification where available;
- safe assembly and atomic rename to the final filename.

Do not repeat IDM’s marketing claim of a fixed “up to N times faster” as a general engineering guarantee. Improvement depends on the server, path, existing bottleneck, connection limits, protocol version, and local disk/CPU.

### 7.4 “Stable fast” versus “unstable slow” option

Under the likely segmented-download interpretation:

#### Stable/high-speed mode

- More concurrent range requests.
- Larger amount of data in flight.
- Aggressive timeouts/retries may recover quickly from a slow connection.
- Can achieve higher aggregate throughput when a single connection underutilizes the path.
- Costs more sockets, memory, server requests, and competition with other traffic.

#### Unstable/low-speed mode

- Fewer simultaneous connections, sometimes one.
- Smaller chunks/checkpoints so failed work is cheaper to retry.
- Longer timeouts and exponential backoff to avoid retry storms.
- Resume metadata is saved frequently.
- Lower peak throughput, but less churn on lossy/intermittent links and less chance of server throttling.

There is no universal rule that “more connections are better on a stable network and fewer are always better on an unstable network.” A good downloader measures throughput, loss/timeouts, and server responses, then adapts concurrency and chunk size.

AWS documents the same general trade-off for S3: concurrent byte-range GETs can increase throughput and make interrupted retries smaller, but more/smaller ranges create more requests and cost ([S3 performance guidelines](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html), [multipart download discussion](https://aws.amazon.com/blogs/developer/introducing-multipart-download-support-for-aws-sdk-for-net-transfer-manager/)).

### 7.5 If the bullet referred to video quality instead

Adaptive bitrate streaming stores a video in multiple qualities and divides each into short segments. The player estimates bandwidth and buffer health:

```text
High/stable bandwidth:   request 1080p segments  ───> higher quality
Falling/unstable link:   request 480p segments   ───> fewer stalls
Recovered bandwidth:    cautiously move upward
```

This is not “TCP versus UDP.” YouTube/Netflix-style on-demand playback can use ordinary HTTP/HTTPS segment requests while the player changes the representation. The system deliberately reduces visual quality to preserve smooth playback.

## 8. Real-system examples

Public documentation rarely exposes every current transport detail. The verified observations below are separated from illustrative HLD choices.

### 8.1 WhatsApp

**Verified public observation:** Meta describes WhatsApp clients receiving pushed messages over a persistent connection and identifies its interop client protocol as XMPP-based; media can be downloaded separately. This is evidence for persistent client/server messaging, but it is **not evidence that WhatsApp uses WebSocket** ([Meta engineering](https://engineering.fb.com/2024/03/06/security/whatsapp-messenger-messaging-interoperability-eu/)). Meta also documents independent persistent secure connections and per-device fan-out in its multi-device architecture ([multi-device engineering post](https://engineering.fb.com/2021/07/14/security/whatsapp-multi-device/)).

**Illustrative HLD decomposition:**

```text
Sender app == persistent encrypted channel ==> messaging gateway
                                                    |
                                      durable message / routing service
                                                    |
Recipient online <== persistent encrypted channel ==+
Recipient offline <== push notification + later catch-up

Large encrypted media: app ---> object/media service ---> recipient
```

Protocol reasoning:

- Persistent bidirectional channel for low-latency messages and receipts.
- Durable server-side state/application ACKs for offline delivery; TCP alone is insufficient.
- HTTPS/object delivery for large media rather than forcing it through the chat control channel.
- Voice/video needs a real-time media stack, NAT traversal, and often relay/SFU infrastructure.

Interview follow-ups: per-conversation ordering, duplicates, offline queues, multiple devices, reconnect cursor, hot groups, encryption metadata, and backpressure.

### 8.2 YouTube

**Verified public observation:** Google documents DASH as HTTP-based and used for YouTube live ingestion; it also documents multi-resolution encodes for adaptive bitrate streaming ([YouTube DASH guide](https://developers.google.com/youtube/v3/live/guides/encoding-with-dash), [Google media guide](https://developers.google.com/media/vp9/settings/vod)).

**Illustrative viewer path:**

```text
Player --HTTPS--> manifest from playback service
Player --HTTPS--> video segment 1 (720p) -- CDN edge
Player --HTTPS--> video segment 2 (1080p) - CDN edge
bandwidth drops
Player --HTTPS--> video segment 3 (480p) -- CDN edge
```

Why HTTP works well: segments are cacheable immutable objects; CDNs understand HTTP; retries are bounded to segments; the player chooses quality based on buffer and estimated throughput.

Live chat is a different requirement and might use a streaming/persistent channel. Do not force the same protocol choice onto video bytes and chat events.

### 8.3 Netflix

**Verified public observation:** Netflix’s Open Connect overview says its appliances store encoded video/image files and serve them to clients over HTTP/HTTPS ([Open Connect overview](https://openconnect.netflix.com/Open-Connect-Overview.pdf)).

```text
Control plane: client --HTTPS--> playback/control services
                                 |
                                 +--> chooses encodes/nearby delivery options

Data plane:    client --HTTP(S) segment requests--> Open Connect appliance
```

Interview lesson: separate a lower-volume control plane—auth, catalog, playback authorization, manifests—from the enormous cache-friendly media data plane.

### 8.4 Uber

**Illustrative design; not a claim of a current private protocol:**

```text
Rider app --HTTPS--> request trip / payment / receipt
Driver app == persistent updates ==> regional real-time gateway
Driver location --> ingestion stream --> matching/ETA services
Trip-state events ==> rider and driver connections
```

- Trip creation/payment needs reliable authenticated requests and application idempotency.
- Location is frequently updated state; a newer location can make an older update irrelevant. The mobile-to-server protocol may still use reliable HTTPS/persistent transports for deployment/security reasons—the data semantics do not force raw UDP.
- Rider-visible trip transitions must be durable and ordered within the trip; transient map animation points may tolerate dropping/coalescing.

Interview lesson: choose reliability **per event type**, not once for the entire product.

### 8.5 Twitter/X

**Illustrative product design:**

```text
Create/read post ------ HTTPS API
Images/video ---------- HTTPS + CDN
Timeline refresh ------ HTTPS pagination / conditional requests
Low-latency updates --- SSE, streaming HTTP, or WebSocket where justified
Backend fan-out ------- internal queues/logs, not millions of direct DB polls
```

A user refreshing a home timeline does not require a permanent full-duplex channel. Live spaces, typing, or high-frequency event streams have different needs. “The product is real time” is too broad; specify acceptable freshness and direction.

### 8.6 Amazon / S3

**Verified public observation:** S3 supports range GETs, and AWS recommends concurrent requests/byte-range fetches for high throughput in suitable large-object workloads ([S3 download documentation](https://docs.aws.amazon.com/AmazonS3/latest/userguide/download-objects.html), [S3 performance guidelines](https://docs.aws.amazon.com/AmazonS3/latest/userguide/optimizing-performance-guidelines.html)).

```text
Download manager
   |-- HTTPS Range: bytes=0-... ------> S3/object endpoint
   |-- HTTPS Range: bytes=... --------> S3/object endpoint
   |-- HTTPS Range: bytes=... --------> S3/object endpoint
   +-- retries one failed part, validates, assembles
```

Interview lesson: large file reliability comes from resumable chunks, validators/checksums, retries, and durable progress—not from assuming one never-ending TCP connection will survive.

## 9. Trade-offs, alternatives, and when to avoid each choice

| Choice | Main strengths | Main costs | Avoid as default when… |
|---|---|---|---|
| HTTPS request/response | Universal, secure, cache/CDN/tool support, stateless scaling | Client initiation; polling can waste work | Updates are continuous, low-latency, and server-initiated |
| SSE | Simple one-way browser stream over HTTP, reconnection/event IDs | Server → client only, text-oriented API | Client must frequently push on the same channel or binary/full-duplex behavior is central |
| WebSocket | Full duplex, low per-message overhead, browser support | Stateful connections, reconnect/replay/backpressure complexity | Updates are rare or one-way; ordinary HTTP is enough |
| TCP | Mature reliable ordered stream, flow/congestion control | Head-of-line delay; app framing needed | Stale data should not block fresh data and custom/selective reliability is justified |
| UDP | Minimal datagram service, app controls timing/recovery | Must add safety, security, congestion/reliability as needed | Loss/reordering cannot be tolerated and no proven higher-level protocol handles them |
| QUIC/HTTP/3 | Secure multiplexed streams, improved loss isolation/mobility behavior | UDP reachability/fallback, implementation/observability maturity | Platform/network support is insufficient and HTTP/2 meets needs |
| FTP/FTPS | Legacy ecosystem and partner workflows | Multiple connections/firewall complexity; plain FTP insecure | Building a new public web transfer service |
| SFTP | Secure operational file access over SSH | Not browser-native; account/key management | Files are public/cacheable CDN objects |
| SMTP | Federated store-and-forward mail delivery | Spam/reputation/retry/bounce complexity | You only need in-product real-time messaging |
| IMAP | Rich server mailbox synchronization | Protocol and state-sync complexity | A simple application-owned JSON API is sufficient |
| POP3 | Simple retrieval model | Weak rich multi-device synchronization model | Server-side folders/read-state/multi-device sync matter |
| P2P | Peers contribute bandwidth/storage; decentralized scale | Discovery, NAT, trust, availability, legal/ops issues | Central control, privacy, mobile cost, predictable performance dominate |
| WebRTC | Purpose-built real-time media/data and NAT traversal | Complex negotiation, TURN/SFU cost, media operations | Payload is ordinary durable application data or a large static file |

## 10. Common mistakes and precise corrections

1. **“TCP is slow; UDP is fast.”**  
   Say what kind of latency/throughput matters and what loss semantics are acceptable.

2. **“UDP loses data.”**  
   The network can also lose IP packets carrying TCP; TCP detects and recovers. UDP itself does not recover.

3. **“HTTP versus TCP?”**  
   They occupy different layers. Ask HTTP/1.1 or HTTP/2 over TCP, HTTP/3 over QUIC, or a different application protocol.

4. **“WebSocket guarantees messages.”**  
   It gives a channel over reliable transport while connected. Durable storage, acknowledgements, dedupe, replay, and offline delivery are application concerns.

5. **“WhatsApp uses WebSocket because chat.”**  
   Product category does not prove protocol. Public Meta material describes a persistent connection and an XMPP-based protocol for interoperability, not WebSocket.

6. **“WebRTC is P2P UDP.”**  
   WebRTC includes signaling-adjacent setup, ICE/STUN/TURN, encrypted media, data channels, and can use relays/SFUs. It is not guaranteed to be direct.

7. **“P2P is a protocol.”**  
   It is a topology/architecture. Name discovery, transport, integrity, NAT traversal, and incentives.

8. **“SFTP is secure FTP.”**  
   SFTP is the SSH File Transfer Protocol. FTPS is FTP protected by TLS.

9. **“SMTP downloads email.”**  
   SMTP submits/relays; IMAP or POP3 provides mailbox access to clients.

10. **“MTA is an email protocol.”**  
    MTA is the component that relays mail, usually using SMTP.

11. **“TCP means exactly once.”**  
    A lost response after a committed operation still creates retry ambiguity. Use application idempotency and deduplication.

12. **“More download connections multiply Internet speed.”**  
    They may utilize available capacity better; they do not create extra link capacity and can hurt fairness/cost.

13. **“OSI has to map every protocol to exactly one layer.”**  
    Layering is a model. TLS and QUIC demonstrate why boundaries are useful but not perfectly rigid.

## 11. Interview angle

### 11.1 How protocol selection appears in HLD

It is usually a follow-up inside a larger design:

- “How does the client receive new chat messages?”
- “Why WebSocket rather than polling or SSE?”
- “How do you recover after a dropped connection?”
- “Why HTTP for video instead of sending an endless socket stream?”
- “What happens on a lossy mobile network?”
- “How do users behind NAT establish a call?”
- “Does TCP guarantee the payment happens once?”
- “How do you distribute a large binary to thousands of machines?”

The interviewer is testing whether you connect protocol properties to requirements and failure modes.

### 11.2 General answer spine: reconstruct the answer live

Use the cue **D-R-E-F-S**:

1. **D — Direction:** request/response, one-way push, or full duplex?
2. **R — Reliability semantics:** loss, order, duplicates; transport ACK versus business ACK.
3. **E — Experience target:** latency, freshness, throughput, buffering.
4. **F — Fit with environment:** browser, NAT/firewall, mobile, security, CDN/proxy support.
5. **S — Scale and survival:** connection count, backpressure, retry, reconnect, replay/resume, fallback.

Example:

> “Chat is bidirectional and we want sub-second delivery, so I’d start with a secure persistent channel such as WebSocket for a browser client. That only solves live transport. I would persist messages, assign IDs/sequence positions, use application acknowledgements and dedupe, and expose a catch-up API after reconnect. I would compare SSE if client-to-server traffic can remain ordinary HTTP, and I’d plan connection routing, slow consumers, keepalives, and fallback.”

### 11.3 Real, self-reported interview questions/experiences found online

These are **sourced from public candidate posts** and are not independently verified:

1. **Design live commenting for live streams.** A candidate reported an Apollo system-design round with up to millions of streams/users and discussed WebSocket managers, Redis, Kafka, Cassandra, history, moderation, and top comments ([LeetCode candidate post](https://leetcode.com/discuss/interview-experience/6624380/)).  
   What it tests: persistent connection routing, fan-out, hot streams, slight-delay tolerance, durable history, moderation pipeline, and backpressure.

2. **Design an AI chat system.** A candidate reported an Amazon L5 HLD round about their current AI chat system and mentioned real-time messaging with WebSocket, stateless backend, rate limiting, and vector storage ([LeetCode candidate post](https://leetcode.com/discuss/post/6769948/)).  
   What it tests: streaming response direction, reconnect behavior, conversation persistence, model work isolation, and rate/cost controls.

3. **Networking protocols used in a project.** A candidate for an OCI/cloud-networking role reported being asked which protocols they knew and to explain WebSocket use in their project ([LeetCode candidate post](https://leetcode.com/discuss/interview-experience/5266926/)).  
   What it tests: whether you can explain a real choice, its failure behavior, and alternatives rather than list acronyms.

4. **Distribute a 2 GB binary to roughly 1,500 servers.** A candidate described a system-design interview with limited network capacity and no budget for more network devices ([LeetCode candidate post](https://leetcode.com/discuss/interview-question/system-design/193953/distribute-binary-file-daily-for-thousands-of-servers)).

### 11.4 Plausible synthesized follow-ups

The following are **my synthesis**, not claimed real company questions:

- A million WebSocket clients are connected across 200 gateways. How does a message find the correct gateway?
- A client reconnects after 90 seconds offline. How does it detect and recover missed events without duplicating effects?
- A payment request times out over TCP. When may the client retry, and how do you prevent a double charge?
- A group call grows from 4 to 100 participants. Why does mesh fail, and how would an SFU change bandwidth?
- UDP is blocked on a corporate network. What fallback does the call or HTTP/3 client use?
- A 20 GB download is 90% complete when the server replaces the file at the same URL. How do ETag/If-Range and checksums prevent corruption?
- A download manager uses 32 connections and harms other traffic. How would you adapt concurrency and enforce fairness?
- Why is SSE better than WebSocket for a one-way notification feed? When would that conclusion change?
- How does SMTP store-and-forward differ from a live WebSocket message?
- Why can TCP head-of-line blocking affect HTTP/2 streams, and what changes in HTTP/3?

### 11.5 Answer spines for common follow-ups

#### “TCP or UDP?”

**Cues:** data meaning → loss tolerance → ordering → latency/freshness → recovery → congestion/security → network support.

> “I would not choose from the acronym first. For payments/files, every byte matters and must be ordered, so I want reliable transport and application idempotency. For live media, freshness matters more than recovering every stale packet, so I want a media protocol with selective reliability, congestion control, and jitter handling, commonly over UDP with fallbacks. QUIC is a reminder that UDP underneath does not imply unreliable application delivery.”

#### “Why WebSocket?”

**Cues:** full duplex → frequency/latency → compare SSE/polling → durable layer → reconnect → scale.

#### “How does resume work?”

**Cues:** stable file identity/validator → completed byte map → Range/206 → retry part → checksum → atomic assembly; restart if representation changed.

#### “How would you distribute a large binary?”

**Cues:** immutable version + manifest/hash → object/chunk store → rack-aware staged tree or P2P → bounded concurrency → resume/checksum → rollout/rollback → monitor network saturation.

#### “Does reliable transport give exactly once?”

**Cues:** lost response ambiguity → operation ID/idempotency key → durable dedupe result → safe retry → audit/reconciliation.

## 12. Compact review sheet

### Category map

```text
Application protocols: HTTP, WebSocket, FTP/FTPS/SFTP, SMTP, IMAP, POP3
Transport protocols:   TCP, UDP, QUIC (implemented over UDP)
Network protocol:      IP
Security layer:        TLS (used by HTTPS and other secured application protocols)
Architecture/topology: client-server, P2P
Real-time suite/API:   WebRTC
Software role:         MTA
```

### One-line selection rules—with caveats

- Start with **HTTPS** for interoperable request/response APIs and object delivery.
- Add **SSE** for simple one-way browser push; use **WebSocket** for frequent full-duplex browser events.
- Use **WebRTC** for interactive voice/video; plan signaling, ICE, TURN, and SFU scaling.
- Use **SFTP** for secure operational/B2B file exchange; **HTTPS + CDN/ranges** for public application downloads.
- Use **SMTP** to submit/relay mail; **IMAP** for rich synchronized mailbox access; **POP3** for a simpler retrieval model.
- Consider **P2P** when peers can safely contribute resources and its discovery/trust/NAT costs are justified.
- Choose **reliability semantics** from the data, then choose TCP/QUIC/datagram behavior; do not use “TCP slow, UDP fast.”

## 13. Say-out-loud 30-second narrative

> “A network protocol is a contract for one part of communication, and layered models keep those responsibilities separate. In practice I map application needs—request/response, one-way push, full duplex, file transfer, email, or real-time media—onto HTTP, SSE/WebSocket, HTTPS ranges or SFTP, SMTP/IMAP, or WebRTC, then understand their transport over TCP, UDP, or QUIC. I choose based on direction, loss and ordering semantics, latency versus completeness, browser/NAT/security constraints, and operational scale. Reliable transport still needs application idempotency, acknowledgements, reconnect and replay. For large downloads, byte ranges and adaptive parallel segments improve utilization and resume behavior, but they do not create bandwidth.”

## 14. True prerequisite gaps to study next

This chapter gives enough foundation to discuss protocol choices, but these underlying topics deserve dedicated study next because they support nearly every HLD network discussion:

1. **DNS end to end:** record types, recursive resolution, caching/TTL, load balancing, failure behavior, and email MX records.
2. **IP routing and subnet/NAT basics:** public/private addresses, routing tables, load balancers, firewalls, NAT traversal, and IPv4 versus IPv6.
3. **TLS and certificates:** handshake, certificate chain/hostname validation, termination at load balancers, mTLS, and what end-to-end encryption means.
4. **TCP deeper mechanics:** congestion window, slow start, retransmission timeout, flow control, keepalive, and connection pooling.

Related but **not required to understand this chapter**: REST versus RPC/gRPC, message queues/pub-sub, CDNs, API gateways/load balancers, and backpressure.
