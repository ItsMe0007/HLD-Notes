# Hashing and Consistent Hashing

> Beginner-first HLD study notes on using hashes to route keys across changing groups of servers. The main goal is to understand exactly why modulo hashing causes widespread remapping, what consistent hashing guarantees, and what extra machinery a production system still needs.

## Scope, assumptions, and evidence labels

The supplied rough bullets were:

- **Hashing**
- why consistent hashing is needed;
- how much rehashing is required with different approaches, such as modulo hashing and "counting hash."

### Terminology correction: "counting hash"

**"Counting hash" is not a standard name for a server-partitioning algorithm.** A *Counting Bloom filter*, *Count-Min Sketch*, and *Count Sketch* do exist, but they solve membership/frequency-estimation problems and are unrelated to moving keys when servers change.

Given the topic and the [published description of Shrayansh Jain's consistent-hashing lecture](https://www.linkedin.com/posts/jainshrayansh_6-consistent-hashing-in-hindi-with-example-activity-6972208518288273408-y64c), this note assumes **"counting hash" meant "consistent hashing."** Do not use "counting hash" for this concept in an interview.

## 1. Plain-language foundation

### 1.1 The problem before the terminology

Imagine a library with three storage rooms. Every book must always be sent to the same room; otherwise the librarian will look in the wrong place.

One rule could be:

```text
turn the book ID into a number
divide it by the number of rooms
use the remainder as the room number
```

With three rooms:

```text
room = book_number % 3
```

This is simple and spreads books reasonably well. The trouble appears when a fourth room opens. The divisor changes from `3` to `4`, so the answer changes for most books. The new room has not merely taken one quarter of the books; most existing books now appear to belong somewhere else.

At library scale this means moving books. In a distributed cache it means widespread cache misses. In a database it may mean copying a huge amount of durable data while reads and writes are still arriving.

**Consistent hashing changes the assignment rule so that adding or removing one room disturbs only the room's approximate share, not almost the whole library.**

### 1.2 What is a hash function?

A **hash function** deterministically turns an input of arbitrary size into a value in a fixed output space.

```text
hash("user:42")  ----------------->  73,491,208
      key                               hash value
```

For the same bytes and the same hash algorithm/version, the output must be the same. A tiny input change should normally produce an unrelated-looking output:

```text
hash("user:42") -> 73,491,208
hash("user:43") ->  9,107,644
```

The exact numbers above are illustrative.

For partitioning, the important properties are:

| Property | Plain meaning | Why it matters |
|---|---|---|
| Deterministic | The same input gives the same output | Every router agrees where a key belongs |
| Approximately uniform | Outputs spread across the hash space | Nodes receive similar numbers of ordinary keys |
| Stable | Algorithm, seed, and key encoding do not change unexpectedly | A software deploy does not remap the entire cluster |
| Fast | It is inexpensive to calculate | Every request may require a hash lookup |
| Good diffusion | Patterns in input do not remain obvious patterns in output | Sequential IDs do not all concentrate on one node |

A cryptographic hash such as SHA-256 is designed for stronger security properties, including resistance to finding collisions or reversing the input. Partition routing often uses a faster non-cryptographic hash because it usually needs stable uniform distribution, not password security. If inputs are attacker-controlled, denial-of-service and collision risks still need deliberate treatment.

> **Hashing is not encryption.** Encryption is reversible with a key; ordinary hashing is designed as a one-way mapping. Hashing also does not compress data into a recoverable smaller form.

### 1.3 Inline prerequisite: key, node, partition, shard, and routing

These terms are required for the rest of the chapter:

- A **key** identifies the unit being assigned, such as `user_id`, `video_id`, cache key, tenant ID, or connection ID.
- A **node** is a physical or virtual server/process that can own work or data.
- A **partition** is one logical subset of the full keyspace.
- A **shard** commonly means a partition placed on a database/storage node. Teams sometimes use *shard* and *partition* interchangeably, so define the term in an interview.
- **Partitioning** decides which subset owns a key.
- **Routing** gets a request to the owner.
- **Replication** keeps extra copies. It is separate from partitioning.

```text
                  partitioning rule
key = user:42 --------------------------> partition P7
                                              |
                     placement map says P7 is on nodes B, D, F
                                              |
                                              v
                                      route request to B
```

Consistent hashing can help with the first two arrows. It does not by itself send the request, store the value, or create replicas.

### 1.4 Hash collisions are not the same as server remapping

A **collision** occurs when two different inputs produce the same finite hash value.

```text
hash(key A) = 123
hash(key B) = 123
```

Collisions are mathematically unavoidable when an unlimited input space is mapped into a finite output space. A storage node therefore still retains and compares the full original key. The hash selects a location; it must not be treated as proof that two original keys are equal.

This is different from **remapping**, where a key's hash value stays the same but the node-ownership rule changes.

### 1.5 "Rehashing" versus "remapping" versus "moving"

People often say "rehash the keys" when a cluster changes. Three operations can be hidden inside that phrase:

1. **Recompute a hash value**: run the hash function again.
2. **Remap ownership**: the same hash value now resolves to a different node.
3. **Move data/state**: copy the actual value to its new owner and eventually remove the old copy.

With consistent hashing, existing key hash values normally do **not** change. What is minimized is **ownership remapping**. A durable store must then perform safe data movement for the affected keys. This note uses *remap* when that is the precise meaning.

## 2. Why hashing is needed in system design

### 2.1 One machine is finite

Suppose one cache server can hold only part of the working set. We add more cache servers and need an answer to this question on every request:

> Given key `K` and the currently available nodes, which node should handle `K`?

A useful assignment should provide:

- **determinism**: all routers choose the same owner;
- **distribution**: ordinary keys do not pile onto one node;
- **affinity**: repeated operations for a key reach the node holding its state;
- **incremental scalability**: adding capacity does not disturb unrelated keys;
- **failure containment**: losing one node affects only a bounded share; and
- **capacity awareness**: stronger machines can receive more work if required.

Hash-based partitioning is attractive because a router can calculate the destination from a key instead of searching every node.

### 2.2 Where the pattern appears

```text
Distributed cache             Sharded database             Stateful worker fleet

cache key                     user_id                      driver_id
   |                              |                            |
 hash                            hash                         hash
   |                              |                            |
cache node                    data shard                   owning worker
```

Examples include:

- distributing cache entries across memory servers;
- assigning database rows or objects to shards;
- routing the same user to a stateful WebSocket or session gateway;
- assigning jobs to workers while preserving per-key order;
- choosing an origin or edge cache for an object; and
- selecting a replica set in a distributed key-value store.

Hashing is not automatically the right choice. Range queries, transactions spanning many keys, locality requirements, and extremely hot individual keys may favor a different partitioning scheme or require additional mechanisms.

## 3. The simple approach: modulo hashing

### 3.1 The rule

For `N` nodes numbered `0` through `N - 1`:

```text
node_index = stable_hash(key) % N
```

With stable membership this is excellent in its simplicity:

- lookup is constant-time;
- no ring or routing table is needed beyond the node list;
- a uniform hash gives an approximately even key count; and
- every router can independently calculate the same answer.

The divisor `N`, however, is embedded in the assignment. Change `N`, and the mapping changes globally.

### 3.2 Worked example: three nodes become four

Use hash values `0` through `11` as a small, perfectly uniform example.

| Hash `h` | Old owner `h % 3` | New owner `h % 4` | Moved? |
|---:|---:|---:|---|
| 0 | 0 | 0 | No |
| 1 | 1 | 1 | No |
| 2 | 2 | 2 | No |
| 3 | 0 | 3 | Yes |
| 4 | 1 | 0 | Yes |
| 5 | 2 | 1 | Yes |
| 6 | 0 | 2 | Yes |
| 7 | 1 | 3 | Yes |
| 8 | 2 | 0 | Yes |
| 9 | 0 | 1 | Yes |
| 10 | 1 | 2 | Yes |
| 11 | 2 | 3 | Yes |

Only 3 of 12 keys remain on the same numbered node. **Nine of 12, or 75%, remap.**

Notice that the new node finally owns only 25% of the keys, yet 75% moved. Many keys move between old nodes unnecessarily because the modulus changed.

### 3.3 Exact expected remapping for an adjacent node-count change

Let:

- `K` = number of keys;
- `N` = number of nodes before the change;
- hash outputs be uniform over a sufficiently large space; and
- surviving node labels keep their numbers.

#### Add one node: `N -> N + 1`

For a key to remain on the same numbered node:

```text
h % N = h % (N + 1)
```

Across a full repeating period of `N(N + 1)` consecutive hash values, only `N` values satisfy that equality. Therefore:

```text
fraction unchanged = 1 / (N + 1)
fraction moved     = N / (N + 1)

expected keys moved = K * N / (N + 1)
```

#### Remove one node: `N -> N - 1`

Similarly:

```text
fraction unchanged = 1 / N
fraction moved     = (N - 1) / N

expected keys moved = K * (N - 1) / N
```

### 3.4 Modulo-remapping table

| Change | Fraction moved with modulo hashing | Example |
|---|---:|---:|
| `3 -> 4` | `3/4 = 75%` | 750,000 of 1M keys |
| `4 -> 3` | `3/4 = 75%` | 750,000 of 1M keys |
| `10 -> 11` | `10/11 ~= 90.91%` | about 909,091 of 1M keys |
| `100 -> 101` | `100/101 ~= 99.01%` | about 990.1M of 1B keys |
| `100 -> 99` | `99/100 = 99%` | 990M of 1B keys |

The formula is about **keys**, not necessarily bytes, requests, or migration duration. If one remapped key is a 50 GB object or receives half the traffic, key count understates the operational impact.

### 3.5 When modulo hashing is still the right answer

Use it when:

- membership is fixed or changes extremely rarely;
- the data can be discarded and rebuilt cheaply;
- a separate fixed set of logical buckets hides physical-node changes;
- a coordinated full reshuffle is acceptable; or
- simplicity matters more than incremental resizing.

Modulo hashing is not "bad hashing." It is a poor direct mapping from keys to a **frequently changing set of physical nodes**.

## 4. Consistent hashing from first principles

### 4.1 Decouple the hash space from the number of nodes

Modulo hashing uses a different destination space whenever `N` changes. Consistent hashing instead uses one fixed hash space, such as:

```text
0 ------------------------------------------ 2^64 - 1
```

The endpoints are treated as connected, making a logical ring:

```text
                         0
                    .----------.
                90 /              \ 10
                  |                |
                70 \              / 30
                    '----------'
                         50
```

The ring is an abstraction. The machines are not physically arranged in a circle.

### 4.2 Put both nodes and keys in the same fixed space

1. Hash each node identifier to a ring position.
2. Hash each data key to a ring position.
3. Starting at the key, walk clockwise to the first node position.
4. That node owns the key. If no node is found before the maximum value, wrap to the first node.

Example:

```text
Ring positions: 0 to 99

Node A = 20
Node B = 50
Node C = 80

                 A(20)
              .----*----.
          10 /           \ 30       key x at 35 -> B
            |             |          key y at 70 -> C
         90 |             | 50 B     key z at 95 -> wrap -> A
            \             /
          C(80)*---------'
```

Using "first node clockwise," ownership intervals are:

```text
A owns (80, 20]  -- including the wraparound
B owns (20, 50]
C owns (50, 80]
```

The clockwise convention is arbitrary. Counter-clockwise also works if every participant uses it consistently.

### 4.3 Adding a node changes one local interval

Add node `D` at position `65`:

```text
Before

A(20) -------- B(50) ---------------- C(80) -------- wrap to A
                 <------ C owns ------>

After

A(20) -------- B(50) ---- D(65) ----- C(80) -------- wrap to A
                 <- D owns -><- C owns ->
```

Only keys in `(50, 65]` change owner, from `C` to `D`. Keys owned by `A` and `B`, and the remaining keys on `C`, keep their owners.

### 4.4 Removing a node changes one local interval

Remove `B`:

```text
Before:  A(20) -------- B(50) -------- D(65) -------- C(80)
                          B owns (20,50]

After:   A(20) ----------------------- D(65) -------- C(80)
                          D takes (20,50]
```

Only the removed node's interval changes ownership. This is a placement rule; if `B` failed before copying durable data, the values are unavailable unless replicas exist.

### 4.5 How much remaps with consistent hashing?

For `K` uniformly hashed keys and equal-capacity nodes:

#### Add one node: `N -> N + 1`

The new node should finish with about `1/(N + 1)` of the keyspace, so approximately that share moves to it:

```text
expected fraction moved = 1 / (N + 1)
expected keys moved     = K / (N + 1)
```

#### Remove one of `N` nodes

The removed node owned about `1/N` of the keyspace:

```text
expected fraction moved = 1 / N
expected keys moved     = K / N
```

### 4.6 Side-by-side: the central answer

| Membership change | Modulo hashing | Consistent hashing, ideal expectation |
|---|---:|---:|
| Add: `N -> N + 1` | `K * N/(N + 1)` keys move | `K/(N + 1)` keys move |
| Remove: `N -> N - 1` | `K * (N - 1)/N` keys move | `K/N` keys move |
| `100 -> 101`, `K = 1B` | about **990.1M** move | about **9.9M** move |
| `100 -> 99`, `K = 1B` | **990M** move | about **10M** move |

At 100 nodes, adding one node changes roughly 99% of assignments with direct modulo hashing versus roughly 1% with ideal consistent hashing.

These are expected values, not a promise that every physical node holds exactly the same number of keys. The next section explains why.

The table measures **primary ownership changes**. With replication, physical copy work can differ: a promoted successor may already hold a replica, but the system may then need to create another copy elsewhere to restore the replication factor.

### 4.7 Why this movement is close to the minimum possible

After adding an equally weighted node, that node must receive roughly `1/(N + 1)` of the final load from somewhere. An assignment scheme cannot give it that share while moving less than that share of existing keys. Consistent hashing is **monotonic** in this sense: keys move to the new node rather than being needlessly shuffled among old nodes.

Likewise, when a node is removed, every key it owned must go somewhere else; keys owned by surviving nodes need not move.

### 4.8 What does "consistent" mean here?

It does **not** mean database consistency, strong consistency, or "the key always stays on the same server." It means the mapping remains as stable as possible when the set of destinations changes.

The original consistent-hashing literature discusses properties commonly summarized as:

- **Balance**: keys should be spread roughly evenly across the buckets/nodes.
- **Monotonicity**: when a node is added, keys that move should move to the new node rather than shuffle among old nodes.
- **Spread**: if participants temporarily have slightly different membership views, one key should map to only a limited number of possible destinations across those views.
- **Load**: disagreement among views should not make one destination receive an unbounded number of different keys.

The simple interview ring mainly demonstrates balance and monotonicity. Spread/load depend on the full algorithm, number of replicas/virtual points, and membership behavior; a hand-drawn ring alone is not a proof of safe operation under inconsistent views.

## 5. Why the basic ring is not enough: virtual nodes

### 5.1 Random node positions create uneven ownership

Hashing four physical nodes once does not place them at perfect quarter points:

```text
Bad-luck placement

       A B
       **
     /    \
    |      |                         D owns a very large arc
    |      * C
     \    /
       * D
```

With few random points:

- one physical node may own a much larger arc than another;
- one node failure can transfer its entire interval to one successor;
- adding a node can take an unpredictable amount from one neighbor; and
- different hardware capacities are not represented naturally.

The expected movement formula remains useful across random placements, but variance can be operationally unacceptable.

### 5.2 A virtual node is a ring position, not another machine

Give each physical node many positions by hashing identifiers such as:

```text
hash("node-A#0") -> A virtual node
hash("node-A#1") -> A virtual node
hash("node-A#2") -> A virtual node
...
```

```text
Physical node A owns: A1, A2, A3, A4, ...
Physical node B owns: B1, B2, B3, B4, ...

Ring:  A1 -- B3 -- A4 -- B1 -- A2 -- B4 -- A3 -- B2 -- wrap
```

Each physical node now owns many small, scattered intervals. Statistical averaging makes total ownership smoother.

The 2007 Amazon Dynamo paper describes this approach: basic random placement caused non-uniform load and ignored hardware differences, so Dynamo assigned each physical node multiple ring positions, called **virtual nodes** ([Amazon Dynamo paper](https://assets.amazon.science/ac/1d/eb50c4064c538c8ac440ce6a1d91/dynamo-amazons-highly-available-key-value-store.pdf)).

### 5.3 What virtual nodes improve

| Problem with one point per node | What virtual nodes change |
|---|---|
| Uneven keyspace | Many small samples smooth each physical node's total share |
| One failure overloads one successor | Failed intervals can transfer to several surviving physical nodes |
| New node takes data from one neighbor | It can receive small intervals from many nodes |
| Different machine capacities | Give a larger-capacity node more virtual positions or weight |
| Coarse movement | Small intervals provide finer migration units |

### 5.4 What virtual nodes cost

- more ring entries and routing metadata;
- more ownership intervals to track, stream, repair, and monitor;
- more complex replica selection because adjacent virtual positions may belong to the same physical node;
- potentially more small files/partitions or operational work, depending on storage design; and
- a tuning decision: too few means imbalance, too many means metadata and movement overhead.

There is no universal correct number such as "always use 100 or 256 virtual nodes." Choose it from measured balance, cluster size, failure-domain rules, capacity heterogeneity, bootstrap time, and routing-metadata cost. Any fixed recommendation without those conditions is **unverified and should not be memorized**.

### 5.5 Virtual nodes distribute keys, not necessarily work

Suppose one billion cold keys are evenly distributed, but `video:world-cup-final` receives half of all reads. One key still maps to one primary interval.

```text
Even key count                         Uneven traffic

A: 25% of keys                         A: 10% requests
B: 25% of keys                         B: 15% requests
C: 25% of keys                         C: 65% requests  <-- hot key
D: 25% of keys                         D: 10% requests
```

Consistent hashing plus virtual nodes does not split an individual hot key. Possible remedies include replication, CDN caching, request coalescing, salting a splittable key, adaptive routing, or treating that workload specially.

## 6. From ring diagram to a production mechanism

### 6.1 Choose the partition key from access patterns

The algorithm cannot rescue a bad key.

| Candidate key | Benefit | Risk |
|---|---|---|
| `user_id` | Keeps one user's state together | Celebrity or large-tenant skew; cross-user operations scatter |
| `video_id` | Direct video lookup; ordinary videos distribute | Viral video becomes hot |
| `city_id` | Geographical locality and city-scoped operations | A few cities dominate load; capacity changes by time zone |
| `tenant_id` | Tenant isolation and easy tenant lookup | One enterprise tenant may exceed a node |
| Random object ID | Usually good distribution | Range scans and related-object queries scatter |

Ask:

- Which operations must be co-located or ordered?
- Which operations may span nodes?
- Can one key become much hotter or larger than average?
- Must a tenant or geographic region remain in a failure/compliance boundary?
- Can the key be split later without changing product semantics?

### 6.2 Use a stable, explicitly versioned hash contract

Every participant must agree on:

- exact key bytes and character encoding;
- hash algorithm and seed;
- signed/unsigned integer interpretation;
- ring width and wrap behavior;
- node identifier format;
- clockwise/counter-clockwise rule;
- virtual-node construction; and
- tie-breaking if two positions collide.

Do not use a language runtime's default `hash()` unless its stability across processes, versions, and deployments is explicitly guaranteed. Some runtimes randomize hashes for security; different languages may serialize the same logical key differently.

Changing the hash function or key encoding can remap nearly everything. Treat that change like a full data migration with old/new versions, not a harmless code refactor.

### 6.3 Ring lookup data structure

Store the virtual-node positions in sorted order:

```text
positions = [7, 19, 26, 44, 63, 81, 95]
owners    = [A,  B,  C,  A,  C,  B,  A]
```

For key hash `h`:

1. binary-search for the first position `>= h`;
2. return that position's physical owner; and
3. if no such position exists, wrap to index `0`.

If there are `V` total virtual positions:

| Operation | Typical cost with sorted structure |
|---|---:|
| Key lookup | `O(log V)` |
| Memory | `O(V)` |
| Add/remove positions | structure-dependent; often `O(v log V)` for `v` positions plus publishing a new map |

Request routing may cache the result, use a tree, or precompute a lookup table. Do not claim the ring itself performs network routing.

### 6.4 Minimal implementation pseudocode

This is enough to explain the core data structure in an interview; production code also needs concurrency control, immutable/versioned snapshots, persistence, metrics, and migration coordination.

```text
state:
    ring = ordered_map<position, physical_node_id>
    positions_by_node = map<physical_node_id, list<position>>

add_node(node_id, vnode_count):
    positions = []
    for replica_index from 0 to vnode_count - 1:
        salt = 0
        do:
            position = stable_hash("node:" + node_id +
                                   "#" + replica_index +
                                   "#" + salt)
            salt = salt + 1
        while ring already contains position

        ring[position] = node_id
        positions.append(position)

    positions_by_node[node_id] = positions

remove_node(node_id):
    for position in positions_by_node[node_id]:
        ring.remove(position)
    positions_by_node.remove(node_id)

get_owner(key):
    if ring is empty:
        return error("no available nodes")

    key_position = stable_hash("key:" + canonical_bytes(key))
    entry = ring.first_entry_at_or_after(key_position)

    if entry does not exist:
        entry = ring.first_entry()       // wrap around

    return entry.physical_node_id
```

Important implementation observations:

- Store a node's generated positions so removal does not scan the entire ring.
- Domain prefixes such as `node:` and `key:` make the hashed inputs explicit; they are not a substitute for collision handling.
- Publish a new immutable ring version only after the membership decision is authorized. Do not let every request mutate shared routing state.
- A lookup returning an owner is not enough for replicated data; walk forward and select distinct eligible physical nodes/failure domains.
- Adding entries to the data structure is not the same as cutting traffic over to the new data owner. The migration protocol controls that moment.

### 6.5 Who owns the ring metadata?

Three common models are:

```text
1. Client-side routing

Client with ring map -----------------> correct storage node

2. Central router/proxy

Client ---> router with ring map -----> correct storage node

3. Any-peer routing

Client ---> any node --forward-------> owner node
```

| Model | Advantages | Costs/failure concerns |
|---|---|---|
| Client-side | No extra proxy hop; direct | Every client needs compatible map/version; stale clients misroute |
| Router/proxy | Clients stay simple; policy centralized | Extra hop; router fleet and metadata service must scale |
| Any peer + forwarding | Simple external client; decentralized entry | Extra internal hop; peers need convergent membership; forwarding loops need protection |

A control plane, configuration service, service discovery system, or gossip protocol must distribute membership. Consistent hashing does not discover nodes or establish that a failed node is truly dead.

### 6.6 Planned addition: safe database rebalancing

The ring tells us **which keys should move**. It does not make movement atomic. A simplified planned workflow is:

```text
old owner                    new owner                    router/control plane
    |                            |                                  |
    |---- snapshot/copy range -->|                                  |
    |---- stream new changes --->|                                  |
    |                            |---- verify caught up ------------>|
    |                            |                                  |
    |<----------- publish new ring version / switch routing --------|
    |                            |                                  |
    |---- forward or reject stale-version requests ---------------->|
    |                            |                                  |
    |---- drain grace period ----|                                  |
    |---- delete old copy after verification                        |
```

Production choices include:

- snapshot plus change-log catch-up;
- temporary dual writes;
- old-owner forwarding;
- versioned routing epochs and redirects;
- checksums/count comparison before cutover;
- bandwidth and I/O throttling; and
- rollback while the old copy remains intact.

Dual writes alone are not magically atomic: one write can succeed while the other fails. A durable log, retry/idempotency, or database-supported migration protocol is needed according to the consistency requirement.

### 6.7 Cache rebalancing can be deliberately simpler

For a cache whose source of truth is elsewhere, moved entries may be allowed to become cold misses:

```text
request -> new owner -> miss -> database/source -> fill new cache owner
```

This avoids copying cache values, but a membership change can create a burst of misses. Even the smaller consistent-hashing fraction may overload the source database if the affected keys are hot. Warm the new node gradually, coalesce duplicate fills, rate-limit misses, or serve temporarily from the old owner.

Netflix's EVCache documentation explicitly notes that adding a sharded cache instance changes consistent-hash results, making some entries unreachable until they expire or are stored on the new owner; it also describes verification before rehashing to minimize movement ([Netflix EVCache features](https://netflix.github.io/EVCache/features/)).

### 6.8 Unplanned failure is not the same as planned removal

If a cache node disappears, losing its entries may be acceptable. If a database node disappears, "the next node owns the range now" does not recover bytes that existed only on the failed disk.

Durable designs add replication:

```text
key K hashes here
       |
       v
    primary B ---- next distinct node C ---- next distinct node D
        copy 1             copy 2                    copy 3
```

Replica selection must skip virtual positions belonging to the same physical node and should respect racks, zones, and regions. Otherwise three logical replicas may fail together.

Replication then introduces separate questions:

- Which replica accepts writes?
- What read/write consistency is promised?
- How are divergent versions repaired?
- What happens during a network partition?
- How is a recovering node brought up to date?

The Amazon Dynamo design combined consistent hashing with replication, preference lists, sloppy quorums, hinted handoff, object versioning, and repair mechanisms. The hashing ring was only one part of its availability design ([Amazon Dynamo paper](https://assets.amazon.science/ac/1d/eb50c4064c538c8ac440ce6a1d91/dynamo-amazons-highly-available-key-value-store.pdf)).

### 6.9 Ring disagreement and membership churn

During a membership change, routers may temporarily hold different ring versions:

```text
Router R1: version 41 says key -> A
Router R2: version 42 says key -> D
```

Possible safeguards:

- attach a ring/epoch version to requests;
- have old owners redirect or forward to the new owner;
- reject versions that are too old and force metadata refresh;
- publish membership changes through a strongly consistent control plane;
- use bounded transition periods; and
- damp failure flapping so a node does not repeatedly leave and rejoin the ring.

Gossip-based membership can be highly available and decentralized but is eventually consistent. Uber later reported that as its Ringpop-based push system grew, slower membership convergence could cause inconsistent topology views, message loss, timeouts, and errors; it replaced that generation with centralized topology management using ZooKeeper and Helix ([Uber Real-Time Push Platform](https://www.uber.com/us/en/blog/real-time-push-platform/)). This is a valuable example of consistent hashing's surrounding control-plane trade-off.

## 7. What consistent hashing guarantees—and what it does not

| Claim | Correct? | Precise explanation |
|---|---|---|
| It minimizes remapping when membership changes | Yes, under its assumptions | Roughly the joining/leaving node's share changes owners |
| It makes key hashes consistent across nodes | Only if configured correctly | All participants must share the exact stable hash contract |
| It evenly balances keys | Approximately, especially with enough vnodes | Randomness still has variance |
| It evenly balances traffic | No | A single hot key can dominate one owner |
| It moves data automatically | No | A migration/streaming protocol must act on the new ownership map |
| It detects failed nodes | No | Health checking, service discovery, gossip, or a control plane does that |
| It prevents data loss | No | Replication, durability, backup, and repair provide that |
| It guarantees zero downtime | No | Safe cutover, stale routing, capacity, and replication determine downtime |
| It solves consistency | No | Read/write/replica protocols determine consistency |
| It removes the need for a shard key | No | It requires a key and can amplify a poor choice |
| It makes cross-shard transactions easy | No | Those remain distributed operations |
| It is the same as a hash table | No | It is a distributed assignment strategy over changing membership |

The original 1997 Karger et al. work introduced consistent hashing for a changing population of web caches ([ACM paper record](https://doi.org/10.1145/258533.258660)). Amazon's Dynamo later used a ring-based variant to achieve incremental partition scaling ([Amazon Science publication page](https://www.amazon.science/publications/dynamo-amazons-highly-available-key-value-store)).

## 8. Alternatives and related strategies

### 8.1 Comparison table

| Strategy | Lookup idea | Movement when one equal node is added | Strength | Main cost/limit |
|---|---|---:|---|---|
| Direct modulo | `hash(key) % N` | `N/(N+1)` | Minimal metadata and very simple | Widespread remap when `N` changes |
| Ring consistent hashing | Next position clockwise | About `1/(N+1)` | Incremental change; weights/vnodes | Ring metadata, imbalance, control-plane complexity |
| Fixed logical hash slots | `hash(key) % S`, then `slot -> node` | Only slots explicitly reassigned | Precise, controllable movement | Slot map and migration orchestration |
| Rendezvous/HRW hashing | Score `(key,node)`; choose highest | About `1/(N+1)` | Simple membership model; no ring | Naive lookup scores every node: `O(N)` |
| Jump consistent hash | Deterministically jump among numbered buckets | About `1/(N+1)` for adding at end | `O(log N)` lookup, constant memory | Best fit for sequential bucket counts; arbitrary removal/weights need more design |
| Directory lookup | Look up `key/range -> owner` in metadata | Exactly what operator chooses | Flexible placement, locality, exceptions | Metadata availability, caching, and scale |
| Range partitioning | Ordered key ranges | Can split/move selected ranges | Efficient range scans and locality | Skew from ordered/popular ranges; range management |
| Round robin | Next request goes to next healthy node | No key ownership concept | Good for interchangeable stateless requests | No per-key affinity or data placement |

### 8.2 Fixed logical slots: separate key placement from machines

Instead of hashing directly to `N` physical nodes, hash into a fixed number `S` of logical slots:

```text
slot = hash(key) % 16,384

slot 0..5000       -> node A
slot 5001..10000   -> node B
slot 10001..16383  -> node C
```

When node D joins, explicitly transfer selected slots to it. Keys in other slots do not move.

Redis Cluster is an important correction to a common interview claim: current Redis Cluster documentation describes **16,384 fixed hash slots**, with nodes owning subsets of those slots; cluster reconfiguration moves slots between nodes. That is not the classic "hash both nodes and keys onto a ring" algorithm ([Redis Cluster specification](https://redis.io/docs/latest/operate/oss_and_stack/reference/cluster-spec/)).

Benefits over a purely random ring include precise balancing, easier inspection, and explicit placement. Costs include maintaining a slot map and deciding which slots to move.

### 8.3 Rendezvous hashing (highest-random-weight hashing)

For each key, calculate a stable score for every candidate node:

```text
score(key, A) = hash(key || A)
score(key, B) = hash(key || B)
score(key, C) = hash(key || C)

owner = node with highest score
```

When a node joins, it takes only keys for which it has the new highest score. When a node leaves, only its keys choose their next-highest node. This has the same attractive minimal-disruption property without a ring. Weighted variants exist.

The straightforward algorithm costs `O(N)` scores per lookup, which can be fine for small worker sets and expensive for very large ones. Optimized variants exist, but should be named only if their mechanics are understood.

### 8.4 Jump consistent hash

Jump consistent hash maps a key to one of a sequentially numbered set of buckets using constant memory and expected `O(log N)` time. The 2014 Lamping and Veach paper reports balance and minimal remapping for bucket-count growth ([Jump consistent hash paper](https://arxiv.org/abs/1406.2294)).

It is appealing when buckets are numbered `0..N-1` and are added at the end. Classic jump hashing is less convenient for arbitrary named-node removal, topology-aware placement, or weighting without another indirection layer.

### 8.5 Range partitioning

Hashing destroys natural ordering. If the system needs queries such as:

```text
all events from 2026-07-01 through 2026-07-16
all usernames from "sam" through "sat"
```

ordered ranges may be preferable. A range can split when large, but monotonically increasing timestamps can create a hot last range. Many databases combine strategies: hash a partition key for distribution and preserve ordering inside a partition for local range queries.

### 8.6 When consistent hashing is a poor fit

Avoid or reconsider it when:

- the service is stateless and any instance can serve any request—round robin or least-load is simpler;
- membership is fixed and a full rebuild is cheap—modulo may be enough;
- strict range locality is the dominant access pattern;
- every operation spans many keys or requires global transactions;
- a control plane already provides fixed partitions and explicit placement;
- load is dominated by a few indivisible hot keys; or
- a managed database/cache already owns partitioning—do not build a competing application ring without a concrete need.

## 9. Trade-offs, failure cases, and common mistakes

### 9.1 Trade-off summary

| Benefit | Corresponding cost/question |
|---|---|
| Much less remapping under node changes | More routing metadata and migration machinery |
| Deterministic per-key affinity | A hot key stays concentrated unless separately handled |
| Decentralized calculation is possible | All participants must agree on membership and hash contract |
| Virtual nodes smooth balance | More tokens/ranges and replica-placement complexity |
| Easy incremental capacity in theory | Migration itself consumes network, disk I/O, CPU, and source capacity |
| Weighted ownership can reflect capacity | Weight changes also trigger movement and need stable control |

### 9.2 Edge cases to be ready for

#### The cluster is already overloaded when a new node is added

The old nodes must serve live traffic **and** stream data to the new node. Adding capacity can initially increase load and cause a bootstrap death spiral.

Mitigations: reserve headroom, throttle migration, prioritize foreground work, add nodes in controlled batches, and verify that the network and source disks can sustain rebalancing.

#### A node repeatedly flaps

Rapid leave/join events repeatedly remap the same ranges and destroy cache warmth.

Mitigations: suspicion windows, health hysteresis, delayed eviction, minimum membership-change intervals, or temporarily route around the node without immediately changing durable ownership.

#### Routers use different ring versions

Writes can reach old and new owners.

Mitigations: epochs, redirects/forwarding, idempotent operations, old/new read fallback, and an authoritative membership publication process.

#### Two physical nodes share a failure domain

Clockwise replicas might sit in the same rack or availability zone.

Mitigation: build topology constraints into the preference/placement list; "next point" must mean next **eligible distinct physical/failure-domain node**.

#### One value is far larger than the others

Uniform key counts do not imply uniform bytes. Track keys, bytes, request rate, CPU cost, and storage I/O separately. Split or specially place unusually large values.

#### One tenant or celebrity is hot

Virtual nodes cannot split an indivisible partition key. Replicate reads, salt only if the access pattern can fan out/aggregate, cache at multiple layers, rate-limit, or isolate the tenant.

#### The hash function changes during a deploy

Nearly every key can remap even though membership did not change. Version the hash scheme and perform an intentional migration.

#### A node-position collision occurs

Even a wide hash space is finite. Specify a deterministic tie-breaker, re-salt the virtual-node identifier, or store an ordered tuple such as `(hash, node_id)`.

#### Multi-key operations land on different shards

Batch reads require fan-out; transactions may need coordination or a redesigned aggregate boundary. Consistent hashing does not preserve relationships unless related records share the chosen partition key.

#### A key is attacker-controlled

An adversary may target a known hot owner or exploit a weak hash. Consider a keyed/secret seed, admission control, rate limits, and a hash appropriate to the threat model. All legitimate routers must still share the seed securely.

### 9.3 Common interview mistakes and corrections

| Mistake | Better statement |
|---|---|
| "Consistent hashing means no keys move" | Only roughly the joining/leaving node's share remaps |
| "Rehashing falls from 100% to 1%" | State `N`; modulo add moves `N/(N+1)`, consistent add moves about `1/(N+1)` |
| "The ring balances all load" | It approximately balances hashed units; hot keys and unequal values remain |
| "Virtual nodes are replica servers" | A vnode is a logical ring position owned by a physical node |
| "The next three vnodes are three replicas" | They may belong to one machine; choose distinct physical nodes/failure domains |
| "If a node dies, its data moves clockwise" | Ownership changes; the bytes survive only if replicated or recoverable |
| "Consistent hashing provides high availability" | It reduces remapping; replication, failure detection, capacity, and routing provide availability |
| "Use Redis because it uses consistent hashing" | Redis Cluster uses fixed hash slots according to its current specification |
| "Hash the request" | Name the stable partition key and justify it from access patterns |
| "Use the default language hash" | Use a stable, cross-process, explicitly versioned hash contract |
| "Add a node with zero downtime" | Explain copy/catch-up/cutover/versioning or cache-miss behavior |
| "More virtual nodes are always better" | More smooth balance but increase metadata and operational work |

## 10. Real-system examples

### 10.1 Amazon Dynamo: verified historical production design

Amazon's 2007 Dynamo paper describes a highly available key-value store used by some Amazon core services. It used consistent hashing for incremental partitioning and virtual nodes for more even load, capacity heterogeneity, and spreading the work of node changes. It also replicated each key to multiple distinct hosts and used additional failure/reconciliation mechanisms ([Amazon Science](https://www.amazon.science/publications/dynamo-amazons-highly-available-key-value-store); [paper PDF](https://assets.amazon.science/ac/1d/eb50c4064c538c8ac440ce6a1d91/dynamo-amazons-highly-available-key-value-store.pdf)).

```text
Amazon application
       |
       v
coordinator hashes shopping-cart key
       |
       v
primary ring owner ---> replica 2 ---> replica 3
       |                    |              |
   failure/versioning/quorum/repair mechanisms around the copies
```

Interview lesson: saying "Amazon used consistent hashing" is incomplete. The useful answer separates **partition placement** from **replication and conflict handling**.

The public paper is a **2007 production snapshot**, not a claim that Amazon retail or DynamoDB uses that exact architecture today.

### 10.2 Netflix EVCache: consistent hashing inside a replicated cache

Netflix's EVCache documentation describes a distributed cache built from memcached. Within a zone, data is sharded with the Ketama consistent-hashing algorithm; data is mirrored across zones. The documentation also calls instance addition disruptive because some entries become unreachable under the changed mapping, and describes operational techniques for deployments and resizing ([EVCache features](https://netflix.github.io/EVCache/features/)).

```text
Netflix service
      |
 EVCache client
      |
      +-- zone A: consistent hash(key) -> one memcached shard
      +-- zone B: consistent hash(key) -> mirrored cache copy
      +-- zone C: consistent hash(key) -> mirrored cache copy
```

Interview lesson: even a cache needs a failure/cold-start plan. Minimal remapping is not zero remapping, and the origin may see a miss spike.

This is a **public project/documentation snapshot**; do not turn its dated implementation details or figures into universal Netflix claims.

### 10.3 Uber Ringpop: benefit and eventual limitation

Uber's 2016 Ringpop article described a library combining SWIM-style membership, a consistent-hash ring, and request forwarding. One use was a geospatial service that distributed ephemeral active-driver state across workers. Requests could enter any node and be forwarded to the key owner ([Uber Engineering, 2016](https://www.uber.com/en-SE/blog/ringpop-open-source-nodejs-library/)).

```text
rider asks for nearby drivers
            |
            v
        any worker
            |
      hash geographic key
            |
       handle locally?
         /      \
       yes       no --forward--> owning worker's in-memory state
```

Uber later documented limits in Ringpop-based systems: peer-to-peer membership convergence became problematic as ring size grew, and some architectures were replaced or redesigned ([Uber push-platform evolution](https://www.uber.com/us/en/blog/real-time-push-platform/); [Uber fulfillment rearchitecture](https://www.uber.com/us/en/blog/fulfillment-platform-rearchitecture/)).

Interview lesson: a design can be valid at one scale and later lose to a centralized control plane or a different datastore. Consistent hashing does not remove membership-coordination costs.

### 10.4 WhatsApp-style connection gateways: illustrative, not a production claim

Suppose long-lived connections keep small ephemeral per-user buffers on gateway nodes:

```text
phone --WebSocket--> edge/load balancer
                           |
                    hash(user_id)
                           |
             +-------------+-------------+
             v             v             v
          gateway A     gateway B     gateway C
             |             |             |
             +------ durable/shared message state ------+
```

Consistent hashing can improve affinity: the same user normally reaches the same gateway, and adding a gateway moves only a share of users. But:

- an existing TCP/WebSocket connection does not teleport when ownership changes;
- the load balancer may keep it until reconnect/drain;
- user activity is not uniform;
- durable undelivered messages should not exist only in gateway memory; and
- presence may require a directory because one account can use multiple devices.

This is an **illustrative WhatsApp-like design**, not evidence of WhatsApp's internal routing algorithm.

### 10.5 YouTube-style video caches: illustrative

```text
viewer requests video_id=V
           |
           v
regional cache router --hash(V)--> cache owner
           |                           |
           |<--------- hit ------------+
           |
         miss
           v
origin/object storage -> fill/cache response
```

Consistent hashing can reduce the number of videos that lose their cache owner during fleet resizing. It does not solve:

- one viral video's traffic;
- geographic nearest-edge selection;
- bitrate/segment differences;
- origin protection during cold fill; or
- replication of popular segments.

Use DNS/anycast/geographic routing to choose a region or edge tier, then a partitioning algorithm inside that tier. This is an **illustrative YouTube-like cache**, not a statement about YouTube's current internal architecture.

### 10.6 Twitter/X-style timelines and celebrity traffic: illustrative

Hashing `user_id` can place a user's home-timeline material together:

```text
normal users ----hash(user_id)----> many timeline shards

celebrity with 100M followers ----> one user key is still one hot key
                                      |
                       replicate/cache/fan-out strategy required
```

Consistent hashing helps when timeline-cache nodes join or leave; it does not solve the celebrity problem. The system may replicate popular reads, mix fan-out-on-write with fan-out-on-read, or special-case very large accounts.

This is an **illustrative Twitter/X-like design**. Twitter publicly described its Manhattan store as using independent shards and requiring engineers to choose partition keys so strongly consistent operations stay within a key, but that publication does not establish that classic ring consistent hashing was the placement method ([Twitter/X Engineering, Manhattan](https://blog.x.com/engineering/en_us/a/2016/strong-consistency-in-manhattan)).

### 11.3 Answer spine: **KEY-RING**

Use these cues to reconstruct an answer live.

#### K — Key and workload

- What object or request is being assigned?
- Name the partition key: user ID, cache key, video ID, tenant ID.
- Is the state durable or disposable?
- Are requests/bytes uniform, or can one key become hot?

#### E — Existing membership behavior

- How often do nodes join, leave, fail, or autoscale?
- Is key affinity required?
- What does reassignment cost: cold misses, reconnects, or data copying?

#### Y — Why naive modulo fails

- `owner = hash(key) % N` is simple with fixed membership.
- Adding one node moves `N/(N+1)` of keys; removing one moves `(N-1)/N`.
- Example: `100 -> 101` remaps about 99.01%.

#### R — Ring rule

- Fixed hash space; map nodes and keys into it.
- Key goes to first node clockwise, with wraparound.
- Addition steals one interval; removal transfers only the leaving owner's intervals.

#### I — Imbalance and virtual nodes

- One random point per server is uneven.
- Many virtual positions smooth ownership and spread failure/migration work.
- Weight by capacity if needed.
- Vnodes still do not solve hot keys.

#### N — Node-change mechanics

- Add: about `K/(N+1)` remaps; remove: about `K/N`.
- Explain copy/catch-up/cutover for durable data or cold-fill behavior for cache.
- Version ring metadata and handle stale routers.

#### G — Guarantees and gaps

- Add replication across distinct failure domains for availability/durability.
- Membership/failure detection is separate.
- Cover consistency, migration headroom, hot keys, hash stability, and monitoring.

### 11.4 Example live answer

> "I first need a stable partition key; suppose this is a distributed cache keyed by object ID and nodes autoscale. Direct `hash(key) % N` is fine while `N` is fixed, but changing from `N` to `N+1` remaps `N/(N+1)` of keys—about 99% when going from 100 to 101 nodes—causing a cache-miss storm. I would use a fixed hash space, place keys and multiple virtual positions per physical node on a ring, and route each key to the first position clockwise. A joining equal node then takes about `1/(N+1)` of keys, while removing one of `N` nodes affects about `1/N`. Virtual nodes smooth random imbalance and spread movement across machines. I would still design membership/version distribution, cache warming and origin protection, hot-key handling, and replication if losing cached state is unacceptable; consistent hashing itself does not provide those."

### 11.5 Synthesized practice follow-ups and answer cues

The following are **my own plausible synthesis**, not sourced interview reports.

#### "Why does `3 -> 4` move 75% under modulo?"

Answer cues:

- compare `h % 3` and `h % 4` over `h = 0..11`;
- only `0, 1, 2` keep the same remainder;
- 9 of 12 move;
- new server owns 25%, but old nodes also exchange keys unnecessarily.

#### "How much moves when adding the 101st server to 100?"

Answer cues:

- modulo: `100/101 ~= 99.01%`;
- ideal consistent hash: `1/101 ~= 0.99%`;
- multiply by keys for count, by actual value sizes for bytes;
- call them expectations under uniform hashes/equal weights.

#### "Why do we need virtual nodes?"

Answer cues:

- a few random node points create unequal arcs;
- many scattered points smooth total ownership;
- failure/addition work comes from or goes to many peers;
- capacity weighting becomes possible;
- cost is metadata/range/replica-placement complexity;
- hot keys remain.

#### "A database node died. Can I just route its keys clockwise?"

Answer cues:

- routing clockwise changes the logical owner, not the location of lost bytes;
- replicas across distinct failure domains are required;
- say how reads/writes choose replicas and what consistency is promised;
- rebuild the replacement from replicas, with throttling and repair.

#### "How do you add a node with zero downtime?"

Answer cues:

- consistent hashing only limits the affected ranges;
- snapshot/copy, capture concurrent writes, catch up, verify;
- publish new epoch, redirect/forward stale clients, drain old owner;
- retain rollback copy until safe;
- throttle migration and keep failure headroom;
- for a disposable cache, lazy refill may replace copying, but protect origin.

#### "Why not round robin?"

Answer cues:

- round robin is good for interchangeable stateless requests;
- it does not ensure the next request reaches the node holding that key's state;
- if all state is shared/external, affinity may be unnecessary and round robin is simpler;
- do not use consistent hashing without a state/locality need.

#### "Does consistent hashing solve a celebrity hot key?"

Answer cues:

- it balances keys, not request frequency or bytes;
- replicate/cache the hot value, coalesce fills, split if semantics allow;
- adaptive/bounded-load routing may override pure affinity;
- isolate a tenant/key if it threatens ordinary traffic.

#### "How do you choose replicas on a vnode ring?"

Answer cues:

- start from primary owner and walk the ring;
- skip vnodes of already-selected physical nodes;
- enforce rack/zone/region separation;
- maintain a preference list larger than required for temporary failures;
- define leader/quorum/conflict behavior separately.

#### "Every client has a different ring view. What happens?"

Answer cues:

- wrong-owner requests, duplicate/missed writes, forwarding loops;
- authoritative versioned membership/epochs;
- owner redirects or forwards with hop limit;
- clients refresh on version mismatch;
- idempotency and transition reads/writes protect correctness.

#### "Would you implement this yourself?"

Answer cues:

- usually choose a managed/distributed store whose partitioning is proven;
- custom implementation is justified for infrastructure components or special stateful routing;
- name operational requirements: membership, migration, replication, repair, observability;
- avoid placing a second, conflicting ring above a database without need.

#### "What should we monitor during rebalancing?"

Answer cues:

- range/key/byte and request distribution per node;
- migration bytes/sec, remaining ranges, ETA, failures/retries;
- foreground p95/p99 latency and success rate;
- disk/network/CPU/queue saturation;
- cache hit rate/origin load or replica lag;
- stale-ring redirects, forwarding rate, and version distribution;
- hot keys/tenants and capacity headroom.

### 11.6 Interviewer red flags

- Drawing a ring before naming the partition key and workload.
- Saying "almost all" without being able to calculate a simple case.
- Claiming exact balance from one node position per server.
- Treating virtual nodes as replicas.
- Claiming a failed node's data automatically appears on its successor.
- Ignoring how clients learn membership and ring versions.
- Ignoring cache-miss or migration load during scaling.
- Using consistent hashing for a stateless service where ordinary load balancing is sufficient.
- Stating that a product uses consistent hashing without a source or dated qualifier.

## 12. Compact revision sheet

### 12.1 Formula card

Let `K` be the number of uniformly hashed keys and `N` the original number of equal-capacity nodes.

| Change | Direct modulo keys moved | Consistent-hash keys moved, expected |
|---|---:|---:|
| Add one: `N -> N+1` | `K * N/(N+1)` | `K/(N+1)` |
| Remove one: `N -> N-1` | `K * (N-1)/N` | `K/N` |

Memory cue:

```text
Modulo:     almost everybody re-evaluates to a different numbered bucket.
Consistent: only the arriving/leaving node's approximate share changes owner.
```

### 12.2 Five facts to retain

1. Use a stable hash of an explicitly chosen partition key.
2. Direct modulo is simple but couples every assignment to physical node count.
3. A fixed hash ring localizes ownership changes.
4. Virtual nodes smooth ownership and spread movement; they do not fix hot keys.
5. Membership, data migration, replication, consistency, and failure detection remain separate design problems.

### 12.3 Decision checklist

- [ ] Do I actually need per-key affinity or data placement?
- [ ] What exact key am I hashing, and can it become hot or huge?
- [ ] Is the state disposable cache data, ephemeral process state, or durable data?
- [ ] How often does membership change?
- [ ] What movement formula applies?
- [ ] Do I need virtual nodes or weights?
- [ ] Where does authoritative membership/ring version come from?
- [ ] How are stale routers handled?
- [ ] How does planned copy/cutover work?
- [ ] What happens on sudden failure?
- [ ] Are replicas on distinct physical failure domains?
- [ ] Can migration or cold misses overload the source?
- [ ] How are hot keys and uneven bytes/CPU detected and mitigated?
- [ ] Would fixed slots, rendezvous hashing, a directory, or ordinary load balancing be simpler?

## 14. Say-out-loud 30-second narrative

> "Hashing gives me a deterministic way to assign a key to a server, but direct `hash(key) mod N` couples every assignment to the current server count. When I add a node, modulo hashing moves `N over N-plus-one` of the keys—about 99% from 100 to 101 nodes. Consistent hashing puts keys and server positions in one fixed ring and chooses the first server clockwise, so only about the new server's `1 over N-plus-one` share moves; removing a node affects about `1 over N`. I use virtual nodes to smooth random imbalance and support weights. Then I separately design stable membership versions, safe migration or cache warming, replicas across failure domains, consistency, and hot-key handling, because consistent hashing only minimizes remapping—it does not provide availability or move data by itself."
