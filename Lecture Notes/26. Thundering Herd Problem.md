# Thundering Herd Problem

> Beginner-first HLD notes expanded from: **a full server queue, new requests plus retries, memory pressure, latency-driven timeouts, exponential backoff, jitter, and a token bucket whose behavior responds to available capacity**.
>
> The central interview lesson is: **a thundering herd is a coordination and overload problem. Many actors react to the same event at nearly the same time, so work that is individually reasonable becomes a synchronized surge. Backoff with jitter is one client-side defense, but a safe design also bounds retries, rejects excess work early, preserves idempotency, controls concurrency, and recovers gradually.**

## Scope, assumptions, and evidence labels

I interpret **"Token bucket generation based on current system remaining capacity"** in two possible ways:

1. a **retry budget**, where retries spend tokens and successful calls replenish them; or
2. an **adaptive admission limiter**, where the rate or concurrency limit changes from overload signals such as latency, throttling, or queueing.

Both are covered. They are related, but they are not the same mechanism. A normal token bucket refills at a configured rate; it does not magically know the server's remaining capacity.

Evidence labels used in this chapter:

- **Verified concept** — supported by a first-party specification, official documentation, or first-party engineering source linked nearby.
- **Dated production example** — a company publicly described this behavior at the linked publication date; it is not a claim about its private current architecture.
- **Illustrative design** — a plausible design for a familiar product, not a claim about that company's private implementation.
- **Self-reported interview experience** — a candidate or anonymous poster described the interview publicly; it is not an official or independently verified company question bank.
- **Illustrative number** — selected to explain mechanics; it is not a production capacity or universal default.

Technical documentation and interview reports were checked on **2026-07-17**. Time-sensitive product behavior is either dated or linked to current documentation.

### Corrections to the rough bullets, up front

1. **A full queue does not always mean a request is denied.** It depends on the admission policy. A well-protected server usually rejects early; a poorly bounded system may continue allocating memory, spawning tasks, or blocking callers until it crashes.
2. **Traffic during failure is `new/original requests + retries`, but retries can also multiply across layers.** If three layers each make four total attempts, one user action can generate up to `4^3 = 64` lowest-level attempts.
3. **Memory is not the only full-capacity failure.** CPU, threads, event-loop time, sockets, file descriptors, database connections, downstream quotas, and garbage collection can saturate first.
4. **A timeout does not prove that the server did no work.** The operation may still be queued, running, or even committed. A retry can duplicate work unless the operation is idempotent or uses an idempotency key.
5. **Exponential backoff alone does not eliminate synchronization.** Identical clients that fail together still retry together at `1s, 2s, 4s, ...`. Jitter supplies the missing randomness.
6. **The full-jitter formula needs the cap inside the random range:**

   ```text
   backoff_cap(k) = min(max_backoff, base * 2^k)
   sleep(k)       = Uniform(0, backoff_cap(k))
   ```

   Writing `min(max_wait, Random(0, base * 2^k))` produces a spike at `max_wait` once the random upper range exceeds the cap, because every random value above the cap collapses to the same sleep value.
7. **`k` must be defined.** In this chapter, `k = 0` means the first retry after the initial failed attempt. Some libraries start the exponent at 1; confirm the convention instead of memorizing a sequence.
8. **Backoff and jitter do not create capacity.** They spread and reduce retry traffic. The server still needs bounded queues, admission control, load shedding, prioritization, and often a circuit breaker or concurrency limiter.
9. **A retry token bucket and a traffic rate limiter solve different questions.** One asks, “May this client spend reliability budget on another attempt?” The other asks, “May this request enter the service now?”
10. **“Use a token bucket based on remaining capacity” is underspecified.** Capacity is multi-dimensional and changes with request cost. An adaptive limiter needs a feedback signal, control algorithm, scope, and failure behavior; raw free memory is rarely a sufficient signal.

## 1. Plain-language foundation

### 1.1 Start with one everyday picture

Imagine a concert gate temporarily closes. Ten thousand people wait outside. The moment the gate opens, everyone sees the same signal and runs toward one narrow entrance.

The problem is not that any one person is unreasonable. The problem is that:

- a large population waited for the same event;
- everyone reacted at nearly the same time; and
- all of them competed for one limited resource.

That is a **thundering herd**.

In software, the “people” may be clients, application servers, worker processes, threads, scheduled jobs, or cache misses. The “gate opening” may be a timeout, cache expiration, lock release, server recovery, network reconnection, or scheduled time. The “narrow entrance” may be an API, database, cache shard, lock, queue, CPU, or network link.

```text
Many actors             One shared trigger            Scarce resource
-----------             ------------------            ---------------
clients 1..N ----+
workers 1..N -----+----> timeout / expiry / unlock ---> API / DB / cache
jobs 1..N --------+      recovery / scheduled time     CPU / connection pool
```

A useful one-sentence definition is:

> A thundering herd occurs when many actors become ready or retry together because of a correlated event, creating a synchronized surge that wastes work or overwhelms a shared resource.

### 1.2 Three ingredients

Most herds have all three:

1. **A population:** many clients, waiters, jobs, or requests.
2. **Correlation:** they observe the same failure, clock boundary, expiration, unlock, or recovery.
3. **A bottleneck:** only a limited number can make progress.

Remove any one ingredient and the problem becomes smaller:

- fewer actors means a smaller burst;
- de-synchronization spreads the burst; and
- more safe capacity lets more work finish.

This explains why solutions fall into three families:

```text
Reduce population       Break correlation       Protect/increase bottleneck
-----------------       -----------------       ---------------------------
retry budget            jitter                  concurrency limit
coalesce duplicates     TTL staggering          bounded queue
drop obsolete work      randomized schedules    load shedding
circuit breaker          gradual ramp-up          autoscaling/capacity
```

### 1.3 A retry storm is one kind of thundering herd

Suppose a server fails at 12:00:00. One million clients receive an error. If every client retries exactly one second later, the server receives a million calls at 12:00:01. If it fails again, the same synchronized wave returns later.

```text
Time          Original requests       Retry requests       Total offered load
12:00:00      very high               0                    very high
12:00:01      new traffic             first retry wave     even higher
12:00:03      new traffic             second retry wave    high again
12:00:07      new traffic             third retry wave     high again
```

This retry-specific form is often called a **retry storm**. Microsoft’s Retry Storm antipattern documentation also notes that excessive retry attempts during recovery can intensify the original problem and describes the situation as a thundering herd. [Microsoft Azure Architecture Center: Retry Storm antipattern](https://learn.microsoft.com/en-us/azure/architecture/antipatterns/retry-storm/)

### 1.4 The term is broader than retries

Common forms include:

| Trigger | Herd behavior | Bottleneck |
|---|---|---|
| Service outage or timeout | All clients retry | Recovering service |
| Hot cache entry expires | All misses recompute/fetch it | Database/origin |
| Lock becomes available | Every waiter wakes; only one wins | Lock/CPU scheduler |
| Network returns | Devices reconnect and resynchronize | Connection gateway/session store |
| Cron boundary | Thousands of jobs start at midnight | Shared database/cluster |
| Leader election finishes | Followers query or register together | New leader |
| Deployment/recovery | Traffic moves to fresh instances immediately | Cold instances/caches |
| Rate-limit window resets | Clients spend the new quota together | API/downstream resource |

The name **cache stampede** or **dogpile** usually refers to the cache-expiration form. A retry storm, cache stampede, reconnect storm, and waiter wake-up herd share the same underlying pattern, but their best mitigations differ.

### 1.5 Why a herd is worse than the same average traffic spread out

A server that can handle 1,000 requests each second might comfortably process 60,000 requests across a minute. It may still fail if all 60,000 arrive in one second.

```text
Smooth load                         Synchronized herd

RPS                                 RPS
^  ##########                       ^          #
|  ##########                       |          #
|  ##########                       |          #
+-------------> time                +----------#----> time
   within capacity                            huge instantaneous burst
```

The average is identical; the peak is not. System design must account for **burstiness, concurrency, and instantaneous capacity**, not only daily or per-minute averages.

## 2. Inline prerequisites

These concepts are required to understand the topic, so they are explained here rather than deferred.

### 2.1 Request, attempt, retry, and logical operation

Keep these words separate:

- A **logical operation** is what the user wants, such as “place order 123.”
- The **initial attempt** is the first network call for it.
- A **retry** is another attempt after a failure or ambiguous result.
- A **request attempt** is one individual call reaching some layer.

One logical operation can therefore produce several attempts:

```text
Place order 123
    |
    +-- attempt 1 -> timeout
    +-- attempt 2 -> timeout
    +-- attempt 3 -> success
```

If all dashboards call these “requests,” it becomes difficult to distinguish user demand from retry-generated load. Good observability records a stable logical-operation or idempotency ID and a separate attempt number.

### 2.2 Throughput, latency, concurrency, and capacity

- **Throughput** is completed/admitted work per unit time, such as requests per second.
- **Latency** is how long one operation takes.
- **Concurrency** is how many operations are in flight simultaneously.
- **Capacity** is the safe amount of work the system can sustain while meeting its reliability and latency objectives.

In steady state, a useful simplified form of Little’s Law is:

```text
average concurrency ~= throughput * average latency
```

If a service receives 1,000 requests/second:

```text
at 100 ms latency: 1,000 * 0.1 = about 100 requests in flight
at   2 s latency: 1,000 * 2.0 = about 2,000 requests in flight
```

The arrival rate stayed the same, but a latency increase created roughly twenty times as much in-flight work. That consumes more threads, sockets, memory, and connection-pool slots.

### 2.3 Queue, worker pool, and backpressure

A typical server has a limited execution resource and a waiting area:

```text
arrivals --> [bounded queue] --> [worker/thread/connection pool] --> dependency
                  |
                  +--> reject when full
```

- The **queue** holds admitted work that has not begun execution.
- The **worker pool** bounds parallel execution.
- **Backpressure** is a signal or mechanism that makes upstream producers slow or stop when downstream cannot keep up.

A queue absorbs a short burst; it does not create throughput. If arrivals remain above service capacity, the queue eventually fills. A large queue can hide overload temporarily while increasing latency and memory usage.

### 2.4 Timeout versus deadline

A **timeout** limits how long one activity waits, such as a connection timeout or read timeout. A **deadline** is the latest time by which the whole logical operation remains useful.

```text
overall deadline: 2 seconds

API gateway work       100 ms
service A work         250 ms
remaining for B        1,650 ms, not a fresh 2 seconds
```

With **deadline propagation**, each downstream call receives the remaining end-to-end time. Without it, every layer can wait for its own full timeout and continue work after the user has already given up.

### 2.5 Idempotency and ambiguous outcomes

An operation is **idempotent** when repeating it has the same externally visible effect as performing it once.

```text
GET /profile/42                 usually naturally idempotent
PUT /users/42 with full state   intended to be idempotent
POST /payments                  not naturally idempotent
```

A timeout is ambiguous:

```text
client ---- create payment ----> server ----> database COMMIT
client <--- response lost ------- network failure
client sees timeout and retries
```

The first attempt succeeded, even though the client did not see the response. The retry can charge twice. For non-idempotent operations, send a stable **idempotency key** for the logical operation and store/replay the first committed result.

Idempotency is a prerequisite for safe retries; it does not itself control retry load.

### 2.6 Rate limit, concurrency limit, load shedding, and circuit breaker

These mechanisms overlap but answer different questions:

| Mechanism | Question it answers |
|---|---|
| Rate limiter | How quickly may this subject start work over time? |
| Concurrency limiter | How many operations may be in flight right now? |
| Load shedding | Which work should be rejected because current load is unsafe? |
| Circuit breaker | Should calls to this unhealthy dependency be temporarily short-circuited? |
| Retry budget | How much extra traffic may retries create? |

A fixed rate limit may admit 1,000 requests/second even after latency grows from 100 ms to 5 seconds. A concurrency limiter responds more directly to that growing in-flight pressure.

### 2.7 Token bucket, briefly

A token bucket contains up to `B` tokens and refills at `r` tokens/second. An operation costing `c` tokens is admitted only if at least `c` are available.

```text
tokens_now = min(B, tokens_previous + elapsed_seconds * r)

if tokens_now >= cost:
    tokens_now -= cost
    allow
else:
    reject_or_wait
```

- `r` controls the long-term average rate.
- `B` controls allowed burst size.
- Refill is usually calculated lazily; no background token generator is required.

For retries, tokens can represent **retry permission** rather than ordinary request quota. That distinction becomes important in Section 8.

## 3. Why the thundering herd matters in system design

### 3.1 It converts a partial failure into a larger failure

A small network interruption can synchronize thousands of clients. Their retries overload a previously healthy dependency, which then times out, producing more retries. The failure spreads.

```text
small fault
   |
   v
errors/timeouts --> synchronized retries --> higher offered load
      ^                                      |
      |                                      v
      +----------- more latency/queueing <---+
```

This is a **positive feedback loop**: the response to overload creates more overload.

Google’s SRE guidance describes how missed RPC deadlines waste server work and cause clients to retry, and recommends randomized exponential backoff, per-request retry limits, retry budgets, and avoiding retries at multiple layers. [Google SRE: Addressing Cascading Failures](https://sre.google/sre-book/addressing-cascading-failures/)

### 3.2 It attacks the recovery path

A failed server may restart with:

- an empty cache;
- cold JIT/runtime state;
- unopened database connections;
- incomplete membership information;
- ongoing replication or log replay; and
- less capacity because other instances are still down.

Sending the full backlog immediately can make it fail again. Recovery must often be **slower than normal serving**:

```text
restart -> health probe -> tiny traffic -> warm state -> gradual ramp -> normal share
```

### 3.3 It wastes work even when the server survives

Examples:

- 10,000 threads wake for one lock; 9,999 immediately sleep again.
- 5,000 cache misses run the same database query.
- a response arrives after its caller’s deadline and is discarded;
- a retry succeeds while the original attempt is still running;
- multiple layers retry the same failure independently.

The system spends CPU, bandwidth, and dependency capacity without producing proportional user value.

### 3.4 It damages tail latency and fairness

The first few requests fill the queue. Later interactive requests wait behind stale retries or background work. Even if average latency looks acceptable, p99/p99.9 latency and important tenants may suffer.

Priority and freshness matter:

```text
prefer:   new checkout request with 1.5 s remaining
over:     fourth analytics retry whose deadline already expired
```

### 3.5 Autoscaling alone may arrive too late

Autoscaling needs time to detect load, allocate capacity, start instances, pass health checks, warm caches, and open connections. A retry wave can recur in seconds. Worse, adding cold instances can shift traffic to capacity that is not ready.

Autoscaling is useful for sustained demand; it is not a substitute for immediate overload protection.

## 4. What happens when the server queue fills

### 4.1 Healthy overload behavior: reject early and cheaply

If a bounded queue has no safe room, an overload-aware server should generally reject before doing expensive work:

```text
request
   |
   v
[cheap admission check] -- no capacity --> rejection + retry guidance
   |
  yes
   v
[bounded queue] --> [workers] --> [dependency]
```

The exact response depends on why admission failed:

- **HTTP `429 Too Many Requests`**: the particular client/tenant/API key exceeded a rate policy. RFC 6585 says a `429` response may include `Retry-After`. [RFC 6585: 429 Too Many Requests](https://www.rfc-editor.org/rfc/rfc6585.html#section-4)
- **HTTP `503 Service Unavailable`**: the service is temporarily unable to handle the request because of overload or maintenance. RFC 9110 allows a server to send `Retry-After` to suggest when to try again. [RFC 9110: 503 Service Unavailable](https://www.rfc-editor.org/rfc/rfc9110.html#name-503-service-unavailable), [RFC 9110: Retry-After](https://www.rfc-editor.org/rfc/rfc9110.html#name-retry-after)
- **gRPC `RESOURCE_EXHAUSTED`**: often quota or capacity exhaustion; retryability depends on service semantics and pushback.
- **gRPC `UNAVAILABLE`**: a transient availability failure that may be retriable under a bounded policy.

Do not treat all of these as “retry immediately.” The response should say whether the condition is retriable and, where possible, provide a safe delay or pushback signal.

### 4.2 Unhealthy overload behavior

Without bounds, a server may:

- allocate one task/future/object per request until memory is exhausted;
- accept sockets faster than it can process them;
- block all request threads waiting for a connection pool;
- spend most CPU on context switching or garbage collection;
- fail health checks, causing the load balancer to remove it;
- shift its traffic to remaining servers, overloading them too; and
- restart into the same traffic and crash-loop.

```text
one server fails
      |
      v
less fleet capacity -> more load per survivor -> more latency/timeouts
      ^                                        |
      |                                        v
      +---------- retries and more failures ---+
```

Memory pressure is therefore one possible symptom, not the definition of full capacity.

### 4.3 Why a bigger queue is not a complete fix

A queue is helpful when a brief burst can drain before deadlines expire. If the arrival rate remains greater than processing rate, a larger queue only changes **where requests wait**.

Assume an **illustrative** worker pool completes 100 requests/second and 1,000 requests arrive at once:

- with no queue, roughly 100 can begin during the first second and the rest fail quickly;
- with a 900-item queue, the last request waits roughly 9 seconds before it even starts;
- if the client deadline is 2 seconds, most queued work is already useless.

Google’s SRE discussion of queue management emphasizes that queued requests consume memory and increase latency, and that early rejection can be preferable for relatively steady traffic. [Google SRE: Queue Management](https://sre.google/sre-book/addressing-cascading-failures/#queue-management)

### 4.4 Queue age is often a better signal than queue length alone

Ten queued 10-second jobs can be worse than one hundred 1-millisecond jobs. Useful signals include:

- oldest item age;
- estimated wait time;
- remaining request deadline;
- in-flight work weighted by estimated cost;
- worker utilization;
- dependency latency; and
- rejection/timeout rates.

Admission can reject an item whose estimated completion time exceeds its deadline:

```text
if estimated_queue_wait + estimated_service_time > remaining_deadline:
    reject_as_deadline_exceeded
```

This estimate will be imperfect, but processing clearly obsolete work is usually worse.

## 5. How latency and timeout create a retry feedback loop

### 5.1 The sequence

```text
1. Offered load approaches capacity.
2. Queueing increases latency.
3. Clients hit their timeout/deadline.
4. Clients retry.
5. Original attempts may still occupy server resources.
6. Total offered load increases.
7. Queueing and latency become worse.
```

The crucial point is step 5: **client cancellation and server cancellation are not automatically the same event**.

### 5.2 Late work

```text
Client                   Service A                    Database
  |---- request ----------->|                            |
  |                         |---- slow query ----------->|
  |                         |                            |
  | x timeout at 1 s        |                            |
  |---- retry ------------->|                            |
  |                         |---- duplicate query ------>|
  |                         |<--- first result at 3 s ---|
  |                         |    result now unwanted      |
```

Mitigations:

- propagate the absolute deadline;
- cancel downstream work where the protocol and operation support cancellation;
- check remaining deadline before expensive stages;
- do not dequeue expired requests;
- make state changes idempotent;
- use bounded retries; and
- measure late completions, not only client timeouts.

### 5.3 A timeout is not an error classification

“Timeout” tells the client that it did not receive a result in time. It does not say:

- whether the server received the request;
- whether execution began;
- whether a database transaction committed;
- whether the response was merely lost; or
- whether another attempt is safe.

Retry policy needs operation semantics, not only transport status.

### 5.4 Pick timeouts from the end-to-end objective

Suppose the user-facing deadline is 2 seconds. A policy of “three attempts, each with a 2-second timeout” cannot meet it. The retry loop should stop when the remaining budget cannot fit another useful attempt:

```text
remaining = absolute_deadline - now

if remaining <= minimum_useful_attempt_time:
    do_not_retry

sleep = choose_backoff(...)

if sleep + minimum_useful_attempt_time > remaining:
    do_not_retry
```

Maximum attempts and maximum elapsed retry time are both needed.

## 6. Retry amplification math

### 6.1 Offered load includes originals and all attempts

Let:

- `λ_original` = logical operations per second;
- `A_avg` = average number of attempts per operation, including the initial attempt.

Then:

```text
λ_attempts = λ_original * A_avg

λ_retries = λ_attempts - λ_original
```

If 10,000 operations/second average 1.2 attempts:

```text
attempt traffic = 10,000 * 1.2 = 12,000 attempts/second
retry traffic   = 2,000 attempts/second
```

That extra 20% often appears precisely when capacity is already reduced.

### 6.2 Worst case during a hard outage

If every operation allows `R` retries, the maximum total attempts per logical operation are:

```text
A_max = 1 + R
```

For 100,000 logical operations and 3 retries:

```text
maximum attempts = 100,000 * 4 = 400,000
```

Backoff changes when those attempts arrive; the attempt cap bounds how many can exist.

### 6.3 Expected attempts under a simplified independent failure model

Let each attempt independently fail with probability `p`, and allow `R` retries. Then the expected number of attempts is:

```text
E[A] = 1 + p + p^2 + ... + p^R
```

For `p = 0.2` and `R = 2`:

```text
E[A] = 1 + 0.2 + 0.04 = 1.24 attempts
```

This model is useful for intuition but weak during an outage because failures are usually **correlated**, not independent. If the service is unavailable, `p` is near 1 for the whole population, and the worst case matters.

### 6.4 Multiplication across layers

Suppose:

```text
browser -> API gateway -> service A -> service B -> database
```

If the browser, gateway, and service A each permit 4 total attempts, a single logical operation can cause up to:

```text
4 * 4 * 4 = 64 attempts at service B/database
```

This is why “each layer retries three times” is dangerous. Prefer one well-informed retry layer for a given failure boundary, and make the attempt budget visible end-to-end.

### 6.5 Retry budget as a ratio

A practical policy can cap retry traffic relative to original traffic:

```text
retry_attempts_per_window <= β * original_attempts_per_window
```

For an **illustrative** `β = 0.10`, at most 10 retry attempts are allowed per 100 original attempts in the accounting window. The correct value is workload-specific and should come from spare capacity and reliability goals, not from this example.

The ratio prevents retry traffic from taking over the service during a broad failure.

## 7. Exponential backoff, from first principles

### 7.1 Why wait at all?

Many failures are transient:

- a packet was lost;
- a leader is being elected;
- an instance is restarting;
- a dependency briefly throttled;
- a connection was reset; or
- a lock or quota may become available.

An immediate retry is likely to encounter the same condition and adds load during recovery. Backoff increases the time between attempts.

### 7.2 Capped exponential backoff

Define:

- `base` = initial backoff scale;
- `k` = retry index, with `k = 0` for the first retry;
- `max_backoff` = largest delay scale.

```text
backoff_cap(k) = min(max_backoff, base * 2^k)
```

For `base = 100 ms`, `max_backoff = 2 s`:

| Retry index `k` | Uncapped | Capped delay scale |
|---:|---:|---:|
| 0 | 100 ms | 100 ms |
| 1 | 200 ms | 200 ms |
| 2 | 400 ms | 400 ms |
| 3 | 800 ms | 800 ms |
| 4 | 1,600 ms | 1,600 ms |
| 5 | 3,200 ms | 2,000 ms |
| 6 | 6,400 ms | 2,000 ms |

Without a maximum, delays can become uselessly large or overflow numeric/time types. Without a maximum number of attempts or total deadline, clients can keep retrying forever at the cap.

### 7.3 Why exponential rather than fixed delay?

| Policy | Benefit | Problem |
|---|---|---|
| Immediate retry | Lowest possible recovery latency for a tiny one-off fault | Maximum overload amplification |
| Fixed delay | Simple; adds breathing room | Repeated synchronized waves; does not reduce retry frequency over time |
| Linear backoff | Grows gently | May remain too aggressive during long outages |
| Exponential backoff | Quickly reduces attempt frequency | Still synchronized without jitter; can add user latency |
| Server-directed delay | Uses server knowledge | Signal may be absent, stale, coarse, or identical for everyone |

Exponential growth is not sacred. It is a robust default for transient remote failures because it rapidly lowers pressure while still allowing early recovery attempts.

### 7.4 Backoff without jitter still herds

Ten thousand identical clients fail at the same time. With deterministic delays, all ten thousand retry at the same cumulative times:

```text
delay sequence:       1s, 2s, 4s, 8s
cumulative retry at:  1s, 3s, 7s, 15s
```

Backoff creates wider gaps, but each retry remains a spike:

```text
requests
   ^       |       |           |               |
   |       |       |           |               |
   +-------1s------3s----------7s--------------15s----> time
```

Jitter is what spreads clients inside those intervals.

## 8. Jitter: breaking synchronization

### 8.1 Full jitter

For retry index `k`:

```text
cap(k)   = min(max_backoff, base * 2^k)
sleep(k) = Uniform(0, cap(k))
```

Every delay in the interval is equally likely. Its expected sleep is `cap(k) / 2`.

Example with `cap(k) = 800 ms`:

```text
client A sleeps  37 ms
client B sleeps 412 ms
client C sleeps 779 ms
...
```

The user’s original expression:

```text
min(max_wait, Random(0, base * 2^n))
```

should be written as:

```text
Random(0, min(max_wait, base * 2^n))
```

Why? If the uncapped random range is `0..8s` and `max_wait = 2s`, the original expression maps every sample from `2s..8s` to exactly `2s`, recreating a large synchronized spike at the cap.

### 8.2 Equal jitter

```text
cap(k)   = min(max_backoff, base * 2^k)
sleep(k) = cap(k)/2 + Uniform(0, cap(k)/2)
```

This guarantees at least half the exponential delay while randomizing the other half. It avoids near-zero sleeps but has a higher mean delay than full jitter for the same cap.

### 8.3 Decorrelated jitter

One common form uses the previous sleep:

```text
sleep_next = min(max_backoff, Uniform(base, 3 * sleep_previous))
```

It does not follow a strict doubling sequence. It can spread clients well and avoids identical schedules, but behavior is less intuitive and depends on the previous random value.

### 8.4 Which jitter should you choose?

AWS’s public simulation found un-jittered exponential backoff clearly worse than its jittered alternatives; full and decorrelated jitter involved a trade-off between total work and completion time in that simulation. [AWS Architecture Blog: Exponential Backoff and Jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/)

A defensible interview default is:

- use **full jitter** unless a platform/library already supplies a tested policy;
- use a cap, maximum attempts, and total deadline;
- honor server pushback;
- do not hand-roll retry logic when a mature client library already implements the service’s required semantics.

There is no universally best jitter distribution. The correct policy depends on latency sensitivity, server recovery behavior, population size, and whether a central scheduler already spaces work.

### 8.5 Jitter does not mean “add a tiny random value”

This is weak:

```text
sleep = exponential_delay + Uniform(0, 10 ms)
```

If 100,000 clients are squeezed into a 10 ms band, the peak can remain enormous. The jitter window must be meaningful relative to the herd size and service capacity.

### 8.6 `Retry-After` and jitter

A server may return `Retry-After` as either a delay in seconds or an HTTP date. If it represents a **minimum** time before retrying, clients should not randomly retry earlier.

A safe conceptual policy is:

```text
not_before = parsed Retry-After
sleep      = not_before + Uniform(0, additional_spread)
```

Alternatively, the server can provide a per-client randomized retry time. If every client receives exactly `Retry-After: 60`, all can still return at the same boundary.

The exact combination is API-specific: respect server semantics first, then add positive spreading where allowed.

### 8.7 Deterministic jitter for scheduled work

Pure randomness changes after every restart. For periodic jobs, stable spreading is often better:

```text
offset = hash(job_id) mod period
run at period_start + offset
```

All jobs receive different offsets, but the same job remains predictably scheduled. Use a well-distributed hash and allow operational override.

## 9. Which failures should be retried?

### 9.1 The decision checklist

Retry only if all are true:

1. **The failure is plausibly transient.** Waiting may change the outcome.
2. **The operation is safe to repeat.** It is idempotent, conditionally idempotent, or protected by an idempotency key.
3. **There is time left.** Another attempt can finish within the end-to-end deadline.
4. **Retry budget remains.** Attempts, elapsed time, and global retry quota are not exhausted.
5. **The dependency is accepting probes.** The circuit is not open and server pushback does not prohibit it.

### 9.2 Typical HTTP treatment

This table is a starting point, not a substitute for the API contract:

| Signal | Typical treatment | Why/caveat |
|---|---|---|
| Connection failure before send | Often retry | More likely the application did not process it, but transport details matter |
| Read timeout after send | Maybe retry | Outcome is ambiguous; require idempotency |
| `408 Request Timeout` | Often retry under policy | Request timing issue may be transient |
| `409 Conflict` | Only for a documented transient conflict | Many conflicts require the client to change state first |
| `425 Too Early` | Retry later as specified | Relevant to replay-risk handling; client/library semantics matter |
| `429 Too Many Requests` | Retry after pushback if useful | Respect rate policy and `Retry-After`; do not compete aggressively |
| `500 Internal Server Error` | Sometimes | Could be transient or a deterministic bug triggered by this input |
| `502 Bad Gateway` | Often bounded retry | An intermediary/upstream failure may be transient |
| `503 Service Unavailable` | Often bounded retry | Respect `Retry-After`; overload may worsen with retries |
| `504 Gateway Timeout` | Maybe | Original downstream work may still be running |
| `400`, `401`, `403`, usually `404` | Usually do not retry unchanged | Validation, credentials, permission, or absent resource will not improve with waiting |

Do not blindly retry “all `5xx`” or “anything not `2xx`.” Service-specific error codes are more informative than the status family alone.

### 9.3 Queue consumers have different retry choices

For asynchronous jobs, a user deadline may be minutes or hours rather than seconds. A failed message should usually not be pushed immediately back onto the head of the same hot queue.

```text
main topic --> worker -- transient failure --> delayed retry topic (attempt 1)
                                      |-----> delayed retry topic (attempt 2)
                                      +-----> DLQ after terminal policy
```

Consider:

- delayed retry queues or scheduled visibility;
- per-key ordering requirements;
- poison messages;
- dead-letter queues and replay tooling;
- idempotent consumers;
- retry age and attempt metadata;
- whether later messages may proceed; and
- a retry rate separate from new work.

## 10. Token buckets and “remaining capacity”

### 10.1 Fixed-rate admission token bucket

The ordinary token bucket uses configured refill rate `r` and capacity `B`:

```text
                refill at r tokens/s
                       |
                       v
request(cost c) --> [ bucket, max B ] --> admit if c tokens exist
```

It controls an average rate and bounded burst. It does **not** automatically react when server latency doubles or half the fleet fails, unless something changes `r`.

### 10.2 Retry-quota token bucket

A retry quota protects the service from extra attempts:

```text
initial request: does not spend retry token
failed attempt:  retry allowed only if token exists
successful call: replenish some retry credit
bucket empty:    fail without retry
```

The key idea is that widespread failures consume tokens while successes replenish them. During a healthy period, retry capacity is available for isolated transient faults. During a broad outage, the bucket drains and suppresses the herd.

Current AWS SDK documentation describes a standard retry mode with exponential backoff plus jitter and a retry-quota token bucket; when its tokens are depleted, the SDK returns the error without retrying. It also distinguishes an adaptive mode that adds a client-side rate limiter. Exact SDK defaults and opt-in behavior are version-sensitive, so consult the linked current documentation rather than copying constants. [AWS SDKs and Tools: Retry behavior](https://docs.aws.amazon.com/sdkref/latest/guide/feature-retry-behavior.html)

Scope matters:

- one bucket per process can multiply across 1,000 processes;
- one shared global bucket can become a bottleneck;
- a per-dependency, per-region, or per-tenant bucket often balances isolation and scale;
- high-priority traffic may need reserved retry budget.

### 10.3 Adaptive send-rate token bucket

An adaptive client may vary how quickly send tokens are generated using feedback such as:

- explicit throttling responses;
- recent success/failure ratio;
- round-trip latency;
- server pushback;
- queue delay; or
- a centrally distributed capacity allocation.

A simplified **conceptual**, not universal, controller is additive-increase/multiplicative-decrease:

```text
healthy low-latency interval: r_next = min(r_max, r_current + α)
overload/throttling signal:   r_next = max(r_min, β * r_current), 0 < β < 1
```

The updated `r_next` becomes the token refill rate. This probes upward slowly and cuts load quickly on overload.

Trade-offs:

- feedback arrives after the system has changed;
- noisy latency can cause oscillation;
- one hot tenant can slow unrelated tenants if they share a limiter;
- client-local estimates do not reveal global capacity;
- a malicious or old client may ignore the policy; and
- adaptive pacing can delay the initial request, not just retries.

### 10.4 Adaptive concurrency is often a better model for latency-driven saturation

If the real constraint is in-flight work, think in permits rather than tokens per second:

```text
adaptive_limit L
in_flight C

admit only if C < L
on completion: observe latency/outcome and update L
```

Netflix’s public `concurrency-limits` project includes a Gradient2 algorithm that compares current round-trip time with a longer-term average and reduces the concurrency limit when latency indicates growing queueing. This is a public implementation pattern, not proof that every current Netflix service uses it. [Netflix `concurrency-limits`: Gradient2Limit](https://github.com/Netflix/concurrency-limits/blob/master/concurrency-limits-core/src/main/java/com/netflix/concurrency/limits/limit/Gradient2Limit.java)

### 10.5 “Remaining capacity” is not one number

A server can have:

- 40% CPU free but no database connections;
- free memory but a saturated network link;
- an empty request queue but a throttled third-party dependency;
- healthy averages but a hot partition;
- capacity for cheap reads but not expensive searches.

Better designs use a signal close to the constrained resource and sometimes weighted costs:

```text
profile read       = 1 unit
full route search  = 20 units
video transcode    = 500 units
```

Do not generate tokens from “free memory percentage” alone. Define the protected resource and validate the controller with overload tests.

### 10.6 Server-issued leases or budgets

For many callers, a central controller can lease portions of capacity:

```text
capacity controller
   |-- lease 500 units/s --> gateway A
   |-- lease 300 units/s --> gateway B
   +-- lease 200 units/s --> batch workers
```

This avoids a remote check for every request but introduces:

- temporarily unused capacity in an idle lease;
- over-admission if expired leases are not fenced;
- allocation lag during load changes;
- controller availability requirements; and
- fairness and priority decisions.

## 11. Defense in depth

No single box solves every herd. A robust call path combines controls:

```text
user/client
  |  bounded attempts + full jitter + deadline + idempotency key
  v
edge/API gateway
  |  rate limit + retry budget + priority + global load shed
  v
service A
  |  circuit breaker + adaptive concurrency + bounded queue
  v
service B
  |  cheap rejection + Retry-After/pushback + deadline checks
  v
database/cache
     connection limit + query timeout + duplicate coalescing
```

### 11.1 Bound the population

- cap attempts per logical operation;
- cap total elapsed retry time;
- allow retries at one appropriate layer;
- use a retry budget;
- coalesce identical requests;
- drop expired or superseded work;
- deduplicate jobs/messages;
- stop polling when push is available; and
- expire offline delivery work with a business TTL.

### 11.2 Break correlation

- full/decorrelated jitter for retries;
- TTL jitter for cache entries;
- deterministic offsets for periodic jobs;
- gradual traffic ramp after recovery;
- randomized or server-assigned reconnect slots;
- staggered health checks and refreshes; and
- per-key early refresh rather than simultaneous hard expiry.

### 11.3 Protect the bottleneck

- bounded queue and worker pool;
- concurrency/adaptive limiter;
- rate limit and quotas;
- circuit breaker;
- load shedding and priority;
- bulkheads per dependency/tenant;
- graceful degradation;
- capacity isolation; and
- autoscaling for sustained, not instantaneous, overload.

### 11.4 Make retry effects safe

- natural idempotency or idempotency keys;
- compare-and-set/version conditions;
- deduplication records with a defined retention window;
- stable logical-operation ID across attempts;
- safe payment/order state machines; and
- clear response semantics for replayed outcomes.

### 11.5 Make recovery controlled

- half-open circuit with a small number of probes;
- load-balancer slow start;
- warm caches and pools before full traffic;
- ramp region/instance weight gradually;
- allow only fresh/high-priority work initially;
- drain retry backlog at a bounded rate; and
- stop recovery if latency or rejection rises again.

## 12. Circuit breaker, bulkhead, and retry budget together

### 12.1 Circuit breaker state machine

```text
           failures exceed policy
 CLOSED ------------------------------> OPEN
   ^                                      |
   |                                      | after cool-down
   | probe succeeds                       v
   +---------------------------------- HALF-OPEN
                         probe fails ------> OPEN
```

- **Closed:** normal calls flow; failures are measured.
- **Open:** calls fail fast or use a fallback; the dependency is protected.
- **Half-open:** a limited probe checks recovery.

The half-open state itself can herd if every client independently sends a probe at the same instant. Limit probes per dependency scope and jitter the probe schedule.

### 12.2 Bulkhead

A bulkhead isolates resource pools:

```text
payment calls  --> [20 permits]
search calls   --> [50 permits]
profile calls  --> [30 permits]
```

If payment becomes slow, it exhausts its own permits rather than every thread in the application.

### 12.3 Retry budget

The retry budget limits extra attempts even while the circuit remains closed. It handles partial failures where some retries still succeed, without allowing retries to dominate originals.

### 12.4 Combined behavior

```text
call requested
   |
   +-- deadline expired? ----------> fail
   +-- circuit open? --------------> fallback/fail
   +-- concurrency permit absent? -> shed/fail
   |
   v
attempt dependency
   |
 failure
   +-- non-retryable? -------------> fail
   +-- idempotency unsafe? --------> reconcile/fail
   +-- retry token absent? --------> fail
   +-- no time left? --------------> fail
   |
   v
jittered backoff --> next attempt
```

Netflix’s historical Hystrix documentation is a dated public example of combining circuit breaking, thread-pool/semaphore isolation, rejection when pools/queues are full, fallbacks, and a single half-open probe after a sleep window. Hystrix is historical/deprecated technology, but the design concepts remain useful. [Netflix/Hystrix: How It Works](https://github.com/Netflix/Hystrix/wiki/how-it-works)

## 13. Herd-specific mitigation patterns

### 13.1 Cache stampede

#### Failure

```text
hot key expires
   |
   +-- request 1 --> DB/computation
   +-- request 2 --> DB/computation
   +-- request 3 --> DB/computation
   +-- ... thousands of duplicate fills
```

#### Defenses

**Request coalescing / singleflight**

```text
many misses --> [one leader fills] --> cache
                    |
                    +--> followers await/share result
```

Only one fill runs per key per coalescing scope. Bound follower wait time and handle a failed/stuck leader.

**Serve stale while revalidating**

Return a slightly stale safe value immediately while one worker refreshes it. This trades freshness for availability and origin protection.

**TTL jitter**

```text
effective_ttl = base_ttl + Uniform(-j, +j)
```

This prevents many keys loaded together from expiring together. It does not stop a herd on one extremely hot key by itself.

**Probabilistic early refresh / refresh-ahead**

Refresh before hard expiration, spreading fills over time. Avoid having every reader refresh early.

**Lease/lock per key**

One actor receives a short fill lease. Use expiration and, for correctness-sensitive writes, a fencing/version mechanism so a delayed old holder cannot overwrite a newer result.

Google Cloud CDN publicly documents request coalescing: multiple cache-fill requests for the same cache key are collapsed into one origin request per edge node. This is a verified product example of the general defense. [Google Cloud CDN: Request collapsing](https://docs.cloud.google.com/cdn/docs/caching#request_collapsing)

### 13.2 Reconnect storm

#### Failure

A gateway outage disconnects millions of devices. When DNS, network, or the gateway recovers, all reconnect, authenticate, restore session state, and request missed data together.

#### Defenses

- capped exponential reconnect backoff with full jitter;
- server-assigned “not before” time or reconnect slot;
- connection admission tokens at the edge;
- session resumption to reduce authentication cost;
- incremental state synchronization rather than full history replay;
- per-device deduplication and sequence numbers;
- priority for active sessions/emergency traffic; and
- slow traffic ramp on recovered gateways.

### 13.3 Periodic job herd

#### Failure

```text
00:00 --> billing + backups + reports + cache refresh + cleanup all start
```

#### Defenses

- hash each job/tenant to a stable offset;
- central scheduler with per-resource concurrency;
- leases so only one scheduler owns a job;
- bounded retry queues with jittered due times;
- backpressure from workers;
- priority and deadline separation; and
- catch-up limits after scheduler downtime.

Do not schedule every tenant at “midnight in its timezone” without considering large timezone clusters and daylight-saving changes.

### 13.4 Lock/waiter herd

#### Failure

Many threads wait for one mutex, socket, or resource. A broadcast wake-up makes all runnable even though only one can win. CPU is spent on scheduling and cache-line contention.

#### Defenses

- wake one waiter rather than all when semantics allow;
- semaphore with bounded permits;
- fair/queued lock;
- shard the protected state;
- eliminate the global lock;
- batch work under one holder; or
- use an event-generation counter so waiters determine whether relevant state actually changed.

This OS/concurrency form inspired the term, but HLD interviews more commonly explore retry, cache, reconnect, and scheduled-work forms.

### 13.5 Recovery/cold-start herd

#### Failure

Instances return healthy and immediately receive their full load. Their caches and connection pools are cold, so each request costs more than in steady state.

#### Defenses

- separate “process alive” from “ready for full traffic”;
- pre-open pools and load essential configuration;
- warm only justified hot data;
- load-balancer slow start;
- small canary share then ramp;
- retain stale cache where safe;
- rate-limit cache/database fill; and
- maintain spare recovery capacity.

### 13.6 Rate-limit reset herd

#### Failure

A fixed-window quota resets on a common clock boundary. Clients waiting for quota all resume at once.

#### Defenses

- token bucket or rolling window rather than a sharp global reset;
- per-client reset offsets;
- server-assigned retry times;
- queue/pacing for batch traffic; and
- a separate global capacity limit below the sum of per-client quotas.

## 14. Worked example: overloaded recommendation service

All numbers in this section are **illustrative**.

### 14.1 Initial system

```text
mobile clients --> API gateway --> Recommendation API --> model service
```

Assume:

- model service safe capacity: 1,000 attempts/second;
- normal original traffic: 800 operations/second;
- normal p95 latency: 150 ms;
- gateway timeout: 500 ms;
- gateway permits 2 retries after each timeout;
- retry delay: fixed 100 ms;
- queue: 2,000 requests;
- request is read-only/idempotent.

### 14.2 Disturbance

Half the model workers fail, reducing safe capacity to roughly 500 attempts/second. Original demand remains 800 operations/second.

```text
arrival 800/s > capacity 500/s
              |
              v
queue grows -> latency exceeds 500 ms -> gateway retries
              ^                         |
              |                         v
              +----- original still queued
```

With two retries, the upper bound is:

```text
800 logical operations/s * 3 attempts = 2,400 attempts/s
```

Actual attempt rate depends on timing and successes, but it can greatly exceed both current and original capacity. Fixed 100 ms retry delay creates repeated waves. The 2,000-item queue retains work that may already be outside its deadline.

### 14.3 Safer redesign

```text
mobile
  | no application retry for this optional panel
  v
gateway
  | 2 s end-to-end deadline
  | retry only one safe transport failure
  | full jitter, retry token required
  | global retry budget
  v
Recommendation API
  | adaptive concurrency limit
  | small bounded queue
  | reject expired work
  | circuit breaker for model service
  v
model service
  | overload rejection + pushback
  | cheaper fallback model/cached recommendations
```

Specific changes:

1. **Degrade instead of retrying optional work.** Return cached/popular recommendations if the personalized model is overloaded.
2. **Use a short queue.** Reject early before the deadline is consumed.
3. **One retry at most, only when useful.** A retry needs remaining deadline and a token.
4. **Use full jitter.** For example, choose within `0..100 ms` for the first retry; actual parameters come from latency data.
5. **Adaptive concurrency.** Reduce admitted in-flight calls when observed latency grows.
6. **Circuit breaker.** Stop most calls during clear dependency failure and allow a small number of recovery probes.
7. **Separate original and retry metrics.** Preserve capacity for fresh user operations.
8. **Cancel/expire obsolete work.** Do not run a request after the caller no longer needs it.

### 14.4 Why not just autoscale?

- model workers may take minutes to start;
- the failed dependency or GPU pool may have a hard limit;
- scaling from queue length can react after deadlines already fail;
- new workers may be cold; and
- retry traffic can fall after recovery, leaving expensive excess capacity.

Autoscaling remains useful, but the overload controls keep the service alive while it reacts.

## 15. Real-system examples

These examples deliberately distinguish public evidence from plausible application of the pattern.

### 15.1 Amazon/AWS — jitter and retry-token budgets

**Verified public behavior/pattern.** Amazon’s Builders’ Library explains timeouts, retries, capped exponential backoff, jitter, idempotency, and limiting retries at one layer. It describes using a token bucket in AWS SDK retry behavior so retries proceed while tokens remain and are rate-limited when tokens are exhausted. [Amazon Builders’ Library: Timeouts, retries, and backoff with jitter](https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/)

Current AWS SDK reference documentation separately describes:

- standard mode with exponential backoff, jitter, and retry quota; and
- adaptive mode with client-side rate limiting that can also delay initial attempts.

[AWS SDKs and Tools: Retry behavior](https://docs.aws.amazon.com/sdkref/latest/guide/feature-retry-behavior.html)

Interview takeaway:

```text
isolated transient failure -> spend retry budget
broad service failure      -> tokens drain, retries stop
recovery                   -> successes restore budget gradually
```

Do not copy AWS-specific defaults to an unrelated system. Copy the reasoning: classify, bound, randomize, and budget.

### 15.2 YouTube/Google — cache-fill protection and overload recovery

**Verified public facts:** Google says the majority of Google Global Cache traffic by bandwidth is cacheable YouTube content; GGC dynamically caches based on local demand, and cache-fill can come hierarchically from other GGC nodes or Google core locations. [Google Interconnect: Content Served](https://support.google.com/interconnect/answer/7658599?hl=en), [Google Interconnect: Cache-Fill](https://support.google.com/interconnect/answer/7658593?hl=en)

**Related verified mechanism, not a claim about YouTube’s private GGC implementation:** Google Cloud CDN collapses concurrent fills for the same cache key into one origin request per edge node. [Google Cloud CDN: Request collapsing](https://docs.cloud.google.com/cdn/docs/caching#request_collapsing)

**Illustrative YouTube application:** a newly viral video segment missing from one edge should not cause every viewer request to fetch an identical segment independently from the origin.

```text
viewers --> local edge miss --> one coalesced hierarchical fill --> upstream cache/core
              |                       |
              +-- followers wait/share completed cached segment
```

Further defenses include serving already cached segments, bounding fill concurrency, and routing to serving locations with available capacity. Do not state that YouTube uses a particular lock or jitter formula unless a source confirms it.

### 15.3 Uber — delayed reprocessing and controlled recovery probes

**Dated production example:** Uber Engineering described Kafka-based non-blocking reprocessing and dead-letter queues for its Driver Injury Protection pipeline, explicitly discussing prescribed retry limits/backoff and separating retry processing so it does not block real-time traffic. [Uber Engineering: Building Reliable Reprocessing and Dead Letter Queues with Apache Kafka](https://www.uber.com/blog/reliable-reprocessing/)

**Dated production example:** Uber’s 2020 mobile networking write-up described a failover state machine using failure thresholds, canary probes, a recovery state, and an increasing recovery timeout to avoid excessive switching while a primary domain is intermittently available. [Uber Engineering: Failover Handling in Mobile Networking](https://www.uber.com/blog/eng-failover-handling/)

These illustrate two herd defenses:

```text
failed async work -> delayed/reprocessing path -> bounded attempts -> DLQ

recovered primary -> controlled canary probes -> confidence -> traffic returns
```

### 15.4 Netflix — circuit breaking and bulkheads

**Historical/dating caveat:** Netflix’s public Hystrix design is no longer a current-technology recommendation, but it remains a clear public example of the concepts.

Hystrix checked whether the circuit was open, rejected work when an isolated thread pool/queue or semaphore was full, used fallbacks, and permitted a single request after a sleep window to test recovery. [Netflix/Hystrix: How It Works](https://github.com/Netflix/Hystrix/wiki/how-it-works)

```text
calls to dependency
     |
     +-- circuit open ------------> fallback
     +-- isolated pool full ------> fallback
     +-- allowed -----------------> dependency
                                      |
                                  rolling health stats
```

This protects the rest of the application from one slow dependency and avoids sending the full herd during obvious failure.

### 15.5 Twitter/X — expensive cache misses and event spikes

**Dated production example:** Twitter Engineering wrote that a cache miss for a Tweet read could trigger a cache fill that in turn triggered multiple backend reads, and that real-time events could produce very short traffic spikes. The article focused on reliability testing these expensive paths and service protections. [Twitter Engineering: Push Our Limits—Reliability Testing at Twitter](https://blog.x.com/engineering/en_us/a/2014/push-our-limits-reliability-testing-at-twitter)

Another dated architecture overview described Redis/Memcache as critical to protecting backing stores and serving tweets/timelines under event spikes. Exact figures in that historical post should not be treated as current. [Twitter Engineering: The Infrastructure Behind Twitter—Scale](https://blog.x.com/engineering/en_us/topics/infrastructure/2017/the-infrastructure-behind-twitter-scale)

**Illustrative mitigation for a viral Tweet:** coalesce the first fill, serve safe stale content, stagger expiration, protect the backing store with a fill-concurrency limit, and replicate/cache the resulting hot value closer to readers.

### 15.6 WhatsApp — reconnect-storm design exercise

**Illustrative design only.** I did not find an authoritative public source that establishes WhatsApp’s current reconnect/backoff implementation, so the following is not a claim about its private architecture.

After a regional network outage, a chat service may see millions of long-lived connections recover together:

```text
phones -- reconnect with capped full jitter --> connection gateways
                                                  |
                                           admission slots
                                                  |
                                         resume session by cursor
                                                  |
                                     fetch only missed messages
```

Important design points:

- stable message IDs make retransmission safe;
- sequence/cursor resume avoids downloading complete history;
- server-assigned retry windows prevent old clients from choosing unsafe timing;
- connection establishment and message replay need separate limits;
- active/urgent traffic may receive priority over background media sync; and
- restored gateways should receive gradually increasing traffic.

This is an excellent interview scenario because it combines reconnect herds, idempotency, ordering, backpressure, and recovery ramp-up.

## 16. Trade-offs, alternatives, and when to use or avoid them

| Technique | Helps with | Cost/risk | Use when | Avoid/rethink when |
|---|---|---|---|---|
| Capped exponential backoff | Reduces retry frequency during longer failures | Adds completion latency; still correlated without jitter | Transient remote failures | Operation is permanent/invalid or deadline is too short |
| Full jitter | Spreads synchronized attempts | Some clients retry near zero; nondeterministic timing | Large independent client population | Central scheduler already provides exact safe slots |
| Equal jitter | Guarantees meaningful minimum delay | Higher mean delay; AWS simulation found it worse than full jitter in its scenario | Near-zero retry is especially harmful | Latency budget is tight and full jitter is acceptable |
| Server `Retry-After` | Uses server knowledge | Identical value can create a later herd; clients may ignore it | Server can estimate recovery/policy | Server cannot provide a trustworthy horizon |
| Retry attempt cap | Bounds amplification | May give up before recovery | Almost every retry policy | Never omit it for an unbounded remote failure |
| Total retry deadline | Preserves user usefulness | Fewer recovery attempts | User/API work with an SLO | Long-running async jobs may need scheduled retries instead |
| Retry token bucket | Bounds aggregate retry load | Scope/fairness/tuning complexity | Shared client/dependency with spare-capacity budget | Used as a substitute for server admission control |
| Circuit breaker | Stops calls during obvious dependency failure | False trips; recovery probing/tuning | Dependency failures are correlated and fallback exists | Each call is independent and failures do not predict the next call |
| Bounded queue | Absorbs short bursts and caps memory | Rejection; queue can consume deadlines | Known burst profile | Used to hide sustained overload |
| Adaptive concurrency | Responds to latency/in-flight pressure | Feedback oscillation; algorithm complexity | Capacity varies or request time drives saturation | Work has unobservable/very delayed completion without another signal |
| Request coalescing | Eliminates duplicate identical work | Leader failure; follower latency; per-key state | Hot cache fill/read-through | Requests differ by authorization or result semantics |
| Serve stale | Protects origin and gives fast response | Staleness/correctness risk | Content tolerates bounded staleness | Payments, authorization, inventory, or safety facts require freshness |
| Delayed retry queue | Protects real-time path and preserves async work | Storage/backlog/ordering complexity | Background jobs/webhooks/events | Synchronous caller needs an immediate result |
| Load shedding | Keeps system alive | User-visible errors or degraded quality | Offered load exceeds safe capacity | Never “avoid” in principle; define which work may be shed |
| Autoscaling | Adds capacity for sustained load | Detection/start/warm-up delay; cost | Load persists long enough | Sole defense against a sub-second retry wave |
| Hedged request | Can reduce tail latency by racing a duplicate | Deliberately adds load; can worsen overload | Rare tail requests with strict budget and cancellation | Service is already saturated or requests have side effects |

### 16.1 Backoff versus queuing

- **Client backoff** leaves work outside the server and spreads it, but completion is less predictable and clients must cooperate.
- **Server queueing** gives the server ordering/control, but consumes server resources and can retain expired work.
- **Delayed durable queueing** suits asynchronous jobs, but changes the interaction contract.

### 16.2 Fail open versus fail closed

If the limiter/controller is unavailable:

- **fail open** preserves availability but may overload the protected resource;
- **fail closed** protects capacity/cost/correctness but rejects healthy requests;
- **local conservative fallback** admits a limited amount without central state.

Choose per operation. A payment deduplication check and an optional recommendation refresh should not necessarily share the same failure policy.

### 16.3 Randomness versus fairness

Jitter reduces global synchronization, but random winners can repeatedly get earlier slots. For queued enterprise/batch work requiring fairness, a central scheduler with weighted fair queuing may be better than independent random retries.

## 17. Interview angle

### 17.1 How the topic appears

An interviewer may never say “thundering herd.” It appears as a follow-up:

- “What happens when this dependency goes down for two minutes?”
- “What if one million clients reconnect after the outage?”

## 18. Common mistakes and interviewer red flags

1. **“Use exponential backoff” as the whole answer.** It does not bound attempts, break synchronization by itself, protect the queue, or make writes safe.
2. **Putting the cap outside the random function.** `min(cap, Uniform(0, huge))` creates a probability spike exactly at the cap.
3. **Not defining whether `n` counts failures or retries.** Off-by-one behavior changes delays and tests.
4. **Retrying every error.** Permanent validation/authentication failures do not improve with time.
5. **Retrying a timeout as if nothing happened.** The first attempt may have committed.
6. **Retries at every layer.** Attempt counts multiply rather than add.
7. **No overall deadline.** Each attempt starts a fresh timeout and violates the user’s latency budget.
8. **No retry budget.** A widespread outage consumes all capacity with repeat work.
9. **Fixed backoff with tiny jitter.** A narrow random band leaves a large peak.
10. **Trusting clients to protect the service.** Old, buggy, or malicious clients ignore backoff; enforce server-side admission too.
11. **Using an enormous queue as “reliability.”** It can convert fast failures into timeouts and memory pressure.
12. **Processing expired queued work.** The caller has left, so the work produces no value.
13. **Returning `429` for every overload.** `429` commonly expresses a client-specific rate policy; global temporary incapacity is often `503`. State the semantics.
14. **Giving the same `Retry-After` to everyone without spreading.** It can schedule the next herd.
15. **Circuit breaker half-open for every caller.** Recovery probes themselves become a herd.
16. **One local retry bucket per instance without fleet math.** A thousand instances multiply retry permission.
17. **A global retry bucket with no tenant isolation.** One failing tenant can consume everyone’s reliability budget.
18. **Inferring capacity from CPU or free memory alone.** The actual bottleneck may be elsewhere.
19. **An unstable adaptive controller.** Delayed/noisy feedback can oscillate between over-admission and underutilization.
20. **Immediate requeue to the same queue.** A failing message can hot-loop or block fresh work.
21. **No idempotency retention analysis.** If deduplication expires before the longest retry, a late retry can repeat the effect.
22. **Autoscaling as the first and only fix.** The herd often arrives faster than scale-up and can kill cold instances.
23. **Monitoring only final success rate.** Successful retries can hide an expensive retry storm.
24. **Conflating cache stampede and hot key.** A hot key is continuously popular; a stampede is a synchronized miss/fill event. They can occur together.
25. **Conflating rate and concurrency.** A safe RPS at 50 ms latency may be unsafe at 5 seconds.

## 19. Observability: how to know a herd is happening

### 19.1 Traffic and retry metrics

- logical operations/second;
- initial attempts/second;
- retries/second by attempt number;
- `attempts / logical_operation` distribution;
- retry-budget token balance and denials;
- success after retry versus success on first attempt;
- errors by retryable/non-retryable classification;
- received `Retry-After`/pushback and compliance; and
- retry traffic by caller, tenant, operation, and dependency.

A rising retry ratio while original demand stays flat is a strong clue.

### 19.2 Saturation and queue metrics

- in-flight requests and concurrency limit;
- queue depth **and oldest age**;
- estimated queue wait;
- worker/thread/event-loop utilization;
- connection-pool wait and exhaustion;
- CPU, memory, garbage collection, sockets, file descriptors;
- rejected/shed requests by reason and priority; and
- expired/deadline-exceeded requests before and after execution.

### 19.3 Latency and abandoned-work metrics

- end-to-end latency by attempt;
- dependency latency;
- timeout rate;
- cancellation propagation success;
- work completed after caller deadline;
- duplicate/idempotency-key hits; and
- original and retry overlap for the same logical operation.

### 19.4 Cache and scheduler herd metrics

- cache hits, misses, fills, and coalesced followers per key;
- concurrent fill count and fill latency;
- origin QPS on cache expiry;
- TTL expiration distribution;
- jobs due per time bucket;
- retry queue depth and oldest age;
- DLQ inflow; and
- reconnect attempts/successes by backoff stage.

### 19.5 Look for correlated waves

The signature is often periodic:

```text
error spike -> retry spike -> latency spike -> error spike -> retry spike
```

Graph original traffic and retry traffic separately on the same time axis. Aggregate-minute dashboards can hide sub-second bursts, so choose resolution appropriate to the retry windows.

## 20. Testing and operational readiness

### 20.1 Load-test beyond the average

Test:

- a sudden burst, not only steady RPS;
- 10%, 50%, and 90% capacity loss;
- latency increase without explicit errors;
- one hot key/partition/tenant;
- empty cache/cold restart;
- retry traffic from many instances;
- full queue and full connection pool;
- recovery while a backlog exists; and
- synchronized scheduled jobs.

### 20.2 Fault injection scenarios

```text
dependency returns 503 + Retry-After
dependency silently slows beyond client timeout
response is lost after database commit
cache key expires under peak traffic
all mobile connections drop and reconnect
circuit moves to half-open across many clients
retry state store becomes unavailable
```

Validate both correctness and offered load.

### 20.3 Invariants worth asserting

- retry attempts never exceed the configured count;
- total elapsed time never exceeds the logical deadline materially;
- no attempt occurs before a mandatory server pushback time;
- non-idempotent logical operations cause at most one committed effect;
- queue memory remains bounded;
- retry traffic cannot consume reserved original-traffic capacity;
- half-open probes remain bounded;
- expired work does not begin expensive execution; and
- recovered instances ramp rather than immediately taking full weight.

### 20.4 Operational controls

Have runtime controls for:

- disabling or reducing retries without a full deployment;
- changing retry budget and concurrency limit;
- forcing a circuit open;
- changing traffic priority/shedding policy;
- pausing retry queues;
- draining backlog at a chosen rate; and
- observing the effect safely.

Configuration rollout itself should be staggered; changing every client to the same new schedule simultaneously can create the correlation you are trying to avoid.

## 21. A compact design recipe

When adding retries to any call, walk through this checklist:

### Semantics

- [ ] What logical operation is being attempted?
- [ ] Is it naturally idempotent, protected by an idempotency key, or unsafe to retry?
- [ ] Which exact errors are transient for this API?
- [ ] Can a timeout hide a committed side effect?

### Time and count

- [ ] What is the end-to-end deadline?
- [ ] Which one layer owns retries?
- [ ] What are maximum total attempts and maximum elapsed retry time?
- [ ] Is there enough remaining time for the next sleep plus attempt?

### Spreading and pushback

- [ ] Is backoff capped?
- [ ] Is meaningful full/decorrelated jitter applied?
- [ ] Is `Retry-After` or protocol pushback honored?
- [ ] Could a common cap/reset time create a new spike?

### Capacity protection

- [ ] Is retry traffic bounded by a budget/token bucket?
- [ ] Is the queue bounded and deadline-aware?
- [ ] Is concurrency bounded near the constrained resource?
- [ ] Are circuit breaker, bulkhead, priority, and load shedding defined?

### Recovery and operations

- [ ] Are half-open probes and restored traffic ramped?
- [ ] Is retry backlog drained at a controlled rate?
- [ ] Are originals and retries measured separately?
- [ ] Can operators reduce retries quickly during an incident?

## 22. Compact revision sheet

### 22.1 One mental model

```text
Thundering herd = many actors + correlated trigger + scarce resource

Control it by:
reduce actors + spread timing + protect bottleneck + make repeats safe + ramp recovery
```

### 22.2 Formula sheet

```text
# Capped exponential scale, k=0 for first retry
cap(k) = min(max_backoff, base * 2^k)

# Full jitter
sleep(k) = Uniform(0, cap(k))

# Equal jitter
sleep(k) = cap(k)/2 + Uniform(0, cap(k)/2)

# One decorrelated-jitter form
sleep_next = min(max_backoff, Uniform(base, 3 * sleep_previous))

# Attempt traffic
attempt_rate = logical_operation_rate * average_attempts

# Simplified concurrency relationship
average_in_flight ~= throughput * average_latency

# Multi-layer worst-case amplification
lowest_level_attempts_per_operation <= product(attempts_allowed_at_each_retrying_layer)
```

### 22.3 Mechanism cues

| Mechanism | Cue |
|---|---|
| Backoff | “Try less frequently as failure continues.” |
| Jitter | “Do not let identical clients share a retry clock.” |
| Attempt/deadline cap | “Reliability work is finite and must remain useful.” |
| Retry budget | “Retries may spend only bounded spare capacity.” |
| Bounded queue | “A burst buffer, not a throughput generator.” |
| Concurrency limit | “Bound pressure caused by slow in-flight work.” |
| Circuit breaker | “Stop calling a dependency that is predictably failing.” |
| Bulkhead | “One dependency or tenant cannot consume every permit.” |
| Coalescing | “Many identical misses share one fill.” |
| Idempotency key | “Many attempts, at most one logical effect.” |
| Slow start | “Recovery capacity is earned gradually.” |

### 22.4 Thirty-second answer spine cue

```text
trigger -> herd -> bottleneck -> feedback loop
client: classify/cap/deadline/jitter/budget
server: bounded queue/concurrency/shed/circuit
correctness: idempotency
recovery: probes + ramp
observe: originals vs retries + queue age
```

## 24. Say-out-loud 30-second narrative

> “A thundering herd happens when many clients or workers react to the same timeout, expiry, reconnect, or recovery event and hit one bottleneck together. I first bound retries by error type, idempotency, attempt count, total deadline, and a retry budget; then I use capped exponential backoff with meaningful jitter to break synchronization.
