# B+ Trees and Database Indexing

> Beginner-first HLD notes expanded from: **headers, data pages, offsets/slot arrays, rows per page, storage sectors and blocks, B+ tree order, clustered and secondary indexes, page splits, adding columns/indexes later, metadata, page-to-block mapping, and safe query cancellation**.
>
> The central interview lesson is: **an index is a separate, page-oriented access path maintained by the database. A B+ tree keeps its keys ordered and its height small so that an exact lookup or the start of a range takes only a few page visits. The leaf payload depends on the storage engine: it may be the whole row, a `(page, slot)` row locator, or the clustered primary key.**

## Scope, assumptions, and evidence labels

The bullets mix several abstraction layers and several database-engine designs. This chapter first builds one generic mental model, then uses these concrete engines to show where implementations differ:

- **PostgreSQL heap tables** for a page with an item-identifier array and index entries that normally reference heap tuples.
- **MySQL InnoDB** for a table whose clustered-index leaves hold the row and whose secondary-index leaves hold the primary key.
- **SQL Server rowstore** for especially explicit documentation of pages, slot arrays, clustered leaves, and heap row IDs.

When vendor documentation says **B-tree**, its implementation commonly has the database-index properties usually taught as a **B+ tree**: data/row locators at the leaf level, a shallow balanced structure, and linked leaf pages. Do not win an interview argument over the product's naming; describe the actual leaf payload and scan behavior.

Evidence labels used below:

- **Verified implementation** - behavior supported by first-party product documentation linked nearby.
- **Generic model** - a portable conceptual model; exact fields and algorithms vary by engine/version.
- **Illustrative number** - arithmetic chosen to explain mechanics, not a production measurement or universal limit.
- **Illustrative design** - a plausible design for a familiar product, not a claim about that company's private current architecture.
- **Self-reported interview experience** - a candidate's public account, not an official or independently verified company question bank.

### The most important corrections up front

1. **A database page, an OS virtual-memory page, a filesystem block, a device logical sector, and an SSD NAND page are different units.** They can happen to have related sizes, but they belong to different layers.
2. **“Block” is overloaded.** PostgreSQL often calls its fixed-size database pages blocks; a filesystem and a storage device also use the word for different units. Always qualify it.
3. **A row locator is not simply a random row number.** A heap-oriented engine can use `(file/relation, page number, slot number)`. The slot resolves through a page directory to the row's current byte offset. InnoDB secondary indexes instead use the primary-key value to find the clustered row.
4. **“Values only in B+ tree leaves” does not imply a second copy of the entire table.** A clustered/index-organized tree's leaf level *is* the table. A secondary tree copies only its indexed keys, locator, and optional included columns.
5. **Only one clustered ordering can be the table's canonical row organization.** A second offset array cannot make the same bytes simultaneously ordered by a different key. Additional access paths are secondary indexes or separate materialized copies.
6. **Adding a column does not always update every old data page immediately.** Modern engines can represent some additions using catalog metadata and materialize the value later. Other alterations still require a scan or rewrite.
7. **Canceling a modifying query does not mean the database never touched memory or disk.** It means uncommitted effects are not made logically visible. WAL/redo, undo or MVCC, and transaction state preserve atomicity and crash recovery.
8. **An index is not automatically faster.** For a query returning much of a table, a sequential scan can be cheaper than thousands of scattered row lookups through an index.

## 1. Plain-language foundation

### 1.1 The problem an index solves

Imagine one million paper customer forms in boxes. To find customer `42`, there are two broad choices:

1. Open forms one by one until the right one appears.
2. Maintain a sorted directory saying approximately where each customer is stored.

A database without a usable index may do the equivalent of the first choice: a **table scan**. An index provides the second access path.

```text
Table/data pages                         Index pages
-------------------------------          ---------------------------
page 0: rows 901, 17, 444 ...            sorted key -> row location
page 1: rows 81, 42, 712 ...             17 -> (page 0, slot 1)
page 2: rows 5, 66, 100 ...              42 -> (page 1, slot 1)
                                         66 -> (page 2, slot 1)
```

The index costs extra storage and extra work on writes, but it can avoid inspecting almost every row.

### 1.2 Why the database groups bytes into pages

Storage is not normally fetched one field at a time. The database manages fixed-size chunks called **pages** (also called database blocks in some products). A page is the unit it can identify, cache, latch, checksum, log, evict, and commonly read/write.

This changes how to think about algorithm cost:

```text
CPU-only textbook question:     How many key comparisons?
Database-internals question:    How many page visits, and are they cached or random I/O?
```

A binary tree has only two children per node, so it can be tall. A B+ tree packs many separator keys and child pointers into each page. Its large **fan-out** keeps the tree shallow.

### 1.3 One query, from SQL to bytes

```text
SELECT * FROM orders WHERE order_id = 9001;
                    |
                    v
          parser + planner/optimizer
                    |
        consult schema, indexes, statistics
                    |
       choose table scan or index lookup
                    |
           visit B+ tree pages
          root -> internal -> leaf
                    |
         obtain row or row locator
                    |
        fetch database data page if needed
                    |
     decode row using table/row metadata
                    |
             return logical columns
```

The SQL name `orders` is a logical name. It is not itself a disk address. Catalog and allocation metadata connect that name to stored objects and pages.

### 1.4 Inline prerequisites: four tiny concepts

These are required for the rest of the chapter, so they are defined here.

- A **key** is the value by which an index is ordered, such as `order_id` or `(conversation_id, sent_at)`.
- A **pointer/locator** tells the engine where to continue: another index page, a heap row, or a clustered key.
- A **transaction** groups database operations into one logical unit that commits or rolls back.
- A **cache/buffer pool** keeps copies of database pages in RAM so repeated access need not reach storage.

## 2. Pages, headers, rows, slots, and offsets

### 2.1 First separate all the “page/block” layers

| Layer | Unit | What it means | Who manages it |
|---|---|---|---|
| Database file format | **Database page / DB block** | Fixed-size formatted unit containing rows, index entries, or metadata | DBMS/storage engine |
| Database RAM cache | **Buffer frame** | RAM slot that holds one database page in the usual buffer-pool design | DBMS |
| Virtual memory / page cache | **OS page or folio** | Unit used by the kernel's memory and file cache; modern kernels can group pages in larger folios | OS kernel |
| Filesystem | **Filesystem block** | Allocation and/or I/O-mapping unit within a file | Filesystem |
| Block device interface | **Logical block/sector (LBA)** | Addressable range exposed by a disk/SSD | Device + OS block layer |
| SSD internals | **NAND page / erase block** | Flash programming and erase units hidden behind the controller | SSD firmware/media |

Concrete product examples, not universal defaults:

- PostgreSQL is normally built with an **8 KiB** database-page size; its page-layout documentation describes a **24-byte page header** and four-byte item identifiers. [PostgreSQL database page layout](https://www.postgresql.org/docs/current/storage-page-layout.html)
- InnoDB's default index-page size is **16 KiB**, selected when the instance is initialized. [InnoDB index physical structure](https://dev.mysql.com/doc/refman/8.4/en/innodb-physical-structure.html)
- SQL Server uses **8 KiB** data pages and documents eight-page, 64 KiB extents. [SQL Server page and extent architecture](https://learn.microsoft.com/en-us/sql/relational-databases/pages-and-extents-architecture-guide)

The OS base page is often 4 KiB on common machines, but that is **environment-dependent—verify the host rather than assuming it**. Database and OS page sizes do not need to match.

### 2.2 Generic slotted data-page layout

Variable-length rows create a problem: if row 2 grows, fixed byte positions for all later rows would change. A **slotted page** adds a level of indirection.

```text
Low address                                                     High address
+----------------------+-------------------+-------------------+---------------+
| Page header          | Slot/item array   |    free space     | row bytes     |
| page id/type         | slot 0 -> offset -|------------------>| R0            |
| free-space bounds    | slot 1 -> offset -|------------+      | R2            |
| checksum/LSN/...     | slot 2 -> offset -|-----+      |      | R1            |
+----------------------+-------------------+-----|------|------+---------------+
                                              rows can be packed/moved here
```

The exact direction in which the slot directory and row area grow is engine-specific. The reusable idea is:

```text
stable logical slot number -> current byte offset -> row bytes
```

PostgreSQL calls the entries **item identifiers**. Each records a byte offset, byte length, and status bits. The item identifier can remain in place even when the row bytes move during page compaction, so a PostgreSQL item pointer/`CTID` is based on `(page number, item-identifier index)`, not a permanent raw byte address. [PostgreSQL database page layout](https://www.postgresql.org/docs/current/storage-page-layout.html)

SQL Server's documented slot-array entry stores the byte offset of its row relative to the start of the page. [SQL Server page and extent architecture](https://learn.microsoft.com/en-us/sql/relational-databases/pages-and-extents-architecture-guide)

### 2.3 What can be in the header?

A **generic** page header can contain:

- page type: heap data, index internal, index leaf, overflow, allocation metadata, and so on;
- page identity and format/version;
- lower/upper free-space boundaries;
- number or boundary of slot entries;
- checksum;
- log sequence number or related recovery information;
- flags and transaction-cleanup hints;
- sibling-page links or tree-level information for an index page.

Do not confuse these three metadata locations:

```text
Catalog metadata     “Table orders has columns id, status, created_at.”
Page header          “This is page 418; free space starts here; page LSN is X.”
Row header           “This row version was created/deleted by these transactions.”
```

An engine does not have to put every item above in every page header. For example, PostgreSQL reserves an access-method-specific special area, which B-tree pages use for data such as sibling links.

### 2.4 Does the offset point to a random row number?

Short answer: **no**. Three values are often collapsed into the word “offset”:

```text
Database page number      Which formatted page?
Slot/item number          Which directory entry inside that page?
Byte offset               Where do this row's bytes currently begin?
```

A heap row identifier can conceptually be:

```text
RID = (file_or_relation_id, page_id, slot_id)
```

Then the engine loads the page and evaluates:

```text
page.slot_array[slot_id] -> byte offset -> row header + row fields
```

The slot is not necessarily “the 57th logical row in the table” and does not imply insertion order. Slots can be reused; deleted rows leave reclaimable space; row versions can move or be forwarded depending on the engine. Treat physical row locators as implementation details unless the database explicitly promises stability.

### 2.5 How many database rows fit on one page?

There is no single answer. At a first approximation:

```text
rows_per_page ≈ floor(
    usable_page_bytes
    / (average_inline_row_bytes + slot_bytes + per-row overhead)
)
```

Factors that change it:

- fixed versus variable-width columns;
- null bitmap and variable-column offset table;
- row/MVCC header;
- alignment and padding;
- page header/trailer or special area;
- reserved free space/fill factor;
- compression;
- off-row large values;
- updates that leave dead/old row versions;
- the actual distribution of `VARCHAR` values, not merely declared maxima.

**Illustrative arithmetic:** with an 8,192-byte page, a 24-byte page header, and an assumed 100 bytes of row-plus-row-header plus a four-byte slot per row:

```text
floor((8192 - 24) / (100 + 4)) = 78 rows
```

That is a teaching estimate, **not a PostgreSQL guarantee**. It ignores alignment, fill factor, dead tuples, special space, and variable row sizes.

Very large values need not fit fully inline. Engines can use overflow/LOB pages; PostgreSQL can move large values to a TOAST relation, for example. Consequently, the logical row size and inline bytes on its main data page can differ.

### 2.6 Why headers and slot arrays are worth the overhead

| Benefit | Why it matters |
|---|---|
| Row movement within a page | Compaction need not update every external row locator if the slot remains stable |
| Variable-length records | Rows do not require fixed global offsets |
| Free-space tracking | Inserts can find a page with room |
| Recovery/checks | Page LSN/checksum information helps detect or recover damaged/incomplete state |
| Concurrency | Page identity and access-method fields support safe latching and navigation |

The trade-off is a few bytes of metadata per page/row and more complex storage-engine code.

## 3. How table `T` becomes a file, page, buffer, and device request

### 3.1 The database does not remember every row's disk sector in the table schema

Instead, multiple mappings compose:

```text
SQL name T
  |
  | catalog lookup
  v
internal table/relation ID + storage object/tablespace/partition IDs
  |
  | access method + allocation metadata
  v
database page number(s)
  |
  | database file + byte offset
  v
filesystem logical blocks/extents
  |
  | filesystem/device mapping
  v
logical block addresses exposed by device
  |
  | SSD controller's flash translation layer, if SSD
  v
physical media locations
```

The lower layers may remap locations without changing the logical table or database page number.

### 3.2 How the engine decides which page of `T` to read

It depends on the chosen access path.

**Table/heap scan**

```text
catalog says where T's storage begins
        -> allocation/segment metadata enumerates its pages
        -> scan page 0, page 1, page 2, ...
        -> test each visible row against the predicate
```

**B+ tree lookup**

```text
catalog says index I belongs to T and gives I's root
        -> root separator selects child page
        -> internal separator selects leaf page
        -> leaf yields row / heap RID / clustered key
        -> fetch base row if the leaf did not cover the query
```

**Insert into a heap**

```text
free-space metadata suggests a T page with enough room
        -> validate current free space
        -> insert row, or allocate a new page
```

PostgreSQL exposes a free-space map by relation and block number; its values are approximate, which is enough to guide placement without maintaining an expensive perfectly current directory. [PostgreSQL `pg_freespacemap`](https://www.postgresql.org/docs/current/pgfreespacemap.html)

### 3.3 Page number to file offset

In a simple unsegmented file model:

```text
byte_offset = database_page_number * database_page_size
```

Real systems add tablespaces, partitions, relation forks, multi-file segments, extents, and allocation maps. The principle remains: the database has a stable logical page identity and translates it to a file/range. PostgreSQL stores tables and indexes as relations, normally in separate files, and uses catalog identifiers plus relation file nodes; its storage layout also includes free-space and visibility-map forks. [PostgreSQL database file layout](https://www.postgresql.org/docs/current/storage-file-layout.html)

### 3.4 Database page to filesystem block to device sector

Suppose only for illustration:

```text
DB page:              16 KiB
filesystem block:      4 KiB
device logical block:  4 KiB

One aligned DB page spans four filesystem/device logical blocks.
```

But no rule says those four logical blocks must be neighboring NAND cells. A filesystem can fragment a file; RAID/LVM/cloud storage can add mapping; an SSD controller remaps LBAs for wear leveling and garbage collection.

Terminology worth saying precisely:

- A **database block** may be exactly synonymous with database page in one engine.
- A **filesystem block** maps part of a file to storage.
- A **device sector/logical block** is an address exposed by the device.
- A **NAND erase block** is a much lower-level flash unit and is not the same as any of the above.

### 3.5 Database page versus memory page: same or different size?

Usually a database's own buffer pool allocates a **buffer frame sized to hold one database page** (plus in-memory bookkeeping). That makes page identity, latching, dirty tracking, and replacement straightforward.

The OS page cache is another layer. A database page can span multiple OS pages, or a larger OS folio can contain the file ranges of multiple database pages. Some databases use buffered I/O and therefore benefit from the OS page cache; others use direct I/O in some configurations to reduce duplicate caching. There is no universal “best” size relationship.

| Choice | Advantage | Cost/risk |
|---|---|---|
| Larger DB page | More rows/keys per I/O, higher B+ fan-out, good sequential throughput | More read/write amplification for tiny random accesses; larger latch/cache unit; more damage per torn/corrupt page |
| Smaller DB page | Finer-grained random I/O/cache and less unused data fetched | More headers, deeper/wider metadata structures, potentially more I/O requests |
| DB page aligned to underlying I/O units | Simpler/full aligned I/O | Underlying stack still has multiple changing units; alignment alone does not guarantee atomicity |

Workload, row width, device behavior, compression, concurrency, and engine recovery design decide what is best. This is normally an engine/build/instance decision, not a per-query knob.

### 3.6 If one memory unit contains five DB pages, does changing one force all five to storage?

There are three answers at three layers:

1. **Database buffer pool:** normally the one changed DB page is marked dirty. The database need not logically dirty the other four.
2. **OS page cache/folio:** dirty state may be tracked at OS-page, filesystem-block, or sub-folio granularity depending on the I/O path. A larger cached unit does not mean every byte became logically modified. Linux's `iomap`, for example, can track per-filesystem-block dirty state when a page-cache folio is larger than a filesystem block. [Linux `iomap` buffered-I/O state](https://docs.kernel.org/next/filesystems/iomap/operations.html)
3. **SSD internals:** small host writes can cause controller-level read/modify/program, garbage collection, and write amplification. NAND is written in pages and erased in larger blocks, so firmware may move still-live data before erasing. [Micron NAND organization overview](https://in.micron.com/content/dam/micron/global/public/products/technical-marketing-brief/client-vs-enterprise-performance-use-cases-tech-brief.pdf)

So: **no, database correctness does not require rewriting five DB pages merely because a larger memory container holds them.** But the lower storage stack may perform more physical work than the changed bytes suggest. That can affect SSD endurance.

Databases mitigate the cost with buffering, group commit, sequential WAL/redo, checkpoints, page alignment, and engine/device-specific protections. Durability also requires correct cache flush/FUA behavior; a device can otherwise acknowledge a volatile cached write before it reaches nonvolatile media. [Linux write-back cache control](https://kernel.org/doc/html/v6.0/block/writeback_cache_control.html)

## 4. B+ tree from first principles

### 4.1 Why not just a sorted array?

A sorted array is excellent for reads:

- binary search finds one key in `O(log N)` comparisons;
- adjacent entries make range scans easy.

But inserting in the middle may shift a large suffix. A large on-disk file cannot cheaply move half its contents for every random insert.

A B+ tree keeps **sorted pages**, not one giant contiguous array. When one page fills, the engine splits a bounded page and changes a small path of parent pages.

### 4.2 Why not a binary search tree or AVL/red-black tree?

Binary trees optimize CPU comparisons, but a node with two children creates many levels. If each node access can mean storage I/O, height dominates.

```text
Binary tree:                       B+ tree page:
          50                       [10 | 20 | 35 | 50 | 72 | 90]
        /    \                      /    |    |    |    |    |   \
      25      75                  many child-page ranges
     / \      / \

few children per node             hundreds of children may fit per page
greater height                    very small height
```

The B+ tree is balanced: all leaves remain at the same depth. An exact lookup follows one root-to-leaf path; a range lookup follows one path to the first key and then walks linked leaf pages.

### 4.3 Anatomy

```text
                         ROOT (index page)
                    [ 30 | 60 | 90 ]
                   /      |      |      \
                  v       v      v       v
             internal  internal internal internal
               pages     pages    pages    pages
                  |        |        |        |
                  v        v        v        v
LEAF CHAIN: [1..29] <-> [30..59] <-> [60..89] <-> [90..]
              |             |             |            |
          row payload   row payload   row payload  row payload
```

- **Root/internal entries:** separator key plus child-page pointer. A separator guides search; it need not represent a separate table row stored there.
- **Leaf entries:** every searchable key plus its payload.
- **Sibling links:** let range scans continue without returning to the root.
- **Balanced depth:** every search reaches a leaf in the same number of tree levels.

PostgreSQL documents B-tree indexes as multilevel structures whose pages at each level form doubly linked lists. [PostgreSQL B-tree implementation](https://www.postgresql.org/docs/current/btree.html)

### 4.4 B-tree versus B+ tree

Textbooks use the names more strictly than vendors.

| Property | Textbook B-tree | Textbook B+ tree |
|---|---|---|
| Record payload | May appear in internal and leaf nodes | Only at leaves |
| Internal nodes | Search keys and possibly records | Separator keys + child pointers |
| Leaf linkage | Not essential to definition | Usually linked for ordered scan |
| Range scan | Possible, but traversal is less leaf-linear | Find first leaf, then scan sibling chain |
| Fan-out | Internal record payload can reduce it | Internal nodes stay compact, increasing fan-out |

In an interview, after giving the distinction, add:

> “Vendors often call their structure a B-tree even when its leaf-oriented, linked-page behavior matches what textbooks call B+ tree. I would verify the selected engine's leaf payload.”

### 4.5 What does order `M` mean?

**Warning: definitions differ across books and courses. State your convention before calculating.**

One common convention:

- order `M` = maximum number of children in an internal node;
- therefore maximum internal separator keys = `M - 1`;
- non-root internal nodes have at least `ceil(M / 2)` children;
- root has 2 to `M` children when it is not also a leaf;
- all leaves are at the same depth;
- leaf capacity may be defined separately as `L` entries, usually at least roughly half full under the textbook invariant.

Another convention uses `d` as minimum degree and allows `d..2d` entries. Berkeley's CS 186 notes use that convention, while University of Washington material gives an `M`-as-maximum-fan-out convention. [Berkeley CS 186 B+ tree notes](https://cs186berkeley.net/notes/note4/) and [University of Washington CSE 373 B+ tree properties](https://courses.cs.washington.edu/courses/cse373/15wi/lectures/lecture15.pdf)

Real engines calculate capacity from bytes rather than an abstract classroom `M`:

```text
fan_out ≈ floor(
    usable_internal_page_bytes
    / (average_separator_bytes + child_pointer_bytes + entry_overhead)
)
```

Variable key length, prefix/truncation compression, page headers, fill factor, and implementation details all matter.

**Illustrative height intuition:** if fan-out were 500, three branching levels could distinguish on the order of `500^3 = 125,000,000` ranges. This is only intuition—real occupancy and leaf capacity differ—but it explains why database trees are often shallow.

Increasing `M`/fan-out usually:

- decreases or preserves height;
- increases the number of keys searched inside one page;
- requires a larger page or smaller entries;
- can increase the bytes read/latch contention per page if achieved by increasing page size.

There is no free “increase `M`” knob independent of page and key size.

### 4.6 “Only leaves contain values”—is that a copy of the whole table?

“Value” means the **leaf payload**, and there are three important models.

#### Model A: leaf holds the full row

```text
clustered leaf entry:
(order_id=9001) -> [all inline columns for order 9001]
```

The leaf level **is the table's primary row storage**. It is not an extra full copy beside another identical heap. InnoDB's clustered index behaves this way: each InnoDB table has one clustered index, normally the primary key, and its leaf path reaches the row data. [InnoDB clustered and secondary indexes](https://dev.mysql.com/doc/refman/8.4/en/innodb-index-types.html)

#### Model B: leaf holds key + heap row locator

```text
secondary/heap index leaf:
(email='a@x.com') -> (heap page 418, slot 7)
```

The table rows remain in heap/data pages. The index copies the indexed key and locator, not every table column. PostgreSQL B-tree indexes normally work in this family.

#### Model C: leaf holds key + clustered key

```text
InnoDB secondary leaf:
(status='PAID') -> primary_key=9001
```

The engine searches the secondary tree, then searches the clustered tree using `9001`. InnoDB explicitly stores primary-key columns in each secondary-index record. A long primary key therefore makes every secondary index wider. [InnoDB clustered and secondary indexes](https://dev.mysql.com/doc/refman/8.4/en/innodb-index-types.html)

An index can additionally contain **included/covering columns** so a query can be answered from the index alone. That intentionally duplicates selected values to save base-table lookups.

Internal separator keys do repeat a small amount of key information already represented at the leaves, but they do **not** repeat each leaf's full row payload. That small duplication is what lets an internal page route a search while staying compact.

### 4.7 Exact and range lookups

```sql
-- Exact
SELECT * FROM orders WHERE order_id = 9001;

-- Range
SELECT * FROM orders
WHERE created_at >= '2026-07-01'
  AND created_at <  '2026-08-01'
ORDER BY created_at;
```

Exact search:

```text
root -> internal -> target leaf -> row/locator
```

Range search:

```text
root -> internal -> first matching leaf
                            |
                            v
                  leaf -> leaf -> leaf until upper bound
```

That second behavior is why ordered B+ trees support equality, ranges, prefix order, `MIN`/`MAX`, and compatible `ORDER BY` well.

## 5. Inserts, database page splits, deletes, and concurrency

### 5.1 A leaf-page split step by step

Suppose one leaf can hold four entries.

```text
Before inserting 35:

parent:       [50]
             /    \
leaf A: [10 20 30 40] <-> leaf B: [50 60 70]

Insert 35 into A -> overflow:

temporary: [10 20 30 35 40]

Allocate and redistribute:

leaf A: [10 20] <-> new leaf C: [30 35 40] <-> leaf B: [50 60 70]

Copy separator into parent:

parent:       [30 | 50]
             /    |    \
        leaf A  leaf C  leaf B
```

Generic split work:

1. Traverse to the target leaf.
2. Acquire the required short-lived page **latches**.
3. Allocate a new page.
4. Redistribute entries in key order.
5. Repair sibling links.
6. Insert a separator and child pointer into the parent.
7. If the parent overflows, split upward.
8. If the root splits, allocate a new root; tree height grows by one.
9. Generate recovery log records as required.

A **latch** protects in-memory data structures for a very short critical section. A transaction **lock** protects logical rows/keys/ranges across transaction work. Interviewers often test this distinction.

### 5.2 Why splitting only a page is better than shifting an array

The work is bounded around a root-to-leaf path and a few fixed-size pages. It does not move every subsequent table entry. Worst-case insertion remains `O(log_M N)` tree levels, although page allocation, WAL, contention, and I/O latency matter in practice.

### 5.3 Split trade-offs and insert patterns

| Pattern | Behavior |
|---|---|
| Monotonically increasing key | Inserts concentrate at the rightmost leaf; good locality and typically high fill, but the right edge can become a concurrency hotspot |
| Random key, such as random UUID | Inserts spread across leaves; less single-right-edge concentration, but more random I/O and more page splits/fragmentation |
| Lower fill factor | More space and index pages now, fewer near-future splits |
| Higher fill factor | Better density/read cache efficiency now, less headroom for random inserts |

For one concrete implementation, InnoDB documents leaving space for future growth and says sequentially inserted index pages tend to end up denser than randomly inserted ones. Exact occupancy values are product/version-specific. [InnoDB index physical structure](https://dev.mysql.com/doc/refman/8.4/en/innodb-physical-structure.html)

### 5.4 Does a split guarantee 50/50 forever?

No. A teaching algorithm often redistributes approximately half. Engines can optimize the split point based on position, key compression, duplicate handling, concurrency, or right-edge behavior. Immediately after later inserts or deletes, occupancies diverge again.

### 5.5 Deletion, merge, and bloat

Textbook deletion may:

- delete from a leaf;
- borrow/redistribute from a sibling if underfull;
- merge siblings if needed;
- update parent separators;
- cascade upward and possibly shrink the root.

Production engines may delay or avoid aggressive merging because merging creates writes, contention, and the risk that pages soon split again. MVCC engines can retain dead row/index versions until cleanup. Therefore “B+ tree nodes are always exactly between half and fully occupied” is a useful classroom invariant, not a universal observation on live engine pages.

## 6. Clustered versus secondary/nonclustered indexes

### 6.1 Plain-language distinction

**Clustered/index-organized access path:** the table's row storage follows one key order at the leaf level.

**Secondary/nonclustered index:** a separate ordered structure whose leaves locate rows in the base storage.

```text
Clustered by order_id                    Secondary by customer_id

leaf/data pages:                         separate leaf pages:
[100,row][101,row][102,row]              [7 -> 101][7 -> 300][9 -> 102]
[200,row][201,row][202,row]                        |
[300,row][301,row][302,row]                        v
                                          heap RID or clustered-key lookup
```

“Physically sorted” is often used casually. A more accurate phrase is:

> The clustered leaf level maintains **logical key order** and nearby keys tend to be colocated in nearby leaf pages. Filesystem/device blocks need not be physically adjacent, and page splits can fragment allocation.

### 6.2 How can clustered order exist inside a page if row bytes move?

The slot/item directory can carry the **logical index order** while row bytes occupy whatever free regions work.

```text
slot order by clustered key:       0       1       2
                                   |       |       |
                                   v       v       v
actual row bytes on page:       [key 30] [key 10] [key 20]

slot[0] -> bytes for key 10
slot[1] -> bytes for key 20
slot[2] -> bytes for key 30
```

SQL Server explicitly documents that rows within a clustered leaf page are not necessarily in byte order, while slot-array entries are kept in index-key order. [SQL Server index architecture and design guide](https://learn.microsoft.com/en-us/sql/relational-databases/sql-server-index-design-guide)

Thus “clustered order” does not mean every insertion shifts the raw bytes of every later row in the file.

### 6.3 Can we create two clustered indexes by making two offset arrays?

No, not over one canonical set of row bytes.

Suppose rows must be ordered both by `order_id` and by `created_at`:

```text
order_id order:     1, 2, 3, 4
created_at order:   3, 1, 4, 2
```

Two slot arrays on one page could describe two **within-page** orders, but they do not solve:

- which page should contain a row;
- global order across pages;
- where a new row belongs in both page sequences;
- splitting/merging under two incompatible organizations;
- locating ranges without a global tree for the second order.

The second global ordering needs another structure. Options:

1. **Secondary B+ tree**: key plus locator; normal choice.
2. **Covering secondary index**: copies enough columns to answer selected queries.
3. **Materialized view/copy**: full second representation, accepting storage and consistency cost.
4. **Analytical columnstore/search engine**: specialized additional representation for a different workload.

An InnoDB table has one clustered index; SQL Server likewise permits one rowstore clustered index per table. PostgreSQL heap storage has no continuously maintained clustered index in this same sense—its `CLUSTER` command rewrites into an index order once, but later writes do not preserve that order automatically. [PostgreSQL `CLUSTER`](https://www.postgresql.org/docs/current/sql-cluster.html) This vocabulary is engine-specific.

### 6.4 What is stored as the “offset” in a nonclustered index?

Again, it depends on base-table organization.

| Base row organization | Secondary leaf locator | Next step |
|---|---|---|
| Heap with stable slot IDs | `(file/relation, page, slot)`-like RID/TID | Load heap page, use slot to reach row |
| Clustered/index-organized table | Clustered key, commonly primary key | Search clustered tree for full row |
| Covering index | Locator may exist, but selected query needs no base-row visit | Return from secondary leaf after visibility checks as needed |

SQL Server documents a heap RID as `(file ID, page number, row number on page)` and uses the clustered-index key as the row locator when a clustered index exists. [SQL Server index architecture and design guide](https://learn.microsoft.com/en-us/sql/relational-databases/sql-server-index-design-guide)

PostgreSQL's item pointer is page number plus item-identifier index. InnoDB's secondary record contains the row's primary-key columns. These are not interchangeable designs.

### 6.5 Primary key is not universally the same as clustered index

- A **primary key** is a logical uniqueness and non-null identity constraint.
- A **clustered index** is a physical/storage organization choice.

InnoDB normally clusters on the primary key and chooses/generates another clustered key if no suitable primary key exists. SQL Server commonly creates a clustered index for a primary-key constraint by default when no conflicting choice exists, but it can be declared nonclustered. PostgreSQL primary keys create unique indexes but heap rows remain separate.

Interview-safe statement:

> “A primary key and clustered key often coincide in InnoDB, but they are distinct concepts. I need the storage engine before claiming they are the same.”

### 6.6 Clustered versus secondary trade-offs

| Concern | Clustered/index-organized | Secondary/nonclustered |
|---|---|---|
| Full-row lookup by cluster key | Direct at leaf | May need extra base-row lookup |
| Range on cluster key | Excellent locality | Good index-leaf locality; base rows may be scattered |
| Number per table | One canonical organization | Many, within engine limits and practical write budget |
| Wide key cost | Wider internal/leaf entries; often propagates into secondary locators | Wider secondary tree and cache footprint |
| Insert pattern | Can cause right-edge hotspot or random splits | Each maintained secondary has its own split/write behavior |
| Change clustered key | Often expensive because row organization and secondary locators change | Changing a secondary definition is localized to that index build |

## 7. Column `C` had no index; what happens when we add one later?

Suppose:

```sql
CREATE INDEX idx_orders_status_created
ON orders(status, created_at);
```

### 7.1 Build phases in the generic model

```text
existing table pages
       |
       | scan all visible rows; extract (status, created_at, locator)
       v
sort/bulk-build leaf entries
       |
       v
build upper B+ levels
       |
       | account for concurrent changes if online/concurrent build
       v
validate + mark index usable in catalog
```

Creating an index later does **not normally rewrite every base row** merely to add an index pointer. The new index is its own storage object. Building it does require reading the existing table and writing the new index, plus temporary sort space and recovery/replication log volume.

### 7.2 Offline versus online/concurrent build

| Build style | Benefit | Cost/risk |
|---|---|---|
| Block writers during build | Simpler snapshot and usually faster/cheaper | Application writes wait; unacceptable for a large hot table |
| Online/concurrent build | Reads/writes continue | More scans/phases, longer runtime, more resource usage, and product-specific restrictions/failure cleanup |

PostgreSQL's concurrent build must account for transactions and uses multiple phases/scans; if it fails, it can leave an `INVALID` index that is ignored for reads but may need cleanup/rebuild. [PostgreSQL `CREATE INDEX`](https://www.postgresql.org/docs/current/sql-createindex.html) and [PostgreSQL `REINDEX`](https://www.postgresql.org/docs/current/sql-reindex.html)

MySQL distinguishes `INSTANT`, `INPLACE`, and `COPY` algorithms per operation. Creating a secondary index is not the same metadata-only operation as an instant column addition; online DDL still consumes CPU, I/O, memory, and temporary/log space. [InnoDB online DDL operations](https://dev.mysql.com/doc/refman/8.4/en/innodb-online-ddl-operations.html)

### 7.3 After the build

New inserts, deletes, and updates to relevant columns must maintain the new tree transactionally. Every index is therefore a permanent tax on writes and storage.

The optimizer may still ignore it because:

- the predicate returns too many rows;
- the leftmost composite prefix is not constrained effectively;
- statistics estimate a table scan as cheaper;
- a function/type conversion makes the predicate incompatible with the index;
- a better index exists;
- selected columns cause too many random base-row fetches;
- the table is so small that scanning is cheaper.

Use the engine's `EXPLAIN` facility and actual workload measurements. “Index exists” is not proof that “query uses it” or “query is faster.”

### 7.4 Safe rollout checklist for a large production table

1. Write the exact high-value query patterns and required ordering.
2. Check existing overlapping indexes.
3. Estimate new-index bytes, build scratch space, WAL/redo/replication volume, and replica lag risk.
4. Select the engine's correct online/concurrent syntax and understand its locks.
5. Set/observe timeouts and progress; rehearse cancel/failure cleanup.
6. Build during a tolerable window.
7. Refresh/verify statistics if required.
8. Compare `EXPLAIN`/latency/write overhead before and after.
9. Remove only genuinely redundant indexes after observing the workload.

## 8. What happens to pages when a new column is added?

### 8.1 Why this question is subtle

Imagine an old physical row:

```text
[id][name][status]
```

After schema change:

```text
[id][name][status][country]
```

How can the engine parse both? The table catalog says what the current logical columns are, while row/page format information tells the engine which physical form a particular row uses. Engines can choose either:

- rewrite every existing row into the new form now; or
- store metadata saying how old rows should logically expose the missing column and materialize it later.

### 8.2 Cases to reason about

| Change | Possible implementation | Why it may be expensive |
|---|---|---|
| Add nullable column with no default | Metadata-only; old rows read as `NULL` | Usually little data work, but takes schema/catalog locks |
| Add constant default | Metadata can remember a missing value in capable engines | Older engines/unsupported forms may rewrite all rows |
| Add volatile/computed-per-row default | Evaluate for each row and store/rewrite | Values cannot all be represented by one catalog constant |
| Add `NOT NULL` | May need proof/validation scan or staged constraint | Engine must establish that no visible row violates it |
| Change data type/layout | Often table rewrite | Existing bytes need a new encoding |
| Add stored generated column | Often calculation/rewrite | Existing rows need materialized derived bytes |

Always check the exact engine and version.

### 8.3 Verified PostgreSQL example

PostgreSQL documents that adding a column with no default does not require rewriting the table. Since PostgreSQL 11, adding a column with a **constant** default can also avoid updating every row immediately; the default is returned when an old row is read and becomes physical when the row/table is rewritten. A volatile default still requires row-by-row work. [PostgreSQL modifying tables](https://www.postgresql.org/docs/current/ddl-alter.html)

Conceptually:

```text
Catalog after ALTER:
column: country
old-row missing value: 'IN'

Old page remains: [id][name][status]
Reader returns:    [id][name][status]['IN']

Later rewritten row may become:
                   [id][name][status]['IN']
```

### 8.4 Verified InnoDB example

MySQL 8.4 documents `ALGORITHM=INSTANT` as the default for supported `ADD COLUMN` operations: the operation modifies metadata rather than rebuilding the table, subject to explicit limitations such as row format/table features and row-size checks. [InnoDB online DDL column operations](https://dev.mysql.com/doc/refman/8.4/en/innodb-online-ddl-operations.html)

InnoDB retains metadata needed to interpret rows created under different instant-DDL versions; its `INNODB_COLUMNS` metadata exposes the initial default associated with an instantly added column. [MySQL `INNODB_COLUMNS`](https://dev.mysql.com/doc/refman/8.4/en/information-schema-innodb-columns-table.html)

### 8.5 Are all data pages updated?

Answer spine:

```text
1. “It depends on engine, version, default, constraints, and row format.”
2. “A supported nullable/constant-default addition can be metadata-only.”
3. “Old rows/pages can retain their earlier physical form.”
4. “Readers synthesize NULL/default using metadata.”
5. “New or later rewritten rows can store the field physically.”
6. “Unsupported changes require scan/rewrite and much more I/O/logging.”
```

Even a metadata-only alteration is not “free”: it takes a schema lock, updates crash-safe catalog state, invalidates caches/plans as required, and can be delayed by long-running transactions.

### 8.6 Longer-term page effects of the new column

When new/updated rows physically store a wider field:

- fewer average rows may fit per page;
- heap updates may need a different page if no space remains;
- clustered leaf rows may grow and split more often;
- buffer-cache efficiency may fall;
- WAL/redo and replica traffic can grow;
- indexes change only if the new column is indexed/included, or if row relocation requires locator maintenance in that engine.

The `ALTER` can be instant while the **future storage cost is not**.

### 8.7 Production migration pattern

For an engine/change that cannot safely be instant:

```text
expand:    add nullable column
deploy:    writers populate it for new rows
backfill:  old rows in small resumable batches
validate:  check completeness / add constraint safely
switch:    readers depend on it
contract:  remove temporary compatibility path later
```

Avoid one enormous transaction: it can hold locks, generate replication lag, enlarge logs, block cleanup, and create a painful rollback.

## 9. What lives in table metadata and database metadata?

### 9.1 Data dictionary/system catalog

The data dictionary is itself persistent database state describing database objects. Typical **table-level metadata** includes:

- internal table/object identifier and owner;
- schema and table name;
- columns: order, type, length/precision, nullability, default, generated expression;
- primary, unique, foreign-key, and check constraints;
- indexes: access method, key columns/order, included columns, predicate, uniqueness, root/storage identifier;
- partitions and partition bounds;
- storage engine, tablespace/file mapping, compression/fill settings;
- row-format/schema-version information;
- triggers and dependencies;
- privileges;
- approximate row/page counts and optimizer statistics references.

Typical **database/cluster-level metadata** includes:

- databases/schemas/catalogs and ownership;
- roles/users and grants;
- character sets, collations, locales/time-zone-related settings;
- tablespaces/data files;
- extensions, functions, types, views, sequences;
- configuration and version/compatibility information;
- replication/recovery/checkpoint state in engine-specific structures.

MySQL describes `INFORMATION_SCHEMA` as access to metadata such as database/table names, column data types, and privileges, backed by its transactional data dictionary. [MySQL `INFORMATION_SCHEMA` introduction](https://dev.mysql.com/doc/refman/8.4/en/information-schema-introduction.html) and [MySQL data dictionary](https://dev.mysql.com/doc/refman/8.4/en/data-dictionary.html)

### 9.2 Optimizer statistics are metadata, not exact truth

The planner can track:

- approximate table row/page count;
- fraction of nulls;
- number of distinct values;
- common values/frequencies;
- histograms/ranges;
- average column width;
- correlations or extended multicolumn statistics.

These values can be sampled, stale, and approximate. PostgreSQL explicitly documents `pg_statistic` estimates such as null fraction, average width, and distinct count as planner inputs. [PostgreSQL `pg_statistic`](https://www.postgresql.org/docs/current/catalog-pg-statistic.html)

That is why an optimizer can choose a poor plan even when a theoretically suitable index exists: its cost decision is only as good as its statistics and cost model.

### 9.3 Metadata versus allocation maps

```text
Catalog:          “Index idx_orders_status exists for table orders.”
Index metadata:   “Its root page/storage object is R.”
Allocation map:   “These extents/pages belong to that storage object.”
Page header:      “This particular page is leaf level and links to siblings.”
```

No one structure must contain every mapping.

## 10. Canceling a query after it partially changed pages

### 10.1 First split reads from writes

**Canceled `SELECT`:** it normally made no user-data change. It may have loaded pages into caches, used temporary space, updated statistics/hints in engine-specific ways, or acquired/released locks. Cancellation stops further work and discards partial result processing as the protocol/API defines.

**Canceled `UPDATE`/`DELETE`/`INSERT`:** it may already have:

- changed dirty buffer-pool pages;
- written WAL/redo/undo records;
- created new MVCC row versions;
- updated some index pages;
- spilled temporary data;
- waited on or acquired locks.

Those physical actions do not mean a partial logical transaction becomes visible.

### 10.2 Atomicity is logical, not “no byte was touched”

```text
UPDATE 1,000 rows
   |
   | first 600 have physical changes in buffers/logs
   v
cancel/error
   |
   +-- transaction/statement is marked failed
   +-- undo or MVCC visibility makes changes noncommitted/invisible
   +-- locks/resources released at rollback boundary
   +-- later cleanup can reclaim dead/undo versions and page space
```

Engines use different recovery models:

- **Undo-oriented:** use undo records to reverse changes.
- **MVCC-oriented:** uncommitted versions remain invisible and are later cleaned; index/heap physical work may remain until vacuum/purge.
- Most transactional engines combine redo/WAL with undo/MVCC mechanisms in engine-specific ways.

PostgreSQL's WAL rule is that data-file changes are written only after the corresponding WAL records are flushed, allowing crash recovery to redo needed changes. Commit need not flush every changed data page immediately. [PostgreSQL WAL introduction](https://www.postgresql.org/docs/current/wal-intro.html)

### 10.3 What if a dirty page was already flushed before cancellation?

Still not necessarily a problem. The page can contain an uncommitted version or an update described by logs. Transaction status/undo/MVCC determines visibility. On rollback or recovery, the engine can treat the change as aborted and later reclaim or reverse it.

This gives the critical distinction:

```text
physical page contains bytes from attempted work
                         !=
other transactions can observe a committed partial result
```

### 10.4 Cancellation is a request, not time travel

Cancellation is usually checked at safe interrupt points. There is a race:

- cancel arrives while command is running -> command errors/stops;
- command finishes or commits just before cancel is processed -> cancel may have no effect;
- client disconnects -> server/pool behavior and transaction scope decide cleanup.

PostgreSQL explicitly says a cancel request aborts the current command if it matches the active backend, but it can arrive too late and the client must wait for the backend's result to know the outcome. An error inside an explicit transaction leaves it in failed state until rollback. [PostgreSQL protocol message flow](https://www.postgresql.org/docs/current/protocol-flow.html) and [protocol transaction-status indicator](https://www.postgresql.org/docs/current/protocol-message-formats.html)

Application rule: after timeout/cancel, **do not blindly assume failure** when the result is ambiguous. Check transaction/connection state; for externally retried business operations use an idempotency key or durable operation status.

### 10.5 Statement, transaction, and autocommit boundaries

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

If the second update is canceled, the safe response is normally to roll back the transaction. In PostgreSQL the transaction is failed until `ROLLBACK` (or rollback to an appropriate savepoint). Other engines can have different statement-error semantics, so never generalize one product's behavior without naming it.

In autocommit mode, one statement is usually its own transaction. A cancel before successful commit should prevent a partial committed statement, but the “commit won just before cancel” race still exists.

### 10.6 What transactions do not roll back

Database atomicity does not automatically undo an email already sent, a payment API already called, or a Kafka message already published outside the same atomic resource. Use patterns such as transactional outbox, idempotent consumers, and reconciliation for external side effects.

### 10.7 Canceling DDL or index builds

DDL is engine-specific. Transactional DDL may roll back catalog changes. An online/concurrent index build can leave temporary or invalid artifacts requiring documented cleanup. Before canceling a production build, know:

- whether the operation is transactional;
- which locks it currently holds;
- whether rollback itself will be long;
- how much temporary/index space remains;
- whether the engine leaves an invalid index;
- whether replicas must replay all prior work/rollback.

## 11. Designing useful indexes in HLD

### 11.1 Start from access patterns, not columns

Weak reasoning:

> “`created_at` looks important, so I will index it.”

Interview-ready reasoning:

> “The hot query fetches the newest 50 messages for one conversation and pages backward. I will evaluate a composite ordered index on `(conversation_id, sent_at DESC, message_id)` because equality on the leading key narrows one conversation, the remaining keys provide order and a stable cursor, and the query stops after 50 entries.”

### 11.2 Index-choice checklist

For each important query, ask:

1. What are the equality predicates?
2. What range predicate follows?
3. What order and limit are required?
4. How selective is the result?
5. Which columns are returned—can/should the index cover them?
6. Are writes frequent enough that maintenance cost matters?
7. Are keys narrow and stable?
8. Does uniqueness need enforcement?
9. Can a partial/filtered index exclude cold or irrelevant rows?
10. What happens at scale: hot root/leaf, scattered heap reads, cache footprint, replication log?

### 11.3 Composite B+ tree order and the leftmost prefix

An index on `(A, B, C)` is lexicographically ordered:

```text
(A=1,B=1,C=1)
(A=1,B=1,C=9)
(A=1,B=2,C=1)
(A=2,B=1,C=1)
```

It naturally groups by `A`, then within equal `A` by `B`, then by `C`.

Typically strong:

```sql
WHERE A = ?
WHERE A = ? AND B = ?
WHERE A = ? AND B = ? AND C BETWEEN ? AND ?
```

Typically not a direct seek on this index:

```sql
WHERE B = ?
```

MySQL documents that any leftmost prefix of a multicolumn index can be used for lookup. PostgreSQL can use conditions on subsets and has skip-scan optimizations in some cases, but says leading-column constraints are generally the most effective. [MySQL multiple-column indexes](https://dev.mysql.com/doc/refman/8.4/en/multiple-column-indexes.html) and [PostgreSQL multicolumn indexes](https://www.postgresql.org/docs/current/indexes-multicolumn.html)

Do not reduce the rule to “equality columns always first.” Ordering, ranges, skip-scan support, selectivity, compression, and the full query mix matter. Test on the chosen engine.

### 11.4 Covering/index-only access

Suppose:

```sql
SELECT created_at, total
FROM orders
WHERE customer_id = ?
ORDER BY created_at DESC
LIMIT 20;
```

A tree keyed on `(customer_id, created_at)` and including `total` may answer the projection from the index without fetching the base row.

Benefits:

- fewer random base-page visits;
- less I/O and latency for that query.

Costs:

- larger leaf entries and cache footprint;
- more write work when included values change;
- lower fan-out/more pages;
- engine visibility rules can still require a base-page check in some cases.

### 11.5 Selectivity and why the optimizer may scan

An index on Boolean `is_active` might return 95% of a table. Walking the index and then fetching 95% of scattered rows can cost more than reading the table sequentially.

An index is most helpful when it:

- excludes a large part of the table;
- provides needed order/limit;
- covers the result;
- avoids expensive joins or uniqueness checks.

Low-cardinality columns can still be useful as a leading part of a composite/partial index when the combined access pattern is selective.

### 11.6 Write amplification from indexes

One logical insert can cause:

```text
base row write
+ primary/clustered index maintenance
+ secondary index 1 maintenance
+ secondary index 2 maintenance
+ ...
+ WAL/redo/undo
+ replication traffic
+ possible page splits
```

Over-indexing costs:

- disk and RAM;
- lower write throughput/higher latency;
- longer recovery and replication replay;
- more vacuum/purge/rebuild work;
- longer schema migrations;
- optimizer planning complexity and redundant structures.

### 11.7 Alternatives to B+ trees

| Structure | Prefer when | Avoid/limitation |
|---|---|---|
| Hash index/table | Exact equality is dominant | No natural range/order; collision/resize semantics depend on engine |
| LSM tree | Very high write throughput, sequential flush/compaction trade is acceptable | Read/write amplification via compaction, tombstones, range/read tuning complexity |
| Inverted/full-text index | Token/term search, relevance | Not a general row-order index; heavy maintenance |
| Spatial R-tree/GiST-like index | Multidimensional location/geometry | More complex predicates and overlap behavior |
| Bitmap index/bitmap scan | Analytical combinations of low-cardinality predicates | Update contention/maintenance can be unsuitable for hot OLTP depending on implementation |
| BRIN/zone map | Huge table naturally correlated with physical order; tiny summary index | Lossy/coarse: reads candidate page ranges and rechecks rows |
| Columnstore | Large analytical scans and aggregates | Point writes/row retrieval and update patterns may not fit |

Use B+ tree when ordered equality/range lookup, stable logarithmic navigation, and page-local scans match the workload. Avoid treating it as the answer to full-text, nearest-neighbor, arbitrary multidimensional, or append-heavy workloads without examining specialized alternatives.

## 12. Real-system examples

All designs here are **illustrative**. They explain how to reason; they do not assert private current architectures of these companies.

### 12.1 WhatsApp-like conversation history

Requirement: fetch the newest 50 messages in conversation `C`, then page backward.

```text
messages primary/access key:
(conversation_id, sent_at, message_id)

conversation C leaf range:
[C,10:01,m1] [C,10:02,m2] [C,10:02,m3] ... [C,11:30,m99]
                                                     ^
                                                     start newest, scan backward
```

Why `message_id` after time? Timestamps can collide; it gives deterministic order and a cursor.

At very large distributed scale, `conversation_id` may also be a partition key and an LSM-backed store may replace a local B+ tree as the primary storage engine. The access-pattern reasoning remains: colocate a conversation and sort by time/ID.

### 12.2 YouTube-like video lookup and channel page

```text
videos:
PK/cluster key: video_id
secondary:      (channel_id, published_at DESC, video_id)

GET /video/X           -> direct video_id lookup
channel newest videos  -> one ordered channel range, stop at page size
```

Video bytes themselves belong in object/blob storage; the relational row stores metadata and object identifiers. A B+ tree over the video blob would be the wrong abstraction.

### 12.3 Uber-like driver location

A normal B+ tree on `(latitude, longitude)` is lexicographic, not “nearest in 2D.” Nearby latitude does not guarantee nearby longitude position in the key order.

```text
rider location
      |
      v
spatial cell/geohash or spatial index
      |
candidate nearby drivers
      |
exact distance + availability filters
```

A B+ tree can still index a one-dimensional spatial cell key or `(city_id, status, updated_at)`, but nearest-neighbor/geometric queries may need a spatial index.

### 12.4 Netflix-like “continue watching”

```text
watch_state:
unique:    (profile_id, title_id)
secondary: (profile_id, last_watched_at DESC, title_id)
```

The unique index makes update/upsert of one title state efficient; the ordered secondary index renders the newest continue-watching rail without sorting all profile rows. Frequent progress updates mean every extra index should justify its write cost.

### 12.5 Twitter/X-like author tweets and timelines

```text
author_tweets key: (author_id, created_at DESC, tweet_id)

author profile -> contiguous author/time range
home timeline  -> may be precomputed/fanned out into a separate timeline store
search text    -> inverted search index, not ordinary B+ tree
```

One product naturally needs multiple representations because “tweets by author,” “home feed rank,” and “full-text search” are different access patterns. Secondary indexes are not always sufficient across extreme partitioned scale.

### 12.6 Amazon-like orders

```text
orders:
cluster/PK: order_id
secondary:  (customer_id, created_at DESC, order_id)
secondary:  (merchant_id, status, created_at, order_id)
```

```text
customer -> latest orders       merchant -> pending orders by age
          |                               |
          v                               v
(customer,time,id) range        (merchant,status,time,id) range
```

Potential issue: a status change updates the merchant secondary index because the indexed key changed. If the query only cares about open orders and the engine supports it, a partial/filtered index may be smaller than indexing every historical order.

## 13. One worked example end to end

Table:

```sql
CREATE TABLE orders (
    order_id     BIGINT PRIMARY KEY,
    customer_id  BIGINT NOT NULL,
    status       VARCHAR(20) NOT NULL,
    created_at   TIMESTAMP NOT NULL,
    total_cents  BIGINT NOT NULL
);

CREATE INDEX idx_orders_customer_time
ON orders(customer_id, created_at DESC, order_id);
```

Query:

```sql
SELECT order_id, status, created_at, total_cents
FROM orders
WHERE customer_id = 42
  AND created_at < :cursor_time
ORDER BY created_at DESC, order_id DESC
LIMIT 20;
```

Generic heap-oriented path:

```text
1. Catalog resolves table and idx_orders_customer_time.
2. Optimizer estimates selectivity from statistics and chooses the index.
3. Root/internal pages locate first leaf entry below (42, cursor_time).
4. Engine scans adjacent leaf entries for customer 42.
5. Each entry yields a heap (page, slot)-like locator.
6. Buffer manager checks whether each heap page is already cached.
7. On a miss, file/page mapping issues I/O through OS/storage layers.
8. Slot entry yields byte offset; row header and schema metadata decode fields.
9. MVCC/locking rules decide visibility.
10. Stop after 20 visible matching rows.
```

InnoDB path difference:

```text
secondary leaf gives (customer_id, created_at, order_id)
                                        |
                                        v
                         order_id searches clustered tree
                                        |
                                        v
                              clustered leaf has row
```

If the secondary index included all projected fields, it might cover the query and avoid repeated clustered lookups, at the cost of a larger secondary tree and more write maintenance.

Now update `status`:

```sql
UPDATE orders SET status = 'SHIPPED' WHERE order_id = 9001;
```

- The primary/clustered lookup finds row `9001`.
- The row version or in-place representation changes per engine.
- `idx_orders_customer_time` does not key on `status`; if it does not include `status`, its ordered key need not change.
- WAL/redo/undo/MVCC records protect recovery and rollback.
- The page can be dirty in RAM and flush later; commit durability primarily depends on the log protocol, not immediate flushing of that data page.

### 14.3 Own plausible synthesis questions and answer spines

The questions below are **practice synthesis**, not claimed real company prompts.

#### Q1. Why B+ tree instead of a balanced binary tree?

Answer spine:

```text
storage works in pages
-> many keys/pointers per B+ node
-> high fan-out
-> low height/few page visits
-> all records/locators at leaves
-> linked leaves make ranges sequential
-> page split bounds insert reorganization
```

#### Q2. Why not one sorted array?

Answer spine:

```text
array: great binary search and range locality
but middle insert shifts a large suffix
B+ tree: sorted fixed-size pages
only root-to-leaf path plus split changes
keeps updates practical while preserving order
```

#### Q3. Increasing order `M`—what changes?

Answer spine:

```text
first define M convention
larger fan-out -> usually shorter tree
but M comes from page bytes / entry width
larger page or smaller keys has cache/I/O/comparison trade-offs
not an independent free knob
```

#### Q4. Do B+ tree leaves duplicate the table?

Answer spine:

```text
ask which index/storage engine
clustered leaf with full row -> leaf IS table
heap secondary -> indexed key + (page,slot) locator
InnoDB secondary -> indexed key + primary key
covering index intentionally copies selected columns
```

#### Q5. Can a table have two clustered indexes?

Answer spine:

```text
one set of base rows has one canonical global order
second slot array only gives within-page order, not page placement/global splits
use secondary tree, covering index, or separate materialized representation
```

#### Q6. What happens when a page splits under concurrent traffic?

Answer spine:

```text
locate full leaf
short page latches
allocate + redistribute
repair sibling link
parent separator update; cascade/root split possible
WAL/redo makes it recoverable
distinguish latches from transaction locks
mention right-edge hotspot/random split trade-off
```

#### Q7. Add an index to a live huge table—what worries you?

Answer spine:

```text
full scan + sort/build + new storage
lock/concurrent DML behavior
online build extra phases/resources
temporary space + WAL/redo + replica lag
failure/cancel invalid artifacts
verify optimizer plan and write cost after rollout
```

#### Q8. Cancel halfway through an update: are half the rows persisted?

Answer spine:

```text
physical buffers/logs may contain partial work
but transaction atomicity prevents partial committed visibility
undo or MVCC marks/reclaims aborted versions
WAL/redo makes flushed pages crash-safe
cancel can race with completion
external side effects need idempotency/outbox
```

#### Q9. Design an index for the latest 50 messages in a chat

Answer spine:

```text
access pattern: equality conversation + descending time range + limit
index/primary order: (conversation_id, sent_at DESC, message_id)
message_id breaks time ties and supports cursor
partition by conversation if distributed
cost: hot conversations, writes, retention, secondary projections
```

### 14.4 General index-design answer spine

Use the cue **Q-K-L-P-W-O**:

```text
Q  Query: predicates, order, limit, returned columns
K  Key: composite order and uniqueness/tie-breaker
L  Leaf: full row, RID, clustered key, or covering payload?
P  Pages: locality, fan-out, splits, cache, expected rows/pages read
W  Writes: maintenance, WAL/redo, build/backfill, replication
O  Optimizer/operations: statistics, EXPLAIN, online build, monitoring/rollback
```

This reconstructs an answer without memorizing a paragraph.

### 14.5 Common interview mistakes

- Saying “index lookup is `O(log N)`, done” without discussing page I/O, range scans, or base-row fetches.
- Treating B-tree and B+ tree vendor names as proof of leaf layout.
- Saying every leaf stores full rows.
- Saying every secondary leaf stores a byte offset.
- Saying primary key always means clustered index.
- Claiming clustered rows are in physically contiguous disk sectors.
- Claiming two offset arrays create two clustered indexes.
- Ignoring duplicate keys and the need for a row locator/tie-breaker.
- Recommending one index per column.
- Ignoring leftmost composite ordering.
- Ignoring the optimizer and `EXPLAIN`.
- Claiming `ADD COLUMN` always rewrites or never rewrites the table.
- Claiming cancellation means no page/log bytes were written.
- Forgetting build locks, disk headroom, WAL/redo, replica lag, and cleanup.

## 15. Compact trade-off table

| Decision | Good when | Cost / avoid when |
|---|---|---|
| B+ tree index | Equality + ordered/range access; ordered pagination; uniqueness | Extra storage/write amplification; not full-text or naturally multidimensional |
| Cluster on narrow stable key | Frequent point/range reads by that key; good leaf locality | Only one; random/wide/mutable key causes splits and secondary-key bloat |
| Secondary index | Important alternate access path is selective | Extra lookup to base row unless covered; maintenance on every relevant write |
| Covering index | Very hot query needs few columns and avoids random row fetch | Large leaves, lower fan-out, duplicate values, more update work |
| Lower fill factor | Many random inserts/updates expected | More pages, memory, and I/O immediately |
| Online/concurrent build | Hot table cannot block writes | Longer/more complex/resource-heavy; special failure states |
| Metadata-only column add | Engine/version and column definition support it | Still schema locks; future rows wider; limitations must be checked |
| Larger DB pages | Sequential/range-heavy workload and wide fan-out help | Small random I/O amplification and coarser cache/latch unit |

## 16. Final mental model

```text
Logical layer
  table, columns, constraints, query, transaction
        |
        v
Catalog + optimizer
  object IDs, indexes, statistics, chosen access path
        |
        v
Storage structures
  heap/clustered rows + B+ root/internal/leaf pages + allocation maps
        |
        v
Row location
  full row at leaf OR clustered key OR page+slot -> byte offset
        |
        v
Buffer and recovery
  cached frames, dirty pages, latches/locks, WAL/redo, undo/MVCC
        |
        v
I/O stack
  database page -> file range -> filesystem blocks -> device LBAs -> media mapping
```

If you can walk down this diagram and name where each decision occurs, you understand the topic rather than merely remembering that “databases use B+ trees.”

## Say-out-loud 30-second narrative

> “A database stores rows and indexes in fixed-size pages cached in RAM. A slotted page uses a header and directory, so a page-and-slot locator still finds a row when its bytes move. A B+ tree packs many separators into each internal page, keeping lookups shallow, while linked leaves support ranges. Leaves hold either the clustered row, a heap locator, or a clustered key. Indexes trade faster selected reads for storage, write amplification, and splits; metadata, transactions, WAL, and MVCC keep schema changes, cancellation, and recovery safe.”
