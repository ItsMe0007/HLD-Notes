# Distributed Concurrency Control and Transaction Isolation

> Beginner-first HLD notes expanded from: **why `synchronized` is insufficient in a distributed system, transactions and valid states, optimistic and pessimistic concurrency control, shared/exclusive locks, auto-increment gaps, isolation levels, consistency ordering, read anomalies, and deadlock strategy**.
>
> The central idea is: **when several requests race to change the same fact, make one authoritative component enforce the invariant with an atomic decision.** That decision might be a database constraint, conditional write, transaction, row/range lock, single-writer queue, or carefully engineered distributed lock. The right mechanism depends on contention, retry cost, scope, and failure model.

## 0. Scope, assumptions, corrections, and evidence labels

### 0.1 How the rough bullets are interpreted

- **"Using synchronised won't work"** means Java's `synchronized` keyword (or an in-process mutex) cannot coordinate separate service instances. It is still useful inside one process.
- **"Transaction / inconsistent-consistent states"** means both the lifecycle of a transaction and the rule that a correct transaction should move committed data from one valid state to another.
- **"Read repeatable"** is corrected to **Repeatable Read**.
- **"Serialized"** is corrected to **Serializable**. Serializable transactions behave *as if* they ran one at a time; the database need not literally execute them one at a time.
- **"Consistency order"** is interpreted as distributed read/write ordering: linearizable, sequential, causal, eventual, and client-session guarantees. These are distinct from SQL isolation levels.
- **"Deadlock strategy"** includes prevention, detection, victim rollback, timeout, retry, and the extra difficulty of detecting a distributed wait cycle.
- **The auto-increment question** is answered for both interpretations: merely locking an existing row does not consume a generated ID, but a transaction that allocates an ID and then rolls back normally leaves a gap.

### 0.2 Corrections worth saying explicitly in an interview

1. A transaction does not automatically prevent every race. Its statements still run under a particular isolation/concurrency-control policy.
2. Optimistic locking usually does not hold a blocking lock while the application thinks. It detects a conflicting version at write time.
3. Pessimistic locking does not mean "use Read Uncommitted." It means reserve conflicting access before performing the protected work, for example with `SELECT ... FOR UPDATE`.
4. A plain database row lock, an application mutex, a distributed lease, and an idempotency key solve different problems.
5. A transaction's **atomic commit**, its **isolation**, and a replicated store's **consistency model** are different dimensions.
6. Shared/exclusive locks are a conceptual compatibility model. In an MVCC database, an ordinary read may use a snapshot instead of taking a long-lived shared row lock.
7. Auto-generated primary keys promise uniqueness under their documented rules, not gaplessness, business ordering, or proof that every smaller ID committed.
8. Deadlocks are expected possibilities in lock-based systems. Production code must be prepared to retry a chosen victim transaction.
9. Two-phase locking (**2PL**) is concurrency control. Two-phase commit (**2PC**) is an atomic commit protocol. They are not the same.
10. A distributed lock with a TTL is not equivalent to a local mutex. A paused or partitioned old owner can wake after its lease expires; fencing is needed when stale work could corrupt an external resource.

### 0.3 Evidence and confidence labels

- **Verified behavior:** supported by first-party documentation or an original paper linked nearby.
- **SQL-standard model:** the portable isolation-level teaching model; an engine may provide stronger or different behavior.
- **Generic implementation:** a useful model whose details vary by database and version.
- **Illustrative design:** a plausible design for a named product, not a claim about its current private architecture.
- **Self-reported interview prompt:** a public candidate report, not an official company question bank.
- **Unverified:** a claim that should not be repeated as fact without checking. No universal conflict rate, lock timeout, retry count, or throughput number is assumed here.

## 1. Plain-language foundation

### 1.1 Concurrency means work overlaps

Two users can act on the same data before either user has seen the other's result.

Suppose one concert seat is available:

~~~text
Database: seat A7 = AVAILABLE

Request 1 reads AVAILABLE
Request 2 reads AVAILABLE
Request 1 writes BOOKED_FOR_ALICE
Request 2 writes BOOKED_FOR_BOB
~~~

Both requests made a locally reasonable decision from the value they read. Together they produced an invalid outcome: Alice may have been told she won even though Bob's later write replaced hers.

This is a **race condition**. The result depends on timing.

### 1.2 Start with the invariant, not the lock

An **invariant** is a business rule that must remain true. Examples:

- one seat has at most one active reservation;
- inventory must never fall below zero;
- one driver has at most one active trip;
- an idempotency key produces at most one payment effect;
- an account's debit must not exceed the allowed balance;
- a username is unique.

The design question is not initially "optimistic or pessimistic lock?" It is:

> Which component owns this fact, what exact invariant must it enforce, and what single atomic operation can enforce it?

For the seat, one safe database operation could be:

~~~sql
UPDATE seats
SET status = 'HELD', held_by = :user_id, version = version + 1
WHERE seat_id = :seat_id
  AND status = 'AVAILABLE';
~~~

Then inspect the affected-row count:

- `1` row: this request acquired the seat;
- `0` rows: the precondition was false, so another request won or the seat was unavailable.

The check and the update happen as one database statement. There is no gap in which application code reads `AVAILABLE`, pauses, and later blindly writes.

### 1.3 Why `synchronized` appears to work locally

Inside one Java process:

~~~java
public synchronized BookingResult book(String seatId) {
    // only one thread holding this object's monitor enters at a time
}
~~~

A Java object has an intrinsic monitor. A thread entering the synchronized method acquires that object's monitor, and another thread targeting the same monitor waits. Oracle's Java tutorial describes this as exclusive access to the same object's intrinsic lock ([Oracle: intrinsic locks and synchronization](https://docs.oracle.com/javase/tutorial/essential/concurrency/locksync.html)).

That can protect in-memory state shared by threads in **that process**.

### 1.4 Why it fails after horizontal scaling

~~~text
                         +--> Booking instance A -- monitor A --+
Client -> Load balancer -+                                      +--> shared DB
                         +--> Booking instance B -- monitor B --+
~~~

Instance A and instance B have different memory and different monitor objects. Both can enter their own synchronized method simultaneously. Even in one JVM, two different service objects or class loaders may mean different monitors. A `static synchronized` method locks a `Class` object in that JVM; it does not create a cluster-wide lock.

The authoritative state must therefore coordinate the requests. Common choices are:

1. database constraint or atomic conditional write;
2. database transaction with appropriate isolation;
3. optimistic version check;
4. pessimistic row/range lock;
5. a single writer for a partition/key;
6. a consensus-backed coordination/lock service when the protected resource is outside the database.

### 1.5 Safety and liveness

Concurrency control balances two types of property:

- **Safety:** nothing bad happens. A seat is not sold twice.
- **Liveness:** something good eventually happens. Requests do not wait or retry forever.

A global lock can improve safety but hurt liveness during failures. An always-available multi-region write path can improve liveness but may expose conflicts. State which property the business prioritizes for each operation.

## 2. Prerequisites explained inline

These ideas are necessary for this chapter, so they are introduced here rather than deferred to "study next."

### 2.1 Shared mutable state and the read-modify-write pattern

The dangerous pattern is:

~~~text
READ current value
COMPUTE a new value in application memory
WRITE new value
~~~

If two workers read the same version, one can overwrite the other. The fix is not always a lock; it can be an atomic database operation such as `counter = counter + 1`, a conditional compare-and-set, or a constraint.

### 2.2 Atomic operation

An operation is **atomic** at a stated boundary if observers cannot see it half-applied and competing operations cannot interleave inside that boundary.

Examples with different scopes:

- CPU compare-and-swap: one memory location on one machine;
- SQL statement: according to one database engine's execution rules;
- local transaction: its included work in one transactional resource;
- distributed transaction: several compatible resources coordinated to one commit decision.

Always name the boundary. "It is atomic" is incomplete without scope.

### 2.3 ACID

A database transaction is commonly described with ACID:

- **Atomicity:** all included changes commit, or none become committed.
- **Consistency:** correct transaction logic plus database constraints move committed data between valid states.
- **Isolation:** concurrent transactions are prevented from producing/observing histories forbidden by the chosen guarantee.
- **Durability:** a reported commit survives failures covered by the database's contract.

ACID consistency is not the same as replica consistency in Section 10. The database cannot infer every business rule; an application can atomically commit a logically wrong value unless constraints or transaction logic reject it.

### 2.4 Commit and rollback

- **Commit:** make the transaction's outcome final and visible under the database's rules.
- **Rollback/abort:** discard its uncommitted logical effects.

A rollback does not necessarily rewind every auxiliary generator or external effect. Auto-increment values, sent emails, and third-party charges are classic examples. Only work inside the transaction's actual boundary is rolled back.

### 2.5 MVCC and snapshots

Many databases use **multi-version concurrency control (MVCC)**. Instead of making every reader wait behind a writer, the database retains enough row-version information for a reader to see a suitable committed snapshot.

~~~text
row versions:  v1 (committed) -> v2 (uncommitted by T1) -> v3 ...

T2's snapshot may read v1 while T1 is producing v2.
~~~

Consequences:

- ordinary reads may not take shared row locks;
- an X lock held by a writer may block another writer or locking read while a snapshot read continues;
- isolation level determines which snapshot(s) a transaction may observe;
- old versions consume cleanup/undo resources until no relevant transaction needs them.

PostgreSQL and InnoDB both document MVCC-based concurrency, but their exact snapshots and lock behaviors differ. Never generalize one engine's edge case to every SQL database.

### 2.6 Timeout and unknown outcome

A timeout means **the caller stopped waiting**. It does not prove the server did nothing.

~~~text
client ---- BOOK A7 ----> server commits hold
client <--- response X--- network drops response
client times out: outcome is UNKNOWN
~~~

Use a stable operation/idempotency key, query status, and retry according to the API contract. Idempotency handles duplicate delivery; concurrency control decides which competing operation may change the state. They complement each other.

### 2.7 Local versus distributed transaction

A local transaction has one transaction manager/resource. A distributed transaction coordinates several transactional resources or services. Atomic commit across compatible databases, Sagas, and compensation are developed in [Lecture 19: Distributed Transactions](./19.%20Distributed%20Transactions.md).

This chapter focuses on **concurrent access**, including access distributed across many application instances. A system can have local database transactions while its callers are distributed.

## 3. Transactions and consistent/inconsistent states

### 3.1 A transaction is a protected unit of work

Consider transferring 100 from account A to B in one database:

~~~sql
BEGIN;

UPDATE accounts SET balance = balance - 100 WHERE id = 'A';
UPDATE accounts SET balance = balance + 100 WHERE id = 'B';

COMMIT;
~~~

The intended invariant might be:

~~~text
A.balance + B.balance remains unchanged
and A.balance does not violate its overdraft rule
~~~

Between the two statements, the transaction's private working state may temporarily show the debit without the credit. Atomicity prevents other transactions from seeing a committed half-transfer under normal transactional rules, and rollback prevents that half-transfer from becoming the final committed state.

### 3.2 "Consistent state" has to be defined

Suppose the initial committed state is:

~~~text
A = 500, B = 300, total = 800
~~~

A valid result is:

~~~text
A = 400, B = 400, total = 800
~~~

An invalid result is:

~~~text
A = 400, B = 300, total = 700
~~~

But the database knows only declared rules and executed logic. It can enforce a check such as `balance >= 0` on one row. A cross-account invariant may require transaction logic, a ledger model, or stronger constraints. "C in ACID" does not mean the database understands money.

### 3.3 Generic transaction lifecycle

Exact engine names vary, but this is a useful conceptual state machine:

~~~text
                last statement completed
ACTIVE ---------------------------------> PARTIALLY COMMITTED
  |                                                |
  | statement/error/crash                          | commit record durable
  v                                                v
FAILED                                           COMMITTED
  |
  | undo/rollback
  v
ABORTED -----> retry as a new transaction, or terminate
~~~

- **Active:** statements are executing.
- **Partially committed:** application work finished, but the commit is not yet safely final. This is a textbook/conceptual state; product internals use different terms.
- **Committed:** the outcome is final under the durability contract.
- **Failed:** normal execution cannot continue.
- **Aborted:** uncommitted logical effects have been rolled back; the operation may be retried if safe.

### 3.4 Atomicity is not isolation

Two transactions can each commit atomically and still create a bad combined result if their interleaving is permitted.

~~~text
Inventory = 1

T1: read 1                 T2: read 1
T1: write 0                T2: write 0
T1: commit                 T2: commit
~~~

Both commits were all-or-nothing, but one sale was lost from the count. The concurrency policy must prevent or detect the conflict.

### 3.5 Transaction boundaries should be short

Do not normally keep a database transaction or row lock open while:

- a user fills an address form;
- a remote payment provider responds;
- an email is sent;
- a long CPU job runs;
- a retry sleeps.

These actions increase lock duration, old-version retention, deadlock exposure, connection usage, and failure ambiguity. For a checkout, commit a short **reservation** transaction, call payment outside it, and then commit a short confirm/release transaction.

## 4. The core anomalies before the formal names

### 4.1 Lost update

~~~text
Initial quantity = 10

T1 reads 10, plans 10 - 3 = 7
T2 reads 10, plans 10 - 4 = 6
T1 writes 7
T2 writes 6

Final quantity = 6, but it should be 3
~~~

T2 overwrote T1's update. Solutions include:

- atomic arithmetic: `SET quantity = quantity - :n` with a nonnegative predicate;
- version-based OCC;
- a pessimistic row lock;
- serializable isolation with required retry;
- single-writer ownership.

### 4.2 Check-then-act race

~~~text
if username "sam" does not exist:
    insert "sam"
~~~

Two requests can both pass the check. A `UNIQUE(username)` constraint makes the authoritative database decide the winner atomically. The application must treat the constraint violation as an expected conflict, not a mysterious server error.

### 4.3 Write skew

Two doctors are on call; the rule is at least one must remain on call.

~~~text
T1 reads: Alice=on, Bob=on     T2 reads: Alice=on, Bob=on
T1 sets Alice=off              T2 sets Bob=off
T1 commits                     T2 commits
~~~

The transactions wrote different rows, so a per-row version check may not conflict. The cross-row invariant is broken. Protect the full invariant using serializable isolation, an invariant-owning summary row, explicit range/predicate locks, or a redesigned data model.

### 4.4 Duplicate effect

Two retries are not necessarily competing updates; they may be the **same logical request** delivered twice. A unique idempotency key or operation record prevents a double charge. OCC alone does not identify two requests as the same business operation.

### 4.5 One-statement conditional update is often the best first answer

For decrementing inventory:

~~~sql
UPDATE inventory
SET available = available - :requested
WHERE sku = :sku
  AND available >= :requested;
~~~

Check the row count. This avoids a separate read and may be simpler than a manually managed lock. Confirm the exact semantics and retry behavior of the selected database, especially when multiple rows or replicas are involved.

## 5. A map of concurrency-control choices

Use the strongest simple mechanism at the component that owns the invariant.

| Mechanism | Plain-language idea | Best fit | Main cost/failure |
|---|---|---|---|
| Unique/check constraint | Let the database reject invalid committed state | Uniqueness and local row rules | Only expresses supported constraints |
| Atomic conditional write | Change only if the current state still permits it | Counters, state transitions, single item | Caller must handle zero rows/conflict |
| Optimistic concurrency control | Work without holding a lock; validate version before commit | Low contention, cheap retry | Conflicts waste work; retry storms |
| Pessimistic database lock | Reserve the row/range before acting | High contention, expensive conflict | Waiting, deadlocks, lock duration |
| Serializable transaction | Allow DB to reject any non-serializable history | Multi-row invariants | Abort/retry or blocking; engine cost |
| Single writer per key/partition | Route changes for one owner through an ordered worker | Commands/events naturally partition by key | Hot partitions, failover, queue lag |
| Distributed lock/lease | Coordinate processes through a shared lock service | External/non-DB critical section | Partitions, stale holders, fencing, availability |
| Reservation/escrow | Allocate bounded rights in short transactions | Tickets, inventory, quotas, long workflows | Expiry/reconciliation complexity |

### 5.1 A practical selection order

~~~text
Can a database constraint express the invariant?
  yes -> use it as the final guardrail
  no
   |
Can one atomic conditional statement enforce it?
  yes -> prefer that, check the result
  no
   |
Is contention low and retry cheap?
  yes -> OCC/version check
  no
   |
Can a short DB transaction lock all relevant rows/ranges?
  yes -> pessimistic lock or serializable transaction
  no
   |
Can commands be routed to one writer per invariant key?
  yes -> partitioned single-writer design
  no
   |
Does an external resource truly require mutual exclusion?
  yes -> proven distributed coordination + lease/session + fencing
  otherwise -> redesign ownership/reservation boundary
~~~

This is a heuristic, not a proof. Multiple mechanisms are commonly layered: idempotency key + unique constraint + short transaction + retry.

## 6. Optimistic concurrency control (OCC)

### 6.1 Plain-language idea

OCC says:

> "Conflicts are probably uncommon. Let everyone work, but reject a write if the data changed since that worker read it."

It is also called **optimistic locking**, although the common version-check pattern does not hold a blocking lock during the application work.

### 6.2 Read, compute, validate, write

~~~text
1. READ       item and version 7
2. COMPUTE    proposed change in application memory
3. VALIDATE   stored version is still 7
4. WRITE      new value and version 8 atomically
~~~

If another writer already changed version 7 to 8, the validation fails.

### 6.3 SQL version-column example

~~~sql
-- Read
SELECT seat_id, status, version
FROM seats
WHERE seat_id = 'A7';

-- Later, conditional write
UPDATE seats
SET status = 'HELD',
    held_by = :user_id,
    version = version + 1
WHERE seat_id = 'A7'
  AND version = :version_read
  AND status = 'AVAILABLE';
~~~

The application must check affected rows. Zero rows is an expected **conflict**, not success.

### 6.4 Two writers timeline

~~~text
Alice                              Bob
  | read A7 version 7                | read A7 version 7
  |                                  |
  | UPDATE ... WHERE version=7       |
  | succeeds; version becomes 8      |
  |                                  | UPDATE ... WHERE version=7
  |                                  | affects 0 rows -> conflict
~~~

Bob can show "seat just became unavailable" or reload/retry according to product semantics. Blind automatic retry is wrong if it would silently change the user's choice.

### 6.5 HTTP and NoSQL forms

- HTTP can return an `ETag`, then require `If-Match: <etag>` on update.
- DynamoDB can use a version attribute and a conditional expression. AWS describes optimistic locking as detecting conflicts at write time; a mismatched version causes the conditional write to fail ([AWS DynamoDB optimistic locking](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/BestPractices_OptimisticLocking.html)).
- Google Cloud Storage generation-match preconditions allow a mutation only when the target is still the expected generation ([Google Cloud Storage request preconditions](https://docs.cloud.google.com/storage/docs/request-preconditions)).

These are compare-and-set forms at different APIs.

### 6.6 When OCC works well

- concurrent writes to the same item are rare;
- reads are common and writes are less common;
- the work between read and write is short;
- retry/reload is cheap;
- waiting behind a lock would harm latency;
- users may edit different records independently.

### 6.7 When OCC performs badly

- a flash sale makes thousands of requests fight for one remaining item;
- conflicts are common;
- the attempted work is expensive;
- a failed attempt already called an irreversible external system;
- repeated retries cause starvation or a retry storm;
- the invariant spans rows that are not all included in validation.

Approximate wasted work under a particular workload is a measurement question, not a universal OCC statistic.

### 6.8 Retry policy

A robust retry loop needs:

1. re-read the current state;
2. decide whether the original intention is still meaningful;
3. retry only a bounded number of times;
4. add randomized exponential backoff under contention;
5. preserve the same business idempotency key;
6. surface a conflict/manual choice when silently merging is unsafe.

For an account transfer, "retry with whatever balance now exists" may be correct only if the debit predicate is rechecked. For document editing, overwriting a collaborator's change may require a merge UI rather than retry.

### 6.9 OCC edge cases

- **ABA problem:** value A changes to B and back to A. Comparing only the business value misses the intervening updates; compare a monotonically changing version/generation.
- **Deleted and recreated record:** a reused natural key can look unchanged; version the incarnation or use a non-reused identity.
- **Version overflow:** choose a sufficiently large type and define behavior; do not assume infinite integers.
- **Replica read:** reading version 7 from a lagging replica while the leader holds version 9 causes avoidable conflicts. Critical OCC reads should meet the required freshness guarantee.
- **Multi-region last-writer-wins:** a per-item version scheme may not behave as expected if each region accepts independent writes and reconciles later. AWS explicitly warns about this interaction for DynamoDB global tables in its current guidance ([AWS concurrent-update strategies](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/BestPractices_ImplementingVersionControl.html)).
- **Write skew:** validate every item/predicate participating in the invariant, or use serializable isolation/redesign.

## 7. Pessimistic concurrency control

### 7.1 Plain-language idea

Pessimistic control says:

> "A conflicting action is likely or too expensive to undo. Reserve the protected state before making the decision."

Other transactions trying to acquire an incompatible lock wait, fail immediately, or time out according to the API.

### 7.2 Database row-lock example

~~~sql
BEGIN;

SELECT status, held_by
FROM seats
WHERE seat_id = 'A7'
FOR UPDATE;

-- Application verifies AVAILABLE, then:
UPDATE seats
SET status = 'HELD', held_by = :user_id
WHERE seat_id = 'A7';

COMMIT;
~~~

The `FOR UPDATE` locking read reserves the selected row against conflicting changes until transaction end under the database's rules. InnoDB documents that locking behavior depends on the access path: a unique lookup can lock the record, while range/non-unique scans can lock scanned records and gaps ([MySQL 8.4: locks set by statements](https://dev.mysql.com/doc/refman/8.4/en/innodb-locks-set.html)).

### 7.3 Why indexes matter to locking

The database often locks index records/ranges it scans. A missing or unsuitable index can make a statement scan and lock far more data than intended, increasing contention and deadlocks. This connects to [Lecture 20: B+ Trees and Database Indexing](./20.%20B%2B%20Trees%20and%20Database%20Indexing.md).

### 7.4 Lock only the short reservation

Unsafe user-flow design:

~~~text
BEGIN -> lock seat -> call payment for 30 seconds -> COMMIT
~~~

Safer workflow shape:

~~~text
Tx 1: AVAILABLE -> HELD(until=...) -> COMMIT
          |
          +-- payment call outside DB transaction
          |
Tx 2: HELD -> BOOKED or AVAILABLE -> COMMIT
~~~

The reservation is durable state, not a database connection held for the user's entire checkout. Expiry requires a recovery worker and careful rules for a late payment result.

### 7.5 When pessimistic control fits

- contention for the same item is high;
- retrying expensive work would be worse than waiting;
- the protected work is short and entirely inside the transaction boundary;
- deterministic ordering/fairness is important and supported;
- conflicts must be discovered before proceeding.

### 7.6 Costs and risks

- waiting raises tail latency;
- blocked transactions consume connections/resources;
- long transactions retain locks and MVCC history;
- deadlocks are possible;
- a crashed session needs database/lease recovery;
- coarse locks reduce concurrency;
- a hot row remains a serialization bottleneck even if the system has many servers.

### 7.7 `NOWAIT` and `SKIP LOCKED`

Some databases offer variants such as:

- **NOWAIT:** fail instead of waiting for a conflicting lock;
- **SKIP LOCKED:** skip locked rows, useful for multiple workers claiming independent jobs.

`SKIP LOCKED` is usually inappropriate for a user asking for one specific seat because silently returning a different/incomplete result changes semantics. Vendor behavior must be checked before relying on either feature.

## 8. Shared (S) and exclusive (X) locks

### 8.1 The basic meanings

- A **shared lock (S)** allows reading the protected item. Several compatible readers can share it.
- An **exclusive lock (X)** allows changing the protected item and conflicts with other S/X lock requests.

MySQL's InnoDB documentation describes S and X row-level lock modes in these terms ([MySQL 9.1: InnoDB locking](https://dev.mysql.com/doc/refman/9.1/en/innodb-locking.html)).

### 8.2 Compatibility matrix

| Existing lock | Request S | Request X |
|---|---:|---:|
| None | grant | grant |
| S | grant | wait/fail |
| X | wait/fail | wait/fail |

This simplified matrix assumes the same protected object and different transactions. Real engines also have intention, update, key-share, gap, next-key, predicate, metadata, and advisory modes.

### 8.3 Lock granularity

| Granularity | Benefit | Cost |
|---|---|---|
| Database/table | Simple, few lock records | Blocks unrelated rows |
| Page | Middle ground in some engines | Unrelated rows on page interfere |
| Row/index record | High concurrency for different rows | More lock bookkeeping |
| Key range/predicate | Protects "no row in this range" and phantoms | Can cover many current/future rows |
| Advisory/business key | Application-defined coordination | Database may not enforce data relationship |

"Row lock" is not always literally attached to heap-row bytes. InnoDB, for example, describes row locks on index records.

### 8.4 MVCC caveat

Do not say "every SELECT takes S and therefore X blocks every reader." In MVCC engines, a normal `SELECT` often reads an earlier committed version. A locking read such as `FOR SHARE`/`FOR UPDATE` has different behavior. State the engine and query form.

### 8.5 Lock conversion

Two transactions can each take S and later request X:

~~~text
T1 holds S on row R; requests X on R
T2 holds S on row R; requests X on R
~~~

Each waits for the other's S lock to disappear: a conversion deadlock. Some engines provide an update-lock concept to reduce this pattern; exact modes are vendor-specific.

### 8.6 Two-phase locking (2PL)

Classic 2PL has:

~~~text
Growing phase:   acquire/upgrade locks; do not release
Shrinking phase: release/downgrade locks; do not acquire new locks
~~~

2PL produces conflict-serializable schedules under its assumptions. **Strict 2PL** holds exclusive locks until commit/abort, preventing other transactions from reading/writing uncommitted changes through those locks and simplifying recovery.

Do not confuse it with 2PC:

| 2PL | 2PC |
|---|---|
| Concurrency-control protocol | Atomic-commit protocol |
| Controls interleaving | Controls commit/abort across participants |
| Uses growing/shrinking lock phases | Uses prepare/decision phases |
| Can deadlock | Can block in prepared uncertainty |

One distributed transaction can use both 2PL for isolation and 2PC for atomic commit.

## 9. Isolation levels and read problems

### 9.1 Isolation asks which interleavings are allowed

Higher isolation makes concurrent execution look more like one-at-a-time execution, usually at the cost of more blocking, version bookkeeping, or transaction aborts.

The four familiar names are a SQL-standard interface, not four universally identical implementations.

### 9.2 Dirty read

T2 reads a value that T1 has not committed:

~~~text
T1: UPDATE balance 500 -> 0      (not committed)
T2: SELECT balance -> sees 0
T1: ROLLBACK
~~~

T2 acted on a value that never became committed. This is a **dirty read**.

### 9.3 Non-repeatable read

T1 reads the same row twice and sees different committed values:

~~~text
T1: SELECT price -> 100
                              T2: UPDATE price = 120; COMMIT
T1: SELECT price -> 120
~~~

The row still exists, but its value changed between T1's reads.

### 9.4 Phantom read

T1 repeats a predicate query and sees a different set of rows:

~~~text
T1: SELECT * FROM bookings WHERE show_id=9;  -> 10 rows
                              T2: INSERT matching booking; COMMIT
T1: repeat same predicate                    -> 11 rows
~~~

The new matching row is a **phantom**. Protecting only the ten rows that existed initially cannot by itself lock a row that did not yet exist; range/predicate locking or a suitable serializable/MVCC mechanism is needed.

### 9.5 SQL-standard teaching table

| Isolation level | Dirty read | Non-repeatable read | Phantom read | Mental model |
|---|---:|---:|---:|---|
| Read Uncommitted | possible | possible | possible | May expose uncommitted concurrent work |
| Read Committed | prevented | possible | possible | Each statement sees a committed view |
| Repeatable Read | prevented | prevented | possible by standard | Re-read rows remain stable |
| Serializable | prevented | prevented | prevented | Outcome is equivalent to some serial order |

This table describes the standard phenomena-based minimum. An implementation may prohibit more anomalies than the minimum.

### 9.6 Read Uncommitted

- Weakest named level in the standard model.
- Can permit dirty, non-repeatable, and phantom reads.
- Rarely suitable for correctness-sensitive application decisions.
- It is not a technique for pessimistic locking.

Vendor caveat: PostgreSQL accepts the name but implements it as Read Committed, so dirty reads are not actually exposed there ([PostgreSQL 18: transaction isolation](https://www.postgresql.org/docs/current/transaction-iso.html)).

### 9.7 Read Committed

- A statement sees committed rows according to the engine's statement-snapshot rules.
- Two statements in one transaction can see different newly committed data.
- Prevents dirty reads but still permits non-repeatable reads and phantoms in the standard model.
- Often a practical default, but a multi-statement read-modify-write still needs an atomic predicate, lock, version check, or stronger isolation.

### 9.8 Repeatable Read

- Prevents non-repeatable reads in the standard model.
- The standard table still permits phantoms.
- Snapshot implementations can preserve a transaction-level view, but may still permit write skew/serialization anomalies.

PostgreSQL's implementation is stronger than the table with respect to phantoms: its Repeatable Read prevents phantom reads, while serialization anomalies can still occur. This is why memorizing only the three-anomaly table is insufficient.

### 9.9 Serializable

Serializable means the committed result is equivalent to **some** order in which the transactions ran one at a time.

The engine may implement it with locks, predicate/range locks, serialization-graph checks, MVCC conflict detection, or a combination. It may:

- block one transaction;
- abort one with a serialization failure;
- require the application to retry the entire transaction.

PostgreSQL documents that it rolls back a transaction when the concurrent pattern cannot correspond to any serial execution ([PostgreSQL 18: `SET TRANSACTION`](https://www.postgresql.org/docs/current/sql-set-transaction.html)).

Serializable is not a substitute for:

- idempotency around external retries;
- atomic commit across unrelated services;
- fresh reads from asynchronous replicas;
- correct constraints and application logic.

### 9.10 Beyond the three read anomalies

| Anomaly | What goes wrong | Typical defense |
|---|---|---|
| Lost update | One writer overwrites another's result | Atomic update, OCC, lock, serializable |
| Write skew | Transactions write different rows after reading a shared predicate | Serializable, predicate/range protection, invariant row |
| Read skew | Related facts are read from inconsistent points in time | Transaction snapshot/stronger isolation |
| Dirty write | One uncommitted write overwrites another | Mainstream transactional engines prevent it |

Isolation guarantees are defined over complete histories, not just the three popular examples.

### 9.11 Choosing an isolation level

1. Write the invariant and transaction read/write set.
2. Identify anomalies that can break it.
3. Check the selected database's actual semantics, not only the level name.
4. Add constraints/atomic predicates as final guardrails.
5. Expect and safely retry deadlock/serialization victims.
6. Load-test the contention pattern and observe abort/wait rates.

Use the lowest isolation that is **demonstrably correct**, not automatically the lowest available.

## 10. Consistency ordering in distributed systems

### 10.1 Why this is a separate axis

Suppose a leader commits `order=PAID`, but a read replica has not applied it yet:

~~~text
Client -> Leader: write PAID -> success
Client -> Replica: read order -> PENDING
~~~

The local write transaction may have been atomic and serializable on the leader. The read still returned stale data because replica visibility is a different guarantee.

"Consistency" is overloaded:

| Phrase | Question it answers |
|---|---|
| ACID consistency | Did valid transaction logic/constraints preserve invariants? |
| Isolation/serializability | Are concurrent transaction interleavings equivalent to allowed histories? |
| Replica consistency model | In what order and when do clients observe distributed operations? |
| CAP consistency | During a partition, does the system preserve a single-copy-like model rather than serve conflicting results? |

### 10.2 Linearizability

Each operation appears atomic at one point between invocation and response, and the order respects real time.

~~~text
Write W finishes before read R begins
=> R must not be ordered before W
~~~

Useful for:

- acquiring a lock;
- claiming one seat;
- leader metadata;
- a balance/inventory decision where stale state would violate safety.

Herlihy and Wing's original definition treats operations as taking effect in an order consistent with real-time precedence ([Linearizability paper](https://www.cs.cmu.edu/~wing/publications/HerlihyWing90.pdf)).

Cost/trade-off: a system preserving a linearizable decision generally must coordinate with an authoritative leader/quorum and may reject or delay operations when a partition prevents safe coordination.

### 10.3 Sequential consistency

All operations appear in one total order that preserves each client's program order, but that total order need not respect real-time order between different clients.

If Alice's write completed before Bob began his read, a sequentially consistent history may still place Bob's read before Alice's write when no per-client program-order rule connects them. Linearizability would not.

### 10.4 Causal consistency

If operation B could have been influenced by A, everyone must observe A before B.

~~~text
Message A: "Meeting is at 5"
Reply B:   "That time works"

Seeing B before A violates their causal order.
~~~

Independent concurrent actions may be observed in different orders. This can fit feeds, comments, collaboration, or messaging where causality matters more than one global total order.

### 10.5 Eventual consistency

If no new updates occur, replicas eventually converge under the store's reconciliation rules. By itself this does not say:

- how stale a read may be;
- which concurrent write wins;
- whether a user reads their own write immediately;
- whether causality is preserved;
- how quickly convergence occurs.

Those need separate guarantees and measured service objectives.

### 10.6 Client/session guarantees

Weaker replicated stores can still offer useful per-client properties:

- **Read your writes:** after I write X, my later reads see X or something newer.
- **Monotonic reads:** once I see version 8, I do not later see version 7.
- **Monotonic writes:** my writes are applied in my issued order.
- **Writes follow reads:** my write is ordered after the version on which it depends.

Sticky routing, version tokens, leader reads, or waiting until a replica reaches a required log position can implement such properties.

### 10.7 Serializability versus linearizability

This is a high-value follow-up:

| Property | Unit/scope | Preserves real-time order? |
|---|---|---:|
| Serializability | Multi-operation transactions | Not necessarily |
| Linearizability | Individual operations on an object/service abstraction | Yes |
| Strict serializability | Transactions | Yes; serializable plus real-time order |

A serializable history can choose a serial order different from wall-clock completion when transactions overlap or the definition permits it. Strict serializability adds the real-time constraint.

### 10.8 Which model for which data?

- Seat ownership, payment state transition, and distributed-lock metadata usually need strong/linearizable decisions at their authoritative boundary.
- View counts, follower counts, recommendations, and analytics can often be eventually consistent.
- Conversation replies may need per-conversation/causal ordering, not one global order for every message in the world.
- A product can use several models: strong for the fact, eventual for derived counters and feeds.

## 11. The auto-increment gap question

### 11.1 Short answer

> Yes, **if T1 allocated an auto-generated value and later rolled back**, that value is normally not reused, so a gap can remain. T2 can allocate and commit a later value. If T1 only locked an existing row and never requested a generated value, that lock alone does not consume an ID.

### 11.2 Exact timeline

~~~text
Initial next value = 101

T1: INSERT ... -> generator allocates 101  (not committed)
T2: INSERT ... -> generator allocates 102; COMMIT
T1: ROLLBACK

Committed IDs: ..., 100, 102
Gap: 101
~~~

The row inserted by T1 disappears, but the generator does not rewind through T2's allocation. Reusing 101 would require coordination and could collide with values already exposed to concurrent sessions or external logs.

### 11.3 Verified vendor behavior

- MySQL InnoDB says an allocated auto-increment value is lost if its transaction rolls back; it is not reused, in all documented auto-increment lock modes ([MySQL 8.0: AUTO_INCREMENT handling](https://dev.mysql.com/doc/refman/8.0/en/innodb-auto-increment-handling.html)).
- PostgreSQL says `nextval` is atomic for concurrent callers but is not reclaimed after abort, so sequences cannot provide gapless values ([PostgreSQL 18: sequence functions](https://www.postgresql.org/docs/current/functions-sequence.html)).
- SQL Server says failed statements and rolled-back transactions can advance identity and create gaps ([Microsoft: `@@IDENTITY`](https://learn.microsoft.com/en-us/sql/t-sql/functions/identity-transact-sql?view=sql-server-ver17)).

So the general interview answer is reliable, while exact allocation locks, caching, restart behavior, and multi-row insert ordering are engine-specific.

### 11.4 What the row lock changes

- If T1 holds an X lock on an unrelated row, T2 may still insert and allocate an ID.
- If T2 needs a conflicting lock or unique key held by T1, T2 may wait.
- An auto-increment generator has its own concurrency mechanism. Do not assume a row X lock freezes the generator/table.
- InnoDB's `AUTO-INC` modes can serialize portions of ID allocation for statements, but rollback gaps still remain.

### 11.5 Never attach business meaning to ID continuity

Do not infer:

- `id=102` committed after `id=101`;
- every smaller ID exists;
- number of rows equals maximum ID;
- a missing ID means deletion or fraud;
- IDs form a gapless invoice/legal document sequence.

If the business requires gapless audited document numbering, design a separate serialized allocation workflow under the jurisdiction's rules. It reduces concurrency and needs explicit treatment of cancellations/voids; an auto-increment primary key is not that mechanism.

## 12. Distributed concurrency control

### 12.1 Put authority where the data lives

If all application instances write one primary database, the database is already a shared serialization point:

~~~text
App A ----+
          +--> primary DB: UNIQUE / conditional UPDATE / transaction
App B ----+
~~~

This is normally safer than taking a separate Redis lock and then writing the database, because a lock service and database can disagree after failures. Prefer the database's own atomic primitive when the invariant is in that database.

### 12.2 Strategy A: constraints and conditional writes

Examples:

- `UNIQUE(show_id, seat_id)` for at most one active ownership record, possibly modeled with status/history carefully;
- `UPDATE ... WHERE available >= requested` for stock;
- state transition `WHERE status = 'PENDING'`;
- OCC `WHERE version = expected`.

This couples the decision directly to the source of truth.

### 12.3 Strategy B: single writer per key

Route all commands for the same invariant key to one ordered partition/actor:

~~~text
book(A7) --+
cancel(A7) +--> partition hash(A7) --> one consumer --> seat store
book(B3) --+          different key can run in parallel
~~~

Benefits:

- simple sequential reasoning per key;
- no simultaneous writers inside that partition;
- good fit for event-sourced/actor-like state.

Requirements and costs:

- the routing key must match the invariant boundary;
- broker ordering is usually per partition, not global;
- failover must prevent two active owners/split brain;
- commands still need IDs/deduplication;
- a celebrity/hot item can overload one partition;
- cross-key invariants need additional coordination.

### 12.4 Strategy C: database pessimistic locking

Many application instances can issue `SELECT ... FOR UPDATE` against one authoritative primary. The database's lock manager coordinates them. This is distributed **caller** concurrency with centralized database coordination; it does not require an application-level distributed lock.

### 12.5 Strategy D: distributed lock/lease service

Use a coordination system when processes must serialize access to something that cannot perform the authoritative condition itself, for example:

- electing one scheduler/leader;
- coordinating an external file/appliance;
- preventing two workers from running one non-idempotent maintenance job;
- protecting a cross-process critical section whose resource honors fencing.

The service must define:

- mutual-exclusion/linearizability guarantee;
- behavior under partitions and failover;
- ownership identity and safe unlock;
- lease/session expiry and renewal;
- fairness and queueing, if any;
- recovery after an ambiguous acquire response;
- fencing token support;
- durability and availability assumptions.

### 12.6 Why a Redis `SET NX` plus TTL needs care

A single-instance lease can be useful when occasional duplicate work is acceptable, but its failure guarantee must match the business. Redis documentation explains that asynchronous primary-replica failover can allow a second client to acquire a lock that the old primary had granted before replication, violating mutual exclusion ([Redis distributed locks](https://redis.io/docs/latest/develop/clients/patterns/distributed-locks/)).

The lesson is not "Redis locks never work." It is:

> Define the acceptable failure behavior, then choose and configure an algorithm whose documented model provides it.

### 12.7 Leases and the stale-holder problem

Leases expire so that a crashed holder does not block forever:

~~~text
Client A gets lease
Client A pauses for a long GC/network delay
Lease expires
Client B gets new lease and writes
Client A resumes and still believes its old work is valid
~~~

A TTL freed the lock but did not stop A's process. If A can still reach the protected resource, it can corrupt B's newer work.

### 12.8 Fencing tokens

A lock service can issue monotonically increasing tokens:

~~~text
A acquires -> token 41
A pauses; lease expires
B acquires -> token 42 -> resource records 42 and accepts B
A resumes  -> sends 41 -> resource rejects stale token
~~~

The resource must persist the largest accepted token and reject lower tokens. If the external resource cannot check fencing tokens, the lock alone cannot fully fence a paused former owner.

Hazelcast's FencedLock documentation uses this exact model and notes the need to pass the token to external resources ([Hazelcast FencedLock](https://docs.hazelcast.com/hazelcast/5.5/data-structures/fencedlock)). Its CP subsystem chooses consistency over availability during a partition when configured for CP behavior ([Hazelcast CP subsystem](https://docs.hazelcast.com/hazelcast/5.7/cp-subsystem/cp-subsystem)). Other products have different guarantees; this is an example, not a default recommendation.

### 12.9 Safe release

Never implement release as an unconditional `DELETE lock_key`. Client A's lease may expire, client B may acquire the same key, and delayed A may delete B's lock. Release must atomically verify an ownership token/session before deleting.

### 12.10 Read replicas can invalidate the reasoning

Unsafe:

~~~text
acquire lock via authoritative service
read stale availability from async replica
make decision
write leader
~~~

The critical read must satisfy the freshness needed by the invariant. A strong lock does not make an unrelated replica current.

### 12.11 Cross-shard invariants

If two items are on different shards, a per-shard transaction/lock cannot by itself enforce a cross-shard invariant. Options include:

- redesign ownership so invariant data shares a shard;
- reservation/escrow rights allocated per shard;
- distributed transaction/2PC among compatible participants;
- Saga with explicit temporary states when atomicity is not required;
- one coordinator/ledger owning the invariant.

## 13. Why `synchronized` is not the fix - complete answer

### 13.1 What it does guarantee

For threads that use the **same Java monitor**:

- at most one holds that monitor at a time;
- monitor release/acquisition creates Java memory-visibility ordering;
- an exception still releases the intrinsic lock when the synchronized method/block exits.

### 13.2 What it does not guarantee

- coordination across JVMs, containers, hosts, or serverless invocations;
- coordination between two different object instances;
- database rollback;
- protection from a script/admin/other service that bypasses the method;
- idempotency after client retry;
- durability after process crash;
- fairness or bounded waiting;
- atomicity across a remote database and external API.

### 13.3 Why adding more application servers defeats it

Horizontal scaling deliberately makes many independent processes able to handle requests. A local mutex serializes only a subset of traffic and becomes correctness theater: tests on one instance pass, while production requests routed to different instances race.

### 13.4 Better interview response

> "`synchronized` can protect same-monitor threads inside one JVM, but each service instance has its own memory. I would enforce the invariant at the shared authority: ideally a unique constraint or conditional database update, then use OCC or a short row/range lock if needed. If the protected resource is external, I would use a coordination service with explicit partition semantics, lease ownership, and fencing, not assume a TTL key behaves like a JVM mutex."

## 14. Deadlock strategy

### 14.1 What a deadlock is

~~~text
T1 holds lock on account A; waits for B
T2 holds lock on account B; waits for A

T1 --waits-for--> T2
 ^                 |
 |                 v
 +----waits-for----+
~~~

Neither can proceed because neither releases its held lock while waiting.

### 14.2 Four classic necessary conditions

A deadlock requires all four in the classic model:

1. **Mutual exclusion:** a resource has an incompatible exclusive use.
2. **Hold and wait:** a transaction holds one resource while requesting another.
3. **No preemption:** the resource is not simply taken away; the transaction must release/rollback.
4. **Circular wait:** a cycle exists in the wait-for relation.

Break any one and that particular deadlock cannot form. Database designs usually focus on consistent ordering, short transactions, detection, and victim rollback rather than eliminating all locking.

### 14.3 Prevention

- Acquire resources in one global order, such as increasing account ID.
- Keep transactions small and do not wait for users/network calls while holding locks.
- Use a suitable index so the query does not lock a large scan/range.
- Lock the full needed set in a consistent phase when practical.
- Avoid S-to-X upgrade patterns; request the required mode deliberately.
- Reduce lock granularity only when it does not create excessive lock bookkeeping.
- Use atomic one-statement updates instead of multi-step locking when possible.
- Partition hot resources or use reservation/escrow to reduce one hot lock.

MySQL recommends short transactions and consistent order when updating several rows/tables ([MySQL 8.4: InnoDB deadlocks](https://dev.mysql.com/doc/refman/8.4/en/innodb-deadlocks.html)).

### 14.4 Detection and victim selection

A lock manager can maintain a **wait-for graph**:

~~~text
node = transaction
edge T1 -> T2 = T1 waits for a lock held by T2
cycle = deadlock
~~~

When it detects a cycle, it chooses a victim and rolls it back. Possible internal heuristics include work already done, locks held, age, rollback cost, or priority. Exact selection is engine-specific and can change; application correctness must not depend on which transaction loses.

### 14.5 Timeout

If detection is disabled, too expensive, or cannot see all participants, a lock wait timeout can break indefinite waiting. But timeout alone cannot distinguish:

- a true deadlock;
- a slow transaction that would eventually finish;
- overload or a partition.

Therefore it may abort useful work and often detects the problem later than a wait-cycle detector.

### 14.6 Application recovery

Treat a deadlock victim like a transient concurrency failure:

1. roll back the **entire** transaction;
2. retry from the beginning with a stable business idempotency key;
3. use bounded retries and randomized backoff;
4. re-read data and recheck all business predicates;
5. do not repeat an external side effect unless its idempotency/status protocol makes that safe;
6. alert/investigate if the rate or age exceeds an objective.

Do not retry only the last statement; earlier reads and decisions belonged to the aborted snapshot/lock context.

### 14.7 Distributed deadlocks

With several lock managers, no node may initially see the complete cycle:

~~~text
Service/DB A sees: T1 waits for T2
Service/DB B sees: T2 waits for T1
~~~

Approaches include a global/merged wait-for graph, edge-chasing probes, transaction timestamps with wait-die/wound-wait policies, leases, or timeouts. They add messages, failure ambiguity, and false suspicions. Avoid holding independent cross-service locks when ownership redesign, ordered acquisition, or a transaction/workflow protocol can express the operation more safely.

### 14.8 Deadlock versus starvation versus livelock

| Problem | Meaning |
|---|---|
| Deadlock | Participants wait in a cycle; none can progress |
| Starvation | One participant repeatedly loses/waits while others progress |
| Livelock | Participants keep reacting/retrying but make no useful progress |

Aggressive OCC retry can livelock or starve even though it holds no blocking database lock.

## 15. Trade-offs and decision matrix

### 15.1 Optimistic versus pessimistic

| Question | Optimistic | Pessimistic |
|---|---|---|
| Assumption | Conflicts are uncommon | Conflicts likely or costly |
| When conflict appears | Validate/write time | Before/during protected access |
| Waiting | Usually no application-held blocking lock | Other workers wait/fail/timeout |
| Wasted work | Conflicting attempts may be discarded | Less speculative work |
| Deadlock | Version checks alone do not deadlock, but surrounding DB work can | Natural risk with multiple locks |
| Starvation/retry storm | Possible under contention | Queue starvation/fairness issues possible |
| Long user think time | Version token can survive without DB lock | Do not keep DB lock open |
| Best example | Edit profile/document version | Short high-contention inventory decision |

### 15.2 Lock granularity

- **Coarse lock:** fewer lock operations and simple invariant, but unrelated requests block.
- **Fine lock:** more parallelism, but more bookkeeping and complex deadlock order.
- **Per-key sharding:** parallel across keys, but one hot key cannot be parallelized without changing the invariant/algorithm.

### 15.3 Strong versus weak ordering

- Stronger ordering simplifies correctness at the cost of coordination latency and reduced partition availability.
- Weaker ordering can improve locality/availability but pushes conflict, stale-read, and user-session logic upward.
- Use strong consistency for the authoritative state transition; publish derived projections asynchronously where staleness is acceptable.

### 15.4 Wait, reject, queue, or reserve

For a contested resource, choose product semantics explicitly:

- **Wait:** fair queue, but long tail latency and cancellations.
- **Reject immediately:** simple and bounded, but users retry.
- **Queue command:** smooths bursts, but result becomes asynchronous and queue position matters.
- **Temporary reservation:** improves checkout UX, but requires expiry/reconciliation and can reduce apparent inventory.

### 15.5 When to avoid a distributed lock

Avoid it when:

- a unique constraint/conditional write on the authoritative store suffices;
- the critical section spans a long or irreversible workflow;
- the external resource cannot honor fencing/idempotency;
- the team cannot state partition/failover guarantees;
- a single-writer/ownership boundary is simpler;
- correctness requires atomic multi-resource commit, not merely mutual exclusion.

## 16. Real-system examples

> Every named-system design below is **illustrative**, not a claim about the company's current private implementation. The point is to practice matching guarantees to product invariants.

### 16.1 Amazon-like last-item purchase

Invariant: confirmed/reserved units never exceed sellable stock.

~~~text
Purchase API
    |
    | idempotency_key = order-91/sku-7
    v
Inventory authority
    |
    | atomic: available >= 1 ? available--, reserved++ : reject
    v
Reservation(order-91, expires_at, state=HELD)
    |
    +--> payment outside inventory transaction
             |
             +--> confirm reservation or release/reconcile
~~~

- Use an atomic conditional decrement or short pessimistic transaction.
- Do not lock one stock row during the payment call.
- A unique reservation operation ID prevents duplicate reservation.
- The visible "items left" cache may be stale; the authoritative reserve result decides.
- For very hot stock, preallocate escrow/reservation rights to partitions or queue purchase commands, accepting the resulting complexity.

### 16.2 Uber-like driver assignment

Invariant: one driver has at most one active accepted trip.

~~~sql
UPDATE drivers
SET state = 'ASSIGNED', trip_id = :trip, version = version + 1
WHERE driver_id = :driver
  AND state = 'AVAILABLE'
  AND version = :expected;
~~~

Two dispatchers can propose the same driver; only one conditional transition succeeds. A uniqueness constraint on active assignment can be a second guardrail. Location/ETA projections can be eventually consistent, while assignment ownership needs a stronger authoritative decision.

### 16.3 WhatsApp-like conversation ordering

One global order for all messages is unnecessary. A conversation can have a per-conversation sequence/causal relation:

~~~text
conversation-42 partition:
  m100 "Meet at 5"
  m101 reply-to=m100 "OK"
~~~

- The reply should not be shown without its causal parent when the product requires that context.
- Independent conversations can progress concurrently.
- Offline devices need deduplication, per-device/client message IDs, and reconciliation.
- Read receipts or unread counts may lag without compromising the durable message order.

### 16.4 YouTube-like views and likes

Different facts need different control:

- Approximate/aggregated view counts can be buffered and eventually merged; a globally locked counter would be a hotspot.
- "One like by user U on video V" can use `UNIQUE(user_id, video_id)` plus idempotent insert/delete.
- The displayed like count can lag behind the authoritative like edges.
- Creator payment/accounting totals require a stronger auditable aggregation than the public approximate counter.

### 16.5 Netflix-like playback state

Resume position is often a conflict-tolerant per-profile value:

~~~text
TV writes episode 3 @ 08:10, version 30
Phone writes episode 3 @ 07:50 from stale version 29
~~~

The service might reject stale versions, merge using server-assigned ordering, or use product-specific "furthest meaningful progress" rules. Last-write-wins based only on unsynchronized device clocks is risky. Entitlement and simultaneous-stream limits need stronger authoritative checks than the convenience resume marker.

### 16.6 Twitter/X-like likes, follows, and counters

- `UNIQUE(user_id, tweet_id)` makes a like edge idempotent.
- A derived like count can be eventually updated and temporarily differ from the edge set.
- Follow/unfollow commands need operation ordering so an old retry does not undo a newer intention; versions or command IDs help.
- Feed generation can tolerate replica/caching staleness more than account/security state.

### 16.7 One pattern across all examples

~~~text
strong authoritative fact
   seat owner / stock reservation / driver assignment / unique like edge
                         |
                         +---- asynchronous events ---->
                                  caches, counters, feeds, analytics
                                  (bounded/eventual staleness)
~~~

Do not impose one global consistency mechanism on every derived view.

## 17. Worked HLD: high-concurrency ticket booking

### 17.1 Requirements and invariants

Functional:

- show available seats;
- temporarily hold selected seats;
- pay and confirm;
- expire abandoned holds;
- safely retry requests.

Correctness:

- at most one active owner/hold per seat;
- a confirmed booking is not expired by the hold sweeper;
- one checkout operation is not charged twice;
- a late payment outcome is reconciled, not silently dropped.

The availability map may be stale; the **hold attempt** is authoritative.

### 17.2 Components

~~~text
Clients
   |
Load balancer
   |
Booking API ----> cache/read replicas (seat-map hints only)
   |
   +-----------> primary booking DB
   |                seats / holds / bookings / idempotency
   |
   +-----------> Payment provider
   |
   +-----------> outbox -> events -> email/cache/analytics
                         ^
Expiry/recovery worker --+
~~~

### 17.3 Example state machine

~~~text
AVAILABLE -> HELD -> CONFIRMED
               |
               +-> EXPIRED/RELEASED -> AVAILABLE
~~~

Transitions use a version/current-state predicate. Do not let a late expiry update overwrite `CONFIRMED`:

~~~sql
UPDATE seats
SET state = 'AVAILABLE', hold_id = NULL, version = version + 1
WHERE seat_id = :seat
  AND state = 'HELD'
  AND hold_id = :hold
  AND hold_expires_at < :database_now;
~~~

### 17.4 Hold transaction

For one seat, a conditional update is enough. For several adjacent seats, one short transaction must claim all or roll all back:

~~~text
BEGIN
  claim A7 if AVAILABLE
  claim A8 if AVAILABLE
  insert hold(operation_id unique, expires_at)
COMMIT
~~~

Acquire locks/modify seats in canonical seat-ID order to reduce deadlocks. A unique operation ID handles repeated client submissions. If any seat is unavailable, roll back the group.

### 17.5 Payment and unknown outcome

1. Commit the hold.
2. Send the hold/checkout ID as the payment idempotency key.
3. If payment response is lost, query the payment operation rather than assume failure.
4. In a new short transaction, change the exact matching live hold to `CONFIRMED`.
5. If payment succeeds after expiry, apply a defined business path: reacquire if possible, otherwise refund/manual review. Do not pretend it never happened.

### 17.6 Hot-show scaling

- Partition by show/venue only if the partition boundary preserves the seat invariant.
- Cache the seat map for browsing, never as the final ownership authority.
- Apply admission control/virtual waiting room to protect the primary during a burst.
- Queue or shed duplicate refreshes.
- Consider per-show command partitioning, but plan for a single hot show.
- Avoid one lock for the whole venue unless its simplicity is worth severe serialization.

### 17.7 Failure checklist

| Failure | Safe behavior |
|---|---|
| Booking API crashes after hold commit | Retry/query by operation ID returns the hold |
| Response lost | Client does not create a second logical operation |
| Expiry worker races with confirm | State + hold ID + version predicate lets one valid transition win |
| Two workers expire same hold | Conditional update makes repeats harmless |
| Database deadlock | Entire hold transaction rolls back and retries with jitter |
| Cache says available but DB says held | DB result wins; invalidate/update cache asynchronously |
| Payment succeeds but confirm fails | Durable recovery/reconciliation continues confirmation or refund |

### 18.3 Answer spine: concurrent booking/oversell

Use these cues to reconstruct an answer live:

~~~text
INVARIANT
  one owner / count never below zero
AUTHORITY
  primary inventory/booking store, not cache
ATOMIC DECISION
  unique constraint or UPDATE ... WHERE available/status/version
USER FLOW
  short durable hold; payment outside transaction; confirm/release
RETRIES
  operation ID + idempotency; conflict is expected
FAILURES

  partition by invariant key, admission control, hot-key plan

  conflict, wait, expiry, oversell invariant, reconciliation
~~~

### 18.4 Answer spine: optimistic versus pessimistic

~~~text
CONFLICT RATE -> low favors OCC; high may favor waiting/reservation
RETRY COST    -> cheap favors OCC; expensive favors pessimistic
SCOPE         -> one item, range, or cross-row invariant
DURATION      -> never hold DB lock through user/remote call
MECHANICS     -> version conditional vs SELECT FOR UPDATE
FAILURE       -> retry storm/starvation vs deadlock/wait timeout
GUARDRAIL     -> constraint/atomic predicate remains authoritative
~~~

### 18.5 Answer spine: isolation levels

~~~text
PURPOSE       -> control visible concurrent interleavings
PHENOMENA     -> dirty, non-repeatable, phantom; add lost update/write skew
LADDER        -> RU, RC, RR, Serializable
VENDOR CAVEAT -> same label can be stronger/different; PG RU=RC, PG RR no phantom
SERIALIZABLE  -> may block/abort; retry whole transaction
BOUNDARY      -> does not solve external idempotency or replica lag
~~~

### 18.6 Answer spine: distributed lock

~~~text
WHY LOCAL FAILS -> different processes/monitors
FIRST ALTERNATIVE -> condition/constraint at source of truth
LOCK CONTRACT -> mutual exclusion, partition/failover, owner, lease/session
STALE OWNER -> monotonic fencing token checked by resource
UNKNOWN OUTCOME -> acquire/release IDs and status recovery
AVAILABILITY -> CP coordination may reject minority during partition
OPERATIONS -> expiry, waiters, contention, renewal, stuck work
~~~

### 18.7 Answer spine: deadlock

~~~text
DEFINE        -> circular wait in wait-for graph
PREVENT       -> canonical order, short tx, good indexes, no network calls
DETECT        -> cycle detection or timeout; choose victim
RECOVER       -> rollback entire tx, bounded retry + jitter + idempotency
MEASURE       -> deadlock rate, lock wait, victim query/keys
DISTRIBUTED   -> fragmented graph makes detection harder
~~~

### 18.8 A strong two-minute answer shape

> "I start by naming the invariant and its authoritative owner. If the rule is local to one database, I prefer a unique constraint or one conditional update because the check and write are atomic at the source of truth. Under low contention, a version column gives optimistic concurrency and the loser reloads or retries safely. Under high contention or costly retries, I may use a short `FOR UPDATE` transaction or a reservation, but never hold it across payment or user think time. Java `synchronized` cannot coordinate separate instances. A distributed lock is a later option for external resources and needs partition semantics, leases, owner-safe release, and fencing. I also define isolation, idempotency, deadlock retry, stale-replica behavior, and hot-key scaling rather than treating 'use a transaction' as the complete answer."

## 19. Common mistakes and edge cases

| Weak statement | Interview-ready correction |
|---|---|
| "Use `synchronized`" | "It protects the same monitor in one JVM; enforce the invariant at shared authority" |
| "Transactions solve race conditions" | "Only with an operation/isolation level that protects the invariant" |
| "Read then update in a transaction is safe" | "Not automatically; show the permitted interleaving and use predicate/version/lock/serializable" |
| "Pessimistic means Read Uncommitted" | "Pessimistic reserves conflicting access before the work; RU is a weak visibility level" |
| "Optimistic locking has no locks" | "The application holds no long-lived blocking lock, but the storage engine still synchronizes the atomic conditional write internally" |
| "Repeatable Read always permits phantoms" | "That is the standard minimum table; PostgreSQL's RR prevents them, while other anomalies can remain" |
| "Serializable runs one transaction at a time" | "It produces an outcome equivalent to some serial order, often through blocking or abort/retry" |
| "X lock blocks every read" | "Snapshot reads in MVCC may continue; locking reads/writers have different rules" |
| "Redis lock makes it distributed-safe" | "State algorithm, failover/partition model, lease ownership, and fencing" |
| "TTL prevents stale lock holders" | "TTL frees the lease; fencing prevents an old holder's late side effect" |
| "Auto increment is consecutive" | "It is a unique-ID mechanism; rollback, conflict, cache, or crash can leave gaps" |
| "Retry deadlock statement" | "Retry the whole transaction from a fresh read, idempotently" |
| "Idempotency prevents seat contention" | "It deduplicates the same operation; concurrency control arbitrates different operations" |
| "A cache lock protects DB truth" | "The DB can be written without that lock or disagree after failure; prefer an atomic DB guardrail" |
| "Strong consistency everywhere" | "Use strong authority for invariants and weaker derived projections where staleness is acceptable" |

### 19.1 Edge-case drills

#### Two users request the last item

Use one authoritative conditional decrement/reservation. One succeeds; the other gets a normal sold-out conflict.

#### Same user clicks twice

Same business idempotency key returns the existing operation. It is not a second competitor.

#### Conflict response is lost

Query by operation ID. A timeout is not proof of failure.

#### OCC write succeeds, response is lost, caller retries with old version

The version check will fail, but that alone does not tell whether this caller's earlier attempt succeeded. The operation/idempotency record answers that question.

#### Lock service grants lease but response is lost

Acquire has unknown outcome. Use a client request/owner ID and a status/recovery protocol; do not blindly assume no lease exists.

#### Client A releases after its lease expired and B acquired

Ownership-checked release prevents A from deleting B's lock.

#### Transaction reads from replica and writes primary

Stale state can cause an invalid decision or avoidable conflict. Route correctness reads to the authoritative snapshot/leader or use a version token with an adequate freshness guarantee.

#### Serializable transaction fails

This can be normal under contention. Retry the complete transaction using fresh reads and idempotent external boundaries.

#### Lock order is canonical but deadlocks still occur

Hidden locks from foreign keys, triggers, range scans, indexes, or lock upgrades may violate the assumed order. Inspect the engine's deadlock graph/query plan.

#### Auto-increment gap appears without an explicit rollback

Failed inserts, conflict-handling paths, allocation caches, or database crashes can also create gaps depending on engine. Do not diagnose deletion from a gap alone.

#### Hold expires exactly as payment completes

Use database/server time, state + hold-ID/version predicates, durable payment operation status, and a defined reconciliation/refund policy. Wall-clock comparison alone does not erase a completed external charge.

## 20. Testing, metrics, and operational readiness

### 20.1 Correctness tests

- Run many workers against a one-unit inventory and assert at most one success.
- Randomize delays between reads and writes to widen race windows.
- Repeat the same idempotency key concurrently and assert one business effect.
- Kill a worker after commit but before response.
- Pause a lock holder beyond its lease, then verify fencing rejects stale work.
- Race hold expiry against confirmation.
- Force transactions to acquire two rows in opposite order and verify victim retry.
- Read from a delayed replica and prove the design does not use it for authoritative decisions.
- Property-test invariants after long random command histories.

A test that passes once is not proof. Concurrency bugs require repeatable stress, fault injection, and invariant checking.

### 20.2 Database/OCC metrics

- conditional-write conflict rate by resource/key;
- optimistic retry count and exhausted retries;
- serialization failures and deadlock victims;
- lock wait count/time and oldest waiting transaction;
- transaction duration and oldest open transaction;
- rows scanned/locked versus rows changed;
- hot keys/partitions;
- replica lag for correctness-sensitive reads.

### 20.3 Distributed-lock metrics

- acquire latency, failures, and wait queue length;
- lease renewal latency/failure;
- lease expiry while work is active;
- stale fencing-token rejections;
- ownership-mismatch release attempts;
- current holder age and critical-section duration;
- coordination quorum/leader availability;
- ambiguous acquire/release recoveries.

### 20.4 Business invariant metrics

- negative inventory or duplicate active ownership: should trigger immediate investigation;
- expired-but-paid and paid-but-unconfirmed operations;
- reconciliation mismatch count/age;
- duplicate idempotency-key payload conflicts;
- manual-review queue age;
- holds expired, confirmed, released, and abandoned by show/SKU.

Technical locks can look healthy while the business invariant is broken; measure both.

## 21. Compact review sheet

~~~text
START WITH THE INVARIANT
  one seat owner, stock >= 0, one active driver trip

LOCAL MUTEX
  same memory + same monitor only
  Java synchronized does not coordinate service instances

TRANSACTION
  atomicity != isolation != distributed replica consistency
  keep boundaries short; external effects are outside unless explicitly transactional

OCC
  read version -> compute -> conditional write where version matches
  low contention / cheap retry
  beware retry storms, stale replicas, ABA, write skew

PESSIMISTIC
  reserve row/range before protected work
  high contention / expensive retry / short transaction
  beware waits, deadlocks, long locks

S/X
  S+S compatible; X conflicts with S/X
  MVCC snapshot reads may not take S row locks

ISOLATION
  RU -> RC -> RR -> Serializable (standard ladder)
  dirty -> non-repeatable -> phantom
  add lost update and write skew; check vendor semantics

ORDERING
  linearizable respects real time
  sequential respects per-client program order, not cross-client real time
  causal orders dependencies
  eventual promises convergence, not immediate freshness
  strict serializable = serializable transactions + real-time order

AUTO-INCREMENT
  allocated value usually not rolled back; gaps are normal
  unique identifier, not gapless business sequence

DEADLOCK
  circular wait
  prevent with canonical order + short tx + good indexes
  detect cycle/timeout -> rollback victim -> retry whole tx with jitter/idempotency

DISTRIBUTED LOCK
  use only when authoritative conditional write/single writer is insufficient
  define partitions, leases, owner-safe release, ambiguous outcomes, fencing
~~~

## 22. Say-out-loud 30-second narrative

> "Distributed concurrency control protects an invariant when requests on different threads or service instances race on shared state. Java `synchronized` works only for the same monitor in one JVM, so I put the atomic decision at the authoritative store: ideally a constraint or conditional update, then OCC for rare cheap conflicts or a short pessimistic lock for likely expensive ones. Isolation controls transaction interleavings, while linearizability and eventual consistency describe distributed visibility. I expect deadlock or serialization retries, keep transactions short, use idempotency for duplicate delivery, and require leases plus fencing if I truly need a distributed lock."

## 23. True prerequisite gaps and study next

### True prerequisite gaps remaining

**None required to understand this chapter at HLD interview level.** Shared mutable state, atomic operations, ACID, commit/rollback, MVCC, timeouts, and local versus distributed transaction boundaries were introduced inline.

### Study next - related, not prerequisites

1. **MVCC internals:** undo/version chains, snapshots, vacuum/purge, and serialization graphs.
2. **Consensus and coordination:** Raft/Paxos, sessions, leases, leader election, and quorum availability.
3. **Escrow and reservation algorithms:** splitting limited rights safely across partitions.
4. **Distributed SQL concurrency:** timestamp ordering, TrueTime-style uncertainty, per-shard consensus, and cross-shard 2PC.
5. **Conflict-free replicated data types (CRDTs):** when concurrent updates can merge instead of lock.
