# SQL vs NoSQL

> Beginner-first HLD notes expanded from: **structure, nature, scalability, properties, and NoSQL types: key-value, document, column-wise, and graph databases**.
>
> The central interview lesson is: **do not choose a database from the label alone. Start from the data relationships, access patterns, correctness rules, transaction boundary, and scale; then choose the database whose natural model makes those requirements easiest to satisfy.**

## Scope, assumptions, and evidence labels

I assume the rough bullet **"column wise"** means a **wide-column / column-family NoSQL database**, such as Apache Cassandra. That is different from a **columnar analytical database or warehouse**, which stores values by column to accelerate scans and aggregations. Section 6.5 separates them explicitly.

The phrase **"SQL database"** is commonly used in interviews to mean a **relational database management system (RDBMS)** such as PostgreSQL or MySQL. Strictly speaking, SQL is a query language and the relational model is a data model. Some non-relational databases have SQL-like languages, and modern relational databases can store JSON. The chapter uses the common interview wording while preserving this distinction.

Evidence labels used in this chapter:

- **Verified concept** - supported by first-party documentation linked nearby.
- **Illustrative design** - a reasonable design for a familiar product; it is not a claim about that company's private current architecture.
- **Self-reported interview experience** - a candidate described the prompt in a public post. It is useful practice evidence, not an official or independently verified company question bank.
- **Illustrative number** - chosen only to explain mechanics; it is not claimed production traffic.

No unsourced production scale numbers are stated. Product features can change, so the References section links the documentation used for current feature claims.

### The most important corrections up front

1. **SQL does not mean "small" or "vertical scaling only."** Relational databases can use indexes, replicas, table partitioning, and application-level or distributed sharding. Distributed SQL systems also exist. Cross-shard joins, constraints, and transactions can make horizontal scaling harder, but it is not impossible.
2. **NoSQL does not mean "no schema."** Every useful application assumes fields, types, keys, and invariants somewhere. The schema may be enforced by the database, validation rules, application code, or readers.
3. **SQL does not own ACID, and NoSQL does not imply BASE/eventual consistency.** MongoDB and DynamoDB support multi-item transactions in defined scopes; Neo4j documents ACID transactions. A relational read replica can return stale data. Data model and consistency model are different axes.
4. **NoSQL is not one database model.** Key-value, document, wide-column, and graph databases solve different problems. "Use NoSQL" is incomplete without naming the family and explaining the access pattern.
5. **Structured versus unstructured is not the main decision.** PostgreSQL can query JSON, MongoDB can validate documents, and videos should normally go to object/blob storage rather than either SQL rows or MongoDB documents.
6. **Read-heavy versus write-heavy is also insufficient.** Either family can serve heavy reads with indexes, caches, and replicas. Write behavior depends on storage engine, indexes, durability, replication, contention, and partitioning - not merely the SQL/NoSQL label.
7. **A database choice is often per workload, not per whole product.** One system may use relational storage for orders, key-value storage for sessions, wide-column storage for event history, a search index for text, and object storage for media.
8. **Horizontal scale is not automatic safety.** A bad partition key can create one hot shard even when a cluster has hundreds of nodes.

## 1. Plain-language foundation

### 1.1 A database is an organized source of remembered state

Suppose an online shop must remember:

- customers;
- products;
- stock;
- shopping carts;
- orders and payments.

Keeping this only in application memory fails when the process restarts. Plain files survive a restart, but concurrent updates, searching, crash recovery, and enforcing correctness quickly become difficult. A database provides an organized way to store, find, change, protect, and recover that state.

The database choice controls which questions are easy and which are expensive.

```text
Easy question in one model              Potentially awkward question in another
----------------------------------      ---------------------------------------
Find order by order_id                  Traverse friends-of-friends repeatedly
Fetch session by token                  Join arbitrary entities by many fields
Fetch messages for chat + time range    Atomically update unrelated partitions
Traverse user -> follows -> follows     Scan every key for an ad-hoc report
```

There is no universally best database because systems ask different questions and require different correctness guarantees.

### 1.2 SQL/relational: organized linked tables

A relational database stores data in tables. Each table has rows and named columns, and keys express identity and relationships.

```text
USERS                               ORDERS
+---------+--------+               +----------+---------+---------+
| user_id | name   |               | order_id | user_id | status  |
+---------+--------+               +----------+---------+---------+
| 42      | Asha   |<--------------| 9001     | 42      | PAID    |
+---------+--------+  foreign key  +----------+---------+---------+
```

The relationship is stored as data: `ORDERS.user_id` refers to `USERS.user_id`. SQL can join the tables:

```sql
SELECT o.order_id, u.name
FROM orders o
JOIN users u ON u.user_id = o.user_id
WHERE o.order_id = 9001;
```

The database can also enforce rules such as:

- every order ID is unique;
- an order's user must exist;
- quantity cannot be negative;
- either all steps of a transaction commit or none do.

### 1.3 NoSQL: several non-relational ways to organize data

**NoSQL** is an umbrella term, commonly read as "not only SQL." It includes several models:

```text
Key-value        session:abc -> { userId: 42, expiresAt: ... }

Document         { orderId: 9001, customer: {...}, items: [...] }

Wide-column      (chat42, 2026-07, timestamp) -> message columns

Graph            (Asha)-[:FOLLOWS]->(Ben)-[:FOLLOWS]->(Chen)
```

These shapes are not interchangeable:

- A key-value store is natural when the application knows the exact key.
- A document store is natural when one aggregate is usually read and written together.
- A wide-column store is natural for very large partition-key-oriented histories.
- A graph database is natural when relationship traversal is itself the main query.

### 1.4 The same order can be modeled differently

Relational form separates reusable facts:

```text
orders          order_items          products
order 9001 ---> item(product 7) ---> product 7 current details
           \--> item(product 9) ---> product 9 current details
```

Document form can keep the read unit together:

```json
{
  "orderId": 9001,
  "userId": 42,
  "status": "PAID",
  "items": [
    { "productId": 7, "nameAtPurchase": "Keyboard", "priceAtPurchase": 2500 },
    { "productId": 9, "nameAtPurchase": "Mouse", "priceAtPurchase": 800 }
  ]
}
```

Neither representation is automatically correct. For an order history, copying the product name and price **at purchase time** is valuable because the historical snapshot should not change when the catalog changes. For current product stock, duplicated copies would be dangerous because every copy must stay synchronized.

The decision follows meaning and update behavior, not fashion.

### 1.5 The best beginner default

When requirements and future queries are uncertain, a relational database is often a sensible starting point because it supports constraints, transactions, indexes, joins, and evolving queries well. This is a **heuristic, not a law**.

Choose a specialized NoSQL model when a concrete requirement makes it a better fit, for example:

- exact-key session access with very low latency;
- a huge message history queried by conversation and time;
- documents whose nested shape varies substantially by product category;
- repeated multi-hop traversal through a dense relationship network.

The interview-quality phrase is:

> "I would start with the simplest database that safely meets the known access patterns and invariants. I will introduce a specialized store only where its data model or scaling behavior removes a demonstrated bottleneck."

## 2. Inline prerequisites

These concepts are necessary to understand SQL versus NoSQL, so they are explained here rather than deferred.

### 2.1 Record, table, row, column, collection, and document

- A **record** is one logical item, such as one user or order.
- A relational **table** groups rows with named columns.
- A **row** is one table record; a **column** is one attribute such as `email`.
- A document database **collection** groups documents.
- A **document** is a nested object, commonly represented like JSON/BSON.
- In a wide-column database, a **partition** groups rows that share a partition key; clustering columns order rows inside it.

Words overlap between products. Always learn the exact semantics of the selected database rather than inferring them from a familiar word such as "table."

### 2.2 Schema

A **schema** describes allowed structure: field names, types, required fields, relationships, constraints, and sometimes indexes.

```text
Relational database:    DB rejects an order with an invalid foreign key.
Document validation:    DB rejects a document whose start_date is a string.
Application-only:       Service code checks that status is an allowed value.
Reader assumption:      Reporting job assumes price is numeric.
```

MongoDB uses a flexible schema by default but supports schema-validation rules for types and ranges. PostgreSQL supports typed columns and also JSON/JSONB. Therefore, "SQL has schema; NoSQL has none" is false. [MongoDB schema validation](https://www.mongodb.com/docs/manual/core/schema-validation/) and [PostgreSQL JSON types](https://www.postgresql.org/docs/current/datatype-json.html) are verified examples.

### 2.3 Primary key, foreign key, and partition key

- A **primary key** uniquely identifies a record: `order_id = 9001`.
- A **foreign key** says one record refers to another valid record: `orders.user_id -> users.user_id`.
- A **partition key** determines which partition/shard stores data in a distributed database.
- A **clustering/sort key** commonly determines order or uniqueness inside one partition.

The same field can play more than one role. In Cassandra, all performant queries are normally designed around the partition key; in DynamoDB, a composite primary key contains a partition key and sort key. [Cassandra's architecture overview](https://cassandra.apache.org/doc/stable/cassandra/architecture/overview.html) and [DynamoDB partition-key guidance](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-partition-key-design.html) describe these roles.

### 2.4 Query and access pattern

A **query** asks the database to read or change data. An **access pattern** is a recurring query shape plus its performance and consistency need.

```text
Weak requirement:    "Store messages."

Useful access patterns:
1. Append one message to a conversation.
2. Fetch the newest 50 messages for conversation C.
3. Page backward from timestamp T.
4. Deduplicate a retry by client_message_id.
5. Search a user's messages by arbitrary text.   <- different index/workload
```

NoSQL modeling is especially access-pattern-first. A table designed for query 2 may be intentionally duplicated into another table or index for query 4.

### 2.5 Index

An **index** is an auxiliary structure that accelerates a lookup, like a book's index. Without a useful index, the database may scan many records.

```text
Table: millions of orders

No index on user_id:  inspect many/all rows
Index on user_id:     navigate to matching row locations
```

Indexes trade faster reads for:

- extra storage;
- extra work on each write;
- cache/memory pressure;
- maintenance and compaction cost.

An index is not magic. A low-selectivity field, a mismatched column order, or a query that wraps the indexed column in an incompatible expression may still perform poorly.

### 2.6 Transaction and ACID

A **transaction** groups operations into one logical unit. A money transfer illustrates why:

```text
1. subtract 500 from account A
2. add      500 to account B
3. record the ledger entry
```

Committing only step 1 loses money. The required properties are summarized as ACID:

| Property | Plain meaning | Important nuance |
|---|---|---|
| **Atomicity** | All transaction changes commit, or none do | Not the same as thread-safe application code |
| **Consistency** | A valid transaction preserves declared/application invariants | Not CAP consistency; the database cannot infer missing business rules |
| **Isolation** | Concurrent transactions behave according to an isolation level | Different levels allow or prevent different anomalies |
| **Durability** | Once commit is acknowledged, results survive the failures covered by the guarantee | Depends on logging, replication, acknowledgement, and failure model |

PostgreSQL's transaction tutorial shows the all-or-nothing and hidden-intermediate-state behavior. [PostgreSQL transactions](https://www.postgresql.org/docs/current/tutorial-transactions.html)

### 2.7 Normalization and denormalization

**Normalization** separates facts to reduce inconsistent duplication:

```text
product 7 has one canonical current price row
order_items refer to product 7
```

**Denormalization** intentionally copies or precomputes data to make a known read fast:

```text
timeline_by_user contains copied tweet IDs
order item stores name_at_purchase and price_at_purchase
```

Denormalization trades read simplicity for:

- multiple writes;
- write amplification;
- temporary or permanent divergence;
- repair/backfill logic;
- harder deletions and privacy requests.

Relational systems can denormalize; NoSQL systems can reference other records. The difference is emphasis, not a hard boundary.

### 2.8 Replica, partition, and shard

- **Replication** copies the same data to more than one node for availability, durability, locality, or read scale.
- **Partitioning/sharding** splits different data across nodes for capacity and throughput.
- A **replica** may lag if replication is asynchronous.
- A **shard key/partition key** routes a record to its owning shard.

```text
                    Sharding: different data
              +--> Shard A: users 0..999
Client/Router-+--> Shard B: users 1000..1999
              +--> Shard C: users 2000..2999
                         |
                         +--> replica of Shard C: same data
```

For deeper mechanics, see [05. Server Scaling - Zero to Million to Billion](./05.%20Server%20Scaling%20-%20Zero%20to%20Million%20to%20Billion.md), [06. Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md), and [09. Key-Value Store](./09.%20Key-Value%20Store.md). The decision-level concepts needed here are self-contained.

### 2.9 Consistency models

- **Linearizable read/write:** operations appear to act on one up-to-date copy and respect real-time order.
- **Read-your-writes:** a client sees its own completed update.
- **Causal consistency:** causally related events are observed in causal order.
- **Eventual consistency:** if updates stop and communication/repair continue, replicas eventually converge.

These describe what clients may observe. They are not synonyms for SQL and NoSQL.

### 2.10 CAP and PACELC in one page

During a network partition, replicas cannot both guarantee:

- **C - linearizable consistency** for the affected operation; and
- **A - every request to a non-failing node receives a successful response** in the formal CAP sense.

A system may reject/delay an operation to preserve one view, or keep serving and reconcile divergent/stale state. CAP does **not** say "choose two of three during normal operation," because a distributed system cannot choose to prevent network partitions.

PACELC adds the healthy-path question: **else**, when there is no partition, does an operation trade lower **latency** for stronger **consistency** by coordinating with more replicas?

This choice can differ by operation and configuration. Do not label every SQL database "CP" and every NoSQL database "AP." See [02. CAP, PACELC and Replication Strategies](./02.%20CAP,%20PACELC%20and%20Replication%20Strategies.md) for the full treatment.

### 2.11 Database, cache, search engine, warehouse, and object store

Do not force every byte into the SQL/NoSQL choice:

| Need | Natural component |
|---|---|
| Authoritative orders and payments | Transactional database |
| Disposable acceleration copy | Cache |
| Full-text relevance and inverted index | Search engine |
| Large scans and aggregations over history | Analytical warehouse/columnar engine |
| Video, image, backup, large file | Object/blob storage |
| Events between services | Log/message broker |

A component may internally use database techniques, but its system role and guarantees differ.

## 3. Why this choice matters in system design

The database is where system correctness and load meet. A poor fit can make the normal request path expensive or make an important invariant nearly impossible to maintain.

### 3.1 It shapes the request path

For "show an order," a normalized relational model might join several indexed tables. A document model might fetch one embedded order. The document read can be simpler, but updating shared current product data across many documents can be harder.

```text
Relational read                         Embedded document read

orders -----+                          orders collection
items ------+--> join --> response      order_id --> one document --> response
products ---+
```

The right choice depends on whether the joined facts are shared/current or form one stable aggregate.

### 3.2 It shapes correctness

An order placement may need all of these to agree:

- order created once despite retries;
- payment authorization linked correctly;
- inventory not oversold;
- total derived from trusted prices;
- status moves only through valid transitions.

Unique constraints, conditional writes, transactions, isolation, and idempotency all matter. "NoSQL scales" is not a substitute for a correctness plan.

### 3.3 It shapes scaling boundaries

A partition key can make a request local to one shard:

```text
conversation_id = C42
        |
        v
hash(C42) --> shard 7 --> messages for C42
```

That is efficient until one conversation or celebrity key becomes disproportionately hot. A system that scales by key also inherits the key's traffic distribution.

### 3.4 It shapes future changes

Unknown future queries favor flexible querying and relationships. Extremely stable, high-volume access patterns can justify a specialized model. A design that embeds everything for today's page may require a costly backfill when tomorrow's page needs a different grouping.

### 3.5 It shapes operations and team risk

The theoretically perfect database can still be the wrong production choice if the team lacks:

- backup and restore practice;
- monitoring and on-call knowledge;
- migration tooling;
- capacity models;
- security and compliance support;
- managed-service availability in the deployment environment.

Operational maturity is a real trade-off and an excellent interview signal.

## 4. Four independent axes: the mental model that prevents most mistakes

Do not compress a database into one word. Evaluate at least four axes:

```text
1. DATA MODEL        tables | documents | key-value | wide-column | graph
2. QUERY INTERFACE   SQL | CQL | commands/API | Cypher | custom query language
3. STORAGE ENGINE    B-tree/page-oriented | LSM/SSTable | in-memory | hybrid
4. DISTRIBUTION      single node | leader/replicas | sharded | leaderless | consensus
```

Examples:

- Cassandra is a wide-column database, exposes SQL-like CQL, uses an LSM-style storage engine, and is partitioned/replicated across nodes.
- PostgreSQL is relational, exposes SQL, commonly uses page/B-tree structures and write-ahead logging, and can have replicas and partitioned tables.
- Neo4j is a property-graph database but documents ACID transactions. "NoSQL" did not determine its transaction property.
- DynamoDB is key-value/document-oriented and distributed, but also provides transactional APIs for coordinated all-or-nothing changes.

Therefore these implications are invalid:

```text
NoSQL              -/-> eventual consistency
Relational         -/-> single machine forever
SQL-like language  -/-> relational joins and foreign keys
JSON document      -/-> schema-free
LSM tree           -/-> NoSQL only
```

## 5. Relational / SQL databases in technical depth

Examples include PostgreSQL and MySQL. Product details vary, so reason from the chosen product rather than assuming every RDBMS has identical features.

### 5.1 Structure

Data is represented through relations/tables with typed columns and keys.

```sql
CREATE TABLE users (
    user_id      BIGINT PRIMARY KEY,
    email        TEXT NOT NULL UNIQUE
);

CREATE TABLE orders (
    order_id     BIGINT PRIMARY KEY,
    user_id      BIGINT NOT NULL REFERENCES users(user_id),
    status       TEXT NOT NULL,
    total_paise  BIGINT NOT NULL CHECK (total_paise >= 0),
    created_at   TIMESTAMP NOT NULL
);

CREATE INDEX orders_by_user_time
    ON orders(user_id, created_at DESC);
```

The schema communicates and enforces part of the domain model.

### 5.2 Relationships and joins

Joins combine related rows without copying every fact into every record.

```sql
SELECT o.order_id, o.status, u.email
FROM orders o
JOIN users u ON u.user_id = o.user_id
WHERE o.user_id = 42
ORDER BY o.created_at DESC
LIMIT 20;
```

Joins are powerful when:

- data has many relationships;
- the same entity is reused in several contexts;
- query patterns evolve;
- correctness benefits from one canonical fact.

Joins can become expensive when they scan large inputs, miss appropriate indexes, cross shards, or produce huge intermediate results. "SQL supports joins" does not mean every join is cheap.

### 5.3 Constraints

Common constraints include:

- primary key / unique;
- foreign key;
- `NOT NULL`;
- `CHECK`;
- data type and length/range restrictions.

Constraints reject invalid state regardless of which application path performs the write. Application validation is still useful for friendly errors, but it does not replace database enforcement when multiple writers or races exist.

### 5.4 Transactions and isolation

A relational database is a natural fit when a transaction changes several related rows:

```text
BEGIN
  create order
  insert line items
  reserve stock using a conditional update
  write payment-attempt reference
COMMIT
```

Isolation level matters. Two transactions can both read "stock = 1" and try to buy it. The design may use a conditional update, row lock, optimistic version, serializable isolation, or a reservation model. Merely saying "ACID" does not explain which anomaly is prevented.

### 5.5 Schema evolution

A database-enforced schema makes assumptions explicit, but changes require coordination:

1. add a nullable/backward-compatible column;
2. deploy code that can read old and new representations;
3. backfill old rows if required;
4. switch writers/readers;
5. enforce `NOT NULL` or remove the old field after verification.

Large blocking migrations are an operational risk. The safe pattern is often **expand, migrate, contract**.

### 5.6 Query flexibility

SQL supports filtering, joins, aggregation, sorting, subqueries, and window functions. This is valuable for changing product requirements and operational investigation. It is not a promise that one transactional primary should run unbounded analytical queries; read replicas, CDC pipelines, or a warehouse may isolate that workload.

### 5.7 Scaling relational databases

A typical progression is:

```text
1. one well-sized primary + good schema/indexes
2. cache frequently reused results
3. add replicas for stale-tolerant reads and failover
4. partition very large tables for maintenance/query pruning
5. archive/stream analytical work elsewhere
6. shard by a stable ownership key if one write domain is insufficient
7. consider distributed SQL when cross-node SQL/transactions justify its cost
```

PostgreSQL officially documents range/list/hash table partitioning and replication/high-availability options, which directly disproves "SQL can only scale vertically." [PostgreSQL partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html) and [PostgreSQL high availability and replication](https://www.postgresql.org/docs/current/high-availability.html)

The hard part of application sharding is not just routing rows. It includes:

- choosing a stable shard key;
- resharding and rebalancing;
- cross-shard joins and transactions;
- global uniqueness;
- secondary indexes across shards;
- hotspot tenants;
- failover and replica lag;
- schema migrations on every shard.

### 5.8 Strengths

- rich and evolving query patterns;
- explicit relationships and joins;
- database-enforced constraints;
- mature multi-row transaction semantics;
- established tools, migrations, backups, and expertise;
- normalization reduces accidental inconsistency.

### 5.9 Costs and cases to question

- a rigid schema can make uncoordinated shape changes harder;
- cross-shard relational operations are complex and may add latency;
- very high ingest with simple key-local queries may fit an LSM-based distributed store better;
- recursive or variable-depth graph traversal can be awkward or slow at high relationship density;
- one overloaded primary is not fixed merely by adding read replicas.

Do not avoid SQL merely because a table will be large. First ask whether it can be indexed, partitioned, archived, replicated, and served within the required scale.

## 6. NoSQL families in technical depth

### 6.1 Key-value databases

Examples: **Redis** and **Amazon DynamoDB**, with a major warning: they have different durability, persistence, query, consistency, and managed-service semantics. A data-model label does not make products substitutes.

#### Plain model

```text
key                                      value
---------------------------------------  -------------------------------
session:7f91                             { userId: 42, expiresAt: ... }
idempotency:createOrder:user42:req88     { orderId: 9001, status: done }
video:abc:like_count                     105234
```

The application supplies a key. The store returns or updates the associated value. Some products understand richer values such as hashes, sets, sorted sets, or JSON; Redis documents many native data types rather than only opaque strings. [Redis data types](https://redis.io/docs/latest/develop/data-types/)

#### Natural access patterns

- get one item by exact key;
- put/update/delete one item;
- increment a counter atomically if supported;
- fetch a key range or partition-key-plus-sort-key range if the product supports it;
- use TTL/expiration for sessions or temporary data.

#### Strengths

- simple routing and predictable key-based access;
- easy partitioning when keys distribute load evenly;
- often low request overhead;
- natural for sessions, idempotency records, feature flags, counters, caching, and precomputed views.

#### Trade-offs and traps

- asking by a value that is not indexed may require another index/table or a scan;
- one hot key can bottleneck one partition;
- cross-key transactions may be limited, more expensive, or product-specific;
- values can become application-owned blobs that are difficult to migrate or query;
- Redis used as an evicting cache is not equivalent to DynamoDB used as a durable source of truth.

#### Use when

The principal request can be phrased as "given this known key, get or update its value," and the correctness/durability features of the chosen product match the requirement.

#### Avoid as the only store when

The core workload needs arbitrary filters, changing joins, deep traversals, or cross-key invariants that the product cannot efficiently enforce.

### 6.2 Document databases

Example: **MongoDB**.

#### Plain model

A document stores one nested object:

```json
{
  "_id": "product-7",
  "name": "Running shoe",
  "category": "footwear",
  "attributes": {
    "sizes": [7, 8, 9, 10],
    "material": "mesh",
    "waterproof": false
  },
  "sellerOffers": [
    { "sellerId": 81, "price": 3499 },
    { "sellerId": 96, "price": 3599 }
  ]
}
```

Another product category can have different attributes without adding dozens of mostly empty relational columns.

#### Embedding versus referencing

**Embed** child data when it:

- belongs to one parent;
- is usually read with that parent;
- stays reasonably bounded in size;
- benefits from one-document atomic updates;
- is not independently shared and updated everywhere.

**Reference** another document when it:

- has its own lifecycle;
- is shared by many parents;
- grows without a safe bound;
- is commonly queried independently;
- would cause frequent large rewrites if embedded.

```text
Good embedding candidate:  order -> purchased line-item snapshots
Likely reference:           order -> customer account's current canonical profile
Dangerous embedding:        user -> every event user has ever generated
```

#### Strengths

- object-like nested data maps naturally to application responses;
- flexible shape helps heterogeneous entities and gradual evolution;
- one aggregate can often be read without a join;
- locality and single-document atomicity can simplify an aggregate.

#### Trade-offs and traps

- duplicated fields need an update/repair policy;
- unbounded arrays/documents become hotspots and grow expensive;
- joins/lookup features exist but should not be assumed equivalent in cost or semantics to a relational design;
- flexible schema can silently accumulate incompatible versions;
- indexes on many optional/nested fields consume storage and write work;
- multi-document transactions exist, but using them everywhere may discard the model's locality advantage.

MongoDB documents flexible schema plus optional validation, and it supports transactions across collections on replica sets/sharded clusters with stated performance considerations. That is direct evidence against "document means schema-free and non-transactional." [MongoDB schema validation](https://www.mongodb.com/docs/manual/core/schema-validation/) and [MongoDB transactions for consistency](https://www.mongodb.com/docs/manual/data-modeling/enforce-consistency/transactions/)

#### Use when

The main unit is a self-contained aggregate/document, its nested shape varies, and most writes/reads stay within that boundary.

#### Avoid or reconsider when

The domain is dominated by many-to-many relationships, frequent arbitrary joins, strict cross-document invariants, or unbounded embedded collections.

### 6.3 Wide-column / column-family databases

Example: **Apache Cassandra**.

#### Plain model

The data model is designed around a **partition key** and ordered rows inside each partition. It can look table-like through CQL, but it is not a relational table with arbitrary join behavior.

```sql
CREATE TABLE messages_by_conversation_bucket (
    conversation_id TEXT,
    month_bucket    TEXT,
    sent_at         TIMESTAMP,
    message_id      UUID,
    sender_id       TEXT,
    body            TEXT,
    PRIMARY KEY ((conversation_id, month_bucket), sent_at, message_id)
) WITH CLUSTERING ORDER BY (sent_at DESC);
```

Mental layout:

```text
Partition key: (conversation C42, month 2026-07)

sent_at desc        message_id       sender       body
---------------------------------------------------------
12:01:05            m103             user 9       "yes"
12:00:50            m102             user 42      "ready?"
11:58:10            m101             user 9       "hello"
```

One efficient query is known in advance:

```text
Get newest messages where
conversation_id = C42 and month_bucket = 2026-07,
starting before timestamp T.
```

Cassandra says data modeling should consider query patterns and that rows sharing a partition key are localized on the same replicas. It also warns that all rows under one key can create a hotspot. [Cassandra CQL data definition](https://cassandra.apache.org/doc/latest/cassandra/developing/cql/ddl.html)

#### Why the bucket is present

If the partition key were only `conversation_id`, a very old or extremely active conversation could grow without bound and concentrate traffic. Adding a time bucket limits partition size, but the application may need to query several buckets when paging far back. This is a classic locality-versus-distribution trade-off.

#### Strengths

- horizontal partitioning and replication are first-class;
- high sustained write throughput for key-oriented workloads;
- rows inside a partition can be ordered by clustering columns;
- multi-datacenter, availability-oriented deployment patterns;
- TTL/time-series/event-history patterns can fit well.

#### Trade-offs and traps

- schema is designed per query; new access patterns often require a new table/index/pipeline;
- joins and arbitrary cross-partition aggregation are not the natural path;
- denormalized copies create write amplification;
- bad partition keys cause hotspots or oversized partitions;
- tombstones, compaction, repair, and consistency settings are operational concerns;
- last-write-wins and clock behavior require care for conflicting writes.

Cassandra documents a partitioned wide-column model with eventually consistent semantics, tunable consistency, and CQL features; it also documents its commit log, memtables, immutable SSTables, and compaction. [Cassandra architecture overview](https://cassandra.apache.org/doc/stable/cassandra/architecture/overview.html) and [Cassandra storage engine](https://cassandra.apache.org/doc/stable/cassandra/architecture/storage-engine.html)

#### Use when

The workload is very large, write-heavy or geographically distributed, queries are stable and partition-key-oriented, and the team can operate its consistency/repair model.

#### Avoid or reconsider when

The product needs ad-hoc relational queries, strong cross-partition constraints, small-scale simplicity, or constantly changing access patterns.

### 6.4 Graph databases

Example: **Neo4j**.

#### Plain model

A property graph stores **nodes**, **relationships/edges**, labels/types, and properties:

```text
(Asha:User {id:42})
    |
    +--[:FOLLOWS {since:"2025-01"}]--> (Ben:User {id:77})
                                               |
                                               +--[:FOLLOWS]--> (Chen:User {id:91})

(Card:PaymentInstrument)-[:USED_ON]->(Order)-[:SHIPPED_TO]->(Address)
```

The relationship is a first-class stored object, so repeated variable-depth traversal is natural:

- friends of friends;
- shortest paths;
- fraud rings connecting accounts/devices/cards/addresses;
- dependency and knowledge graphs;
- recommendation candidates through shared relationships.

#### Why not just a relational join table?

A relational database can model edges:

```text
follows(follower_id, followee_id)
```

For direct followers or one/two known hops, an indexed relational table may be entirely sufficient. A graph database becomes attractive when traversal depth varies, relationships carry rich properties, and path exploration is central and frequent.

#### Strengths

- relationship traversal matches the model;
- paths and neighborhoods are expressive;
- avoids manually composing many self-joins for variable-depth traversals;
- relationships can have types and properties.

#### Trade-offs and traps

- not every social feature needs a graph database;
- graph partitioning is difficult because connected traversals may cross machines;
- supernodes/very high-degree nodes can become hotspots;
- simple key lookups, append-only histories, and bulk analytics may fit other systems better;
- a second graph copy introduces synchronization and deletion complexity if another database remains authoritative.

Neo4j documents a property-graph approach and ACID transactions. Its cluster documentation describes leader/writer behavior and read scaling, showing again that graph data model, transactions, and distribution must be evaluated separately. [Neo4j introduction](https://neo4j.com/docs/operations-manual/current/introduction/), [Neo4j transactional behavior](https://neo4j.com/docs/operations-manual/current/database-internals/), and [Neo4j clustering](https://neo4j.com/docs/operations-manual/current/clustering/introduction/)

#### Use when

Multi-hop relationship traversal is a core online workload and materially simpler/faster than maintaining equivalent application-side joins or precomputed views.

#### Avoid or reconsider when

Queries are mostly exact-key, one-hop, tabular reporting, or append-by-time; or when the team cannot keep the graph synchronized with the system of record.

### 6.5 Wide-column is not analytical columnar storage

This is a frequent terminology trap.

| Term | Layout intuition | Optimized workload | Examples |
|---|---|---|---|
| **Wide-column / column-family NoSQL** | Partitioned rows, sparse/flexible columns, clustering inside a key-owned partition | Key-oriented online reads/writes at distributed scale | Cassandra, HBase |
| **Columnar analytical database/warehouse** | Values of the same column stored/encoded together | Scan a few columns across many rows; aggregate/compress | BigQuery, ClickHouse, Redshift |

```text
Row-oriented page:
[order1: id,user,total,status][order2: id,user,total,status]

Analytical columnar chunks:
[all ids...][all users...][all totals...][all statuses...]
```

If an interviewer asks for "daily average watch time over a year," an analytical columnar system is likely relevant. If the request is "fetch the newest messages in conversation C42," a wide-column operational model may fit. They share the word *column* but solve different problems.

## 7. Corrected comparison: structure, nature, scalability, properties

This expands the four original comparison bullets without turning tendencies into false absolutes.

| Dimension | Relational / SQL tendency | NoSQL tendency | Correction / interview nuance |
|---|---|---|---|
| **Structure** | Tables, rows, typed columns, primary/foreign keys, joins | Key-value entries, documents, wide partitions, or graph nodes/edges | NoSQL has several structures; SQL databases can also store JSON |
| **Nature of data** | Strong fit for related entities and shared canonical facts | Strong fit for aggregate, key-local, sparse, high-volume, or traversal-specific data depending on family | "Structured/unstructured" is too weak; videos belong in object storage |
| **Schema** | Usually database-enforced and explicitly migrated | Often flexible, validation/application enforced, and versioned per record | Flexible schema still requires compatibility and validation |
| **Query model** | Rich joins, filters, aggregation, evolving/ad-hoc queries | Usually optimized around known keys, document fields, partition ranges, or graph paths | Product-specific secondary indexes and query features vary |
| **Normalization** | Normalization is common; denormalization remains possible | Denormalization/query-specific copies are common | Duplication moves join work from read time to write/repair time |
| **Transactions** | Mature multi-row/table transactions are a common strength | Atomicity/transactions vary by product and scope; several support multi-item transactions | Verify scope, isolation, cost, partition constraints, and failure semantics |
| **Constraints** | Unique, foreign key, check, not-null commonly enforced | Often fewer cross-record constraints; conditional writes/validation may exist | Application checks alone can race |
| **Consistency** | Primary reads can be strong; replicas may be stale | Can be eventual, tunable, causal, or strong depending on product/configuration | Consistency is independent of data model |
| **Vertical scale** | Common and often sufficient for a long time | Also possible | Scaling up is not inherently bad; it is operationally simple until limits/cost dominate |
| **Horizontal scale** | Replication and partitioning exist; general cross-shard semantics can be hard | Many products are designed around key-based distribution and limited cross-partition work | NoSQL's restrictions are often what make distribution tractable |
| **Availability** | Depends on replication, failover, quorum, and operation | Same | SQL/NoSQL label does not give an availability SLO |
| **Storage engine** | Often page/B-tree oriented, but not universally | Often LSM for write-heavy stores, in-memory for some key-value uses, but not universally | Compare the actual engine and workload |
| **Operational maturity** | Broad tooling and expertise are common | Varies greatly by product and team | Existing safe operations can outweigh a theoretical benchmark win |
| **Best default question** | "Can relational storage safely meet this?" | "What concrete requirement justifies this specific NoSQL family?" | State evidence and an evolution path, not ideology |

## 8. "Nature of data" explained correctly

### 8.1 Structured, semi-structured, and unstructured

- **Structured:** fields and types are regular, such as orders and ledger entries.
- **Semi-structured:** records have recognizable fields but may vary, such as product metadata or events in JSON.
- **Unstructured:** the database does not usefully interpret the internal content, such as a video or PDF byte stream.

The usual storage split is:

```text
Object storage: video bytes at object key /videos/abc/1080p
Database:       video_id, owner_id, title, status, object_key, duration
Search index:   title/transcript tokens for full-text retrieval
Warehouse:      aggregated watch events
```

Putting a large video inside a document does not make MongoDB the "unstructured-data database." The binary object's size, delivery, range reads, replication cost, and CDN integration make object storage the natural component.

### 8.2 Relationship shape is often more important

Ask:

- Are facts shared by many records?
- Are relationships many-to-many?
- Do queries traverse a known one hop or an unknown number of hops?
- Does one aggregate have a clear bounded ownership boundary?
- Are records fetched by exact key, partition range, arbitrary predicate, or graph path?

These answers distinguish relational, document, wide-column, key-value, and graph models much better than "structured versus unstructured."

### 8.3 Update shape matters too

```text
Read together + update together          -> embedding may fit
Read together + update independently     -> reference or duplicate with a policy
Shared canonical fact                    -> normalize/source of truth
Append immutable history by owner/time   -> wide-column/event/log pattern may fit
Traverse relationships repeatedly        -> graph or precomputed adjacency pattern
```

## 9. Properties: ACID, BASE, constraints, and consistency

### 9.1 ACID is not a database-selection slogan

"Need ACID, therefore SQL" can be a useful first hint, but it is incomplete.

Ask instead:

1. Which invariant must never break?
2. Which records participate?
3. Are they on one item, one partition, one shard, or several services?
4. Which isolation anomaly must be prevented?
5. What durability acknowledgement is required?
6. What happens on timeout - did the transaction commit or not?

For example, DynamoDB documents ACID transaction APIs across items/tables, MongoDB documents multi-document transactions under supported deployments, and Neo4j documents ACID graph transactions. [DynamoDB transactions](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/transactions.html)

The existence of a transaction feature does not mean it is free or unlimited. Check item/partition scope, contention, latency, throughput consumption, retry behavior, and isolation semantics in the selected product.

### 9.2 BASE is a design tendency, not the definition of NoSQL

BASE is commonly expanded as:

- **Basically Available**;
- **Soft state**;
- **Eventually consistent**.

It describes an availability/convergence-oriented family of choices. It is not a precise universal contract shared by all NoSQL products, and it does not mean "randomly wrong data is acceptable."

An eventually consistent design still needs explicit rules:

- which fields may be stale;
- acceptable staleness or convergence objectives, if bounded;
- conflict detection and resolution;
- read-your-writes needs;
- repair and anti-entropy;
- behavior during partition;
- monitoring for non-convergence.

### 9.3 ACID consistency is not CAP consistency

```text
ACID C: transaction preserves encoded application/database invariants
CAP  C: operations behave like one linearizable copy during distributed faults
```

A transaction can preserve `balance >= 0` on one primary while a remote asynchronous replica temporarily returns the old balance. Different questions are being answered.

### 9.4 Strong and eventual behavior can coexist

One product or system may offer:

- strong reads for checkout;
- eventually consistent replicas for browsing history;
- a conditional write for idempotency;
- an asynchronous search index;
- a warehouse updated minutes later.

Specify consistency per operation and user promise rather than applying one adjective to the whole architecture.

## 10. Scalability in technical depth

### 10.1 Four different scaling problems

"Scale" can mean:

1. **data capacity** - bytes no longer fit comfortably;
2. **read throughput** - too many or too expensive reads;
3. **write throughput** - one write leader/disk/partition is saturated;
4. **geographic latency/availability** - users and failure domains span regions.

Each has different remedies. Read replicas do not increase one leader's write capacity. Sharding does not automatically reduce a single hot key. A cache does not make authoritative writes durable.

### 10.2 Vertical scaling

Use a larger machine, more memory, faster disks, or more CPU.

Advantages:

- simplest transaction and operational model;
- no cross-shard operations;
- often the cheapest engineering solution at modest scale.

Limits:

- hardware and cost ceilings;
- larger failure blast radius;
- maintenance/failover needs remain;
- write throughput may still bottleneck on serialization or storage.

Vertical scaling is not an interview failure. A good design adds complexity in response to requirements.

### 10.3 Replication

```text
                 async replication
Writes --> Leader --------------------> Replica 1 --> stale-tolerant reads
                   \------------------> Replica 2 --> failover/read traffic
```

Replication can improve availability, durability, read capacity, and locality. It introduces lag, failover, consistency, and split-brain questions. Synchronous quorum/consensus reduces some risk but adds latency and can reduce availability when insufficient replicas communicate.

### 10.4 Horizontal partitioning / sharding

```text
route(user_id)
  0..999       -> shard A
  1000..1999   -> shard B
  2000..2999   -> shard C
```

Good shard keys provide:

- even data distribution;
- even request distribution;
- locality for common operations;
- a stable ownership boundary;
- manageable resharding.

Bad cases:

- `country` where most traffic is from one country;
- `celebrity_id` where one celebrity receives a disproportionate load;
- monotonically increasing key with range partitioning that sends all new writes to one shard;
- tenant ID when one tenant is much larger than every other tenant;
- conversation ID without time bucketing when a partition can grow forever.

### 10.5 Why many NoSQL systems scale out naturally

Many distributed NoSQL APIs deliberately constrain operations to a partition key. That gives the router a clear answer:

```text
request contains key -> hash/range -> owning replica set -> local operation
```

Avoiding arbitrary cross-partition joins and constraints reduces coordination. The trade-off is moved to data modeling: the application must know access patterns, maintain duplicated views, and handle cross-partition workflows.

### 10.6 Why relational sharding is harder, not impossible

A general relational query may combine data owned by several shards:

```text
SELECT ... JOIN ... GROUP BY ...
          |
          +--> shard A
          +--> shard B
          +--> shard C
```

The system must scatter, gather, combine, order, and perhaps transact across nodes. Distributed SQL databases coordinate this automatically to varying degrees, but coordination still has physical latency and availability costs.

If all rows for one customer live on one shard, many transactions remain local. This is why a strong ownership key is valuable in both SQL and NoSQL.

### 10.7 Storage engines: B-tree versus LSM intuition

This is a likely advanced follow-up, not a SQL/NoSQL identity rule.

**B-tree/page-oriented intuition:**

- update indexed pages in a structure suited to point and range lookups;
- mature general-purpose indexing;
- random-write/page-management and write-ahead-log costs.

**LSM-tree intuition:**

- append writes to a log and memory table;
- flush sorted immutable files;
- compact files in the background;
- high write throughput, but reads may consult multiple structures and compaction creates write amplification.

Cassandra is a documented LSM-style example. Some relational and non-relational products use different or hybrid engines. Compare the actual database and workload.

### 10.8 Scale checklist for either family

- What is the partition/shard key?
- Is load uniform, or can one key become hot?
- Are reads point, range, scan, join, or traversal?
- Are writes inserts, updates, counters, or append-only?
- How many indexes does every write update?
- Does a transaction cross partitions?
- Which replicas acknowledge before success?
- Can reads be stale?
- How is data rebalanced?
- What happens when a node, zone, or link fails?
- How are backups restored and verified?

## 11. One e-commerce domain modeled four ways

This is not four competing complete designs. It shows where each model is natural.

### 11.1 Relational source of truth for orders

```text
users 1---N orders 1---N order_items N---1 products
                  |
                  +---1 payment_attempts
```

Natural requirements:

- create an order and line items atomically;
- enforce unique order ID and idempotency key;
- query all orders for a user;
- reconcile payments and orders;
- support operations/reporting queries that evolve.

Good fit: relational database, potentially partitioned/sharded later by a stable customer/order ownership rule.

### 11.2 Document product catalog

```json
{
  "productId": "shoe-7",
  "category": "running-shoes",
  "title": "Trail Runner",
  "attributes": {
    "terrain": "trail",
    "dropMm": 8,
    "waterproof": true
  },
  "mediaObjectKeys": ["catalog/shoe-7/front.webp"]
}
```

Natural requirement: different categories have different nested attributes, and the product page reads one aggregate. Good document fit, although a relational JSONB model might also meet the scale and operational requirements.

### 11.3 Key-value cart/session

```text
cart:user42 -> {
  version: 8,
  items: [{productId:7, qty:2}],
  expiresAt: ...
}
```

Natural requirement: fetch/update by user ID with TTL and optimistic version/conditional update. Good key-value fit. Final price and inventory must still be verified against authoritative services at checkout.

### 11.4 Graph fraud view

```text
(Account)-[:USED]->(Card)-[:USED_ON]->(Order)-[:SHIPPED_TO]->(Address)
     \                                              /
      +--------------[:LOGGED_IN_FROM]->(Device)---+
```

Natural requirement: find suspicious rings and paths connecting accounts, devices, cards, and addresses. Good graph workload. The graph may be a derived view populated from authoritative transaction events; freshness and deletion handling must be defined.

### 11.5 One request crosses systems carefully

```text
Client
  |
  v
Checkout service
  |-- read cart ----------------------> key-value store
  |-- validate prices/inventory -----> authoritative services/DBs
  |-- transaction: create order -----> relational order DB
  |-- publish OrderCreated via outbox -> event stream
                                        |--> search/analytics
                                        |--> fraud graph update
                                        +--> notification
```

Do not attempt one distributed transaction across every specialized store unless the requirements truly demand it and the infrastructure supports it. A transactional outbox plus idempotent consumers commonly provides reliable asynchronous propagation; derived stores can be rebuilt from authoritative events.

## 12. Selection framework: how to choose in an interview

Use the mnemonic **Q-T-C-S-O-E**:

```text
Q  Queries and access patterns
T  Transaction and invariant boundary
C  Consistency, availability, durability
S  Scale, partition key, hotspots
O  Object/relationship shape
E  Ecosystem, operations, and evolution
```

### Step 1: Clarify the workload

Ask or state assumptions:

- What are the top reads and writes?
- Is this a source of truth, cache, search view, or analytics copy?
- Are queries known and stable or ad hoc and evolving?
- Point lookup, range by time, join, full-text search, aggregation, or graph traversal?

### Step 2: State the correctness rules

- Which invariant must never break?
- What must commit atomically?
- Is one item enough, or several rows/documents/partitions?
- Can a read be stale? Which user flow cannot tolerate it?
- What happens on duplicate retries and timeouts?

### Step 3: Model relationships and ownership

- Are entities heavily related and shared?
- Is there one bounded aggregate read/written together?
- Is variable-depth traversal the core query?
- Is data naturally owned by user, tenant, conversation, or time bucket?

### Step 4: Estimate the relevant scale

You do not need fake precision. Determine the order of magnitude and bottleneck:

- records and bytes;
- peak reads/writes per second;
- record size and growth/retention;
- read-to-write ratio;
- geographic distribution;
- skew: celebrities, large tenants, viral content.

### Step 5: Propose the simplest fitting baseline

Examples:

- relational for connected transactional business data;
- key-value for exact-key sessions/idempotency/precomputed views;
- document for bounded heterogeneous aggregates;
- wide-column for massive stable partition-range workloads;
- graph for core multi-hop traversal;
- object store for media;
- search engine for full-text retrieval;
- warehouse for analytical scans.

### Step 6: Defend distribution and failure behavior

Say:

- partition key and hotspot mitigation;
- replication and read path;
- consistency per operation;
- transaction scope;
- failover/timeout/retry behavior;
- backup and recovery.

### Step 7: Name the trade-off and evolution path

Example:

> "I will use PostgreSQL initially because orders have cross-row invariants and evolving operational queries. The trade-off is that one write primary has a ceiling. I can first index and partition tables, offload stale-tolerant reads, and shard by customer region/ID only when measured load requires it. Product search and analytics will be asynchronous derived views rather than expensive queries on the order primary."

That is much stronger than "SQL because ACID."

## 13. Decision matrix: use, avoid, and alternatives

| Workload | Likely starting point | Why | Main caution / alternative |
|---|---|---|---|
| Bank ledger / money movement | Relational or suitable distributed SQL | constraints, transactions, audit queries | still design idempotency, isolation, and append-only ledger semantics |
| Order management | Relational | related entities, state transitions, reconciliation | shard only with a clear ownership key; document snapshot can complement |
| Login session / token lookup | Key-value | exact key, TTL, low overhead | decide whether loss/eviction is acceptable; use a durable product if authoritative |
| Idempotency key | Key-value or relational unique row | conditional create / uniqueness | TTL must exceed retry window; store final result/status |
| Heterogeneous product catalog | Document or relational + JSONB | category-specific nested fields | search engine for text/facets; canonical stock/price may remain relational |
| Chat message history by conversation/time | Wide-column or sharded relational | append and ordered range by owner/time | bucket partitions; handle ordering, deduplication, edits, deletion |
| Social follower direct lookup | Relational edge table or wide-column adjacency lists | one-hop keyed access is simple | graph DB is not automatically required |
| Fraud-ring / multi-hop relationship exploration | Graph | paths and variable-depth traversal | synchronization, supernodes, graph partitioning |
| Full-text product search | Search engine | inverted indexes/relevance | derived, usually not the authoritative store |
| Video/image bytes | Object storage + CDN | large immutable objects/range delivery | metadata stays in a database |
| Clickstream analytics | Event log + analytical columnar warehouse | append ingestion and large scans | not Cassandra merely because both mention columns |
| Small CRUD application with uncertain future queries | Relational | simplicity, joins, constraints, ecosystem | avoid premature polyglot/sharding complexity |

### 13.1 When relational is usually a strong fit

- several related entities change together;
- correctness uses unique/foreign-key/check constraints;
- query patterns evolve;
- joins and aggregations are common;
- the workload fits one primary or can be partitioned with manageable complexity;
- operational familiarity favors it.

### 13.2 When to question relational as the only model

- massive, stable, partition-key-oriented writes exceed a practical leader/shard design;
- nested aggregates vary heavily and are normally retrieved as one document;
- multi-hop path traversal is the main online operation;
- exact-key low-latency access needs a dedicated cache/key-value view;
- search or analytics is overloading the transactional store.

### 13.3 When a specialized NoSQL store is justified

State all three:

1. the **specific family/product capability**;
2. the **specific access pattern or scale constraint** it fits;
3. the **trade-off accepted**.

Example:

> "I choose Cassandra for `messages_by_conversation_bucket` because writes append at high volume and reads always supply conversation plus time bucket. This gives partition-local ordered reads and scale-out writes. I accept query-specific denormalization, no relational joins in the request path, and the need to manage hotspots, compaction, repair, and consistency settings."

### 13.4 When to avoid premature polyglot persistence

Every additional database adds:

- another failure mode and on-call skill;
- CDC/event propagation;
- inconsistent derived views;
- backup, restore, access control, and observability work;
- versioned contracts and backfills;
- cost and vendor dependency.

Use several stores when workload differences justify them, not to display a longer architecture diagram.

## 14. Real-system examples

All examples in this section are **illustrative designs**, not claims about the companies' current private backends. They show how to reason about familiar product requirements.

### 14.1 WhatsApp-like messaging

Requirements split naturally:

```text
                         +--> relational: account, device, policy metadata
Client --> Chat service -+--> wide-column: messages by conversation + time bucket
                         +--> key-value: presence/session/recent delivery state
                         +--> object store: encrypted image/video attachment bytes
                         +--> search index: optional client/server-permitted search view
```

Reasoning:

- Account/device state has constraints and evolving relationships: relational is a reasonable baseline.
- Message history is append-oriented and fetched by conversation/time: wide-column or sharded relational can fit.
- Presence is ephemeral, exact-key, high-churn state: key-value with TTL can fit.
- Attachments are large objects: object storage, not a "NoSQL blob document."

Follow-ups:

- Partition key: `(conversation_id, time_bucket)` prevents unbounded partitions.
- Ordering: server sequence per conversation or stable sort tuple; global order is unnecessary.
- Deduplication: unique `client_message_id` per sender/conversation.
- Group chat hotspot: bucket/subpartition and cache fan-out metadata; one huge group can still be hot.
- Consistency: message sender needs read-your-writes; presence can be stale.

### 14.2 YouTube-like video platform

```text
Upload
  |--> object storage: original and transcoded video chunks
  |--> relational/document metadata: owner, title, visibility, processing state
  |--> event log: view/watch events
  |      \--> columnar warehouse: aggregate watch analytics
  |--> search index: titles, tags, transcript
  \--> key-value/cache/CDN metadata: hot playback manifests
```

Database choice follows access pattern:

- Rights/ownership and publishing state need correctness.
- Product metadata can use relational columns plus JSONB or a document model depending on variability and joins.
- Video bytes go to object storage/CDN.
- Search and recommendations are derived workloads, not reasons to make one primary database do everything.

### 14.3 Uber-like ride system

```text
Driver app --location updates--> geo/location service --> fast geo index / ephemeral store
Rider app ----request ride-----> trip service ---------> relational trip/payment DB
                                          |
                                          +--> event stream --> historical location store
                                                              --> analytics/ML
```

Reasoning:

- Trip lifecycle, fare, payment, and idempotent booking have important invariants: relational is a strong fit.
- Current driver locations are high-churn and queried geospatially: a specialized in-memory/geospatial index may fit.
- Historical pings are append-by-driver/time and large: wide-column/time-series/analytical pipeline can fit.
- A road network is graph-shaped, but route computation may use a specialized routing engine rather than the trip database.

Do not choose one database for both payment correctness and every GPS ping merely because both belong to "Uber."

### 14.4 Netflix-like streaming service

| Data | Illustrative store | Why |
|---|---|---|
| Account, plan, billing entitlement | Relational | constraints and transactional state |
| Title metadata | Document or relational + JSON | heterogeneous nested metadata versus shared relationships |
| Playback session/resume position | Key-value | exact account/profile/title key, frequent update |
| Video segments | Object storage + CDN | large immutable media delivery |
| Viewing events | Log -> wide-column/warehouse | high-volume append and later analytics |
| Recommendations | Derived feature/vector/graph systems | specialized retrieval, not authoritative billing state |

### 14.5 Twitter/X-like social feed

```text
Write tweet --> tweet source of truth
                  |
                  +--> fan-out workers --> timeline_by_user (key/wide-column view)
                  +--> search index
                  +--> analytics

Follow graph --> relational edge table, adjacency store, or graph DB
                chosen from actual one-hop vs multi-hop access patterns
```

Reasoning:

- Tweet creation needs idempotency and durable identity.
- Home timeline is a precomputed/denormalized query view, often key-oriented.
- Celebrity fan-out creates hotspots and write amplification; a hybrid fan-out-on-write/fan-out-on-read design can treat celebrities differently.
- A graph DB is justified only if online multi-hop traversal is core. Direct `followers(user_id)` can be served by a relational or adjacency-list store.

### 14.6 Amazon-like marketplace

```text
Browse path:   catalog document/relational DB --> search index --> cache
Checkout path: cart KV --> pricing/inventory --> relational order/payment transaction
After commit:  outbox/event --> fulfillment, email, fraud graph, analytics
Media path:    object storage --> CDN
```

Key separation:

- cart contents are not authoritative prices or stock;
- order history stores price-at-purchase snapshots intentionally;
- inventory requires a conditional/transactional reservation strategy;
- search results may be stale, but checkout revalidates authoritative facts;
- fraud graph can lag if the policy accepts it, while a synchronous risk decision may query a current feature service.

## 15. Edge cases and failure modes interviewers probe

### 15.1 Hot partition / celebrity problem

One logical key receives far more traffic than others.

Mitigations depend on operation:

- cache hot immutable/read-mostly values;
- request coalescing and CDN for public content;
- split/bucket a write-heavy logical key;
- add a random suffix for counters, then aggregate;
- isolate very large tenants;
- use hybrid fan-out for celebrity timelines;
- rate-limit abusive traffic.

Random sharding harms range queries and aggregation, so state that cost.

### 15.2 Retry after timeout

The client cannot tell whether a timed-out write committed.

```text
Client --> create order --> DB commits
Client <-- response lost X
Client --> retries create order
```

Use an idempotency key with a unique/conditional insert and retain the operation result. "The database is ACID" alone does not prevent two separate successful transactions from creating duplicate orders.

### 15.3 Cross-shard transaction

Transferring data/money across shard A and shard B needs coordination, a redesign that keeps the invariant local, or a durable workflow/saga with compensations. Do not silently assume local transaction semantics extend across arbitrary partitions.

### 15.4 Replica lag

A user updates a profile on the leader and immediately reads from a lagging replica.

Options:

- read the leader for that flow;
- sticky/session routing;
- causal token/bookmark;
- wait until the replica reaches the write position;
- show pending local state;
- accept staleness only where product semantics allow it.

### 15.5 Flexible-schema drift

Old documents store `price` as text, new ones as integer, and one writer omits currency.

Mitigations:

- document version field;
- database validation where supported;
- backward/forward-compatible readers;
- dual-read/dual-write only during controlled migration;
- backfill with checkpoints and observability;
- reject unknown invalid states rather than silently guessing.

### 15.6 Unbounded embedding / wide partition

Embedding every comment in one post or every message under one unbucketed conversation causes growth, contention, and large reads. Split child records and page them; bucket time-series/wide-column partitions.

### 15.7 Denormalized update failure

One source write succeeds but updating a derived timeline/search/graph copy fails.

Use:

- one authoritative source;
- transactional outbox or CDC;
- idempotent consumers;
- retry/dead-letter policy;
- freshness/error metrics;
- reconciliation/backfill jobs;
- ability to rebuild derived views.

Dual-writing independently from application code creates an inconsistent failure window unless coordinated.

### 15.8 Deletes, privacy, and retention

One user record may have copies in caches, search indexes, analytics stores, graph views, backups, and denormalized tables. Define:

- authoritative deletion event;
- propagation and legal retention rules;
- tombstone versus physical deletion;
- cache invalidation;
- proof/monitoring of completion;
- backup expiry and restore-time re-deletion procedure.

Polyglot persistence multiplies this work.

### 15.9 Secondary-index explosion

Adding an index for every possible document field or SQL predicate slows writes and consumes storage. Identify high-value queries, inspect query plans/metrics, and isolate search/analytics when the access pattern differs fundamentally.

### 15.10 Database or region outage

State:

- replica placement across failure domains;
- automated/manual failover;
- tolerated data loss (`RPO`) and recovery time (`RTO`);
- whether failover preserves consistency;
- DNS/client connection behavior;
- backup restore testing.

Replication is not backup: deletion or corruption can replicate too.

## 16. Common mistakes and corrected answers

| Mistake | Why it fails | Better statement |
|---|---|---|
| "SQL is structured; NoSQL is unstructured" | Both store structured representations; relational can hold JSON, document DBs validate schema | Compare relationships, aggregate boundaries, access patterns, and enforcement |
| "NoSQL has no schema" | Readers and writers always assume some shape | Schema is flexible and may be enforced in DB, app, or consumers; version it |
| "NoSQL scales horizontally; SQL vertically" | Relational replication/partitioning/sharding exist; NoSQL can still hotspot | Distribution is often more native in NoSQL, while general cross-shard SQL is harder |
| "SQL is ACID; NoSQL is BASE" | NoSQL products can offer ACID transactions; SQL replicas can be stale | Evaluate transaction and consistency guarantees by product, scope, operation, configuration |
| "Use NoSQL for write-heavy systems" | Storage engine, indexes, replication, durability, and contention determine write cost | Describe the write path and why a specific engine/model fits |
| "Use SQL for read-heavy systems" | Both families can cache/index/replicate; query shape matters | Point lookup, range, join, scan, search, or traversal is the useful distinction |
| "Use MongoDB because fields may be added" | Relational schemas and JSON columns can evolve too | Explain heterogeneous bounded aggregates and read/write locality |
| "Use graph DB for any social network" | Direct follower lookup is a simple adjacency query | Use graph when repeated variable-depth traversal is central |
| "Cassandra is columnar analytics" | Wide-column OLTP and columnar OLAP are different | State partition-key range access versus column scans/aggregation |
| "Store images in NoSQL because they are unstructured" | Large object delivery has different economics and APIs | Store bytes in object storage; store metadata/object key in DB |
| "One database for consistency" | One overloaded database doing OLTP, search, and analytics may still be unsafe | Keep one source of truth; build explicit derived views where justified |
| "Microservices require a database per endpoint" | Creates needless distributed transactions and operations | Choose ownership boundaries; one service may own several tables in one DB |
| "ACID prevents duplicate requests" | Two independent transactions can both be valid | Add idempotency key and unique/conditional write |
| "Eventual consistency means five seconds" | Eventual alone has no fixed bound | State a measured/contracted staleness bound if one exists |

## 18. Answer spines: cues to reconstruct answers live

An answer spine is a sequence of cues, not a memorized paragraph.

### 18.1 Direct question: "SQL versus NoSQL?"

Use **definition -> dimensions -> decision -> nuance**:

```text
Definition
  SQL commonly means relational tables/keys/joins.
  NoSQL is an umbrella: KV, document, wide-column, graph.

Dimensions
  access pattern -> relationships/shape -> transaction boundary
  -> consistency -> partition/scaling -> operations/evolution

Decision
  SQL: relational invariants + evolving joins/queries.
  Specific NoSQL family: concrete model/scale fit.

Nuance
  neither owns ACID, consistency, JSON, or horizontal scaling.
  choose product/configuration and state the trade-off.
```

### 18.2 Database choice inside a system design

Use **source -> query -> invariant -> key -> failure -> evolution**:

1. **Source:** authoritative store or derived view?
2. **Query:** top read/write patterns and latency.
3. **Invariant:** what changes atomically; acceptable staleness.
4. **Key:** partition/shard key, locality, skew/hotspots.
5. **Failure:** retry/idempotency, replicas, lag, failover, repair.
6. **Evolution:** new query, resharding, migration, rebuild derived view.

### 18.3 "Why not PostgreSQL?"

Do not attack PostgreSQL generically. Say:

```text
PostgreSQL is my baseline and can handle JSON, indexes, replicas, and partitions.
For this specific workload, however:
  - all reads are by (conversation, time bucket),
  - writes are sustained append at a scale beyond the proposed leader/shard setup,
  - cross-record joins/constraints are not required on the message path,
  - availability requirements favor partition-local replicated writes.
Therefore a wide-column store is worth its query rigidity and operational cost.
```

If those facts are absent, staying with PostgreSQL may be the better answer.

### 18.4 "Why not MongoDB now that it has transactions?"

```text
Feature existence != natural workload fit.
Check:
  aggregate boundary; relationship density; query evolution;
  transaction scope/isolation/cost; duplication; indexes; team operations.
If frequent cross-document transactions and joins dominate,
the document boundary is fighting the domain and relational may be simpler.
```

### 18.5 "Would you ever use both?"

```text
Yes, only with explicit ownership:
  relational DB = authoritative order
  search/graph/KV = derived query view
Propagation = outbox/CDC + idempotent consumers
Correctness = critical writes validate against source
Operations = freshness metrics, retry, reconciliation, rebuild, deletion
```

### 18.6 Follow-up ladder and response cues

| Interviewer follow-up | Response cues |
|---|---|
| How does it scale? | bottleneck type -> cache/replica/partition -> key -> skew -> rebalancing |
| What if a node fails? | replica placement -> acknowledgement -> retry/idempotency -> failover -> consistency |
| Can it be eventually consistent? | name operation -> user promise -> stale behavior -> convergence/repair |
| What about transactions? | invariant -> records/shards -> isolation -> timeout outcome -> alternative workflow |
| What if schema changes? | expand -> compatible reader/writer -> backfill -> validate -> contract |
| What if queries change? | new index/view/table -> CDC/backfill -> dual-read validation -> cutover |
| Why not cache? | cache accelerates; source of truth/durability/invalidation remain |
| Why not one DB? | workload isolation benefit versus propagation and operational complexity |
| What is the partition key? | locality + uniformity + stable ownership + bounded size + hotspot treatment |

## 19. Worked interview answer: Amazon-like cart and checkout

### 19.1 Clarify requirements

Assumptions:

- add/remove cart items;
- show cart quickly;
- cart can expire after inactivity;
- checkout must not create duplicate orders;
- price and stock shown earlier may be stale;
- an accepted order/payment state must be durable and auditable.

### 19.2 Separate correctness domains

```text
Cart: user convenience state; exact-key; frequent; some staleness may be tolerable.
Order: source of truth; unique; related rows/state transitions; strong correctness.
Catalog search: derived and stale-tolerant.
Product media: object storage.
```

### 19.3 Choose stores

- **Cart:** key-value by `user_id`, with TTL and a version/conditional update. A relational row with JSON can also work at smaller scale; the key-value choice is justified only by the exact-key workload and operational requirements.
- **Orders/payments:** relational database. Use unique constraints for `order_id` and `idempotency_key`; use a transaction for order plus line-item creation and a carefully designed inventory reservation/payment workflow.
- **Catalog:** relational + JSONB or document database depending on category variability and relationship/query needs; full-text search goes to a search index.
- **Media:** object storage/CDN.

### 19.4 Request path

```text
POST /checkout (Idempotency-Key: K88)
  |
  +--> read cart(user42)
  +--> fetch authoritative prices
  +--> attempt inventory reservations
  +--> DB transaction:
  |      unique insert idempotency K88
  |      create order + immutable price snapshots
  |      create line items
  |      write outbox event
  +--> commit
  +--> return stored result for K88

Outbox worker --> OrderCreated --> payment/fulfillment/search/analytics consumers
```

### 19.5 Follow-up defenses

- **Two checkouts race:** unique idempotency key and conditional inventory reservation.
- **Response is lost:** retry returns the recorded result rather than creating another order.
- **Cart price is stale:** checkout revalidates and asks user to accept changes if product semantics require it.
- **Search lags:** acceptable for browsing; checkout never trusts it for price/stock.
- **Database write succeeds, event publish fails:** transactional outbox retries after commit.
- **One customer becomes hot:** most customers are naturally distributed, but abusive/retry traffic needs rate limiting; one user's cart cannot be sped up infinitely by adding shards.
- **Need multi-region:** identify whether orders have a home region, how IDs route, and whether cross-region failover may sacrifice availability or increase coordination latency.

This answer shows why "SQL or NoSQL?" is often correctly answered with **both, at different ownership boundaries**, without making the design needlessly polyglot.

## 20. Rapid revision sheet

### 20.1 One-line models

```text
Relational:   related facts in tables; joins + constraints + transactions.
Key-value:    known key -> value; simple and partition-friendly.
Document:     bounded nested aggregate; flexible shape and locality.
Wide-column:  partition key + ordered clustering; query-first distributed history.
Graph:        nodes + first-class edges; repeated path traversal.
```

### 20.2 Never say these alone

```text
"NoSQL because scale."
"SQL because structured."
"Mongo because schema changes."
"Graph because social network."
"Cassandra because write-heavy."
"ACID prevents duplicates."
```

Always attach:

```text
access pattern + invariant + partition/transaction boundary
+ consistency/failure behavior + accepted trade-off
```

### 20.3 Fast choice cues

| Cue | Candidate model |
|---|---|
| Many shared facts, joins, constraints | Relational |
| Known exact key, TTL, simple value | Key-value |
| Bounded nested object read together | Document |
| Massive stable partition/time-range query | Wide-column |
| Variable-depth path/traversal | Graph |
| Full-text relevance | Search index |
| Huge bytes | Object storage |
| Large historical scans | Columnar analytics |

### 20.4 Final pre-answer checklist

- Did I name the top reads and writes?
- Did I identify the source of truth?
- Did I state the invariant and transaction boundary?
- Did I specify a partition/shard key and hotspot risk?
- Did I state read consistency and replica behavior?
- Did I distinguish data model from storage engine/distribution?
- Did I name one rejected alternative and why?
- Did I avoid claiming every database in a family behaves identically?
- Did I give an evolution path instead of designing for imaginary scale?

## 22. Say-out-loud 30-second narrative

> "SQL usually means a relational model: tables, keys, joins, constraints, and strong multi-row transaction support. NoSQL is not one alternative; key-value, document, wide-column, and graph stores optimize different access patterns. I choose by starting with the queries, relationship shape, invariant and transaction boundary, required consistency, and partition key. Neither side owns ACID, JSON, or horizontal scale. I normally start relational when requirements are uncertain, then justify a specialized store with a concrete workload, while stating its hotspot, duplication, consistency, and operational trade-offs."

## 23. Remaining true prerequisite gaps

**None are blocking for this chapter.** The required basics - schema, keys, indexes, transactions/ACID, normalization, replication, sharding, and consistency - were introduced inline.

For deeper interview mastery, the next true prerequisite gaps exposed by common follow-ups are:

- **database indexes and query plans** - needed to defend why a SQL query is fast or slow;
- **transaction isolation and concurrency control** - needed to reason about overselling, double booking, and write races;
- **storage engines (B-tree, WAL, LSM, SSTables, compaction)** - needed for read/write-amplification follow-ups.

These are deeper prerequisites for advanced follow-ups, not reasons to postpone using the selection framework in this chapter.
