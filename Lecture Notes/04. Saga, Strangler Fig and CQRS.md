# Saga, Strangler Fig, and CQRS: Safe Distributed Workflows, Migrations, and Read Models

> Beginner-first HLD study notes expanded from the supplied bullets. These three patterns solve different problems. The goal is to know which problem you are solving, what can fail, and how the system recovers.

## Scope, assumptions, spelling, and evidence labels

I interpret your bullets as follows:

- **"controller route x% traffic"** means a proxy, API gateway, load balancer, or feature-flag router gradually directing traffic from a legacy implementation to a replacement. This chapter explains why random request-level routing can be unsafe for writes.
- **"shared DB vs Individual DB"** means shared data/schema ownership versus database-per-service ownership. "Individual" does **not** necessarily mean a separate physical database server for every service.
- **"what happen if rollback event consumer fails"** means a saga participant fails while executing a compensating action. This is covered as a first-class failure, not dismissed as "send it to a DLQ."
- The correct spellings are **choreography** and **orchestration**.

Evidence labels used below:

- **Verified technical guidance**: supported by an original paper, an original pattern author, or first-party architecture documentation linked nearby.
- **Self-reported interview experience**: a candidate publicly reported the question or discussion. It is evidence of interview style, not a verified company question bank.
- **Illustrative product design**: a reasonable interview design for a named product. It is **not** a claim about that company's current private architecture.

No production traffic percentage, retry count, timeout, or "number of steps" rule is universal. Such values must come from the business operation, measured failure behavior, and service-level objectives.

## 1. The one-page mental map

### 1.1 The three patterns answer different questions

| Pattern | Plain-language question it answers | Main axis |
|---|---|---|
| **Strangler Fig** | How can we replace an old system gradually instead of betting on one big rewrite? | Change and migration |
| **Saga** | How can one business action update several independently owned services without one cross-service database transaction? | Write workflow and failure recovery |
| **CQRS** | What if the model best suited for safe writes is a poor model for fast or convenient reads? | Read/write modeling |

They can be used together, but one does not imply the others.

```text
                     migration problem
Legacy system  ------------------------------>  New services
                         Strangler Fig                  |
                                                       |
                                    cross-service write workflow
                                            Saga       |
                                                       v
                                      authoritative write state
                                                       |
                                    project/transform changes
                                            CQRS       |
                                                       v
                                         query-optimized views
```

### 1.2 A concrete combined example

Imagine moving checkout out of an old e-commerce monolith:

1. A **Strangler Fig façade** routes the migrated checkout capability to new services while catalog remains in the monolith.
2. Order, Inventory, and Payment now own separate data. A **saga** coordinates `reserve inventory -> authorize payment -> confirm order`.
3. The normalized order write model is inconvenient for the customer's "My Orders" page. **CQRS** builds a denormalized order-summary read model.

The patterns form a useful combination, but each still has a separate job.

### 1.3 The most important non-example

Splitting every endpoint, adding Kafka, and creating a read database does not automatically produce these patterns:

- two deployments are not a Strangler migration unless responsibility is progressively displaced and the old path is retired;
- a chain of events is not a correct saga unless failure, compensation, durable state, duplicates, and recovery are designed; and
- a primary plus a read replica is not necessarily a meaningful CQRS domain split if both sides still use the same conceptual model.

## 2. Inline prerequisites

These concepts are required for the patterns, so they are explained here rather than deferred to "study next."

### 2.1 Local transaction and ACID

A **database transaction** groups changes into one unit. In a relational database, ACID commonly means:

- **Atomicity**: all included changes commit, or none do.
- **Consistency**: application/database rules move data from one valid state to another.
- **Isolation**: concurrent transactions do not expose forbidden intermediate effects to one another.
- **Durability**: committed data survives the failures covered by the database's durability guarantee.

Example inside one database:

```text
BEGIN
  INSERT order
  DECREMENT inventory
  INSERT payment_record
COMMIT
```

If the decrement fails, the database can roll back the entire transaction.

### 2.2 Why a normal transaction stops at a service boundary

With database-per-service ownership:

```text
Order Service       Inventory Service       Payment Service
     |                      |                       |
 Order DB              Inventory DB             Payment DB
```

The Order Service's local transaction cannot simply roll back commits already made in the other databases. A distributed transaction protocol such as **two-phase commit (2PC)** can coordinate compatible resource managers, but it adds coordination, blocking/availability concerns, and tight operational coupling, and many service/datastore combinations do not support it. A saga is an application-level alternative when temporary intermediate states and semantic compensation are acceptable.

### 2.3 Invariant

An **invariant** is a rule the business must preserve, for example:

- available inventory must not become negative;
- one payment idempotency key must not capture twice; or
- a shipped order must have a valid shipping address.

Before choosing a pattern, state the invariant. If an invariant truly requires immediate atomicity across two proposed services, the service boundary may be wrong.

### 2.4 Command, query, and event

| Term | Meaning | Example |
|---|---|---|
| **Command** | A request expressing intent; it may succeed or be rejected | `ReserveInventory(orderId, items)` |
| **Query** | A request for information that does not change business state | `GetOrderSummary(orderId)` |
| **Event** | A fact that has already happened | `InventoryReserved(orderId, reservationId)` |

Do not name a request `PaymentCharged` before charging the payment. That is a fact-shaped command. Use `ChargePayment`; publish `PaymentCharged` only after success.

### 2.5 Synchronous versus asynchronous interaction

- **Synchronous**: the caller waits for a response, such as HTTP or gRPC. This is easy to follow but adds latency and runtime availability coupling.
- **Asynchronous**: a producer sends a message and the consumer handles it later. This buffers bursts and reduces temporal coupling, but introduces lag, duplicates, ordering questions, and more difficult debugging.

A saga can use either or both. "Saga" does not mean "Kafka."

### 2.6 Eventual consistency

**Eventual consistency** means copies or derived views can temporarily differ but are expected to converge when new changes stop and processing succeeds. It does not promise "consistent within a few seconds" unless the system defines and meets such a bound.

Saga intermediate states and asynchronously updated CQRS read models are commonly eventually consistent.

### 2.7 Idempotency

An operation is **idempotent for a given business key** when retrying it does not apply the business effect again.

```text
Reserve(order-123, sku-7, qty=1, operation-id=op-456)
```

The Inventory Service records `op-456`. If it receives the same request again, it returns the stored result rather than reserving another unit.

Idempotency does not mean "ignore every repeated-looking request." Two intentional purchases of the same item need different operation IDs.

### 2.8 Source of truth and materialized view

- The **source of truth** is the authoritative state used to decide correctness.
- A **materialized view** is a stored, precomputed representation derived from authoritative state to make particular queries fast.

A CQRS read model is often a materialized view. If it is deleted, the authoritative write data must still be safe; whether the view can be rebuilt depends on the retained history or another backfill source.

# Part I - Shared Database versus Database per Service

## 3. Plain-language foundation

### 3.1 The real question is ownership, not machine count

Suppose Order and Payment both use one PostgreSQL cluster. There are several very different arrangements:

```text
A. Shared ownership (high coupling)

Order Service -----+
                   +----> same order/payment tables
Payment Service ---+

B. Private logical ownership on shared infrastructure

Order Service   ----> order_schema.*      --+
                                               same physical cluster
Payment Service ----> payment_schema.*    --+
```

Arrangement B can still preserve database-per-service semantics if:

- each service has separate credentials;
- only the owner accesses its schema/tables directly; and
- cross-service access uses APIs or events.

Microsoft explicitly notes that services can share a physical database server; the coupling problem is sharing schemas/tables and bypassing ownership boundaries ([Azure data considerations](https://learn.microsoft.com/en-us/azure/architecture/microservices/design/data-considerations)).

### 3.2 Common deployment choices

| Topology | Logical ownership | Isolation/cost |
|---|---|---|
| Every service reads/writes common tables | Shared | Lowest initial friction; highest coupling |
| Private tables per service in one schema | Intended to be private, but easier to bypass | Low infrastructure cost; weak boundary unless permissions enforce it |
| Schema per service in one database | Private schema | Clearer permissions; shared database blast radius/resources |
| Logical database per service on one cluster | Private database | Stronger boundary; shared cluster operations |
| Separate cluster/server per service | Private infrastructure | Strongest resource/failure isolation; highest operational cost |

"Database per service" can use any of the last three. Never create a database **per service instance**; ten replicas of Order Service must see the same authoritative order data.

### 3.3 One fact, one authoritative owner

Data may be copied into read models and analytics stores, but one domain should own changes to an authoritative fact.

```text
Order Service owns: order status, accepted price, order lifecycle
Payment Service owns: authorization/capture/refund lifecycle
Inventory Service owns: stock and reservations
```

The Payment Service should not run:

```sql
UPDATE orders SET status = 'PAID' WHERE order_id = ...;
```

It should report the payment fact through a reply/event; the Order owner validates and applies its own state transition.

## 4. Why the choice matters in HLD

### 4.1 Shared database advantages

- One local ACID transaction can protect rules spanning several modules.
- SQL joins, constraints, reporting, and ad hoc investigation are straightforward.
- Fewer database fleets, backup plans, credentials, and schema pipelines must be operated.
- Local data access is often simpler and lower latency than network composition.
- It is a natural fit for a monolith or modular monolith.

### 4.2 Shared database disadvantages

- A schema change must remain compatible with every accessing service version.
- Services can silently depend on one another's tables and internal meanings.
- Long transactions, locks, hot tables, connection exhaustion, or a database outage affect many services.
- Independent scaling and technology choice are limited.
- Broad credentials increase security blast radius.
- Teams need coordinated database releases, weakening independent deployment.
- A "quick join" can become a permanent bypass around an intended service API.

AWS calls out both development coupling from coordinated schema changes and runtime coupling from shared transactions/locks in its [shared-database-per-service guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-data-persistence/shared-database.html).

### 4.3 Database-per-service advantages

- The owner can change its schema behind a stable contract.
- Services can scale storage independently and use a justified storage model.
- Credentials enforce a smaller security boundary.
- load, lock, and failure blast radii can be separated.
- A team can own behavior, data, migration, and operations end to end.

### 4.4 Database-per-service disadvantages

- A local ACID transaction cannot cover the full cross-service workflow.
- Cross-service joins become API composition, replicated read models/CQRS, or offline analytics.
- Data copies introduce propagation lag, reconciliation, deletion, and lineage problems.
- More stores mean more provisioning, monitoring, backups, restores, upgrades, and expertise.
- Network calls and messages introduce timeouts, duplicates, partial failure, and contract evolution.
- Poor boundaries create chatty services and distributed complexity without autonomy.

### 4.5 The compressed comparison

| Decision axis | Shared database | Database per service |
|---|---|---|
| Cross-domain transaction | Easy local ACID | Saga/2PC/redesign required |
| Cross-domain query | SQL join | API composition or projected view |
| Schema evolution | Coordinated | Owner-local behind contract |
| Team autonomy | Lower | Higher if boundaries are real |
| Operational overhead | Lower | Higher |
| Failure/resource isolation | Shared | Can be isolated |
| Polyglot storage | Limited | Possible, not mandatory |
| Best default for | Modular monolith, small team, unclear domain | Mature boundaries and independent teams/workloads |

## 5. When shared data is reasonable, transitional, or a smell

### 5.1 Reasonable

Use a shared database deliberately when:

- the application is a modular monolith;
- one team deploys the modules together;
- strong cross-module transactions are central;
- the product/domain is still changing rapidly; or
- the operational cost of many stores is not justified.

Shared storage is not automatically bad design. Pretending multiple services are independent while they freely edit one another's tables is the problem.

### 5.2 Transitional during extraction

A newly extracted service may initially use legacy tables to reduce migration risk. Treat this as a named migration phase with:

- one declared writer for each table/fact;
- restricted permissions;
- compatible schema changes;
- dependency monitoring; and
- an exit criterion and target owner.

Without an exit criterion, "temporary" shared ownership tends to become permanent.

### 5.3 A boundary smell

If two proposed services continually need:

- one transaction;
- synchronous calls in both directions;
- direct joins on every request; and
- coordinated schema/releases,

they may be one cohesive service or bounded context. First reconsider the split; do not automatically add a saga and CQRS to repair a poor boundary.

### 5.4 Alternatives for cross-service reads

| Need | Usual option | Main trade-off |
|---|---|---|
| Small, low-fan-out, fresh query | API composition | Runtime latency and partial failure |
| High-read or multi-domain query | CQRS/materialized view | Staleness and projection operations |
| Ad hoc analytics/reporting | Warehouse/lake/ETL | Not an online transaction path |
| One strongly consistent invariant | Keep data in one owner/transaction | Larger service boundary |

### 5.5 Database-choice answer spine

Use **O-A-Q-O-M**:

1. **O - Ownership:** who is the authoritative writer for each fact?
2. **A - Atomicity:** which invariants require one transaction?
3. **Q - Queries:** how will cross-domain reads work?
4. **O - Operations:** what isolation and fleet cost can the team support?
5. **M - Migration:** is shared storage permanent or a controlled intermediate phase?

# Part II - Saga Pattern

## 6. Saga in plain language

### 6.1 The problem

A customer places an order. The business action spans three owners:

```text
Order Service      Inventory Service      Payment Service
 create order       reserve stock          authorize money
```

If all used one database, one transaction might commit or roll back together. With separate stores, Inventory may commit a reservation and Payment may then reject the card. The system needs a defined way to reach a valid final state.

### 6.2 The idea

A **saga** is a sequence of local transactions. Each participant atomically changes its own data. If a later step cannot complete, the workflow runs **compensating transactions** for already completed, compensatable steps.

The original Saga paper by Hector Garcia-Molina and Kenneth Salem was published in 1987 ([ACM SIGMOD, *Sagas*](https://doi.org/10.1145/38713.38742)). Modern microservice sagas apply the local-transaction and compensation idea across service-owned state.

```text
Forward path:
T1 Create pending order
T2 Reserve inventory
T3 Authorize payment
T4 Confirm order

Failure at T3:
C2 Release inventory
C1 Cancel pending order
```

This is not a database rollback. `Release inventory` and `Cancel order` are new committed business actions.

### 6.3 Compensation is semantic undo, not time travel

Suppose a payment was captured and then shipping failed. A refund does not erase the original payment:

- the customer may see both charge and refund;
- fees or exchange-rate differences may exist;
- refund settlement may take time;
- audit records must remain; and
- another concurrent action may have occurred.

Microsoft's compensating-transaction guidance emphasizes that compensation is application-specific, may not restore the exact starting state, is eventually consistent, and can itself fail ([Azure Compensating Transaction pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/compensating-transaction)).

Do not mechanically assume every compensation runs in exact reverse order. Business dependencies determine the safe order, and independent compensations may sometimes run in parallel. For example, releasing a scarce inventory reservation might be more urgent than updating a noncritical reporting record.

### 6.4 Compensatable, pivot, and retryable steps

A useful design classifies saga steps:

- **Compensatable transaction:** can be semantically undone, such as releasing a reservation.
- **Pivot transaction:** the point of no return. After it commits, the saga should normally finish through forward recovery rather than compensate earlier work.
- **Retryable transaction:** a later step designed to eventually succeed when retried, such as sending a notification after the committed business decision.

The pivot is a business choice, not always payment. Dispatching a physical parcel, submitting a market trade, or sending an irreversible external instruction may be the point of no return.

Design the order of work to delay irreversible actions:

```text
validate -> reserve -> authorize/hold -> commit decision -> capture/dispatch -> notify
             ^             ^                  ^
       easy to release  can void hold       possible pivot
```

## 7. A saga as an explicit state machine

Do not model a saga as "some events happen." Model allowed states and transitions.

```text
PENDING
   |
   v
INVENTORY_RESERVED
   |
   v
PAYMENT_AUTHORIZED
   |
   v
CONFIRMED --------------------------> COMPLETED
   |
 failure before pivot
   v
COMPENSATING -> INVENTORY_RELEASING -> CANCELED
                       |
                  repeated failure
                       v
                  MANUAL_REVIEW
```

Example durable workflow record:

```text
saga_id:             saga-901
business_key:        order-123
state:               COMPENSATING
completed_steps:     [CREATE_ORDER, RESERVE_INVENTORY]
pending_action:      RELEASE_INVENTORY
attempt_count:       4
next_attempt_at:     2026-07-16T12:30:00Z
last_error_code:     INVENTORY_TIMEOUT
workflow_version:    7
```

The record lets a restarted worker continue instead of guessing from logs.

### 7.1 Intermediate states are part of the product

While the saga runs, users and other services can see temporary states. Define their meaning:

| State | User/API behavior |
|---|---|
| `PENDING` | Show "processing"; do not promise shipment |
| `CONFIRMED` | Business decision committed |
| `CANCEL_PENDING` | Stop fulfillment; compensation still running |
| `REFUND_PENDING` | Show refund initiated; support can inspect it |
| `MANUAL_REVIEW` | Do not silently claim success or cancellation |

Returning HTTP `200 Order complete` before the workflow's promised state is durable creates a product contract the backend may not satisfy. A common alternative is `202 Accepted` plus an operation/order status endpoint for long-running work.

## 8. Choreography-based saga

### 8.1 Plain-language model

In **choreography**, there is no component directing every step. Each service reacts to events and publishes the next fact.

```text
Client
  |
  v
Order Service --OrderCreated------------> Event broker
                                               |
                                               v
                                        Inventory Service
                                               |
                            +------------------+------------------+
                            |                                     |
                  InventoryReserved                    InventoryRejected
                            |                                     |
                            v                                     v
                     Payment Service                         Order cancels
                            |
                 +----------+-----------+
                 |                      |
         PaymentAuthorized       PaymentFailed
                 |                      |
                 v                      +--> Inventory releases
           Order confirms                   Order cancels
```

Events announce facts; participants decide their own valid local transition.

### 8.2 Advantages

- No workflow coordinator is required for a small, naturally event-driven flow.
- Participants remain responsible for their own domain logic.
- New independent reactions, such as analytics, may subscribe without changing the producer.
- The broker can buffer temporary consumer outages.

### 8.3 Disadvantages and edge cases

- The end-to-end workflow is spread across handlers and schemas.
- It becomes difficult to answer "where is order 123 stuck?" without correlation and tracing.
- Event cycles and accidental dependencies can grow as steps increase.
- Multiple consumers may race during compensation.
- Timeout decisions have no obvious owner unless one participant explicitly owns them.
- Changing the process can require coordinated event-contract changes.
- A published event may have many consumers, so its compatibility surface can become wide.

### 8.4 Good fit

Use choreography when the process is short, reactions are genuinely independent, and no single component needs a rich global view of deadlines and branching. There is no reliable universal rule such as "choreography for at most four steps."

## 9. Orchestration-based saga

### 9.1 Plain-language model

In **orchestration**, a durable coordinator records the workflow and tells participants which command to execute next.

```text
                         +----------------------+
Client ----------------->| Order Saga           |
                         | Orchestrator + state |
                         +----+---------+-------+
                              |         |
             ReserveInventory |         | AuthorizePayment
                              v         v
                       Inventory      Payment
                        Service       Service
                              ^         ^
                              | reply   | reply
                              +----+----+
                                   |
                      next command or compensation
```

The orchestrator decides process flow. Inventory still decides whether stock can be reserved; Payment still owns payment rules.

### 9.2 Advantages

- The workflow, branches, deadlines, and compensation order are explicit.
- One durable record makes status, operations, and audit easier.
- Complex workflows can be changed without every participant knowing the whole process.
- Centralized timeout and retry policies are easier to reason about.
- Participants can expose focused commands and replies rather than subscribing to many unrelated facts.

### 9.3 Disadvantages and edge cases

- The coordinator can accumulate domain logic and become a "god orchestrator."
- Its state, versioning, availability, and concurrency need careful design.
- Every participant depends on coordinator contracts.
- A synchronous orchestrator can become a latency bottleneck if designed as one in-memory process.

### 9.4 Is the orchestrator a single point of failure?

Not inherently. A production orchestrator can be:

- replicated as multiple workers;
- backed by durable workflow state;
- partitioned by saga/business key;
- protected by leases, compare-and-set, or optimistic concurrency; and
- resumed by another worker after a crash.

The **logical role** is central, but the runtime need not be one fragile server. The real risk is centralizing too much business knowledge or losing durable progress.

### 9.5 Good fit

Use orchestration when the business process has branching, deadlines, human steps, several compensations, or a strong need for end-to-end visibility and controlled evolution.

### 9.6 Choreography versus orchestration

| Dimension | Choreography | Orchestration |
|---|---|---|
| Flow definition | Emerges across event handlers | Explicit in coordinator/workflow |
| Participant knowledge | Knows events it reacts to | Knows commands/replies |
| End-to-end visibility | Requires reconstructed trace | Natural from workflow state |
| Simple independent reactions | Strong fit | May be unnecessary machinery |
| Complex branches/timeouts | Becomes difficult | Strong fit |
| Coupling | Event/schema and semantic coupling | Coordinator-contract coupling |
| Failure recovery owner | Distributed | Explicit coordinator |
| Main risk | Event spaghetti/cycles | God orchestrator/central bottleneck |

Real systems often combine them: orchestration manages the critical order lifecycle, while facts such as `OrderConfirmed` are choreographed to analytics and notification consumers.

## 10. Making a saga reliable

### 10.1 The database-plus-message atomicity gap

A service often needs to:

1. update its database; and
2. publish an event/reply.

These two dangerous orders exist:

```text
commit DB -> crash -> event never sent

send event -> DB transaction rolls back -> consumer acts on a fact that is false
```

### 10.2 Transactional outbox

Write the business change and an outbox row in the **same local database transaction**:

```text
BEGIN
  UPDATE inventory_reservation ...
  INSERT outbox(event_id, type, payload, status='NEW')
COMMIT
```

A relay later publishes the outbox record. If it crashes after publish but before marking the row sent, it may publish again, so consumers still need idempotency. The outbox closes the "committed but never announced" gap; it does not create end-to-end exactly-once effects. See [Azure's transactional outbox explanation](https://learn.microsoft.com/en-us/azure/architecture/databases/guide/transactional-out-box-cosmos) and the [Transactional Outbox pattern](https://microservices.io/patterns/data/transactional-outbox).

### 10.3 Idempotent consumer/inbox

The consumer records processed message or operation IDs atomically with its business change:

```text
BEGIN
  INSERT inbox(message_id)       -- unique constraint
  APPLY business transition
  INSERT outbox(next_event)
COMMIT
```

If `message_id` already exists, return the prior outcome or acknowledge without reapplying the effect.

### 10.4 Ordering and stale messages

Messages can arrive late or out of order. Protect the domain state machine:

```text
Expected: PENDING -> RESERVED -> CONFIRMED
Late event: PaymentAuthorized arrives after order is CANCELED
Action: reject/no-op according to state/version; do not resurrect the order
```

Useful controls include:

- partitioning related messages by aggregate/business key;
- sequence numbers or entity versions;
- compare-and-set/optimistic concurrency; and
- explicit allowed state transitions.

Global ordering is usually unnecessary and expensive. Preserve the order that the invariant actually needs, often per order/account.

### 10.5 Timeouts and unknown outcomes

A timeout means **unknown**, not definitely failed.

```text
Orchestrator -> Payment: Authorize(op-456)
Payment commits authorization
Reply is lost
Orchestrator times out
```

Blindly starting a new authorization can double-charge. Retry with the same idempotency key or query the provider by that key. Decide failure only when the business outcome is known or the contract makes the retry safe.

### 10.6 Retry classification

- **Transient infrastructure error**: retry with bounded exponential backoff, jitter, and a deadline appropriate to the business.
- **Permanent business rejection**: do not retry the same impossible request; compensate or choose an alternate path.
- **Unknown outcome**: query/deduplicate before deciding.
- **Poison message/code defect**: isolate it, alert, repair code/data, then replay safely.

A dead-letter queue is a parking place, not a completed recovery strategy.

### 10.7 Concurrency and saga isolation

Sagas do not provide the isolation of one serializable database transaction. Another workflow can see or change intermediate state.

Mitigations include:

- explicit `PENDING`/`RESERVED` states that other operations respect;
- reservation records instead of decrement-and-hope;
- short reservation expiry with a durable expiry process;
- optimistic version checks;
- escrow/allocated quotas;
- commutative operations where possible; and
- keeping truly atomic invariants inside one service.

Example: Inventory should atomically check and reserve stock locally. A saga must not read a stale quantity in Order Service and later ask Inventory to decrement unconditionally.

### 10.8 Observability and operations

Track at least:

- saga ID and business ID;
- current state and workflow version;
- step attempt count and last classified error;
- age in state and deadline;
- outbox/inbox backlog and oldest age;
- compensation success/failure;
- stuck/manual-review count; and
- trace/correlation IDs across commands and events.

Operators need safe commands such as `retry step`, `reconcile with provider`, `force forward after verified external outcome`, and `start approved manual compensation`. Every action should be audited.

## 11. Your key follow-up: what if the rollback-event consumer fails?

First correct the wording:

> In a saga, this is not a database rollback. It is a **compensating command/event-driven action**, and that action is another distributed operation that can fail.

### 11.1 Example

```text
1. Inventory reserves item R-77               success
2. Payment authorization                      rejected
3. Workflow requests ReleaseReservation(R-77)
4. Inventory compensation consumer            crashes
```

The order must not be marked fully canceled with cleanup complete merely because the compensation message was sent.

### 11.2 Recovery sequence

1. **Persist the obligation.** The saga remains `COMPENSATING` or `CANCEL_PENDING` with `RELEASE_INVENTORY` outstanding.
2. **Use durable delivery.** The broker redelivers if the consumer did not acknowledge, or the orchestrator retries the command.
3. **Make compensation idempotent.** `ReleaseReservation(R-77, compensation-op-9)` can run more than once without increasing stock twice.
4. **Classify and retry.** Retry infrastructure failures; do not endlessly retry an invalid business request without investigation.
5. **Reconcile.** A scheduled scanner compares stuck saga state with Inventory's authoritative reservation state.
6. **Alert and escalate.** After the automatic policy/deadline, move to `MANUAL_REVIEW` while retaining the outstanding obligation.
7. **Prefer forward recovery after a pivot.** If the irreversible business decision already occurred, complete the promised action rather than pretending it can be undone.
8. **Expose honest status.** The user may see `cancellation pending` or `refund pending`; support sees the error and remediation history.

```text
Compensation request
       |
       v
durable broker/outbox -> consumer -> local idempotent transaction
       |                    |
       | timeout/failure    +--> CompensationSucceeded
       v
retry scheduler -> reconciliation scan -> alert/manual repair
       |
       +---- saga stays open until terminal state is verified
```

### 11.3 Why idempotency matters during compensation

The consumer might release the reservation, commit, and crash before acknowledging. The broker then redelivers. Without a unique reservation/operation check, the second execution may add stock again and create inventory from nothing.

Correct local model:

```text
reservation R-77: ACTIVE -> RELEASED

release when ACTIVE:  change to RELEASED and return quantity
release when RELEASED: return prior success; do not add quantity again
```

### 11.4 What if compensation is permanently impossible?

Examples include a parcel already handed to a carrier or an external transfer that cannot be canceled. The design choices are:

- take the compensatable action earlier, such as an authorization hold rather than immediate capture;
- delay the irreversible pivot;
- use a different compensating business action, such as refund/return/store credit;
- complete forward and handle an exception workflow; or
- keep the invariant inside one atomic boundary if no temporary inconsistency is acceptable.

If no valid compensation or forward recovery exists, "use Saga" is not a complete design.

### 11.5 Concise answer spine for this follow-up

Use **P-R-I-R-E**:

1. **P - Persist** the incomplete compensation state.
2. **R - Retry** through durable delivery with error classification.
3. **I - Idempotency** prevents duplicate undo effects.
4. **R - Reconcile** authoritative state and alert on age/deadline.
5. **E - Escalate** to manual or forward recovery; never silently call it complete.

---

## 12. Saga failure matrix

| Failure | Risk | Design response |
|---|---|---|
| Participant crashes before local commit | Step did not happen | Safe retry |
| Participant commits but cannot publish | Workflow stalls | Transactional outbox |
| Publisher sends twice | Duplicate next step | Event ID + idempotent inbox/handler |
| Consumer commits then crashes before ACK | Redelivery repeats effect | Atomic inbox/business change |
| Reply is lost | Outcome unknown | Same idempotency key + status lookup |
| Messages arrive out of order | Invalid state resurrection | Per-key order, version, guarded state machine |
| Business rule rejects step | Forward path impossible | Defined compensation/alternate branch |
| Compensation consumer is down | Resources remain held | Persist, retry, reconcile, alert |
| Compensation permanently fails | Saga cannot reach normal terminal state | Manual/alternate/forward recovery |
| Two sagas update same entity | Isolation anomaly | Local invariant, reservation, version check |
| Orchestrator crashes | Workflow appears stuck | Durable state + another worker resumes |
| Broker unavailable | Unpublished work accumulates | Outbox backlog, backpressure, alerts |
| Poison message | Partition/workflow can stall | Quarantine without forgetting correctness gap; repair/replay |

---

## 13. Saga trade-offs, alternatives, and decision rules

### 13.1 Benefits

- Preserves service data ownership.
- Avoids holding one distributed lock for a long business process.
- Supports asynchronous and long-running workflows.
- Makes business-specific recovery explicit.
- Can remain available when a participant is temporarily unavailable by waiting/retrying.

### 13.2 Costs

- Temporary inconsistency is user-visible unless hidden by state design.
- Compensation and isolation logic are domain-specific and difficult.
- Message delivery, deduplication, timeouts, and observability add infrastructure/code.
- Testing the failure combinations is much harder than testing one transaction.
- Operational repair becomes part of the product.

### 13.3 Alternatives

| Alternative | Prefer it when | Trade-off |
|---|---|---|
| One local ACID transaction | The invariant belongs in one service/store | Coarser boundary, but simplest correctness |
| 2PC/distributed transaction | Compatible controlled resources, short transaction, atomic visibility required | Coordination, blocking/availability and operational coupling |
| Try-Confirm-Cancel (TCC) | Resources support explicit holds and confirm/cancel | Every participant must implement reservation lifecycle |
| Durable workflow engine | Orchestration needs timers, retries, human tasks, visibility | Engine operations/versioning; business semantics still yours |
| Async job/state machine without compensation | Work only needs eventual forward completion | Not suitable when prior effects must be undone |
| Manual process | Rare, high-value exception is cheaper to review | Slow and operationally expensive; must be audited |

A workflow engine can implement an orchestration-based saga; it does not invent correct compensation or invariants for you.

### 13.4 Use Saga when

- one business process legitimately crosses independently owned services;
- each service can commit locally;
- temporary states are acceptable and modeled;
- completed steps have valid compensation, or the workflow can recover forward; and
- the organization can operate durable messages/workflows and reconciliation.

### 13.5 Avoid or redesign when

- the invariant cannot tolerate any intermediate inconsistency;
- compensation is impossible and forward recovery cannot meet the promise;
- the steps are actually one cohesive aggregate split across arbitrary services;
- a short transaction in one database solves the problem cleanly; or
- the team lacks basic idempotency, outbox, observability, and repair capability.

### 13.6 Common mistakes

- Saying "Saga gives ACID across microservices."
- Calling compensation a rollback that erases history.
- Publishing events after a commit without an outbox/equivalent.
- Assuming the broker guarantees exactly-once business effects.
- Treating timeout as failure instead of unknown outcome.
- Retrying permanent business rejection.
- Marking a saga done when compensation was merely requested.
- Using a DLQ as the final recovery plan.
- Letting the orchestrator edit participant databases.
- Using a saga to preserve an invariant that should live in one service.

### 13.7 Say-out-loud 30-second Saga narrative

> "A saga handles one business workflow across service-owned databases as a sequence of local transactions. Each service preserves its own invariant, and if a later step fails, earlier compensatable actions are reversed with new business transactions rather than a database rollback. I choose choreography for simple independent reactions and orchestration for explicit branches, deadlines, and visibility. Reliability requires a durable state machine, outbox, idempotent consumers, guarded ordering, timeout handling, and reconciliation. If compensation fails, the saga stays open, retries safely, alerts, and moves to manual or forward recovery; it never silently claims the workflow is complete."

# Part III - Strangler Fig Pattern

## 14. Strangler Fig in plain language

### 14.1 The problem with a big-bang rewrite

An old system still serves customers while a replacement is built. A complete rewrite often has one risky cutover date, while the old system keeps changing and hidden behavior must be rediscovered.

The **Strangler Fig pattern** gradually places new functionality around the old system. A façade intercepts calls and sends migrated capabilities to the new implementation; unmigrated capabilities continue to use the old implementation. Eventually the old responsibility is removed.

Martin Fowler introduced the metaphor to emphasize incremental replacement and reduced cutover risk ([original Strangler Fig Application](https://martinfowler.com/bliki/OriginalStranglerFigApplication.html)).

```text
Phase 1
Client -> façade/proxy -> legacy system -> legacy DB

Phase 2
                         +-> new Order Service -> Order DB
Client -> façade/proxy --+
                         +-> legacy system -----> legacy DB

Phase 3
Client -> façade/proxy -> new services -> owned databases
                           X legacy order path retired
```

### 14.2 What is actually being migrated?

Choose a **business capability or seam**, not random classes:

- endpoint/path;
- tenant/customer cohort;
- product or region;
- event consumer;
- scheduled job;
- domain workflow; or
- data ownership boundary.

The goal is to transfer responsibility and eventually delete/decommission the old path.

### 14.3 Strangler Fig versus canary deployment

| Strangler Fig | Canary deployment |
|---|---|
| Moves a capability from a legacy system to a replacement over time | Releases a new version to a small audience before broad rollout |
| Often changes architecture, ownership, and data | Often changes only implementation version |
| Completion means legacy responsibility is retired | Completion means the new version serves all traffic |

A Strangler migration can **use** canary traffic shifting, feature flags, shadowing, or blue-green deployment. Those mechanisms are not the pattern itself.

---

## 15. A safe Strangler migration flow

### 15.1 Phase 0 - Discover and baseline

Before extracting:

- map callers, batch jobs, database triggers, files, queues, reports, and admin tools;
- characterize current behavior, including relied-upon quirks;
- add contract/characterization tests around the seam;
- record latency, error, throughput, and business-success baselines;
- identify the authoritative data writer; and
- define rollback/roll-forward and decommission criteria.

The happy-path HTTP endpoint is rarely the whole capability.

### 15.2 Phase 1 - Introduce an interception point

```text
Clients -> gateway/facade/router -> legacy
```

Initially it can pass all traffic through. Make it highly available, observable, and thin. It should not become a new home for domain logic.

Possible interception points:

- reverse proxy/API gateway;
- load balancer;
- application façade;
- event/message bridge;
- feature-flagged routing inside a caller; or
- anti-corruption adapter at the new boundary.

### 15.3 Phase 2 - Build the replacement behind a compatible seam

The new service can expose the old contract through an adapter while using a cleaner internal model. An **anti-corruption layer** translates legacy names/semantics so they do not infect the new domain.

Do not simply copy legacy table structure and call it a new bounded context unless that model genuinely fits.

### 15.4 Phase 3 - Prepare and verify data

Common sequence:

1. backfill historical data;
2. capture new changes with an outbox/CDC/event stream;
3. catch up to a known watermark/version;
4. compare counts, checksums, invariants, and sampled business results;
5. shadow or dark-read the new path; and
6. define acceptable mismatch and lag thresholds from business requirements.

### 15.5 Phase 4 - Shift a controlled slice

Start with internal/test users or a stable low-risk cohort. Observe technical and business metrics. Increase only when readiness gates pass.

### 15.6 Phase 5 - Transfer authoritative ownership

Changing read routing is easier than changing the writer. Explicitly decide:

- when the new system becomes the source of truth;
- how in-flight work is drained;
- whether the legacy system stays updated during a rollback window;
- how caches are invalidated; and
- what happens to old clients and jobs.

After new-only writes begin, rollback is **not** merely setting traffic back to 0% unless the legacy store has been kept current or a reverse migration exists.

### 15.7 Phase 6 - Decommission, do not just abandon

Remove:

- legacy routes and code;
- old writers/readers and credentials;
- sync/dual-write jobs;
- stale tables, queues, dashboards, and alerts;
- compatibility flags after their safe lifetime; and
- obsolete operational runbooks.

Prove from telemetry that no callers or data dependencies remain. A legacy system receiving 0% web traffic may still be used by batch jobs or reports.

Azure and AWS both describe the façade-based incremental route from legacy to replacement and highlight proxy and data-store concerns ([Azure Strangler Fig](https://learn.microsoft.com/en-us/azure/architecture/patterns/strangler-fig), [AWS Strangler Fig](https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/strangler-fig.html)).

---

## 16. Your bullet: "controller route x% traffic"

### 16.1 Where routing can live

The "controller" is usually one of:

```text
DNS/edge -> load balancer -> API gateway/reverse proxy -> application router
                                 |
                         feature flag/cohort rule
```

Keep the rule in a controlled routing/flag layer. A normal MVC controller can route, but mixing migration logic into domain handlers makes removal and auditing harder.

### 16.2 Five useful routing strategies

| Strategy | Example | Appropriate use |
|---|---|---|
| Capability/path | `/v2/orders/*` -> new | Cleanly separated function |
| Stable entity cohort | `hash(customerId) % 100 < x` | Progressive stateful migration |
| Tenant/region/product | tenant A or region B -> new | Operationally coherent slice |
| Explicit header/flag | employee/beta users -> new | Testing and controlled preview |
| Random request percentage | random x% -> new | Stateless/read-only canary, or shared compatible state |

The hash formula is illustrative. The key point is **stable assignment**, not the exact function.

### 16.3 Why random request-level splitting can break writes

Suppose order 123 is created in the new system, but the next random request goes to legacy:

```text
POST /orders/123       -> new store
PATCH /orders/123/pay  -> legacy store (order missing/stale)
GET /orders/123        -> random result changes between requests
```

For stateful capabilities, route consistently by the entity/tenant owner, or ensure both implementations use one authoritative compatible state during that phase.

Sticky sessions alone are weak: users switch devices, workers process messages without sessions, and one user can act on many entities. Route using the business ownership key when possible.

### 16.4 Shadow traffic is not normal traffic splitting

**Shadow/mirrored traffic** sends a copy to the new system while the legacy response remains authoritative.

```text
                      +-> legacy -> response to client
request -> duplicator-+
                      +-> new -> compare result only
```

Safe for pure reads when privacy/capacity are handled. For writes, the shadow path must suppress external effects or use an isolated sandbox/idempotent simulation. Mirroring a payment or email command can charge or notify twice.

### 16.5 A progressive traffic plan

Do not memorize `1% -> 10% -> 50% -> 100%`. Use gates:

1. internal/test cohort;
2. shadow reads and compare;
3. low-risk stable production cohort;
4. larger cohorts after latency/error/business invariants pass;
5. all reads for the migrated capability;
6. authoritative write transfer;
7. rollback window/roll-forward validation; and
8. legacy retirement.

### 16.6 Traffic-cutover answer spine

Use **U-S-W-O-R**:

1. **U - Unit:** path, user, tenant, region, or entity being migrated.
2. **S - Stable:** keep related stateful operations on one side.
3. **W - Writer:** name the sole authoritative writer at each phase.
4. **O - Observe:** compare technical and business invariants.
5. **R - Recover:** define rollback or roll-forward including data, not only routing.

---

## 17. Data migration: the hardest part of Strangler Fig

### 17.1 Safe ownership phases

```text
Phase A: legacy is authoritative
legacy write -> legacy DB -> outbox/CDC -> new DB
reads: legacy; new is compared

Phase B: cohort cutover
router sends owned cohort -> new service
ownership rule says which system writes each entity

Phase C: new is authoritative
new write -> new DB -> optional reverse feed during rollback window
reads: new

Phase D: legacy retired
reverse feed and old schema removed
```

Entity/cohort ownership must be durable and discoverable. Two writers independently editing the same fact invites lost updates and conflict.

### 17.2 Why application dual write is dangerous

```text
request -> write legacy DB succeeds
        -> write new DB times out
```

The caller cannot atomically commit two independent databases. A retry might duplicate one side. Prefer:

- one authoritative write plus outbox/CDC replication;
- replayable, idempotent change records;
- explicit watermarks/checkpoints; and
- reconciliation.

If temporary dual write is unavoidable, record durable per-write status, use idempotency/versioning, define repair, and keep the phase short. "Write both" alone is not a consistency design.

### 17.3 Backfill plus live changes

A robust migration handles:

- updates occurring while the backfill runs;
- deletes/tombstones, not only inserts;
- event order and entity versions;
- different IDs and field semantics;
- legacy duplicates or invalid rows;
- privacy/retention rules in both stores;
- checksum/invariant comparison;
- retryable chunks and resume checkpoints; and
- the point at which backfill and change stream meet without a gap.

### 17.4 Shared database as a bridge

The new service may temporarily use the legacy database. Advantages:

- less initial data movement;
- existing transactions still work; and
- behavior can be extracted before storage.

Disadvantages:

- schema and release coupling remain;
- the monolith can bypass new service rules;
- ownership is ambiguous;
- independent scaling/failure isolation are limited; and
- the migration can stall in a distributed monolith.

Use permissions and one-writer rules, then migrate data ownership as a separate controlled step.

### 17.5 Schema evolution: expand, migrate, contract

1. **Expand:** add backward-compatible fields/tables/API behavior.
2. **Migrate:** deploy readers/writers that understand both forms and backfill.
3. **Contract:** remove old fields only after all callers and rollback versions stop using them.

Do not combine an incompatible schema drop with the first traffic cutover.

---

## 18. Strangler Fig trade-offs, edge cases, and decision rules

### 18.1 Benefits

- Delivers value and learning incrementally.
- Limits each cutover's blast radius.
- Allows measured comparison with existing behavior.
- Supports rollback before irreversible ownership changes.
- Lets teams prioritize high-value/high-change capabilities.

### 18.2 Costs

- Old and new systems coexist, increasing temporary operational cost.
- A façade, adapters, sync, flags, and compatibility layers must be maintained.
- Data ownership/cutover can be more difficult than code extraction.
- The migration can take long enough for temporary mechanisms to become permanent.
- End-to-end debugging crosses both architectures.

### 18.3 Hidden edge cases interviewers probe

- scheduled jobs and database triggers still writing old tables;
- callbacks/webhooks returning to the wrong implementation;
- long-running operations crossing the cutover time;
- caches containing values from different authorities;
- auth/session semantics differing across systems;
- old mobile clients with long upgrade cycles;
- reports/admin tools bypassing the façade;
- event consumers and replay after ownership moves;
- rollback after the new database accepted writes; and
- feature flags whose combinations were never tested.

### 18.4 Use Strangler Fig when

- the legacy system is large, valuable, and replaceable in meaningful seams;
- it can coexist with replacement components;
- requests/events can be intercepted; and
- gradual risk reduction matters more than a fast complete shutdown.

### 18.5 Avoid or choose another migration when

- the system is small enough for a simple replacement;
- no meaningful interception/boundary exists without changing everything;
- legal/security constraints prohibit prolonged dual operation;
- the old system must be shut down immediately; or
- coexistence/data synchronization costs exceed a tested cutover approach.

### 18.6 Common mistakes

- Defining Strangler Fig as "send x% traffic to microservices."
- Randomly splitting state-changing requests without stable ownership.
- Letting old and new become independent writers of the same fact.
- Mirroring irreversible commands.
- Treating application dual write as atomic.
- Moving code but leaving every table and release coupled forever.
- Putting business logic in the gateway.
- Calling the migration complete at 100% traffic while batch/report dependencies remain.
- Assuming routing rollback also rolls data back.
- Never deleting flags, adapters, old code, or credentials.

### 18.7 Say-out-loud 30-second Strangler Fig narrative

> "The Strangler Fig pattern replaces a legacy system capability by capability behind a façade instead of using one big-bang rewrite. I first map hidden callers and establish tests and metrics, then build a compatible new path, backfill and synchronize data, shadow it, and shift stable cohorts. For writes I name one authoritative owner at every phase; random x-percent routing and uncoordinated dual writes are unsafe. Each increase is gated by technical and business checks, and rollback includes data synchronization, not just a router flag. The migration finishes only when old code, data access, jobs, and compatibility machinery are actually retired."

# Part IV - CQRS

## 19. CQRS in plain language

### 19.1 Start with normal CRUD

A simple application can use one model for both updates and reads:

```text
Create/Update Order ----+
                        +--> Order model/table
Get/List Order ---------+
```

This is usually the best starting point.

As the domain grows, safe writes and convenient reads may want very different shapes:

- Writing an order needs validation, pricing rules, state transitions, and normalized authoritative data.
- Reading "My Orders" needs one fast row/card containing order, payment summary, shipment ETA, and display text.

Forcing one model to do both can create complicated joins and a domain model polluted with presentation concerns.

### 19.2 The idea

**Command Query Responsibility Segregation (CQRS)** uses a different model for changing state than for reading state.

```text
Commands -> command model -> authoritative write data
                                      |
                                      | changes/events
                                      v
Queries  <- query model   <- query-optimized read data
```

Martin Fowler's concise definition is that CQRS uses different models to update information and to read information, while warning that it adds risky complexity for most systems ([Fowler, CQRS](https://martinfowler.com/bliki/CQRS.html)).

### 19.3 Commands are business intent

Prefer:

```text
PlaceOrder
CancelOrder
ChangeDeliveryAddress
```

over generic commands such as:

```text
SetOrderStatus("CONFIRMED")
UpdateOrderField("total", 999)
```

The command side validates authorization, current state, invariants, and allowed transitions.

### 19.4 Queries do not change business state

A query can read a denormalized projection and return it directly. It should not secretly confirm an order or consume business inventory. Incidental caching, access logs, and metrics are not normally considered business-state mutations.

Practical command handlers may return an accepted/rejected result, ID, or version even though the original method-level Command-Query Separation principle says commands do not return a value. Do not turn the command response into an arbitrary read model.

---

## 20. CQRS is a spectrum

### 20.1 Entry step - separate interfaces, same model and database

```text
Command handlers --+
                   +--> same domain model / same DB
Query handlers -----+
```

This is useful **Command-Query Separation** and layering, but it is not yet the full CQRS model split because both paths share one conceptual model. It introduces no cross-store projection lag and can be a sensible stopping point.

### 20.2 Level 2 - Separate models, same database

```text
Rich write/domain model -> normalized tables
Simple query model      -> SQL view/materialized table in same DB
```

The database transaction may update both synchronously. Read and write models differ even though storage is shared.

### 20.3 Level 3 - Separate models and stores

```text
Client
  |
  +--> Command API -> domain handler -> Write DB + Outbox
  |                                      |
  |                                      v
  |                                 Event broker
  |                                      |
  |                                      v
  +<-- Query API <---- Read DB <---- Projector
```

This enables independent scaling and specialized schemas but normally introduces projection lag and more operations.

### 20.4 What CQRS does not require

CQRS does **not** inherently require:

- separate physical databases;
- microservices;
- asynchronous messages;
- event sourcing;
- Kafka; or
- eventual consistency.

Those are possible implementation choices. Separate stores synchronized asynchronously are eventually consistent; separate models updated in one local transaction need not be.

---

## 21. Why and when CQRS is useful

### 21.1 Benefits

- Read and write models can match their own jobs.
- Read-heavy and write-heavy workloads can scale independently.
- Read schemas can be denormalized and precomputed for low latency.
- The command model can focus on invariants and business language.
- Query APIs can expose only safe display data.
- Multiple read models can serve different screens/search/reporting needs.

### 21.2 Costs

- More models, handlers, schemas, deployments, and tests.
- Separate stores introduce lag and read-your-write problems.
- Projection messages can be duplicated, reordered, delayed, or poisoned.
- Rebuilds, event/schema evolution, and reconciliation need explicit design.
- Debugging "write succeeded but screen is stale" spans multiple components.
- Teams may duplicate logic incorrectly between command and query sides.

Microsoft's CQRS guidance similarly describes independent optimization/scaling and warns about synchronization and significant added complexity ([Azure CQRS pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/cqrs)).

### 21.3 Strong signals for CQRS

- Write validation/domain rules are complex, while reads are simple but differently shaped.
- Reads vastly dominate writes and need specialized precomputation.
- Several stable query views aggregate multiple service owners.
- Read and write sides need different scaling, security, or storage characteristics.
- Eventual consistency for those reads is acceptable and can be explained to users.

### 21.4 Signals to keep CRUD

- Basic admin CRUD with little domain logic.
- Read and write shapes substantially overlap.
- Traffic does not justify independent scaling.
- Immediate read-after-write from one model is required everywhere.
- The team cannot operate projection lag, replay, and reconciliation.

Before CQRS, try the simpler tool that addresses the measured problem:

- indexes/query tuning;
- cache;
- read replicas;
- a SQL view or materialized view;
- a reporting database for only the expensive reports; or
- a better API/read DTO over the same model.

### 21.5 CQRS decision spine

Use **A-M-S-L-R**:

1. **A - Asymmetry:** what differs between read and write requirements?
2. **M - Models:** what shape and invariant belongs on each side?
3. **S - Synchronization:** same transaction, CDC, outbox/events, or replay?
4. **L - Lag:** what staleness/read-your-write behavior is acceptable?
5. **R - Repair:** how are duplicates, gaps, schema changes, and rebuilds handled?

---

## 22. Keeping the read model correct

### 22.1 Typical update flow

```text
1. PlaceOrder command
2. Validate and commit Order + outbox event
3. Relay publishes OrderPlaced(orderId, version=1)
4. Projector updates order_summary row
5. Query API serves order_summary
```

### 22.2 Projection idempotency

If `OrderPlaced(version=1)` arrives twice, the projection must not create duplicate order cards. Store processed event IDs or enforce entity version rules.

### 22.3 Ordering

```text
OrderPlaced v1
OrderCanceled v2
```

If v2 arrives before v1, blindly applying arrival order can resurrect an order. Partition by order ID where possible, record versions, and reject/buffer stale/gapped updates according to the projection design.

### 22.4 Checkpoints and atomicity

A projector needs to update its view and processing checkpoint safely. If it records the checkpoint before the view, a crash loses the update. If it updates the view then crashes before the checkpoint, it processes again. An atomic local transaction plus idempotent update handles the second case safely.

### 22.5 Poison events

Skipping a malformed event and continuing may leave every later result wrong. A DLQ can prevent total blockage, but track the gap, alert, repair the schema/data/handler, replay it, and verify convergence.

### 22.6 Rebuild and replay

If the authoritative source retains a complete, valid history, a new projection can replay from the beginning. Otherwise rebuild from:

- an authoritative snapshot/backfill query;
- retained change-data-capture history plus a snapshot; or
- another versioned export.

Kafka is a log, but it is not automatically a permanent event store. Retention, compaction, event completeness, schema history, and replay side effects must support the rebuild promise.

### 22.7 Schema evolution

- Version event meaning compatibly.
- Add fields before requiring them.
- Keep projectors tolerant of older events.
- Upcast/translate historical versions during replay when necessary.
- Do not reuse one event name for a different business meaning.
- Test a full projection rebuild, not only live consumption.

### 22.8 Reconciliation

Periodically compare read views with authoritative state using counts, versions, checksums, or sampled business invariants. Projection health is not just "consumer process is running"; measure lag, oldest unprocessed event, gap count, and divergence.

### 22.9 Deletion, privacy, and retention

Copied read models are real data stores. A user deletion or retention policy must propagate to every projection, cache, search index, backup policy, and replay path. Replaying old events must not silently recreate data that policy requires to remain deleted; use deletion/tombstone and retention design appropriate to the legal requirement.

## 23. Read-your-write and user experience

### 23.1 The problem

```text
Client -> CancelOrder -> write DB commits version 8
Client -> GetOrder    -> read model still at version 7: shows CONFIRMED
```

The system is eventually consistent but the experience looks broken.

### 23.2 Options

| Technique | How it helps | Cost/trade-off |
|---|---|---|
| Return command result/version | UI immediately knows accepted state | Response is not the whole query model |
| Optimistic UI with `pending` | Hides short projection lag honestly | Must handle eventual rejection/failure |
| Minimum-version token | Query waits/routes until projection >= version | Extra coordination and timeout behavior |
| Read write source for critical follow-up | Stronger freshness | Adds write-side read load/coupling |
| Session overlay | Merge user's recent accepted changes over stale view | More client/server state complexity |
| Synchronous projection | Immediate view | Couples command latency/availability to view update |

Choose per use case. A public feed may tolerate seconds of lag; a just-submitted payment status may require an explicit pending state or authoritative lookup. Do not call every read "eventually consistent" without a UX contract.

### 23.3 Version-token example

```text
CancelOrder response: { orderId: 123, acceptedVersion: 8 }

GetOrder(123, minimumVersion=8)
  if projection.version >= 8: return it
  else: wait briefly, route to authoritative view, or return 202/pending
```

The fallback must be bounded so one lagging projector does not exhaust request threads.

## 24. CQRS versus Event Sourcing

These are often taught together and therefore confused.

### 24.1 Event Sourcing in one paragraph

**Event Sourcing** stores the sequence of domain state changes as the authoritative record rather than storing only current state.

```text
OrderPlaced -> ItemAdded -> AddressChanged -> OrderConfirmed
                                      |
                                      +--> replay to reconstruct state
```

Snapshots may speed reconstruction, but the event stream remains authoritative. Event sourcing requires durable event semantics, versioning, replay, audit/privacy decisions, and concurrency control.

### 24.2 Relationship table

| Question | CQRS | Event Sourcing |
|---|---|---|
| Core idea | Different models for commands and queries | Events are the authoritative state history |
| Requires event log as source of truth? | No | Yes |
| Can use current-state write tables? | Yes | Usually projections/snapshots derive current state |
| Main motivation | Model/workload asymmetry | Preserve intent/history and reconstruct state |
| Can exist without the other? | Yes | Yes, though often paired |

CQRS without Event Sourcing:

```text
Write DB stores current order row
Outbox publishes changes
Read DB stores order card
```

Event Sourcing with CQRS:

```text
Event store is write source of truth
separate projections serve queries
```

Azure documents the common combination but treats them as separate patterns ([Event Sourcing pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)).

### 24.3 Read replica versus CQRS

A primary and identical read replica separate physical load, which is a lightweight read/write separation. If queries still use the same schema and conceptual model, it does not provide the main modeling benefit people usually mean by CQRS. It may be all the system needs.

### 24.4 Cache versus CQRS read model

- A cache is usually disposable acceleration with eviction/invalidation behavior.
- A CQRS read model is an explicitly maintained query representation with schema, projection logic, checkpoints, and consistency semantics.

A read model can be kept in memory or cached, but calling every cache "CQRS" makes the term useless.

## 25. CQRS failure matrix and common mistakes

| Failure | Symptom | Response |
|---|---|---|
| Write commits, event missing | Read model never changes | Transactional outbox/CDC |
| Duplicate event | Double count/duplicate row | Idempotent projection/event ID |
| Out-of-order event | Old state overwrites new | Per-key order + versions |
| Projector down | Increasing staleness | Durable log, autoscaling, alert on lag |
| Poison event | Partition stalls or view gap | Quarantine, repair, replay, reconcile |
| Read store lost | Queries unavailable | Rebuild/backfill plan and tested restore |
| Schema change | Projector/replay breaks | Compatible versioning/upcasting |
| Lag after command | User sees old state | Pending UX/version token/fresh fallback |
| Replay triggers side effects | Duplicate emails/payments | Projections must be side-effect-safe; isolate integration effects |
| Deletion not propagated | Privacy/retention violation | Tombstones, inventory of projections, audited deletion |

Common mistakes:

- Saying CQRS means "one write API and one read API" with no model difference or reason.
- Using CQRS everywhere instead of within a justified bounded context.
- Claiming separate databases or Event Sourcing are mandatory.
- Updating the write database and broker non-atomically.
- Ignoring projection lag and read-your-write UX.
- Assuming broker order is global or permanent.
- Skipping poison events without repairing the view.
- Duplicating business invariants on the read side and trusting them for commands.
- Creating many projections without ownership, retention, rebuild, and cost controls.

### 25.1 Say-out-loud 30-second CQRS narrative

> "CQRS separates the model that validates and applies business commands from the model optimized for queries. I start with CRUD and use CQRS only when read and write shapes, scale, or security differ enough to justify two models. They may share a database, or a write store can publish changes through an outbox to asynchronously maintained read projections. With separate stores I explicitly design projection idempotency, per-entity ordering, lag, read-your-write behavior, schema evolution, reconciliation, and rebuild. CQRS does not require microservices or Event Sourcing; those are independent choices that can be combined when their own trade-offs fit."

## 26. End-to-end illustrative Amazon-style checkout migration

> **Illustrative product design:** this applies the patterns to familiar Amazon-like behavior. It is not a claim about Amazon's current internal architecture.

### 26.1 Before migration

```text
Web/mobile -> legacy commerce monolith -> commerce DB
                 catalog
                 checkout
                 inventory
                 orders
                 payment integration
```

### 26.2 Strangle the Order History read first

This is a lower-risk seam than payment capture:

```text
                       +-> New Order Query API -> order_summary read DB
Gateway /orders GET ---+
                       +-> Legacy (fallback/unmigrated cohort)

Legacy order changes -> CDC/outbox -> projection -> order_summary
```

Use shadow reads, compare results, then route stable cohorts. CQRS creates the screen-shaped read model.

### 26.3 Transfer order write ownership

Backfill Order DB, catch up changes, stop/drain legacy writes for the owned cohort, and make the new Order Service authoritative. Do not allow both to independently allocate the same order ID or change the same lifecycle.

### 26.4 Introduce checkout saga

```text
Client -> Checkout/Order Saga
            |
            +--> Inventory: Reserve
            |        |
            |        +--> reservation DB
            |
            +--> Payment: Authorize
            |        |
            |        +--> payment DB/provider
            |
            +--> Order: Confirm
                     |
                     +--> order DB + OrderConfirmed outbox
                                      |
                                      v
                              Order summary projection
```

If payment rejects, release inventory and cancel the pending order. If release fails, retain `CANCEL_PENDING`, retry idempotently, reconcile, and alert.

### 26.5 Cutover and retirement

Route by stable customer/order cohort, compare order and financial invariants, maintain a data-aware rollback window, then remove legacy order writes, reads, sync, jobs, tables, credentials, and flags.

| Pattern | Contribution |
|---|---|
| Strangler Fig | Controlled responsibility transfer and retirement |
| Saga | Cross-service checkout state and compensation |
| CQRS | Fast, denormalized order-history view |

## 27. Illustrative real-product examples

These examples use recognizable product behavior to build interview intuition. They are not assertions about current private implementation details.

### 27.1 Twitter/X home timeline - CQRS-shaped problem

```text
PostTweet command -> authoritative Tweet store -> TweetCreated
                                                   |
                                                   v
                                  timeline projection/fan-out workers
                                                   |
                                                   v
User opens app -> Timeline Query API -> precomputed timeline/read store
```

Why CQRS may fit:

- validating/storing a tweet differs from rendering millions of personalized timelines;
- read volume and latency needs differ from the write path; and
- a home timeline can often tolerate bounded propagation lag.

Follow-ups: celebrity fan-out, deleted tweets, block/privacy changes, projection repair, and read-after-write for the author.

### 27.2 YouTube video processing and views - workflow plus projections

```text
Upload command -> video metadata state machine -> transcode jobs -> publish
                                              |
VideoView events -> aggregation -> count/analytics read models
```

Publishing can be a durable forward workflow; not every failure should "undo" the upload. View counters/search/creator dashboards are different projections with different freshness needs. This illustrates why a state machine and CQRS-style views can be more appropriate than forcing every action into one saga rollback.

### 27.3 Uber trip-domain extraction - Strangler-shaped migration

```text
App -> gateway -> stable city/tenant cohort -> New Trip Service
             \-> remaining cohort          -> Legacy Dispatch
```

A safe migration would keep one owner for each trip, route follow-up requests/events by trip ID, shadow read-only calculations, and account for driver callbacks and long-running active trips. Randomly moving each request could split one trip across systems.

### 27.4 Netflix catalog availability - CQRS and saga boundaries

Title rights, regional availability, artwork, and localization may be validated on a write side and projected into region/device-specific catalog views. A publishing workflow may coordinate several checks, but once a title is publicly released, recovery might be "unpublish and correct metadata" rather than pretending release never occurred.

### 27.5 WhatsApp message sending - a warning against pattern forcing

```text
ACCEPTED -> PERSISTED -> DELIVERED -> READ
```

This is naturally a durable message state machine. A classic compensating saga is usually a poor description: if a recipient already read a message, a later delete is a new user-visible action, not rollback. Idempotent message IDs, ordering, retries, acknowledgments, and device synchronization are the central concerns.

### 27.6 Amazon-style checkout - Saga plus CQRS

Inventory reservation, payment authorization, order confirmation, and fulfillment cross business owners, so a saga may fit. The "My Orders" page can use a CQRS read model combining summaries without querying every owner synchronously. Shipment dispatch is a likely irreversible/pivot concern; after it, forward recovery or return/refund is more realistic than rollback.

## 31. Interview answer spines

### 31.1 "Explain the Saga pattern"

Use **I-S-C-R-O**:

1. **I - Invariant:** state the cross-service business rule.
2. **S - Steps:** list local transactions and durable states.
3. **C - Compensation/pivot:** define semantic undo and point of no return.
4. **R - Reliability:** outbox, idempotency, ordering, timeouts, retries.
5. **O - Operations:** trace, reconcile, alert, manual/forward recovery.

Spoken spine:

> "Because each service owns its database, I cannot use one normal local transaction. I model the workflow as local commits with explicit states. Each completed step has a business compensation until the pivot; after the pivot I recover forward. I coordinate by choreography for simple reactions or a durable orchestrator for complex flow. Outbox, idempotency, state-version checks, and reconciliation handle crash/retry behavior, and the user sees pending states honestly."

### 31.2 "Choreography versus orchestration"

Use **F-V-C-R**:

- **F - Flow:** distributed event reactions versus explicit coordinator.
- **V - Visibility:** reconstructed trace versus durable central workflow state.
- **C - Complexity/coupling:** event web versus coordinator contracts.
- **R - Recovery:** distributed timeout ownership versus explicit retry/compensation owner.

Conclude from the workflow; do not use an arbitrary number-of-steps rule.

### 31.3 "What if compensation fails?"

Use **P-R-I-R-E** from Section 11:

> persist incomplete obligation -> durable retry -> idempotent compensation -> reconcile/alert -> manual or forward recovery.

Add: a timeout is unknown; a DLQ does not make the business state correct.

### 31.4 "Shared DB or database per service?"

Use **O-A-Q-O-M**:

> ownership -> atomic invariants -> cross-domain queries -> operational capacity -> migration phase.

State that physical sharing can be safe with private schemas/credentials; shared table ownership is the coupling problem.

### 31.5 "How would you migrate a monolith?"

Use **I-S-D-V-S-D**:

1. **I - Intercept** behind a façade.
2. **S - Seam** by business capability and characterize behavior.
3. **D - Data** backfill/sync with one writer.
4. **V - Verify** by shadowing and invariant comparison.
5. **S - Shift** stable cohorts with gates.
6. **D - Decommission** old routes, data access, jobs, and flags.

### 31.6 "Route x% traffic"

Use **U-S-W-O-R**:

> migration unit -> stable cohort -> authoritative writer -> observe invariants -> data-aware recovery.

Say explicitly that random per-request percentage is generally unsafe for state-changing flows with separate stores.

### 31.7 "Explain CQRS"

Use **A-M-S-L-R**:

> read/write asymmetry -> separate models -> synchronization mechanism -> lag/UX -> rebuild/repair.

End with the simpler alternative you rejected and why the benefit is worth the complexity.

## 32. Follow-up drills with model reasoning

### Drill A: Payment times out after inventory was reserved

Do not label payment failed. Retry `AuthorizePayment` with the same idempotency key or query the provider. If it was authorized, continue; if definitively rejected, release inventory. A reservation TTL can prevent indefinite holding, but expiry must be coordinated with late payment results so a late authorization does not confirm an order whose stock was released.

### Drill B: The compensation event is in a DLQ

The infrastructure isolated the message; the business obligation still exists. Keep the saga non-terminal, alert based on age/severity, repair the consumer/message, replay idempotently, and reconcile Inventory state. A support path may resolve it manually with an audit record.

### Drill C: Why not put the whole saga in the API gateway?

The gateway should handle edge routing, authentication, throttling, and protocol concerns. Checkout workflow is domain/application behavior and belongs in the owning service or a dedicated durable orchestrator. Putting it in the gateway couples edge deployment to domain rules and creates a central business-logic bottleneck.

### Drill D: Can the orchestrator update Order, Inventory, and Payment databases directly?

No. That destroys participant ownership and centralizes schemas. It should send commands and consume replies/facts; each participant validates and commits locally.

### Drill E: Can I roll back a Strangler cutover by changing 20% back to 0%?

Only if the old path can correctly serve data written during the cutover. Before write ownership moves, routing rollback may be easy. After new-only writes, you need reverse synchronization, a reversible migration, or roll-forward. State the data recovery plan before shifting writes.

### Drill F: Why not always use stable user-ID routing?

The business entity may be the real ownership unit. One user can operate on a shared order, merchant account, or trip that other users/workers also access. Route by order/tenant/trip ownership where possible, and make background/event routing use the same rule.

### Drill G: Does CQRS make reads strongly consistent?

No. Separate models can be updated synchronously in one database or asynchronously across stores. Consistency follows the synchronization/read path. With async projections, explicitly define lag and read-your-write behavior.

### Drill H: Why not use Event Sourcing for every CQRS system?

Event Sourcing adds an authoritative immutable history, replay/versioning, audit/privacy, and event-model obligations. If current-state writes plus an outbox can build the required read models, Event Sourcing may add cost without sufficient benefit.

### Drill I: Both services need the same customer address

Customer owns the canonical profile address. Order may copy a shipping-address snapshot accepted at purchase because historical fulfillment must not change when the profile changes. This is deliberate domain data, not an accidental cache. Define which changes propagate and which snapshots are immutable.

### Drill J: Can a CQRS read model be trusted to authorize a command?

Not if it can be stale relative to the invariant. Use the authoritative command-side state or a versioned guarantee. A read projection can help screen obvious requests, but the command side must enforce the final rule.

## 33. Pattern selection table

| Symptom | First candidate | Important warning |
|---|---|---|
| Risky full legacy rewrite | Strangler Fig | Data ownership is harder than request routing |

## 34. Commonly confused statements corrected

| Weak/incorrect statement | Interview-ready correction |
|---|---|
| "Each microservice needs its own server/database instance" | "Each service needs exclusive logical data ownership; physical infrastructure can be shared with enforced schemas/credentials" |
| "Saga rolls back every database" | "Saga commits local steps and later commits semantic compensations" |
| "Kafka gives exactly once" | "Broker guarantees have scope; business effects still need idempotency and atomic local state/message handling" |
| "Timeout means the step failed" | "Timeout means the caller does not know the outcome" |
| "Orchestrator is always a single point of failure" | "Its role is logically central; durable state and replicated workers remove a single runtime process" |
| "DLQ handles failed rollback" | "DLQ isolates delivery failure; the saga remains incomplete until repaired/reconciled" |
| "Strangler means send some traffic to new code" | "It progressively transfers and retires a capability, including data and hidden dependencies" |
| "Set traffic back to 0% to roll back" | "Routing rollback works only if old state remains valid" |
| "Dual write keeps databases consistent" | "Two independent writes have a partial-failure gap; prefer one writer plus outbox/CDC and reconciliation" |
| "CQRS needs separate databases" | "CQRS needs separate read/write models; storage separation is optional" |
| "CQRS is Event Sourcing" | "They are independent patterns often combined" |
| "Read model is the source of truth" | "The read model is a derived query view unless explicitly designed otherwise" |

## 37. Final say-out-loud 30-second narrative

> "These patterns solve different distributed-system problems. Strangler Fig reduces migration risk by intercepting traffic and progressively transferring a capability, with stable cohorts and one data writer until the legacy path is truly retired. Database-per-service then improves ownership but removes easy cross-service transactions and joins. A saga coordinates cross-service writes through local commits, explicit states, semantic compensation, and durable recovery; if compensation fails, the obligation remains open and is retried, reconciled, or escalated. CQRS separates a correctness-focused command model from query-optimized views, but earns its complexity only when read/write needs differ and lag, replay, and repair are deliberately handled."
