# Distributed Key-Value Store

> Beginner-first HLD notes expanded from: scalability, decentralization, eventual consistency, partitioning, consistent-hash replication, preference lists, `R + W > N`, load-balancer/routing types, `GET`/`PUT`, vector clocks, conflict resolution, gossip, and Merkle trees.
>
> The concrete design in this chapter is a **Dynamo-style, leaderless, eventually consistent key-value store**. That is one valid point in the design space—not the definition of every key-value store.

## Scope, assumptions, and evidence labels

I assume the rough bullet **“LB Types”** refers to the ways a client request can be routed to the correct storage node: a generic load balancer, a partition-aware client, or a smart proxy/coordinator. I also briefly distinguish Layer 4 and Layer 7 load balancers. If the lecture meant something else by “LB,” the routing section is still a required part of this design.

The original 2007 Amazon Dynamo paper is the main reference because the bullet sequence closely matches that design. **Amazon Dynamo and Amazon DynamoDB are not the same system.** Dynamo was an internal system described in a paper; DynamoDB is a later managed AWS product with different and evolving capabilities.

Evidence labels used in this chapter:

- **Verified concept** — supported by a paper or first-party documentation linked nearby.
- **Source-era production example** — a company publicly described the design at the linked publication date; its current private architecture may differ.
- **Illustrative product mapping** — shows how the idea could apply to a familiar product; it is not a claim about that product’s private backend.
- **Self-reported interview experience** — an anonymous candidate or user reported the prompt. It is useful practice evidence, not an official or independently verified question bank.
- **Illustrative number** — chosen to make the mechanics concrete, not claimed production traffic.

### The most important corrections up front

1. **A key-value store is a data model, not one consistency model.** Redis-like caches, Dynamo-style AP stores, and Raft-backed strongly consistent stores can all expose key-value APIs.
2. **Consistent hashing partitions keys; it does not replicate them.** Replication is a second placement step that selects several distinct physical nodes.
3. **The preference list is per key/token range.** It is the ordered set of nodes preferred to store that key, not a global list of the “best” servers.
4. **`R + W > N` means quorum intersection under specific assumptions.** It does not alone guarantee linearizability, prevent concurrent writes, repair stale replicas, or even guarantee overlap when sloppy quorums write to fallback nodes outside the normal replica set.
5. **Vector clocks detect causal order versus concurrency; they do not automatically merge business meaning.** The application still needs a conflict policy.
6. **Gossip is not consensus and normally is not the user-data replication path.** It spreads membership, liveness, and topology knowledge.
7. **A Merkle tree detects which ranges differ; it does not choose the correct value.** Version comparison and reconciliation still decide what to copy or merge.
8. **Eventual consistency does not mean “consistent after five seconds.”** It means replicas converge if updates stop, communication resumes, and the repair/reconciliation mechanisms keep making progress.

## 1. Plain-language foundation

### 1.1 A key-value store is a giant dictionary

In ordinary code, a map/dictionary lets us associate a unique key with a value:

```text
"user:42"        -> { name: "Asha", plan: "premium" }
"video:abc123"   -> { title: "Hashing", durationSec: 740 }
"session:9f..."  -> { userId: 42, expiresAt: ... }
```

The basic operations are:

```text
PUT(key, value)    store or replace the value under key
GET(key)           retrieve the value under key
DELETE(key)        make the key absent
```

The key-value store usually treats the value as an opaque byte sequence. It may not understand that the bytes represent a name, cart, image metadata, or serialized JSON. The application owns that meaning.

That narrow interface is valuable because a server can answer an exact-key lookup without parsing a SQL query, joining tables, or searching every record.

### 1.2 One-machine version

The smallest useful implementation could be:

```text
Client ---> one server ---> in-memory hash map
                         \
                          +--> append-only disk log
```

- The hash map makes exact-key lookup fast.
- The append-only log makes acknowledged data recoverable after restart.
- On startup, the server replays the log to rebuild memory.

This is enough for a small dataset and one server’s throughput. It is not enough when:

- the data no longer fits on one machine;
- one machine cannot serve the read/write rate;
- the machine, disk, rack, or availability zone can fail;
- users are geographically far from that one copy.

### 1.3 Distributed version

A distributed key-value store spreads data and copies across many machines:

```text
                    key determines owners
Client ---> Router / Coordinator -----------------------------+
             |                                                 |
             +--> Node A: some keys + local storage            |
             +--> Node B: other keys + replicas                |
             +--> Node C: other keys + replicas                |
             +--> Node D: other keys + replicas <--------------+
```

Now a `PUT` is no longer just “write into a map.” The system must answer:

- Which nodes own this key?
- How many copies should exist?
- How many copies must acknowledge before success?
- What if one owner is down?
- What if two disconnected sides accept different values?
- How does an old replica catch up?
- How does every node learn that servers joined or left?

The rest of the chapter is the answer to those questions.

### 1.4 Simple API does not mean simple internals

The interface can be three functions while the implementation handles:

- partitioning and rebalancing;
- replication and failure domains;
- durable local storage;
- request routing;
- timeouts and ambiguous retries;
- consistency guarantees;
- concurrent versions and reconciliation;
- node membership and failure detection;
- background repair, backups, monitoring, and overload.

This is exactly why “design a distributed key-value store” is a strong infrastructure interview prompt.

### 1.5 Key-value store, cache, and relational database are different axes

| Term | What it says | What it does **not** say |
|---|---|---|
| Key-value store | Records are primarily addressed by key | Whether data is in RAM/disk, strong/eventual, durable/evictable |
| Cache | Data is a disposable or reconstructable acceleration copy | It may still expose a key-value interface |
| Relational database | Tables, relations, constraints, and query model | It may internally use key-value indexes |
| Document store | Value has queryable document structure | It may also use a partition key |
| Ordered key-value store | Keys have an order and often support range scans | It need not expose SQL |

Do not say “key-value means in-memory” or “key-value means eventually consistent.” Neither is generally true.

## 2. Inline prerequisites

These concepts are required to understand the design, so they are explained here instead of deferred.

### 2.1 Node, shard, partition, replica, and coordinator

- A **node** is a participating server/process.
- A **partition/shard** is a subset of the logical dataset. In this chapter, a hash-token range is a partition.
- A **replica** is one copy of a partition/key on a node.
- The **replication factor `N`** is the intended number of copies of each key.
- A **coordinator** handles one client operation: it locates replicas, sends requests, waits for enough responses, and reconciles results. In a leaderless system this is a temporary role, not a permanent leader.

One physical node usually owns many token ranges and therefore stores many primary and replica partitions.

### 2.2 Failure domains

Three copies are useful only if their failures are sufficiently independent:

```text
Bad placement:                     Better placement:

Rack 1                             Zone A   Zone B   Zone C
  Node A  copy 1                   copy 1   copy 2   copy 3
  Node B  copy 2
  Node C  copy 3
```

The first design survives one disk but not a rack power failure. Replica placement should understand host, rack, zone, and sometimes region.

### 2.3 Latency, throughput, availability, and durability

- **Latency:** time for one request, normally discussed with percentiles such as p50/p95/p99.
- **Throughput:** work completed per second, such as reads/s or bytes/s.
- **Availability:** whether requests can complete successfully under stated failures.
- **Durability:** whether an acknowledged write survives later failures.

They are not interchangeable. A `PUT` can be low-latency but not durable if success means “stored only in RAM.” A stale `GET` can be available but not strongly consistent.

### 2.4 Consistency models needed here

- **Linearizability:** each operation appears to take effect atomically between invocation and response, respecting real-time order.
- **Causal consistency:** causes are observed before their effects; unrelated concurrent operations need not share one global order.
- **Read-your-writes:** a client/session sees its own completed updates.
- **Monotonic reads:** after seeing version 7, the same session does not later fall back to version 6.
- **Eventual consistency:** if updates stop and communication/repair continue, replicas converge.

Eventual consistency alone does not guarantee read-your-writes, monotonic reads, a maximum staleness time, or a single conflict-free value during convergence.

### 2.5 Hash and consistent hashing

A hash function maps a key to a fixed-size number:

```text
hash("user:42") -> token 63
```

Naive partitioning might use `hash(key) mod nodeCount`. Changing the node count changes most answers, causing huge data movement.

**Consistent hashing** places both tokens and node ownership on a logical ring. Adding/removing a node changes only nearby token ranges rather than remapping nearly everything. For deeper coverage, see [06. Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md); the mechanics needed today are repeated in Section 6.

### 2.6 Wall-clock time versus logical causality

- A wall clock says “it is 10:15:03.” Machine clocks can be skewed or move backward.
- A logical clock says “this event happened after that event” without pretending to know perfect physical time.

Vector clocks are logical clocks. They are used because concurrent distributed writes may not have a trustworthy universal timestamp order.

### 2.7 Quorum versus consensus

A **quorum** is simply a sufficiently sized subset of replicas used for an operation. Quorum intersection is a set-math property.

**Consensus** is a protocol such as Raft/Paxos that makes nodes agree on one ordered decision despite failures.

Waiting for two of three independent Dynamo replicas is not automatically the same as running Raft among three replicas. This distinction prevents many incorrect interview answers.

## 3. Requirements and design position

### 3.1 Functional requirements

Core scope:

- `PUT(key, value, context?)`
- `GET(key)`
- `DELETE(key, context?)`
- optional per-request consistency setting;
- values small enough to replicate as units;
- exact-key access is the main query.

Explicitly out of the core design:

- arbitrary joins and predicates;
- cross-key ACID transactions;
- full-text search;
- large blob streaming;
- globally ordered scans.

These can be added, but each changes the architecture and interview scope.

### 3.2 Non-functional goals from the rough bullets

#### Scalability

- Add nodes to increase aggregate storage and throughput.
- Avoid moving most data when membership changes.
- Spread work across many nodes.
- Prevent one global data-path leader from limiting the whole cluster.

#### Decentralization

- Any reachable service node can coordinate a request.
- Nodes share membership/topology information without one required registry in the request path.
- No permanent primary owns all writes for the cluster.

Decentralization is a means, not a sacred requirement. A consensus-backed metadata/control plane can simplify safety while the data plane remains sharded. “No master anywhere” can make operations harder without improving user-visible availability.

#### Eventual consistency and high availability

- Continue serving some reads/writes during node or network failures.
- Permit temporary replica divergence.
- Detect causally ordered versus concurrent versions.
- Converge through hinted handoff, read repair, and anti-entropy.

### 3.3 Assumed service targets

These are qualitative because the rough bullets contain no workload numbers:

- low-latency exact-key reads and writes;
- data larger than one machine;
- node and rack/zone failure tolerance;
- durability via several failure-independent copies;
- availability favored over single-value consistency during a partition;
- no silent loss of concurrent writes: return/merge conflicts rather than arbitrarily erase one.

In an interview, ask for value size, request rate, read/write ratio, key distribution, regions, retention, durability, consistency, and whether multi-key transactions are required before choosing the final architecture.

## 4. From a single-node store to a distributed store

### 4.1 Local storage engine: the part each node still needs

Distribution does not remove local persistence. A common write-optimized node uses an LSM-style path:

```text
PUT
 |
 +--> append commit/WAL record to durable storage
 |
 +--> update in-memory memtable
 |
 +--> acknowledge at configured durability point

Later:
memtable --> immutable sorted table (SSTable) --> background compaction
```

A `GET` checks memory and relevant disk tables, often using Bloom filters and indexes to avoid unnecessary disk reads.

Important local trade-offs:

| Choice | Benefit | Cost |
|---|---|---|
| Append/WAL | Sequential, recoverable write path | Replay and log management |
| Memtable | Fast recent reads/writes | Needs memory and flush policy |
| SSTables/LSM | High write throughput | Read amplification and compaction |
| B-tree | Strong point/range read behavior | Random/in-place write cost and page management |
| Compaction | Reclaims old versions/tombstones, improves reads | CPU, disk I/O, temporary space, write amplification |

The exact engine is replaceable. Apache Cassandra explicitly describes Dynamo-style clustering with an LSM local storage engine in its [architecture documentation](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html).

### 4.2 Why one bigger server eventually stops being enough

Vertical scaling is often the right first step, but has ceilings:

- finite storage/IOPS/network/CPU;
- larger blast radius;
- expensive high-end hardware;
- maintenance and failure affect the only copy;
- geographic distance remains.

Horizontal scaling partitions different keys across nodes. Replication then restores redundancy for each partition.

### 4.3 High-level component view

```text
                              +--------------------------+
Client / SDK ---------------->| routing / coordinator    |
                              | hash key, find replicas  |
                              +------------+-------------+
                                           |
                    +----------------------+----------------------+
                    |                      |                      |
                    v                      v                      v
              +-----------+          +-----------+          +-----------+
              | Storage A |          | Storage B |          | Storage C |
              | WAL + mem |          | WAL + mem |          | WAL + mem |
              | SSTables  |          | SSTables  |          | SSTables  |
              +-----------+          +-----------+          +-----------+
                    ^                      ^                      ^
                    +---- repair / hints / replica exchange ----+

Every node also participates in membership/failure knowledge (for example, gossip).
```

In a fully symmetric implementation, a storage node can also be the coordinator. The boxes are roles, not necessarily separate fleets.

## 5. The complete design at a glance

```text
1. PARTITION
   key --hash--> token on ring --> primary token owner

2. REPLICATE
   walk ring / topology policy --> ordered preference list of N distinct nodes

3. ROUTE
   client-aware SDK, smart proxy, or any-node coordinator

4. PUT
   coordinator versions write --> sends to preferred reachable replicas
   --> succeeds after W durable acknowledgements
   --> fallback nodes keep hints when normal owners are unavailable

5. GET
   coordinator asks replicas --> waits for R responses
   --> compares version metadata --> returns one value or concurrent siblings
   --> repairs stale copies in background

6. MEMBERSHIP
   nodes disseminate topology/heartbeat state through gossip-like exchanges

7. HEAL
   hinted handoff for temporary misses
   + read repair for keys that are read
   + Merkle-tree anti-entropy for cold or silently divergent ranges
```

The original design lineage is documented in the 2007 paper [“Dynamo: Amazon’s Highly Available Key-value Store”](https://www.cs.cmu.edu/afs/cs/academic/class/15712-f08/www/readings/Vogels07.pdf). Apache Cassandra and Riak are useful open documentation examples of related mechanisms, but their exact implementations differ.

## 6. Partitioning with consistent hashing

### 6.1 The hash ring mental model

Imagine the hash output wraps around in a circle:

```text
                         token 0
                            |
                   Node A   |    Node B
                      *-----+-----*
                    /               \
                   /                 \
            Node D *                 * Node C
                   \                 /
                    \               /
                      *-----------*
                       higher tokens
```

For a key:

1. Compute `t = hash(key)`.
2. Move clockwise to the first token owner.
3. That owner is the first preferred replica.

Example:

```text
hash("cart:user-42") = token 63

clockwise owners after 63:
Node C, Node D, Node A, Node B, ...
```

With replication factor `N = 3`, the normal replicas might be C, D, and A.

### 6.2 Virtual nodes

One physical node owning one large contiguous range can create imbalance. Instead, each physical node owns many small logical positions called **virtual nodes** or **vnodes**:

```text
Ring positions:
A1, C1, B1, A2, D1, C2, A3, B2, D2, ...

Physical Node A owns: A1, A2, A3, ...
```

Benefits:

- adding one server takes small ranges from many servers;
- removing one server spreads replacement load across many survivors;
- heterogeneous servers can receive different numbers/weights of ranges;
- load and data tend to balance more smoothly.

Costs:

- more placement metadata;
- more replica-neighbor relationships;
- repairs and range movement become more fragmented;
- too many vnodes can increase operational work.

### 6.3 Adding a node

```text
Before:                  After Node E joins:

A owns (D, A]            E takes selected subranges from A/B/C/D
B owns (A, B]            only those ranges stream to E
C owns (B, C]            most other keys do not move
D owns (C, D]
```

Safe rebalancing needs more than updating a ring diagram:

1. Decide the new ownership epoch/topology.
2. Stream/copy the affected ranges while old owners continue serving.
3. Handle writes occurring during the copy.
4. Verify the new replica set is complete.
5. Atomically or carefully switch routing ownership.
6. Delete old copies only after a safe grace period.

Otherwise a membership change can lose writes or briefly create missing ranges.

### 6.4 Partitioning failure: hot keys and skew

Uniform hashing balances many independent keys. It does **not** split one extremely hot key:

```text
millions of GET("live-score:final")
                    |
                    v
             same replica set
```

Mitigations:

- CDN/edge or application cache for read-hot immutable values;
- replicate/cache hot values more widely;
- request coalescing so one miss triggers one backend read;
- isolate heavy tenants;
- split a write-hot logical key into buckets and aggregate reads;
- apply backpressure/rate limits.

Key splitting changes semantics and is harder for operations that require an exact atomic total.

### 6.5 Partition key design remains an application decision

Even a managed system cannot save a low-cardinality or monotonically hot partition key from all problems. AWS documentation explains that DynamoDB hashes the partition key to determine placement and recommends high-cardinality, uniformly active keys in its [partition documentation](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/HowItWorks.Partitions.html).

That is a current DynamoDB product fact, not proof that DynamoDB internally uses the exact 2007 Dynamo ring.

## 7. Replication, preference lists, and quorums

### 7.1 `N`: intended number of replicas

Let:

```text
N = number of normal replicas for a key
W = number of write acknowledgements required for PUT success
R = number of replica responses required for GET success
```

With `N = 3`, each key should normally live on three distinct physical nodes, preferably across independent failure domains.

More replicas can improve failure tolerance and read capacity, but cost:

- `N` times logical storage before compression/dedup effects;
- more network traffic per write;
- more repair work;
- more possible stale/divergent copies;
- larger write quorum if stronger overlap is desired.

### 7.2 Preference list

The **preference list** for a key is an ordered list of nodes that should store it.

```text
hash(key) -> token 63

ring walk: C1 -> D1 -> C2 -> A1 -> B1 -> ...
physical nodes: C    D     C     A     B

effective distinct-node preference list:
[C, D, A, B, ...]

for N = 3, normal replica set = [C, D, A]
```

We skipped `C2` because C was already selected. Multiple vnodes on one physical machine do not provide independent copies.

A topology-aware list also skips/selects candidates to spread replicas across racks/zones.

The list is often longer than `N`. The first `N` are the normal owners; later nodes are fallback candidates during failure.

### 7.3 Strict quorum versus sloppy quorum

#### Strict/fixed quorum

Only the normal `N` replicas count. If too few are reachable, the operation fails or waits.

```text
normal replicas = [A, B, C]
A down, W = 3  -> PUT cannot succeed
```

This preserves the fixed replica-set assumptions but reduces availability.

#### Sloppy quorum

Walk farther down the preference list and temporarily store the write on reachable fallback nodes:

```text
normal replicas: A, B, C
A unavailable

write to: B, C, D(fallback holding a hint for A)
```

When A returns, D hands the hinted value to A and may remove the temporary copy.

Sloppy quorum improves availability, but “three acknowledgements” may not come from the same three nodes a later read contacts. That weakens the simple quorum-overlap argument.

### 7.4 Hinted handoff

A fallback replica stores:

```text
value + version metadata + "intended for Node A"
```

Later:

```text
Node A returns
     ^
     | replay hint
Node D ----------------> A
```

Hints are a temporary repair optimization, not the sole correctness mechanism:

- the hint holder can fail;
- hints can expire or be lost;
- a node may be absent longer than the hint window;
- silent corruption is not necessarily represented by a hint.

Apache Cassandra documents hints as one of several convergence mechanisms alongside read repair and anti-entropy repair in its [hint documentation](https://cassandra.apache.org/doc/stable/cassandra/managing/operating/hints.html).

### 7.5 The `R + W > N` overlap rule

Assume one fixed replica set of size `N`.

If a successful write touched any `W` replicas and a later read contacts any `R` replicas, then:

```text
R + W > N
```

forces the two sets to overlap in at least one node.

Example with `N = 3`, `W = 2`, `R = 2`:

```text
write acknowledgers = {A, B}
possible read set    = {B, C}
intersection        = {B}
```

No two subsets of sizes two and two can be disjoint inside a three-node universe.

### 7.6 What quorum overlap actually buys

With the needed assumptions, overlap means at least one read response can contain the acknowledged version. The coordinator must still:

- request the right replica set;
- receive version metadata, not just an arbitrary value;
- compare all `R` responses correctly;
- prefer a causally newer version over an older one;
- handle concurrent incomparable versions;
- repair stale replicas eventually.

Apache Cassandra describes `W + R > RF` as a common way to make acknowledged writes visible to later reads within its consistency model in its [Dynamo architecture documentation](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html#tunable-consistency).

### 7.7 Why `R + W > N` is **not** a complete strong-consistency proof

The slogan is frequently overclaimed. It does not alone guarantee linearizability because:

1. **Sloppy quorums can use different node universes.** A write may succeed on fallback nodes that a read does not query.
2. **Concurrent writes can both succeed.** Set intersection does not serialize them.
3. **Version resolution may be unsafe.** A wall-clock last-write-wins rule can select the wrong write under clock skew.
4. **A timeout is ambiguous.** The client may see failure even though enough replicas eventually stored the write.
5. **Topology can change.** Stale placement views can make coordinators contact different replica sets.
6. **Durability of an acknowledgement is implementation-defined.** RAM acknowledgement is not an fsynced copy.
7. **Reading `R` responses is not enough if the coordinator returns the first response without reconciliation.**

Quorum intersection is a useful ingredient, not a substitute for a specified consistency protocol.

### 7.8 Should `W > N/2` too?

`W > N/2` ensures any two write acknowledgement sets in the same fixed `N` replicas overlap. That can matter in some quorum-register algorithms.

In a Dynamo-style store, it still does not make writes mutually exclusive or ordered. A shared replica may accept both concurrent versions. Vector clocks then expose the fork instead of preventing it.

### 7.9 Common `N/R/W` choices

For illustrative `N = 3`:

| `W` | `R` | What it favors | Important consequence |
|---:|---:|---|---|
| 1 | 1 | Lowest coordination latency/availability | No guaranteed overlap; stale reads likely under lag |
| 1 | 3 | Fast writes, expensive reads | Overlap in fixed set; read needs every normal replica |
| 3 | 1 | Expensive writes, fast reads | Fixed-set overlap; write fails if any required replica unavailable |
| 2 | 2 | Balanced quorum | Fixed-set overlap; tolerates one unavailable normal replica for each operation |
| 3 | 3 | Maximum fixed-set participation | Highest latency and lowest availability; still needs conflict semantics |

These are mechanics, not universal recommendations. Multi-region systems often use local and global quorums with different latency/consistency properties.

### 7.10 Availability calculation intuition

Ignoring correlated failures, an operation can complete when at least `W` write replicas or `R` read replicas are reachable. Increasing `R` or `W` generally:

- increases the chance of seeing newer/durable state;
- increases latency because more responses are required;
- decreases availability because more replicas must respond;
- increases exposure to tail latency.

Hedged/speculative reads can reduce tail latency but add load. Retry storms can turn one slow replica into a cluster-wide overload, so budgets and backpressure matter.

## 8. Routing and “LB types”

The system must route a key to a coordinator and ultimately to its replicas.

### 8.1 Option A: generic load balancer to any node

```text
Client --> L4/L7 Load Balancer --> any healthy Node B
                                      |
                                      +-- B hashes key and coordinates A/C/D
```

Benefits:

- simple clients;
- centralized health checks, TLS, authentication, and rate limiting;
- nodes remain symmetric.

Costs:

- chosen coordinator may not own the key, adding an internal hop;
- load balancer can hide topology/locality;
- the LB tier needs high availability and capacity.

#### Layer 4 versus Layer 7

- **L4** routes TCP/UDP connections using network/transport information. It is protocol-agnostic and low overhead but usually cannot inspect the key in an application request.
- **L7** understands HTTP/gRPC/application fields. It can authenticate, rate-limit, or route by tenant/key if the protocol exposes it, at the cost of more processing and configuration.

Neither automatically knows consistent-hash ownership unless that logic is added.

### 8.2 Option B: token-aware client library

```text
Client SDK has topology map
hash(key) -> preferred replicas
Client -------------------------> Node C (owner/coordinator)
```

Benefits:

- often one fewer network hop;
- client can prefer local rack/zone;
- load can spread directly across replicas.

Costs:

- every language SDK needs routing logic;
- topology updates must reach clients safely;
- stale clients can misroute requests;
- authentication, retries, and observability become distributed across clients.

A contacted wrong node should redirect/forward or return fresh topology rather than corrupt data.

### 8.3 Option C: smart stateless proxy/coordinator tier

```text
Client --> smart proxy fleet --> exact replica set
```

Benefits:

- thin clients;
- centralized routing policy and observability;
- proxies can be independently scaled and updated.

Costs:

- extra hop;
- proxy fleet becomes a critical dependency;
- topology cache correctness and overload must be managed.

### 8.4 Practical hybrid

A common design combines:

- DNS/L4 load balancing across regional endpoints;
- an L7 gateway for auth/quotas;
- token-aware coordinators or SDKs inside the trusted network.

The interview answer should say **who knows the ring, who performs retries, and what happens when that view is stale**. Merely drawing “Load Balancer” does not answer routing.

## 9. `PUT` operation in depth

Assume `N = 3`, `W = 2`, normal replicas `[A, B, C]`.

### 9.1 Happy path

```text
Client             Coordinator B              A          B          C
  | PUT(k,v,ctx)         |                     |          |          |
  |--------------------->|                     |          |          |
  |                      | hash(k), preference list       |          |
  |                      | create next version            |          |
  |                      |---- write(k,v,ver) ------------>|          |
  |                      |---------- write --------------->|          |
  |                      |---------------- write ------------------->|
  |                      |<--------- durable ack ----------|          |
  |                      |<---------------- durable ack ---------------|
  |<------ success ------|       W = 2 reached                        |
```

Step by step:

1. Client sends key, serialized value, optional causal context, request ID, and perhaps a consistency/durability level.
2. Coordinator validates limits and authorization.
3. It hashes the key and obtains the preference list.
4. It creates a new version from the supplied context.
5. It sends the mutation to normal replicas, often all `N` even if it waits for only `W`.
6. Each replica appends to its WAL/commit log, updates memory, and acknowledges at the configured durability point.
7. Coordinator returns success after `W` qualifying acknowledgements.
8. Slow replicas continue asynchronously if the implementation permits.

### 9.2 What does “acknowledgement” mean?

The design must define it:

| Ack point | Latency | Failure risk |
|---|---|---|
| Accepted in process memory | Lowest | Process/power loss can erase it |
| Written to OS page cache | Low | Machine/power failure may erase it |
| WAL fsynced on one disk | Higher | Survives process restart; disk/host can still fail |
| Durably stored on `W` failure-independent replicas | Higher | Better host/zone durability |

Saying “`W = 2` is durable” is incomplete until the acknowledgement point and failure domains are named.

### 9.3 One normal replica is unavailable

```text
normal: [A, B, C], A unavailable
fallback preference: D

B --> B stores normal copy
B --> C stores normal copy
B --> D stores copy + hint "deliver to A"

W = 2 can succeed after B and C; D improves temporary redundancy.
```

If only B and D are reachable, a sloppy quorum might still acknowledge `W = 2`. Availability improves, but a strict read from `[A,B,C]` must include B or it may miss that acknowledged version.

### 9.4 Timeout and retry ambiguity

```text
Client PUT request-id=77
       |
       +--> replicas store value
       X response lost / client times out
```

The client cannot infer that the write failed. Retrying blindly can:

- create another causal version;
- repeat a non-idempotent application update;
- race with another writer.

Mitigations:

- idempotency key/request ID with a bounded deduplication record;
- conditional write using expected version/context;
- make `PUT(key, completeValue)` overwrite semantics explicit;
- return version tokens and support safe retry/read-back;
- distinguish transport timeout from known rejection.

Exactly-once execution is not created by a retry flag; it requires durable deduplication or idempotent semantics.

### 9.5 Conditional updates

A plain leaderless `GET` then `PUT` is subject to races:

```text
Alice GET balance=10     Bob GET balance=10
Alice PUT balance=7      Bob PUT balance=4
```

Vector clocks can detect concurrent descendants, but they do not enforce `balance >= 0` or atomically serialize decrements.

For hard invariants, use:

- compare-and-set with a strong per-key protocol;
- single leader/consensus per shard;
- escrow/reservation techniques;
- an application workflow that can compensate.

## 10. `GET` operation in depth

Assume `N = 3`, `R = 2`, replicas `[A, B, C]`.

### 10.1 Happy path with one latest version

```text
Client          Coordinator B            A               B               C
  | GET(k)           |                    |               |               |
  |----------------->|                    |               |               |
  |                   |--- read(k) ------>|               |               |
  |                   |------ read(k) ------------------->|               |
  |                   |----------- optional read ------------------------>|
  |                   |<-- v5 ----------|                                |
  |                   |<----- v4 ------------------------|                |
  |                   | compare versions: v5 dominates v4                |
  |<---- value v5 ----|                                                   |
  |                   |---- async repair v5 to stale B -----------------> |
```

Step by step:

1. Coordinator hashes the key and finds preferred replicas.
2. It sends enough requests to obtain `R` responses; it may contact extras to reduce tail latency.
3. Responses contain value plus version/tombstone metadata.
4. Coordinator compares versions.
5. If one version causally dominates all others, return it.
6. If versions are concurrent, merge automatically or return siblings/context according to policy.
7. Repair stale responders asynchronously or before response, depending on latency/correctness policy.

### 10.2 Digest reads

To reduce bandwidth, a coordinator can request:

- one full value;
- digests/version metadata from other replicas.

If digests match, return the full value. If they differ, fetch full conflicting values and reconcile. This helps when values are larger than version metadata, but complicates the read path.

### 10.3 Read repair

When a `GET` discovers an old/missing replica, it sends the newer/resolved version back.

Benefit: frequently read (“hot”) keys heal naturally.

Limitation: cold keys may never be read, so read repair cannot be the only repair method.

### 10.4 What a `GET` can return

Depending on the API and policy:

```text
not found
one resolved value + version context
several concurrent sibling values + contexts
timeout / insufficient replicas
```

“Not found” is itself versioned state when deletes exist. Otherwise an old value can be mistaken for live data and resurrected.

### 10.5 Session guarantees are not automatic

A user writes through A/B and then reads from C. With low `R/W`, C may be stale.

Options for read-your-writes:

- carry the returned version and require a read at least that new;
- route a session to replicas known to have acknowledged its write;
- temporarily read at stronger consistency;
- have a coordinator wait/fetch until the version is visible;
- use a strong-consistency mode for that operation.

Do not relabel ordinary eventual consistency as read-your-writes without one of these mechanisms.

## 11. Data versioning and vector clocks

### 11.1 Why a single “version number” is insufficient

On one leader, version 8 follows version 7. In a leaderless partition, two nodes can both independently create what they think is the next version.

```text
starting cart = {book}

network partition

Region A adds pen              Region B adds mug
cart = {book, pen}             cart = {book, mug}
```

Neither write necessarily happened after the other. Calling one “version 2” and the other “version 2” detects equality, not causal relationship. Calling them version 2 and 3 based on arrival order may falsely imply causality.

### 11.2 Vector-clock structure

A vector clock is a map from participant identity to logical counter:

```text
{A: 2, B: 1, C: 0}
```

Interpretation: this version’s history includes two tracked events from A and one from B.

The identity may represent a coordinator/node or another chosen actor. The choice affects metadata size and semantics.

### 11.3 Creating and updating a vector clock

Start with no history:

```text
V0 = {}
```

Write through A:

```text
V1 = {A: 1}
```

A client reads V1, changes the value, writes through A:

```text
V2 = {A: 2}
```

`V2` is a descendant of `V1`.

### 11.4 Comparing two vector clocks

Vector `X` dominates `Y` when:

- every counter in X is greater than or equal to the corresponding counter in Y; and
- at least one is strictly greater.

Treat missing entries as zero.

Examples:

```text
{A:2, B:1} dominates {A:1, B:1}

{A:2, B:0} and {A:1, B:1} are concurrent:
first is ahead on A, second is ahead on B
```

Three outcomes:

| Comparison | Meaning | Normal action |
|---|---|---|
| Equal | Same causal history | Values should agree; investigate if bytes differ |
| One dominates | Older version is an ancestor | Keep descendant, repair ancestor |
| Neither dominates | Concurrent branches | Preserve/merge/reject according to conflict policy |

### 11.5 Complete concurrent-write example

Initial cart:

```text
value = {book}
clock = {A:1}
```

Two clients both read that version. During a partition:

```text
Client 1 via B:                  Client 2 via C:
{book, pen}                      {book, mug}
{A:1, B:1}                      {A:1, C:1}
```

Comparison:

- left is newer on B;
- right is newer on C;
- neither dominates;
- they are siblings, not “old versus new.”

Application merges both:

```text
{book, pen, mug}
```

Merge the causal contexts by per-entry maximum, then increment the new coordinator A:

```text
max({A:1,B:1}, {A:1,C:1}) = {A:1,B:1,C:1}
new resolved write via A          = {A:2,B:1,C:1}
```

The resolved version now dominates both siblings.

### 11.6 Why the client must return context

If the client sends only a new value without the version it read, the store cannot know whether the write:

- descends from an existing version;
- is based on stale state;
- is a blind overwrite;
- resolved several siblings.

A Dynamo-style API therefore returns causal context with `GET` and expects it back on `PUT`. Riak’s documentation similarly explains causal context, siblings, and dotted version vectors in its [causal-context guide](https://docs.riak.com/riak/kv/latest/learn/concepts/causal-context/index.html).

### 11.7 Vector-clock limitations

- Metadata grows as more identities participate.
- Pruning old entries can create false concurrency or lose causal detail.
- Node replacement/identity reuse needs care.
- A vector says versions are concurrent; it does not say how to merge values.
- Large sibling sets can explode storage and response sizes.
- Clients that drop context create unnecessary siblings.
- It tracks partial causal order, not one total order.

The original Dynamo paper describes truncating old clock entries when clocks grow beyond a threshold, which is a pragmatic loss of causal information rather than a mathematically free optimization.

### 11.8 Dotted version vectors and other successors

**Dotted version vectors (DVVs)** distinguish an individual update (“dot”) from the summarized causal history. They can track causality more precisely in some client/server patterns and are used by later Riak versions.

Other choices include:

- hybrid logical clocks for time-like ordering plus logical causality;
- per-key sequence numbers under a leader;
- consensus log index/term;
- CRDT-specific causal metadata;
- wall-clock timestamp for last-write-wins when its data-loss risk is acceptable.

Choose version metadata together with the consistency and conflict model; it is not a decorative field.

## 12. Conflict resolution

Vector clocks answer “ancestor or concurrent?” The system/application must answer “what should the final state be?”

### 12.1 Last-write-wins (LWW)

Choose the value with the greatest timestamp or deterministic tie-breaker.

Benefits:

- one value returned;
- simple storage and API;
- deterministic convergence if all replicas apply the same rule.

Costs:

- silently discards a concurrent write;
- depends on clock quality if physical timestamps are used;
- “last by timestamp” may not be last in real time;
- poor fit when both user intents matter.

LWW is conflict **selection**, not conflict avoidance.

### 12.2 Application/client merge

Return siblings and let domain logic reconcile:

```text
cart additions:     union items, with explicit remove semantics
contact fields:     merge independently if fields do not conflict
document edits:     show conflict or use domain-aware merge
```

Benefits: preserves business meaning.

Costs: complex clients, sibling growth, more round trips, and every consumer must handle conflicts correctly.

### 12.3 Server-side deterministic merge

Put domain-aware merge logic in the service so all clients get one rule. This reduces client complexity but couples storage to value semantics and makes schema/version evolution harder.

### 12.4 CRDTs

A Conflict-free Replicated Data Type constrains state/operations so concurrent updates merge deterministically and replicas converge without choosing one update arbitrarily.

Examples:

- grow-only set;
- observed-remove set;
- grow-only or positive/negative counter;
- last-write-wins register (still has LWW semantics).

CRDTs are not magic:

- the data type must match product semantics;
- metadata and tombstones can grow;
- invariants such as “stock never below zero” are not generally solved by a mergeable counter;
- deletes and resets need precise rules.

### 12.5 Serialize or reject conflicts

For a username claim, payment balance, lock owner, or last seat, preserving two concurrent “successes” is often wrong. Use a leader/consensus/conditional transaction and reject one writer.

### 12.6 Shopping-cart merge trap

Set union is a popular example:

```text
A adds pen, B adds mug -> union keeps both
```

But deletion breaks the naive rule:

```text
A removes book, B has an older cart containing book
plain union resurrects book
```

Correct choices include operation logs, observed-remove sets, item-level versions/tombstones, or product rules that surface the conflict. “Merge carts with union” is not a complete design.

### 12.7 Comparison table

| Policy | Keeps all intent? | Complexity | Clock sensitivity | Good fit |
|---|---:|---:|---:|---|
| LWW | No | Low | Often high | Cache/config where loss is acceptable |
| Client merge | Potentially | High | Low with causal metadata | Rich domain conflicts |
| Server merge | Potentially | Medium/high | Depends | One well-defined domain |
| CRDT | For modeled operations | Medium/high | Usually less physical-clock dependence | Counters, sets, collaborative state |
| Consensus/CAS | One ordered winner | High coordination | No wall-clock ordering required | Hard invariants |

## 13. Gossip protocol

### 13.1 Plain-language analogy

Instead of one central announcer telling every node the cluster state, each node periodically calls a few peers:

```text
round 1: A tells B
round 2: A tells D, B tells C
round 3: C tells E, D tells F, ...
```

Information spreads epidemically. With versioned state and repeated exchanges, all healthy, connected nodes tend toward the same membership view.

### 13.2 What gossip can disseminate

- node identity and endpoint;
- heartbeat/liveness evidence;
- token ownership/topology version;
- schema/config version;
- load or capacity summaries.

Exact content is implementation-specific.

### 13.3 A basic exchange

```text
Node A chooses random Node D

A -> D: digest of states A knows
D -> A: newer/missing state
A -> D: states D requested
```

Version/generation numbers prevent older gossip from overwriting newer membership state.

### 13.4 Failure detection is suspicion, not certainty

If A stops hearing B, A cannot know whether:

- B crashed;
- the network is partitioned;
- B is paused/overloaded;
- A itself is isolated.

Failure detectors therefore produce local suspicion based on heartbeat arrival patterns and timeouts. Different nodes can temporarily disagree about who is up.

This is why automatically deleting/reassigning a node on a short timeout is dangerous. A network partition could make both sides believe the other is dead and move data simultaneously.

### 13.5 Seeds do not have to be permanent masters

New nodes need at least one known address to join. **Seed nodes** provide bootstrap contacts. Once joined, ordinary gossip can be peer-to-peer; seeds need not coordinate client data.

Seed availability still matters for new-node bootstrap, so configure several across failure domains.

### 13.6 Gossip is not user-data replication

Normally:

```text
client values --> replication protocol / replica writes
cluster facts  --> gossip
```

Sending every user value through gossip would be slow, redundant, and hard to acknowledge. Gossip is best for small, versioned control-state dissemination.

### 13.7 Gossip is not consensus

Gossip aims for eventual dissemination. It does not by itself guarantee:

- one globally agreed membership view at an instant;
- one leader;
- one total order of writes;
- atomic configuration changes.

A safe topology-change system may combine gossip for distribution with operator approval, epochs, or a consensus-backed metadata service.

Apache Cassandra’s first-party [Dynamo architecture guide](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html#distributed-cluster-membership-and-failure-detection) describes periodic gossip plus local failure-detector decisions and intentionally avoids automatically removing a node merely because it is temporarily unreachable.

## 14. Replica repair: hints, read repair, and Merkle trees

Replica divergence is normal in an eventually consistent design. Convergence needs multiple mechanisms because each covers a different gap.

### 14.1 Repair toolbox

| Mechanism | Trigger | Best at | Misses / cost |
|---|---|---|---|
| Hinted handoff | Failed write delivery to a known owner | Short temporary outages | Hint can expire/fail; not silent corruption |
| Read repair | A read observes unequal replicas | Hot/read keys | Cold keys may never heal; adds read work |
| Anti-entropy repair | Scheduled/background replica comparison | Cold keys, missed hints, silent divergence | CPU/disk/network intensive |
| Full restore/rebuild | Node replacement/disaster | Large loss | Heavy data streaming and operational work |

### 14.2 Why not compare every byte of every replica?

Sending every key/value across the network to discover a tiny difference would cost roughly the whole dataset per comparison.

A Merkle tree summarizes ranges hierarchically.

### 14.3 Merkle tree foundation

Hash each leaf bucket, then hash pairs of child hashes upward:

```text
                         Root H(ABCD)
                       /              \
                 H(AB)                  H(CD)
                /     \                /     \
            H(range A) H(range B)  H(range C) H(range D)
```

Two replicas compare roots:

- equal root (assuming a suitable hash and identical tree construction) -> no detected difference;
- different root -> compare children;
- recurse only into differing branches;
- identify mismatched leaf ranges, then exchange actual keys/versions there.

### 14.4 Repair example

```text
Replica A tree root = 91af
Replica B tree root = 44c2       roots differ

left child:  A=1200, B=1200     equal, skip half the range
right child: A=9981, B=73aa     differ, descend

eventually locate token subrange [70, 75]
exchange only keys/versions in that range
```

The tree saves comparison bandwidth when most data agrees. Repairing the mismatched range can still transfer substantial data, depending on leaf granularity.

### 14.5 Trees must line up

Replicas can compare trees only when both summarize the same logical key/token range with compatible boundaries and hash rules. Membership changes can invalidate or require rebuilding trees because ownership ranges changed.

The original Dynamo design maintained trees per key range owned by a virtual node. Riak documents a related hash-tree exchange in its [active anti-entropy guide](https://docs.riak.com/riak/kv/latest/learn/concepts/active-anti-entropy/index.html).

### 14.6 Merkle tree limitations

- Building/updating trees consumes CPU, memory/disk, and I/O.
- Coarse leaves can cause over-streaming for a small difference.
- Fine leaves enlarge tree metadata and comparison work.
- Equal hashes are probabilistic evidence; use a strong enough hash.
- The tree identifies disagreement, not the semantically correct winner.
- Concurrent updates during tree construction need snapshot/version handling.
- Repair traffic can compete with foreground requests.

Apache Cassandra’s [repair documentation](https://cassandra.apache.org/doc/latest/cassandra/managing/operating/repair.html) explicitly warns that repair is disk/network intensive and describes full versus incremental repair.

### 14.7 Deletes and tombstones

Deleting a key cannot immediately erase every trace:

```text
A receives DELETE(k) -> removes value
B was offline         -> still has old value
A forgets deletion
B returns             -> anti-entropy sees B's value and may resurrect it
```

Instead, store a versioned **tombstone** meaning “this key was deleted.” Replicate and repair it like a value. Garbage-collect it only after the system can safely assume all relevant replicas have received it or their old data can no longer rejoin.

Premature tombstone deletion is a classic eventual-consistency data-resurrection bug.

### 14.8 Repair is not backup

If an operator deletes or corrupts data and that mutation wins everywhere, repair faithfully spreads it. Backups and point-in-time recovery protect against logical mistakes and correlated corruption; replicas primarily protect online availability/durability.

## 15. End-to-end failure scenarios

### 15.1 Normal operation

```text
PUT k=v1 -> replicas A/B acknowledge -> client success
                                \
                                 C receives shortly after

GET k -> query B/C -> compare same version -> v1
```

No conflict; background mechanisms are mostly idle.

### 15.2 One replica fails during a write

```text
normal replicas [A,B,C], A down
PUT -> B + C + fallback D(hint for A)
W reached -> success
A returns -> D hands off -> A catches up
```

If D also fails before handoff, read repair or anti-entropy must still find the gap.

### 15.3 Network partition with concurrent writes

```text
             network partition
        A/B  X------------------X  C/D

left accepts cart + pen       right accepts cart + mug
clock {A:1,B:1}               clock {A:1,C:1}
```

When communication heals:

1. read/anti-entropy encounters both versions;
2. vector comparison says concurrent;
3. application/CRDT merges or policy selects one;
4. resolved descendant is written;
5. repair propagates it to all normal replicas.

An answer that ends at “eventual consistency” without steps 2–5 has not explained convergence.

### 15.4 Coordinator fails after replicas wrote

```text
replicas stored v7
coordinator crashes before client response
client sees timeout
```

This is an **unknown outcome**, not a definite failure. The client uses the same idempotency key, performs a read-back, or issues a context-aware retry.

### 15.5 Stale topology views

Coordinator X thinks owners are `[A,B,C]`; coordinator Y thinks new owners are `[B,C,D]`.

Safe systems need:

- topology epochs/versions;
- forwarding or retry hints;
- an overlap/streaming protocol during ownership transfer;
- fencing against an obsolete owner accepting unsafe writes;
- repair after convergence.

Consistent hashing solves the mathematical placement change, not the distributed cutover protocol.

### 15.6 Entire zone fails

If replicas span A/B/C zones and one zone fails, `N=3, R=2, W=2` can continue if the two surviving replicas are reachable and provisioned for failover load.

Capacity is part of availability: surviving two zones can be logically sufficient but collapse under the failed zone’s traffic.

## 16. Eventual consistency, CAP, and PACELC

### 16.1 Precise convergence promise

A useful statement is:

> If no new updates occur, network communication resumes, replicas remain repairable, and reconciliation/repair continues, all replicas for a key eventually converge to the same resolved state.

It does not specify a deadline. A separate bounded-staleness SLO would say something measurable such as a percentile replication-delay target.

### 16.2 CAP position

During a network partition, the Dynamo-style design in this chapter favors availability for selected operations:

- both reachable sides may accept writes;
- replicas may diverge;
- conflicts are reconciled later.

Therefore it sacrifices linearizable single-copy behavior during that partition.

See [02. CAP, PACELC and Replication Strategies](./02.%20CAP,%20PACELC%20and%20Replication%20Strategies.md) for the full theorem. The short interview-safe statement is: the forced consistency/availability choice occurs **during a partition**, and can be made per operation—not once for the company.

### 16.3 PACELC position

Even without a partition:

- waiting for more/farther replicas improves freshness/durability;
- waiting increases latency and tail exposure;
- replying after fewer/local acknowledgements lowers latency but permits lag.

That ordinary-day latency/consistency choice is PACELC’s point.

### 16.4 Eventual core plus stronger operations

One product can use:

| Operation | Guarantee | Reason |
|---|---|---|
| Load recommendations | Eventual/cache | Staleness is tolerable |
| Update profile bio | Read-your-writes for owner, eventual globally | Good UX without global serialization |
| Add cart item | Mergeable/AP | Concurrent additions can be preserved |
| Claim unique username | Strong CAS/consensus | Exactly one winner |
| Charge wallet | Transactional/strong | Hard money invariant |

“Our database is AP” is less useful than naming behavior for the exact operation.

## 17. Strongly consistent alternative

Not every distributed key-value store should use Dynamo’s leaderless conflict model.

### 17.1 Leader/consensus per shard

```text
hash(key) -> shard 17

                 leader B
Client PUT -------> B ---- replicated log ----> A, C
                    |
                    +-- commit after quorum agreement
```

Raft/Paxos-style replication orders writes for each shard. Followers apply the same committed log.

Benefits:

- linearizable per-key/shard reads and writes can be provided;
- conditional update and invariants are easier;
- no application-visible sibling versions for ordered writes;
- topology/leadership uses explicit terms/epochs.

Costs:

- minority partition cannot safely accept writes;
- leader/consensus adds coordination and latency;
- leader hotspots and elections must be handled;
- multi-shard transactions remain harder.

### 17.2 Real examples of the alternative

- **Source-era X/Twitter Manhattan:** X described a strong-consistency service using per-shard replicated logs and an elected writer in [“Strong consistency in Manhattan”](https://blog.x.com/engineering/en_us/a/2016/strong-consistency-in-manhattan).
- **Source-era Uber Docstore:** Uber described partitions running Raft with a leader and followers, offering strict serializability within a partition in [“Evolving Schemaless into a Distributed SQL Database”](https://www.uber.com/en-US/blog/schemaless-sql-database/).
- **Source-era Meta ZippyDB:** Meta described a managed, strongly consistent, geographically distributed key-value store built using RocksDB plus replication and shard-management infrastructure in [its ZippyDB article](https://engineering.fb.com/2021/08/06/core-infra/zippydb/).

These show that “key-value” and “eventually consistent leaderless” are separate decisions.

### 17.3 Choosing between leaderless and leader-based

| Requirement | Leaderless/eventual | Leader/consensus |
|---|---|---|
| Write on both sides of a partition | Often yes | Only quorum/leader side |
| Preserve concurrent writes | Version/merge siblings | Serialize or reject |
| Hard invariant / CAS | Awkward or extra protocol | Natural per shard |
| Coordination latency | Can be lower | Quorum/leader coordination |
| Application conflict burden | Higher | Lower |
| Availability during partition | Higher for allowed operations | Lower on minority side |

## 18. Local data and API design details

### 18.1 Suggested record envelope

```text
key: bytes
value: bytes | null
version_context: vector/DVV/HLC metadata
tombstone: boolean
expires_at: optional timestamp
checksum: corruption detection
schema_codec_version: optional
request_id: optional deduplication reference
```

Values need size limits. Large objects increase:

- replication and repair bandwidth;
- tail latency;
- memory pressure;
- compaction amplification.

Store large media in object/blob storage and keep metadata/reference in the KV store.

### 18.2 Example API

```http
PUT /v1/kv/{base64url-key}
Idempotency-Key: 018f...
If-Version: <opaque causal context, optional>
Consistency: quorum

<opaque value bytes>
```

```http
GET /v1/kv/{base64url-key}
Consistency: one | quorum | all
```

Illustrative response:

```json
{
  "value": "<base64 bytes>",
  "context": "<opaque version token>",
  "siblings": []
}
```

HTTP is only an example. A binary protocol/gRPC may reduce overhead. Never make clients construct vector clocks manually; treat context as opaque.

### 18.3 Delete API

`DELETE` writes a tombstone with a new version. Conditional delete should name the expected causal version. Immediate physical erasure is an asynchronous compaction/retention concern.

Privacy deletion may require a separate audited workflow because replication, hints, backups, and logs all contain copies.

### 18.4 TTL and expiry

TTL introduces clock and visibility questions:

- Is expiry evaluated by coordinator or every replica?
- What happens under clock skew?
- Is an expired record a tombstone?
- Can read repair resurrect it?
- When is disk space reclaimed?

For safety, store an authoritative expiry timestamp/version and define deterministic read/compaction rules. Do not rely only on a best-effort background sweeper.

### 18.5 Secondary indexes

An exact-key store does not automatically answer “find every user whose city is Pune.” A distributed secondary index is another partitioned, replicated dataset that must stay consistent with base writes.

Options:

- denormalize an index record in the application;
- asynchronous index via change stream (eventually consistent);
- synchronous transactional index update (more coordination);
- send search use cases to a search system.

The index can have a different hotspot pattern from the base key.

## 19. Scalability, overload, and operational edge cases

### 19.1 Horizontal scale is not automatically linear

Adding nodes helps only if bottlenecks are partitionable. Non-partitioned bottlenecks include:

- one hot key/tenant;
- central proxy or metadata service;
- all-node gossip/schema storms;
- repair/compaction bandwidth;
- downstream network/storage limits;
- cross-shard transactions or scans.

State “near-linear for a uniform independent-key workload until another shared resource dominates,” not “infinite linear scale.”

### 19.2 Rebalancing can cause an outage while adding capacity

New nodes trigger streaming, cache coldness, compaction, and replica changes. Throttle transfers, reserve headroom, prioritize foreground traffic, and verify completion.

If a cluster normally runs near 100% disk or network utilization, it has no safe recovery/rebalance capacity.

### 19.3 Tail latency and stragglers

Waiting for `R/W` fastest of `N` can avoid one slow replica, but slow requests continue consuming resources. Use:

- deadlines propagated end to end;
- bounded speculative requests;
- cancellation of unnecessary work;
- per-tenant queues and concurrency limits;
- circuit breakers for unhealthy replicas.

### 19.4 Retry storms

If every client immediately retries a timeout three times, an overloaded store receives more work and fails harder.

Use exponential backoff with jitter, retry budgets, idempotency, load shedding, and `Retry-After`-like guidance.

### 19.5 Noisy neighbors

Multi-tenant stores need:

- per-tenant rate and storage quotas;
- admission control;
- resource isolation for compaction/repair;
- hot-key and large-value detection;
- fair scheduling;
- tenant-aware capacity attribution.

### 19.6 Data corruption

Replication can reproduce corruption. Use checksums at rest/in transit, scrub data, compare replicas, retain backups, and distinguish logical versions from corrupted bytes.

### 19.7 Split brain and stale owners

Eventual membership plus sloppy writes increases the need for explicit topology epochs and safe handoff. A node that was removed and later returns must not simply rejoin with old data as an authoritative replica.

Use replacement identities, fencing/epochs, controlled bootstrap, and full repair.

### 19.8 Disaster recovery

Define:

- **RPO:** acceptable data-loss window;
- **RTO:** acceptable restoration-time target;
- backup frequency/retention;
- cross-region replica semantics;
- allocator/topology recovery;
- restore tests.

Replication across zones is not automatically multi-region disaster recovery.

## 20. When to use or avoid this design

### 20.1 Good fits for a key-value model

- exact-key session/profile/config lookup;
- cache and precomputed recommendation lookup;
- carts with explicit merge semantics;
- feature flags or metadata, if consistency needs are met;
- per-entity state where operations stay within one key/partition;
- high-throughput write/read workloads with known access patterns.

### 20.2 Good fits specifically for the leaderless eventual design

- temporary staleness is acceptable;
- high partition-time write availability matters;
- conflicts can be merged or surfaced;
- keys are independent;
- low coordination latency is more valuable than one global order.

### 20.3 Avoid or add a stronger subsystem when

- money, inventory, locks, or uniqueness require one winner;
- queries depend on joins, arbitrary predicates, or many secondary indexes;
- multi-key atomic transactions dominate;
- values are huge blobs;
- clients cannot tolerate/resolve siblings;
- the team cannot operate repair, compaction, backups, and topology changes;
- a managed database or a single relational database already meets scale.

Do not build a distributed database merely to demonstrate distributed-systems vocabulary. Operational simplicity is a valid design requirement.

### 20.4 Alternatives

| Alternative | Prefer when | Main trade-off |
|---|---|---|
| Single PostgreSQL/MySQL | Moderate scale, relational queries/transactions | Vertical/failover limits but great simplicity |
| Sharded SQL | Relational model plus higher scale | Complex resharding/cross-shard transactions |
| Redis/Valkey cache | Very low latency, disposable/reconstructable data | Memory cost and durability/consistency choices |
| DynamoDB/Cassandra/Riak-like store | Key-based large-scale availability | Data-model discipline and weaker guarantees/config complexity |
| etcd/Consul/ZooKeeper-like store | Small critical metadata, coordination | Not a bulk high-throughput general data store |
| Object storage | Large immutable blobs | Higher latency and limited fine-grained updates |
| Search engine | Text/predicate search | Index lag and not primary transactional truth |

## 21. Real-system examples

### 21.1 Amazon: original Dynamo and modern DynamoDB

**Source-era verified:** The 2007 Dynamo paper describes an internal highly available key-value store designed for Amazon services where rejecting cart-like updates could harm customer experience. It combines consistent hashing, preference lists, configurable `N/R/W`, versioning, hinted handoff, gossip, and Merkle-tree anti-entropy: [original Dynamo paper](https://www.cs.cmu.edu/afs/cs/academic/class/15712-f08/www/readings/Vogels07.pdf).

```text
cartId -> serialized cart + causal versions
```

**Current product distinction:** DynamoDB supports partition-key based distribution and offers eventual and strong read options for certain operations; global-table options have their own documented semantics. See AWS’s current [read-consistency documentation](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/HowItWorks.ReadConsistency.html). Do not infer that its current internals or guarantees are identical to the 2007 Dynamo paper.

Lesson: start from the business behavior—cart writes should remain accepted—then choose conflict handling. Do not start with “AP because Dynamo.”

### 21.2 X/Twitter: Manhattan

**Source-era verified:** X/Twitter described Manhattan as its internal distributed key-value store used for core nouns including Tweets, users, and direct messages, with pluggable storage engines including LSM/B-tree/RocksDB options in [“Adopting RocksDB within Manhattan”](https://blog.x.com/engineering/en_us/topics/infrastructure/2021/adopting-rocksdb-within-manhattan).

It also separately described eventually consistent and strong-consistency modes/services. This is an important lesson: one platform can expose different consistency paths instead of forcing every dataset into one trade-off.

```text
(datasetId, partitionKey, localKey) -> value + TTL/tombstone metadata
```

### 21.3 YouTube: Bigtable within a larger architecture

**Source-era verified:** Google Cloud described YouTube using Bigtable for use cases including metadata/reporting, recommendations-related data, view metrics, and advertising functionality in [“How YouTube uses Bigtable”](https://cloud.google.com/blog/products/databases/youtube-runs-on-bigtable).

Bigtable is an ordered/wide-column key-value model, not the exact leaderless Dynamo design.

```text
video/channel key -> metadata or metric-oriented columns
```

Lesson: “key-value” can include ordered keys and structured column families; the workload-specific table design matters more than the label.

### 21.4 Netflix: EVCache

**Source-era verified:** Netflix’s open-source Rend documentation says EVCache is Netflix’s main caching solution and describes recommendation batches loading results into it: [Netflix Rend repository](https://github.com/Netflix/rend).

```text
memberId + recommendationModelVersion -> precomputed recommendation payload
```

This is a cache use case, not evidence that Netflix uses the Dynamo mechanisms in this chapter. The source also illustrates a practical edge case: changing value sizes interacted badly with Memcached slab allocation, motivating chunking.

Lesson: value-size distribution and memory allocator behavior can matter as much as ring diagrams.

### 21.5 Uber: Docstore

**Source-era verified:** Uber describes Docstore as a horizontally scalable distributed database with application-chosen partition keys and Raft-replicated partitions in [“Evolving Schemaless into a Distributed SQL Database”](https://www.uber.com/en-US/blog/schemaless-sql-database/).

```text
trip/customer partition key -> related ordered records
```

It is a useful counterexample to “distributed KV must be leaderless”: Uber chose strict serializability at partition level for business-critical use cases.

### 21.6 WhatsApp: key directory as a consistency-sensitive mapping

**Verified public behavior, illustrative storage mapping:** WhatsApp publicly describes storing users’ public encryption keys in an auditable key directory and batching high-volume key changes in [“Deploying key transparency at WhatsApp”](https://engineering.fb.com/2023/04/13/security/whatsapp-key-transparency/).

Conceptually there is a lookup:

```text
user/device identity -> public encryption key material + audit proof context
```

But a plain eventually consistent LWW key-value store would be insufficient for the security guarantee: unauthorized or equivocated key replacement must be detectable. The published system adds an append-only auditable directory and proofs.

Lesson: a problem can look like `identity -> key` while requiring much stronger integrity semantics than ordinary KV retrieval. This section does **not** claim WhatsApp uses the Dynamo architecture for messages or presence.

### 21.7 Illustrative cross-product mapping

These are not production architecture claims:

| Product operation | Possible key/value | Likely concern |
|---|---|---|
| WhatsApp presence | `userId -> lastSeen/status` | High availability; staleness may be acceptable |
| YouTube video metadata | `videoId -> title/owner/status` | Read scale, cache, delete propagation |
| Uber driver location | `driverId -> recent location` | TTL, high write rate, freshness; specialized geo index needed for nearby search |
| Netflix recommendations | `memberId:modelVer -> list` | Cacheability, large values, batch refresh |
| X/Twitter post lookup | `postId -> post record` | Durable low-latency exact-key reads |
| Amazon cart | `cartId -> cart state` | Concurrent updates and merge semantics |

Notice that “find nearby drivers” and “search posts” are not exact-key queries; they need indexes/specialized services around the KV source data.

## 22. Trade-off summary

### 22.1 Core design decisions

| Decision | Chosen here | Alternative | Trade-off |
|---|---|---|---|
| Partitioning | Consistent-hash token ranges + vnodes | Range sharding/directory | Incremental movement vs ordered scans/control metadata |
| Replication | Leaderless `N` replicas | Leader + followers | Partition availability vs ordered writes |
| Write success | Tunable `W` | All/one/consensus commit | Durability/freshness vs latency/availability |
| Read success | Tunable `R` + version comparison | Leader read / lease read | Read scale/availability vs simpler freshness |
| Temporary failure | Sloppy quorum + hints | Strict normal owners | Availability vs weaker fixed-set overlap |
| Versioning | Vector/DVV-like causal context | LWW timestamp / log index | Preserve concurrency vs metadata/API complexity |
| Conflict | Domain merge/siblings | LWW or reject | Intent preservation vs application complexity |
| Membership | Gossip-like dissemination | Central/consensus registry | Decentralization vs instantly authoritative view |
| Repair | Hints + read repair + Merkle anti-entropy | Log catch-up/snapshots | Availability/convergence vs background cost |

### 22.2 Tension between the three stated goals

```text
Scalability
  wants partitioning and independent shards
       |
       v
Decentralization
  removes one ordering authority
       |
       v
Eventual consistency
  accepts temporary divergence
       |
       v
Versioning + conflict resolution + repair become mandatory
```

The mechanisms are not a random checklist. Each repairs a consequence introduced by an earlier choice.

## 23. Common mistakes and precise corrections

### Mistake 1: “Consistent hashing replicates data.”

Correction: it maps keys to a primary token range. A separate preference/placement policy chooses `N` distinct replica nodes.

### Mistake 2: “`R + W > N` guarantees strong consistency.”

Correction: it guarantees set overlap only for quorums drawn from the same `N`-replica universe. Correct version comparison, topology, concurrency control, and sloppy-quorum behavior still matter; it is not alone a linearizability proof.

### Mistake 3: “Eventual consistency means the write will appear in a few seconds.”

Correction: no bound is implied. State a measured replication-lag SLO separately.

### Mistake 4: “Vector clock tells us the latest value.”

Correction: it can show dominance or concurrency. Concurrent clocks have no single latest value until a conflict policy resolves them.

### Mistake 5: “Last-write-wins means the real last user action wins.”

Correction: usually it means the greatest supplied/assigned timestamp wins; skew and delay can make that different from real-time order.

### Mistake 6: “Gossip replicates the key-value records.”

Correction: in this architecture it disseminates membership/topology/failure information. Replica writes use the request/replication protocol.

### Mistake 7: “Gossip tells everyone exactly who is alive.”

Correction: failure detection is local suspicion in an asynchronous network. Views can differ temporarily.

### Mistake 8: “Merkle trees repair data.”

Correction: they efficiently locate differing ranges. Version comparison and streaming/reconciliation perform the repair.

### Mistake 9: “Replication is backup.”

Correction: replicas copy bad deletes/corruption too. Maintain independent backups and test restores.

### Mistake 10: “Three copies means two failures are safe.”

Correction: it depends on placement, `R/W`, simultaneous failures, durability of copies, and available failover capacity.

### Mistake 11: “Any node can write, so there is no coordination.”

Correction: the coordinator still locates replicas, creates version context, sends writes, waits for `W`, handles retries, and reconciles reads. Leaderless means no permanent per-key leader, not zero communication.

### Mistake 12: “A load balancer solves partition routing.”

Correction: a generic LB selects a service endpoint. Some component still needs the token map and replica preference list.

### Mistake 13: “Hotspots disappear with more virtual nodes.”

Correction: vnodes spread many keys and range movement. One hot key still targets one replica set.

### Mistake 14: “Delete is just remove from the hash map.”

Correction: distributed eventual stores need versioned tombstones and safe garbage collection to prevent resurrection.

### Mistake 15: “No master is always more scalable.”

Correction: a sharded leader system can scale well; a small consensus metadata plane can improve safety. Choose decentralization based on failure and throughput goals.

### 24.3 Answer spine: `K-V-S-T-O-R-E`

Use cues, not a memorized paragraph.

#### K — Key, workload, and requirements

- Exact-key `GET/PUT/DELETE`; value size and TTL?
- Read/write QPS, data size, regions, key skew?
- Durability and latency targets?
- Single-key or multi-key transactions?

#### V — Visible guarantees and versions

- Eventual, causal/session, or linearizable?
- What can a read return after a completed write?
- How are concurrent versions detected/resolved?

#### S — Shard and scale

- Hash partitioning/consistent ring/vnodes.
- Rebalance protocol and hot-key exception.
- Local WAL/memtable/SSTable engine.

#### T — Three knobs: `N`, `R`, `W`

- Failure-domain-aware preference list.
- Fixed versus sloppy quorum; hinted handoff.
- Explain overlap and its limitations.

#### O — Operation paths and outcomes

- Walk through `PUT` and `GET` packet by packet.
- Define durable ack.
- Cover timeout ambiguity, retries, and idempotency.

#### R — Routing, repair, and recovery

- Token-aware client vs LB/proxy/any-node coordinator.
- Gossip for membership.
- Hints, read repair, Merkle anti-entropy, tombstones, backup.

#### E — Edge cases and evolution

- Hot key, large value, node/zone failure, stale topology, overload.
- Compare leaderless design with consensus-per-shard.
- Metrics, capacity headroom, disaster recovery.

### 24.4 A strong opening answer

> “I’ll first clarify whether we need a durable exact-key database or an evictable cache, the value-size and traffic distribution, and the client-visible consistency guarantee. I’ll assume a durable multi-node store whose priority is low-latency availability, so I’ll start with a Dynamo-style design and call out where a consensus-backed shard would be better. I’ll hash keys onto token ranges, use virtual nodes for balanced movement, and place `N` replicas across failure domains from a preference list. Any node can coordinate; a `PUT` waits for configurable `W` durable acknowledgements and a `GET` compares `R` versioned responses. `R + W > N` gives fixed-set overlap, not automatic linearizability. Vector or dotted-version context detects concurrent writes, while application merge or CRDTs resolve them. Gossip spreads membership, and hints, read repair, and Merkle anti-entropy heal different forms of replica divergence.”

### 24.5 Suggested 40-minute flow

```text
0–5 min    clarify API, scale, value size, regions, guarantees
5–8 min    estimate partitions/storage/throughput qualitatively or numerically
8–14 min   top-level components and local storage engine
14–20 min  partitioning, vnodes, preference list, placement
20–27 min  PUT/GET, N/R/W, durability, retries
27–32 min  vector clocks, conflicts, session behavior
32–36 min  gossip, hints, read repair, Merkle repair
36–40 min  failures, hotspots, monitoring, strong alternative
```

Adjust to interviewer signals; do not spend half the interview deriving the ring before stating the consistency contract.

### 24.6 Synthesized follow-up questions and answer cues

The questions below are **my own plausible synthesis**, not attributed to a particular company.

#### “Does `R=2, W=2, N=3` guarantee strong consistency?”

Answer cues:

- fixed-set quorum intersection, at least one shared replica;
- read must compare versions and choose descendant;
- concurrent versions can still exist;
- sloppy quorums/topology changes can violate same-set assumption;
- no automatic real-time serialization;
- use consensus/linearizable quorum-register protocol for a stronger guarantee.

#### “What happens if the coordinator crashes after writing but before replying?”

Answer cues:

- outcome unknown, not definitely failed;
- idempotency token/dedup or context-aware retry;
- read-back may discover committed version;
- never retry a non-idempotent logical mutation blindly.

#### “Why not just use last-write-wins?”

Answer cues:

- easy deterministic one-value result;
- loses one concurrent intent;
- clock skew/rollback changes winner;
- acceptable for some cache/profile fields, dangerous for carts/money;
- vector + domain merge or consensus alternatives.

#### “How does a vector clock detect concurrency?”

Answer cues:

- map actor -> counter;
- X dominates Y if every component `>=` and one `>`;
- neither dominates -> concurrent siblings;
- merge contexts by component-wise max, increment resolver;
- metadata/pruning/client-context limitations.

#### “Why do we need both hinted handoff and Merkle trees?”

Answer cues:

- hint knows a specific write missed a temporarily unavailable owner;
- hints can expire/fail and do not find silent corruption;
- Merkle anti-entropy compares entire shared ranges efficiently;
- read repair heals only accessed keys;
- complementary, not substitutes.

#### “Does gossip elect the latest ring?”

Answer cues:

- gossip disseminates versioned state eventually;
- nodes may temporarily disagree;
- failure suspicion is local;
- safe topology changes need epochs/operator/consensus/fencing;
- seeds bootstrap; not permanent masters.

#### “How do you prevent a deleted value from returning?”

Answer cues:

- versioned tombstone replicated/repaired;
- retain beyond maximum repair/missed-replica window;
- repair before tombstone GC;
- control old-node rejoin;
- backup/privacy deletion handled separately.

#### “A key goes viral. Add more nodes?”

Answer cues:

- one key still maps to one replica set;
- cache/edge, request coalescing, extra read replicas;
- split write key only if operations can aggregate;
- isolate/rate-limit tenant, protect backend;
- monitor per-key/per-partition skew.

#### “How do you add a node without losing writes?”

Answer cues:

- topology epoch and planned ownership;
- stream snapshot/ranges;
- capture concurrent writes or dual-write/forward;
- verify replicas;
- switch routing then retire old data;
- throttle and preserve headroom.

#### “Could we use Raft instead?”

Answer cues:

- yes, per shard;
- serializes writes, simplifies CAS and conflicts;
- only quorum side writes during partition;
- leader/quorum latency and elections;
- choice follows invariant and availability needs.

#### “What exactly does eventual mean?”

Answer cues:

- updates stop, communication and repair progress;
- deterministic reconciliation;
- replicas converge eventually;
- no time bound or session guarantees unless separately promised/measured.

### 24.7 Interviewer red flags

- Starting with Dynamo mechanics before clarifying cache versus durable database.
- Calling Dynamo and DynamoDB identical.
- Drawing a hash ring but not explaining physical-node-distinct replicas.
- Saying `R + W > N` equals linearizability.
- Ignoring sloppy-quorum fallback placement.
- Returning the first `GET` response despite querying `R` replicas.
- Using vector clocks but not carrying client context.
- Mentioning conflict resolution without a product rule.
- Saying gossip provides consensus or transports all values.
- Saying Merkle trees identify the “newest” value.
- Omitting tombstones, ambiguous retries, and topology changes.
- Claiming horizontal scale fixes a single hot key.
- Treating replicas as backups.
- Giving exact capacity/latency numbers with no workload or evidence.

## 25. Failure matrix

| Failure | Immediate behavior | Recovery | Important risk |
|---|---|---|---|
| One normal replica down | Use remaining/fallback; require `R/W` | Hint/read/anti-entropy | Reduced redundancy |
| Coordinator down before response | Client sees timeout/unknown | Retry with idempotency/read-back | Duplicate logical mutation |
| Network partition | Both sides may accept in AP mode | Version compare + merge + repair | Invariant violation/siblings |
| Gossip delay | Some nodes have stale topology | Further gossip/authoritative epoch | Misrouting/dual ownership |
| Hint holder fails | Handoff missed | Read repair/anti-entropy | Longer inconsistency window |
| Disk corruption | Checksum/read error or mismatched replica | Repair from good copy/restore | Corruption replicated as valid data |
| Merkle repair overload | Foreground tail latency rises | Throttle/schedule | Recovery makes outage worse |
| Zone failure | Surviving zones serve if quorum/capacity | Re-replicate/restore zone | Correlated loss and overload |
| Premature tombstone GC | Old value may reappear | Manual correction/full repair | Data resurrection |
| Hot key | One replica set saturates | Cache/split/isolate | Adding generic capacity may not help |
| Full disk | Writes/compaction fail | Shed load/add space/rebalance | Compaction needs temporary space |
| Clock skew under LWW/TTL | Wrong winner/expiry | Fix clock, reconcile | Silent data loss |

## 26. Observability and operational readiness

### 26.1 Client-path metrics

- `GET`/`PUT`/`DELETE` rate and p50/p95/p99 latency;
- success, timeout, overload, and insufficient-replica rates;
- bytes per operation and value-size distribution;
- chosen consistency level and achieved acknowledgements;
- coordinator-to-replica fan-out and speculative-request rate;
- idempotent replay and conflict/sibling rates.

### 26.2 Partition and node metrics

- bytes/keys/QPS per token range, node, tenant, and hot key;
- disk space plus compaction headroom;
- WAL/fsync latency, memtable flush, SSTable count;
- compaction queue, write/read amplification;
- cache/Bloom-filter hit behavior;
- CPU, memory, disk IOPS, network, queue depth.

### 26.3 Replication and convergence metrics

- replica lag/version divergence;
- outstanding hints, age, replay failures;
- read-repair rate;
- Merkle tree build/compare duration;
- mismatched ranges and bytes streamed;
- under-replicated ranges and time to restore `N` copies;
- tombstone age and repair-before-GC compliance.

### 26.4 Membership and topology metrics

- gossip convergence/version skew;
- local UP/DOWN disagreement;
- topology epoch seen by clients/coordinators;
- range movement progress and throttling;
- seed/bootstrap failures;
- stale-owner requests and forwards.

### 26.5 SLO examples without invented thresholds

Define, measure, and choose actual thresholds from requirements:

- percentile `GET/PUT` latency by consistency level;
- successful-operation availability during one-node/one-zone loss;
- acknowledged-write durability objective;
- time until an under-replicated range regains target replication;
- percentile visibility lag for eventual reads;
- recovery time and recovery point objectives.

Do not claim “five nines” without defining the operation, region, consistency level, measurement window, and exclusions.

### 26.6 Tests to run

- kill coordinator before/after each acknowledgement point;
- isolate replica groups with a network partition;
- skew clocks if LWW/TTL uses physical time;
- fill disks and pause compaction;
- drop hints and prove anti-entropy heals;
- take a cold replica offline beyond tombstone retention;
- add/remove/replace nodes during live writes;
- overload one key and one tenant;
- restore from backup into a clean cluster;
- fail a zone and verify capacity, not merely quorum math.

## 27. Security and multi-tenancy

The Dynamo paper mechanisms address distribution, not the full service boundary.

Add:

- authentication and per-key/namespace authorization;
- TLS in transit and encryption at rest;
- tenant-separated key namespaces and quotas;
- maximum key/value/request sizes;
- rate limiting and abuse protection;
- audit logs for administrative/data access;
- checksum/integrity validation;
- secrets management and rotation;
- privacy deletion across replicas, hints, logs, and backups;
- protection against key enumeration and side-channel leakage.

Avoid placing credentials or personal data directly in human-readable keys because keys often appear in metrics, logs, traces, and partition-debug tooling.

## 28. Compact revision sheet

### 28.1 Ten facts to retain

1. A KV store maps unique keys to opaque values; it can be cache/durable and strong/eventual.
2. Partitioning spreads different keys; replication creates copies of each partition.
3. Consistent hashing reduces movement when nodes change; vnodes improve balance and rebalancing granularity.
4. A preference list is the ordered, failure-domain-aware replica candidate list for a key.
5. `N` is replica count, `W` write acknowledgements, `R` read responses.
6. `R + W > N` yields fixed-set overlap, not automatic linearizability.
7. Vector clocks distinguish causal descendants from concurrent siblings; they do not merge business meaning.
8. Gossip spreads cluster knowledge; it is not consensus or the normal user-data path.
9. Hints heal known temporary misses, read repair heals accessed keys, and Merkle anti-entropy finds cold divergence.
10. Tombstones, idempotent retries, topology epochs, hotspots, overload, backup, and repair are part of correctness—not optional polish.

### 28.2 Formula/logic card

```text
primary token = hash(key)
normal replicas = first N distinct, topology-valid nodes in preference order

fixed-set read/write intersection:       R + W > N
fixed-set write/write intersection:      2W > N

vector X dominates Y:
  for every actor i: X[i] >= Y[i]
  and for at least one i: X[i] > Y[i]

neither vector dominates -> concurrent versions
```

Remember: intersection formulas say sets overlap. The protocol determines what the overlap means.

### 28.3 Mechanism-to-problem map

```text
too much data / QPS       -> partitioning
membership changes       -> consistent hashing + controlled rebalance
node/disk failure        -> replication across failure domains
temporary owner down     -> sloppy quorum + hinted handoff
stale hot replica        -> read repair
stale cold replica       -> Merkle anti-entropy
concurrent writes        -> vector/DVV + conflict policy
cluster knowledge        -> gossip + failure detector
hard invariant           -> CAS / leader / consensus
hot single key           -> cache / extra replicas / split semantics
ambiguous timeout        -> idempotency + read-back/context
delete resurrection      -> tombstone + repair-aware GC
```

### 28.4 Design checklist

- [ ] Clarified durable DB versus cache and exact-key access patterns.
- [ ] Named value size, QPS, data size, skew, regions, retention, and SLO assumptions.
- [ ] Defined what a completed read/write guarantees.
- [ ] Selected partition scheme and explained node add/remove.
- [ ] Chosen physical-failure-domain-aware replica placement.
- [ ] Defined `N/R/W` and durable acknowledgement point.
- [ ] Distinguished strict from sloppy quorum and explained hints.
- [ ] Walked through full `PUT` and `GET` paths.

## 30. Say-out-loud 30-second narrative

> “A distributed key-value store maps each key to a partition, commonly using consistent hashing and virtual nodes, then copies it to `N` failure-independent replicas from a preference list. Any node can coordinate a write, wait for `W` durable acknowledgements, and read `R` versioned responses. `R + W > N` gives fixed-replica-set overlap, but not automatic linearizability, especially with concurrency or sloppy quorums. Vector clocks detect causal descendants versus concurrent siblings, while domain merge, CRDTs, LWW, or consensus decides conflicts. Gossip spreads membership knowledge, hinted handoff and read repair fix common misses, and Merkle-tree anti-entropy finds cold divergence so replicas eventually converge.”
