# Fault-Tolerant Systems, Cascading Failures, and Resilience4j

> Beginner-first HLD notes expanded from: **fault-tolerant systems, cascading failures, and the Resilience4j RateLimiter, Bulkhead, TimeLimiter, CircuitBreaker, and Retry modules**.
>
> The central interview lesson is: **failures are unavoidable; a resilient design prevents one slow or failed dependency from consuming every shared resource, multiplying its own load through retries, and taking unrelated features down with it.** Timeouts bound waiting, bulkheads bound concurrent damage, rate limiters bound admission rate, circuit breakers fail fast during sustained trouble, retries recover only carefully classified transient failures, and fallbacks preserve useful behavior.

## Scope, assumptions, and evidence labels

I assume the intended topic is **synchronous service-to-service calls in a Java/Spring microservice system**, because the bullets name Resilience4j. The underlying HLD patterns apply to other languages and to proxies/service meshes too.

## Corrections to the rough bullets, up front

1. **The phrase is "fault-tolerant system."** Fault tolerance does not mean that nothing fails. It means the system continues its required behavior, perhaps in a degraded form, for a stated set of faults.
2. **A cascading failure is not merely several failures at once.** It is a failure that spreads or grows through dependency/resource relationships and positive feedback.
3. **Resilience4j is a Java fault-tolerance library, not the fault-tolerant architecture itself.** It cannot create redundant data, repair a database, make a non-idempotent operation safe, coordinate a global fleet limit, or invent a correct fallback.
4. **Rate and concurrency are different dimensions.** A rate limiter controls how quickly work may start; a bulkhead controls how much work may be in flight at one time.
5. **Resilience4j's RateLimiter is not automatically a distributed token bucket.** Its documented local implementations replenish permissions in configured cycles. Each application process normally has its own limiter state.
6. **A circuit breaker does not limit concurrency.** In CLOSED state it can allow every concurrent caller through. Resilience4j explicitly recommends a Bulkhead when concurrent calls must be restricted.
7. **A timeout does not guarantee the underlying work stopped.** The caller may stop waiting while the remote service, database, future, or blocking thread continues. Cancellation has to propagate and be honored.
8. **A retry does not automatically avoid recomputing work X.** It avoids recomputation only when X is outside the retried function and its result can safely be reused.
9. **"Never retry 4xx" is too absolute.** Most unchanged 4xx requests should not be retried, but 408 may be repeated, 429 explicitly invites a delayed retry policy, and 409/412 might require refresh-and-recompute rather than a blind identical retry.
10. **"Retry only 5xx" is also too broad.** 501 is usually a missing capability, a deterministic 500 can fail forever, and even a transient 502/503/504 retry is unsafe for an unprotected non-idempotent operation.
11. **A network error is not automatically safe to retry.** A failure before connecting is different from a read timeout after the server might have committed a payment.
12. **Idempotency is necessary but not sufficient.** An idempotent retry can still overload a recovering server, exceed the user's deadline, or consume a retry budget.
13. **Fixed retry intervals can synchronize clients.** Plain exponential backoff can remain synchronized too. Jitter is what breaks correlation.
14. **Retries at multiple layers multiply.** If three layers each allow four total attempts, one user operation can create up to 4 x 4 x 4 = 64 attempts at the deepest dependency. This is a derived worst case, also used as an example in Google's SRE guidance.
15. **Fallback is not synonymous with fake success.** A fallback must be semantically honest: stale recommendations may be acceptable; a fabricated payment confirmation is not.

### Rough-bullet expansion map

| Rough bullet | Corrected and expanded treatment |
|---|---|
| Fault tolerant system | Fault model, continued core behavior, redundancy, containment, recovery, graceful degradation, observability, and testing |
| Cascading failures | Dependency propagation, resource exhaustion, retry amplification, positive feedback, blast radius, and recovery storms |
| Rate limiter | Admission rate, Resilience4j's fixed-cycle local permissions, outbound quota protection, local-vs-global scope |
| Bulkhead | Concurrent-call isolation, noisy neighbors, semaphore vs thread-pool bulkheads, queueing, fleet-level limits |
| Time limiter | Attempt duration, end-to-end deadlines, cancellation limits, blocking vs asynchronous calls |
| Circuit breaker states | CLOSED, OPEN, HALF_OPEN, special Resilience4j states, windows, minimum samples, slow-call threshold, recovery probes |

| Fixed/exponential/jitter/custom | Formulas, thundering herd, caps, retry budgets, Retry-After, result predicates, and deadline-aware custom policies |

## 1. Plain-language foundation

### 1.1 Start with a simple story

Service A needs Service B to answer a user request:

~~~text
User -> Service A -> Service B
~~~

Normally B responds in 100 ms. One day B becomes extremely slow and takes 20 seconds.

Without protection:

1. A keeps accepting requests.
2. Every request holds an A worker thread, memory, socket, and perhaps a connection while waiting for B.
3. A's available workers run out.
4. Requests that do not even need B can no longer get a worker.
5. Users time out and clients retry.
6. The retries create even more calls to A and B.
7. A now looks unhealthy, so traffic shifts to its remaining replicas.
8. Those replicas overload too.

The original problem was "B is slow." The final incident is "A, B, and unrelated features are unavailable." That growth and spread is a **cascading failure**.

~~~text
                        +---------------------------+
                        |                           |
                        v                           |
B slows -> A waits -> A resources fill -> A slows/fails
                           |                  |
                           v                  v
                     queues/timeouts       retries
                           |                  |
                           +------ more load--+
~~~

The loop is dangerous because it is **positive feedback**: the system's reaction to trouble makes the trouble larger.

### 1.2 What "fault tolerant" actually means

A **fault** is an underlying problem: a machine crashes, a network link drops packets, a dependency becomes slow, a disk fills, or an operator deploys a bad configuration.

A **failure** is the externally visible inability to provide required behavior: checkout cannot place an order, playback cannot start, or the latency SLO is missed.

A **fault-tolerant system** is designed so that specified faults do not become unacceptable system-wide failures.

Examples:

- one application instance crashes, but a load balancer routes to healthy replicas;
- the recommendation service fails, but video playback still works;
- one tenant sends excessive traffic, but other tenants retain capacity;
- a transient network failure causes one safe retry, not a duplicate charge;
- a region fails, and the system serves a defined subset from another region.

Fault tolerance is always relative to a **fault model**:

> "The system tolerates the loss of one stateless instance" is meaningful.  
> "The system never fails" is not.

### 1.3 Reliability, availability, fault tolerance, and resilience

These terms overlap but are useful to separate:

| Term | Beginner meaning |
|---|---|
| Reliability | The system behaves correctly over time. |
| Availability | The system is ready to serve at a given time. |
| Fault tolerance | It continues required behavior despite specified faults. |
| Resilience | It absorbs disruption, limits damage, recovers, and adapts. |

Resilience is broader than putting a circuit breaker annotation on a method. A resilient design covers the full lifecycle:

~~~text
anticipate -> detect -> contain -> degrade safely -> recover -> learn
~~~

### 1.4 The ship analogy behind "bulkhead"

A ship's hull is divided into watertight compartments. If one compartment is punctured, water should not flood the whole ship.

Software bulkheads partition limited resources:

~~~text
Without isolation                    With per-dependency isolation

one shared worker pool               [pool for Payment] -> Payment
  |                                  [pool for Search ] -> Search
  +-> Payment                         [pool for Email  ] -> Email
  +-> Search
  +-> Email

Payment hangs: all features fail      Payment hangs: its partition fills;
                                      Search and Email retain capacity
~~~

The goal is not to make Payment succeed. The goal is to keep its failure from consuming everything.

### 1.5 The electrical analogy behind "circuit breaker"

An electrical breaker stops current when a circuit is unsafe. A software circuit breaker temporarily stops calls to a dependency when recent outcomes show that calls are likely to fail or be too slow.

- **CLOSED:** calls flow normally; outcomes are measured.
- **OPEN:** calls fail immediately without touching the dependency.
- **HALF_OPEN:** a small number of probe calls test recovery.

~~~text
                  threshold breached
           +-----------------------------+
           |                             v
       [CLOSED]                       [OPEN]
           ^                             |
           | probe results healthy       | wait duration passes
           |                             v
           +------------------------ [HALF_OPEN]
                                      |
                                      +-- probes unhealthy --> OPEN
~~~

"Closed" is the healthy electrical condition: the path conducts. "Open" means the path is interrupted.

### 1.6 One-line purpose of each Resilience4j pattern

| Pattern | Plain question it answers |
|---|---|
| RateLimiter | "May another call start at this rate?" |
| Bulkhead | "Is there room for another call in flight?" |
| TimeLimiter | "Is this call still worth waiting for?" |
| CircuitBreaker | "Do recent outcomes suggest this dependency is worth calling?" |
| Retry | "Is this particular failure likely and safe to try again?" |
| Fallback | "What useful, honest behavior remains if the primary path fails?" |

They are complementary controls, not five names for the same mechanism.

## 2. Required prerequisites, explained inline

These concepts are necessary for understanding the rest of the chapter, so they are introduced here rather than postponed.

### 2.1 Synchronous call and dependency graph

In a synchronous call, A waits for B before A can finish:

~~~text
User waits
  |
  v
A waits for B
             |
             v
             B waits for C
~~~

The user's availability and latency now depend on every critical edge in that call tree. A fan-out is even more sensitive:

~~~text
                         +-> Inventory
Request -> Checkout -----+-> Pricing
                         +-> Payment
                         +-> Recommendation (optional)
~~~

If Checkout requires every branch, the slowest required branch controls completion. If Recommendation is optional, Checkout should not let that optional branch destroy the core path.

### 2.2 Throughput, latency, concurrency, and Little's Law

- **Throughput:** completed/admitted work per unit time, such as requests/second.
- **Latency:** time one operation takes.
- **Concurrency:** number of operations currently in flight.
- **Capacity:** work the system can safely sustain while meeting its objectives.

For a stable flow, a useful simplified relationship is:

~~~text
average in-flight concurrency ~= throughput x average latency
~~~

**Illustrative calculation:**

~~~text
1,000 requests/s x 0.1 s = about 100 requests in flight
1,000 requests/s x 5.0 s = about 5,000 requests in flight
~~~

The incoming rate did not change. Latency did, and concurrency grew 50 times. That is why a rate limiter alone cannot protect a slow dependency: the same requests/second can consume far more sockets, threads, heap, and connections.

### 2.3 Queue and backpressure

A queue stores work waiting for execution:

~~~text
arrivals -> [bounded queue] -> [workers] -> dependency
                |
                +-> reject when full
~~~

A queue can absorb a short burst, but it does not create capacity. When arrivals stay above completions:

- queue length rises;
- queue waiting time rises;
- callers hit deadlines before work begins;
- memory grows if the queue is unbounded;
- retrying callers add more work.

**Backpressure** makes upstream producers slow, wait, or reject when downstream cannot keep up. Early, explicit rejection is often safer than an enormous queue of already-doomed requests.

### 2.4 Timeout, deadline, and cancellation

A **timeout** limits one wait, such as:

- DNS resolution timeout;
- TCP connection timeout;
- TLS handshake timeout;
- pool-acquisition timeout;
- response/read timeout;
- one Resilience4j TimeLimiter attempt.

A **deadline** is the absolute latest time the whole user operation remains useful.

~~~text
Overall request budget: 2,000 ms

gateway/auth                   100 ms
Service A local work           250 ms
network and response reserve   150 ms
remaining for attempts       1,500 ms
~~~

If each downstream layer starts a fresh two-second timeout, a two-second user request can generate much more than two seconds of hidden downstream work. **Deadline propagation** passes the remaining budget down the call tree.

Cancellation is a signal: "the result is no longer needed." It only saves resources when every layer propagates and honors it. A Java future cancellation or thread interruption cannot force arbitrary remote work or uncooperative blocking code to stop.

### 2.5 Idempotency and ambiguous outcomes

An operation is **idempotent** when repeating the same logical operation has the same intended effect as doing it once.

~~~text
GET /users/42                         normally idempotent
PUT /users/42 with a complete state   intended to be idempotent
POST /payments                        not naturally idempotent
~~~

The hardest retry case is an ambiguous outcome:

~~~text
Client ---- charge(idempotency-key=K) ----> Payment
Payment -------------------------------> DB COMMIT
Client <--------- response lost ----------- network

Client sees a timeout. Did the charge happen? It cannot tell.
~~~

For a payment-like POST:

1. Generate one stable idempotency key per logical payment.
2. Reuse that key across attempts.
3. The server atomically associates the key with the committed result.
4. A duplicate request returns the first result instead of performing the effect again.

See [13. Idempotency](./13.%20Idempotency.md) for the deeper treatment. Idempotency makes duplicate effects safe; it does not make unlimited retry traffic safe.

### 2.6 HTTP method semantics are not enough by themselves

HTTP defines PUT, DELETE, and safe methods such as GET as idempotent in their intended effects. It also says a client should not automatically retry a non-idempotent request unless it knows the request semantics are idempotent or knows the original was never applied. [RFC 9110, Idempotent Methods](https://www.rfc-editor.org/rfc/rfc9110.html#name-idempotent-methods)

Important nuance:

- A nominally idempotent GET can still be expensive and worsen overload.
- An application can make POST effectively idempotent with a correct idempotency-key contract.
- A method name cannot repair a badly implemented server endpoint.

### 2.7 Semaphore, worker thread, and thread pool

A **semaphore** is a counter of permits. A call acquires a permit before entering and releases it on completion:

~~~text
permits = 3

call 1 -> acquire -> running
call 2 -> acquire -> running
call 3 -> acquire -> running
call 4 -> no permit -> wait briefly or reject
~~~

A **thread pool** owns a bounded set of worker threads, often plus a bounded queue. Work is submitted to the pool and executes on a different thread.

The distinction matters:

- a semaphore limits concurrency with low overhead but gives no new thread isolation;
- a thread-pool bulkhead isolates blocking work onto dedicated threads but adds scheduling, context propagation, queueing, and memory overhead.

### 2.8 Sliding window

A circuit breaker needs recent evidence, not all-time history. A **sliding window** summarizes:

- the last N calls (**count-based**), or
- calls in the last N seconds (**time-based**).

It can calculate recent failure rate and slow-call rate. A minimum sample size prevents one failure at startup from immediately representing "100% failure" and opening the circuit.

## 3. Why this is needed in system design

### 3.1 Remote calls fail differently from local function calls

A local function normally returns a value or throws while the process is alive. A remote call adds partial-failure states:

- the request never left;
- it left but did not arrive;
- it arrived and is queued;
- it executed but the response was lost;
- it is only slow;
- one instance is bad while others are healthy;
- the server deliberately throttled the caller;
- an intermediary returned the error;
- the result committed after the caller timed out.

The caller often cannot distinguish these from the outside. Good resilience policy encodes what to do under uncertainty without producing duplicates or a traffic storm.

### 3.2 Shared resources connect otherwise unrelated failures

Suppose A calls Payment and Search using one common connection/worker pool:

~~~text
                +-> Payment (hung)
requests -> A --+
                +-> Search (healthy)

Shared pool:
[P][P][P][P][P][P][P][P]  all occupied by Payment
~~~

Search itself is healthy, but A cannot reach it. A logical dependency became a resource dependency. Bulkheads deliberately cut this connection.

### 3.3 Retries can turn success-oriented logic into an attack

Retries are selfishly rational: one caller increases its chance of success. Across thousands of callers, the extra load can prevent the dependency from recovering.

If a logical arrival rate is 10,000/s and every failed call makes two retries:

~~~text
initial attempts: 10,000/s
retry attempts:   up to 20,000/s
offered attempts: up to 30,000/s
~~~

This is an **illustrative upper bound** before considering backoff, deadlines, successes, and multiple layers.

Google's SRE guidance warns that retries at several layers multiply. With four total attempts at each of three layers, one user action can create 64 lowest-level attempts. It recommends randomized exponential backoff, per-request limits, server-wide retry budgets, clear retriable errors, and avoiding retries at multiple layers. [Google SRE: Addressing Cascading Failures](https://sre.google/sre-book/addressing-cascading-failures/)

### 3.4 Recovery is often weaker than steady state

A restarted service can have:

- cold caches;
- new connection pools;
- runtime/JIT warm-up;
- replication/log replay;
- only a fraction of replicas healthy;
- a large queued backlog.

Opening the traffic floodgate immediately can knock it down again:

~~~text
restart -> small health/probe traffic -> warm -> gradual ramp -> normal share
~~~

Circuit-breaker HALF_OPEN probes, load-balancer slow start, jitter, and admission control all protect this recovery path.

### 3.5 Autoscaling and redundancy are not complete answers

Autoscaling has measurement, provisioning, startup, and warm-up delay. It may also scale the caller faster than the bottlenecked database. Redundancy can make matters worse if traffic from a failed replica instantly overloads survivors.

Resilience therefore needs both:

~~~text
capacity/redundancy + demand control/isolation
~~~

## 4. A fault-tolerance toolbox, not a single pattern

### 4.1 Defense-in-depth view

~~~text
Client request
    |
    v
[overall deadline]
    |
    v
[admission/rate limit] -- no permission --> reject/defer
    |
    v
[circuit breaker] ----- OPEN -----------> fast failure/fallback
    |
    v
[bulkhead] ------------ full -----------> fast failure/fallback
    |
    v
[per-attempt timeout] -> exceeded ------> cancel signal/failure
    |
    v
Downstream dependency
    |
 transient + safe + budget remains
    v
[bounded retry with backoff + jitter]
    |
 exhausted/permanent
    v
[honest fallback or explicit failure]
~~~

The exact nesting changes semantics and must be chosen deliberately; Section 10 analyzes it.

### 4.2 Pattern comparison

| Pattern | Primary signal | Primary resource protected | Typical failure response | What it does not solve |
|---|---|---|---|---|
| Rate limiter | Calls per time period | Dependency/API throughput quota | reject or wait | Long in-flight calls, global coordination by default |
| Bulkhead | Current concurrency/queue capacity | threads, connections, dependency capacity | reject or briefly wait | Whether the dependency is healthy |
| Time limiter | Elapsed time | caller wait and resource holding time | timeout/cancel signal | Guaranteed remote cancellation |
| Circuit breaker | Recent failure/slow-call rate | caller and unhealthy dependency | short-circuit | Concurrent overload while CLOSED |
| Retry | Classified transient result/exception | availability opportunity | another attempt later | Persistent failure, duplicate effects, retry storms |
| Fallback | Primary path unavailable | user-visible functionality | stale/default/partial/queued/error | Truthfulness and business correctness automatically |

### 4.3 What Resilience4j is

Resilience4j is a lightweight Java library that decorates functions/calls with independent modules such as CircuitBreaker, Retry, RateLimiter, Bulkhead, and TimeLimiter. It integrates with Spring Boot, Reactor, metrics, and other frameworks. [Resilience4j official repository](https://github.com/resilience4j/resilience4j)

It is useful because it provides:

- thread-safe state machines and counters;
- reusable configuration and named registries;
- annotations and functional decorators;
- event streams and Micrometer/Actuator integration;
- consistent exceptions for rejection/short-circuit/timeout.

### 4.4 What Resilience4j is not

It does not by itself provide:

- multi-region redundancy or failover;
- a distributed global rate-limit counter;
- a durable idempotency-key store;
- message durability or a dead-letter queue;
- end-to-end deadline propagation across protocols;
- guaranteed cancellation of remote work;
- capacity planning;
- correct business fallbacks;
- service recovery or incident response.

Think of it as **client-side traffic and failure control around one call edge**, inside a larger resilient architecture.

### 4.5 Instance scope is a design decision

Use a distinct named protection instance for each remote backend or materially different call class:

~~~text
inventory.read
inventory.reserve
payment.authorize
payment.refund
recommendation.home
~~~

Sharing one circuit breaker across Inventory and Payment lets Inventory failures open Payment's circuit. Sharing one bulkhead lets a slow optional dependency take capacity from a critical one. The Resilience4j repository explicitly recommends unique instances for protected backends so metrics, configuration, and failure isolation remain independent. [Resilience4j instance-management guidance](https://github.com/resilience4j/resilience4j#41-instance-management-when-to-share-vs-not-share-instances)

Do not create unbounded per-user circuit-breaker instances, however; that produces high-cardinality state and metrics. Instance granularity should correspond to a bounded operational protection domain.

## 5. Rate limiter

### 5.1 Plain-language purpose

A rate limiter controls how quickly calls may begin:

~~~text
allowed: 10 calls each second

time:    | second 1 | second 2 |
permits: |    10    |    10    |
~~~

Typical caller-side use:

- an external vendor allows 100 requests/second;
- one downstream can safely accept a configured amount from this caller;
- background work must not overwhelm an online API;
- one local feature needs a coarse protective cap.

### 5.2 Resilience4j behavior

**Verified behavior:** Resilience4j documents two in-memory implementations:

- an AtomicRateLimiter that divides time into cycles and updates immutable state atomically;
- a SemaphoreBasedRateLimiter that uses a semaphore and scheduler to refresh permissions.

The key settings are:

| Setting | Meaning |
|---|---|
| limitForPeriod | Permissions created for each refresh period |
| limitRefreshPeriod | Duration of one refresh cycle |
| timeoutDuration | How long a caller may wait to acquire permission |

[Official Resilience4j RateLimiter guide](https://resilience4j.readme.io/docs/ratelimiter)

Important corrections:

- This is a fixed-cycle permission model, not necessarily a token bucket.
- A nonzero timeoutDuration can make callers wait; it is **not** the downstream response timeout.
- The registry/state is normally inside one JVM.

### 5.3 Local versus distributed rate

Suppose there are 20 caller instances and each local limiter allows 100 calls/s:

~~~text
theoretical fleet admission <= 20 x 100 = 2,000 calls/s
~~~

That is a **derived ceiling** under the simple independent-limit assumption, not a guaranteed measured rate. If the vendor quota is 100 calls/s for the whole fleet, a local limit of 100 on every replica is wrong.

Options for a fleet-wide quota:

- enforce centrally at an API gateway or rate-limit service;
- allocate/lease sub-budgets to instances;
- designate a key owner/shard;
- accept bounded overshoot with conservative local limits;
- use the vendor's own 429 feedback as a final guard, not as the only control.

See [12. Rate Limiter](./12.%20Rate%20Limiter.md) for distributed algorithm depth.

### 5.4 Fixed-cycle boundary burst

With 10 permissions per second, 10 calls can arrive at the end of one cycle and 10 more at the start of the next:

~~~text
cycle 1                      cycle 2
---------|-------------------|---------
         10 calls            10 calls
         <--- nearly 20 calls in a tiny interval --->
~~~

This near-2L boundary burst is a **derived property** of fixed windows/cycles. If the downstream needs smooth pacing, use a smaller refresh period, token/leaky-bucket-style control, or a bounded worker queue with a controlled drain rate.

### 5.5 Rate limiter versus bulkhead

Consider 100 calls/s:

- at 10 ms each, average concurrency is about 1;
- at 5 seconds each, average concurrency is about 500.

The rate is identical. The resource pressure is not. Rate limit to respect throughput quotas; bulkhead/concurrency-limit to protect in-flight capacity. Often use both.

### 5.6 Use and avoid

Use a Resilience4j RateLimiter when:

- a local Java call edge needs a known per-instance rate;
- a caller must pace an outbound integration;
- a short wait or immediate rejection is acceptable.

Do not treat it as sufficient when:

- the quota is globally exact across replicas/regions;
- request costs differ greatly;
- the main problem is latency-driven concurrency;
- waiting for a permit would consume the entire user deadline.

## 6. Bulkhead and the noisy-neighbor problem

### 6.1 What is a noisy neighbor?

A noisy neighbor is one tenant, feature, dependency, partition, or workload that consumes disproportionate shared capacity and harms others.

~~~text
Shared capacity: 10 slots

Tenant A: [A][A][A][A][A][A][A][A][A][A]
Tenant B: no slot
Tenant C: no slot
~~~

The "neighbor" does not have to be malicious. It might have:

- a sudden legitimate spike;
- an expensive query pattern;
- a retry bug;
- a large customer's batch import;
- one slow downstream shard.

### 6.2 Isolation dimensions

Bulkheads can be created by:

- downstream dependency;
- endpoint or operation class;
- critical vs optional traffic;
- tenant or tenant tier;
- synchronous vs batch work;
- region, cell, queue, shard, process, container, thread pool, connection pool, or semaphore.

The right boundary matches the blast radius you are willing to accept.

~~~text
Critical checkout calls -> reserved capacity -> Payment
Batch reconciliation    -> separate capacity -> Payment
~~~

If batch traffic fills its partition, checkout retains its reservation.

### 6.3 Semaphore bulkhead

Resilience4j's SemaphoreBulkhead limits concurrent executions with permits:

~~~text
Caller thread -> acquire permit -> invoke dependency on same execution path
                    |
                    +-> no permit: reject or wait up to maxWaitDuration
~~~

Advantages:

- low overhead;
- no extra queue/thread hop;
- suitable for many synchronous, asynchronous, or nonblocking models when the permit is held until actual completion;
- simple concurrency cap.

Costs and limitations:

- no separate thread isolation;
- a blocking dependency still blocks the caller's thread;
- cancellation/timeouts must be provided separately;
- a long maxWaitDuration creates another queue of waiting caller threads;
- incorrect reactive decoration can release the permit when a publisher is created rather than when work actually completes; use the supported Reactor integration and test the lifecycle.

### 6.4 Thread-pool bulkhead

Resilience4j's ThreadPoolBulkhead uses a fixed/bounded thread pool and bounded queue:

~~~text
caller -> [bounded queue] -> [dedicated threads] -> dependency
              |
              +-> full: reject
~~~

Advantages:

- blocking or poorly behaved client code is isolated from the caller's main worker pool;
- per-dependency threads and queue expose saturation clearly;
- caller can receive an asynchronous CompletionStage.

Costs:

- queueing and context-switch overhead;
- memory for queued work;
- queue delay can consume deadlines;
- logging/security/trace context must cross threads correctly;
- too many per-dependency pools waste threads and memory;
- nested pools can deadlock/starve if they wait on each other;
- it is often a poor fit for a truly nonblocking event-loop pipeline;
- virtual threads reduce thread cost but do not create downstream database connections or capacity, so concurrency still needs bounding.

Resilience4j officially provides SemaphoreBulkhead and a fixed thread-pool bulkhead with a bounded queue. [Official Bulkhead guide](https://resilience4j.readme.io/docs/bulkhead)

### 6.5 Comparison

| Question | Semaphore bulkhead | Thread-pool bulkhead |
|---|---|---|
| Executes on caller thread? | Usually yes | No, dedicated worker |
| Adds queue? | Optional permit wait; no work queue | Bounded work queue |
| Thread isolation? | No | Yes |
| Overhead | Lower | Higher |
| Best fit | trusted/fast client, async or nonblocking pipeline, concurrency cap | blocking/untrusted library, explicit worker isolation |
| Main risk | caller threads remain blocked | queue latency, thread/context overhead |

### 6.6 Sizing a bulkhead

A starting estimate uses healthy throughput and latency:

~~~text
needed concurrency ~= target throughput x healthy service time
~~~

Then constrain it by:

- downstream's tested safe concurrency;
- connection-pool size;
- per-instance traffic share and load skew;
- the end-to-end deadline;
- headroom for normal variance;
- priority reservations;
- total fleet concurrency.

**Illustrative example only:**

~~~text
per-instance target = 200 calls/s
healthy p95-ish service time used for planning = 0.15 s
base estimate = 200 x 0.15 = 30 in flight
~~~

Do not turn 30 into a default. Load-test the real request-cost distribution. If 20 caller replicas each use 30, the downstream can see up to roughly 600 simultaneous calls under even distribution, and potentially uneven pressure by shard.

### 6.7 Common noisy-neighbor mistake

One shared bulkhead called "backend" is not isolation:

~~~text
All dependencies -> [one bulkhead] -> everything
~~~

At minimum, isolate independent backends and criticality classes. For high-cardinality tenants, use a bounded number of cells/shards, hierarchical limits, or heavy-hitter isolation rather than one in-memory object per arbitrary user.

Microsoft's architecture guidance describes bulkheads at connection-pool, process/container, queue, service-instance, and tenant boundaries, while noting the utilization and management cost of partitions. [Azure Bulkhead pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/bulkhead)

## 7. Time limiter, timeouts, and deadlines

### 7.1 Plain-language purpose

A TimeLimiter says:

> "If this asynchronous work has not produced a result by T, stop making this caller wait and treat the attempt as timed out."

Resilience4j's TimeLimiter decorates a Future or CompletionStage and configures:

- timeoutDuration;
- whether cancellation should be requested on the running Future.

[Official Resilience4j TimeLimiter guide](https://resilience4j.readme.io/docs/timeout)

### 7.2 It is not every network timeout

A remote call can wait in several phases:

~~~text
DNS -> TCP connect -> TLS -> connection-pool acquire -> request write
    -> server queue/work -> response headers -> response body read
~~~

Configure the underlying HTTP/RPC client too. An outer TimeLimiter cannot precisely identify which phase is stuck, and timing out the Future does not necessarily close a socket or roll back server work.

### 7.3 Attempt timeout versus overall deadline

Suppose:

~~~text
overall budget = 1,000 ms
max attempts   = 3
attempt timeout = 700 ms
backoff = 100 ms, then 200 ms
~~~

Worst-case schedule:

~~~text
attempt 1: 700
sleep 1:   100
attempt 2: only 200 ms remain
attempt 3: impossible
~~~

Starting every attempt with a fresh 700 ms would violate the user budget. Before every attempt:

~~~text
remaining = overallDeadline - now

if remaining <= minimumUsefulAttemptTime + responseReserve:
    do not retry
else:
    attemptTimeout = min(configuredAttemptTimeout, remaining - reserve)
~~~

### 7.4 Timeout does not equal cancellation

~~~text
caller -------- request --------> server
caller -- timeout at 500 ms
server -------- commits at 800 ms
~~~

The caller has stopped waiting, but the effect happened. Safe handling needs:

- idempotency/deduplication for repeated effects;
- protocol cancellation propagation where supported;
- database/query timeout;
- HTTP client socket/request cancellation;
- downstream deadline checks before expensive stages;
- observability for "completed after caller timeout."

### 7.5 Choosing timeouts

Do not choose "2 seconds because it sounds safe." Start from:

1. the user's end-to-end latency objective;
2. measured dependency latency distribution by operation and payload;
3. acceptable false-timeout rate;
4. remaining budget after upstream/local work;
5. cold connection/handshake behavior;
6. retry count and backoff;
7. downstream's own server-side deadline.

Very long timeouts allow resource accumulation. Very short timeouts create false failures and retry load. Google's SRE chapter shows how a small fraction of very long requests can exhaust a worker fleet and stresses deadline/cancellation propagation. [Google SRE: Latency and Deadlines](https://sre.google/sre-book/addressing-cascading-failures/#latency-and-deadlines)

## 8. Circuit breaker in depth

### 8.1 CLOSED

In CLOSED:

- calls are allowed;
- results are recorded in the sliding window;
- classified failures and slow calls contribute to rates;
- the breaker opens only after the minimum number of calls is present and a configured threshold is met.

~~~text
last window:
S S F S F F S F S F

failure rate = 5 / 10 = 50%
~~~

Whether 50% should open the circuit depends on configuration and the operation. The number is illustrative.

**Common mistake:** CLOSED does not mean "B is definitely healthy." It means the breaker currently permits calls.

### 8.2 OPEN

In OPEN:

- the protected function is not called;
- callers get a fast CallNotPermittedException or a configured fallback;
- threads/connections are not spent waiting for the known-bad path;
- the dependency receives relief.

OPEN is a deliberate failure mode. It exchanges a slow, resource-consuming, likely failure for a fast, bounded, observable failure.

### 8.3 HALF_OPEN

After the open wait duration, the breaker allows only a configured number of probe calls:

~~~text
many incoming calls
     |
     +--> probe 1 -> dependency
     +--> probe 2 -> dependency
     +--> probe 3 -> dependency
     +--> remaining calls rejected
~~~

- healthy probe rates -> CLOSED;
- unhealthy failure/slow-call rates -> OPEN again.

HALF_OPEN protects a recovering dependency from receiving the full backlog. In a fleet, however, every caller instance has its own local circuit. If 500 pods probe at once, "three probes per pod" can still mean 1,500 calls. This is a **derived fleet example** and motivates jittered recovery, load-balancer slow start, shared downstream admission control, and conservative probe counts.

### 8.4 Failure-rate and slow-call thresholds

A service can be harmful before it returns errors. If most calls take nearly the full timeout, caller resources are already filling. Resilience4j can open based on:

- failure rate;
- slow-call rate, where "slow" means exceeding a configured duration.

This is verified in the official state-machine guide. [Resilience4j CircuitBreaker guide](https://resilience4j.readme.io/docs/circuitbreaker)

Classify outcomes carefully:

- timeout/connection failures: usually record;
- 502/503/504: commonly record when mapped as transient dependency failure;
- local 400 validation errors: normally do not blame the dependency;
- BulkheadFullException: indicates local saturation, not necessarily remote failure;
- RequestNotPermitted/CallNotPermitted: do not recursively count as a new remote failure;
- fallback success: do not let it hide the primary failure from breaker metrics.

### 8.5 Count-based versus time-based window

| Window | Good for | Risk |
|---|---|---|
| Last N calls | stable sample size independent of traffic rate | at high traffic it represents a very short time; at low traffic old outcomes linger |
| Last N seconds | consistent recency interval | low traffic may not reach minimumNumberOfCalls |

Tune the minimum sample with the window. Without it:

- tiny samples cause false opens;
- huge samples react too slowly;
- a threshold with no traffic-volume context is meaningless.

### 8.6 Circuit breaker does not cap parallel calls

Resilience4j explicitly states that the protected function call is not synchronized. If 20 callers arrive while CLOSED, all 20 may execute even if the sliding window size is 15. Use a Bulkhead to limit parallel execution. [Resilience4j CircuitBreaker concurrency note](https://resilience4j.readme.io/docs/circuitbreaker)

This is a frequent interview trap:

~~~text
slidingWindowSize = 100
~~~

means "measure the last 100 outcomes," not "allow 100 concurrent calls."

### 8.7 Special Resilience4j states

Beyond the three normal states, Resilience4j documents:

- **METRICS_ONLY:** allow calls and record metrics but do not open on thresholds;
- **DISABLED:** allow calls without normal breaker metrics/events;
- **FORCED_OPEN:** reject calls until manually transitioned/reset.

Useful operational applications:

- METRICS_ONLY for a shadow rollout;
- FORCED_OPEN as an emergency kill switch;
- DISABLED for controlled troubleshooting.

Do not make manual states the normal recovery mechanism. They need authorization, audit, alerting, and a rollback plan.

### 8.8 Circuit breaker versus health check

A health check answers whether an instance/process should receive traffic. A caller-side breaker answers whether **this caller's recent experience of this operation** justifies making another call.

They differ because:

- one endpoint or shard may fail while the process is alive;
- a caller may have a network-path problem other callers do not;
- making the whole application "DOWN" whenever one optional dependency's circuit opens can cause restarts or load-balancer removal, worsening the incident.

Resilience4j's Spring guide disables circuit-breaker health indicators by default because mapping OPEN to application DOWN may not be the desired behavior. [Resilience4j health endpoint guide](https://resilience4j.readme.io/docs/getting-started-3)

### 8.9 When to avoid or reconsider

A circuit breaker adds state, delayed recovery, tuning, and false-positive risk. It is less useful for:

- a pure in-process function that fails immediately without scarce resources;
- a call whose every failure is permanent per request rather than dependency health;
- very low traffic where a rate-based window cannot collect useful evidence;
- a path where a load balancer/service mesh already provides the desired ejection and application-specific fallback is unnecessary.

Even then, timeouts and resource bounds may still be necessary.

## 9. Retry in depth

### 9.1 Why retry exists

Distributed systems experience transient faults:

- one lost packet;
- a brief leader election;
- a connection closed during deployment;
- one unhealthy replica selected by a load balancer;
- a short overload period;
- a temporary DNS resolver/server failure.

The same logical request may succeed moments later. Microsoft's Retry pattern guidance recommends retries for short-lived transient faults and warns against them for long-lasting or deterministic business/code failures. [Azure Retry pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/retry)

### 9.2 The "work X before calling B" point

The rough bullet is correct only with the right retry boundary.

#### Good scope when X is safely reusable

~~~text
x = expensivePureComputation(request)

retryOnlyRemoteCall:
    call B with x
~~~

If the first call to B fails, the retry reuses x.

#### Wrong scope

~~~text
retryWholeMethod:
    x = expensiveComputation(request)
    writeAuditRow()
    call B with x
~~~

Now every attempt recomputes x and might repeat writeAuditRow.

#### Reuse is safe only if

- X is pure or its side effects are separately idempotent;
- X remains valid across the retry interval;
- X is not tied to a transaction/lock that has already ended;
- credentials/nonces/timestamps in X have not expired;
- the retry is still for the same logical operation;
- memory cost of retaining X is acceptable.

A retry library re-executes the decorated function. It does not infer which earlier work should be cached.

### 9.3 The retry decision: five gates

Retry only if **all** gates pass:

~~~text
1. Transient?       Could another attempt plausibly succeed soon?
2. Safe?            Is duplicate execution safe/deduplicated?
3. Budget?          Are attempts/retry tokens still available?
4. Deadline?        Is enough useful time left?
5. Load-aware?      Will delay/jitter avoid worsening overload?
~~~

If one answer is no, fail, fall back, queue for later, or require explicit reconciliation.

### 9.4 Corrected HTTP and transport matrix

This table is a starting policy, not a substitute for the downstream API contract.

| Signal | Blind identical retry? | Better reasoning/action |
|---|---|---|
| Local validation failure | No | Fix request/code; permanent for unchanged input |
| DNS NXDOMAIN / invalid hostname | Usually no | Likely configuration/permanent; distinguish from transient resolver SERVFAIL |
| TLS certificate/hostname error | No | Security/configuration failure |
| Connection failed before request was sent | Often conditionally | Backoff, alternate healthy endpoint, deadline and budget; non-idempotent ambiguity is lower but prove send phase |
| Connection/read timeout after send | Only if idempotent/deduplicated | Server may have executed; outcome is ambiguous |
| 400, 405, 410, 422 | Normally no | Unchanged request will usually fail again |
| 401 | Not the same blind request | Refresh credentials once if contract allows, then retry with changed authorization |
| 403 | Normally no | Permission/policy failure; retrying is usually noise |
| 404 | Normally no | Only poll/retry if API explicitly has eventual-creation semantics |
| 408 Request Timeout | Conditionally yes | RFC 9110 says an outstanding request may be repeated; still require safety/budget |
| 409 Conflict / 412 Precondition Failed | Not blindly | Fetch current version, resolve/recompute, then issue a new conditional request |
| 425 Too Early | Protocol-specific retry | Retry without TLS early data as specified; not a generic app retry rule |
| 429 Too Many Requests | Yes, delayed and bounded | Honor Retry-After/rate contract; add spreading; do not hammer immediately |
| 500 Internal Server Error | Maybe | Could be transient or deterministic bug; use API knowledge and safe semantics |
| 501 Not Implemented | No | Capability is absent; retrying unchanged call is pointless |
| 502 Bad Gateway | Often conditionally | Intermediary/upstream transient possibility; idempotency, backoff, budget |
| 503 Service Unavailable | Often conditionally | Temporary overload/maintenance; honor Retry-After |
| 504 Gateway Timeout | Only if idempotent/deduplicated | Upstream outcome can be ambiguous |

RFC 9110 defines idempotent-method retry constraints, says 408 requests may be repeated, and describes 503 as temporary overload/maintenance that may include Retry-After. [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html) RFC 6585 defines 429 and says it may include Retry-After. [RFC 6585, 429 Too Many Requests](https://www.rfc-editor.org/rfc/rfc6585.html#section-4)

The interview-quality answer is therefore:

> "I do not retry by status class alone. I classify the exact failure, determine whether the server may have applied the operation, require idempotency or a stable dedupe key for ambiguous effects, honor server guidance, and check attempt, deadline, and load budgets."

### 9.5 Maximum attempts: vocabulary trap

Resilience4j's v2.2.0 Retry guide defines maxAttempts as including the initial call. Thus:

~~~text
maxAttempts = 3
means
1 initial attempt + at most 2 retries
~~~

[Resilience4j Retry configuration](https://resilience4j.readme.io/docs/retry)

Other SDKs use "max retries" to mean retries after the initial attempt. State the convention in configuration reviews and metrics.

### 9.6 Fixed interval

~~~text
sleep(k) = constant
~~~

Advantages:

- simple;
- predictable;
- reasonable for one isolated client and a known short recovery interval.

Risks:

- clients that fail together retry together;
- constant pressure continues during a long outage;
- can align with rate-limit/window boundaries.

~~~text
failure at t=0
all clients retry at t=1, t=2, t=3 ...
                 ^    ^    ^
                 synchronized spikes
~~~

### 9.7 Capped exponential backoff

Let k = 0 for the first retry after the initial attempt:

~~~text
cap(k) = min(maxDelay, baseDelay x 2^k)
sleep(k) = cap(k)
~~~

**Illustrative sequence:**

~~~text
base = 100 ms, max = 2 s
100, 200, 400, 800, 1600, 2000, 2000 ... ms
~~~

Benefits:

- rapidly reduces pressure;
- gives longer faults more recovery time;
- caps unbounded delay.

But identical clients still schedule identical 100/200/400 ms waves.

### 9.8 Exponential backoff plus jitter

One common form is **full jitter**:

~~~text
cap(k)   = min(maxDelay, baseDelay x 2^k)
sleep(k) = Uniform(0, cap(k))
~~~

This spreads retry attempts across each growing range.

Other variants:

- equal jitter: retain part of the backoff and randomize the rest;
- decorrelated jitter: make the next range depend on the previous random delay;
- deterministic per-host jitter: useful for recurring work that should differ across a fleet but remain reproducible.

AWS's first-party simulation showed that capped exponential backoff still produced clusters, while jitter spread calls and reduced work in that simulated contention workload. Its exact measured reductions are simulation-specific, not universal production guarantees. [AWS: Exponential Backoff and Jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/)

### 9.9 Retry-After and custom logic

Custom retry logic can consider:

- exception/status/result body;
- method and idempotency key;
- attempt number;
- remaining deadline;
- server Retry-After;
- retry token budget;
- dependency/circuit state;
- request priority and cost;
- failover endpoint;
- whether an earlier attempt is still running.

If Retry-After is a minimum wait, a useful policy shape is:

~~~text
sleep = max(serverMinimumDelay, clientJitteredBackoff)
~~~

Then cap it by the remaining deadline. Do not apply jitter that makes the client retry **before** the server's minimum.

Resilience4j supports predicates on results/exceptions and interval functions/bi-functions based on attempt plus result or exception. It also exposes fixed, exponential, randomized, and custom interval mechanisms. Exact factory/property behavior should be verified for the pinned version. [Resilience4j Retry guide](https://resilience4j.readme.io/docs/retry)

### 9.10 Retry budget

Maximum attempts protects one logical request. A **retry budget** protects the whole caller/service:

~~~text
normal traffic succeeds -> slowly replenish retry tokens
retry attempt           -> consume a token
no token                -> fail/fallback; do not retry
~~~

This prevents a widespread outage from turning every request into its maximum retry sequence. Budgets can be:

- fixed retries per second;
- a percentage of original traffic;
- token bucket replenished by successes;
- per dependency/priority/tenant.

Do not let low-priority retries consume capacity needed for new critical requests.

### 9.11 Retry at one layer

~~~text
Browser -> Gateway -> Service A -> Service B -> Database
~~~

Choose the layer with:

- the best semantic knowledge;
- the idempotency contract;
- the end-to-end deadline;
- the ability to classify failures;
- the closest sensible failover choice.

Avoid independent retries at every layer. If a database driver already retries, Service B, A, the gateway, and the mobile app must account for it in one attempt budget.

### 9.12 Retry alternatives

| Situation | Better than an immediate synchronous retry |
|---|---|
| Long outage | circuit open + fallback/failure |
| Slow, noncritical work | durable queue and later processing |
| Non-idempotent ambiguous effect | lookup/reconciliation by idempotency key |
| Known bad instance | load-balancer failover/ejection |
| User can tolerate stale data | cache/stale read |
| Tail latency with healthy replicas | carefully bounded hedged request, with cancellation and idempotency |
| Permanent validation/business failure | return explicit error |

Hedging is not ordinary retry: it sends a duplicate before the first fails to reduce tail latency. It consumes extra load and must be rare, idempotent, and cancelable.

## 10. Combining the patterns: order changes behavior

### 10.1 There is no universally correct annotation stack

Consider these two conceptual orders.

#### Retry outside CircuitBreaker

~~~text
Retry(
  CircuitBreaker(
    remoteCall
  )
)
~~~

Each attempt passes through the breaker:

- the breaker sees every attempt;
- it can open during one logical operation;
- breaker metrics count attempts;
- Retry must not retry CallNotPermittedException forever.

#### CircuitBreaker outside Retry

~~~text
CircuitBreaker(
  Retry(
    remoteCall
  )
)
~~~

The breaker sees the final logical result after internal attempts:

- one logical request contributes one breaker result;
- persistent failure is detected more slowly in raw attempts;
- retry activity is hidden from breaker call counts;
- all retries may occur before the circuit can open.

Neither is automatically correct. State what the breaker is intended to measure: individual dependency attempts or final logical operations.

### 10.2 A reasonable conceptual envelope

For a synchronous, idempotent read, one defensible starting design is:

~~~text
Fallback(
  OverallDeadline(
    BoundedRetry(
      RatePermitPerAttempt(
        CircuitBreaker(
          Bulkhead(
            PerAttemptTimeout(
              RemoteCall
            )
          )
        )
      )
    )
  )
)
~~~

Why this shape:

- fallback is outside so the primary failure remains visible to inner metrics;
- one overall deadline bounds attempts plus sleeps;
- each retry consumes admission permission;
- every attempt checks circuit state;
- the bulkhead is held only during an attempt, not during backoff sleep;
- each attempt has a timeout.

This still needs exception classification:

- do not count RateLimiter rejection as remote failure;
- do not count Bulkhead rejection as remote failure unless that is intentionally the breaker's signal;
- do not retry circuit-open or local-overload exceptions unless a policy explicitly makes sense;
- do not let the fallback make the primary path appear healthy.

Different goals justify different nesting. For example, a global rate permit outside Retry means one logical operation consumes one permit and its retries bypass the rate cap - often unsafe for protecting a vendor quota.

### 10.3 Spring annotation default order

The Resilience4j Spring guide documents this default aspect nesting:

~~~text
Retry(
  CircuitBreaker(
    RateLimiter(
      TimeLimiter(
        Bulkhead(
          Function
        )
      )
    )
  )
)
~~~

It also documents properties for changing aspect priority and recommends functional chaining when a different order is required. [Resilience4j aspect order](https://resilience4j.readme.io/docs/getting-started-3)

Consequences of the documented default:

- Retry is outermost, so inner protections are re-entered per attempt.
- TimeLimiter is per attempt rather than an end-to-end budget unless another outer deadline exists.
- Bulkhead is released before the retry sleeps, which is generally desirable.
- RateLimiter permission is acquired per attempt.

Do not stack annotations by visual order and assume that is runtime nesting. Write an integration test that asserts attempt count, metric count, permit use, timeout behavior, and fallback selection.

### 10.4 Fallback rules

A good fallback is:

- cheaper and more reliable than the primary;
- independent of the failed dependency, or protected separately;
- bounded by its own deadline/resources;
- semantically honest;
- observable as degraded service;
- appropriate to data freshness and correctness.

Examples:

| Primary failure | Possible fallback | Dangerous fallback |
|---|---|---|
| Recommendations | popular items or omit rail | another unprotected remote recommender |
| Product reviews | cached/stale reviews with age | report "no negative reviews" |
| Email receipt | durable queue for later | fail a committed purchase |
| Payment authorization | explicit pending/failure + reconciliation | claim payment succeeded |
| Surge pricing | business-approved degraded policy | silently invent an arbitrary price |

Fallbacks can also overload. If every primary failure reads one shared fallback database, the fallback becomes the next cascade.

## 11. Resilience4j implementation notes for Spring/Java

This section is implementation awareness for HLD interviews, not a version-pinned copy-paste recipe.

### 11.1 Modules and configuration shape

The Spring Boot starter supports named CircuitBreaker, Retry, RateLimiter, Bulkhead, ThreadPoolBulkhead, and TimeLimiter instances in application configuration. Actuator/AOP are expected runtime integrations; Reactor integration is needed for reactive use. [Official Spring Boot setup](https://resilience4j.readme.io/docs/getting-started-3)

An **illustrative configuration shape**:

~~~yaml
resilience4j:
  circuitbreaker:
    instances:
      inventoryRead:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 50
        minimumNumberOfCalls: 20
        failureRateThreshold: 50
        slowCallDurationThreshold: 500ms
        slowCallRateThreshold: 60
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3

  retry:
    instances:
      inventoryRead:
        maxAttempts: 3
        waitDuration: 100ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2

  bulkhead:
    instances:
      inventoryRead:
        maxConcurrentCalls: 30
        maxWaitDuration: 0

  ratelimiter:
    instances:
      inventoryRead:
        limitForPeriod: 100
        limitRefreshPeriod: 1s
        timeoutDuration: 0

  timelimiter:
    instances:
      inventoryRead:
        timeoutDuration: 400ms
        cancelRunningFuture: true
~~~

Every value above is **illustrative only**. The retry snippet shows exponential backoff but not a proven full-jitter distribution. For production, use the pinned version's supported randomized/custom interval function, incorporate the overall deadline, and test the exact delay distribution.

### 11.2 Exception and result classification

Do not use broad defaults such as "retry every Exception" or "count every throwable as breaker failure."

Define domain-aware categories:

~~~text
ValidationException            -> no retry, not breaker failure
AuthenticationException        -> refresh once or fail, not dependency-health failure
ConnectionException            -> conditional retry, breaker failure
DependencyTimeoutException     -> conditional retry, breaker slow/failure
HTTP 503                       -> conditional retry, breaker failure
HTTP 429                       -> delayed retry/rate feedback, usually not service failure
BulkheadFullException          -> local overload, shed/fallback
CallNotPermittedException      -> circuit already open, do not hammer retry
~~~

The exact mapping depends on the HTTP/RPC client because many clients turn status codes into exception subclasses.

### 11.3 Functional decoration versus annotations

Annotations are concise. Functional decoration makes scope and order explicit.

~~~text
preparedX = computeXOnce()

decoratedRemoteAttempt =
    timeLimiter(
      bulkhead(
        circuitBreaker(
          () -> callB(preparedX)
        )
      )
    )

result = retry(decoratedRemoteAttempt)
~~~

This is conceptual pseudocode. It highlights that computeXOnce is outside the retried function.

Prefer explicit decoration when:

- order is business-critical;
- different attempts need changing timeouts;
- Retry-After/result-aware logic is needed;
- one overall deadline spans all attempts;
- annotation proxy behavior is hard to see.

### 11.4 Spring AOP self-invocation trap

Resilience4j annotations are implemented through Spring AOP proxies. If one method calls another annotated method on the same object through this, the call bypasses the proxy and therefore the advice. Spring's official documentation explicitly describes this self-invocation behavior. [Spring AOP proxying](https://docs.spring.io/spring-framework/reference/core/aop/proxying.html)

~~~java
class CheckoutService {
    void outer() {
        inventoryCall(); // same-object call can bypass proxy advice
    }

    @CircuitBreaker(name = "inventoryRead")
    void inventoryCall() { ... }
}
~~~

Typical repair: move the protected client method into a separate injected bean or use explicit functional decoration. Also test final/private/proxy visibility constraints for the selected Spring version.

### 11.5 Async/reactive lifecycle trap

For an asynchronous call, the protection must wrap the **completion signal**, not merely the creation of a Future/Mono:

~~~text
wrong lifetime: acquire -> create async object -> release -> remote work still running
right lifetime: acquire -> subscribe/start -> remote completion/error -> release
~~~

Use the library's supported integration for CompletionStage/Reactor and verify concurrency metrics during real in-flight work.

### 11.6 Metrics and events

Resilience4j publishes events and integrates with Actuator/Micrometer. The official Spring guide shows circuit call/state/failure metrics and tags such as successful, failed, and not permitted. [Resilience4j metrics guide](https://resilience4j.readme.io/docs/getting-started-3)

Keep instance names bounded. Never tag metrics with raw request IDs, idempotency keys, or unlimited tenant IDs.

## 12. End-to-end worked example: Checkout A calls Inventory B

### 12.1 Requirements

Assume:

- Checkout computes a price/tax payload X.
- It calls Inventory to reserve stock.
- Reservation uses a stable reservation operation ID.
- The user deadline is 1.5 seconds.
- Inventory sometimes has short transient network failures.
- Checkout must not claim success without a confirmed reservation.

All numbers are illustrative.

### 12.2 Naive design

~~~text
Checkout:
  compute X
  call Inventory with no timeout
  on any error, retry 10 times immediately
~~~

Problems:

- Checkout workers can block indefinitely.
- compute X might repeat if retry wraps the whole method.
- immediate retries amplify an outage.
- duplicate reservations occur without an idempotency key.
- no concurrency isolation exists.
- users can wait longer than the request deadline.
- optional Checkout features share the same resources.

### 12.3 Protected design

~~~text
User
  |
  | overall deadline = 1.5s
  v
Checkout
  |
  +-- compute X once (pure/reusable)
  +-- reservationId = stable logical id
  |
  +-- Retry: max 3 total attempts, only classified transient failures
        |
        +-- check remaining deadline and retry budget
        +-- jittered capped backoff
        +-- outbound rate permit per attempt
        +-- Inventory circuit breaker
        +-- Inventory semaphore bulkhead
        +-- per-attempt timeout
        +-- reserve(reservationId, X)
  |
  +-- confirmed -> commit Checkout outcome
  +-- ambiguous -> query/reconcile by reservationId; do not invent success
  +-- unavailable -> explicit failure/pending policy
~~~

### 12.4 Failure walkthrough

1. Attempt 1 times out after the request may have reached Inventory.
2. Because reservationId makes the operation deduplicated, another attempt is safe from duplicate reservation effects.
3. The bulkhead releases the attempt permit.
4. Retry checks the shared retry budget and remaining user deadline.
5. It sleeps a jittered delay outside the bulkhead.
6. Attempt 2 reacquires rate, circuit, and bulkhead permission.
7. Inventory returns the already-created reservation for reservationId.
8. Checkout returns success inside the overall deadline.

If Inventory remains unhealthy:

- failures/slow calls open its circuit;
- later Checkout calls fail fast instead of waiting;
- unrelated Search/Account pools remain available;
- a business-approved message is returned;
- alerts and dashboards show the degraded dependency;
- HALF_OPEN sends limited recovery probes.

### 12.5 What each mechanism contributed

| Mechanism | Contribution |
|---|---|
| Stable reservation ID | duplicate-effect safety and reconciliation |
| Scope of retry | X is not recomputed |
| Per-attempt timeout | one attempt cannot wait forever |
| Overall deadline | attempts and sleeps cannot exceed user usefulness |
| Bulkhead | Inventory cannot consume all Checkout concurrency |
| RateLimiter | Checkout does not exceed assigned Inventory/vendor rate |
| CircuitBreaker | sustained Inventory failure becomes fast failure |
| Backoff + jitter | retries reduce and spread pressure |
| Retry budget | outage-wide retry amplification is bounded |
| Honest fallback | no false reservation confirmation |

### 13.1 WhatsApp-style messaging

**Illustrative design:**

~~~text
Send message
   |
   +-- stable clientMessageId
   +-- enqueue/store durably
   +-- retry transient transport failures with backoff+jitter
   +-- dedupe at receiver/server by clientMessageId

Media upload path -- separate bulkhead --> object/media service
Text path -------- separate bulkhead --> message service
~~~

Why:

- a timeout must not create duplicate visible messages;
- offline delivery is better modeled as durable asynchronous retry than a long synchronous HTTP loop;
- a huge media backlog should not consume all text-message capacity;
- message send and "typing indicator" deserve different retry/priority policies.

Avoid: returning "delivered" as a fallback when only local enqueue is confirmed. The UI should distinguish queued, sent, delivered, and read states.

## 14. Cascading-failure anatomy and containment

### 14.1 Slow dependency cascade

~~~text
1. B latency rises
2. A in-flight requests rise (throughput x latency)
3. A thread/connection pools saturate
4. A queue delay rises
5. clients hit timeouts
6. clients retry
7. offered load rises
8. B and A slow further
9. load balancer shifts traffic to surviving replicas
10. survivors overload
~~~

Controls by stage:

| Stage | Control |
|---|---|
| B latency rises | per-attempt timeout, slow-call breaker |
| A in-flight grows | bulkhead/adaptive concurrency |
| queue grows | bounded queue, queue deadline, load shedding |
| clients retry | retry classification, backoff+jitter, budget |
| survivors overload | rate/admission control, slow start, cells |
| optional work competes | priority/bulkhead/graceful degradation |

### 14.2 Retry cascade across layers

~~~text
Mobile (3 attempts)
  -> Gateway (3 attempts per mobile attempt)
      -> Service A (3 attempts per gateway attempt)
          -> B

deepest worst-case attempts = 3 x 3 x 3 = 27
~~~

This is a **derived maximum** where 3 means total attempts at each layer. Centralize the retry decision or propagate an attempt budget in request metadata.

### 14.3 Health-check/restart cascade

An overloaded instance can miss a liveness check, be restarted, lose warm cache/connections, and shift its traffic to survivors. The restart makes total capacity lower, so more instances miss checks.

Use:

- separate liveness ("process cannot recover") from readiness ("do not send normal traffic now");
- cheap, isolated health endpoints;
- restart backoff;
- load-balancer slow start;
- graceful drain;
- avoid making one optional dependency's open circuit mark the whole process dead.

### 14.4 Fallback cascade

~~~text
Primary A fails
   -> every caller invokes fallback B
       -> B was sized for 5% fallback traffic, now receives 100%
           -> B fails
~~~

Load-test fallbacks at incident volume. Prefer local cache/static/omission when business-correct. Protect a network fallback with its own timeout, circuit, and bulkhead.

### 14.5 Autoscaling cascade

Scaling callers can increase downstream pressure:

~~~text
API pods x2 -> database call concurrency x2 -> DB slower
           -> API latency rises -> autoscaler adds more API pods
~~~

Scale the bottleneck or limit caller concurrency; do not assume horizontal scale at one tier creates end-to-end capacity.

## 15. Trade-offs, alternatives, and when to use/avoid

### 15.1 Summary table

| Mechanism | Use when | Main benefit | Main cost/risk | Avoid as sole answer when |
|---|---|---|---|---|
| Retry | failure is transient, safe, budgeted | masks brief faults | extra latency/load/duplicates | persistent overload or unsafe write |
| Circuit breaker | recent outcomes predict sustained failure | fast failure and recovery relief | false opens, tuning, delayed recovery | one-off per-request business errors |
| Semaphore bulkhead | cap in-flight work cheaply | preserves capacity | no thread isolation | untrusted blocking code |
| Thread-pool bulkhead | isolate blocking client code | separate worker failure domain | threads, queueing, context overhead | nonblocking event loop |
| Rate limiter | protect rate quota/admission | bounded starts over time | bursts/local scope/waiting | latency-driven in-flight saturation |
| Time limiter | bound caller wait | frees caller path sooner | false timeouts, work may continue | no underlying client/server timeout |
| Cache/stale fallback | slightly stale read is acceptable | very fast degradation | freshness/correctness | payment/security truth |
| Durable queue | work can complete asynchronously | absorbs transient outage | delayed result, duplicates, backlog | immediate synchronous confirmation required |
| Failover | independent healthy replica/region exists | continued primary behavior | correlation, consistency, cost | common-mode dependency failure |
| Service mesh/proxy | polyglot fleet wants centralized traffic controls | consistent operational policy | semantic blind spots, extra hop/control plane | idempotency/business fallback |
| Cell/shuffle shard | blast radius must be structurally bounded | tenant/failure containment | capacity fragmentation/operations | tiny simple system |

### 15.2 Client library versus service mesh

**Application library advantages:**

- knows method semantics and idempotency;
- can choose business fallback;
- can reuse X outside retry;
- can inspect result bodies;
- can propagate domain operation IDs.

**Mesh/proxy advantages:**

- language-independent;
- centralized rollout/observability;
- consistent connection, outlier, and rate policies;
- can protect services whose application teams omitted controls.

Use both carefully. Never let both independently retry without one shared attempt/deadline budget.

### 15.3 Queue versus synchronous retry

Use a queue when:

- completion can be delayed;
- the request should survive caller restart;
- outage duration can be minutes/hours;
- backpressure should be explicit;
- dead-letter/reconciliation is required.

The worker still needs bounded attempts, delayed retry, idempotency, visibility timeout/ack semantics, and DLQ policy. A queue moves and persists pressure; it does not eliminate it.

### 15.4 Graceful degradation versus correctness

Classify features:

| Class | Example | Failure behavior |
|---|---|---|
| Critical and correctness-sensitive | payment, authorization | fail/pending/reconcile; never fabricate |
| Critical but stale-tolerant | some configuration/catalog reads | bounded stale cache if policy allows |
| Optional | recommendations, avatars | omit/default |
| Asynchronous side effect | email, analytics | durable queue, retry later |

This classification should appear early in an HLD interview because it determines fallbacks, budgets, and isolation.

## 16. Configuration method: derive, do not guess

### 16.1 Inputs

For each protected operation collect:

- criticality and fallback semantics;
- request rate and burst distribution;
- healthy p50/p95/p99 latency by payload;
- timeout/error distribution by cause;
- downstream safe QPS/concurrency;
- connection-pool and worker capacity;
- replica count and traffic skew;
- user end-to-end deadline;
- idempotency/dedupe capability;
- upstream/downstream retry behavior;
- recovery time distribution;
- SLO/error budget.

### 16.2 Timeout

Choose from the remaining end-to-end budget and measured latency distribution. Include handshake/cold-path cases intentionally. Observe false-timeout rate and work completed after timeout.

### 16.3 Bulkhead

Estimate healthy concurrency with throughput x latency, then cap by the downstream's tested safe concurrency and fleet total. Prefer zero or very short wait for user-facing overload; long permit waits are hidden queues.

### 16.4 Circuit breaker

Choose:

- window type/size;
- minimum calls;
- failure and slow-call definitions;
- thresholds;
- open duration;
- HALF_OPEN probe count;
- automatic/manual transition behavior;
- excluded exceptions.

Trade-off:

~~~text
small/low threshold -> fast detection, more false opens
large/high threshold -> stable evidence, more damage before opening
~~~

### 16.5 Retry

Choose:

- maximum total attempts;
- retriable results/exceptions;
- base/multiplier/cap/jitter;
- Retry-After behavior;
- per-request and service retry budgets;
- one owning retry layer;
- remaining-deadline test;
- idempotency/reconciliation.

The best retry count is often small. Success-after-retry metrics should prove that attempts add value rather than merely add load.

### 16.6 Rate limit

Derive from:

- downstream/vendor assigned quota;
- per-operation cost;
- fleet versus instance scope;
- burst tolerance;
- criticality/priority allocation;
- behavior when permission is unavailable;
- interaction with autoscaling.

### 16.7 Rollout

1. Emit metrics without enforcement where possible (for example, METRICS_ONLY for a breaker).
2. Replay/load-test recorded traffic and injected failure.
3. Enable for a small cohort/instance.
4. Watch rejections, retries, fallbacks, latency, and business outcome.
5. Expand gradually.
6. Keep an audited kill switch and rollback.

## 17. Observability and testing

### 17.1 Metrics to expose

#### End-to-end

- logical operations/s;
- total attempts/s;
- retry amplification = attempts / logical operations;
- success, failure, degraded-success, and pending rates;
- p50/p95/p99 latency and deadline exceeded;
- dependency time by phase if available.

#### Retry

- retries attempted by reason;
- success on attempt 1/2/3;
- exhausted retries;
- skipped due to deadline;
- skipped due to retry budget;
- Retry-After delay;
- duplicate/idempotency-key hits.

#### Circuit breaker

- state and transitions;
- buffered/minimum calls;
- failure and slow-call rates;
- not-permitted calls;
- duration OPEN;
- HALF_OPEN probe outcomes;
- forced/manual state changes.

#### Bulkhead

- current/max concurrent calls;
- permit wait and rejections;
- thread-pool active count;
- queue depth/age and queue rejection;
- saturation by dependency/priority.

#### Rate limiter

- successful/failed permission acquisitions;
- permit wait;
- configuration changes;
- local instance versus fleet aggregate.

#### Time limiter

- timeouts by operation;
- cancel requests and observed cancellation;
- downstream work completing after timeout;
- remaining deadline at attempt start.

#### Fallback

- fallback invocation/success/failure;
- data age/staleness;
- business outcome under degradation;
- fallback dependency saturation.

### 17.2 Alert on impact and control failure

An OPEN circuit is a symptom/protection action, not automatically the incident's root cause. Alerts should connect:

~~~text
user impact + dependency health + protection activity + saturation
~~~

Examples:

- high fallback rate plus healthy user success may be a degradation warning;
- rising retries plus falling success-after-retry suggests harmful retry;
- bulkhead rejections with low dependency CPU may indicate a client/pool misconfiguration;
- circuit flapping OPEN/CLOSED suggests thresholds or unstable recovery;
- timeouts without cancellations imply zombie work.

### 17.3 Tests

#### Unit/state tests

- CLOSED -> OPEN only after minimum sample and threshold;
- OPEN -> HALF_OPEN timing;
- healthy/unhealthy probes;
- correct exception/result classification;
- retry attempt convention;
- deadline stops later attempts;
- backoff cap and jitter range;
- permit release on success, exception, timeout, and cancellation.

Use a controllable clock/scheduler; avoid slow real sleeps.

#### Integration tests

Inject:

- connection refused;
- reset before and after request send;
- delayed headers/body;
- 429 with Retry-After;
- 400/500/501/502/503/504;
- remote success after caller timeout;
- fallback failure;
- partial unhealthy instances;
- Spring same-class self-invocation.

#### Load/chaos tests

Prove:

1. one dependency slows;
2. its bulkhead fills and rejects predictably;
3. unrelated features retain latency/capacity;
4. retry traffic remains inside budget and is de-synchronized;
5. the breaker opens after intended evidence;
6. recovery probes do not create a fleet-wide herd;
7. fallback handles full incident volume;
8. queues and connection pools remain bounded.

Chaos engineering is useful only with abort conditions, observability, and a known blast radius.

## 18. Common mistakes and edge cases

1. **Retrying every exception.** Business, authentication, programming, and cancellation exceptions are not transient dependency failures.
2. **Retrying all 5xx.** 501 and deterministic 500 failures show why status class alone is insufficient.
3. **Never retrying any 4xx.** Ignores 408, 429, and remediation-based conflict/auth flows.
4. **Retrying a timed-out POST without a stable idempotency key.**
5. **Generating a new idempotency key for each attempt.** That defeats deduplication.
6. **Wrapping local work X and side effects inside Retry unintentionally.**
7. **Sleeping inside a bulkhead.** If the bulkhead surrounds Retry, permits/threads may be held during backoff.
8. **No overall deadline.** Three "short" per-attempt timeouts plus sleeps exceed user latency.
9. **Timeout equals cancellation assumption.** Underlying work continues and consumes capacity.
10. **Circuit breaker as concurrency limiter.** It allows concurrent calls while CLOSED.
11. **One breaker for all dependencies.** One backend opens every path and corrupts metrics.
12. **Breaker per user/request.** No useful shared sample; unbounded state/cardinality.
13. **Counting 4xx validation as backend failure.** A caller bug opens a healthy dependency's circuit.
14. **Counting local bulkhead/rate rejection as remote failure without intent.**
15. **Fallback catches everything and returns success.** Monitoring and callers cannot see real failure.
16. **Fallback calls another remote dependency without protection.**
17. **Every layer retries.** Attempts multiply.
18. **Fixed or unjittered exponential backoff across a fleet.**
19. **No retry budget.** Every request uses maximum attempts during an outage.
20. **Retrying 429 immediately.** Makes throttling worse; honor Retry-After/rate contract.
21. **Resilience4j local RateLimiter treated as a fleet-global quota.**
22. **Huge queue in a thread-pool bulkhead.** It turns overload into high latency and memory pressure.
23. **Long maxWaitDuration on a semaphore.** Caller threads form an invisible queue.
24. **Shared connection pool despite separate semaphores.** The lower shared resource still couples failures.
25. **Ignoring replica count in bulkhead/rate sizing.**
26. **Every pod enters HALF_OPEN together.** Recovery herd.
27. **Marking application DOWN when an optional circuit opens.** Orchestrator restarts can amplify failure.
28. **Spring self-invocation bypasses annotations.**
29. **Reactive protection wraps object creation, not subscription/completion.**
30. **Default library settings copied into production.** Defaults cannot know your SLO, traffic, semantics, or fleet.
31. **Logging every retry at error level.** Creates log storms; aggregate metrics and sample structured logs.
32. **No distinction between logical request and attempt in tracing.** Debugging amplification becomes difficult.
33. **Retrying after the user cancelled.** Wastes work unless durable business completion is explicitly required.
34. **Fallback data has no freshness label.**
35. **Circuit flapping.** Open duration/probe thresholds are too aggressive or the backend is marginal.

Use cues to reconstruct the answer live.

#### F - Fail fast and contain

### 19.5 Example interview answer

> "I would first classify whether the dependency is critical and whether failed operations are safe to repeat. Every call gets a propagated end-to-end deadline and a shorter per-attempt timeout. I isolate the dependency with its own concurrency bulkhead and, if it has a throughput quota, a rate limit whose fleet scope is explicit. A circuit breaker watches only relevant failures and slow calls, opens after a meaningful sample, and permits limited recovery probes. I retry only transient, idempotent or deduplicated operations, at one layer, with a small attempt cap, capped exponential backoff plus jitter, Retry-After support, a shared retry budget, and a remaining-deadline check. When the primary path is unavailable, I return an honest stale/partial/queued result only where business semantics allow, then validate all of this with attempt, rejection, fallback, and recovery metrics under load and fault injection."

## 20. Compact revision sheet

### 20.1 Ten facts to retain

1. A cascading failure grows/spreads through positive feedback and shared resources.
2. Fault tolerance is defined against a fault model and may use graceful degradation.
3. Timeout bounds waiting; it does not guarantee work stopped.
4. Rate limiter controls starts/time; bulkhead controls in-flight concurrency.
5. A circuit breaker observes health and fails fast; it does not cap concurrency.
6. CLOSED allows calls, OPEN rejects, HALF_OPEN admits limited recovery probes.
7. Retry only when transient + safe + budgeted + within deadline + load-aware.
8. HTTP status class alone is not a retry policy.
9. Exponential backoff reduces frequency; jitter breaks synchronization.
10. Pattern order determines whether limits/metrics apply per attempt or per logical request.

### 20.2 Formula card

~~~text
average concurrency
    ~= throughput x average latency

deepest attempt worst case
    = product of total attempts allowed at each retrying layer

capped exponential delay
    cap(k) = min(maxDelay, baseDelay x 2^k)

full jitter
    sleep(k) = Uniform(0, cap(k))

retry amplification
    = total attempts / logical operations

independent local fleet concurrency ceiling
    ~= replicas x per-replica bulkhead limit

independent local fleet rate ceiling
    ~= replicas x per-replica rate limit
~~~

The last two are simplified derived ceilings; routing skew, shared pools, adaptive limits, and actual traffic change observed values.

### 20.3 Pattern-selection card

~~~text
brief transient fault + safe operation -> bounded retry
sustained unhealthy dependency         -> circuit breaker
too many in-flight calls               -> bulkhead/concurrency limit
too many starts per time               -> rate limiter
caller waits too long                  -> timeout/deadline/cancellation
optional feature fails                 -> honest fallback/degrade
long asynchronous outage               -> durable queue/delayed retry
one tenant harms others                -> per-tenant/tier cell/bulkhead
fleet-global quota                     -> distributed/central admission
~~~

### 20.4 Design checklist

- What exact fault is being tolerated?
- Which user behavior must remain available?
- Is the path critical, optional, stale-tolerant, or asynchronous?
- Is the operation idempotent or deduplicated?
- What is the overall deadline and remaining budget?
- Which errors are transient, permanent, overload, or ambiguous?
- Which layer owns retries, and what hidden SDK retries exist?
- Are attempts bounded, backed off, jittered, and budgeted?
- Does each attempt reacquire rate/bulkhead/circuit permission as intended?
- Are bulkheads separated by dependency/criticality?
- Is the rate/concurrency scope local, fleet, or global?
- What outcomes count toward the breaker?
- Is the circuit window statistically meaningful for the traffic?
- Can fleet-wide HALF_OPEN probes overwhelm recovery?
- Does timeout propagate cancellation/deadline?
- Is fallback correct, independent, bounded, and observable?
- Are queues bounded and deadline-aware?
- Do metrics distinguish logical requests from attempts?
- Has failure **and recovery** been load-tested?
- Is there a shadow rollout, kill switch, and rollback?

## 22. Say-out-loud 30-second narrative

> "A fault-tolerant system assumes dependencies will fail and contains the blast radius instead of pretending failures can be eliminated. I give each call an end-to-end deadline and per-attempt timeout, isolate slow dependencies and noisy neighbors with per-operation bulkheads, rate-limit starts when there is a throughput quota, and use a circuit breaker to turn sustained failure or slowness into fast rejection with limited recovery probes. I retry only transient operations that are idempotent or deduplicated, at one layer, with a small attempt cap, capped exponential backoff plus jitter, Retry-After, a retry budget, and a remaining-deadline check. Fallbacks must be cheap and honest, and I verify the design through attempt, saturation, circuit, timeout, fallback, and recovery metrics under fault-injected load."
