# Distributed Transactions: Two-Phase Commit, Three-Phase Commit, and Saga

> Beginner-first HLD notes expanded from the supplied bullets. The goal is to understand what can go wrong when one business operation changes several independent systems, then choose deliberately between atomic commit, compensation, or redesigning the data boundary.

## 0. Scope, corrections, assumptions, and evidence labels

Your bullets are interpreted as:

- **“2 phase - blocking protocol”** means **Two-Phase Commit (2PC)**. This is correct, but “blocking” has a precise meaning: a prepared participant can be forced to wait for the final decision after certain failures.
- **“3 phase - non blocking protocol”** means **Three-Phase Commit (3PC)**. This needs an important correction: classic 3PC is non-blocking only under stronger assumptions such as reliable communication among surviving sites and reliable failure detection. It does not remain safely non-blocking through arbitrary network partitions.
- **“SAGA Pattern”** means a **Saga**: a sequence of committed local transactions with business-specific compensating actions or forward recovery. “Saga” is a name, not an acronym. A Saga does not provide the same all-at-once ACID semantics as 2PC.

The term **distributed transaction** is used in two ways:

1. **Strict database meaning:** several transactional resources reach one atomic commit/abort decision.
2. **Broader business meaning:** several services cooperate to complete one business operation, possibly with temporary inconsistency and compensation.

2PC and 3PC target the first meaning. Saga targets the second.

Evidence labels used below:

- **Verified technical behavior:** supported by an original paper or first-party documentation linked nearby.
- **Self-reported interview prompt:** a public candidate report; useful evidence about interview style, not a company-confirmed question bank.
- **Illustrative product design:** a reasonable HLD for a named product, not a claim about its current private architecture.
- **Unverified:** a claim that should not be repeated as fact without checking. No universal latency, timeout, retry count, or maximum participant count is assumed in this chapter.

Protocol and product details were checked on **17 July 2026**.

## 1. Plain-language foundation

### 1.1 Start with one database

Suppose one database stores both account balances:

~~~text
BEGIN
  subtract Rs 500 from account A
  add      Rs 500 to account B
COMMIT
~~~

The database can expose this as one transaction:

- if both changes succeed, commit both;
- if either change fails, roll back both;
- other transactions need not observe a half-finished transfer; and
- after commit, the database makes the result durable according to its guarantee.

This is a **local transaction** because one transaction manager and one transactional resource control all changes.

### 1.2 Now split the data

Suppose account A is in database A and account B is in database B:

~~~text
Transfer Service
      |
      +---- subtract Rs 500 ----> Database A
      |
      +---- add Rs 500 ---------> Database B
~~~

A naive implementation does this:

1. Database A commits the debit.
2. The network fails before Database B receives the credit.

The user has lost money from A without receiving it in B. Calling the second database again is also not automatically safe: perhaps B committed but its reply was lost. A timeout tells the caller **“I do not know the outcome,”** not **“the operation definitely failed.”**

That uncertainty is the heart of distributed transactions.

### 1.3 What the system is trying to protect

The business first states an **invariant**, a rule that must remain true. Examples:

- a transfer must not create or destroy money;
- an item must not be sold beyond available stock;
- an order must not be marked paid if the payment was rejected;
- one payment operation ID must not charge twice.

Then the design chooses the required consistency:

~~~text
Option A: hide every partial result and expose one atomic decision
          -> local transaction or distributed atomic commit such as 2PC

Option B: allow named intermediate states, then finish or compensate
          -> Saga / durable workflow

Option C: change the ownership boundary so one service can enforce the
          invariant in one local transaction
          -> often the simplest and strongest solution
~~~

The best first question is not “2PC or Saga?” It is:

> Which invariant must be atomic, who owns the authoritative data, and can that data live inside one transactional boundary?

### 1.4 A one-screen mental map

| Mechanism | Plain-language idea | Consistency visible to applications | Main failure cost |
|---|---|---|---|
| Local ACID transaction | One database decides everything | Atomic within that database | Database availability and contention |
| 2PC | Every participant promises first; one coordinator then decides | Atomic commit across compatible participants | Prepared work can block during uncertainty |
| 3PC | Add a pre-commit state so survivors can sometimes decide after a failure | Atomic under its failure/timing assumptions | Extra phase; unsafe assumptions during partitions |
| Saga | Commit one local step at a time; compensate or continue on failure | Temporary intermediate states are visible | Compensation, isolation, and recovery complexity |

Do not read the table as “3PC is a better 2PC” or “Saga is an asynchronous 2PC.” They make different promises.

## 2. Prerequisites explained inline

These concepts are required to understand the protocols, so they are introduced here rather than deferred to “study next.”

### 2.1 ACID

A database transaction is commonly described by ACID:

- **Atomicity:** its included changes commit or abort as one unit.
- **Consistency:** database constraints and correct application logic move the data between valid states. The database cannot invent missing business rules.
- **Isolation:** concurrent transactions do not observe or create outcomes forbidden by the selected isolation guarantee.
- **Durability:** a reported commit survives the failures covered by the database’s durability contract.

2PC is principally an **atomic commit protocol**. It does not, by itself, provide all of ACID across arbitrary services:

- participants still need local transactions and durable recovery;
- concurrency control must supply the required isolation;
- application logic must define valid states; and
- a coordinator must durably preserve the global decision.

### 2.2 Commit, rollback, and a write-ahead log

- **Commit** makes a transaction’s result final.
- **Rollback/abort** discards its uncommitted changes.
- A **write-ahead log (WAL)** records recovery information durably before related data pages are treated as safely committed.

A participant that votes **YES** in 2PC must persist enough information to survive a crash and later obey either final decision. A promise held only in RAM would disappear on restart and make the protocol unsafe.

### 2.3 Locks and versions

A transaction may reserve rows using locks or preserve tentative versions using another concurrency-control mechanism.

The important point for 2PC is:

> After voting YES, a participant must keep the prepared result and any resources needed to honor it until COMMIT or ABORT is learned.

That is why an unresolved prepared transaction can block conflicting work. PostgreSQL’s current documentation warns that a prepared transaction continues holding its locks and should not remain prepared for long ([PostgreSQL 18: PREPARE TRANSACTION](https://www.postgresql.org/docs/18/sql-prepare-transaction.html)).

### 2.4 Coordinator and participants

~~~text
                    global decision
                 +-------------------+
                 | Transaction       |
                 | Coordinator       |
                 +-------------------+
                    /       |       \
                   /        |        \
          Participant A  Participant B  Participant C
             Order DB      Inventory DB   Payment DB
~~~

- The **coordinator** runs the commit protocol and records the global outcome.
- A **participant/resource manager** owns one local transaction branch and votes on whether it can commit.
- A **global transaction ID** correlates the same unit of work across every participant and recovery log.

In XA terminology, a transaction manager coordinates XA-capable resource managers. Java’s official XA interface exposes prepare, commit, rollback, and recovery operations for this purpose ([Java SE 25 XAResource](https://docs.oracle.com/en/java/javase/25/docs/api/java.transaction.xa/javax/transaction/xa/XAResource.html)).

### 2.5 Crash, message loss, timeout, and network partition

These failures are not interchangeable:

| Failure | What it means |
|---|---|
| Process crash | One process stops; it may later recover from durable state |
| Machine/site crash | A larger failure domain becomes unavailable |
| Message loss/delay | Sender cannot assume whether the receiver acted |
| Timeout | A local waiting limit expired; it is not proof of remote failure |
| Network partition | Live nodes are split into groups that cannot communicate |
| Byzantine failure | A node can lie or behave arbitrarily; classic 2PC/3PC do not solve this |

A protocol described as non-blocking under crash failures may still block or become unsafe under partitions. Always ask for the failure model.

### 2.6 Atomic commit versus consensus

These problems are related but not identical:

- **Atomic commit:** commit only if every required participant is able to commit; one NO must cause abort.
- **Consensus:** processes agree on one proposed value under a defined failure model.

Replicating the coordinator’s decision with Raft or Paxos can make the decision record more available, but it does not automatically make a non-transactional payment API, email provider, or database into a 2PC participant.

Gray and Lamport show the close relationship: classic 2PC is a zero-fault-tolerant special case of their Paxos Commit formulation, while fault-tolerant consensus can keep making progress when a sufficient quorum is working ([Gray and Lamport, Consensus on Transaction Commit](https://www.microsoft.com/en-us/research/publication/consensus-on-transaction-commit/)).

### 2.7 Idempotency

An operation is **idempotent for a business operation ID** when retrying the same logical request does not repeat its business effect.

~~~text
Charge(payment_operation_id = op-91, amount = Rs 500)
~~~

The payment service stores op-91 with its result. A retry returns that result instead of charging again. This is required for Saga steps and is also useful for safely redelivering commit/abort messages.

The repository’s [Idempotency chapter](./13.%20Idempotency.md) develops this topic in depth.

## 3. The distributed commit problem

### 3.1 Correctness properties

An atomic commit protocol aims for:

1. **Agreement:** correct participants do not reach mixed final outcomes.
2. **Commit validity:** commit is allowed only when every required participant voted YES.
3. **Abort validity:** a NO vote, invalid local work, or a required pre-decision failure can lead to abort.
4. **Durability:** once decided, the outcome survives covered crashes.
5. **Termination/liveness:** participants eventually learn an outcome under the protocol’s stated assumptions.

Safety and liveness are different:

- **Safety:** “never debit only one side.”
- **Liveness:** “the transfer eventually finishes instead of waiting forever.”

2PC protects safety by sometimes sacrificing liveness. A prepared participant waits rather than guessing and risking a mixed result.

### 3.2 Atomicity is not isolation

Suppose two distributed transactions concurrently update the same business entities. 2PC can ensure that each one either commits everywhere or aborts everywhere, yet the final result can still violate an invariant if the underlying reads and writes were not isolated correctly.

~~~text
Atomic commit asks:  Did all branches commit together?
Isolation asks:      Was their concurrent interleaving valid?
~~~

An interview answer that says “2PC gives ACID automatically” is too broad. State what the local databases, lock/version protocol, and global transaction layer guarantee.

### 3.3 Avoid distribution when the invariant is naturally local

If debit and credit are entries in the same authoritative ledger, keep them in one local transaction even if downstream services receive events later:

~~~text
                 one local transaction
Transfer API -> [ Ledger DB: debit entry + credit entry + outbox ]
                                      |
                                      +---- async event ----> notifications
~~~

Splitting one invariant across services and then rebuilding atomicity with coordination can be worse than keeping a larger, cohesive ownership boundary.

# Part I — Two-Phase Commit (2PC)

## 4. 2PC in plain language

### 4.1 The wedding-vow intuition

The coordinator does not immediately tell everyone to commit. It first asks:

> “Can you promise that, if I later say COMMIT, you definitely can commit?”

Every YES voter writes that promise durably and waits. Only after all required participants say YES does the coordinator choose COMMIT. Any NO causes ABORT.

This separates:

1. **Prepare/vote:** establish whether commit is possible.
2. **Decision:** tell everyone to commit or abort.

### 4.2 A controlled example

This bank-transfer example assumes both databases support the same distributed transaction mechanism. It is not a claim that unrelated banks expose XA participation.

~~~text
Global transaction X91

Database A: prepare debit Rs 500
Database B: prepare credit Rs 500

If both promise YES -> COMMIT both
If either says NO    -> ABORT both
~~~

### 4.3 Phase 0: the work happens

The protocol’s name counts commit phases, not the earlier business work:

1. Coordinator assigns global transaction ID X91.
2. Database A performs the debit tentatively in a local transaction.
3. Database B performs the credit tentatively in a local transaction.
4. Neither branch exposes a final commit yet.

Then 2PC begins.

## 5. Phase 1 — Prepare / vote

~~~text
Coordinator                 Database A                 Database B
    |                           |                          |
    |------ PREPARE X91 ------->|                          |
    |------ PREPARE X91 ---------------------------------->|
    |                           | validate constraints     |
    |                           | persist PREPARED         |
    |                           | keep needed resources    |
    |<-------- YES X91 ---------|                          |
    |<-------- YES X91 ------------------------------------|
~~~

Each participant checks whether it can commit:

- did its local statements succeed?
- do constraints allow the result?
- can it persist the prepared state?
- can it retain the locks/versions and recovery data?

A **YES** means:

> “I surrender my right to decide alone. I can later commit or abort, and I will wait for the global decision.”

A participant that has voted YES must not simply time out and roll back. The coordinator may already have durably decided COMMIT and another participant may already have committed.

A **NO** can be returned if the local work failed or cannot be prepared. The participant can abort its branch.

## 6. Phase 2 — Global decision

### 6.1 All votes are YES

~~~text
Coordinator                 Database A                 Database B
    |                           |                          |
    | persist GLOBAL_COMMIT X91 |                          |
    |------ COMMIT X91 -------->|                          |
    |------ COMMIT X91 ----------------------------------->|
    |                           | persist/commit            |
    |                           | release resources         |
    |<--------- ACK ------------|                          |
    |<--------- ACK ---------------------------------------|
~~~

The coordinator must durably record COMMIT before sending it. After that point, recovery must continue driving the transaction to commit.

### 6.2 Any vote is NO, or phase 1 cannot complete

~~~text
Coordinator                 Database A                 Database B
    | persist GLOBAL_ABORT X91  |                          |
    |------ ABORT X91 --------->|                          |
    |------ ABORT X91 ------------------------------------>|
    |                           | roll back                 |
    |                           | release resources         |
~~~

Before a global commit decision exists, abort is the safe fallback when a required participant cannot prepare.

### 6.3 Why ACKs are often shown after the two phases

An ACK allows the coordinator to know when it may forget or archive the transaction. It does not create a third commit decision phase. Lost decision messages can be resent because COMMIT X91 and ABORT X91 should be handled idempotently.

## 7. The durable state machines

### 7.1 Participant

~~~text
WORKING
  | \
  |  \ local failure / vote NO
  |   +------------------------------> ABORTED
  |
  | vote YES + durable prepare
  v
PREPARED (in doubt; cannot decide alone)
  |                              |
  | receive COMMIT               | receive ABORT
  v                              v
COMMITTED                       ABORTED
~~~

### 7.2 Coordinator

~~~text
COLLECTING VOTES
  |                       |
  | every required YES    | any NO / pre-decision failure
  v                       v
log GLOBAL_COMMIT       log GLOBAL_ABORT
  |                       |
  +---- notify all --------+
~~~

The logs are part of the algorithm, not an optional observability feature.

## 8. Why 2PC is a blocking protocol

### 8.1 The critical failure window

~~~text
1. A and B both vote YES and enter PREPARED.
2. Coordinator crashes before A and B learn the final decision.
3. A and B cannot safely guess.
~~~

Possible realities include:

- the coordinator had not chosen COMMIT, so abort may be correct;
- it durably chose COMMIT but its reply was delayed;
- another participant received COMMIT before the coordinator crashed.

If a prepared participant cannot reach the recovered coordinator or another participant that knows the decision, it must wait. Skeen’s original analysis calls 2PC blocking because operational sites can have to wait for failed sites to recover while database locks remain held ([Skeen, Nonblocking Commit Protocols, 1981](https://www.cs.utexas.edu/~lorenzo/corsi/cs380d/papers/Ske81.pdf)).

### 8.2 What “blocking” does and does not mean

It means:

- the **in-doubt transaction** cannot safely finish at some participant; and
- its retained locks/resources can block conflicting transactions.

It does not necessarily mean:

- the entire database process freezes;
- every unrelated transaction stops; or
- a coordinator crash always blocks every 2PC transaction.

The blast radius depends on the prepared transaction’s resources and workload.

### 8.3 “Make the coordinator highly available” helps, but is not the whole proof

Replicated coordinator state, automatic failover, and recovery workers reduce the duration and probability of blocking. However:

- the global decision must be replicated consistently, not merely cached;
- a partition can still separate participants from the decision;
- every participant must support recovery by global transaction ID; and
- incompatible external systems still cannot join the atomic transaction.

A consensus-backed decision store improves availability. It does not remove 2PC’s need for every resource manager to prepare and honor the decision.

## 9. 2PC failure matrix

| Failure point | Safe behavior | Why |
|---|---|---|
| Participant fails before voting | Coordinator times out and aborts | No global commit can be chosen without all required YES votes |
| Participant votes NO | Global abort | Atomic commit validity requires all YES |
| Participant crashes after YES | On recovery, restore PREPARED and learn decision | The YES promise must survive the crash |
| Coordinator crashes before a durable decision | Recovery uses its log; if no COMMIT could have been sent, abort can be driven | Decision must be logged before notification |
| Coordinator crashes after logging COMMIT but before notifying all | Recovery resends COMMIT | The durable decision is irreversible |
| COMMIT message arrives twice | Participant returns the same committed outcome | Decision handling must be idempotent |
| Participant commits but its ACK is lost | Coordinator resends; participant reports committed | Lost reply does not undo the commit |
| Coordinator is unreachable while participant is PREPARED | Participant queries recovery peers if protocol permits; otherwise waits | Guessing can create a mixed outcome |
| Operator force-aborts one prepared branch | Possible heuristic mixed/hazard outcome | Another branch may have committed |

### 9.1 Heuristic decisions

A **heuristic outcome** occurs when a participant or operator makes a unilateral final choice to escape indefinite uncertainty.

- **Heuristic commit:** branch was forced to commit.
- **Heuristic rollback:** branch was forced to abort.
- **Heuristic mixed:** different branches ended differently.
- **Heuristic hazard:** the system cannot determine the complete result.

This can restore local availability at the cost of global atomicity and manual reconciliation. It is a last-resort operational action, not a normal timeout policy. The Java XA API explicitly models heuristic error outcomes, which is a clue that real recovery is more complicated than the happy-path sequence.

## 10. 2PC costs and practical constraints

### 10.1 Latency

The critical path includes:

- a prepare request and votes;
- durable log work at participants;
- a durable global decision;
- the final decision messages; and
- sometimes acknowledgements before cleanup.

For N participants, communication remains O(N) in a centralized topology, but the exact message and storage-write count depends on read-only optimizations, batching, replication, and implementation. Do not quote one universal latency number.

### 10.2 Contention

Participants retain prepared state and often locks across network and storage delays:

~~~text
longer transaction duration
        -> locks held longer
        -> more waiting/deadlocks
        -> lower throughput under contention
~~~

### 10.3 Compatibility and coupling

Every participant must implement the transaction protocol and compatible recovery semantics. Common obstacles are:

- a message broker or NoSQL store does not support the chosen 2PC/XA mechanism;
- an external card processor cannot expose a prepared transaction;
- one participant has a much longer outage than the rest;
- cross-region latency makes the lock window expensive;
- independent service deployment becomes tied to a shared transaction manager.

### 10.4 Operations

Production design needs:

- globally unique transaction IDs;
- coordinator and participant recovery logs;
- monitoring of prepared transaction age;
- a recovery scanner;
- reconciliation tooling;
- alerts for heuristic outcomes;
- a safe runbook for resolving stuck transactions; and
- limits so abandoned prepared transactions do not exhaust resources.

PostgreSQL exposes prepared transactions in the pg_prepared_xacts view and recommends disabling prepared transactions when no external manager will close them promptly ([PostgreSQL: Two-Phase Transactions](https://www.postgresql.org/docs/current/two-phase.html)).

## 11. When 2PC fits and when it does not

### Use 2PC when

- atomic visibility is a hard requirement;
- all participants are transactional resources that support the same protocol;
- transactions are short and participant count is controlled;
- temporary unavailability is preferable to inconsistent commit;
- latency and lock contention are acceptable; and
- the team can operate coordinator and in-doubt recovery safely.

Examples include:

- an application server coordinating two XA-capable databases;
- one controlled database platform coordinating multiple shards; or
- a database plus another XA-capable transactional resource.

### Avoid or reconsider 2PC when

- a participant is an email, shipping, human approval, or third-party payment API;
- the workflow lasts minutes, hours, or days;
- the system must remain writable through network partitions;
- compensating business actions are acceptable;
- participants use incompatible storage technologies; or
- one cohesive invariant was split across services without a good ownership reason.

### 11.1 Verified implementation examples

**PostgreSQL:** PREPARE TRANSACTION stores a transaction’s prepared state so a later session can execute COMMIT PREPARED or ROLLBACK PREPARED. Its documentation says the feature is for external transaction managers, not ordinary interactive application use ([PostgreSQL 18 documentation](https://www.postgresql.org/docs/18/sql-prepare-transaction.html)).

**Google Spanner:** the published Spanner design says a transaction contained within one Paxos group can avoid the distributed transaction manager; transactions spanning multiple Paxos groups use 2PC, and the transaction-manager state is stored in the underlying Paxos groups. This is an important real-world pattern: **consensus replicates each group’s state, while 2PC coordinates the atomic outcome across groups** ([Spanner paper, Section 2.1](https://storage.googleapis.com/gweb-research2023-media/pubtools/1974.pdf)).

## 12. 2PC interview answer spine

Use **P-P-D-R**:

1. **Participants:** name the coordinator, resource managers, and global transaction ID.
2. **Promise:** PREPARE asks each participant to durably promise YES or vote NO; YES retains prepared state/resources.
3. **Decision:** all YES means durable global COMMIT; any NO means ABORT; notify every participant.
4. **Recovery:** after YES, a participant cannot decide alone; coordinator failure can block, so discuss logs, retries, HA, monitoring, and heuristic danger.

One compact sentence:

> “2PC gets atomic commit by turning each YES vote into a durable promise, but that promise creates an in-doubt state that may hold resources while the final decision is unavailable.”

# Part II — Three-Phase Commit (3PC)

## 13. 3PC in plain language

### 13.1 The idea

2PC jumps from “everyone promised YES” to “COMMIT.” A participant can be uncertain while another participant might already have committed.

3PC inserts a shared **PRE-COMMIT** state:

~~~text
CanCommit?  ->  PreCommit  ->  DoCommit
vote first      align state     make final
~~~

The extra state is intended to ensure that survivors can distinguish:

- an early state where abort is safe; from
- a later state where the system has crossed far enough that commit can be completed.

### 13.2 The three phases

#### Phase 1 — CanCommit?

~~~text
Coordinator ------ CAN_COMMIT? ------> all participants
Coordinator <--------- YES/NO -------- all participants
~~~

- Any NO leads to ABORT.
- All YES allows the coordinator to proceed.

#### Phase 2 — PreCommit

~~~text
Coordinator logs PRE_COMMIT
Coordinator -------- PRE_COMMIT ------> all participants
Coordinator <----------- ACK ---------- all participants
~~~

Participants enter a durable pre-commit state. No participant should already be committed while another is still in the earlier uncertain state if the protocol assumptions hold.

#### Phase 3 — DoCommit

~~~text
Coordinator logs COMMIT
Coordinator -------- DO_COMMIT -------> all participants
participants commit and acknowledge
~~~

### 13.3 State-machine intuition

~~~text
WAITING / UNCERTAIN
  | \
  |  \ NO or safe early termination
  |   +-------------------------------> ABORT
  |
  | unanimous YES + PRE_COMMIT learned
  v
PRE_COMMIT / COMMITTABLE
  |
  | DO_COMMIT or valid termination protocol
  v
COMMIT
~~~

The pre-commit state is a **buffer state**. Skeen’s original paper derives three-phase protocols by adding a “prepare to commit” state to satisfy its non-blocking conditions ([Skeen, 1981](https://www.cs.utexas.edu/~lorenzo/corsi/cs380d/papers/Ske81.pdf)).

## 14. What “non-blocking” really means

Under the paper’s model, surviving operational sites can use their states and a termination protocol to choose a replacement coordinator and finish without waiting for a failed site to recover.

It does **not** mean:

- no operation ever waits;
- no locks/resources are held;
- the system survives arbitrary message delay while staying available;
- timeouts reveal the truth; or
- 3PC bypasses CAP.

### 14.1 The assumptions are the correction to your bullet

Skeen explicitly assumes:

1. point-to-point communication remains possible between any two operational sites — the network itself never fails; and
2. the network can detect and reliably report site failures, for example through a trustworthy timeout.

These assumptions exclude the hard ambiguity of a network partition, where a live but unreachable node looks like a crashed node.

Therefore the interview-ready statement is:

> “Classic 3PC is non-blocking for its modeled crash failures with reliable communication and failure detection; it is not unconditionally non-blocking in an asynchronous, partitionable network.”

### 14.2 Partition counterexample intuition

Imagine the network divides after PRE_COMMIT reaches only one side:

~~~text
Partition L                         Partition R
received PRE_COMMIT                still in earlier state
times out                          times out
believes commit can finish         believes abort is safe
~~~

If both partitions continue independently based only on timeouts, they can make conflicting choices. If they refuse to risk a conflicting choice, at least one side must wait. 3PC cannot obtain both perfect safety and full availability merely by adding a phase.

The lesson is broader:

> A timeout is a scheduling tool and failure suspicion, not proof that another node is dead.

## 15. 3PC failure reasoning

| Participant’s durable knowledge | Failure intuition under 3PC assumptions | Important caveat |
|---|---|---|
| Before YES / early waiting state | Abort can be safe | Must run the specified termination protocol |
| PRE_COMMIT known | Survivors can elect a coordinator and drive commit | Relies on surviving sites being able to communicate and compare states |
| COMMIT known | Resend/finish commit | Decision handling still needs durable logs and idempotency |
| Network divided | Timeout cannot distinguish crash from partition | Safe non-blocking progress is not generally available on both sides |

Simplified termination intuition after electing a replacement coordinator:

- if any reachable site has the final COMMIT decision, propagate COMMIT;
- if a reachable site is in PRE_COMMIT and no final abort exists, align the survivors to that committable state and drive COMMIT;
- if all reachable sites remain in the early uncertain state, ABORT can be safe because the separated protocol states imply that nobody could already have committed; and
- if the collected states appear contradictory, an assumption was violated or recovery data is corrupt, so do not guess.

Those rules are only an intuition for the classic state separation. A real implementation must follow its exact protocol, logging order, election, and termination rules.

Do not invent timeout transitions in an interview without stating the failure model. Explain the state distinction and then say the exact recovery rule belongs to the specified 3PC variant.

## 16. 3PC trade-offs and practical position

### Advantages under its assumptions

- removes the particular 2PC uncertainty in which a survivor may have to wait for a failed site;
- makes the commit progression more explicit; and
- is valuable for understanding how protocol states and failure assumptions affect liveness.

### Costs and limitations

- one additional coordination phase adds messages, logging, and latency;
- locks/tentative resources remain open for longer;
- correct failure detection assumptions are difficult in real asynchronous networks;
- network partitions break the simple non-blocking story;
- coordinator election and termination logic add complexity; and
- every participant must implement the same protocol and recovery rules.

Skeen’s paper itself notes that the extra phase imposes substantial message overhead. Do not propose 3PC simply because a study sheet calls it “better than 2PC.” First verify that the target platform actually supports the required protocol and assumptions.

### When to use or avoid

For an HLD interview:

- discuss 3PC mainly when explicitly asked, when comparing commit protocols, or when explaining why assumptions matter;
- do not present it as the default solution for microservices;
- prefer a database’s proven distributed transaction implementation when strict ACID is required; and
- prefer a Saga or another workflow model when application-visible intermediate states and compensation are acceptable.

## 17. 3PC interview answer spine

Use **C-P-D-A**:

1. **CanCommit:** gather votes.
2. **PreCommit:** durably add a buffer state and acknowledgements.
3. **DoCommit:** issue the final decision.
4. **Assumptions:** non-blocking only with reliable survivor communication/failure detection; partitions invalidate the simple claim and the extra phase costs latency.

If the interviewer says “3PC is non-blocking,” respond:

> “Yes under the classic model’s assumptions. I would qualify that before applying it to a real partitionable network.”

# Part III — Saga

## 18. Saga in plain language

### 18.1 The core difference

A Saga does not keep every service’s transaction uncommitted until one global decision.

Instead:

1. each service commits a short local transaction;
2. the workflow records what succeeded;
3. a later failure triggers forward recovery, an alternative action, or compensating transactions; and
4. the system eventually reaches a valid business state.

~~~text
2PC:
  tentative A + tentative B + tentative C
                 |
          one atomic decision

Saga:
  commit A -> commit B -> commit C
                |
            if C fails
                v
         compensate B -> compensate A
~~~

Because A and B were real commits, other work may observe the intermediate state. A Saga trades cross-service isolation and instantaneous atomicity for shorter local transactions, looser coupling, and application-controlled recovery.

The original Saga paper described long-lived work as a sequence of transactions that can interleave with other transactions, with compensating transactions used for recovery ([Garcia-Molina and Salem, Sagas, 1987](https://doi.org/10.1145/38713.38742)).

### 18.2 Amazon-like checkout example — illustrative design

~~~text
Customer
   |
   v
Order workflow
   |
   +--> T1 Create PENDING order
   |
   +--> T2 Reserve inventory
   |
   +--> T3 Authorize payment
   |
   +--> T4 Confirm order
   |
   +--> T5 Request fulfillment
~~~

If payment authorization is rejected:

~~~text
T1 Create PENDING order       -> C1 Cancel order
T2 Reserve inventory          -> C2 Release reservation
T3 Authorize payment fails    -> no charge to undo
~~~

If the request timed out, do not immediately release inventory and assume payment failed. Query the payment operation by idempotency key first: the authorization may have succeeded while its response was lost.

## 19. Compensation is not rollback

A database rollback erases uncommitted changes as if they never committed. A Saga compensation is a **new committed business action**.

Examples:

| Forward action | Possible compensation | Why it is not exact time travel |
|---|---|---|
| Reserve stock | Release reservation | Another customer may have waited |
| Authorize card | Void authorization | Provider state and audit history remain |
| Capture payment | Issue refund | Fees, exchange rate, settlement delay, and records may remain |
| Book hotel | Cancel booking | Cancellation policy may charge a fee |
| Send email | Send correction | The first email cannot be unsent |
| Ship parcel | Attempt return/intercept | Physical action may be irreversible |

Compensation must preserve business truth and auditability. It may not restore the exact starting state, may run in a business-defined order rather than strict reverse order, and can itself fail. Microsoft’s compensating-transaction guidance explicitly treats compensation as application-specific, resumable, idempotent work that can require human intervention ([Azure Compensating Transaction pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/compensating-transaction)).

## 20. Compensatable, pivot, and retryable steps

Classify each step before implementation:

- **Compensatable:** can be meaningfully counteracted, such as releasing a reservation.
- **Pivot:** the point of no return or business commitment.
- **Retryable:** after the pivot, should eventually succeed through safe retries.
- **Irreversible:** cannot honestly be undone, such as an external physical action.

Design irreversible actions late:

~~~text
validate -> reserve -> authorize/hold -> pivot -> capture/dispatch -> notify
              easy to release             forward recovery after commitment
~~~

Payment authorization and capture are different tools:

- **authorize/hold:** reserve the ability to charge; often voidable;
- **capture:** actually collect the funds; refund may be the later compensation.

The exact pivot depends on the domain, not on a universal rule.

## 21. Choreography versus orchestration

### 21.1 Choreography

Services react to events without one component directing the whole workflow:

~~~text
OrderCreated
     |
     v
Inventory Service -- InventoryReserved --> Payment Service
                                              |
                            PaymentAuthorized-+
                                              v
                                         Order Service
~~~

Advantages:

- less central workflow control;
- services remain event-driven; and
- suitable for a small, simple sequence.

Costs:

- the flow is spread across many codebases;
- cycles and accidental coupling are easy to create;
- global status and debugging are harder;
- changing the sequence affects several consumers.

### 21.2 Orchestration

A durable orchestrator stores the workflow state and sends commands:

~~~text
                  +----------------------+
                  | Saga Orchestrator    |
                  | state: PAYMENT_WAIT  |
                  +----------------------+
                    |       |        |
             reserve|  authorize     |confirm
                    v       v        v
                Inventory Payment   Order
~~~

Advantages:

- one explicit state machine;
- easier timeouts, retries, compensation, and observability;
- clearer ownership for complex branching.

Costs:

- orchestration logic is another system to design and operate;
- a naive single process is a failure point;
- the orchestrator can accumulate too much business logic.

“Central coordinator” does not automatically mean “single machine.” Replicated workers plus durable workflow state can make the logical coordinator highly available. Azure’s Saga guidance distinguishes choreography and orchestration and calls out durable state, idempotence, monitoring, and isolation anomalies ([Azure Saga pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/saga)).

## 22. Making a Saga reliable

A diagram with arrows is not yet a reliable Saga. The following mechanics close the failure gaps.

### 22.1 Explicit durable state

Store each instance as a state machine:

~~~text
saga_id: checkout-91
state: PAYMENT_PENDING
order_id: O91
inventory_operation_id: reserve-91
payment_operation_id: authorize-91
attempt: 3
deadline: ...
last_error: timeout
version: 7
~~~

On orchestrator restart, another worker resumes from state, not from memory.

### 22.2 Transactional outbox

A service often needs to update its database and publish an event. Two independent writes create a gap:

~~~text
DB commit succeeds -> process crashes -> event never published
event published -> DB rolls back       -> event describes a fact that did not happen
~~~

Store the business change and an outbox row in one local transaction:

~~~text
BEGIN
  update order state
  insert outbox(message_id, type, payload)
COMMIT

relay later publishes the outbox message
~~~

The relay can publish twice, so consumers still deduplicate. See the repository’s [Messaging Queues chapter](./15.%20Messaging%20Queues.md) and the [Transactional Outbox pattern](https://microservices.io/patterns/data/transactional-outbox).

### 22.3 Idempotent participant operations

Every command has a stable operation ID and a stored result:

~~~text
ReserveInventory(operation_id = reserve-91, order_id = O91)
~~~

The same operation ID must not reserve twice. A new intentional reservation receives a new ID.

### 22.4 Inbox/deduplication and monotonic transitions

Participants can store processed message IDs or enforce a unique business operation key. State transitions should reject stale commands:

~~~text
PAYMENT_AUTHORIZED -> PAYMENT_REFUNDED     allowed
PAYMENT_REFUNDED   -> PAYMENT_AUTHORIZED   reject stale late success
~~~

Use optimistic versions or compare-and-set updates so an old retry cannot move a completed workflow backward.

### 22.5 Unknown outcomes

On timeout:

1. mark the outcome UNKNOWN;
2. query the participant by operation ID;
3. retry idempotently if appropriate;
4. compensate only after the domain’s evidence and policy permit it.

“Timeout equals failure” is one of the most dangerous payment and reservation mistakes.

### 22.6 Compensation can fail

If ReleaseInventory fails:

- retain a COMPENSATION_PENDING state;
- retry with backoff for transient failure;
- make the compensation idempotent;
- alert when its age exceeds the business objective;
- reconcile authoritative systems; and
- route an irrecoverable case to manual review with a complete audit trail.

A dead-letter queue stores a failed message. It does not repair the business invariant.

### 22.7 Reconciliation

A periodic reconciler finds silent divergence:

~~~text
orders CONFIRMED but no payment authorization
payments CAPTURED but order CANCELLED
inventory reservation expired while order still PENDING
workflow state unchanged beyond deadline
~~~

Recovery paths deserve the same design attention as the happy path.

## 23. Saga isolation problems

Because local steps commit independently, another request can observe or change intermediate data.

### 23.1 Dirty business read

Order O91 is PENDING with reserved stock. Another workflow treats the reservation as a final sale even though payment later fails.

### 23.2 Lost update

Two Sagas read the same available quantity and overwrite one another.

### 23.3 Nonrepeatable decision

A later step re-reads data that changed after an earlier validation.

### 23.4 Countermeasures

- **Semantic lock/status:** mark an entity PENDING so other workflows apply special rules.
- **Reservation with expiry:** hold stock or funds for a bounded period.
- **Optimistic version check:** update only if the expected version still matches.
- **Commutative update:** use an increment/decrement or ledger entry rather than overwrite.
- **Reread before irreversible action:** validate current authoritative state near the pivot.
- **Partition/serialize by entity:** process one order/account key in order where justified.
- **Redesign ownership:** move a hard invariant under one transactional owner.

A Saga is not automatically correct merely because every step has compensation.

## 24. When Saga fits and when it does not

### Use a Saga when

- a business operation spans independent service-owned stores;
- temporary intermediate states are acceptable and can be modeled;
- participants cannot join one atomic transaction;
- the workflow is long-running;
- meaningful compensation or forward recovery exists; and
- availability and service autonomy matter more than instantaneous global isolation.

### Avoid or reconsider a Saga when

- users must never observe an intermediate state;
- one invariant requires immediate atomic enforcement;
- compensation is impossible or legally/business-wise unacceptable;
- the workflow has no durable owner, idempotency, or reconciliation plan; or
- the service split created artificial cross-service chat and transactions.

Possible better choices:

- one local transaction in a modular monolith or larger service;
- a distributed database that exposes the required ACID transaction;
- a reservation/Try-Confirm-Cancel protocol;
- an append-only ledger plus derived views; or
- asynchronous projection when the second write is only a search/cache/read model.

The repository’s [Saga, Strangler Fig, and CQRS chapter](./04.%20Saga,%20Strangler%20Fig%20and%20CQRS.md) covers Saga workflow implementation and its relationship to service data ownership in greater breadth.

## 25. Saga interview answer spine

Use **L-C-D-I-R**:

1. **Local commits:** every participant performs a short local transaction.
2. **Compensation/pivot:** define how completed steps are counteracted and where forward recovery replaces undo.
3. **Durable coordination:** choreography or an orchestrator, explicit states, outbox, correlation IDs.
4. **Idempotency and isolation:** duplicates, unknown outcomes, versions, reservations, semantic locks.
5. **Recovery:** compensation retries, reconciliation, observability, and manual repair.

One compact sentence:

> “A Saga gives up one hidden cross-service commit and instead makes intermediate business states, compensation, idempotency, and recovery explicit.”

# Part IV — Choosing the right mechanism

## 26. These mechanisms are not always direct competitors

One system can use several layers:

~~~text
Checkout Saga
   |
   +--> Payment Service local ACID transaction
   |
   +--> Inventory Service backed by a distributed SQL database
                               |
                               +--> internal 2PC across replicated shards
~~~

The application may use a Saga between services while a database internally uses consensus and 2PC. “We use Saga” does not imply “there is no 2PC anywhere,” and the reverse is also true.

## 27. Decision table

| Question | Local transaction | 2PC | 3PC | Saga |
|---|---|---|---|---|
| Are partial states visible? | No within isolation boundary | Normally no global partial commit | Normally no under assumptions | Yes, as explicit business states |
| Must all participants support a protocol? | One resource only | Yes | Yes | No common DB protocol; service contracts still required |
| Failure recovery | DB rollback/recovery | Durable prepare and global decision | Extra pre-commit termination logic | Retry, compensate, reconcile |
| Blocking risk | Local locks/contention | Prepared participants can block | Reduced for modeled crashes | No global prepared lock, but workflows can be stuck |
| Network partition posture | Depends on DB | Preserve safety; may lose availability | Non-blocking claim does not survive arbitrary partitions | Can accept work selectively; convergence is application-defined |
| Isolation | Supplied by DB | Must be designed across participants | Must be designed across participants | Not automatic; semantic controls needed |
| Long-running work | Poor fit if locks retained | Poor fit | Usually poor fit | Natural fit |
| External side effects | Usually outside transaction | Only if resource is a real participant | Same compatibility issue | Model idempotency and compensation |
| Main complexity | Schema/concurrency | Coordinator, logs, prepared recovery | Assumptions, extra phase, termination | Business state machine and compensation |

## 28. Alternatives and complements

### 28.1 Change the service boundary

Put facts that enforce one invariant under one owner. Publish events after the local commit for other services.

### 28.2 Distributed SQL/database-native transaction

Let a database designed for distributed ACID hide much of the protocol. The trade-off does not disappear; it moves into the database’s latency, quorum, partition, and locality behavior.

### 28.3 Try-Confirm/Cancel (TCC)

TCC exposes reservation semantics:

~~~text
TRY:     reserve stock / hold funds
CONFIRM: make reservations final
CANCEL: release reservations
~~~

It is application-aware and useful when resources naturally support holds. It still needs deadlines, idempotency, and recovery.

### 28.4 Transactional outbox

When the “second system” is a broker, cache, search index, analytics store, or notification service, often only the authoritative database write must be atomic. Commit it with an outbox, then update derivatives asynchronously.

### 28.5 Append-only ledger

For money or inventory, record immutable movements with unique operation IDs rather than destructively overwriting a balance. Corrections become new entries; balances are derived. This improves auditability but still requires correct atomic ownership and concurrency control.

## 29. Practical decision flow

~~~text
What invariant must be preserved?
              |
              v
Can one authoritative owner enforce it in one local transaction?
        | yes                           | no
        v                               v
 Use local ACID                 Must partial states be hidden?
 + outbox for derivatives          | yes             | no
                                   v                 v
                      Are all participants       Model a Saga/TCC
                      short-lived and protocol-  with explicit states,
                      compatible?                compensation, and recovery
                         | yes       | no
                         v           v
                    Consider 2PC   Revisit boundary or
                    or a proven    use reservations /
                    distributed DB business workflow

If considering 3PC:
  state the failure/timing assumptions and platform support first.
~~~

## 30. Trade-off sentence to use in an interview

> “I would start by keeping the invariant local. If strict atomicity must span compatible short-lived resources, I would consider a proven 2PC-capable database or transaction manager and accept coordination/availability cost. If services are autonomous or work is long-running, I would model a Saga with named intermediate states, idempotent steps, compensation, and reconciliation. I would not claim 3PC solves partitions; its non-blocking property depends on stronger assumptions.”

# Part V — Real-system reasoning

## 31. Evidence boundary

The Google Spanner and PostgreSQL examples above are verified from public first-party material.

The following WhatsApp, YouTube, Uber, Netflix, Twitter/X, and Amazon examples are **illustrative product designs**. They show how to reason in an interview; they are not claims about those companies’ current private implementations.

## 32. Amazon-like order checkout

### Likely choice: Saga plus local transactions

~~~text
                    Checkout Orchestrator
                   /         |          \
                  v          v           v
             Order DB   Inventory DB   Payment provider
              PENDING    RESERVED       AUTHORIZED
                   \         |          /
                    +---- CONFIRMED ----+
~~~

Why not broad 2PC?

- an external payment provider may not support prepare/commit;
- fulfillment can last a long time;
- stock and payment have natural reserve/void/refund operations;
- the user can be shown PENDING rather than locking resources indefinitely.

Important follow-ups:

- expire reservations;
- use a payment idempotency key;
- distinguish authorization from capture;
- query unknown outcomes;
- compensate or route to manual review;
- never make “send confirmation email” part of atomic correctness.

## 33. Uber-like split payment

A ride costs Rs 1,000: Rs 100 from an internal wallet and Rs 900 from a card.

~~~text
Trip Billing Saga
   |
   +--> reserve Rs 100 wallet credit
   +--> authorize Rs 900 card
   +--> capture/settle according to trip state
~~~

Failure questions:

- Card declines: release wallet reservation.
- Card times out: query by operation ID before retrying or releasing.
- Wallet commits but card later succeeds after compensation starts: enforce versioned state and a single billing operation ID.
- Refund fails: keep REFUND_PENDING, retry, reconcile, alert, and expose support status.

If wallet debit and credit belong to one internal ledger, keep that ledger movement local. Do not distribute what can remain one ledger transaction.

## 34. WhatsApp-like message send

A message write, push notification, recipient delivery, and read receipt do not need one atomic global commit.

~~~text
Sender -> Message Service
              |
              +-- local transaction: store message + outbox
              |
              +-- async delivery ---> recipient devices
              |
              +-- later status <----- delivered/read receipts
~~~

Why a Saga/2PC may be unnecessary:

- recipient devices can be offline;
- delivery is retryable, not something to hold in a database prepare state;
- SENT, DELIVERED, and READ are legitimate monotonic states;
- a notification is derivative, not part of message durability.

This is a state machine plus reliable messaging, not “rollback the send if one phone is offline.”

## 35. YouTube-like upload pipeline

~~~text
UPLOADING -> STORED -> TRANSCODING -> READY
                |           |
                |           +--> retry failed rendition
                +--> cleanup abandoned chunks after expiry
~~~

Video storage, transcoding, thumbnail generation, moderation, indexing, and notification are long-running. A durable workflow with idempotent tasks fits better than 2PC.

Not every failed derived task requires compensating the original upload. If one thumbnail rendition fails, forward retry may be better than deleting the video.

## 36. Netflix-like subscription activation

~~~text
Create subscription PENDING
   -> authorize/capture payment
   -> activate entitlement
   -> publish confirmation
~~~

A Saga can void/refund payment if entitlement activation reaches a nonrecoverable failure. After the payment/entitlement pivot, forward recovery may be better than immediately revoking access.

Playback telemetry, recommendations, and “continue watching” updates are derivatives; they normally should not join the subscription’s strict transaction.

## 37. Twitter/X-like post fan-out

~~~text
Post API -> authoritative Post DB + outbox
                            |
                            +--> timeline fan-out
                            +--> search index
                            +--> notification
                            +--> analytics
~~~

The post’s authoritative creation can be one local transaction. Timeline and search copies can converge asynchronously.

Using 2PC across millions of derived timeline writes would couple post availability to a huge fan-out. A delete is also a new fact propagated to derived stores; it is not time travel that guarantees nobody ever saw the original.

## 38. The recurring product lesson

Classify effects into:

| Effect type | Example | Usual treatment |
|---|---|---|
| Authoritative invariant | ledger entries, confirmed reservation | local ACID or proven distributed ACID |
| Reversible reservation | stock hold, card authorization | Saga/TCC with expiry |
| Derived view | timeline, search index, recommendation | outbox/event and asynchronous rebuild |
| Retryable side effect | notification, transcoding | durable queue/workflow |
| Irreversible external action | shipment dispatch, sent email | delay until pivot; use forward recovery/correction |

This classification is usually more valuable than naming a protocol immediately.

## 42. Fast follow-up drills

### Q: Why is 2PC blocking?

After voting YES, a participant has promised to accept the coordinator’s decision. If it cannot learn whether COMMIT was chosen, it cannot safely commit or abort, so it retains prepared state and possibly locks.

### Q: Can the participant abort after a timeout?

Before voting YES, often yes according to the protocol. After a durable YES, not unilaterally: another branch may already have committed. It must learn the decision or accept a heuristic consistency risk.

### Q: Why does 3PC add PRE_COMMIT?

It separates an early abort-safe state from a later commit-safe state so surviving sites can terminate after modeled crash failures. That reasoning relies on reliable communication/failure detection and does not safely generalize to arbitrary partitions.

### Q: Why not use Saga everywhere?

Saga exposes intermediate states, lacks automatic isolation, needs meaningful compensation, and increases application/recovery complexity. A local transaction is stronger and simpler when the invariant fits one owner.

### Q: Why not use 2PC everywhere?

All resources must participate, prepared work retains resources, latency grows with coordination and durability, and partitions/coordinator uncertainty reduce availability. External and long-running actions are usually poor participants.

### Q: Does a refund restore the original state?

Not exactly. It is a new auditable transaction; fees, exchange rates, timing, or other concurrent actions can differ.

### Q: If the orchestrator is down, is the Saga lost?

It should not be. Durable workflow state, replicated workers, stable operation IDs, and outbox/inbox records let another worker resume. A purely in-memory orchestrator is not production-safe.

### Q: Are 2PC and two-phase locking the same?

No.

- **2PC** coordinates commit/abort across participants.
- **Two-phase locking (2PL)** is a concurrency-control discipline for acquiring and releasing locks to provide serializability properties.

A system can use both, either, or neither.

# Part VII — Common mistakes, edge cases, and review sheet

## 43. Common mistakes corrected

| Weak or incorrect statement | Interview-ready correction |
|---|---|
| “A timeout means the service failed” | “A timeout means the caller does not know the result; query or retry idempotently” |
| “2PC gives all ACID properties” | “2PC coordinates atomic commit; isolation, consistency rules, and durability also depend on the participants/system” |
| “After YES, a participant can time out and roll back” | “After YES it is in doubt and cannot decide alone without risking mixed outcomes” |
| “The coordinator is just a single point of failure” | “It is logically central; durable replicated state reduces runtime failure, but participants can still be cut off from the decision” |
| “3PC is always non-blocking” | “Only under its failure/communication assumptions; arbitrary partitions break that claim” |
| “3PC means three databases” | “It means three protocol phases, independent of participant count” |
| “Saga rolls back databases” | “Saga commits local steps and later commits semantic compensations” |
| “Compensation restores the exact old world” | “It restores a valid business state while preserving history and concurrent effects” |
| “DLQ solved the failed compensation” | “It only isolated a message; the business obligation remains open” |
| “Exactly-once messaging prevents duplicate charges” | “Business idempotency at the effect boundary is still required” |
| “Kafka replaces 2PC” | “Broker transactions have a defined scope; unrelated databases/APIs do not automatically join it” |
| “Microservices require Saga” | “Only cross-owner workflows do; reconsider boundaries and keep local invariants local” |
| “Every Saga failure should compensate” | “Retry or forward recovery may be safer, especially after the pivot” |
| “2PC and Saga cannot coexist” | “They can operate at different layers or for different workflows” |

## 44. Edge cases an interviewer may inject

### Duplicate client request

Use a client idempotency key tied to the intended business operation. Return the existing transfer/order result.

### Coordinator crashes after sending COMMIT to only one participant

The durable global decision must be COMMIT because it was logged before sending. Recovery resends COMMIT to the others. A participant still waiting must not guess ABORT.

### Participant forgets prepared state after restart

The protocol is broken. A YES vote must have durable recovery state.

### Late success arrives after Saga compensation began

Use operation IDs, versioned monotonic states, and participant status queries. A late event must not blindly overwrite COMPENSATING or REFUNDED.

### Compensation becomes impossible

Move to MANUAL_REVIEW or another explicit exception state, alert, preserve evidence, and apply a domain-approved correction. Do not falsely label the Saga CANCELLED.

### Two Sagas reserve the last item

Inventory must enforce the stock invariant locally with an atomic conditional update, lock, or serialized ledger/reservation rule. Saga coordination cannot repair an oversell caused by a broken local invariant.

### One participant is read-only

Some 2PC interfaces allow a read-only vote/optimization so that participant needs no second-phase commit work. Treat this as an implementation optimization, not a change to the core safety reasoning.

### A transaction spans regions

Include WAN latency, participant availability, failure domains, lock duration, and data-locality design. Do not state a universal maximum region or participant count.

## 45. Metrics and alerts

### 2PC / distributed database transaction

- prepared/in-doubt transaction count;
- age of oldest prepared transaction;
- prepare-to-decision latency;
- commit/abort/heuristic outcome rate;
- locks and blocked queries caused by prepared transactions;
- coordinator recovery backlog;
- participant timeout and decision-redelivery rate.

### Saga

- workflow completion, failure, compensation, and manual-review rate;
- age in every state;
- oldest outbox/inbox message;
- retries and unknown-outcome count;
- compensation age and failure count;
- reconciliation mismatches;
- duplicate/idempotency conflict count;
- end-to-end p50/p95/p99 workflow time, with business-specific objectives.

Metrics need dimensions such as workflow type, participant, state, error class, and version. Never put card details or other sensitive payloads into logs.

## 46. Compact comparison cheat sheet

~~~text
LOCAL TRANSACTION
  one owner, strongest simple option
  keep hard invariants here when possible

2PC = PREPARE + DECIDE
  YES is a durable promise
  all YES -> COMMIT; any NO -> ABORT
  prepared participant can block while decision is unknown
  requires compatible short-lived transactional resources

3PC = CAN_COMMIT + PRE_COMMIT + DO_COMMIT
  adds a buffer state and termination path
  non-blocking only under stronger network/failure assumptions
  extra phase adds cost; partitions defeat the simple claim

SAGA = LOCAL COMMITS + RECOVERY
  explicit intermediate states
  compensate before pivot; retry/forward recover after pivot
  choreography or durable orchestration
  outbox + idempotency + unknown-outcome query + reconciliation
  no automatic cross-service isolation
~~~

### Study next — related, not prerequisites

1. **Consensus in depth:** Raft/Paxos, quorums, leader terms, replicated logs, and why consensus does not make external side effects transactional.
2. **Distributed database internals:** sharding, per-shard consensus, 2PC across shards, timestamp ordering, and serializable isolation.
3. **Durable workflow engines:** timers, leases, workflow replay/versioning, human tasks, and exactly-once illusions.
4. **Try-Confirm/Cancel and escrow:** application reservations, expiry, and high-contention inventory/money invariants.
5. **Reconciliation engineering:** invariant checks, repair ledgers, audit trails, and operational runbooks.
