# Microservices: Monoliths, Service Size, and Domain-Driven Decomposition

> Beginner-first HLD study notes expanded from the supplied bullets. The goal is to learn how to justify an architectural boundary, not to memorize that "microservices scale."

## Scope, assumptions, and evidence labels

Your bullets describe breaking a monolith into loosely coupled, independently deployable services by using Domain-Driven Design (DDD), EventStorming, subdomains, and bounded contexts while avoiding a distributed monolith. I interpret “take out all the events, sequence them, and divide by bounded context” as a **discovery workflow**, not a mechanical rule that turns every event or bounded context into a network service.

This chapter explains both decomposition by business capability and decomposition by DDD subdomain/bounded context. These approaches overlap, but they are not identical; EventStorming supplies collaborative evidence that helps refine both.

Evidence labels used in this chapter:

- **Verified technical guidance** means the claim is backed by a first-party engineering source, an architecture guide, or a well-known original pattern source linked nearby.
- **Public production snapshot** means a company publicly described that architecture at a particular time. It is not a claim about the company's private architecture today.
- **Illustrative design** means a reasonable interview decomposition for a named product. It is not a claim that WhatsApp, YouTube, Netflix, Amazon, Uber, or Twitter/X currently uses those exact services.
- **Self-reported interview experience** means a candidate publicly reported the question. It is useful evidence of question style, but it is not an independently verified company question bank.

Specific service counts and team-size rules age quickly. Where one is useful, the date and confidence are stated explicitly.

## 1. Plain-language foundation

### 1.1 Start with a simple application

Imagine a small online bookstore. It must:

- show books;
- let users add books to a cart;
- accept an order;
- take payment;
- arrange delivery; and
- send a confirmation.

All of that business behavior must live somewhere in software. The first architectural question is not "Which cloud service should I use?" It is:

> How should I divide the software into parts that people can understand, change, deploy, and operate safely?

Two broad answers are **a monolith** and **microservices**.

### 1.2 Four words that must not be mixed up

Before comparing the architectures, distinguish these terms:

| Term | Plain meaning | Example |
|---|---|---|
| **Module** | A logical code boundary inside an application | `orders`, `catalog`, or `payments` package |
| **Process** | A running operating-system program | One Java/JVM or Node.js process |
| **Deployment unit** | A versioned artifact released as one unit | One JAR, executable, container image, or serverless function |
| **Instance** | One running copy of a deployment | Five running copies of the Order Service |

One service can have many instances. One deployment can contain many modules. A container is a packaging/runtime mechanism; it does not automatically make the code a microservice.

### 1.3 What is a monolith?

A **monolithic application** packages the application's main capabilities into one deployable unit. The bookstore might have clean internal modules, but catalog, cart, orders, and payments are built and released together.

```text
Clients
   |
   v
+-------------------------------------------+
|           Bookstore application           |  one deployment
|                                           |
|  Catalog | Cart | Order | Payment | Email |
+----------------------+--------------------+
                       |
                       v
                +--------------+
                | one database |
                +--------------+
```

A monolith does **not** necessarily mean:

- one source-code file;
- bad design;
- one server;
- no load balancer;
- no caching;
- no background jobs; or
- inability to handle large traffic.

Several identical instances of a stateless monolith can sit behind a load balancer:

```text
                    +--> monolith instance 1 --+
Client -> LB/API ---+--> monolith instance 2 --+--> database/cache
                    +--> monolith instance 3 --+
```

That is horizontal scaling. The application is still monolithic because all capabilities are deployed together.

### 1.4 A modular monolith: the important middle option

A **modular monolith** is one deployment with deliberately enforced internal business modules.

```text
+---------------------------------------------------+
|                  one deployment                   |
|                                                   |
|  +---------+    public interface    +----------+  |
|  | Catalog | ---------------------> |  Orders  |  |
|  +---------+                        +----------+  |
|       ^                                  |        |
|       |                                  v        |
|  +---------+                        +----------+  |
|  | Search  |                        | Payments |  |
|  +---------+                        +----------+  |
|                                                   |
|  Modules cannot reach into each other's internals |
+---------------------------------------------------+
```

It keeps local function calls, straightforward testing, and usually one operational unit while creating boundaries that may later be extracted. The discipline matters: if every module imports every other module's internals and all code freely edits all tables, it is a **big ball of mud**, not a useful modular monolith.

### 1.5 What is a microservice architecture?

A **microservice architecture** builds one product as a set of services that:

- represent focused business responsibilities;
- communicate over a network through explicit contracts;
- can be changed and deployed independently in the intended operating model;
- own their behavior and normally own their data;
- are operated by teams accountable for their production behavior; and
- can be replicated/scaled independently when useful.

Martin Fowler and James Lewis described the style as independently deployable services built around business capabilities, with decentralized control and automated deployment ([Fowler/Lewis, *Microservices*](https://martinfowler.com/articles/microservices.html)).

```text
                          +------------------+
Client -> API Gateway --->| Catalog Service  |---> Catalog DB
                 |        +------------------+
                 |
                 +------->+------------------+
                 |        | Cart Service     |---> Cart DB
                 |        +------------------+
                 |
                 +------->+------------------+      +------------+
                          | Order Service    |----->| event bus  |
                          +------------------+      +-----+------+
                                  |                       |
                              Order DB             +------+------+
                                                   v             v
                                              Payment       Notification
                                               Service         Service
                                                  |               |
                                             Payment DB       Message log
```

The services together are still **one distributed application** from the user's point of view. Moving code into different processes does not remove dependencies; it turns some code dependencies into network, data, contract, and organizational dependencies.

### 1.6 The defining word is not "small"

There is no correct number of:

- lines of code;
- API endpoints;
- database tables;
- classes;
- containers; or
- engineers

that turns something into a microservice.

The useful goal is:

> a cohesive business boundary that can evolve and operate with meaningful independence.

"Micro" is a warning against a service becoming an unmanageable mini-monolith. It is not a command to make every service tiny.

### 1.7 What microservices are not

| Common confusion | Correction |
|---|---|
| "Every REST API is a microservice" | A network API may expose a monolith, a macroservice, or a microservice. The boundary and independence matter. |
| "A Docker container is a microservice" | A container is packaging. A badly coupled monolith in 30 containers can be worse than one clean deployment. |
| "One table equals one service" | Tables are storage structures, not business capabilities. This often creates chatty CRUD services. |
| "Each service must use a different language/database" | Technology freedom is an option, not a requirement. Standardization often lowers operating cost. |
| "Serverless means microservices" | Functions are a compute/deployment model. Many functions can implement one service, or one tangled application. |
| "Microservices automatically mean event-driven" | Services can use synchronous requests, asynchronous messages, or both. |
| "A repository is a service" | Repo boundaries and runtime/deployment boundaries are separate decisions. A monorepo can contain independent services. |

### 1.8 The distributed monolith anti-pattern

A **distributed monolith** has multiple deployables but keeps monolithic coupling. Typical signs are:

- services must be released together;
- one request crosses a long chain of synchronous calls;
- services directly query or modify one shared schema;
- a change to one domain requires changes in many repositories;
- local development needs the entire fleet;
- one service failure makes the whole product unusable; and
- no team can own a feature end to end.

```text
Client -> A -> B -> C -> D -> E -> shared database
          all must be up; all versions move together
```

This pays network and operational costs without gaining deployment or team autonomy.

## 2. Inline prerequisites needed for the comparison

These concepts are explained here because monolith-versus-microservices reasoning depends on them.

### 2.1 Local call versus network call

Inside one process, a module can call a function directly:

```text
OrderModule.placeOrder() -> InventoryModule.reserve()
```

Across services, the call becomes a network interaction:

```text
Order Service --HTTP/gRPC--> Inventory Service
```

The remote call adds:

- serialization and deserialization;
- network latency;
- authentication/authorization between services;
- timeouts;
- retries and possible duplicate operations;
- versioned contracts; and
- partial failure: caller healthy, callee unhealthy, network uncertain.

A remote call is not merely a slower function call. It has different failure semantics.

### 2.2 API/contract

An **API contract** says what callers may send, what they can receive, and what errors/compatibility rules apply. HTTP/JSON, gRPC/Protobuf, and event schemas are contract mechanisms.

Independent deployment requires compatible evolution. If Service A cannot deploy until B changes in lockstep, their operational independence is weak even if they live in separate repositories.

### 2.3 Database transaction and invariant

A database transaction commonly provides an all-or-nothing boundary for several changes. An **invariant** is a rule that must always remain true, such as:

- inventory cannot fall below zero;
- a payment is captured at most once for an idempotency key; or
- an order total equals its accepted line-item prices.

In a monolith using one relational database, one transaction can often protect an invariant across modules. With database-per-service, a normal local database transaction cannot atomically update several services' stores. The business workflow needs a distributed approach such as a **saga**, introduced in Section 13.

### 2.4 Availability and failure isolation

**Availability** means the required operation can provide an acceptable response. **Failure isolation** means a failure's effect is contained within an intended boundary.

Separating processes can help isolation, but it does not guarantee it. If checkout synchronously requires recommendations, a Recommendation Service outage can still break checkout. The architecture must use:

- optional dependencies and graceful degradation;
- strict timeouts;
- bounded retries;
- circuit breakers;
- resource isolation/bulkheads; and
- capacity planning.

### 2.5 Deployment independence

Independent deployment means a team can safely release a compatible service change without coordinating a simultaneous fleet-wide release. It depends on more than separate artifacts:

- backward/forward-compatible contracts;
- automated build/test/deployment;
- safe database migrations;
- feature flags or progressive rollout;
- fast rollback or roll-forward; and
- observability.

### 2.6 Service data ownership

"Database per service" is fundamentally an **ownership rule**:

> Only the owning service may interpret and mutate its private data model directly; other services use its API or events.

The stores may initially share a physical database server for cost reasons, but separate schemas/credentials and enforced ownership preserve the boundary. Giving every service write access to every table removes autonomy.

### 2.7 Synchronous and asynchronous communication

- **Synchronous request/response:** the caller waits for a result. Useful when the result is required now, but it adds runtime coupling and latency.
- **Asynchronous messaging:** the producer publishes a command/event and continues; a consumer processes it later. This can reduce temporal coupling and absorb bursts, but introduces eventual consistency, duplicates, ordering questions, and harder debugging.

Neither is universally better. The business requirement decides which delay and failure behavior is acceptable.

## 3. Why microservices are needed in system design

Microservices mainly address the difficulty of changing and operating a **large, complex product built by many teams**. Traffic matters, but it is not the only or always the first driver.

### 3.1 The scaling problem has several meanings

When someone says "the monolith does not scale," ask what is failing to scale:

| Kind of scale | Actual problem | How microservices might help |
|---|---|---|
| **Traffic scale** | One capability receives disproportionate load | Replicate only that service |
| **Data scale** | Different data has different access/retention needs | Give domains separate stores and partitioning policies |
| **Development scale** | Too many teams change one codebase/release train | Give teams owned deployable boundaries |
| **Change scale** | Small changes require broad regression/release work | Localize change and deployment |
| **Reliability scale** | One fault or resource leak risks everything | Isolate processes/resources and degrade selectively |
| **Security/compliance scale** | Sensitive and public paths need different controls | Isolate trust/data boundaries |
| **Technology/lifecycle scale** | One stack cannot fit all workloads or upgrades | Allow justified per-service choices |

A monolith can often solve traffic scale by adding instances, caches, queues, replicas, and good database design. Microservices become compelling when **independent ownership and evolution** are important as well.

### 3.2 A concrete example: product catalog versus video transcoding

Suppose a marketplace adds seller videos:

- Catalog reads are request/response, read-heavy, and latency-sensitive.
- Video transcoding is CPU-heavy, asynchronous, bursty, and may take minutes.

Keeping both in one deployment means scaling catalog web instances to acquire transcoding CPU, or building complex resource isolation inside the process. Separate services let each use a suitable execution and scaling model:

```text
Catalog traffic -> Catalog Service -> catalog read store

Upload -> Media Service -> object storage -> queue -> Transcoder workers
                                                   scale by queue depth
```

This is a strong split because workload, failure mode, release cadence, and data lifecycle differ.

### 3.3 The organizational reason

If one team owns UI, another owns "backend logic," and another owns the database, a small order feature may require three handoffs. Business-aligned services instead aim for a cross-functional team that owns a capability from API through storage and production.

This is related to **Conway's Law**: system structures tend to resemble the communication structures of the organizations that build them. A good design intentionally aligns teams and business boundaries instead of allowing accidental organizational silos to dictate the architecture. Microsoft recommends aligning ownership to domain boundaries while revisiting either the boundary or team structure when one context spans many teams ([Azure domain analysis](https://learn.microsoft.com/en-us/azure/architecture/microservices/model/domain-analysis)).

### 3.4 The prerequisite capabilities: the microservice premium

Microservices do not create autonomy merely by being split. A team needs a platform and operating model that make many services manageable:

- CI/CD and automated testing;
- repeatable infrastructure provisioning;
- service discovery and traffic management;
- centralized logs, metrics, traces, and correlation IDs;
- alerting tied to service-level objectives;
- secret/configuration management;
- contract testing and compatibility rules;
- safe schema migration and release strategies;
- resilience patterns and load testing;
- clear service ownership/on-call responsibility; and
- incident response and dependency visibility.

Containers and Kubernetes can help operate services, but they are optional implementation choices, not the reason for the architecture.

## 4. Monolith versus microservices: the core comparison

The foundation comes first; this table is the interview-compressed view.

| Dimension | Monolith / modular monolith | Microservices |
|---|---|---|
| Deployment | One application releases together | Services aim to release independently |
| Calls | Mostly in-process: fast and simple | Network calls/messages: latency and partial failures |
| Data transactions | One-store ACID transactions are straightforward | Local transactions are simple; cross-service workflows need sagas/events/compensation |
| Scaling | Scale the deployment as a unit | Scale hot services independently |
| Fault boundary | Process/resource failure can affect all modules | Process isolation can limit blast radius, if dependencies degrade safely |
| Code refactoring | Moving code across modules is relatively easy | Moving responsibility/data across services is operationally harder |
| Team coordination | Easy for a small team; one release path becomes crowded at scale | Business-aligned teams can move autonomously; cross-service features still require coordination |
| Technology | Consistent stack, simpler upgrades/skills | Selective polyglot options, but more standards and operational burden |
| Testing | End-to-end/local integration is easier | Contract, component, integration, and production-like tests are more complex |
| Debugging | One process and often one trace | Requires distributed tracing and cross-service log correlation |
| Data querying | Joins and reporting are usually straightforward | Cross-domain queries need API composition, read models, or data pipelines |
| Infrastructure cost | Fewer deployables and lower baseline overhead | More compute overhead, networking, pipelines, observability, and ownership cost |
| Security | Fewer network surfaces; broad in-process trust can be risky | Fine-grained isolation possible; far more identities, endpoints, and secrets to secure |
| Best fit | Small/medium product, uncertain domain, small team, transactional simplicity | Complex domain, many autonomous teams, clear boundaries, uneven scaling/reliability needs |

### 4.1 Advantages of a monolith

1. **Simple mental and operational model.** One application is easier to run locally, deploy, observe, and roll back.
2. **Fast local interactions.** Function calls avoid network latency, serialization, and remote-failure handling.
3. **Straightforward data consistency.** A relational transaction can protect multiple related changes.
4. **Easy early refactoring.** When domain knowledge is incomplete, moving code and changing boundaries is cheaper inside one process.
5. **Lower platform cost.** Fewer pipelines, endpoints, credentials, dashboards, alerts, and idle capacity pools.
6. **Simpler end-to-end testing.** Most behavior can run in one test process/database.
7. **Consistent technology.** Shared libraries and one stack reduce cognitive load for a small team.

### 4.2 Disadvantages of a monolith

These become material as product and organization complexity grow:

1. **Deployment coupling.** A small catalog fix releases with unrelated order/payment code.
2. **Larger regression and blast surface.** One bad deployment, memory leak, or resource exhaustion can affect many capabilities.
3. **Coarse scaling.** If search alone is hot, the whole application may need replication.
4. **Team contention.** Many teams share a build, code ownership rules, and release calendar.
5. **Longer build/test/release cycles.** This is not inevitable, but a poorly structured large codebase accumulates cost.
6. **Boundary erosion.** Without enforced modules, expedient imports and shared tables create tangled dependencies.
7. **Technology lockstep.** A runtime/framework upgrade affects the entire application.

### 4.3 Advantages of microservices

Every advantage is conditional on good boundaries and operations:

1. **Independent change and deployment.** A team can release its capability at its cadence through a stable contract.
2. **Independent scaling.** Scale transcoding, search, or timeline fan-out without multiplying payment workers.
3. **Fault and resource isolation.** A leak in recommendations need not crash checkout; CPU-heavy work can have separate limits.
4. **Clear business ownership.** One accountable team owns behavior, data, SLOs, and production support.
5. **Smaller cognitive scope per team.** A team can understand its domain without loading the whole product into working memory.
6. **Selective technology choice.** Search may use a search index while payments use a relational database. This should be need-driven, not novelty-driven.
7. **Security and compliance isolation.** Payment/tokenization data can have stricter access, audit, and deployment controls.
8. **Replaceability and incremental modernization.** A stable interface lets the internal implementation evolve or be replaced gradually.

### 4.4 Disadvantages and hidden costs of microservices

1. **Distributed-systems failures.** Requests time out, arrive twice, arrive late, or succeed while the response is lost.
2. **Higher latency.** A call chain pays multiple network, queueing, serialization, and processing delays.
3. **Data consistency becomes a workflow concern.** One business operation may span independent databases and temporarily expose intermediate states.
4. **Operational fleet complexity.** Each service needs deployment, configuration, telemetry, ownership, patching, and capacity.
5. **Harder debugging.** A user error may cross a gateway, four services, a broker, and two databases.
6. **Contract/version evolution.** Producers and consumers deploy at different times; breaking changes require migration plans.
7. **Harder testing.** Test doubles can lie; full integration environments become expensive and flaky.
8. **Cross-service reporting/query difficulty.** Database joins do not cross service ownership safely.
9. **Duplicate platform code or over-centralized shared libraries.** Duplication raises maintenance cost; a shared library can recreate lockstep coupling.
10. **Security surface expansion.** More network endpoints, identities, policies, certificates, and supply-chain artifacts exist.
11. **Cost inefficiency at small scale.** Many services reserve baseline CPU/memory and generate telemetry/network cost.
12. **Service sprawl and ownership decay.** Tiny services become orphaned or require many touch points for one feature.

### 4.5 Advantages are not guarantees

| Claim | What must be true for it to hold |
|---|---|
| "Microservices improve reliability" | Failure is isolated, dependencies are optional/bounded, capacity is separated, and fallbacks exist. |
| "Teams deploy independently" | Contracts and database changes are compatible; shared libraries and release gates do not force lockstep. |
| "Services scale independently" | The bottleneck is actually inside that service; its database and dependencies can also scale. |
| "Different technologies improve flexibility" | The benefit exceeds training, patching, tooling, and on-call complexity. |
| "Small codebases are easier" | A business change does not require understanding twenty services and six teams. |

### 4.6 Your rough bullets, expanded and corrected

| Rough bullet | Interview-ready meaning |
|---|---|
| **Loosely coupled** | A service depends on a small, explicit, stable contract rather than another service's code, tables, deployment timing, or internal model. Loose coupling means **controlled dependency**, not no dependency. |
| **Independently code, test, deploy** | One team should be able to change and release its service without a coordinated fleet release. It still needs contract and integration tests because the product works as a whole. Separate repositories or containers alone do not create independence. |
| **Less communication overhead** | This is true only for **organizational coordination** when a team owns a capability end to end. Runtime communication overhead actually increases because local calls become network calls/messages; cross-service features can also increase team coordination when boundaries are poor. |
| **Scale independently** | A hot capability can add instances without scaling the whole application, provided its database and downstream dependencies can also handle the load. It is an option, not an automatic outcome. |
| **DDD approach** | Use business language, rules, subdomains, bounded contexts, and invariants to discover cohesive boundary candidates. DDD can organize a modular monolith too; it does not command a microservice deployment. |
| **EventStorming approach** | Collaboratively narrate business-significant events, expose missing knowledge and conflicting language, then use those signals to propose and test bounded contexts. It is discovery evidence, not a mechanical one-event/one-service mapping. |
| **Prevent a distributed monolith** | Give each service genuine data and decision ownership, keep contracts compatible, minimize mandatory synchronous chains, and verify that teams can deploy and operate independently. |

## 5. When to use, avoid, or delay microservices

### 5.1 Prefer a monolith or modular monolith when

- the product/problem fit is still uncertain;
- one small team can own the product;
- the domain and its boundaries are changing rapidly;
- simple transactions across several features dominate;
- traffic is modest or scales fairly uniformly;
- fast local development and low operating cost matter most;
- the organization lacks mature delivery/observability/on-call capabilities; or
- service isolation would not create meaningful independent ownership.

Starting with a monolith does not mean ignoring architecture. Use business modules, private data access, internal APIs, dependency rules, and automated tests. Martin Fowler's **monolith-first** argument is that early service boundaries are hard to get right and the operational premium is not justified for simpler systems; he also notes this is guidance based on experience rather than a universal law ([Monolith First](https://martinfowler.com/bliki/MonolithFirst.html)).

### 5.2 Prefer microservices when several of these are true

- many long-lived teams need genuinely independent roadmaps and releases;
- the product is complex enough to have stable business boundaries;
- capabilities have sharply different scale, latency, availability, security, or compute needs;
- release contention in the existing system is a measured bottleneck;
- process/resource isolation materially limits failure blast radius;
- a domain needs a different data lifecycle or justified technology;
- organizational ownership can align with services; and
- automation/observability/resilience foundations already exist or are deliberately funded.

### 5.3 Requirements that are weak reasons by themselves

- **"We expect millions of users."** A replicated monolith, caches, CDN, queues, and database scaling may be enough.
- **"We are moving to cloud/Kubernetes."** Infrastructure migration does not require domain decomposition.
- **"Microservices are modern."** Architecture is an economic trade-off, not a fashion label.
- **"We want different languages."** Polyglot systems increase cost and should follow a workload need.
- **"We need high availability."** A poorly designed service chain can reduce availability.
- **"The monolith code is messy."** Splitting tangled code can distribute the mess. First discover/enforce modules and ownership.

### 5.4 A practical decision scorecard

Use **SPLIT** in an interview:

1. **S - Scope and certainty:** Is the domain complex and understood, or still being discovered?
2. **P - People and pace:** How many autonomous teams? Are releases blocking one another?
3. **L - Load differences:** Do capabilities scale or consume resources differently?
4. **I - Isolation:** Are availability, security, compliance, and blast-radius boundaries different?
5. **T - Tooling and transaction cost:** Can the organization operate distributed services, and can the domain tolerate distributed workflows?

If the case is weak, propose a modular monolith with future extraction seams. That is a complete HLD decision, not a temporary embarrassment.

## 6. How big or small should a microservice be?

### 6.1 There is no numeric formula

Do not answer with line counts, table counts, endpoint counts, or a rigid team size. Historical phrases such as "two-pizza team" are organizational heuristics, not a service-size specification. Fowler and Lewis explicitly warned that the word *micro* causes an unhelpful focus on size ([How big is a microservice?](https://martinfowler.com/articles/microservices.html#HowBigIsAMicroservice)).

The right question is:

> Which responsibilities must stay together, and which can change, deploy, scale, fail, and be owned independently?

### 6.2 Cohesion and coupling: the central test

- **High cohesion:** responsibilities inside a service are strongly related and change for the same business reason.
- **Low coupling:** a change inside the service rarely forces simultaneous changes elsewhere.

Good boundary:

```text
Order Service
  create order
  validate order state transitions
  calculate accepted order total
  cancel order
  own order records
```

Suspicious boundary:

```text
CreateOrderService
UpdateOrderService
OrderStatusService
OrderLineService
OrderTotalService
```

Those tiny services likely share one invariant and call each other on every operation. They should probably be one cohesive Order Service.

### 6.3 Factors that determine service granularity

#### A. Business responsibility and reason to change

Keep things together when the same business policy changes them together. Separate them when different stakeholders, rules, and release cadences govern them.

Example: catalog descriptions and inventory quantities both mention a product, but they change for different reasons:

- merchandising changes title/images/category;
- warehouse operations changes stock/reservations.

That suggests different contexts even though both use a `productId`.

#### B. Transactional invariant and aggregate boundary

Data that must be atomically consistent usually belongs inside one service/aggregate boundary. Splitting it creates distributed coordination on the critical path.

Example: an order and its line items need one consistent accepted total. Do not create a remote LineItem Service solely because line items are a separate table.

#### C. Independent deployability

Ask: "Can I deploy this service alone without a synchronized release?"

If every change requires three services to deploy together, merge/rethink the boundary or introduce a stable contract. Deployment independence is a stronger measure than small code size.

#### D. Runtime chattiness and latency

If two proposed services exchange many fine-grained calls for one request, the boundary may cut through cohesive behavior.

```text
Bad checkout path:
Checkout -> Order -> OrderLine -> Price -> Tax -> Inventory -> Customer -> Address
             many sequential synchronous calls
```

Some calls are legitimate across contexts, but repeated, bidirectional, low-level calls are a warning. A coarse API, local copy/read model, asynchronous event, or merged boundary may be better.

#### E. Data ownership and access pattern

Separate when data has distinct ownership, retention, sensitivity, throughput, or storage access patterns. Keep together when both sides constantly require transactional joins and coordinated writes.

#### F. Scaling profile

A separately scalable hot path is a valid extraction candidate:

- search queries versus catalog administration;
- video transcoding versus metadata reads;
- timeline fan-out versus account settings.

But do not split only because one endpoint is currently slow. Profile the bottleneck; it may be a query/index/cache problem.

#### G. Reliability and blast radius

Separate work whose failure should not consume the critical path's resources. Notifications, analytics export, and recommendations can often fail or lag without failing order acceptance.

Conversely, if a "separate" service is mandatory for every request, the end-to-end availability is coupled.

#### H. Security, privacy, and compliance

Payment instruments, identity documents, or health data may deserve a smaller trust boundary, restricted team access, separate audit rules, and isolated deployment. More services also mean more network security work, so isolate intentionally.

#### I. Team ownership and cognitive load

A long-lived team should be able to understand and operate its owned capability. If five teams must edit one service, it may be too broad. If one team owns 80 unrelated tiny services and cannot keep them patched/on-call, the fleet is too fine-grained.

Do not let today's org chart blindly define the domain. Teams reorganize. Prefer stable business boundaries, then align teams to them where possible.

#### J. Technology/runtime need

Different compute or storage needs can support a split: GPU inference, stream processing, full-text search, or batch transcoding. "The team wants to try Go" is not sufficient.

#### K. Lifecycle and release cadence

Stable tax calculation and rapidly experimenting recommendations may deserve different release risk and ownership. Functions retired together often belong together.

#### L. Operational fixed cost

Every service adds pipelines, alerts, dashboards, security identities, dependency upgrades, on-call knowledge, capacity floors, and incident paths. A split must repay this continuing tax.

### 6.4 Signs a service is too large

- unrelated capabilities and vocabularies live together;
- several teams frequently conflict in the same code/release;
- most releases change only a small isolated area but retest/redeploy everything;
- one workload dominates resources for unrelated work;
- different data needs are forced into one model;
- failures regularly spread across unrelated capabilities;
- ownership is vague; or
- the code cannot be understood within one team's cognitive capacity.

### 6.5 Signs services are too small

- one user action crosses many sequential services;
- services mirror entities/tables rather than business behavior;
- services share a database or frequently join each other's tables;
- most changes touch several services and teams;
- APIs are fine-grained getters/setters;
- services cannot make useful decisions without repeatedly asking neighbors;
- distributed transactions are the default, not the exception;
- local development/testing requires the full fleet;
- more effort goes into contracts/operations than domain behavior; or
- services are always deployed together.

### 6.6 Split and merge are both valid evolution

Boundaries are hypotheses. As the domain changes:

- **split** when one part gains distinct ownership, load, policy, lifecycle, or failure needs;
- **merge** when two services always change/call/deploy together and no longer earn their boundary; and
- **move behavior** when deeper domain knowledge shows the original owner was wrong.

Architecture quality includes the ability to correct boundaries, not pretending the first diagram is permanent.

### 6.7 Service-size review checklist

Use **C-DATA**:

1. **C - Cohesion:** Does it express one coherent business responsibility and vocabulary?
2. **D - Deployability:** Can it release independently through compatible contracts?
3. **A - Atomicity/data:** Are required invariants and owned data inside the boundary?
4. **T - Traffic/team:** Does separate scaling and one-team ownership create value?
5. **A - Availability/operations:** Does it isolate failures enough to repay operational cost?

## 7. Decomposition pattern 1: by business capability

### 7.1 What is a business capability?

A **business capability** is something the business is able to do to create or support value. It describes **what the business does**, not which code framework or database it uses.

For an online marketplace:

- product catalog management;
- pricing and promotions;
- inventory management;
- order management;
- payment processing;
- fulfillment/shipping;
- customer support; and
- recommendations.

Good capability names often combine a meaningful business noun and responsibility. Avoid technical layers such as:

- UI Service;
- Business Logic Service;
- Database Service;
- Validation Service; or
- Utility Service.

Those layers force a business change through several teams/services.

### 7.2 Capability decomposition example

```text
Online commerce domain
|
+-- Sell products
|   +-- Catalog management
|   +-- Pricing and promotions
|   +-- Search/discovery
|
+-- Buy products
|   +-- Cart management
|   +-- Order management
|   +-- Payment processing
|
+-- Fulfill products
|   +-- Inventory management
|   +-- Shipment planning
|   +-- Returns/refunds
|
+-- Support customers
    +-- Accounts
    +-- Notifications
    +-- Customer service
```

This is a capability map, not yet a final list of deployables. A broad capability may remain a module, become one service, or be refined using domain analysis.

### 7.3 How to find capabilities

1. Start with business outcomes and end-to-end user journeys.
2. Ask domain experts what work the business performs.
3. List commands and important events: `PlaceOrder`, `PaymentAuthorized`, `ItemReserved`, `ShipmentDispatched`.
4. Group behavior that serves the same outcome and changes under the same policy.
5. Identify owners, data, external systems, regulations, and SLOs.
6. Draw dependencies and test whether groups can make decisions without fine-grained neighbor calls.

Potential discovery inputs include:

- business process maps;
- product roadmaps;
- user stories and support workflows;
- existing module/data dependencies;
- regulatory boundaries;
- expert interviews; and
- event-storming workshops.

### 7.4 Why it works

Business capabilities are generally more stable than a framework or UI layer. A cross-functional capability team can own its API, rules, data, deployment, and operations. AWS describes the pattern as a way to produce relatively stable, loosely coupled services and business-value-aligned teams, while warning that it needs deep business understanding ([AWS: Decompose by business capability](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-decomposing-monoliths/decompose-business-capability.html)).

### 7.5 Limitations and mistakes

- A capability may be too broad to be one service: "Sales Service" could become a new monolith.
- The organization chart may not match the true business boundaries.
- Two capabilities may share hidden invariants or data ownership.
- Naming a capability does not define its model or contract.
- A generic technical capability such as identity might be purchased/platform-provided, not custom-built as a product service.
- End-to-end workflows still cross capabilities; orchestration and consistency must be designed.

Capability decomposition is a strategic starting point. DDD gives a deeper vocabulary for finding model and consistency boundaries.

## 8. Domain-Driven Design (DDD) foundation from zero

### 8.1 DDD is not "make one service per domain noun"

**Domain-Driven Design** is an approach for tackling complex business software by building a model and shared language with domain experts. It asks the architecture and code to reflect the business's rules and distinctions.

DDD is especially useful when business rules are the hard part: pricing, lending, logistics, insurance claims, reservations, or marketplace matching. It can be excessive for a tiny CRUD admin tool with little domain logic.

DDD does not require microservices. A modular monolith can and often should use DDD boundaries.

### 8.2 Essential vocabulary

#### Domain

The overall problem area the software serves.

```text
Example domain: online retail
```

#### Subdomain

A meaningful part of that domain with its own knowledge and rules.

```text
Catalog, pricing, inventory, orders, fulfillment
```

DDD commonly classifies subdomains:

| Type | Meaning | Retail example | Investment tendency |
|---|---|---|---|
| **Core** | Creates competitive differentiation | personalized discovery or fulfillment optimization | Model carefully; keep strategic control |
| **Supporting** | Necessary and business-specific, but not the main differentiator | a custom seller-onboarding workflow | Build as needed without overengineering |
| **Generic** | Common problem solved similarly elsewhere | authentication, email delivery, payroll | Prefer a standard product/provider when suitable |

The classification depends on the business. Recommendations might be core for a streaming company and generic integration for a small retailer.

#### Domain model

A purposeful representation of the business concepts, behavior, and rules needed by the software. It is not a copy of every real-world detail or a database schema diagram.

#### Ubiquitous language

A shared, precise vocabulary used by engineers and domain experts in conversation, documentation, APIs, tests, and code.

For example, decide whether `reserve`, `allocate`, and `hold` mean different things. Ambiguous language often reveals ambiguous boundaries.

#### Bounded context

A **bounded context** is the boundary within which one domain model and language have a specific meaning.

The same real-world thing can have different models in different contexts:

```text
"Product" in Catalog context:
  title, description, images, category, attributes

"SKU" in Inventory context:
  warehouse, onHand, reserved, available

"Order line" in Ordering context:
  acceptedTitle, acceptedPrice, quantity, tax class
```

There should not be one giant `Product` object containing every field for every team. Context boundaries let each model evolve for its purpose.

#### Entity

An object defined by persistent identity, even when attributes change. An `Order` remains the same order after its status changes.

#### Value object

An object defined by its value rather than identity, commonly immutable: `Money(500, INR)`, an address snapshot, or a date range.

#### Aggregate and aggregate root

An **aggregate** is a group of domain objects protected as one consistency/transaction boundary. One entity is the **aggregate root**, the only entry point for changes.

```text
Order aggregate
|
+-- Order (root)
    +-- OrderLine 1
    +-- OrderLine 2
    +-- shipping-address snapshot

Invariant: accepted order total matches accepted lines.
One local transaction changes this aggregate.
```

An aggregate is not automatically a microservice. A service/bounded context can own many aggregate instances and several aggregate types.

#### Domain event

A business-significant fact that already happened:

- `OrderPlaced`;
- `InventoryReserved`;
- `PaymentAuthorized`; or
- `VideoTranscodingCompleted`.

"Row 17 updated" is a database fact, not a useful domain event. Events let contexts react without sharing internal tables, but published events require delivery, versioning, idempotency, and privacy design.

### 8.3 Problem space versus solution boundary

This distinction prevents a common mistake:

- **Subdomain** describes a part of the business problem space.
- **Bounded context** describes where a particular model/language applies in the software solution.
- **Microservice** is a runtime/deployment/ownership boundary.

They often align, but are not guaranteed to map one-to-one. One bounded context may initially be one module/service; a very large context may have several internal deployables behind one context-level API. Conversely, combining unrelated bounded contexts into one service weakens the boundary.

Microsoft's current Azure guidance calls each bounded context a microservice **candidate** and offers the heuristic "no smaller than an aggregate and no larger than a bounded context" ([Azure tactical DDD](https://learn.microsoft.com/en-us/azure/architecture/microservices/model/tactical-domain-driven-design)). Treat it as a useful design check, not a mathematical law.

### 8.4 Context map

A **context map** records how bounded contexts relate:

```text
[Catalog] --ProductPublished event--> [Search]
    |
    +--published product view--------> [Ordering]

[Ordering] --ReserveItems command---> [Inventory]
    |
    +--AuthorizePayment command------> [Payments]
```

It should make ownership and direction explicit. Ordering may store a snapshot of product name/price instead of reaching into Catalog's database.

### 8.5 EventStorming: discovering the boundaries collaboratively

#### 8.5.1 What it is, in plain language

**EventStorming** is a collaborative workshop in which business experts, product people, engineers, and other stakeholders build a visible story of how the business works. The main building blocks are facts that matter to the business, written as events in the past tense:

```text
OrderSubmitted -> ItemsReserved -> PaymentAuthorized -> OrderConfirmed
```

The goal is not to make a pretty architecture diagram. The goal is to reveal knowledge that was previously scattered across people, teams, code, tickets, and operating procedures:

- What actually happens, including exceptions?
- Why does it happen?
- Who or what starts it?
- Which rules make decisions?
- Which words mean different things to different people?
- Where is ownership unclear?
- Which facts must be immediately consistent?

The official EventStorming site describes it as a flexible workshop format for collaboratively exploring complex business domains. It can be used for organizational discovery, process modelling, or software design ([EventStorming official overview](https://www.eventstorming.com/)).

The official spelling is **EventStorming**, one word. You will still commonly see “event storming” in articles and conversations.

#### 8.5.2 The minimum notation you need

Colors are a useful workshop convention, but the meaning of each note matters more than memorizing colors. A common legend is:

| Building block | Plain meaning | Checkout example |
|---|---|---|
| **Domain event** | A business-significant fact that has already happened; use past tense | `OrderSubmitted`, `PaymentDeclined` |
| **Command** | An intention/request to make something happen; it may be rejected | `ReserveItems`, `AuthorizePayment` |
| **Actor** | A person or role that issues a command | Customer, support agent |
| **Policy** | “Whenever this happens, do that” business reaction | After `ItemsReserved`, request payment authorization |
| **External system** | A dependency outside the model or organization boundary | Payment provider, courier API |
| **Read model** | Information needed by an actor or rule to decide | Cart summary, available stock view |
| **Aggregate** | A local decision/consistency boundary that handles commands and emits events | Order, Stock Reservation, Payment |
| **Hot spot** | An uncertainty, disagreement, exception, risk, or unanswered question | “Can a coupon expire during payment retry?” |

Keep three distinctions clear:

1. A **command can fail**; an event states that something already happened.
2. A **domain event on the workshop wall is a modelling fact**, not automatically a Kafka topic or public integration event.
3. An **event-driven exploration does not require an event-driven implementation**. A final boundary may expose HTTP, gRPC, messages, or local module calls.

#### 8.5.3 Three useful zoom levels

EventStorming is a family of related workshop formats rather than one fixed recipe:

| Format | Scope | Main question | Likely output |
|---|---|---|---|
| **Big Picture** | A whole business line or broad domain | “How does this business actually work, and where are the conflicts/opportunities?” | Shared narrative, pivotal events, hot spots, ownership and boundary signals |
| **Process Modelling** | One end-to-end workflow and its variations | “How does this process reach a stable outcome?” | Happy path, alternate paths, policies, actors, decisions, unresolved cases |
| **Software Design** | A critical process in implementation-level detail | “Which models make decisions, protect consistency, and collaborate?” | Commands, events, aggregates, read models, boundaries, candidate contracts |

The official material recommends choosing or chaining these formats: use a broad exploration to find the important area, then zoom into one process and its software behavior. It also treats extracting bounded contexts as analysis performed from the validated narrative, not as an automatic sticky-note grouping ([Avanscoperta EventStorming formats](https://www.avanscoperta.it/en/eventstorming/)).

#### 8.5.4 A practical workshop flow for monolith decomposition

This is a robust learning/interview version of the workflow. Real facilitators adapt it to the domain.

##### Step 0: state the question and invite multiple perspectives

Do not ask only the monolith's developers. Include people who know policy, operations, customer exceptions, compliance, and external integrations. A useful framing is:

> “How does an order move from customer intent to a settled, fulfillable commitment today, including failure and support paths?”

For a legacy migration, keep two models conceptually separate:

- **as-is:** what the business and monolith do today;
- **to-be:** the business behavior and ownership you want.

Otherwise the group may accidentally document the desired clean story while missing legacy jobs and manual repair paths that must be migrated.

##### Step 1: storm the important domain events

Participants write facts independently and in parallel:

```text
CartCheckedOut
OrderSubmitted
CouponRejected
ItemsReserved
ReservationExpired
PaymentAuthorized
PaymentDeclined
OrderConfirmed
OrderCancelled
RefundRequested
RefundCompleted
```

“Take out all the events” should be interpreted as **surface all relevant business events and perspectives**, not log every technical action. Avoid notes such as `ControllerCalled`, `RowInserted`, or `JSONParsed` unless that technical fact itself has business significance.

The first pass can be intentionally messy. The official **Chaotic Exploration** pattern uses inconsistent perspectives and granularity to expose where the organization's official story differs from reality ([EventStorming: Chaotic Exploration](https://www.eventstorming.com/patterns/chaotic-exploration/)).

##### Step 2: enforce a narrative or timeline

Arrange events roughly from earlier to later and separate alternate/parallel paths:

```text
                                       +--> ItemsRejected --> OrderRejected
                                       |
OrderSubmitted --> ReserveItems -------+
                                       |
                                       +--> ItemsReserved --> AuthorizePayment
                                                                  |
                                      +---------------------------+------------------+
                                      |                                              |
                                      v                                              v
                              PaymentAuthorized                              PaymentDeclined
                                      |                                              |
                                      v                                              v
                               OrderConfirmed                               ReservationReleased
```

Ordering the narrative helps expose gaps:

- What starts the flow?
- What happens between these facts?
- Is the missing item a command, decision, event, or manual action?
- What are the failure, timeout, retry, cancellation, and expiry paths?
- Which steps can occur concurrently?
- What stable event ends each branch?

Do **not** force one total sequence when the business is concurrent or conversational. Some events can happen in different orders; model the valid states and exit conditions instead. The official EventStorming patterns explicitly warn that imposing a sequence on intrinsically non-sequential behavior can create a misleading tangle ([EventStorming: Conversational System](https://www.eventstorming.com/patterns/conversational-system/)).

##### Step 3: mark pivotal events and hot spots

A **pivotal event** changes the phase of the business story: the order becomes a commitment, fulfillment begins, or cancellation is no longer freely reversible.

Hot spots record questions without stopping exploration:

- Payment timed out: is its outcome failed or unknown?
- Can stock be released after payment authorization?
- Which team decides whether a risky order is accepted?
- Does “cancelled” mean requested, approved, or completed?
- Can customer support override an automated fraud decision?

These disagreements are valuable. They often reveal hidden policies, overloaded words, missing states, and possible context boundaries.

##### Step 4: add causes and decisions

For each important event, ask what caused it:

```text
[Actor / prior event]
        |
        v
     COMMAND ------> [AGGREGATE / decision owner] ------> DOMAIN EVENT
                         ^
                         |
                    READ MODEL / facts
```

Example:

```text
Customer
   |
   v
SubmitOrder ---> [Order] ---> OrderSubmitted
                                  |
                                  v
                         [reservation policy]
                                  |
                                  v
ReserveItems --> [Stock Reservation] --> ItemsReserved / ItemsRejected
```

This converts a list of facts into behavior. It also identifies who owns each business decision rather than merely who stores a row.

##### Step 5: read the story out loud and search for missing behavior

One person narrates the complete model while the rest challenge it. Use precise sentences:

> “When an order is submitted, the reservation policy asks Inventory to reserve the requested SKUs. Inventory either records `ItemsReserved` with an expiry or records `ItemsRejected`. It never emits both for the same reservation version.”

Speaking it aloud exposes vague arrows that look plausible on a diagram. Walk these variants too:

- normal success;
- business rejection;
- dependency unavailable;
- response lost after a successful write;
- duplicate command/event;
- event arriving late or out of order;
- cancellation racing with completion;
- scheduled expiry;
- manual support correction; and
- reconciliation after inconsistent state.

##### Step 6: look for bounded-context signals

Cluster behavior around a coherent purpose, model, and decision owner. Draw a proposed boundary where several signals reinforce one another:

| Boundary signal | Example |
|---|---|
| A word changes meaning | `Order` is an editable proposal in Sales but an accepted record in Fulfillment |
| Different business rules/policies | Inventory decides reservability; Payments decides authorization status |
| Distinct invariants/transactions | Available stock must not go below the allowed threshold |
| Clear data/decision owner | Only Payments interprets provider attempts and reconciliation |
| Different lifecycle | Catalog content evolves independently from an accepted order-line snapshot |
| Different security/compliance | Card/token handling requires tighter access and audit controls |
| Different scaling/failure needs | Search reads and payment authorization have different load and criticality |
| Different change vocabulary/experts | Merchandising and warehouse teams change for different reasons |

Possible result:

```text
+---------------- Ordering context ----------------+
| SubmitOrder -> OrderSubmitted                     |
| ConfirmOrder -> OrderConfirmed                    |
| RejectOrder  -> OrderRejected                     |
+-----------------------+---------------------------+
                        | ReserveItems command
                        v
+--------------- Inventory context ----------------+
| Stock Reservation -> ItemsReserved/ItemsRejected  |
+-----------------------+---------------------------+
                        | ItemsReserved event
                        v
+---------------- Payments context -----------------+
| Payment -> PaymentAuthorized/PaymentDeclined      |
+---------------------------------------------------+
```

The cross-boundary arrows suggest contracts for later design. They do not yet dictate whether the transport is HTTP, gRPC, a queue, or an in-process module call.

##### Step 7: create a context map and challenge each candidate

For every proposed context record:

- its purpose and ubiquitous language;
- commands, queries, and events at its boundary;
- data and decisions it exclusively owns;
- upstream/downstream relationships;
- facts copied as snapshots/projections;
- immediate invariants versus eventually consistent workflow state;
- expected team owner; and
- reliability, latency, security, and scaling requirements.

Then test the candidate with the **C-DATA** checklist from Section 6.7. Merge boundaries when they need constant synchronous conversation or one atomic invariant. Split a context only when a distinct reason to change/scale/secure/own outweighs another service's fixed operational cost.

##### Step 8: choose implementation boundaries conservatively

The output is a set of **boundary hypotheses**, not deployment instructions:

```text
EventStorming evidence
        |
        v
bounded-context candidates
        |
        +--> modules in one modular monolith (often the safe first step)
        |
        +--> coarse independently deployed services (when justified)
        |
        +--> keep together and revisit after measurement
```

One bounded context can start as one module. A large bounded context may later contain several internal components or deployables. Never create one service per event, command, aggregate, sticky-note cluster, or database table.

#### 8.5.5 How EventStorming helps prevent a distributed monolith

It helps by making the coupling visible **before** the architecture hardens:

- language/model ownership becomes explicit;
- commands identify the service allowed to make a decision;
- events expose cross-context facts instead of shared table access;
- aggregates reveal which rules need a local transaction;
- hot spots expose unknown or tightly coupled workflow behavior;
- context maps reveal dependency direction and contract needs; and
- alternate/failure-path walkthroughs reveal synchronous chains and hidden coordination.

But the workshop cannot guarantee good services. You can still build a distributed monolith if implementation introduces:

- shared writable tables;
- cyclic synchronous calls;
- release-time lockstep;
- a shared domain library that leaks every internal model;
- commands routed through many services for simple decisions;
- no clear source of truth; or
- one feature requiring several team approvals and repositories.

Validate the running system with change/deployment data, distributed traces, dependency depth, feature touch count, and incident blast radius. Boundary discovery and operational evidence form a feedback loop.

#### 8.5.6 Public production migration example

A 2016 Findmypast engineering write-up described using EventStorming to identify business subdomains, first isolating them behind internal APIs and functional tests **inside the monolith**, then separating database ownership, extracting a service, canarying it, and finally deleting redundant monolith code ([Findmypast: Extracting a Microservice from a Monolith](https://tech.findmypast.com/extracting-a-microservice-from-a-monolith/)).

Treat this as a **public historical case study**, not a universal recipe or a claim about Findmypast's current architecture. Its durable lesson is that discovering a boundary is only the beginning; code isolation, behavior characterization, data ownership, cutover, verification, and retirement are separate migration steps.

#### 8.5.7 Common EventStorming mistakes

| Mistake | Correction |
|---|---|
| Invite only engineers | Include domain experts and operational/support perspectives; the missing knowledge is often outside code |
| Start by naming services | Start from business facts, decisions, rules, and language; services are a later hypothesis |
| Record only the happy path | Add rejection, timeout, expiry, retry, duplicate, race, and manual-repair paths |
| Treat every technical log as a domain event | Prefer facts meaningful to the business |
| Treat every event as an integration message | Publish only deliberately designed contracts; many modelling events stay internal |
| Force every event into one timeline | Show parallelism, branches, loops, and valid termination states |
| Draw a boundary around each aggregate | An aggregate is a local consistency boundary; a bounded context/service may own several aggregates |
| Make each bounded context a service immediately | Begin as modules/coarse services when domain or operational readiness is uncertain |
| Preserve the monolith's tables as the domain model | Use current code/data as evidence, not as the definition of the desired business boundary |
| End when the wall looks organized | Capture purpose, glossary, ownership, context map, open hot spots, and validation experiments |

## 9. Decomposition pattern 2: by DDD subdomain and bounded context

### 9.1 Step-by-step method

#### Step 1: describe the business, not the technology

Write important user journeys, commands, policies, decisions, and events.

```text
Customer places order
  -> validate purchasable items
  -> freeze accepted prices
  -> reserve inventory
  -> authorize payment
  -> confirm or reject order
```

Do not begin with Kafka, Kubernetes, SQL, or endpoint lists.

#### Step 2: identify subdomains

Group related knowledge and rules:

- Catalog knows how items are described and published.
- Pricing knows price lists, promotions, and effective periods.
- Inventory knows stock, reservations, and warehouse allocation.
- Ordering knows order lifecycle and accepted commercial facts.
- Payments knows authorization, capture, refund, and provider reconciliation.

Classify them as core, supporting, or generic for this particular business.

#### Step 3: find language/model boundaries

Look for words that change meaning, different owners, separate policies, and different life cycles. These are clues for bounded contexts.

Example: `customer` can mean:

- login identity in Identity;
- buyer and saved preferences in Customer Profile;
- billing party in Payments; and
- recipient/address snapshot in Ordering.

#### Step 4: identify aggregates and invariants

For each context ask:

- What must be valid immediately?
- Which objects change atomically?
- Who is allowed to make this decision?
- Can another context receive the result later?

Keep an invariant inside one aggregate/service where practical. Across contexts, define an explicit business workflow and acceptable temporary states.

#### Step 5: draft service candidates

Map bounded contexts/capabilities to coarse candidates. Do not split every aggregate or noun immediately.

```text
+------------------+    +------------------+
| Catalog context  |    | Inventory context|
| Product aggregate|    | Stock/Reservation|
+------------------+    +------------------+

+------------------+    +------------------+
| Ordering context |    | Payments context |
| Order aggregate  |    | Payment aggregate|
+------------------+    +------------------+
```

#### Step 6: design contracts and ownership

For each candidate specify:

- commands/queries/events exposed;
- data exclusively owned;
- upstream/downstream contexts;
- synchronous versus asynchronous interactions;
- compatibility policy;
- identity/reference strategy; and
- which facts are copied as snapshots/read models.

#### Step 7: test real scenarios and failures

Walk through:

- happy path;
- timeout after a write;
- duplicate message;
- dependency down;
- out-of-order event;
- cancellation/refund;
- schema/API version skew;
- traffic spike;
- data repair/reconciliation; and
- team ownership/release.

A boundary that works only in the happy-path box diagram is not finished.

#### Step 8: evaluate non-functional differences

Microsoft emphasizes that boundaries must consider architecture characteristics as well as domain logic: performance, availability, security, fault tolerance, testability, and agility ([Azure domain analysis](https://learn.microsoft.com/en-us/azure/architecture/microservices/model/domain-analysis)).

#### Step 9: start coarse and evolve

Implement as modular-monolith modules or coarse services if knowledge is incomplete. Measure change coupling, call patterns, incidents, scaling, and ownership. Split or merge when evidence justifies it.

### 9.2 DDD e-commerce example

```text
                             queries
 Customer app ------------------------------------+
      |                                            |
      v                                            v
+-------------+        published views       +-----------+
| API/BFF     | ---------------------------->| Catalog   |
+------+------+                               +-----+-----+
       |                                            |
       | place order                                | ProductPublished
       v                                            v
+-------------+   reserve command/event       +-----------+
| Ordering    | ----------------------------->| Inventory |
| owns Order  |                               | owns Stock|
+------+------+                               +-----+-----+
       | authorize                                  |
       v                                            | result event
+-------------+                                     |
| Payments    | <-----------------------------------+
| owns Payment|
+------+------+ 
       |
       +-- PaymentAuthorized/Failed --> Ordering

Ordering stores accepted price/name snapshots.
It does not join Catalog tables during order history reads.
```

Important nuance: price ownership can be a separate Pricing context, while Ordering owns the **accepted price snapshot** for an order. Both facts are valid because they answer different questions:

- Pricing: "What price would apply now?"
- Ordering: "What price did this buyer accept then?"

### 9.3 Strengths

- boundaries reflect business language and rules;
- conflicting models do not contaminate one universal schema;
- invariants and decision ownership become explicit;
- architecture stays relatively stable when technology changes;
- events represent business facts rather than database internals; and
- teams can own coherent contexts.

### 9.4 Risks

- requires sustained access to domain experts;
- initial domain knowledge may be wrong;
- DDD terminology can create ceremony without better decisions;
- forcing one context into one tiny service can create chattiness;
- context maps and contracts require maintenance; and
- distributed transactions/reporting remain real engineering problems.

## 10. Business capability versus DDD decomposition

They are often complementary lenses, not mutually exclusive choices.

| Question | Business capability lens | DDD lens |
|---|---|---|
| Starts from | What the organization must be able to do | The domain's concepts, language, rules, and models |
| Typical output | Capability map | Subdomains, bounded contexts, context map, aggregates/events |
| Best at | High-level stable business alignment | Finding semantic, consistency, and model boundaries |
| Typical names | Order management, fulfillment, customer support | Ordering context, Inventory context, Payment context |
| Key participants | Business leaders, product owners, architects, SMEs | Domain experts, product, engineers, architects |
| Main risk | Capabilities too broad or copied from org chart | Too much modeling ceremony or premature one-context-one-service mapping |
| Relationship | Gives top-down candidates | Refines candidates and tests their internal/external boundaries |

Practical combined method:

```text
Business goals and journeys
          |
          v
Capability map (what the business does)
          |
          v
Subdomains (where distinct business knowledge lives)
          |
          v
Bounded contexts (where one model/language applies)
          |
          v
Aggregates/invariants + interactions
          |
          v
Candidate modules/services
          |
          v
Operational checks: team, scale, failure, security, deployment cost
```

AWS explicitly notes that decomposition patterns can be combined, for example using business capability first and subdomain analysis to refine it ([AWS decomposition patterns](https://docs.aws.amazon.com/prescriptive-guidance/latest/modernization-decomposing-monoliths/decomposing-patterns.html)).

### 10.1 Do not decompose primarily by these

#### Technical layer

```text
UI Service -> Logic Service -> Data Service
```

Every feature crosses every service. Teams cannot deliver business value independently.

#### Entity or table

```text
Customer Service, Address Service, OrderLine Service, Status Service
```

This usually creates remote getters/setters and breaks aggregate invariants.

#### CRUD verb

```text
CreateOrder Service, UpdateOrder Service, ReadOrder Service
```

Command/query scaling can later justify CQRS-style separation, but splitting by verb without a measured need fragments one lifecycle.

#### Current team chart alone

If boundaries merely mirror temporary reporting lines, reorganizations and cross-capability workflows will constantly reshape the architecture.

#### Technology

```text
Java Service, Python Service, SQL Service
```

Technology is an implementation detail unless it corresponds to a genuinely different workload/capability.

## 11. What changes technically after the split?

The service boxes are the easy part. The difficult part is moving **calls, data, transactions, failure handling, testing, and ownership** across boundaries.

### 11.1 Communication choices

#### Synchronous HTTP or gRPC

Use when the caller needs a result to continue now:

- query item availability for a checkout decision;
- fetch an authorization decision;
- request a price quote; or
- issue a command whose immediate acceptance/rejection matters.

Benefits:

- simple request/response mental model;
- immediate result/error;
- mature tooling; and
- easy client semantics for queries.

Costs:

- caller and callee must both be available at that time;
- latency adds across the call path;
- retries can duplicate commands;
- slow dependencies consume caller resources; and
- long chains create cascading failure.

Use deadlines/timeouts, bounded retries with jitter, idempotency for retried commands, circuit breakers where useful, and a clear fallback. Do not blindly retry every error.

#### Asynchronous queue or event stream

Use when work can occur later or multiple consumers react independently:

- transcode a video after upload;
- update a search index after a catalog change;
- send notifications after an order event;
- generate analytics; or
- continue a multi-service business workflow.

Benefits:

- producer and consumer do not need to be up simultaneously;
- queues absorb bursts and provide backpressure;
- consumers can evolve/scale independently; and
- noncritical work leaves the user request path.

Costs:

- result is not immediate;
- duplicates are normal in many delivery models;
- ordering is limited, often per key/partition;
- queues can build a large delay/backlog;
- debugging spans time and systems; and
- event schema/privacy/retention must be governed.

#### Decision table

| Need | Usually favor | Caveat |
|---|---|---|
| User needs answer before continuing | Synchronous request | Bound the dependency with a deadline/fallback |
| Work may finish later | Queue/command | Expose pending status and retry safely |
| Many consumers react to a fact | Event publication | Producer must not know consumer internals |
| High-volume burst smoothing | Queue/stream | Monitor lag; queue is not infinite capacity |
| Strong decision across one invariant | Keep local to one owner | Avoid inventing a remote transaction |
| Read data from several domains | API composition or read model | Avoid direct database joins across ownership |

### 11.2 Avoid long synchronous dependency chains

Suppose each service is available 99.9% of the time and a request requires five independent services. A simplistic multiplication gives:

```text
0.999^5 ~= 0.995
```

or about 99.5% for the whole path under those simplifying assumptions. **This is an illustration, not a capacity/SLA calculation**: real failures are correlated, dependencies have different SLOs, retries change load, and some paths degrade rather than fail. The lesson is that mandatory dependencies compose.

Prefer:

- fewer synchronous hops;
- parallel calls when results are independent;
- cached/local materialized views for read-only facts;
- asynchronous noncritical work;
- coarse-grained APIs; and
- explicit graceful degradation.

### 11.3 API Gateway and Backend for Frontend (BFF)

An **API Gateway** is an entry point that can route requests, authenticate callers, enforce rate limits, terminate TLS, and hide internal service locations.

A **Backend for Frontend** is a gateway/API tailored to one client type, such as mobile versus web.

```text
Mobile ---> Mobile BFF --+
                         +--> domain services
Web ------> Web BFF -----+
```

Do not move all business logic into the gateway. Otherwise it becomes a new monolith and every feature release depends on it.

### 11.4 Service discovery

Service instances are added, removed, or replaced. **Service discovery** maps a logical service name to healthy instances. A platform load balancer, DNS, registry, or service mesh can provide this. It is an operational prerequisite, not a domain boundary method.

### 11.5 Contract evolution

Producer and consumer versions coexist. Safer evolution is usually additive:

1. add a field/endpoint/event version without removing the old behavior;
2. deploy producers/consumers that tolerate both;
3. migrate usage and observe it;
4. remove the old behavior only after consumers are gone.

Avoid assuming "we deploy independently" while requiring synchronized breaking changes. Consumer-driven contract tests can catch some incompatibilities, but they do not replace integration or semantic testing.

## 12. Data ownership, queries, and consistency

### 12.1 Database per service does not mean database server per container

Logical ownership is more important than physical topology.

Possible arrangements:

```text
Strong isolation:
Order Service -> Order database cluster
Payment Service -> Payment database cluster

Cost-conscious early stage:
Order Service -> orders schema ---+
Payment Service -> payments schema +--> same physical DB cluster
                 separate users; no cross-writes
```

The second option shares failure/capacity infrastructure but can still enforce model ownership. It is an evolutionary compromise, not the same isolation as separate clusters.

### 12.2 Why a shared database is tempting

It makes joins and cross-module transactions easy. But if Service A directly reads/writes B's tables:

- B cannot change its schema independently;
- business rules can be bypassed;
- ownership becomes ambiguous;
- releases coordinate around database migrations;
- one query can exhaust shared capacity; and
- technology/storage evolution is constrained.

During migration, a shared database may be a temporary step. State the risk and a path toward exclusive ownership rather than claiming full independence.

### 12.3 How to perform cross-service reads

#### API composition

A gateway/composer calls owning services and combines results.

```text
Order Details API
   +--> Ordering: order state
   +--> Shipment: tracking state
   +--> Payments: payment summary
```

Use for low/medium fan-out and freshness requirements. Handle partial results and latency.

#### Materialized read model / CQRS projection

Consumers build a query-optimized copy from events.

```text
Ordering ----OrderChanged-----+
Payments ----PaymentChanged---+--> OrderHistory Projector -> OrderHistory read DB
Shipping ----ShipmentChanged--+
```

The read is fast and does not fan out, but the view is eventually consistent and needs rebuild/repair/versioning logic.

**CQRS prerequisite:** Command Query Responsibility Segregation separates the write model from the read model because their needs differ. It does not inherently require microservices or event sourcing.

#### Analytics/data platform

For broad reporting, replicate governed domain data/events into a warehouse/lake. Do not make operational services perform uncontrolled cross-database analytics joins on the request path.

### 12.4 Data duplication is sometimes correct

In a monolith, duplication is often treated as normalization failure. Across bounded contexts, copying a stable fact or snapshot can reduce coupling.

Examples:

- Ordering stores the product title and accepted price at purchase time.
- Search stores an index projection of catalog attributes.
- Shipping stores the recipient address snapshot used for that shipment.

Rules for responsible duplication:

- one context remains the source/authority for each type of decision;
- copies have explicit freshness semantics;
- events or jobs update them;
- consumers tolerate duplicates/out-of-order updates;
- reconciliation/rebuild exists; and
- sensitive data is not copied casually.

### 12.5 Consistency is a business decision

Different facts need different consistency:

| Fact/operation | Likely requirement | Reason |
|---|---|---|
| Prevent overselling a unique item | Strong decision at Inventory owner | Two successful reservations cannot exceed supply |
| Search reflects new title | Eventual consistency often acceptable | A short indexing delay is tolerable |
| Accepted order price | Immutable snapshot in Ordering | History must not change with today's price |
| Recommendation update | Eventual/batch | Freshness is useful but not correctness critical |
| Payment idempotency | Strong uniqueness/deduplication | Retry must not double-capture |
| Notification delivery | At-least-once attempt plus dedup/preferences | Message can lag; duplicate user message is undesirable |

Do not apply "eventual consistency" to every microservice interaction. Keep critical decisions within one owner where possible, and state where temporary inconsistency is acceptable.

## 13. Transactions across microservices: saga, outbox, and idempotency

This section is a prerequisite for understanding the cost of splitting Ordering, Inventory, and Payments.

### 13.1 Why one ACID transaction no longer fits

In a monolith:

```text
BEGIN
  insert order
  decrement/reserve inventory
  insert payment record
COMMIT
```

With database-per-service, no service should directly update all three stores. A distributed two-phase commit exists in some technology stacks, but it introduces coordinator/availability/coupling constraints and is not the default answer for long-running business workflows involving external payment providers.

### 13.2 Saga in plain language

A **saga** is a sequence of local transactions. Each service commits its own change and triggers the next step. If a later step fails, the workflow performs a business-specific **compensating action** for already completed steps.

```text
1. Ordering: create order PENDING
2. Inventory: reserve items
3. Payments: authorize payment
4. Ordering: mark CONFIRMED

If payment fails:
5. Inventory: release reservation
6. Ordering: mark REJECTED
```

A compensation is not necessarily a database rollback. A captured payment may require a refund, which is a new auditable business transaction and might itself fail temporarily.

### 13.3 Orchestration versus choreography

#### Orchestration

A saga orchestrator tells participants what step to perform and records workflow state.

```text
              +-------------------+
              | Order orchestrator|
              +--+-------+------+-+
                 |       |      |
                 v       v      v
             Inventory Payment Ordering state
```

Benefits:

- workflow is visible in one place;
- timeouts, retries, and compensation order are easier to reason about;
- good for complex or regulated flows.

Costs:

- orchestrator can accumulate too much domain logic;
- participants are coupled to commands/workflow; and
- orchestrator reliability/state must be designed.

#### Choreography

Services react to events without a central coordinator.

```text
OrderPlaced -> Inventory reserves -> InventoryReserved
                                     |
                                     v
                              Payment authorizes
                                     |
                               PaymentAuthorized
                                     |
                                     v
                              Ordering confirms
```

Benefits:

- loose publisher awareness;
- easy to add independent reactions;
- natural for simple event flows.

Costs:

- overall workflow becomes hard to see;
- event cycles and accidental coupling can grow;
- timeout/compensation ownership is unclear; and
- debugging needs excellent tracing.

Use choreography for simple reactions and orchestration when one business process needs explicit state/control. Real systems often combine them.

### 13.4 The dual-write problem

Suppose Ordering does:

```text
1. commit order to database
2. publish OrderPlaced event
```

It can crash after step 1 and before step 2. Reversing the order can publish an event for a database write that later fails. This is the **dual-write problem**.

### 13.5 Transactional outbox

Write the domain change and an outbox record in the same local transaction. A relay later publishes the outbox record.

```text
Ordering local transaction
  +-- insert Order(PENDING)
  +-- insert Outbox(OrderPlaced, eventId=E1)
  +-- COMMIT

Outbox relay -> broker -> consumers
```

Publishing may occur more than once if the relay crashes around acknowledgement, so consumers still need idempotency/deduplication. Change-data capture is another way to relay committed outbox/log changes.

### 13.6 Idempotency

An operation is **idempotent** when applying the same logical request more than once has the same intended effect as applying it once.

For commands such as payment capture:

```text
POST /captures
Idempotency-Key: checkout-123-capture
```

The owner stores the key and result atomically with the business effect. A retry returns the stored result instead of charging again.

"Exactly once" is usually an end-to-end business property created with identity, durable state, deduplication, and correct transactional boundaries; it is not guaranteed merely because TCP, Kafka, or a queue was used.

### 13.7 Saga design questions interviewers ask

- What are the intermediate states visible to the user?
- Which step is retriable, idempotent, or irreversible?
- What if compensation fails?
- How long can a reservation remain pending?
- Who detects timeouts and resumes stuck workflows?
- Are events ordered per order ID?
- How are duplicates and poison messages handled?
- How can support/operators inspect and repair a saga?
- Does the workflow need orchestration or simple event reactions?
- Could changing the service boundary keep the critical invariant local instead?

The last question is important: a saga is not a license to split one invariant into arbitrary tiny services.

## 14. Reliability, observability, and testing

### 14.1 Partial failure is normal

For every remote dependency ask:

```text
Can it be slow?
Can it be unavailable?
Can the request succeed but the response be lost?
Can I retry safely?
Can I serve a degraded result?
How do I know it recovered?
```

### 14.2 Core resilience mechanisms

| Mechanism | Purpose | Common mistake |
|---|---|---|
| Timeout/deadline | Stop waiting after the result is no longer useful | Default/infinite timeouts exhaust resources |
| Retry | Recover from transient failure | Retrying non-idempotent work or retrying overload creates a storm |
| Exponential backoff + jitter | Spread retry load | Every client retries on identical schedule |
| Circuit breaker | Temporarily stop calls likely to fail | Treating it as a substitute for capacity/fallback design |
| Bulkhead/resource isolation | Prevent one dependency from consuming all pools | One shared thread/connection pool spreads failure |
| Rate limit/load shed | Protect capacity under overload | Queue everything until latency collapses |
| Fallback/degradation | Keep critical function useful | Returning silently incorrect data for a critical decision |
| Health checks | Route around known unhealthy instances | A shallow "process alive" check misses dependency failure |

Retries belong at a carefully chosen layer. If gateway, Service A, and Service B each retry three times, one user request can multiply into a large load spike.

### 14.3 Observability

Logs alone are insufficient. A mature service fleet needs:

- **metrics:** request rate, errors, latency percentiles, saturation, queue lag, business outcomes;
- **structured logs:** searchable fields and stable error/event identifiers;
- **distributed traces:** one trace/span context across gateway, services, broker, and stores;
- **correlation/business IDs:** order ID, payment ID, saga ID, idempotency key;
- **dependency maps:** caller/callee ownership and SLOs;
- **deployment markers:** connect behavior changes to releases; and
- **alerts/runbooks:** actionable symptom-based response.

Observe user outcomes, not only servers. "All pods are green" does not prove customers can place orders.

### 14.4 Testing strategy

No single layer is enough:

1. **Unit/domain tests:** fast verification of aggregate rules and policies.
2. **Component/service tests:** run one service with real owned storage and controlled dependencies.
3. **Contract tests:** verify compatible request/event shapes and expectations.
4. **Integration tests:** verify real broker, database, identity, and selected service interactions.
5. **End-to-end tests:** cover a small set of critical user journeys; avoid making the entire test suite depend on the full fleet.
6. **Resilience tests:** inject latency, errors, duplicates, unavailable dependencies, and backlog.
7. **Production verification:** canaries, synthetic checks, SLO monitoring, and rollback/roll-forward.

Too many brittle end-to-end tests are a symptom of uncertain contracts and boundaries. Too many mocks can create false confidence because the mock behaves unlike the real dependency.

### 14.5 Security changes

Microservices turn in-process trust into many network trust decisions:

- authenticate workloads, not only end users;
- authorize each service operation with least privilege;
- encrypt data in transit and at rest where required;
- rotate secrets/certificates;
- validate untrusted input at boundaries;
- propagate user identity/claims deliberately, not blindly;
- audit sensitive actions;
- prevent one compromised service from reading every database; and
- patch many runtimes/images/dependencies.

An API Gateway does not secure internal service-to-service traffic by itself.

## 15. Migrating a monolith to microservices

### 15.1 Do not start with a big-bang rewrite

A big-bang rewrite runs old and new product development against a distant cutover, delays feedback, and combines domain, data, code, and operational risk. Prefer incremental extraction unless there is a strong reason otherwise.

### 15.2 Strangler Fig pattern

The **Strangler Fig** pattern routes selected behavior to new services while the monolith continues serving the rest. Over time, extracted functionality replaces the old implementation.

```text
Phase 1
Client -> Router/Facade -> Monolith

Phase 2
                        +--> New Catalog Service
Client -> Router/Facade-+
                        +--> Monolith (orders, payment, users...)

Phase 3
                        +--> Catalog Service
Client -> Router/Facade-+--> Order Service
                        +--> remaining monolith
```

AWS describes the pattern as an incremental migration that lowers transformation risk compared with a big-bang replacement, while warning about unclear domains, proxy failure, data synchronization, and inter-service complexity ([AWS Strangler Fig guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/strangler-fig.html)).

### 15.3 Choose the first extraction carefully

Good candidates often have:

- clear business/data ownership;
- few inbound/outbound dependencies;
- a measured pain: scale, release pace, reliability, or technology;
- lower correctness risk; and
- an easy routing seam.

Examples might be notifications, media processing, document generation, or a well-bounded catalog read path. Do not automatically extract the easiest code if it creates no learning or value.

### 15.4 Migration sequence

Use **EDGE**:

1. **E - Establish the boundary:** map behavior/data dependencies, define contract and owner.
2. **D - Divert through a seam:** gateway, facade, internal interface, event, or branch-by-abstraction.
3. **G - Gradually move logic/data/traffic:** shadow, compare, canary, reconcile, and make rollback possible.
4. **E - Eliminate the old path:** stop dual ownership, remove legacy code/schema/access, and update runbooks.

Detailed steps:

1. Measure the current pain and define success metrics.
2. Modularize/tangle-map the monolith before physical extraction where needed.
3. Establish one authoritative owner for writes.
4. Create a stable API/event contract and anti-corruption adapter.
5. Backfill/migrate data with verification.
6. Route a small percentage or read-only traffic first when safe.
7. Observe correctness, latency, failure, and support impact.
8. Move write ownership with an explicit cutover strategy.
9. Remove old writes/reads; do not keep permanent dual ownership.
10. Repeat only while the benefit remains greater than the premium.

### 15.5 Anti-Corruption Layer (ACL)

An ACL translates between a legacy model/API and the new context so the new domain model is not contaminated by old meanings.

```text
New Ordering model -> ACL/adapter -> Legacy ERP schema/API
```

It is especially useful when an external or legacy system uses different terminology and data shapes. It adds code and latency, so its ownership and retirement plan should be clear.

### 15.6 Data migration options

| Option | When useful | Main risk |
|---|---|---|
| Read old data through monolith API | Early extraction | Runtime dependency remains |
| New service owns new writes; backfill history | Clean cutover possible | Ordering and reconciliation during transition |
| Change-data capture to new store | Large/continuous migration | Schema semantics, lag, duplicates |
| Temporary dual write | Rare, tightly controlled | Inconsistent outcomes; needs repair and source-of-truth rule |
| Shadow read/compare | Validate new implementation | Privacy/cost and comparison false positives |

Avoid application-level dual writes without a recovery design. Say which store is authoritative at every phase.

### 15.7 Success metrics

Do not call a migration successful because there are more repositories. Measure:

- lead time and deployment frequency;
- change failure and rollback rate;
- incident/blast-radius changes;
- p95/p99 latency and error rate;
- build/test duration;
- cross-team coordination per feature;
- number of synchronous dependencies/touch points;
- data reconciliation errors;
- infrastructure/on-call cost; and
- whether the owning team can actually deploy/operate independently.

## 16. Common mistakes and corrections

### 16.1 "Microservices are always more scalable"

**Correction:** They permit independent scaling. A monolith can scale horizontally, and the database may remain the bottleneck in either design.

### 16.2 "Microservices isolate all failures"

**Correction:** Process isolation helps only when calls, capacity, and fallback behavior prevent propagation. Mandatory synchronous chains can amplify failure.

### 16.3 "One noun/table per service"

**Correction:** Model behavior, invariants, and reasons to change. `Order` and `OrderLine` are usually one aggregate, not remote CRUD services.

### 16.4 "Use DDD by creating Domain Service classes"

**Correction:** Strategic DDD first discovers subdomains, language, and bounded contexts. A DDD *domain service* is an in-model object for behavior that does not naturally belong to an entity/value object; it is not synonymous with a network microservice.

### 16.5 "One bounded context must equal one microservice forever"

**Correction:** It is a candidate mapping. A context may be a module first or hide several internal deployables behind one contract. Preserve semantic ownership and low coupling.

### 16.6 "Every service chooses any technology"

**Correction:** Purposeful polyglot choice can help, but a small platform team supporting 12 stacks loses leverage. Offer a paved road/default stack with exceptions justified by workload.

### 16.7 "Events remove coupling"

**Correction:** They reduce temporal/directional coupling but create schema and semantic coupling. Consumers depend on event meaning, order, delivery, retention, and compatibility.

### 16.8 "A queue guarantees exactly once"

**Correction:** Plan for duplicate delivery and uncertain outcomes. Use stable event/operation IDs, idempotent consumers, unique constraints, and reconciliation.

### 16.9 "Shared libraries are always reuse"

**Correction:** A library for stable cross-cutting code can help. A shared domain library used by every service can force synchronized upgrades and leak models. Prefer contract schemas and intentional duplication when independent evolution matters.

### 16.10 "Start with the smallest possible services"

**Correction:** Start with boundaries you understand. Coarse services/modules are easier to split than a fleet of incorrect chatty boundaries is to untangle.

### 16.11 "A service owns data because it has its own ORM code"

**Correction:** Ownership requires access control and behavior enforcement. Other services must not bypass it through tables, replicas, or shared ORM models.

### 16.12 "Microservices solve a messy organization"

**Correction:** Architecture exposes and may amplify unclear ownership. Team boundaries, product ownership, funding, on-call, and decision rights must also change.

### 16.13 "The API gateway coordinates every workflow"

**Correction:** Keep routing and cross-cutting edge concerns at the gateway. Put domain workflows in the owning domain/application service or an explicit saga orchestrator.

### 16.14 "Never duplicate data"

**Correction:** Private projections and immutable snapshots are often necessary for autonomy. Define source of truth, freshness, repair, and privacy.

## 17. Real-product examples

### Important evidence note

Except for the explicitly cited Uber production snapshot, the service maps below are **illustrative interview designs based on the public behavior of the named products**. They show how to reason about boundaries; they do not claim to reveal each company's current private implementation. Large companies continuously evolve their architectures, and names such as "WhatsApp Service" or "YouTube Service" are too broad to be useful boundaries.

### 17.1 WhatsApp-style messaging system (illustrative design)

Candidate business contexts:

| Context/service candidate | Owns | Different reason to split |
|---|---|---|
| Identity/Devices | accounts, devices, keys/registration metadata | security lifecycle and device management |
| Messaging | message envelope, recipient routing, delivery state | very high-volume low-latency core path |
| Conversation/Groups | membership and admin rules | strongly consistent group-policy decisions |
| Media | upload metadata, object references, processing | large blobs and bandwidth differ from small messages |
| Presence | last-seen/online ephemeral state | high-churn data; loss/staleness semantics differ |
| Push Notification | APNs/FCM delivery attempts | external-provider failure must not block message acceptance |
| Abuse/Safety | reports, rate/abuse decisions | sensitive policy and review workflow |

```text
Sender
  |
  | 1. upload bytes (for media)
  +--------------------> Media Service -> object storage
  |                           |
  |                           +--> media reference
  |
  | 2. send encrypted envelope + optional media reference
  v
Messaging Service -> durable mailbox/routing -> recipient connections
       |
       +--> delivery event -> Push Notification (if recipient offline)
       +--> delivery receipt state

Presence failure should not prevent durable message acceptance.
Push-provider failure should be retried independently.
```

Boundary lesson:

- Keep the critical message acceptance/routing path short.
- Treat presence as advisory; a stale presence result should not lose a message.
- Keep media bytes outside message databases; the message can carry an object reference.
- Group membership may need an authoritative version so a sender cannot use an arbitrarily stale member list for security-sensitive delivery.

Trade-off follow-up: splitting Groups from Messaging adds a membership lookup/version question. Caching membership improves latency but creates revocation staleness. The answer must state what correctness/security the product requires.

### 17.2 YouTube-style video platform (illustrative design)

Candidate contexts:

- Upload/session management;
- Media processing/transcoding;
- Video metadata and visibility;
- Playback/manifest authorization;
- Search indexing;
- Recommendations;
- Comments/engagement;
- Creator analytics; and
- CDN/origin delivery.

```text
Creator -> Upload Service -> object storage (original)
                              |
                              +-- VideoUploaded event --> Transcoding workers
                                                         | 240p/720p/1080p...
                                                         v
                                                   encoded segments
                                                         |
                                      TranscodingCompleted event
                                           |             |             |
                                           v             v             v
                                      Metadata       Search index   Analytics
                                      publishable

Viewer -> Playback API -> check visibility/entitlement -> manifest
Viewer -------------------------------------------------> CDN segments
```

Boundary lesson:

- Upload acknowledgement should not wait for every rendition to finish.
- Transcoding is asynchronous, compute-heavy, retriable, and scales by queue depth.
- Search and recommendations consume projections; brief staleness is often acceptable.
- Playback control returns authorization/metadata/manifests; large video bytes should be delivered by CDN/origin infrastructure, not streamed through ordinary application microservices.
- A video can expose states such as `UPLOADING`, `PROCESSING`, `READY`, or `FAILED`, making eventual progress visible instead of pretending one global transaction exists.

### 17.3 Uber-style ride system

#### Public production snapshot

Uber wrote in 2020 that it had moved from primarily two monolithic services (circa 2012-2013) into a large service architecture, then faced complexity from the resulting service network. Its Domain-Oriented Microservice Architecture (DOMA) grouped related services into domains, used domain gateways, and constrained dependency layers. The article reported roughly 2,200 critical microservices grouped into about 70 domains at that time. **Verified as a 2020 public snapshot; not a current 2026 count** ([Uber Engineering, *Introducing Domain-Oriented Microservice Architecture*](https://www.uber.com/blog/microservice-architecture/)).

That example matters because it shows both halves of the trade-off:

- services helped team autonomy, ownership, deployment, and fault separation during growth;
- too many service touch points and deep dependencies later harmed developer velocity and debugging;
- the remedy was stronger domain-level boundaries and gateways, not endlessly smaller services.

#### Illustrative ride-request decomposition

```text
Rider app -> Edge/BFF -> Trip/Ride domain gateway
                            |
                  +---------+---------+
                  |                   |
                  v                   v
             Trip Service       Matching domain
                  |                   |
             Trip state          candidate drivers
                                      |
                                      v
                                 Dispatch offer

Location stream -----------------> Location domain
Pricing quote --------------------> Pricing domain
Payment after trip ---------------> Payment domain
Notifications <-------------------- trip/matching events
```

Boundary lesson:

- Matching, live location, pricing, trip lifecycle, and payments have different models and workload profiles.
- The top-level ride flow still needs clear orchestration/state; service boundaries do not make the business workflow disappear.
- A gateway can hide many internal services so product callers depend on one stable domain interface.
- Location telemetry is high-volume and freshness-oriented; payment state is correctness/audit-oriented. Their stores and failure policies should differ.

### 17.4 Netflix-style streaming product (illustrative design)

Candidate contexts:

- Account/household/profile;
- Subscription/billing;
- Catalog metadata and regional availability;
- Entitlement/playback policy;
- Personalization/recommendations;
- Playback session/manifest;
- Content encoding/origin; and
- CDN delivery/telemetry.

```text
TV/mobile client -> Edge/BFF -> Playback Session
                                   |-- Account/Profile
                                   |-- Entitlement/Policy
                                   |-- Catalog availability
                                   +-- create manifest/session
                                                |
                                                v
Client --------------------------------------> CDN video segments

Playback events -> telemetry stream -> QoE analytics/recommendations
```

Boundary lesson:

- Do not place recommendations on the hard playback path if a generic/trending fallback works.
- Catalog availability and playback entitlement may look similar but answer different questions: "Is this title offered in this region?" versus "May this profile play it now?"
- Large media delivery belongs to a CDN plane; application services control the session but should not proxy every byte.
- A BFF can tailor payloads and call patterns for TVs, phones, and browsers, but domain rules remain in domain owners.

Trade-off follow-up: if playback synchronously needs four services, define cached policy/catalog views, deadlines, and which failures deny playback versus degrade noncritical UI.

### 17.5 Twitter/X-style social feed (illustrative design)

Candidate contexts:

- Identity/account;
- Social graph/follow relationships;
- Post/Tweet write and storage;
- Timeline generation/read;
- Fan-out workers;
- Search indexing;
- Media;
- Notifications; and
- Trust/safety/moderation.

```text
Author -> Post Service -> Post store
              |
              +-- PostCreated event --+--> Timeline fan-out/read model
                                      +--> Search index
                                      +--> Notification rules
                                      +--> Analytics

Reader -> Timeline Service -> timeline IDs/read model -> hydrate posts/media
```

Boundary lesson:

- Post acceptance should not synchronously wait for search indexing, analytics, or every follower timeline update.
- Timeline can be a derived read model; the Post Service remains owner of the authoritative post.
- Fan-out-on-write gives fast reads but expensive writes for high-follower accounts. Fan-out-on-read reduces write amplification but increases read work; a hybrid is common as an interview design.
- Delete/moderation events must propagate to derived timelines, caches, and search. Eventual projection does not remove the need for urgent safety invalidation paths and repair.

### 17.6 Amazon-style marketplace/commerce system (illustrative design)

Candidate contexts:

- Catalog;
- Offers/Pricing;
- Search/Discovery;
- Cart;
- Ordering;
- Inventory;
- Payments;
- Fulfillment/Shipping;
- Returns/Refunds;
- Seller management; and
- Notifications.

```text
Browse path:
Customer -> Storefront/BFF -> Search projection / Catalog / Offer views

Checkout command:
Customer -> Ordering -> create PENDING order with accepted snapshots
                        |
                        v
                   Checkout saga
                     | reserve inventory
                     | authorize payment
                     | create fulfillment request
                     v
              CONFIRMED / REJECTED / REVIEW

OrderConfirmed event --> Fulfillment
                     +-> Notification
                     +-> Analytics
```

Boundary lesson:

- Catalog content, current offer price, inventory availability, and accepted order facts have different owners.
- Cart is often a mutable intent list; an order is an auditable business commitment. Do not make them one model merely because checkout converts one into the other.
- Checkout needs idempotency, an explicit saga state machine, reservation expiry, compensation, and support reconciliation.
- Search/recommendation can be stale projections; inventory reservation cannot rely only on a stale search result.

### 17.7 Cross-example summary

| Product path | Keep on critical synchronous path | Move/degrade asynchronously when possible |
|---|---|---|
| WhatsApp-style send | durable message acceptance and authorization/routing decision | push notification, analytics, nonessential presence UI |
| YouTube-style upload | upload session and durable object acceptance | transcoding, indexing, recommendations, analytics |
| Uber-style ride request | trip state plus required matching/pricing policy | notifications, analytics, many projections |
| Netflix-style playback | entitlement and playback-session decision | recommendations, telemetry processing |
| Twitter/X-style post | authoritative post write and safety checks required at write | timeline fan-out, search, analytics, notifications |
| Amazon-style checkout | order command, inventory/payment business decisions | email, analytics, noncritical recommendation updates |

This is a boundary heuristic: preserve the smallest correctness-critical decision path, then decouple derived/noncritical reactions.

## 18. Worked design exercise: choosing boundaries for checkout

### 18.1 Requirements

Functional:

- view products;
- add to cart;
- place an order;
- prevent overselling;
- authorize payment;
- view order status; and
- receive confirmation.

Non-functional:

- browsing is much heavier than checkout;
- checkout must not double-charge;
- email may arrive later;
- search may lag catalog briefly;
- order history must preserve accepted name/price;
- one small team is launching version 1.

### 18.2 Initial decision: modular monolith

For one small team and an uncertain domain, start with one deployment but enforce modules:

```text
+------------------------------------------------+
|                  Commerce app                  |
|                                                |
| Catalog | Cart | Ordering | Inventory | Payment|
|          private module APIs/data access       |
+-----------------------+------------------------+
                        |
                  relational DB
               schemas/ownership conventions

Async worker: notifications
Object/search infrastructure as needed
```

Why:

- transaction and local refactoring simplicity;
- low operational overhead;
- faster domain learning;
- boundaries are represented in code, so extraction remains possible.

### 18.3 First likely extractions

Suppose measurements later show:

- catalog/search receives 100x checkout read traffic;
- notification provider failures slow request threads;
- a separate payments/compliance team must isolate credentials/releases.

Possible evolution:

1. Move notifications behind an event/queue.
2. Extract search projection/read service if independent scaling is valuable.
3. Extract Payments behind an idempotent contract and strict security boundary.
4. Keep Ordering and its order-line invariant together.
5. Extract Inventory only when its warehouse/reservation model and ownership are sufficiently clear; introduce a saga for checkout.

### 18.4 Why not start with twelve services?

- business boundaries and product fit are not yet stable;
- cross-service workflow/observability would delay learning;
- one team gains little release autonomy from splitting itself;
- distributed data errors are more dangerous than the current scaling problem; and
- independent scaling can first be addressed with worker pools, caches, and module-level resource separation.

### 18.5 Why not stay monolithic forever?

If measured team contention, isolated scaling, compliance, or failure requirements dominate, the monolith's coupling cost can exceed the distributed premium. Architecture should evolve from observed forces rather than loyalty to either style.

## 20. Answer spines: cues to reconstruct answers live

### 20.1 "What is a monolith and a microservice architecture?"

Use **D-BIND**:

1. **D - Deployment boundary:** monolith releases capabilities together; services are separate deployables.
2. **B - Business boundary:** good services are organized around cohesive capabilities/contexts, not technical layers.
3. **I - Independence:** aim for independent change, scale, data ownership, and team operation.
4. **N - Network:** local calls become remote calls/messages with latency and partial failure.
5. **D - Data/distributed cost:** cross-service transactions, queries, tracing, contracts, and operations become harder.

Compact answer spine:

> "A monolith packages the application's capabilities as one deployment; it can still be modular and horizontally scaled. Microservices place cohesive business capabilities behind independently deployable network contracts, usually with owned data. This can improve team autonomy, isolated scaling, and blast-radius control, but turns local calls and transactions into distributed failure, consistency, versioning, and observability problems. I choose between them from domain/team complexity and measured isolation needs, not traffic or fashion alone."

### 20.2 "How do you choose monolith versus microservices?"

Use **SPLIT**:

- **S:** scope/domain certainty;
- **P:** people, ownership, and release pace;
- **L:** load/resource differences;
- **I:** isolation for failure/security/compliance;
- **T:** tooling maturity and transaction/operational cost.

Conclusion cue:

> Small team + uncertain domain + shared invariants -> modular monolith. Many autonomous teams + stable contexts + distinct scale/isolation + mature platform -> microservices. State an evolutionary trigger.

### 20.3 "How large should a microservice be?"

Use **C-DATA**:

- **Cohesion:** one coherent business purpose/reason to change.
- **Deployability:** can release without synchronized neighbors.
- **Atomicity/data:** invariants and authoritative writes stay local.
- **Traffic/team:** separate scale and one-team cognitive ownership add value.
- **Availability/operations:** boundary isolates failure enough to repay its fixed cost.

Close with smells:

> Chatty calls, shared tables, and lockstep changes mean too small/wrong; unrelated models, teams, scaling, and release contention mean too large.

### 20.4 "How do you decompose by business capability?"

Cues:

> user/business outcomes -> map what the business does -> group related rules/change -> cross-functional ownership -> test data/invariants/dependencies -> refine broad capabilities -> validate with failure and team scenarios.

Do not say "UI, backend, database." Give a concrete capability map such as Catalog, Inventory, Ordering, Payments, and Fulfillment.

### 20.5 "How do you use DDD to identify services?"

Use **DOMAIN**:

1. **D - Discover** journeys, commands, events, and business policies with experts.
2. **O - Organize** knowledge into core/supporting/generic subdomains.
3. **M - Model** bounded contexts with their own ubiquitous language.
4. **A - Aggregate** the data/rules that require one transactional invariant.
5. **I - Interactions**: define context map, APIs/events, ownership, and consistency.
6. **N - Non-functional/evolution**: test scale, failure, security, team, and start coarse.

Critical distinction cue:

> Subdomain = problem area; bounded context = model/language boundary; aggregate = local consistency boundary; microservice = deployable/operational boundary.

### 20.6 "How would you use EventStorming to find service boundaries?"

Use **S-T-O-R-M**:

1. **S - Scope and stakeholders:** choose a business journey; include people with rules, operations, exceptions, and technical knowledge.
2. **T - Timeline and trouble:** storm past-tense domain events, arrange narratives/branches, and mark pivotal events and hot spots.
3. **O - Origins and outcomes:** add actors, commands, policies, read models, aggregates, external systems, and failure outcomes.
4. **R - Regions of meaning:** cluster by purpose, language, invariants, decision/data owner, lifecycle, and non-functional needs; propose bounded contexts and a context map.
5. **M - Measure before microservices:** test coupling, transactions, team ownership, scale, and failure behavior; implement as modules or coarse services and evolve.

Close with the guardrail:

> "EventStorming produces shared understanding and boundary hypotheses. It does not mean one event, aggregate, or bounded context must become one network service, and a workshop event is not automatically a published message."

### 20.7 "Business capability versus DDD decomposition?"

Cues:

> capability asks what the business can do -> useful top-down map; DDD asks where distinct knowledge/model/language/invariants live -> refines semantic boundaries; combine them; neither mechanically produces one tiny service per box.

### 20.8 "How do you migrate a monolith?"

Use **EDGE**:

- **Establish** candidate from measured pain and map dependencies/data.
- **Divert** calls through a facade/gateway/abstraction seam.
- **Gradually** move behavior, data, and traffic with shadow/canary/reconciliation.
- **Eliminate** old ownership/path after verification.

Add:

> one authoritative writer at each phase, compatible contracts, rollback, observability, and a Strangler approach instead of a big bang.

### 20.9 "How do transactions work across services?"

Use **LOCAL**:

1. **L - Local transactions:** each service commits only owned state.
2. **O - Orchestrate/choreograph** a saga for the business workflow.
3. **C - Compensate** completed steps; model pending/failure states.
4. **A - Atomic outbox** bridges database commit to event publication.
5. **L - Logical idempotency:** stable keys/deduplication make retries safe; monitor/repair stuck flows.

### 20.10 "Why not use a shared database?"

Cues:

> direct table access bypasses owner -> schema/release/data-rule coupling -> ambiguous authority and shared failure -> use API/events/projections -> physical cluster may be shared temporarily with separate schemas/credentials -> state migration plan.

### 20.11 "What if a dependency is down?"

Cues:

> classify dependency as correctness-critical or optional -> deadline -> retry only transient/idempotent with backoff/jitter -> circuit/resource isolation -> degrade/queue/reject explicitly -> surface pending status -> observe SLO and reconcile.

Example:

- recommendations down: show popular titles;
- payment authorization down: keep order pending or reject safely; do not pretend payment succeeded.

### 20.12 "How do you query across services?"

Cues:

> no cross-owner DB join -> low fan-out/fresh result: API composition -> high-read/complex query: event-built materialized view/CQRS -> analytics: warehouse/lake -> disclose staleness, rebuild, privacy, and partial-result behavior.

### 20.13 "Why might you merge two services?"

Cues:

> always change together -> chatty bidirectional calls -> shared invariant/data -> lockstep deploy -> one team and same scaling/failure policy -> boundary adds more cost than autonomy -> merge while retaining internal modules.

## 21. Interview follow-up drills with model reasoning

### Drill A: "We have high traffic. Therefore microservices?"

Not enough information. Ask which capability and bottleneck. A stateless monolith can scale horizontally; caches, CDN, queues, database indexes/replicas/partitioning may address traffic. Microservices are justified if workloads differ enough for isolated scaling or team/release/failure boundaries also matter.

### Drill B: "Catalog and Inventory both talk about products. One service?"

Not necessarily. Catalog owns presentation/classification; Inventory owns stock and reservations. Their rules, owners, write rates, and consistency differ. They share a stable product/SKU identity and published facts, not one universal model. Ordering should not rely on a stale catalog/search projection to reserve stock.

### Drill C: "Should Payment Service call Order Service to update status?"

Avoid giving Payment authority to modify Order internals. Payment can publish `PaymentAuthorized`/`PaymentFailed` with stable IDs; the Ordering owner applies a valid state transition. For an explicitly orchestrated saga, the orchestrator can command steps. In either case define duplicates, ordering, outbox, and state-machine validation.

### Drill D: "Can services share a database?"

They can physically share infrastructure, particularly during transition, but should have exclusive logical schema/table ownership and credentials. Direct cross-writes destroy independence. If strong transactional coupling constantly requires shared tables, question whether the services should be separate.

### Drill E: "Does separate deployment mean zero coordination?"

No. Product features and contract changes sometimes need coordination. Independent deployment means compatible versions can roll out separately without a mandatory simultaneous release. Use additive contracts, expand-migrate-contract schema changes, feature flags, and consumer visibility.

### Drill F: "A payment completed, but the response timed out. What does Ordering do?"

The outcome is unknown, not automatically failed. Retry/query Payment using the same idempotency key/payment ID. Payment returns the recorded result. Ordering remains `PAYMENT_PENDING` until it receives/reconciles the authoritative outcome; never create a second capture with a fresh key just because the first response was lost.

### Drill G: "Notifications are separate, so order confirmation is reliable?"

Separation prevents provider slowness from blocking order commit only if Ordering durably records an event/outbox. Notification then consumes idempotently, retries with backoff, tracks provider response, respects preferences, and sends poison messages to an inspectable failure path. The order can succeed while notification is delayed.

### Drill H: "How would you detect a wrong boundary?"

Measure change coupling, synchronized deployments, call volume/latency, incidents, shared-data violations, feature touch points, and ownership ambiguity. Too many cross-boundary transactions and bidirectional calls suggest merging or moving behavior. Independent scaling and release with a stable contract validate a useful split.

### Drill I: "Should each bounded context be a separate repository?"

Repository strategy is independent. A monorepo can enforce separate builds/owners/deployments; multiple repos can still be tightly coupled. Choose based on tooling and ownership, while measuring actual contract and deployment independence.

### Drill J: "Can a monolith use DDD?"

Yes. DDD is a modeling/design approach, not a deployment requirement. A modular monolith with bounded-context modules may be the safest starting point because it preserves local refactoring while the team learns the domain.

### Drill K: "How does the system handle a read that needs Order, Payment, and Shipment?"

For an infrequent freshest view, compose bounded API calls with deadlines and partial-result semantics. For a high-volume order-history screen, build a materialized read model from versioned events and show its freshness. Do not give a reporting service direct read access to all private operational schemas.

### Drill L: "Microservices or SOA?"

**SOA (Service-Oriented Architecture)** is a broad family of systems built from networked services. Microservices are a more specific style emphasizing smaller business-aligned services, independent deployment/ownership, decentralized data, and lightweight automation. Historical SOA implementations often used larger enterprise services and centralized integration middleware, but terminology varies; avoid presenting a universal size cutoff.

### Drill M: "The EventStorming wall has 80 events. Should we create 80 services?"

No. Events describe business facts across a narrative; service boundaries protect coherent models, decisions, and ownership. Cluster signals into bounded-context candidates, identify aggregates/invariants inside them, then test deployability, data ownership, chattiness, scaling, team ownership, and operational cost. Eighty events might remain inside four modules in a modular monolith, become a few coarse services, or reveal that more discovery is needed.

## 22. Compact review sheet

### 22.1 One-screen mental model

```text
Monolith:
  one deployment
  can be modular and horizontally scaled
  local calls + simple transactions + low operating cost
  but capabilities/teams/releases/scaling share fate

Microservices:
  cohesive business services over the network
  independently deployable, owned data, explicit contracts
  gain team/release/scale/failure isolation where boundaries work
  pay latency + partial failure + distributed data + fleet operations

Choose boundary from:
  business reason to change
  bounded context + aggregate/invariant
  deployability + data ownership
  traffic/failure/security needs
  team cognitive ownership
  chattiness + operational fixed cost

Decompose:
  capability map -> EventStorming narratives/hot spots -> subdomains
  -> bounded contexts -> aggregates
  -> contracts/data owners -> failure/workflow tests -> evolve

EventStorming guardrail:
  workshop events reveal business behavior
  they are not automatically services, APIs, or broker messages

Warning signs:
  shared DB + lockstep deploy + chatty calls = distributed monolith
  unrelated models + team/release contention = service too large
```

### 22.2 Vocabulary corrections

| Instead of saying | Say |
|---|---|
| "Microservices scale; monoliths do not" | "Services permit independent scaling; both styles can scale horizontally" |
| "A microservice is under N lines" | "Size follows cohesion, invariants, independent deployment, ownership, and operating cost" |
| "One database per service" | "Each service exclusively owns its model/data access; physical isolation can vary" |
| "Events decouple everything" | "Events reduce temporal coupling but create schema, semantic, ordering, and delivery dependencies" |
| "Saga rolls everything back" | "Saga uses local transactions and business compensations; temporary states and failed compensation are explicit" |
| "DDD domain equals service" | "Subdomain is problem space, bounded context is model boundary, aggregate is consistency boundary, service is deployment/ownership boundary" |
| "Fault isolation comes free" | "Isolation requires bounded dependencies, resource separation, timeouts, fallbacks, and observability" |
| "Kubernetes means microservices" | "Kubernetes operates workloads; it does not discover good domain boundaries" |
| "Start with microservices for future scale" | "Start at the coarsest boundary that meets today's needs and preserves useful seams" |
| "One EventStorming sticky equals one Kafka event/service" | "Workshop events expose business behavior; publication and deployment boundaries are separate design decisions" |

### 22.3 Metrics worth mentioning

- deployment frequency and lead time;
- change failure/rollback rate;
- number of teams/services touched per feature;
- build/test duration;
- p50/p95/p99 latency and error rate;
- downstream call count and dependency depth;
- queue lag and oldest-message age;
- saga duration, failures, compensation, and stuck-workflow count;
- contract compatibility failures;
- service ownership/orphan count;
- data reconciliation drift;
- incident blast radius and recovery time;
- per-capability resource cost; and
- user/business success rate, not only infrastructure health.
