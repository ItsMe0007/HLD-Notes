# Idempotency and Idempotency Keys in System Design

> Beginner-first HLD notes expanded from the supplied bullets. The goal is to understand why duplicate requests happen, what HTTP does and does not promise, and how to make one logical action have one intended effect across retries, concurrent requests, multiple servers, queues, and failures.

## Scope, terminology, and confidence labels

The phrase **idempotency handle** in the rough notes is usually called an **idempotency key**, **request token**, **client token**, or **request ID**. This chapter uses *idempotency key*.

I interpret the bullets as asking about both:

1. **HTTP method semantics:** why `GET`, `PUT`, and `DELETE` are defined as idempotent while `POST` is not idempotent by default; and
2. **application-level implementation:** how an API makes a business operation such as charging a card or creating an order safe to retry sequentially or concurrently across many servers.

Labels used in the examples:

- **Verified public behavior** is backed by a standard or first-party public documentation linked in the note.
- **Illustrative design** is how one could design a named product in an interview. It is not a claim about that company's private current architecture.
- Public interview reports are **candidate-authored and unverified by the company**. They are evidence of realistic prompt shapes, not official question banks.

## 1. Plain-language foundation

### 1.1 The double-payment story

Suppose you tap **Pay ₹500**. Your phone sends a request to the payment server. The server charges the card, but its response is lost because the network disconnects.

```text
Phone                     Payment API                    Bank
  |                            |                           |
  |--- charge ₹500 ----------->|--- charge ₹500 ---------->|
  |                            |<-- success ----------------|
  |<--- response lost ----X    |
  |
  | What happened? The phone cannot tell.
```

The phone now faces an **ambiguous outcome**:

- perhaps the request never reached the server;
- perhaps it reached the server but failed before charging;
- perhaps it charged successfully and only the response was lost.

If the phone never retries, a valid payment might be lost. If it blindly retries, the customer might be charged twice.

**Idempotency makes that retry safe.** The client attaches a stable identifier to the logical payment:

```http
POST /payments HTTP/1.1
Idempotency-Key: pay_attempt_7f41...
Content-Type: application/json

{"order_id":"order-91","amount":500,"currency":"INR"}
```

Every retry for *this same intended payment* reuses the same key. The server records what happened for that key. It either performs the operation once or returns the result/status of the earlier attempt instead of charging again.

```text
first request:  key K -> perform payment -> remember K -> payment P123
retry:          key K -> K already known -> return payment P123
```

### 1.2 The core definition

An operation is **idempotent** when applying the same operation multiple times has the same **intended effect** as applying it once.

In mathematical notation, if `f` is the operation:

```text
f(f(x)) = f(x)
```

System-design translation:

```text
"Set the account's mailing city to Pune" repeated 5 times
    -> final city is Pune

"Add ₹500 to the balance" repeated 5 times
    -> balance increases by ₹2,500; not idempotent
```

The word **same** matters:

- same logical intent;
- same relevant parameters;
- same caller/security scope;
- inside the server's documented deduplication time window.

A different payment, even for the same amount and recipient, is a new intent and needs a new key.

### 1.3 Same effect does not mean same response

Idempotency is primarily about the requested **effect**, not about returning byte-for-byte identical responses.

For example:

```text
DELETE /photos/42  -> 204 No Content   (photo existed and is now absent)
DELETE /photos/42  -> 404 Not Found    (photo is already absent)
```

The responses differ, but after either one or two identical deletes the intended resource state is the same: photo `42` is absent.

An API can offer a **stronger application contract** and replay the original response exactly. Stripe publicly documents this style. Another API can return the resource's latest status; PayPal publicly documents that style for supported operations. Both can preserve the one-effect property, but clients observe different response semantics.

### 1.4 Idempotent is not the same as safe

These terms are often confused:

- A **safe** HTTP operation is essentially read-only from the user's point of view. `GET` is supposed to be safe.
- An **idempotent** operation may change state, but repeating it does not add further intended effects. `DELETE` changes state and is therefore unsafe, yet it is idempotent.

```text
                         Idempotent?
                     no             yes
Safe?     yes       unusual       GET, HEAD, OPTIONS, TRACE
          no        POST*          PUT, DELETE

* POST is not guaranteed idempotent by its method semantics,
  but a particular POST API can add an idempotency-key contract.
```

### 1.5 Idempotency is also not these things

| Related term | What it means | Why it is different |
|---|---|---|
| Deduplication | Detect/remove repeated identifiers or data | A mechanism often used to implement idempotency; merely spotting a duplicate does not define the correct stored result or recovery behavior |
| At-most-once delivery | Try to deliver zero or one time | Can lose work; idempotent retries commonly use at-least-once attempts to avoid loss |
| Exactly-once processing | A claim tied to a specific processing boundary | It might cover a broker transaction but not an external email, bank, or database side effect |
| Atomicity | A group of changes happens all-or-nothing | Helps couple the idempotency record and business mutation; does not identify repeated intent by itself |
| Locking | Temporarily excludes concurrent actors | Stops overlap while held, but does not remember a completed result after the lock is released |
| Optimistic concurrency | Prevents a stale writer from overwriting newer state | Solves competing updates, even when their idempotency keys differ |
| Rate limiting | Restricts request frequency | Reduces abuse/load, but cannot prove a payment happened only once |

## 2. Prerequisites explained inline

These concepts are necessary for the rest of the chapter, so they are introduced here rather than deferred to “study next.”

### 2.1 Minimal HTTP request/response model

An HTTP client sends:

- a **method**, such as `GET`, `POST`, `PUT`, `PATCH`, or `DELETE`;
- a **target URI**, such as `/orders/42`;
- headers, such as authentication and `Idempotency-Key`;
- optionally a body.

The server replies with:

- a numeric **status code**, such as `201` or `409`;
- headers;
- optionally a response body.

HTTP method semantics are a contract visible to clients, proxies, gateways, and retry libraries. They are not automatically enforced by the network. A developer can incorrectly build `GET /transfer-money`; doing so violates the expected `GET` semantics.

### 2.2 Why distributed requests are retried

Networks and processes can fail at any point. A timeout only tells the caller, “I did not receive a conclusive response in time.” It does **not** prove that the server did nothing.

Common retry sources include:

- a user double-clicking;
- a mobile app retrying after reconnecting;
- an SDK retrying a timeout or `503`;
- a gateway retrying an upstream connection failure;
- a worker receiving the same at-least-once message again;
- a client process crashing and recovering an unfinished command;
- two tabs/devices submitting the same business action.

Retries are useful because many transient failures disappear on a later attempt. Idempotency lets the system use retries for availability without multiplying business effects.

### 2.3 Race condition, atomic operation, and transaction

A **race condition** occurs when correctness depends on which concurrent operation happens first.

This is unsafe:

```text
Server A: check key K -> absent
Server B: check key K -> absent
Server A: charge
Server B: charge
```

An operation is **atomic** when other actors cannot observe or interleave with a partial step. A database unique constraint plus an atomic insert-if-absent can elect exactly one owner for key `K`.

A database **transaction** can group the idempotency record and local business mutation so they commit or roll back together:

```text
BEGIN
  claim key K
  create order O
  record K -> completed, order O
COMMIT
```

If the process crashes before commit, none of those changes becomes committed. If it commits, all of them do.

### 2.4 Cache versus source of truth

A **cache** is normally a faster, disposable copy of authoritative data. Cache copies can be stale, evicted, or temporarily inconsistent.

If correctness depends on “only one payment,” a collection of independent application-server caches is not a safe authority:

```text
Server A cache: K absent       Server B cache: K absent
         |                              |
         +-------- both execute --------+
```

“Synchronize caches very fast” is not a correctness guarantee. Even a tiny propagation delay leaves a race window. Correctness needs one atomic decision at an authoritative owner, such as a suitable database constraint, a strongly consistent conditional write, or a deliberately chosen single primary with understood durability/failover behavior.

**Strongly consistent for this decision** means that after one claim succeeds, a competing claim cannot still be accepted as if the key were absent. It does not merely mean that copies will probably converge soon.

## 3. Are `GET`, `PUT`, and `DELETE` idempotent? How exactly?

[RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html#name-idempotent-methods) defines safe methods plus `PUT` and `DELETE` as idempotent. This is a statement about the methods' **intended semantics**.

### 3.1 Method-by-method table

| HTTP method | Safe? | Idempotent by standard semantics? | Beginner explanation |
|---|---:|---:|---|
| `GET` | Yes | Yes | Retrieve a representation. Repeating the read should not request another business mutation. |
| `HEAD` | Yes | Yes | Same semantics as `GET` but without response content. |
| `OPTIONS` | Yes | Yes | Ask about communication options/capabilities. |
| `TRACE` | Yes | Yes | Diagnostic loop-back semantics; commonly disabled for security/operational reasons. |
| `PUT` | No | Yes | Create or replace the state at a client-known target URI with the supplied desired state. Repeating the same replacement reaches the same desired state. |
| `DELETE` | No | Yes | Request removal of the target resource association. Once absent, repeating the request leaves it absent. |
| `POST` | No | No guarantee | Ask the target resource to process the body according to its own semantics, often creating a new subordinate resource or triggering an action each time. A particular POST can add an idempotency contract. |
| `PATCH` | No | No guarantee | Apply change instructions. Some instructions are repeatable; others, such as increment or append, accumulate. It can be designed to be idempotent. |

### 3.2 Why `PUT` is idempotent

`PUT` expresses a **desired resource state**, normally at a URI already known by the client.

```http
PUT /users/42/preferences HTTP/1.1
Content-Type: application/json

{"theme":"dark","language":"en"}
```

Apply it once:

```text
/users/42/preferences = {theme: dark, language: en}
```

Apply the identical request five times:

```text
/users/42/preferences = {theme: dark, language: en}
```

It remains the same desired state. RFC 9110 says `PUT` creates or replaces the state of the target resource. If a `PUT` creates the previously absent representation, `201 Created` is appropriate; if it modifies an existing one, `200 OK` or `204 No Content` is appropriate.

#### A misuse that is not idempotent

```http
PUT /wallets/42

{"action":"increment","amount":500}
```

That body describes a command that accumulates, not the replacement state of `/wallets/42`. Repetition changes the balance repeatedly. Calling it `PUT` does not magically make the implementation idempotent; the API violates the method's intended semantics.

Better options are:

- `PUT /wallets/42` with an absolute desired representation/version where that model is valid;
- `POST /wallets/42/credits` with an idempotency key for a distinct credit operation;
- a ledger entry with a unique business transaction ID.

#### A subtlety: concurrent different PUTs

Idempotency only discusses repetition of the **same request**. It does not resolve two different desired states:

```text
PUT A: set language = en
PUT B: set language = hi
```

Whichever arrives last might win. To prevent stale overwrites, use optimistic concurrency such as `ETag` + `If-Match`, a version field, or a database compare-and-set. An `ETag` is a server-provided version marker; `If-Match` says “apply my update only if the resource still has the version I read.” This is a different problem.

### 3.3 Why `DELETE` is idempotent

Consider this state transition:

```text
before: order_note_42 exists
DELETE once: order_note_42 absent
DELETE again: order_note_42 still absent
```

`DELETE` is not safe because it changes state on the first successful call. It is idempotent because repeating the same deletion does not request another target-resource effect.

The status may vary:

```text
first call  -> 204 No Content
second call -> 404 Not Found
```

That does not violate idempotency. Some APIs instead return `204` on every call to make deletion easier for clients. Either response policy should be documented.

#### Delete implementation pitfalls

- If every retry sends another customer email or refunds money again, the broader business operation is not safely idempotent.
- A soft delete that overwrites `deleted_at = now()` on every retry changes metadata each time. Usually preserve the original deletion time.
- “Delete the oldest pending job” is not a stable resource target: every retry can delete a different job. Model the chosen job ID explicitly.

### 3.4 Why `GET` is idempotent and safe

`GET /orders/42` asks for a representation. Repeating it can return different data because another user updated the order or time passed, but the repeated `GET` itself should not request a state-changing business effect.

Allowed incidental effects can still differ per call:

- access logs;
- metrics;
- cache warming;
- an advertisement accounting event.

RFC 9110 explicitly distinguishes these incidental server effects from what the user requested. A dangerous endpoint such as `GET /accounts/42/withdraw?amount=500` violates safe-method semantics and can be triggered by crawlers, prefetchers, or link scanners.

### 3.5 Why `POST` is not idempotent by default—but can be made idempotent

`POST /orders` commonly means, “create a new order chosen/identified by the server.” Repeating it can create `order-101`, then `order-102`, so HTTP cannot assume repetition is safe.

```http
POST /orders

{"item":"book","quantity":1}
```

However, the application can define a stronger contract:

```http
POST /orders
Idempotency-Key: checkout-session-91-attempt-1

{"item":"book","quantity":1}
```

Now the method is still `POST`, but this endpoint promises to associate repeated uses of that key and payload with one logical order.

Another option, when the client is allowed to choose the resource ID, is:

```http
PUT /orders/client-order-7f41

{"item":"book","quantity":1}
```

### 3.6 What about `PATCH`?

[RFC 5789](https://www.rfc-editor.org/rfc/rfc5789.html) says `PATCH` is neither safe nor inherently idempotent, but a particular patch can be constructed idempotently.

```text
idempotent-style patch:     replace /status with "PAID"
non-idempotent-style patch: increment /reward_points by 10
non-idempotent-style patch: append "A" to /tags array
```

Even an idempotent patch can overwrite a newer concurrent change. Conditional requests such as `If-Match: <etag>` protect the expected base version.

## 4. Why idempotency is needed in system design

### 4.1 It resolves “did it happen?”

In a local function call, a return value often tells the caller what happened. Across a network, the business commit and the caller receiving the response are separate events.

```text
              ambiguity window
                      |
request -> validate -> commit effect -> serialize response -> network -> client
                         ^                                  X
                         effect happened                    reply lost
```

The caller needs a stable way to ask, “Is this the same operation I previously attempted?” That is what the idempotency key represents.

### 4.2 It enables availability through safe retries

Without safe retry, the client must choose between:

- **at-most-once attempt:** fewer duplicates but possible lost work; or
- **at-least-once attempt:** eventual progress but possible repeated effects.

Idempotent handling allows at-least-once attempts to produce an **effectively-once business result within a defined boundary and retention window**.

Do not casually claim universal “exactly once.” The guarantee only holds where the identifier, atomic state, downstream effects, expiry, and failure recovery all preserve it.

### 4.3 It appears at every retry boundary

```text
Mobile client
  | retry HTTP request
  v
Order API
  | retry Payment API
  v
Payment service
  | retry bank request
  v
External processor

Order event -> broker redelivery -> fulfillment consumer
```

Protecting only the mobile-to-order boundary is insufficient if the order service retries the payment call without a stable downstream key. Each component that can repeat a non-idempotent effect needs either:

- an idempotent operation;
- an idempotency key propagated/derived for that boundary;
- an atomic business uniqueness constraint;
- or explicit reconciliation where atomicity is impossible.

### 4.4 High-value use cases

- payments, refunds, payouts, and wallet transfers;
- order, booking, ride, or ticket creation;
- provisioning a VM, volume, or subscription;
- sending a notification when duplicates are undesirable;
- processing at-least-once events;
- inventory reservation;
- multipart/resumable uploads;
- webhook consumption;
- scheduling or dispatching a background job.

For a harmless, cheap, naturally idempotent state set, a full result-record system can be unnecessary complexity. The required strength follows the consequence of duplication.

## 5. Expanding and correcting the rough bullets

| Rough bullet | Correct, interview-ready version |
|---|---|
| “GET, PUT, DELETE are idempotent?” | RFC 9110 defines safe methods (`GET`, `HEAD`, `OPTIONS`, `TRACE`) plus `PUT` and `DELETE` as idempotent in their intended semantics. Endpoint implementations can still violate those semantics. |
| “PUT DELETE how” | `PUT` repeats the same desired replacement at the same URI. `DELETE` repeats the desired absence of the same target. The response can differ even when the intended effect does not. |
| “POST isn't” | `POST` has no general idempotency guarantee because repeated processing may create/trigger multiple things. A specific POST can be made retry-safe using a key, client-chosen business ID, or uniqueness constraint. |
| “Different response codes: 200, 201, 400, 409, 429” | Statuses describe different conditions; idempotency does not require the same status. `429` is rate limiting, not duplicate detection. `422` is also important for same-key/different-payload misuse in the current IETF draft. |
| “Sequential / Parallel” | Completed sequential duplicates can replay a stored outcome. Concurrent duplicates require an atomic claim so only one worker executes; losers wait, poll, receive `409`, or receive `202` according to the contract. |
| “Contract with client, why depend on client?” | The client normally knows whether two attempts represent the same user intent. The server still owns enforcement: require/validate/scope the key, compare payloads, atomically persist state, and document expiry. A two-step server-generated operation ID is an alternative. |
| “Locking in Parallel” | A lock may serialize in-flight requests but is not enough alone. A durable result record or business invariant is still needed after the lock expires/releases. Prefer an atomic unique claim/state machine over holding a distributed lock through a long external call. |
| “Multiple servers => cache synchronization fast” | All instances must consult one authoritative atomic idempotency owner or correctly partition keys to one owner. Independent local caches—even quickly synchronized—leave races, eviction, restart, and failover gaps. |

## 6. The idempotency-key contract

### 6.1 What the key means

An idempotency key is not merely “a random header.” It is a protocol statement:

> For this authenticated caller, operation scope, and payload, all attempts using this key represent one logical intent.

The effective lookup identity is usually closer to:

```text
(tenant/account, API operation or route, idempotency key)
```

not just:

```text
idempotency key
```

Scoping prevents one customer from colliding with or retrieving another customer's result, and it allows the same opaque key to be used on a different documented operation if that is the chosen contract.

### 6.2 Why the client often generates it

Only the caller reliably knows whether two requests express one intent or two separate intents.

Identical payloads are ambiguous:

```text
Pay ₹500 to the electricity provider today
Pay another ₹500 to the same provider today
```

These can be two legitimate payments with identical bodies. A server that deduplicates solely by payload hash could incorrectly suppress the second one.

Conversely, two retry payloads might differ in irrelevant fields such as a client timestamp while still representing the same intent. Pure payload equality cannot always infer user intent.

The client's responsibility does **not** mean the server trusts the client for correctness. The server must enforce the contract. Common client patterns are:

- generate a high-entropy UUID/ULID when the user begins one logical action;
- store it with the pending local action so it survives connection loss or app restart;
- reuse it for automatic/manual retries of that action;
- generate a new key only when the user deliberately starts a new action.

A key generated afresh inside every HTTP retry attempt is useless.

### 6.3 Alternatives to depending on a client-generated key

#### Alternative A: server-created operation resource

```text
POST /payment-intents          -> 201, intent_id = pi_42
POST /payment-intents/pi_42/confirm
GET  /payment-intents/pi_42
```

The first step creates a stable server ID; later calls target that operation. The first creation can itself need safe-retry handling, or the client can provide a business ID.

#### Alternative B: client-chosen resource URI

```text
PUT /orders/client-order-7f41
```

This naturally says “make this identified order exist in this state.” It is suitable only when allowing client-selected identifiers and replacement semantics makes sense.

#### Alternative C: natural business uniqueness

Examples:

- one refund per `(original_payment_id, refund_reference)`;
- one payroll transfer per `(company_id, payroll_run_id, employee_id)`;
- one message per `(conversation_id, sender_device_id, client_message_id)`.

A database unique constraint on that invariant can be stronger and longer-lived than a short idempotency-cache window.

#### Alternative D: server/SDK-generated and persisted token

An official SDK can create a token once for the logical command and transparently reuse it on transport retries. The important part is persistence across attempts, not which piece of code called the random-number generator.

### 6.4 Payload fingerprint

The server should remember a fingerprint or selected semantic fields so that this error is caught:

```text
key K + {amount: 500, currency: INR}
key K + {amount: 5000, currency: INR}  <-- same key, different intent/payload
```

Do not silently return the earlier ₹500 result for the ₹5,000 request.

A robust fingerprint can include:

- authenticated tenant/account ID;
- HTTP method and normalized route/operation name;
- API version;
- normalized semantic request fields;
- possibly a hash of canonicalized content.

Canonicalization needs care: JSON whitespace and object-key order should not create different meanings; volatile request timestamps or trace IDs often should not be included; secrets should not be copied into logs. In high-assurance systems, storing selected validated fields can be clearer than relying on a naïve raw-body hash.

### 6.5 Expiry and retention window

Idempotency records cannot always be stored forever. The API must document how long the key is remembered.

Choose retention from:

- the maximum client/SDK retry horizon;
- offline mobile retry behavior;
- delayed queue/webhook redelivery;
- the cost of a duplicate;
- legal/privacy retention requirements;
- record volume and storage cost.

After expiry, the same key might be treated as new. Therefore a short TTL is not a permanent exactly-once guarantee. Financial/business invariants often also use permanent transaction/order IDs and ledger uniqueness.

Time-specific public examples such as provider retention periods can change; check the current endpoint documentation rather than memorizing a universal number.

### 6.6 What the server must publish

A usable contract documents:

- which operations support or require the key;
- header/field name, allowed format, maximum length, and scope;
- whether the same key with different parameters is rejected;
- retention/expiry behavior;
- what happens for an in-flight duplicate;
- which results are saved, including validation errors and `5xx` responses;
- whether replays return the original response or latest operation state;
- which statuses are safe to retry and whether to use `Retry-After`;
- whether the client can query a status URL after an ambiguous result.

The current [IETF Idempotency-Key HTTP Header Internet-Draft](https://datatracker.ietf.org/doc/html/draft-ietf-httpapi-idempotency-key-header) describes a common contract, but **as checked on 2026-07-16 it is an Internet-Draft, not a final RFC; details can change**.

## 7. Response codes and retry behavior

### 7.1 Status-code table

| Status | Typical meaning in an idempotent API | Retry guidance |
|---:|---|---|
| `200 OK` | Existing resource updated, operation result returned, or completed outcome replayed | Usually no retry needed |
| `201 Created` | The operation created a resource; include `Location` when appropriate | A replay may repeat `201` if the API replays the original response, or may return an equivalent/latest result according to contract |
| `202 Accepted` | Async work was accepted but not yet completed; ideally return an operation/status URI | Poll/status-check; do not create a new key merely because it is pending |
| `204 No Content` | Successful `PUT`/`DELETE` with no response content | No retry needed |
| `400 Bad Request` | Malformed request, invalid key syntax, or missing key when the API requires one | Fix the request; whether to reuse the key depends on whether execution began and the documented contract |
| `404 Not Found` | Resource absent; a repeated `DELETE` can legitimately return this and remain idempotent | Usually terminal unless eventual creation/replication is expected |
| `409 Conflict` | Common policy for the same key still being processed; can also mean a resource-state conflict | Same-key retry later if it means “in progress”; inspect machine-readable problem type rather than status alone |
| `412 Precondition Failed` | `If-Match`/`If-None-Match` condition failed | Refresh resource/version; this is optimistic concurrency, not key deduplication |
| `422 Unprocessable Content` | Current IETF draft recommendation for a known key reused with a different payload | Do not retry unchanged; use the correct original payload or start a truly new intent with a new key |
| `429 Too Many Requests` | Rate limit exceeded; not an idempotency duplicate code | Wait per `Retry-After`/backoff, then use the **same key** if retrying the same intent |
| `500`, `502`, `503`, `504` | Server/upstream failure or timeout; outcome may be known, failed-before-execution, or ambiguous | Follow the API contract; retry the same intent with the same key, bounded backoff, and jitter when permitted |

`429` is standardized for rate limiting by [RFC 6585](https://www.rfc-editor.org/rfc/rfc6585.html#section-4). It should not be used to mean “I have already seen this idempotency key.”

The draft's `400`/`409`/`422` mapping is a useful convention, not a universal law. Existing APIs use different policies. A stable machine-readable error body, such as [RFC 9457 Problem Details](https://www.rfc-editor.org/rfc/rfc9457.html), lets the client distinguish “key in progress” from an unrelated `409`.

**Backoff** means increasing the delay between repeated failures. **Jitter** adds a small random variation so thousands of clients do not retry at exactly the same instant.

### 7.2 Original response versus latest status

There are two common policies:

#### Policy 1: replay the original result

```text
first completed response: 201 {payment_id: P1, status: processing}
later same-key replay:     201 {payment_id: P1, status: processing}
```

Advantages:

- deterministic retries;
- client can treat retry as the missing response;
- simple audit of the command's original outcome.

Costs:

- response storage;
- potentially stale status;
- privacy/size concerns;
- deciding whether to cache a `500` is subtle.

[Stripe's public idempotency documentation](https://docs.stripe.com/api/idempotent_requests) says it saves the first request's resulting status code and body once endpoint execution begins and returns that result on later uses of the key, including saved `500` results. This is provider-specific behavior, not a rule every API must copy.

#### Policy 2: return the operation/resource's latest status

```text
first completed response: 201 {payment_id: P1, status: processing}
later same-key replay:     201 {payment_id: P1, status: completed}
```

Advantages:

- immediately useful current state;
- less need to store a large original response if resource ID is enough.

Costs:

- retry response is not byte-for-byte stable;
- the client must understand that this is a status view.

[PayPal's public idempotency guidance](https://developer.paypal.com/reference/guidelines/idempotency/) says supported retries return the latest status of the earlier request and warns that simultaneous same-ID requests can cause the second request to fail. Again, that is one provider's contract.

### 7.3 Which failures should be finalized?

A useful classification is:

| Failure point/type | Usually finalize the key? | Reason |
|---|---:|---|
| Missing/invalid key, authentication failure, malformed body before handler execution | Often no | No business operation began; corrected request may be allowed |
| Terminal business decline after execution began | Usually yes | Repeating should not run a second business decision/effect unexpectedly |
| Transient infrastructure failure known to be before any execution | Often no | Same-key retry should get a chance to execute |
| Unknown outcome after an external request | Keep `PENDING`/`UNKNOWN`, not blindly failed | A retry must recover/query before deciding to repeat the effect |
| Successful business commit | Yes | Store outcome/resource reference before responding |
| Internal `500` after business commit | Yes or recoverable state | Otherwise a retry can duplicate the committed effect |

There is no correct universal “cache every error” rule. Define it based on exactly when execution starts and whether the outcome is known.

## 8. Sequential duplicate requests

Sequential means the later retry reaches the service after the first attempt has finished or at least persisted a visible state.

### 8.1 Minimal record

```text
IdempotencyRecord
  scope               tenant + operation
  key                 client token
  request_fingerprint validated semantic input
  status              PROCESSING | SUCCEEDED | FAILED_TERMINAL | UNKNOWN
  resource_id         resulting order/payment/job ID
  http_status          optional saved response status
  response_body        optional saved/reconstructable response
  created_at
  updated_at
  expires_at
  lease_owner/lease_until  optional for recoverable workflows

UNIQUE(scope, key)
```

### 8.2 Sequential flow

```text
Client                     API                     Idempotency store
  |                         |                              |
  |--- key K, payload P --->|--- atomic claim K,P -------->|
  |                         |<-- winner/new ----------------|
  |                         |--- perform operation          |
  |                         |--- store result R ----------->|
  |<-- R -------------------|                              |
  |
  |--- retry K,P ---------->|--- read K ------------------->|
  |                         |<-- completed R ---------------|
  |<-- replay/equivalent R--|                              |
```

Decision rules:

1. Validate authentication, basic request form, and key format.
2. Atomically insert/claim `(scope, key)` with the request fingerprint.
3. If this call won, execute through the appropriate transactional/state-machine path.
4. Mark a terminal result and store the needed response/resource reference.
5. If the record already exists:
   - reject a fingerprint mismatch;
   - replay/reconstruct a completed result;
   - handle `PROCESSING` according to the concurrent-request contract;
   - recover an expired/unknown attempt rather than blindly re-executing.

### 8.3 Why `check -> perform -> save key` is broken even sequentially

```text
check K -> absent
perform charge -> succeeds
process crashes before save K
retry sees K absent
perform charge again
```

Recording only **after** the effect leaves a crash window. Recording only **before** the effect creates the opposite window:

```text
save K as done
process crashes before charge
retry sees K done and suppresses the payment
```

The record needs states and atomic coupling/recovery, not a single boolean “seen.”

## 9. Parallel duplicates, locking, and atomic claims

### 9.1 The classic race

```text
Time       Server A                         Server B
t1         GET key K -> absent              GET key K -> absent
t2         execute payment                  execute payment
t3         SET K -> complete                SET K -> complete

Result: two effects, one dedupe record
```

Making `GET` and `SET` individually atomic does not make the **read-decide-write sequence** atomic.

### 9.2 Prefer an atomic winner election

Use a uniqueness constraint or conditional insert:

```sql
INSERT INTO idempotency_records
  (tenant_id, operation, idem_key, request_hash, status)
VALUES
  (:tenant, :operation, :key, :hash, 'PROCESSING')
ON CONFLICT (tenant_id, operation, idem_key) DO NOTHING;
```

Exactly one concurrent insert wins. Others learn that a record exists and follow its state. PostgreSQL documents `ON CONFLICT` as handling unique/exclusion conflicts; equivalent mechanisms include DynamoDB conditional `PutItem`, a consensus-backed compare-and-set, or an appropriate atomic operation in another store.

This approach is often clearer than “take a distributed lock, then remember later,” because the claim itself is durable state and can eventually contain the result.

### 9.3 What should the losing concurrent request receive?

Choose and document one policy:

| Policy | Behavior | Good for | Cost/risk |
|---|---|---|---|
| Wait briefly | Block until the winner completes, then return its result | Fast operations with tight latency bounds | Consumes connection/thread/task capacity; timeout still possible |
| `409 Conflict` | Say same key is currently outstanding | Simple synchronous APIs; matches current IETF draft suggestion | Client must retry/poll correctly; `409` has other meanings too |
| `202 Accepted` + status URI | Tell both callers the operation is pending | Long-running async operations | Requires an operation resource and polling/webhook contract |
| Return current `PROCESSING` representation | Same operation ID/status to all callers | Explicit state-machine APIs | Client must handle nonterminal responses |

Do not let the losing request start the business effect merely because the winner is slow.

### 9.4 Do we need a lock?

Sometimes, but separate the purposes:

- **Unique claim:** determines that all same-key attempts refer to one record/operation.
- **Lock or lease:** determines which worker may advance a recoverable operation at this moment.
- **Saved outcome:** makes later retries return/reconstruct the earlier result.

A row lock can serialize a short transaction. An optimistic conditional state change can elect a worker:

```text
UPDATE operation
SET state = 'EXECUTING', owner = W2, version = version + 1
WHERE id = K AND state = 'PENDING' AND version = 7
```

A lease is useful for a long workflow but introduces expiry questions. If worker A pauses past lease expiry and worker B takes over, A might resume and act late. **Fencing tokens**—monotonically increasing ownership versions accepted by the downstream resource—can reject stale owners when the downstream supports them.

### 9.5 Why a Redis lock alone is not the answer

A Redis `SET key value NX PX ttl` can atomically acquire a time-limited claim on one Redis primary. But:

- lock expiry does not prove the side effect stopped;
- after release, future retries still need the saved outcome;
- a process can crash after an external effect and before storing completion;
- asynchronous replication/failover can lose a recently acquired key;
- eviction or accidental deletion can erase dedupe memory;
- long pauses can outlive a lease.

Redis's own [distributed-lock documentation](https://redis.io/docs/latest/develop/clients/patterns/distributed-locks/) explains that simple primary-replica failover cannot preserve mutual exclusion because replication is asynchronous. Redis can still be an excellent low-latency idempotency store when its failure guarantees match the business risk, persistence is configured appropriately, and the state-machine gaps are addressed. It is not automatically a financial correctness proof.

### 9.6 Avoid holding a database transaction across a slow external call

This pattern is tempting:

```text
BEGIN
lock idempotency row
call bank for 20 seconds
update row
COMMIT
```

It holds database connections/locks, increases contention, and still cannot atomically commit the bank and local database. For an external side effect, persist an operation state, commit, call the external system with its own idempotency reference, then record/query/reconcile the result.

## 10. Multiple API servers and the cache-synchronization question

### 10.1 Correct baseline architecture

```text
                          +--> API server 1 --+
Client -> Load balancer --+--> API server 2 --+--> authoritative idempotency owner
                          +--> API server 3 --+       | atomic claim/result
                                                       v
                                                 domain DB / workflow
                                                       |
                                                       v
                                                downstream service
```

The API servers can remain stateless. Any server can receive the retry because all of them use the same logical authority.

The authority can be:

- a relational database with a unique constraint and transactions;
- a strongly consistent key-value conditional write;
- a sharded state service where `hash(scope, key)` selects one owner;
- Redis when its atomicity, persistence, failover, and tolerated duplicate risk are explicitly acceptable;
- the domain table itself through a natural unique business key.

### 10.2 Why local cache synchronization is insufficient

Assume invalidations propagate in one millisecond. Two requests can still arrive inside that millisecond:

```text
0.000 ms  A checks local cache: K absent
0.100 ms  B checks local cache: K absent
0.200 ms  A starts effect
0.300 ms  B starts effect
0.800 ms  A publishes "K exists"
1.800 ms  B receives invalidation -- too late
```

Other failure modes:

- application restart clears local memory;
- autoscaling creates a cold instance;
- invalidation message is delayed/lost;
- key eviction makes an old operation appear new;
- a partition isolates cache peers;
- sticky sessions break on failover and do not cover concurrent devices.

Local caches may accelerate reads of **completed** results, but a miss must go to the authority, and a cache must not independently grant the right to execute.

### 10.3 Shared cache does not automatically mean strong correctness

A single Redis primary gives atomic command execution at that primary. But durability and failover are separate dimensions. Redis documents asynchronous replication and notes that `WAIT` does not turn the deployment into a strongly consistent CP system; acknowledged writes can still be lost in some failover/persistence configurations ([Redis replication documentation](https://redis.io/docs/latest/operate/oss_and_stack/management/replication/)).

Choose based on consequence:

| Operation | Possible authority choice | Reasoning |
|---|---|---|
| Duplicate analytics event, small overcount acceptable | TTL dedupe in Redis | Low latency; rare loss may be tolerable |
| Send a noncritical push notification once if possible | Durable cache/store + provider message ID | Balance cost and occasional duplicate/loss policy |
| Create order and local inventory reservation | Domain database transaction + unique key | Couples claim and business mutation |
| Charge/refund money | Durable operation/ledger uniqueness + downstream idempotency + reconciliation | Duplicate or lost effects are high impact |

### 10.4 Sharding and hot keys

At high scale, route by a stable hash of `(tenant, operation, key)`. All retries for that identity must reach the same logical shard. Normal random keys distribute well; one extremely hot key from a retry storm remains hot because all duplicates correctly converge on one record.

Mitigations include:

- request coalescing/single-flight at servers as an optimization;
- bounded waiting rather than every request polling aggressively;
- returning `409`/`202` plus backoff;
- caching the completed immutable outcome;
- rate limiting abusive retry storms—without treating rate limiting as the idempotency mechanism.

### 10.5 Multi-region choices

If the same key can reach India and Europe concurrently, independent regional stores can both accept it.

Options:

1. **Home region per tenant/key:** route all writes for that scope to one region. Lower implementation complexity; cross-region latency and home-region availability matter.
2. **Globally consistent database/consensus:** nodes coordinate and agree on one ordered winner before accepting the claim. This gives one conditional decision across regions, with higher write latency/cost and possible unavailability during partitions.
3. **Preallocated regional business IDs/budgets:** useful for some domains, but difficult for arbitrary one-off duplicate intent.
4. **Asynchronous reconciliation:** accept possible duplicates temporarily and merge/compensate later. Only valid when the business can tolerate it; dangerous for irreversible side effects.

This is a PACELC/CAP-style trade-off: during partitions, simultaneously preserving low-latency regional writes, availability, and one global winner is not free.

## 11. Failure analysis: where implementations break

### 11.1 Failure matrix

| Failure point | What the client sees | Danger | Required design response |
|---|---|---|---|
| Before request reaches API | Timeout/connection error | Lost operation if never retried | Retry same intent/key |
| After API receives, before claim | Timeout/error | Usually no effect | Same-key retry can claim |
| After durable claim, before local mutation | Timeout; record says `PROCESSING` | Permanent stuck record or false suppression | Same DB transaction for fast local work, or lease/recovery state machine |
| Local mutation committed, completion record not committed | Timeout/`500` | Retry repeats local effect | Commit both in one transaction or use natural business unique constraint/reconciliation |
| External processor succeeded, local state update lost | Timeout/`500`; local state unknown | Double external effect on retry | Reuse downstream key, query by reference, persist intent, reconcile |
| Completion saved, response lost | Timeout | Client repeats | Replay saved/reconstructed result |
| Dedupe primary fails before replication | Timeout/failover | Promoted replica may not know key | Stronger persistence/consistency, business constraint, or explicitly tolerate risk |
| Retry arrives after key expiry | Looks new | Duplicate business effect | Longer retention and/or permanent natural transaction ID |
| Same key, different payload | Client bug/attack | Wrong result returned for new operation | Fingerprint comparison and deterministic `422`/documented conflict |
| Different keys, same real-world action | Double click generated new key; two devices | Key layer sees two valid intents | Business invariant/unique reference in addition to API key |

### 11.2 The hardest gap: local database plus external side effect

You generally cannot place your database commit and an unrelated bank/email provider commit in one ordinary ACID transaction.

```text
Local DB                         External bank
   |                                  |
   |--- mark attempt pending          |
   |--- request, downstream key K --->|
   |<-- timeout, outcome unknown --X  |
```

The correct next action is not necessarily “call charge again” or “mark failed.” It is often:

1. keep the attempt `PENDING`/`UNKNOWN`;
2. query the processor using the downstream reference;
3. retry using the same downstream idempotency key if the provider contract permits;
4. accept an authenticated webhook that updates the same operation state idempotently;
5. reconcile local ledger/operation records with processor reports.

### 11.3 Internal mutation pattern: one transaction

When the idempotency record and business rows share one transactional database, the simplest strong pattern is:

```text
BEGIN
  atomically claim (tenant, operation, key)
  validate same fingerprint
  INSERT order with UNIQUE(client_order_reference)
  UPDATE inventory with required invariant
  store outcome/resource_id in idempotency record
COMMIT
send response
```

If commit succeeds but the response is lost, the retry finds the committed result. If the process dies before commit, the whole transaction rolls back.

Keep this transaction short. Do not perform remote network calls inside it.

### 11.4 External workflow pattern: durable state machine

```text
RECEIVED
   |
   | durable claim + validated intent
   v
READY_TO_SUBMIT
   |
   | call provider with stable downstream key
   v
SUBMITTED / UNKNOWN  ---- query/webhook/reconcile ----+
   |                                                 |
   +-----------------------> SUCCEEDED or FAILED <---+
```

Transitions must be conditional so two workers cannot both advance the same state blindly. A terminal status should be immutable or follow an explicit legal transition model.

### 11.5 Queue consumers and message redelivery

At-least-once brokers may redeliver when a consumer performs the effect and crashes before acknowledging/committing its offset.

```text
read event E
write DB effect
crash before broker acknowledgement
read event E again
```

Common **idempotent consumer/inbox** pattern:

```text
BEGIN DB TRANSACTION
  INSERT event_id E INTO inbox ON CONFLICT DO NOTHING
  if newly inserted:
      apply local business mutation
COMMIT
acknowledge broker message
```

The inbox claim and local mutation share a transaction. If the external effect is an SMS or email provider call, the same external-side-effect gap still exists; propagate a provider key if supported or use a recoverable send state and accept/document the remaining duplicate-versus-loss trade-off.

Kafka's public documentation distinguishes exactly-once processing within Kafka transactions from writes to external systems that require coordination. “Kafka/Flink exactly once” is not by itself proof that a bank charge, email, or database outside that transaction happens once.

### 11.6 Outbox pattern for reliable event publication

Another dual-write problem is committing an order and publishing `OrderCreated`:

```text
write order succeeds
publish event fails
```

The **transactional outbox** stores the event in the same database transaction as the order. A relay publishes it later, possibly more than once. Consumers use an inbox/event ID or idempotent business mutation to tolerate repeats.

```text
BEGIN
  INSERT order
  INSERT outbox(event_id, OrderCreated payload)
COMMIT

relay -> publish at least once -> consumer inbox/dedupe
```

Outbox solves reliable local state + event publication. It does not automatically make every downstream real-world effect exactly once.

## 12. Idempotency versus other correctness mechanisms

### 12.1 Different keys can still violate a business invariant

If a double-click accidentally creates `K1` and `K2`, both are new at the key layer. Add a domain constraint when the business itself has a stable identity:

```text
UNIQUE(merchant_id, merchant_order_id)
UNIQUE(original_payment_id, refund_reference)
UNIQUE(subscription_id, billing_period)
```

Idempotency keys handle transport retries. Business uniqueness handles logically duplicated commands that arrive under different transport keys.

### 12.2 Idempotency does not guarantee ordering

```text
request A: PUT progress = 90 seconds
request B: PUT progress = 120 seconds

network reorder: B arrives, then stale A arrives
final progress = 90 seconds
```

Both requests can be individually idempotent and the final outcome still be wrong. Use a version/sequence number, compare-and-set, monotonic `max(progress)`, or a domain-specific ordering rule.

### 12.3 Idempotency does not prevent lost updates

Two editors read version 7 and submit different replacements. Repeating each replacement is idempotent, but last-write-wins discards one editor's change. Use `ETag`/`If-Match`, versions, merge logic, or serialized ownership.

### 12.4 Idempotency does not replace authentication or authorization

An attacker must not retrieve another user's stored response merely by guessing a key. Scope by authenticated principal, recheck authorization, use unguessable keys, and avoid leaking whether another tenant used an opaque value.

### 12.5 Idempotency does not replace rate limiting

One key can be replayed a million times, consuming database/cache/network capacity even if the business effect occurs once. Rate limiting, request coalescing, and backoff still matter. Conversely, allowing only one request per second does not prevent a retry ten seconds later from duplicating a charge.

### 12.6 Idempotency does not make compensation automatic

If a workflow legitimately reaches multiple services and fails halfway, a saga may compensate completed steps. Every forward and compensating command should itself be idempotent. A refund is not “the same key as charge”; it is a separate operation with its own identity and invariant.

## 13. A concrete implementation blueprint: create payment

### 13.1 API contract

```http
POST /v1/payments HTTP/1.1
Authorization: Bearer ...
Idempotency-Key: 018f2d1e-...
Content-Type: application/json

{
  "merchant_order_id": "shop-42-order-91",
  "amount_minor": 50000,
  "currency": "INR",
  "payment_method_token": "pm_..."
}
```

Assumptions:

- one key is scoped to merchant + `CreatePayment` operation;
- amount is stored in minor currency units to avoid floating-point ambiguity;
- same key with different semantic fields is rejected;
- the endpoint returns a payment operation ID;
- the payment processor accepts a downstream idempotency reference or status lookup;
- the API documents key retention and retry behavior.

### 13.2 Data model

```text
payment_attempt
  attempt_id                 primary key
  merchant_id
  idempotency_key
  request_fingerprint
  merchant_order_id
  amount_minor
  currency
  state                      READY | SUBMITTING | UNKNOWN |
                             SUCCEEDED | DECLINED
  provider_reference         nullable
  result_code
  lease_version              integer
  created_at, updated_at

UNIQUE(merchant_id, idempotency_key)       -- table is dedicated to CreatePayment
UNIQUE(merchant_id, merchant_order_id)     -- if the business permits one payment per order
```

### 13.3 Request-handling pseudocode

```text
handleCreatePayment(caller, key, request):
    validate authentication, authorization, key syntax, and request shape
    scope = (caller.merchant_id, "CreatePayment")
    fingerprint = canonicalSemanticFingerprint(scope, request)

    attempt, inserted = atomicInsertIfAbsent(scope, key, fingerprint, READY)

    if not inserted:
        if attempt.fingerprint != fingerprint:
            return 422 KEY_REUSED_WITH_DIFFERENT_REQUEST
        if attempt.state is terminal:
            return responseFor(attempt)
        return 202 with /payment-attempts/{attempt.id}

    enqueue/advance attempt by attempt_id
    return 202 with /payment-attempts/{attempt.id}
```

Worker outline:

```text
advancePayment(attempt_id):
    acquire state transition using version/lease
    call provider with downstream_key = stable(attempt_id)

    if provider gives terminal response:
        conditionally record SUCCEEDED or DECLINED
    else if timeout/ambiguous:
        record UNKNOWN
        query/reconcile using provider reference/downstream key
```

Important properties:

- the API servers do not keep correctness state in process memory;
- parallel same-key calls converge on one `payment_attempt`;
- different payload reuse is visible;
- external ambiguity remains recoverable instead of becoming a second charge;
- status lookup separates long payment processing from one HTTP connection;
- a domain order constraint can catch different-key duplicates.

### 13.4 If everything is in one local database

For a short order creation with no remote call, process synchronously in one transaction and return `201`. The asynchronous state machine is not mandatory; it is introduced here because real payments cross an external boundary.

## 14. Trade-offs and alternatives

| Technique | How it works | Strengths | Weaknesses / avoid when |
|---|---|---|---|
| Naturally idempotent state assignment | Set desired state, add to a mathematical set, remove identified item | Simple; little extra state | Does not handle order, external effects, increments, append, or different-key duplicates |
| Client-chosen ID + `PUT` | Same URI identifies same resource | Aligns with HTTP semantics; resource becomes dedupe record | Client ID policy/security; replacement semantics may not match an action |
| Idempotency record in domain SQL DB | Unique key; claim and local mutation in one transaction | Strong local correctness and durability | Adds writes/storage; hot keys; external calls still need workflow |
| Separate strongly consistent key-value store | Conditional put/read with TTL | Scalable low-latency claims; clear service boundary | Must coordinate with domain DB; cross-store dual-write problem |
| Redis record/lock | Atomic claim and fast replay | Very low latency and TTL support | Persistence, eviction, async failover, lease, and external-effect gaps; choose guarantees explicitly |
| Natural business unique constraint | Database enforces invoice/order/period identity | Durable; catches different transport keys | Only works when business identity/invariant is well defined; must return existing result cleanly |
| Payload-hash-only dedupe | Treat identical bodies as duplicates | No client integration | Cannot distinguish two legitimate identical actions; canonicalization/retention errors |
| Pessimistic/distributed lock | Serialize concurrent workers | Useful for short high-contention critical section | Does not remember outcomes; leases can expire; poor fit for slow remote work |
| Optimistic conditional update | Version/expected state controls transition | No long-held lock; detects conflict | Requires retry/merge; solves concurrency, not request identity alone |
| Inbox at consumer | Unique event ID + mutation in one DB transaction | Handles at-least-once redelivery for local effects | External effect still needs its own handling; storage/TTL |
| Transactional outbox | Commit local state and event record together | Prevents lost publication; relay can retry | Delivery is typically at least once; consumers still need idempotency |
| Status resource / operation intent | Create durable operation and poll/webhook | Excellent for long-running/ambiguous work | More API and lifecycle complexity |

### 14.1 When to use strong idempotency handling

Use it when duplicate effects are costly or users/SDKs must safely retry:

- money, orders, bookings, resource provisioning;
- external side effects;
- at-least-once workflows;
- mobile/offline clients;
- operations likely to exceed caller timeouts.

### 14.2 When a simpler approach is enough

- repeated `PUT` of a simple desired configuration already has safe semantics;
- inserting into a set with a unique member key naturally deduplicates;
- a best-effort analytics counter tolerates a documented small overcount;
- the action is cheap/reversible and retry ambiguity is not material.

Do not build a distributed lock service for a naturally idempotent `SET status = 'READ'` unless another invariant requires it.

## 15. Real-system-shaped examples

### 15.1 WhatsApp-like message sending — illustrative design

The mobile client can assign a stable `client_message_id` when the user presses Send and persist it with the pending message. Reconnect/resend uses the same ID.

```text
Phone outbox                 Messaging API                Message store
    | client_msg=M7               |                            |
    |--- send M7 ---------------->|--- insert UNIQUE --------->|
    |<--- ACK lost --------X       |    (chat, sender, M7)      |
    |--- resend M7 -------------->|--- find same message ----->|
    |<--- ACK server_msg=S99 -----|                            |
```

The unique tuple prevents duplicate server messages. Recipient devices can also deduplicate delivered `server_message_id`s because delivery can be retried.

Idempotency alone does not preserve conversation order. Message sequence/order rules are separate. Multi-device sending also needs a key namespace that avoids device collisions.

**Confidence:** architecture is illustrative, not a claim about WhatsApp's private implementation.

### 15.2 YouTube resumable upload — verified public protocol

YouTube's public resumable-upload API creates a unique upload-session URI, then uses `PUT` requests and byte ranges. After interruption, the client queries how many bytes the server has stored and resumes from the next byte instead of creating a new video/upload from scratch.

```text
Client -- POST start session --> YouTube API
Client <-- Location: upload_session_U

Client -- PUT U bytes 0..999999 --> API
network fails
Client -- PUT U status query ----> API
Client <-- 308, Range 0..999999 -- API
Client -- PUT U bytes 1000000.. --> API
```

The session ID and byte offset make retry intent explicit. The current public guide says a completed status query returns the completion response and documents session expiry and resumable `5xx` behavior: [YouTube Data API resumable uploads](https://developers.google.com/youtube/v3/guides/using_resumable_upload_protocol).

Lesson: idempotency may be modeled as a durable **operation resource plus progress state**, not only as a generic header.

### 15.3 Uber-like ride request — illustrative design

The rider app creates one stable key when the user requests a ride:

```text
POST /ride-requests
Idempotency-Key: rider7-request-attempt-91
{pickup, destination, product}
```

The ride service creates one `ride_request_id` and returns/replays it. Driver matching consumes a unique `RideRequested(ride_request_id)` event through an inbox so API retries do not launch multiple dispatch workflows.

Edge cases:

- a user intentionally requests a second ride: new intent/key;
- two devices create different keys: enforce any product rule about one active ride per rider separately;
- cancel and request are different operations and need ordering/state-transition rules;
- rate limiting multiple clicks is useful load protection but is not the one-ride correctness mechanism.

**Confidence:** illustrative design based on product behavior, not a claim about Uber internals.

### 15.4 Netflix-like playback progress — illustrative design

An absolute update is naturally repeatable:

```http
PUT /profiles/P1/videos/V9/progress

{"position_ms":120000,"session_id":"S7","sequence":18}
```

Sending sequence `18` again should not move progress twice. But request `17` can arrive after `18`; idempotency alone would allow a stale overwrite. Store the highest accepted sequence per session or apply a domain rule such as monotonic progress within a viewing session.

Seeking backward is legitimate, so blindly storing `max(position_ms)` is not universally correct. The sequence/session contract must express intent.

**Confidence:** illustrative design, not a claim about Netflix internals.

### 15.5 Twitter/X-like follow or like — illustrative design

Set membership is naturally idempotent:

```text
follow:   ensure (user_A, user_B) exists in Following set
unfollow: ensure (user_A, user_B) is absent
like:     ensure (user_A, post_X) exists in Likes set
unlike:   ensure (user_A, post_X) is absent
```

A unique constraint on the pair makes repeat “add” converge; deletion of the identified pair makes repeat “remove” converge. A separate event/notification ID is needed if the first follow sends a notification—otherwise repeating the set operation could still notify repeatedly.

**Confidence:** illustrative data model, independent of X's current public endpoint method or private storage.

### 15.6 Amazon/AWS resource provisioning — verified public pattern

AWS publicly describes client request identifiers such as `ClientToken` for idempotent operations including EC2 instance launch. A timeout after requesting an instance can be retried with the same token so the workflow identifies the same requested provisioning intent rather than launching another resource. The later response can be semantically equivalent without being identical because the instance state may have advanced.

```text
Controller -> RunInstances(ClientToken=K) -> EC2 workflow
response lost
Controller -> RunInstances(ClientToken=K) -> same logical launch/result
```

Source: [Amazon Builders' Library — Making retries safe with idempotent APIs](https://aws.amazon.com/builders-library/making-retries-safe-with-idempotent-APIs/).

This example also explains why “search for an identical resource after timeout” is weaker: another workflow could have created an apparently similar instance, and two legitimately identical requests may both be intended. A client token expresses intent directly.

## 16. Edge cases, security, and operational details

### 16.1 Same key, different API version or route

Scope by operation and version or keep stable semantic comparison rules. Otherwise `K` on `/payments` could collide with `K` on `/refunds`, or a schema upgrade could make the saved fingerprint ambiguous.

### 16.2 Key stored in the wrong place

Do not rely on:

- one browser tab's volatile memory when a page refresh should retry the same action;
- an API server's local map in a horizontally scaled fleet;
- a cache configured to evict correctness records silently;
- logs as the authoritative lookup path.

### 16.3 Key reuse after a terminal failure

If the earlier response is a terminal decline, same-key replay should normally return that result. If the user changes card/payment details, it is a new intent and should use a new key—unless the API explicitly models one payment intent with multiple confirm attempts as separate sub-operations.

### 16.4 Request body contains nondeterministic values

Client timestamps, rotating tokens, trace IDs, and field order can create false mismatches. Define semantic fields and canonicalization. Never ignore a changed amount, currency, destination, item list, or other effect-defining field.

### 16.5 Large or sensitive stored responses

Storing every raw response can be expensive and can duplicate personal/payment data. Alternatives:

- store status + resource ID and reconstruct an authorized response;
- encrypt sensitive fields;
- apply scoped retention/deletion;
- avoid storing authentication credentials or raw payment secrets;
- cap response size and key length.

### 16.6 Denial-of-service by unique keys

An attacker can send endless unique keys and fill the store. Protect with:

- authentication and authorization before expensive reservation where possible;
- rate limits and quotas;
- key-format/length bounds;
- TTL and storage quotas per tenant;
- capacity monitoring;
- rejecting extremely high-cardinality unauthenticated scopes.

### 16.7 Key guessing and cross-tenant leaks

Use high-entropy identifiers and scope every lookup to the authenticated caller. Do not return another tenant's saved response or reveal payload details on collision.

### 16.8 Cancellation races

`cancel(K2)` can race with `create(K1)`. Separate idempotency keys do not order them. Model legal states and conditional transitions:

```text
CREATING -> ACTIVE -> CANCELLING -> CANCELLED
```

Define what cancellation means before creation completes and ensure both commands can be retried.

### 16.9 Clock and TTL errors

Use the authoritative store's expiry/clock behavior where possible. Clock skew between API servers should not independently decide that a record expired. Never equate “lease expired” with “previous external action definitely stopped.”

### 16.10 Poisoned result

Saving a transient `500` forever can permanently poison a key; not saving a `500` after a committed effect can duplicate it. Classify failure by whether execution began and outcome is known. Provide an operator recovery path for stuck `UNKNOWN` records.

## 17. Observability and testing

### 17.1 Useful metrics

- new claims versus completed replays;
- in-flight same-key conflicts/waits;
- key/payload mismatch count;
- age and count of `PROCESSING`/`UNKNOWN` records;
- recovery/reconciliation success and failure;
- store latency, conditional-write conflicts, errors, evictions, and failovers;
- result-record retention/storage cardinality;
- downstream duplicate-reference responses;
- retries by status and client version;
- business duplicate invariant violations.

Use bounded labels in metrics. Raw idempotency keys have **high cardinality**—almost every value is different—and should normally be searchable logs/traces, not metric labels because they can make the metrics system expensive or unusable.

### 17.2 Logging and tracing

Correlate, with redaction:

```text
trace_id
authenticated tenant/account
operation name
hashed/redacted idempotency key
attempt/operation ID
request fingerprint version
state transition and owner/version
downstream reference
original versus replay flag
```

Do not log full payment credentials or sensitive response bodies.

### 17.3 Tests interviewers like to hear

1. Send the same key/payload sequentially 100 times; assert one business record/effect.
2. Send it concurrently through many API instances; assert one atomic winner.
3. Reuse the key with a changed amount; assert deterministic rejection.
4. Kill the worker after claim but before local mutation; assert recovery/progress.
5. Kill it after local commit but before response; assert replay, not duplicate.
6. Timeout after the external provider succeeds; assert status query/same downstream key.
7. Fail over the idempotency store; verify behavior matches the promised guarantee.
8. Retry after TTL expiry; verify/document what happens.
9. Use different keys for the same merchant order; verify the business constraint.
10. Deliver the same queue event after consumer crash; verify inbox + mutation atomicity.
11. Send out-of-order updates with distinct keys; verify version/sequence handling.
12. Flood unique keys; verify quota, TTL, and storage protection.

Fault injection at each boundary is more convincing than only a happy-path unit test.

### 18.1 How idempotency appears

It is rarely a standalone definition question at senior HLD depth. It appears when you design:

- payment gateway/wallet/refund;
- order, booking, or ride creation;

- webhook ingestion;

### 18.3 Answer spine: `R-E-P-L-A-Y`

Use this as reconstruction cues, not a memorized paragraph.

#### R — Requirements and risk

- What exact business effect must not repeat?
- How costly is a duplicate versus lost/delayed work?
- What is the retry/retention horizon?
- Is the work synchronous, asynchronous, or external?

#### E — Express one intent

- Natural idempotent state assignment, client-chosen resource ID, business reference, or client key?
- Scope `(caller, operation, key)`.
- Same logical retry reuses the key; new intent gets a new one.
- Compare semantic payload/fingerprint.

#### P — Persist an atomic claim

- Unique constraint/conditional put at one authoritative owner.
- One winner for parallel requests across all servers.
- State is more than “seen”: processing, succeeded, terminal failure, unknown.

#### L — Link the business effect

- Same DB: transactionally couple claim, mutation, and outcome.
- External service: propagate downstream key/reference, persist intent, query/reconcile.
- Queue: inbox + mutation transaction; outbox for reliable publication.

#### A — Answer duplicates consistently

- Completed: replay original or return semantically equivalent/latest result.
- In flight: wait, `409`, or `202` + status URI.
- Payload mismatch: reject.
- Map `400`/`409`/`422`; do not misuse `429`.

#### Y — Yield safely under scale and failure

- Shared/partitioned authority, not local caches.
- TTL/late retries, hot keys, failover, multi-region trade-off.
- Authentication, cardinality defense, metrics, fault-injection tests.

### 18.4 Example interview opening

> “First I want to identify the effect and failure boundary. I will assume `POST /payments` can time out after the processor acts, clients and gateways may retry, duplicate charges are unacceptable, and a delayed result is preferable to a second charge. The client creates one stable idempotency key per logical payment; the server scopes it by merchant and operation, stores a semantic fingerprint, and uses a unique conditional insert so parallel calls across API servers elect one durable payment attempt. Same-database mutations and the result record commit together. Because the bank is external, I persist an operation state and reuse a downstream reference rather than holding a lock across the call; ambiguous outcomes are queried and reconciled. Completed retries return the same payment, in-flight retries return `202` plus a status URI, mismatched payloads fail, and I will then cover retention, business uniqueness, failover, and multi-region routing.”

### 18.6 Follow-up answer cues

| Follow-up | Answer cues |
|---|---|
| `DELETE` then `404` | Intended target state remains absent; idempotency is effect, not identical response |
| Bad `PUT` | Method name is a contract; increment/action body violates replacement semantics |
| Idempotent `POST` | Endpoint-level key/business ID; `PUT` only when client-known target/replacement semantics fit |
| Client key | Client knows intent; body equality cannot distinguish legitimate identical actions; server still validates/enforces |
| Two identical real payments | New intent gets new key; never dedupe by payload alone |
| Parallel servers | Unique `(scope,key)` conditional insert at shared authority; one winner |
| Atomic `GET` + `SET` | Composition is not atomic; race between calls; use insert-if-absent/script/transaction as appropriate |
| Losing request | Wait boundedly, `409`, or `202`/operation status; never execute again |
| Lock insufficient | No durable result after release; crash/lease/external effect gaps |
| Lease expiry | Stale worker can resume; conditional state/fencing token/downstream idempotency |
| Bank succeeded, local unknown | Stable provider key/reference, query/webhook, `UNKNOWN` state, reconciliation |
| Local commit, reply lost | Retry reads committed result; mutation + record same transaction |
| Redis failover | Async replication can lose recent record; stronger store/business uniqueness or explicitly tolerate |
| Fast cache sync | Any propagation window races; misses/restarts/partitions/eviction; one atomic authority |
| Store response | Exact replay versus resource ID/latest view; storage, PII, staleness trade-off |
| Errors | Finalize terminal/committed outcomes; do not finalize known pre-execution transient failures; preserve unknown |
| Mismatch | Compare semantic fingerprint; reject, commonly `422` by current draft |
| TTL | Derived from retry/redelivery/risk; late replay can be new; durable business reference for longer guarantee |
| Different keys, same order | Natural business unique constraint in addition to transport key |
| Kafka exactly-once | Name the transaction boundary; external email/SMS/DB requires sink coordination/idempotent effect |
| Out-of-order PUT | Idempotency is not ordering; sequence/version/ETag/domain merge |
| Multi-region | Home owner or global consensus; latency/availability/consistency trade-off |
| `429` | Rate limiting; same-key retry after `Retry-After` if same intent |
| Storage attack | Auth, quotas/rate limit, TTL, length bounds, per-tenant capacity |

### 18.7 Common interview mistakes

- Defining idempotency as “same response every time.”
- Saying `DELETE` is not idempotent because it returns `404` later.
- Saying HTTP makes any implementation idempotent merely because the method is named `PUT`.
- Saying all `POST`s are non-idempotent and cannot be made retry-safe.
- Generating a new key for every retry.
- Deduplicating only by request payload.
- Looking in a cache, executing, and storing afterward.
- Using independent in-memory maps on each API server.
- Saying “Redis is fast” without atomicity, durability, eviction, failover, and crash-window reasoning.
- Treating a temporary lock as the completed-result record.
- Holding a database transaction/lock through a long bank call.
- Marking a timed-out external call `FAILED` and immediately calling again.
- Claiming universal exactly-once delivery/processing.
- Ignoring different-key duplicates and business invariants.
- Ignoring ordering/lost-update problems because each call is individually idempotent.
- Returning `429` for a duplicate rather than a rate limit.
- Having no TTL/expiry contract, or claiming a TTL gives a permanent guarantee.
- Ignoring authentication, response leakage, and unique-key storage attacks.

## 19. Compact revision sheet

### 19.1 Ten facts to retain

1. **Idempotent means repeated identical intent has the same intended effect as one execution; responses may differ.**
2. **Safe is read-only intent; idempotent can mutate once. `DELETE` is unsafe but idempotent.**
3. **HTTP defines safe methods plus `PUT` and `DELETE` as idempotent; `POST` and `PATCH` have no default guarantee.**
4. **A particular `POST` can be retry-safe using an idempotency key or business identity.**
5. **The client normally supplies a stable key because it knows intent; the server owns validation, scoping, atomic enforcement, result storage, and expiry.**
6. **Parallel duplicates require one atomic winner, not `check` then `set`.**
7. **Across many servers, use one authoritative/shared or correctly sharded state owner; fast local-cache synchronization is not enough.**
8. **Couple local mutation and outcome record in one transaction; use downstream keys, state machines, status queries, and reconciliation for external effects.**
9. **Idempotency does not solve different-key duplicates, ordering, lost updates, authorization, or rate limiting.**
10. **`429` means rate limiting. Current draft conventions use `400` missing key, `409` in progress, and `422` key/payload mismatch, but APIs must document their own contract.**

### 19.2 One-screen flow

```text
Request(scope, key, payload)
          |
          v
validate + semantic fingerprint
          |
          v
atomic INSERT IF ABSENT at authority
     | new                         | existing
     v                             v
execute/recover             fingerprint same?
     |                       | no -> reject mismatch
     |                       | yes
     v                       v
local DB? same tx       terminal -> replay/result
external? state +       in-flight -> wait/409/202
downstream key +
query/reconcile
     |
     v
store terminal result before response
```

### 19.3 Design checklist

- What exact effect is protected?
- What defines one logical intent?
- Who creates and persists the key?
- What is the key's caller/operation scope?
- Which payload fields are compared?
- Where is the atomic insert/unique constraint?
- What states exist beyond `seen`?
- What happens to a concurrent loser?
- Can the business mutation share a transaction with the record?
- How is an external call identified, queried, and reconciled?
- Are queue consumers and publishers using inbox/outbox where needed?
- Is a natural business unique constraint also required?
- What result/status is replayed?
- Which errors finalize the key?
- How long is retention, and what happens after expiry?
- What are cache, persistence, failover, sharding, and multi-region guarantees?
- How are authentication, abuse, storage, observability, and recovery handled?

> “Idempotency means repeating the same logical operation has the same intended effect as doing it once, even though the HTTP response may differ. I scope that key by caller and operation, compare the semantic payload, and use a unique atomic claim in a shared authoritative store so parallel requests across servers elect one operation.
