# Service Discovery (HLD)

> Beginner-first notes expanded from the supplied bullets: why hard-coded instance IPs fail, what a service registry does, how registration and discovery work, how health is determined, Eureka's in-memory data model and peer replication, latency and caching, stale endpoints, timing trade-offs, alternatives, failure cases, and interview follow-ups.
>
> The central idea is: **call a stable logical service name, discover the changing set of healthy instances behind that name, and then choose an instance.** Discovery finds candidates; load balancing chooses among them.

## 0. Scope, assumptions, corrections, and evidence labels

### Assumptions

- The focus is internal service-to-service discovery in a system with multiple instances per service.
- Netflix Eureka is the main concrete implementation because it appears in the rough bullets. The chapter also compares DNS, a load balancer, Kubernetes Services, Consul, managed cloud discovery, and a service mesh.
- "Client" can mean a service that calls another service. It does not necessarily mean an end-user's phone or browser.
- Product defaults and public architectures change. Eureka-specific behavior was checked against current Spring Cloud Netflix 4.0.4 documentation and Netflix's public Eureka documentation on **18 July 2026**.

### Corrections to the rough bullets

1. **Hard-coding one IP creates a single-instance dependency, but hard-coding by itself is not always a single point of failure.** A hard-coded list can contain several instances. Its real problem is that the list becomes stale and must be changed manually.
2. **Service discovery and load balancing are different responsibilities.** Discovery returns eligible endpoints; a client library, proxy, or load balancer selects one. Eureka does not proxy the business request through the registry.
3. **Eureka clients do not normally query the server for every business request.** They keep an in-memory registry and refresh it periodically. This avoids a registry network hop on the hot path.
4. **The Eureka client cache is not best described as a simple per-entry TTL cache.** Eureka periodically fetches registry deltas/full state. Server-side lease expiry, eviction scans, server response caches, peer replication, and client refresh intervals all contribute to staleness.
5. **A heartbeat is weak evidence of liveness, not proof of readiness or correctness.** A process may send heartbeats while its request threads, database dependency, or credentials are broken.
6. **Graceful deregistration handles planned shutdown only.** A crashed process cannot send a deregistration request, so lease expiry or an external controller/health checker is still required.
7. **The Eureka server's core registry is nested, not exactly one flat `Map<String, Lease<InstanceInfo>>`.** Conceptually and in the public source it is application name -> instance ID -> lease.
8. **`lastRenewalTimestamp` is server-side lease state.** It should not be casually listed as an `InstanceInfo` field. `InstanceInfo` holds endpoint identity and metadata; `Lease<InstanceInfo>` tracks renewal timing.
9. **A cluster removes the single-server failure, not every single point of failure.** Common configuration, a shared network, bad registrations, software bugs, and the bootstrap mechanism can still fail the entire discovery plane.
10. **Eureka peers replicate registry mutations but do not run a strongly consistent consensus protocol for every update.** Different peers and clients can temporarily show different views; the design favors availability and eventual convergence.

### Evidence labels used below

- **Verified implementation behavior:** supported by current first-party documentation or public source code.
- **Historical public production example:** the company publicly described this architecture at the linked date; it is not a claim about its private 2026 design.
- **Illustrative system design:** a reasonable use of the named product in an interview; not a claim about its real private implementation.
- **Candidate-reported interview experience:** a public self-report, not an official or independently verified company question bank.
- **Illustrative number:** learning arithmetic or a hypothetical target, not a production benchmark.

# Part I - Plain-language foundation

## 1. Start with the problem, not the product

Imagine an order service that calls an inventory service:

```text
Order Service --------------------> 10.20.1.17:8080
                                      Inventory instance
```

This works until the inventory process restarts on another machine:

```text
old instance: 10.20.1.17:8080  [gone]
new instance: 10.20.4.92:8080  [ready]
```

The order service still knows only the old address. Someone must edit configuration and restart/reload callers. Now suppose inventory scales to four instances:

```text
inventory-1  10.20.4.92:8080
inventory-2  10.20.8.31:8080
inventory-3  10.20.3.44:8080
inventory-4  10.20.7.10:8080
```

Every caller would have to know all four addresses, notice additions/removals, stop calling failed instances, and decide how to spread traffic. In a large microservice system this list changes continuously because of:

- horizontal scaling;
- rolling and blue/green deployments;
- process, VM, container, rack, or zone failures;
- rescheduling by an orchestrator;
- port allocation changes; and
- maintenance and traffic draining.

The caller should instead ask for a stable logical name:

```text
"inventory-service" -> currently usable endpoints
```

That translation from **service identity** to **current network locations** is service discovery.

## 2. The hotel analogy

Think of a hotel:

- You ask the front desk for "housekeeping," not for "employee 47 in room B-19."
- The employee on duty can change without changing the service name.
- The roster records which employees are on shift.
- A dispatcher chooses an available employee.
- If an employee goes off shift, the roster must be updated.

In a distributed system:

| Hotel concept | System concept |
|---|---|
| "Housekeeping" | Logical service name |
| Employee | Service instance |
| Roster | Service registry/catalog |
| Clock in/out | Register/deregister |
| "I am still on shift" | Heartbeat/lease renewal |
| Dispatcher chooses employee | Load balancing |

The analogy has a limit: a network cannot always tell whether an instance is dead or merely unreachable. That ambiguity drives many of the trade-offs in this chapter.

## 3. Essential vocabulary

### Service versus instance

- A **service** is a logical capability: `inventory-service`.
- An **instance** is one running copy: `inventory-7f88 at 10.20.4.92:8080`.

```text
inventory-service
  |-- instance A: 10.20.4.92:8080
  |-- instance B: 10.20.8.31:8080
  `-- instance C: 10.20.3.44:8080
```

Registration is normally at **instance level** under a service-level name. Registering only "inventory exists" would not tell callers where its individual copies are.

### Endpoint

An **endpoint** is the address needed to connect, usually hostname/IP + port + protocol. It may also include metadata such as zone, version, weight, or supported protocol.

### Service registry/catalog

A **service registry** stores the current mapping:

```text
service name -> set of instance records
```

It is a dynamic address book. Eureka Server, Consul's catalog, AWS Cloud Map, and Kubernetes EndpointSlices can play this role in different architectures.

### Registration

Registration tells the control plane that an instance exists and where it can receive traffic. Deregistration removes it during a graceful shutdown. Registration may be done by the instance itself or by the platform that created it.

### Discovery

Discovery asks, "Which instances currently serve this logical name?" Discovery may return a list of endpoints or expose a stable virtual address that hides the list.

### Lease and heartbeat

A **lease** is a time-limited registration. A heartbeat renews it. If renewals stop long enough, the registry may expire the record.

Do not confuse these terms:

- **heartbeat interval:** how often evidence is sent;
- **lease expiration:** how long the registry tolerates missing evidence;
- **registry refresh interval:** how often a consumer updates its local view;
- **cache TTL:** how long a cache entry may be reused in systems that use TTL semantics.

They affect one another but are not the same timer.

## 4. Required prerequisites, explained inline

### 4.1 IP address, port, hostname, and DNS

- An **IP address** identifies a network interface for routing packets.
- A **port** identifies a listening process/service at that address.
- A **hostname** is a stable name that can be resolved to address information.
- **DNS** is one possible discovery mechanism. It maps names to records such as IP addresses and can return several records.

DNS answers are cached by applications, operating systems, and recursive resolvers. Therefore, changing DNS does not mean every caller instantly sees the new answer. See [DNS](<./29. DNS.md>) for the full resolution path.

### 4.2 Horizontal scaling and load balancing

**Horizontal scaling** adds more instances rather than making one machine larger. Once several instances exist, traffic needs to be distributed among eligible instances. That is load balancing.

Service discovery answers:

```text
Which candidates exist and appear eligible?
```

Load balancing answers:

```text
Which eligible candidate should receive this call?
```

See [Load Balancers](<./17. Load Balancers.md>) for algorithms, health checking, and proxy layers.

### 4.3 Liveness, readiness, and health

- **Liveness:** is the process alive enough that restarting it might help?
- **Readiness:** should new traffic be sent to it now?
- **Health:** an overloaded umbrella term; always ask what is actually being tested.

A process can be alive but not ready while warming caches. It can be ready for `/status` yet unable to process checkout because its database pool is exhausted. Discovery should normally expose only instances eligible for the relevant traffic, which is closer to readiness than mere process liveness.

### 4.4 Control plane versus data plane

- The **control plane** maintains membership, health, configuration, and routing information.
- The **data plane** carries actual business calls and responses.

```text
Control plane:  Inventory instance ---> Eureka (register/heartbeat)
                Order client      <--- Eureka (registry refresh)

Data plane:     Order client      ---> Inventory instance (business call)
```

Eureka is normally in the control plane, not inline in every business call. This separation explains why cached discovery can keep serving traffic during a registry outage.

### 4.5 Eventual consistency

With multiple registry peers and client caches, an update reaches different observers at different times. **Eventual consistency** means that, if updates and failures stop and communication resumes, those views are expected to converge. It does not mean they are identical at every moment.

For discovery, temporary inconsistency can mean:

- one caller sees a new instance while another does not; or
- one caller still sees a failed instance after another has removed it.

The data plane must be designed to tolerate that.

# Part II - Why hard-coded instance addresses fail

## 5. Expanding and correcting each rough bullet

| Rough point | Precise explanation |
|---|---|
| Single point of failure | Hard-coding **one** instance makes every call depend on it. Hard-coding several removes that exact SPOF but still leaves a stale, manually maintained membership list. |
| No load balancing | One hard-coded address cannot spread load. A hard-coded list could use a local algorithm, but it still cannot automatically learn membership or health changes. |
| Tight coupling | Callers depend on physical deployment details: addresses, ports, sometimes zones and versions. Replacing or scaling the callee forces caller configuration changes. |
| Difficulty in testing | Addresses embedded in code make local, staging, failover, and fault-injection tests awkward. Configuration injection can solve this for a small static system; a registry is not mandatory merely for testability. |

Additional problems omitted from the bullets:

- **Autoscaling does not work cleanly.** New instances receive no traffic until every caller learns them.
- **Failures are not removed automatically.** Callers continue using dead endpoints.
- **Rolling deployments become risky.** Old and new versions cannot be added/drained dynamically.
- **Configuration fan-out grows.** A change to one service may require reloads across many dependent services.
- **Environment leakage becomes possible.** A staging caller can accidentally retain a production IP.
- **IPs are identities they were never meant to be.** An IP can be reassigned to a different process; a service name expresses intent more safely.
- **Topology cannot guide routing.** A caller cannot prefer same-zone instances unless it has current zone metadata.

## 6. When hard-coded/configured endpoints are still acceptable

Service discovery adds operational machinery. Do not use it automatically for every system.

Static configuration can be reasonable when:

- there is one stable database proxy or managed endpoint;
- the system has only a few rarely changing servers;
- failover is manual and its recovery objective permits that;
- an embedded/air-gapped installation has fixed topology; or
- a prototype does not yet justify a registry.

A useful evolution path is:

```text
hard-coded address
    -> environment/configured hostname
    -> DNS or stable load-balancer address
    -> dynamic registry/client or platform-native discovery
    -> service mesh when traffic policy/identity justify it
```

Use the simplest mechanism that satisfies update speed, availability, scale, security, and operational requirements.

# Part III - The complete discovery flow

## 7. Components and responsibilities

```text
                         CONTROL PLANE

     register/renew        replicate state       fetch/watch
Service instance ------> Registry cluster <------ Caller/Proxy
                              |
                        health/metadata

                          DATA PLANE

Caller/Proxy -- choose endpoint --> Service instance
             -- business request ->
```

| Component | Responsibility |
|---|---|
| Service instance/provider | Advertise identity and address; report or expose status; gracefully drain on shutdown. |
| Registry/catalog | Store membership and metadata; expire/update records; answer queries; distribute changes. |
| Discovery client/resolver | Convert logical name into a current candidate set; cache/watch updates. |
| Load balancer | Choose one candidate using round-robin, least requests, weights, zone, latency, or other policy. |
| Health mechanism | Decide whether an endpoint should remain eligible. It can be registry-, platform-, proxy-, or client-driven. |

## 8. Registration patterns

### 8.1 Self-registration

The service instance registers itself and renews its lease.

```text
Inventory instance ---> POST register ---> Eureka
Inventory instance ---> heartbeat ------> Eureka
Inventory instance ---> deregister -----> Eureka
```

**Advantages**

- Works without a sophisticated orchestrator.
- The instance knows its own application metadata and readiness lifecycle.
- Fits Eureka's client library model.

**Costs and risks**

- Every language/runtime needs compatible client behavior or a sidecar.
- Discovery concerns leak into application code/dependencies.
- A buggy or compromised service can advertise the wrong address/status.
- The instance might announce itself before it is ready or disappear without deregistering.

### 8.2 Third-party/platform registration

An orchestrator, controller, agent, or deployment platform observes instances and updates discovery.

```text
Scheduler creates Pod/Task
         |
         v
Controller updates EndpointSlice/Cloud Map
         |
         v
Clients/proxies learn endpoint
```

**Advantages**

- Application code can remain unaware of the registry.
- Lifecycle truth is centralized in the platform.
- Polyglot services behave consistently.
- Security policy can restrict registry writes to controllers.

**Costs and risks**

- The controller/platform becomes critical infrastructure.
- Platform health may not equal application readiness unless probes are accurate.
- External/non-platform workloads need integration.

Kubernetes commonly uses this pattern: controllers create and update EndpointSlices from Pods selected by a Service rather than requiring every application to speak a custom registry protocol.

## 9. Discovery and routing patterns

### 9.1 Client-side discovery - Eureka's common model

```text
                periodic registry fetch
Order Service <------------------------ Eureka
     |
     | local list: [I1, I2, I3]
     | choose I2
     `-------------------------------> Inventory I2
```

The calling process contains a discovery client and load balancer. It resolves `inventory-service`, chooses an instance, and calls it directly.

**Benefits**

- No central proxy hop on each business request.
- The caller can continue from a warm cache during a registry outage.
- Routing can use caller context, zone, latency, or passive failure observations.

**Costs**

- Every client stack needs correct discovery, balancing, retries, metrics, and security.
- Cached views differ and become stale.
- Library upgrades must be rolled out to many services.
- The caller sees internal topology.

### 9.2 Server-side discovery through a proxy/load balancer

```text
Caller ---> stable proxy/LB ---> chosen service instance
                 ^
                 |
          watches/queries registry
```

The caller knows one stable proxy address. The proxy owns discovery and load balancing.

**Benefits**

- Simple, language-agnostic callers.
- One place to deploy sophisticated routing and outlier detection.
- Internal endpoints are hidden from callers.

**Costs**

- Adds a network hop and proxy resources.
- A poorly designed proxy tier is a bottleneck or failure domain.
- Central routing configuration can have a large blast radius.

The proxy should normally watch/cache registry state rather than synchronously querying the registry for every request.

### 9.3 DNS-based discovery

```text
Caller -- DNS query: inventory.internal --> [10.1.1.4, 10.1.2.7]
Caller -----------------------------------> selected endpoint
```

**Benefits**

- Nearly every runtime already understands DNS.
- No application-specific registry library is required.
- SRV records can include ports and priorities where clients support them.

**Costs**

- Multiple cache layers make freshness hard to predict.
- Some clients cache longer than intended or use only one returned address.
- DNS usually exposes less health/routing metadata than a native API.
- Record ordering is not a complete load-balancing or failure-recovery strategy.

DNS is service discovery, but not every discovery requirement fits DNS.

### 9.4 Sidecar/service-mesh discovery

```text
Order app ---> local sidecar ===== network =====> inventory sidecar ---> app
                  ^                                   ^
                  `------ control-plane updates ------'
```

The application calls a local proxy or intercepted address. Proxies receive endpoint and policy updates from a mesh control plane.

**Benefits**

- Language-independent discovery and traffic policy.
- Consistent mTLS, metrics, retries, and outlier detection.
- Application code uses logical destinations.

**Costs**

- Extra proxy hop, CPU/memory, configuration, certificates, and operational complexity.
- Mesh-wide misconfiguration can affect many services.
- Retrying at several layers can accidentally multiply traffic.

See [Service Mesh](<./28. Service Mesh.md>) after understanding this chapter.

### 9.5 Stable virtual IP (VIP) or platform Service

A platform may give `inventory-service` a stable virtual IP. Callers use it while platform networking routes to current backends.

```text
Caller ---> inventory VIP ---> platform routing ---> ready Pod
```

This hides endpoint membership and resembles server-side discovery, even when packet forwarding is distributed across nodes rather than performed by one central proxy.

## 10. Side-by-side pattern comparison

| Pattern | Who sees endpoint list? | Per-call registry hop? | Main strength | Main cost |
|---|---|---:|---|---|
| Client-side registry | Caller library | No, normally cached | Direct calls and rich local routing | Client complexity and stale caches |
| Proxy/server-side | Proxy/LB | No, if proxy watches/caches | Simple polyglot clients | Proxy hop and fleet operations |
| DNS | DNS-aware caller/resolver | Usually no; DNS is cached | Universal interface | Limited metadata and cache uncertainty |
| Sidecar/mesh | Local proxy | No synchronous registry call normally | Central policy with local data plane | Mesh complexity/resources |
| Stable VIP | Platform/network layer | No | Very simple client contract | Platform dependence and less caller control |

# Part IV - Eureka from startup to request

## 11. Eureka roles

### Eureka Server

The server maintains an in-memory registry of application instances and exposes operations to:

- register an instance;
- renew its lease;
- cancel/deregister it;
- update status;
- fetch all applications, one application, or changes; and
- replicate mutations to peer Eureka servers.

Spring's current documentation states that the server has no backend store for this registry and that both servers and clients use in-memory state/caches.

### Eureka client as a provider

An application instance uses the client to register itself, publish metadata/status, send heartbeats, and deregister on graceful shutdown.

### Eureka client as a consumer

The same application can fetch the registry and resolve another service's logical ID. A client-side load balancer such as Spring Cloud LoadBalancer then selects an instance.

One application can be both:

```text
Order Service
  provider role: registers ORDER-SERVICE so others can find it
  consumer role: discovers INVENTORY-SERVICE and PAYMENT-SERVICE
```

## 12. A normal lifecycle

```text
1. START
   Inventory instance starts and initializes dependencies.

2. REGISTER / BECOME READY
   It advertises instance ID, app name, host/IP, port, status, URLs,
   zone/version/other metadata, and lease settings.

3. REPLICATE
   The receiving Eureka server stores the lease and replicates the
   registration to its peers.

4. DISCOVER
   Order-service clients periodically fetch a full registry or deltas
   and cache the eligible inventory instances locally.

5. LOAD BALANCE
   For a business call, the local load balancer selects one cached
   inventory endpoint and connects directly to it.

6. RENEW
   Inventory periodically heartbeats to renew its lease.

7a. GRACEFUL STOP
   It is taken out of traffic/drained, then sends cancel/deregister.

7b. CRASH
   No cancel is possible; missing renewals eventually cause eviction,
   unless safety/self-preservation policy intentionally retains it.
```

## 13. Eureka's verified default timings - examples, not universal laws

Current Spring Cloud Netflix configuration documentation lists these defaults:

| Setting | Documented default | Meaning |
|---|---:|---|
| Lease renewal / heartbeat interval | 30 seconds | How often the instance renews |
| Lease expiration duration | 90 seconds | Missing-heartbeat duration before the instance becomes eligible for removal |
| Client registry fetch interval | 30 seconds | How often a discovery client refreshes registry information |
| Server response-cache update interval | 30 seconds | One server-side source of propagation delay |
| Self-preservation | enabled | Avoid aggressive eviction during a widespread renewal drop |

These are **verified product defaults as documented on 18 July 2026**, not recommended values for every workload and not a promise of exact detection time. Netflix's public Eureka communication guide also warns that caching and periodic fetches can make a change take up to roughly two minutes to reach all clients in its described configuration. Treat that number as **historical/configuration-dependent**, not a universal SLA.

## 14. Registration is not instant global visibility

A new instance can be registered on one server but not yet present in:

- another peer's in-memory view;
- the server's response cache;
- a client's next delta/full fetch; or
- a local load balancer's filtered list.

Therefore:

```text
"POST /register succeeded" != "every caller can route to it now"
```

Similarly, deregistration acknowledgement does not prove that every caller has forgotten the instance. Deployment draining must account for propagation and in-flight requests.

# Part V - How does the server know whether an instance is up?

## 15. Four complementary signals

### 15.1 Graceful status change and deregistration

On planned shutdown, an instance can mark itself `OUT_OF_SERVICE` or otherwise not ready, stop receiving new work, drain in-flight work, deregister, and exit.

A safer conceptual sequence is:

```text
mark not ready / out of service
           |
           v
allow discovery/routing change to propagate
           |
           v
stop new work and drain existing work
           |
           v
deregister and terminate
```

The exact ordering and drain delay depend on the platform. The invariant is: **do not terminate while discovery still intentionally sends new traffic, and do not wait forever for perfect global propagation.**

Deregistration cannot handle power loss, process kill, kernel failure, or a network partition.

### 15.2 Heartbeat and lease expiry

The instance periodically renews a lease:

```text
heartbeat received -> lastRenewalTimestamp moves forward
heartbeat missing  -> lease ages
lease too old      -> eligible for eviction
```

This handles silent crashes, but a missing heartbeat is ambiguous:

```text
Possibility A: instance died
Possibility B: registry is unreachable from the instance
Possibility C: registry received traffic slowly
Possibility D: long GC/process pause delayed the heartbeat
Possibility E: network partition separated healthy components
```

No timeout can perfectly distinguish all of these. A short timeout removes real failures quickly but creates more false removals. A long timeout tolerates pauses but routes to dead instances longer.

### 15.3 Application readiness/health status

Spring's Eureka integration documents that, by default, a heartbeat is used and application health is not automatically propagated; after registration an application can remain announced as `UP`. Enabling or implementing a health handler lets application status influence Eureka.

Health design requires care:

- A shallow check may say `UP` while important functionality is broken.
- A deep check that fails whenever a shared database blips may simultaneously remove every service instance, creating a larger outage.
- Different endpoints may have different readiness. An instance could serve reads while writes are unavailable.
- Health endpoints must be cheap, bounded, authenticated where necessary, and not leak sensitive information.

Use readiness to answer **"should I receive new traffic?"**, not **"is every dependency in the universe perfect?"**

### 15.4 Passive data-plane health

Clients and proxies see real request outcomes:

```text
connection refused
timeout
repeated 5xx
very high latency
```

They can temporarily eject an endpoint before the registry removes it. This is called passive health checking or outlier detection.

It reduces the damage from stale discovery but needs guardrails:

- use short, explicit connection and request timeouts;
- eject after evidence rather than one random error;
- periodically probe/reintroduce the instance;
- avoid every client making permanently different decisions;
- retry only within a deadline and budget; and
- retry writes only when idempotency/operation semantics make it safe.

## 16. Eureka self-preservation

Suppose a network failure blocks heartbeats from a large fraction of otherwise healthy services. Blindly expiring every missing lease would delete most of the registry and amplify the partition into a broad outage.

Eureka monitors the received renewal rate against an expected threshold. Its self-preservation behavior can stop or limit mass eviction when renewals fall unexpectedly low. Current Spring configuration documents self-preservation as enabled by default and exposes a renewal threshold setting.

```text
Large renewal drop
       |
       +-- assumption 1: all those services died -> evict quickly
       |
       `-- assumption 2: control-plane/network problem -> retain entries
```

Eureka deliberately leans toward assumption 2 for availability. The price is stale endpoints. Clients must therefore use timeouts, failure-aware balancing, and bounded retries.

**Interview insight:** discovery health is a trade-off between two errors:

| Error | Meaning | Consequence |
|---|---|---|
| False positive | Dead/unready endpoint is advertised healthy | Failed or slow calls until ejection/refresh |
| False negative | Healthy endpoint is removed | Lost capacity; possible overload of remaining instances |

There is no timer that eliminates both.

# Part VI - How Eureka stores the mapping

## 17. Conceptual data model

At the HLD level, the registry needs a structure like:

```text
ServiceName -> InstanceId -> Lease(InstanceInfo)
```

Example:

```text
INVENTORY-SERVICE
  |-- inventory-a -> Lease(10.20.4.92:8080, UP, zone-a, ...)
  `-- inventory-b -> Lease(10.20.8.31:8080, UP, zone-b, ...)

PAYMENT-SERVICE
  `-- payment-a   -> Lease(10.30.5.11:8443, UP, zone-a, ...)
```

The public Eureka source represents the core registry as a concurrent nested map equivalent to:

```java
ConcurrentHashMap<String, Map<String, Lease<InstanceInfo>>> registry;
```

- Outer key: application/service name.
- Inner key: instance ID, unique within that application.
- Inner value: a server-side lease whose holder is `InstanceInfo`.

So a flat conceptual key such as `appName/instanceId` is reasonable in a diagram, but it is not the exact core structure shown in Eureka's public source.

## 18. What belongs in `InstanceInfo`?

Representative endpoint identity and metadata include:

- instance ID;
- application/service name;
- hostname and IP address;
- secure and/or non-secure port;
- current and overridden status;
- home, status, and health-check URLs;
- virtual host/service identifiers;
- data-center/zone-related information;
- arbitrary metadata such as version, weight, or capabilities; and
- timestamps/version markers used to reconcile updates.

Do not let arbitrary metadata become an ungoverned second configuration database. Define allowed keys, sizes, ownership, and compatibility semantics.

## 19. What belongs in `Lease<InstanceInfo>`?

The lease wraps the instance record and tracks lifecycle timing such as:

- the `InstanceInfo` holder;
- registration time;
- last renewal time;
- lease duration;
- service-up time; and
- eviction/cancellation timing.

This corrects the supplied bullet: **last renewed timestamp is lease state, not merely an `InstanceInfo` attribute.** `InstanceInfo` may also carry advertised lease configuration and its own update timestamps, but those should not be confused with the registry's authoritative observation of the latest heartbeat.

## 20. Why use an in-memory map?

Discovery reads are frequent, records are relatively small, and membership is reconstructible from live instances and peers. In-memory state gives low-latency lookup and avoids placing a database on every registry read.

The trade-offs are:

- a restarted standalone server loses its local authoritative view;
- peers, re-registration, and client caches are required for recovery;
- memory and payload size grow with services x instances x metadata;
- large full-registry fetches can become expensive; and
- replication and caches create eventual consistency.

Eureka uses delta fetches to reduce transfer when the change rate is much lower than total registry size. Clients reconcile delta results and can fall back to a full fetch if their view does not match.

## 21. Concurrency and update identity

Many instances register and renew concurrently, so a real registry needs:

- thread-safe maps or a serialized state machine;
- unique instance IDs;
- idempotent register/renew/cancel behavior where possible;
- version/last-dirty information so an old replicated update does not overwrite a newer one;
- bounded metadata and request size; and
- a change log/revision for efficient watches or deltas.

Never use only `IP:port` as a permanent identity. Addresses can be reused. An instance incarnation/generation ID helps prevent a delayed heartbeat from an old process reviving a record now owned by a replacement.

# Part VII - What if the Eureka server goes down?

## 22. One server

With a single server:

- new instances cannot reliably register;
- instances cannot renew leases;
- cold/new clients cannot fetch a registry;
- existing warm clients can temporarily use their cached endpoint lists; and
- clients do not learn deployments, scaling changes, or newly failed instances.

Business traffic may continue because the server is not normally on the request path, but "cached operation" has limits:

```text
Warm caller + still-live cached endpoint -> may keep working
Warm caller + all cached endpoints dead  -> fails
Cold caller + no registry                -> cannot discover
New healthy provider + no registry       -> receives no discovered traffic
```

Therefore a standalone server is still a control-plane single point of failure even if the data plane degrades gracefully for a while.

## 23. Peer-aware Eureka cluster

Production Eureka can run several peer-aware servers, ideally across failure domains:

```text
                 zone A             zone B             zone C
               +---------+        +---------+        +---------+
services ----> | Eureka A| <----> | Eureka B| <----> | Eureka C|
clients  <---- | registry| <----> | registry| <----> | registry|
               +---------+        +---------+        +---------+
```

Current Spring documentation says every Eureka server is also a Eureka client by default. Peer service URLs let the servers register with/locate one another, and registry state is synchronized/replicated among connected peers.

The rough bullet is directionally right, with an important clarification:

- peers do not magically discover one another from an empty system;
- their seed URLs/DNS/configuration solve the **bootstrap problem**; and
- application registration/renewal/cancel/status mutations are then replicated among peers.

Clients should know multiple registry endpoints or a highly available stable method to locate them, often preferring a same-zone server and failing over to another zone.

## 24. Replication is not the same as consensus

Eureka peer replication is designed for availability and convergence. It does not make every update a linearizable, quorum-committed transaction before acknowledging it.

During a partition:

```text
zone A clients + Eureka A     X     Eureka B + zone B clients

View A may continue changing        View B may continue changing
```

Both sides can continue serving discovery information, so their views can diverge. After connectivity returns, replication, renewals, re-registration, timestamps, and later fetches help the system converge.

This is often a sensible discovery trade-off: refusing all discovery because a quorum is unavailable could stop healthy same-side service calls. But stale/wrong membership must be tolerated by the data plane.

## 25. Does clustering eliminate the SPOF?

It eliminates **one Eureka process** as the SPOF when deployed and addressed correctly. It does not eliminate:

- all peers in one zone/rack/account;
- bad shared configuration or certificate expiry;
- a software bug deployed to every peer;
- overload from synchronized full-registry fetches;
- poisoning by unauthorized registrations;
- a network partition between providers, clients, and peers;
- dependency on DNS/configuration used to locate the registry; or
- correlated operator error.

High availability requires independent failure domains, tested failover, capacity headroom, safe rollout, and a recovery path, not merely `replicas: 3`.

## 26. What if all registry servers fail?

The desired degradation policy is:

1. Warm clients retain last-known-good endpoints rather than immediately clearing them.
2. Local/proxy outlier detection stops repeatedly calling endpoints that actually fail.
3. Business calls use deadlines, circuit breakers, and bounded retries.
4. Cold-start behavior is explicit: fail startup, start unready, or use a narrowly scoped backup registry/static seed.
5. When the registry returns, clients refresh with jitter so they do not create a recovery storm.

Do not claim the system is fully available merely because clients cache. Cache survival and endpoint survival are different.

# Part VIII - Latency, caching, and stale endpoints

## 27. Does discovery add an extra hop?

### Client-side Eureka

Usually **no extra network hop per business request**:

```text
Background/control plane:
Caller client ---- periodic fetch ----> Eureka

Per request/data plane:
Caller -- local lookup + selection ---> Service instance
```

There is a small local library lookup/selection cost, plus background network and CPU cost for registry refresh. Startup/cold-cache discovery does call the server.

### Server-side discovery

Yes, the proxy/load balancer is an intentional data-plane hop:

```text
Caller -> proxy -> service
```

But the registry itself still need not be queried on each request because the proxy can watch/cache endpoints.

### DNS

A DNS lookup may occur before connecting, but several layers cache answers, and one connection can carry many requests. DNS is not automatically queried once per HTTP request.

### Service mesh

One or two local/remote proxy traversals are added, but endpoint discovery is pushed/cached in proxies rather than synchronously fetched for every call.

## 28. Why local caching matters

Local caching provides:

- fast resolution;
- low registry read load;
- fewer network dependencies on the request path; and
- temporary operation during registry failure.

It also creates:

- delayed new-instance adoption;
- delayed dead-instance removal;
- different callers seeing different memberships; and
- memory/payload cost in every client.

This is not a cache bug. It is an explicit availability-versus-freshness trade-off.

## 29. A stale-instance failure timeline

```text
t0      Inventory I2 sends its last heartbeat
t1      I2 crashes
t2      Some caller still has I2 in local cache and selects it
t3      Call times out/refuses; caller may eject I2 locally
t4      Server decides I2's lease is expired and eviction is allowed
t5      Peer/server response views are updated
t6      Caller performs its next registry refresh and removes I2
```

The stale window is not just one TTL. It may include:

```text
failure-detection time
+ eviction scheduling delay
+ peer replication delay
+ server response-cache delay
+ client refresh delay
+ DNS/proxy cache delay, if present
+ self-preservation behavior
```

These components overlap, so blindly summing maximums is often too pessimistic. The correct production answer is to measure **failure-to-no-more-traffic time** at chosen percentiles.

## 30. Timing knobs and trade-offs

| Knob | Lower/shorter gives | Lower/shorter costs | Higher/longer gives | Higher/longer costs |
|---|---|---|---|---|
| Heartbeat interval | Faster evidence and potentially detection | More writes/network; synchronized bursts | Less control-plane load | Coarser evidence |
| Lease expiration | Faster removal | False eviction on pauses/partitions | More tolerance | Dead endpoint retained longer |
| Registry refresh | Fresher clients | More server/network/CPU load | Better scalability/outage insulation | Staler membership |
| Server response-cache refresh | Fresher responses | More serialization/cache churn | Cheaper reads | More propagation delay |
| DNS TTL | Faster record change visibility | More DNS queries | Lower DNS load | Staler DNS answers |
| Passive ejection threshold | Quicker local protection | Eject healthy nodes on transient errors | Fewer false ejections | More repeated failures |

The supplied statement "TTL should be set precisely" needs refinement:

> There is no single perfect TTL. Choose coordinated detection, expiration, refresh, and retry budgets from the service's failure-recovery objective and expected pauses/network loss; load test the control plane; then measure stale-routing behavior.

For Eureka specifically, current documentation says lease expiration must be greater than the renewal interval and warns that making renewal intervals shorter than the established defaults can conflict with server calculations. Change coupled defaults only with an understood model and tests.

## 31. Defenses against stale discovery

- Connection and request timeouts.
- Passive outlier ejection.
- Circuit breakers per destination/instance.
- Bounded retries against a **different** instance when safe.
- Overall deadlines so each retry has a shrinking budget.
- Idempotency keys or idempotent operations for retried writes.
- Graceful readiness transitions and draining during deployment.
- Push/watch updates where faster propagation is required.
- Same-zone preference with cross-zone fallback.
- Reconciliation/full fetch after missed deltas.
- Jittered refresh and heartbeat schedules to prevent thundering herds.

Do not solve a stale endpoint by retrying indefinitely. That converts one failed instance into caller thread exhaustion and a retry storm.

# Part IX - Service discovery versus neighboring components

## 32. Discovery versus load balancer

```text
Discovery:     INVENTORY -> [I1, I2, I3]
Load balancing: choose I2 for this request
Routing/proxying: send the request to I2
```

They can live in one product/library but remain separate decisions.

| Question | Discovery | Load balancing |
|---|---|---|
| What exists? | Primary concern | Consumes the answer |
| What is eligible/healthy? | Stores/exposes signals | Filters or adds local signals |
| Which target gets this call? | Usually not the final decision | Primary concern |
| Does it carry business traffic? | Registry usually does not | Proxy LB does; client LB does not proxy |

## 33. Discovery versus API gateway

An API gateway is an application-aware entry point that authenticates, rate-limits, transforms, and routes API traffic. Service discovery supplies dynamic backend destinations.

```text
External client -> API Gateway -> internal service
                       |
                       `-- uses discovery to learn service endpoints
```

The gateway may consume the registry, but it is not the registry. See [API Gateway, Service Discovery, Regions and DNS](<./27. API Gateway, Service Discovery, Regions and DNS.md>).

## 34. Discovery versus DNS

DNS **can implement discovery**. A richer registry API may additionally support:

- arbitrary metadata and tags;
- readiness/health status;
- zones and weights;
- watches/delta streams;
- authenticated registration APIs; and
- application-specific filtering.

DNS is often the interface placed in front of such a registry because it gives broad compatibility.

## 35. Discovery versus service mesh

Service discovery answers where destinations are. A service mesh usually adds workload identity, mTLS, authorization, telemetry, retries, circuit breaking, traffic splitting, and proxy-based routing. A mesh therefore **uses/includes discovery** but is a larger system.

## 36. Discovery versus configuration management

Both distribute changing data, but their semantics differ:

- service discovery data is high-churn instance membership and routing metadata;
- configuration data is application behavior/policy and often changes less frequently.

Do not casually combine them into one unbounded key-value store. Their consistency, security, retention, payload, and blast-radius requirements differ.

# Part X - Designing a service discovery system in an interview

## 37. Clarify requirements first

### Functional requirements

- Register and update a service instance.
- Gracefully deregister or mark it out of service.
- Renew a time-limited lease.
- Resolve a service name to eligible endpoints.
- Return/filter metadata such as zone, version, protocol, and weight.
- Detect or accept health/readiness changes.
- Notify consumers through polling, delta fetch, long poll, or watch stream.
- Support administrative drain/override and audit it.

### Non-functional requirements

- High read availability and low lookup latency.
- Bounded membership propagation time.
- Survive instance and registry-node failures.
- Scale to the expected services, instances, clients, and change rate.
- Prevent unauthorized registration and discovery poisoning.
- Remain useful during network partitions and control-plane outages.
- Provide observable, recoverable operations.

### Questions to ask the interviewer

- Is discovery internal, external, or both?
- Are clients Java-only, polyglot, or legacy?
- Is there already Kubernetes/ECS/Nomad or must we build on VMs?
- Is a stale endpoint acceptable for seconds? For a minute?
- Should discovery remain available without a quorum?
- Do clients connect directly or through a proxy?
- How many services, instances, clients, regions, zones, registrations/sec, renewals/sec, and lookups/sec?
- Are cross-region calls normal, fallback-only, or forbidden?
- What security/tenant boundaries exist?

## 38. API sketch

```http
PUT /v1/services/{service}/instances/{instanceId}
{
  "host": "10.20.4.92",
  "port": 8080,
  "protocol": "https",
  "status": "READY",
  "zone": "az-a",
  "version": "2026.07.18",
  "metadata": {"weight": "100"},
  "incarnation": "8f30..."
}

POST /v1/services/{service}/instances/{instanceId}:renew
If-Match-Incarnation: 8f30...

POST /v1/services/{service}/instances/{instanceId}:drain

DELETE /v1/services/{service}/instances/{instanceId}

GET /v1/services/{service}?status=READY&zone=az-a

GET /v1/changes?afterRevision=918273       # poll/long-poll
WATCH /v1/services/{service}               # conceptual stream
```

Desired semantics:

- Register/update is idempotent for `(service, instanceId, incarnation)`.
- A new incarnation prevents an old delayed heartbeat from renewing a replacement's lease.
- Responses carry a monotonically ordered revision within a defined scope.
- Watches resume from a revision and fall back to a snapshot if the history has expired.
- Payloads and metadata are bounded.

## 39. Data structures

```text
services: serviceName -> map(instanceId -> InstanceLease)

InstanceLease:
  endpoint
  protocol/ports
  status
  zone/region/version/weight/tags
  incarnation
  registeredAt
  lastRenewedAt
  leaseExpiresAt
  revision

expiry index:
  min-heap/timing-wheel buckets by leaseExpiresAt

change log:
  revision -> REGISTER | UPDATE | RENEW? | DEREGISTER | EXPIRE
```

Do not necessarily publish every heartbeat to every watcher. Heartbeats can update lease state internally while membership/status changes generate the externally relevant change stream. Otherwise, renewal volume overwhelms consumers.

## 40. Back-of-the-envelope control-plane load

Suppose, purely as an **illustrative number**, there are 100,000 instances and each heartbeats every 30 seconds:

```text
renewals/sec = 100,000 / 30 ~= 3,333 average
```

Add failover bursts, retries, registrations, and reads. If 50,000 clients fetch one whole registry every 30 seconds, the payload/serialization cost can dominate. Common mitigations are:

- fetch only relevant services;
- deltas or watches rather than full snapshots;
- server response caches;
- compression;
- local agents/sidecars that fan out updates;
- jittered schedules;
- rate limits and backpressure; and
- partitioning by region/namespace/tenant when isolation permits it.

The arithmetic is a design prompt, not a claim about Eureka capacity.

## 41. Read distribution: snapshot plus changes

A robust pattern is:

```text
1. Client fetches snapshot at revision R.
2. Client subscribes to changes after R.
3. Server emits R+1, R+2, ...
4. If client falls too far behind/history is compacted, fetch a new snapshot.
```

This gives fast updates without requiring an unbounded change history. The watch path needs:

- per-client buffers with bounds;
- slow-consumer eviction/reconnect;
- heartbeats on the watch connection;
- resume tokens/revisions;
- duplicate-tolerant application; and
- a recovery path from missed events.

Eureka's full registry plus delta fetch/reconciliation is a polling-oriented version of this general idea.

## 42. Replication choices

### Availability-oriented peer replication

```text
write accepted locally -> async/best-effort replicate -> eventual convergence
```

Good when stale membership is preferable to discovery unavailability. Requires stale-endpoint tolerance and conflict/reconciliation rules. Eureka is closer to this side.

### Consensus-backed catalog

```text
leader proposes update -> quorum commits ordered log -> state changes
```

Good when authoritative ordered membership/configuration is important. A minority partition cannot accept writes, so write availability is lower during quorum loss. Consul documents Raft consensus and persistent catalog state.

### Platform-authoritative derived registry

An orchestrator's desired/observed state is the source. Controllers derive endpoint objects. Kubernetes EndpointSlices are an example.

There is no universally correct choice. State the consistency boundary and failure behavior, not just "use CAP."

## 43. Partitioning and multi-region design

Most service traffic should stay local to reduce latency, cost, and correlated dependency:

```text
Global logical service
  |-- region IN: registry + endpoints in region IN
  |-- region EU: registry + endpoints in region EU
  `-- region US: registry + endpoints in region US
```

Typical policy:

1. Discover same-zone healthy endpoints.
2. Fall back to another zone in the same region.
3. Cross region only if the service/data semantics permit it.

Cross-region discovery needs explicit answers for:

- whether data is present/consistent in the remote region;
- whether credentials and policy allow the call;
- latency and egress budget;
- residency requirements;
- failback after recovery; and
- how much remote registry state is replicated.

Do not add a cross-region endpoint simply because it is alive. A payment instance with no access to the user's regional ledger may be technically reachable and functionally wrong.

## 44. Bootstrap: who discovers the discovery service?

Every discovery system has a small stable root:

- configured seed addresses;
- DNS records;
- cloud metadata/API;
- an orchestrator API address; or
- a local agent with known socket/port.

```text
config/DNS seed -> locate registry cluster -> discover everything else
```

This is not circular if the bootstrap layer is simpler and more stable than general service membership. Keep it small, redundant, and independently testable.

## 45. Security design

A malicious registry write can redirect trusted service traffic to an attacker. Treat discovery as a security-critical control plane.

- Authenticate service and registry identities, preferably with workload identity/mTLS where appropriate.
- Authorize which principal may register which service name and namespace.
- Validate advertised addresses, ports, protocols, zones, and metadata.
- Prevent one tenant/team from overwriting another's instance IDs.
- Encrypt registration, renewal, replication, and discovery traffic as required.
- Audit register, status override, drain, and delete operations.
- Rate-limit and bound payloads to prevent memory/serialization abuse.
- Use short-lived credentials with safe rotation; monitor expiry.
- Restrict registry administration separately from ordinary instance renewal.
- Never assume that "returned by discovery" means "authorized for this request." Services/proxies still enforce authentication and authorization.

Spring's Eureka documentation supports basic authentication and TLS/client certificates, but the exact security architecture must be designed, not assumed from enabling a starter.

## 46. Observability and operational signals

### Registry/control plane

- register, renew, cancel, expire, and status-change rates;
- renewal success rate versus expected rate;
- registry size by service/zone/status;
- peer replication queue, failures, and lag;
- full-fetch versus delta-fetch rate and payload size;
- response latency/error rate and cache hit/age;
- self-preservation state and duration;
- watch clients, reconnects, slow-consumer drops;
- memory, GC pauses, CPU, threads, and network;
- authentication/authorization failures; and
- revision/registry divergence across peers.

### Client/data plane

- age and source of the local registry snapshot;
- number of eligible endpoints per service/zone;
- resolution failures and empty sets;
- per-instance connection failures, timeouts, 5xx, and latency;
- outlier ejections/reintroductions;
- retry rate and retry amplification;
- cross-zone/region fallback traffic; and
- failure-to-no-more-traffic time during drills.

An alert without a runbook is incomplete. Operators need to know whether to restore peers, disable a bad health rule, drain a poisoned registration, preserve cached state, or reconcile the registry.

# Part XI - Failure cases and edge cases

## 47. Failure matrix

| Scenario | What can happen | Design response |
|---|---|---|
| Instance crashes without deregistering | Stale endpoint remains until detection/refresh | Lease expiry + passive ejection + timeouts |
| Instance sends heartbeat but request path is broken | Registry may still show `UP` | Readiness/health integration + real traffic metrics/outlier detection |
| Registry node dies | Some clients fail registry operations | Multiple peers/endpoints; warm caches; failover |
| All registry nodes unavailable | Warm clients have old membership; cold clients fail | Last-known-good cache, explicit cold-start policy, restore quickly |
| Network partition lowers all heartbeats | Mass false eviction is possible | Self-preservation/thresholds; same-side availability; reconciliation |
| Peer views diverge | Different clients route differently | Version/timestamp rules, renew/re-register, reconciliation, tolerant data plane |
| Bad instance registers another service's name | Traffic hijack or wrong routing | Strong identity, per-service ACLs, validation, audit |
| New instance announces before warm-up | Requests fail during startup | STARTING/not-ready state; publish only after readiness |
| Old instance deregisters after replacement reuses ID | Replacement can be deleted | Unique incarnation/generation and conditional operations |
| Client misses delta history | Its map becomes incomplete | Hash/revision validation and full snapshot fallback |
| Every client refreshes simultaneously | Registry thundering herd | Jitter, server cache, compression, backoff |
| Health check depends on shared DB | All instances removed during DB blip | Separate readiness semantics; degrade carefully; avoid cascading health |
| Zone has no local healthy instances | Empty local pool | Defined cross-zone fallback and capacity headroom |
| Graceful shutdown is too fast | Cached callers hit terminated process | Mark/drain, propagation allowance, termination grace period |
| Retry chooses same stale instance | Repeated failure and latency | Exclude attempted/ejected endpoint within retry budget |
| Long-lived connection survives deregistration | Existing traffic continues | Connection drain/GOAWAY/close policy; deregistration controls new discovery, not existing sockets |

## 48. Common mistakes

- Saying "service discovery is Eureka" as if a pattern and one implementation are identical.
- Calling the registry for every business request in a client-side design.
- Saying Eureka itself load-balances network traffic; the client/load-balancer uses its endpoint data.
- Treating heartbeat, liveness, readiness, and business correctness as synonyms.
- Assuming deregistration handles crashes.
- Registering a process as `UP` before it is ready.
- Killing an instance immediately after deregistration without allowing for cache propagation/in-flight work.
- Treating local cache freshness as one TTL in Eureka.
- Tuning heartbeat and lease timers independently or extremely low.
- Clearing the last-known-good cache when the registry temporarily fails.
- Retrying every failed write without idempotency or a deadline.
- Retrying at client, sidecar, gateway, and service simultaneously.
- Claiming three registry replicas guarantee high availability while all share one failure domain.
- Calling peer replication strongly consistent.
- Ignoring the bootstrap mechanism.
- Using `IP:port` without an incarnation as permanent instance identity.
- Letting arbitrary instances register any service name.
- Routing cross-region without checking data and policy semantics.
- Adding Eureka on top of Kubernetes without explaining what gap Kubernetes Services/EndpointSlices leave.
- Treating service discovery as a substitute for authentication, authorization, timeouts, or circuit breakers.

# Part XII - Alternatives and when to use or avoid them

## 49. Option comparison

| Option | Good fit | Advantages | Trade-offs / avoid when |
|---|---|---|---|
| Static config/hostnames | Small, stable topology | Simplest; easy to reason about | Manual changes; poor autoscaling/failure response |
| DNS discovery | Polyglot systems; moderate metadata needs | Universal client support; operationally familiar | Cache/TTL behavior; limited health and routing metadata |
| Stable proxy/load balancer | Simple clients; centralized policy | Hides endpoints; strong proxy health/routing | Extra hop; proxy fleet and config blast radius |
| Eureka client-side discovery | Spring/Java-heavy dynamic VM environments; availability-oriented reads | Direct calls, warm-cache resilience, mature lease model | Client coupling; eventual/stale views; in-memory registry |
| Consul | Multi-platform catalog, DNS/API, health checks, stronger catalog ordering | Agents, DNS, metadata, Raft-backed persisted control plane | Operate consensus/gossip/ACLs; quorum affects writes |
| Kubernetes Service + EndpointSlice/CoreDNS | Workloads already on Kubernetes | Platform-managed registration, readiness, VIP/DNS, no app registry SDK | Cluster/platform coupling; external/multi-cluster needs extra design |
| AWS Cloud Map/ECS integration | AWS-managed workloads | Managed catalog, DNS/API, ECS lifecycle integration | Cloud coupling, cost/quotas, DNS/API semantics |
| Service mesh | Polyglot services needing mTLS and rich traffic policy | Discovery plus identity, telemetry, retries, routing | Significant control/data-plane complexity and resources |

## 50. Eureka versus Consul - do not compare only product names

At a high level:

- Eureka commonly uses self-registering clients, in-memory peer registries, periodic client fetches, and availability-oriented eventual convergence.
- Consul documents a catalog replicated by Raft consensus, persisted control-plane state, agents, health checks, and DNS/API discovery.

This does not make one universally better. Ask:

- Do registry writes require ordered authoritative state or must same-side writes continue during a partition?
- Can every application embed a client library?
- Is DNS important?
- Who performs health checks?
- Who will operate quorum, upgrades, ACLs, backups, and disaster recovery?

## 51. Eureka versus Kubernetes native discovery

If all services already run on Kubernetes, a separate Eureka layer may duplicate:

- Pod lifecycle observation;
- ready endpoint tracking;
- stable Service names/VIPs;
- CoreDNS integration; and
- EndpointSlice distribution.

Eureka may still be justified for hybrid VM/container discovery, application-specific metadata, migration compatibility, or cross-platform needs. State that gap explicitly. Otherwise prefer the platform's existing control plane.

## 52. When to avoid client-side discovery

Avoid or reconsider it when:

- many languages make consistent libraries hard to maintain;
- security rules should hide internal topology;
- routing policy changes frequently and must roll out centrally;
- applications cannot implement safe retries/outlier detection; or
- a platform/service mesh already provides the needed abstraction.

## 53. When to avoid a central proxy tier

Avoid or reconsider it when:

- its latency/throughput cost violates the workload budget;
- extremely high fan-out would create expensive proxy concentration;
- callers need destination-specific local decisions; or
- the organization cannot operate the proxy tier across failure domains.

Distributed sidecars/node proxies may be a middle ground, but they still add operational complexity.

# Part XIII - Real-system examples

## 54. Netflix and Eureka - historical verified example

Netflix's public Eureka wiki describes a regional Eureka cluster with at least one server per availability zone in its deployment model. Services register and renew leases; registrations and renewals replicate among Eureka nodes; clients periodically fetch and cache registry information and then call service instances.

```text
Netflix service A ---- register/heartbeat ---> regional Eureka peers
Netflix service B <--- cached registry -------/
       |
       `---- client-side selection/direct call ---> service A instance
```

Netflix also documented using Eureka information behind more sophisticated client-side balancing and for operations such as taking deployments out of traffic. This is a **historical public production description**, not a claim that every Netflix service still uses the identical 2014/2018 architecture in 2026.

## 55. Uber Hyperbahn/HAProxy - historical verified example

Uber Engineering publicly wrote in 2016 that service discovery and routing were crucial to its SOA and described using HAProxy and Hyperbahn. Another Uber post described producers and consumers registering with a Hyperbahn routing mesh, consumers addressing producers by service name, and the mesh tracking failures/SLA violations.

```text
Consumer -> local/network routing layer -> Provider instances
                        ^
           registration + health/routing knowledge
```

This demonstrates a **proxy/routing-mesh approach**, contrasting with an application directly selecting from Eureka's cached instance list. It is historical; do not present it as Uber's guaranteed current architecture.

## 56. Twitter/X Finagle naming - historical public implementation

Twitter's public Finagle documentation separates a logical name from its concrete address and shows names being resolved through delegation into ZooKeeper-backed server sets. The resolved address is represented as dynamically changing state.

```text
/s/crawler
    -> namespace/delegation rules
    -> ZooKeeper server set
    -> changing set of concrete addresses
```

The deeper lesson is indirection: callers express **what** they need, while a naming layer binds it to **where** it currently lives. This public documentation is not proof that all X services use that exact path today.

## 57. Google/Borg and a YouTube-style service

Google's Borg paper describes name-service integration and a Borg Name Service that publishes task hostname/port information. Since the paper discusses Google production infrastructure generally, it supports a Google/Borg example, not a specific claim about YouTube's current private stack.

**Illustrative YouTube design using the same principle:**

```text
Upload API -> logical "transcode-service"
                  |
                  v
           current transcode workers
```

Transcode workers are scheduled, replaced, and scaled dynamically. The upload/orchestration tier should resolve a logical worker service or queue/worker capability rather than embed worker IPs. This diagram is interview synthesis, not a sourced YouTube implementation claim.

## 58. AWS Cloud Map / Amazon ECS - verified platform example

AWS documents Cloud Map as a managed mapping from logical names to backend resources, discoverable through DNS or a `DiscoverInstances` API. Amazon ECS can synchronize launched tasks into Cloud Map, so callers can resolve an internal service to current task addresses.

```text
ECS starts/stops tasks -> Cloud Map catalog -> DNS/API discovery -> caller
```

This is a verified **AWS platform** example. It is not evidence about Amazon.com's private retail microservice discovery architecture.

**Illustrative Amazon shopping flow:** an order service discovers inventory instances, prefers the same zone, and falls back only according to regional data policy. This use is plausible interview synthesis, not a claim about Amazon retail internals.

## 59. Kubernetes example

Kubernetes documents that a Service gives a stable abstraction over changing Pods. The control plane creates EndpointSlices containing backend addresses and endpoint conditions such as `ready`, `serving`, and `terminating`; clients can use a Service VIP/DNS or query EndpointSlices.

```text
Deployment replaces Pods
        |
EndpointSlice controller updates ready endpoints
        |
CoreDNS / Service routing / native client
        |
Caller reaches a ready Pod
```

This is third-party/platform registration rather than every Pod implementing Eureka APIs.

## 60. WhatsApp-style persistent connection routing - illustrative only

Meta publicly describes WhatsApp clients connecting to Meta infrastructure and has published parts of WhatsApp's client/server architecture, but the sources reviewed for this chapter do **not** establish a current internal service-registry implementation.

An interview design still needs discovery behind the front door:

```text
Phone === long-lived connection ===> Chat gateway instance
                                        |
                                        `-> discovers message/presence/media services
```

For a long-lived connection, discovery usually selects the gateway at connection establishment. Removing the gateway from discovery stops **new** connections but does not move existing sockets. Graceful drain, reconnect behavior, and session/routing state therefore matter. This is explicitly an **illustrative design**, not a claim about WhatsApp internals.

# Part XIV - Interview angle

## 61. How the topic appears in interviews

Service discovery may appear as:

- a direct prompt: "Design a service discovery system";
- a microservices concept question: "How does Eureka work?";
- a comparison: "API gateway versus service discovery";
- a failure follow-up: "What if the registry goes down?";
- a scaling follow-up: "How do new instances receive traffic?";
- a health follow-up: "How do you remove a dead instance?"; or
- an implicit requirement inside chat, ride-sharing, notifications, video processing, or payments.

The interviewer is usually testing whether you can reason about **dynamic membership, health ambiguity, caching, load balancing, consistency, and failure**, not whether you memorized Eureka annotations.

## 65. High-value follow-ups and concise answer cues

### "Is service discovery the same as load balancing?"

No. Discovery produces the current eligible endpoint set; load balancing chooses one endpoint for a request. Eureka supplies registry data; a client library/proxy performs selection and the business call.

### "What if Eureka goes down?"

With one node, registration/renewal/cold discovery fail, while warm clients may temporarily use cached endpoints. With peers across zones, clients fail over and mutations replicate. If all peers fail, last-known-good caches help only while cached endpoints remain usable; new membership is unavailable. Restore the control plane, refresh with jitter, and rely on data-plane timeouts/outlier detection meanwhile.

### "Does Eureka add an extra hop?"

Not normally for client-side discovery. Registry fetches occur at startup and periodically in the background; the client chooses from its local cache and calls the service directly. A proxy-based pattern adds a proxy hop, not a registry lookup per request.

### "How does Eureka know an instance died?"

Graceful shutdown sends cancel, but crashes are detected by missing heartbeat renewals and lease expiry/eviction. Self-preservation may retain entries during a widespread renewal loss. Heartbeats alone do not prove readiness, so use health status plus data-plane failure detection.

### "Why can the local cache be stale?"

Failure detection, peer replication, server response caching, periodic client refresh, and self-preservation are asynchronous. Staleness buys lower lookup latency and registry-outage tolerance. Bound its impact with deadlines, ejection, safe retries, draining, and measured propagation SLOs.

### "Should I make the TTL very small?"

There is no one TTL in this flow. Shorter heartbeat/expiry/refresh values increase freshness but raise load and false eviction during pauses/partitions. Coordinate the timers from failure objectives, respect implementation assumptions, use jitter, and load/failure test them.

### "Register the service or the instance?"

Each running instance registers under a stable service name. The registry maps one service to many instance records. The service name is the query identity; instance ID distinguishes membership and lifecycle.

### "Can a heartbeat say `UP` while the service is broken?"

Yes. It may prove only that the client thread can reach the registry. Propagate meaningful readiness carefully and combine it with proxy/client observations of real traffic. Avoid deep checks that remove every instance during one shared-dependency blip.

### "Eureka or Kubernetes Service?"

If workloads already live entirely in Kubernetes and need standard VIP/DNS/readiness behavior, use native Services/EndpointSlices unless Eureka fills a named gap such as hybrid workloads or compatibility. Avoid duplicating control planes by habit.

### "What consistency model should a registry use?"

It depends. Availability-oriented eventual views let healthy partition sides continue but can advertise stale endpoints. Consensus gives authoritative ordered updates but a minority cannot accept writes. State the failure preference and design clients accordingly.

## 66. Compact cheat sheet

```text
WHY
  endpoints change during scaling, failure, deployment, and rescheduling
  callers use logical service identity, not physical IPs

CORE MODEL
  service name -> instance ID -> lease(instance info)
  discovery finds candidates; load balancing chooses one

REGISTRATION
  self-register (Eureka client) or third-party/controller (Kubernetes/ECS)
  graceful: not-ready/drain/deregister
  crash: missing heartbeat -> lease expiry -> eviction policy

HEALTH
  heartbeat != readiness != correctness
  false positive: dead endpoint retained
  false negative: healthy endpoint removed
  add passive outlier detection, timeouts, bounded safe retries

EUREKA
  in-memory peer registries; clients cache and periodically fetch deltas/full state
  client-side discovery normally has no registry hop per business request
  peer replication/eventual convergence; self-preservation favors availability

STALE WINDOW
  detection + eviction + server cache + peer propagation + client refresh
  not one simple TTL

HA
  peers across failure domains + multiple client endpoints + bootstrap
  warm cache helps during outage; cold clients/new membership still fail

ALTERNATIVES
  static config, DNS, proxy/LB, Kubernetes Service, Cloud Map, Consul, mesh

INTERVIEW
  requirements -> pattern -> data model -> health -> read/cache -> HA/consistency
  -> stale defenses -> security/operations -> trade-off
```

## 67. Final design checklist

- What logical name does the caller use?
- Who registers instances: application, agent, controller, or orchestrator?
- What makes an instance ready, draining, and dead?
- Is the record instance-level and protected by an incarnation ID?
- How are graceful shutdown and sudden crash handled differently?
- Who holds the endpoint list and who selects an endpoint?
- Is the registry/proxy on the request path?
- What is the snapshot/delta/watch recovery model?
- What is the maximum acceptable stale-routing window?
- How do heartbeat, lease, eviction, server cache, and client refresh interact?
- What happens during a client-registry partition?
- What happens when one peer, a zone, or every registry peer fails?
- What can a warm client do versus a cold client?
- Is replication eventual or consensus-backed, and why?
- How does local-zone preference and cross-zone/region fallback work?
- What stable bootstrap locates the discovery system?
- Can an unauthorized workload poison a service name?
- Are retries bounded, deadline-aware, different-target, and safe for writes?
- Are refreshes/heartbeats jittered and payloads bounded?
- Are renewal loss, peer lag, cache age, empty pools, ejections, and stale-routing time monitored?
- Does an existing platform already solve the requirement more simply?

## 68. Say-out-loud 30-second narrative

> "Service discovery lets callers use a stable service name while the actual instance IPs and ports change during scaling, deployments, and failures. Each instance registers under that name, usually with metadata and a renewable lease; a client, proxy, DNS layer, or platform resolves the name, and a load balancer chooses an eligible endpoint. With Eureka, servers keep an in-memory peer-replicated registry, while clients periodically cache it and call instances directly, so there is normally no registry hop per request. Those caches and health timeouts create stale endpoints, so I combine graceful draining, readiness, lease expiry, timeouts, passive ejection, bounded safe retries, multi-zone registry peers, security, and propagation monitoring."
