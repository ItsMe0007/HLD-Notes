# Chat Application Design

> Beginner-first HLD study notes for a WhatsApp/Discord-style chat system. The core problem is not merely opening a WebSocket. It is accepting a message durably, ordering it within a conversation, routing it to every intended device, recovering after disconnects, and making retries look exactly-once to the user even though distributed delivery is usually at-least-once.

## Scope, assumptions, and evidence labels

The rough bullets ask about:

- functional and non-functional requirements;
- back-of-the-envelope estimates;
- peer-to-peer, WebSocket, WebRTC, polling, and long polling;
- allocating connection servers and mapping logged-in users;
- database choice, partitioning, replication, and message queues;
- whether one load balancer is enough;
- direct-chat versus group-chat servers;
- offline messages and heartbeats; and
- why WhatsApp emphasizes local databases while Discord keeps durable server history.

This chapter designs **text chat first**: one-to-one and group messages, multiple devices, online delivery, offline sync, receipts, presence, and media attachments. Voice/video calling is an extension because it has a different data path and normally uses WebRTC. The chosen baseline keeps durable server-side message history for convenient multi-device sync; a stricter WhatsApp-like, device-history variant is shown separately.

Research and product behavior were checked on **2026-07-16**.

### Evidence labels

- **Protocol/official documentation**: an RFC or official product/project document linked nearby.
- **First-party production snapshot**: an engineering description published by the company on a particular date. It may not be the architecture today.
- **Public product behavior**: a feature documented by its owner; it does not reveal the private backend.
- **Self-reported interview experience**: a candidate's public report. It establishes question style, not an official question bank.
- **Illustrative design/number**: a reasoned interview assumption, not a claim about a named company's production system.

### The most important corrections up front

1. **WebSocket and WebRTC are not competing answers to the same question.** A WebSocket is a persistent client-server, two-way messaging channel. WebRTC is a suite for interactive audio/video/data paths, often peer-to-peer when the network permits. Use WebSocket for chat events and signaling; use WebRTC for calls.
2. **A WebSocket does not make a message durable.** If the server process dies after reading a frame but before storing it, the message is gone unless the client retries. Durability, ordering, deduplication, and receipts are application responsibilities.
3. **“Exactly once” is normally a user-visible effect, not a network guarantee.** Retries can create duplicate deliveries. A client-generated idempotency key plus server deduplication makes duplicates display once.
4. **Offline push notifications are hints, not the message source of truth.** APNs/FCM may delay, collapse, reorder, throttle, or drop notifications. Reconnect and sync must read from the chat service's durable state.
5. **Do not store `user -> server` as permanent truth in the main database.** Connections move and die frequently. Keep a leased, expiring `user/device -> gateway connection` directory; rebuild it on reconnect.
6. **One logical load-balancer endpoint need not mean one machine.** Production uses a redundant regional load-balancer fleet or managed service. A single appliance would be a bottleneck and failure point.
7. **NoSQL is not automatically “the chat database.”** Choose from access patterns and guarantees. Relational storage is excellent for users, memberships, and smaller systems; a wide-column/key-value log is attractive for very large, append-heavy message history.
8. **Direct and group chats should share a conversation abstraction, not necessarily every execution path forever.** A common `conversation_id` and API reduce duplication; large-group fan-out can later use specialized workers and isolation.
9. **WhatsApp is not purely peer-to-peer and not purely local.** Meta documents a client/server architecture with persistent client connections, temporary offline server messages, device/key mappings, and encrypted state synchronization. “Local database” describes where delivered history is principally read, not the whole system.
10. **End-to-end encryption does not require history to exist only locally.** A server can retain ciphertext while only client-controlled keys can decrypt it. Meta explicitly describes such a server-based encrypted-history design for Messenger.

### Rough-bullet expansion map

| Rough bullet | Expanded answer |
|---|---|
| P2P, WebSocket, or WebRTC? | Client-server WebSocket for text/events; HTTP for bulk/history/media control; WebRTC for calls; P2P text only for unusual products |
| Polling/long polling/WebSocket | Compare directionality, latency, resource cost, reconnect behavior, and operational complexity |
| Server allocation | Balance new sockets across gateway capacity; keep correctness outside the gateway; use a leased route directory |
| Login mapping | Map each **device connection**, not just a user, because one user may have phone, web, and desktop sessions |
| Database | Hybrid is common: SQL for metadata/membership, partitioned log store for messages, object storage for media, ephemeral store for presence |
| Partition/replication | Partition messages by conversation plus time bucket; replicate across failure domains; keep one ordering authority per conversation |
| Message queue | Use after durable acceptance for fan-out, push, indexing, moderation, and analytics; bridge DB-to-broker with an outbox/log |
| Single LB | One DNS name can front many regional LBs; sockets reconnect and resume after LB/gateway failure |
| Direct versus groups | Common `conversation_id` and ingress; different fan-out strategies based on group size and traffic |
| Offline messages | Durable conversation log/inbox cursor plus push hint; sync by sequence/cursor on reconnect |
| Heartbeat | Detect dead/zombie sockets, refresh route leases, and measure RTT; jitter and tune for battery/NAT/LB timeouts |
| WhatsApp local vs Discord server DB | Privacy, retention, search, multi-device, moderation, and product semantics drive the choice |

## 1. Plain-language foundation

### 1.1 What a chat system really does

Alice types “hello” to Bob. A reliable system must do more than move five characters:

```text
Alice's phone       Chat service          Durable state       Bob's devices
     |                    |                     |              phone  laptop
     | send hello         |                     |                |      |
     +------------------->| authenticate        |                |      |
     |                    | authorize chat      |                |      |
     |                    | assign order        |                |      |
     |                    | persist ----------->|                |      |
     | accepted           |                     |                |      |
     |<-------------------+                     |                |      |
     |                    | route/fan out ---------------------->|      |
     |                    | route/fan out ----------------------------->|
     |                    |<---------------- delivery ACK -------|      |
     | delivered          |                     |                |      |
     |<-------------------+                     |                |      |
```

The design must answer:

- Was Alice allowed to write to this conversation?
- What if Alice taps send twice or retries after a timeout?
- What order do Alice's and Bob's simultaneous messages have?
- What if Bob has three devices, or none is online?
- What if a gateway, broker, database replica, or whole region fails halfway?
- Does “delivered” mean one device, every device, or Bob actually opened it?
- How does Bob recover everything missed during a disconnect?

### 1.2 Inline prerequisite: request/response versus server push

Ordinary HTTP is commonly taught as:

```text
client --request--> server
client <-response-- server
```

The server cannot use an already-completed request to spontaneously send a later chat message. Polling repeatedly creates requests; long polling holds one request open; Server-Sent Events keeps a server-to-client stream; WebSocket upgrades into a persistent two-way channel.

This assumes basic DNS, TLS, HTTP, and TCP knowledge. The minimal chain is:

```text
DNS finds an edge address
  -> TCP (or another transport) establishes connectivity
  -> TLS authenticates/encrypts the connection
  -> HTTP authenticates or upgrades the session
  -> WebSocket carries application events
```

See [01. Network Protocols](./01.%20Network%20Protocols.md) for the full prerequisite.

### 1.3 Inline prerequisite: a persistent connection is stateful

A normal API server can handle request 1 and request 2 on different machines. A live WebSocket is attached to one gateway process for its lifetime. That gateway holds state such as:

- connection/socket handle;
- authenticated `user_id` and `device_id`;
- last activity and heartbeat state;
- outbound buffer/backpressure state; and
- possibly the last session cursor.

This does **not** mean the gateway should own durable truth. If it dies, a client should connect elsewhere and resume from durable sequence/cursor state.

### 1.4 Inline prerequisite: message, event, and notification differ

- A **message** is durable user content: text, reaction, image metadata, etc.
- An **event** is a state change sent between components or to clients: `message_created`, `typing_started`, `receipt_advanced`.
- A **push notification** wakes or alerts a mobile device. It may say “new messages available” without carrying authoritative history.

Typing indicators can be lossy events. A paid support message should be durable. Treating both identically wastes resources or loses important data.

### 1.5 Delivery states must have precise meanings

```text
local pending -> server accepted -> delivered -> read
      |               |                |          |
  UI optimistic    durable store    device ACK  explicit client receipt
```

Recommended definitions for this chapter:

- **Pending**: only the sender client currently knows the message.
- **Accepted/sent**: the service durably stored the message and its deduplication identity.
- **Delivered**: at least one recipient device acknowledged receipt. If the product means “all active devices,” say so explicitly.
- **Read**: a recipient client explicitly advanced its read cursor past this message. It is not inferred merely from network delivery.

The word “sent” is ambiguous in interviews. Define it before drawing ticks/checkmarks.

### 1.6 Inline prerequisite: acknowledgement is not proof of the next stage

An ACK only proves what its contract says:

```text
TCP ACK          -> bytes reached the peer's TCP stack
WebSocket write  -> local library accepted bytes; not durable business state
server ACK       -> message met the server's stated acceptance rule
device ACK       -> recipient app received/stored it, according to its contract
read receipt     -> recipient client reports a read high-water mark
```

Never use a low-level network ACK as a user-facing delivery receipt.

### 1.7 Inline prerequisite: idempotency

An operation is **idempotent** when retrying it has the same intended effect as doing it once. Alice creates a stable ID before the first send:

```text
client_message_id = random UUID/128-bit value
dedupe key        = (sender_device_id, client_message_id)
```

If Alice times out and retries, the service returns the already-created `message_id` instead of inserting another message. The dedupe record must be committed atomically with, or be inseparable from, durable acceptance of the message.

### 1.8 Inline prerequisite: ordering is scoped

A global order across every conversation is expensive and unnecessary. Chat normally needs:

- deterministic order **within one conversation**;
- stable pagination/resume order; and
- no promise that unrelated chats share an order.

Server time alone is unsafe because clocks differ and two writes can share/reverse timestamps. Assign a monotonically increasing `conversation_seq` or an equivalent totally ordered key at the conversation's ordering authority.

### 1.9 Inline prerequisite: fan-out

**Fan-out** means turning one input into deliveries/work for many destinations:

```text
one group message -> member A devices
                  -> member B devices
                  -> member C devices
                  -> ...
```

Fan-out is cheap for a direct chat and potentially the dominant cost for a huge group. **Fan-out on write** creates recipient work when the message is sent; **fan-out on read** stores once and lets members fetch it when they read.

### 1.10 Inline prerequisite: end-to-end encryption

With **end-to-end encryption (E2EE)**, content is encrypted on a sender device and only authorized recipient devices possess keys to decrypt it. Transport TLS is different: TLS protects one network hop/session, but a server terminating TLS can normally see plaintext. An E2EE server can route and store ciphertext without having the content key, though it still sees some routing metadata.

## 2. Requirements before architecture

### 2.1 Questions to clarify with the interviewer

Ask only questions that can change the design:

1. One-to-one only, groups, broadcast channels, or all three?
2. Maximum ordinary and exceptional group sizes?
3. Text only, or images/video/files? Calls?
4. Should history synchronize across devices and after reinstall?
5. Is end-to-end encryption required?
6. Required retention: forever, configurable, or delete after delivery?
7. Are edits, deletes, reactions, replies, typing, presence, and search in scope?
8. What does delivered/read mean with multiple devices?
9. Scale: DAU, concurrent connections, messages/day, media/day, geographic spread?
10. Is per-conversation ordering required? What availability/latency targets matter most?

### 2.2 Functional requirements selected for this design

Core:

- authenticate a user and each device;
- create one-to-one and group conversations;
- send/receive text messages in near real time;
- fetch paginated history;
- work when recipients are offline and sync after reconnect;
- support multiple devices per user;
- show accepted, delivered, and read states;
- provide approximate online/last-seen presence; and
- attach media using object storage.

Secondary:

- typing indicators, reactions, edits/deletes, replies;
- push notifications;
- group membership/roles; and
- abuse reporting and rate limits.

Out of the main path:

- voice/video calls, implemented as WebRTC signaling plus media infrastructure;
- full-text search details;
- large public broadcasts/livestream chat, discussed as a different fan-out class; and
- encryption protocol internals, though their architectural consequences are included.

### 2.3 Non-functional requirements

These are **illustrative interview targets**, not claims about WhatsApp or Discord:

| Property | Example target/meaning | Design consequence |
|---|---|---|
| Connected delivery latency | p99 under 500 ms within a region | Persistent channel, short durable write path, asynchronous side effects |
| Availability | 99.99% for sending/reconnect sync | Redundant gateways, replicated storage, no single LB/broker |
| Durability | No acknowledged message lost after `accepted` | ACK only after replicated durable acceptance |
| Ordering | Total order per conversation, not global | Conversation-partitioned sequencer/leader |
| Delivery semantics | At-least-once transport; exactly-once visible result | Client IDs, dedupe, idempotent consumers |
| Consistency | Strong membership authorization and message acceptance; eventual presence/receipts | Separate critical from ephemeral state |
| Scalability | Horizontal for sockets, message rate, fan-out, and storage | Partitioned gateways, logs, broker, databases |
| Security | TLS; authorization on every action; optional E2EE | Device identity/key management, metadata still protected |
| Recoverability | Resume missed events after disconnect or gateway loss | Durable cursors/sequence numbers and replay API |
| Backpressure | Slow clients cannot exhaust gateway memory | Bounded buffers, coalescing, disconnect/resume |

### 2.4 CAP/PACELC applied without slogans

During a network partition, a conversation's ordering authority should avoid accepting conflicting sequence histories in two isolated regions. A common choice is:

- route a conversation to a home region/leader;
- fail writes or fail over under controlled fencing if that authority is unavailable; and
- allow stale history reads/presence where acceptable.

Presence and typing favor availability/eventual consistency. Membership changes and durable message authorization need stronger consistency. Different data in one product can make different CAP choices. See [02. CAP, PACELC and Replication Strategies](./02.%20CAP,%20PACELC%20and%20Replication%20Strategies.md).

## 3. Back-of-the-envelope estimation

### 3.1 State assumptions first

The following is a **worked illustrative workload**, deliberately not attributed to a real company:

| Input | Assumption |
|---|---:|
| Daily active users | 100 million |
| Messages sent per DAU per day | 20 |
| Peak-to-average message factor | 5x |
| Concurrent connected devices | 50 million |
| Average stored text-message envelope | 1 KB |
| Average destination device deliveries per logical message | 2.5 |
| Replication factor | 3 |
| Illustrative heartbeat interval | 60 seconds |

Media bytes are excluded; they normally dominate storage/network and use a separate object path.

### 3.2 Logical messages per second

```text
logical messages/day = 100,000,000 * 20
                     = 2,000,000,000/day

average messages/s   = 2,000,000,000 / 86,400
                     ~= 23,148/s

peak messages/s      = 23,148 * 5
                     ~= 115,741/s
```

“Message QPS” is incomplete. We also need delivery fan-out, receipts, history reads, reconnects, presence, and heartbeats.

### 3.3 Delivery-event rate

If a logical message creates 2.5 destination-device deliveries on average:

```text
delivery events/day      = 2B * 2.5 = 5B
average delivery events/s ~= 57,870/s
peak delivery events/s    ~= 289,352/s   (using the same illustrative 5x)
```

The 2.5 factor is highly sensitive to group sizes and device counts. One message to a 100,000-member broadcast cannot be modeled using an ordinary-chat average; it needs a separate workload class.

### 3.4 Message storage

```text
logical new text data/day = 2B * 1 KB = 2 TB/day
logical/year              ~= 730 TB/year
RF=3 raw replicas         ~= 2.19 PB/year
```

This excludes indexes, database overhead, tombstones, backups, cross-region copies, edits, compression, and retention deletion. It is a sizing starting point, not a provisioned-storage number.

### 3.5 Concurrent-connection memory and gateway count

Assume, only for estimation, that the gateway's total amortized memory per connection is 50 KB:

```text
50M connections * 50 KB ~= 2.5 TB gateway-fleet RAM
```

If load testing shows one gateway can safely hold 250,000 sockets at its hard limit:

```text
theoretical minimum = 50M / 250K = 200 gateways
at 60% target occupancy ~= 334 gateways
```

Then add rolling-deploy and failure-domain reserve. Socket capacity must be benchmarked with the real runtime, TLS, buffers, payload distribution, file descriptors, NIC, CPU, and heartbeat behavior; 50 KB and 250,000 are **illustrative, unverified assumptions**.

### 3.6 Heartbeats may exceed message rate

```text
50M connections / 60 seconds ~= 833,333 client heartbeats/s
```

If each receives an ACK/pong, that is roughly 1.67 million heartbeat frames/s across the fleet. The exact interval is a battery, NAT, load-balancer idle-timeout, and failure-detection trade-off. Jitter is essential so clients do not all fire on the minute.

### 3.7 Text-path network, excluding overhead

At the illustrative peak:

```text
ingress payload  ~= 115,741 * 1 KB = 116 MB/s ~= 0.93 Gb/s
delivery payload ~= 289,352 * 1 KB = 289 MB/s ~= 2.31 Gb/s
```

Add TLS/WebSocket framing, protocol fields, receipts, replication, broker traffic, inter-region traffic, and sender-device echo. Media must be estimated independently.

### 3.8 What these estimates actually change

- Connection gateways scale on **concurrent sockets and frames**, not only send QPS.
- Heartbeat efficiency deserves as much attention as the message API.
- Text history needs partitioned storage and retention policy.
- Fan-out workers scale separately from ingress.
- A huge group/broadcast tier needs isolation from ordinary chats.
- Media uploads/downloads must bypass chat servers.

For the full estimation method, see [08. Back-of-the-Envelope Estimation](./08.%20Back-of-the-Envelope%20Estimation.md).

## 4. APIs, events, and data model

### 4.1 Separate control/bulk APIs from the live event channel

Use HTTPS for operations that are naturally request/response or bulk:

```http
POST /v1/sessions
POST /v1/conversations
POST /v1/conversations/{conversation_id}/members
GET  /v1/conversations/{conversation_id}/messages?before_seq=9000&limit=50
POST /v1/media/uploads
GET  /v1/sync?cursor=opaque_device_cursor
```

Use secure WebSocket (`wss://`) for low-latency commands/events:

```json
{
  "type": "send_message",
  "request_id": "req-77",
  "conversation_id": "c-123",
  "client_message_id": "deviceA-uuid-9",
  "content": {"kind": "text", "body": "hello"}
}
```

```json
{
  "type": "message_accepted",
  "request_id": "req-77",
  "message_id": "m-9001",
  "conversation_id": "c-123",
  "conversation_seq": 9001,
  "server_time": "2026-07-16T10:00:00Z"
}
```

The same send command could be HTTP `POST`; WebSocket is a latency/connection efficiency choice, not a correctness requirement. Supporting an HTTP fallback can simplify retries and restricted networks.

### 4.2 Core records

```text
User
----
user_id, profile, status, created_at

Device
------
device_id, user_id, public_key?, push_token, revoked_at, last_sync_cursor

Conversation
------------
conversation_id, type(DIRECT|GROUP|CHANNEL), created_by,
home_region, next_seq/ordering_metadata, retention_policy, membership_version

Membership
----------
conversation_id, user_id, role, joined_seq, left_seq, mute_settings

Message
-------
conversation_id, bucket, conversation_seq, message_id,
sender_user_id, sender_device_id, client_message_id,
kind, payload_or_ciphertext, media_ref?, created_at, membership_version

UserConversationState
---------------------
user_id, conversation_id, last_delivered_seq, last_read_seq,
last_visible_seq, mute_state, unread_count_hint
```

### 4.3 One `conversation_id` for direct and group chat

A message should refer to a conversation, not embed a special direct-versus-group schema:

```text
conversation_id -> type + membership + policy
message         -> conversation_id + sequence + payload
```

For a direct chat, the system may enforce one canonical conversation for a pair using:

```text
direct_pair_key = hash(min(userA,userB), max(userA,userB))
```

Whether a new thread can exist between the same two people is a product requirement, not a database fact.

### 4.4 Use high-water marks for receipts

Writing one read-receipt row per message per group member is explosive. If a user has read through sequence 9000, store:

```text
(user_id, conversation_id) -> last_read_seq = 9000
```

This means every earlier visible message is read, subject to membership/history rules. Individual exceptions may need separate state, but the common path is compact.

### 4.5 Edits and deletes are events, not time travel

An append-oriented model can represent:

```text
seq 9001 MESSAGE_CREATED target=m1
seq 9002 MESSAGE_EDITED  target=m1
seq 9003 REACTION_ADDED  target=m1
seq 9004 MESSAGE_DELETED target=m1
```

Clients materialize current state. Compaction/snapshots prevent replaying an infinite event history. “Delete for everyone” may be a tombstone propagated to clients; whether old devices/backups truly erase content is a privacy/product contract that must be stated honestly.

## 5. Polling, long polling, SSE, WebSocket, P2P, and WebRTC

### 5.1 Short polling

```text
Client: anything new? ----> Server: no
Client: anything new? ----> Server: no
Client: anything new? ----> Server: message
```

**Good:** simplest, stateless requests, works through ordinary HTTP infrastructure.

**Bad:** either high latency (long interval) or many empty requests (short interval); synchronized polling causes bursts.

**Use:** tiny prototypes, infrequent updates, fallback, or admin screens. Avoid as the primary path for a high-scale real-time chat.

### 5.2 Long polling

```text
Client: anything new? ------------------------->
Server: [holds request until event/timeout] ---> response
Client: immediately opens the next request ---->
```

**Good:** near-real-time server-to-client delivery using HTTP; historically useful where WebSockets were unavailable.

**Costs:** repeated headers/requests, race windows between polls, server resources for outstanding requests, awkward bidirectional behavior, and reconnect bookkeeping.

Meta's 2008 Facebook Chat production snapshot described an Erlang channel service holding long-polled HTTP requests and partitioning online-user state. It is useful historical evidence that long polling can scale with specialized infrastructure, not a recommendation that it is simpler today: [Facebook Chat engineering post](https://engineering.fb.com/2008/05/13/web/facebook-chat/).

### 5.3 Server-Sent Events (SSE)

SSE keeps a one-way HTTP response stream from server to client. The client sends via ordinary HTTP.

**Good:** built-in event stream semantics, automatic browser reconnect support, simpler than WebSocket when only server push is needed.

**Costs:** one-way, browser connection/proxy constraints, and still requires an upload path. It is a legitimate design for server-heavy notifications, but chat is naturally bidirectional.

### 5.4 WebSocket

[RFC 6455](https://www.rfc-editor.org/rfc/rfc6455.html) defines an opening HTTP handshake followed by framed two-way communication over TCP. Either side can send independently.

```text
client ================= persistent WSS ================= gateway
       send_message  ->
       <- message_created
       read_cursor   ->
       <- typing/presence
```

**Why selected:** low-latency bidirectional events, less repeated HTTP overhead, and a natural connection on which the server can push messages.

**Costs:** stateful connection fleet, load-balancer idle timeouts, reconnect/resume, authentication refresh, heartbeat traffic, deployment draining, backpressure, and connection-count limits.

WebSocket preserves frame/message order on one TCP connection, but it does **not** provide durable storage, cross-reconnect replay, end-to-end receipts, or cross-server conversation ordering.

### 5.5 Peer-to-peer text chat

Pure P2P means Alice's device sends directly to Bob's device. It sounds attractive because servers carry fewer content bytes, but ordinary durable chat needs the server for:

- discovering devices and exchanging keys/connection candidates;
- NAT/firewall traversal and relays;
- delivery while Bob is offline;
- multi-device fan-out and history sync;
- group membership and ordering;
- abuse/rate enforcement and push notifications; and
- account recovery.

For a group of `N` fully meshed peers, connections or sender copies can approach `O(N^2)` overall. Direct IP exposure can also be a privacy concern. P2P text is reasonable for LAN/off-grid/ephemeral products whose defining requirement accepts those limitations; it is not the default for WhatsApp-like reliable messaging.

### 5.6 WebRTC

[RFC 8825](https://www.rfc-editor.org/rfc/rfc8825.html) describes WebRTC's goal as interactive audio/video/data along the most direct feasible path. It still needs **signaling** to exchange offers, answers, and network candidates; WebRTC deliberately does not prescribe the application's signaling protocol.

Most devices sit behind **NAT**, where a router maps a private device address to a public-facing address, and firewalls may block unsolicited paths. WebRTC uses **ICE** to test candidate paths, often **STUN** to learn a public-facing address, and **TURN** to relay traffic when a direct path cannot be established. Therefore “WebRTC is always serverless P2P” is false.

```text
Signaling/control:
Alice --WebSocket--> chat/call service --WebSocket--> Bob

Media when direct works:
Alice <================ WebRTC/SRTP ================> Bob

Media when direct fails:
Alice <====> TURN relay <====> Bob

Group call:
participants <====> SFU (selective forwarding unit)
```

Use WebRTC for voice/video/screen share. A WebRTC data channel can carry text, but it does not solve durable offline history, so it rarely replaces the main chat backend.

### 5.7 Decision table

| Technique | Direction | Persistent | Offline durability built in? | Best fit |
|---|---|---:|---:|---|
| Short polling | Client asks | No | No | Simple/infrequent updates |
| Long polling | Server answers later | Per request | No | Compatibility/fallback |
| SSE + HTTP send | Server stream + client POST | Yes, one-way | No | Server-heavy event feeds |
| WebSocket | Two-way | Yes | No | Live chat/control events |
| WebRTC | Peer/media-oriented | Session | No | Calls and interactive media/data |
| Push notification | Provider -> device | Provider managed | Best effort, bounded | Wake/alert offline mobile client |

### 6.2 Why the gateway and chat service are separate

The gateway specializes in millions of mostly idle, long-lived connections and bounded I/O. The chat service specializes in business correctness: membership, idempotency, ordering, storage, and fan-out decisions.

Benefits:

- gateway restarts do not move durable chat ownership;
- socket fleet and message-processing fleet scale independently;
- HTTP, WebSocket, and future transports can share one chat core;
- a slow client cannot directly block a conversation shard; and
- specialized runtimes can be chosen only where useful.

At small scale these can be one deployable application with clear internal modules. Microservices are an evolution, not a requirement for correctness.

### 6.3 The synchronous acceptance path should be short

On send, do only what defines successful acceptance:

1. authenticate the device;
2. authorize current conversation membership and rate policy;
3. deduplicate `client_message_id`;
4. assign per-conversation order;
5. durably store/replicate message plus a reliable publication intent; and
6. ACK the sender.

Fan-out, push, indexing, analytics, link previews, and most moderation work should be asynchronous unless the requirement says the message must be blocked before acceptance.

## 7. Login mapping, server allocation, and load balancing

A **load balancer** accepts a public connection and selects a healthy backend gateway. For a WebSocket, it then keeps forwarding that connection's traffic to the chosen gateway until the connection closes. A **connection gateway** is the edge application server that owns the live socket; a **route directory** tells internal delivery services which gateway currently owns a user's device connection.

### 7.1 Authentication and connection establishment

```text
1. Device --HTTPS login/refresh--> Auth service
2. Auth service returns short-lived access token + refresh mechanism
3. Device --WSS Upgrade + token--> load balancer
4. LB chooses a healthy gateway with connection capacity
5. Gateway validates token/device/account status
6. Gateway creates connection_id and registers a leased route
7. Client sends last durable sync/session cursor
8. Gateway/chat service replays missed events, then marks session ready
```

Do not rely on a token validated only months ago. Support expiry/refresh or server-driven reauthentication and immediate device revocation.

### 7.2 Map devices, not merely users

The route directory contains something like:

```text
(user_id, device_id)
    -> {region, gateway_id, connection_id, session_id, lease_expiry}
```

A secondary index may resolve all devices for a user:

```text
user_id -> set of active device routes
```

Multiple simultaneous connections can exist because of phone, web, desktop, reconnect overlap, or app duplication. Give each session a generation/fencing token so a stale connection cannot overwrite a newer route or ACK state.

### 7.3 Why the mapping is a lease

A clean disconnect can delete the mapping immediately, but machines and networks often fail silently. Heartbeats refresh a short TTL. If refresh stops, the route expires.

The directory is **soft state**:

- losing it delays online delivery but does not lose messages;
- clients reconnect or the delivery service falls back to pending state;
- it can be sharded by `user_id` and replicated for availability; and
- database-level permanent transactions on every heartbeat are unnecessary.

An in-memory distributed store, a custom sharded registry, or gateway-owned partitions with discovery can implement it. Redis is a possible component, not a magic requirement.

### 7.4 How to allocate new connections

For a long-lived socket, the balancing decision happens mainly at connection time. Useful inputs include:

- active connection count;
- event-loop lag/CPU;
- outbound buffered bytes;
- memory and file-descriptor headroom;
- region/AZ; and
- whether a gateway is draining.

**Recommended baseline:** locality/AZ-aware least-connections or power-of-two-choices. Do not use round-robin blindly if connections have very different lifetimes.

### 7.5 Sticky session versus external routing directory

Once a TCP/WebSocket connection exists, it is inherently attached to one gateway. Cookie stickiness for later new connections is not necessary for correctness if session state/cursors are external or replayable.

| Choice | Benefit | Cost |
|---|---|---|
| Sticky reconnect | Better warm cache, possibly faster resume | Imbalance, dead target handling, makes gateway state tempting |
| Any healthy gateway + directory | Flexible recovery and draining | Route lookup and replay required |
| Consistent-hash user to gateway | Stable placement under normal churn | Hot users, membership changes, capacity-aware balancing is harder |

Consistent hashing can choose a **home shard/region** or cache owner, but strict gateway affinity should not be the only correctness mechanism. See [06. Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md).

### 7.6 “Single LB?”

Use one friendly endpoint, not one physical balancer:

```text
                         DNS / Anycast / traffic manager
                       /               |                \
                 Region A          Region B          Region C
                 LB fleet          LB fleet          LB fleet
                  / | \             / | \             / | \
             gateways...       gateways...       gateways...
```

- Multiple LB nodes/AZs remove the appliance single point of failure.
- Health checks stop new connections to unhealthy/draining gateways.
- Existing sockets do not magically move when an LB/gateway fails; clients reconnect with exponential backoff plus jitter and resume.
- LB idle timeout must exceed the effective heartbeat interval.
- Capacity planning includes concurrent flows, new connections/s, TLS handshakes/s, bytes/s, and packets/s, not only HTTP RPS.

#### L4 versus L7 for WebSockets

- An **L4 balancer** routes TCP connections with low protocol overhead. TLS may terminate at the gateway (or at another proxy), and application-aware routing is limited.
- An **L7 balancer/reverse proxy** can terminate TLS, inspect the HTTP upgrade, apply host/path/auth-related policy, and then proxy the upgraded connection. It consumes more resources and its WebSocket/idle/draining behavior must be configured explicitly.

Either can work. Choose from required routing/policy and measured connection/TLS capacity, then make the tier redundant. “WebSocket requires L4” and “L7 automatically understands every WebSocket failure mode” are both overstatements.

### 7.7 Reconnect storms

If a region or gateway tier fails, millions of clients may reconnect simultaneously. Defenses:

- exponential backoff with full jitter;
- per-account/device reconnect rate limits;
- admission control and `Retry-After`/server hints;
- staged DNS/traffic shifts;
- session resume instead of full history bootstrap; and
- reserve capacity for failure, not just steady state.

## 8. End-to-end message flow, ordering, and deduplication

### 8.1 Online one-to-one send

```text
Alice       Gateway A       Chat shard       Store/outbox       Fan-out       Gateway B       Bob
  | send(id=K) |                |                  |                |              |             |
  +----------->| auth/session   |                  |                |              |             |
  |            +-------------->| membership       |                |              |             |
  |            |               | dedupe K         |                |              |             |
  |            |               | assign seq=42    |                |              |             |
  |            |               +----------------->| durable commit |              |             |
  |            |               |<-----------------+                |              |             |
  | accepted   |<--------------+                  |                |              |             |
  |<-----------+               |                  +--------------->| lookup route |             |
  |            |               |                  |                +------------->| push frame  |
  |            |               |                  |                |              +------------>|
  |            |               |                  |                |              | device ACK  |
  |            |               |                  |                |<-------------+<------------+
  | delivered  |<-------------------------------------------------+              |             |
```

The sender can optimistically render the pending bubble immediately and reconcile it with `message_id`/sequence when accepted.

### 8.2 Where sequence comes from

All writes for one conversation route to one logical ordering authority at a time:

```text
hash(conversation_id) -> partition -> current leader/shard
```

That authority assigns increasing sequence numbers and appends them in order. Implementations include:

- a database partition leader with conditional/transactional increments;
- a replicated log partition keyed by conversation;
- a conversation actor/worker whose state is durably checkpointed; or
- time-sortable message IDs plus an explicit tie-break/order rule if strict gap-free sequence is not required.

A single global counter is unnecessary. Gap-free numbering is also usually unnecessary; stable total order matters more than consecutive integers.

### 8.3 Retry ambiguity and exactly-once appearance

Suppose the store commits, but the server ACK is lost:

```text
client send K -> server commit -> ACK lost -> client retries K
```

Without dedupe, two bubbles appear. With a uniqueness/conditional rule on `(sender_device_id, client_message_id)`, the retry returns the existing message. Consumers and clients should also dedupe by stable `message_id` because brokers and reconnect replay may redeliver.

### 8.4 Do not ACK before the durability contract is met

Possible ACK contracts:

| ACK point | Latency | Failure behavior |
|---|---:|---|
| Gateway memory | Lowest | Gateway crash can lose an “accepted” message; normally unacceptable |
| Single storage node | Low | Node loss can lose it |
| Replicated durable quorum/committed log | Higher | Strong default for accepted/sent |
| Recipient device ACK | Unbounded if offline | Wrong for sender acceptance; useful as delivered status |

State the choice. This design ACKs `accepted` after replicated durable commit, then sends delivery/read changes asynchronously.

### 8.5 The database/broker dual-write problem

This is unsafe:

```text
write DB succeeds -> process crashes -> publish broker never happens
```

Possible solutions:

- **transactional outbox:** commit the message and outbox record together; a relay publishes and marks progress;
- **change-data capture:** stream committed DB changes;
- **log as source of acceptance:** commit first to a replicated ordered log, then materialize storage; or
- a datastore/broker combination with a supported atomic transaction, after studying its limits.

Fan-out must be replayable. A temporary broker/delivery outage should increase delay/backlog, not erase accepted messages.

### 8.6 Slow recipient and gateway backpressure

Each connection gets a bounded outbound queue. When it grows:

1. coalesce replaceable events such as presence and typing;
2. stop loading more durable messages for that socket;
3. mark the device behind with a durable cursor;
4. disconnect if necessary; and
5. require sync/resume on reconnect.

An unbounded per-socket buffer lets one slow mobile client exhaust gateway memory.

## 9. Offline users, pending messages, and reconnect sync

### 9.1 “Offline queue” needs a durable definition

There are two broad models.

#### Model A: conversation log plus per-device/user cursor

```text
Conversation log: seq 1 2 3 4 5 6 7 ...
Bob delivered through:          ^ 5
Bob reconnects -> fetch > 5
```

**Strengths:** one canonical message copy, natural history, compact cursors.

**Costs:** finding all unread conversations may need an inbox index; group membership/history visibility must be handled carefully.

#### Model B: fan-out-on-write inbox entries

```text
message m42 -> inbox(Alice), inbox(Bob), inbox(Bob-laptop)...
```

**Strengths:** quick “what is pending for this user/device?” reads and independent delivery tracking.

**Costs:** write/storage amplification, huge-group fan-out, cleanup complexity.

**Recommended hybrid:** store content once in the conversation log; create compact inbox/update references or per-user stream entries for discovery/delivery. Do not copy a 10 MB attachment into every inbox.

### 9.2 Offline send flow

```text
1. Accept and durably append message.
2. Fan-out worker resolves recipient devices.
3. No live route is found, or socket delivery fails.
4. Leave/advance durable pending-inbox state; do not treat absence as loss.
5. Send a privacy-aware APNs/FCM notification as a wake/alert hint.
6. Device reconnects, authenticates, and presents last cursor.
7. Sync API returns ordered missing events.
8. Device stores them locally and ACKs a high-water mark.
```

Meta's 2023 WhatsApp device-verification post explicitly describes a client waking and retrieving an offline message from the WhatsApp server: [official engineering post](https://engineering.fb.com/2023/04/13/security/whatsapp-device-verification-protects-your-account/).

### 9.3 Why push is not the queue

Official platform behavior makes the limitation concrete:

- [Apple documents APNs as best effort](https://developer.apple.com/documentation/usernotifications/sending-notification-requests-to-apns), able to reorder, throttle, store only a limited pending notification set, or not deliver.
- [FCM documents bounded message lifespan and delayed delivery](https://firebase.google.com/docs/cloud-messaging/customize-messages/setting-message-lifespan); acceptance by FCM does not mean device delivery.
- [FCM also says delivery order is not guaranteed](https://firebase.google.com/docs/cloud-messaging/customize-messages/collapsible-message-types).

Therefore send “new content available; sync” or a safe preview. The client fetches authoritative ordered data from the chat service.

### 9.4 Per-device versus per-user progress

Keep both if product semantics need them:

```text
device delivered cursor -> what that device has durably received
user read cursor         -> what the account/person has marked read
```

If Bob reads on his phone, the system sends a read-cursor event to Bob's laptop and Alice. The laptop may still need the message content even though the user-level read cursor advanced.

### 9.5 Cursor design

A cursor can be:

- per conversation: `last_seq` for each active conversation;
- per user/device event stream: one monotonically ordered inbox sequence; or
- opaque token encoding/checkpointing multiple shards.

A single device stream simplifies reconnect (`fetch after device_event_seq=1234`) but creates fan-out references. Per-conversation cursors avoid a central user stream but need an index to discover which conversations changed. Large systems often combine them.

### 9.6 Retention and expired offline messages

Define policy:

- server-authoritative history: retain according to product/account policy;
- WhatsApp-like relay: delete ciphertext after every destination device receives it or after a bounded TTL;
- disappearing messages: expiry event/policy propagates to clients, with honest backup/screenshot limitations;
- account/device inactive too long: require a snapshot/history bootstrap or declare old relay messages expired.

Never let an offline delivery queue grow forever by accident.

## 10. Direct chat versus group chat

### 10.1 Share the domain model and ingress

Both types need authentication, membership authorization, deduplication, sequence assignment, durable append, receipts, and history. A common model avoids two incompatible products:

```text
send(conversation_id, client_message_id, payload)
                     |
                     v
              Conversation policy
              /                   \
        DIRECT/small group      large group/channel
        ordinary fan-out       specialized fan-out
```

Do **not** create separate direct-chat and group-chat gateway fleets merely because the UI has two tabs. Split services when scale, isolation, or different guarantees justify the operational cost.

### 10.2 One-to-one delivery

For a direct conversation:

- validate both current participants/blocking policy;
- append once to the conversation log;
- fan out to the recipient's active devices and the sender's other devices;
- update compact inbox/cursor state; and
- optionally generate one notification per recipient account, coalesced across devices.

The sender's other devices matter. Otherwise a message sent from phone does not appear on desktop.

### 10.3 Small groups: fan-out on write/delivery

For tens or hundreds of members, workers can enumerate members and create delivery tasks/references:

```text
one stored message
      |
      +--> member/device routes online
      +--> inbox references offline
      +--> coalesced push notifications
```

Partition the work into batches, make each batch idempotent, and checkpoint progress. A worker crash must not restart millions of duplicate notifications without dedupe.

### 10.4 Huge groups/channels: fan-out on read or hybrid

Copying one event to every member is expensive for a 100,000-member channel, especially if most members are inactive.

| Strategy | Write cost | Read cost | Best for |
|---|---:|---:|---|
| Fan-out on write | High: roughly O(members/devices) | Low | Small active groups |
| Fan-out on read | Low: append once | Readers fetch/merge | Huge/broadcast groups |
| Hybrid | Push active members, pull for inactive | Moderate | Mixed populations |

For a large channel, online subscribers may be organized by gateway or regional subscription shard. Publish one message per gateway/shard, then let it multicast locally, instead of one cross-service call per socket.

### 10.5 Hot conversation problem

Partitioning by `conversation_id` gives locality and order, but one viral channel becomes a hot key. Mitigations:

- isolate large channels into a separate workload class;
- batch fan-out and publish per region/gateway shard;
- cache recent history;
- use time buckets for storage;
- shard read delivery while keeping one logical order authority; and
- relax strict total order only if the product permits partitioned substreams.

Randomly salting writes removes the hot key but also destroys easy order. It is not a free fix.

### 10.6 Membership races

What if Bob leaves while Alice sends? Define a serialization rule using membership versions or sequence boundaries:

```text
Bob joined_seq = 100
Bob left_seq   = 500
Bob may read messages where 100 <= message_seq < 500
```

The send path authorizes against a committed membership version. The message records that version or sequence. This prevents an asynchronous fan-out worker from delivering a pre/post-membership message using a later, ambiguous member list.

### 10.7 When separate services become justified

Separate large-group/channel fan-out when it needs:

- different SLOs and overload isolation;
- different partitioning/storage;
- moderation before publication;
- hierarchical gateway multicast;
- substantially different retention/history rules; or
- independent capacity/deploy cadence.

Keep shared identity, conversation IDs, client protocol shapes, and common correctness libraries where possible.

### 10.8 Group delivery/read receipts can become another fan-out problem

For a three-person chat, showing exactly who read each message is cheap. In a 100,000-member channel, every reader emitting a receipt that is then broadcast to every member can exceed message traffic.

Options:

- disable individual read receipts above a group-size threshold;
- expose an aggregate count sampled/eventually updated;
- fetch the reader list only on demand with pagination;
- publish receipt deltas only to the sender/moderators; and
- store per-user high-water marks without broadcasting each change.

The receipt product requirement should change with the group workload class.

## 11. Database choice, partitioning, and replication

### 11.1 Start from access patterns

Before choosing a product, separate three terms:

- **Partitioning/sharding** divides different records among nodes, usually using a partition key, so capacity scales horizontally.
- **Replication** keeps copies of the same record in different failure domains so a node/AZ loss need not lose it.
- **SQL versus NoSQL** describes broad data models/interfaces, not “small versus large.” SQL emphasizes relational schemas/transactions; NoSQL includes key-value, document, and wide-column models with different query/consistency trade-offs.

Core queries are:

1. append a message to one conversation with idempotency and order;
2. fetch the latest/older messages for a conversation in sequence order;
3. find conversations changed for a user/device;
4. read/update membership and roles;
5. advance delivered/read cursors; and
6. expire/delete data by retention policy.

There is no common requirement to scan every message by arbitrary fields on the critical path. Full-text search belongs in a derived index when server-readable content permits it.

### 11.2 Sensible storage split

| Data | Good starting choice | Why |
|---|---|---|
| Users/devices/conversations/membership | Relational DB | Constraints, transactions, administration, clear relationships |
| Message history at moderate scale | Same relational DB, partitioned tables | Simpler operations and consistency |
| Message history at extreme scale | Wide-column/key-value/log-oriented store | High append throughput, horizontal partitions, range reads by conversation |
| Presence/routes | Ephemeral distributed KV/custom memory tier | TTL/lease-heavy, rebuildable state |
| Media | Object storage + CDN | Large immutable blobs, cheap capacity/egress path |
| Search | Search index populated asynchronously | Text/inverted-index queries; derived/rebuildable |
| Analytics | Event lake/warehouse | Keeps analytical scans off operational storage |

“Use NoSQL because millions of users” is weak. A partitioned SQL system may work well; a poorly keyed NoSQL system can hotspot and fail.

### 11.3 Message table for a wide-column store

```text
PRIMARY/PARTITION KEY: (conversation_id, time_bucket)
CLUSTER/SORT KEY:      conversation_seq

columns:
message_id, sender_id, client_message_id, kind,
payload/ciphertext, media_ref, created_at, membership_version
```

Benefits:

- append and range-read match the key layout;
- newest-page and backward pagination are efficient;
- time bucket bounds a partition's size; and
- old buckets can expire/archive as units.

Costs:

- reading across bucket boundaries needs application logic;
- an oversized/current hot bucket can still hotspot;
- membership/user inbox queries need separate tables/indexes; and
- cross-partition transactions are limited/expensive in many NoSQL stores.

### 11.4 Discord's concrete storage lesson

Discord's 2023 first-party snapshot reported storing trillions of messages in ScyllaDB after Cassandra. It partitioned messages by **channel plus a static time bucket**, used chronologically sortable Snowflake IDs, and replicated a partition across three nodes in the described setup. It also reported that hot channel/bucket partitions could raise latency for unrelated work on the same nodes: [How Discord Stores Trillions of Messages](https://discord.com/blog/how-discord-stores-trillions-of-messages).

This is valuable evidence for the pattern and its failure mode, not proof that every chat app should use ScyllaDB.

### 11.5 Relational design is a valid evolution path

An MVP can use:

```sql
messages(
  conversation_id,
  conversation_seq,
  message_id,
  sender_id,
  client_message_id,
  payload,
  created_at,
  PRIMARY KEY (conversation_id, conversation_seq),
  UNIQUE (sender_device_id, client_message_id)
)
```

Partition the table by hash/time when needed. Relational transactions make membership, dedupe, sequence, and outbox easier to reason about. Move only after measurements show a limit or operational/cost advantage.

### 11.6 Replication strategy

Within a region:

- keep replicas in different availability zones/failure domains;
- ACK only after the configured durable quorum/leader commit;
- use leader/fencing rules to avoid two sequence authorities; and
- test node/AZ loss while backlog and compaction are active.

Across regions:

- assign each conversation a home write region for order;
- asynchronously replicate history elsewhere for disaster recovery/local reads; or
- use a globally consistent database if its latency/cost fits.

Multi-leader writes to the same conversation create conflict and order problems. “Last write wins” by wall clock is usually a poor chat-ordering rule.

### 11.7 Data model variants and trade-offs

| Model | Strength | Weakness |
|---|---|---|
| One canonical conversation log | No content copies, easy history/order | Needs inbox/change index for users |
| Per-user inbox copy | Fast inbox and delivery progress | Write/storage amplification |
| Per-device queue | Precise device delivery | Explodes with devices; cleanup/relink hard |
| Event log + materialized views | Replayable, flexible consumers | More operational complexity and eventual views |

The usual answer is a canonical log plus compact derived per-user/device state.

### 11.8 Deletion, compaction, and tombstones

Deletes in LSM/wide-column stores often create tombstones that disappear later during compaction. Heavy expiry can hurt reads and compaction. Plan:

- bounded time buckets;
- TTL distribution/jitter where possible;
- archive/drop whole partitions instead of individual rows when semantics allow;
- tombstone/compaction monitoring; and
- legal/privacy deletion workflows across replicas, backups, caches, and indexes.

For the underlying comparison, see [10. SQL vs NoSQL](./10.%20SQL%20vs%20NoSQL.md).

## 12. Where to use a message queue, and where not to

A **message broker/queue** is a service-to-service buffer: a producer publishes work/events, the broker retains and routes them according to its contract, and consumers process and acknowledge them. It decouples component speed and failure. It is distinct from a user's durable chat history or “offline inbox,” though a retained log can participate in those designs.

### 12.1 What the broker does here

After durable acceptance, publish a `message_created` event for independent consumers:

```text
message_created
      |
      +--> online delivery/group fan-out
      +--> push notifications
      +--> inbox/unread materialization
      +--> search indexing (if allowed)
      +--> abuse/moderation pipeline
      +--> analytics/audit/metrics
```

This keeps slow or failed side effects off the sender's critical path and absorbs bursts.

### 12.2 What the broker should not be assumed to do

- It is not automatically the long-term message database.
- It does not automatically provide one queue per user efficiently.
- It does not make consumers exactly-once.
- It does not solve the database/publish dual write.
- It does not guarantee global order across partitions.
- It cannot substitute for client reconnect cursors unless retention and indexing explicitly support that use.

### 12.3 Partitioning broker events

Key events by `conversation_id` when per-conversation consumer order matters:

```text
partition = hash(conversation_id) % P
```

All events for one conversation then enter one broker partition in producer order. Different conversations process concurrently. One viral conversation can hotspot a partition, so large channels may need their own topic/tier or hierarchical fan-out.

### 12.4 Kafka-like durable log

**Good for:** high-throughput retained event streams, replay, multiple independent consumers, partition ordering.

**Costs:** partition planning, consumer lag, rebalances, hot keys, and more operational complexity than a simple work queue. Kafka's official design documentation describes an idempotent producer that prevents retry duplicates in the log under its supported scope: [Apache Kafka design](https://kafka.apache.org/41/design/design/). Application-level dedupe is still needed end-to-end.

### 12.5 RabbitMQ/work-queue style broker

**Good for:** explicit work dispatch, acknowledgements, routing patterns, per-job retries/dead-lettering.

**Costs:** millions of durable per-user queues can be operationally expensive; ordering under multiple consumers/requeue requires care; replayable history is not its primary abstraction.

Use shared sharded work queues/topics plus database/inbox state rather than creating a physical broker queue for every user by reflex.

### 12.6 Redis Pub/Sub versus Streams

[Redis documents Pub/Sub as at-most-once](https://redis.io/docs/latest/develop/pubsub/): an offline/disconnected subscriber loses the published item. That makes it suitable for replaceable live nudges only when durable truth exists elsewhere.

Redis Streams retain entries and support stronger consumer patterns. They can serve moderate durable pipelines, but memory/cost, persistence, trimming, failover, consumer recovery, and partition scale still need explicit design.

### 12.7 Failure and retry policy

Each consumer needs:

- stable event ID/message ID;
- idempotent side effect or dedupe record;
- retry with exponential backoff;
- poison-event quarantine/dead-letter process;
- lag/age monitoring;
- bounded retry policy; and
- a replay/reconciliation job against durable truth.

For online delivery, a duplicate event can cause a duplicate frame. The recipient client must upsert/dedupe by `message_id`.

### 12.8 Broker overload and graceful degradation

Prioritize:

1. durable message acceptance and history sync;
2. actual message delivery;
3. delivery/read receipts;
4. push alerts;
5. typing/presence;
6. analytics/link previews.

Drop/coalesce ephemeral events first. Do not let analytics backlog reject user messages unless the system's safety envelope truly requires it.

## 13. Heartbeats and presence

### 13.1 What a heartbeat solves

A heartbeat helps:

- detect half-open/zombie connections;
- keep NAT/firewall/LB state alive;
- refresh the route-directory lease;
- measure round-trip latency; and
- carry the last received sequence/cursor for resume diagnostics.

[RFC 6455](https://www.rfc-editor.org/rfc/rfc6455.html#section-5.5.2) defines WebSocket Ping/Pong control frames. An application can also define a higher-level heartbeat containing session progress.

### 13.2 Heartbeat flow

```text
Server Hello: heartbeat_interval = H
Client waits randomized offset
Client -> HEARTBEAT(last_received_device_seq)
Server -> HEARTBEAT_ACK(server_time)

No ACK for configured misses -> close, expire route, reconnect+resume
```

Discord's public Gateway documentation is a concrete example: it provides a heartbeat interval, uses jitter before the first heartbeat, ACKs heartbeats, and resumes a session using the last sequence after disconnect: [Discord Gateway](https://docs.discord.com/developers/events/gateway).

That documentation is for Discord's developer gateway; do not assume the consumer client uses identical constants or internals.

### 13.3 Choosing an interval

Short interval:

- faster failure detection and route cleanup;
- more frames, CPU, radio wakeups, and battery usage.

Long interval:

- lower overhead;
- slower offline detection and risk of intermediary idle timeout.

Tune by client type/network state. Foreground desktop, foreground mobile, and background mobile need not use the same policy. Ensure:

```text
LB/NAT idle timeout > expected heartbeat gap + jitter/tolerance
route lease TTL     > heartbeat interval * allowed misses
```

Do not copy a magic 30-second value without measuring and checking infrastructure timeouts.

### 13.4 Jitter prevents synchronization

If 50 million clients ping at `12:00:00`, the system sees a heartbeat tsunami. Randomize initial and reconnect timing while keeping the required maximum gap.

### 13.5 Presence is approximate

“Online” can mean:

- at least one valid connection lease exists;
- app is foreground/active;
- user interacted recently; or
- device is reachable for push.

These are not equivalent. Use a state machine such as:

```text
ACTIVE -> CONNECTED_IDLE -> SUSPECT -> OFFLINE
```

Presence can be eventually consistent and coalesced. A brief network switch should not spam every friend with online/offline events.

### 13.6 Presence fan-out is dangerous

Naively broadcasting every status change to every contact costs roughly:

```text
presence changes * contacts/subscribers
```

Meta's 2008 Facebook Chat post identified the same worst-case fan-out problem. Mitigate with subscriptions only for currently visible contacts/conversations, batching, debounce, privacy rules, and cached approximate states.

## 14. Why WhatsApp uses local history while Discord stores server history

### 14.1 First correct the premise

The shorthand is directionally useful but incomplete:

- WhatsApp clients keep and read delivered history from local databases, but WhatsApp servers still authenticate, map devices/keys, route over persistent connections, hold undelivered encrypted messages, and synchronize encrypted application state/history during device linking.
- Discord also uses local caches for performance, but its durable message history is server-authoritative and queryable across clients; Discord publicly describes a large server-side ScyllaDB message store.

### 14.2 WhatsApp's documented model

Meta's 2021 multi-device production snapshot says:

- every device has its own identity key;
- the server maps an account to device identities;
- a sender performs client fan-out, encrypting for destination devices;
- messages are not stored on the server after delivery; and
- a newly linked companion receives an encrypted recent-history bundle, then reads history from its own local database.

Source: [How WhatsApp enables multi-device capability](https://engineering.fb.com/2021/07/14/security/whatsapp-multi-device/).

Meta's 2024 interoperability description also says WhatsApp uses a client/server architecture, pushes messages over persistent connections, uses Signal Protocol for E2EE, and has an XMPP-based client protocol with optimized stanzas: [official interoperability engineering post](https://engineering.fb.com/2024/03/06/security/whatsapp-messenger-messaging-interoperability-eu/).

These are dated public descriptions; private implementation details can evolve.

### 14.3 Discord's documented model

Discord's 2023 engineering post says it migrated trillions of server-stored messages from Cassandra to ScyllaDB. Messages were partitioned by channel plus time bucket; history is durable server-side: [Discord engineering](https://discord.com/blog/how-discord-stores-trillions-of-messages).

Discord's public Gateway API uses persistent WebSockets, sequence numbers, heartbeat ACKs, and session resume/replay: [Discord Gateway documentation](https://docs.discord.com/developers/events/gateway). This is direct evidence for the developer-facing gateway, not a complete disclosure of consumer-client internals.

### 14.4 Product trade-off table

| Dimension | Device-centric/local-history emphasis | Server-authoritative history emphasis |
|---|---|---|
| Privacy/data minimization | Server retains less delivered content | Server retains content/ciphertext for policy duration |
| Multi-device | Requires device fan-out/history transfer/key sync | Any authorized device can query central history |
| Offline reinstall/new device | Backup or device transfer is important | Login plus server sync is straightforward |
| Server-side search | Hard/impossible on E2EE plaintext | Easy for plaintext; encrypted search remains hard |
| Moderation/compliance | Relies more on metadata/reports/client evidence | Server can enforce/search content if it can decrypt |
| Storage cost | More client storage/bandwidth complexity | Large provider storage/retention cost |
| Editing/deletion consistency | Must propagate across devices/backups | Central history simplifies canonical state |
| User expectation | Private messenger | Community/workspace knowledge archive |

WhatsApp optimizes private person-to-person messaging and privacy; Discord communities treat channels as durable shared spaces where a user can join/reconnect and read server history. Product semantics lead to different storage architecture.

### 14.5 E2EE and server storage can coexist

The server can store:

```text
message metadata + ciphertext
```

while authorized clients hold decryption keys. Meta's 2023 Messenger design explicitly describes server-based storage for encrypted message history with keys under user control: [Building end-to-end security for Messenger](https://engineering.fb.com/2023/12/06/security/building-end-to-end-security-for-messenger/).

So the true decision axes are:

- who holds decryption keys;
- how long ciphertext remains;
- how new devices receive keys/history;
- what metadata the server sees;
- how recovery/backups work; and
- which server-side features are possible.

### 14.6 Local database is still useful in every design

Even a server-authoritative product benefits from an encrypted client database for:

- instant startup and scroll;
- offline reading/composing;
- retry queue with client message IDs;
- search cache or local E2EE search;
- last sync/read cursors; and
- reduced repeated history downloads.

“Local versus server DB” is rarely exclusive. The crucial question is which copy is authoritative and recoverable.

## 15. Media attachments and calls

### 15.1 Do not stream media through chat application servers

```text
Client --request upload--> Media service
Client <-signed URL/key---- Media service
Client ===== upload bytes directly =====> Object storage
Client --send message(media_ref, hash, key?)--> Chat service
Recipient <-message event---------------------- Chat service
Recipient ===== download via CDN/object ======> Media path
```

Benefits:

- chat gateways remain optimized for small events;
- resumable/multipart upload handles large files;
- object storage provides cheap durable blobs;
- CDN reduces origin/region egress; and
- media processing/virus scanning can scale independently when encryption policy permits.

### 15.2 Media authorization

Use short-lived signed URLs or an authenticated media proxy. Validate that the requester is/was authorized for the conversation and respect membership/retention policy. An unguessable object key alone is not durable authorization.

For E2EE media, the client can encrypt before upload and send the content key inside the E2EE message. The server can scan metadata/size/behavior but cannot scan plaintext without changing the privacy model.

### 15.3 Calls are a parallel subsystem

The chat WebSocket carries:

```text
call_offer, call_answer, ICE candidates, ringing, reject, hangup
```

WebRTC carries audio/video using direct paths when possible, TURN when relaying is required, and usually an SFU for efficient multi-party forwarding. Calls require separate bandwidth, regional relay/SFU capacity, congestion, NAT traversal, QoS, and abuse design.

## 16. Multi-region design

### 16.1 Route users near themselves, conversations to an ordering home

```text
Alice India -> India gateway ----\
                                Conversation home/leader (chosen region)
Bob Europe  -> Europe gateway ---/       |
                                         +--> replicated history/events
```

Gateways should be close to clients for connection latency. Conversation writes need one logical ordering authority; cross-region forwarding is sometimes the price of strict per-conversation order.

### 16.2 Choosing the home region

Possibilities:

- creator's region;
- majority/weighted member locality;
- hash of conversation into available regions;
- organization/tenant data-residency region; or
- dynamically migrated home with a fenced handoff.

Migration needs an epoch/term so the old and new leader cannot both accept sequence numbers.

### 16.3 Region failure choices

| Choice | Behavior | Trade-off |
|---|---|---|
| Wait for home recovery | Preserves simple order | Write unavailability |
| Promote async replica with fencing | Faster recovery | May lose acknowledged data unless replication contract prevented it |
| Synchronous cross-region quorum | Stronger durability | Higher normal latency/cost |
| Accept independently in two regions | High write availability | Conflict/order reconciliation; often wrong for chat |

State whether `accepted` promises regional or multi-region durability. Do not claim both low global latency and synchronous global durability for free.

### 16.4 Connection relocation

On regional evacuation:

- traffic manager directs new/reconnecting clients elsewhere;
- session/device cursor is portable;
- route-directory leases expire in the failed region;
- clients use backoff/jitter;
- remote gateways fetch durable missed events; and
- capacity reserve absorbs the shifted connections.

## 17. Security, privacy, and abuse

### 17.1 Threats to include in HLD

- stolen/replayed access tokens;
- revoked device retaining a connection;
- unauthorized group send/history fetch;
- spam, scraping, contact discovery abuse;
- oversized/decompression-bomb payloads;
- malicious files and links;
- message replay/duplication;
- enumeration of conversation/media IDs;
- metadata leakage in logs/notifications;
- compromised encryption-key directory; and
- denial of service through sockets, heartbeats, fan-out, or reconnect storms.

### 17.2 Controls on the critical paths

- TLS for client-server and service-to-service traffic.
- Short-lived access tokens bound to account/device/session where practical.
- Authorization against current membership for send, history, media, and receipt actions.
- Per-user/device/IP/conversation rate limits and group-size quotas.
- Strict payload/frame/file size limits; schema validation.
- Idempotency keys and anti-replay windows.
- Signed media access with short expiry.
- Device revocation propagated to gateways.
- Encryption of stored data; E2EE when required.
- Audit events without plaintext secrets/content in ordinary logs.

### 17.3 E2EE architectural consequences

The server can still see some metadata required to route: accounts/devices, timing, message size, conversation membership, and delivery state, unless advanced privacy techniques hide portions. The server cannot safely provide plaintext search, link previews, content moderation, or recovery unless those occur on clients or the key model explicitly enables them.

Key management becomes a subsystem:

- device identity/public-key directory;
- prekeys/session establishment;
- group sender keys/key epochs;
- verification/key transparency;
- membership-triggered rotation; and
- secure new-device/history recovery.

Do not invent an encryption protocol in an interview. State that a reviewed protocol such as Signal is used and focus on key/device/product consequences.

### 17.4 Notifications and logs leak content

Even with E2EE, a plaintext lock-screen preview or log line can expose a message. Let users choose previews; avoid content in analytics/traces; redact tokens, phone numbers, keys, and message bodies.

### 17.5 Moderation versus encryption is a real trade-off

Options include:

- user-initiated reporting that uploads selected content/context;
- client-side safety checks;
- behavioral/metadata abuse signals;
- rate/forward limits;
- blocking and invitation controls; and
- server-side scanning only for products where server-readable content is part of the declared model.

There is no architecture that gives unrestricted server plaintext moderation while also truthfully claiming the server cannot decrypt content.

## 18. Failure modes, edge cases, and common mistakes

### 18.1 Failure matrix

| Failure | User-visible risk | Recovery/design |
|---|---|---|
| Gateway dies | Sockets drop | Reconnect with jitter, authenticate, resume cursor |
| Route directory loses entry | Online device appears offline | Durable pending state + push; heartbeat/reconnect rebuilds route |
| Store unavailable | Cannot meet durable acceptance | Reject/defer send; keep client pending and retry idempotently |
| ACK lost after commit | Sender retries | Dedupe by client message ID; return existing result |
| Broker unavailable | Delivery/indexing delayed | Outbox/backlog; accepted message remains durable |
| Fan-out worker crashes mid-group | Missing or duplicate tasks | Idempotent batches and checkpoints |
| Push provider delays/drops | No timely alert | Reconnect/sync is authoritative |
| Slow client | Gateway memory growth | Bounded buffer, coalesce, disconnect/resume |
| One storage partition hot | Tail latency/cascade | Buckets, isolation, hierarchical fan-out, admission control |
| Region fails | Many disconnects and home unavailable | Failover policy, reserve capacity, fenced promotion |

### 18.2 Simultaneous messages

Alice and Bob send concurrently. Their clocks are irrelevant to canonical order. Both reach the conversation authority; it assigns sequences 42 and 43. UI may first show Alice's local pending message elsewhere, then reconcile to server order without duplicating it.

### 18.3 Client clock skew

Keep `client_created_at` for UI/debug context if desired, but use server-assigned sequence/time for canonical ordering and retention. A malicious/faulty client must not place a message years in the past/future.

### 18.4 Duplicate live and replay delivery

A device may receive `m42` live, disconnect before ACK persistence, then receive `m42` again during sync. Upsert by `message_id`; advancing a cursor is monotonic and idempotent.

### 18.5 Receipt reordering

`read_seq=100` may arrive before a delayed `delivered_seq=90`. Store high-water marks using `max(old, new)` and enforce `read <= visible/authorized sequence`, rather than moving state backward.

### 18.6 User blocked or removed during queued fan-out

Recheck the correct policy/version at delivery or encode membership boundaries when the event was accepted. Decide whether already-accepted messages remain deliverable after a block/removal; this is a product/security requirement.

### 18.7 New group member and old history

Options:

- no history before `joined_seq`;
- last N messages;
- all retained history; or
- administrator-configured.

This affects authorization, encryption keys, storage queries, and membership records. Do not assume.

### 18.8 Deletes and offline devices

An offline device may later receive both create and delete events. Apply in sequence. If deletion retention expires before the device returns, a snapshot/tombstone policy must prevent resurrecting old cached content.

### 18.9 Common interview mistakes

1. Drawing a WebSocket and declaring the design complete.
2. Saying WebSocket guarantees delivered/read or durable exactly-once messages.
3. ACKing before durable replicated acceptance.
4. Using server timestamps as the only ordering mechanism.
5. Requiring global order across unrelated chats.
6. Omitting client idempotency IDs and retry ambiguity.
7. Treating APNs/FCM as the offline source of truth.
8. Storing one permanent `user -> chat server` row without lease/expiry or multi-device support.
9. Using one physical load balancer.
10. Assuming sticky sessions eliminate reconnect/resume.
11. Keeping unbounded gateway send buffers.
12. Writing a database row for every heartbeat/presence twitch.
13. Putting media bytes through the WebSocket/chat service.
14. Creating a broker queue for every user without capacity analysis.
15. Using Redis Pub/Sub for durable offline delivery.
16. Dual-writing DB and broker without an outbox/log.
17. Choosing NoSQL before stating access patterns/consistency.
18. Partitioning only by conversation and ignoring a viral hot channel.
19. Fan-out-on-write to millions of channel members.
20. Splitting direct/group into separate stacks before identifying a different scaling need.
21. Ignoring sender's other devices.
22. Calling presence strongly accurate.
23. Claiming E2EE means servers cannot store ciphertext.
24. Claiming local storage means WhatsApp is P2P.
25. Omitting abuse, membership races, revocation, and reconnect storms.

## 19. Trade-offs, alternatives, and an evolution path

### 19.1 Major decision table

| Decision | Chosen baseline | Choose the alternative when |
|---|---|---|
| Live transport | WebSocket | SSE/long poll for simpler one-way/fallback; polling for infrequent updates |
| Text topology | Client-server | P2P only when offline history/central policy is intentionally absent |
| Calls | WebRTC + signaling | Traditional telephony/streaming stack if interoperability/broadcast dominates |
| Send API | WebSocket command, HTTP fallback | HTTP-only if send rate is low and only receive needs a stream |
| Authoritative history | Server-side durable log plus client cache | Device-centric relay for strict data minimization/privacy product |
| Ordering | One authority per conversation | Relax for huge public chat if total order is not worth hot-key cost |
| Direct/group model | Shared conversation API/data model | Specialized large-channel fan-out behind the abstraction at scale |
| Small-group fan-out | Fan-out on write/delivery | Fan-out on read for huge mostly inactive audiences |
| Message store | SQL initially; wide-column at demonstrated scale | Stay SQL if it meets cost/latency/operations |
| Event delivery | Durable broker after acceptance | Direct synchronous call only at small scale with reconciliation |
| Presence | Expiring approximate lease | Stronger state only if a business workflow truly depends on it |
| Multi-region writes | Conversation home region | Global-consistent DB if its latency/cost is acceptable |

### 19.2 Availability versus strict order

If a conversation's leader/home is partitioned from a user, the system can:

- reject/delay sends to preserve a single order; or
- accept in multiple places and later reconcile conflicts.

Private chat generally prefers a clear order and no phantom acknowledgement. A live public comment stream may tolerate partial order for availability. State the product decision instead of saying “AP system” for every component.

### 19.3 Latency versus durability

ACKing from memory is fast but can lie. Cross-region synchronous replication is safer but slow. A common middle ground is synchronous multi-AZ durability in the home region and asynchronous DR replication, with the exact acknowledged-loss risk stated.

### 19.4 Server history versus device history

Choose server-authoritative history when users expect seamless multi-device login, long-lived community archives, server search, and centralized compliance. Choose device-centric history when minimized server retention and E2EE privacy dominate, accepting more complex backup, device linking, key fan-out, and recovery.

### 19.5 Fan-out on write versus read

The deciding variables are not just group size:

- active-member fraction;
- device count;
- read frequency;
- acceptable join/reconnect latency;
- push policy;
- message rate and celebrity skew; and
- whether each recipient needs personalized filtering.

A 10,000-member highly active work channel may justify different treatment from a one-million-follower broadcast where only 0.1% are online.

### 19.6 Simplicity versus specialized infrastructure

For a new product, prefer:

```text
one deployable app (HTTP + WebSocket)
    + relational DB
    + object storage
    + managed push provider
```

Then evolve from measurements:

```text
Stage 1: single app, SQL, local/outbox worker
Stage 2: redundant gateways + external route/presence store
Stage 3: separate chat/fan-out workers + durable broker
Stage 4: partition message history + conversation routing
Stage 5: regional gateways/home-region writes + DR
Stage 6: isolate huge channels, hierarchical fan-out, specialized storage
```

Kafka, Cassandra/Scylla, custom sequencers, and dozens of services on day one increase failure modes before they solve a demonstrated bottleneck.

### 19.7 When to avoid WebSocket

Avoid it as the only mechanism when:

- events are very rare and seconds/minutes of latency are fine;
- a platform/network cannot maintain long-lived sockets reliably;
- the client runs only briefly in background;
- server-to-client is one-way and SSE is operationally simpler; or
- a webhook is more appropriate for server-to-server delivery.

Even with WebSocket, retain HTTP history/sync and push-notification integration.

### 19.8 When to avoid a durable queue on the critical path

Do not add a broker merely to draw another box. If the message database plus transactional outbox and a small worker satisfies scale, that may be safer. Introduce a broker when buffering, replay, independent consumer scaling, or isolation provides measurable value.

## 20. Real-system examples and what they teach

The public sources below reveal only selected behavior/snapshots. Architectural lessons marked “inference” are reasoned applications, not claims about private current implementations.

### 20.1 WhatsApp: private, device-aware messaging

**First-party snapshot/behavior:** Meta describes Signal-based E2EE, persistent client-server connections, an account-to-device identity mapping, client fan-out to devices, temporary offline server messages, and local message history after encrypted device linking. Its 2024 interoperability post described infrastructure already handling over 100 billion messages/day; this is a **dated first-party scale statement, not a current 2026 capacity claim**.

```text
sender device --encrypted device copies--> WhatsApp servers
                                            |         |
                                      online push   temporary offline relay
                                            |         |
                                      recipient devices -> local DB
```

Sources: [WhatsApp multi-device architecture (2021)](https://engineering.fb.com/2021/07/14/security/whatsapp-multi-device/) and [messaging interoperability architecture (2024)](https://engineering.fb.com/2024/03/06/security/whatsapp-messenger-messaging-interoperability-eu/).

**Lesson:** privacy and device/key semantics shape fan-out, history, backup, and recovery. Local history does not remove central routing infrastructure.

### 20.2 Discord: durable community/channel history

**First-party production snapshot:** Discord reported migrating trillions of messages to ScyllaDB, using channel-plus-time-bucket partitions and chronologically sortable IDs. It explicitly discusses hot partitions and operational issues in the prior Cassandra design.

```text
channel message -> durable server history (channel, bucket, ordered ID)
                      |
                      +--> live gateway events
                      +--> later history reads by authorized members
```

Source: [How Discord Stores Trillions of Messages (2023)](https://discord.com/blog/how-discord-stores-trillions-of-messages).

**Lesson:** community history and cross-device access favor a server-authoritative log. Correct partition keys can still have hot-key and maintenance failure modes.

### 20.3 Meta Messenger: ordered user updates and independent consumers

**First-party production snapshot:** Meta's 2014 Iris design described a totally ordered per-user queue of messaging updates with separate progress pointers for the mobile app and long-term storage. If an app was offline or disk was slow, one pointer could stop while the other advanced; recent data came from the queue/backing store and older history from the storage tier.

Source: [Building Mobile-First Infrastructure for Messenger](https://engineering.fb.com/2014/10/09/production-engineering/building-mobile-first-infrastructure-for-messenger/).

**Lesson:** a user/device update stream plus independent cursors is a powerful model for real-time delivery, offline recovery, and materialized storage. This historical custom system is not a command to build a proprietary queue.

### 20.4 YouTube Live Chat: broadcast fan-out and moderation

**Public product behavior:** YouTube Live Chat supports real-time messages during streams/Premieres, replay with archived streams, rate limits, slow mode, blocked-word/potentially inappropriate message handling, and moderator actions: [YouTube Live Chat help](https://support.google.com/youtube/answer/15270973?hl=en) and [moderation help](https://support.google.com/youtube/answer/9826490?hl=en).

**Illustrative inference:** this is not ordinary group chat. A few hot stream IDs can have enormous read fan-out, most viewers never write, moderation can precede visibility, and “Top chat” may be a filtered view rather than one identical feed. Use hierarchical regional/gateway multicast, aggressive rate limits, and a replay log; isolate it from private-chat fan-out.

### 20.5 Uber rider-driver chat: scoped identity and lifecycle

**Public product behavior:** Uber documents in-app rider/driver messaging after a trip is accepted, notifications for messages, read-aloud behavior for drivers, and protection of personal information: [Uber Help](https://help.uber.com/driving-and-delivering/article/contacting-a-rider?nodeId=4f6cfc84-ae65-4292-ac8b-abd96d165271).

**Illustrative inference:** make the conversation ACL/lifecycle derive from `trip_id`: only matched participants and support systems can use it, templates/voice-safe quick replies reduce driving distraction, notification is important, and retention/access after trip completion follows a different policy than friend chat.

```text
trip state accepted -> create/enable scoped conversation
trip active         -> messages + push + safety controls
trip ended          -> restrict new sends; retain per policy/support needs
```

### 20.6 Netflix: the useful counterexample is separating control from bytes

Netflix is not a general chat product. Its relevance is the **data-plane separation lesson**: Netflix publicly separates application/cloud control work from large video delivery through Open Connect, its purpose-built CDN: [Netflix Open Connect overview](https://openconnect.netflix.com/Open-Connect-Overview.pdf).

**Illustrative chat inference:** keep small real-time chat metadata/control events on gateways, but send large image/video/file bytes through object storage/CDN. Scaling 1 KB messages and 100 MB videos with the same servers is inefficient.

### 20.7 Twitter/X Chat: encrypted content can remain server-stored

**Current public product statement:** X's notice effective April 23, 2026 says its newer Chat feature encrypts message content before it leaves the sender device and while it is stored on X infrastructure; a PIN protects private-key storage in the described design: [X EECC notice](https://help.x.com/en/rules-and-policies/eecc-notice).

**Lesson:** E2EE/server encryption and server-retained ciphertext are compatible. The hard questions shift to key recovery, device authorization, metadata, revocation, and what a PIN-protected key model promises.

### 20.8 Amazon Buyer-Seller Messaging: conversation context is authorization

**Public product behavior:** Amazon describes Buyer-Seller Messaging as a way to resolve order issues and answer product/order questions, with conversation lookup via order/buyer context: [Amazon seller guidance](https://sell.amazon.com/blog/amazon-order-management).

**Illustrative inference:** key the conversation to `order_id` or a support case, enforce buyer/seller/support roles, attach templates and translation as asynchronous/enrichment features, and apply commerce audit/retention policies. This is not an open social graph where any user may message any other.

### 20.9 Cross-system comparison

| Product/workload | Dominant pressure | Design lesson |
|---|---|---|
| WhatsApp | Private device-to-device semantics and offline relay | Device/key fan-out, local history, temporary ciphertext |
| Discord | Durable community history | Server log, channel/time partitions, hot-key awareness |
| Messenger/Iris | Ordered sync to independent destinations | Per-user update stream and cursors |
| YouTube Live Chat | Few extremely hot broadcasts | Hierarchical fan-out, moderation, rate limits |
| Uber | Trip-scoped, safety-sensitive chat | Business object drives ACL and lifecycle |
| Netflix analogy | Huge media bytes | Split real-time control from object/CDN data plane |
| X Chat | E2EE plus stored ciphertext | Encryption is about keys, not absence of storage |
| Amazon | Order-scoped buyer/seller communication | Contextual authorization, retention, audit |

## 21. Operations, monitoring, and validation

### 21.1 Gateway signals

- active connections and new connections/s by region/AZ/gateway;
- connection acceptance/rejection and TLS/auth latency;
- heartbeat rate, missed ACKs, zombie detection;
- reconnect/resume success rate and reconnect storm indicators;
- event-loop lag, CPU, memory, file descriptors, NIC saturation;
- outbound buffer bytes and slow-client disconnects; and
- connection distribution/skew and drain duration.

### 21.2 Message-path signals

- acceptance latency and error rate;
- dedupe hit rate and conflicting client IDs;
- sequence assignment/contention by conversation shard;
- durable write latency, quorum failures, replication lag;
- outbox age and unpublished count;
- broker consumer lag/oldest-event age;
- accepted-to-live-delivery and accepted-to-device-ACK latency; and
- per-conversation/group fan-out backlog and hot-key skew.

### 21.3 User-correctness signals

- duplicate message render reports;
- missing/gapped sequence detection;
- receipt cursor regressions rejected;
- offline sync duration, pages, and failures;
- push sent/provider-accepted versus later sync/delivery—not provider acceptance alone;
- membership authorization failures; and
- expired/deleted content resurrection.

### 21.4 Storage and queue health

- partition sizes and hot partition latency;
- compaction/tombstone/backlog pressure;
- disk/storage growth versus retention forecast;
- replica repair/rebalance health;
- dead-letter counts and poison-event reasons;
- search/materialized-view divergence; and
- backup restore/reconciliation test results.

### 21.5 Tests that matter

1. Kill a gateway with live sockets; verify jittered reconnect and gap-free deduped resume.
2. Commit a message then drop its ACK; verify retry returns the same message.
3. Stop the broker after DB acceptance; verify outbox later delivers.
4. Crash a group fan-out worker halfway; verify complete, idempotent recovery.
5. Delay/reorder receipts and live/replay events.
6. Fill one client's outbound buffer while others remain healthy.
7. Drive one viral conversation without harming ordinary chats.
8. Lose an AZ/region under peak connections and backlog.
9. Revoke a device with a live socket.
10. Add/remove a group member concurrently with sends and history fetch.

## 22. Interview angle

### 22.1 What the interviewer is usually testing

The boxes are secondary. The interview probes whether you can:

- clarify chat semantics before choosing technology;
- distinguish live transport from durable delivery;
- reason about stateful connections and stateless/replayable correctness;
- define ACKs, idempotency, order, offline sync, and multiple devices;
- choose data keys from access patterns;
- handle group fan-out and hot keys;
- use queues without losing accepted messages;
- discuss failures, backpressure, privacy, and product trade-offs; and
- adapt when the interviewer narrows the problem.

### 22.2 A 40-45 minute structure

| Time | Focus |
|---:|---|
| 0-5 min | Scope, scale, group/device/history/E2EE semantics |
| 5-9 min | Back-of-the-envelope: sends, connections, fan-out, storage |
| 9-14 min | APIs/events/data model and delivery-state definitions |
| 14-23 min | WebSocket gateways, routing directory, core architecture |
| 23-31 min | One message end-to-end: dedupe, order, durable ACK, broker |
| 31-36 min | Offline reconnect and group fan-out/hot channels |
| 36-41 min | Database partitioning/replication and multi-region failure |
| 41-45 min | Trade-offs, bottlenecks, security, monitoring |

Follow the interviewer's signals. A narrow “WhatsApp Web reconnect” question may deserve more client/session state and less global storage design.

### 22.4 Answer spine: `C-H-A-T-S`

Use these cues to reconstruct the design instead of memorizing prose.

#### C - Clarify contract and capacity

- direct/groups/channels, devices, history, E2EE, media/calls;
- define accepted/delivered/read and per-conversation order;
- estimate sends/s, fan-out/s, sockets, heartbeat load, storage.

#### H - Hold connections, not durable truth

- HTTP for login/history/media; WSS for live events; WebRTC for calls;
- redundant regional LB and gateway fleets;
- leased `(user, device) -> gateway connection` mapping;
- heartbeat, bounded buffers, reconnect with jitter.

#### A - Accept safely

- authenticate and authorize membership;
- client idempotency key;
- route by conversation and assign sequence;
- replicated durable append plus outbox;
- sender ACK only after the stated durability contract.

#### T - Transfer, track, and tolerate offline

- broker-driven idempotent fan-out;
- active routes receive live frames;
- offline state remains in log/inbox cursor; push is only a hint;
- device ACK/read high-water marks;
- replay after reconnect, dedupe by message ID.

#### S - Scale and surface trade-offs

- SQL metadata, partitioned message log, object-store media, ephemeral presence;
- small-group push versus large-group pull/hybrid;
- conversation home region and replication/failure policy;
- E2EE/history/search/moderation trade-offs;
- hot keys, reconnect storms, broker lag, and observability.

### 22.5 Example opening answer

> “I’ll scope this to one-to-one and groups, text plus media references, multiple devices, offline sync, and accepted/delivered/read receipts; calls will be a WebRTC extension. Before drawing boxes I’ll define delivered as at least one recipient-device ACK and require order only within a conversation. I’ll estimate logical sends, device fan-out, concurrent sockets, heartbeat rate, and storage. My baseline uses regional WebSocket gateways with a leased device-route directory, a conversation-keyed chat service that deduplicates, orders, and durably appends, then a broker/outbox path for online fan-out, offline inbox state, push hints, and derived consumers.”

### 22.7 Follow-up answer cues

| Follow-up | Answer cues |
|---|---|
| ACK lost after commit | Stable client ID; unique dedupe key; return same message/sequence |
| Multiple devices | Route per device; content sync per device; define delivered any/all; user read cursor separate |
| Gateway dies | Durable truth elsewhere; lease expires; jitter reconnect; resume by cursor; client dedupe |
| WebRTC for text | Signaling/NAT/TURN still needed; both online; no durable history; group/multi-device cost |
| Queue per user | Operational cardinality; shared sharded topics; canonical log + compact inbox/cursor |
| Huge group | Isolate workload; fan-out-on-read/hybrid; gateway multicast; cached log; hot-key plan |
| Preserve order at hot key | One logical sequencer; shard fan-out/read, not order authority; batch; admit/limit |
| Offline sync | Durable conversation/user event log; opaque/high-water cursor; push hint; pagination/replay |
| Redis Pub/Sub | At-most-once; live nudge only; use durable log/Streams/broker plus DB |
| Mapping | Leased soft state per device; TTL/fencing; reconstruct on reconnect |
| Heartbeat | NAT/LB timeout, detection goal, battery, connection count, jitter, adaptive intervals |
| Region split | Home leader/fencing; preserve order or explicitly accept conflict; state durability/availability trade-off |
| Membership race | Membership version/join-left sequence; idempotent fan-out; product rule for accepted-before-leave |
| E2EE + history | Store ciphertext; client-controlled keys; new-device/key recovery is hard; metadata remains |
| Presence mismatch | Presence is approximate lease, not delivery guarantee; state definitions/debounce |

### 22.8 Red flags in a candidate answer

- starts with “Kafka + Cassandra + WebSocket” before clarifying semantics;
- never defines sent/delivered/read;
- ignores multiple devices and sender-device sync;
- claims exactly-once without retry/dedupe mechanics;
- makes push notifications authoritative;
- has no resume cursor or gateway-failure story;
- stores media in the message DB or pushes it through gateways;
- draws a single LB/broker/database without redundancy/failure domains;
- assumes group fan-out is the same at 10 and 10 million members;
- says E2EE, moderation, search, and server plaintext access all coexist with no key trade-off; or
- cannot explain why each partition key matches a query.

## 23. Compact revision sheet

### 23.1 Five concepts to retain

1. **A connection is transport; a durable ordered log is truth.**
2. **At-least-once delivery plus idempotency gives exactly-once appearance.**
3. **Route each device using leased soft state; replay after gateway failure.**
4. **Store once by conversation, track compact per-user/device cursors, and vary group fan-out by size/activity.**
5. **Privacy/history/product requirements determine local versus server storage and key ownership.**

### 23.2 Formula card

```text
average logical sends/s = DAU * sends/user/day / 86,400
peak sends/s            = average sends/s * peak factor
delivery events/s       = sends/s * average destination-device fan-out
connection RAM          = concurrent sockets * memory/socket
gateway count           = concurrent sockets / safe sockets/gateway
heartbeat/s             = concurrent sockets / interval seconds
logical storage/day     = messages/day * bytes/message
replicated storage      = logical storage * replication factor (+ other overhead)
group fan-out work      ~= messages/s * active destination devices/message
```

### 23.3 One-message checklist

```text
[ ] authenticate device
[ ] authorize membership/block/rate policy
[ ] dedupe client_message_id
[ ] route to conversation authority
[ ] assign per-conversation order
[ ] durably replicate message + publication intent
[ ] ACK sender with canonical ID/sequence
[ ] asynchronously fan out idempotently
[ ] live deliver or retain pending cursor/inbox reference
[ ] push only as a hint
[ ] accept device delivery/read high-water marks
[ ] replay/dedupe on reconnect
```

### 23.4 Architecture choice card

```text
text/events      -> WebSocket (HTTP fallback/history)
calls            -> WebRTC + signaling + TURN/SFU
message content  -> conversation-partitioned durable store
metadata/ACL     -> SQL/distributed SQL
presence/routes  -> TTL/lease store
media            -> object storage + CDN
async effects    -> outbox/change stream + durable broker
offline          -> durable log/inbox cursor + push hint
ordering         -> one logical authority per conversation
```

### 23.5 Design-review checklist

- Are requirements, group limits, device semantics, history, E2EE, and receipts defined?
- Are message rate, fan-out, connections, heartbeat, storage, media, and peak/skew estimated?
- Can any gateway/LB/broker/store node fail without acknowledged-message loss?
- Is retry safe at producer, consumer, and client display?
- Is per-conversation order preserved without a global bottleneck?
- Is the DB/broker publication gap closed?
- Can every client resume from a durable cursor?
- Are slow clients and reconnect storms bounded?
- Does the group strategy change for huge/hot channels?
- Do partition keys match history/inbox queries and bound partition size?
- Are media, presence, typing, search, analytics, and push kept off the critical path as appropriate?
- Are membership races, revocation, E2EE keys, abuse, deletion, and backups addressed?

## 25. Say-out-loud 30-second narrative

> “A chat system is a durable per-conversation log connected to users through a large fleet of stateful gateways. I would use HTTP for login, history, and media control, WebSockets for live text events, and WebRTC only for calls. Each send carries a client idempotency ID; the conversation shard authorizes it, assigns per-conversation order, durably replicates it with an outbox, and only then ACKs. Fan-out resolves every recipient device; online devices get live events, while offline devices catch up from a durable cursor and push is only a wake-up hint. I would use leased connection mappings, bounded buffers, small-group push versus large-group pull, partitioned history, and explicit failure, privacy, and multi-device semantics.”
