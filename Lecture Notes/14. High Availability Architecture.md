# High Availability Architecture: Active-Passive, Active-Active, Database Failover, and Disaster Recovery

> Beginner-first HLD notes expanded from the supplied bullets. The goal is to understand what “the service stays available” really promises, how redundancy and failover create that property, why state makes active-active difficult, what happens when a failed database returns, and where disaster recovery begins.

## Scope, assumptions, and confidence labels

I interpret the rough phrase **“99.99x uptime”** as the family of availability targets around **99.99% (“four nines”)** and **99.999% (“five nines”)**. `99.99x` is not itself a standard target; the exact percentage, measurement method, and time window must be stated.

I also separate three ideas that the bullets currently combine:

1. **High availability (HA):** keep the service usable through selected failures, usually by redundant live capacity and failover.
2. **Data resilience:** keep acknowledged data durable and correct through failures, and make corrupted or deleted data recoverable.
3. **Disaster recovery (DR):** restore acceptable business service after a large or exceptional incident, within explicit recovery-time and data-loss goals.

Labels used for company examples:

- **Verified public behavior** is supported by first-party documentation linked in the note.
- **Historical production example** comes from a dated company engineering article; it must not be assumed to describe the company’s entire current architecture.
- **Illustrative interview design** is a reasonable way to design a named product in an interview. It is not a claim about that company’s private implementation.
- Public interview experiences are **candidate-authored and unverified by the company**. They show realistic question shapes, not an official question bank.

## 1. Plain-language foundation

### 1.1 The one-server shop

Imagine a small online shop with one application server and one database:

```text
Customers
    |
    v
+-----------+       +-----------+
| App       |------>| Database  |
| Server A  |       | DB-A      |
+-----------+       +-----------+
```

It works until either box stops working:

- if `Server A` crashes, no request can reach the application;
- if `DB-A` crashes, the application might be running but cannot complete database-dependent work;
- if the network link, power supply, or data center containing both fails, both disappear together.

Each indispensable thing whose failure can stop the service is a **single point of failure (SPOF)**.

The first idea behind high availability is simple: **do not make the service depend on one copy of something that can fail.** Add another application server and another database copy, place them in separate failure domains, detect failures, and route work to a healthy copy.

```text
                         Failure domain / zone A
                     +---------------------------+
Customers            |  +--------+    +-------+ |
    |                 |  | App A1 |    | DB-A  | |
    v                 |  +--------+    +-------+ |
+----------------+    +---------------------------+
| Logical traffic|                 |
| entry / LB     |                 | replication
+----------------+                 |
    |                 +---------------------------+
    |                 |  +--------+    +-------+ |
    +---------------->|  | App B1 |    | DB-B  | |
                      |  +--------+    +-------+ |
                      +---------------------------+
                         Failure domain / zone B
```

But merely drawing two boxes is not enough. We must answer:

- Which copy is serving now?
- How is failure detected without mistaking a slow node for a dead one?
- Who redirects traffic?
- Is the second database current enough to take over?
- Could both databases accept conflicting writes?
- Can the remaining capacity carry all traffic?
- What happens to requests that were in progress?
- How does the recovered node safely rejoin?

Those questions are the real topic.

### 1.2 Availability in one sentence

**Availability is the fraction of intended service that users can successfully obtain during a defined measurement window.**

That wording deliberately asks four questions:

1. **Which service?** Login, watching a video, placing an order, and viewing old orders can have different targets.
2. **Which users or requests?** Does a slow response count as unavailable? Do invalid client requests count?
3. **What is success?** A `200` response containing wrong data should not be called available.
4. **Over what window?** A month, a rolling 28 days, a quarter, or a year?

Availability can be measured by time:

```text
time availability = usable time / total eligible time
```

or, more usefully for many online APIs, by requests or business events:

```text
request availability = successful eligible requests / all eligible requests
```

For a checkout service, a business-oriented indicator might be:

```text
successful valid checkout attempts completed within 2 seconds
-------------------------------------------------------------
all valid checkout attempts
```

A process being “up” is not enough if every checkout times out.

### 1.3 SLI, SLO, SLA, and error budget

These four terms are often compressed into “we need four nines,” but they have different jobs:

- **SLI — Service Level Indicator:** the measured number, such as the proportion of valid checkout requests completed correctly within two seconds.
- **SLO — Service Level Objective:** the target for that indicator, such as `99.99% over a rolling 28-day window`.
- **SLA — Service Level Agreement:** a customer-facing contract that may contain an availability commitment, exclusions, support terms, and financial/service-credit consequences.
- **Error budget:** the allowed failure implied by the SLO, `1 - SLO`; teams can use its burn rate to balance release speed with reliability work.

```text
measurement                target                     contract
SLI: 99.992%  ---------->  SLO: at least 99.99% ---> SLA: terms/remedy if promise missed
                                  |
                                  +-> error budget = 0.01%
```

An internal SLO is often stricter than the external SLA so the team has safety margin. Also read the denominator carefully: a provider’s SLA might exclude planned maintenance, invalid traffic, unsupported configurations, or failures outside its control. In an interview, define the user-centered SLI first instead of hiding a weak experience behind exclusions.

### 1.4 What “four nines” means

An availability target of `99.99%` allows an error fraction of:

```text
100% - 99.99% = 0.01% = 0.0001
```

Over a 365-day year:

```text
allowed unavailable time = (1 - 0.9999) × 365 × 24 × 60
                         = 52.56 minutes
```

The same target, measured over one million eligible requests, allows about 100 failed requests. Whether those failures are spread out or occur in one damaging incident is a separate product concern.

| Availability target | Informal name | Unavailability per 365-day year | Average unavailability per month* | Failed requests per 1,000,000 |
|---:|---|---:|---:|---:|
| 99% | two nines | 3 days 15 h 36 min | 7 h 18 min | 10,000 |
| 99.9% | three nines | 8 h 45 min 36 s | 43 min 50 s | 1,000 |
| 99.95% | — | 4 h 22 min 48 s | 21 min 55 s | 500 |
| 99.99% | four nines | 52 min 34 s | 4 min 23 s | 100 |
| 99.999% | five nines | 5 min 15 s | 26.3 s | 10 |
| 99.9999% | six nines | 31.5 s | 2.63 s | 1 |

\*Monthly values use `365/12` average days. A calendar month has a different duration. The arithmetic is exact to the shown rounding; an SLA may use a different formula, exclusions, or window.

Google’s SRE guidance calls `1 - SLO` an **error budget**. A 99.9% request SLO over one million requests therefore has a budget of 1,000 errors ([Google SRE Workbook](https://sre.google/workbook/error-budget-policy/)). The budget makes the target operational: releases and reliability work can be governed by how quickly the service is spending it.

### 1.5 “No single point of failure” is not “cannot fail”

Removing known SPOFs improves availability, but the system can still fail because of:

- a bad deployment sent to every replica;
- shared credentials expiring;
- a dependency or DNS provider failing;
- every replica receiving the same overload;
- a network partition isolating components;
- data corruption replicating everywhere;
- an operator applying the same incorrect configuration globally;
- capacity that was sufficient before failover but insufficient afterward.

The more accurate goal is:

> For each failure the design claims to tolerate, no single instance or failure domain should be able to stop the critical user journey, and the failover path must be tested.

### 1.6 Active-passive in ordinary language

Two environments exist, but only one is responsible for serving a particular workload at a time.

```text
normal:

Users ---> Active A ---> database A
             |
             +---- replicate ----> Standby B (not serving this workload)

after safe failover:

Users -X-> failed A
   |
   +-----------------------------> Promoted B
```

The standby may be:

- **cold:** mostly absent; restore/provision it after the incident;
- **warm:** running at reduced capacity and receiving data, then scale/promote it;
- **hot:** fully running and nearly ready, but not serving the relevant live workload.

Naming varies across vendors and teams. Always describe what is actually running, where writes go, and what activation requires instead of relying only on “warm” or “hot.”

### 1.7 Active-active in ordinary language

Two or more environments serve production work at the same time.

```text
                        +---> Active region A ---> state A
Users ---> global route-|
                        +---> Active region B ---> state B
```

For stateless application servers this is straightforward: any server can process a request. For writable state it raises a hard question:

> If both regions can change the same logical record while they cannot communicate, which result is correct?

Active-active compute does **not** automatically imply multi-writer data. A common and useful design is:

- application servers active in both regions;
- reads local to either region;
- each user, order, account, or shard has one write-owning region at a time.

That keeps much of the availability and latency benefit while avoiding arbitrary conflicts.

### 1.8 Disaster recovery in ordinary language

HA is the live machinery that absorbs expected failures. **Disaster recovery is the prepared way to restore service after damage exceeds that machinery or affects an entire site, region, control plane, dataset, or organization.**

DR asks two business questions:

- **RTO — Recovery Time Objective:** how long may this service remain below its required level after the incident?
- **RPO — Recovery Point Objective:** how far back in time may recovered data be? In other words, how much recent data loss can the business tolerate?

```text
last recoverable point          disaster             service restored
         |                         |                         |
---------+-------------------------+-------------------------+------> time
         <--------- RPO ---------->|<--------- RTO --------->
             potentially lost          recovery duration
```

RPO and RTO are **objectives**, not claims proved by drawing a second region. Replication lag, backup frequency, detection time, restore speed, traffic switching, dependency recovery, and human decisions determine what is actually achieved. Google’s DR planning guide uses these definitions and stresses that lower RPO/RTO normally cost more and add complexity ([Google Cloud Architecture Center](https://docs.cloud.google.com/architecture/dr-scenarios-planning-guide)).

## 2. Important vocabulary: related, but not interchangeable

| Term | Plain meaning | Example | What it does **not** guarantee |
|---|---|---|---|
| Availability | Users can obtain the intended service now | Checkout succeeds within its latency limit | That stored data will survive later corruption |
| Reliability | The system continues to perform correctly over time | Few incorrect or failed payment attempts | A specific recovery time unless defined |
| Fault tolerance | Continue correctly despite specified faults | One app instance dies; peers keep serving | Survival of every possible correlated failure |
| Resilience | Resist, adapt to, and recover from disruption | Degrade recommendations, preserve playback, then heal | Zero user impact |
| Redundancy | Extra capacity/copies exist | Three app instances in different zones | Correct failover, independent failures, or current data |
| Durability | Acknowledged data is not lost over the promised horizon | A committed order survives a node crash | The order API is currently reachable |
| Data integrity | Data remains correct and internally valid | No duplicated debit or broken ledger balance | Continuous access |
| Backup | An independent recoverable copy/version | Immutable snapshot plus transaction logs | Immediate failover; a restore takes time |
| Replication | Changes are copied to live replicas | Primary streams its log to a standby | Protection from a bad delete/corruption copied to every replica |
| HA | Architecture/operation for a high service-availability target | Multi-zone app and automatic DB failover | Recovery from every region-wide or logical disaster |
| DR | Recovery strategy for exceptional, large-impact incidents | Restore/fail over to another region under a runbook | Seamless normal operation |

### 2.1 Availability versus scalability

- **Scalability** asks, “Can capacity grow with load?”
- **Availability** asks, “Can service continue when something fails?”

The same horizontal replica fleet can help both, but neither implies the other:

- 100 servers behind one fragile database can scale reads but still have a database SPOF.
- two lightly loaded servers in separate zones can tolerate one server failure but may not handle a traffic surge.
- auto-scaling may add capacity eventually, but it cannot help during the boot-up interval if no spare capacity exists.

### 2.2 HA versus data resilience

Consider two bad designs:

```text
A. Available but unsafe:
   Both regions answer writes, silently overwrite conflicts,
   and return success. Uptime looks excellent; user data is wrong.

B. Durable but unavailable:
   Five independent backups preserve every byte,
   but restoring the only database takes eight hours.
```

A serious design gives availability **and** correctness requirements separately.

### 2.3 Replication versus backup

Replication is useful for fast failover, but it rapidly copies current state—including accidental deletion, bad application writes, or some forms of corruption. Backups preserve older recovery points and can be isolated or made immutable. Microsoft’s reliability guidance explicitly treats redundancy, replication, and backup as distinct techniques ([Microsoft Learn](https://learn.microsoft.com/en-us/azure/reliability/concept-redundancy-replication-backup)).

For important state, use both:

```text
Primary DB ---> live replica(s)       for HA / low RTO
     |
     +-------> versioned backup + logs for corruption / deletion / DR
```

## 3. Prerequisites explained inline

These concepts are necessary to understand HA architecture, so they are introduced here rather than deferred to “study next.”

### 3.1 Load balancer

A **load balancer (LB)** accepts traffic through a stable logical endpoint and selects a backend instance. It can remove an unhealthy backend from rotation.

```text
             +--> App 1
Client --> LB+--> App 2
             +--> App 3
```

The LB may operate at a transport level (connections) or application level (for example, HTTP paths and headers). For today’s topic, the crucial jobs are health-aware routing and hiding individual backend addresses.

### 3.2 Health check and heartbeat

A **health check** asks whether a component is ready to do useful work. A **heartbeat** is a repeated “I am alive” signal used by peers or monitoring.

Good readiness is deeper than “the process exists”:

```text
weak check:  TCP port accepts a connection
better check: process can serve its critical path and has required local state
too-deep check: every instance fails because one optional global dependency is slow
```

The check must distinguish:

- **liveness:** should this process be restarted?
- **readiness:** should new traffic be routed here?
- **dependency health:** which functions can this instance currently provide?

### 3.3 Failure domain and blast radius

A **failure domain** is a boundary inside which components can fail together: a process, host, rack, power unit, availability zone, region, cloud account, DNS/control plane, deployment group, or operator credential.

The **blast radius** is the users, traffic, data, or components affected by one fault.

Two replicas on one host protect against a process crash but not a host crash. Two hosts on one power circuit protect against one host but not that circuit. Redundancy must cross the failure boundary named in the requirement.

### 3.4 Stateless versus stateful service

A **stateless** application instance keeps no unique durable business state that is required on the next request. Any healthy peer can handle a retry, assuming shared state and authentication are reachable.

A **stateful** component owns changing information: a database, queue partition, session, file, lock, or stream offset. Failing it over requires transferring or reconstructing both data and ownership.

This is why duplicating web servers is easy while duplicating databases correctly is not.

### 3.5 Replica, leader, follower, and quorum

A **replica** is a copy of state maintained from a stream/log of changes.

- A **leader/primary** normally orders and accepts writes.
- A **follower/standby/secondary** copies that order and may serve reads.
- A **quorum** is a sufficiently large intersecting subset, often a majority, used to make one authoritative decision even if some nodes cannot communicate.

With three voting nodes, a majority is two. A 2-node side of a network split can continue making decisions; the isolated 1-node side must not independently elect another leader if the protocol preserves single ownership.

### 3.6 Synchronous versus asynchronous replication

Suppose the leader receives write `W1`.

**Synchronous acknowledgement path:**

```text
Client -> Leader -> Standby/quorum durable confirmation -> ACK client
```

The client waits for another failure domain. This raises write latency and can reduce write availability when enough replicas are unreachable, but it can protect acknowledged writes within the promised failure model.

**Asynchronous acknowledgement path:**

```text
Client -> Leader -> ACK client

                  ---- later ----> Standby
```

The client gets a faster acknowledgement and the leader may keep writing without a remote replica, but acknowledged writes still in the replication gap can disappear from the promoted standby if the leader is permanently lost.

“Synchronous” is incomplete unless the ACK point is specified: received in memory, written to an operating-system buffer, durably flushed, committed by a quorum, or applied and readable.

### 3.7 Network partition and the minimum CAP idea

A **network partition** occurs when required nodes cannot communicate reliably even though more than one of them may still be running.

If two sides can both accept a write to the same invariant while unable to coordinate, the system must either:

- reject/wait on at least one side to preserve one consistent order; or
- keep accepting on both sides and permit divergence that must later be reconciled.

That is the practical CAP tension needed for active-active design. “High availability means eventual consistency” is too broad: a quorum-based system can remain available for failures that leave a majority and preserve strong consistency, but it will reject unsafe writes on a minority side during a partition.

### 3.8 Idempotency for uncertain retries

If a request commits but its response is lost, the client cannot tell whether it happened. After failover it may retry. An **idempotency key** identifies the same logical operation so a retry returns/reconstructs the first result instead of applying the effect twice.

```text
POST /payments  Idempotency-Key: K
    -> commit payment P123
    -> response lost

retry with K
    -> find K/P123
    -> do not charge again
```

HA increases the need for retries. Without idempotency and reconciliation, failover can turn a transient outage into duplicate orders or payments.

### 3.9 Fencing and split brain

**Split brain** means two nodes or regions both believe they own the same exclusive role and accept conflicting work.

**Fencing** makes the old owner unable to keep acting before the new owner is allowed to take over. Examples include:

- revoke its storage lease;
- use a monotonically increasing leader epoch and reject commands from older epochs;
- remove network/storage access;
- power it off through an independent management path;
- require a majority-issued lease before writing.

Failure detection alone is not ownership transfer. “I cannot hear the primary” does not prove the primary cannot still reach clients or storage.

## 4. Why high availability is needed in system design

### 4.1 Failures are normal at scale

With many processes, machines, disks, network paths, deployments, and dependencies, some component will eventually be slow, unreachable, overloaded, or incorrect. HA changes the question from:

> “How do I stop every component from ever failing?”

to:

> “How does the user journey continue, degrade safely, or recover quickly when a particular component or failure domain fails?”

### 4.2 Business requirements differ by operation

An entire product rarely needs one uniform target.

| Example capability | Possible requirement shape | Sensible failure behavior |
|---|---|---|
| Browse catalog | Very high read availability; bounded staleness acceptable | Serve cache or slightly old replica |
| Add item to cart | High write availability; mergeable state | Queue/retry or merge per-item changes |
| Reserve last unit | Correctness dominates unrestricted write availability | Require authoritative owner/quorum; reject if uncertain |
| Charge payment | No double charge; durable audit trail | Idempotent command, one write owner, reconcile unknown outcomes |
| Recommendations | Helpful but non-critical | Omit or serve defaults |
| Admin analytics | Large delay may be acceptable | Restore later from backup |

The goal is **graceful degradation**, not pretending every optional feature is equally critical.

### 4.3 Availability is an end-to-end property

If a request must pass through five serial dependencies and each is independently 99.99% available, the idealized path availability is:

```text
0.9999^5 = 0.99950009999 ≈ 99.9500%
```

That is about 4.38 hours of unavailable time per 365-day year, not 52.56 minutes.

This multiplication is an illustrative model; real component failures are often correlated and SLO definitions differ. The lesson is sound: **a user journey cannot be more available than a mandatory serial dependency unless the design can bypass, cache, degrade, or otherwise tolerate that dependency’s failure.**

### 4.4 Redundancy helps only when failures are sufficiently independent

For two components each 99.9% available, if their failures were independent and either one could serve the request:

```text
combined availability = 1 - P(both unavailable)
                      = 1 - 0.001 × 0.001
                      = 99.9999%
```

But if both share the same power, deployment, database, certificate, quota, or operator action, independence is false. Correlated failure dominates. AWS’s Builders’ Library explicitly emphasizes minimizing correlated failures rather than merely adding copies ([AWS Builders’ Library](https://aws.amazon.com/builders-library/minimizing-correlated-failures-in-distributed-systems/)).

### 4.5 Repair speed matters as well as failure frequency

For a simple repairable component with binary up/down state, a traditional steady-state approximation is:

```text
availability ≈ MTBF / (MTBF + MTTR)
```

- **MTBF:** mean time between failures;
- **MTTR:** mean time to restore/repair service.

This is not a complete modern service SLI, but it reveals two routes to improvement:

- fail less often: safer changes, isolation, capacity, simpler dependencies;
- restore faster: detection, automation, failover, rollback, practiced runbooks.

## 5. Expand and correct the rough bullets

| Rough bullet | Correct, interview-ready expansion |
|---|---|
| “High Availability Architecture” | HA is an end-to-end property measured against a defined SLI/SLO and failure model. It needs redundancy, health detection, routing, safe state ownership, spare capacity, and tested recovery—not just multiple servers. |
| “Data Resilient” | Data resilience is related but separate: acknowledged writes need defined durability; corruption/deletion needs backups/PITR; conflicts and retries need correctness controls. A service can be available while its data is wrong. |
| “99.99x uptime” | State an exact target and window. `99.99%` is four nines and permits about 52m 34s/year; `99.999%` is five nines and permits about 5m 15s/year. Also define whether availability is time-, request-, or business-event-based. |
| “No single point failure” | Audit the whole critical path and common failure domains: DNS, LB, app, identity, database, queue, configuration, certificates, network egress, capacity, and deployment/control plane. Redundant copies that fail together do not remove the SPOF. |
| “Active Passive, Active Active” | Active-passive gives one owner and simpler conflicts but has detection/promotion/routing delay and idle-cost trade-offs. Active-active uses all sites and can reduce failover interruption, but state ownership, partition behavior, conflicts, capacity, and cost become harder. |
| “Single LB ??” | One **logical** LB endpoint is normal; one physical/process LB is a SPOF. Use a managed distributed LB or redundant LB nodes across failure domains, plus a resilient discovery/routing mechanism. Audit DNS, health checks, control plane, and backend capacity too. |
| “DR -> Disaster Recovery” | Correct. DR is a tested business recovery plan driven by RTO/RPO. It includes application, data, dependencies, routing, security/configuration, people/runbooks, validation, and failback. HA and DR overlap but are not synonyms. |
| “Multi Master, Conflict Resolution” | Usually say **multi-leader** or **multi-writer**. If the same record can be written in multiple places, choose conflict avoidance, synchronous ordering, deterministic merge/CRDT, domain resolution, rejection, or compensation. Last-write-wins converges but can discard intent. |
| “DB down then UP, lost updates” | Distinguish (1) acknowledged writes absent from an async standby after failover, (2) concurrent stale updates overwriting each other, and (3) a client losing the response and retrying. Each needs a different control. The old primary must be fenced and resynchronized, not simply returned to service. |
| “delay in going up” | Recovery time includes detection, decision, fencing, log replay/catch-up, promotion, routing/DNS/connection refresh, cache warming, capacity ramp, and validation. “Process started” is not “service ready.” |

## 6. The building blocks of an HA design

### 6.1 A multi-zone single-region baseline

```text
                         Region R1

          +-------------------------------------------+
Internet  |  distributed / redundant logical LB      |
   ------>|   node in zone A       node in zone B     |
          +-----------+-------------------+-----------+
                      |                   |
             +--------v-----+    +--------v-----+
Zone A       | App A1, A2   |    | App B1, B2   |       Zone B
             +--------+-----+    +--------+-----+
                      |                   |
                      +---------+---------+
                                |
                        +-------v--------+
                        | DB leader A    |
                        +-------+--------+
                                | replicated across
                        +-------v--------+
                        | DB standby B   |
                        +----------------+
```

This can tolerate selected instance and zone faults if:

- traffic entry really spans zones;
- health checks stop routing to bad targets;
- app instances are replaceable and do not keep irreplaceable local session state;
- the database’s acknowledgement and promotion rules meet the RPO;
- the surviving zone has sufficient app, network, database, and quota capacity;
- dependencies also tolerate the zone loss;
- failover behavior has been exercised.

It is not automatically protected from region loss, bad global deployment, account compromise, or logical data corruption.

### 6.2 Redundancy patterns

| Pattern | Meaning | Normal utilization | Failure tolerance intuition |
|---|---|---:|---|
| `N+1` | Enough capacity for demand plus one failed unit | Spare one unit | One unit may fail if remaining units carry load |
| `N+2` | Two spare units/failure allowance | More headroom | Better maintenance plus failure coverage |
| `2N` | A full duplicate environment | Roughly half or less if one side reserved | Either side can carry the full intended load |
| Sharded + replicas | Each shard has several copies | Capacity distributed | One copy failure per shard may be tolerated; a missing shard still breaks part of service |

These labels describe capacity/copies, not correctness or routing. A `2N` environment with the same broken deployment in both halves can still fail completely.

### 6.3 The availability control loop

```text
observe -> decide -> isolate/fence -> reroute/promote -> verify -> rebuild
   ^                                                        |
   +---------------- continuous monitoring -----------------+
```

Every step can fail:

- **observe:** health signal is missing, delayed, or misleading;
- **decide:** automation flaps or an operator delays declaration;
- **isolate/fence:** the old owner remains able to write;
- **reroute/promote:** DNS is cached, connections are pinned, or replica is behind;
- **verify:** shallow checks mark an unusable node healthy;
- **rebuild:** the system remains in a fragile one-copy state.

### 6.4 Static stability and failover capacity

Suppose two zones each handle 50% of traffic at 70% CPU. If one zone disappears, the other would need roughly twice its previous work—about 140% of its capacity. It cannot.

Options include:

- keep enough pre-provisioned headroom for one-zone loss;
- shed optional work;
- reserve capacity that can activate safely and quickly;
- make each cell smaller so one failure affects fewer users;
- scale up, but account for provisioning time and quota limits.

AWS calls a design **statically stable** when it continues through a dependency/AZ impairment without needing immediate control-plane changes or new capacity. Their guidance favors enough pre-provisioned capacity to survive a zone loss because launching resources during the incident may itself fail ([AWS Builders’ Library](https://aws.amazon.com/builders-library/static-stability-using-availability-zones/)).

## 7. Active-passive architecture in depth

### 7.1 Core shape

```text
                    health / failover controller
                              |
                              v
Clients -> traffic director -> ACTIVE site A
                 |                  |
                 |                  +-- writes --> DB-A
                 |                                  |
                 |                        replicate |
                 |                                  v
                 +--------------------> PASSIVE site B / DB-B
                                        no production writes
```

The defining property is **one active owner for the scoped workload**. A passive site can still:

- receive replicated data;
- run health checks and smoke tests;
- serve non-authoritative reads if explicitly allowed;
- perform backups or analytics;
- keep a minimal application footprint warm.

### 7.2 Cold, pilot-light, warm, and hot variants

| Variant | What exists before disaster | Activation work | Relative cost | Relative RTO | Typical risk |
|---|---|---|---:|---:|---|
| Backup and restore / cold | Backups, templates, perhaps no compute | Provision, restore, configure, validate | Lowest | Longest | Restore or dependency surprise |
| Pilot light | Core data/minimum critical services | Provision most app capacity and route | Low–medium | Long | Untested surrounding infrastructure |
| Warm standby | Full stack at reduced scale with current-ish data | Promote, scale, route, warm | Medium | Minutes-ish, design-dependent | Capacity ramp or lag |
| Hot standby | Full or near-full stack ready, not serving scoped writes | Fence, promote/route | High | Short | Split brain or stale standby |

The RTO words are relative, not guarantees. A well-automated restore can beat a poorly practiced “hot” failover.

### 7.3 Safe failover sequence

For a stateful active-passive service, use this mental sequence:

```text
1. DETECT     Is the active actually unhealthy for this user journey?
2. DECIDE     Does policy permit failover? Is the standby eligible/current enough?
3. FENCE      Prevent the old active from writing or serving as authority.
4. RECOVER    Replay logs / verify data / complete recovery on standby.
5. PROMOTE    Give standby a newer ownership epoch or leader role.
6. ROUTE      Shift new traffic; drain or expire old connections.
7. RETRY      Clients retry uncertain operations idempotently.
8. VERIFY     Check business transactions, errors, latency, and data integrity.
9. REBUILD    Create a new standby and restore redundancy.
```

Fencing precedes promotion when there is any possibility the old active can still operate. A third witness or majority-based coordination is often used because two nodes alone cannot safely distinguish “the peer is dead” from “the link between us is broken.”

### 7.4 Where failover time comes from

An approximate breakdown is:

```text
T_failover = T_detect
           + T_decide
           + T_fence
           + T_replay_or_catch_up
           + T_promote
           + T_route_and_client_refresh
           + T_warm_capacity
           + T_verify
```

Some stages overlap. The formula is a debugging checklist, not a standard equation.

Aggressive failure detection shortens downtime but raises false-positive risk. A false failover can itself cause split brain, cache loss, connection storms, or a needless outage. Use multiple signals, consecutive failures, and business-impact checks; tune detection to the failure and risk.

### 7.5 Failback is a separate operation

When site A returns, do not automatically send traffic back merely because a process is green.

After failover, site B contains new writes. Site A is now stale and may have a divergent history. A deliberate failback normally requires:

1. keep A fenced;
2. resynchronize A from current authority B;
3. validate data and capacity;
4. decide whether returning has business value;
5. perform a controlled role switch or keep B primary;
6. monitor and preserve a rollback path.

Unnecessary immediate failback creates a second risky event while the team is already recovering.

### 7.6 Benefits and costs

**Benefits**

- one write owner makes ordering and uniqueness easier;
- simpler mental model and conflict avoidance;
- passive capacity can cost less if cold/warm;
- failures can be contained to one active site.

**Costs**

- non-zero detection/promotion/routing delay;
- standby capacity may sit mostly unused;
- async replication may leave non-zero RPO;
- the standby can silently decay if it never receives real traffic;
- failover and failback are operationally sensitive;
- the promoted site must absorb full load.

### 7.7 When to prefer active-passive

- one global writer or strict ordering is natural;
- conflicting writes would be expensive or unsafe;
- the required RTO permits a controlled promotion;
- the workload does not justify full duplicate active capacity;
- compliance or locality requires one active data owner;
- batch consumers, auditing, or payments favor one processing owner.

## 8. Active-active architecture in depth

### 8.1 “Active-active” must name the layer

The phrase is ambiguous. A system can be:

| Layer | Possible design |
|---|---|
| Edge/CDN | Active-active globally for cached content |
| Regional load balancers | Both accept traffic |
| Stateless API servers | Both regions execute requests |
| Reads | Both regions serve local replicas |
| Writes | Only a home region owns each key, or both can write the same key |
| Queue consumers | Both process disjoint partitions, or one consumer group is active |
| Batch computation | Both compute independently, but only one publishes authoritative output |

Say exactly which work is active in each place.

### 8.2 Common routing models

```text
Latency routing:   user -> nearest healthy region
Weighted routing:  80% -> A, 20% -> B
Geo routing:       India -> A, Europe -> B
Home routing:      user/account/shard owner determines region
Overflow routing:  A until busy, then B
```

Routing needs health awareness, but existing long-lived connections do not instantly move when DNS or routing changes. Clients may need reconnect and retry logic.

### 8.3 Stateless active-active

```text
                 +--> Region A API fleet --+
Client -> global |                          +--> authoritative state
route            +--> Region B API fleet --+
```

This is the easiest form if any instance can reconstruct needed request state. It provides:

- capacity utilization in both regions;
- low latency through proximity;
- little or no application warm-up on region loss;
- routine exercise of both code paths.

But if both regions depend on one database in A, that database and its network path remain a SPOF/latency bottleneck.

### 8.4 Active-active with per-key ownership

```text
                 +--> Region A owns users/shards A-M
Global router ---|
                 +--> Region B owns users/shards N-Z

Both regions are active globally,
but each key has one write owner at a time.
```

During normal operation, users get local ownership where possible. On failure, ownership transfers with an epoch/fence. This avoids simultaneous writes to one key while using both regions.

Trade-offs:

- cross-region latency when a user is far from the home region;
- moving ownership safely is a distributed coordination problem;
- cross-key transactions spanning owners are harder;
- skew can create hot regions.

### 8.5 Fully multi-writer active-active

```text
Region A writer ---- async replication ---- Region B writer
       ^                                           ^
       |                                           |
    users A                                     users B

same logical key may be changed in both regions
```

This may preserve local writes through a partition, but it cannot wish away conflicting intent. The data model must define which operations:

- commute and can be merged;
- require one global order;
- can tolerate last-write-wins;
- must be rejected when coordination is unavailable;
- require compensation after a temporary over-commit.

### 8.6 Capacity after losing one active site

If two regions each run at 60% of their own maximum and split traffic evenly, one surviving region would be asked to handle roughly 120%. “Both are active” does not prove failover capacity.

Choose one:

- each region is provisioned for 100% of total failover load;
- surviving regions have explicit headroom plus tested auto-scaling time;
- traffic shedding protects critical operations;
- a third region shares the load;
- degraded SLOs are accepted during failover.

### 8.7 Benefits and costs

**Benefits**

- all environments continuously serve and are less likely to rot unnoticed;
- regional loss can require only traffic redistribution for stateless paths;
- local traffic can reduce latency;
- infrastructure is used during normal operation;
- can preserve selected operations through a partition.

**Costs**

- multi-region state, ordering, and conflict resolution;
- correlated deployment/configuration risk remains;
- cross-region replication bandwidth and egress cost;
- harder debugging, testing, capacity planning, and data residency;
- a global routing/control dependency can still fail;
- overprovisioning is still needed if a peer must carry failed load;
- consistency-sensitive workflows may need a single owner anyway.

### 8.8 When to prefer active-active

- very low regional-failover interruption is required;
- global users benefit materially from local reads/compute/writes;
- the state can be partitioned by owner or safely merged;
- both environments can carry failure load;
- the organization can operate and test the extra complexity;
- cost is justified by latency, availability, or business-continuity requirements.

## 9. Active-passive versus active-active

| Dimension | Active-passive | Active-active |
|---|---|---|
| Normal serving | One environment for scoped workload | Two or more environments |
| Traffic switch | Required on active failure | Often reweight/remove failed site |
| RTO potential | Depends on detection, promotion, warm-up, routing | Very low for stateless paths; state can still delay |
| Write conflicts | Easier to avoid with one active writer | Central design problem if same key is writable anywhere |
| Normal resource use | Passive capacity underused | Capacity serves normal work |
| Failure capacity | Passive must scale/carry full load | Survivors must absorb failed share |
| Operational exercise | Passive may drift unless tested | Both paths exercised continuously |
| Consistency | Easier single order | May require ownership, quorum, merge, or compensation |
| Cost | Cold/warm can be cheaper | Usually higher infra/data/operational cost |
| Failback | Role reversal and resync | Reintroduce and rebalance; state repair may still be needed |
| Best fit | Strict owner, controlled RTO, lower cost | Global latency, stringent RTO, partitionable/mergeable work |

Do not answer “active-active is better.” The correct choice follows the invariant, RTO/RPO, latency, partition behavior, capacity, cost, and operational maturity.

## 10. “Single LB?” — removing the traffic-entry SPOF

### 10.1 One logical endpoint can hide many LB nodes

This diagram is conceptually fine:

```text
Client ---> api.example.com / logical LB ---> app fleet
```

as long as the logical LB is implemented redundantly. For example, AWS documents that enabling an Availability Zone creates an Elastic Load Balancing node in that zone and recommends/for an Application Load Balancer requires multiple zones, allowing traffic to healthy targets in another zone ([AWS ELB documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html)).

The user sees one DNS name; the service behind it is not one process.

### 10.2 Self-managed active-passive LBs

One traditional design is:

```text
                       floating virtual IP
                               |
                    +----------+----------+
                    |                     |
               LB-A active <---heartbeat---> LB-B standby
                    |
               backend fleet
```

On failure, LB-B acquires the virtual IP after fencing/lease rules. This can be appropriate inside one network, but the heartbeat, shared network, VIP ownership, and failover coordinator must be designed carefully.

### 10.3 Active-active LBs

```text
                       DNS / anycast / upstream routing
                         /                         \
                     LB-A                           LB-B
                  zone/site A                    zone/site B
                       \                           /
                            healthy backend fleet
```

DNS may publish several addresses; anycast may advertise the same address from several sites; or an upstream provider may route to redundant nodes. These mechanisms have different convergence and failure behavior.

### 10.4 The LB questions interviewers actually care about

1. **Is it one logical or one physical LB?**
2. **How are LB nodes discovered?** DNS, anycast, virtual IP, provider fabric?
3. **What checks backend health?** Does the check cover real readiness?
4. **What happens to existing connections?** WebSockets and long polling may reconnect, not migrate.
5. **What happens when one zone fails?** Is the other zone reachable and large enough?
6. **Can a bad configuration take out all nodes?** Use staged changes and blast-radius isolation.
7. **Is the control plane required for existing traffic?** Prefer a data plane that keeps using last-known-good state if configuration service is impaired.
8. **Are DNS, TLS certificates, WAF, authentication, and rate limits also resilient?**

### 10.5 Health-check traps

- **False positive:** unhealthy backend remains in rotation and harms users.
- **False negative:** healthy backend is removed; remaining backends overload.
- **Cascading health failure:** every app’s check fails because one optional dependency is down, removing all capacity.
- **Flapping:** a borderline instance repeatedly enters/leaves rotation.
- **Shallow green:** `/health` returns `200`, but checkout queries fail.
- **Deep synchronized load:** health checks themselves overload a weak dependency.

Mitigations include consecutive-success/failure thresholds, jittered checks, readiness states, dependency-specific degradation, passive out-of-band probes, and business-transaction canaries.

### 10.6 DNS failover is not instantaneous

A DNS traffic director can remove an unhealthy site from new resolutions, but:

- resolvers and clients cache records according to TTL and sometimes longer;
- clients may keep existing TCP/TLS connections;
- some clients do not retry another returned address correctly;
- making TTL extremely small increases DNS query load and does not control every cache;
- the DNS service and domain configuration are themselves dependencies.

Therefore state the expected routing convergence and client retry behavior; do not claim “zero downtime because DNS switches.”

## 11. Data-layer high availability

Stateless app replicas can be replaced freely. Databases must preserve both **bytes** and **one authoritative history**. Data-layer HA therefore needs an explicit answer for:

1. where each write is accepted;
2. when the client is told the write succeeded;
3. which replicas may serve which reads;
4. who becomes leader after failure;
5. how an obsolete leader is fenced;
6. how replicas catch up or are rebuilt;
7. how backups recover from logical damage.

### 11.1 Single-leader replication

```text
                       write
Client ------------------------------> Leader A
                                        |      \
                              log/WAL   |       \ log/WAL
                                        v        v
                                    Follower B  Follower C

reads: leader for strongest freshness path;
       followers when their permitted staleness is acceptable.
```

Advantages:

- one place orders writes;
- uniqueness, conditional updates, and transactions are easier;
- no normal multi-leader write conflicts;
- followers can scale selected reads.

Costs:

- leader is a write bottleneck per shard;
- remote writers pay latency to the leader;
- failover has an election/promotion interval;
- follower reads can be stale;
- async failover can lose acknowledged writes;
- a stale old leader must be fenced.

Large systems normally shard the data and give each shard its own leader; “single leader” need not mean one database machine for the entire product.

### 11.2 Synchronous quorum / consensus-backed leader

With three voting replicas, the leader can acknowledge only after a majority has durably accepted the ordered record:

```text
Client -> Leader A -> A + one of {B,C} form a majority -> ACK
```

If A fails, a new majority containing a replica with every committed entry elects a new leader under the protocol’s rules. This can provide strong protection for committed data and single ordering, but:

- a region/partition without a majority cannot safely accept writes;
- every synchronous failure domain on the ACK path adds latency;
- overloaded or slow quorum members can hurt tail latency;
- election and replay still cause a short availability gap;
- the business operation still needs idempotency for lost responses.

Consensus protects the replicated log boundary. It does not make an email, bank call, or another independent database atomically happen exactly once.

### 11.3 Asynchronous primary/standby

```text
time --->

Leader A:    W1 commit, ACK ---- W2 commit, ACK ---- crashes
Standby B:       W1 arrives ----------- lag --------- promoted
                                      W2 absent
```

If A is permanently lost, `W2` is missing after promotion despite its client having received success. This is an **RPO/durability loss due to replication lag**, not the same as a concurrency lost update.

Asynchronous replication is reasonable when:

- low write latency or write availability is more important than RPO = 0;
- recent loss can be reconstructed from an external log/client retry/reconciliation;
- the business explicitly accepts the replication window;
- cross-region round-trip time is too expensive for the normal commit path.

### 11.4 Read replicas and lag

A follower can be alive yet behind:

```text
Leader version:   120
Follower version: 116
apply lag:        four log positions
```

User-visible problems include:

- create an order, immediately read “not found”;
- change a password, old credential still appears valid on a stale path;
- update profile, then see the old value;
- inventory appears available after it was sold.

Controls:

- route correctness-sensitive reads to the leader/linearizable path;
- use read-your-writes session routing;
- return a commit/version token and require a replica at least that current;
- wait briefly for replay or reroute when the replica is behind;
- define a maximum staleness SLO;
- remove lagging replicas from eligible read/failover sets;
- monitor both received-log and applied-log positions.

### 11.5 The acknowledgement contract

An interviewer may ask, “The write was replicated—are we safe?” Ask what replicated means.

| ACK condition | What has happened | Remaining failure risk |
|---|---|---|
| Leader memory | Process accepted bytes | Process/host loss may lose them |
| Leader local durable storage | One durable copy | Disk/site loss; no failover copy |
| Follower received in memory | Network copy exists | simultaneous process/power loss |
| Follower durably logged | Another durable failure-domain copy | protocol/election must preserve it |
| Majority committed | Protocol recognizes ordered majority record | majority/storage disaster, software/logical corruption |
| Replica applied | State is queryable there | application side effects elsewhere may still be incomplete |

Use the datastore’s actual contract; do not infer it from marketing terms like “sync” or “multi-AZ.”

### 11.6 Database proxy / service discovery after promotion

Applications need to find the current writer. Common approaches:

- a stable database proxy endpoint whose routing changes;
- service discovery updated by the failover controller;
- a virtual IP within one network;
- a driver aware of several hosts and leader role;
- a consensus/metadata service carrying the current epoch and owner.

Connection pools complicate failover: existing connections to the dead or demoted primary must fail fast, be discarded, resolve/discover the new endpoint, authenticate, and retry only safe operations. A DNS change alone does not rewrite open database connections.

### 11.7 Replica is not automatically eligible for promotion

A standby may be unsuitable because it is:

- too far behind the accepted RPO;
- missing required logs;
- in the same failed domain;
- running incompatible schema/software;
- corrupted or failing integrity checks;
- unable to sustain leader load;
- configured as a delayed replica intentionally;
- isolated from a quorum or witness.

Failover automation should choose from **eligible** candidates, not merely reachable ones.

### 11.8 Backups, point-in-time recovery, and delayed replicas

Live replicas target low RTO. For bad deletes, corruption, ransomware, or a buggy write replicated everywhere, also keep:

- versioned full/incremental backups;
- continuous transaction-log/WAL archive for point-in-time recovery (PITR);
- checksums and restore-time integrity validation;
- copies in an independent failure/security domain;
- immutable or write-protected retention where appropriate;
- a known-good recovery point and tested restore procedure.

A deliberately delayed replica can create a short safety window before a bad change reaches it, but it is not a substitute for independent backups.

## 12. Multi-master / multi-leader and conflict resolution

### 12.1 Terminology correction

**Master-master** is an older common phrase. Prefer:

- **multi-leader:** several replicas can order/accept writes;
- **multi-writer:** several sites can update the data;
- **leaderless:** clients/coordinators write to several replicas without a permanent leader.

The exact replication and conflict model matters more than the label.

### 12.2 How a conflict appears

Initial state:

```text
profile.city = "Delhi", version 7
```

During a network partition:

```text
Region A reads v7, writes city="Pune"     -> version A8
Region B reads v7, writes city="Mumbai"  -> version B8
```

Neither value causally follows the other. They are concurrent. When replication resumes, the system cannot infer the user’s intent from the strings alone.

### 12.3 First ask whether conflicts can be avoided

The safest conflict resolution is often **not creating the conflict**:

1. **Single write region:** all writes go to one region; others are read/standby.
2. **Home region per key:** each user/account/order has one owner.
3. **Shard ownership:** each partition has one leader; ownership moves with fencing.
4. **Synchronous quorum/order:** coordinate before acknowledging the write.
5. **Escrow/reservations:** pre-allocate independent quotas to regions, such as 40 seats to A and 60 to B.
6. **Append-only unique events:** regions add distinct immutable records rather than overwrite one shared value.

These designs trade some routing latency, partition-side write availability, or modeling flexibility for simpler correctness.

### 12.4 Conflict-resolution choices

#### A. Last-write-wins (LWW)

Choose the value with the greatest timestamp or ordered version.

```text
A8 timestamp 10:01:03.100 -> Pune
B8 timestamp 10:01:03.250 -> Mumbai wins
```

Benefits:

- deterministic convergence;
- low metadata and implementation complexity;
- acceptable for disposable or simple “latest setting” state.

Risks:

- silently discards one valid intent;
- wall clocks can be skewed;
- retry timing can make an older business intent appear newer;
- whole-object LWW can erase changes to unrelated fields;
- convergence does not mean business correctness.

AWS documents LWW for concurrent updates in eventually consistent DynamoDB global-table modes and warns that application-level conflict handling may be required ([DynamoDB concurrent-update guidance](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/BestPractices_ImplementingVersionControl.html)). This is a verified product example of a trade-off, not a universal recommendation.

#### B. Field-level or domain-specific merge

```text
Region A: shipping_address changed
Region B: notification_language changed

merged: keep both independent field changes
```

The application understands which fields can combine. For the same non-mergeable field, expose a conflict, apply precedence, or ask the user.

#### C. Set/operation merge and CRDT-style data

Some operations can be designed to commute:

- add unique element to a set;
- increment independent counter components and sum them;
- keep per-item cart quantities with deterministic merge rules;
- use a grow-only collection where deletion is not required.

A **CRDT** packages state/operation rules so replicas that receive the same updates can converge despite reordering and duplication. It does not automatically preserve arbitrary business invariants. A counter can converge while still allowing inventory to go below zero if every region sells independently.

#### D. Causal/version-vector conflict detection

Track enough version history to distinguish:

- update B follows A, so B supersedes A;
- update A and B are concurrent, so preserve both siblings for resolution.

This avoids pretending every event has a reliable global wall-clock order but adds metadata and resolution complexity.

#### E. Optimistic concurrency / reject stale writer

```text
UPDATE account
SET address = ... , version = 8
WHERE id = 42 AND version = 7
```

Exactly one competing update succeeds locally; the other receives a conflict and rereads/merges/retries. Across asynchronously independent regions, local version checks alone may not see a concurrent remote update, so ownership or a cross-region protocol is still needed for global protection.

#### F. Application compensation or reconciliation

Allow temporary divergence, detect invariant violations later, and repair the business outcome:

- oversold inventory -> refund/backorder one order;
- duplicate external action -> reconcile ledger and reverse one;
- conflicting calendar reservation -> notify and reschedule.

This preserves write availability but transfers complexity and user impact to recovery. It is acceptable only when compensation is defined and affordable.

### 12.5 Choose resolution per data type

| Data | Possible design | Why |
|---|---|---|
| User theme preference | LWW/direct state assignment | A lost intermediate preference is usually tolerable |
| “Liked video” set | Idempotent add/remove with explicit semantics | Repeated operations can be made safe; beware add-vs-remove race |
| Shopping cart | Per-item merge/domain rules | Different item edits can combine; quantity conflicts need policy |
| Document editing | Operation transform/CRDT/domain merge | Preserve concurrent user edits rather than discard one whole document |
| Bank ledger | Append-only uniquely identified entries + authoritative balance rules | Never resolve money by arbitrary wall-clock LWW |
| Last inventory unit | Single owner/quorum/escrow | Must enforce non-negative stock before accepting sale |
| View counter | Sharded counters + async aggregation | Exact immediate global total often unnecessary |
| Username uniqueness | One authoritative index or reservation protocol | Two regions cannot independently promise same unique name |

### 12.6 Conflict-resolution decision tree

```text
Can writes be routed to one owner without violating requirements?
  |
  +-- yes --> single owner + fenced failover
  |
  +-- no --> Must a global invariant be checked before ACK?
              |
              +-- yes --> synchronous coordination/quorum,
              |          escrow, or reject during partition
              |
              +-- no --> Do operations merge without losing intent?
                           |
                           +-- yes --> operation/domain/CRDT merge
                           |
                           +-- no --> expose/reject conflict,
                                      choose documented LWW,
                                      or compensate later
```

### 12.7 Common multi-writer mistake

“Use master-master so the database never goes down” is incomplete. The next interviewer questions are:

- Can both accept writes during a network partition?
- Can the same row/key be changed?
- How are IDs generated?
- How are uniqueness, foreign keys, stock, and balances enforced?
- Is replication synchronous or asynchronous?
- Which update wins, and could it erase acknowledged work?
- How is each site prevented from consuming more than global capacity?
- What happens when the partition heals?

If these are unanswered, multi-writer has moved the outage into a data-integrity incident.

## 13. What happens when the database goes down and then comes back?

There is no single answer; it depends on whether a new primary was promoted and whether any side accepted writes.

### 13.1 Case A: follower goes down, leader remains healthy

```text
Leader A ---> Follower B (down)
    |
    +-------> Follower C (still current)
```

If policy permits, A continues serving. B later:

1. restarts;
2. finds its last durable log position;
3. fetches and replays missing log entries;
4. remains ineligible for reads/promotion until sufficiently caught up;
5. rejoins the replica set.

If missing logs were already discarded, B needs a new base snapshot plus later logs. Catch-up can consume network, disk, and leader I/O, so throttle it or isolate repair traffic to avoid harming production.

### 13.2 Case B: primary goes down, no standby is promoted

The service may reject writes or all database work while waiting. When A returns, it performs crash recovery from its durable log, validates state, and resumes as the same primary.

This can preserve simple ownership but has RTO equal to repair/restart time. It is not high availability for a long primary outage.

### 13.3 Case C: primary goes down and standby is promoted

```text
before:
  A = primary, B = standby

failure:
  A unreachable
  controller fences A
  B promoted with epoch 12
  clients write W_new to B

recovery:
  A returns with old epoch 11 and stale/divergent data
```

A must **not** simply resume as primary. Safe rejoin:

```text
1. keep A fenced from clients and write storage
2. prove B/epoch 12 remains authoritative
3. find common history point
4. discard/rewind A's divergent old-primary timeline as policy requires
5. copy/replay B's current history into A
6. validate A
7. admit A as a follower
```

PostgreSQL’s official failover documentation explicitly warns that a restarted old primary must be informed it is no longer primary and describes fencing/STONITH to prevent both systems writing; it also describes rebuilding the standby after failover ([PostgreSQL failover documentation](https://www.postgresql.org/docs/current/warm-standby-failover.html)). PostgreSQL’s `pg_rewind` is one concrete tool for synchronizing an old primary after timelines diverge ([PostgreSQL `pg_rewind`](https://www.postgresql.org/docs/current/app-pgrewind.html)). These product details illustrate the general rule.

### 13.4 Case D: both sides accepted writes — split brain

```text
partition:
  A accepts WA1, WA2
  B accepts WB1, WB2

partition heals:
  histories diverged
```

Recovery now depends on the data model:

- consensus-based design should prevent one side from accepting committed writes without quorum;
- multi-leader design uses its conflict rules and repair protocol;
- single-leader system that accidentally split must identify authoritative history and reconcile orphaned business actions manually/with tooling;
- external side effects may require compensation even if database rows are merged.

Never “take whichever database has more rows” or blindly copy both ways. Ordering, uniqueness, deletes, and side effects need semantic reconciliation.

### 13.5 Three different meanings of “lost update”

#### Meaning 1: acknowledged write lost on failover

```text
Client -> A: W1
A ACKs before B receives W1
A is permanently lost
B promoted without W1
```

Controls: synchronous/quorum commit, lower accepted RPO, durable external log, or business reconciliation.

#### Meaning 2: concurrent stale writer overwrites a newer value

```text
X reads version 5
Y reads version 5
X writes version 6: address=Pune
Y blindly writes version 6: address=Mumbai  # X's update disappears
```

Controls: transaction locking, compare-and-set/version condition, serialization, or merge.

#### Meaning 3: response lost, client does not know outcome

```text
DB commits W1
response is lost
client thinks it may have failed
```

The update may be safe in the database but “lost” to the caller’s knowledge. Controls: idempotency key, operation-status lookup, transactional result record, and reconciliation.

Name the failure before proposing a fix.

### 13.6 Why a recovered database takes time to become ready

The delay may include:

- OS/process restart and storage attachment;
- crash recovery and log replay;
- downloading missing snapshot/log segments;
- integrity or consistency checks;
- schema/configuration validation;
- replica catch-up to an eligibility threshold;
- cache/buffer-pool warming;
- leader election or lease expiry;
- fencing confirmation;
- service discovery and DNS propagation;
- application connection-pool retry/backoff;
- rebuilding indexes or repairing replicas;
- load ramp and business-level verification.

Separate states explicitly:

```text
DOWN -> STARTING -> RECOVERING -> CATCHING_UP -> READ_ONLY_READY
     -> WRITER_ELIGIBLE -> SERVING -> FULLY_REDUNDANT
```

Routing writes to `STARTING` because its port opened is a common outage amplifier.

### 13.7 What happened to writes that arrived while the DB was down?

Possible contracts:

1. **Fail fast:** return `503`; client retries idempotently.
2. **Bounded in-memory wait:** risky if the app dies; suitable only for non-durable work.
3. **Durable queue/log first:** accept `202` after durably enqueuing; worker writes when DB returns.
4. **Fail over to authoritative replica:** preserve synchronous API if promotion succeeds.
5. **Serve reads/degraded mode only:** reject mutations but keep cached browsing.

Do not return success merely because an app put a critical write in local memory. Define whether the API means “accepted durably” or “business effect completed.”

### 13.8 Queue buffering is not infinite availability

A queue can decouple a temporary database outage:

```text
API -> durable queue -> DB writer -> database
```

But ask:

- How large can the backlog grow before storage/quota is exhausted?
- Is ordering required per account/key?
- Can workers process messages more than once?
- Are writes idempotent?
- Will the recovery burst overload the database?
- What latency does the user observe?
- What if the queue is also unavailable?

This changes the contract to asynchronous acceptance; it does not make every operation instantly complete.

## 14. Disaster recovery in depth

### 14.1 HA and DR overlap but cover different scopes

```text
HA examples                          DR examples
-----------                          -----------
process crash                        region/site loss
host failure                         widespread control-plane outage
one-zone impairment                  ransomware/account compromise
routine database failover            replicated logical corruption
rolling maintenance                  long dependency/provider outage
                                     catastrophic operator error
```

Some multi-region active architectures make regional recovery part of normal HA. The distinction is operational: DR is invoked when normal redundancy is insufficient or a coordinated recovery plan is needed.

### 14.2 Begin with business impact, not technology

For each critical capability define:

- maximum tolerable outage / target RTO;
- maximum tolerable recoverable-data gap / target RPO;
- minimum degraded functionality during recovery;
- data integrity and regulatory requirements;
- incident types covered;
- who may declare failover;
- priority/order of dependent service recovery.

Example:

| Capability | Illustrative RTO | Illustrative RPO | Degraded behavior |
|---|---:|---:|---|
| Browse product catalog | 5 minutes | 1 hour | Serve last good catalog |
| Place paid order | 15 minutes | Near zero for accepted payments | Pause checkout if ledger authority uncertain |
| Recommendations | 24 hours | 24 hours | Hide personalized rail |
| Finance audit export | 8 hours | 0 for ledger entries | Generate later; do not fabricate |

These numbers are **illustrative design inputs**, not claims about any named company. The business chooses them.

### 14.3 RPO is not always “backup every X minutes”

If a snapshot is taken every 24 hours but transaction logs are archived every minute, PITR may support a much lower RPO than 24 hours. Conversely, a replica reporting a few seconds of average lag may fall far behind during the exact disaster.

Measure achieved recovery points under stress and include:

- unshipped log backlog;
- failed/corrupt backups;
- cross-service consistency points;
- external systems not included in the snapshot;
- queue messages and object storage state;
- the time at which data was durably acknowledged, not merely received.

### 14.4 Common DR strategies

| Strategy | Compute before disaster | Data before disaster | Relative cost | Relative RTO | Best fit |
|---|---|---|---:|---:|---|
| Backup and restore | None/minimal | Independent backups/logs | Low | Long | Non-critical systems, corruption recovery |
| Pilot light | Critical core only | Replicating/current-ish | Low–medium | Long–medium | Infrastructure can be created predictably |
| Warm standby | Reduced full stack | Replicated | Medium | Medium–short | Business-critical workload |
| Hot active-passive | Full ready stack | Near-current replicated state | High | Short | Strict one-writer workload |
| Multi-region active-active | Both serve | Partitioned, synchronously replicated, or conflict-aware | Highest/complex | Potentially shortest | Stringent RTO/global latency |

Azure’s architecture guidance similarly presents active-active, active-passive, and passive/cold multi-region approaches as cost/RTO/RPO trade-offs and warns that dependent services must be included ([Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/web-apps/guides/multi-region-app-service/multi-region-app-service?tabs=paired-regions)). Exact achievable times are system-specific.

### 14.5 A complete DR plan covers more than servers and DB

Inventory and recover:

- compute images, containers, and deployment artifacts;
- databases, object storage, caches where needed, queues, search indexes;
- network, DNS, certificates, firewall/WAF, egress/NAT;
- identity, secrets, keys, and access to recovery accounts;
- configuration, feature flags, schemas, and service discovery;
- third-party endpoints and their regional restrictions;
- observability, alerting, incident communication, status page;
- data-processing offsets and in-flight workflows;
- legal/data-residency requirements;
- people, roles, contact lists, and authorization to declare DR.

A DR environment that cannot retrieve secrets or publish DNS is not ready.

### 14.6 The DR runbook

A usable runbook is executable under pressure:

```text
1. Confirm incident and activation criteria.
2. Freeze risky deployments/changes.
3. Establish incident command and communication.
4. Decide authoritative data/recovery point.
5. Fence damaged/old writers.
6. Restore or promote dependencies in order.
7. Validate integrity and critical business transactions.
8. Shift traffic gradually; watch saturation/error/burn rate.
9. Reconcile in-flight and external operations.
10. Operate in DR mode with explicit limitations.
11. Rebuild redundancy.
12. Plan controlled failback; preserve evidence and postmortem actions.
```

Automation shortens time and reduces typing mistakes, but irreversible decisions—such as choosing an older recovery point—may require human authorization.

### 14.7 DR drills prove the claim

Test at several levels:

- restore a random backup into an isolated environment;
- verify checksums, row counts, invariants, and sample business queries;
- simulate instance, zone, and region failures;
- fail over while under representative load;
- disconnect cross-region replication to test partition behavior;
- rotate on-call staff through the runbook;
- measure achieved RTO/RPO rather than marking “test passed”;
- test failback and re-protection, not just failover;
- test corrupted data and compromised credentials, not only clean power loss.

AWS’s reliability principles explicitly include automatic recovery, horizontal distribution, change automation, and testing recovery procedures rather than assuming they work ([AWS Well-Architected Framework](https://docs.aws.amazon.com/wellarchitected/2024-06-27/framework/rel-dp.html)).

### 14.8 DR can fail because the disaster is replicated

Examples:

- delete table in primary; replication deletes it in secondary;
- bad deployment corrupts records in both active regions;
- compromised global admin deletes replicas and backups;
- one shared encryption key becomes unavailable;
- configuration automation pushes an empty routing file globally.

Mitigations include isolated recovery accounts, immutable/versioned backups, least privilege, multi-person controls for destructive actions, delayed recovery copies, staged rollouts, and offline/independent documentation.

### 14.9 Failover versus failback

- **Failover:** move service away from the failed/damaged authority.
- **Failback:** later return to an original or newly preferred environment.

Failback needs its own RPO/integrity plan because DR site now contains post-failover writes. Many teams sensibly keep the DR site as current primary until the original is fully resynchronized and a scheduled switch is safe.

## 15. A systematic “no SPOF” audit

Start from a critical user journey and draw every mandatory dependency.

```text
User
 |
 DNS
 |
 CDN/WAF/LB
 |
 API gateway/auth
 |
 Order service ----> inventory service ----> inventory DB
 |
 payment service --> external processor
 |
 order DB
 |
 queue --> fulfillment worker
```

For each node and edge ask:

1. What if one process dies?
2. What if its host/rack/zone/region dies?
3. What if the network is one-way, slow, or partitioned?
4. What if it returns incorrect responses rather than failing closed?
5. What if it is overloaded?
6. What if its configuration/credentials expire?
7. What if all replicas receive one bad change?
8. What if a retry duplicates work?
9. What if recovery capacity is unavailable?
10. What state must be rebuilt, and from where?

### 15.1 Often-forgotten SPOFs and correlated dependencies

- authoritative DNS or registrar account;
- certificate issuance/renewal;
- identity provider or token-key distribution;
- one NAT gateway/egress path;
- one message-broker partition/leader;
- one configuration/feature-flag service;
- one schema registry;
- one time source if leases/timestamps rely on clocks;
- one storage bucket/account/region;
- one database proxy or service-discovery cluster;
- one deploy pipeline with global reach;
- one global rate limiter;
- one secret or encryption key;
- one third-party API;
- one on-call expert/runbook held only in the failed system;
- quotas and IP/subnet capacity needed during failover;
- one hot shard/key even when the fleet is large.

### 15.2 Control plane versus data plane

- **Data plane:** handles live user traffic/data operations.
- **Control plane:** changes configuration, membership, routing, or provisioning.

Prefer **static stability** where possible: the data plane keeps serving with last-known-good configuration when the control plane is impaired. If every request asks a central configuration service which backend exists, that control service becomes a request-path SPOF.

### 15.3 Cells reduce blast radius

Instead of one enormous global deployment, divide users into self-contained **cells**:

```text
Global directory
   |
   +--> Cell 1: LB + app + cache + DB shards (users 1-1M)
   +--> Cell 2: LB + app + cache + DB shards (users 1M-2M)
   +--> Cell 3: LB + app + cache + DB shards (users 2M-3M)
```

A cell failure affects its assigned subset rather than every user. The directory/routing layer must itself be resilient, and cross-cell workflows become more complex, but containment can be more valuable than trying to make one giant system perfect.

### 15.4 Avoid shared fate in deployments

Redundant infrastructure does not help if one release breaks all copies simultaneously. Use:

- canary or one-cell/one-zone rollout;
- health and business-metric gates;
- automatic rollback with human oversight;
- backward-compatible database changes;
- independent failure-domain batches;
- feature flags with safe defaults;
- last-known-good configuration;
- sufficient time to observe slow-burn errors.

## 16. Graceful degradation and overload protection

HA does not require every feature to work fully through every incident. Preserve the critical path and shed optional work.

### 16.1 Example degradation ladder

```text
Level 0: full service
Level 1: stale cache for catalog; recommendations disabled
Level 2: reads available; new writes queued or selectively rejected
Level 3: only account/payment status and safety-critical operations
Level 4: explicit maintenance/error response; no false success
```

The order must be agreed with product/business teams before the incident.

### 16.2 Mechanisms

- timeouts so failed dependencies do not occupy every thread;
- bounded retries with exponential backoff and jitter;
- circuit breakers to stop repeatedly calling a known-bad dependency;
- bulkheads: separate thread pools/queues/resources per dependency or tenant;
- load shedding and admission control;
- priority queues for critical work;
- cached/default responses for optional reads;
- asynchronous acceptance when the contract permits;
- rate limits and per-tenant quotas;
- backpressure so producers do not overwhelm slower consumers.

Retries can improve transient availability but can also multiply an outage. If every layer retries three times across three layers, one user request can create up to `3 × 3 × 3 = 27` downstream attempts in the worst simple model. Choose one appropriate retry boundary, cap attempts, add jitter, and make effects idempotent.

### 16.3 Fail open versus fail closed

When a dependency is unavailable:

- **fail open:** allow the action, preserving availability but risking policy/invariant violation;
- **fail closed:** reject the action, preserving safety but reducing availability.

Examples:

| Dependency | Likely choice | Reason |
|---|---|---|
| Optional recommendations | Open/omit | Safety unaffected |
| Security authorization for money movement | Closed | Unauthorized action is worse than temporary rejection |
| Best-effort abuse limiter for public reads | Possibly open with local limits | Preserve access while bounding damage |
| Exact inventory reservation | Closed if authority unavailable | Avoid oversell unless compensation is an accepted policy |

State the business consequence, not a universal rule.

## 17. Observability, SLOs, and testing

### 17.1 Measure what users experience

Useful signals include:

- good/eligible requests by endpoint and region;
- p50/p95/p99 latency and timeout rate;
- SLO error-budget burn rate;
- healthy/ready capacity by failure domain;
- saturation: CPU, memory, connections, queue depth, disk/network;
- database replication receive/apply lag;
- leader election/promotion count and duration;
- stale-read or minimum-version wait rate;
- conflict, duplicate, and reconciliation rate;
- retry amplification and circuit-breaker state;
- backup freshness, success, and verified restore age;
- achieved RTO/RPO from drills;
- time spent in degraded or single-replica state.

### 17.2 Alert on symptoms and causes

- **Symptom alert:** users cannot checkout or latency SLO is burning rapidly. This pages the service owner.
- **Cause alert:** one follower is lagging or one disk is filling. This may create a ticket/page depending on urgency.

Paging on every failed pod creates noise; ignoring user symptoms because “all processes are green” misses outages.

### 17.3 Test matrix

| Fault | Expected behavior | What to verify |
|---|---|---|
| One app process killed | LB removes/replaces it | No material SLO impact; connection behavior |
| One zone isolated | Other zone carries load | Capacity, dependency placement, no retry storm |
| Primary DB killed | Safe promotion | RTO, committed-write RPO, fencing, pool recovery |
| Old primary returns | Rejoins only as follower | No split brain; correct timeline resync |
| Replica lag injected | It becomes ineligible | Freshness routing and promotion policy |
| Cross-region partition | Defined C/A behavior | No forbidden invariant or silent conflict |
| Queue redelivery | One intended effect | Consumer idempotency and ordering |
| Bad config canary | Rollout stops/rolls back | Blast radius and last-known-good state |
| Region removed under peak load | Survivors stay within SLO/degraded SLO | Static headroom, quotas, scaling delay |
| Corrupt/delete test data | Restore selected point | Backup isolation, integrity, measured RPO/RTO |

Use a controlled test environment and safety boundaries. Chaos engineering is an evidence-gathering method, not random production destruction.

### 17.4 Availability review questions before launch

- What exact SLI/SLO and window apply?
- What failure domains are in scope?
- Where are the current SPOFs?
- What does a write ACK promise?
- Which requests are safe to retry?
- How much replication lag is allowed?
- What triggers and authorizes failover?
- How is the old owner fenced?
- Can surviving capacity handle loss under peak load?
- What degraded modes exist?
- What is backed up, where, and when was it last restored?
- What are achieved—not merely target—RTO and RPO?
- How is failback performed?
- Which global changes can defeat all redundancy?

## 18. Real-system examples

The examples below deliberately separate public evidence from interview illustration. Do not present an illustrative diagram as a fact about a company’s current private system.

### 18.1 WhatsApp-style chat — illustrative interview design

Requirements that drive HA:

- users keep long-lived connections;
- a message accepted as durable should survive a gateway crash;
- offline recipients receive the message after reconnecting;
- duplicates are preferable to silent loss only if clients can deduplicate;
- presence/typing indicators may be lost or stale without losing chat history.

```text
                          global / regional routing
                         /                         \
                 Region A gateways          Region B gateways
                   /     |                       |     \
              Conn A  Conn B                 Conn C  Conn D
                         \                       /
                          chat/user shard owners
                                  |
                         replicated message log
                          /                  \
                 delivery workers       offline inbox
```

HA reasoning:

- **Connection gateways:** active-active and disposable; the mobile client reconnects to another healthy gateway.
- **Session state:** store only rebuildable connection mapping in memory; keep durable message state elsewhere.
- **Message ID:** client/server assigns a stable ID so retries and redelivery can be deduplicated.
- **Conversation ordering:** one owner/partition per conversation can assign sequence; global ordering across all chats is unnecessary.
- **Presence/typing:** degrade or allow staleness; do not make durable message send depend on it.
- **Database/log:** replicate across failure domains with an ACK contract matching “sent” semantics.
- **Region partition:** route each conversation/user to an owner, or define merge/order semantics; do not casually allow two regions to assign conflicting sequence to the same chat.
- **Recovery:** replay durable undelivered messages; clients deduplicate by message ID.

Interviewer follow-up: “A sender got no ACK, reconnected through another region, and retried. How do you avoid showing the recipient two copies?” Answer with a stable message ID, durable deduplication/inbox state at the authoritative message boundary, and an idempotent acknowledgement—not a promise of magical exactly-once networking.

### 18.2 YouTube-style video platform — illustrative interview design

Different paths need different availability strategies:

```text
Viewer -> DNS/edge -> CDN/cache -------> immutable video segments
                    |                         ^
                    +-> metadata API -> replicated metadata/read cache

Uploader -> upload API -> object storage -> durable queue -> transcoders
                                                 |
                                                 +-> rendition metadata publish
```

- **Playback segments:** globally active caches/CDN; immutable segment IDs make replicas/caching safe. If recommendation service fails, direct playback can still work.
- **Metadata reads:** serve replicated/cached title and thumbnail with bounded staleness.
- **Upload:** object chunks use content/upload IDs and resumable idempotent parts. A durable queue decouples transcoding.
- **Transcoding:** workers active across a fleet; duplicated job execution must publish deterministic rendition IDs or use a completion claim.
- **Publish state:** only expose a rendition after the object exists and metadata transition is durable.
- **DR:** object data and metadata need independent recovery plans; recreating derived renditions may be possible from original uploads, reducing their durability requirement.

Graceful degradation:

```text
recommendations down -> show subscriptions/trending/default rail
comments down        -> keep playback
one rendition absent -> choose another quality
upload processing slow -> accept durably and show "processing"
```

The interview insight is to define availability per capability. “YouTube is up” should not depend on comments, recommendations, ads, upload, and playback all being perfect simultaneously.

### 18.3 Uber — verified historical production example plus illustration

Uber’s 2020 engineering article on multi-region Kafka describes both active-active and active-passive consumption patterns ([Uber Engineering](https://www.uber.com/en-NL/blog/kafka/)). It reports that:

- producers normally wrote to a local regional Kafka cluster and could fail to another region;
- messages were asynchronously replicated into aggregate clusters;
- some applications consumed actively in several regions;
- a dynamic-pricing example computed in multiple regions but assigned one primary updater;
- stronger-consistency consumers such as payment processing and auditing commonly used an active-passive consumption mode with synchronized progress/checkpoints.

This historical example is valuable because “Uber is active-active” would hide several different ownership choices:

```text
Trip events
   | produce locally
   +--> Kafka region A --async replicate--+
   +--> Kafka region B --async replicate--+--> regional computations active
                                                   |
                                          one designated publisher/owner
                                                   |
                                            active data lookup path
```

Different layers are active according to their semantics:

- recomputable pricing state can be calculated in both places;
- only an assigned owner publishes a conflicting update;
- payment/audit progress favors one active consumer to simplify correctness;
- offsets/checkpoints are state that must be included in failover, not an implementation footnote.

**Illustrative ride-booking extension:** region-local API fleets can be active-active, while a city/geo shard owns matching decisions. If the live map or ETA model is impaired, existing-trip safety/status remains higher priority than new matching or surge precision.

### 18.4 Netflix — historical production example plus illustration

Netflix published an **Active-Active for Multi-Regional Resiliency** design in 2013, discussing traffic across multiple AWS regions and replicated data ([Netflix TechBlog](https://medium.com/netflixtechblog/active-active-for-multi-regional-resiliency-c47719f6685b)). This is a historical engineering source, not a claim that every Netflix subsystem in 2026 has one uniform design.

An interview-friendly decomposition is:

```text
Viewer -> global traffic -> active API region A --+
                        \-> active API region B --+-> regional state/cache

Video bytes -> distributed CDN/edge path
Control/metadata -> replicated service with per-data consistency choices
```

- playback/control API can fail away from an impaired region;
- cached/immutable video delivery is naturally easier to make active in many locations;
- account entitlement and billing need stricter authority than recommendations;
- “continue watching” can often accept convergent latest-progress semantics, but a late device update must not always move playback backward—domain-aware monotonic progress may be better than raw LWW;
- recommendations can be stale or omitted, preserving playback.

This illustrates that active-active is often **per request path and data type**, not one database switch.

### 18.5 Twitter/X-style timeline — illustrative interview design

```text
Create post -> write API -> home shard leader -> durable post store
                                    |
                                    +-> event log -> fanout workers
                                                      |
                                            home-timeline caches

Read timeline -> active read API -> cache -> post hydration store
```

HA choices:

- **Post creation:** one leader per author/post shard; stable post ID makes retry idempotent.
- **Fanout:** durable event log; workers may redeliver, so cache insertion uses post ID to deduplicate.
- **Timeline reads:** active across zones/regions; stale ranking is tolerable within an explicit window.
- **Celebrity fanout:** read-time merge avoids one giant write fanout and reduces blast radius.
- **Degradation:** if fanout lags, fetch recent author posts at read time or show a slightly stale timeline rather than reject post creation.
- **Conflict:** counters for likes/reposts can be sharded and aggregated; authorization/delete state needs a fresher path so deleted content is not indefinitely served from caches.

The key interview distinction is that delayed timeline appearance is not the same as losing the authoritative post.

### 18.6 Amazon-style ordering — illustrative design with verified AWS building blocks

```text
Customer -> redundant edge/LB -> order API fleet
                                  |
                          idempotent order command
                                  |
                          order-shard leader/quorum
                                  |
                          transactional outbox
                                  |
                      durable event bus / workers
                         /                   \
                   inventory              payment
```

- API and edge can be active across several zones.
- An order has one authoritative state machine; repeated `PlaceOrder(K)` returns the same order.
- Inventory reservation requires conditional authority, quorum, or escrow—not LWW.
- A transactional outbox couples the order state change to eventual event publication without a fragile database-plus-broker dual write.
- Payment uses a propagated idempotency key and reconciliation with the external processor.
- Browse/search/recommendations can degrade independently from checkout.
- Database replicas and backups cover different failure classes.

**Verified AWS examples, not claims about Amazon.com internals:** AWS ELB uses load-balancer nodes across enabled Availability Zones ([ELB documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html)); DynamoDB global-table guidance distinguishes write-to-any-region and write-to-one-region patterns and explicitly discusses conflict/idempotency implications ([DynamoDB global-table write modes](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-global-table-design.prescriptive-guidance.writemodes.html)).

### 18.7 Cross-system summary

| Product path | Natural HA pattern | State caution |
|---|---|---|
| Chat gateways | Active-active stateless connections | Reconnect and deduplicate durable messages |
| Video playback | Active-active CDN/cache | Metadata/entitlement has different consistency |
| Ride/pricing compute | Active-active regional compute | One publisher or merge policy for authoritative result |
| Timeline reads | Active-active caches/read replicas | Staleness and delete propagation |
| Order/payment | Redundant compute + single/quorum write authority | Idempotency, invariant, audit, reconciliation |
| Analytics/batch | Active-active or standby depending cost/RTO | Checkpoint/offset and derived-data recovery |

There is no rule that one company chooses one pattern for every subsystem.

## 19. Worked HLD: making an order service highly available

This walkthrough shows how to build the answer instead of beginning with “two regions.”

### 19.1 Step 1 — clarify the contract

Functional scope:

- create an order from a previously priced cart;
- reserve inventory;
- initiate payment;
- query order status.

Illustrative non-functional requirements:

- order-status reads target 99.99% request availability;
- no two successful orders reserve the same last unit;
- no duplicate charge for one checkout intent;
- acknowledged order intent survives one-zone failure;
- region disaster RTO/RPO must be chosen with the interviewer; assume RTO 15 minutes and near-zero RPO for accepted order/payment records **only for this exercise**.

### 19.2 Step 2 — identify critical versus degradable paths

Critical:

- idempotency record/order state;
- inventory authority;
- payment attempt/audit state;
- status query.

Degradable:

- recommendations;
- rich analytics;
- non-critical email;
- search index freshness.

### 19.3 Step 3 — baseline HA layout

```text
                              Region R1

Clients -> distributed LB -> Order APIs across zone A + zone B
                                  |
                          order DB shard leader
                              /          \
                   voter/replica B    voter/replica C
                                  |
                         transactional outbox
                                  |
                          replicated durable broker
                            /                \
                    Inventory worker     Payment worker
                            |                  |
                    inventory authority   payment provider

Cross-region:
  order/audit replication + independent PITR backups + warm application stack
```

### 19.4 Step 4 — request flow

```text
1. Client sends PlaceOrder(idempotency_key=K, cart_version=17).
2. Any API instance validates syntax/authentication.
3. Authoritative transaction claims K and creates PENDING order once.
4. Outbox record commits in the same local DB transaction.
5. Publisher emits OrderCreated; duplicate publish is safe by event ID.
6. Inventory service conditionally reserves stock under order ID.
7. Payment service calls provider with stable attempt ID.
8. Saga/state machine records success or compensation requirement.
9. Client polls/subscribes to status using order ID; retry K finds same order.
```

This does not make inventory and an external payment one ACID transaction. It gives durable state, idempotent steps, and explicit compensation/reconciliation.

### 19.5 Step 5 — failure reasoning

| Failure | Expected behavior |
|---|---|
| One API dies | LB retries/routes new request; same key prevents duplicate order |
| One zone dies | surviving API/DB quorum carries critical path if capacity remains |
| DB leader dies before commit | transaction absent; retry with K can create it |
| DB commits but response is lost | retry K returns existing order |
| DB leader dies after local-only async ACK | violates near-zero-RPO assumption; require stronger commit or change requirement |
| Broker publish fails after DB commit | outbox publisher retries from durable row |
| Event is redelivered | consumer inbox/business key makes effect idempotent |
| Payment provider times out | query/reconcile using stable provider attempt ID; do not blindly issue a new charge |
| Inventory unavailable | keep order pending briefly or reject/compensate; never report confirmed without reservation |
| Region R1 lost | activate/fail to R2 only after fencing/authority and recovery-point checks |
| Old database returns | resync as follower; never accept its old ownership epoch |
| Bad delete/corruption replicates | PITR/backup recovery, not ordinary replica failover |

### 19.6 Step 6 — regional strategy choice

Possible answer:

> “I would keep stateless APIs deployable in both regions but use one write-owning home region per order shard. That gives active-active compute/read potential without concurrent mutation of the same order. A region transfer uses an epoch and quorum/fencing. If the interviewer requires local writes during a total inter-region partition, I would ask whether oversell/duplicate business conflicts can be compensated; otherwise I will reject unsafe writes rather than claim fully available multi-writer ordering.”

That is stronger than naming “multi-master” without invariant behavior.

### 19.7 Step 7 — capacity and operations

- surviving zone/region needs tested peak critical-path capacity;
- shed analytics and email first;
- alert on order-success SLI, payment unknowns, outbox lag, broker backlog, replication lag, and quorum health;
- regularly kill an API, fail a DB leader, replay duplicate events, restore backups, and conduct regional game days;
- measure failover and reconciliation outcomes against the stated SLO/RTO/RPO.

## 20. Trade-offs, alternatives, and when to use or avoid them

### 20.1 Decision table

| Choice | Prefer when | Avoid / be cautious when | Main trade-off |
|---|---|---|---|
| Multi-zone active app fleet | Most online services | Dependencies remain single-zone | Modest complexity for strong instance/zone resilience |
| Active-passive database | One write order/invariants matter | Promotion RTO is unacceptable | Simpler correctness versus failover interval |
| Async replica | Latency/write availability and non-zero RPO accepted | Any acknowledged-write loss is unacceptable | Faster ACK versus replication-gap loss |
| Sync/quorum replication | Committed-write durability/order dominates | Remote latency or minority availability is unacceptable | Coordination safety versus latency/availability |
| Read replicas | Read scale; stale reads are classified | Read must immediately reflect write | Throughput versus freshness |
| Active-active stateless compute | Global latency/very low compute failover | All sites still share one fragile state dependency | Fast routing failover versus distributed operations |
| Per-key home region | Keys partition naturally | Cross-key/global transactions dominate | Conflict avoidance versus routing/ownership complexity |
| Multi-writer async state | Local partition-side writes are essential and mergeable | Money, uniqueness, scarce inventory without escrow | Availability/local latency versus conflicts |
| LWW | Latest simple assignment may overwrite safely | User intent/ledger/invariant must never disappear | Easy convergence versus silent loss |
| CRDT/domain merge | Operations have provably safe merge semantics | Arbitrary constraints/transactions | Availability versus data-model complexity |
| Durable queue buffering | Completion may be async | User requires immediate authoritative result | Outage absorption versus delayed state/backlog |
| Cold DR | Long RTO acceptable, low cost important | Business cannot wait for provision/restore | Cost versus recovery time |
| Warm/hot standby | Shorter RTO with one owner | Team never tests it or cannot fund capacity | Readiness versus cost/drift |
| Active-active region | Extremely low RTO/global locality justifies it | State semantics/operations immature | Continuity/latency versus highest complexity |

### 20.2 When not to chase another nine

Moving from 99.9% to 99.99% cuts the yearly time budget from about 8h 46m to about 52m 34s. That often requires:

- fewer mandatory dependencies;
- faster automated recovery;
- multi-zone data and compute;
- safe, smaller deployments;
- spare capacity;
- better observability/on-call;
- continuous restore/failover tests;
- possibly multi-region architecture.

If the user/business harm does not justify that cost and complexity, a lower honest SLO plus good recovery may be the better design. Over-complex architecture can itself reduce reliability.

### 20.3 When active-active is especially dangerous

- global counters or scarce inventory must never exceed a limit;
- uniqueness must be decided across all regions;
- external side effects cannot be reversed/deduplicated;
- application teams do not own explicit conflict semantics;
- capacity depends on emergency provisioning that has not been tested;
- schema/deploy changes cannot be rolled out compatibly;
- observability cannot distinguish replication lag/conflict by region;
- “active-active” is being proposed only to avoid discussing failover.

### 20.4 Alternatives to whole-system multi-region

- multi-zone HA plus cross-region backup/PITR;
- active-active edge/read cache with single-region writes;
- per-tenant or per-shard home regions;
- cold/warm standby only for critical state;
- queue writes during short outages;
- cell architecture to limit affected population;
- manual but practiced DR where automated false failover is riskier;
- dual provider only for one critical dependency, not an undifferentiated multi-cloud rewrite.

## 21. Common mistakes and corrections

### Mistake 1: “Add two servers, so there is no SPOF”

They may share one LB, database, zone, certificate, deployment, or capacity pool.

**Correction:** audit the end-to-end critical path and failure domains.

### Mistake 2: “The load balancer is a single box in my diagram”

A diagram icon often represents a logical distributed service.

**Correction:** say how the LB itself is redundant/discovered and what happens to connections and DNS.

### Mistake 3: “99.99% means 52 minutes of maintenance whenever we want”

The measurement may be request-based, scheduled maintenance may or may not be excluded, and one 52-minute global outage can have different harm from scattered errors.

**Correction:** define SLI, SLO/SLA, eligible events, window, latency threshold, and error-budget policy.

### Mistake 4: “Replication means no data loss”

Async replicas can lag; sync semantics differ; corruption/delete can replicate.

**Correction:** state ACK point, RPO, promotion rule, and independent backup/PITR.

### Mistake 5: “Backup is HA”

A backup can preserve data but may take hours to restore.

**Correction:** replicas/failover target low RTO; backups target recoverability and older points.

### Mistake 6: “Active-active has zero downtime”

Detection, traffic convergence, connection retries, state repair, and overload still cause impact.

**Correction:** state achieved failover SLI/RTO per path, not “zero.”

### Mistake 7: “Active-active means both databases accept all writes”

Compute can be active-active while each key has one writer.

**Correction:** name active layers and write-ownership scope.

### Mistake 8: “Use last-write-wins to solve conflicts”

LWW produces convergence by discarding a loser; it may violate user intent.

**Correction:** begin with the invariant and prefer avoidance, quorum, merge, or explicit compensation where needed.

### Mistake 9: “The old primary is back, route traffic to it”

After promotion it is stale and may have a divergent timeline.

**Correction:** fence, rewind/resnapshot, validate, and rejoin as follower.

### Mistake 10: “Automatic failover is always better”

False detection can trigger a damaging regional switch or split brain.

**Correction:** automate frequent, well-understood faults; use eligibility, fencing, multiple signals, guardrails, and human decisions where consequences are irreversible.

### Mistake 11: “Health endpoint is green, so the service is available”

It may not complete the business path or may be saturated.

**Correction:** combine readiness with end-to-end SLIs/canaries and capacity signals.

### Mistake 12: “Auto-scaling will handle failover”

It takes time and may hit quota, image, IP, control-plane, or cold-cache constraints.

**Correction:** calculate surviving capacity and test scale-up under the failure.

### Mistake 13: “Retry every failure”

Retries can duplicate side effects and amplify overload.

**Correction:** classify retryable errors, bound attempts, use backoff/jitter, and make business operations idempotent.

### Mistake 14: “DR is a second region in a diagram”

No one may know how to activate it; secrets, data, DNS, or dependencies may be missing.

**Correction:** define RTO/RPO, runbook, authority, dependencies, validation, failback, and measured drills.

### Mistake 15: “High availability requires eventual consistency everywhere”

Consistency/availability choices occur under particular partitions and per operation. Majority systems can be strongly consistent and highly available for failures that retain quorum.

**Correction:** state failure, operation, quorum/owner, and what the minority side does.

## 22. How this appears in system-design interviews

### 22.1 The typical progression

An interviewer rarely needs a standalone “design HA” monologue. They often let you draw:

```text
client -> LB -> service -> DB
```

and then ask:

- “What are the single points of failure?”
- “What if the load balancer goes down?”
- “What if the primary database dies?”

## 24. Answer spines: cues to reconstruct the answer live

An **answer spine** is a sequence of cues. It prevents memorized paragraphs from collapsing when the interviewer changes one requirement.

### 24.1 “Design this service for high availability” — `FAILOVER`

1. **F — Function and failure scope:** critical user journey; instance/zone/region failures in scope.
2. **A — Availability objective:** SLI, exact SLO/window, degraded SLO, RTO/RPO.
3. **I — Invariants and SPOFs:** money, uniqueness, ordering; trace mandatory dependencies.
4. **L — Layout and load:** redundancy across failure domains; capacity after one fails.
5. **O — Ownership of state:** leader/quorum/home shard/multi-writer; write ACK and read freshness.
6. **V — Verification and failover:** health signals, election, fencing, promotion, routing.
7. **E — Errors and degradation:** timeout, bounded retry, idempotency, queue, shed optional work.
8. **R — Recovery:** rebuild redundancy, backups/PITR, DR drill, reconciliation, failback.

Spoken skeleton:

> “I’ll define availability for the checkout path and the failures we must tolerate. I’ll make stateless APIs redundant across zones behind a logically distributed load balancer and reserve capacity for a zone loss. For state, I’ll preserve one order owner/quorum per shard, state when a write is committed, and use fenced promotion. Retries carry idempotency keys. Optional features degrade independently. I’ll monitor the user SLI and replication lag, test failover under load, and use independent PITR backups plus a regional DR plan with measured RTO/RPO.”

### 24.2 “Active-active versus active-passive” — `SCOPE`

1. **S — Scope the word active:** app, read, write, consumer, or full region?
2. **C — Correctness:** one owner or simultaneous writes; conflicts/invariants?
3. **O — Objectives:** RTO, RPO, latency, partition availability.
4. **P — Peak failure capacity:** can the survivor carry the failed share?
5. **E — Economics and execution:** idle cost, complexity, drift, tests, failback.

Spoken skeleton:

> “Active-passive has one serving owner for the scoped workload and promotes/routes to a standby, so ordering is simpler but failover has detection and promotion time and standby cost. Active-active uses multiple sites continuously and can improve locality and routing failover, but surviving capacity and state semantics become harder. I would often use active-active stateless compute with one write owner per key. The choice follows RTO/RPO, invariants, partition behavior, capacity, and operating cost.”

### 24.3 “What if the load balancer fails?”

Cues:

```text
logical vs physical -> redundant nodes/failure domains -> discovery/routing
-> backend health -> existing connections -> capacity -> config/DNS common fate
```

Spoken skeleton:

> “The one LB icon is a logical endpoint, not one process. I would use LB nodes in multiple failure domains reached through resilient DNS, anycast, or a safely transferred VIP. Each node routes only to ready backends. I would define how new and existing connections fail over, ensure the surviving zone has capacity, and stage configuration because DNS, certificates, and one global bad rule can still defeat the redundancy.”

### 24.4 “The primary database failed; walk me through it” — `DFPRR`

1. **Detect:** multiple signals; avoid reacting to one slow probe.
2. **Fence:** old primary loses authority/storage/epoch.
3. **Promote:** choose eligible sufficiently current replica; state possible RPO.
4. **Route and retry:** update proxy/discovery; clear pools; idempotent uncertain operations.
5. **Recover:** verify, rebuild a standby, resync old primary as follower, later fail back deliberately.

Add:

> “If replication was asynchronous, an acknowledged write not present on the promoted replica may be lost. If the business forbids that, the original ACK path needed synchronous/quorum durability or another reconstructable durable record.”

### 24.5 “How do you resolve multi-master conflicts?” — `AVOID-MERGE`

Cues:

```text
invariant first
-> avoid with one owner/home shard/quorum/escrow if possible
-> identify concurrent writes with versions/causality
-> merge only if operations have domain-safe semantics
-> otherwise reject/expose/LWW by explicit policy/compensate
-> repair and observe conflict rate
```

Spoken skeleton:

> “I would not pick a conflict algorithm before naming the invariant. For inventory or balances I would avoid concurrent ownership using a leader/quorum or escrow. For mergeable state I would retain causal versions and apply a domain merge or CRDT. LWW only fits data where discarding one concurrent assignment is acceptable; convergence alone is not correctness. Non-mergeable accepted conflicts need explicit compensation and reconciliation.”

### 24.6 “Explain four nines”

Cues:

```text
exact SLI -> exact window -> 0.01% error budget
-> 52m34s/year or 100 failed per million
-> latency/correctness counts -> end-to-end dependencies -> measurement/exclusions
```

Spoken skeleton:

> “99.99% means a 0.01% error budget over a stated window. Time-based over 365 days that is about 52 minutes 34 seconds; request-based it is about 100 failed eligible requests per million. I still need to define success, including a latency threshold and correctness, plus exclusions and scope. The target applies to the user journey, so mandatory dependency availability and correlated failures matter.”

### 24.7 “HA versus DR”

Cues:

```text
HA = continue through scoped routine faults
DR = restore after exceptional/wide damage
RTO = time; RPO = data point
replicas for fast failover; independent backup for logical damage
test failover + restore + failback
```

### 24.8 “How do you find SPOFs?”

Cues:

```text
choose business journey -> dependency graph -> each node and edge
-> process/host/zone/region -> network partition -> capacity
-> state ownership/ACK -> global config/identity/DNS -> recovery dependency
-> correlated failure and test evidence
```

### 24.9 “Can a queue keep us available while the DB is down?”

Cues:

> Change API semantics to durable acceptance (`202`) → queue itself replicated → bounded backlog → ordering per key → idempotent consumer → backpressure → database catch-up rate → user status and failure/reconciliation.

### 24.10 “Why not make everything active-active?”

Cues:

> Stateless/read paths benefit → state invariants may require coordination → every region needs failure capacity → global changes still correlate → replication/egress/operations cost → choose per capability → simplest design meeting measured RTO/RPO.

## 25. Follow-up drills with model reasoning

### Drill A: “Two active regions each run at 50% traffic. Aren’t we safe?”

Not until capacity is quantified. If each is already above 50% of its own maximum resource, it cannot necessarily take 100% of total traffic. Include peak traffic, connection count, database/write capacity, broker partitions, network egress, quotas, caches, and auto-scaling delay. Test region loss under peak-like load and define which work is shed.

### Drill B: “If the database has a synchronous replica, RPO is zero, right?”

Ask:

- Was every successful transaction required to wait for it?
- Was the record durably flushed or merely received?
- Is it in an independent failure domain?
- Can the failover protocol elect a node without the record?
- What if both copies are corrupted/deleted?
- Does the claim cover external side effects?

It can give RPO 0 for committed database records under a stated failure model; it does not guarantee universal zero data loss.

### Drill C: “Why not promote the old primary when it returns? It has the missing W2.”

The new primary may have accepted `W3…W20`; histories have diverged. Promoting the old node would discard or conflict with newer authority. Keep it fenced, preserve orphan evidence for business reconciliation, and resynchronize it from the current authority. Prevention is a stronger original ACK/quorum policy if W2 must never be orphaned.

### Drill D: “How does active-passive handle a network partition?”

The standby must not promote merely because it cannot hear the active. Use a witness/quorum/lease and fencing so only the side with authority can write. A minority/unfenced side rejects. If both accept, the design has become accidental multi-leader and needs conflict recovery.

### Drill E: “Would you use multi-writer for a shopping cart?”

Potentially, but define operations. Adding different products can merge as per-item state/events. Concurrent quantity set, remove-versus-add, coupon constraints, stock reservation, and checkout do not all merge trivially. The cart view can be available and convergent while final checkout uses authoritative price/inventory validation.

### Drill F: “Can a quorum system be highly available?”

Yes, for failures leaving a quorum it can automatically elect/continue while preserving strong consistency. It is not CAP-available on every isolated side of a partition: the minority rejects/waits. “Highly available” is an SLO claim; “available” in CAP is a strict partition-response property.

### Drill G: “A `200 OK` was returned in eight seconds against a two-second SLO. Available?”

If the SLI defines success within two seconds, no. Availability should reflect usable user service; otherwise slow timeouts can be mislabeled success. The exact latency threshold and eligible-request rules belong in the SLI.

### Drill H: “We have three live replicas. Why pay for backups?”

Replicas protect live service from hardware/process failure and support low RTO. They usually copy bad writes and deletes. Independent versioned backups/PITR provide an older clean point and security/failure-domain separation. A backup only counts when restoration and integrity are tested.

### Drill I: “Can DNS give instant regional failover?”

No general instant guarantee. TTL/resolver behavior, existing connections, client retry behavior, health-detection time, and backend readiness affect convergence. A global proxy/anycast design can change routing differently, but it too has health and control-plane behavior to define.

### Drill J: “Should the auth service fail open to preserve availability?”

Usually not for security-sensitive or money-changing actions: accepting an unauthorized action is worse than temporary rejection. Cached signed-token verification may keep service available without contacting auth on every request, provided key freshness/revocation risks are explicitly accepted. Public non-sensitive reads may degrade differently.

### Drill K: “What if a global code bug corrupts both active regions?”

Active-active infrastructure is not protection from shared software fate. Halt rollout, fence writers if continuing causes damage, choose a clean recovery point, restore/PITR from isolated versions, deploy the fixed backward-compatible code, validate invariants, reconcile external actions, and progressively reopen. Prevent with canaries/cells, staged schemas, limited global blast radius, and immutable backups.

### Drill L: “Does active-active eliminate DR?”

No. It can make certain regional infrastructure faults routine HA events. DR is still needed for multi-region software corruption, security compromise, dependency failure, account/control-plane problems, data deletion, and cases where surviving regions cannot serve. Active-active itself needs repair and rebalancing after one side fails.

### Drill M: “A WebSocket server dies. How is the connection failed over?”

The connection itself normally dies; it is not moved. The client reconnects through resilient routing, authenticates, presents last received sequence/checkpoint, and resumes from durable state. Gateways are disposable; message/session state needed for continuation must be externalized or reconstructable.

### Drill N: “Why can aggressive health checks reduce availability?”

Transient latency or packet loss may falsely eject healthy instances. Synchronized checks add load, and mass ejection overloads survivors. Use thresholds/hysteresis, jitter, multiple signals, readiness distinction, slow start, and minimum healthy-capacity safeguards.

### Drill O: “What should happen to a payment request during regional failover?”

The outcome may be unknown. Preserve one logical payment-attempt ID across retries and regions, query the authoritative/provider status before issuing another effect, record the result durably, and reconcile discrepancies. Never infer “timeout means not charged.”

## 26. Compact review sheet

### 26.1 One-screen mental model

```text
Availability = successful usable service / eligible demand or time
Four nines   = 99.99% = 0.01% error budget

HA requires:
  redundant copies across named failure domains
  + health detection
  + safe traffic/ownership failover
  + enough survivor capacity
  + correct retry/degradation
  + tested recovery

Active-passive:
  one owner -> detect -> fence -> promote -> route -> rebuild

Active-active:
  name active layer -> route -> define per-key write ownership
  -> coordinate, merge, reject, or compensate conflicts

Database failure:
  ACK semantics determine RPO
  old primary must be fenced and resynced after promotion

DR:
  RPO = acceptable recovery-point gap
  RTO = acceptable recovery time
  replication != backup
  failover != failback
```

### 26.2 Vocabulary corrections

| Instead of saying | Say |
|---|---|
| “99.99x uptime” | “99.99% request availability over a rolling 28-day window, with success under X ms” |
| “No single point failure” | “No single instance/zone in the stated failure model stops this critical path” |
| “One LB” | “One logical endpoint backed by redundant LB nodes and resilient discovery” |
| “Replica means no data loss” | “The ACK waits for these replicas at this durability point; async lag gives this RPO risk” |
| “Master-master solves HA” | “Multi-writer improves local write availability but uses this ownership/conflict policy” |
| “Last writer wins solves conflicts” | “LWW converges by discarding a concurrent value; this data can tolerate that” |
| “Database is up again” | “It has recovered, caught up, passed integrity/readiness, and is eligible for this role” |
| “Failover is automatic” | “Detection, eligibility, fencing, promotion, routing, and verification are automated under these guardrails” |
| “We have DR” | “This tested runbook achieved RTO X and RPO Y for the named disaster” |
| “Active-active has no downtime” | “This stateless path fails away within the measured SLO; state and connections behave as follows” |

### 26.3 Metrics worth mentioning

- user/business success SLI and error-budget burn;
- p95/p99 latency and timeout rate;
- ready capacity per zone/region and failover headroom;
- database receive/apply lag and commit quorum latency;
- leader-election/failover duration;
- old-epoch/fencing rejection count;
- retry volume, duplicate suppression, and reconciliation backlog;
- queue depth/oldest-message age;
- conflict rate and chosen resolutions;
- backup age, restore verification age, achieved RPO/RTO;
- duration in degraded/single-replica state.

## 27. Say-out-loud 30-second narrative

> “High availability means a defined user journey stays usable through specified failures, not just that a process is running. I remove single points by placing redundant compute and state across independent failure domains, routing only to healthy capacity, and reserving enough capacity after a failure. Active-passive keeps one owner and uses detect, fence, promote, and reroute; active-active serves from several sites but must define write ownership and conflict rules. Database acknowledgements determine possible data loss, and an old primary must resynchronize before rejoining. HA handles live failures, while tested backups and a DR plan meet explicit RPO and RTO for larger disasters.”
