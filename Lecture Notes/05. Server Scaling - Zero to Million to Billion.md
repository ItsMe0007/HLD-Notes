# Server Scaling: From Zero to Millions to Billions

> Beginner-first HLD study notes for understanding how an internet application evolves as load grows. The goal is to diagnose a bottleneck and justify the next change, not to memorize a fixed ladder of technologies.

## Scope, assumptions, and evidence labels

No rough bullets were supplied beyond **"server scaling, 0 to million to billion."** I therefore use a generic read-heavy internet application with user accounts, dynamic API requests, media/static files, and background work. Whenever the workload matters, the assumptions are stated explicitly.

The words *million* and *billion* refer to **users only in illustrative examples**. They are not capacity thresholds. One billion registered accounts that rarely log in may create less load than one million users maintaining live video connections.

Evidence labels used in this chapter:

- **Verified technical guidance**: backed by first-party documentation, an engineering publication, or an original paper linked nearby.
- **Public production snapshot**: a company publicly described an architecture or number at a particular date. It must not be treated as a claim about the company's private architecture today.
- **Self-reported interview experience**: a candidate publicly reported a question. It shows question style, but is not an independently verified company question bank.
- **Illustrative design**: a reasonable interview design for a named type of product. It is not a claim that a real company uses those exact components.

All calculations are estimates from explicitly stated assumptions, not universal capacity claims. There is no reliable rule such as "5,000 requests per second means shard the database." Hardware, code, queries, payloads, latency targets, and traffic shape change the answer.

## 1. Plain-language foundation

### 1.1 What does scaling mean?

Suppose one cashier can serve ten customers per minute. When more customers arrive, the queue grows. The shop can respond in several ways:

- make the cashier faster;
- add more cashiers;
- let customers do some work themselves;
- pre-package popular items;
- send slow tasks to a back room; or
- open another branch closer to customers.

Server scaling is the same idea applied to software:

> **Scaling is increasing a system's ability to handle more work, data, users, or geographic reach while still meeting its correctness, latency, availability, and cost goals.**

Scaling is not merely "add servers." The overloaded resource may be the database, network link, cache, message partition, lock, third-party API, or even one unusually popular key.

### 1.2 What is a server?

The word **server** is overloaded:

| Meaning | Plain explanation | Example |
|---|---|---|
| Physical/virtual machine | A computer with CPU, memory, disk, and network | A cloud VM |
| Process | A running program that accepts requests | A Java, Go, or Node.js application |
| Application role | A logical responsibility | Web server, cache server, database server |
| Service | A capability exposed over a network | Order Service with 20 running instances |

One machine can run several server processes. One logical service can run on thousands of machines. In interviews, say which meaning you intend.

### 1.3 The smallest useful request path

Imagine a user opens `shop.example.com/products/42`:

```text
1. Name lookup              2. Network request          3. Application work

shop.example.com --DNS--> IP address
       User --------------------HTTP/TLS--------------------> App server
                                                              |
                                                              v
                                                          Database
                                                              |
       User <-------------------HTTP response------------------+
```

The browser asks **DNS** where the site lives, establishes a network connection, sends an **HTTP** request, and receives a response. The application may read or change durable state in a **database**.

At larger scale the path gains more components, but it is still this same conversation repeated many times.

### 1.4 Inline prerequisite: DNS, HTTP, and TLS

These are required to understand the front door of a scalable system:

- **DNS (Domain Name System)** translates a hostname into an IP address or another name. DNS-based traffic management can direct users toward a region, but cached DNS answers mean changes are not instantaneous.
- **HTTP** is a request/response application protocol. A request has a method such as `GET` or `POST`, headers, and possibly a body. HTTP does not make business operations safe to repeat automatically.
- **TLS** encrypts and authenticates the connection used by HTTPS. TLS may terminate at an edge proxy or load balancer, after which internal connections still need an explicit security model.
- **Connection** and **request** are different. HTTP/2 or HTTP/3 can carry multiple requests over a connection; WebSocket and streaming systems may keep connections open for a long time.

This matters because 100,000 short HTTP requests per second and 100,000 mostly idle persistent connections stress different resources.

### 1.5 Inline prerequisite: stateful versus stateless

**State** is information that must survive or influence later operations: a user session, cart, order, uploaded object, or account balance.

A **stateless application instance** does not keep the only important copy of a user's state in its local memory or disk. Any healthy instance can handle the next request because durable/shared state lives in a database, cache, or object store.

```text
State trapped in one instance                Externally stored state

User A ---> App 1 [session A]                User A ---+
User A -X-> App 2 [no session]                         v
                                                Load balancer
                                                     |
                                             +-------+-------+
                                             v               v
                                           App 1           App 2
                                             +-------+-------+
                                                     |
                                                     v
                                                Session store
```

Stateless compute is easier to add, remove, restart, and balance. This does **not** mean the whole system is stateless; it means state is placed in components designed to own it.

### 1.6 Inline prerequisite: latency, throughput, concurrency, and saturation

These words describe different things:

| Term | Meaning | Example |
|---|---|---|
| **Latency** | Time one operation takes | p95 response time is 250 ms |
| **Throughput** | Work completed per unit time | 8,000 successful requests/second |
| **Offered load** | Work clients try to send | 10,000 requests/second arrive |
| **Goodput** | Useful work completed within correctness/latency goals | 7,500 valid responses/second before timeout |
| **Concurrency** | Work in progress at the same time | About 2,000 active requests |
| **Utilization** | Fraction of a resource being used | CPU averages 70% |
| **Saturation** | A resource has more demand than it can promptly serve | DB connection wait queue grows |

Average latency hides a bad tail. If 99 requests take 20 ms and one takes 5 seconds, an average can look acceptable while the unlucky user suffers. Interviews commonly discuss p50, p95, or p99 latency: the latency below which 50%, 95%, or 99% of observations fall.

A useful relationship is **Little's Law**:

```text
average in-flight work ~= throughput x average time in system

At 5,000 requests/second and 0.2 second average latency:
in-flight requests ~= 5,000 x 0.2 = 1,000
```

Reducing latency can therefore reduce memory, threads, connections, and queue occupancy even before adding machines.

### 1.7 Scalability, elasticity, performance, and availability are not synonyms

| Property | Question |
|---|---|
| **Performance** | How fast or efficient is the system at a particular load? |
| **Scalability** | Can capacity grow without the design collapsing or cost growing unreasonably? |
| **Elasticity** | Can capacity expand and contract as demand changes, often automatically? |
| **Availability** | Can users obtain the intended service at a particular moment? |
| **Reliability** | Does the system continue to behave correctly over time and failures? |
| **Durability** | Once data is acknowledged as saved, will it survive the failures in the guarantee? |

A single very large database may perform well but be difficult to scale further. Ten app instances in one failure domain may scale throughput but still become unavailable together. Two replicas may improve availability while providing no additional write capacity.

### 1.8 Vertical and horizontal scaling

**Vertical scaling (scale up)** gives one node more resources:

```text
4 CPU, 16 GB RAM  --->  32 CPU, 128 GB RAM
```

**Horizontal scaling (scale out)** adds nodes:

```text
                         +--> App 1
Client -> Load balancer -+--> App 2
                         +--> App 3
```

| Choice | Advantages | Costs/limits |
|---|---|---|
| Scale up | Simple; often no application redesign; useful early | Hardware ceiling; larger failure blast radius; upgrades may disrupt; price can jump nonlinearly |
| Scale out | Incremental capacity; redundancy; commodity nodes | Requires distribution, routing, statelessness/partitioning, coordination, and better operations |

They are complements. A horizontally scaled fleet still needs sensible instance sizes; a database is often scaled vertically before the team accepts sharding complexity.

### 1.9 There are many independent dimensions of scale

A system may need to scale along one or several axes:

- request rate;
- simultaneous connections;
- stored bytes and number of records;
- read-to-write ratio;
- size or CPU cost per request;
- number of tenants and noisy-neighbor risk;
- number of geographic regions;
- team/deployment activity;
- fan-out, such as one celebrity post delivered to millions of followers; and
- traffic burstiness, such as a flash sale or live sports final.

This is why "supports a billion users" is incomplete.

## 2. Why scaling is needed in system design

### 2.1 Growth exposes finite resources

Every component has limits:

- CPU time for compression, ranking, encryption, or business logic;
- RAM for live objects, buffers, indexes, and caches;
- disk capacity and I/O operations;
- network bandwidth and connection tracking;
- database locks, query workers, and connection count;
- queue partitions and consumer throughput;
- third-party quotas; and
- human ability to deploy and operate the system.

When demand approaches a limit, latency often rises before requests fail. Slow requests remain in flight longer, consume more resources, time out, and may be retried. An ordinary capacity problem can become a positive feedback loop.

Google's SRE material warns that overload can cause cascading failure as traffic shifts to remaining replicas, while Amazon's Builders' Library distinguishes raw throughput from useful **goodput** and recommends load testing past the breaking point ([Google SRE, *Addressing Cascading Failures*](https://sre.google/sre-book/addressing-cascading-failures/); [Amazon Builders' Library, *Using load shedding to avoid overload*](https://aws.amazon.com/builders-library/using-load-shedding-to-avoid-overload/)).

### 2.2 Scaling is also about failure and geography

More users create a larger consequence when a component fails. Global users also experience speed-of-light latency to a distant region. Scaling therefore introduces questions beyond capacity:

- How much failure can the design tolerate?
- Which data must be strongly consistent?
- Can users accept stale reads?
- Can a region operate if another region or the inter-region link fails?
- What is the recovery point objective (RPO: tolerable data loss) and recovery time objective (RTO: tolerable restoration time)?
- What does the system intentionally stop doing under overload?

### 2.3 The governing principle

> **Start with the simplest design that meets today's requirements and has a safe next step. Measure, find the limiting resource, and scale that bottleneck.**

This principle avoids two opposite mistakes:

- ignoring predictable growth until an emergency; and
- paying the correctness, cost, and operational complexity of planetary scale before users exist.

Scaling should be an evidence-driven loop:

```text
requirements
    |
    v
measure current behavior ---> locate bottleneck ---> choose smallest useful change
        ^                                                   |
        |                                                   v
        +---------------- load/failure test <--------- verify trade-offs
```

## 3. Translate "users" into actual load

### 3.1 Ask these questions before drawing architecture

1. Are counts registered users, daily active users (DAU), peak concurrent users, requests/second, or open connections?
2. What does one active user do per day or session?
3. What percentage of operations are reads versus writes?
4. What is the average and p99 payload or processing cost?
5. How bursty is traffic? What peak-to-average factor should be planned?
6. How much data is retained, and for how long?
7. What latency, availability, durability, and consistency are required?
8. Are users concentrated in one country or global?
9. Which operation is critical: browsing, checkout, posting, playback, or messaging?
10. What growth horizon and budget matter?

If the interviewer supplies no figures, state assumptions, calculate only enough to expose the dominant constraints, and invite correction.

### 3.2 Core estimation formulas

```text
average RPS = daily operations / 86,400

peak RPS = average RPS x assumed peak factor

egress bytes/second = response RPS x average response bytes

new storage/day = writes/day x average stored bytes per write

raw retained storage = new storage/day x retention days

approximate concurrency = RPS x average latency in seconds
```

Then add overhead explicitly for indexes, replication, backups, metadata, encoding, and headroom. Do not hide those inside a fake precision factor.

### 3.3 Worked example: the same behavior at one million and one billion DAU

**Illustrative assumptions, not production claims:**

- each daily active user makes 20 dynamic requests/day;
- 10% are writes;
- peak traffic is 10 times the daily average;
- average response is 20 KB; and
- average request latency is 200 ms.

| Quantity | 1 million DAU | 1 billion DAU |
|---|---:|---:|
| Dynamic requests/day | 20 million | 20 billion |
| Average RPS | about 231 | about 231,481 |
| Assumed peak RPS | about 2,315 | about 2.315 million |
| Peak write RPS | about 232 | about 231,481 |
| Peak response egress | about 46.3 MB/s | about 46.3 GB/s |
| Approx. in-flight at 200 ms | about 463 | about 463,000 |

The arithmetic is simple; the interpretation is the interview skill:

- A million DAU may fit on a modest fleet if requests are cheap.
- A billion DAU cannot be represented by multiplying one box indefinitely.
- If the response were a 5 Mbps video stream, bandwidth would dominate instead of API CPU.
- If all users arrived during one live event, concurrency and burst capacity would dominate.
- If every write updated one global counter, contention would dominate even at much lower RPS.

### 3.4 Why there are no universal "magic numbers"

One request might be a cache hit using microseconds of CPU; another might scan a table, render video, or call five dependencies. Even QPS can hide cost differences. Google's overload guidance explicitly notes that queries per second alone can be misleading because different queries consume very different resources ([Google SRE, *Handling Overload*](https://sre.google/sre-book/handling-overload/)).

Prefer measured statements:

> "A load test shows one instance sustains 800 of this request per second at our p99 target. Peak forecast is 8,000 RPS. I need at least 10 instances for throughput, then I add failure headroom and validate the full dependency path."

Do not say:

> "One server handles 10,000 RPS."

### 3.5 Find the bottleneck using signals, not guesses

| Suspected limit | Useful evidence |
|---|---|
| App CPU | CPU saturation, run queue, profiling, latency rising with CPU |
| App memory | Heap growth, GC pauses, out-of-memory restarts, working-set size |
| Network | bytes/sec, packets/sec, retransmits, connection limits |
| Database CPU/I/O | slow-query plan, scanned rows, buffer hit rate, IOPS, lock waits |
| Database connections | pool wait time, active/idle connections, rejected connections |
| Cache | hit rate by route/key, eviction rate, memory, hot-key traffic, miss latency |
| Queue/worker | backlog age, enqueue/dequeue rate, consumer lag, retry/DLQ rate |
| Downstream service | dependency latency/error rate, timeout count, quota exhaustion |
| One partition/tenant | per-key or per-shard load distribution, throttling, skew |

Scale only after verifying that the proposed change relieves the measured limit. Adding app instances will not repair an unindexed query; it may create more database connections and make the outage worse.

## 4. The evolution: zero to millions to billions

### 4.1 Read this as a toolbox, not a mandatory staircase

The following stages are ordered for teaching. Real systems skip, combine, or reorder them.

- A static documentation site may use object storage plus a CDN from day one and never need an application database.
- A payment ledger may need careful replication and recovery with only a few requests per second.
- A live chat product may need persistent connections and partitioned messaging before it has a million users.
- A well-built modular monolith can run on many instances; traffic alone does not require microservices.

The trigger for each change is a requirement or observed bottleneck, not a user-count milestone.

### 4.2 Stage 0: developer machine and first request

```text
Browser ---> application ---> local database/file
```

At this stage, optimize for learning:

- validate that users want the product;
- keep the code easy to change;
- collect basic request/error/latency metrics;
- define the important data and invariants; and
- keep configuration and secrets out of source code.

This is not a production architecture. The machine can be offline, local data can disappear, and other users cannot reliably reach it.

### 4.3 Stage 1: one production server

```text
Internet
   |
 DNS
   |
   v
+------------------------------------------------+
| One production host                            |
| reverse proxy/TLS -> application -> database   |
|                              \-> local files   |
+------------------------------------------------+
                 |
                 v
           backup copied elsewhere
```

This can be a legitimate first production design for low traffic. It is cheap, simple, fast to debug, and has no distributed coordination.

Minimum production discipline still matters:

- HTTPS, authentication, input validation, and least-privilege access;
- off-host, tested backups;
- logs, metrics, alerts, and a health endpoint;
- repeatable deployment and rollback;
- resource limits and request timeouts; and
- a recovery runbook.

**Limits:** the host is a single failure domain; app and database compete for resources; deployments can interrupt service; local files prevent easy replacement; and vertical growth has a ceiling.

**Use while:** it meets the SLO and recovery requirements. "Simple" is a benefit, not an embarrassment.

### 4.4 Stage 2: separate compute, durable data, and large objects

The first useful split is often by resource and durability needs:

```text
                         +----------------+
Client ---> App host --->| Managed/remote | 
             |           | database       |
             |           +----------------+
             |
             +----------> Object storage: images/video/backups
```

Why:

- application releases no longer manipulate the only database process;
- database resources and backups can be managed independently;
- object storage handles large immutable blobs better than one server's disk; and
- the app host can eventually be replaced without losing durable state.

Trade-offs:

- every database call is now a network call;
- credentials and network policy matter;
- the database can still be a single logical bottleneck; and
- managed services reduce operational work but add vendor constraints and cost.

### 4.5 Stage 3: tune and scale up before distributing blindly

Before adding architectural machinery, remove waste:

- create appropriate database indexes based on query plans;
- eliminate full scans and N+1 queries;
- bound result sizes and paginate;
- compress appropriately;
- pool and reuse database/network connections;
- profile expensive application paths;
- move large files out of the relational database if access patterns warrant it;
- batch safe operations; and
- give the bottleneck more CPU, memory, faster storage, or network.

An **index** is an auxiliary data structure that helps the database find rows without scanning the entire table. It speeds selected reads but consumes storage and makes writes more expensive because the index must also be updated.

Vertical scaling is often the fastest, lowest-risk response, particularly for a relational primary. Avoid presenting it as a permanent strategy with no ceiling.

### 4.6 Stage 4: put cacheable content at the edge with a CDN

A **content delivery network (CDN)** is a geographically distributed set of edge servers that can serve cached content closer to users ([Cloudflare, *What is a CDN?*](https://www.cloudflare.com/learning/cdn/what-is-a-cdn/)).

```text
                         cache hit
User ---> nearby CDN edge ----------> image/CSS/video segment
              |
              | cache miss
              v
          origin/object storage
```

Good early CDN candidates:

- versioned JavaScript and CSS;
- images, downloads, and video segments;
- public pages or API responses whose cache key and freshness are safe; and
- TLS termination and edge protection, depending on provider.

Benefits:

- lower latency for distant users;
- lower origin bandwidth and request load;
- absorption of repeated requests for popular content; and
- an edge layer for DDoS/WAF/rate controls.

Hard parts:

- selecting the correct cache key, including relevant query parameters, headers, language, and authorization state;
- invalidating or versioning changed content;
- avoiding private-user data leaks through a shared cache;
- handling a cold cache or mass expiry; and
- deciding what happens when the origin is unavailable.

**Common mistake:** "A CDN caches the whole application automatically." Dynamic personalized writes normally still reach the origin, and cache rules must be designed.

### 4.7 Stage 5: multiple stateless application instances behind a load balancer

When one application instance is the measured limit, scale it horizontally:

```text
                           +--> App instance 1 --+
Client -> CDN/WAF -> LB ----+--> App instance 2 --+--> DB/cache/object store
                           +--> App instance 3 --+
```

A **load balancer (LB)** accepts traffic and chooses a healthy backend. Common policies include round robin, least outstanding requests, weighted routing, or a hash when affinity is required.

Required design work:

#### Make instances replaceable

- keep durable state outside local memory/disk;
- deploy immutable/reproducible versions;
- stop accepting new work during shutdown and drain in-flight requests;
- distinguish **liveness** (process is not irrecoverably stuck) from **readiness** (safe to receive traffic); and
- warm connections/caches before receiving full traffic.

#### Handle sessions deliberately

| Session approach | Benefit | Cost |
|---|---|---|
| Signed/encrypted client token | No server session lookup; easy horizontal scale | Revocation and token size; do not put secrets in readable tokens |
| Shared session store | Central revocation and server-controlled state | Extra dependency and lookup; must be highly available |
| Sticky sessions | Quick bridge for legacy stateful apps | Uneven load; failed instance loses affinity; complicates scaling/deploys |

Sticky sessions are not always wrong, but they are weaker than making the application replaceable.

#### Separate scaling from high availability

- Multiple instances in one rack/zone can handle more traffic but share a failure domain.
- Instances in multiple availability zones tolerate more infrastructure failures.
- The database, cache, queue, and load balancer also need an availability story.
- Headroom must cover failures; running every node near 100% leaves no capacity when one disappears.

AWS's reliability guidance recommends horizontal scaling and stateless systems where possible, but those are principles, not a guarantee that every dependency is resilient ([AWS Well-Architected Reliability Pillar](https://docs.aws.amazon.com/wellarchitected/latest/reliability-pillar/welcome.html)).

#### Autoscaling is delayed feedback, not magic

Autoscaling observes a signal, decides, starts capacity, and waits for it to become ready:

```text
traffic spike -> metric rises -> policy reacts -> instance starts -> app warms -> capacity arrives
```

Design for that delay:

- keep predictive/scheduled capacity for known events;
- retain headroom for sudden bursts and failures;
- scale on the metric closest to work, such as request concurrency or queue backlog, when CPU is misleading;
- use cooldown/hysteresis to prevent oscillation; and
- load test cold starts and dependency limits.

Scaling app instances can overload a database by multiplying connection pools. Put a global connection budget behind per-instance pool sizes.

### 4.8 Stage 6: add application caching for hot repeated reads

A cache stores a copy of data in a faster location. It reduces latency and load only when requests have useful locality and staleness is acceptable.

#### Cache-aside flow

```text
                1. GET key
App ------------------------------> Cache
 |                                    |
 |<----------- hit -------------------+  return value
 |
 |             miss
 +-- 2. SELECT --------------------> Database
 |<---------------------------------- value
 +-- 3. SET key + TTL -------------> Cache
```

On update, a common sequence is:

```text
write authoritative database -> invalidate cached key -> next read repopulates
```

Redis documents this cache-aside shape for repeated read-heavy workloads ([Redis, *Cache-aside*](https://redis.io/docs/latest/develop/use-cases/cache-aside/)).

#### The central cache questions

1. What is the authoritative source of truth?
2. What exactly is the key?
3. How stale may the value be?
4. How is it invalidated or versioned?
5. What happens on a miss or cache outage?
6. Can the database survive that fallback load?
7. Is one key much hotter or larger than others?

#### Important failure modes

- **Stale data:** the cached value outlives a write.
- **Cache stampede:** a popular key expires and many callers regenerate it simultaneously. Use request coalescing/single-flight, jittered TTLs, or refresh-ahead where appropriate.
- **Hot key:** one item receives more traffic than one cache node/network path can serve. Replicate the value, use local caches, split the key if semantically possible, or protect it with request coalescing.
- **Cold start:** after a restart, misses transfer a surge to the database. Warm gradually or prewarm a justified hot set.
- **Cache penetration:** repeated requests for nonexistent keys reach the database. Cache safe negative results briefly or validate early.
- **Eviction surprise:** memory pressure removes data before TTL. Treat cache contents as disposable unless the chosen system is intentionally the source of truth.

#### When not to cache

- low-reuse data where hit rate will be poor;
- security-sensitive data without a correct identity-aware key;
- values that require immediate correctness unless invalidation is rigorous; or
- an unbounded query space that turns memory into a second uncontrolled database.

### 4.9 Stage 7: scale database reads with replicas

In a common primary-replica arrangement, writes go to one primary and changes are copied to read-only replicas:

```text
                              +--> Read replica 1 ---> stale-tolerant reads
App ---> Primary DB ----------+--> Read replica 2 ---> reports/search support
           ^ writes           |
           +------------------+ replication stream
```

Read replicas reduce load on the primary for read-heavy workloads. Amazon RDS documentation, for example, describes asynchronous read replicas and warns that they can be stale ([AWS RDS, *Working with read replicas*](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReadRepl.html)).

Design consequences:

- **Replica lag:** a just-written value may not yet appear on a replica.
- **Read-your-writes:** after a user changes a profile, route that user's immediate read to the primary, carry a version/watermark, or update the UI optimistically within a safe contract.
- **Failover:** promotion changes the writer and must avoid two primaries accepting conflicting writes. Read scaling and automatic high-availability failover are related but not identical features.
- **Long queries:** reporting on replicas prevents some primary load, but can still cause lag or resource contention.
- **Connections:** replicas need their own routing and pool management.

Read replicas do not scale write throughput. If writes or total data exceed one primary's practical boundary, partitioning or a different storage model may eventually be needed.

### 4.10 Stage 8: move deferrable work behind durable queues

Not every task must finish before responding to the user. A message queue or event log decouples accepting work from processing it:

```text
Client -> API -> durable DB change/outbox -> Queue ---> Worker 1 -> email
                                                  +--> Worker 2 -> thumbnail
                                                  +--> Worker 3 -> analytics
```

Use asynchronous processing for work such as:

- email/push delivery;
- image/video transformation;
- feed materialization;
- search indexing;
- analytics and audit pipelines;
- slow third-party synchronization; and
- retryable batch processing.

Benefits:

- absorbs short bursts;
- lets workers scale independently;
- isolates slow consumers from interactive latency; and
- provides a durable place to retry.

Costs and required correctness:

- a `202 Accepted` response means accepted for processing, not necessarily completed;
- users may observe eventual consistency;
- duplicates can occur, so consumers need **idempotency**;
- ordering is normally scoped to a partition/key, not the entire system;
- poison messages require bounded retries, diagnosis, and a dead-letter workflow;
- backlog age, not just message count, must be monitored; and
- queue capacity does not fix a consumer that can never catch up.

An operation is **idempotent** when repeating the same logical request does not repeat its business effect. For example, charging with a unique payment idempotency key must not create a second charge.

Amazon SQS describes queues as a way to decouple and scale distributed components ([Amazon SQS documentation](https://docs.aws.amazon.com/sqs/)). Apache Kafka partitions topics so producers and consumers can scale across brokers; ordering is guaranteed within a topic-partition, not across all partitions ([Apache Kafka documentation](https://kafka.apache.org/documentation/)).

#### The database-to-queue atomicity gap

This is a common interview follow-up:

```text
1. save order in DB       succeeds
2. publish OrderCreated   process crashes before publish
```

Writing the database and broker separately can lose an event or publish it twice. The **transactional outbox** pattern writes both the business row and an outbox row in one database transaction; a relay later publishes the outbox record. Publishing remains at-least-once, so consumers still deduplicate.

### 4.11 Stage 9: partition data and write load

**Partitioning** divides a dataset into subsets. **Sharding** commonly means placing those subsets on independently scalable database nodes.

```text
                         shard(user_id)
Request -> shard router ------------------+--> Shard 0: users A-F
                                         +--> Shard 1: users G-M
                                         +--> Shard 2: users N-Z
```

The partition key controls where data lives and which node handles an operation.

#### Common strategies

| Strategy | Example | Strength | Main risk |
|---|---|---|---|
| Hash | `hash(user_id) mod N` or virtual buckets | Spreads random IDs | Range queries scatter; naive modulo makes rebalancing expensive |
| Range | time or customer ID ranges | Efficient range scans | Sequential/current range becomes hot |
| Directory | lookup table maps tenant to shard | Flexible placement/moves | Directory becomes critical infrastructure |
| Geographic | user home region | Locality and data residency | Traveling users and cross-region operations |
| Tenant | all tenant data together | Isolation, moves, per-tenant controls | One giant tenant becomes a hot shard |

Managed partitioned stores make routing easier, but key design still matters. DynamoDB documents that poorly distributed partition keys can create hot partitions and throttling ([AWS DynamoDB, *Designing partition keys to distribute workload*](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-partition-key-uniform-load.html)).

#### Consistent hashing, briefly

With naive `hash(key) mod N`, changing the number of shards changes many mappings. **Consistent hashing** maps keys and nodes onto a logical ring so adding/removing a node moves only portions of keyspace. Virtual nodes improve balance.

It helps placement, but does not solve:

- a single celebrity/hot key;
- cross-shard transactions;
- secondary indexes;
- global uniqueness;
- rebalancing bandwidth; or
- application queries that do not contain the partition key.

#### The sharding tax

Before sharding, plan for:

- routing and shard-map changes;
- online resharding while reads/writes continue;
- scatter-gather query latency;
- joins and transactions crossing shards;
- globally unique ID generation;
- per-shard schema migration, backup, and restore;
- uneven tenants and hot partitions;
- referential integrity across shards; and
- failure isolation so one shard does not stall all requests.

Shard only when the storage/write/locality requirement justifies this permanent complexity. First exhaust indexing, query fixes, archival, vertical scaling, caching, read replicas, and workload-specific storage where appropriate.

### 4.12 Stage 10: split independently changing or scaling capabilities

As product and team complexity grow, one deployment may contain very different workloads:

- image processing is CPU-heavy;
- chat holds long-lived connections;
- search uses a specialized index;
- checkout needs strict business invariants; and
- notifications are asynchronous and bursty.

A modular monolith can still scale horizontally. Extract a service when there is a clear reason such as:

- an independent scaling profile;
- failure isolation;
- a stable business boundary with its own data ownership;
- a specialized runtime/storage need; or
- independent deployment/team ownership whose value exceeds network and operational cost.

```text
                         +--> Catalog service ---> catalog store/cache
Client -> API gateway ---+--> Order service -----> order DB
                         +--> Realtime gateway --> connection state
                                      |
                                      v
                                  event stream -> workers
```

Microservices do not automatically create capacity. A synchronous chain of five services may add latency and five failure opportunities. A hundred services sharing one database are not independently scalable in their data layer.

### 4.13 Stage 11: global and billion-scale operation

At sufficiently large global load, one region becomes a latency, capacity, and blast-radius problem. A common model is multiple **cells**: repeated, mostly independent slices containing the resources needed for a subset of users/tenants.

```text
                              +---------------- Global control plane ----------------+
                              | placement, config, deploy metadata, global directory |
                              +--------------------------+----------------------------+
                                                         |
User -> DNS/Anycast/edge -> global traffic policy --------+----------------------+
                          |                                                      |
                          v                                                      v
                 Region A / Cell A1                                      Region B / Cell B1
              +----------------------+                                +----------------------+
              | LB -> app fleet      |                                | LB -> app fleet      |
              | cache + queue        |<---- deliberate replication --->| cache + queue        |
              | local data shards    |                                | local data shards    |
              +----------------------+                                +----------------------+
                          |                                                      |
                          +--> Cell A2 ...                                       +--> Cell B2 ...
```

#### Why cells help

- limit the number of users affected by a bad deployment, hot tenant, or failed shard;
- scale by adding a repeated unit;
- test evacuation and recovery one slice at a time; and
- keep most requests local to their data.

Cells trade global simplicity for routing, placement, and cross-cell workflow complexity.

#### Multi-region choices

| Model | How writes work | Advantages | Costs |
|---|---|---|---|
| Active-passive | One primary region writes; standby takes over | Simpler consistency/conflict model | Failover time; idle cost; data loss depends on replication/RPO |
| Active-active, home region | Each entity has one write home; other regions route/replicate | Local writes for home users; fewer conflicts | Directory/routing; traveling users; home-region outage |
| Multi-writer | Several regions accept writes to same logical data | Lowest local write latency and regional independence | Conflict resolution or coordination; hard invariants; complex testing |

The design must say:

- which region owns each write;
- whether replication is synchronous or asynchronous;
- what users see during lag or partition;
- how conflicting changes merge or fail;
- how a region is evacuated and later rejoined;
- how data residency and deletion propagate; and
- which degraded features remain available.

#### Control plane versus data plane

- The **data plane** serves user traffic: read profile, send message, play video.
- The **control plane** changes configuration and placement: deploy version, create tenant, move shard, update policy.

Keeping the data plane able to use last-known-good configuration when the control plane is unavailable prevents a management outage from becoming a total serving outage.

#### Billion scale is about distribution and isolation

At this point, capacity comes from partitioned work:

- users/data partitioned among cells and shards;
- content served from edges;
- asynchronous tasks divided among queue partitions;
- independent services scaled by workload;
- regional traffic kept local; and
- overload/failure contained rather than allowed to spread.

One global, strongly consistent write path for every feature is rarely affordable in latency or availability. Reserve coordination for invariants that truly require it.

## 5. Reliability when load exceeds capacity

Adding normal capacity is only half the design. The system also needs a deliberate response when demand exceeds that capacity.

### 5.1 The overload failure loop

```text
more traffic
    |
    v
resource saturates -> latency rises -> more work stays in flight -> clients time out
       ^                                                        |
       |                                                        v
       +--------------- retries add more traffic <--------------+
```

Queues can move this loop rather than remove it. An unbounded in-memory queue keeps accepting work until latency and memory collapse.

### 5.2 Protection mechanisms and their different jobs

| Mechanism | Plain-language job | Key trade-off |
|---|---|---|
| Rate limiting | Restrict a caller/tenant/operation over time | Fairness versus legitimate bursts |
| Admission control | Accept work only if capacity/budget exists | Some requests fail early |
| Backpressure | Tell upstream to slow down | Requires cooperative protocol/clients |
| Load shedding | Reject lower-priority/excess work cheaply | Controlled partial unavailability |
| Graceful degradation | Do less-expensive work | Lower result quality/features |
| Timeout/deadline | Stop waiting/working after usefulness expires | Too short causes false failures; too long holds resources |
| Retry with backoff+jitter | Retry transient failure without synchronized storms | Adds load and duplicate risk |
| Circuit breaker | Temporarily stop calls to a failing dependency | May reject calls during recovery or mask partial health |
| Bulkhead | Isolate resource pools by operation/tenant | Lower utilization efficiency; more configuration |
| Queue bound | Cap waiting work/age | Must reject, spill, or degrade when full |

The system should preserve **goodput**: complete as much useful, timely work as possible. Amazon recommends rejecting excess work before overload turns latency into broad unavailability; Google recommends degraded responses and carefully bounded retries ([Amazon load-shedding guidance](https://aws.amazon.com/builders-library/using-load-shedding-to-avoid-overload/); [Google SRE overload guidance](https://sre.google/sre-book/handling-overload/)).

### 5.3 Safe retries require four decisions

1. **Which failures are transient?** Retry a timeout/overload only when another attempt may help; do not retry validation errors.
2. **Is the operation idempotent?** A timed-out write may have committed even though its response was lost.
3. **Where is the retry performed?** Retrying at every layer multiplies traffic. Choose one appropriate layer and a retry budget.
4. **When is it retried?** Use exponential backoff, randomized jitter, and an overall deadline.

Example retry amplification:

```text
Client retries 3x -> API retries 3x -> service retries DB 3x

One logical request can produce up to 3 x 3 x 3 = 27 attempts at the bottom.
```

### 5.4 Graceful degradation examples

- Serve a cached/stale product page but disable checkout if price/inventory cannot be verified.
- Show a chronological feed if ranking is unavailable.
- Stop autoplay previews while preserving video playback.
- Accept a message durably and delay nonessential presence/read-receipt updates.
- Reject analytics/crawler traffic before human checkout traffic.

Degradation rules are business decisions. They must be designed and tested before an incident.

### 5.5 Capacity planning must include failure headroom

If a fleet needs 10 instances to serve peak traffic, deploying exactly 10 means one failure causes overload. Capacity planning should consider:

- loss of a node, zone, or region required by the availability target;
- deployment capacity while old and new versions overlap;
- traffic forecast error and burstiness;
- autoscaling/warm-up delay;
- cache-cold and replica-recovery load; and
- dependency quotas.

The exact margin is workload- and risk-specific. Establish it with load and failure tests rather than a copied percentage.

## 6. Symptom-to-decision guide

| Observed symptom | Verify first | Likely options | New trade-off to mention |
|---|---|---|---|
| App CPU saturated; DB healthy | Profile confirms parallel request work | Optimize; scale up; add stateless instances/LB | Coordination, deploys, session state, cost |
| Static/media bandwidth dominates | Cacheability and access geography | CDN, object storage, compression | Invalidation, cache keys, egress, cold misses |
| Same reads repeatedly hit DB | Query correctness and reuse/hit potential | Cache-aside, local cache, materialized view | Staleness, stampede, invalidation |
| Read-heavy DB CPU/I/O | Index/query plan, pool behavior | Tune; cache; read replicas | Replica lag and read routing |
| Write primary is bottleneck | Lock/contention/key distribution | Batch; data-model change; partition/shard; purpose-fit store | Cross-shard operations and resharding |
| Burst overwhelms slow side work | Can work be delayed? Backlog recovery rate? | Durable queue and worker autoscaling | Eventual consistency, duplicates, DLQ |
| One tenant/key overloads one shard | Per-key distribution | Isolate tenant, split/salt key, replicate hot reads, rate limit | More routing/merge logic |
| Far-away users have high latency | Network versus server latency | CDN; regional read/compute; home-region data | Replication, consistency, residency |
| Dependency failure consumes all threads | Deadlines/retry count/pool sharing | Circuit breaker, bulkhead, bounded concurrency | False rejection, tuning |
| Fleet collapses after traffic spike | Goodput curve, queues, retries, health behavior | Admission control, load shedding, degradation | Intentional rejected work |
| Deployments are the scaling bottleneck | Build/deploy coupling, ownership | Modularize; independent service boundary | Distributed operations and contracts |

This table is a diagnostic map, not a product-selection chart. Every change should have a measurable expected effect and a rollback path.

## 8. Lessons from real systems

The safest way to use company architecture stories is to extract a design lesson, not copy the whole stack. Workload, team, time period, and constraints matter.

### 8.1 Netflix: separate control/API work from the video data path

**Public production snapshot:** Netflix describes Open Connect as its purpose-built content delivery network. It places Open Connect Appliances in or near ISP networks so video can be delivered locally; Netflix's current Open Connect site says it partners with over a thousand ISPs ([Netflix Open Connect](https://openconnect.netflix.com/en/)). An earlier 2016 explanation distinguished general application logic in AWS from the video delivery path through Open Connect ([Netflix, *How Netflix Works With ISPs Around the Globe*](https://about.netflix.com/en/news/how-netflix-works-with-isps-around-the-globe-to-deliver-a-great-viewing-experience)).

```text
Open app / search / choose title
Viewer ----------------------------> application/control services

Press Play
Viewer <---------------------------- nearby Open Connect cache
                                              ^
                                              |
                                  content filled before/on demand
```

**Lesson:** classify traffic. A personalized metadata API and a multi-megabit video stream need different scaling strategies. Moving large reusable bytes close to viewers reduces origin bandwidth, transit distance, and latency.

**Do not infer:** that every product should build its own CDN. A commercial CDN is the sensible choice until economics, control, and traffic volume justify purpose-built delivery.

### 8.2 YouTube: optimize the dominant pipeline and cache bytes intelligently

**Public production snapshots:** Google's 2023 HALP paper says YouTube uses a CDN and describes a learned DRAM cache-eviction policy deployed in production since early 2022; the reported historical result was an average 9.1% reduction in peak byte misses for 1.8% CPU overhead ([Google Research, *HALP*](https://research.google/pubs/halp-heuristic-aided-learned-preference-eviction-policy-for-youtube-content-delivery-network/)). A 2021 YouTube engineering article described specialized Video Coding Units for the expensive transcoding workload ([YouTube, *Reimagining video infrastructure*](https://blog.youtube/inside-youtube/new-era-video-infrastructure/)).

```text
Creator -> upload -> durable source object -> transcoding fleet -> renditions
                                                            |
Viewer <- CDN edge/cache <-----------------------------------+
```

**Lesson:** scaling is workload-specific.

- Upload storage, transcoding compute, metadata, recommendations, comments, and video delivery are separate bottlenecks.
- A cache's **byte miss ratio** may matter more than object hit count when objects have very different sizes.
- At enormous repeated compute volume, specialized hardware can become economical—but it is far beyond an early product's needs.

### 8.3 Uber: scale the organization and independent workloads, not only RPS

**Public production snapshot:** Uber wrote in 2015 that it began with a monolith for one offering in one city, then moved toward service-oriented/microservice architecture as cities, products, code coupling, deployment activity, and the engineering organization grew ([Uber Engineering, *Service-Oriented Architecture: Scaling the Uber Engineering Codebase As We Grow*](https://www.uber.com/en-LT/blog/service-oriented-architecture/)).

```text
Early: one product/city                  Later: different workload boundaries

Rider + driver + payment                trip lifecycle service
        |                               location/matching service
     monolith                           pricing service
                                        payment service
```

**Lesson:** one traffic number does not force microservices. A boundary becomes useful when capabilities have different data, failure, scaling, or release needs. Organizational scalability is real, but it must pay for network calls, versioned contracts, distributed tracing, and data consistency.

### 8.4 WhatsApp: connection scale and fan-out are different from request scale

**Public production snapshot:** Meta's 2021 explanation of WhatsApp multi-device says companion devices connect independently. For one-to-one messages, the sending client encrypted and transmitted to each device in the sender's and receiver's device lists—a client-fan-out approach in that described architecture ([Meta Engineering, *How WhatsApp enables multi-device capability*](https://engineering.fb.com/2021/07/14/security/whatsapp-multi-device/)).

```text
Sender phone
   |-- encrypted copy --> Receiver phone
   |-- encrypted copy --> Receiver laptop
   +-- encrypted copy --> Sender's linked desktop
```

**Lesson:** model the unit of scale accurately.

- A messaging gateway may be limited by open connections, memory per connection, heartbeats, and reconnect storms rather than normal HTTP RPS.
- One logical message can create several deliveries.
- Offline delivery, ordering, deduplication, encryption keys, and device membership are part of capacity and correctness.

**Do not infer:** that this is a complete or current WhatsApp backend description. It is one public, dated explanation of a multi-device path.

### 8.5 Twitter/X: cache architecture and celebrity fan-out need explicit treatment

**Public production snapshot:** a 2017 Twitter engineering post described specialized cache clusters for users, Tweets, and timelines. It reported historical cache-cluster rates of 10M-50M QPS and described Haplo as a timeline cache written by timeline/fan-out services ([Twitter Engineering, *The Infrastructure Behind Twitter: Scale*](https://blog.x.com/engineering/en_us/topics/infrastructure/2017/the-infrastructure-behind-twitter-scale)). These are dated public figures, not current X architecture claims.

Two feed strategies illustrate the scaling problem:

```text
Fan-out on write                          Fan-out on read

Author posts                              Reader opens feed
    |                                          |
    +--> append to follower A inbox            +--> fetch posts from followees
    +--> append to follower B inbox            +--> merge/rank now
    +--> append to ...
```

| Strategy | Advantage | Cost |
|---|---|---|
| Fan-out on write | Fast feed read | Expensive write amplification; celebrity problem; storage duplication |
| Fan-out on read | Cheap post creation | Expensive read/merge; latency for users following many accounts |
| Hybrid | Treat ordinary and celebrity accounts differently | More logic and consistency cases |

**Lesson:** averages hide skew. A normal user and an account with millions of followers should not necessarily take the same path.

### 8.6 Amazon: persistent state determines availability trade-offs

**Public production snapshots:** Amazon's 2007 Dynamo paper described a highly available key-value store used by core services, including workloads such as shopping carts. It partitioned and replicated data and accepted version conflicts in some failure cases to prioritize availability ([Amazon Science, *Dynamo: Amazon's highly available key-value store*](https://www.amazon.science/publications/dynamo-amazons-highly-available-key-value-store)).

A later DynamoDB paper reported that during the 66-hour 2021 Prime Day event, Amazon systems made trillions of DynamoDB API calls with a peak of 89.2 million requests/second. This is a **verified historical number from Amazon's 2022 paper**, not a general DynamoDB guarantee or a current traffic claim ([Amazon Science, *Amazon DynamoDB: A scalable, predictably performant, and fully managed NoSQL database service*](https://www.amazon.science/publications/amazon-dynamodb-a-scalable-predictably-performant-and-fully-managed-nosql-database-service)).

**Lesson:** at large scale, failures are normal and data semantics drive architecture.

- Shopping-cart availability may justify reconciling concurrent versions.
- A payment ledger may instead reject or coordinate an operation rather than merge balances later.
- "Use NoSQL for scale" is not a design. State the access pattern, partition key, consistency need, hot-key behavior, and transaction boundary.

### 8.7 Cross-system comparison

| System story | Dominant scaling lesson |
|---|---|
| Netflix | Put high-volume reusable content near viewers; separate data delivery from control APIs |
| YouTube | Split ingest, compute, metadata, and delivery; optimize cache by actual byte cost |
| Uber | Product/team/deployment growth can justify service boundaries independently of raw traffic |
| WhatsApp | Count connections, devices, deliveries, and reconnect storms—not just API requests |
| Twitter/X | Skew and fan-out dominate average traffic; use specialized/hybrid paths |
| Amazon | Partitioned replicated state requires explicit availability/consistency/conflict choices |

## 9. Trade-offs and alternatives

### 9.1 Component decision table

| Technique | Use when | Avoid/postpone when | Price you pay |
|---|---|---|---|
| Larger single node | Fastest measured relief; workload still fits one node | Need independent failure domains or beyond hardware ceiling | Bigger blast radius, ceiling, nonlinear price |
| Horizontal app fleet | App work is parallelizable and instances can be replaceable | Bottleneck is shared DB/downstream | LB, session/state placement, deployments, more connections |
| CDN | Content is reusable or users are geographically distant | Highly personalized/private responses cannot be keyed safely | Invalidation, edge debugging, provider/egress policy |
| Distributed cache | Repeated reads or expensive computation have high reuse | Low hit rate or immediate freshness dominates | Staleness, stampede, hot keys, memory cost |
| Read replicas | Read-heavy DB; stale reads are acceptable for chosen routes | Primary writes/locks are bottleneck; all reads need immediate freshness | Lag, routing, failover complexity, cost |
| Durable queue | Work can finish asynchronously or bursts need smoothing | Caller requires immediate final result/invariant | Eventual completion, duplicates, ordering, backlog operations |
| Sharding | One write/storage/locality boundary is no longer sufficient | Indexing, archival, cache, replicas, or scale-up solve it | Routing, resharding, cross-shard transactions/queries |
| Microservices | Clear domain/scaling/failure/deployment boundary | Small team/product; boundaries unstable; shared DB remains | Network failure, contracts, observability, operational load |
| Multi-region active-passive | Regional disaster recovery with simpler write ownership | Failover time/RPO cannot meet requirements | Standby cost, rehearsals, promotion/failback |
| Multi-region active-active | Regional independence/local writes are essential | Strong global invariants dominate or team cannot operate it | Conflicts/coordination, testing, high cost |
| Cells | Need repeatable scale and blast-radius isolation | Small system without automation/placement needs | Directory, routing, fleet consistency, capacity fragmentation |

### 9.2 VMs, containers, orchestration, and serverless

These are execution/operations choices, not scaling laws:

| Option | Strength | Watch for |
|---|---|---|
| Virtual machines | Familiar isolation and control; good for steady workloads | Image patching, slower provisioning, bin-packing |
| Containers + orchestrator | Portable packaging, scheduling, rolling updates, service discovery | Control-plane/cluster complexity; resource tuning; does not fix app state |
| Serverless functions/platform | Per-request elasticity, low idle operations, managed scaling | Cold starts, concurrency/quotas, runtime limits, vendor coupling, dependency overload |

"Put it on Kubernetes" or "make it serverless" does not answer how the database, state, ordering, overload, or regional failure works.

### 9.3 SQL versus NoSQL is not small versus large

Relational databases can scale very far with good schemas, indexes, replicas, partitioning, and operations. NoSQL systems can fail under a poor partition key.

Choose from requirements:

- access/query patterns;
- transaction and constraint needs;
- consistency model;
- data shape and evolution;
- write/read throughput and storage;
- hot-key distribution;
- multi-region behavior; and
- operational maturity.

Polyglot storage can be justified—for example, relational orders, object media, search index, and ephemeral cache—but every datastore adds backup, security, monitoring, and expertise costs.

### 9.4 Cost is a first-class non-functional requirement

At scale, architecture changes which unit is expensive:

- CDN/cache trades memory/edge fees for origin compute and network egress;
- compression trades CPU for bytes;
- replication trades storage/network for availability/read capacity;
- overprovisioning trades money for burst/failure headroom;
- serverless trades per-unit premium for low idle cost and operational simplicity;
- active-active regions trade substantial cost/complexity for latency and resilience.

Discuss cost per useful request, message, stored GB-month, or streamed GB—not merely total machine count.

## 10. Edge cases and common mistakes

### 10.1 Mistakes in the scaling story

1. **Equating registered users with load.** Convert behavior into peak RPS, connections, bytes, writes, and storage.
2. **Jumping directly to billion-scale components.** Start with a working path, then evolve around stated constraints.
3. **Scaling every layer equally.** The bottleneck moves; measure each resource.
4. **Calling microservices a traffic solution.** A replicated monolith may scale; a chat gateway and image worker may need separate scale for specific reasons.
5. **Ignoring the database.** Adding app nodes can amplify DB connections and queries.
6. **Assuming autoscaling is instantaneous.** Startup, warm-up, signal delay, and quotas create a gap.
7. **Treating a queue as infinite capacity.** Backlog must drain faster than it grows over the relevant window.
8. **Treating cache as free.** State freshness, stampedes, key security, outage fallback, and hit rate.
9. **Treating replicas as synchronous and fresh.** State what replication and read semantics are used.
10. **Sharding without a query-driven key.** A uniformly distributed key that every query must scatter across is not a win.
11. **Ignoring skew.** One tenant, celebrity, live event, or current-time partition can dominate.
12. **Claiming "exactly once" end to end.** Explain broker guarantee scope, idempotent effects, deduplication, and transactional boundaries.
13. **Retrying at every layer.** This creates retry storms and duplicated writes.
14. **Unbounded timeouts, queues, result sets, or concurrency.** Bounded work is easier to protect and estimate.
15. **Confusing scaling with high availability.** Extra capacity in one failure domain does not survive that domain.
16. **Adding multi-region arrows without a write owner.** Define conflicts, consistency, failover, RPO, and RTO.
17. **Forgetting deletion and privacy.** Cached, replicated, indexed, backed-up, and analytical copies need a lifecycle.
18. **Ignoring deployment load.** Rolling deploys temporarily reduce or duplicate capacity and can cold-start caches.
19. **No overload mode.** "Autoscaling will handle it" is not a safe answer beyond tested limits.
20. **Naming products without semantics.** "Redis + Kafka + Cassandra" says nothing about keys, messages, guarantees, or failure recovery.

### 10.2 Subtle scenarios an interviewer may probe

#### Cache fails completely

If every request falls through at once, the database may collapse. Options include local fallback caches, request coalescing, database admission control, stale serving, feature degradation, and gradually warming a recovered cache. A cache used for capacity has become a critical dependency even if it is logically disposable.

#### A celebrity account joins one shard

Hashing by `user_id` distributes ordinary users but cannot split one user's traffic. Separate that hot user's fan-out/read path, cache/replicate immutable data, subdivide by another dimension, and rate-limit abusive demand. Uniform key count is not uniform work.

#### Traffic rises, but CPU stays low

Possible bottlenecks include database pool waits, remote I/O, lock contention, rate limits, network, thread pools, or a downstream API. CPU-based autoscaling may do nothing. Measure queue/wait time and scale or protect the constrained resource.

#### One region disappears

Routing traffic elsewhere is safe only if the destination has capacity and the required data. Otherwise failover causes a secondary overload. Pre-plan capacity, replication, write ownership, DNS/traffic-manager behavior, dependency reachability, and failback.

#### A message is processed twice

Give the logical operation a stable ID. Store the processed ID or enforce a unique business constraint in the same transaction as the side effect where possible. Acknowledge the message only after durable processing. Deduplication retention must cover the redelivery window.

#### All cache entries expire at midnight

Synchronized TTLs create a thundering herd. Add randomized TTL jitter, refresh ahead, or versioned gradual migration; protect the database even if prevention fails.

#### An app deployment doubles DB connections

During rolling deployment both old and new instances coexist. If each has a pool of 100, the total may exceed the DB limit. Budget connections across maximum surge capacity, use a proxy/pooler if appropriate, and test deployment—not only steady state.

## 11. Interview angle

### 11.1 How server scaling appears in interviews

It may be the whole prompt:

- "How would you scale this service from 100 users to 100 million?"

More often it is a sequence of follow-ups after a basic design:

- "Traffic grows 10x tomorrow. What breaks first?"
- "Your database is at 90% CPU; what do you do?"
- "Now users are global."
- "What happens if the cache or an availability zone fails?"
- "One customer produces half the traffic."
- "How do you keep writes correct with retries?"

The interviewer is evaluating whether you can:

1. turn ambiguity into workload and quality requirements;
2. build a correct simple system first;
3. locate and remove successive bottlenecks;
4. understand state, consistency, and failure;
5. explain trade-offs rather than list technologies; and
6. operate the design: observe, test, deploy, recover, and control cost.

### 11.2 Sourced questions from public candidate posts

These are **self-reported interview experiences**, not verified official company questions. They are included because the user specifically requested real posts/threads.

| Public post | What the candidate reported | Scaling lesson to practice |
|---|---|---|
| [Oracle interview experience, LeetCode](https://leetcode.com/discuss/interview-experience/5808980) | One round included "scale a microservice to a million users"; the same post mentions notification-system HLD follow-ups | Refuse to treat user count as sufficient; ask traffic, state, availability, and growth pattern before choosing components |
| [Adobe senior SWE interview experience, LeetCode](https://leetcode.com/discuss/post/5784907/) | A legacy monolith described as supporting 100K users had to scale to one million | Diagnose whether replication of the monolith is safe; sessions, DB contention, routing, and migration matter more than merely multiplying by ten |
| [Apollo EM interview experience, LeetCode](https://leetcode.com/discuss/post/6624380/) | Design live comments with one million simultaneous streams and ten million connected users, seven-day history, top comments, and some delivery delay allowed | Model persistent connections, comment fan-out, hot streams, partitioning, history retention, moderation, and eventual delivery separately |
| [HashedIn SDE1 interview experience, LeetCode](https://leetcode.com/discuss/interview-experience/1204788/hashedin-sde1-bangalore-apr-2021) | The candidate reported a follow-up to scale a design to 10M+ users | Show staged evolution and quantify load; do not bolt every distributed component onto the first diagram |

What these posts actually establish is narrow: candidates publicly described receiving scale-oriented prompts. They do not establish an official rubric, the accuracy of all details, or a reusable "correct answer."

### 11.3 Answer spine: **SCALE**

Use these cues to reconstruct a live answer rather than memorize a paragraph.

#### S — Scope the requirements

- What does the product do?
- Registered, active, concurrent, or connected users?
- Read/write mix, payload, peak shape, retention, geography?
- Latency, availability, durability, consistency, RPO/RTO?
- Which flow is critical and which work may be delayed?

#### C — Calculate the dominant load

- operations/day -> average RPS -> peak RPS;
- RPS x payload -> bandwidth;
- writes x bytes x retention -> storage;
- RPS x latency -> in-flight concurrency;
- identify skew/fan-out, not only averages.

Use round numbers and state assumptions. The goal is an order of magnitude and bottleneck, not false precision.

#### A — Assemble the simplest correct path

- client -> edge/LB -> application -> authoritative data store;
- large blobs -> object storage/CDN;
- include authentication, metrics, backups, and timeouts;
- keep state ownership and core invariant explicit.

#### L — Locate and lift one limit at a time

- app limit -> optimize/scale up or stateless horizontal fleet;
- repeated reads -> cache;
- read-primary limit -> replicas;
- deferrable burst -> queue/workers;
- write/storage limit -> partition/shard;
- global latency/blast radius -> regional cells;
- independent workload/team boundary -> service extraction.

For each addition say: **trigger -> mechanism -> benefit -> new problem**.

#### E — Engineer failure and explain trade-offs

- failure domains and headroom;
- cache miss/outage, replica lag, duplicates, hot keys;
- timeouts, retry budget, idempotency, backpressure/load shedding;
- observability, load/failure tests, safe rollout/rollback;
- cost and what you intentionally postpone.

### 11.4 A 45-minute pacing template

| Time | Focus |
|---:|---|
| 0-5 min | Clarify functional/non-functional requirements and scale vocabulary |
| 5-10 min | Back-of-envelope estimates; identify likely dominant axis |
| 10-20 min | Draw a simple end-to-end design and one critical read/write flow |
| 20-35 min | Deep dive into bottleneck, data model/partitioning, and failure correctness |
| 35-42 min | Evolve for larger scale, overload, multi-zone/region, observability |
| 42-45 min | Summarize trade-offs, limits, and next change |

Adapt to interviewer steering. A system design interview is a conversation; do not race through a rehearsed catalog of boxes.

### 11.5 Example live walkthrough: "scale our API from zero to a billion users"

An interview-quality response can proceed like this:

1. **Clarify:** "Does billion mean registered or daily active? What peak RPS, regions, payloads, and write ratio? I'll assume the workload from Section 3 unless you prefer another."
2. **State invariant:** "User/account writes are authoritative in the database; cached profile reads may be slightly stale, but authentication and payment-like changes cannot be stale."
3. **Start:** one deployable application, relational DB, object storage, backups, metrics, and HTTPS.
4. **First bottleneck:** tune indexes/queries and scale vertically based on tests.
5. **Read/bandwidth growth:** CDN for static/media; cache repeated public reads.
6. **App growth/failure:** stateless replicas behind a multi-zone LB with external session state and deployment headroom.
7. **Database read growth:** route explicitly stale-tolerant reads to replicas; preserve read-your-writes where needed.
8. **Slow/bursty work:** durable queue, scalable idempotent consumers, outbox for DB-to-event reliability.
9. **Write/data growth:** choose a high-cardinality partition key from access patterns, plan hot-key and resharding behavior, minimize cross-shard transactions.
10. **Global growth:** assign users/tenants to home-region cells, serve reusable bytes at the edge, replicate intentionally, and define regional failure/degraded mode.
11. **Protect:** deadlines, retry budgets, admission control, load shedding, rate limits, and load/failure tests beyond the expected peak.
12. **Close:** "I would add each layer only after its trigger is measured; the next likely bottleneck is X, and the principal trade-off is Y."

### 11.6 Own plausible follow-ups and answer cues

The following are **synthesized practice questions**, not sourced interview reports.

#### "Why not start with microservices and sharding so we never migrate?"

Answer cues:

- requirements and boundaries will change while the product is being discovered;
- distributed calls, partial failure, contracts, resharding, and cross-shard invariants are permanent costs;
- preserve escape hatches: modular boundaries, opaque IDs, object storage, stateless app, observability;
- introduce the specific distribution only when its benefit is measurable.

#### "The cache gives a 95% hit rate. Can the database be sized for only 5%?"

Answer cues:

- steady-state yes for performance planning, no for safe failure planning;
- cache outage/cold start can send 100% or concentrated hot traffic to DB;
- protect DB with admission control, stale serving, coalescing, gradual warm-up;
- inspect byte/work-weighted hit rate and misses by route, not one global percentage.

#### "Why doesn't adding ten app servers give ten times throughput?"

Answer cues:

- shared serialized bottleneck: DB, lock, cache key, network, downstream quota;
- coordination and contention overhead; uneven load;
- use end-to-end load test and per-resource saturation;
- Amdahl-like limit: the non-parallel portion caps speedup.

#### "How do you keep a user from seeing stale data after a write?"

Answer cues:

- identify where read-your-writes is required;
- route immediate reads to primary, use session/version watermark, or update client view;
- do not globally require strong reads if only one flow needs them;
- define timeout/failure behavior when primary is unavailable.

#### "Autoscaling is enabled, so why did the service fail during a spike?"

Answer cues:

- detection/provision/warm delay; max quota; poor signal; cold caches;
- dependency cannot scale or connection pools overload it;
- need headroom, predictive capacity, bounded queues, shedding, and end-to-end tests;
- check goodput and tail latency, not instance count alone.

#### "How would you handle one viral item?"

Answer cues:

- measure per-key skew;
- CDN/cache/local replication for reads;
- request coalescing and TTL jitter;
- partition counters/fan-out then aggregate if exact immediacy is unnecessary;
- priority/rate limits; isolate the hot path from ordinary keys.

#### "Can we accept writes in every region?"

Answer cues:

- ask which invariant and conflict semantics;
- home-region/single-writer avoids same-key conflict with some remote latency;
- multi-writer needs coordination or deterministic/application merge;
- network partition forces a choice between rejecting/delaying some operations and accepting divergence;
- payment balance and profile nickname deserve different answers.

#### "What do you monitor at a billion-user scale?"

Answer cues:

- user-visible SLI: success/correctness and p95/p99 latency;
- offered load versus goodput;
- saturation and queues at each dependency;
- cache hit/miss and replica lag;
- per-cell/shard/tenant skew, not only global averages;
- retry/shed/rate-limit volume, deployment version, cost per unit;
- alert on symptoms and exhausted error budget, then diagnose causes.

### 11.7 Interviewer red flags in an answer

- "Kafka makes it scalable" without message key, partition count reasoning, consumer behavior, or backlog recovery.
- "Redis makes it fast" without cache key, TTL, invalidation, and outage fallback.
- "NoSQL handles billions" without access pattern, partition key, consistency, and hot-key plan.
- "Kubernetes autoscales it" without startup delay and dependency capacity.
- "Use consistent hashing" as if it solved skew and resharding correctness.
- "Active-active" without saying who accepts conflicting writes.
- a giant final diagram with no request flow or evolution trigger.

### 12.1 The scaling ladder in one table

| Pressure | First reasonable move | Later move if still constrained |
|---|---|---|
| Wasteful code/query | Profile, index, bound, batch, pool | Data-model or algorithm change |
| One node resource | Scale up | Horizontalize the parallel portion |
| Static/media bytes | Object storage + CDN | Specialized delivery/edge strategy |
| App request compute | Stateless app replicas + LB | Split independently scaling hot capability |
| Repeated reads | Cache | Partition/replicate cache, materialize views |
| DB reads | Tune/cache/read replicas | Read-optimized store/model |
| Slow/bursty side work | Queue + worker fleet | Partition stream, isolate consumers |
| DB writes/data | Batch/model/archive | Shard/partition or purpose-fit distributed DB |
| Geographic latency/failure | CDN, DR, regional reads | Home-region cells or justified active-active |
| Overload | Headroom, bounded work | Admission control, shedding, degradation |
| Team/deploy coupling | Modular monolith boundaries | Independently owned/deployed services |

### 12.2 For every new box, answer seven questions

1. Which measured requirement or bottleneck caused it?
2. What data/state does it own?
3. How is work routed or partitioned to it?
4. What correctness/consistency guarantee does it provide?
5. What happens when it is slow, unavailable, duplicated, or stale?
6. How is it observed, tested, deployed, recovered, and paid for?
7. What simpler option was rejected, and why?

## 15. True prerequisite gaps to study next

The inline explanations are enough to understand the scaling journey. For implementation-level mastery, the remaining prerequisite gaps are:

- **End-to-end networking:** DNS, TCP/QUIC, TLS, HTTP, proxies, and persistent connections.
- **Database fundamentals:** indexes/query plans, ACID transactions, locking, isolation levels, backups, and recovery.
- **Distributed-data fundamentals:** replication, consistency models, quorum, CAP/PACELC, partitioning, and consensus boundaries.
- **Asynchronous processing:** delivery semantics, idempotency, ordering, transactional outbox, backpressure, and dead letters.
- **Reliability measurement:** SLIs/SLOs/error budgets, percentiles, capacity/load testing, RPO/RTO, and incident recovery.
