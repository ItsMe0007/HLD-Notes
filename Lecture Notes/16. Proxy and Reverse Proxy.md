# Proxy and Reverse Proxy for System Design (HLD)

> Beginner-first study notes expanded from the supplied bullets:
> **differences, advantages and use cases; load balancer, API gateway, CDN; proxy, VPN, and firewall.**
>
> The goal is not to memorize boxes. It is to learn which party an intermediary represents, what work it performs, where trust changes, and what can fail.

## Scope, assumptions, and evidence labels

I assume **proxy** in your first bullet means a **forward proxy**, because “proxy versus reverse proxy” is normally shorthand for “forward proxy versus reverse proxy.” “Proxy” is also the broad family name: both forward and reverse proxies are proxies.

The words in this area overlap:

- A reverse proxy can also perform load balancing, API-gateway policies, caching, and firewall-like filtering.
- A CDN, Layer 7 load balancer, and API gateway commonly behave as reverse proxies.
- One NGINX, Envoy, HAProxy, cloud service, or custom process can perform several roles.

Therefore, compare **responsibilities**, not just product names or diagram boxes.

Evidence labels used below:

- **Verified standard/documentation:** supported by a standards body or first-party documentation linked in the note.
- **Public production example:** described by the company in a public engineering source. It might describe architecture at the publication date, not the company’s complete current private architecture.
- **Illustrative design:** a reasonable way the named product could use the concept; it is not a claim about its private implementation.
- **Candidate-reported interview question:** a self-reported public interview experience. It is useful practice evidence, not an official or independently verified company question bank.

No unsourced production-scale number is presented as a current fact. Worked numbers are explicitly illustrative.

## 1. Plain-language foundation: a proxy is a middleman

### 1.1 Direct communication

Without an intermediary, a client connects to a server:

```text
Client ------------------------------> Server
       request              response
```

For a web request, the browser might:

1. use DNS to find the IP address for `shop.example`;
2. create a TCP or QUIC connection to that address;
3. establish TLS for HTTPS;
4. send an HTTP request;
5. receive the HTTP response.

### 1.2 Communication through a proxy

A proxy stands between the two endpoints:

```text
Client ------------> Proxy ------------> Server
       connection 1          connection 2

Client <------------ Proxy <------------ Server
```

The proxy receives traffic from one side and relays it to the other. For an application-level proxy, there are normally **two separate connections**:

- client to proxy;
- proxy to destination/backend.

That separation lets the proxy inspect, reject, route, cache, transform, record, or prioritize traffic. It also creates another hop, another failure point, and another trusted component.

Some lower-layer load-balancing designs use NAT, packet forwarding, or direct server return instead of two full application connections. That is why “every load balancer is an HTTP reverse proxy” is false.

### 1.3 Forward versus reverse: the one naming rule

Ask:

> **Which side arranged the intermediary, and which side does it represent?**

```text
FORWARD PROXY: represents clients

Client A --\
Client B ----> [ Forward Proxy ] ----> many Internet servers
Client C --/


REVERSE PROXY: represents servers

many Internet clients ----> [ Reverse Proxy ] ---> Server A
                                                -> Server B
                                                -> Server C
```

- A **forward proxy** is chosen or imposed on the client side. It controls client **egress**—traffic going out toward destinations.
- A **reverse proxy** is deployed by the service owner. It controls service **ingress**—traffic coming in toward backends.

The words do **not** mean “requests go forward and responses go in reverse.” Both types carry requests and responses in both directions.

### 1.4 The fastest memory cue

```text
Forward proxy hides/controls a set of CLIENTS from destinations.
Reverse proxy hides/controls a set of SERVERS from clients.
```

“Hides” is a useful first approximation, not a security guarantee. A proxy can deliberately forward the original IP or identity, logs can reveal it, and traffic correlation can still be possible.

## 2. Prerequisites explained inline

These concepts are required for the rest of the chapter. They are introduced here so no prior networking background is assumed. See [Network Protocols](./01.%20Network%20Protocols.md) for the deeper version.

### 2.1 Client, server, and origin

- A **client** initiates an operation, such as a browser asking for a page.
- A **server** accepts a request or connection and provides a service.
- An **origin server** is the authoritative backend from which a proxy or CDN can obtain the response.
- A process can be both: an API gateway is a server to the mobile app and a client to an internal service.

These are roles during an interaction, not permanent machine types.

### 2.2 DNS, IP address, and port

- **DNS** maps a hostname such as `api.example.com` to one or more network addresses.
- An **IP address** is used for routing to a network endpoint.
- A **port** selects a network service at that address, such as TCP port 443 for typical HTTPS.

For a public reverse proxy, DNS normally returns an address belonging to the proxy/edge layer, not an application server:

```text
api.example.com --DNS--> reverse-proxy public address
                               |
                               +--> private application addresses
```

DNS does not itself relay the HTTP request. It only helps the client select an address.

### 2.3 Layers: L4 versus L7

- **Layer 4 (transport):** TCP/UDP connections, IP addresses, and ports. An L4 intermediary can route without understanding an HTTP path.
- **Layer 7 (application):** HTTP method, hostname, path, headers, cookies, or gRPC metadata. An L7 proxy can route `/payments` differently from `/catalog`.

“L4” and “L7” are convenient OSI-model shorthand. Real implementations do not always fit a perfectly clean layer.

### 2.4 HTTP and TLS

- **HTTP** defines requests and responses.
- **TLS** provides encryption, integrity protection, and peer authentication, usually server authentication.
- **HTTPS** is HTTP carried through TLS.

If a proxy **terminates TLS**, the TLS session ends at the proxy. The proxy can read HTTP and usually opens a new connection to the backend. That backend connection may be plaintext, TLS-encrypted, or mutually authenticated with mTLS.

If a proxy uses **TLS passthrough**, it routes encrypted bytes without terminating the client’s TLS session. It has less application information available for routing and policy.

### 2.5 Public and private addresses

Backends often use private addresses and accept traffic only from the reverse-proxy tier. This prevents clients from bypassing authentication, rate limits, WAF rules, and logging at the proxy.

Merely placing a proxy in the diagram does not hide or protect an origin if its public address remains reachable.

## 3. Forward proxy in detail

### 3.1 Plain-language model

Imagine an office in which employees are not allowed to contact arbitrary websites directly. Their browsers send web traffic to an approved company machine. That machine checks policy and makes the outbound request.

```text
Employee browser        Corporate forward proxy        Website
      |                           |                        |
      | request example.com       |                        |
      |-------------------------->|                        |
      |                           | new outbound request   |
      |                           |----------------------->|
      |                           |<-----------------------|
      |<--------------------------|                        |
```

The website sees a connection from the proxy. The proxy may know the employee, device, original destination, and—depending on protocol and TLS mode—request content.

### 3.2 Who configures it?

Common forms are:

- **Explicit proxy:** the browser, operating system, or application is configured with a proxy address.
- **Proxy auto-configuration:** client policy selects a proxy based on the requested URL.
- **Transparent/intercepting proxy:** the network redirects traffic without normal application configuration. HTTPS interception has important certificate and privacy implications.
- **Application-specific proxy:** an application deliberately uses an HTTP or SOCKS proxy.
- **Egress proxy:** workloads in a data center or Kubernetes cluster must use a controlled outbound path.

### 3.3 HTTP through a forward proxy

For plaintext HTTP, a configured HTTP proxy can receive an absolute target and act on the HTTP message:

```http
GET http://example.com/articles/42 HTTP/1.1
Host: example.com
```

For ordinary HTTPS proxying, the client commonly asks the proxy to create a tunnel using `CONNECT`:

```http
CONNECT example.com:443 HTTP/1.1
Host: example.com:443
```

After a successful response, the client performs TLS through the tunnel to `example.com`:

```text
Browser ===== encrypted TLS bytes =====> example.com
              carried by proxy tunnel
```

With a basic CONNECT tunnel, the proxy can observe connection metadata and destination but cannot normally read the encrypted HTTP contents.

### 3.4 TLS-inspecting forward proxy

A corporate inspection proxy can read HTTPS only by becoming a TLS endpoint:

```text
Browser --TLS #1--> Inspection proxy --TLS #2--> Website
                     reads HTTP
```

For the browser to accept this, the organization normally installs and trusts a corporate certificate authority on managed devices. This is not end-to-end TLS between the browser and website; it creates two TLS-protected legs. It introduces:

- a powerful security and privacy trust point;
- certificate/key-management risk;
- application compatibility problems such as certificate pinning;
- legal and policy obligations;
- a high-value target containing sensitive plaintext.

Do not say “an HTTPS proxy can always see passwords.” A tunnel-only forward proxy cannot read properly encrypted application data. An authorized TLS-inspection proxy can.

### 3.5 SOCKS and non-HTTP proxies

A SOCKS proxy relays network sessions without needing HTTP semantics. It can support more kinds of application traffic, but it cannot automatically apply HTTP-path rules or HTTP caching unless another HTTP-aware layer is added.

This reveals a general rule:

> More application awareness enables richer policy, but increases processing, protocol coupling, and exposure to parsing/security bugs.

## 4. Reverse proxy in detail

### 4.1 Plain-language model

A customer requests `https://shop.example/orders/42`. DNS points `shop.example` at the company’s public reverse-proxy tier. The customer does not select a private order-service instance.

```text
Browser                Reverse proxy              Internal services
   |                         |                           |
   | GET /orders/42          |                           |
   |------------------------>| route by host/path        |
   |                         |--------------------------> Order service
   |                         |<--------------------------|
   |<------------------------|                           |
```

To the browser, the reverse proxy behaves like the server for `shop.example`. To the internal order service, the reverse proxy behaves like a client.

The current HTTP semantics standard calls this role a **gateway**, also known as a **reverse proxy** ([RFC 9110, Section 3.7](https://www.rfc-editor.org/rfc/rfc9110.html#section-3.7)).

### 4.2 A typical request lifecycle

1. DNS routes the hostname to a public edge/reverse-proxy address.
2. The client creates a connection to the proxy.
3. The proxy may terminate TLS and validate the HTTP request.
4. It selects a route and a healthy backend.
5. It may enforce authentication, authorization, rate limits, or WAF rules.
6. It opens or reuses a backend connection.
7. It sends a carefully constructed upstream request.
8. The backend responds.
9. The proxy may buffer, stream, cache, compress, or transform the response.
10. The proxy returns a response to the client and records metrics/logs.

NGINX’s first-party documentation describes this receive-forward-fetch-return behavior and also documents header modification and response buffering ([NGINX reverse proxy documentation](https://docs.nginx.com/nginx/admin-guide/web-server/reverse-proxy)).

### 4.3 One public endpoint, many private implementations

The external contract can remain stable while backends change:

```text
api.example.com
      |
      v
+---------------- Reverse Proxy ----------------+
| /users/*     -> user-service-v3               |
| /orders/*    -> order-service                 |
| /legacy/*    -> old monolith                  |
| 5% /search/* -> search-v2 canary              |
+-----------------------------------------------+
```

Clients need not know service addresses, deployment versions, instance count, private protocols, or failover decisions.

## 5. Forward proxy versus reverse proxy

| Question | Forward proxy | Reverse proxy |
|---|---|---|
| Whom does it represent? | Clients | Servers/services |
| Who normally deploys/configures it? | Client, client organization, or egress-network owner | Service/backend owner |
| Primary traffic boundary | Outbound/egress | Inbound/ingress |
| Does the client know it exists? | Usually yes for an explicit proxy; not always for interception | Client knows the public service, but usually not its internal proxy/backend distinction |
| Does the destination/backend know the original peer? | Destination directly sees proxy; identity may be forwarded | Backend directly sees proxy; client identity may be forwarded |
| Typical selection | Destination chosen from the client’s request | Backend chosen from service-owned route configuration |
| Common jobs | Egress allow/deny, audit, shared cache, privacy mediation, outbound identity | TLS termination, routing, load balancing, auth, WAF, caching, canary, origin protection |
| Example software | Squid, an HTTP CONNECT proxy, SOCKS proxy, controlled egress gateway | NGINX, HAProxy, Envoy, cloud L7 load balancer, API gateway, CDN edge |
| Main security danger | Open-proxy abuse, sensitive browsing logs, TLS-inspection trust | Public chokepoint, origin bypass, header spoofing, request smuggling, key exposure |

### 5.1 What each side can see

```text
Forward proxy:
  destination server knows proxy directly;
  proxy may know original client and destination.

Reverse proxy:
  client knows public proxy endpoint directly;
  proxy knows selected backend;
  backend may know original client only through trusted metadata.
```

### 5.2 “Protects the client/server” is incomplete

The popular shortcut is:

- forward proxy protects clients;
- reverse proxy protects servers.

Use it only as a memory cue. Protection exists only when policies are correct:

- A malicious or compromised forward proxy can spy on or alter traffic it can read.
- A reverse proxy cannot protect an origin that is still publicly accessible.
- Either proxy can become a bottleneck, denial-of-service target, or broad blast-radius component.
- Neither automatically supplies authentication, authorization, encryption, malware protection, or anonymity.

## 6. Why proxies are needed in system design

The deeper reason is **indirection**: endpoints depend on a stable intermediary rather than on each other’s physical identity.

### 6.1 Centralize cross-cutting concerns

Without a proxy, every application server may need to independently implement:

- public TLS certificates;
- authentication hooks;
- rate limiting;
- request size checks;
- access logs and tracing;
- compression;
- routing and canary rules;
- protection from malformed traffic.

A reverse proxy can provide a consistent enforcement point. A forward proxy can do the corresponding job for outbound traffic.

### 6.2 Decouple clients from topology

Instances are added, removed, moved, upgraded, or marked unhealthy without publishing every address to clients.

### 6.3 Create a trust boundary

The proxy can be the only component allowed to:

- receive Internet traffic;
- reach private backends;
- make Internet egress connections;
- possess a public TLS private key;
- attach verified identity metadata.

### 6.4 Adapt protocols and connection behavior

A proxy can:

- accept HTTP/2 and use HTTP/1.1 to a legacy backend;
- accept public JSON/HTTP and call an internal gRPC service;
- reuse backend connection pools;
- buffer a slow client’s upload or a fast backend’s response;
- enforce timeouts and maximum sizes.

Every adaptation costs CPU/memory and creates semantic edge cases. The proxy must understand both sides correctly.

### 6.5 Improve operations

Because traffic crosses one controlled point, operators can observe:

- request rate and bandwidth;
- client-side versus backend-side latency;
- route and backend selected;
- TLS/protocol versions;
- status/error categories;
- retry, timeout, rejection, and cache behavior.

The cost is concentration: an error in shared configuration can affect every service behind it.

## 7. Forward-proxy advantages, use cases, and limits

| Use case | Why a forward proxy helps | Important limit/trade-off |
|---|---|---|
| Corporate web access control | Central allowlists, denylists, malware scanning, and user policy | Privacy-sensitive logs; encrypted contents require authorized TLS inspection |
| Workload egress control | Only the proxy may reach the Internet; destinations and identities can be restricted | Must support required protocols and scale without blocking deployments |
| Stable outbound IP | External partners allowlist one/few proxy IPs rather than every workload | Proxy/region failure can break all partner access |
| Shared client-side cache | Repeated public content can be served without refetching | HTTPS tunneling and personalized content reduce cacheability |
| Audit/compliance | Central record of which workload/user contacted which destination | Logs are sensitive and can become a surveillance or breach risk |
| Privacy mediation | Destination sees proxy address rather than direct client address | Proxy itself is trusted; fingerprinting and traffic correlation may remain |
| Network translation | HTTP-to-another-protocol or controlled tunnel creation | Translation can lose semantics and complicate debugging |

### 7.1 When a forward proxy is a good fit

Use one when you control clients/workloads and need a shared egress policy or identity.

Examples:

- CI workers may download packages only through an inspected repository proxy.
- Payment services may call a partner only through an egress proxy with a stable allowlisted IP.
- School browsers may use an explicit web proxy for content policy.
- Serverless/private workloads may use an egress gateway instead of receiving arbitrary public connectivity.

### 7.2 When it is not enough

- It is not a VPN for arbitrary IP connectivity unless it provides a tunnel designed for that scope.
- It is not a firewall for all network paths if applications can bypass it.
- It is not anonymity if the proxy records identity or leaks it onward.
- It is not secure if anyone can use it as an **open proxy**.

## 8. Reverse-proxy advantages and technical capabilities

### 8.1 Host and path routing

```text
Host: video.example.com      -> media service
Host: api.example.com
  /users/*                   -> user service
  /payments/*                -> payment service
  /v1/legacy/*               -> monolith
```

L7 routing supports service decomposition and migrations without changing the public hostname.

### 8.2 TLS termination and certificate centralization

The proxy presents the public certificate, negotiates TLS, and can offload repeated cryptographic work from application servers. Backends can still use TLS/mTLS; “terminated at the edge” should not automatically mean “plaintext everywhere internally.”

### 8.3 Origin isolation

Backends can use private addresses and firewall/security-group rules that accept only the proxy tier. This makes edge policy harder to bypass.

### 8.4 Load balancing and health-aware routing

A reverse proxy can select among instances and stop sending new traffic to unhealthy ones. This is an overlap with load balancing, explored in Section 9.

### 8.5 Connection management

The proxy can maintain many client connections while pooling/reusing a controlled number of backend connections. This protects backends from connection churn, but the proxy must have adequate sockets, memory, file descriptors, CPU, and timeout policies.

### 8.6 Buffering, streaming, and slow-client isolation

Buffering lets a backend finish quickly while the proxy slowly sends data to a client. It can also validate an upload size before contacting the application.

But buffering is harmful for some flows:

- server-sent events;
- WebSockets;
- live streaming;
- large uploads/downloads;
- low-latency incremental responses.

The design must deliberately choose buffering or streaming per route.

### 8.7 Caching and compression

The proxy can cache safe responses, compress content, and answer conditional requests. Incorrect cache keys or directives can leak one user’s data to another.

### 8.8 Central policy

Possible controls include:

- authentication and coarse authorization;
- per-user/API-key/IP rate limits;
- request body/header limits;
- WAF rules;
- geographic or tenant policy;
- bot and abuse controls;
- CORS and security headers.

Do not put deep domain authorization only at the proxy. A payment service must still verify that the authenticated user may access the specific payment.

### 8.9 Release and failure controls

The proxy can implement:

- weighted canaries;
- blue/green routing;
- request mirroring/shadowing;
- circuit breaking;
- load shedding;
- maintenance responses;
- connection draining.

Mirrored requests must not create real writes. Either mirror only safe traffic or send it to a target that cannot perform side effects.

## 9. Reverse proxy versus load balancer

### 9.1 Different questions

- **Reverse proxy question:** “Should clients talk to an intermediary that represents my backend?”
- **Load-balancer question:** “Which eligible target should receive this traffic so work and failure are distributed?”

A reverse proxy can forward to one backend and still provide TLS, routing, caching, or origin protection. A load balancer is valuable because there are multiple targets, zones, or regions to select among.

### 9.2 The overlap

```text
                  reverse-proxy behavior
Client ---> [ inspect HTTP + choose backend ] ---> App A
                                                -> App B
                  load-balancing behavior
```

An L7 HTTP load balancer normally acts as a reverse proxy because it accepts the client-side HTTP/TLS connection and communicates with a backend.

An L4 load balancer may instead:

- proxy TCP/UDP connections;
- translate destination addresses with NAT;
- forward packets;
- use direct server return, where a backend sends the response without returning through the load balancer.

Those designs can balance traffic without being an HTTP reverse proxy.

### 9.3 Load-balancing mechanics in brief

| Method | Good for | Failure/skew concern |
|---|---|---|
| Round robin | Similar requests and similar-capacity servers | Long requests can create unequal active load |
| Weighted round robin | Mixed server capacities or gradual rollout | Weights become stale |
| Least connections | Connections with uneven duration | One connection may represent very different work |
| Least latency/time | Avoid slower targets | Feedback can oscillate; needs enough samples |
| Hash/consistent hash | Affinity for cache/session/key | Hot keys and remapping; state still needs resilience |
| Random/power of choices | Simple scalable selection | Requires useful load signal for stronger variants |

Health checks must test readiness, not merely whether a process accepts TCP. A backend can be alive but unable to reach its database.

AWS documents the basic load-balancer contract as distributing traffic across targets, monitoring their health, and routing only to healthy targets ([Elastic Load Balancing documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/what-is-load-balancing.html)). NGINX documents Layer 7 methods including round robin, least connections, and hashing ([NGINX HTTP load balancing](https://docs.nginx.com/nginx/admin-guide/load-balancer/http-load-balancer/)).

### 9.4 Interview-safe conclusion

> A load balancer is a traffic-distribution role. A reverse proxy is a backend-facing intermediary role. Many L7 products perform both; neither term completely contains the other across all layers and implementations.

## 10. Reverse proxy versus API gateway

### 10.1 Foundation

An **API gateway** is a reverse-proxy-style entry point specialized for publishing and managing APIs.

```text
Mobile/Web client
       |
       v
+------------- API Gateway --------------+
| route | authenticate | quota | schema   |
| version | transform | observe | protect |
+----------------------------------------+
       |
       +----> User API
       +----> Trip API
       +----> Payment API
```

AWS describes API Gateway as a service for creating, publishing, maintaining, monitoring, and securing REST, HTTP, and WebSocket APIs; it can handle traffic management, authorization, monitoring, and version management ([AWS API Gateway documentation](https://docs.aws.amazon.com/apigateway/latest/developerguide/welcome.html)).

### 10.2 Responsibility comparison

| Area | General reverse proxy | API gateway |
|---|---|---|
| Main abstraction | Network/application intermediary for backends | Managed API contract and policy entry point |
| Routing | Host/path/backend routing | API resource/method/version/tenant routing |
| Authentication | Possible | Usually central and first-class |
| Quotas/API keys | Possible | Common first-class capability |
| Request/response transformation | Possible | Often schema/protocol aware |
| Developer portal/lifecycle | Uncommon | May publish docs, keys, plans, versions |
| Aggregation | Possible custom behavior | Sometimes composes multiple backend calls |
| Non-HTTP protocols | Product dependent | Usually focused on supported API protocols |
| Load balancing | May be central | May delegate to an internal load balancer/service discovery |

### 10.3 API gateway dangers

- It can become a **god gateway** containing business logic owned by every team.
- Complex transformations hide coupling and make version migrations harder.
- Synchronous aggregation increases tail latency and partial-failure cases.
- A global auth/config error has a large blast radius.
- Per-request heavyweight middleware increases cost and latency.

Keep universal edge policy in the gateway; keep domain decisions in domain services or a deliberate backend-for-frontend layer.

### 10.4 “Gateway” is overloaded

Three different uses appear in interviews:

1. **Default/network gateway:** a router used to leave a local network.
2. **HTTP gateway/reverse proxy:** RFC terminology for an intermediary representing origins.
3. **API gateway:** an API-management and policy role.

State which meaning you intend.

## 11. Reverse proxy versus CDN

### 11.1 Foundation

A **content delivery network (CDN)** is a geographically distributed edge system whose primary job is to serve content near users and reduce origin work. In HTTP designs, CDN points of presence act like globally distributed caching reverse proxies.

```text
User in Delhi ---> nearby CDN edge --cache hit--> response
                         |
                         | cache miss
                         v
                  regional/origin proxy ---> origin
```

AWS documents that CloudFront routes a request to a low-latency edge location and, on a cache miss, retrieves content from a configured origin ([CloudFront documentation](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/Introduction.html)).

### 11.2 Comparison

| Question | Reverse proxy | CDN |
|---|---|---|
| Where deployed? | Often one region/data center or each region | Many global edge locations |
| Primary purpose | Controlled entry/routing to backends | Low-latency distribution and origin offload |
| Cache required? | No | Central capability, though dynamic pass-through is possible |
| Origin relationship | Usually near/inside service infrastructure | Fetches from one or more origins over a wider network |
| Other edge features | TLS, routing, WAF, auth, compression | Often TLS, DDoS/WAF, routing, edge compute, image/media optimization |

### 11.3 CDN correctness questions

Before caching, define:

- cache key: host, path, query, selected headers, encoding, tenant;
- freshness: `Cache-Control`, TTL, revalidation;
- invalidation strategy;
- whether cookies or authorization make a response private;
- stale-on-error behavior;
- range requests and media segments;
- cache stampede protection;
- origin shielding and request collapsing.

“Use a CDN for static files” is a useful starting point, not the full model. CDNs can pass through or selectively cache dynamic content, but personalized data needs careful rules.

## 12. Proxy versus VPN versus firewall

These tools can all sit “between a user and a server,” but solve different primary problems.

### 12.1 VPN: a protected virtual network path

A **virtual private network (VPN)** creates an authenticated, usually encrypted tunnel that carries network traffic across another network.

```text
Laptop == encrypted tunnel over Internet ==> Company VPN endpoint
                                                |
                                                +--> private Git server
                                                +--> internal database
                                                +--> internal APIs
```

Common forms:

- **Remote-access VPN:** one device joins/reaches an organization network.
- **Site-to-site VPN:** two networks communicate through a protected tunnel.
- **Full tunnel:** most device traffic goes through the VPN.
- **Split tunnel:** only selected destinations use it.

NIST describes IPsec as a network-layer security framework for protecting communications over IP networks ([NIST SP 800-77 Rev. 1](https://csrc.nist.gov/pubs/sp/800/77/r1/final)).

### 12.2 Proxy versus VPN

| Question | Proxy | VPN |
|---|---|---|
| Typical scope | Selected application/protocol/session | IP traffic for a device, subnet, or selected routes |
| Application awareness | Can be high, e.g. HTTP headers and paths | Usually carries packets without HTTP semantics |
| Client setup | App/OS proxy setting or network interception | VPN client/tunnel and routes |
| Main purpose | Relay plus policy/routing/caching/identity | Private connectivity and protected network path |
| Encryption | Not guaranteed; depends on client-proxy and proxy-server legs | Tunnel normally encrypts traffic between VPN endpoints |
| Destination sees | Proxy’s connection/address | Often VPN egress address after routing/NAT |
| Internal resource access | Only what proxy exposes | Can make many private network services reachable |

A consumer “VPN” product may combine tunneling, DNS, firewall, and proxy functions. Product packaging does not erase the conceptual distinction.

A VPN also does not create magical end-to-end confidentiality. Traffic is protected **to the VPN endpoint**; after that it still depends on HTTPS/TLS and destination security. The VPN operator or company becomes part of the trust model.

### 12.3 Firewall: a policy enforcement boundary

A **firewall** allows or blocks network traffic according to security policy. NIST defines it as a device or program controlling traffic between networks or hosts with different security postures ([NIST firewall glossary](https://csrc.nist.gov/glossary/term/firewall)).

Common levels:

- packet filter: source/destination address, port, protocol;
- stateful firewall: understands connection state;
- application-aware firewall: understands more protocol content;
- **WAF:** focuses on HTTP/web attacks and application-layer rules.

### 12.4 Proxy versus firewall

| Question | Proxy | Firewall |
|---|---|---|
| Primary action | Receive/relay or originate a second connection | Permit, reject, or drop traffic |
| Must be an endpoint? | Usually yes for proxied flow | No; can filter packets inline |
| Can hide topology? | Often | Not necessarily |
| Can cache/transform? | Application proxies can | Not a normal firewall responsibility |
| Can enforce policy? | Yes | Yes—this is the firewall’s central job |
| Can one product do both? | Yes, an application proxy can apply firewall-like rules | Yes, a next-generation firewall may proxy applications |

### 12.5 They often work together

**Employee egress:**

```text
Managed laptop
     |
     +== VPN ==> company network
                    |
                 Firewall
                    |
              Forward proxy
                    |
                 Internet
```

**Public service ingress:**

```text
Internet
   |
Network firewall / DDoS controls
   |
CDN + WAF
   |
Reverse proxy / API gateway / load balancer
   |
Private application services
```

Some of these are logical functions combined into fewer managed products. Draw separate boxes only when the distinction matters to requirements, ownership, scaling, or failure.

## 13. Other commonly confused mechanisms

### 13.1 Proxy versus DNS

- DNS gives the client an address.
- A proxy receives and relays traffic.
- DNS can distribute clients across proxy addresses, but has caching/TTL and limited per-request awareness.

### 13.2 Proxy versus HTTP redirect

```text
Redirect:
  Client -> old.example
  Client <- 302 Location: new.example
  Client ---------------------> new.example

Proxy:
  Client -> old.example proxy -> new backend
  Client never has to request the backend URL directly
```

A redirect changes what the client contacts. A reverse proxy keeps relaying behind the same public URL.

### 13.3 Proxy versus NAT

- **NAT** rewrites network addresses/ports in packets.
- An application proxy terminates one interaction and creates another.
- NAT does not normally understand HTTP paths, cache responses, or validate API keys.
- Both can cause the far side to observe an intermediary address.

### 13.4 Proxy versus service discovery

Service discovery answers “which instances exist?” A proxy/load balancer can consume that registry and answer “which healthy instance should receive this request?”

Client-side load balancing can use service discovery and connect directly, avoiding a central data-plane hop. It moves retry, balancing, TLS, and compatibility logic into clients/libraries.

### 13.5 Service-mesh sidecar

A service-mesh proxy beside a workload often performs both directions:

```text
Service A -> local sidecar -> network -> sidecar -> Service B
             outbound/                  inbound/
             forward role              reverse role
```

The labels depend on the connection being discussed. “Envoy is a reverse proxy” is incomplete; the same software can perform edge, forward, reverse, and sidecar roles.

## 14. Choosing the right component

Start with the requirement, not a fashionable box.

| Requirement | Primary role to consider | Why |
|---|---|---|
| Control employee/workload Internet access | Forward/egress proxy | Destination policy, stable identity, audit |
| Expose private backends behind one public name | Reverse proxy | Topology hiding and centralized ingress |
| Spread work over replicas | Load balancer | Health-aware target selection |
| Authenticate, quota, version, and publish APIs | API gateway | API lifecycle and policy |
| Serve cacheable content near global users | CDN | Edge cache and origin offload |
| Connect a device/site to a private network | VPN | Protected routed tunnel |
| Permit/deny traffic between trust zones | Firewall/WAF | Security policy enforcement |

### 14.1 A decision sequence

1. **Direction:** Are we controlling client egress or service ingress?
2. **Scope:** One HTTP API, all web traffic, arbitrary TCP, or routed IP networks?
3. **Visibility:** Must the component understand host/path/header/body?
4. **State:** Is caching, session affinity, or connection persistence required?
5. **Policy:** Auth, authorization, rate limits, WAF, compliance, or stable identity?
6. **Topology:** One backend, replicas, services, regions, or global edges?
7. **Failure:** What happens if the intermediary or its configuration plane fails?
8. **Trust:** Where does TLS end, and who may assert client identity?
9. **Operations:** Who owns configuration, rollout, observability, and incident response?
10. **Cost:** Is the extra hop and shared blast radius justified?

## 15. Combined architectures: order is not universal

### 15.1 Small application

```text
DNS -> NGINX reverse proxy -> one application process
```

Even with one backend, the proxy can provide a public certificate, host routing, static files, request limits, and safe process isolation. It is a reverse proxy but not meaningfully distributing load.

### 15.2 Scaled public API

```text
Users
  |
  v
DNS / anycast routing
  |
  v
CDN + DDoS/WAF edge
  |
  v
regional L4 load balancer
  |
  v
API-gateway / L7 reverse-proxy cluster
  |
  +------> User service replicas
  +------> Catalog service replicas
  +------> Order service replicas
```

This diagram separates roles for learning. A cloud CDN may include WAF and L7 routing; an API gateway may load balance; a managed load balancer may terminate TLS. Production might use fewer boxes.

### 15.3 Internal microservices

```text
API gateway -> service-level load balancer/proxy -> service instances
                            |
                         discovery
```

Alternatively, the gateway or client may use service discovery and client-side balancing directly. Adding another proxy hop is justified only if it owns a real policy, compatibility, isolation, or operational need.

### 15.4 Controlled partner call

```text
Payment service -> egress forward proxy -> partner API
                         |
                    stable public IP
                    allowlist + audit
```

The incoming user request might have already crossed a reverse proxy. A single end-to-end operation can contain both proxy directions.

### 15.5 Why “CDN -> gateway -> LB” is not a law

Possible valid arrangements include:

- CDN -> L7 load balancer that already performs gateway rules;
- CDN -> API gateway -> service discovery;
- network load balancer -> API-gateway cluster;
- API gateway -> internal L4/L7 load balancer;
- global load balancer/DNS -> regional CDN or reverse proxy;
- no CDN for a private, uncacheable, regional API.

State what each hop contributes. Delete a hop that adds no distinct value.

## 16. TLS patterns through proxies

| Pattern | Client TLS endpoint | Backend leg | Proxy sees HTTP? | Typical use |
|---|---|---|---|---|
| Forward CONNECT tunnel | Destination server | End-to-end tunnel | No plaintext normally | HTTPS through corporate/egress proxy without inspection |
| Forward TLS inspection | Forward proxy | New TLS to destination | Yes | Managed-device security/compliance |
| Reverse TLS termination | Reverse proxy | Plaintext or separate protocol | Yes | L7 routing, WAF, auth |
| Reverse TLS bridging | Reverse proxy | New TLS/mTLS | Yes | L7 controls plus encrypted/authenticated backend leg |
| Reverse TLS passthrough | Backend | Same encrypted stream routed at L4 | No HTTP plaintext | End-to-end TLS to backend, limited L7 features |

### 16.1 Termination is a trust decision

If public TLS ends at the reverse proxy:

- private keys must be protected and rotated;
- proxy memory can contain plaintext;
- backend encryption must be chosen explicitly;
- logs must not capture credentials/tokens/bodies indiscriminately;
- compliance boundaries may change.

### 16.2 Protocol information without decryption

An L4 proxy may use destination IP/port and sometimes TLS handshake metadata, but encrypted client hello and newer privacy mechanisms can reduce visible metadata. Do not design a future-proof routing policy that assumes every TLS field is always observable.

### 16.3 Mutual TLS

mTLS can authenticate:

- client to edge proxy;
- edge proxy to backend;
- service proxy to service proxy.

Identity does not automatically transfer across a terminated connection. The proxy must attach a verified identity, and the backend must trust it only from authenticated proxies.

## 17. Client identity and forwarding headers

### 17.1 The problem

The backend’s direct peer is the reverse proxy:

```text
Client 203.0.113.10 -> Proxy 10.0.0.5 -> Backend 10.0.1.20
Backend socket sees peer: 10.0.0.5
```

The backend may still need the original client address for abuse analysis, approximate location, auditing, or rate limits.

### 17.2 Standard and common headers

The standardized `Forwarded` header can carry information such as:

```http
Forwarded: for=203.0.113.10;proto=https;host=api.example.com
```

Common non-standard fields include:

- `X-Forwarded-For`;
- `X-Forwarded-Proto`;
- `X-Forwarded-Host`;
- `X-Real-IP`.

[RFC 7239](https://www.rfc-editor.org/rfc/rfc7239.html) defines `Forwarded` and explicitly warns that the field can be modified by clients or intermediaries and cannot be trusted without a trusted-proxy model.

### 17.3 Safe trust-chain rule

At the public edge:

1. remove or overwrite client-supplied forwarding/identity headers;
2. derive identity from the actual authenticated connection;
3. append information in a controlled format;
4. authenticate the edge-to-backend path;
5. configure the backend with the exact trusted proxy hops;
6. never accept a “trusted user” header from a path that can bypass the proxy.

Blindly trusting the first `X-Forwarded-For` value lets an attacker spoof IP-based authorization or evade rate limits.

### 17.4 Preserve public request context

Backends may need the original host and scheme to generate correct redirects, absolute links, and secure cookies. Misconfigured `Host` or `X-Forwarded-Proto` frequently causes:

- redirect loops between HTTP and HTTPS;
- redirects to private backend names;
- incorrect cookie security/domain attributes;
- tenant-routing mistakes.

## 18. Availability and failure behavior

### 18.1 The intermediary can become the outage

```text
Healthy clients -> FAILED SINGLE PROXY -X-> healthy backends
```

The backends being replicated does not help if every request depends on one failed proxy.

### 18.2 Make the proxy tier highly available

Typical design:

```text
                 +--> Proxy A --+
DNS/anycast/L4 LB               +--> backend pools
                 +--> Proxy B --+
                 +--> Proxy C --+
```

Design considerations:

- at least enough capacity to survive one failure or a defined zone failure;
- health checks from the component that actually routes traffic;
- configuration replicated and versioned;
- safe, staged configuration rollout;
- connection draining before shutdown;
- limits so one tenant/route cannot exhaust all workers;
- multiple failure domains, not replicas on one host/rack/zone;
- a plan for data-plane behavior if the configuration/control plane is unavailable.

### 18.3 Control plane versus data plane

- **Data plane:** handles live requests.
- **Control plane:** publishes routes, certificates, policy, discovery, and configuration.

A robust proxy can continue serving last-known-good configuration during a temporary control-plane failure. A bad configuration rollout can still be more dangerous than an instance crash, so validation, canaries, rollback, and audit history matter.

### 18.4 Timeouts and deadline budget

Suppose a client allows 2 seconds:

```text
client budget: 2.0 s
  proxy queue/connect/TLS: some portion
  backend work:           some portion
  response transfer:      some portion
```

The proxy cannot safely give every downstream attempt the full 2 seconds and then retry repeatedly. Propagate a deadline, reserve response time, and cap retries.

### 18.5 Retries can duplicate work

Failure window:

```text
Proxy -> backend: POST /charge
Backend commits charge
Backend response is lost
Proxy retries -> possible duplicate charge
```

Retry only when:

- the operation is safe/idempotent, or uses a stable idempotency key;
- the failure type makes retry useful;
- there is time in the deadline;
- there is a retry budget and backoff;
- retrying another unhealthy backend will not amplify overload.

### 18.6 Common gateway status clues

According to HTTP semantics:

- **502 Bad Gateway:** the gateway/proxy received an invalid response from an inbound server.
- **503 Service Unavailable:** the service is temporarily unable to handle the request, commonly overload or maintenance.
- **504 Gateway Timeout:** the gateway/proxy did not receive a timely response from an upstream/inbound server.

These are starting clues, not complete root causes. Inspect proxy and backend metrics, connection errors, deadlines, health state, and tracing. See [RFC 9110 status codes](https://www.rfc-editor.org/rfc/rfc9110.html#name-server-error-5xx).

## 19. Performance and capacity reasoning

### 19.1 What the proxy consumes

Capacity depends on:

- requests or connections per second;
- concurrent connections;
- request/response byte rate;
- TLS handshakes versus connection reuse;
- HTTP parsing and policy work;
- compression/decompression;
- buffering;
- cache hit ratio;
- logging and telemetry;
- long-lived WebSocket/SSE connections;
- backend latency and timeouts.

“This proxy supports X RPS” without workload, hardware, protocol, payload, and latency is not transferable.

### 19.2 In-flight concurrency

Little’s Law gives a useful approximation:

```text
in-flight requests ~= request rate x average time in system
```

**Illustrative arithmetic, not a production benchmark:**

```text
10,000 requests/s x 0.100 s average = about 1,000 in-flight requests
```

At the same request rate, a backend slowdown from 100 ms to 1 second creates roughly ten times the in-flight work. Memory, sockets, queues, and timeout pressure rise even though incoming RPS did not.

Tail latency matters more than the average for overload behavior.

### 19.3 Byte and buffer pressure

Traffic crosses both client-facing and backend-facing interfaces. Plan ingress and egress byte rates for both legs.

If the proxy buffers whole bodies:

```text
buffer memory/disk pressure
  ~= concurrent buffered messages x buffered bytes per message
```

Large uploads can exhaust memory or temporary disk. Set size limits, stream where appropriate, and consider direct signed upload to object storage.

### 19.4 Queues need bounds

An unlimited proxy queue converts overload into:

- growing latency;
- memory exhaustion;
- timeouts for work that never had a chance to finish;
- synchronized retry storms.

Use bounded concurrency/queues, load shedding, per-tenant limits, and clear overload signals.

### 19.5 High availability is also capacity planning

If normal traffic needs all proxy instances, one failure causes overload. Size for the declared failure case, such as losing one instance or one zone, plus safe rollout headroom. The exact margin is workload-specific and must be measured.

## 20. Security model and common threats

### 20.1 The proxy is a high-value trust point

It may hold:

- TLS private keys;
- user tokens and cookies;
- routing to private services;
- an identity trusted by every backend;
- logs containing client and request data;
- permission to contact arbitrary external destinations.

Treat it as security-critical infrastructure, not just a performance utility.

### 20.2 Threats and mitigations

| Threat | Why it happens | Mitigation direction |
|---|---|---|
| Open forward proxy abuse | Anyone can relay spam, attacks, scraping, or hidden traffic | Strong client authentication, network restrictions, destination policy, quotas |
| Origin bypass | Backend remains publicly reachable | Private networking; firewall/security group allows only approved proxy paths |
| Header spoofing | Backend trusts client-supplied identity/IP headers | Strip at edge; rebuild; authenticated proxy chain; exact trusted-hop config |
| SSRF/confused deputy | User-controlled URL makes proxy contact metadata/private services | Destination allowlist, DNS/IP validation, block private/reserved ranges, revalidate redirects |
| Request smuggling/desync | Frontend and backend parse ambiguous HTTP framing differently | Consistent strict parsing, normalize/reject ambiguity, patched software, protocol-safe translation |
| Cache poisoning/leak | Cache key omits tenant/auth/varying input | Explicit cache policy/key; do not cache private responses; test poisoning cases |
| TLS-key compromise | Public private keys concentrated at proxy | Managed key storage, least privilege, rotation, isolation, incident plan |
| DDoS/resource exhaustion | Public chokepoint accepts costly connections/bodies | Edge filtering, SYN/connection controls, body/header limits, rate limiting, bounded work |
| Config blast radius | One policy/route/certificate change affects many services | Schema validation, review, canary, last-known-good, rapid rollback |
| Sensitive logging | Central logs capture tokens, cookies, bodies, client IPs | Allowlist fields, redact, minimize, encrypt, access-control, retention policy |
| Proxy loop | Route points to itself or cyclic chain | Hop limits, loop detection, configuration validation, `Via`/trace diagnostics |

### 20.3 WAF is not all application security

A WAF can block known patterns and malformed requests, but it cannot reliably decide domain questions such as:

> “May this authenticated user refund this particular order?”

Backends still validate authorization, business invariants, input, idempotency, and data access.

### 20.4 Direct return and asymmetric paths

Some L4 load balancers use direct server return:

```text
Client -> load balancer -> backend
Client <------------------ backend
```

This can reduce return-path load, but complicates network configuration, observability, filtering symmetry, and connection handling. It is a load-balancing design, not the normal two-connection L7 reverse-proxy flow.

## 21. Protocol and application edge cases

### 21.1 WebSocket

The proxy must support the HTTP upgrade/extended CONNECT behavior, retain a long-lived bidirectional connection, use appropriate idle timeouts, and drain carefully. Ordinary per-request retries do not apply after a connection is established.

### 21.2 Server-sent events and streaming responses

Disable or reduce buffering where incremental delivery matters. Compression buffers and intermediary timeouts can invisibly destroy “real-time” behavior.

### 21.3 gRPC

The route must support the required HTTP/2 semantics, trailers, streaming, message limits, deadlines, and gRPC status handling. A proxy that downgrades incorrectly can break the protocol.

### 21.4 HTTP/3 and QUIC

An edge can terminate HTTP/3/QUIC and use a different protocol internally, but then connection migration, telemetry, retry, and loss behavior differ between legs. UDP reachability and fallback matter.

### 21.5 Large file upload

Questions to ask:

- Does the proxy buffer or stream?
- What is the maximum body size?
- Can a slow upload occupy a backend worker?
- Can clients upload directly to object storage with a short-lived signed URL?
- How is malware scanning performed?
- What happens after a partial upload or retry?

### 21.6 Long-running request

Increasing every timeout is usually not the best fix. For work lasting minutes:

```text
POST /jobs -> 202 Accepted + job_id
GET /jobs/{id} or callback/event -> result
```

This releases proxy connections and makes retries/progress explicit.

### 21.7 Sticky sessions

Routing a user repeatedly to one instance can preserve in-memory session state, but:

- the instance can fail;
- scaling/remapping moves users;
- uneven session activity creates skew;
- deployments require draining;
- it complicates multi-region failover.

Prefer stateless application instances or externalized replicated session state when feasible. Use affinity only for a named reason.

### 21.8 Cache with authorization

Never assume GET means public. A response can be user-specific. The proxy must obey cache controls and include every representation-changing dimension in the cache key—or not cache it.

### 21.9 Health-check trap

`GET /health -> 200` proves little if the handler is hard-coded while the service cannot reach required dependencies. Separate:

- **liveness:** should the process be restarted?
- **readiness:** should it receive new traffic?
- **deep dependency health:** useful diagnostically, but can cause cascading removal if used carelessly.

### 21.10 Service discovery staleness

The proxy needs a policy for:

- stale instance lists;
- DNS TTLs;
- endpoints removed during deployments;
- failed health-check propagation;
- connection pools pointing at retired instances.

Drain and deregister before termination when possible.

## 23. Trade-offs, alternatives, and when to use or avoid

### 23.1 Core trade-offs

| Benefit | Cost introduced |
|---|---|
| Stable endpoint and hidden topology | Extra hop and dependency |
| Central security/policy | Large blast radius and high-value target |
| TLS/certificate centralization | Key concentration and plaintext trust boundary |
| Load balancing/failover | Health-check errors, retry duplication, routing skew |
| Connection pooling/buffering | Memory, sockets, latency, head-of-line and streaming issues |
| Caching/compression | Staleness, poisoning/leaks, CPU |
| Protocol transformation | Semantic mismatch and harder debugging |
| Central observability | Sensitive logs and cardinality/cost |
| Easier backend rollout | Complex shared configuration/control plane |

### 23.2 Alternatives

#### Direct client-to-server

Use when there is one simple trusted endpoint and no meaningful central policy. It has fewer hops but couples clients to the endpoint and pushes edge concerns into the server.

#### DNS-based distribution

DNS can select regions or addresses without proxying every byte. It is useful for global steering but reacts through resolver caching/TTL and lacks per-request HTTP awareness.

#### Client-side load balancing

Clients discover instances and choose one. It removes a central data-plane hop and can be efficient internally, but every client stack must correctly implement discovery, TLS, retries, health, and policy.

#### Application code

The app can authenticate, route, compress, and log directly. This is reasonable at small scale. Duplication and inconsistent policy grow as services multiply.

#### Service mesh

Sidecars or node proxies provide service-to-service identity, traffic policy, and telemetry. They add operational/control-plane complexity and do not automatically replace an Internet edge gateway.

#### Managed edge/gateway

A cloud-managed service reduces undifferentiated operations and may supply global capacity. Trade-offs include cost model, limits, less low-level control, and vendor coupling.

### 23.3 Use a reverse proxy when

- multiple internal services/instances must share a public endpoint;
- origin addresses should remain private;
- TLS, routing, policy, observability, or compatibility is genuinely shared;
- controlled rollout/failover is required;
- slow clients or connection churn must be isolated from backends.

### 23.4 Avoid or simplify it when

- it adds no distinct responsibility;
- the workload is a tiny trusted internal tool and the operational cost exceeds the benefit;
- protocol passthrough is required and the proposed L7 proxy does not correctly support it;
- very large data can flow directly to object storage/CDN using limited signed access;
- client-side discovery already provides a simpler reliable internal path;
- several consecutive proxies duplicate TLS, parsing, retries, logging, and timeouts without clear ownership.

The decision is not “proxies are good” or “extra hops are bad.” It is whether the control point earns its complexity.

## 25. Answer spines: cues to reconstruct answers live

Do not memorize paragraphs. Use these cue sequences.

### 25.1 “What is a proxy?”

**`MIDDLEMAN -> TWO LEGS -> CONTROL -> COST`**

1. Middleman that receives and relays traffic.
2. Usually creates client-to-proxy and proxy-to-destination legs.
3. This enables routing, policy, caching, identity, or observation.
4. It adds latency, trust, capacity needs, and a failure point.

### 25.2 “Forward proxy versus reverse proxy?”

**`REPRESENTS -> CONFIGURED BY -> FLOW -> USE -> TRUST`**

1. Forward represents clients; reverse represents servers.
2. Client/organization configures forward; service owner deploys reverse.
3. Draw both paths.
4. Give one use each: egress policy versus ingress routing/TLS.
5. Say identity can be forwarded and neither automatically means secure/anonymous.

### 25.3 “Why use a reverse proxy?”

**`STABLE -> ROUTE -> SHIELD -> SHARED -> RELEASE -> FAILURE`**

1. Stable public endpoint decouples clients from backends.
2. Route to services/versions/healthy instances.
3. Keep origins private.
4. Centralize TLS, coarse auth, limits, WAF, logs.
5. Support canary/draining/connection pooling.
6. Then mention proxy HA and blast radius.

### 25.4 “Reverse proxy versus load balancer/API gateway/CDN?”

**`ROLE -> LAYER -> SCOPE -> PRIMARY JOB -> OVERLAP`**

1. Reverse proxy: represents backends.
2. LB: distributes among targets; L4 or L7.
3. API gateway: API policy/lifecycle; usually L7 reverse-proxy behavior.
4. CDN: global edge cache/distribution; usually reverse-proxy behavior for HTTP.
5. One product can combine them—name the required responsibility.

### 25.5 “Design a reverse proxy/load balancer”

**`CONTRACT -> PATH -> SELECT -> SAFETY -> SCALE -> OPERATE`**

1. **Contract:** protocol, RPS/bytes/connections, latency, L4/L7, routes, regions.
2. **Path:** DNS, TLS endpoint, request flow, backend connection, response path.
3. **Select:** discovery, eligibility/health, algorithm, affinity, canary.
4. **Safety:** timeouts, retry rules, idempotency, limits, auth, header trust.
5. **Scale:** stateless proxy fleet, partitioning if needed, N+1/zone failure, bounded queues.
6. **Operate:** metrics/traces/logs, config validation/canary/rollback, draining.

### 25.6 “Where should TLS terminate?”

**`VISIBILITY -> TRUST -> BACKEND LEG -> KEYS -> PROTOCOL`**

1. Need L7 visibility or pure passthrough?
2. Which component is trusted with plaintext?
3. Plaintext, TLS, or mTLS to backend?
4. Key storage/rotation and compliance.
5. HTTP/2, gRPC, WebSocket, HTTP/3 compatibility.

### 25.7 “What if the reverse proxy fails?”

**`REPLICAS -> ENTRY -> CAPACITY -> CONFIG -> DRAIN -> DEGRADE`**

1. Multiple instances across failure domains.
2. Healthy entry steering via managed LB/DNS/anycast.
3. Capacity after a failure, not only at steady state.
4. Last-known-good configuration; staged rollback.
5. Drain connections during rollout.
6. Bounded shedding and useful errors when overloaded.

## 26. Synthesized interviewer follow-up drills

The following are **my plausible synthesis**, not claimed as sourced interview questions.

### Drill A: “Can a reverse proxy exist with one backend?”

Yes. Reverse proxy describes whom it represents, not backend count. It can still terminate TLS, route hostnames, enforce limits, serve static files, or isolate the app. It is not doing meaningful load distribution until alternatives exist.

### Drill B: “Can a load balancer exist without being a reverse proxy?”

Yes. An L4 design can forward/translate packets and use direct server return without interpreting HTTP or creating the normal two-leg L7 flow. DNS can also distribute clients without proxying their traffic.

### Drill C: “Does an HTTPS forward proxy see the request body?”

Not with an ordinary CONNECT tunnel: it relays encrypted TLS bytes. It can see content only if TLS terminates there, such as authorized enterprise inspection with a trusted corporate CA, or if the application explicitly sends plaintext to it.

### Drill D: “Where should rate limiting happen?”

At the earliest trusted layer that has the needed identity and can reject cheaply, often an edge/gateway. Distributed enforcement needs shared or partitioned state and failure policy. Domain-specific quotas may still belong in the service. Do not trust spoofable client headers.

### Drill E: “The proxy returns 504. Is the backend down?”

Not necessarily. The proxy failed to receive a timely response. Causes include backend saturation, dependency slowness, connection-pool exhaustion, network loss, a too-short proxy timeout, or queueing. Trace both legs and compare deadline/latency metrics.

### Drill F: “Why not retry every failed request on another server?”

Writes can duplicate; retries consume remaining deadline and amplify an overloaded fleet. Retry selected transient failures only, with idempotency, a retry budget, backoff, and another healthy target.

### Drill G: “How does the backend know the real client IP?”

Through a trusted metadata chain such as `Forwarded`/`X-Forwarded-For` or a lower-layer proxy protocol. The public edge must strip spoofed values and the backend must trust only known proxy hops.

### Drill H: “Should the API gateway come before or after the load balancer?”

Ask which load balancer. A regional L4 LB may distribute connections across gateway instances, so it is before them. The gateway may then select/load-balance service instances. A managed L7 gateway might already balance, eliminating a separate box.

### Drill I: “Why not put business logic in the gateway?”

Universal policy belongs there; domain behavior creates coupling, shared releases, and broad failures. Put orchestration in an explicitly owned backend-for-frontend/domain layer when needed.

### Drill J: “How do you deploy a proxy configuration safely?”

Schema/static validation, dependency checks, tests for routes/headers, versioned config, small canary, traffic/error/latency monitoring, automatic or fast rollback, and last-known-good data-plane behavior.

### Drill K: “How do WebSockets change the design?”

Long-lived connections make connection count and draining more important than request RPS, require protocol support and idle-timeout tuning, reduce ordinary per-request balancing opportunities, and need reconnect jitter to avoid a storm after failure.

### Drill L: “The CDN and gateway both retry. What can go wrong?”

Retries multiply across layers. One client request can fan into several origin attempts, exceed deadlines, and duplicate writes. Assign retry ownership, cap total attempts, propagate deadlines, and normally avoid caching/retrying unsafe methods.

### Drill M: “Why is the origin still being attacked despite a CDN?”

Attackers may know or discover its address and bypass the CDN. Restrict origin ingress to authenticated CDN/proxy paths, rotate exposed addresses where appropriate, and ensure no alternate DNS/service path bypasses policy.

### Drill N: “A proxy gives us anonymity, correct?”

No. The destination may see the proxy address, but the proxy may log the client, application headers can identify the user, browser fingerprinting and traffic correlation remain, and a misconfiguration can leak addresses.

### Drill O: “When is direct upload better?”

For large objects, let the API authorize and issue a narrow, expiring signed upload URL, then upload directly to object storage. This removes proxy bandwidth/buffering pressure while preserving scoped access and completion validation.

## 27. Common interview mistakes and corrections

1. **“Proxy means forward proxy.”** Proxy is the family; say forward proxy when contrasting it with reverse.
2. **Defining direction from arrow movement.** Define which side is represented and who configures it.
3. **“Every reverse proxy is a load balancer.”** It may forward to one backend or perform only TLS/routing.
4. **“Every load balancer is a reverse proxy.”** L4 packet/NAT/DSR and DNS designs disprove this.
5. **Treating product names as concepts.** NGINX, Envoy, HAProxy, and cloud services can perform multiple roles.
6. **Rigidly drawing CDN -> gateway -> LB.** Explain each responsibility and remove duplicates.
7. **“A CDN only serves static files.”** Caching is central, but dynamic routing/pass-through/edge policy are common.
8. **“API gateway is just another name for load balancer.”** It focuses on API contract, policy, and lifecycle.
9. **Confusing API gateway with default network gateway.** State the meaning.
10. **“VPN and proxy are the same because both change visible IP.”** Their normal traffic scope and primary purpose differ.
11. **“Firewall is a proxy.”** Some products proxy, but filtering policy and relaying are different roles.
12. **“HTTPS means the reverse proxy cannot inspect HTTP.”** It can when TLS terminates there.
13. **“HTTPS means a forward proxy reads everything.”** CONNECT tunneling normally preserves encrypted contents.
14. **“TLS terminates at the edge, so traffic is end-to-end encrypted.”** There are separate legs; name backend protection.
15. **Trusting `X-Forwarded-For` from anyone.** Strip, rebuild, and trust exact proxy hops.
16. **Claiming the origin is hidden while it remains public.** Restrict its network path.
17. **Retrying all 5xx/timeouts.** Consider method semantics, commit ambiguity, deadlines, budgets, and overload.
18. **Ignoring proxy availability.** A single front door is a single failure point unless made redundant.
19. **Sizing only by RPS.** Bytes, connections, TLS, latency, buffering, compression, and policy dominate different workloads.
20. **Enabling unlimited queues.** They turn overload into latency and memory failure.
21. **Buffering streams.** WebSocket, SSE, gRPC streaming, and media need deliberate behavior.
22. **Making sessions sticky by default.** It hides server state and complicates failure/scale.
23. **Caching authenticated responses casually.** Incorrect keys can leak private data.
24. **Using only a hard-coded liveness check for routing.** Readiness must show ability to serve safely.
25. **Putting every team’s code in the gateway.** Shared edge logic can become a deployment and ownership bottleneck.
26. **Logging all headers/bodies “for observability.”** Tokens, cookies, and personal data can leak.
27. **Assuming 502/503/504 identify one root cause.** They indicate a failure category at a boundary; investigate.
28. **Citing vendor throughput as universal capacity.** Benchmark the actual protocol, policy, payload, and hardware.

## 28. Compact review sheet

### 28.1 Core picture

```text
FORWARD = client-side representative / egress

clients -> forward proxy -> destinations

REVERSE = server-side representative / ingress

clients -> reverse proxy -> backends
```

### 28.2 Roles

```text
Reverse proxy : represent/hide/control backends
Load balancer : choose among eligible targets
API gateway   : publish/manage/protect APIs
CDN           : global edge delivery/cache
VPN           : protected routed tunnel
Firewall/WAF  : allow/block by security policy
```

They can be combined in one product.

### 28.3 Request path checklist

```text
DNS
 -> client connection
 -> TLS endpoint
 -> proxy policy/routing
 -> backend selection
 -> backend connection
 -> response buffering/streaming/cache
 -> logs/metrics/traces
```

### 28.4 Design checklist

```text
direction and represented side
L4 versus L7
protocols and long-lived connections
TLS termination/passthrough and backend encryption
routing, discovery, health, affinity
identity/header trust
timeouts, retries, idempotency
limits, buffering, cache
proxy HA and failure capacity
config rollout/rollback and observability
```

### 28.5 One-line distinctions

| Confusion | Correct cue |
|---|---|
| Forward vs reverse | Represents clients vs represents servers |
| Reverse proxy vs LB | Intermediary role vs target-distribution role |
| Reverse proxy vs gateway | General backend intermediary vs API policy/lifecycle specialization |
| Reverse proxy vs CDN | Regional/general entry vs global delivery/cache specialization |

| Proxy vs firewall | Relay/second leg vs allow-deny policy |
| Proxy vs redirect | Proxy keeps relaying vs client contacts new URL |
| Proxy vs DNS | Traffic intermediary vs name-to-address selection |

## 29. Say-out-loud 30-second narrative

> “A proxy is an intermediary that usually creates one connection from the caller and another to the destination. A forward proxy represents clients and controls egress; a reverse proxy represents backends and controls ingress. Reverse proxies can terminate TLS, hide origins, route, cache, enforce policy, and often load-balance. Load balancer, API gateway, and CDN describe specialized responsibilities that can overlap with reverse proxying, while a VPN provides a network tunnel and a firewall primarily permits or blocks traffic. In a design I also cover proxy redundancy, trusted forwarding headers, backend TLS, bounded timeouts and retries, and streaming behavior.”
