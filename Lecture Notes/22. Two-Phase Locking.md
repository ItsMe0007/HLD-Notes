# Two-Phase Locking (2PL)

> Beginner-first HLD notes expanded from: **growing and shrinking phases; basic, conservative/static, strict, and rigorous/strong-strict 2PL; deadlocks; timeouts; wait-for graphs; victim selection; wait-die and wound-wait; cascading aborts; and the concurrency trade-offs of stronger variants**.
>
> The central idea is: **2PL lets transactions overlap, but controls when they may acquire and release locks so that the committed result is equivalent to some safe one-at-a-time order.** It guarantees conflict serializability under its model; it does **not** automatically guarantee deadlock freedom, freedom from cascading aborts, or good performance.

## 0. Scope, assumptions, corrections, and evidence labels

### 0.1 How the rough bullets are interpreted

- **“Growing Phase, Shrinking Phase”** are the two phases of the basic protocol. They are not separate 2PL types.
- **“Conservative” and “Static” 2PL** refer to the same main idea: obtain every required lock before the transaction starts doing its work.
- **“Strong Strict (Rigrous)”** is corrected to **strong strict 2PL**, also commonly called **rigorous 2PL**. “Rigorous” is the standard spelling.
- **Strict 2PL and rigorous 2PL are not identical** under the definitions used in this chapter:
  - strict 2PL holds all **exclusive/write locks** until commit or abort;
  - rigorous 2PL holds **all shared/read and exclusive/write locks** until commit or abort.
- Some books, courses, and products use **strict 2PL** loosely for the all-locks-until-end rule. In an interview, define the term before relying on the label.
- **“Timeout → false abort?”** is interpreted as: can a timeout abort a transaction that was merely slow or blocked but not deadlocked? Yes. That is an unnecessary or false-victim abort, not proof that the database detected a cycle.
- **“WFG → cycle, self cycle?”** is interpreted as both how a wait-for graph detects deadlock and whether a transaction can normally have an edge to itself.
- **“Wait Die – T1 wait”** is only correct if `T1` is the **older requester** and the lock holder is younger.
- **“Wound Wait – T2 aborts”** is only correct if older `T1` requests a lock held by younger `T2`; the older requester wounds, or aborts, the younger holder.
- **Cascading aborts do not require rigorous 2PL to solve.** Strict 2PL is sufficient in the usual S/X-lock model. Rigorous 2PL is stronger and can reduce concurrency further because it also retains shared locks.

### 0.2 Corrections worth saying explicitly in an interview

1. **2PL does not mean “take two locks.”** It means lock acquisition and lock release occur in two non-overlapping phases.
2. **2PL and 2PC are unrelated protocols with similar names.** Two-phase locking controls concurrent interleavings; two-phase commit coordinates commit/abort across resources.
3. **Basic 2PL guarantees conflict serializability, not deadlock freedom.** Its hold-and-wait behavior can create deadlocks.
4. **A wait is not a deadlock.** A deadlock is a circular dependency that cannot resolve without intervention.
5. **A timeout is not cycle detection.** It is a policy that stops waiting after a threshold and may abort a transaction that would soon have progressed.
6. **A wait-for graph cycle is the key deadlock condition** in the standard single-instance lock model. A long chain without a cycle is blocking, not deadlock.
7. **Strictness is about hiding uncommitted writes.** Strict 2PL holds X locks to transaction end, preventing dirty reads and dirty writes through those locks and therefore preventing cascading aborts.
8. **Rigorous 2PL does not eliminate deadlocks.** Holding every lock longer may increase blocking and can still create cycles.
9. **Conservative 2PL avoids deadlock by eliminating hold-and-wait**, but it needs the full read/write lock set in advance and may reserve items long before they are used.
10. **Wait-die and wound-wait use timestamps for deadlock prevention, not for timestamp-order concurrency control.** Locks still control access.
11. **A deadlock victim must retry the whole transaction**, not blindly resume after the failed statement, unless a particular database explicitly provides different semantics.
12. **Real databases may combine MVCC, locks, latches, predicate tracking, and validation.** Do not assume a product implements every isolation level using textbook 2PL.

# Part I — Plain-language foundation

## 1. The problem: safe overlap

### 1.1 A transaction is a group of database work

A **transaction** is a unit of work that either commits as a whole or aborts. For example, transferring ₹100 from account `A` to account `B` may contain:

~~~text
1. subtract 100 from A
2. add 100 to B
3. commit
~~~

If only one transaction ever ran at a time, reasoning would be easy—but throughput would be poor. Databases therefore overlap transactions:

~~~text
time ------------------------------------------------------------>

T1: read A ---- write A ---------------- read B ---- write B -- commit
T2:       read C ---- write C -- commit
T3:                   read D -------- write D ----------- commit
~~~

Overlapping is safe when unrelated work proceeds independently and conflicting work is ordered correctly.

### 1.2 The danger: both transactions make a reasonable local decision

One concert seat is available:

~~~text
Database: seat A7 = AVAILABLE

T1 reads A7 = AVAILABLE
T2 reads A7 = AVAILABLE
T1 writes A7 = HELD_BY_ALICE
T2 writes A7 = HELD_BY_BOB
~~~

Each transaction made sense in isolation. Their interleaving did not. Possible failures include a lost update, two success responses for one seat, or a business invariant violation.

### 1.3 A lock is temporary permission

A **lock** says which transaction currently has permission to use a data item in a particular way.

- A **shared (`S`) lock** permits reading. Multiple transactions may normally hold compatible S locks on the same item.
- An **exclusive (`X`) lock** permits writing. It conflicts with another transaction’s S or X lock on the same item.

Simplified compatibility matrix for the **same item and different transactions**:

| Existing lock | Request `S` | Request `X` |
|---|---:|---:|
| none | grant | grant |
| `S` | grant | wait/fail |
| `X` | wait/fail | wait/fail |

The lock manager does not usually block a transaction on its own compatible/reentrant lock. Real databases also have update, intention, key-share, range, gap, predicate, metadata, and advisory lock modes.

### 1.4 Locks alone are not enough

Suppose every read and write takes the appropriate lock, but a transaction may release and acquire locks in any order:

~~~text
T1: lock X(A), write A, unlock A
T2:                         lock X(A), write A, unlock A
T2: lock X(B), write B, unlock B
T1:                         lock X(B), write B, unlock B
~~~

The relative order on `A` is `T1 → T2`, while the relative order on `B` is `T2 → T1`. There may be no single one-at-a-time transaction order that explains the result.

2PL adds a discipline:

> Once a transaction starts releasing locks, it may never acquire another lock.

That single rule prevents a transaction from weaving in and out of other transactions in ways that create a cyclic conflict order.

### 1.5 What 2PL is trying to achieve

The goal is not to execute transactions literally one at a time. The goal is to make their conflicting effects equivalent to **some** serial order:

~~~text
Concurrent execution observed by the database
                |
                | equivalent conflicting order
                v
        T2, then T1, then T3
~~~

This property is called **conflict serializability**. It gives a powerful reasoning model while still allowing non-conflicting work to overlap.

## 2. Prerequisites explained inline

These ideas are necessary to understand 2PL, so they belong here rather than in “study next.”

### 2.1 Schedule

A **schedule** is the order in which operations from concurrent transactions actually occur.

Notation used in examples:

| Notation | Meaning |
|---|---|
| `S1(A)` | transaction `T1` obtains a shared lock on `A` |
| `X1(A)` | `T1` obtains an exclusive lock on `A` |
| `R1(A)` | `T1` reads `A` |
| `W1(A)` | `T1` writes `A` |
| `U1(A)` | `T1` unlocks `A` |
| `C1` | `T1` commits |
| `A1` | `T1` aborts |

### 2.2 Serial and serializable

- A **serial schedule** completes one transaction before beginning the next.
- A **serializable schedule** may interleave transactions but has the same relevant effect as some serial schedule.

~~~text
Serial:        T1 T1 T1 T1 | T2 T2 T2

Interleaved:   T1 T1 | T2 | T1 | T2 T2
               ^ may still be serializable
~~~

Serializable does not mean wall-clock order, commit order, or “no waiting.” It means an equivalent legal one-at-a-time order exists according to the serializability definition being used.

### 2.3 Conflicting operations

Operations from different transactions **conflict** when they touch the same logical item, at least one writes, and swapping their order could change the outcome:

| Pair on same item | Conflict? |
|---|---:|
| read–read | no |
| read–write | yes |
| write–read | yes |
| write–write | yes |

### 2.4 Precedence/serialization graph

For each transaction, create a node. Add `Ti → Tj` if a conflicting operation of `Ti` occurs before the corresponding conflicting operation of `Tj`.

~~~text
T1 writes A before T2 reads/writes A  =>  T1 ----> T2
~~~

If the precedence graph is acyclic, the schedule is **conflict serializable**. A topological order of that graph gives an equivalent serial order.

Do not confuse this graph with a **wait-for graph**:

| Graph | Edge means | Cycle means |
|---|---|---|
| precedence/serialization graph | one transaction’s conflicting operation came before another’s | schedule is not conflict serializable |
| wait-for graph | one active transaction is blocked waiting for another | current deadlock |

The nodes may both be transactions, but the edges answer different questions.

### 2.5 Dirty read, cascading abort, and strict schedule

A **dirty read** happens when `T2` reads a value written by uncommitted `T1`.

~~~text
T1 writes A = 50       (not committed)
T2 reads A = 50
T1 aborts
~~~

`T2` used a value that officially never existed. If `T2` must also abort, that is a **cascading abort**.

A schedule is commonly called **strict** when no other transaction may read **or overwrite** an item written by `T1` until `T1` commits or aborts. Holding `T1`’s X lock until its end enforces this in the standard lock model.

### 2.6 Commit and abort

- **Commit:** make the transaction’s outcome final under the database’s durability/visibility contract.
- **Abort/rollback:** discard its uncommitted logical effects and release its transaction locks.

Rollback itself can be expensive: the database may have to undo changes, restore indexes, emit recovery records, and hold locks until undo is safe.

### 2.7 Lock manager

Conceptually, a lock manager maintains:

~~~text
Lock table
  resource A:
    granted: X to T1
    waiting: S by T2, X by T3

Transaction table
  T1: running, holds A
  T2: waiting for T1
  T3: waiting according to queue/compatibility policy
~~~

Its responsibilities include:

- grant compatible requests;
- queue or reject incompatible requests;
- remember owners and waiters;
- support conversions such as S → X;
- release locks on commit/abort;
- participate in deadlock detection or prevention;
- preserve fairness according to product policy.

### 2.8 Locks versus latches

- A **transaction lock** protects logical database contents and may be held for milliseconds, seconds, or longer. It participates in transaction isolation and deadlocks.
- A **latch/mutex** protects an in-memory database structure for a very short critical section, such as a buffer or B+ tree page while it is being modified internally.

Both can block, but 2PL is about transaction locks, not the database engine’s short-lived implementation latches.

### 2.9 2PL versus 2PC

| Two-phase locking (`2PL`) | Two-phase commit (`2PC`) |
|---|---|
| concurrency-control protocol | distributed atomic-commit protocol |
| growing then shrinking | prepare/vote then decide |
| orders conflicting reads/writes | makes participants agree on commit/abort |
| can deadlock | can block when a prepared participant cannot learn the decision |

A distributed transaction may use both: 2PL for isolation at participants and 2PC for the shared commit decision.

# Part II — The 2PL protocol

## 3. Growing and shrinking phases

### 3.1 Growing phase

During the **growing phase**, a transaction may:

- acquire new S or X locks;
- upgrade a lock, such as `S(A) → X(A)`, if the request can be granted;
- read and write protected items;
- wait for conflicting locks.

It may **not release** a lock.

~~~text
Growing:    acquire A ---- acquire B ---- upgrade C ---- acquire D
                 lock set only stays the same size or grows
~~~

### 3.2 Lock point

The instant when a transaction obtains its final lock is its **lock point**.

~~~text
T1: S(A) ---- X(B) ---- S(C) ---- ...
                           ^
                           lock point
~~~

The lock point matters because basic 2PL schedules can be serialized in lock-point order. Intuitively, every conflict edge points from the transaction with the earlier relevant lock ownership to a transaction whose lock point must come later.

### 3.3 Shrinking phase

The first unlock moves the transaction into the **shrinking phase**. It may:

- release locks;
- downgrade a lock, such as `X(A) → S(A)`, under the theoretical model;
- continue operations that need only locks it still holds.

It may **not acquire or upgrade any lock** after shrinking begins.

~~~text
Shrinking:  release A ---- release B ---- downgrade C ---- release C
                 no new lock or upgrade is legal
~~~

### 3.4 The one rule to remember

~~~text
           GROWING                         SHRINKING
  acquire / upgrade locks            release / downgrade locks

--------------------------- LOCK POINT ---------------------------> time

  no unlocks before boundary          no new locks after boundary
~~~

Operationally, systems often hold locks until transaction end, so the shrinking phase is compressed into commit/abort cleanup. The conceptual rule still explains the guarantee.

### 3.5 Upgrade and downgrade nuance

An **upgrade** asks for more privilege and behaves like acquiring a stronger lock, so it belongs in growing.

~~~text
S(A) -> X(A)   upgrade: growing only
~~~

A **downgrade** gives up privilege and behaves like a partial release, so it begins or belongs in shrinking.

~~~text
X(A) -> S(A)   downgrade: shrinking only
~~~

Two transactions that both hold S on `A` and both request X can deadlock:

~~~text
T1 holds S(A), wants X(A) ---- waits for T2's S(A)
T2 holds S(A), wants X(A) ---- waits for T1's S(A)
~~~

This is why acquiring the most restrictive known mode first can reduce conversion deadlocks.

## 4. A valid basic-2PL schedule

Suppose:

- `T1` transfers money by updating accounts `A` and `B`;
- `T2` updates unrelated account `C`, then reads `A`.

~~~text
time | T1                                  | T2
-----+-------------------------------------+---------------------------
  1  | X1(A) granted                      |
  2  | R1(A), W1(A)                       |
  3  |                                     | X2(C) granted
  4  | X1(B) granted  <- T1 lock point    | W2(C)
  5  | R1(B), W1(B)                       | S2(A) requested: WAIT
  6  | U1(A)  <- T1 starts shrinking      |
  7  |                                     | S2(A) granted, R2(A)
  8  | U1(B), C1                          | U2(A), U2(C), C2
~~~

This obeys **basic** 2PL:

- `T1` obtained both locks before its first unlock.
- After `U1(A)`, `T1` acquired no new lock.
- `T2` obtained its locks before releasing any.

But notice the recovery problem: `T2` read `A` after `T1` released the X lock but **before `T1` committed**. If `T1` aborted instead of committing, `T2` would have read dirty data. Basic 2PL gives conflict serializability but does not by itself prevent this cascade.

### 4.1 Why 2PL implies conflict serializability—intuition

Assume conflicting `Ti` and `Tj` both access `A`, and `Ti`’s access must occur first. `Ti` holds the conflicting lock before `Tj` can obtain its lock. Therefore `Tj` cannot pass its relevant lock acquisition before `Ti` has released enough privilege; under 2PL, that release occurs after `Ti`’s lock point. `Tj`’s conflicting lock acquisition is no earlier than that, so `Ti`’s lock point precedes `Tj`’s.

Every conflict edge therefore follows increasing lock-point order:

~~~text
Ti -> Tj  implies  lockPoint(Ti) < lockPoint(Tj)
~~~

A cycle would require lock points to be both earlier and later than themselves, which is impossible. Thus the precedence graph is acyclic.

This is an intuition, not a full formal proof; the original IBM paper established the underlying consistency/locking result ([IBM Research: Eswaran, Gray, Lorie, and Traiger, 1976](https://research.ibm.com/publications/the-notions-of-consistency-and-predicate-locks-in-a-database-system)).

### 4.2 What the theorem does not say

The guarantee does **not** mean:

- every serializable schedule can be produced by 2PL;
- deadlocks cannot occur;
- dirty reads cannot occur under basic 2PL;
- transactions commit in lock-point order under every 2PL variant;
- business logic is automatically correct;
- phantom-producing predicates are safe if only existing rows are locked;
- external API calls become transactional.

2PL is sufficient for conflict serializability under its locking assumptions, but it is conservative: it rejects some schedules that happen to be serializable.

# Part III — Types and variants

## 5. The variants at a glance

The variants add rules for different problems. Think of them as dimensions, not magic “levels.”

| Variant | Extra rule | Main benefit | Main cost/gap |
|---|---|---|---|
| basic 2PL | after first unlock, acquire no new locks | conflict serializability | deadlocks and cascading aborts possible |
| conservative/static 2PL | obtain the complete lock set before doing work | deadlock-free under the model | must know set in advance; locks reserved early |
| strict 2PL | hold every X lock until commit/abort | strict, cascadeless recovery | deadlocks remain; conflicting readers/writers wait longer |
| rigorous/strong-strict 2PL | hold every S and X lock until commit/abort | strict schedule; simple release/serialization reasoning | more blocking than strict; deadlocks remain |

### 5.1 Property matrix

| Property | Basic | Conservative alone | Strict | Rigorous |
|---|---:|---:|---:|---:|
| follows 2PL | yes | yes | yes | yes |
| conflict serializable | yes | yes | yes | yes |
| deadlock-free | no | **yes** | no | no |
| prevents dirty reads of uncommitted writes through S/X locks | no | not necessarily | yes | yes |
| prevents cascading aborts | no | not necessarily | yes | yes |
| all X locks held to end | no | not required by conservative rule alone | yes | yes |
| all S locks held to end | no | not required | no | yes |
| full lock set needed before work | no | **yes** | no | no |

Variants can be combined. For example, a system could preclaim all locks **and** hold them to commit, yielding conservative + rigorous behavior.

## 6. Basic 2PL

### 6.1 Rule

1. Grow by acquiring/upgrading locks without releasing.
2. After the first release/downgrade, shrink without acquiring/upgrading.

### 6.2 What it gives

- conflict-serializable schedules;
- concurrency for transactions whose lock sets do not conflict;
- a clear lock point and serialization argument.

### 6.3 What it does not give

- deadlock freedom;
- strictness or cascadeless recovery;
- freedom from starvation;
- protection for logical predicates unless appropriate range/predicate locks exist;
- good performance under hot-item contention.

Basic 2PL is the foundation used to understand the stronger variants, but many production systems prefer stricter lock retention because recovery becomes simpler.

---

## 7. Conservative or static 2PL

### 7.1 Rule

Before executing its data operations, the transaction requests **all locks it will need**:

~~~text
declare/request { X(A), X(B), S(C) }
              |
       all granted together?
          /             \
        yes              no
        |                |
  begin operations    hold none; wait/retry
~~~

The important condition is all-or-nothing preclaiming. If the transaction kept `A` while waiting for `B`, it would reintroduce hold-and-wait and could deadlock.

### 7.2 Why it is deadlock-free

One necessary condition for lock deadlock is **hold-and-wait**: a transaction holds one resource while waiting for another.

Conservative 2PL does not let a transaction begin holding a partial set while waiting for the rest. Therefore a circular hold-and-wait chain cannot form under the model.

### 7.3 Example

`T1` needs `A` then `B`; `T2` needs `B` then `A`.

With basic 2PL:

~~~text
T1 holds A, waits B
T2 holds B, waits A  => deadlock
~~~

With conservative 2PL:

~~~text
T1 requests {A, B} atomically -> granted
T2 requests {A, B}            -> waits while holding neither
T1 finishes and releases
T2 receives {A, B}
~~~

### 7.4 Trade-offs

- The transaction must know its complete read/write set before executing. Data-dependent traversal makes this hard:

  ~~~text
  read customer -> discover region -> read region rules
                -> discover inventory partitions -> lock chosen rows
  ~~~

- It may lock an item long before using it, increasing wait time for others.
- It may request locks for conditional branches it never takes.
- A large all-or-nothing request can be difficult for a distributed lock manager.
- It prevents deadlocks but not necessarily cascading aborts: the conservative rule alone does not require X locks to be held until commit.
- It may cause starvation or long waits unless the scheduler is fair.

### 7.5 When it fits

- stored procedures with a small, predictable lock set;
- batch steps whose exact objects are declared in advance;
- safety-critical workloads where aborting for deadlock is especially expensive;
- systems willing to trade concurrency for deterministic lock acquisition.

### 7.6 When it fits poorly

- ad hoc queries;
- graph traversals or joins whose rows are discovered during execution;
- large range scans;
- workflows whose data-dependent branches touch unpredictable objects;
- cross-service operations where “lock everything first” would create a long fragile critical section.

## 8. Strict 2PL

### 8.1 Rule

Strict 2PL follows ordinary 2PL and adds:

> Every exclusive (`X`) lock is retained until the transaction commits or aborts.

Shared locks may be released after the transaction enters the shrinking phase, provided it never acquires another lock afterward.

~~~text
T1: S(A) --- X(B) --- X(C) --- U1(A) --- work --- C1 --- release B,C
                              ^ allowed after lock point
                                X locks stay to the end
~~~

### 8.2 Why it prevents cascading aborts

If `T1` writes `A`, it holds `X(A)` until commit/abort. Another transaction cannot acquire the conflicting S or X lock needed to read or overwrite `A` through the pure S/X-lock model.

~~~text
T1: X(A), W(A) --------------------------- C1, U(A)
T2:           S(A) requested -- WAIT ----- S(A), R(A)
~~~

`T2` can see only `T1`’s committed value, or the restored old value after `T1` aborts. No transaction becomes dependent on an uncommitted write, so abort does not cascade through dirty reads.

### 8.3 Why recovery is simpler

During rollback, no other transaction can have overwritten a value still protected by the victim’s X lock. The recovery manager can undo the victim without disentangling another committed transaction’s write to the same item.

### 8.4 Remaining limitations

- deadlocks still occur;
- S locks released before transaction end may let a writer proceed sooner, but only after the transaction is done acquiring locks;
- long write transactions retain X locks and increase blocking;
- readers in a pure lock-based engine wait behind X locks;
- strictness does not solve missing predicate/range protection;
- external side effects are outside the database lock guarantee.

---

## 9. Rigorous or strong-strict 2PL

### 9.1 Rule

Rigorous 2PL adds the strongest common retention rule:

> Hold **every S and X lock** until commit or abort.

No normal lock is released while the transaction is active:

~~~text
T1: acquire S(A) -- acquire X(B) -- acquire S(C) -- work -- COMMIT
                                                               |
                                                        release all locks
~~~

The transaction grows throughout execution; shrinking is effectively the release-at-end step.

CMU’s database notes call this **Strong Strict 2PL (also known as Rigorous 2PL)** and state that all locks are released at commit ([CMU 15-445: Two-Phase Locking](https://15445.courses.cs.cmu.edu/spring2023/notes/16-twophaselocking.pdf)).

### 9.2 What it adds beyond strict 2PL

- Shared locks are also held to transaction end.
- Lock release is operationally simple: release the transaction’s lock set at commit/abort.
- Under the standard model, conflict order aligns cleanly with transaction completion/commit ordering.
- It is automatically strict and cascadeless.

### 9.3 Concurrency cost

Suppose `T1` reads `A` early, later works on unrelated `B`, and commits much later.

With strict 2PL, after reaching its lock point, `T1` may release `S(A)`, allowing `T2` to update `A` sooner.

With rigorous 2PL:

~~~text
T1: S(A), R(A) -------- long work on B -------- C1, release S(A)
T2:       X(A) request ----------- WAIT ------- X(A), W(A)
~~~

The longer shared-lock lifetime delays writers. Therefore rigorous 2PL can allow less concurrency than strict 2PL.

### 9.4 What it still does not solve

- Deadlock: two transactions can hold different locks and request each other’s.
- Starvation: repeated victimization or unfair queues remain possible without policy.
- Hotspot throughput: one popular item still serializes conflicting access.
- Phantom safety: row locks alone may not protect a predicate such as “all seats in section A.”
- Long transactions: holding more locks longer worsens blocking and lock-table memory usage.

### 9.5 Terminology warning

You may encounter:

~~~text
strict 2PL       = X locks held to end
strong strict    = all locks held to end
rigorous 2PL     = all locks held to end
~~~

But some authors call the all-locks rule simply **strict 2PL**. A robust interview sentence is:

> “Using the definitions I know, strict 2PL retains write locks to transaction end, while rigorous—or strong-strict—2PL retains both read and write locks. I’ll use those definitions here.”

# Part IV — Problem 1: deadlock

## 10. From ordinary waiting to deadlock

### 10.1 Blocking is normal

~~~text
T1 holds X(A) and is still running
T2 requests S(A) and waits
T1 commits and releases X(A)
T2 proceeds
~~~

This is **blocking**, not deadlock. There is a path to progress: `T1` is not waiting for `T2`.

### 10.2 Deadlock is circular waiting

~~~text
time | T1                            | T2
-----+-------------------------------+-----------------------------
  1  | X1(A) granted                 |
  2  |                               | X2(B) granted
  3  | X1(B) requested: waits for T2 |
  4  |                               | X2(A) requested: waits for T1
~~~

Now:

~~~text
T1 holds A and waits for B held by T2
T2 holds B and waits for A held by T1

T1 ----------waits for----------> T2
 ^                                  |
 |                                  |
 +-----------waits for--------------+
~~~

Neither can reach commit, so neither releases its held lock. The cycle must be broken externally, usually by aborting one transaction.

### 10.3 Four necessary conditions—brief prerequisite

The classic Coffman conditions are useful vocabulary:

1. **Mutual exclusion:** a conflicting resource cannot be shared.
2. **Hold and wait:** a transaction holds one resource while waiting for another.
3. **No preemption:** locks are not normally taken away without aborting the owner.
4. **Circular wait:** a cycle of wait dependencies exists.

Deadlock prevention breaks at least one condition. For example:

- conservative 2PL breaks hold-and-wait;
- global lock ordering breaks circular wait;
- wound-wait selectively preempts younger transactions through abort.

### 10.4 Deadlock versus related problems

| Problem | Meaning | Does a WFG cycle necessarily exist? |
|---|---|---:|
| blocking | one transaction waits for another that can still progress | no |
| deadlock | a circular set cannot progress without intervention | yes, in standard lock model |
| starvation | a transaction repeatedly fails to obtain service | no |
| livelock | transactions keep reacting/retrying but make no useful progress | no stable wait cycle required |
| long-running transaction | owner simply takes a long time | no |

Killing every long waiter treats symptoms and can hide the difference.

## 11. Wait-for graph (WFG)

### 11.1 Construction

Create one node per active transaction. Add an edge:

~~~text
Ti ----> Tj
~~~

when `Ti` is blocked requesting a lock that conflicts with a lock held by `Tj`.

For the two-account deadlock:

~~~text
T1 waits for B held by T2  =>  T1 -> T2
T2 waits for A held by T1  =>  T2 -> T1

             +------+
             |      v
            T1     T2
             ^      |
             +------+
~~~

The graph has a directed cycle, so a deadlock exists.

### 11.2 A three-transaction cycle

~~~text
T1 holds A, wants B
T2 holds B, wants C
T3 holds C, wants A

T1 -> T2 -> T3 -> T1
~~~

Aborting any one transaction on the cycle can release locks and break this particular cycle.

### 11.3 Multiple blockers

If `T1` requests X on a row with compatible S locks held by both `T2` and `T3`, it may need both to release:

~~~text
T1 -> T2
T1 -> T3
~~~

Implementation details vary: some lock managers record edges to every blocker, to the effective queue predecessor, or use an implicit graph derived from lock queues. The correctness argument must match the queue and compatibility policy.

### 11.4 Cycle rule and scope

For ordinary reusable locks with one instance per logical resource, a WFG cycle is the standard necessary and sufficient deadlock condition.

Distributed systems complicate observation:

- different lock managers see only fragments;
- messages describing edges can be delayed or reordered;
- a transaction may finish while a global graph snapshot is assembled;
- a network partition can resemble a failed or blocked owner.

The conceptual cycle remains useful, but constructing a consistent global WFG is more expensive and may need centralized aggregation, probes/edge chasing, or careful snapshot logic.

### 11.5 Detection cost

For a fixed graph snapshot, a depth-first-search or strongly-connected-components approach can find cycles in `O(V + E)` time, where `V` is the number of transaction nodes and `E` is the number of wait edges. A lock manager often does not rescan the entire graph: after adding a new wait edge from `Ti`, it can search the affected dependency path to see whether it reaches `Ti` again.

The asymptotic bound is not the whole operational cost. Building a consistent graph, synchronizing with changing lock queues, and checking frequently under a hotspot may cost more than the graph algorithm itself.

## 12. Can a wait-for graph have a self-cycle?

### 12.1 Canonical answer: normally no

In the standard transaction-level WFG, edges are defined between **different transactions**:

~~~text
Ti -> Tj, where i != j
~~~

A transaction’s existing lock is normally compatible with its own request or treated as a conversion. It does not wait for itself to commit before it can proceed. Therefore a self-edge `T1 → T1` is not a normal deadlock representation.

PostgreSQL, for example, explicitly notes that a transaction’s own locks never conflict in its locking model ([PostgreSQL `LOCK` documentation](https://www.postgresql.org/docs/16/sql-lock.html)).

### 12.2 Lock upgrade does not create a canonical self-edge

If `T1` holds S(A) and requests X(A):

- if no other transaction holds a conflicting lock, the upgrade succeeds;
- if `T2` also holds S(A), `T1` waits for **T2**, so the edge is `T1 → T2`;
- if both upgrade, the cycle is `T1 → T2 → T1`, not two self-cycles.

### 12.3 When an apparent self-deadlock can occur

An apparent self-cycle usually means the graph’s node is coarser than the actual execution unit. Examples include:

- two parallel tasks/threads inside one session waiting on resources held by each other;
- two branches of one distributed transaction represented under one global ID;
- application advisory locks combined with database locks but modeled as one transaction node;
- reentrant-lock misuse in an application layer;
- a lock-manager bug, stale edge, or ID reuse artifact.

In those cases, model tasks/branches/resources at enough granularity. The interview-safe answer is:

> “Not in the canonical transaction wait-for graph: a transaction does not conflict with itself. If I see a self-loop, I would suspect a modeling artifact or multiple execution units collapsed into one node.”

## 13. Timeout: simple, useful, and imprecise

### 13.1 Policy

~~~text
request lock
   |
wait up to D
   |
   +-- lock granted -> continue
   |
   +-- D expires ----> fail/abort according to DB policy
~~~

Timeout avoids indefinite waiting without maintaining or traversing a WFG.

### 13.2 Can timeout cause a false abort?

Yes:

~~~text
T1 holds A and will release it at 1001 ms
T2's lock timeout is 1000 ms
T2 aborts at 1000 ms even though no cycle existed
~~~

`T2` was a slow waiter, not deadlocked. The timeout chose availability/latency bounds over preserving its work. This is sometimes informally called a **false deadlock** or false-positive abort, but strictly speaking no deadlock was detected—the system simply stopped waiting.

### 13.3 Choosing a timeout is a trade-off

| Timeout too short | Timeout too long |
|---|---|
| unnecessary aborts | slow detection of real deadlocks if timeout is the only mechanism |
| retry storms under legitimate contention | connections/threads remain occupied |
| wasted completed work | tail latency grows |
| unstable performance during brief spikes | locks and transactions stay blocked longer |

There is no universal correct duration. Base it on normal transaction/lock-wait distributions, latency objectives, rollback cost, workload bursts, and whether exact deadlock detection also exists.

### 13.4 Product example: delayed detection versus timeout

PostgreSQL has a `deadlock_timeout`: it waits before **running the expensive deadlock check**, rather than treating every wait as a deadlock. Its documentation explains the trade-off between needless checks and slower reporting ([PostgreSQL lock-management settings](https://www.postgresql.org/docs/current/runtime-config-locks.html)). This is distinct from a general statement/query timeout.

### 13.5 Database lock timeout versus client timeout

Do not merge these:

- A **database lock-wait timeout** is observed inside the database, which knows whether the lock request was granted and documents whether the statement or transaction is rolled back.
- A **client/network timeout** means the caller stopped waiting for a response. The server may have committed just before the response was lost.

~~~text
client ---- COMMIT ----> database commits
client <--- success X--- response lost
client timeout => outcome UNKNOWN, not definitely aborted
~~~

Use an operation/idempotency key and a status lookup for unknown external outcomes. Never retry a non-idempotent business action merely because the client timed out.

## 14. WFG detection and victim selection

### 14.1 Detection flow

~~~text
incompatible lock request
          |
          v
add/derive wait edge
          |
          v
check affected graph for a cycle
      /             \
   no cycle        cycle
      |              |
 keep waiting    choose victim
                     |
                 abort/undo
                     |
               release its locks
                     |
               wake eligible waiters
~~~

Checks may run on every wait, after a delay, periodically, or through a graph algorithm optimized around the new waiter. More frequent checks reduce deadlock duration but consume CPU/latch time.

### 14.2 Detection is not resolution

Finding a cycle does not itself restore progress. The system must:

1. choose at least one victim;
2. abort and undo it safely;
3. release its locks;
4. wake and reschedule waiters;
5. return a retryable error to the application;
6. prevent pathological repeated victimization where possible.

### 14.3 Victim-selection objective

A practical objective is:

> Break the cycle while minimizing total rollback cost, business harm, and repeated starvation.

Possible signals:

| Signal | Why it matters | Caveat |
|---|---|---|
| work already completed | more completed work may mean more waste | elapsed time is not the same as undo cost |
| rows/pages modified | approximates physical/logical undo | one row can still trigger expensive work |
| log/undo bytes generated | closer estimate of rollback effort | engine-specific |
| locks held | abort may unblock many waiters | many locks can also mean costly rollback |
| number of cycles involving transaction | one victim might break several cycles | graph can change during choice |
| number/priority of blocked transactions | improves global progress | can unfairly punish one long transaction |
| transaction age | protects old transactions from starvation | old may be very expensive to undo |
| previous abort count | avoids choosing same victim repeatedly | requires history across retries |
| explicit business/deadlock priority | protects critical work | misuse can starve low-priority work |
| estimated work remaining | tempting optimization | usually hard or impossible to predict reliably |
| external side effects | abort may require reconciliation | ideally external calls are outside open DB transactions |

Your bullet **“effort till now, effort remaining, effort to rollback, number of cycles”** is therefore directionally good. Refine it as:

- effort done is only a proxy;
- remaining effort is often unknown;
- actual undo/log cost is more actionable;
- cycle participation and number of waiters unblocked measure system-wide benefit;
- age, priority, and past victim count protect fairness.

### 14.4 Real product examples

- **MySQL/InnoDB product behavior:** InnoDB says it tries to choose a small transaction, using the number of inserted, updated, or deleted rows as transaction size ([MySQL deadlock detection](https://dev.mysql.com/doc/refman/8.4/en/innodb-deadlock-detection.html)).
- **SQL Server product behavior:** it first considers configured deadlock priority; when priorities match, it chooses the transaction estimated cheaper to roll back using log bytes written so far ([Microsoft `SET DEADLOCK_PRIORITY`](https://learn.microsoft.com/en-us/sql/t-sql/statements/set-deadlock-priority-transact-sql?view=sql-server-ver17)).
- **PostgreSQL product behavior:** PostgreSQL documents that exactly which transaction is aborted is difficult to predict and applications should not rely on it ([PostgreSQL explicit locking](https://www.postgresql.org/docs/15/explicit-locking.html)).

These differences are why an interview answer should describe policy choices before claiming what “the database” always does.

### 14.5 Victim retry rule

Retry the **entire logical transaction** from a clean transaction boundary:

~~~text
attempt transaction
   |
deadlock victim?
   |
rollback complete
   |
bounded retry with jitter/backoff
   |
re-read current state and re-run business checks
~~~

Do not reuse values read before the abort; another transaction has progressed. Bound retries, add jitter, respect request deadlines, and surface persistent failure instead of looping forever.

## 15. Prevention option: consistent lock ordering

### 15.1 Rule

Define a total order over lockable resources and always acquire locks in that order.

For account transfers:

~~~text
lock account with smaller account_id first
lock account with larger account_id second
~~~

Even when transferring `B → A`, acquire `A` then `B` if `A < B`.

### 15.2 Why it works

All wait edges move in the same resource order. A circular wait would require eventually moving backward to a smaller resource, which the rule forbids.

~~~text
A < B < C

allowed acquisition: A -> B -> C
forbidden:           B -> A
~~~

### 15.3 Practical strengths

- no need to know every lock before starting, only the order rule;
- no deadlock detector needed for cycles fully covered by the order;
- simpler and often more concurrent than conservative preclaiming;
- easy for symmetric operations such as transfers between two IDs.

### 15.4 Practical limits

- every code path, service, trigger, stored procedure, and background job must obey the same order;
- data-dependent operations may discover a lower-ordered resource late;
- lock upgrades can violate the spirit of the ordering;
- hidden locks on indexes, foreign keys, ranges, and metadata can still create unexpected cycles;
- different resource classes need one coherent composite order.

PostgreSQL and MySQL both recommend acquiring multiple objects in a consistent order; MySQL also recommends short transactions, appropriate indexes, and full transaction retry ([PostgreSQL explicit locking](https://www.postgresql.org/docs/15/explicit-locking.html), [MySQL deadlock handling](https://dev.mysql.com/doc/refman/8.4/en/innodb-deadlocks-handling.html)).

## 16. Prevention option: conservative/static 2PL

Conservative 2PL, covered in Section 7, is a deadlock-prevention strategy because it prevents partial ownership while waiting.

### 16.1 Comparison with lock ordering

| Question | Conservative/static | Global lock order |
|---|---|---|
| must know complete set first? | yes | no, but future acquisitions must respect order |
| holds partial set while waiting? | no | possibly yes |
| deadlock-free if rule fully obeyed? | yes | yes |
| reserves resources early? | often | less necessarily |
| supports data-dependent discovery? | poorly | better if discoveries are monotonic in order |

### 16.2 Why “less concurrency” is accurate

Conservative 2PL may reserve `C` at transaction start even though it will not use `C` until much later. Another transaction that needs only `C` waits despite no immediate operation conflict.

~~~text
T1 preclaims A, B, C | uses A ----- uses B ----- uses C ----- commit
T2 needs only C      | ------------------- WAIT ------------- run
~~~

The protocol trades utilization and concurrency for deadlock avoidance and predictable ownership.

## 17. Timestamp-based prevention: wait-die

### 17.1 Timestamp prerequisite

Assign each transaction a unique age when it first starts:

~~~text
smaller/earlier timestamp = older transaction

TS(T1) = 10  -> older
TS(T2) = 20  -> younger
~~~

If a transaction aborts because of the prevention policy and retries, it normally keeps its **original timestamp**. Otherwise it would always restart as the youngest and could starve forever.

This timestamp is being used to decide **who waits and who aborts**. The database is still using locks; this is not the separate timestamp-ordering concurrency-control protocol.

### 17.2 Rule

When requester `Ti` asks for a lock held incompatibly by `Tj`:

~~~text
if Ti is older than Tj:
    Ti WAITS
else:
    Ti DIES (aborts and later retries with original timestamp)
~~~

Mnemonic:

> **Old waits; young dies.**

### 17.3 Your `T1 wait` case

Assume `T1` is older and `T2` is younger:

~~~text
TS(T1) = 10, TS(T2) = 20
T2 holds X(A)
T1 requests X(A)

older T1 requests from younger T2 -> T1 waits
~~~

Reverse request:

~~~text
T1 holds X(B)
T2 requests X(B)

younger T2 requests from older T1 -> T2 dies/aborts
~~~

So “Wait-die → `T1` waits” is incomplete unless `T1` is older and the holder is younger.

### 17.4 Why cycles are impossible

Wait-die permits wait edges only from older to younger:

~~~text
older ----waits for----> younger
~~~

Age must strictly decrease or timestamp must strictly increase along every wait edge, depending on how it is written. A path cannot return to its starting timestamp, so a cycle cannot form.

### 17.5 Non-preemptive nature

Wait-die is **non-preemptive**:

- an older requester does not abort the younger holder; it waits;
- a younger requester aborts itself rather than taking a lock from the older holder.

### 17.6 Trade-offs

- A young transaction may abort repeatedly while an old holder runs.
- It can cause unnecessary aborts even when waiting would not create a full cycle.
- Retaining the original timestamp reduces starvation risk but does not make poor retry loops harmless.
- Because holders are not preempted, an already-running younger holder can finish its critical section rather than lose completed work.
- The policy is simple and prevents deadlock without WFG cycle detection.

The timestamp-prevention family traces to Rosenkrantz, Stearns, and Lewis’s distributed-concurrency-control work ([ACM DOI: *System Level Concurrency Control for Distributed Database Systems*](https://doi.org/10.1145/320251.320260)). University database notes describe wait-die as older-waits/younger-aborts and emphasize retaining the old timestamp on restart ([Emory: wait-die](https://www.cs.emory.edu/~cheung/Courses/554/Syllabus/8-recv%2Bserial/deadlock-waitdie.html)).

## 18. Timestamp-based prevention: wound-wait

### 18.1 Rule

When requester `Ti` asks for a lock held incompatibly by `Tj`:

~~~text
if Ti is older than Tj:
    Ti WOUNDS Tj -> abort younger holder Tj
else:
    Ti WAITS for older holder Tj
~~~

Mnemonic:

> **Old wounds; young waits.**

### 18.2 Your `T2 aborts` case

Assume `T1` is older and `T2` is younger:

~~~text
TS(T1) = 10, TS(T2) = 20
T2 holds X(A)
T1 requests X(A)

older T1 wounds younger holder T2
T2 aborts; after its locks are safely released, T1 can proceed
~~~

Reverse request:

~~~text
T1 holds X(B)
T2 requests X(B)

younger T2 waits for older T1
~~~

So “wound-wait → `T2` aborts” is correct only for the first age/ownership arrangement.

### 18.3 Why cycles are impossible

Wound-wait permits surviving wait edges only from younger to older:

~~~text
younger ----waits for----> older
~~~

Timestamps strictly move toward older transactions along the path, so they cannot circle back.

### 18.4 Preemptive nature

Wound-wait is **preemptive**: an older requester can force a younger lock holder to abort. This prioritizes older transactions and may release multiple locks held by the wounded transaction.

### 18.5 Trade-offs

- It may abort a younger transaction after that transaction has already done substantial work.
- A younger requester can wait instead of repeatedly self-aborting against an older holder.
- Preemption can unblock an old/high-priority transaction quickly.
- It still makes conservative choices: it may abort in a conflict that would never have completed a cycle.
- Safe abort and undo must finish before the requester may consume uncommitted state.

### 18.6 Wait-die versus wound-wait

| Conflict | Wait-die | Wound-wait |
|---|---|---|
| older requests lock held by younger | older waits | younger holder aborts |
| younger requests lock held by older | younger requester aborts | younger waits |
| preempt holder? | no | yes, when holder is younger |
| permitted WFG edge direction | older → younger | younger → older |
| deadlock possible under correct policy? | no | no |
| restart timestamp | original | original |

Both ultimately favor older transactions. The memorable difference is **who is forced to abort at the moment of conflict**.

### 18.7 Four-case drill

Let `O` be old and `Y` young:

~~~text
WAIT-DIE
O asks Y -> O waits
Y asks O -> Y dies

WOUND-WAIT
O asks Y -> O wounds Y
Y asks O -> Y waits
~~~

If you can reconstruct this square, you do not need to memorize a paragraph.

# Part V — Problem 2: cascading aborts

## 19. How basic 2PL still permits dirty dependency

### 19.1 Example

`T1` obtains all locks before releasing any, so it obeys basic 2PL:

~~~text
time | T1                                      | T2
-----+-----------------------------------------+---------------------------
  1  | X1(A)                                   |
  2  | X1(B)   <- final acquisition            |
  3  | W1(A = 50)                              |
  4  | U1(A)   <- shrinking begins             |
  5  |                                         | S2(A), R2(A = 50)
  6  | error while processing B                |
  7  | A1; restore old A                       |
  8  |                                         | A2 required: it read dirty A
~~~

No 2PL rule was broken:

- `T1` acquired `A` and `B` before releasing `A`;
- after releasing `A`, it acquired no new lock.

Yet `T2` observed `T1`’s uncommitted write. Serializability alone does not guarantee an easy recovery order.

### 19.2 Cascade chain

~~~text
T1 writes A, releases A before commit
          |
          v
T2 reads dirty A, writes C, releases C before commit
          |
          v
T3 reads dirty C

T1 aborts -> T2 aborts -> T3 aborts
~~~

One failure wastes work across many transactions and complicates recovery bookkeeping.

---

## 20. Recoverable, cascadeless, and strict schedules

These terms form a useful strength hierarchy.

### 20.1 Recoverable

If `T2` reads a value written by `T1`, `T2` must not commit before `T1` commits.

~~~text
T2 reads-from T1  =>  C1 must occur before C2
~~~

Dirty reads may still happen, so cascading abort remains possible; commit order merely avoids a committed transaction depending on an aborted one.

### 20.2 Cascadeless

If `T2` reads a value written by `T1`, it may do so only **after `T1` commits**.

~~~text
W1(A) -- C1 -- R2(A)
~~~

No dirty reads means no read-dependency cascade.

### 20.3 Strict

After `T1` writes `A`, no other transaction may **read or write `A`** until `T1` commits/aborts.

~~~text
W1(A) -------- C1/A1 -------- R2(A) or W2(A)
~~~

This prevents dirty reads and dirty writes, simplifying undo.

### 20.4 Relationship

~~~text
strict  =>  cascadeless  =>  recoverable

reverse implications do not generally hold
~~~

Strict 2PL generates strict schedules under the standard S/X model; rigorous 2PL does too.

---

## 21. Which variant resolves cascading aborts?

### 21.1 Minimum common answer: strict 2PL

Holding X locks until commit/abort is enough to stop others from acquiring S or X on uncommitted written items. Therefore:

> **Strict 2PL prevents cascading aborts; rigorous 2PL also prevents them but is stronger than necessary for that purpose.**

### 21.2 Why your “strong strict/rigorous” bullet is understandable

Many courses emphasize rigorous/strong-strict 2PL because its rule is extremely simple—release all locks only at transaction end. That certainly fixes the dirty-read schedule.

But attributing the fix only to rigorous 2PL misses the exact trade-off:

| Variant | X lock after write | S lock after last read | Cascading abort prevented? |
|---|---|---|---:|
| basic | may release before commit | may release | no |
| strict | held to commit/abort | may release after lock point | yes |
| rigorous | held to commit/abort | held to commit/abort | yes |

Rigorous retains shared locks longer, which can delay writers without adding more cascading-abort protection than strict 2PL already supplies.

### 21.3 Recovery versus concurrency trade-off

~~~text
more early release
    -> more possible concurrency
    -> harder recovery / possible dirty dependency

more lock retention
    -> simpler recovery / stronger strictness
    -> more blocking, larger lock footprint, longer waits
~~~

The right answer is not “always use the strongest.” It is to state the required isolation/recovery property and the workload cost.

# Part VI — Technical depth for HLD

## 22. Lock granularity

What exactly is being locked?

| Granularity | Benefit | Cost/risk |
|---|---|---|
| database/table | few lock records; simple | unrelated rows block each other |
| partition/page | middle ground | false contention among rows in same unit |
| row/index record | unrelated rows proceed | more lock-manager memory/CPU |
| key range/predicate | protects absence and phantom-producing ranges | can cover many current/future records |
| advisory/business key | application-defined resource | database may not connect it to actual row changes |

2PL’s theorem assumes the chosen lockable items correctly represent conflicts. If the business invariant is “there must be no booking in this time range,” locking only currently existing rows may not protect the empty gaps where another transaction inserts a conflicting booking.

### 22.1 Phantom example

~~~text
T1: SELECT COUNT(*) FROM seats WHERE section='A' AND status='AVAILABLE'
    -> locks existing matching rows only

T2: INSERT a new AVAILABLE seat in section A
~~~

If the repeated predicate must remain stable, the database needs a range/predicate technique, a serializable mechanism, or a redesigned invariant-owning row. Row-level 2PL is only as complete as its conflict representation.

### 22.2 Intention locks—brief prerequisite

With hierarchical granularity, a transaction may lock rows while another asks to lock the entire table. **Intention locks** announce lower-level ownership at higher levels so compatibility can be checked without scanning every row lock.

~~~text
Database
  |
Table Orders  <- intention-exclusive says: X locks exist below
  |
Row order-42  <- X lock
~~~

They are a lock-management scalability mechanism, not a replacement for row X locks.

### 22.3 Lock escalation

An engine may replace many fine-grained locks with a coarser lock to reduce lock-table overhead:

~~~text
10,000 row locks  ->  one table/partition lock
~~~

This saves memory/management work but can suddenly block unrelated transactions. Exact thresholds and behavior are product-specific; do not state a universal number.

## 23. 2PL, isolation levels, and MVCC

### 23.1 2PL is an implementation family, not a SQL isolation-level name

SQL exposes names such as Read Committed, Repeatable Read, and Serializable. A database can implement them using combinations of:

- locks and 2PL-like retention;
- multi-version concurrency control (MVCC);
- predicate/range locks;
- serializable snapshot isolation and dependency detection;
- optimistic validation;
- engine-specific mechanisms.

Do not say “Serializable always means rigorous 2PL.” PostgreSQL, for example, documents MVCC and serializable conflict detection rather than a simple textbook all-read-locks-to-commit implementation ([PostgreSQL concurrency control](https://www.postgresql.org/docs/17/mvcc.html)).

### 23.2 MVCC changes reader behavior

In a pure lock-only model:

~~~text
writer holds X(A) -> reader requesting S(A) waits
~~~

In an MVCC engine, a normal reader may instead read an older committed version:

~~~text
A_v1 committed ---- A_v2 uncommitted by writer
reader snapshot sees A_v1 without waiting for A_v2
~~~

Writers and locking reads may still take transaction locks, and strict write-lock retention remains important. Product behavior depends on isolation level and statement form such as ordinary `SELECT` versus `SELECT ... FOR UPDATE`.

### 23.3 Safety question to ask

When someone says “the database locks the row,” ask:

1. Which engine and isolation level?
2. Which SQL statement acquired it?
3. Is it a row, index record, key range, table, or advisory lock?
4. Which modes conflict?
5. How long is the lock retained?
6. Can snapshot readers bypass it?
7. What happens on timeout, deadlock, savepoint rollback, and connection loss?

## 24. Performance and operational costs

### 24.1 Blocking amplifies latency

One transaction holding a hot lock can create a queue:

~~~text
T1 holds hot row
   |
   +-- T2 waits
   +-- T3 waits
   +-- T4 waits
   +-- T5 waits
~~~

If `T1` slows, downstream wait time accumulates. This is **lock convoying**. Tail latency can rise even when average transaction time looks acceptable.

### 24.2 Long transactions are especially harmful

Do not normally keep a transaction open while:

- waiting for user input;
- calling a remote payment or shipping service;
- performing long CPU work;
- sleeping before a retry;
- streaming a large response;
- waiting for an asynchronous workflow step.

Long transactions retain locks and connections, increase deadlock exposure, enlarge rollback work, and in MVCC engines may also retain old versions.

### 24.3 Index choice changes lock footprint

A poor access path can scan and lock far more records/ranges than intended. A suitable index can:

- find target rows faster;
- reduce the number of records examined/locked;
- shorten lock duration;
- reduce collision probability.

But an update may also lock index entries, uniqueness checks, or foreign-key-related records. “One SQL row” does not guarantee one internal lock.

### 24.4 Fairness and starvation

A lock manager may use FIFO-like queues, priority, compatibility-aware granting, or vendor-specific rules. Risks include:

- a writer waiting behind a continuous stream of compatible readers;
- a low-priority transaction repeatedly chosen as deadlock victim;
- wait-die retries happening too aggressively;
- wound-wait repeatedly aborting young long-running work.

Mitigations include aging, retained timestamps, bounded retries with jitter, admission control, fair lock queues, and breaking large work into correct shorter transactions.

### 24.5 Metrics worth monitoring

- lock-wait count and duration by resource/query;
- blocked-session count and age of oldest blocker;
- deadlocks and victims by transaction type;
- rollback duration and undo/log volume;
- retry success and retry-exhaustion rate;
- locks held per transaction and lock-table pressure;
- hot rows/keys/ranges;
- transaction duration percentiles;
- connection-pool occupancy caused by blocked transactions;
- client timeouts whose database outcome is unknown.

Never optimize by looking only at total deadlock count. A rare deadlock with safe fast retry may be healthier than eliminating all deadlocks using a global table lock.

## 25. Centralized versus distributed 2PL

### 25.1 Central database

One database lock manager can see lock ownership and local wait relationships:

~~~text
App A ----+
App B ----+--> one DB lock manager --> data
App C ----+
~~~

This does not mean one application process; many service instances rely on the shared database as the authoritative coordinator.

### 25.2 Sharded/distributed database

Transactions may acquire locks from several shard lock managers:

~~~text
                 +--> Shard A lock manager: T1 holds account A
Transaction T1 --+
                 +--> Shard B lock manager: T1 waits for account B

Transaction T2 holds B and waits for A
~~~

No one shard sees the full cycle. Options include:

- a centralized/global WFG service;
- periodically exchanging wait edges;
- probe/edge-chasing algorithms;
- timestamp prevention such as wait-die/wound-wait;
- transaction-level timeouts;
- ordered partition access;
- avoiding multi-shard lock scope through data placement or single ownership.

Distributed observation adds stale-edge and failure-detector complications. A timeout may conflate congestion, partition, crashed coordinator, and true circular wait.

### 25.3 Interaction with 2PC

A distributed transaction may:

1. acquire locks under distributed 2PL;
2. prepare participants under 2PC;
3. retain locks while the global decision is made/delivered;
4. release after commit/abort.

~~~text
2PL protects isolation during execution
2PC protects one commit outcome across participants
~~~

If a participant is prepared but cannot learn the decision, strict lock retention can block other transactions. This is one reason cross-shard synchronous transactions have availability and tail-latency costs.

### 25.4 HLD design preference

Before proposing a global distributed lock graph, ask whether the invariant can be placed under one owner:

- co-locate accounts/items that transact together;
- use an atomic conditional write at the authoritative shard;
- route commands for one key to a single writer;
- reserve locally and coordinate a longer workflow asynchronously;
- use a Saga when semantic compensation is valid.

Do not distribute a lock if a simpler ownership boundary can eliminate the cross-node critical section.

# Part VII — Alternatives and selection

## 26. Alternatives to 2PL

| Approach | Core idea | Better when | Main trade-off |
|---|---|---|---|
| atomic conditional update | write only if current predicate/version matches | one-row state transition or counter | limited to expressible atomic boundary |
| optimistic concurrency control (OCC) | execute without long blocking; validate before commit | conflicts are rare and retries cheap | wasted work/retry storms under contention |
| MVCC/snapshot reads | readers use committed versions | read-heavy workloads | write conflicts and serializability still need handling |
| timestamp-order protocol | enforce operation order using read/write timestamps | engine designed for non-lock ordering | aborts instead of waits; different recovery complexity |
| serializable snapshot isolation | track dependency patterns and abort dangerous structures | read-heavy serializable workloads | serialization failures require retry |
| single writer/actor per key | route mutations of one key through ordered owner | natural partition key and command model | hot key throughput and failover complexity |
| deterministic transaction ordering | decide transaction order before execution | predictable distributed database workload | batching/order service and slow transaction issues |
| escrow/reservation | allocate bounded rights in advance | inventory/quota with commutative decrements | reservation expiry/reconciliation |
| coarse serialization | one table/global/semaphore lock | very low throughput, simple critical operation | severe concurrency loss and availability coupling |

### 26.1 2PL versus OCC

~~~text
2PL: conflict -> wait before unsafe access
OCC: conflict -> discover at validation/commit and abort someone
~~~

- Prefer lock-based pessimism when conflict is common or wasted work is expensive.
- Prefer optimism when conflict is rare, lock holding would be long, and retry is cheap.
- A hot item is a serialization point either way: 2PL queues contenders; OCC may repeatedly abort them.

### 26.2 2PL versus single writer

A per-key single writer makes ordering explicit in routing/queues rather than dynamically acquiring several locks. It can eliminate same-key lock deadlocks, but multi-key operations still need ordering, coordination, or redesign.

### 26.3 When to use/accept 2PL

- the database already provides appropriate transactional locking;
- invariants span a small known set of rows/ranges;
- conflicts are common enough that early waiting is cheaper than late abort;
- transactions are short;
- deadlock retry is safe and implemented;
- recovery simplicity from strict lock retention is valuable.

### 26.4 When to avoid or redesign

- the operation waits on people or remote services;
- a transaction would lock huge unpredictable ranges;
- hot locks dominate throughput;
- cross-region/shard lock latency exceeds the business goal;
- the invariant can be enforced with one atomic conditional statement;
- asynchronous reservation/workflow semantics are acceptable;
- retries cannot safely reproduce side effects;
- the database’s actual MVCC/serializable mechanism already offers a better fit.

# Part VIII — Real-system examples

## 27. Evidence boundary for named systems

The following are **illustrative designs**, not claims about current internal implementations at Amazon, Uber, WhatsApp, Netflix, YouTube, or Twitter/X. Their private architectures evolve and use multiple storage engines. The examples show where 2PL reasoning helps and where it does not.

## 28. Amazon-like inventory reservation

### 28.1 Invariant

Confirmed/reserved quantity must not exceed available inventory for each SKU.

### 28.2 Short local transaction

For one SKU, an atomic conditional update is usually simpler than manually discussing a multi-step lock protocol:

~~~sql
UPDATE inventory
SET available = available - :quantity
WHERE sku = :sku
  AND available >= :quantity;
~~~

Affected rows:

- `1`: reservation succeeded;
- `0`: insufficient inventory or missing SKU.

The database may internally use locks or MVCC conflict handling, but the application expresses one atomic state transition.

### 28.3 Multi-SKU order with strict 2PL

~~~text
Order Service
     |
     | begin DB transaction
     | sort SKUs by ID
     v
lock SKU-10 -> lock SKU-42 -> validate quantities -> decrement -> commit
     ^              ^                                      |
     +-------- X locks held under strict policy ------------+
~~~

Consistent SKU order reduces deadlocks. Strict lock retention prevents another transaction from reading/overwriting uncommitted decrements through conflicting locks.

### 28.4 Edge cases

- One SKU is hot: lock queues dominate; use admission control, partitioned inventory rights, or reservation/escrow design.
- Query lacks a good SKU index: it may touch/lock more records than expected.
- Payment call inside transaction: avoid; it makes inventory locks depend on a remote service.
- Deadlock victim: retry the full reservation with the same order/idempotency identity, then re-evaluate current stock.
- Client times out after commit: look up reservation by operation ID instead of blindly decrementing again.

## 29. Ticket/seat booking

### 29.1 Two seats, reverse choice order

~~~text
User 1 selects A7 then A8
User 2 selects A8 then A7

T1 holds A7, waits A8
T2 holds A8, waits A7 -> deadlock
~~~

Canonical ordering—always lock lower seat ID first—removes this cycle.

### 29.2 Better workflow boundary

Do not hold a database transaction while the user enters payment details:

~~~text
AVAILABLE
    |
short transaction: conditional hold with expiry
    v
HELD(order_id, expires_at)
    |
payment outside lock-holding transaction
    +---- success -> short confirm transaction -> SOLD
    |
    +---- fail/expiry -> short release transaction -> AVAILABLE
~~~

2PL can protect each short state transition. It should not turn the entire human/payment workflow into one long locked transaction.

### 29.3 Interview insight

The interviewer is usually testing whether you separate:

- row-level concurrency at the authoritative seat record;
- idempotency for duplicate booking requests;
- reservation expiry;
- payment’s unknown outcome;
- cross-service workflow recovery.

“Use a lock” is only the first line of the answer.

## 30. Uber-like driver assignment

### 30.1 Invariant

A driver must not have two active trips.

### 30.2 Candidate claim pattern

Multiple dispatch workers may race:

~~~text
Dispatcher A ----+
                 +--> authoritative driver state
Dispatcher B ----+
~~~

Possible atomic transition:

~~~sql
UPDATE drivers
SET state = 'ASSIGNED', active_trip_id = :trip
WHERE driver_id = :driver
  AND state = 'AVAILABLE';
~~~

Exactly one contender changes the row. This avoids a separate read-then-lock conversation for a single driver.

For claiming from a pool, some databases support locking-read patterns such as `FOR UPDATE SKIP LOCKED`. That can help workers claim different available rows, but exact fairness and vendor semantics must be checked.

### 30.3 Why not rigorous 2PL across the full ride?

Holding a driver’s transaction locks from request through pickup and trip completion would be disastrous: the workflow lasts minutes, crosses devices and services, and must survive disconnects. Commit a short assignment transition, then represent ride progress as durable states/events.

## 31. Netflix-like subscription/payment activation

### 31.1 Local invariant versus remote workflow

Within one billing ledger database, strict transactional locking can protect a short ledger/entitlement update.

But this is a poor design:

~~~text
BEGIN
lock subscription row
call external payment provider -------- long/uncertain wait
activate entitlement
COMMIT
~~~

A better boundary:

~~~text
create PENDING subscription (short transaction)
             |
authorize/capture using stable payment operation ID
             |
confirm ACTIVE or move to FAILED/REVIEW (short transaction)
~~~

Use idempotency, status queries, and workflow recovery across the provider boundary. 2PL cannot roll back an external card charge.

## 32. WhatsApp-like message send

A global rigorous-2PL transaction across sender storage, recipient devices, push notification, and read receipts would couple message availability to offline devices.

~~~text
Sender -> Message Service
              |
              +-- short local transaction: message + outbox
              |
              +-- asynchronous delivery -> recipient devices
              |
              +-- later delivered/read acknowledgements
~~~

Locks may protect a short local write or sequence allocation. They should not be held until every phone acknowledges. `SENT`, `DELIVERED`, and `READ` are legitimate durable states, not one transaction waiting to commit.

## 33. YouTube-like upload and Netflix-like media pipelines

~~~text
UPLOADING -> STORED -> TRANSCODING -> READY
                |           |
                |           +--> retry failed rendition
                +--> cleanup expired partial upload
~~~

Video upload, object storage, transcoding, moderation, indexing, and notification are long-running. Use durable workflow/state-machine techniques. 2PL may protect individual metadata transitions, but holding locks across the entire pipeline would reduce availability and make crash recovery worse.

## 34. Twitter/X-like post fan-out

~~~text
Post API -> authoritative Post DB + outbox
                            |
                            +--> home timelines
                            +--> search index
                            +--> notifications
                            +--> analytics
~~~

Use one short transaction for authoritative post creation and outbox insertion. Derived fan-out copies can update asynchronously. Do not use distributed 2PL across millions of timelines: those copies are rebuildable views, not co-owners of the post-creation invariant.

## 35. Product decision summary

| System example | Good 2PL-sized boundary | Boundary to avoid locking across |
|---|---|---|
| Amazon-like inventory | short reservation/decrement of authoritative SKU rows | payment, shipping, human checkout |
| seat booking | short AVAILABLE → HELD/SOLD transition | hold expiry and payment interaction |
| Uber-like dispatch | short AVAILABLE → ASSIGNED transition | entire trip lifecycle |
| Netflix-like subscription | local ledger/entitlement state transition | external payment provider and email |
| WhatsApp-like chat | local message durability/sequence metadata | recipient delivery/read receipts |
| YouTube-like upload | one metadata state change | upload/transcode/moderation pipeline |
| Twitter/X-like post | post + outbox local commit | fan-out, indexing, analytics |

The recurring HLD lesson is:

> Use locks to protect short authoritative invariants, not to make a long distributed workflow pretend to be one local critical section.

## 39. Main answer spine

Use **R-P-G-G-R-O**:

1. **Reason:** concurrent transactions need a serializable conflict order.
2. **Phases:** grow by acquiring/upgrading; after first release, shrink only.
3. **Guarantee:** conflict serializability, explained by lock-point order.
4. **Gaps:** basic 2PL can deadlock and can allow cascading aborts.
5. **Refinements:** conservative avoids deadlock; strict holds X locks to end; rigorous holds S and X to end.
6. **Operations:** detect/prevent deadlocks, choose a cheap/fair victim, retry the whole transaction, keep transactions short.

-> deadlock + dirty dependency remain

## 41. Fast follow-up answers

### Q: Why is it called two-phase locking?

Because each transaction has a lock-growing period followed by a lock-shrinking period. It is not about two locks and not the prepare/commit phases of 2PC.

### Q: What marks the phase boundary?

The first lock release begins shrinking. Equivalently, the last acquisition occurs at the lock point. After a release/downgrade, no new lock or upgrade is legal.

### Q: What does basic 2PL guarantee?

Conflict serializability, assuming locks correctly represent every relevant conflict and the protocol is obeyed.

### Q: Why can basic 2PL deadlock?

Growing transactions can hold different locks while requesting more. That permits circular hold-and-wait, such as `T1` holding A and waiting B while `T2` holds B and waits A.

### Q: Why can basic 2PL cascade aborts?

After acquiring its last lock, a transaction may release an X lock before commit. Another transaction can then read that uncommitted value; if the writer aborts, the reader must also abort.

### Q: What is the smallest common 2PL refinement that prevents cascades?

Strict 2PL: retain all X locks until commit/abort. Rigorous 2PL also works but additionally retains S locks.

### Q: Does rigorous 2PL avoid deadlocks?

No. It simplifies strictness and lock release but retains hold-and-wait; transactions may still acquire items in opposite orders.

### Q: Does conservative 2PL prevent cascading abort?

Not by its preclaiming rule alone. It prevents deadlock. To prevent cascading aborts too, combine it with strict X-lock retention or an equivalent rule.

### Q: How do you detect deadlock?

Build or derive a WFG with waiter-to-holder edges. A directed cycle indicates deadlock in the standard lock model. Then choose and roll back a victim.

### Q: Is a WFG self-loop normal?

No. A transaction’s locks normally do not conflict with itself. A self-loop suggests execution units were collapsed, an advisory/application lock issue, stale graph state, or a bug.

### Q: Why is timeout not exact detection?

It uses elapsed waiting as a proxy. A non-deadlocked transaction may exceed the threshold and be aborted just before the holder releases.

### Q: Who should be the victim?

Prefer a victim that breaks the cycle at low rollback/business cost while preserving fairness: consider log/undo work, locks and waiters, transaction priority/age, past aborts, and cycle participation. Product policies differ.

### Q: Wait-die in one sentence?

Old requester waits for young holder; young requester dies against old holder. It is non-preemptive.

### Q: Wound-wait in one sentence?

Old requester wounds/aborts young holder; young requester waits for old holder. It is preemptive.

### Q: Why keep the original timestamp after retry?

So an repeatedly aborted transaction ages in priority relative to new arrivals instead of restarting forever as the youngest.

### Q: What should application code do after a deadlock error?

Wait for rollback completion, retry the entire transaction from a clean boundary with bounded jitter/backoff, re-read state, preserve the logical operation ID, and stop after a policy limit/deadline.

## 42. Schedule drills

### Drill 1: does it obey 2PL?

~~~text
S1(A), R1(A), U1(A), X1(B), W1(B), C1
~~~

**No.** `T1` acquired `X(B)` after releasing `S(A)`.

### Drill 2: basic or strict?

~~~text
X1(A), W1(A), S1(B), R1(B), U1(A), U1(B), C1
~~~

- It may obey **basic 2PL** if `S1(B)` was the final acquisition and all unlocks followed.
- It is **not strict 2PL** because X(A) was released before commit.

### Drill 3: strict or rigorous?

~~~text
S1(A), R1(A), X1(B), W1(B), U1(A), C1, U1(B)
~~~

- It can be **strict 2PL**: X(B) remains to commit.
- It is **not rigorous**: S(A) was released before commit.

### Drill 4: WFG

~~~text
T1 waits for a lock held by T2
T2 waits for locks held by T3 and T4
T3 waits for a lock held by T1
T4 is running
~~~

Edges:

~~~text
T1 -> T2
T2 -> T3
T2 -> T4
T3 -> T1
~~~

Cycle: `T1 → T2 → T3 → T1`. `T4` is not on that cycle. Aborting `T4` would not break it.

### Drill 5: timestamps

`T1` is oldest, `T3` youngest. `T3` requests a lock held by `T1`.

- wait-die: `T3` aborts;
- wound-wait: `T3` waits.

Now `T1` requests a lock held by `T3`:

- wait-die: `T1` waits;
- wound-wait: `T3` aborts.

# Part X — Common mistakes and edge cases

## 43. Common mistakes corrected

| Weak/incorrect statement | Interview-ready correction |
|---|---|
| “2PL means two locks” | “It divides lock acquisition and release into growing and shrinking phases” |
| “Growing is a 2PL type” | “Growing and shrinking are phases; basic, conservative, strict, and rigorous are variants” |
| “2PL prevents deadlocks” | “Basic, strict, and rigorous 2PL can deadlock; conservative 2PL prevents it under its model” |
| “A wait means deadlock” | “A wait becomes deadlock only when dependencies form a cycle” |
| “Timeout detected a deadlock” | “Timeout bounded the wait and may abort a non-deadlocked transaction” |
| “Any WFG path is deadlock” | “A directed cycle is required in the standard model” |
| “A lock upgrade is a self-cycle” | “It waits on other transactions’ incompatible locks; two upgraders form a two-node cycle” |
| “Abort the youngest is always optimal” | “Age is one fairness heuristic; rollback cost, priority, waiters, and repeat victimization also matter” |
| “Remaining effort tells us the best victim” | “Remaining effort is usually uncertain; log/undo work is more measurable” |
| “Conservative 2PL is just taking locks in order” | “It preclaims the complete set without holding a partial set; global ordering is a different prevention rule” |
| “Conservative 2PL prevents cascading aborts” | “Not unless combined with strict X-lock retention or another dirty-read prevention rule” |
| “Strict and rigorous are always synonyms” | “Terminology varies; define whether X-only or all locks are held to end” |
| “Rigorous 2PL fixes deadlock” | “It fixes dirty dependency/release simplicity, not circular wait” |
| “Wait-die means T1 waits” | “Only if T1 is the older requester and the holder is younger” |
| “Wound-wait means T2 aborts” | “Only if older T1 requests a lock held by younger T2” |
| “Timestamp prevention is timestamp ordering” | “The timestamp chooses wait/abort while locks still control access” |
| “Retry the failed statement” | “Normally roll back and retry the entire transaction with fresh reads” |
| “Client timeout means rollback” | “Client timeout can leave an unknown commit outcome” |
| “Every SELECT takes S until commit” | “MVCC engines may serve ordinary reads from snapshots; check engine, isolation, and statement” |
| “Serializable means rigorous 2PL” | “Serializable is a behavior guarantee; engines implement it in different ways” |
| “Row lock protects a range predicate” | “Phantom prevention may need key-range/predicate protection or another serializable mechanism” |
| “Use global lock to eliminate all deadlocks” | “It serializes unrelated work; compare operational cost with safe victim retry” |

## 44. Edge cases an interviewer may inject

### 44.1 Two transactions acquire the same rows but in different index orders

Even SQL statements that look symmetrical may visit records in different plans/orders. Use appropriate indexes, explicit canonical ordering where the engine honors it, small transactions, and deadlock retry. Do not promise that application source-line order reveals every internal lock order.

### 44.2 A deadlock victim already sent an email

The email is outside database rollback. Avoid external side effects inside the open transaction. Use an outbox committed with database state, then send asynchronously and idempotently.

### 44.3 Savepoint rollback releases some locks

Vendor rules differ. PostgreSQL documents that locks acquired after a savepoint are released when rolling back to it. Textbook “all locks to transaction end” summaries do not substitute for product-specific savepoint behavior.

### 44.4 A transaction participates in two cycles

Choosing it may break both and unblock more work:

~~~text
Cycle 1: T1 -> T2 -> T1
Cycle 2: T1 -> T3 -> T4 -> T1
~~~

But rollback cost and priority still matter; maximizing cycles broken is not the sole objective.

### 44.5 Deadlock detector becomes the bottleneck

At extreme contention, many threads may wait on the same item and graph checks themselves consume significant coordination. MySQL documents that disabling immediate deadlock detection and relying on lock-wait timeout can sometimes be more efficient in high-concurrency hotspot cases—but that is a product/workload-specific option, not a universal recommendation ([MySQL deadlock detection](https://dev.mysql.com/doc/refman/8.4/en/innodb-deadlock-detection.html)).

### 44.6 The victim is always the same transaction type

This may be starvation. Track retry count and victim history, use aging/priority carefully, shorten or split the expensive transaction without breaking atomicity, and fix inconsistent lock ordering.

### 44.7 A queue has no cycle but nothing useful completes

Look for:

- a stalled/crashed lock owner;
- a transaction waiting on user/remote input;
- connection-pool exhaustion;
- starvation from queue policy;
- an app-level lock outside the database graph;
- a client that abandoned a session without cleanup.

WFG cycle detection solves only circular lock waits.

### 44.8 A transaction does not know all rows in advance

Conservative 2PL is a poor fit. Alternatives include canonical monotonic discovery, predicate/range locking, OCC/serializable validation, one invariant-owning summary row, or redesigning the operation around a short atomic transition.

### 44.9 The same logical request retries after deadlock

Use two layers correctly:

- transaction retry re-executes database logic after a concurrency abort;
- idempotency identity prevents the logical business effect from occurring twice across ambiguous client/network retries.

One does not replace the other.

# Part XI — Decision and review sheets

## 45. Deadlock strategy map

~~~text
Can every transaction acquire resources in one global order?
  |
  +-- yes -> enforce order; still implement defensive retry
  |
  +-- no
       |
       +-- complete lock set known cheaply in advance?
       |      |
       |      +-- yes -> consider conservative/static preclaiming
       |      +-- no
       |
       +-- aborts acceptable and ages available?
       |      |
       |      +-- yes -> consider wait-die or wound-wait
       |      +-- no/engine already detects
       |
       +-- WFG detection + victim rollback + bounded retry

Timeout may bound all paths, but is not exact deadlock detection.
~~~

In an application using a managed relational database, the normal answer is often:

1. keep transactions short;
2. access objects consistently;
3. use good indexes;
4. let the engine detect unavoidable deadlocks;
5. retry the complete transaction safely;
6. investigate if frequency becomes operationally significant.

You usually do not implement wait-die yourself on top of arbitrary SQL unless you own the transaction/lock scheduler.

## 46. Variant selection map

~~~text
Need conflict serializability with locks?
  -> basic 2PL rule

Need deadlock freedom and know all locks first?
  -> conservative/static preclaiming

Need no dirty reads/writes and simple undo?
  -> strict: retain X locks to end

Need the simplest all-locks-at-end discipline / commit-aligned ordering?
  -> rigorous/strong-strict: retain S and X to end

Need high read concurrency?
  -> examine MVCC/optimistic/SSI alternatives and actual DB semantics
~~~

These choices can be combined; they are not a single mutually exclusive ladder.

## 47. Production checklist

Before approving a lock-based design, answer:

### Correctness

- What exact invariant is protected?
- Which component owns it?
- Are the lock items rows, keys, ranges, tables, or business resources?
- Are phantoms/absent rows part of the invariant?
- Which lock modes conflict?
- Is the result actually serializable/strict enough for the business?

### Duration and ordering

- When is each lock acquired and released?
- Is the transaction short and free of remote/user waits?
- Is there a canonical acquisition order?
- Can lock upgrades be avoided?
- Is the full lock set predictable, and would preclaiming help?

### Failure

- How is deadlock detected or prevented?
- How is the victim selected?
- Does the database roll back the statement or whole transaction on each error?
- Does retry re-read all state from a clean transaction?
- Is retry bounded, jittered, and deadline-aware?
- Can a client timeout leave an unknown committed outcome?

### Operations

- Which queries/resources are hottest?
- Are indexes minimizing scan and lock footprint?
- What are lock-wait, deadlock, retry, and rollback metrics?
- Can support identify blocker, waiter, resource, and transaction age?
- Is there an outbox/idempotency strategy for effects outside the database?

## 48. Compact cheat sheet

~~~text
2PL
  not “two locks”; not 2PC
  GROW: acquire/upgrade, no release
  SHRINK: release/downgrade, no acquire
  lock point = final acquisition
  guarantees conflict serializability

BASIC
  deadlock possible
  early X release possible -> cascading abort possible

CONSERVATIVE / STATIC
  preclaim complete lock set, hold none while waiting for set
  deadlock-free
  needs future set; reserves early; lower concurrency

STRICT
  hold X locks until commit/abort
  strict + cascadeless; simpler recovery
  S may release after lock point

RIGOROUS / STRONG STRICT
  hold S and X until commit/abort
  simpler all-at-end release
  less concurrency than strict; still can deadlock

WFG
  waiter -> holder
  directed cycle = deadlock in standard lock model
  self-loop is not normal at transaction granularity

TIMEOUT
  simple bound, no graph
  may abort non-deadlocked waiter -> unnecessary/false-victim abort

VICTIM
  rollback/log cost + locks/waiters + cycles + priority/age + past aborts
  retry whole transaction from clean state

WAIT-DIE (non-preemptive)
  old asks young -> old waits
  young asks old -> young dies

WOUND-WAIT (preemptive)
  old asks young -> old wounds young
  young asks old -> young waits

Both keep original timestamp on retry and prevent wait cycles.
~~~

## 49. Say-out-loud 30-second narrative

> “Two-phase locking makes concurrent transactions conflict-serializable by separating lock growth from lock release: once a transaction releases its first lock, it cannot acquire another. Basic 2PL can still deadlock and can release written data before commit, causing cascading aborts. Conservative 2PL preclaims every lock to avoid deadlock; strict 2PL holds write locks to the end; rigorous, or strong-strict, holds both read and write locks to the end. For deadlocks I use consistent ordering or prevention, or detect wait-for-graph cycles, roll back a cheap fair victim, and retry the whole short transaction safely.”
