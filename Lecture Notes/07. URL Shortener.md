# URL Shortener

> Beginner-first HLD study notes for designing a TinyURL/Bitly-style service. The central lesson is not “use Base62.” It is how to turn a small key-value mapping into a low-latency, highly available, abuse-resistant system—and how to choose an ID-generation method whose failure modes match the requirements.

## Scope, assumptions, and evidence labels

The supplied rough bullets were:

- why a hash is not used;
- why Base62 is used;
- counter approaches: an incremental database counter, ticket server, Snowflake, and ZooKeeper; and
- whether a Redis Cluster can provide the counter.

This chapter expands those bullets into the complete URL-shortener design because the ID generator only makes sense inside the create and redirect flows.

### The most important correction up front

**Base62 and a counter are not alternatives.** They solve different stages:

```text
ID generation                 Text encoding                 Persistence
--------------                -------------                 -----------
counter / Snowflake /  -----> integer or random bits -----> short code -----> code -> long URL
random generator                e.g. 12345                    e.g. 3d7           mapping
```

- A **counter, ticket server, Snowflake generator, or random generator** creates a unique candidate ID.
- **Base62** represents a non-negative integer using 62 URL-friendly characters.
- A **database uniqueness constraint/conditional insert** is the final authority that prevents two records from owning the same public code.

Saying “I use Base62 to guarantee uniqueness” is incomplete. Base62 preserves uniqueness only if the input integers are already unique and the encoding is implemented canonically.

### Evidence labels used in this chapter

- **Verified technical source**: an RFC, official project documentation, original code, or first-party engineering post linked nearby.
- **Public product behavior**: behavior documented by the product owner. It is not a claim about the product's private backend architecture.
- **Public production snapshot**: an architecture described publicly at a particular date; it may not describe the system today.
- **Self-reported interview experience**: a candidate's public report. It shows question style, not a verified company question bank.
- **Illustrative design**: a reasonable design using a familiar product. It is not a claim that the real company implements it that way.

## 1. Plain-language foundation

### 1.1 What is a URL shortener?

Suppose this is difficult to print, dictate, or fit into a message:

```text
https://shop.example.com/products/123?campaign=summer&source=newsletter
```

A URL-shortening service gives it a compact alias:

```text
https://sho.rt/3d7KaQ
```

When someone opens the short URL, the service looks up the alias and tells the browser to visit the original URL.

```text
Creator                         URL-shortener service
   |  long URL                         |
   +---------------------------------->|
   |  https://sho.rt/3d7KaQ            |
   |<----------------------------------+

Visitor                         URL-shortener service                    Destination
   |  GET /3d7KaQ                      |                                      |
   +---------------------------------->|                                      |
   |                                  | look up 3d7KaQ -> long URL            |
   |  302 Location: https://shop...   |                                      |
   |<----------------------------------+                                      |
   |  GET https://shop...                                                    |
   +------------------------------------------------------------------------>|
```

The shortener normally does **not** download and resend the destination page. It returns an HTTP redirect response containing a `Location` header, and the visitor's browser contacts the destination.

### 1.2 The service is fundamentally a key-value lookup

At its core, the stored record is:

```text
short code  ->  destination URL
3d7KaQ      ->  https://shop.example.com/products/123?campaign=summer
```

This looks easy, and at small scale it is. The HLD challenge comes from the surrounding questions:

- How is every code made unique across many create servers?
- Can redirects remain fast when reads greatly outnumber writes?
- What happens immediately after creating a link if replicas lag?
- Can a popular link overload one cache or partition?
- Can users change or delete links that browsers have cached permanently?
- How are phishing, malware, enumeration, and spam handled?
- Do old links have to work for years after product and storage migrations?

### 1.3 Inline prerequisite: URL, URI, domain, path, and query

These terms are required for the rest of the design:

```text
https://sho.rt/3d7KaQ?preview=true
\___/   \____/ \____/ \__________/
scheme   host    path      query
```

- A **URL** identifies where/how to access a resource. In interviews, “URL” is the normal term for the web address being shortened.
- The **domain/host** such as `sho.rt` is resolved through DNS to the service's network endpoint.
- The **path segment** such as `3d7KaQ` carries the short code.
- The **query string** begins after `?` and may carry options. Do not accidentally treat it as part of the code.
- A **fragment** begins after `#`. Browsers normally do not send the fragment to the web server; if it is part of a stored destination, it is returned in the `Location` value and then interpreted by the destination client.

[RFC 3986](https://www.rfc-editor.org/rfc/rfc3986.html) defines uppercase letters, lowercase letters, digits, `-`, `.`, `_`, and `~` as unreserved URI characters. Base62 chooses only the alphanumeric subset.

### 1.4 Inline prerequisite: an identifier is not its encoding

The number twelve can be written in several ways:

```text
decimal: 12
binary:  1100
hex:     c
```

The value did not change; only its textual representation did. Similarly:

```text
unique integer ID 12345 --Base62 encode--> "3d7"
```

Using the alphabet `0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ`, `3d7` decodes as:

```text
3 * 62^2 + 13 * 62 + 7 = 12345
```

The alphabet order is a contract. Changing it later would make old codes decode to different values.

### 1.5 Inline prerequisite: HTTP redirects

For a browser to follow a short link, the shortener returns a `3xx` response:

```http
HTTP/1.1 302 Found
Location: https://shop.example.com/products/123
Cache-Control: no-store
```

The most relevant choices are:

| Status | Meaning | Important behavior for a shortener |
|---|---|---|
| `301 Moved Permanently` | The alias permanently points elsewhere | Heuristically cacheable; a client/CDN may bypass the shortener later |
| `302 Found` | The destination is temporary/changeable | Common default for mutable links and click visibility; cache explicitly as desired |
| `307 Temporary Redirect` | Temporary and method-preserving | Safer if requests other than `GET` are intentionally supported |
| `308 Permanent Redirect` | Permanent and method-preserving | Same permanence warning as `301` |

[RFC 9110](https://www.rfc-editor.org/rfc/rfc9110.html#name-redirection-3xx) says `301` and `308` are permanent and heuristically cacheable, while `307` and `308` unambiguously preserve the request method. For an ordinary public link opened with `GET`, method preservation is rarely the deciding issue.

**Practical default:** use `302` for links that can be edited, disabled, expired, analyzed on every hit, or safety-checked at click time. Use `301`/`308` only when the mapping is truly immutable and bypassing future service logic is acceptable. Explicit `Cache-Control` policy still matters.

### 1.6 What “short” really means

The code is only one part of the address:

```text
https://sho.rt/3d7KaQ
\_____________/\____/
 domain overhead code
```

A shorter domain may save more characters than changing a seven-character code to six. HTTPS is still required; removing it from a displayed link does not remove the protocol from actual navigation.

### 1.7 Not every compact link is a generic URL shortener

There are at least three related patterns:

1. **Generic alias:** random/allocated code maps to any permitted external URL.
2. **Compact resource route:** the path directly contains an existing resource ID, such as a video ID.
3. **Deep/universal link:** a URL carries an application action and opens a native app or falls back to the web.

They can look similar to a user but have different storage, security, and routing needs.

## 2. Why URL shortening is needed—and why it is an HLD exercise

### 2.1 Product reasons

A short link can provide:

- compact sharing in text messages, printed material, slides, and QR codes;
- a stable public alias while the destination changes;
- branded domains and custom aliases;
- click analytics and campaign attribution;
- expiry, disablement, access policy, or one-time use;
- a safety checkpoint before navigation; and
- app deep-linking with a browser fallback.

It also introduces a serious dependency: if the shortening service or its domain disappears, every dependent link may break. Google's 2025 update preserving actively used `goo.gl` links illustrates how short links can remain embedded in documents, videos, and posts long after link creation ([Google Developers Blog](https://developers.googleblog.com/google-url-shortener-links-will-no-longer-be-available/)).

### 2.2 Why interviewers like the problem

The core mapping is simple enough to explain quickly, leaving time to test whether the candidate can reason about:

- requirements and back-of-the-envelope estimates;
- unique distributed ID generation;
- encoding versus hashing;
- database indexes and partitioning;
- read-heavy caching and hot keys;
- asynchronous analytics;
- consistency during create, edit, and delete;
- availability and dependency isolation;
- HTTP semantics; and
- abuse/security.

The interview is not really testing whether you remember `62^7`. It is testing whether each architectural component follows from a requirement and whether you can explain what breaks.

## 3. Requirements before architecture

### 3.1 Questions to clarify with the interviewer

Ask enough to define the design, without spending the whole interview gathering requirements:

- Can anyone create a link, or only authenticated/rate-limited users?
- May one long URL have multiple short links for different owners or campaigns?
- Are custom aliases required?
- Can destinations be edited after creation?
- Do links expire? Can they be disabled or deleted?
- Must links be unguessable/private, or merely unique?
- Are click analytics in scope? How fresh and exact must they be?
- What are the expected creation and redirect rates, traffic peaks, and retention period?
- Is the service global? Can links be created in several regions simultaneously?
- What availability and latency matter most: create, redirect, or both?

### 3.2 Functional requirements selected for this design

Core:

1. Create a short link for a valid `http` or `https` destination.
2. Redirect a visitor from the short link to its destination.
3. Guarantee that one public code never refers to two link records.

Included extensions:

- optional custom aliases;
- optional expiration and disable/delete;
- idempotent create requests; and
- asynchronous click analytics.

Out of the initial critical path:

- a full analytics query engine/dashboard;
- paid plans and billing;
- branded customer domains;
- password-protected and one-time links; and
- content-preview rendering.

These can be added after the core design remains correct.

### 3.3 Non-functional priorities

| Priority | Why |
|---|---|
| Low redirect latency | A redirect adds latency before the destination page even starts loading |
| High redirect availability | One failed shortener can break many otherwise healthy destinations |
| Durable, correct mappings | A wrong redirect is worse than a temporary error |
| Read scalability | Mature links are normally read much more often than links are created |
| Read-after-create behavior | Creators often click or share a link immediately |
| Abuse resistance | The trusted short domain can hide phishing/malware destinations |
| Cost efficiency | Many mappings are cold; a small hot subset drives most reads |

### 3.4 Product decisions that change the architecture

#### Same long URL: same code or a new code?

There is no universal answer.

- **Always create a new code:** simplest semantics for ownership, campaigns, expiry, and analytics.
- **Deduplicate within one account and policy:** useful if the same idempotent request is repeated.
- **Globally deduplicate:** saves records but leaks whether another user created a URL and makes ownership/expiry/analytics ambiguous.

This chapter uses **a new link record per intentional create**, while an `Idempotency-Key` prevents accidental duplicates caused by retries.

#### Mutable or immutable destination?

- Immutable mappings simplify caching and auditability.
- Mutable mappings enable campaign management but require cache invalidation/versioning and make permanent HTTP redirects unsafe.

This design allows edits for owned links and therefore defaults to `302`.

#### Is the code secret?

A code can be unique without being unguessable. Sequential Base62 codes such as neighboring counter values are enumerable. If possession of the link grants access to private data, use sufficient cryptographic randomness **and** treat the destination's authorization as the real security boundary. “Nobody will guess the URL” is not a complete access-control design.

## 4. Back-of-the-envelope estimation

The following numbers are **illustrative design assumptions, not public Bitly/TinyURL statistics**:

```text
new links/day                 = 10 million
redirect:create ratio         = 100:1
peak:average traffic ratio    = 10:1
average stored record         = 500 bytes including rough metadata/index allowance
retention                     = long-lived; do not reuse expired codes
```

### 4.1 Request rates

```text
create average = 10,000,000 / 86,400 ~= 116 requests/s
redirect average = 116 * 100 ~= 11,600 requests/s
redirect peak ~= 116,000 requests/s under the 10x assumption
```

The exact numbers are less important than the consequences:

- creation traffic is modest enough that a transactional database counter may initially be fine;
- the redirect path deserves separate scaling and aggressive caching; and
- analytics cannot synchronously update one database row for every redirect at peak.

### 4.2 Storage

```text
10,000,000 records/day * 500 bytes ~= 5 GB/day
5 GB/day * 365 ~= 1.83 TB/year raw
```

Replication, database page overhead, secondary indexes, backups, and analytics events can multiply this. State assumptions instead of presenting `1.83 TB` as a production forecast.

### 4.3 Code-space capacity

For an alphabet of size `A` and fixed code length `L`:

```text
capacity = A^L
```

With Base62:

| Length | `62^L` possible strings |
|---:|---:|
| 5 | 916,132,832 |
| 6 | 56,800,235,584 |
| 7 | 3,521,614,606,208 |
| 8 | 218,340,105,584,896 |
| 10 | 839,299,365,868,340,224 |
| 11 | 52,036,560,683,837,093,888 |

At the assumed 10 million new links/day, the theoretical seven-character space lasts about:

```text
3.5216 trillion / 10 million per day ~= 352,161 days ~= 964 years
```

That is only a **space calculation**, not an exhaustion promise. Reserve system routes and offensive words, allow safety margin, account for custom aliases and never-reused tombstones, and remember that randomly chosen codes collide before the entire space is occupied.

### 4.4 Average is not peak and keys are not equally popular

One celebrity post can send far more traffic to a single code than the average link receives. Hash partitioning spreads different codes, but it does not split one hot code. The design needs local/CDN caching, request coalescing, replicated cache entries, and origin protection for hot links.

## 5. API and data model

### 5.1 Create API

```http
POST /v1/links
Authorization: Bearer ...
Idempotency-Key: 624b2e...
Content-Type: application/json

{
  "destination_url": "https://shop.example.com/products/123?campaign=summer",
  "custom_alias": null,
  "expires_at": "2027-01-01T00:00:00Z"
}
```

```http
HTTP/1.1 201 Created
Content-Type: application/json

{
  "code": "3d7KaQ",
  "short_url": "https://sho.rt/3d7KaQ",
  "destination_url": "https://shop.example.com/products/123?campaign=summer",
  "expires_at": "2027-01-01T00:00:00Z"
}
```

Possible responses:

- `400` malformed/unsupported destination;
- `409` requested custom alias already exists;
- `413` destination exceeds the accepted size;
- `422` syntactically valid but policy-rejected URL;
- `429` rate limit exceeded; and
- `503` link creation temporarily unavailable.

Why `POST`? Link creation allocates a new server-side resource and is not naturally idempotent. The idempotency key makes safe client retries possible: `(creator_id, idempotency_key)` maps to the already-created result.

### 5.2 Redirect API

```http
GET /3d7KaQ
```

Success:

```http
HTTP/1.1 302 Found
Location: https://shop.example.com/products/123?campaign=summer
Cache-Control: private, no-store
```

The cache policy above prioritizes mutable mappings and server-visible clicks. For immutable links, the service can deliberately send shared-cache directives instead.

Failure semantics should be explicit:

| Condition | Possible response | Note |
|---|---:|---|
| Never existed or intentionally hidden | `404` | Avoid leaking moderation state if necessary |
| Permanently expired/deleted | `410 Gone` | Useful when permanent absence can be disclosed |
| Disabled for abuse | `403`, warning page, or `404` | Product/safety decision |
| Temporary dependency failure | `503` | Never redirect to an uncertain destination |

### 5.3 Separate API and redirect hosts

Using `api.example.com/v1/links` for management and `sho.rt/{code}` for redirects avoids collisions with reserved paths such as `/admin`, `/health`, and `/v1`. If one domain must do both, maintain a reserved-code namespace.

### 5.4 Link record

```text
Link
----
code                 string, primary/unique public lookup key
internal_id          64-bit integer or UUID, optional unique internal identity
destination_url      bounded string/blob
creator_id           nullable account ID
created_at           timestamp
expires_at           nullable timestamp
status               ACTIVE | DISABLED | EXPIRED | DELETED
redirect_type        302 by default; tightly controlled
version              integer for optimistic update/cache invalidation
destination_hash     optional, only for scoped dedup/search
```

The critical query is a point lookup:

```sql
SELECT destination_url, status, expires_at, redirect_type, version
FROM links
WHERE code = ?;
```

That makes a key-value store a natural fit, but a relational database is also completely reasonable—especially initially. “NoSQL because scale” is not a justification. Compare concrete needs: conditional unique insertion, indexes, transactions for owner updates, replication, operational skill, and cost.

### 5.5 Analytics data must be separate

Do not update `click_count = click_count + 1` on the link row for every redirect. That turns every read into a contended write and couples user navigation to analytics availability.

```text
redirect service --async event--> durable stream/queue --> processors --> analytics store
       |
       +--return redirect without waiting for dashboard aggregation
```

An event might contain `event_id`, `code`, time bucket, coarse region, referrer category, user-agent category, and privacy/consent fields. Expect at-least-once delivery to produce duplicates; deduplicate by event ID or accept documented approximate counts.

### 5.6 Never recycle a public code to a different destination

Reusing an expired/deleted code can send an old email, QR code, or browser history entry to an unrelated new owner. Keep a tombstone or an allocation high-water mark so public identifiers are never reassigned. This also means deletion for privacy may remove the destination/owner data while retaining a minimal non-reusable code tombstone.

## 6. Why Base62 is used

### 6.1 It packs an integer into fewer characters

Decimal has ten symbols. Base62 has 62:

```text
0-9  -> 10
a-z  -> 26
A-Z  -> 26
total = 62
```

Each Base62 character carries `log2(62) ~= 5.95` bits of choice, compared with about 3.32 bits for decimal and 4 bits for hexadecimal. A larger safe alphabet gives a shorter representation for the same integer space.

### 6.2 The characters are convenient in URL path segments

Alphanumeric characters are unreserved under RFC 3986, so the code does not need percent-encoding and avoids delimiters such as `/`, `?`, `#`, and `&` that have URL syntax meanings ([RFC 3986, section 2.3](https://www.rfc-editor.org/rfc/rfc3986.html#section-2.3)).

### 6.3 Encoding algorithm

To encode a non-negative integer:

```text
alphabet = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

while number > 0:
    remainder = number % 62
    prepend alphabet[remainder]
    number = number / 62 using integer division
```

To decode, scan left to right:

```text
value = 0
for each character c:
    value = value * 62 + alphabet_index(c)
```

Handle `0` explicitly, reject characters outside the alphabet, detect numeric overflow, and define whether leading zero characters are canonical.

### 6.4 Base62 does not provide these properties

- It does **not** generate uniqueness.
- It does **not** compress and later reconstruct an arbitrary long URL.
- It does **not** encrypt or hide a sequential ID.
- It does **not** prevent someone from enumerating adjacent IDs.
- It does **not** detect a mistyped code.

### 6.5 Operational pitfalls

#### Case sensitivity

`aB3` and `Ab3` are different codes. Therefore:

- the database column/index must use binary or case-sensitive comparison;
- application routers and caches must preserve path case;
- log pipelines must not lowercase the path; and
- support teams must know case matters.

DNS hostnames are case-insensitive; URL paths generally are not. Do not transfer hostname rules to the code.

#### Human ambiguity

`0/O` and `I/l/1` are easy to confuse in print or speech. Alternatives include:

| Alphabet | Advantage | Cost |
|---|---|---|
| Base36 (`0-9a-z`) | Case-insensitive operationally | Fewer codes per character |
| Base58 | Removes commonly confused characters | Smaller space; alphabet conventions vary |
| Base64url | Slightly denser using `-` and `_` | More punctuation and ecosystem edge cases |
| Word-based slugs | Easier to say | Much longer; moderation/language complexity |

Base62 is a common compromise, not a law.

#### Variable length and leading zeros

A direct counter begins with very short codes and grows at powers of 62. If fixed length is required, left-pad with the zero symbol or start from a chosen offset—but specify canonicalization so both `3d7` and `0003d7` do not accidentally become separate aliases.

#### Offensive/reserved strings

Some random or encoded strings will resemble words. Maintain a denylist for generated/custom aliases and retry allocation. Also reserve application paths before public launch. This reduces usable capacity slightly.

### 6.6 Sequential Base62 is predictable

If IDs `12345`, `12346`, and `12347` are encoded directly, their codes are related. Attackers can crawl neighboring records, infer creation volume, find unlisted campaigns, and impose read load.

Options:

- accept enumeration for fully public links and rate-limit scanning;
- generate sufficiently long cryptographically random codes;
- apply a vetted keyed permutation/obfuscation before Base62 encoding; or
- use a hash/HMAC plus collision handling.

Simple “scrambling” is not authorization or cryptographic secrecy. If unpredictability is a requirement, say so and budget entropy explicitly.

## 7. Why a hash of the long URL is not automatically the answer

### 7.1 The tempting design

```text
long URL --hash--> 256-bit digest --take first 7 chars--> short code
```

It looks attractive because every create server can calculate a result independently. But several different issues are hidden in “take the first seven characters.”

### 7.2 A full cryptographic hash is not short

A SHA-256 digest contains 256 bits. Representing all of it in Base62 takes up to about 43 characters. Keeping the full digest gives a large code; truncating it gives a small finite namespace in which collisions must be handled.

This is not a defect in SHA-256. It is the unavoidable result of mapping many possible inputs into fewer public codes.

### 7.3 Truncation creates collisions

If there are `M = 62^7 ~= 3.52 trillion` seven-character codes and `n` uniformly distributed stored hashes, the expected number of colliding pairs is approximately:

```text
expected collision pairs ~= n(n - 1) / (2M)
```

At `n = 1 billion`:

```text
expected pairs ~= 141,980
probability of at least one collision ~= effectively 100%
```

This does **not** mean a seven-character space is almost full—occupancy is only about `0.028%`. It means that with so many pairs of inputs, at least one pair matching is virtually certain. This is the birthday effect.

A unique database constraint can detect a collision, but a deterministic hash of only the long URL returns the same colliding candidate on every retry. The service needs a nonce/salt, probing rule, secondary suffix, or another ID.

### 7.4 “Any collision” and “collision cost” are different

For a random-code generator that chooses a new candidate after a conflict, `n = 1 billion` insertions into a seven-character space produce about 142,000 expected collision attempts in total under the idealized model—roughly `0.014%` extra attempts. A collision is certain to happen somewhere, but conditional insert-and-retry can make it operationally routine.

For deterministic truncated hashing, retrying without changing the input cannot escape the collision. This distinction is frequently missed in interviews.

### 7.5 The same URL does not always mean the same link record

Two customers may shorten the same destination but need different:

- owners and permissions;
- campaign tags and analytics;
- expiry times;
- custom domains;
- redirect behavior; or
- disable/edit histories.

Hashing only the destination forces these distinct resources toward one code. Adding owner, policy, and a nonce to the hash can solve that—but now the design is essentially generating a unique/random ID with extra steps.

### 7.6 URL equivalence is harder than string equality

These strings may or may not refer to equivalent application behavior:

```text
https://EXAMPLE.com:443/a
https://example.com/a
https://example.com/a?
https://example.com/a?x=1&y=2
https://example.com/a?y=2&x=1
```

Host casing and the default HTTPS port can often be normalized, but query parameter order may matter to a destination application. Tracking parameters may look removable but be commercially meaningful. Over-aggressive normalization can merge different destinations; no normalization misses deduplication.

If scoped deduplication is required:

1. define a conservative canonicalization contract;
2. hash the canonical bytes plus the intended deduplication scope;
3. index the full digest for candidate lookup; and
4. verify the full canonical URL before declaring equality, because hashes can collide.

### 7.7 Hashing can still be a valid design

It is wrong to say “hashes cannot be used.” Valid patterns include:

- hash/HMAC `(destination, owner, nonce)`, truncate, conditionally insert, and change the nonce on conflict;
- generate a random code directly, which often achieves the same distribution more simply;
- store a full destination hash as a secondary index for scoped deduplication; or
- use a keyed permutation/HMAC-derived value to make allocated IDs less enumerable, with collision resolution if truncated.

The precise answer is:

> A naive truncated hash of only the long URL is not my default because it has unavoidable collisions, ambiguous deduplication semantics, and no natural retry candidate. I can use hashing if I add scope, a nonce, a uniqueness check, and collision resolution.

### 7.8 Hashing and Base62 are not mutually exclusive

Base62 can encode:

- a counter value;
- a Snowflake integer;
- random bits interpreted as an integer; or
- a hash/HMAC value.

Hashing decides how a value is derived; Base62 decides how a value is written.

## 8. What the ID generator must guarantee

Before naming a technology, define the properties:

| Property | Required? | Why |
|---|---|---|
| Global uniqueness | Yes | Two mappings cannot share a public code |
| Compact numeric range | Desirable | Directly affects Base62 length |
| High creation throughput | Workload-dependent | Redirect throughput does not stress the generator |
| High create availability | Product-dependent | Existing redirects can remain available if create pauses |
| Strict global ordering | Usually no | A shortener needs uniqueness, not gapless chronological numbering |
| Rough time ordering | Optional | Helps storage locality/debugging but can reveal creation time |
| Unpredictability | Requirement-dependent | Public links and capability links have different threat models |
| No gaps | Usually no | Gaps are harmless; demanding gaplessness greatly increases coordination |

**Never promise gapless IDs.** Rollbacks, retries, preallocated ranges, process crashes, and rejected URLs burn values. Uniqueness matters; consecutive numbering usually does not.

## 9. ID-generation approaches in depth

### 9.1 Approach A: database auto-increment or atomic counter

#### Plain-language idea

One database safely hands out `1, 2, 3, ...`. The application Base62-encodes each integer.

```text
Create service ---> database INSERT/sequence ---> ID 12345 ---> Base62 "3d7"
```

The simplest version uses the mapping database itself:

- With an explicit database sequence, fetch `nextval`, Base62-encode it, and insert the ID, code, and destination in one transaction.
- With an `AUTO_INCREMENT` column, insert the row, obtain the generated ID, derive the code, and store it before commit—or decode the public code back to the internal ID on reads. The first form makes custom and generated aliases share one lookup path.

#### Strengths

- simplest design and operational model;
- uniqueness enforced by a mature transactional database;
- compact IDs with no probabilistic collisions;
- no separate ID service; and
- excellent starting point at the illustrative `~116` average creates/s.

#### Weaknesses

- all allocation reaches one logical writer/sequence;
- primary failure can pause creation;
- sharded databases have independent local counters unless coordinated;
- sequential IDs are enumerable; and
- cross-region synchronous access adds latency.

#### Multiple database shards

Local `AUTO_INCREMENT` on every shard can generate duplicates. Possible fixes:

- fixed odd/even or `shard_id + k * shard_count` allocation;
- embed a shard/region ID in the number;
- allocate non-overlapping ranges to shards; or
- use a separate global generator.

Odd/even is easy for two allocators but becomes awkward as membership changes. Range allocation is more flexible.

#### Failure and retry nuance

If a client times out, the insert may have committed. Retrying without an idempotency key can create a second link. The idempotency record must be committed atomically with the link result or reconstructed reliably from the same transaction.

Database sequences may advance even when transactions roll back. That produces gaps, not duplicates, and is normally correct.

#### When to choose it

Use it first when creation load is moderate, one write region is acceptable, and operational simplicity matters more than independent multi-region creation.

### 9.2 Approach B: ticket server

#### Plain-language idea

A dedicated service does only one job: hand out globally unique integers. Mapping databases can then be sharded independently.

```text
Create server A --+
Create server B --+--> Ticket service --> 900001, 900002, 900003 ...
Create server C --+
```

This is a central allocator logically, even if it has replicas or several nodes.

#### Public production snapshot: Flickr

Flickr described a MySQL-backed ticket-server design in 2010. A one-row table used `REPLACE INTO` to advance an auto-increment ID. Two ticket servers independently generated odd and even IDs using increment `2` and offsets `1`/`2`, avoiding synchronous replication/locking between the pair ([Flickr engineering post](https://code.flickr.net/2010/02/08/ticket-servers-distributed-unique-primary-keys-on-the-cheap/)). This is a dated production example, not a statement about Flickr today.

#### Strengths

- mapping storage shards do not coordinate ID creation;
- compact, sortable numeric IDs;
- simple client contract; and
- easier to audit than probabilistic generation.

#### Weaknesses

- per-ID network hop and centralized throughput path;
- allocator outage pauses all new links;
- replication/failover must not move the counter backward and reissue IDs; and
- multiple active allocators require disjoint spaces or consensus.

#### Ticket service versus “single point of failure”

A logical central point need not be one machine. It can use:

- a consensus-backed primary with failover;
- multiple servers owning disjoint arithmetic sequences;
- non-overlapping ranges; or
- regional prefixes.

But every mitigation changes guarantees. Two asynchronous replicas that both believe they are primary can issue duplicates unless fenced or assigned disjoint spaces.

### 9.3 Approach C: range allocation / hi-lo tickets

This is often the best refinement of a central counter.

Instead of asking for one ID per link, a create server atomically reserves a block:

```text
Allocator durable counter = 1,000,000

Worker A: INCREMENT BY 10,000 -> owns [1,000,000, 1,009,999]
Worker B: INCREMENT BY 10,000 -> owns [1,010,000, 1,019,999]

Workers now allocate locally until their blocks are exhausted.
```

#### Benefits

- one allocator write per block rather than per link;
- create continues during a short allocator outage while workers have unused tickets;
- compact global numeric space; and
- no wall-clock dependency.

#### Costs

- worker failure wastes the unused portion of a range;
- large ranges improve throughput/availability but waste more IDs and can reduce observed ordering;
- the durable allocator must never roll its high-water mark backward; and
- range ownership needs an epoch/lease record for audit and recovery.

Do **not** reclaim an uncertain partially used range. Burning IDs is safer than issuing duplicates.

### 9.4 Approach D: Snowflake-style IDs

#### Plain-language idea

Each worker constructs an integer from time, its unique worker identity, and a per-time-unit sequence. Workers can generate independently without a central call for every ID.

The archived original Twitter implementation used:

```text
| unused sign | timestamp delta | datacenter | worker | sequence |
|    1 bit    |     41 bits     |   5 bits  | 5 bits| 12 bits  |
```

The code shows five datacenter bits, five worker bits, twelve sequence bits, a custom epoch, waiting after the sequence wraps, and rejection when the clock moves backward ([archived `IdWorker.scala`](https://github.com/twitter-archive/snowflake/blob/b3f6a3c6ca/src/main/scala/com/twitter/service/snowflake/IdWorker.scala)). X/Twitter announced and open-sourced Snowflake in 2010 ([announcement](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake)).

From that specific layout:

- `2^10 = 1,024` worker/datacenter combinations;
- `2^12 = 4,096` IDs per millisecond per worker; and
- `2^41` milliseconds spans about 69.7 years from the chosen epoch.

These are **verified properties of the archived layout, not universal Snowflake rules**. A new system can choose different bit widths.

#### Strengths

- no network call on the per-ID hot path;
- high aggregate throughput across workers/regions;
- roughly time-sortable IDs; and
- fixed 64-bit-friendly storage in common variants.

#### Weaknesses and edge cases

1. **Worker identity must be unique.** Reusing a worker ID while the old process can still issue IDs creates collisions. Assign identities with configuration, orchestration, or ZooKeeper/etcd-style leases and fencing.
2. **Clocks move backward.** NTP correction, VM restore, or operator error can repeat `(timestamp, worker, sequence)`. The original code rejects until time catches up; alternatives persist last time or use a logical clock, each with costs.
3. **Sequence exhaustion.** More than the per-millisecond allowance makes that worker wait for the next millisecond.
4. **Not strictly globally ordered.** Two workers' clocks can differ. IDs are approximately chronological, not a global transaction order.
5. **Longer public code.** A general non-negative signed 63-bit value needs up to 11 Base62 characters because `62^10 < 2^63 <= 62^11`. A compact sequential counter can stay at seven characters for trillions of allocations.
6. **Information leakage.** Timestamp and worker bits can expose creation time and infrastructure patterns after decoding.

#### When to choose it

Use a Snowflake-style generator when independent, high-throughput, multi-node creation is worth longer codes and operational discipline around worker IDs and clocks.

### 9.5 Approach E: ZooKeeper counter or sequential znode

#### What ZooKeeper is for

ZooKeeper is a replicated coordination service used for naming, configuration, membership, synchronization, and leader election—not a general high-volume data store ([Apache ZooKeeper overview](https://zookeeper.apache.org/doc/current/index.html)). It can order writes and build atomic coordination recipes.

#### Can it generate IDs?

Yes, in several ways:

- maintain a znode value and use versioned compare-and-set;
- use Apache Curator's `DistributedAtomicLong`, which tries optimistic locking and optionally promotes to a mutex ([Curator recipe](https://curator.apache.org/docs/recipes-distributed-atomic-long/)); or
- create persistent sequential znodes and use the appended sequence number.

#### Why it is usually poor on the per-link path

- every allocation is a coordination write through the ensemble;
- one counter path becomes contended;
- retries and ambiguous outcomes must be handled;
- sequential znodes create metadata that must be cleaned up; and
- the sequence suffix is scoped to one parent and uses a signed 32-bit counter that eventually overflows in the documented implementation ([ZooKeeper programmer guide](https://zookeeper.apache.org/doc/r3.5.8/zookeeperProgrammers.html)).

ZooKeeper can be correct yet unnecessarily expensive and operationally dangerous for every link creation.

#### Better uses in this design

Use ZooKeeper to:

- assign/fence Snowflake worker IDs;
- elect a ticket allocator leader;
- publish allocator epochs;
- lease coarse ID ranges; or
- coordinate shard membership.

That keeps a coordination system on the control plane, not the high-QPS data path.

### 9.6 Approach F: Redis `INCR`—including Redis Cluster

#### Single Redis primary

Redis `INCR counter` atomically increments one integer key and returns the new value; the official documentation explicitly presents it as a counter pattern ([Redis `INCR`](https://redis.io/docs/latest/commands/incr/)). It is fast and simple.

The uniqueness question is not only atomicity while the primary is healthy. It is also durability across restart and failover:

```text
1. Redis issues ID 500.
2. Mapping database durably stores code for 500.
3. Redis primary fails before 500 reaches a durable/replicated copy.
4. New primary resumes from 497.
5. IDs 498, 499, 500 may be issued again.
```

A mapping-store unique constraint catches duplicates, but the allocator may collide repeatedly until it advances beyond the old high-water mark.

Redis replication is asynchronous by default. `WAIT` can request acknowledgements, but the official docs state that it does not turn Redis into a strongly consistent CP system and acknowledged writes can still be lost depending on persistence/failover configuration ([Redis replication](https://redis.io/docs/latest/operate/oss_and_stack/management/replication/)).

#### What changes in Redis Cluster?

**A single counter key belongs to one hash slot, and that slot belongs to one primary.** Therefore:

```text
INCR global_url_id
        |
 hash(global_url_id) -> slot 7421 -> Redis primary B
                                      ^
                          every global increment still arrives here
```

Redis Cluster provides sharding and failover for many different keys. It does **not** split one key's increments across primaries. Replicas do not normally add write throughput for that counter.

If each shard has its own counter:

```text
counter:A -> 1, 2, 3 ...
counter:B -> 1, 2, 3 ...
```

then the raw values are not globally unique. Add a shard/region prefix, use disjoint ranges, or interleave arithmetic sequences. At that point uniqueness comes from the allocation scheme—not from Redis Cluster providing a global counter.

The Redis Cluster specification also documents asynchronous replication and lack of strong consistency ([Redis Cluster specification](https://redis.io/docs/latest/operate/oss_and_stack/reference/cluster-spec/)). `WAIT` reduces risk; it does not erase every failover ambiguity.

#### Recommendation

- Redis is reasonable for approximate counters, rate limits, caches, and sometimes a best-effort ID allocator whose collisions are safely checked.
- For durable public-ID uniqueness, prefer the mapping database's own sequence, a durable/fenced range allocator, a Snowflake-style generator, or random candidates plus conditional insert.
- If Redis is used, enable appropriate persistence/replication, monitor the high-water mark, keep the database unique constraint, and design recovery from counter regression. Do not say “Redis is atomic, therefore IDs can never repeat.”

### 9.7 Approach G: cryptographically random code plus conditional insert

#### Plain-language idea

Each create server independently generates a random Base62 string:

```text
candidate = secure_random_base62(length=8 or more)
INSERT ... IF code DOES NOT EXIST
if conflict: generate another candidate and retry
```

#### Strengths

- no central allocation service;
- easy horizontal and multi-region generation if the store can arbitrate uniqueness;
- codes are not sequential; and
- code length directly expresses the desired entropy/capacity.

#### Weaknesses

- collisions are possible and must be handled;
- cross-region conditional uniqueness can still require coordination;
- a weak pseudo-random generator makes codes predictable;
- fixed-length random codes consume a sparse namespace; and
- strict ordering is absent.

At one billion total generated codes, the approximate probability of at least one collision is:

| Random code length | Space | Probability of at least one collision before retry |
|---:|---:|---:|
| 7 | `3.52 * 10^12` | effectively 100% |
| 10 | `8.39 * 10^17` | about 44.9% |
| 11 | `5.20 * 10^19` | about 0.96% |

Those figures describe the probability that **some** collision occurs among a billion draws. With conditional insert and redraw, an individual collision is not a correctness failure. Choose a longer code to make retries and enumeration suitably rare.

### 9.8 Comparison table

| Approach | Per-ID coordination | Compactness | Create availability/scale | Main failure risk | Best fit |
|---|---|---|---|---|---|
| DB auto-increment | One DB write/sequence | Excellent | Good until the logical writer is limiting | Primary outage/regression misconfiguration | Start simple, single write region |
| Ticket server | Network call to allocator | Excellent | Central logical path | Allocator availability/fencing | Sharded storage with modest creates |
| Range/hi-lo allocator | One call per block | Excellent | High; workers allocate locally | Reusing uncertain ranges | Compact IDs at higher scale |
| Snowflake style | None per ID | Up to 11 Base62 chars for common 63-bit form | Very high/distributed | Clock rollback or duplicate worker ID | Multi-region/high-throughput creation |
| ZooKeeper per ID | Ensemble write/CAS | Depends | Poor hot-counter scaling | Contention/operational misuse | Usually avoid per ID; use for coordination |
| Redis `INCR` | One Redis-primary write | Excellent | Fast, but one key is one shard | Counter rollback after failover | Best-effort or carefully bounded use |
| Random code + conditional insert | Store checks collision | Chosen entropy often means longer codes | Highly distributable | RNG/collision/store arbitration | Unpredictable multi-writer codes |

### 9.9 A sensible evolution path

```text
small/moderate scale
    DB sequence + Base62 + unique(code)
              |
              | allocator becomes measurable bottleneck
              v
    durable range allocator + local blocks
              |
              | independent multi-region creation is more important than shortest code
              v
    Snowflake-style or random codes + conditional insert
```

This is not a mandatory migration ladder. If unguessable links are a day-one requirement, start with long random codes. If only one region creates links, a database sequence may remain sufficient for years.

## 10. Chosen high-level design

For the illustrative requirements, choose:

- **database sequence initially**, evolvable to a **durable range allocator**;
- Base62 public codes, optionally passed through a vetted keyed permutation if enumeration reduction is required;
- a primary durable mapping store with hash partitioning by `code` when needed;
- cache-aside redirect reads;
- asynchronous analytics; and
- separate create and redirect services so generator failure cannot take down existing links.

The choice favors compact codes and simple correctness because the assumed create rate is far below redirect traffic. If active-active regional creation were mandatory, choose Snowflake or sufficiently long random codes instead and accept the different trade-offs.

### 10.1 Component diagram

```text
                         CREATE PLANE

Creator --> API gateway/rate limit --> Create service --> URL/policy validation
                                              |
                                      ID/range allocator
                                              |
                                      Base62 / code policy
                                              |
                                              v
                                     Mapping database
                                              |
                                      cache warm/invalidate


                        REDIRECT PLANE

Visitor --> DNS/edge/LB --> Redirect service --> L1 local cache
                                  | miss              |
                                  v                   |
                           distributed cache <--------+
                                  | miss
                                  v
                       replicated mapping database
                                  |
                                  +--> 302 + Location --> Visitor --> Destination
                                  |
                                  +--async click event--> queue/stream --> analytics
```

Create and redirect can begin as modules in one deployable application. Separate scaling/failure domains become valuable when traffic and reliability justify them; microservices are not a prerequisite for correctness.

### 10.2 Create flow step by step

1. Authenticate if required; apply account/IP quotas and abuse controls.
2. Validate size and parse the destination with a standard URL library.
3. Allow only supported schemes, normally `https` and possibly `http`.
4. Check idempotency key. Return the prior result if this request already committed.
5. If a custom alias was requested, validate/normalize it and conditionally reserve it.
6. Otherwise obtain an ID locally from the current allocated range and Base62-encode it.
7. Reject reserved/offensive codes and obtain another candidate if needed.
8. Transactionally/conditionally insert `code -> record` with a unique constraint and the idempotency result.
9. Warm or invalidate the cache after the durable write.
10. Return the short URL only after the mapping is durable under the promised consistency level.

If a candidate conflicts, never overwrite. Generate another or return `409` for a custom alias.

### 10.3 Redirect flow step by step

1. Edge/load balancer sends the request to a nearby healthy redirect instance.
2. Parse and strictly validate the code length/alphabet.
3. Check a small in-process L1 cache.
4. Check the distributed cache.
5. On miss, query the authoritative mapping store by exact code.
6. Verify `ACTIVE` status and `expires_at` at request time.
7. Populate cache with TTL no later than expiry and a versioned value.
8. Emit a click event asynchronously or record edge logs.
9. Return the configured redirect and `Location`.

Do not wait for analytics or visit the destination before redirecting.

### 10.4 Read-after-create consistency

The common bug is:

```text
write primary --> success
immediate redirect --> read replica still behind --> false 404
```

Options:

- synchronously warm cache after the committed write;
- have the redirect service fall back to the primary on a replica miss for recently created codes;
- pin the creator to the write region for a short interval;
- use a store with the required read consistency; or
- return a consistency token/version that the redirect path honors.

Blindly negative-caching the first replica miss makes the error last longer. Use short negative TTLs and primary fallback for plausible fresh codes.

### 10.5 Cache strategy

Use cache-aside:

```text
hit  -> return redirect
miss -> database -> cache -> return redirect
```

Cache value:

```text
{destination_url, status, expires_at, redirect_type, version}
```

Design details:

- Never cache beyond link expiry.
- Version or invalidate on edit/disable.
- Use request coalescing/single-flight so one hot miss creates one database lookup, not thousands.
- Add jitter to TTLs to avoid simultaneous expiry storms.
- Negative-cache invalid codes briefly to protect the database from scans.
- Use a Bloom filter only if invalid-code traffic is severe and its false-positive/refresh behavior is understood; it is an optimization, not a source of truth.
- Keep database capacity/circuit breakers for cache outage; a cache must not be the only copy.

### 10.6 Hot-link handling

One popular code maps to one logical record, so ordinary partitioning does not divide its read rate.

```text
viral code --> CDN/edge cache --> regional caches --> many redirect replicas
                                             \----> database only on cold miss
```

Mitigations:

- cache the redirect response at CDN/edge with a controlled TTL;
- replicate the small mapping across cache nodes/regions;
- use L1 caches in every redirect process;
- coalesce fills;
- keep the mapping immutable or propagate invalidation quickly; and
- derive analytics from edge logs/events rather than forcing every click through one origin.

The trade-off is that edge-cached redirects reduce origin visibility and delay edits/disablement. Safety-sensitive services may use short edge TTLs or real-time denylist checks at the edge.

### 10.7 Partitioning and replication

When one database no longer fits, hash-partition by the full public code:

```text
partition = stable_hash(code) -> logical shard
```

Do not partition by creation timestamp if most reads are by code. Hashing distributes sequential/Base62 codes and ordinary traffic across shards, though a single hot code still needs caching.

Each shard should have replicas across failure domains. Define:

- which replica accepts writes;
- whether redirect reads may be stale;
- failover behavior and RPO/RTO;
- backup/restore testing; and
- how shard routing metadata changes during resharding.

### 10.8 Multi-region design

Redirects are naturally global and read-heavy:

```text
Global traffic manager
        |
   +----+----+----------------+
   v         v                v
Region A  Region B         Region C
cache +   cache +          cache +
replica   replica          replica
```

Creation is harder because code uniqueness and custom-alias reservation are writes.

Options:

1. **Single home write region:** simple global uniqueness; higher remote create latency and regional create outage.
2. **Region-disjoint ranges/prefixes:** active regional creation without per-ID coordination; region bits/ranges affect compactness and rebalancing.
3. **Snowflake workers:** region/worker identity in ID; handle clocks and fencing.
4. **Random codes:** generate locally, but global conditional uniqueness/replication conflicts still need a policy.

Custom aliases require global conflict resolution. Route all reservations for an alias to one authoritative home partition or use a globally consistent conditional-write store. Last-write-wins is unacceptable because it could silently change the destination owned by a public code.

## 11. Correctness, failures, and edge cases

### 11.1 Dependency-failure matrix

| Failure | Redirect behavior | Create behavior | Mitigation |
|---|---|---|---|
| ID allocator unavailable | Existing links unaffected | Use remaining local range, then fail/queue create | Separate planes; range allocation; multi-AZ allocator |
| Distributed cache unavailable | Higher DB load/latency | Usually unaffected | Circuit breaker, L1 cache, DB headroom, gradual recovery |
| Mapping primary unavailable | Cached redirects may work; misses depend on failover | Pause or fail over writes | Replicas, automated/fenced failover, backups |
| Read replica lag | Stale edit or false miss | Write succeeds | Cache warm, primary fallback, consistency token |
| Analytics queue unavailable | Redirect should continue | Usually continue | Buffer/drop by explicit policy; monitor loss |
| One region unavailable | Route to another region | Home-region creates may pause | Replicated mappings, DNS/anycast failover, region-aware IDs |
| Abuse scanner unavailable | Policy-dependent | Fail closed, quarantine, or limited allow | Separate risk tiers and explicit degraded mode |

### 11.2 Ambiguous create retry

Network timeout does not prove failure:

```text
client ----create----> service ----commit----> DB
client <---X response lost
```

Without an idempotency key, retry creates a second code. Store `(creator, idempotency_key) -> result` with the link transaction. Define key expiry long enough for realistic retries and reject reuse of the same key with a different request body.

### 11.3 Cache invalidation on edit/disable

Safe patterns include:

- database commit, then publish an outbox event to invalidate caches;
- short TTL plus version validation;
- write-through update after commit; and
- a real-time denylist at the edge for emergency abuse takedown.

The transactional outbox avoids the dual-write bug where the database changes but the invalidation event is lost. A stale `301` cached in a user's browser may be impossible to revoke, which is why permanent redirects need special caution.

### 11.4 Expiration races

Cache TTL alone is insufficient. Clock skew or an incorrect TTL can serve expired links. Store `expires_at` in the cached value and compare it at request time using synchronized clocks; also cap the cache TTL to expiry.

Background cleanup reclaims storage, not public codes. Keep tombstones/non-reuse information.

### 11.5 Destination validation

At minimum:

- parse with a maintained URL parser, not a permissive regex;
- allow only `http`/`https` schemes;
- reject embedded credentials if product policy disallows them;
- bound the input length;
- normalize only according to an explicit conservative contract;
- detect self-redirect loops and excessive redirect chains where possible; and
- moderate custom aliases.

Do not allow `javascript:`, `data:`, `file:`, or arbitrary custom schemes in a generic web redirect service.

### 11.6 Phishing, malware, and open-redirect abuse

A generic shortener intentionally redirects to user-supplied destinations, so its trusted-looking domain is valuable to attackers. OWASP documents how unvalidated redirects enable phishing and exploit chains ([OWASP cheat sheet](https://cheatsheetseries.owasp.org/cheatsheets/Unvalidated_Redirects_and_Forwards_Cheat_Sheet.html)).

Controls include:

- authentication, quotas, rate limits, and CAPTCHA/risk scoring for creators;
- domain/reputation checks at creation and click time;
- blocklists plus human appeal/takedown workflows;
- warning/preview pages for suspicious destinations;
- detecting bulk campaigns and code enumeration;
- abuse-report endpoints and audit trails;
- limiting redirect chains; and
- transparent destination previews for users.

X publicly says its `t.co` link service checks against potentially dangerous sites and warns users, in addition to measuring clicks ([X Help](https://help.x.com/en/using-x/url-shortener)). That is verified product behavior, not evidence of a particular internal scanner architecture.

### 11.7 SSRF appears when the service fetches the URL

The basic redirect path does not need to contact the destination. But link previews, screenshots, metadata extraction, or malware scanning often do. Fetching an attacker-controlled URL can target:

- loopback addresses;
- private network services;
- cloud metadata endpoints;
- IPv6/link-local ranges; or
- a safe hostname that later resolves to a private address.

Run fetchers in an isolated network sandbox with strict egress rules, resolve and validate every redirect hop, block private/reserved address ranges, enforce time/size/content limits, and do not send internal credentials. See the [OWASP SSRF prevention guidance](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html).

### 11.8 Privacy and logging

Long URLs frequently contain email addresses, document tokens, campaign IDs, search terms, or session-like secrets. Do not indiscriminately log full destinations or query strings. Apply retention, access control, encryption, redaction, and regional privacy requirements to both mapping and click-event data.

The `Referer` header and analytics can reveal the short code and navigation context. Set an intentional `Referrer-Policy` and minimize collected fields.

### 11.9 Enumeration and denial of service

Sequential codes let a scanner enumerate valid mappings. Even random codes can be brute-forced if too short.

Mitigate with:

- sufficient entropy when confidentiality is claimed;
- per-IP/account/ASN rate limiting;
- negative caching and invalid-code filters;
- anomaly detection for sequential scans;
- WAF/bot management; and
- destination-side authorization.

Never return meaningfully different timing/body details for “private but exists” versus “never exists” if that leaks sensitive existence.

### 11.10 Common data-integrity rules

- `code` has a case-sensitive unique constraint.
- Custom alias reservation uses conditional insert, never check-then-insert without atomicity.
- Mapping changes are authorized and audited.
- Code is immutable even if destination is mutable.
- No code is reassigned after delete/expiry.
- Generator output is still checked by the mapping store.
- Analytics failure cannot change redirect correctness.
- Unknown state produces an error, never a guessed/fallback destination.

## 12. Real-system examples and what they teach

These examples deliberately separate documented behavior from architectural inference.

### 12.1 X/Twitter: `t.co`

**Public product behavior:** X says links shared in posts and direct messages are processed through `t.co`; it lists compact posting, click signals, and protection from dangerous sites as reasons ([X Help](https://help.x.com/en/using-x/url-shortener)).

```text
post contains destination
        |
        v
      t.co  --> click measurement / safety decision --> destination
```

Lesson: a shortener may be a platform control point for safety and ranking/analytics, not merely a character-saving utility.

### 12.2 YouTube: `youtu.be`

**Public product behavior:** YouTube's launch post said `youtu.be` was only for YouTube video links and that the short link contains the video's existing ID ([YouTube Blog, 2009](https://blog.youtube/news-and-events/make-way-for-youtube-links/)).

```text
youtube.com/watch?v=VIDEO_ID
             |
             v
      youtu.be/VIDEO_ID
```

Lesson: if a product already has a compact globally unique resource ID, the “shortener” can reuse it and may not need a second arbitrary code-to-URL mapping for the normal case.

### 12.3 WhatsApp: `wa.me`

**Public product behavior:** WhatsApp documents a click-to-chat URL format under `wa.me` ([WhatsApp Help Center](https://faq.whatsapp.com/5913398998672934)).

Conceptually:

```text
wa.me/<phone/action information> --> open chat in app or web
```

Lesson: this is closer to a compact/deep link whose path identifies an application action than a generic arbitrary-destination shortener. A readable identifier can expose sensitive information; compactness and privacy are separate concerns.

### 12.4 Uber: mobile/universal deep links

**Public product behavior:** Uber documents `m.uber.com` deep links that can open native/mobile-web ride flows and carry pickup/drop-off parameters, with fallback choices ([Uber Developers](https://developer.uber.com/docs/riders/ride-requests/tutorials/widget/migration-to-deep-links)).

```text
web/text link --> universal-link resolver
                    | app installed       | no app / chosen fallback
                    v                     v
                Uber app              mobile web/store
```

Lesson: interview scope can expand from “redirect to a web page” to device-aware routing, parameter validation, attribution, and fallback behavior. A deep link is related but not identical to a generic URL shortener.

### 12.5 Amazon: Associates short links

**Public product behavior:** Amazon Associates documents creating short and long product links and incorporating Associate/tracking IDs in the generated short-link workflow ([Amazon Associates Help](https://affiliate-program.amazon.com/help/node/topic/GJMMT7G4C8K4Y3AY)).

Lesson: the mapping record may include owner and campaign attribution; globally deduplicating solely by destination would destroy those semantics.

### 12.6 Netflix: share links as an illustrative extension

**Verified behavior:** Netflix documents sharing shows and movies from mobile devices ([Netflix Help](https://help.netflix.com/en/node/464)). **Its internal share-link architecture is not established by that page.**

An illustrative design might store:

```text
share code -> title_id + locale + campaign + app/web fallback policy
```

Lesson: do not claim that a familiar product uses Base62, Snowflake, or the architecture in this chapter without a first-party source. Use the product to explore requirements: regional catalog availability, app fallback, expired campaigns, and safe content previews.

## 13. Trade-offs and design decisions

### 13.1 Decision table

| Decision | Prefer option A when… | Prefer option B when… |
|---|---|---|
| Counter + Base62 vs random code | Shortest codes and simple collision-free allocation matter | Unpredictability and independent generation matter |
| DB sequence vs ticket service | One DB writer handles create load | Mapping data is sharded and ID allocation should be independent |
| Ticket per ID vs range allocation | Creation rate is low and simplest audit matters | Per-ID hop/bottleneck or allocator outage window matters |
| Range allocation vs Snowflake | Compactness and no clock dependence matter | Multi-region autonomy and very high throughput matter |
| Redis `INCR` vs durable DB allocator | Best-effort/checked allocation and low latency are enough | Counter rollback must not risk reissuing durable IDs |
| `302` vs `301` | Links are mutable, revocable, safety-checked, or analytics-sensitive | Links are immutable and client/edge caching is desired |
| SQL vs key-value store | Transactions, ownership queries, and moderate scale dominate | Massive point lookups and managed horizontal partitioning dominate |
| Global dedup vs new record | Exact same scoped policy should reuse a link | Ownership, campaigns, expiry, or privacy differ |
| Origin analytics vs edge analytics | Every click must reach current policy logic | Viral traffic and low latency require edge caching |

### 13.2 When not to use a URL shortener

Avoid or reconsider it when:

- the original URL is already compact and trustworthy;
- exposing the destination domain is important for user safety;
- the organization cannot promise long-term domain/service ownership;
- an extra redirect materially harms latency or reliability;
- a deep link or direct resource ID solves the problem without a mapping service; or
- the link is being mistaken for authorization to sensitive data.

### 13.3 Recommended design by situation

| Situation | Reasonable starting choice |
|---|---|
| Side project/internal tool | SQL table auto-increment + Base62, cache only after measurement |
| Public branded links, moderate create rate | Durable DB/range allocator + Base62 + abuse controls |
| Multi-region active-active creation | Snowflake-style ID or long secure-random code; globally safe custom aliases |
| Capability/private links | High-entropy secure random token; destination authorization and leak response |
| Existing compact resource ID | Reuse it in a short domain if semantics allow, as `youtu.be` illustrates |

### 14.3 Answer spine: `S-H-O-R-T`

Use cues to reconstruct the answer live instead of memorizing a paragraph.

#### S — Scope and scale

- mutable vs permanent redirects;

### 14.5 Sourced questions versus synthesized follow-ups

The company-specific prompts in Section 14.2 are sourced. The following questions are **my own plausible synthesis** from the design's pressure points, not claims about a particular interview.

#### “Why not hash the long URL and take seven characters?”

Answer cues:

- full hash is long; truncation enters finite code space;
- birthday collisions are inevitable at scale;
- deterministic retry repeats the same collision unless nonce/probing changes;
- same destination may need different owner/campaign/expiry records;
- URL canonicalization is business-sensitive;
- hash can work with scope + nonce + conditional insert + collision resolution.

#### “Why Base62?”

Answer cues:

- 62 alphanumeric, URI-unreserved characters;
- about 5.95 bits/character, compact;
- `62^7 ~= 3.52T` code space;
- encoding only, not uniqueness/encryption;
- case-sensitive storage/routing; Base58/Base36 trade-offs.

#### “Can Redis Cluster generate the global counter?”

Answer cues:

- `INCR` is atomic for one key;
- one key hashes to one slot/primary, so cluster does not shard that counter;
- per-shard counters require disjoint prefixes/ranges and are not one global sequence;
- async replication/failover can roll state back;
- `WAIT` reduces risk but does not make Cluster strongly consistent;
- DB unique constraint plus recovery is still required; durable range allocator is cleaner.

#### “Why not ZooKeeper?”

Answer cues:

- it can implement CAS/sequential IDs correctly;
- every ID becomes a coordination write and one counter path contends;
- ambiguous retries/metadata/sequence-scope and overflow details;
- use it to assign worker IDs, elect allocator, or lease ranges—not usually per URL.

#### “What if a Snowflake worker's clock goes backward?”

Answer cues:

- duplicate risk if timestamp + worker + sequence repeats;
- reject/wait until last timestamp, persist logical last time, or use logical-clock variant;
- alert/quarantine severe skew;
- unique leased worker IDs and fencing are equally important;
- only roughly ordered globally.

#### “301 or 302?”

Answer cues:

- `301` is permanent and heuristically cacheable; reduces origin load;
- cached clients may bypass edits, disablement, analytics, and safety checks;
- `302` keeps alias authoritative and is the mutable default;
- explicit cache headers/CDN policy matter;
- `307/308` preserve methods if non-GET support is intended.

#### “A new link returns 404 immediately. What happened?”

Answer cues:

- write committed on primary, read replica lagged;
- negative cache may amplify it;
- warm cache, primary fallback for fresh code, session consistency/token;
- keep negative TTL short and distinguish authoritative absence.

#### “One link goes viral. Does sharding solve it?”

Answer cues:

- sharding spreads different keys, not one key's traffic;
- CDN/edge + replicated L1/L2 cache;
- request coalescing on miss;
- edge logs for analytics;
- short TTL/emergency denylist if mappings must be revocable.

#### “What if the ID service is down?”

Answer cues:

- redirects should remain unaffected through plane isolation;
- range workers can continue until local block exhausted;
- then fail creation clearly rather than risk duplicate IDs;
- monitor range remaining and allocator SLO;
- random/Snowflake trades central dependency for other risks.

#### “How do custom aliases work in multiple regions?”

Answer cues:

- case-sensitive normalization and reserved names;
- conditional insert in authoritative alias partition;
- route same alias to same home/consensus domain;
- do not resolve conflict by last-write-wins;
- propagate mapping/cache only after winner commits.

#### “Can deleted codes be reused?”

Answer cues:

- no: old QR codes, messages, caches, and browser history persist;
- reuse could redirect old visitors to an unrelated owner;
- erase sensitive record fields but retain non-reusable tombstone/high-water mark;
- `410` versus `404` is a disclosure/product choice.

#### “How do you count clicks exactly?”

Answer cues:

- do not write link row synchronously per click;
- emit unique event to durable stream/outbox/edge logs;
- at-least-once means dedup or documented approximation;
- bot filtering, privacy, late events, and time-zone window semantics;
- “exact” has latency/cost/availability trade-offs—clarify business need.

#### “How do you stop phishing?”

Answer cues:

- shortener is an intentional open redirect, so reputation is a threat surface;
- creator auth/rate limits/risk scoring;
- destination checks at create and click;
- warn/block/takedown/appeal and audit;
- preview fetcher isolated against SSRF;
- unguessable codes do not make malicious destinations safe.

### 14.6 Interviewer red flags

- Saying “Base62 generates a unique ID.”
- Saying hashes never collide, or saying hashes can never be used.
- Choosing seven characters without a capacity/entropy calculation.
- Using Redis Cluster and assuming one counter is distributed across every shard.
- Using ZooKeeper for every ID without discussing contention and purpose.
- Calling Snowflake globally sequential or ignoring clock rollback/worker-ID reuse.
- Returning a short URL before its mapping is durably committed.
- Reading only from a lagging replica immediately after creation.
- Updating click count synchronously on the mapping row.
- Using `301` for editable/revocable links without discussing cached clients.
- Reusing expired codes.
- Claiming that random/unpredictable URLs are authorization.
- Adding a cache with no invalidation, expiry, negative-cache, or outage plan.
- Claiming a named company uses this architecture without a source.

## 15. Monitoring and operational readiness

### 15.1 Redirect SLO signals

- request rate, success/`404`/`410`/`5xx` rate;
- p50/p95/p99 latency by region and cache outcome;
- L1/L2/edge hit ratios;
- database lookup latency and replica lag;
- hot-code/tenant distribution;
- stale-cache/version mismatch rate;
- redirect loops and destination-policy blocks; and
- traffic shifted during regional failover.

### 15.2 Create and generator signals

- create success/latency and idempotent replay rate;
- generated-code collision/denylist retry rate;
- custom-alias conflict rate;
- remaining local range and allocation latency;
- allocator high-water mark and failover epoch;
- IDs burned per range/process crash;
- Snowflake clock rollback/sequence-wait/worker-lease errors if used; and
- Redis persistence/replica acknowledgements and counter regression alarms if used.

### 15.3 Analytics and abuse signals

- event publish failures, queue lag, duplicates, late events, and dropped events;
- unusual create volume by user/IP/ASN;
- sequential invalid-code scans;
- malicious-domain detection/takedown time;
- preview/scanner sandbox egress denials; and
- false-positive appeal outcomes.

### 15.4 Disaster-recovery test

Regularly prove that:

- mapping backups restore to a consistent point;
- restored allocator state cannot reissue any pre-disaster ID;
- DNS/certificates and the short domain are recoverable;
- caches refill without crushing the database;
- old codes still resolve correctly after migration; and
- emergency disablement propagates within the promised time.

## 16. Compact revision sheet

### 16.1 Five concepts to retain

1. A URL shortener is a durable `code -> destination` mapping plus an HTTP redirect.
2. ID generation and Base62 encoding are separate; uniqueness comes from the generator and authoritative conditional write.
3. A naive truncated hash has collisions and awkward dedup/retry semantics, though hash-based designs can work with nonce and collision handling.
4. A Redis Cluster does not distribute one counter key; one key still has one primary, and asynchronous failover can roll state back.
5. The redirect path needs caching, replication, hot-key handling, async analytics, read-after-create correctness, and abuse controls.

### 16.2 Formula card

```text
Base62 capacity:                M = 62^L
Average requests per second:    daily requests / 86,400
Expected random collision pairs n(n - 1) / (2M)
Approx probability any pair:    1 - exp(-n(n - 1) / (2M))
```

Do not apply the birthday formula to a sequential counter encoding: unique integers mapped bijectively to Base62 do not collide until the integer allocation itself repeats or exceeds the supported domain.

### 16.3 One-line choice guide

```text
moderate one-region creates -> DB sequence + Base62
compact IDs at higher scale  -> durable range allocator + Base62
active-active autonomy       -> Snowflake-style or secure random codes
unguessable capability link  -> enough cryptographic randomness + real authorization
```

### 16.4 Design checklist

- [ ] Clarified new-code versus dedup semantics.
- [ ] Clarified mutable, expiring, custom, private, and analytics requirements.
- [ ] Estimated creates, redirects, peaks, storage, retention, and code space.
- [ ] Defined `POST` idempotency and redirect status/cache policy.
- [ ] Used a case-sensitive unique code constraint/conditional insert.
- [ ] Separated ID generation from Base62 encoding.
- [ ] Explained hash collisions and retry/dedup semantics.
- [ ] Chosen generator based on create rate, regions, compactness, and unpredictability.
- [ ] Accounted for Redis/ZooKeeper/Snowflake failure modes if mentioned.
- [ ] Designed read-after-create and negative-cache behavior.
- [ ] Kept analytics off the redirect critical path.
- [ ] Covered cache outage, replica lag, allocator outage, and region failure.
- [ ] Handled viral hot links.
- [ ] Never reused expired/deleted codes.
- [ ] Added phishing, enumeration, URL validation, and preview-SSRF controls.
- [ ] Defined monitoring and disaster recovery.
