# Load Balancers for System Design (HLD)

> Beginner-first study notes expanded from the supplied bullets: **L4 and L7 load balancers; static algorithms (round robin, weighted round robin, IP hash); dynamic algorithms (least connections, weighted least connections, least response time, and `TTFB x active connections`).**
>
> The main idea is not "put a load balancer in front of the servers." It is to decide **what unit of work is being balanced, what information the balancer can see, what makes a target eligible, and what failure behavior the system needs.**

## 0. Scope, assumptions, corrections, and evidence labels

### Assumptions

- The focus is request/connection load balancing for web and service traffic, not CPU thread scheduling or network-path ECMP in depth.
- **IP hash** means selecting a backend from a hash of the client/source IP. A production hash might include more flow fields.
- **Least response time** means a latency-aware policy. Its exact score, measurement window, and tie-breaking rule are product-specific.
- Your bullet `TTFB * Active Connection` refers to a documented **Citrix NetScaler least-response-time formulation**:

  ```text
  score = number of active transactions x average TTFB
  choose the eligible service with the lowest score
  ```

  The historical guide special-cases an idle service so it is selected first. This formula is **not a universal industry definition** of least response time. Current NGINX, Envoy, AWS, HAProxy, and other products expose related but different policies.

### Important correction to the static/dynamic grouping

The grouping is useful, but it is not a protocol standard:

- A **static/input-based** policy does not inspect current backend load for each choice. Round robin, configured weights, and source-IP hash fit this idea.
- A **dynamic/load-aware** policy uses changing observations such as active connections, outstanding requests, measured latency, or CPU.
- Even a "static" scheduler normally uses a **dynamic healthy-target set**. Round robin should skip a server that has failed its health check.

So "static" does not mean the balancer never changes state; it means the **selection score is not based on live load feedback**.

### Evidence labels

- **Verified product behavior:** current first-party documentation or a primary paper linked in the chapter.
- **Historical product behavior:** supported by an older first-party product guide; it may not describe the current product.
- **Public production example:** the company publicly described this design at the publication date; it need not describe its complete current private architecture.
- **Illustrative design:** a reasonable way a named product could use the concept; not a claim about its private implementation.
- **Candidate-reported interview prompt:** a public self-report, not a verified official company question bank.
- **Unverified:** a claim worth checking before repeating as fact. No unsourced production-scale number is presented as current fact.

## 1. Plain-language foundation

### 1.1 Start with one server

Suppose a shopping application has one backend:

```text
Users --------------------> Server A
                              |
                              +--> application + local memory
```

This works until one of two things happens:

1. **Demand exceeds its capacity.** Requests wait, time out, or fail.
2. **The server fails.** There is no other instance to serve users.

We can add servers, which is **horizontal scaling**:

```text
Server A
Server B       Which one should each client call?
Server C
```

Now clients need a way to discover an instance, avoid dead instances, and distribute work without knowing every deployment change.

### 1.2 A load balancer is a dispatcher

Put one stable entry point in front of a pool of interchangeable servers:

```text
                      +--> Server A
Clients --> Load -----+--> Server B
            Balancer  +--> Server C
```

For each new unit of work, the load balancer:

1. identifies the relevant **backend pool**;
2. filters it to **eligible/healthy targets**;
3. applies a **selection algorithm**;
4. forwards or proxies the traffic to the chosen target; and
5. observes success, failure, latency, or connection state as its design permits.

The public address remains stable while servers are added, removed, restarted, or deployed.

### 1.3 The most important correction: equal requests are not equal load

Imagine three requests:

```text
GET /health             -> 2 ms, almost no CPU
GET /report?year=2026   -> 4 s, database + CPU
GET /video/live         -> connection lasts 2 hours
```

Giving each server the same **number of requests** need not give it the same:

- CPU time;
- memory usage;
- open connections;
- in-flight requests;
- bytes transferred;
- database work; or
- user-visible latency.

Therefore the interview question behind every algorithm is:

> **What observable quantity is the best available proxy for the expensive resource in this workload?**

No single algorithm is best for every workload.

### 1.4 What a load balancer does and does not promise

A correctly designed load-balancing tier can help with:

- **scalability:** add targets and give them traffic;
- **availability:** avoid known-unhealthy targets;
- **utilization:** spread work so capacity is not stranded;
- **latency:** avoid overloaded or slow targets;
- **deployment:** canary, slow-start, drain, and shift traffic;
- **indirection:** hide changing backend addresses from callers; and
- **observability/policy:** centralize connection and request metrics.

It does **not** magically:

- create backend capacity;
- make a stateful application stateless;
- replicate database data;
- guarantee that an accepted request completed;
- make a single load-balancer process highly available;
- know true CPU cost before a request runs; or
- prevent overload when every backend is saturated.

## 2. Required prerequisites, explained inline

These ideas are necessary to understand the rest of the chapter. They are brief here so the chapter does not assume prior networking knowledge.

See [Network Protocols](./01.%20Network%20Protocols.md) for a deeper treatment after this inline foundation.

### 2.1 DNS, IP addresses, and ports

- **DNS** maps a hostname such as `api.example.com` to one or more IP addresses.
- An **IP address** identifies a network endpoint for routing.
- A **port** identifies a service at that address, such as TCP port 443 for typical HTTPS.
- A load balancer exposes a **virtual service** or **listener**, usually described by an IP/hostname, protocol, and port.

```text
api.example.com --DNS--> load-balancer address:443
                              |
                              +--> 10.0.1.7:8080
                              +--> 10.0.2.9:8080
```

DNS helps the client reach the load-balancing tier. The local load balancer then chooses a backend. DNS itself can also distribute clients among regions, but its caching and failover behavior differs from per-request proxying.

### 2.2 TCP, UDP, connection, request, and stream

- **TCP** gives applications a reliable, ordered byte stream. A TCP connection is identified by source/destination addresses and ports plus protocol.
- **UDP** sends datagrams without TCP's connection semantics, although a load balancer may track a UDP pseudo-flow for affinity.
- An **HTTP request** is an application operation carried on a connection.
- One TCP connection can carry many sequential HTTP/1.1 requests.
- One HTTP/2 connection can multiplex many concurrent streams/requests.
- A **WebSocket** begins with an HTTP upgrade and then becomes a long-lived bidirectional connection.

This distinction changes algorithm behavior:

```text
L4 choice: often once per TCP connection/flow
L7 choice: can often be once per HTTP request or RPC
```

If one HTTP/2 connection carries 1,000 concurrent RPCs, an L4 balancer cannot spread those RPCs across 1,000 backends after the connection is established. An L7 proxy that understands the RPCs can potentially balance per stream/request.

### 2.3 Horizontal scaling and stateless application servers

Horizontal scaling means adding more instances instead of only buying a larger machine.

See [Server Scaling — Zero to Million to Billion](./05.%20Server%20Scaling%20-%20Zero%20to%20Million%20to%20Billion.md) for the wider scaling journey.

It is easiest when application instances are **stateless** with respect to user sessions:

```text
Request 1 -> Server A --\
Request 2 -> Server C ----> shared database/cache/object store
Request 3 -> Server B --/
```

If Server A alone stores a user's cart in its memory, sending the next request to Server B loses that state. Options are:

1. keep durable/session state in a shared store — usually preferred;
2. carry appropriate state in a signed/encrypted client token;
3. replicate state; or
4. use **session affinity/stickiness** as a deliberate compromise.

Load balancing and statelessness are related, but the balancer does not create statelessness.

### 2.4 Reverse proxy versus load balancer

A **reverse proxy** is an intermediary representing servers. A **load balancer** is a responsibility: selecting among multiple targets.

```text
one reverse proxy product can do:
TLS termination + authentication + routing + caching + load balancing
```

Many L7 load balancers are reverse proxies. Some L4 load balancers instead rewrite/forward packets and may use direct server return, so "every load balancer is an HTTP reverse proxy" is false. See [Proxy and Reverse Proxy](./16.%20Proxy%20and%20Reverse%20Proxy.md) for the full comparison.

### 2.5 Latency, TTFB, response time, throughput, and concurrency

- **Latency/response time:** how long one operation takes.
- **TTFB (time to first byte):** time until the first response byte/header becomes available. It does not measure how long the whole body takes.
- **Throughput:** completed operations or transferred bytes per unit time.
- **Concurrency:** work active at the same moment, such as open connections or in-flight requests.
- **Queue time:** time work waits before it starts running.

For a streaming response, TTFB might be 50 ms while the stream lasts two hours. For a buffered report, TTFB may include nearly all computation. Treating them as equivalent would be a design mistake.

## 3. Why load balancing is needed in system design

### 3.1 Turn a set of machines into one logical service

Clients depend on `api.example.com`, not `api-server-173.internal`. The service owner can change the backend fleet without updating every client.

### 3.2 Scale horizontally

If one instance safely handles `C` units of the measured workload, `N` instances do not automatically provide exactly `N x C`: there is protocol, coordination, skew, database, and failure overhead. Still, a load balancer is what makes a replicated application tier usable as one service.

### 3.3 Route around known failures

The balancer removes targets that fail health criteria and stops assigning them new work. Existing connections and in-flight requests need separate failure semantics; they cannot always be moved transparently.

### 3.4 Support elastic and safe deployment

Instances can pass through a lifecycle:

```text
starting -> ready -> warming -> serving -> draining -> removed
```

The balancer is the enforcement point for "do not send traffic until ready" and "stop new traffic but finish current work."

### 3.5 Keep failure domains explicit

A service may have targets across racks, zones, or regions. The traffic policy decides whether to:

- prefer local-zone capacity for latency and fault isolation;
- spread across zones for utilization;
- reserve capacity for failover; or
- evacuate a failed region.

### 3.6 Balance efficiency against reliability

Perfectly packing every target to 100% utilization leaves no headroom for a failure or burst. Production design intentionally holds spare capacity, sheds low-priority load, or queues bounded work.

## 4. Anatomy of a load-balancing system

### 4.1 The data path

The **data plane** handles live traffic:

```text
Client
  |
  v
[ listener :443 ]
  |
  +-- optional TLS termination
  +-- route/pool selection
  +-- health eligibility filter
  +-- scheduling algorithm
  |
  v
[ chosen target ]
```

Key pieces are:

- **listener/frontend:** address, port, and protocol accepting traffic;
- **route/rule:** for L7, maps host/path/header to a pool;
- **backend pool/target group:** candidate instances for one service;
- **target:** server, VM, container IP, pod, function, or appliance;
- **scheduler:** applies round robin, least connections, etc.;
- **connection table:** remembers flow-to-target mappings when required;
- **timeout/retry/drain logic:** manages failure and lifecycle behavior; and
- **telemetry:** records traffic, errors, latency, saturation, and target choice.

### 4.2 The control path

The **control plane** decides what configuration the data plane should use:

```text
Orchestrator/service registry ----> target membership
Health system --------------------> eligible targets
Operator/deployment system -------> weights, routes, drain state
Certificate/config store ---------> TLS and policy
                         |
                         v
                    LB data-plane nodes
```

Questions that matter:

- How quickly does a newly ready target become visible?
- What happens when two balancer nodes have different membership versions?
- Is configuration pushed, polled, or discovered through DNS/registry?
- Can the last known good configuration continue if the control plane is down?
- How are bad configuration rollouts rolled back?

The control plane should usually be outside the hot request path. Existing data-plane nodes should not stop serving merely because configuration management is briefly unavailable.

### 4.3 The selection pipeline

Backend choice is better modeled as a pipeline than as one algorithm:

```text
all registered targets
   -> correct route / region / zone
   -> administratively enabled
   -> healthy and ready
   -> not draining
   -> under hard connection/request limit
   -> apply RR / hash / least-* policy
   -> chosen target
```

"Use round robin" answers only one stage.

## 5. Layer 4 load balancing

### 5.1 Plain-language meaning

An L4 load balancer mainly understands transport/network information:

- protocol such as TCP or UDP;
- source IP and port;
- destination IP and port; and
- connection/flow state.

It does not need to understand that the bytes represent `GET /payments/42`.

```text
Client 203.0.113.8:51000
          |
          | TCP to 198.51.100.10:443
          v
       [ L4 LB ] --flow mapping--> 10.0.2.7:443
```

### 5.2 One flow must remain coherent

All packets belonging to the same TCP connection must reach the backend that owns that connection state. Sending packet 1 to A and packet 2 to B would not create one valid TCP session unless a much more specialized shared-state design existed.

Common selection inputs form a **5-tuple**:

```text
(source IP, source port, destination IP, destination port, protocol)
```

The exact inputs are product-specific. [AWS documents](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html) that its Network Load Balancer uses a flow hash with protocol, source/destination addresses and ports, plus TCP sequence information, and then keeps an individual TCP connection on one target for its lifetime.

### 5.3 L4 implementation styles

#### L4 proxy

The balancer terminates the client-side transport connection and opens another to the backend:

```text
Client --TCP #1--> L4 proxy --TCP #2--> Backend
```

This makes the balancer a full connection endpoint and consumes per-connection resources, but it provides strong control over both legs.

#### Passthrough/NAT/packet forwarding

The balancer rewrites or forwards packets while the backend terminates the connection. Some designs keep return traffic symmetric through the balancer; others use **direct server return (DSR)**:

```text
request:  Client -> L4 LB -> Backend
response: Client <---------- Backend       (DSR)
```

[Google Cloud documents](https://docs.cloud.google.com/load-balancing/docs/passthrough-network-load-balancer) a passthrough L4 design in which source/destination information reaches backend VMs unchanged and responses return directly. This reduces return-path balancer work, but networking and backend configuration become more specialized.

### 5.4 Benefits

- Protocol-agnostic for TCP/UDP-like traffic.
- Lower parsing and policy overhead than a full application proxy.
- Can preserve client source information naturally in passthrough designs.
- Fits high-throughput connection/packet distribution.
- TLS can terminate at the backend, retaining end-to-end application encryption through the balancer.
- Useful for non-HTTP protocols: databases, custom TCP, gaming UDP, DNS, or SMTP.

### 5.5 Limitations

- Cannot normally route `/payments` separately from `/catalog`.
- Cannot make per-request choices inside an existing TCP connection.
- Cannot apply HTTP-aware authentication, caching, header rules, or WAF inspection without another layer.
- Connection count is a weak estimate when flows have very different request counts or byte rates.
- Flow state and target changes must be handled carefully during LB-node or backend failure.

### 5.6 Choose L4 when

- the protocol is not HTTP or must remain opaque;
- extremely efficient packet/connection forwarding matters;
- TLS must terminate at the backend;
- client IP preservation or DSR is important;
- a later L7 tier already handles application routing; or
- balancing one whole flow to one target is the intended unit.

## 6. Layer 7 load balancing

### 6.1 Plain-language meaning

An L7 balancer understands the application protocol, most commonly HTTP:

```text
Host: api.example.com
Method: POST
Path: /payments
Headers: X-Tenant-ID, Authorization, Cookie, ...
```

It can therefore choose both a **service** and an **instance**:

```text
                         +-- /videos/* --> video pool --> V1/V2/V3
Client --> [ L7 LB ] ----+-- /users/*  --> user pool  --> U1/U2
                         +-- /pay/*    --> pay pool   --> P1/P2
```

### 6.2 TLS implication

For HTTPS, HTTP headers and paths are encrypted. To route by them, the L7 balancer normally **terminates TLS**:

```text
Client ==TLS #1==> L7 LB ==TLS #2 or private link==> Backend
                       reads HTTP
```

The backend leg should be protected according to the threat model; private networking is not a substitute for authentication in every environment. mTLS can authenticate services.

A balancer may route encrypted TLS using limited metadata such as SNI without reading HTTP, but path/header routing requires application visibility.

### 6.3 Benefits

- Host-, path-, method-, header-, cookie-, or tenant-aware routing.
- Per-request/RPC distribution when the protocol and connection behavior permit it.
- Central TLS termination and certificate management.
- Authentication hooks, rate limits, WAF rules, request-size limits, and header normalization.
- Canaries, A/B tests, version routing, redirects, rewrites, caching, compression, and protocol translation.
- Rich per-route and per-status observability.

### 6.4 Costs and limitations

- More CPU, memory, parsing, buffering, and configuration complexity.
- The balancer sees plaintext application data after TLS termination and becomes a high-value trust point.
- Protocol-specific; malformed or ambiguous message parsing creates security risk.
- Buffering can consume memory and break streaming expectations.
- A shared L7 policy/configuration bug can affect many services.
- HTTP/2, gRPC, streaming, and WebSocket behavior must be understood explicitly.

### 6.5 HTTP keep-alive and multiplexing nuance

An L7 proxy has a downstream connection from the client and an independent upstream connection pool to backends:

```text
client connection -> L7 proxy -> reusable connections to A/B/C
```

It may send successive requests from one client connection to different backends. However:

- a transaction stays on the selected backend;
- a WebSocket/long-lived stream normally remains on one backend;
- protocol ordering and connection-bound authentication may constrain reuse; and
- not every product balances HTTP/2 streams in the same way.

### 6.6 Choose L7 when

- traffic is HTTP/HTTPS/gRPC and application-aware routing matters;
- multiple services share one entry point;
- per-request balancing improves distribution;
- TLS, auth, WAF, rate limiting, or canary policy belongs at the edge; or
- rich request-level telemetry is worth the overhead.

## 7. L4 versus L7: interview comparison

| Question | L4 load balancer | L7 load balancer |
|---|---|---|
| Main selection information | Protocol, IPs, ports, flow state | Host, path, method, headers, cookies, RPC metadata, plus lower-layer data |
| Common selection unit | Connection/flow | Request/RPC; long-lived upgrades remain connection-like |
| Needs HTTP parsing? | No | Yes for HTTP-aware behavior |
| HTTPS path routing | No, unless TLS is terminated by an application-aware layer | Yes after TLS termination |
| Protocol coverage | Broad TCP/UDP/custom protocols | Protocol-specific, commonly HTTP/HTTPS/gRPC |
| Typical overhead | Lower | Higher due to termination/parsing/policy |
| Content routing, WAF, cache | Not by itself | Possible |
| Source-IP preservation | Natural in some passthrough designs | Often conveyed through a trusted header/proxy protocol |
| Backend sees client connection directly | In passthrough modes | Usually no; sees proxy connection |
| Best fit | Fast opaque flow distribution | Application-aware traffic management |

### A common two-tier design

This is not redundant duplication; the layers solve different problems:

```text
Internet
   |
   v
[ global DNS/anycast ]       choose region/edge
   |
[ highly efficient L4 ]      spread flows across L7 proxy fleet
   |
[ L7 proxy/gateway ]         TLS, route, auth, per-request policy
   |
[ application instances ]
```

[Google's documented GKE example](https://docs.cloud.google.com/kubernetes-engine/docs/tutorials/exposing-grpc-services-on-gke-using-envoy-proxy) similarly combines an external L4 passthrough balancer with Envoy L7 proxies, allowing the outer tier to distribute TCP connections and Envoy to terminate TLS and balance gRPC per request.

## 8. Where load balancing can happen

### 8.1 Global versus local

```text
                          +--> Region A local LB --> servers
Client -> global traffic -+
          selection       +--> Region B local LB --> servers
                          +--> Region C local LB --> servers
```

- **Global traffic management** chooses a region/edge using DNS, anycast routing, latency, geography, health, policy, or capacity.
- **Local load balancing** chooses an instance inside that region/zone/service pool.

DNS responses are cached, so DNS failover is not as immediate or precise as an L7 proxy decision. Anycast and global proxy networks have different routing/failure properties.

### 8.2 Server-side/proxy load balancing

The caller sees one address and an intermediary selects the target:

```text
Caller -> LB/proxy -> Service instance
```

Benefits: simple clients, centralized security/policy, controlled public exposure.

Costs: extra hop, shared tier, needs its own scaling and HA.

### 8.3 Client-side load balancing

The caller obtains service instances from discovery and chooses one:

```text
                  +--> Instance A
Caller + LB logic +--> Instance B
                  +--> Instance C
       ^
       |
 service discovery
```

Benefits: no separate proxy hop, each client sees its own outstanding requests, natural for internal RPC libraries/service meshes.

Costs: every client/library must implement discovery, policy, retries, telemetry, security updates, and stale membership handling consistently.

Sidecar/on-host proxies are a middle ground: selection is local to the caller but implemented by shared infrastructure rather than application code.

### 8.4 Load balancing is used internally too

It can sit between:

- public clients and edge proxies;
- API gateway and microservices;
- service A and replicas of service B;
- workers and stateless processing nodes;
- query routers and database read replicas; or
- cache clients/proxies and cache nodes.

For data systems, **sharding/partition ownership** is not the same as freely balancing interchangeable requests. A write for shard 7 must reach the correct owner/leader, not simply the least busy database.

## 9. Before choosing an algorithm: define the unit and signal

### 9.1 Selection unit

Ask whether one scheduler decision applies to:

- a packet;
- a UDP pseudo-flow;
- a TCP connection;
- an HTTP request/RPC;
- a WebSocket session;
- a background job; or
- a key/tenant requiring affinity.

Round robin per TCP connection and round robin per HTTP request can produce very different distributions.

### 9.2 Load signal

| Signal | What it approximates | Where it misleads |
|---|---|---|
| Selection/request count | Number of assignments | Requests have unequal cost/duration |
| Active connections | Concurrent connection occupancy | Idle keep-alives and busy multiplexed connections differ |
| Outstanding requests | Work not yet completed | Requests vary greatly in CPU/bytes/downstream work |
| TTFB/latency | Recent responsiveness | Network path, client, queues, cache, and request mix affect it |
| CPU/memory | Direct resource saturation | Metrics are delayed; request may be bottlenecked elsewhere |
| Bandwidth | Network transfer pressure | CPU/database may be the real bottleneck |
| Queue depth | Work waiting | Queue semantics and service rates may vary by target |

### 9.3 Information has a cost

Round robin needs little shared state. Least connections needs current counters. Latency-aware or CPU-aware scheduling needs measurements, smoothing, freshness, and sometimes coordination among many balancer nodes.

Stale or noisy feedback can make a sophisticated policy worse than a simple one.

## 10. Static/input-based algorithms

### 10.1 Round robin

#### How it works

Cycle through eligible targets in order:

```text
Targets: A, B, C

Request/connection:  1  2  3  4  5  6  7
Chosen target:       A  B  C  A  B  C  A
```

The counter wraps after the last target. If B is unhealthy, the eligible sequence becomes A, C until B recovers.

#### Why it works

Over enough selections, each eligible target receives approximately the same **number of assignments**. It is cheap, predictable, and needs no live load metric.

#### Best fit

- Homogeneous target capacity.
- Similar request/connection duration and resource cost.
- Stateless requests.
- Large enough traffic volume for short-term sequence effects to average out.

#### Failure cases

Suppose requests alternate between cheap and expensive:

```text
R1 cheap -> A
R2 heavy -> B
R3 cheap -> C
R4 heavy -> A
R5 cheap -> B
R6 heavy -> C
```

This example happens to balance. But if every third request is heavy, one server may repeatedly get it. More generally, equal counts do not guarantee equal work.

Round robin also behaves poorly when:

- a target is slower despite identical hardware;
- connections have highly variable lifetimes;
- a new cold instance immediately receives its full share;
- multiple independent balancers' counters align and create bursts; or
- a few clients keep very long L4 connections, so no new selection occurs.

#### Common mistake

Do not say "round robin sends every user equally to every server." It sends each **new selection unit** in rotation. A connection may carry many requests, and session affinity may override later choices.

---

### 10.2 Weighted round robin

#### Why weights are needed

Suppose:

```text
A capacity = 1 unit
B capacity = 2 units
C capacity = 3 units
```

Equal round robin wastes C or overloads A. Assign relative weights `1:2:3` so that over a six-selection cycle the intended shares are:

```text
A: 1/6
B: 2/6
C: 3/6
```

A simple conceptual schedule could be:

```text
A, B, C, B, C, C
```

Production implementations often use **smooth weighted round robin** or another scheduling technique so C's assignments are distributed across the cycle instead of arriving as a burst. Exact sequence is implementation-specific; the long-run ratio is the important contract.

#### What a weight should mean

A weight is a relative target share, not necessarily a CPU percentage. Derive it from measured sustainable capacity under the representative workload:

```text
relative weight ~= target sustainable work rate / baseline work rate
```

If a larger instance handles twice the real mixed workload at the same latency/error objective, weight 2 may be a reasonable starting point. Benchmark rather than deriving weights only from core count.

#### Useful cases

- Mixed instance generations or sizes.
- Gradual canary rollout: new version initially gets a small share.
- Capacity differs by target/zone.
- Temporarily reducing traffic to a degraded pool.

#### Trade-offs and traps

- Configured weights become stale as workloads or noisy neighbors change.
- A weight based on CPU may be wrong for memory-, disk-, or database-bound requests.
- `90:10` is an assignment target, not a guarantee for ten requests.
- A tiny canary weight can still receive a high-risk request or hot tenant.
- A recovered instance may need **slow start** instead of immediately receiving full weighted share.
- Sticky sessions can make observed traffic diverge from configured weights.

---

### 10.3 IP hash

#### How it works

Hash a client/source IP, then map it to a backend:

```text
index = hash(source_ip) mod number_of_eligible_targets
target = targets[index]
```

For a stable target list, the same source IP usually maps to the same backend:

```text
203.0.113.10 ----hash----> Server B
203.0.113.11 ----hash----> Server A
203.0.113.10 ----hash----> Server B again
```

#### Why it is used

- Simple source affinity for a stateful session.
- Cache locality when requests from the same source reuse local data.
- Flow consistency in some packet-forwarding designs.

#### Major limitations

##### NAT and corporate proxies

Thousands of users can appear behind one public IP and all map to one backend, creating a hotspot.

##### Mobile clients change addresses

Switching between Wi-Fi and cellular, carrier NAT behavior, or IPv6 privacy addresses can remap the user and break affinity.

##### One user can use many IPs; many users can share one IP

IP is not a reliable authenticated user/session identifier.

##### Pool changes remap broadly under modulo

If the number or ordering of targets changes, many source IPs move. For state/cache affinity, consistent hashing or rendezvous hashing can reduce movement; see [Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md).

##### Unequal work remains

Hashing aims for uniform keys, not uniform request frequency. One busy customer or shared NAT can still dominate a target.

##### Proxies obscure the original source

If an outer CDN/proxy connects to the balancer, the transport source may be the proxy. Trusting an `X-Forwarded-For` value for affinity requires a controlled proxy chain and spoofing protection.

#### Use/avoid guidance

Use IP hash only when approximate source affinity is genuinely useful and its failure behavior is acceptable. Prefer application-level session cookies or stable authenticated IDs when L7 is available, and prefer shared state so any healthy target can serve the user.

## 11. Dynamic/load-aware algorithms

### 11.1 Least connections

#### How it works

Choose the eligible target with the fewest active connections:

```text
A: 12 active connections
B:  4 active connections  <-- choose B
C:  9 active connections
```

When a connection opens, increment its chosen target's counter; when it closes, decrement it. Tie-breakers are product-specific, often round robin or random.

#### Why it can beat round robin

Long connections remain counted, so a target holding many long-lived connections is less likely to receive the next one. [HAProxy's official documentation](https://www.haproxy.com/documentation/haproxy-configuration-tutorials/proxying-essentials/configuration-basics/backends/) recommends least connections for long-lived protocols such as database connections, gRPC streams, and LDAP.

#### What it assumes

Each connection is assumed to represent roughly similar load. That assumption can fail:

```text
Server A: 100 idle keep-alive connections, almost no work
Server B:   5 HTTP/2 connections, 500 active streams
```

Least connections sees A as busier even though B may consume much more CPU.

#### Other failure modes

- A just-started target has zero connections and can receive a sudden herd.
- Independent balancer nodes may have only local, incomplete counters.
- A slow target can have fewer completed/closed connections and remain loaded; depending on arrival patterns, the feedback can help or become unstable.
- Connection pools between proxies and backends can make upstream connection count differ from request concurrency.
- WebSocket connections vary: one may be idle while another carries heavy message traffic.

#### Best fit

- L4 connections with significantly different lifetimes but roughly similar per-active-connection cost.
- Long-lived database, streaming, or socket sessions.
- Situations where the balancer has accurate connection lifecycle state.

For request-oriented HTTP services, **least outstanding requests** is often a more meaningful signal.

---

### 11.2 Weighted least connections

#### Goal

Combine live connection occupancy with configured target capacity.

A common conceptual comparison is:

```text
normalized_load_i = active_connections_i / weight_i
choose the lowest normalized load
```

Example:

| Target | Active connections | Weight | `connections / weight` |
|---|---:|---:|---:|
| A | 20 | 1 | 20 |
| B | 30 | 2 | 15 <- choose |
| C | 40 | 2 | 20 |

B has more raw connections than A but less load relative to its configured capacity.

**Important:** this equation is an interview model, not a promise that every product implements precisely this arithmetic. NGINX states that `least_conn` takes weights into account; HAProxy has its own scheduling details; Envoy documents an effective-weight formula for weighted least request.

#### Benefits

- Handles mixed server sizes better than unweighted least connections.
- Reacts to current occupancy while respecting expected capacity.
- Useful for long-lived heterogeneous connection workloads.

#### Limitations

- Inherits every weakness of connection count.
- Weights can be wrong or stale.
- A weight of 2 assumes twice the relevant capacity, which must be validated.
- Counters may be local to each LB node, so globally perfect balance is not guaranteed.
- Slow start and hard concurrency limits are still needed during recovery/deployments.

---

### 11.3 Least response time

#### Plain-language idea

Send new work toward targets that have recently responded faster, often while also considering current in-flight work.

```text
A: average TTFB  40 ms, 8 active
B: average TTFB 150 ms, 3 active
C: average TTFB  55 ms, 2 active
```

It is not enough to say "choose A because 40 ms is smallest." A may already be busy; B's slower measurements may have come from different endpoints; C might be the best compromise. The product must define a score.

#### Measurement choices

A real design must answer:

- TTFB or full response completion time?
- Exponential moving average, rolling window, percentile, or last sample?
- Per target, per route, per request class, or one mixed metric?
- Does the measurement include network time from this balancer node?
- How are timeouts, failures, and in-flight requests scored?
- How quickly can a recovered target earn a representative sample?

[Current NGINX documentation](https://nginx.org/en/docs/http/ngx_http_upstream_module.html) exposes `least_time header|last_byte [inflight]`: it can use time to response headers or time to the last byte, considers active connections and weights, and can account for incomplete requests. That is related to, but not identical to, every other "least response time" implementation.

#### Benefits

- Can steer away from degraded, overloaded, or slower targets.
- Captures more than raw connection count when targets differ in actual performance.
- Useful when latency is a reasonable signal for queueing/service health.

#### Risks

##### Noisy feedback

The fastest target gets more traffic, becomes slower, then traffic swings elsewhere. Without smoothing, randomization, or damping, the control loop can oscillate.

##### Different request mixes

If A recently served cheap health/read requests and B served expensive reports, their averages are not comparable. Route-level statistics help but cost more state and samples.

##### Cold and stale samples

An idle target may have an old good measurement. A new target has no history. Explicit warm-up policy is required.

##### Coordinated herding

Many LB nodes can observe the same fast target and simultaneously choose it. Power-of-two random choices, local observations, and bounded concurrency reduce this risk.

##### TTFB is not completion time

A service can emit headers quickly and stream slowly. For downloads or streaming, first-byte latency may not reflect connection occupancy or total resource consumption.

#### Best fit

- Request workloads where recent target latency correlates with spare capacity.
- Heterogeneous or variably degraded targets.
- A balancer with sufficient samples, smoothing, and sensible request classes.

Avoid it when the sample is dominated by client/network variance, traffic is too low for stable measurements, or simple policies already meet the SLO.

---

### 11.4 `TTFB x active connections`: your exact bullet

#### Historical documented formulation

[A historical Citrix NetScaler Traffic Management Guide](https://kb.wisconsin.edu/images/group9/13201/NS-Load_Balance_Methods.pdf) described **Least Response Time** for HTTP/SSL services using:

```text
N = number of active transactions x TTFB
choose the service with the least N
```

It defined TTFB as the interval from sending a request packet to receiving the first response packet, used successful HTTP `200` responses for that measurement, and documented a special rule: a service with no active transaction is selected first.

The guide's worked setup was:

| Service | Active transactions | TTFB | Score `N = active x TTFB` |
|---|---:|---:|---:|
| HTTP-1 | 3 | 2 s | 6 |
| HTTP-2 | 7 | 1 s | 7 |
| HTTP-3 | 0 | 2 s | 0 <- selected first |

This combines:

- a **current occupancy signal**: active transactions; and
- a **historical speed signal**: average TTFB.

HTTP-2 is individually fast but already busy. HTTP-1 is slower but less occupied. HTTP-3 is idle, so it receives the next request.

#### Weighted historical variant

The same guide documented:

```text
Nw = N x (10000 / weight)
```

The constant only scales the number; relative weight is what matters. Higher-weight services receive a lower normalized score for the same `N`.

#### Why multiplication is intuitive

A rough queueing intuition is:

```text
more work already active x more time per work item = more expected occupancy
```

It penalizes a target that is either busy, slow, or both.

#### Why it is not a universal law

- `active = 0` makes every idle target score zero regardless of whether its TTFB is 5 ms or 5 s; tie-breaking/special policy matters.
- Averages hide tail latency.
- TTFB may not correlate with total work.
- Multiplication assumes connection/transaction count and average latency are comparable across request types.
- Old samples and few samples distort the score.
- Multiple balancers have different local observations.
- A target might return first byte quickly and then stall.

Use the formula in an interview as **one concrete latency-aware implementation**, then say the exact production scoring policy is product-specific and must be validated against the workload.

---

### 11.5 Least outstanding requests: an important related alternative

Choose the target with the fewest in-progress application requests rather than TCP connections:

```text
A: 12 outstanding HTTP requests
B:  3 outstanding HTTP requests  <-- choose B
C:  8 outstanding HTTP requests
```

This is often better for HTTP/2/gRPC because one connection can carry many requests. [AWS Application Load Balancer](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/edit-target-group-attributes.html) supports a least-outstanding-requests algorithm and documents that it treats converted HTTP/2 requests as multiple HTTP/1.1 requests for this accounting.

It still assumes requests are roughly comparable. A five-second ML inference and a two-millisecond cache hit each count as one.

#### Power of two choices (P2C)

Instead of scanning every target:

1. choose two eligible targets at random;
2. send the request to the less loaded of the two.

```text
random sample: B (7 outstanding), F (2 outstanding)
choose F
```

This gives strong balancing with low selection cost and reduces everyone chasing the single globally "best" server. [Envoy documents](https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/upstream/load_balancing/load_balancers.html) a default two-choice least-request policy for equal-weight hosts. [Twitter/X publicly described](https://blog.x.com/engineering/en_us/topics/infrastructure/2019/daperture-load-balancer) using P2C in Finagle, normally comparing outstanding requests.

## 12. Algorithm comparison and choice

| Algorithm | Live load feedback? | State/cost | Strong use case | Main failure mode |
|---|---|---|---|---|
| Round robin | No | Very low | Similar targets and short, similar work | Equal assignments are not equal load |
| Weighted round robin | Configured capacity only | Low | Predictably different target capacity, canaries | Stale/wrong weights; request cost varies |
| IP hash | No backend-load signal | Low to medium | Approximate source affinity | NAT hotspots, IP changes, pool-change remapping |
| Least connections | Active connections | Medium | Long-lived connections | Connections differ in activity; HTTP/2 multiplexing |
| Weighted least connections | Connections + configured capacity | Medium | Long-lived connections on mixed-capacity targets | Bad weights and misleading connection count |
| Least outstanding requests | In-flight application requests | Medium | Variable-duration request services | One count still hides request cost |
| Least response/least time | Recent latency, often occupancy | High | Latency reflects target load/degradation | Stale/noisy samples, oscillation, mixed request classes |
| P2C least request | Small random sample + local load | Low/medium | Large dynamic pools, distributed proxies | Approximate rather than global minimum |
| Consistent/rendezvous hash | Key mapping, not live load | Medium | Cache/state locality with limited remapping | Hot keys; affinity fights load balance |

### A practical decision tree

```text
What is the selection unit?
|
+-- packet/connection, opaque protocol
|     |
|     +-- similar short connections --------> round robin
|     +-- variable long connection lifetime -> least connections
|     +-- different capacities -------------> weighted variant
|     +-- stable affinity required ----------> flow/IP/key hash, accept trade-offs
|
+-- HTTP request/RPC visible at L7
      |
      +-- similar targets and work ----------> round robin
      +-- variable duration -----------------> least outstanding / P2C
      +-- measured latency is reliable ------> least-time/latency-aware
      +-- different capacities -------------> weighted policy + slow start
      +-- session/cache locality ------------> cookie/key hash or externalize state
```

### A mature interview answer

> "I would begin with round robin because the instances and request costs are expected to be similar. I would measure per-target outstanding requests, latency, CPU, and error rate. If duration skew causes queue imbalance, I would move to least outstanding requests or P2C. I would not select least response time merely because it sounds smarter; its feedback needs stable samples and route-aware smoothing. Health filtering, hard concurrency limits, slow start, and draining remain separate from the scheduler."

## 13. Health checks: who is eligible?

The best scheduling algorithm still fails if it sends traffic to a dead or unready target.

### 13.1 Active health checks

The balancer periodically probes a target:

```text
L4: can a TCP connection be established?
L7: does GET /ready return an accepted status promptly?
gRPC: does the health method report serving?
```

[AWS documents](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) configurable intervals, timeouts, and consecutive healthy/unhealthy thresholds for ALB target health. Exact default numbers change by product and target type, so do not memorize one universal value.

### 13.2 Passive health checks

The data plane observes real traffic:

- connection refused/reset;
- repeated timeout;
- abnormal 5xx/error rate;
- protocol failure; or
- extreme latency.

Passive checks reflect the actual request path but can confuse application/user errors with instance health. One `404` should not evict a server.

### 13.3 Liveness, readiness, shallow, and deep checks

- **Liveness:** is the process alive enough that restarting may or may not help?
- **Readiness:** should this instance receive new traffic now?
- **Shallow/local check:** process, event loop, disk, config, or required local resource works.
- **Deep/dependency check:** calls database, DNS, queue, or another service.

Deep checks catch dependency failures but can create **correlated eviction**:

```text
shared database has a brief issue
    -> every app instance's deep check fails
    -> LB removes every app instance
    -> application becomes completely unreachable
```

[Amazon's health-check guidance](https://aws.amazon.com/blogs/networking-and-content-delivery/choosing-the-right-health-check-with-elastic-load-balancing-and-ec2-auto-scaling/) explicitly discusses shallow versus deep checks. A common design uses readiness based mainly on local ability to serve and exposes dependency health separately, with careful fail-open/fail-closed policy.

### 13.4 Health-state machine

Use thresholds and hysteresis rather than flipping on one sample:

```text
             enough consecutive failures
HEALTHY ------------------------------------> UNHEALTHY
   ^                                             |
   | enough consecutive successes               | keep probing
   +---------------------------------------------+

Optional states: INITIAL, WARMING, DRAINING, QUARANTINED
```

Why consecutive thresholds?

- One packet loss should not cause flapping.
- One lucky successful probe should not immediately restore a broken server.
- Recovery should usually include slow start.

### 13.5 Fail open versus fail closed

If every target is marked unhealthy, options include:

- **fail closed:** return an error; avoids sending to known-bad targets;
- **fail open:** try all registered targets; useful if the health system is wrong and some target can still serve;
- serve a degraded/static response;
- shed lower-priority requests; or
- fail over to another pool/zone/region.

This is a product and risk decision. [AWS ALB documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html), for example, says its target group can fail open to all targets when all registered targets fail health checks. Do not assume all load balancers behave the same way.

### 13.6 Health check storm and false confidence

If 1,000 balancer nodes probe 10,000 targets every second, naive probing itself becomes large load. Systems use distributed health aggregation, jitter, less frequent checks, or local subsets.

A health endpoint returning `200` only proves that endpoint answered. It does not guarantee every real route, dependency, tenant, or data shard works.

## 14. Sessions, affinity, and long-lived connections

### 14.1 Preferred baseline: any target can serve any request

Keep user/session state in a shared durable system or suitable client token:

```text
Client -> any App A/B/C -> shared session/cache/database
```

This gives the balancer freedom to route around failure and distribute work.

### 14.2 Sticky session

The balancer remembers or encodes a client/session-to-target mapping:

```text
session S17 -> Server B
```

Mechanisms include:

- load-balancer-generated cookie;
- application cookie value hashed by the balancer;
- source-IP hash;
- authenticated user/tenant key; or
- consistent hash on an application key.

### 14.3 Benefits

- Preserves local session state for a legacy application.
- Improves local cache reuse.
- Keeps connection/session-bound workflows together.
- Can reduce repeated warm-up for expensive per-user state.

### 14.4 Costs

- Uneven user activity creates hot targets.
- A failed target breaks or remaps its sessions.
- Scaling out does not immediately redistribute existing sticky sessions.
- Draining can take a long time.
- It hides an architectural dependency on local state.
- Cross-zone/region failover becomes harder.

Affinity is a routing preference, not durable storage. If losing the chosen backend loses important state, the application still needs replication or a shared store.

### 14.5 WebSockets and messaging

Once a WebSocket connection is assigned, frames stay on that connection and backend until disconnect/failure. The algorithm affects **new connections**, not migration of a live one.

For millions of long-lived messaging connections, consider:

- connection count and per-connection memory;
- messages/bytes per connection, not only open count;
- reconnect storms after a gateway failure;
- graceful drain duration;
- a shared/durable message path so gateway loss does not lose messages; and
- a presence/directory layer for locating a user's active connections.

## 15. Making the load-balancing tier highly available

### 15.1 The obvious trap

```text
Clients -> ONE load balancer -> many healthy servers
              X
```

This moves the single point of failure from the server to the load balancer.

### 15.2 Redundant LB nodes

```text
                   +--> LB node 1 --+
Clients -> VIP/DNS +--> LB node 2 --+--> backend pool
                   +--> LB node 3 --+
```

Traffic can reach multiple nodes via:

- a virtual IP with active/standby failover;
- anycast/BGP;
- DNS returning multiple addresses;
- an outer network load-balancing tier;
- ECMP from routers; or
- a managed cloud service that hides these details.

[Amazon's Builders' Library](https://aws.amazon.com/builders-library/minimizing-correlated-failures-in-distributed-systems/) describes the general principle of running services behind redundant load balancers rather than making the balancer itself a single point of failure. [Google's Maglev paper](https://research.google/pubs/maglev-a-fast-and-reliable-software-network-load-balancer/) describes a distributed software L4 balancer whose nodes receive traffic from routers through ECMP.

### 15.3 Stateful versus stateless balancer nodes

Round-robin counters can be local and approximate. Active TCP flows require stable mapping/connection state.

Options include:

- route every flow consistently so all its packets reach one LB node;
- replicate connection state to a standby;
- accept that LB-node failure resets affected connections and clients reconnect;
- use stateless consistent mapping for new flows; or
- keep state in specialized shared data-plane structures.

State synchronization improves continuity but adds latency, complexity, and correlated failure risk.

### 15.4 Capacity and failure headroom

If three LB nodes each run at 70% and one fails, the remaining two may need 105% each:

```text
before: total load 210 capacity units / 3 nodes = 70 each
after:  total load 210 / 2 nodes = 105 each -> overload
```

Design for `N+1` or the selected failure scenario, not just normal average load. Include sudden reconnect and retry traffic.

### 15.5 Zonal choices

- **Cross-zone balancing** improves pool-wide utilization but incurs inter-zone latency/cost and can spread a zone problem.
- **Zonal affinity/isolation** keeps traffic local and contains failure but needs enough capacity in every zone.
- During failover, verify the surviving zone has headroom and downstream databases/caches can absorb the shift.

### 15.6 Global failover caveat

DNS TTL is not an exact evacuation timer: resolvers and clients cache, existing connections persist, and some clients retry old addresses. Global design often combines DNS/anycast selection, health, multi-region capacity, and application-level retry/failover.

## 16. TLS, identity, and security

### 16.1 TLS patterns

#### Terminate at L7 balancer

```text
Client ==TLS==> LB ----> Backend
```

Central certificates and HTTP visibility; backend leg still needs protection appropriate to the trust model.

#### Terminate and re-encrypt

```text
Client ==TLS #1==> LB ==TLS #2==> Backend
```

Protects both network legs and allows L7 policy, but adds certificate/service-identity management.

#### TLS passthrough

```text
Client =================TLS=================> Backend
                    L4 forwards bytes
```

The backend owns the client TLS session; application routing/inspection at the balancer is limited.

### 16.2 Preserving client identity

A proxy connection makes the backend see the proxy address. L7 proxies commonly add controlled metadata such as `Forwarded` or `X-Forwarded-For`; L4 proxies may use Proxy Protocol.

Security rule:

> Strip/overwrite client-supplied forwarding headers at the trusted edge and trust them only from authenticated/allowlisted proxy hops.

Otherwise a client can claim another IP and bypass IP-based rate limits, audit, or affinity logic.

### 16.3 The load balancer is not the whole security design

It may provide DDoS absorption, WAF, TLS policy, auth, rate limiting, and request limits, but:

- origins must not be directly bypassable;
- administrative/control-plane access must be protected;
- configuration and certificate secrets need audit/rotation;
- backend authorization must not blindly trust arbitrary headers; and
- L7 parsers must handle ambiguous HTTP consistently to avoid request-smuggling classes of bugs.

## 17. Timeouts, retries, overload, and correctness

### 17.1 A failed connection is ambiguous

```text
Client -> LB -> Server A -> Database commits
                    |
             response is lost
```

The load balancer may know the response did not reach the client, but not whether the operation ran. Retrying a payment/order write on Server B can duplicate the side effect.

Use bounded retries only when:

- the operation is safe/idempotent, or has an idempotency key;
- the retry budget and deadline allow it;
- the failure point is understood; and
- retries do not multiply at every layer.

See [Idempotency](./13.%20Idempotency.md).

### 17.2 Timeout layers

```text
client deadline
  > LB request timeout
      > backend call/downstream timeouts
```

The remaining deadline should shrink down the call chain. Random independent timeouts can leave lower layers doing work after the user has given up.

Streaming and WebSocket idle timeouts need separate configuration from ordinary request timeouts.

### 17.3 When every backend is overloaded

An algorithm can redistribute spare capacity; it cannot invent it. Use:

- hard per-target concurrency limits;
- bounded queues;
- rate limiting/admission control;
- load shedding by priority;
- autoscaling based on meaningful leading signals;
- cached/degraded responses;
- circuit breakers; and
- retry budgets with jitter/backoff.

Unbounded LB queues turn overload into high latency and memory exhaustion. Rejecting early can be safer than accepting work that cannot finish before its deadline.

### 17.4 Retry storms

If each of three layers retries three times, one user request can amplify dramatically. Exact amplification depends on where failures occur, but the principle is clear: define a single end-to-end retry budget, use backoff/jitter, and avoid retrying known-overload responses blindly.

## 18. Deployment lifecycle: warm, serve, drain

### 18.1 Safe scale-out/deployment

```text
1. start process
2. load config/models/caches and connect to dependencies
3. pass readiness
4. register as target
5. slow-start/ramp traffic
6. observe error, latency, saturation
7. reach full weight
```

Registering before readiness creates user failures. Giving a cold instance full least-connections traffic creates a cold-start herd.

### 18.2 Safe removal

```text
1. mark target draining / deregister
2. stop assigning new work
3. let in-flight work and connections finish within a limit
4. close/notify long-lived connections as protocol permits
5. stop process only after drain or timeout policy
```

[AWS calls this](https://docs.aws.amazon.com/elasticloadbalancing/latest/APIReference/API_DeregisterTargets.html) **deregistration delay/connection draining** and documents that new requests stop while in-flight traffic is allowed to complete. The timeout must reflect the workload; a five-second API and a two-hour stream cannot use the same drain assumption.

### 18.3 Canary through weights or routes

```text
v1 pool: 99%
v2 pool:  1%
```

Monitor correctness, latency, errors, and saturation before increasing v2. Weighted traffic alone does not guarantee representative tenants/endpoints; use shadowing or stratified routing carefully when needed.

### 18.4 Recovery slow start

A recovered target may have empty caches, JIT warm-up, new database pools, or model loading. Ramp its effective weight gradually. Otherwise least-connection and round-robin policies can immediately overload the apparently empty instance.

## 19. Observability and operations

### 19.1 What to measure at the LB

By listener, route, pool, zone, target, protocol, and status category:

- requests/connections/bytes per second;
- active connections and outstanding requests;
- accepted, rejected, queued, retried, and timed-out work;
- LB processing latency, backend connect time, TTFB, and full response time;
- 4xx, 5xx, resets, TLS failures, and protocol errors;
- healthy/unhealthy/draining target count;
- target selection distribution and configured/effective weights;
- p50/p95/p99 latency, not only averages;
- CPU, memory, file descriptors, network, and connection-table capacity of the LB tier; and
- config version and rollout status on every data-plane node.

### 19.2 Detect algorithm mismatch

Examples:

- Equal request counts but unequal CPU -> requests have unequal cost.
- Equal connection counts but unequal outstanding streams -> multiplexing makes connection count misleading.
- Least-time target choices oscillate -> feedback too reactive/noisy.
- One source-IP hash target is hot -> shared NAT or hot client.
- New instance gets a spike -> missing slow start.
- Configured `2:1` weights yield `10:1` active sessions -> stickiness or connection duration dominates.

### 19.3 Tail latency matters

Averages can look healthy while a small fraction of users time out. Compare:

- client-observed latency;
- LB time before upstream selection;
- connect/TLS time to backend;
- backend TTFB;
- full-body/stream completion; and
- retry attempts.

This separates a slow balancer, saturated backend, network problem, and slow client.

### 19.4 Test failures, not just happy throughput

Run controlled tests for:

- one target failure;
- one LB-node failure;
- zone loss;
- all health checks failing;
- control-plane outage with last-known config;
- slow rather than dead target;
- connection/retry storm;
- bad certificate/config rollout;
- draining long-lived connections; and
- load above total backend capacity.

## 20. Real-system examples

Public architecture changes over time. Each example states whether it is public evidence or an illustration.

### 20.1 WhatsApp: persistent client connections

**Public fact:** [Meta has described](https://engineering.fb.com/2024/03/06/security/whatsapp-messenger-messaging-interoperability-eu/) WhatsApp clients connecting to its servers using a client/server protocol and Meta servers pushing messages to connected clients over persistent connections. It has also [described companion devices connecting independently](https://engineering.fb.com/2021/07/14/security/whatsapp-multi-device/) in its multi-device architecture.

**Illustrative load-balancing design:**

```text
Phone / companion devices
       |
       | persistent encrypted connections
       v
global/edge traffic tier
       |
       v
connection gateways A/B/C
       |
       +--> presence/connection directory
       +--> durable message routing/storage
```

Interview implications:

- L4 or edge flow balancing can select a connection gateway.
- Least connections is better than blind round robin only if open connections approximate resource use.
- Messages/bytes and reconnect rates are also necessary because connections vary in activity.
- A live connection cannot be reassigned request-by-request; on failure, the client reconnects.
- Durable messages must not exist only in gateway memory.
- Multi-device means one account can have several active connections, so source-IP affinity is the wrong identity model.

This diagram is illustrative; the cited Meta posts do not disclose WhatsApp's complete load-balancing algorithm.

### 20.2 YouTube: two different workloads

**Illustrative YouTube-style design:** metadata/API traffic and video delivery need different policies.

```text
                        +--> L7 API LB --> metadata/search/user services
Viewer -> edge/global --+
                        +--> CDN/video edge selected by location, health,
                             content availability, capacity, and network conditions
```

For API calls, per-request L7 balancing may work. For video segments/streams:

- bandwidth and cache locality may matter more than request count;
- TTFB alone does not measure full segment delivery;
- sending the same content toward a warm cache can beat perfectly even traffic;
- global edge selection and local server selection are separate layers; and
- one viral video is a hot key/content problem, not solved by round robin alone.

This is an illustrative design, not a claim about YouTube's current private routing. [Google's published Maglev system](https://research.google/pubs/maglev-a-fast-and-reliable-software-network-load-balancer/) is a real example of a distributed L4 load balancer serving Google traffic, but the paper does not establish a YouTube-specific algorithm.

### 20.3 Uber: L7 edge and service-mesh balancing

**Public production example:** [Uber described](https://www.uber.com/en-CA/blog/eng-failover-handling/) front-end edge proxies terminating TLS over TCP or QUIC and forwarding HTTPS traffic to backend services. It also [described an on-host L7 service-mesh proxy](https://www.uber.com/blog/better-load-balancing-real-time-dynamic-subsetting/) that discovers backend tasks, excludes unhealthy tasks, and performs load balancing.

```text
Mobile app ==TLS/QUIC==> edge proxy
                             |
                             v
                       nearest data center
                             |
service A -> on-host L7 proxy -> discovered healthy tasks of service B
```

Uber's public dynamic-subsetting work highlights a less obvious scale issue: if every caller proxy connects to every backend, connection count grows multiplicatively. Proxies therefore balance over subsets, trading perfect global choice for manageable connection/state cost. The company described adjusting subset size using near-real-time aggregate traffic reports.

Interview lesson: "least connections across every server" can be impractical when there are thousands of callers and targets. Discovery fan-out and connection-pool size are part of the load-balancer design.

### 20.4 Netflix: Zuul and software load balancing

**Public production/project snapshot:** [Netflix's Zuul documentation](https://github.com/netflix/zuul/wiki/How-We-Use-Zuul-At-Netflix) says outbound calls used Ribbon for software load balancing and even distribution, with filters/metrics and dynamic routing. [A Netflix TechBlog article](https://medium.com/netflix-techblog/netflix-edge-load-balancing-695308b5548c) described historically using Ribbon round robin plus filtering/blacklisting for connection failures, then evolving edge load balancing to react better to slow origins.

```text
Netflix device -> edge/ELB -> Zuul fleet
                                |
                         routing + resilience
                                |
                         origin service pool
```

Lessons:

- Round robin is a sound baseline, not a permanent answer when origins become slow rather than fully dead.
- Health/failure filtering is orthogonal to the scheduling order.
- Rich L7 gateways can route a customer/device to a special cluster for debugging or canary/stress testing.
- These sources are dated architecture snapshots; do not claim this is Netflix's complete current stack in 2026.

### 20.5 Twitter/X: power of two choices

**Public production example:** [Twitter/X Engineering described](https://blog.x.com/engineering/en_us/topics/infrastructure/2019/daperture-load-balancer) Finagle embedding a client-side load balancer in every client and using **power of two choices**: choose two instances randomly and select the one with less load, commonly fewer outstanding requests.

```text
Finagle client
   |
   +-- random candidate A: 9 outstanding
   +-- random candidate B: 3 outstanding  <-- choose B
```

The post explains why a small random comparison can distribute requests well without globally scanning and coordinating every replica. It also discusses **aperture/subsetting** because all-to-all client-to-server sessions become expensive.

Interview lesson: sophisticated distributed systems often prefer a good local approximate choice over an expensive, stale global minimum.

### 20.6 Amazon/AWS Elastic Load Balancing

**Verified product behavior:** [AWS currently documents](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/how-elastic-load-balancing-works.html) different behavior by load-balancer type:

- Application Load Balancer evaluates L7 listener rules, then uses a target-group algorithm such as round robin, least outstanding requests, or weighted random.
- Network Load Balancer selects a target using a flow hash and keeps an individual TCP connection on one target for the connection's lifetime.
- ELB health checks determine eligible targets, and target deregistration supports draining.

```text
                    +--> ALB: HTTP rules + per-target request policy
AWS client traffic -+
                    +--> NLB: transport flow hash + connection affinity
```

Interview lesson: "AWS load balancer" is not one behavior. State the protocol and product class; L4 and L7 choose at different units and expose different policies.

### 20.7 Amazon shopping-style request: illustrative end-to-end example

```text
Customer
   |
global edge / DDoS / CDN
   |
L7 application load balancer
   |  /catalog -> catalog target group (cache-heavy reads)
   |  /cart    -> cart target group
   |  /orders  -> order target group (idempotency required)
   v
stateless services -> caches/databases/queues
```

Possible choices:

- round robin for similar stateless catalog API instances;
- least outstanding requests if some catalog requests are much slower;
- no source-IP session dependency: cart/session state is shared;
- weighted route for an order-service canary;
- no automatic retry of an ambiguous order write without an idempotency key;
- readiness, slow start, and drain on every deployment.

This is illustrative and is not a claim about Amazon.com's private current architecture.

## 21. Worked design: choose a balancer for an API plus WebSockets

### Requirements

- Public HTTPS JSON API.
- `/chat/connect` upgrades to a long-lived WebSocket.
- Other API requests are short and stateless, but report generation varies from 50 ms to 5 s.
- Backends run in three availability zones.
- Deploy without dropping ordinary requests; clients can reconnect sockets.

### Design

```text
                           +--> API pool A/B/C
Client -> redundant L7 LB +      least outstanding requests
       TLS termination    |
                           +--> Gateway pool G1/G2/G3
                                  least connections + hard limits
```

Reasoning:

1. **L7** is needed to route API paths and WebSocket upgrades, terminate TLS, enforce auth/rate limits, and observe HTTP status.
2. For ordinary APIs, **least outstanding requests/P2C** reacts to long reports better than connection-based selection.
3. For new WebSockets, **least connections** is a starting point, but also cap connections and monitor messages/bytes/CPU because socket activity differs.
4. Keep session and undelivered-message state outside gateway memory; the connection itself remains on one gateway.
5. Spread targets across zones and retain failure headroom. Decide whether normal traffic stays zone-local or cross-zone.
6. Use shallow readiness plus separate dependency signals; avoid evicting the whole pool solely because one shared dependency blips.
7. On deploy, mark draining, stop new API/WebSocket assignments, finish requests, notify/close long sockets within a bounded window, then terminate.
8. Do not retry non-idempotent writes blindly; propagate deadlines and idempotency keys.

### Alternatives

- An outer L4 tier can distribute connections across a fleet of L7 proxies.
- If TLS must remain end-to-end to gateways, use L4 passthrough for the WebSocket endpoint and a separate L7 API endpoint.
- If report generation should not occupy request workers, make report creation asynchronous through a queue and balance worker jobs separately.

### 22.3 Answer spine: **BALANCE**

Use these cues to reconstruct an answer live instead of memorizing a paragraph.

#### B — Bound the workload

- Public or internal? One region or global?
- TCP/UDP, HTTP/HTTPS, gRPC, or WebSocket?
- Selection unit: connection, request, job, or key?
- Short uniform work or long/variable work?
- Required availability, latency, throughput, and failure capacity?

#### A — Application visibility: L4 or L7

- L4 for opaque/high-throughput flow distribution.
- L7 for host/path/header/request policy and TLS termination.
- Say whether one connection contains many requests/streams.

#### L — Live eligible target set

- Discovery/registration and readiness.
- Active/passive checks, thresholds, and hysteresis.
- Warming, unhealthy, draining, fail-open/fail-closed behavior.

#### A — Algorithm tied to the load signal

- RR for comparable work/targets.
- Weighted RR for predictable capacity differences/canary.
- Least connections for long-lived connection occupancy.
- Least outstanding/P2C for variable request duration.
- Latency-aware only with trustworthy smoothed metrics.
- Hash only for an explicit affinity/locality need.

#### N — No single point of failure or hidden state

- Multiple LB nodes and failure headroom.
- Session state externalized; affinity trade-off if unavoidable.
- Flow state behavior when an LB or target dies.
- Multi-zone/region policy.

#### C — Correctness and connection lifecycle

- TLS termination/re-encryption/passthrough and client identity.
- Timeouts, safe/idempotent retries, overload/backpressure.
- Register, slow-start, drain, deregister, terminate.

#### E — Evidence in production

- Per-target traffic, outstanding work, latency, error, saturation.
- Health and configuration versions.
- Load-skew and failure tests.
- Explain what would make you change the algorithm.

### 22.4 Example 90-second interview answer

> "First I would clarify whether we are balancing TCP connections, HTTP requests, or background jobs, because that determines the unit and useful load signal. For a public stateless HTTP API, I would use a redundant managed L7 tier so it can terminate TLS, route by host/path, and balance each request across a healthy target group. I would begin with round robin if instances and request costs are similar; if duration is highly variable, I would use least outstanding requests or power-of-two choices rather than least TCP connections, especially with HTTP/2 multiplexing. Targets enter only after readiness, ramp with slow start, and leave through connection draining. I would keep user sessions in a shared store instead of IP affinity, because NAT, IP changes, and failures create skew. The LB tier itself spans zones with failure headroom. I would bound timeouts and retries, require idempotency for ambiguous writes, and shed load when all targets are saturated. Finally I would monitor per-target request share, outstanding work, TTFB/full latency, errors, saturation, health transitions, and config versions to verify that the chosen policy actually balances the limiting resource."

### 22.5 Synthesized follow-ups and answer cues

The following are **my own plausible interview synthesis**, not sourced interview reports.

#### "Why is L4 faster than L7?"

- Less protocol termination/parsing/policy and potentially packet-forwarding/DSR.
- Do not promise a universal latency/throughput number; implementation and hardware dominate.
- L4 loses content-aware routing and request-level balancing.

#### "Round robin gives equal requests. Why can a server still overload?"

- Request cost/duration/bytes differ.
- L4 connections carry different numbers of requests.
- Targets may have different capacity or noisy neighbors.
- Use metrics, weights, least outstanding, workload separation, or admission control.

#### "Why is least connections weak for HTTP/2?"

- One TCP connection can contain many concurrent streams.
- Count outstanding streams/requests at L7 if visible.
- For opaque L4 traffic, accept connection-level granularity or change architecture.

#### "The load balancer dies. What happens?"

- A single LB is a SPOF; use redundant nodes/VIP/anycast/DNS/outer tier.
- New connections move to surviving nodes.
- Existing proxy-terminated connections on the failed node usually reset unless stateful failover exists.
- Clients reconnect with backoff; retain capacity for reconnect storm.

#### "What should `/health` check?"

- Readiness to serve real local work, not merely process existence.
- Avoid making every instance unready solely because one shared dependency blips unless fail-closed is intended.
- Separate shallow readiness from detailed dependency telemetry.
- Use thresholds/hysteresis and slow recovery.

#### "How do you deploy without dropping requests?"

- Start and warm, pass readiness, register, slow-start.
- For old node: drain new assignments, finish in-flight work, handle long streams within policy, then terminate.
- Align orchestrator grace period with LB drain timeout.

#### "Should the LB retry a failed POST?"

- Failure may occur after side effect committed.
- Only retry if safe/idempotent or protected by an idempotency key.
- Use one deadline/retry budget; avoid layered amplification.

#### "Why not always use least response time?"

- Noisy/stale samples, mixed request types, herding, oscillation, measurement cost.
- TTFB may not represent completion/resource cost.
- Simple RR/P2C may be more robust and cheaper.
- Choose based on observed SLO and imbalance.

#### "How do sticky sessions affect scaling?"

- New capacity gets only new/remapped sessions; old targets retain hot users.
- Failure loses/remaps local session state.
- Prefer shared state; if affinity is required, monitor skew and define fallback.

#### "All targets are unhealthy. What should the LB do?"

- Explicit fail-open versus fail-closed decision based on risk.
- Possibly fail over, serve degraded response, or shed load.
- Protect against a bad global health dependency.
- Alert on zero healthy capacity and test the path.

#### "How would you design the load balancer itself for 20,000 machines?"

- Clarify network LB versus task dispatcher and global versus regional.
- Shard/subset target membership; avoid every node probing every target naively.
- Versioned discovery/health/control plane; data plane keeps last good config.
- Multiple stateless scheduling nodes where possible; partition flow state where necessary.
- P2C avoids global scans; regional hierarchy reduces scope.
- Durable/TTL-based exclusions, concurrent API semantics, metrics, and failure headroom.

#### "What happens to an in-flight request when its target fails?"

- It may reset/timeout; the LB cannot teleport partially executed application state.
- Retry only under correctness rules.
- For streaming/WebSocket, reconnect/resume must be application-protocol behavior.
- Health checks prevent future assignments after detection; detection is not instantaneous.

## 23. Common mistakes and interviewer red flags

1. **Drawing one LB and calling the system highly available.** The LB tier needs redundancy and failure capacity.
2. **Saying L4 uses only IP and L7 uses URLs.** L4 commonly uses protocol, IPs, ports, and flow state; L7 adds application semantics.
3. **Ignoring connection versus request.** This breaks reasoning for keep-alive, HTTP/2, gRPC, and WebSockets.
4. **Claiming round robin balances CPU.** It balances assignments, not necessarily work.
5. **Using IP hash as user identity.** NAT and address changes invalidate that assumption.
6. **Calling stickiness availability.** Affinity can increase coupling to one server.
7. **Choosing least response time without defining response time.** TTFB, headers, full body, window, failures, and request class matter.
8. **Presenting `TTFB x active` as universal.** It is one documented historical product formulation.
9. **Treating a `200 /health` as proof the application works.** Health design has false-positive and correlated-failure trade-offs.
10. **Removing a server before draining.** Users receive resets/5xx even though the new version is healthy.
11. **Retrying every failed request.** Ambiguous writes duplicate side effects and overload creates retry storms.
12. **Assuming the scheduler fixes overload.** Once total demand exceeds capacity, use admission control, shedding, queues, or more capacity.
13. **Ignoring warm-up.** New/recovered zero-load targets attract a herd.
14. **Equating a load balancer with an API gateway or CDN.** Responsibilities overlap, but selection, API policy, and global caching/delivery are distinct concerns.
15. **Giving product defaults as universal facts.** Health thresholds, fail-open behavior, algorithms, and drain timeouts vary and change.

## 24. Compact revision sheet

### 24.1 One mental model

```text
Discover targets
   -> filter healthy/ready/not-draining
   -> choose L4 flow or L7 request visibility
   -> select signal/algorithm
   -> forward with bounded lifecycle policy
   -> observe and correct
```

### 24.2 Algorithm memory cues

| Algorithm | Say this to yourself |
|---|---|
| Round robin | "Equal turns, not equal work." |
| Weighted RR | "Turns proportional to measured capacity." |
| IP hash | "Approximate source affinity, NAT and remapping risk." |
| Least connections | "Fewest open flows; good when flow lifetime matters." |
| Weighted least connections | "Occupancy normalized by expected capacity." |
| Least outstanding | "Fewest unfinished application requests." |
| Least response time | "Recent speed plus load, but feedback can lie." |
| `TTFB x active` | "One Citrix formulation, not a universal definition." |
| P2C | "Compare two random choices; cheap and surprisingly balanced." |

### 24.3 Formula card

These are conceptual unless marked product-specific:

```text
IP hash:
  target = targets[hash(source_ip) mod N]

Conceptual weighted least connections:
  normalized_load_i = active_connections_i / weight_i

Historical Citrix least response time:
  score_i = active_transactions_i x average_TTFB_i
  weighted_score_i = score_i x (10000 / weight_i)

Envoy documented weighted least-request effective weight:
  effective_weight = configured_weight / (active_requests + 1)^bias
```

Do not mix formulas from different products as though they were one standard.

### 24.4 Design checklist

- [ ] What protocol and selection unit am I balancing?
- [ ] Do I need L4 opacity/efficiency or L7 request visibility?
- [ ] Where does TLS terminate, and how is the backend leg protected?
- [ ] What is the backend pool and how is membership discovered/versioned?
- [ ] What does healthy and ready mean?
- [ ] What happens on one failure, all-unhealthy, and slow-not-dead behavior?
- [ ] Which load signal matches the constrained resource?
- [ ] Are targets equal capacity; do I need weights and slow start?
- [ ] Are sessions/statelessness/affinity explicit?
- [ ] How do HTTP/2, gRPC, WebSocket, or streaming change granularity?
- [ ] Is the LB tier redundant across failure domains with headroom?
- [ ] Are timeouts, retries, idempotency, queues, and shedding bounded?
- [ ] How are targets registered, canaried, drained, and removed?
- [ ] Which per-target/per-route metrics verify the policy?
- [ ] Have I tested LB-node, target, zone, health, control-plane, and overload failures?
