# The Dual Write Problem: Transactional Outbox, CDC, and Listen to Yourself

> Beginner-first HLD notes expanded from the supplied bullets. The goal is to recognize when one logical action secretly contains two independently failing writes, then choose a source of truth and a recovery design instead of relying on timing, retries, or hope.

---

## 0. Scope, assumptions, corrections, and evidence labels

### Assumptions

- No prior distributed-systems background is assumed. Local transactions, message delivery, ordering, idempotency, 2PC, and CDC are introduced inline.
- The central example is a service that must update its database and publish an event, but the same reasoning applies to database + cache, database + search index, database + object store, or two service databases.
- Product examples involving WhatsApp, YouTube, Uber, Netflix, X/Twitter, and Amazon are **illustrative interview designs**, not claims about those companies' current private architectures.
- Product and version-sensitive claims were checked on **18 July 2026**. No universal latency, retry count, poll interval, retention period, or throughput limit is asserted.

### Important corrections to the rough bullets

1. **Event-driven architecture by itself is not a solution.** `write database -> publish event` is still a dual write. A pattern such as transactional outbox, broker-first/listen-to-yourself, event sourcing, or direct CDC must define the atomic source of truth.
2. **2PC is not universally bad, and its coordinator is not necessarily an unrecoverable single point of failure.** A production coordinator can persist and replicate its decision. The deeper problem is that prepared participants may block when they cannot learn the decision, while latency, locks, availability, protocol support, and operational coupling make 2PC unattractive for many service/broker combinations.
3. **“Does the broker support single-phase commit?” is not the key question.** A broker can commit its own write in one local transaction, but that does not atomically commit a separate database. The relevant question is whether *both resources* can join the same distributed transaction protocol and whether its trade-offs are acceptable.
4. **Publisher ID + sequence number is a Kafka implementation mechanism, not the complete application idempotency design.** Consumers still need a stable application `event_id`; ordering usually needs an `aggregate_id` and `aggregate_version`.
5. **Multiple pollers can reduce throughput bottlenecks, but row claims do not eliminate every duplicate.** A relay can publish successfully and crash before marking the row published. Consumer-side idempotency remains necessary.
6. **Kafka Streams does not automatically repair events that parallel publishers appended in the wrong business order.** Kafka preserves the append order inside a partition. Streams has tools for event-time, late records, windows, and versioned state, but the application must define what ordering means and how to handle gaps or stale versions.
7. **The outbox already is the durable retry record.** If publishing fails, the relay normally leaves/requeues the outbox row and retries with backoff and jitter. Adding another durable database without defining a new atomic boundary just creates another dual write.
8. **Transaction-log tailing does not always mean “no outbox table.”** The canonical transaction-log-tailing pattern reads inserts from an outbox through the WAL/binlog. Direct CDC of business tables can avoid an outbox, but it emits physical row changes and creates stronger schema/filter coupling.
9. **Database logs are not inherently slow.** CDC often tails them continuously and can avoid polling delay. End-to-end latency is deployment-dependent and can grow because of connector lag, broker backpressure, large transactions, failures, or an undersized pipeline.
10. **A write-through cache does not magically restore read-your-writes for listen-to-yourself.** At most, a cache can expose a provisional `PENDING` intent. It must not pretend the asynchronous database mutation has committed successfully.

### Evidence labels

- **Verified technical behavior:** supported by first-party documentation, a standard, or a canonical pattern source linked nearby.
- **Candidate-reported interview experience:** a public self-report; useful evidence of prompt style, not a company-confirmed question bank.
- **Interviewer-authored report:** someone publicly says they asked candidates this topic; the claim is not independently audited.
- **Illustrative product design:** a plausible HLD for a familiar product, not a claim about its private implementation.
- **Deployment-dependent:** measure or verify in the selected database, connector, broker, and failure model.

# Part I — Foundation

## 1. The plain-language problem

### 1.1 One order, two writes

Suppose an Order Service must do two things when checkout succeeds:

1. save order `O-42` in PostgreSQL; and
2. publish `OrderCreated(O-42)` to a broker so Inventory and Email can react.

The tempting code is:

```text
saveOrder(order)                 // write A: database
publish(OrderCreated(order.id))  // write B: broker
```

Each line can succeed independently. If the process crashes between them:

```text
Client          Order Service          Database          Broker
  |                  |                     |                |
  |-- create O-42 -->|                     |                |
  |                  |-- INSERT O-42 ----->|                |
  |                  |<-- COMMIT OK --------|                |
  |                  X process crashes                       |
  |                                                        no event
```

The database says the order exists, but Inventory never learns about it. The first write is durable and the second never happened.

Reversing the lines merely reverses the failure:

```text
publish OrderCreated -> succeeds
process crashes
save order            -> never happens
```

Now consumers may reserve stock or send confirmation for a **ghost order** that is absent from the database.

### 1.2 Definition

A **dual write** occurs when one logical business operation must change two independently committing resources, but there is no atomic boundary covering both.

```text
one business intent
       |
       +--> resource A commits independently
       |
       +--> resource B commits independently
```

“Dual” describes the consistency problem, not necessarily exactly two code statements. One request that updates five services has several related dual-write boundaries.

### 1.3 It is broader than database + Kafka

Common forms include:

| First resource | Second resource | Possible inconsistency |
|---|---|---|
| Primary database | Message broker | State changed but event missing, or phantom event |
| Primary database | Redis cache | Database and cache expose different values |
| Primary database | Search index | Search misses an existing object or returns a deleted one |
| Metadata database | Object storage | Metadata points to no object, or an orphaned object has no metadata |
| Order database | Inventory database | Order accepted without stock reservation |
| Database | Email/payment/external API | Local state and irreversible external effect disagree |

A monolith can have a dual-write bug. Microservices make it more common, but network and transaction boundaries—not deployment count—create the problem.

## 2. Why ordinary error handling cannot make it atomic

### 2.1 The failure matrix

For `A then B`:

| Outcome | A | B | Result |
|---|---:|---:|---|
| A fails | failed | not attempted | Usually safe to return failure |
| A succeeds, B succeeds | committed | committed | Desired result |
| A succeeds, process/network fails before B | committed | absent | Partial state |
| A succeeds, B commits, reply is lost | committed | unknown to caller | Retry may duplicate B |
| A succeeds, B definitively rejects | committed | failed | A needs repair, compensation, or reconciliation |

The dangerous word is **unknown**. A timeout means “I did not receive a conclusive response,” not “the remote write definitely failed.”

### 2.2 Parallel calls do not help

Starting A and B simultaneously reduces elapsed time but not the inconsistency space:

```text
          +--> A succeeds
request --|
          +--> B fails
```

The caller still cannot undo an already committed independent effect atomically.

### 2.3 A retry is a recovery policy, not atomicity

Retries improve the chance that B eventually succeeds, but introduce new questions:

- What if B succeeded and only its acknowledgement was lost?
- Is B idempotent under the same logical operation ID?
- How long can retries continue?
- Where is the retry intent stored if the process restarts?
- What happens while B is unavailable for hours?
- Who detects a permanently invalid message?

An in-memory retry loop disappears with the process. A durable retry record is needed.

### 2.4 “Roll back A” is another distributed write

Calling A again to compensate can also fail or race with other activity:

```text
A commits -> B fails -> compensate A
                         ^ this is a third fallible operation
```

Compensation may be the correct business design, but it creates an explicit eventually consistent workflow; it does not recreate an invisible ACID rollback.

## 3. Inline prerequisites

These are required to understand the solutions, so they are explained now rather than deferred to “study next.”

### 3.1 Local database transaction

A local transaction groups changes controlled by one transactional database:

```sql
BEGIN;
INSERT INTO orders (...);
INSERT INTO audit_rows (...);
COMMIT;
```

If both tables are inside the same supported transaction boundary, the database commits both or neither. This atomicity is the key tool used by the transactional outbox.

### 3.2 Broker acknowledgement and redelivery

A producer normally sends a message and waits for the broker's configured acknowledgement. A consumer later processes it and acknowledges it or commits an offset. Failures around either acknowledgement are ambiguous, so many reliable messaging designs are **at least once**: a message should not be lost, but it can appear more than once.

Therefore:

```text
reliable producer side -> do not lose the event intent
reliable consumer side -> tolerate duplicate delivery
```

See [Messaging Queues](<./15. Messaging Queues.md>) for the full broker foundation.

### 3.3 Idempotency

An idempotent handler makes retrying the same logical message have the same intended effect as handling it once. A consumer might record `(consumer_name, event_id)` under a unique constraint, or enforce the business transition with a unique operation ID/version.

Idempotency does **not** automatically fix out-of-order distinct events. `OrderCancelled(v3)` and `OrderPaid(v2)` have different IDs; deduplicating either does not tell the consumer which state is newer.

See [Idempotency](<./13. Idempotency.md>) for request keys, ambiguous outcomes, and atomic claims.

### 3.4 Ordering scope

“Keep events ordered” is incomplete. Ask:

- global order across every event?
- order per user, order, account, trip, or conversation?
- order within one database transaction?
- event-time order or broker append order?

Most systems need **per-aggregate order**, not one global serial lane. An aggregate is a business entity whose changes must be consistent together, such as one order or account.

Typical envelope fields are:

```json
{
  "event_id": "evt-7f41",
  "event_type": "OrderPaid",
  "aggregate_type": "Order",
  "aggregate_id": "O-42",
  "aggregate_version": 7,
  "schema_version": 3,
  "occurred_at": "2026-07-18T10:30:00Z",
  "correlation_id": "checkout-91",
  "payload": {}
}
```

`event_id` supports deduplication. `aggregate_id` is often the broker partition key. `aggregate_version` lets a consumer reject stale events or notice a gap.

### 3.5 Source of truth

The **source of truth** is the authoritative durable record from which other state can be rebuilt or reconciled.

The core design decision is:

```text
Which single durable commit proves the business intent was accepted?

Database-first/outbox: database transaction is authoritative
Broker-first/listen-to-yourself: broker event is authoritative intent
Event sourcing: event log is authoritative history
```

Without a clear answer, every copy may claim to be authoritative when they disagree.

# Part II — Why 2PC is usually not the default answer

## 4. Two-Phase Commit in one minute

Two-Phase Commit (2PC) coordinates participants that implement a compatible prepare/commit protocol:

```text
Phase 1: PREPARE
Coordinator -> DB:     can you commit?
Coordinator -> Broker: can you commit?
Participants durably promise YES or answer NO

Phase 2: DECIDE
all YES -> coordinator records and sends COMMIT
any NO  -> coordinator records and sends ABORT
```

After a participant votes YES, it may need to retain locks/resources and wait for the final decision. It cannot safely guess if the decision is unavailable.

## 5. Why 2PC is often a poor fit for service + broker dual writes

| Concern | Technical meaning |
|---|---|
| Participant support | Both resources and client stack must support a compatible distributed transaction mechanism; heterogeneous cloud services often do not. |
| Blocking | A prepared participant can be unable to finish while the durable decision is unreachable. |
| Latency | Prepare and decision communication, forced durable records, and lock duration add work; exact cost is deployment-dependent. |
| Availability coupling | The operation's success now depends on all enlisted participants and coordination infrastructure. |
| Resource retention | Prepared transactions may hold locks, connections, or versions until resolved. |
| Operational recovery | In-doubt transactions, coordinator recovery, heuristic outcomes, and monitoring need explicit runbooks. |
| Service autonomy | A cross-service transaction couples deployment, failure, and transaction semantics across owners. |

### Correction: “the coordinator is a single point of failure”

A naive in-memory coordinator is a single point of failure. A real coordinator can durably log and replicate decisions, recover, and resend them. Even then, temporary loss of the decision path can leave prepared participants blocked. So the interview-quality statement is:

> 2PC requires a durable, highly available coordinator, but coordinator/communication failures can still reduce progress and keep prepared participants waiting; this is an availability and operational concern, not necessarily permanent data loss from one unreplicated machine.

### Correction: broker-local transactions are not cross-system transactions

Kafka transactions can atomically cover supported operations inside Kafka, such as consuming from Kafka and producing to Kafka with offsets. That does not automatically include an arbitrary external database write. Likewise, a database transaction does not include a broker call merely because it occurs inside an annotated application method.

The right question is:

```text
What exact resources are inside the transaction boundary?
```

### When 2PC can still be reasonable

Consider it when all participants genuinely support the same mechanism, the transaction is short, strong atomic visibility is required, lock/availability costs are acceptable, and the organization can operate recovery safely. Do not reject it only because “microservices never use 2PC,” and do not select it only because the word “transaction” appears.

For deeper protocol behavior, see [Distributed Transactions](<./19. Distributed Transactions.md>).

# Part III — Event-driven architecture and the solution map

## 6. Event-driven architecture is the transport style, not the atomicity mechanism

In an event-driven system, a producer records that something happened and independent consumers react asynchronously:

```text
OrderCreated ---> Inventory consumer
             +--> Notification consumer
             +--> Analytics consumer
```

This decouples availability and timing, but the event must first be created reliably.

```text
BROKEN:
update orders table -> publish OrderCreated
        database             broker
             still two independent commits

RELIABLE DATABASE-FIRST:
one DB transaction: update order + insert outbox event
then relay the durable event asynchronously
```

Event-driven architecture therefore **exposes the need** for a reliable publication pattern; it is not sufficient on its own.

## 7. The solution families

| Approach | Authoritative first commit | Main trade-off |
|---|---|---|
| Keep both changes in one local transaction | One database | Strongest/simple, but requires one ownership boundary |
| 2PC/distributed transaction | Coordinated global decision | Strong atomicity with support, latency, blocking, and coupling costs |
| Transactional outbox + polling | Business DB transaction | Portable and understandable; polling load, delay, relay plumbing |
| Transactional outbox + CDC | Business DB transaction/WAL | Low-lag relay and no query polling; CDC infrastructure and DB coupling |
| Direct CDC of business tables | Business DB transaction/WAL | No outbox table; physical changes are not clean domain events |
| Listen to yourself/broker-first | Broker record | No DB+broker producer dual write; local state and API confirmation become asynchronous |
| Event sourcing | Event-store append | Complete history/replay; larger modeling and operational commitment |
| Saga/durable workflow | Sequence of local commits | Cross-service business progress and compensation, not one ACID commit |
| Reconciliation | One chosen authority plus repair | Accepts a window and detects/repairs divergence |

The rest of the chapter focuses on the three solutions in the supplied bullets.

# Part IV — Transactional Outbox Pattern

## 8. Plain-language idea

Do not try to atomically write the database and broker. Instead, atomically write the business change and a **promise to publish** inside the same database:

```text
Request
   |
   v
Order Service
   |
   | one local transaction
   +------> orders table: O-42
   +------> outbox table: OrderCreated(O-42)
                  |
                  | after commit
                  v
             Relay/Poller ------> Broker ------> Consumers
```

If the transaction rolls back, neither row exists. If it commits, the event intent is durable even if the service crashes immediately afterward.

## 9. Step-by-step flow

```sql
BEGIN;

INSERT INTO orders(order_id, customer_id, status, version)
VALUES ('O-42', 'C-9', 'CREATED', 1);

INSERT INTO outbox_events(
    event_id, aggregate_type, aggregate_id, aggregate_version,
    event_type, schema_version, payload, occurred_at, status
) VALUES (
    'evt-7f41', 'Order', 'O-42', 1,
    'OrderCreated', 1, '{...}', CURRENT_TIMESTAMP, 'PENDING'
);

COMMIT;
```

Then, independently:

1. a relay finds committed `PENDING` rows;
2. it publishes the event to the broker;
3. it receives the broker acknowledgement;
4. it marks the row `PUBLISHED` or records a durable publication checkpoint; and
5. cleanup removes/archives old completed rows later.

The API can acknowledge the order after step 1's database transaction commits. Downstream effects are eventually consistent.

## 10. What the outbox guarantees—and what it does not

### It provides

- atomicity between the local business mutation and creation of its event intent;
- no committed event intent for a rolled-back business change;
- a durable source from which publication can resume after process/broker failure;
- eventual publication **if** the relay, broker, retry policy, and operations eventually make progress; and
- a clean place to monitor publication backlog.

### It does not provide automatically

- exactly-once delivery or exactly-once business effects;
- global ordering;
- immediate downstream consistency;
- atomicity across the consumer's database and any next event it publishes;
- business compensation across several services;
- correct event schemas, authorization, or PII handling; or
- liveness if an invalid event blocks a partition forever.

The common end-to-end target is:

```text
transactional outbox + at-least-once publication + idempotent consumers
                    + per-aggregate ordering/version rules + reconciliation
```

## 11. A production-oriented outbox schema

For a polling relay, useful fields include:

| Field | Purpose |
|---|---|
| `event_id` | Stable globally unique application event identity |
| `aggregate_type`, `aggregate_id` | Business owner and partition/routing key |
| `aggregate_version` | Per-entity causal order/gap detection |
| `event_type`, `schema_version` | Consumer routing and compatible evolution |
| `payload` | Immutable event body or a reference to it |
| `occurred_at` | Business event time; not sufficient alone for ordering |
| `status` | `PENDING`, `CLAIMED`, `PUBLISHED`, `FAILED`/`QUARANTINED` as designed |
| `available_at`, `attempt_count` | Scheduled retry and retry accounting |
| `lease_owner`, `lease_expires_at` | Recovery from a crashed claimant |
| `published_at`, `last_error` | Operations, audit, and cleanup |

For CDC, an **append-only** outbox is often simpler: insert the immutable event once and let the connector's source offset/checkpoint track progress. Debezium's outbox router expects inserts as the normal outbox operation and uses a unique event ID plus aggregate key; updates and deletes need deliberately configured handling.

Do not put giant blobs into every event. Store large content in object storage and publish a durable identifier, version/checksum, and access-controlled reference—but remember that creating the object and metadata can itself introduce a dual write requiring staging/reconciliation.

## 12. Challenge: “Do brokers support single-phase commit?”

The outbox deliberately relies on **one local database commit** for the authoritative state and event intent. Later broker publication is not in that transaction.

A successful broker acknowledgement means “the broker accepted this record under its configured durability semantics.” It does not mean the relay's `PUBLISHED` database update committed. There is still a small ambiguous window:

```text
relay publishes -> broker durably accepts -> relay crashes before marking row
```

After restart, the relay sees `PENDING` and publishes again. That is why the correct contract is normally at-least-once, not “two local one-phase commits somehow equal one atomic commit.”

## 13. Challenge: extra relay/poller effort

A robust polling relay is a small durable work-queue system. It needs:

- safe row claiming across instances;
- batching and backpressure;
- bounded retry with exponential backoff and jitter;
- leases or recovery for crashed workers;
- poison-event handling;
- ordering rules;
- broker acknowledgement handling;
- metrics, tracing, alerts, and operator tools; and
- cleanup/retention.

Frameworks can reduce boilerplate, but the team still owns the delivery contract and failure behavior.

## 14. Challenge: multiple pollers and the same row

Two common claim strategies are:

### A. Row lock while selecting

```sql
SELECT *
FROM outbox_events
WHERE status = 'PENDING' AND available_at <= CURRENT_TIMESTAMP
ORDER BY occurred_at, event_id
FOR UPDATE SKIP LOCKED
LIMIT 100;
```

PostgreSQL explicitly describes `SKIP LOCKED` as unsuitable for a general consistent view but useful for multiple consumers of a queue-like table. The implementation must decide whether to hold the transaction across the network publish—which can hold locks/connections too long—or claim quickly and use a lease/state transition.

### B. Atomic claim with a lease

Workers atomically move selected rows from `PENDING` to `CLAIMED`, recording an owner and expiry. If a worker dies, another can reclaim the row after the lease expires. Lease duration and heartbeats are deployment-dependent.

Both approaches prevent much simultaneous work, but neither removes the `published -> crash before checkpoint` duplicate window. Consumer idempotency is still required.

## 15. Challenge: idempotency at publisher and consumer

### Kafka producer idempotence is helpful but boundary-limited

Kafka's idempotent producer uses broker-managed producer identity/epoch and sequence information to suppress supported producer retry duplicates. That protects the configured Kafka producer session/protocol boundary; it is not a permanent business event identity and does not atomically include a consumer database.

Use both layers deliberately:

```text
transport identity/mechanism: Kafka producer ID, epoch, sequence
application identity:         stable event_id = evt-7f41
business order:               aggregate_id O-42, aggregate_version 7
```

### Consumer inbox pattern

A consumer can atomically record receipt and apply its local effect:

```sql
BEGIN;

INSERT INTO inbox(consumer_name, event_id, processed_at)
VALUES ('inventory', 'evt-7f41', CURRENT_TIMESTAMP)
ON CONFLICT DO NOTHING;

-- Continue only if the insert won.
UPDATE inventory
SET reserved = reserved + 1
WHERE sku = 'S-8' AND available - reserved >= 1;

COMMIT;
```

The uniqueness scope should normally include the logical consumer/subscription, because different consumers are each entitled to process the same event once.

An inbox row by itself is insufficient if it commits separately from the business effect. Receipt marker + local effect—and any new outbox event—should share one local transaction where possible:

```text
consume event
  -> one local transaction:
       insert inbox marker
       update local business data
       insert next outbox event
  -> commit
  -> acknowledge input
```

## 16. Challenge: parallel pollers and event order

Suppose one order changes twice:

```text
T1 commits OrderPaid(v2)
T2 commits OrderCancelled(v3)
```

Poller B may publish `v3` before Poller A publishes `v2`. If both use the same Kafka key, Kafka will faithfully retain the **wrong append order** it received:

```text
partition for O-42: [OrderCancelled v3, OrderPaid v2]
```

Kafka only guarantees that a consumer reads a partition in the order records were written to that partition. Kafka Streams offers event-time/window/versioned-state tools, but it does not infer the application's desired order and universally sort all records.

### Practical ordering design

1. Define the required scope—usually per `aggregate_id`.
2. Allocate `aggregate_version` atomically with the business mutation.
3. Use `aggregate_id` as the broker key so one entity maps to one partition.
4. Prevent concurrent relay for the same aggregate, or publish in version order with an appropriate scheduling design.
5. Make consumers version-aware:
   - `version == last + 1`: apply;
   - `version <= last`: duplicate/stale, ignore safely;
   - `version > last + 1`: gap—buffer briefly, retry/fetch state, or quarantine according to policy.
6. Reconcile if an expected version never arrives.

Timestamps alone are weak ordering keys: clocks can differ, precision can collide, and event time may intentionally differ from commit order. A database auto-increment value also does not universally prove commit order under concurrent transactions. Choose and document the semantics you actually need.

## 17. Challenge: relay cannot publish

The outbox row is already the durable retry record:

```text
PENDING -> attempt publish
             | success + ack -> PUBLISHED
             | transient fail -> PENDING with later available_at
             | invalid/terminal -> QUARANTINED + alert/operator action
```

Use exponential backoff with jitter for transient broker/network errors, bounded by business objectives. Jitter prevents many relays from retrying in lockstep. Do not hot-loop forever, and do not silently discard after an arbitrary attempt count.

If the broker is unavailable for a long period:

- continue committing business writes only if the database can safely absorb the growing outbox and the business accepts delayed consumers;
- apply admission control/backpressure before storage, transaction-log retention, or downstream recovery becomes unsafe;
- alert on oldest-event age, not only row count; and
- preserve original event identity across every retry.

## 18. Challenge: publication delay

Polling adds a delay up to the scheduling/query/processing path, while CDC can observe committed log changes continuously. Neither has a universal latency. Batch size, poll frequency, index shape, database load, broker acknowledgements, connector lag, and backpressure determine the result.

The design question is not “Is async slow?” but:

```text
What propagation SLO does this business effect require,
and what happens when that SLO is missed?
```

An email can often lag. Fraud blocking or a user-facing order status may need a much tighter and explicitly monitored bound.

## 19. Challenge: cleanup

Never let cleanup race with publication.

A safe policy distinguishes:

- unpublished/claimed rows that must remain recoverable;
- successfully checkpointed rows eligible after a retention window;
- quarantined rows that require audit/operator decisions; and
- rows needed for reconciliation, legal retention, or replay.

Delete/archive in small batches or use time/table partitioning so cleanup does not create long locks, excessive write amplification, or a giant transaction log burst. Retention is a business and operational decision, not a universal number.

# Part V — Transaction-Log Tailing / Change Data Capture

## 20. What is a database transaction log?

Databases normally maintain a durable sequential log of changes for crash recovery and replication. Names vary:

- PostgreSQL: Write-Ahead Log (WAL);
- MySQL: binlog; and
- some managed NoSQL databases: change streams/table streams.

**Change Data Capture (CDC)** reads committed changes from such a source and emits them to another system.

```text
Application -> Database tables
                   |
                   v
             WAL/binlog/stream
                   |
              CDC connector
                   |
                   v
                 Broker
```

The connector stores a position/checkpoint so it can resume after restart. Around failure and checkpoint boundaries it can emit duplicates; Debezium documents at-least-once behavior during abnormal recovery.

## 21. Two designs that are often confused

### 21.1 Transactional outbox + log tailing (canonical relay pattern)

```text
one DB transaction:
  business row + semantic outbox row
                       |
                       v
                  WAL/binlog
                       |
                 Debezium/CDC
                       |
                       v
                     broker
```

The outbox table **still exists**. CDC replaces the query poller. The canonical transaction-log-tailing pattern from the transactional-outbox catalog is specifically a message relay that observes outbox inserts.

Advantages:

- semantic event payload is deliberately created by application/domain code;
- database atomicity still connects business state and event intent;
- no repeated `SELECT` polling of pending rows; and
- log order/transaction metadata can help preserve source commit sequencing within the connector's supported scope.

### 21.2 Direct CDC of business tables (no separate outbox)

```text
UPDATE orders SET status = 'PAID'
              |
              v
CDC sees row before/after image
              |
transform/filter into downstream record
```

This can avoid an outbox table, but the captured fact is “row X changed,” not necessarily the domain event “PaymentCaptured.” A single business event may span several rows; maintenance scripts, backfills, repairs, and schema changes can look like user actions.

Use direct CDC when consumers genuinely want data replication/change streams or the physical transition maps cleanly to the required contract. Prefer a semantic outbox when downstream consumers need stable domain intent independent of table layout.

## 22. CDC/log-tailing challenges

### Duplicate events

If a connector publishes changes and crashes before its source/broker checkpoint is durable, it may replay them. Preserve the stable outbox `event_id` or construct a carefully defined source identity, and keep consumers idempotent.

### Ordering

A connector can preserve a source log's observed order within its documented scope, but end-to-end ordering can change through:

- several database shards/logs with no single global sequence;
- parallel connector tasks or transforms;
- routing to several broker partitions;
- retries/replays; and
- downstream concurrent processing.

Use aggregate keys/versions and state the boundary. Do not promise global order merely because the source log is sequential on one node.

### Latency — correction to the rough bullet

Log tailing is not inherently slow; it often avoids polling intervals. It can still lag because the connector is down or undersized, the broker is slow, a large transaction delays emission, a snapshot is running, or backpressure accumulates. Treat CDC lag as a measurable metric, not a fixed property.

### Filtering and semantic correctness

“Publish valid inserts on table T” needs a precise contract:

- only `INSERT`, or a specific `UPDATE old.status -> new.status` transition?
- what about deletes/tombstones?
- do migration/backfill jobs publish externally?
- how are multi-row transactions represented?
- how are tenant/security filters enforced?
- what happens after a column rename or schema change?
- how are snapshots distinguished from live changes?

For a semantic outbox, configure the connector/transform only for the intended outbox table. Debezium explicitly recommends selectively applying its outbox transformation because connectors can also emit heartbeats, transaction metadata, schema changes, and other record shapes.

### Operational coupling

CDC needs database-specific privileges/configuration, connector upgrades, offset storage, schema evolution, snapshot strategy, and monitoring. A stalled PostgreSQL replication slot or equivalent mechanism can retain logs and consume disk until handled. This exact behavior is product/configuration dependent, but the runbook must exist before production.

## 23. Polling relay versus CDC relay

| Dimension | Polling outbox | CDC outbox |
|---|---|---|
| Database portability | Often works with ordinary SQL access | Connector/database-specific |
| Publication trigger | Periodic/busy polling query | New committed log entry |
| Database load | Repeated indexed queries and status updates | Log decoding/replication overhead |
| Typical latency shape | Tied partly to poll schedule | Continuous, but connector lag can grow |
| Scaling | Claims/leases/`SKIP LOCKED` | Connector/task/partition topology |
| Ordering work | Explicit across pollers | Source order helps, end-to-end scope still required |
| Operational stack | Application worker | CDC platform/connectors/offsets |
| Payload semantics | Explicit outbox event | Explicit outbox event in this comparison |

Choose from team maturity, database support, load, latency SLO, ordering scope, and operational ownership—not from “CDC is always faster” or “polling is always simpler.”

# Part VI — Listen to Yourself Pattern

## 24. Plain-language idea: make the broker first

The database-first outbox says, “commit state and event intent to the database first.” Listen to yourself makes a different choice: “publish the accepted command/event to the broker first; then this same service consumes it to update its local database.”

```text
Client
  |
  | ChangeUserName(command_id=C-7)
  v
User Service API --------publish--------> Broker
                                           |
                   +-----------------------+-------------------+
                   |                                           |
                   v                                           v
         User Service consumer/daemon                   Other consumers
                   |
                   v
              User Database
```

The service “listens to itself” because its own consumer applies the database change from the record it published.

The producer no longer performs `write DB + publish broker`. Its acceptance boundary is one broker write. The database becomes an asynchronous projection/materialized state derived from the broker record.

## 25. API semantics must change honestly

After the broker acknowledges the command, the database may not be updated yet. Therefore the API should not claim “the username is definitely committed in the database.” A truthful response might be:

```http
HTTP/1.1 202 Accepted
Location: /commands/C-7

{"command_id":"C-7","status":"PENDING"}
```

The consumer later applies or rejects the command and records/emits an outcome.

This pattern is strongest when the broker record is accepted as authoritative intent and eventual local state is part of the product contract.

## 26. Challenge: a GET arrives before the consumer applies the write

```text
POST change name -> broker accepted -> 202
GET user          -> database still contains old name
```

This is a **read-your-writes** problem. Options include:

1. **Expose pending status:** return current committed state plus command `C-7: PENDING`.
2. **Session consistency token:** return an event offset/version and allow the read to wait up to a bounded time for the projection to reach it.
3. **Optimistic UI:** the client displays the requested value as pending until confirmation.
4. **Read from a broker-derived materialized view** that tracks the command, if it provides the required consistency contract.
5. **Use a different pattern** if the API must synchronously validate, persist, and immediately return the committed database state.

### Why “write-through cache” is not a complete fix

If the API writes the broker and cache, that is another dual write. If the consumer writes database and cache separately, that is also a dual write. A cache may store a provisional intent keyed by command/version, but it must show `PENDING`, handle failure/rejection, expire/reconcile correctly, and not masquerade as committed database truth.

## 27. Other listen-to-yourself challenges

### Business validation

Checks performed before publishing can become stale before asynchronous application. The authoritative consumer may discover a conflict, invalid state transition, or constraint violation. Model explicit outcomes:

```text
ChangeRequested -> ChangeApplied
                -> ChangeRejected(reason)
```

If the caller requires definitive synchronous rejection, broker-first may be a poor fit.

### Duplicates

The client can retry after an ambiguous publish response. Give every command a stable `command_id`; configure broker producer guarantees appropriately; and make the service's own database consumer idempotent with an inbox/unique business transition.

### Ordering and concurrency

Partition by the entity whose commands must be serialized, such as `user_id`, and carry an expected/current version. One partition's append order can serialize commands for that user, but multi-entity invariants still need a separate design.

### Broker availability

Because the broker is the acceptance boundary, an unavailable broker means new writes cannot be durably accepted through this path. This is conceptually clean, but the availability dependency must be acceptable.

### Replay and side effects

Replaying the topic must not resend irreversible emails, charges, or notifications unintentionally. Separate state-building events from one-time effect execution, retain inbox/effect identities, and define replay modes.

### Client feedback and operations

The system needs command-status queries, lag monitoring, rejection workflows, poison-command handling, and user-visible semantics for “accepted but not yet applied.” This is more than adding a daemon thread.

## 28. Listen to yourself versus event sourcing

They are related but not identical:

- **Listen to yourself:** broker-first publication is used so the same service asynchronously builds local state; the retained record may be a command or event.
- **Event sourcing:** an append-only event store is the authoritative history, and current state is derived by replaying domain events.

A listen-to-yourself implementation may evolve into event sourcing, but calling every broker-first pipeline “event sourcing” overstates its domain model, retention, replay, snapshotting, and evolution guarantees.

## 29. When to use or avoid listen to yourself

Use it when:

- asynchronous command acceptance is a valid product contract;
- the broker/log can be the authoritative durable intent;
- the service is already deeply event-driven;
- projection lag and command-status workflows are acceptable; and
- per-key ordering/idempotent projection are natural.

Avoid or reconsider it when:

- the caller must receive immediate confirmation of a committed database constraint;
- reads must immediately reflect the committed write without session/pending semantics;
- the team cannot operate broker-first state recovery and replay;
- commands involve difficult synchronous validation across current state; or
- an ordinary local transaction + outbox is simpler and meets the requirements.

# Part VII — Choosing a design

## 30. Decision flow

```text
One logical operation changes two resources
                    |
                    v
Can the hard invariant live under one owner/local transaction?
          | yes                          | no
          v                              v
  Do that first                  Is atomic visibility across
                                 compatible short-lived resources required?
                                         | yes          | no/unsupported
                                         v              v
                                  evaluate 2PC      Choose authority
                                                        |
                          +-----------------------------+-------------------+
                          |                             |                   |
                    DB is authority              broker/log is       multi-service
                          |                        authority           workflow
                          v                             v                   v
                 transactional outbox        listen-to-yourself/      Saga + each
                   poller or CDC              event sourcing          step's outbox

For every async branch: idempotency + ordering scope + retry/backpressure
                        + observability + reconciliation
```

## 31. Comparative trade-off table

| Pattern | Write response can mean | Duplicate handling | Ordering | Best fit | Avoid when |
|---|---|---|---|---|---|
| Naive dual write | Both calls appeared successful | Ad hoc | Ad hoc | Non-critical best-effort effect with repair | Missing/phantom effects violate invariants |
| 2PC | Global commit/abort decision | Protocol recovery | Transaction order/isolation dependent | Compatible resources, short strict atomic transaction | Participants unsupported, high availability/autonomy needed |
| Outbox + polling | Business state + event intent committed | Consumer idempotency | Must coordinate pollers/per key | Portable DB-first event publication | DB polling/load/plumbing unacceptable |
| Outbox + CDC | Business state + event intent committed | Consumer idempotency | Source log helps; broker scope remains | Existing CDC platform, low-lag DB-first events | Connector/database coupling cannot be operated |
| Direct CDC | Business row committed | Consumer idempotency | Physical log order scope | Replication/change feed, simple row transition | Stable semantic domain event required |
| Listen to yourself | Command/event durably accepted | Producer + projection idempotency | Broker key/partition + versions | Async broker-first products | Immediate committed-state response required |
| Saga | Workflow entered/advanced a durable state | Per-step idempotency | Workflow/state-machine rules | Several autonomous business services | One local invariant could be simpler |

## 32. Alternatives and complements

### Redesign ownership

The strongest solution is often to place data needed for one hard invariant under one service/database transaction, then publish derived facts. Do not split a transaction boundary merely to claim microservice purity.

### Make the second system derived and rebuildable

For caches and search indexes, choose the primary database as truth. Invalidate/update asynchronously, tolerate a bounded stale window, and rebuild from source. Versioned cache values and reconciliation reduce damage.

### Saga or durable workflow

Outbox solves “local state changed but its message was lost.” Saga solves “several committed local business steps must finish or compensate.” They are complementary:

```text
Saga step = local business update + next event in one outbox transaction
```

See [Saga, Strangler Fig, and CQRS](<./04. Saga, Strangler Fig and CQRS.md>).

### Event sourcing

Append the domain event as the authoritative state transition and derive views. This can remove a database+event dual write, but it requires event modeling, versioning/upcasting, snapshots, replay safety, and a very clear treatment of external effects.

### Reconciliation

Even a reliable design needs a periodic way to detect impossible or missing states. Compare authoritative records, publication checkpoints, consumer versions, or business totals; repair automatically where safe and route ambiguous cases to manual review.

# Part VIII — Familiar-system examples

## 33. WhatsApp: message acceptance and fan-out

**Illustrative interview design, not a private-architecture claim.**

A messaging service may need to store message `M7` and notify delivery/fan-out systems:

```text
Sender -> Message API
             |
             | one transaction
             +--> messages(M7, conversation=C9, seq=104)
             +--> outbox(MessageAccepted M7)
                              |
                              v
                           broker
                         /        \
                  recipient fanout  push notification
```

Use client-generated/stable message intent IDs for retries and a conversation sequence for per-conversation ordering. A duplicate delivery event must not create a duplicate visible message. Global ordering across all chats is unnecessary.

Follow-up: if low-latency broker-first acceptance is used instead, the UI must distinguish “sent to service/broker” from “persisted,” “delivered,” and “read.”

## 34. YouTube: upload metadata and transcoding

**Illustrative design.**

After an upload is finalized, metadata must transition to `UPLOADED` and a transcoding job must be created:

```text
Object upload complete
        |
        v
Video Service transaction
  update video V5 -> UPLOADED
  insert outbox TranscodeRequested(V5, source_version=8)
        |
        v
Broker -> transcode workers -> renditions -> VideoReady event
```

The transcode command is idempotent by `(video_id, source_version, profile)`. Object storage + metadata is itself another boundary: staged object names, checksums, finalize states, expiry of abandoned uploads, and reconciliation handle orphaned objects or missing metadata.

## 35. Uber: trip-state transitions

**Illustrative design.**

When trip `T9` changes from `REQUESTED` to `MATCHED`, dispatch, rider notification, pricing, and analytics may react. Store the trip transition and event intent together, partition events by `trip_id`, and carry `trip_version`.

```text
Trips DB: T9 MATCHED v12 + outbox TripMatched(T9,v12)
                                      |
                                      v
                                   broker
                            +----------+-----------+
                            |          |           |
                         rider UI   driver UI   analytics
```

A late `TripRequested(v11)` consumer update must not overwrite `TripMatched(v12)`. A multi-service trip workflow may also need a Saga; outbox makes each Saga step's publication reliable.

## 36. Netflix: playback activity versus billing

**Illustrative design.**

Playback activity used for recommendations can tolerate eventual propagation and may naturally use an event log with derived views. Subscription billing cannot treat “probably published” as payment truth.

```text
PlaybackStarted -> event log -> recommendations/analytics projections

Billing DB transaction:
  record invoice/payment state + outbox BillingStateChanged
```

The lesson is that one company/product can use different patterns for different invariants. Do not choose one consistency pattern for every event.

## 37. X/Twitter: post creation, timelines, search, notifications

**Illustrative design.**

The authoritative post row and `PostCreated` outbox event can commit together. Timeline fan-out, search indexing, notifications, and analytics are derived consumers:

```text
Post Service DB + outbox
          |
          v
        broker
     /     |       \
timeline  search  notification
```

Temporary lag can mean the post exists on its direct page before search or every follower timeline shows it. Each derived system must be rebuildable/reconcilable, and deletion/privacy changes need higher-priority, audited propagation rather than assuming all stale copies vanish instantly.

## 38. Amazon: order checkout

**Illustrative design plus verified AWS pattern guidance.**

An Order Service can atomically create a `PENDING` order and outbox event. Inventory, payment, and shipping are separate Saga participants; confirmation email is a non-authoritative side effect.

```text
Order DB transaction
  order=PENDING + outbox OrderCreated
                   |
                   v
                 broker
        +------------+-------------+
        |            |             |
   Inventory      Payment      Notification
        |            |
        +------ Saga state/compensation ------+
```

AWS Prescriptive Guidance documents the transactional outbox for database + notification dual writes and describes both polling/RDS-style and CDC/change-stream-style implementations. For DynamoDB or another NoSQL store, atomicity may be limited to a transaction or logical partition/item boundary; verify the chosen product's exact constraints.

```text
1. INVARIANT
   What must not disagree? Which effect is authoritative?

2. BOUNDARIES
   Name resource A and B; each commits independently.

3. FAILURE WINDOWS
   A then B -> lost event; B then A -> phantom event;
   timeout -> unknown outcome; parallel/retry does not create atomicity.

4. REDESIGN FIRST
   Can the hard invariant live in one local transaction/owner?

5. CHOOSE AUTHORITY
   DB-first -> transactional outbox.
   Broker-first -> listen-to-yourself/event log if async semantics fit.
   Compatible strict resources -> evaluate 2PC.

6. OUTBOX FLOW
   one DB transaction = business row + immutable event intent;
   relay via polling or CDC; publish after commit.

7. DELIVERY SEMANTICS
   usually at least once; stable event_id; idempotent consumer/inbox.

8. ORDERING
   define per-aggregate scope; key partition; aggregate_version;
   handle duplicate, stale, and gap.

9. OPERATIONS
   retry/backoff+jitter, poison event, backlog/backpressure,
   cleanup, metrics, reconciliation.

10. TRADE-OFF
    eventual consistency and plumbing in exchange for decoupling and recovery.
```

## 43. High-value follow-ups and concise answer cues

### “Does outbox guarantee exactly once?”

No. It atomically stores state + event intent, but the relay can publish and crash before checkpointing. Target at-least-once publication with idempotent consumers. Define “exactly once” only inside a named boundary.

### “Why not just retry Kafka?”

The process can die, and the publish result can be unknown. Store retry intent durably in the same local transaction as the state change; reuse a stable event ID.

### “Can multiple pollers publish the same event?”

Claims/locks reduce simultaneous work, but a post-publish/pre-checkpoint crash still causes replay. Consumer idempotency remains required.

### “How do you preserve order?”

Define order per aggregate, allocate an atomic version, partition by aggregate ID, serialize/publish carefully, and have consumers reject stale versions or detect gaps. Kafka preserves per-partition append order, not desired business order that producers failed to enforce.

### “What if a consumer also publishes?”

Atomically write inbox marker + local business change + new outbox row, then acknowledge the input after commit. Otherwise the dual-write problem has moved to the consumer.

### “What if the broker remains down?”

The outbox grows. Decide whether writes may continue from business requirements and database capacity; use oldest-age/backlog alerts, backpressure/admission control, recovery rate planning, and reconciliation.

### “Polling or CDC?”

Polling is usually easier to understand and more portable but adds indexed scans/status coordination. CDC avoids query polling and can provide continuous low-lag relay but adds connector, log-retention, privileges, schema, offset, and recovery operations.

### “Why not direct CDC?”

It is good for data-change replication, but a row mutation is not always a stable domain event. Backfills and schema changes leak through. A semantic outbox decouples the public event contract from physical tables.

### “Could I update the cache first so listen-to-yourself reads are fresh?”

That creates another independently failing write unless the cache is derived through the same event stream. Expose a pending intent/version or wait for a projection; do not call provisional cache state committed truth.

# Part X — Edge cases, mistakes, and operations

## 44. Edge cases interviewers use to go deeper

### Database commits, but the API response is lost

The client retries `CreateOrder`. Request-level idempotency must ensure the retry returns/reconstructs the same order and does not create a second business row/outbox event. Producer outbox idempotency and client API idempotency are related but different layers.

### Event payload serialization fails during the transaction

Fail and roll back the local transaction, or construct/validate the event before the mutation while ensuring the final values are correct. Never commit the business row and assume a later process can always reconstruct the exact intended semantic event.

### Consumer marks inbox before its business update

If those commits are separate and the business update fails, redelivery sees the inbox marker and skips the missing effect. Put inbox + effect in one local transaction.

### Consumer performs a non-idempotent external effect

An inbox cannot roll back an email/payment provider. Propagate a stable downstream idempotency key if supported, persist an effect state machine, query ambiguous outcomes, and reconcile. Some effects require manual review.

### One poison event blocks per-key order

Blindly skipping it may violate every later state transition. Quarantine with full context, pause only the affected key/partition where possible, alert, repair/redrive with the same identity, and define whether later versions can safely proceed.

### Outbox payload contains stale reconstructed state

Build the event from the authoritative mutation/result inside the transaction. Do not query later and guess what the original event should have contained; intervening updates may have changed the row.

### A schema migration changes both table and event

Database schema and event schema have different consumers and timelines. Use explicit event schema versions and backward/forward compatibility. Deploy readers before writers where the compatibility plan requires it.

### Multi-region active-active writes

Per-region outboxes produce no automatic global order. Define aggregate ownership/routing, conflict resolution, globally unique event IDs, version semantics, broker replication, failover behavior, and duplicate/reconciliation rules. Do not casually promise one global sequence.

## 45. Common mistakes

- Calling event-driven architecture itself the fix.
- Publishing inside a database transaction and assuming rollback retracts the already visible event.
- Saying `@Transactional` automatically includes Kafka, SQS, email, Redis, or an HTTP call.
- Treating a timeout as a definite publish failure.
- Claiming outbox gives exactly once.
- Using a Kafka producer ID as the permanent business event ID.
- Marking an outbox row published before a durable broker acknowledgement.
- Deleting a row immediately after attempting publish.
- Running parallel pollers without defining per-aggregate order.
- Claiming Kafka Streams automatically reorders arbitrary business events.
- Using timestamps as a complete causal-order solution.
- Recording a consumer inbox separately from the business effect.
- Moving the dual write from producer to consumer and declaring victory.
- Adding a second retry database even though the outbox is already durable.
- Directly exposing physical CDC records as long-lived domain contracts without filtering/evolution design.
- Calling log tailing inherently slow without measuring connector lag.
- Using a cache to hide a pending write while reporting it as committed.
- Ignoring broker outage backlog, database capacity, poison events, cleanup, and reconciliation.
- Using Saga when both changes could safely live in one local transaction.

## 46. Observability and runbook checklist

### Producer/outbox

- pending/claimed/quarantined row count;
- age of the oldest unpublished event;
- business-commit-to-broker-ack latency percentiles;
- publish attempt, retry, and terminal failure rate by error class;
- expired lease/reclaimed row count;
- outbox table/index size and cleanup lag;
- broker acknowledgement latency and error rate.

### CDC

- source-log position versus connector position;
- connector restart/failure and duplicate-recovery count;
- retained WAL/binlog/change-stream pressure;
- snapshot state and schema/transform errors;
- source-to-broker event lag.

### Consumer

- consumer lag and oldest event age;
- idempotency/inbox conflict count;
- stale-version and version-gap count;
- effect failures, DLQ/quarantine age, and redrive results;
- reconciliation mismatches;
- end-to-end business completion time, not just broker throughput.

### Alerts need actions

For every alert, document who owns it, what can be safely retried, how to pause/resume a key or partition, how to inspect payload without exposing secrets, how to redrive with the same ID, and when manual correction is required.

## 47. Security and governance

- Minimize PII/secrets in outbox payloads because events are copied, retained, logged, and replayed.
- Encrypt in transit/at rest according to the threat model and restrict database/CDC/broker access.
- Authorize at the command boundary; consumers should not trust an event merely because it came from a topic.
- Sign or authenticate producer identities where cross-trust-boundary tampering matters.
- Version deletion/privacy workflows and propagate them with auditable priority.
- Avoid putting raw payloads or credentials into retry error columns and logs.

# Part XI — Revision material

## 48. Compact cheat sheet

```text
DUAL WRITE
  one intent -> two independent commits
  A then B: A can commit while B is absent/unknown
  B then A: consumers can see a phantom event
  parallelism, try/catch, and in-memory retry do not create atomicity

2PC
  PREPARE + DECIDE across compatible participants
  can provide atomic commit, but adds blocking/latency/coupling/recovery
  coordinator can be durable/replicated; participants may still wait

TRANSACTIONAL OUTBOX
  one DB transaction = business state + event intent
  relay later by polling or CDC
  normally at least once -> stable event_id + idempotent consumer
  order per aggregate -> key + aggregate_version + gap policy

POLLING
  claim/lease/SKIP LOCKED + retry/backoff/jitter + cleanup
  duplicate window remains after publish/before checkpoint

LOG TAILING / CDC
  canonical: tail semantic outbox inserts
  direct CDC: no outbox, but physical row changes leak into contract
  not inherently slow; monitor lag, offsets, logs, schema, duplicates

LISTEN TO YOURSELF
  broker first; service consumes its own command/event into DB
  removes producer DB+broker dual write
  API is async: accepted/pending, not necessarily applied
  solve read-your-writes, rejection, order, replay, idempotency

END TO END
  choose authority + local atomicity + durable handoff
  + idempotency + ordering scope + backpressure
  + observability + reconciliation
```

## 49. Final design checklist

- What exact invariant or user-visible promise is at risk?
- Which two resources commit independently?
- What happens in both write orders and every acknowledgement-loss window?
- Can one owner/local transaction remove the dual write?
- Which system is the source of truth?
- Is 2PC actually supported and justified?
- If using outbox, are business row and event intent in the exact same transaction?
- Is the event ID stable across retries?
- What delivery guarantee exists at each boundary—not just the broker marketing label?
- Is consumer receipt atomic with its local effect and next outbox event?
- What is the ordering scope, key, and version rule?
- How do multiple workers claim work and recover expired claims?
- What happens on ambiguous publish success?
- How are transient, permanent, and poison failures separated?
- How much backlog can accumulate, and when is admission control applied?
- How are published rows checkpointed and cleaned safely?
- If using CDC, what are filtering, schema, snapshot, offset, privilege, and log-retention plans?
- What does the API tell a user while state is pending?
- Can state be reconciled/rebuilt, and who operates redrive/manual repair?
- Are privacy, authorization, schema evolution, metrics, and runbooks covered?

## 50. Say-out-loud 30-second narrative

> “A dual write happens when one business action must commit to two independent resources, such as an orders database and Kafka. Writing either one first leaves a crash window: database-first can lose the event, while broker-first can create a phantom event, and retries face unknown outcomes. I first try to keep the hard invariant in one local transaction. For database-first publication, I atomically save the business change and an outbox event, then relay it by polling or CDC. Publication is usually at least once, so I use stable event IDs, idempotent consumers, per-aggregate keys and versions, backoff, backlog monitoring, cleanup, and reconciliation. Broker-first listen-to-yourself is an alternative only when accepted-but-pending writes and asynchronous projections fit the API contract.”
