# Rate Limiter

> Beginner-first HLD notes expanded from: **token bucket, leaky bucket, fixed window counter, exact sliding window, sliding-window counter using subwindows or weighting, placement at the load balancer/API gateway/server, cache choice, scaling across 1,000 servers, and atomic Redis operations**.
>
> The central interview lesson is: **a rate limiter is a distributed admission-control system, not merely a counter. First define who is limited, what is counted, the time/burst semantics, and the failure policy; then choose an algorithm, placement, and state architecture that enforce that contract closely enough.**

## Scope, assumptions, and evidence labels

I assume the bullet **"Cache Type"** asks what state should live in an in-process cache versus a shared/distributed cache such as Redis, and whether Redis is truly a cache in this design. Section 12 addresses that explicitly.

I assume **"Sliding Window"** means the exact timestamp-log algorithm, while **"Sliding Window Counter"** means either:

- several smaller fixed counters covering the rolling interval; or
- the two-counter weighted approximation using the previous and current fixed windows.

Both interpretations are covered because they have different accuracy and memory trade-offs.

The word **throttling** is used inconsistently in industry. Some people mean "reject requests above a rate"; others mean "delay/pace them." This chapter always states the action explicitly: **reject, queue/delay, challenge, or degrade**.

Evidence labels used here:

- **Verified concept** - supported by a first-party specification, documentation page, or engineering post linked nearby.
- **Dated production example** - a first-party description of how a system behaved when published; it is not assumed to be that company's private current architecture.
- **Public product behavior** - behavior documented for a public API; exact limits can change by plan, endpoint, account, or date.
- **Illustrative design** - a plausible design for a familiar product; it is not a claim about that company's private architecture.
- **Self-reported interview experience** - a candidate described a prompt publicly; it is neither an official nor independently verified company question bank.
- **Illustrative number** - chosen to explain mechanics or estimation, not claimed production traffic or capacity.

Current documentation was checked on **2026-07-16**. Time-sensitive product limits are either linked and dated or deliberately described without a fixed number.

### The most important corrections up front

1. **A rate limiter does not by itself stop DDoS.** It can reduce Layer-7 abuse and protect downstream resources, but a large attack can saturate the network, TLS terminator, gateway, rate-limit service, or state store before an application-side limiter runs. Edge DDoS protection, WAF/bot detection, connection limits, load shedding, and capacity isolation remain necessary.
2. **Token bucket and leaky bucket are not synonyms.** A token bucket permits a bounded burst and enforces a long-term average. A queue-based leaky bucket emits at a controlled rate, making bursty arrivals wait and therefore adding latency.
3. **"Leaky bucket adds latency" is true only for the queue/shaper form.** A leaky-bucket meter can reject non-conforming traffic immediately without holding a queue. Section 7 separates the two forms.
4. **Fixed window is not a rolling limit.** A `100/minute` fixed counter can admit 100 requests at the end of one minute and 100 more at the start of the next, nearly 200 back-to-back.
5. **The weighted sliding-window counter is approximate.** Its formula assumes requests in the previous window were spread uniformly. A multi-subwindow counter is another approximation; smaller subwindows reduce, but do not eliminate, boundary error.
6. **Putting an independent `100/minute` limiter on each of 1,000 servers creates a fleet limit near `100,000/minute`, not `100/minute`.** The fleet must share state, route each key to one owner, or receive centrally leased portions of a global budget.
7. **Dividing a global quota by the current server count is usually fragile.** Traffic is uneven, autoscaling changes the divisor, a quota may be smaller than the server count, and unused allowance on an idle server cannot automatically help a busy server.
8. **Redis is often the authoritative decision-state store here, not merely a disposable cache.** Losing counters may allow extra requests; losing only cached rules may be recoverable from a rules database. These are different failure consequences.
9. **Atomic Redis commands do not make a multi-command algorithm atomic.** `GET -> compute -> SET` races. Use a suitable single command, a short Lua script/Redis Function, or optimistic locking where appropriate.
10. **Lua atomicity is local to the Redis execution boundary.** It does not make updates across different Redis Cluster hash slots, regions, or Redis plus the application database one transaction.
11. **Rate limits and concurrency limits solve different overload problems.** `100 requests/second` does not cap how many slow requests are simultaneously in flight. Protect a slowing dependency with concurrency/adaptive admission limits as well.
12. **HTTP `429 Too Many Requests` is standardized; the newer `RateLimit` response fields are still an IETF Internet-Draft as of the check date.** Do not describe draft fields as a finalized RFC.

## 1. Plain-language foundation

### 1.1 What problem is a rate limiter solving?

Imagine a restaurant kitchen that can prepare 100 meals per minute. One customer submits 5,000 orders at once. If every order enters the kitchen:

- the order queue explodes;
- normal customers wait behind the abusive customer;
- the kitchen exhausts ingredients or workers;
- retries make the overload worse;
- the whole restaurant may stop functioning.

A rate limiter is the person at the kitchen entrance who applies a rule before admitting more work:

```text
"Customer A may start at most 20 orders per minute,
 with a small burst of 5, and excess orders are rejected."
```

In software, it sits on or near the request path and makes one small decision:

```text
request + trusted identity + applicable rules + recent usage
                              |
                              v
                   ALLOW / REJECT / DELAY
```

The decision prevents one source or workload from consuming an unfair or unsafe share of a finite resource.

### 1.2 A limit is a contract, not just `N requests/minute`

The sentence `100 requests per minute` leaves many questions unanswered:

- **Who?** Per IP, user, API key, tenant, device, endpoint, or the whole fleet?
- **What?** HTTP requests, messages, bytes, CPU-cost units, failed logins, or successful writes?
- **Which minute?** A calendar/fixed minute or every rolling 60-second interval?
- **May it burst?** Can all 100 arrive at once, or must they be evenly spaced?
- **What happens above the limit?** Reject, delay, challenge, serve stale data, or lower quality?
- **What happens if the limiter is unavailable?** Fail open, fail closed, or use a conservative local fallback?
- **Do rejected and retried attempts consume quota?** This affects both recovery and abuse behavior.

A complete policy looks more like this:

```yaml
rule_id: create_order_per_user
match: POST /orders
identity: authenticated_user_id
algorithm: token_bucket
rate: 2 tokens/second
capacity: 10 tokens
cost: 1 token/request
action_when_empty: reject
failure_mode: fail_closed
response: 429 with Retry-After hint
```

### 1.3 Five ideas to keep separate

| Mechanism | Plain meaning | Typical signal/action |
|---|---|---|
| **Rate limiting** | Bound how quickly a subject may start/consume work | Reject or delay above a time-based budget |
| **Quota** | Bound total usage over a longer accounting period | "10,000 units/day"; reject until reset/top-up |
| **Concurrency limiting** | Bound how many operations are in flight simultaneously | Admit only if a slot is free |
| **Load shedding** | Drop lower-priority work because the system is currently overloaded | Reject/degrade based on live health |
| **Backpressure** | Tell upstream producers to slow down or stop | Queue pressure, flow control, delayed consumption |

They can work together. A service might apply a per-user token bucket, a tenant daily quota, a global concurrency cap, and emergency load shedding.

### 1.4 Rate limiting versus throttling

Two common behaviors are both called "throttling":

```text
Rejecting limiter                  Pacing/shaping limiter

request --> [decision] --> app     request --> [bounded queue] --> app
               |
               +--> 429                         constant/smoothed release
```

- **Rejecting** preserves low latency for admitted work but makes clients retry or fail.
- **Pacing** can smooth arrivals for a fragile consumer but increases latency and needs a bounded queue.

Never say only "the request is throttled" in an interview. State whether it is rejected, queued, or degraded.

### 1.5 One concrete example

Suppose an OTP endpoint allows **5 failed validations in 10 minutes per account** and **20 attempts per minute per IP**.

Why two keys?

- Per-account limiting stops an attacker trying many passwords against one victim from many IPs.
- Per-IP limiting slows one machine attacking many accounts.

Why count failed validations rather than every validation?

- A successful user should not be penalized for normal use.
- But the result is known only after authentication logic runs, so the design needs a post-response counter update or a reservation/finalization flow.

This already shows why rate limiting is more than choosing an algorithm: identity, placement, action, and what counts matter as much as the counter math.

## 2. Inline prerequisites

These concepts are required for the rest of the chapter, so they are introduced here rather than deferred.

### 2.1 Throughput, latency, burst, and concurrency

- **Throughput/rate** is work completed or admitted per unit time, such as requests/second.
- **Latency** is time one request takes.
- A **burst** is a short period whose rate is much higher than the long-term average.
- **Concurrency** is how many operations are in progress at once.

If arrivals average `R` requests/second and each spends `L` seconds in the system, a useful steady-state estimate is:

```text
average in-flight concurrency ~= R * L
```

This is Little's Law in a simplified form. If latency rises from 100 ms to 5 seconds while a rate limiter still admits 100 requests/second:

```text
before: 100 * 0.1 = about 10 in flight
after:  100 * 5   = about 500 in flight
```

The request rate did not change, yet memory, threads, connections, and dependency pressure did. That is why a concurrency limit is often paired with a rate limit.

### 2.2 Load balancer, API gateway, and application server

- A **load balancer (LB)** distributes traffic among healthy servers. A Layer-4 LB mainly sees connections/IP/ports; a Layer-7 LB understands HTTP routes, headers, and sometimes identity.
- An **API gateway** is an application-aware entry point that commonly handles authentication, routing, quotas, transformations, logging, and rate limiting.
- An **application server/service** executes business logic and knows facts such as "this GraphQL query costs 30 units" or "this is a payment attempt."

The terms overlap in products. In an interview, describe the capability you need instead of relying on the box label.

### 2.3 Identity and trusted keys

A limiter groups requests by a **rate-limit key**, for example:

```text
rl:<rule_id>:<tenant_id>:<user_id>
```

Possible dimensions include:

- source IP or IP prefix;
- authenticated user ID;
- API key or OAuth client/application ID;
- tenant/account;
- device/session;
- route, method, resource, or operation;
- region; and
- combinations such as `(tenant, user, endpoint)`.

Do not trust an arbitrary client-supplied `X-Forwarded-For` or user ID. The edge must strip/validate forwarding headers, and authentication middleware must supply a verified identity. Otherwise the attacker simply changes the key on every request.

### 2.4 State, cache, and source of truth

A cache is usually a replaceable copy of data held somewhere authoritative. Rate limiting has two different data categories:

```text
Policy/configuration                   Live decision state
--------------------                   -------------------
"free plan = 100/min"                 tokens = 17.4
rule priority/version                  last_refill = ...
stored durably in rules DB             counter for this window = 83
cached at gateways/RLS                 often authoritative in Redis
```

If the rules cache is lost, it can reload from the rules database. If the live token state is lost, clients may regain capacity early. Redis may be an in-memory **state store**, even if people casually call it a cache.

### 2.5 Race condition and atomicity

Assume one token remains and two servers check simultaneously:

```text
Server A: GET tokens -> 1
Server B: GET tokens -> 1
Server A: SET tokens -> 0; allow
Server B: SET tokens -> 0; allow
```

Two requests spent one token. A **race condition** means the result depends on unlucky timing between concurrent operations.

An **atomic read-decide-write** behaves as one indivisible step:

```text
refill -> compare -> decrement -> return decision
```

No other request may observe or change that state halfway through.

### 2.6 Replication, sharding, and consistency

- **Replication** keeps copies of the same state for availability/failover.
- **Sharding/partitioning** divides different keys across nodes for capacity.
- **Consistency** describes whether different readers/regions observe the same updates and ordering.

```text
hash(rule + subject) --> owning Redis shard --> primary + replica(s)
```

Atomicity on one primary prevents local races. Asynchronous replication may still lose a recent increment during failover, causing temporary undercounting. Multi-region replicas may make independent decisions before their counts converge.

### 2.7 Time and clocks

Window algorithms depend on time. Clocks can:

- differ across application servers;
- jump after synchronization;
- have different precision;
- be supplied maliciously if a client timestamp is trusted.

Prefer one authoritative time source for each key's decisions, such as Redis server time or the rate-limit shard's monotonic clock. A monotonic clock is best for elapsed durations on one process; an epoch/server timestamp is useful for persisted/distributed state. Never let the public client choose the accounting time.

### 2.8 TTL and lazy cleanup

A **TTL (time to live)** automatically expires inactive counter keys. Without it, a one-time request from millions of identities can leave millions of permanent keys.

For a fixed window, a key might include the window number and expire after enough time that it can no longer affect a decision:

```text
rl:login:user42:window_29344123  -> 7, TTL roughly 2 windows
```

Token buckets need not run a background refill job. Store `tokens` and `last_refill`; calculate the refill lazily on the next request. Inactive keys can expire once they would have refilled completely.

## 3. Why rate limiting is needed in system design

### 3.1 Fairness and tenant isolation

Without per-tenant admission control, one customer's batch job can occupy every worker or database connection. A limiter creates a resource boundary:

```text
Tenant A burst ----+
Tenant B normal ---+--> [fair admission] --> shared service
Tenant C retry ----+
```

Fairness is not always equality. A paid tenant may receive more capacity, while system/background traffic may have lower priority than an interactive user.

### 3.2 Protecting a finite downstream resource

The frontend fleet may autoscale quickly while the database, payment provider, ML model, or third-party API cannot. Rate limiters cap offered work before it exhausts:

- database connections;
- CPU/GPU time;
- thread pools;
- third-party quota;
- message provider throughput;
- storage write IOPS; or
- a fragile legacy dependency.

The limit should reflect the constrained resource. Counting all requests equally is weak when one request costs 1 ms and another fans out to 10,000 records.

### 3.3 Abuse and security

Examples include:

- password guessing and credential stuffing;
- OTP resend/validation abuse;
- account creation spam;
- scraping;
- comment/message spam;
- inventory hoarding or automated purchasing;
- expensive search/GraphQL queries; and
- enumeration of IDs, emails, or phone numbers.

Rate limiting raises attacker cost but is not the full defense. Combine it with authentication, bot/fraud signals, challenges, IP reputation, device fingerprints, behavioral detection, WAF rules, and audit/alerting.

### 3.4 Cost and business quotas

A request may trigger a paid SMS, map lookup, GPU inference, or partner API call. Weighted quotas make cost visible:

```text
cheap metadata read       = 1 unit
full-text search          = 5 units
video processing request  = 100 units
```

A plan may define both a short-term rate and a monthly quota. The former protects instantaneous capacity; the latter enforces commercial entitlement.

### 3.5 Stability during retries and overload

When a dependency slows, callers time out and retry, increasing load and causing more timeouts: a retry storm. A rate limiter, concurrency limiter, retry budget, exponential backoff, and circuit breaker together break this positive feedback loop.

### 3.6 What a rate limiter cannot guarantee

A limiter does not guarantee:

- enough capacity for every allowed request;
- successful or low-latency responses;
- exact billing/accounting unless designed as a durable ledger;
- exactly-once business execution;
- complete bot identity;
- DDoS absorption at lower network layers; or
- fairness if the chosen identity key is easy to rotate.

The `remaining` value is therefore often a hint, not a promise that the next request must succeed.

## 4. Define requirements before selecting an algorithm

### 4.1 Functional questions

Ask or state assumptions for:

1. What request/operation is protected?
2. Which identity or combination is limited?
3. Is the policy per endpoint, user, API key, tenant, region, or global?
4. Is each request equal cost, or do requests consume weights?
5. Is the requirement fixed-calendar, rolling-window, long-term average, or smoothly paced output?
6. What burst is acceptable?
7. Are excess requests rejected, queued, challenged, or degraded?
8. Do failures/rejections/retries count?
9. Can administrators change rules at runtime?
10. What information is returned to clients?

### 4.2 Non-functional questions

- Decision latency budget: the check is on every admitted request's critical path.
- Decision throughput and peak factor.
- Accuracy: is a 1-5% overshoot acceptable, or is this a security/financial cap?
- Availability: what happens if the limiter or Redis is unavailable?
- Scope: one process, one region, or the whole world?
- Scale: active keys, rules per key, and hot-key skew.
- Durability: may recent state be lost during failover?
- Privacy: may raw user IDs/IPs appear in keys and metrics?
- Operability: shadow mode, rollout, observability, audit, and rollback.

### 4.3 A useful policy model

```text
Policy = Match + Subject + Budget + Time/Burst + Cost + Action + Failure mode
```

Example hierarchy:

```text
1. fleet emergency:       2,000,000 cost units/s globally
2. tenant plan:              10,000 requests/min per tenant
3. user abuse:                  100 requests/min per user
4. expensive endpoint:           5 searches/min per user
5. concurrency:                 200 in-flight calls to partner X
```

All applicable limits can be evaluated. Usually the request is allowed only if every required rule has capacity.

### 4.4 Decision API between a gateway and rate-limit service

An internal interface might be:

```http
POST /v1/decisions
Content-Type: application/json

{
  "descriptors": {
    "tenant_id": "t17",
    "user_id": "u42",
    "route": "POST:/orders"
  },
  "cost": 1,
  "request_id": "optional-stable-attempt-id"
}
```

```json
{
  "allowed": false,
  "rule_id": "create_order_per_user",
  "remaining": 0,
  "retry_after_ms": 840,
  "policy_version": 37
}
```

Security rule: the gateway constructs descriptors from authenticated/trusted request context. It does not pass through arbitrary client-provided tenant or user identifiers.

## 5. Fixed window counter

### 5.1 Plain-language idea

Divide time into fixed boxes and count requests in the current box.

For a limit of 5 requests per minute:

```text
12:00:00 ---------------- 12:00:59 | 12:01:00 ---------------- 12:01:59
             counter = 5            counter resets to 0
```

For request time `t` and window size `W`:

```text
window_id = floor(t / W)
key       = rule + subject + window_id
```

Atomically increment the key. Allow if the new value is at most the limit.

### 5.2 The boundary burst

Limit: 5 per minute.

```text
12:00:59.900  A A A A A | A A A A A  12:01:00.100
                         ^ reset boundary
```

Ten requests arrive within about 200 ms, even though each fixed minute contains only five. In general, nearly `2 * limit` can pass in a very short interval spanning the boundary.

This is not necessarily a bug if the product contract really is a calendar quota such as "1,000 operations each UTC day." It is a problem if the downstream cannot survive bursts or the promise is "no more than N in any rolling interval."

### 5.3 Redis shape

```text
key:   rl:{u42}:search:29344123
value: 37
TTL:   enough to outlive the window, commonly around 2W
```

The braces illustrate a Redis Cluster hash tag if related per-user keys must share a slot; the exact key design depends on the operations.

A safe conceptual operation is:

```text
atomically:
    count = INCR(key)
    if count == 1: set TTL
    return count <= limit
```

Redis documents `INCR` as atomic and shows the rate-limiter pattern. It also warns that `INCR` followed by a conditional `EXPIRE` can leak the key if the client fails between commands; a Lua script closes that gap. See [Redis `INCR` rate-limiter pattern](https://redis.io/docs/latest/commands/incr/).

### 5.4 Trade-offs

| Dimension | Fixed window |
|---|---|
| State per active key | One counter per live window |
| Decision work | O(1) |
| Burst behavior | Boundary can admit nearly 2N back-to-back |
| Accuracy for rolling contract | Poor near boundary |
| Operational complexity | Very low |
| Natural uses | Calendar quotas, coarse abuse control, cheap first-layer limit |

### 5.5 Use and avoid

Use it when:

- simplicity and cost dominate exact rolling fairness;
- boundary bursts are safe;
- the rule is intentionally aligned to a billing/calendar period; or
- it is an early coarse limiter followed by stricter protection.

Avoid it as the only mechanism when:

- a downstream must never see boundary bursts;
- customers expect a rolling promise;
- the endpoint is security-sensitive and boundary gaming matters.

## 6. Exact sliding window log

### 6.1 Plain-language idea

Remember the timestamp of every counted request. For a new request at time `now` and window length `W`:

1. Remove timestamps at or before `now - W` according to the chosen boundary convention.
2. Count timestamps still inside the window.
3. If `count + cost` exceeds the limit, reject.
4. Otherwise add the new timestamp and allow.

```text
now - W                                              now
  |---------------------------------------------------|
        x       x  x                x          x
        <---------- exact timestamps still counted --->
```

### 6.2 Redis sorted-set implementation

One common structure is a sorted set per `(rule, subject)`:

```text
score  = server timestamp
member = unique request/attempt ID
```

Atomic pseudocode:

```text
ZREMRANGEBYSCORE key -inf (now - W)
count = ZCARD key
if count < limit:
    ZADD key now unique_member
    EXPIRE key cleanup_ttl
    allow
else:
    reject
```

The prune, count, conditional add, and expiry must be one atomic server-side operation. Separate round trips race.

The member must be unique even when two requests share the same millisecond; otherwise `ZADD` may update one member instead of representing two requests. A trusted unique attempt ID or `timestamp + sequence/random suffix` works.

For a sorted set containing `N` timestamps where pruning removes `M` entries, Redis currently documents `ZREMRANGEBYSCORE` as `O(log(N) + M)`, `ZCARD` as `O(1)`, and `ZADD` as `O(log(N))` per added element. The entire decision also blocks competing work on that shard while the atomic script runs. See [`ZREMRANGEBYSCORE`](https://redis.io/docs/latest/commands/zremrangebyscore/), [`ZCARD`](https://redis.io/docs/latest/commands/zcard/), and [`ZADD`](https://redis.io/docs/latest/commands/zadd/).

### 6.3 Do rejected attempts enter the log?

Two legitimate policies exist:

**Count only admitted requests**

- Gives a precise admission contract.
- Once the oldest admitted request expires, one new request can pass.
- Repeated rejected probes do not extend the lockout.

**Count every attempt**

- Can punish sustained abuse and extend a block.
- A client retrying too aggressively can keep itself locked out.
- It is better modeled as a separate abuse/penalty rule rather than silently changing the admission algorithm.

State this choice. For an ordinary rolling API quota, storing only admitted operations is often easier for clients to reason about. For brute-force defense, count failed attempts and apply an explicit escalating penalty.

### 6.4 Cost

If the limit is `L` admitted requests per window and there are `K` active subjects, state can approach:

```text
O(K * L) timestamp entries
```

Redis sorted-set operations also have tree/index overhead. A logical 16-byte timestamp/member estimate is not a valid Redis memory estimate; actual per-entry memory can be much higher and must be measured with the chosen encoding/version. This is why exact logs are expensive for millions of high-limit identities.

### 6.5 Trade-offs

| Dimension | Exact sliding log |
|---|---|
| State | One record per counted request in the window |
| Accuracy | Exact relative to timestamp, boundary, and atomicity choices |
| Work | Prune + count + insert; typically more than O(1) counter math |
| Burst | Enforces N in any rolling W interval; may still allow N at once if contract permits |
| Best fit | Low limits, security rules, strict rolling quotas |
| Weak fit | Huge limits, massive key cardinality, ultra-cheap decisions |

Important nuance: an exact rolling limit does **not** smooth N allowed requests. If the log is empty, all N can arrive at once. Use token/leaky bucket semantics if burst shape matters.

## 7. Leaky bucket

### 7.1 Why the term is confusing

"Leaky bucket" is used for two related but behaviorally different mechanisms.

#### Form A: bounded queue / traffic shaper

Requests enter a finite FIFO queue and leave at a steady rate.

```text
bursty arrivals --> [ r1 r2 r3 r4 r5 | bounded queue ] --one every 100 ms--> service
                                  |
                                  +--> overflow is rejected/dropped
```

This form genuinely smooths output. Waiting in the queue adds latency.

#### Form B: meter

Maintain a conceptual water level. Each arrival adds water; time drains it. If adding the request would exceed capacity, reject immediately. It tests whether traffic conforms but does not itself queue or delay the request.

```text
level = max(0, old_level - elapsed * leak_rate)
if level + cost <= capacity:
    level += cost
    allow now
else:
    reject now
```

This meter resembles token-bucket state math with the interpretation reversed. Do not claim every leaky-bucket implementation creates a queue.

### 7.2 Queue latency and capacity

For a queue capacity of `Q` requests and drain rate `r` requests/second, the rough maximum queueing wait for the last accepted item is:

```text
maximum queue wait ~= Q / r seconds
```

This is an **illustrative bound** ignoring processing variation and scheduling overhead. A queue of 1,000 with a 100/s drain can make a request wait about 10 seconds, which may exceed the caller timeout and cause retries. The queue must be bounded and the timeout contract must be aligned.

### 7.3 When it is useful

- A downstream accepts a stable pace but not bursts.
- A worker processes an asynchronous queue at a controlled rate.
- A third-party provider has a strict send rate.
- Network traffic shaping needs smooth output.

NGINX publicly describes its request rate limiting using leaky-bucket terminology, with burst/delay options. Its older explanatory blog carries an editor's correction about the exact burst timing, so use the current module documentation for configuration details; the conceptual queue example remains useful. See [NGINX rate-limiting explanation](https://blog.nginx.org/blog/rate-limiting-nginx).

### 7.4 When to avoid queueing

- Interactive APIs whose callers prefer an immediate `429` to an unpredictable wait.
- Work that will be stale by the time it drains.
- Unbounded or attacker-controlled queues.
- Requests whose client timeout is shorter than the possible wait.

In those cases, reject early or enqueue a durable asynchronous job with a clear `202 Accepted` workflow rather than invisibly holding an HTTP request.

## 8. Token bucket

### 8.1 Plain-language idea

Imagine a bucket holding permission tokens:

- Tokens appear at a steady refill rate `r`.
- The bucket holds at most `B` tokens.
- A request costs one or more tokens.
- If enough tokens exist, spend them and allow immediately.
- Otherwise reject or optionally calculate when enough tokens will exist.

```text
             refill r tokens/s
                    |
                    v
              +-----------+
request cost->|  o o o o  | capacity B
              +-----------+
                    |
                 spend token -> allow
```

### 8.2 State and lazy refill

Store only:

```text
tokens
last_refill_time
```

On a request at time `now`:

```text
elapsed = max(0, now - last_refill_time)
tokens  = min(B, tokens + elapsed * r)

if tokens >= cost:
    tokens -= cost
    allowed = true
else:
    allowed = false

last_refill_time = now
```

Run this read-refill-compare-decrement-write atomically for the key.

No timer needs to add tokens while the user is idle. If the bucket would be full after enough inactivity, the state can expire and a missing key can mean "start full" or another explicitly chosen initial state.

### 8.3 Rate versus burst capacity

- `r` controls the sustainable long-term rate.
- `B` controls how much accumulated idle capacity can be spent at once.

Starting from a full bucket, the traffic admitted over any duration `T` is bounded approximately by:

```text
admitted cost <= B + r * T
```

The exact edge depends on discrete/continuous refill and boundary conventions. This formula expresses the envelope, not a production guarantee.

Example: `r = 10 requests/s`, `B = 50`.

- After idle time, 50 requests may pass immediately.
- After the burst empties the bucket, about 10/s can continue.
- Capacity prevents unlimited accumulation during long inactivity.

### 8.4 Initial state is a product choice

- **Start full:** a new/idle client can burst immediately; common for user-friendly APIs.
- **Start empty:** capacity ramps up; safer for sudden creation of many identities but surprising to clients.
- **Start partially full:** compromise.

An attacker who can create infinite identities can exploit a full initial bucket for each identity. Identity validation and a coarser IP/device/tenant limit remain necessary.

### 8.5 Weighted token bucket

Token cost can reflect estimated work:

```text
GET /video/metadata        cost 1
GET /search?q=...          cost 5
POST /bulk-export          cost 100
GraphQL query              cost from validated complexity
```

Guardrails are required:

- compute cost from parsed/validated server-side semantics;
- cap maximum cost;
- reject malformed or negative cost;
- ensure one batch of 1,000 items is not charged as one cheap request; and
- compare estimated cost with observed downstream load and retune.

### 8.6 Token bucket does not smooth output

A full bucket can release `B` requests at the same instant. This is desirable for human/application bursts but unsafe if a downstream requires evenly paced traffic.

```text
Token bucket:  burst is admitted now, then rate settles
Leaky queue:   burst waits, output is paced
```

### 8.7 Trade-offs

| Dimension | Token bucket |
|---|---|
| State per key | Token balance + last refill timestamp |
| Decision work | O(1) math when atomic at owner |
| Burst | Explicitly bounded by capacity |
| Long-term average | Controlled by refill rate |
| Weighted work | Natural: consume multiple tokens |
| Exact rolling N/W | Different contract; do not describe it as exact rolling-window enforcement |
| Typical fit | Public APIs, per-user/tenant quotas, service admission |

AWS documents that API Gateway uses a token-bucket algorithm and explicitly says throttles are **best-effort targets, not guaranteed ceilings**. This is a valuable production lesson: distributed enforcement and implementation details can permit some overrun even when the conceptual algorithm is clear. See [Amazon API Gateway throttling](https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-throttling.html).

## 9. Sliding-window counters: bounded-memory approximations

An exact timestamp log is accurate but expensive. Sliding-window counters keep a small, bounded number of counters and accept some error.

### 9.1 Variant A: multiple subwindows

Split a 60-second rolling interval into, for example, six 10-second buckets:

```text
oldest                                             newest
+--------+--------+--------+--------+--------+--------+
| 0-10 s |10-20 s |20-30 s |30-40 s |40-50 s |50-60 s|
|   7    |   12   |   9    |   4    |   11   |   3    |
+--------+--------+--------+--------+--------+--------+
```

Maintain a circular ring of `(bucket_epoch, count)` entries. On a request:

1. Find the current subwindow from server time.
2. Reset the slot if its stored epoch is stale; a ring slot is reused over time.
3. Sum counters whose subwindows overlap the last `W` seconds.
4. Optionally weight the partially overlapping oldest subwindow.
5. If estimated usage plus cost is within the limit, increment the current slot.

Do not store only an array position. Store the epoch/version in each slot; after inactivity or a clock jump, an old count in a reused position must not be mistaken for current traffic.

#### Accuracy

If the oldest partially overlapping subwindow is included in full, the result conservatively overcounts by at most the traffic stored in that subwindow. If it is excluded, the result can undercount it. Weighting it assumes the requests were spread across the subwindow.

Smaller subwindows:

- reduce maximum boundary error;
- increase the number of counters and per-decision work; and
- create more rollover/cleanup activity.

The choice is not "accurate or inaccurate" but a tunable **memory/CPU versus error-granularity** trade-off.

### 9.2 Variant B: weighted previous/current fixed windows

Keep only two counters:

- the whole previous fixed window; and
- the current fixed window so far.

Let:

- `W` = window length;
- `e` = elapsed time inside the current window, `0 <= e < W`;
- `P` = previous-window count;
- `C` = current-window count.

The previous window overlaps the rolling interval by a fraction `(W - e) / W`, so estimate:

```text
estimated_usage = P * ((W - e) / W) + C
```

Before admitting a request of cost `c`:

```text
allow only if estimated_usage + c <= limit
```

#### Worked example

Limit: 50 requests per 60 seconds. We are 15 seconds into the current minute. Previous count `P = 42`; current count `C = 18`.

```text
overlap fraction = (60 - 15) / 60 = 0.75
estimate         = 42 * 0.75 + 18
                 = 49.5
```

A new unit-cost request would take the estimate to 50.5, so a strict `<= 50` check rejects it.

This example and formula were used in Cloudflare's dated 2017 engineering description. Cloudflare reported that, on its analyzed dataset, `0.003%` of requests were incorrectly allowed or limited and the average difference from the actual rate was `6%`. Those are **verified figures for that specific historical dataset and implementation, not a universal accuracy guarantee**. See [Cloudflare's sliding-window engineering post](https://blog.cloudflare.com/counting-things-a-lot-of-different-things/).

### 9.3 Why weighting is only an estimate

The formula treats the previous window as though requests were evenly distributed. Consider 40 previous-window requests:

```text
Case 1: all 40 occurred near the old window's beginning
        Most may already be outside the true rolling interval.

Case 2: all 40 occurred near the old window's end
        Most may still be inside the true rolling interval.

Weighted formula sees P = 40 in both cases.
```

Therefore it may:

- reject a request the exact log would allow; or
- allow a request the exact log would reject.

It removes the severe hard reset of fixed windows but does not reproduce exact timestamps.

### 9.4 Cold-start initialization trap

When a key is first seen, what should `P` be?

- `P = 0` gives a new subject a full limit immediately.
- `P = limit` ramps allowance during the first window and can unfairly reject a legitimate new subject.
- Persisting prior state avoids reset abuse but costs storage/durability.

There is no universally correct value; it follows the product's new-identity and abuse policy.

### 9.5 Trade-offs

| Variant | State per key | Decision cost | Accuracy | Useful when |
|---|---:|---:|---|---|
| Exact log | O(requests in W) | Prune/count/add | Exact rolling semantics | Strict, low-to-moderate limits |
| M subwindows | O(M) | Sum/update M, or maintain rolling sum carefully | Tunable granularity | Need bounded state and controllable error |
| Two weighted windows | O(1) | O(1) math | Coarse distribution assumption | Very cheap approximate rolling enforcement |

If maintaining a cached rolling sum for subwindows, rotate and subtract expired buckets atomically. Otherwise concurrent rollover can corrupt the sum even when individual increments are atomic.

## 10. Algorithm comparison and selection

### 10.1 Comparison table

| Algorithm | State per key | Allows initial burst? | Smooths actual output? | Rolling accuracy | Main weakness |
|---|---:|---|---|---|---|
| Fixed window | O(1) | Up to N, plus boundary spike | No | Low near boundary | Nearly 2N across reset |
| Exact sliding log | O(N in window) | Yes, up to N | No | Exact | Memory and sorted-set/log work |
| Multi-subwindow counter | O(M) | Yes | No | Approximate, tunable | Subwindow boundary error |
| Weighted two-window counter | O(1) | Yes | No | Approximate | Assumes uniform prior traffic |
| Token bucket | O(1) | Yes, explicitly B | No | Different rate/burst contract | Burst may hit fragile downstream |
| Leaky queue | O(queue bound) | Arrivals queue | **Yes** | Pacing rather than rolling count | Latency, queue memory, timeouts |
| Leaky meter | O(1) | Configurable tolerance | No queue | Conformance envelope | Terminology/config can confuse |

Here `N` is the allowed request count and `M` the number of subwindows.

### 10.2 Selection cues

Choose **fixed window** when the contract is calendar-based or a cheap coarse cap is sufficient.

Choose **exact sliding log** when every attempt matters, limits are modest, and a true rolling interval is required.

Choose a **subwindow counter** when bounded state matters but you want a known error granularity.

Choose a **weighted two-window counter** when O(1) state and smoother boundaries matter more than exactness.

Choose **token bucket** when you want a sustainable average plus an explicit user-friendly burst.

Choose a **leaky queue** when the downstream requires paced output and waiting is acceptable.

### 10.3 Frequently confused requirements

#### "100 requests per minute"

Could mean:

- at most 100 in each named clock minute -> fixed window;
- at most 100 in any rolling 60 seconds -> sliding log/counter;
- average 100/minute with a burst of 20 -> token bucket;
- output no faster than one every 600 ms -> leaky queue.

Ask which behavior is intended.

#### "No bursts"

An exact sliding log still permits the entire allowance at once if the prior window is empty. "No bursts" points to pacing, a very small token-bucket capacity, or a second short-window limit.

#### "Daily quota"

A continuously refilling token bucket and a UTC-midnight quota are different products. If customers buy 10,000 calls per billing day, a fixed accounting window or durable quota ledger may be the true contract. A short token bucket can still protect instantaneous load.

### 10.4 Combine algorithms when one contract has two timescales

Example:

```text
per user:
    token bucket: 10 requests/s, burst 20       <- instantaneous protection
    daily quota: 10,000 cost units/UTC day     <- commercial entitlement

per service:
    concurrency cap: 500 in flight             <- slowdown protection
```

The request must satisfy each required constraint. Beware that updating several distributed counters atomically is harder than evaluating one key; Section 14.8 covers the consequence.

## 11. Where should the limiter run?

The best production answer is usually **layered**, because each location knows different facts and protects a different resource.

### 11.1 Layered picture

```text
Untrusted client
      |
      v
[CDN / DDoS / WAF edge]  coarse IP, connection, bot, geographic rules
      |
      v
[L4/L7 load balancer]    connection or HTTP-route local safety cap
      |
      v
[API gateway + auth]     API key, user, tenant, route, plan quotas
      |
      v
[Application service]    business action and true cost/context
      |
      v
[Queue / worker]         pace asynchronous work toward dependency
      |
      v
[DB / partner / provider]
```

### 11.2 Client-side limiter

Useful for:

- avoiding accidental bursts;
- SDK retry pacing;
- improving UX before a known reset; and
- reducing needless network calls.

It is not a security boundary. An attacker can modify or bypass the client. Treat it as cooperative optimization only.

### 11.3 CDN/WAF/edge

Advantages:

- rejects abusive traffic before it consumes origin bandwidth, TLS/application work, or gateway capacity;
- sees IP, network, fingerprint, path, and bot signals;
- is geographically close to clients.

Limitations:

- authentication/user context may not yet exist;
- NAT makes many real users share an IP;
- attackers can rotate IPs/identities;
- global counters across all edge locations add coordination latency and are often approximate or scoped per location.

Cloudflare's current documentation says its rate-limit counters include data-center identity and are generally not shared globally across the whole network. That is a concrete example of choosing locality/availability over one exact world-wide counter. See [Cloudflare request-rate calculation](https://developers.cloudflare.com/waf/rate-limiting-rules/request-rate/).

### 11.4 Load balancer

First clarify the layer:

**L4 LB**

- can cap new connections, connections per IP, or packets/bytes;
- usually lacks authenticated user and route semantics;
- is useful before expensive TLS/application work.

**L7 LB/reverse proxy**

- can inspect HTTP method, host, path, headers, and sometimes authenticated metadata;
- can apply per-process or shared route buckets;
- still may not know business cost or response outcome.

A local limiter on each LB instance protects each instance cheaply but does not automatically enforce one fleet-wide per-user quota.

### 11.5 API gateway

Often the best primary location for public API limits because it has:

- authenticated API key/user/tenant;
- normalized route information;
- centralized policy integration;
- a chance to reject before application execution;
- consistent `429`, headers, logging, and observability.

Cost: every request may require a call to shared state, and a gateway rule may still be too coarse for business-specific operations.

Envoy illustrates both choices: its local HTTP rate-limit filter uses a token bucket per Envoy process by default, while global rate limiting calls an external rate-limit service with shared storage. Envoy also documents combining local and global limiting so a local cap reduces load on the global service. See [Envoy local rate limiting](https://www.envoyproxy.io/docs/envoy/latest/intro/arch_overview/other_features/local_rate_limiting) and [Envoy Gateway global versus local rate limiting](https://gateway.envoyproxy.io/docs/concepts/rate-limiting/).

### 11.6 Application/service

Advantages:

- knows that `/graphql` query A costs 1 while query B costs 1,000;
- can count only failed OTP attempts or successful expensive actions;
- can protect one database table, partner, or business workflow;
- supports per-resource rules such as `(seller, product)`.

Limitations:

- abusive traffic already reached the service;
- every service may implement subtly different semantics;
- policy rollout and observability become fragmented.

Use a shared library/sidecar/platform where possible, but keep cost calculation and business descriptors inside trusted service code.

### 11.7 Queue/consumer

For asynchronous work, admission and execution pacing are separate:

```text
API --> validate + enqueue --> durable queue --> paced workers --> provider
           |                                      |
       quota admission                       leaky/token pacing
```

The API can return `202 Accepted`; a worker drains at the provider's safe rate. Bound queue size/age, expose job state, and expire stale work. Do not hide a long unbounded queue behind a synchronous API call.

### 11.8 Recommended interview answer

> "I would use layered limits: a coarse local edge/LB limiter to cheaply absorb obvious abuse and protect the rate-limit service itself; a gateway-level shared limiter for authenticated tenant/user/route quotas; and service or worker limits for true business cost and fragile dependencies. The layers have different keys and do not pretend to be one atomic global rule."

## 12. Cache and state choices

### 12.1 Separate configuration from live counters

```text
Control plane                               Data plane

Admin UI/API                               Request
    |                                         |
Rules service --> durable rules DB            v
    |                                     Gateway/filter
    +--> versioned publish/invalidate          |
              |                                v
              +--> local rules caches --> Rate-limit decision
                                                   |
                                                   v
                                            live state store
```

**Configuration path** can tolerate seconds of propagation if policy rollout semantics are explicit. **Decision path** must be low latency and available because it runs for every request.

### 12.2 Local in-process state

Pros:

- no network hop;
- very low latency;
- survives shared-store slowness as a fallback;
- useful for per-process safety caps and denial caches.

Cons:

- each of 1,000 servers has a different counter;
- server restart loses state;
- autoscaling and load imbalance change effective limits;
- sticky routing is needed for stable per-key ownership;
- world-wide/global policy is not exact.

Good local uses:

- emergency per-instance overload cap;
- cooperative local portion leased from a global budget;
- short cache of "this key is blocked until T" after a global rejection;
- cached versioned rules; and
- fallback behavior during shared-store failure.

### 12.3 Shared Redis-style state

Pros:

- all gateway/app instances see one owner for a key;
- atomic counters/scripts are fast;
- TTL cleans inactive state;
- sharding distributes different identities.

Cons:

- one remote decision per request adds latency and network cost;
- the store becomes critical-path infrastructure and an attack target;
- hot keys can overload one shard;
- failover and multi-region replication may permit under/over-enforcement;
- Redis memory overhead can be material at high cardinality.

Call it a **distributed in-memory state store** when counters are authoritative. A durable policy database remains separate.

### 12.4 Rules cache

Gateways/rate-limit servers should cache rule definitions locally:

- keyed by tenant/route/plan;
- versioned;
- refreshed via push/pub-sub plus periodic polling as repair;
- bounded by TTL/size;
- last-known-good retained during control-plane outage.

Policy changes need defined semantics. If a limit changes from 100 to 10 midway through a window, decide whether the new rule immediately observes old usage, starts new state using `policy_version` in the key, or migrates balances. Versioning every update resets state and can be abused if configs churn; reusing state can make a new policy unexpectedly exhausted.

### 12.5 Hybrid 1: cache negative/block decisions

If the global service says a key is blocked until `T`, the gateway can reject the same key locally until `T` without another shared lookup.

Advantages:

- attacks do not hammer Redis for every rejected request;
- lower latency and cost while blocked.

Risks:

- a policy override/unblock may propagate late;
- inaccurate `T` or clock skew can over-block;
- high-cardinality attackers can fill the denial cache.

Use bounded caches, trusted time, policy versions, and invalidation.

### 12.6 Hybrid 2: centrally leased token chunks

Instead of asking Redis for every request, a server atomically reserves `K` tokens from a global budget, then spends them locally:

```text
Global bucket: 10,000 tokens
       |
       +-- lease 100 --> Gateway A local bucket
       +-- lease 100 --> Gateway B local bucket
       +-- lease 100 --> Gateway C local bucket
```

Benefits:

- roughly one global call per `K` admitted cost units rather than per request;
- very fast local decisions;
- Redis is shielded during peaks.

Trade-offs:

- tokens stranded on an idle/crashed server reduce usable capacity until lease expiry/reclamation;
- smaller `K` is more accurate/fair but causes more global calls;
- larger `K` improves performance but lets one server hold more unused budget;
- lease expiry and window rollover are subtle;
- reusing a lease after a retry/failover requires a unique lease ID/fencing.

If tokens are truly deducted atomically when leased, the central allocation need not overshoot merely because it is batched; the common cost is **underutilization/stranding**. Overshoot appears if servers mint independent local budgets, reuse leases, or replenish before global reconciliation.

### 12.7 Hybrid 3: independent local budgets plus asynchronous aggregation

Each region/server admits locally and reports counts periodically. This maximizes availability and minimizes latency but is approximate:

```text
possible overshoot roughly grows with
number of independent decision makers * unsynchronized local allowance
```

Use it only when the business tolerates bounded inconsistency, and explicitly calculate/budget the maximum overshoot. It is unsuitable for a hard financial or security limit unless paired with a stricter authority.

### 12.8 Cache/store selection table

| Need | Natural choice | Why |
|---|---|---|
| Rules and plan definitions | Durable DB + local versioned cache | Admin/audit durability with fast reads |
| Per-process overload cap | Local memory | No dependency in emergency path |
| Fleet-wide per-user decision | Sharded shared state such as Redis | One atomic owner per key |
| Exact rolling timestamps | Sorted set/log-like structure | Retain each event in window |
| Token/fixed/weighted state | String/hash fields | Small O(1) state |
| Reduce global calls | Leased local token chunks | Batch allocation with bounded trade-off |
| Global financial entitlement | Durable transactional ledger/reservation system | Redis-only best-effort counters may be insufficient |

## 13. What if there are 1,000 servers?

### 13.1 Why independent local counters fail

Global requirement: user `u42` may make 100 requests/minute.

If each of 1,000 servers independently allows 100:

```text
effective theoretical fleet allowance = 1,000 * 100
                                        = 100,000/minute
```

This is an **illustrative worst case**, assuming the client can spread traffic and every server has a full local allowance.

### 13.2 Why `100 / 1000` is not the fix

Giving every server 0.1 requests/minute is nonsensical for discrete requests. Even with larger quotas:

- traffic is not evenly distributed;
- server count changes during autoscaling/deployments;
- one server may have unused capacity while another rejects;
- failures change the effective total;
- clients can deliberately choose paths/connections that spread load.

Static division is acceptable only for a deliberately coarse **per-instance safety cap**, not a fair global subject limit.

### 13.3 Design A: stateless gateways + shared sharded state

```text
                         +---------------- Control plane ----------------+
Admin --> Rules API --> Rules DB --> config publish --> local rule caches
                         +-----------------------------------------------+

Clients
   |
   v
Edge/LB
   |
   v
1,000 gateway/app instances
   |  descriptor: rule + subject + cost
   v
Rate-limit service fleet (stateless compute)
   |
   +--> hash key --> Redis shard 0 primary + replica
   +--> hash key --> Redis shard 1 primary + replica
   +--> hash key --> Redis shard 2 primary + replica
   `--> ...
```

Every request for the same key routes to the same Redis primary, regardless of which application server received it. Different keys spread across shards.

Properties:

- application and rate-limit service instances scale horizontally;
- atomicity is handled at the state owner;
- Redis capacity scales by sharding keys;
- a hot subject/rule remains one hot shard/key and needs separate mitigation;
- each request pays an RLS/store network round trip unless co-location, pipelining, or leasing is used.

### 13.4 Design B: route each key to one stateful rate-limit owner

Use consistent/rendezvous hashing to send `(rule, subject)` to one rate-limit node with local memory:

```text
hash(rule, subject) --> owner node --> local bucket
```

Pros:

- no external store hop on every request;
- very fast atomic local state;
- natural key sharding.

Cons:

- owner failure/state recovery is difficult;
- membership changes move keys and can reset/double state;
- replication/failover must preserve ordering and fencing;
- callers need routing or a proxy layer;
- hot keys still concentrate.

This can be excellent at very high scale but is a more complex stateful service, not "just use consistent hashing."

### 13.5 Design C: local leased budgets

Gateways reserve chunks from global shards as described in Section 12.6. This is a common performance direction when one remote check per request is too costly.

Explain the bounded trade-off rather than claiming exact global state:

- Central leasing can preserve the allocation ceiling if leases are unique and deducted once.
- Unspent leases reduce availability/fairness.
- A failover that reissues a lease without fencing can overspend.
- Per-region preallocation makes region independence explicit.

### 13.6 Shard key design

A typical state key hashes a stable high-cardinality subject:

```text
hash(rule_id + tenant_id + user_id) -> Redis shard
```

Do not shard only by `rule_id`: every user on a popular rule would hit one shard. Do not include a random suffix: requests for the same quota would be split and over-admitted.

Related notes: [06. Hashing and Consistent Hashing](./06.%20Hashing%20and%20Consistent%20Hashing.md) and [09. Key-Value Store](./09.%20Key-Value%20Store.md).

### 13.7 Hot keys

Examples:

- a fleet-global emergency counter;
- an unauthenticated key that collapses all traffic into `anonymous`;
- one huge tenant;
- one viral resource with a per-resource rule.

Mitigations:

- enforce a local first-stage cap before the global check;
- reserve/lease token chunks;
- isolate the tenant/rule on dedicated capacity;
- use a hierarchy: per-instance safety plus global approximate aggregation;
- route the hot key to a specialized replicated owner;
- rate-limit by finer identity where semantics allow; or
- use load shedding/concurrency control based on live capacity instead of one exact global counter.

Randomly sharding one logical counter and summing later lowers contention but weakens immediate global enforcement. State that trade-off.

### 13.8 High-cardinality memory attack

An attacker can generate millions of random API keys, usernames, paths, or spoofable header values so each request creates new state.

Defenses:

- construct keys only from validated/bounded fields;
- cap key length and hash sensitive identifiers;
- use TTL and bounded memory;
- reject impossible/unknown identities before allocating state;
- place coarse IP/network limits before fine identity limits;
- cap metric label cardinality; and
- consider probabilistic sketches for detection, not exact per-identity billing.

### 13.9 Capacity-estimation formulas

If every incoming request gets one global decision:

```text
rate-limit decision QPS ~= incoming request QPS * checks per request
```

If each request evaluates three independent remote limits, naive state-operation load can triple. A script may combine same-slot rules, but different identity dimensions may not colocate.

Live-state count:

```text
active keys ~= active subjects during TTL horizon * applicable rules/subject
```

Logical memory:

```text
logical state ~= active keys * logical bytes/key
actual store memory = logical payload + key/object/index/allocator/replication overhead
```

Illustrative example only: 10 million active keys times 64 logical bytes is about 640 MB of payload, but actual Redis memory can be several times the payload depending on key length, data type, encoding, and allocator. Benchmark with representative keys rather than repeating a universal bytes-per-key claim.

For exact sliding logs:

```text
entries ~= sum of admitted events still inside every active window
```

This often dominates memory rather than the number of identities.

### 13.10 Multi-region options

| Option | Latency/availability | Global accuracy | Main trade-off |
|---|---|---|---|
| One global home region per key | Remote regions may pay WAN RTT/fail on partition | Strongest single-owner story | Latency and regional dependency |
| Sticky regional ownership | Local after routing to home | Strong per key if one fenced owner | Failover complexity |
| Preallocate budget per region | Local decisions, region-independent | Global ceiling if allocation/fencing correct; capacity can strand | Rebalancing and failover |
| Independent regional counters + async merge | Best local availability | Approximate; overshoot during lag/partition | Must tolerate/bound inconsistency |
| Synchronous consensus store | Higher coordination latency | Stronger consistency | Throughput, cost, availability during partition |

An eventually consistent merge cannot undo requests already admitted. "Counters converge later" is not equivalent to enforcing the limit at decision time.

### 13.11 A practical layered 1,000-server answer

```text
1. Each gateway has a generous local emergency token bucket.
2. Authenticated per-user/tenant/route limits call a sharded regional RLS.
3. RLS performs one atomic decision on the key's Redis primary.
4. Hot/high-volume policies may use fenced token leases.
5. Regions receive explicit budget allocations or route a key to a home region.
6. Business-critical hard quotas use a durable entitlement/reservation system.
7. On RLS failure, policy-specific fail-open/closed/fallback behavior applies.
```

There is no requirement that 1,000 application servers talk to one Redis machine. The **logical state is shared** while both decision compute and storage are horizontally partitioned.

## 14. Atomicity in Redis operations

### 14.1 What Redis atomicity does mean

An individual Redis command such as `INCR` executes atomically on its owning server. Redis also guarantees atomic execution of a Lua script or Redis Function: other commands do not interleave while it runs. The script must therefore be short, because it blocks server work during execution. See [Redis programmability and atomic execution](https://redis.io/docs/latest/develop/programmability/).

This is enough to make one key's sequence:

```text
read -> refill/prune -> decide -> update -> set expiry -> return
```

indivisible on one Redis shard.

Here **atomic primarily means no interleaving**, not SQL-style rollback. Redis transactions do not roll back successful commands because another queued command fails, and a script that encounters a runtime error after earlier writes does not provide a general automatic rollback guarantee. Validate arguments and types before the first write and design the script so later operations cannot unexpectedly fail. See [Redis' discussion of errors and rollback](https://redis.io/docs/latest/develop/using-commands/transactions/#why-redis-does-not-support-roll-backs).

### 14.2 What it does not mean

It does not automatically provide:

- a transaction with the application database;
- exactly-once charging across ambiguous network retries;
- atomic updates to arbitrary keys on different Redis Cluster slots;
- synchronous replication to all replicas/regions;
- no data loss during every failover;
- fairness/order across multiple regions writing independently.

Atomicity, durability, consistency, and idempotency are separate properties.

### 14.3 Broken token-bucket implementation

```text
tokens, last = HGET key tokens last
new_tokens = refill(tokens, last, now)
if new_tokens >= cost:
    HSET key tokens (new_tokens - cost), last now
    return ALLOW
return REJECT
```

Two callers can read the same balance and both allow. A pipeline reduces round trips but does **not** make this conditional workflow atomic.

### 14.4 Correct single-key shape with a short script/function

Conceptual Lua-like pseudocode:

```lua
-- KEYS[1] = bucket key
-- ARGV = capacity, refill_per_ms, cost, ttl_ms

local capacity      = tonumber(ARGV[1])
local refill_per_ms = tonumber(ARGV[2])
local cost          = tonumber(ARGV[3])
local ttl_ms        = tonumber(ARGV[4])

local time_parts = redis.call('TIME')
local now = tonumber(time_parts[1]) * 1000
          + math.floor(tonumber(time_parts[2]) / 1000)
local state = redis.call('HMGET', KEYS[1], 'tokens', 'last_ms')

local tokens = tonumber(state[1]) or capacity
local last   = tonumber(state[2]) or now
local elapsed = math.max(0, now - last)
tokens = math.min(capacity, tokens + elapsed * refill_per_ms)

local allowed = 0
if tokens >= cost then
    tokens = tokens - cost
    allowed = 1
end

redis.call('HSET', KEYS[1], 'tokens', tokens, 'last_ms', now)
redis.call('PEXPIRE', KEYS[1], ttl_ms)
return {allowed, tostring(tokens)}
```

This is an explanatory skeleton, not drop-in production Lua: numeric precision, fixed-point arithmetic, validation, missing fields, TTL policy, Redis-version behavior, and return types must be implemented and tested carefully.

Why use server time? It prevents 1,000 application clocks from refilling the same bucket inconsistently. Redis documents `TIME` as returning server seconds and microseconds. See [Redis `TIME`](https://redis.io/docs/latest/commands/time/).

For precise token arithmetic, integer fixed-point units are often safer than unconstrained floating-point values:

```text
1 request token = 1,000 microtokens
refill and balances use bounded integers
```

Guard against overflow, negative elapsed time, invalid config, and an impossible `cost > capacity` according to explicit policy.

### 14.5 `MULTI/EXEC`, `WATCH`, and Lua/Functions

**`MULTI/EXEC`**

- queues commands and executes them sequentially without another client's command interleaving;
- is good for a known unconditional sequence such as increment plus expiry;
- cannot conveniently branch inside the transaction based on a just-returned value.

**`WATCH` + `MULTI/EXEC`**

- provides optimistic compare-and-set;
- client reads state, computes, and commits only if watched keys did not change;
- under a hot key, many clients may abort/retry, increasing contention and latency.

**Lua script / Redis Function**

- executes the conditional read-decide-update next to the data in one call;
- is usually the clearest token/sliding-window implementation;
- must be short, deterministic/effect-safe for the deployed version, versioned, monitored, and tested.

Redis 7 introduced persisted Redis Functions as a first-class alternative to application-shipped `EVAL` scripts; environments and managed services vary, so choose based on supported deployment/operations. Redis' transaction guarantees and optimistic locking are documented at [Redis transactions](https://redis.io/docs/latest/develop/using-commands/transactions/).

### 14.6 `INCR` and expiry: a subtle race

Broken sequence:

```text
count = INCR key
if count == 1:
    EXPIRE key 60
```

If the client dies after `INCR` but before `EXPIRE`, the key may never expire. Future traffic can remain incorrectly limited and memory leaks.

Fixes:

- perform the conditional increment+expiry in Lua/Function; or
- use a fixed-window key and an atomic transaction that sets expiry on every increment, accepting the chosen TTL behavior; or
- use a server command that provides the required combined semantics in the deployed Redis version.

Do not merely say "both commands are atomic." Each is atomic separately; the workflow is not.

### 14.7 Redis Cluster slots

Redis Cluster shards keys into hash slots. Multi-key transactions/scripts generally require their keys to be in the same slot. Hash tags force related keys to colocate:

```text
rl:{user42}:search
rl:{user42}:write
```

Both hash on `{user42}`. Redis documents this single-slot restriction and hash-tag mechanism in the [Redis Cluster specification](https://redis.io/docs/latest/operate/oss_and_stack/reference/cluster-spec/).

Trade-off: putting every rule for a huge tenant under one tenant hash tag can create a hot shard. Colocate only the state that truly needs one atomic operation.

### 14.8 Multiple applicable limits may not be atomically composable

Suppose a request consumes both:

```text
per-user key: user42
per-IP key:   203.0.113.7
per-tenant:   tenant17
```

These dimensions naturally hash to different slots. There is no key naming scheme that colocates every user with every IP and also preserves shared per-IP and per-tenant aggregation.

Options:

1. **Sequentially consume each limit.** If a later rule rejects, earlier counters remain charged. This is conservative but may over-limit.
2. **Precheck then consume.** Races can allow overshoot between phases; not atomic.
3. **Central coordinator/transactional store.** Stronger semantics but higher latency/complexity.
4. **One composite key.** Atomic for `(tenant,user,IP)` but no longer enforces the true aggregate user across IPs or IP across users.
5. **Layered independent enforcement.** Edge IP rule and gateway user/tenant rule are deliberately separate; accept bounded non-atomicity.

Interview-quality answer: state which safety direction is preferred. For abuse control, conservative overcount may be acceptable. For paid quota, use reservations/idempotency and a durable ledger rather than casual sequential decrements.

### 14.9 Network timeout after Redis committed

Timeline:

```text
Gateway --> Redis: spend token
Redis commits and replies
network drops reply
Gateway does not know whether token was spent
Gateway retries
```

The retry may spend twice. Common API rate limiting tolerates occasional conservative overcharging. If it does not:

- include a stable decision/attempt ID;
- atomically remember the ID and its result for the retry horizon;
- return the prior result on duplicate;
- bound dedupe memory with TTL; and
- ensure one business operation's retry semantics match the decision ID.

This adds state and can itself be attacked with high-cardinality IDs. Use it only where the contract justifies it.

Also decide whether a token is refunded when the downstream request never starts or fails. Most admission limiters charge attempts because transactional refund with the business operation is complex and refunds can be gamed. Billing quotas require stronger accounting.

### 14.10 Replication and failover

Redis Open Source replication is asynchronous by default. A primary can acknowledge an increment before a replica receives it; failover can therefore lose recent rate state and temporarily allow extra traffic. Redis' `WAIT` command can reduce the probability by waiting for replica acknowledgements, but Redis explicitly says it does not turn the deployment into a strongly consistent CP system and acknowledged writes can still be lost depending on configuration. See [Redis replication](https://redis.io/docs/latest/operate/oss_and_stack/management/replication/).

Trade-off:

- waiting for replicas improves state safety;
- it adds latency and may reduce availability/throughput;
- many rate limits tolerate bounded extra admission and favor speed;
- hard entitlements may require another storage/consensus design.

### 14.11 Long scripts and hot shards

Redis atomic scripts block other server work while executing. Avoid:

- scanning unbounded logs;
- pruning enormous sets in one call;
- touching unnecessary keys;
- complex policy interpretation inside Redis;
- user-controlled loops; and
- returning huge payloads.

Keep policy matching/cost calculation in the trusted rate-limit service; send Redis a compact, validated atomic state transition. Monitor script latency and slow logs.

### 14.12 Redis implementation checklist

```text
[ ] trusted key and server-side time
[ ] one atomic read-decide-update per state owner
[ ] TTL for inactive state
[ ] fixed-point/overflow/config validation
[ ] unique sorted-set members for exact logs
[ ] same-slot keys for every atomic multi-key operation
[ ] explicit handling of ambiguous client retry
[ ] known failover/replication overshoot behavior
[ ] bounded script runtime and state cardinality
[ ] hot-key and Redis-unavailable fallback
[ ] policy version/change semantics
```

## 15. Failure behavior: the limiter must not become the outage

### 15.1 Fail open, fail closed, or degrade locally

If the gateway cannot reach the rate-limit service/store:

**Fail open** - allow the request.

- Preserves availability.
- Risks overload, abuse, or exceeding third-party cost.
- Reasonable for low-risk reads if downstream has independent overload protection.

**Fail closed** - reject the request.

- Preserves a hard security/cost boundary.
- Makes the rate limiter a dependency whose outage denies service.
- Reasonable for OTP, login abuse, expensive paid operations, or contractual hard limits depending on product policy.

**Local degraded fallback** - use a conservative in-process bucket.

- Gives bounded service during shared-store outage.
- Is not globally exact across servers.
- Needs a short expiry and clear recovery behavior.

Failure mode should be **per rule**, not one global switch:

| Rule | Plausible failure policy | Reasoning |
|---|---|---|
| Product catalog read | Fail open with local emergency cap | Availability favored; reads still protected locally |
| Password/OTP attempts | Fail closed or challenge | Security boundary dominates |
| Paid third-party call | Fail closed or queue | Unbounded calls create cost/contract risk |
| Recommendation refresh | Fail open/degrade to cached response | Non-critical feature |
| Fleet overload cap | Local-only fallback must exist | A remote limiter cannot protect you when it is down |

### 15.2 A robust fallback chain

```text
normal:     local coarse cap -> global shared decision -> application

RLS slow:   local coarse cap -> short timeout -> policy fallback
                                      |
                       +--------------+--------------+
                       |                             |
                   fail closed                 fail open/local
```

Requirements:

- a strict, small timeout so limiter slowness does not consume the whole request budget;
- a circuit breaker around the limiter dependency;
- local emergency rate/concurrency caps;
- cached last-known-good rules;
- metrics separating normal allows from fallback allows;
- alerting before fallback traffic overloads downstream; and
- automatic recovery without every server retrying at once.

### 15.3 Rate-limit service overload

The limiter itself sees attacker traffic. Protect it with:

- cheap local gateway caps before global checks;
- bounded connection/request queues;
- timeouts and load shedding;
- negative/block-decision caches;
- request coalescing or token leasing where safe;
- isolated capacity from the protected service;
- hot-key detection; and
- a compact protocol rather than shipping full HTTP requests.

Do not synchronously log each decision to a slow database. Emit bounded/asynchronous metrics or samples.

### 15.4 Configuration/control-plane outage

Gateways should continue with a last-known-good, versioned configuration. A rules database outage should not erase every rule. Updates should be staged:

```text
validate -> publish version -> shadow/evaluate -> partial rollout -> enforce -> monitor
```

Reject invalid policies such as negative rates, zero window with positive quota, `cost > capacity` without defined handling, unbounded descriptor fields, or conflicting priority.

### 15.5 Region partition

During a WAN partition, choose explicitly:

- route/reject requests that cannot reach the key's home authority, preserving the quota;
- use a previously allocated regional budget, preserving a bounded total;
- admit against an independent regional budget, accepting overshoot; or
- fail closed for sensitive rules and open/fallback for low-risk rules.

CAP/PACELC applies to the counter decision: exact one-owner consistency across partitions competes with local availability and latency. See [02. CAP, PACELC and Replication Strategies](./02.%20CAP,%20PACELC%20and%20Replication%20Strategies.md).

### 15.6 Safe rollout with shadow mode

Before rejecting production traffic:

1. Run the policy in **shadow mode**: calculate decisions but do not enforce.
2. Measure would-block rates by endpoint, tenant tier, identity type, and legitimate/error outcome.
3. Inspect NAT/shared-IP and batch-client false positives.
4. Roll out to a small percentage or selected tenants.
5. Provide an override/kill switch with audit.
6. Monitor downstream health as well as 429 counts.

The open-source Envoy/Lyft rate-limit service supports rule and global shadow modes, a practical example of this operational pattern. See [Envoy proxy rate-limit service](https://github.com/envoyproxy/ratelimit).

## 16. HTTP and client behavior

### 16.1 `429 Too Many Requests`

RFC 6585 defines HTTP `429 Too Many Requests` for rate limiting and says a response **may** include `Retry-After`. The response should explain the condition. See [RFC 6585, section 4](https://www.rfc-editor.org/rfc/rfc6585.html).

Example:

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/problem+json
Retry-After: 2

{
  "type": "https://api.example.com/problems/rate-limit",
  "title": "Rate limit exceeded",
  "rule": "search_per_user",
  "retry_after_ms": 1840
}
```

`Retry-After` can be a delay in seconds or an HTTP date according to HTTP semantics. A conservative delay is safer than promising an exact future admission when multiple limits or dynamic overload rules apply.

### 16.2 `429` versus `503`

- Use **429** when this client/identity/policy has exhausted its allowed usage.
- Use **503 Service Unavailable** when the service as a whole is temporarily unable to handle work, often with `Retry-After`.
- A global emergency limiter might be represented as 503 if the client did nothing individually wrong.

The distinction helps clients and operators understand whether to reduce their own usage or treat the service as unhealthy.

### 16.3 Rate-limit response fields are not yet a final standard

Many APIs use vendor-specific fields such as:

```text
X-RateLimit-Limit
X-RateLimit-Remaining
X-RateLimit-Reset
```

As of 2026-07-16, the IETF HTTPAPI work for standardized `RateLimit`/`RateLimit-Policy` fields is still an **Internet-Draft**, explicitly a work in progress, not a final RFC. Its syntax may change. See [current IETF RateLimit header-fields draft](https://datatracker.ietf.org/doc/draft-ietf-httpapi-ratelimit-headers/).

Do not invent one "universal" header contract. Document your API's exact fields, units, reset meaning, and whether values are hints.

### 16.4 Good client behavior

On `429`:

1. Do not retry immediately in a tight loop.
2. Respect `Retry-After` if present and reasonable.
3. Add jitter so thousands of clients do not retry at the same instant.
4. Use exponential backoff for repeated failures.
5. Bound attempts and total retry time.
6. Retry only idempotent operations or use an idempotency key.
7. Reduce concurrency/batch size and cache results where possible.

```text
delay = max(server_hint, exponential_backoff) + random_jitter
```

If every client retries precisely at the advertised reset boundary, the system creates a thundering herd. Token refill or randomized retry hints can smooth recovery.

### 16.5 What should `remaining` mean?

With one fixed window it may be an integer. With weighted counters, multiple policies, local leases, or dynamic overload controls it may be approximate.

Choices:

- report the most restrictive remaining policy;
- return policy-specific values;
- expose only when near exhaustion;
- omit it if inaccurate or sensitive.

Avoid revealing internal fleet capacity or another user's usage. Rate-limit keys and headers can become privacy/security side channels.

### 16.6 Caching 429 responses

Do not let a shared HTTP cache replay one user's 429 to everyone. Responses keyed by authenticated identity are normally private/non-cacheable unless cache behavior is deliberately designed. A gateway's internal local denial cache is different: it keys on the trusted rate-limit subject and policy version.

## 17. Identity, counting, and edge cases

### 17.1 IP address is useful but imperfect

Problems:

- carrier-grade NAT, offices, schools, and families share one public IP;
- IPv6 clients may rotate addresses within a prefix;
- mobile clients change networks;
- proxies can hide the original address;
- botnets distribute traffic across many IPs;
- forwarding headers are spoofable unless set by trusted infrastructure.

Use IP as one signal/layer, not the universal identity. Combine it with authenticated user/API key, session/device, network reputation, and behavioral signals.

### 17.2 Pre-authentication and post-authentication keys

For login:

```text
before credentials verified:  IP / device / account-name attempt key
after authentication:          stable user / tenant / API-key key
```

Limit both `IP -> many accounts` and `many IPs -> one account`. Avoid exposing whether an account exists: normalize responses and timing where security requires.

### 17.3 Attempts versus successful operations

Possible accounting points:

```text
request arrival -> admitted -> downstream started -> success committed -> response returned
```

- Count arrival for traffic/capacity protection.
- Count admitted attempts for ordinary API fairness.
- Count failed authentication outcomes for brute-force rules.
- Count committed operations for commercial quota if retries must not double-charge.

One policy may need more than one counter. Do not make a slow response-dependent update part of the early edge critical path unless necessary.

Cloudflare's current best-practices documentation includes response-based counting for failed login/OTP status codes, demonstrating that "what counts" can depend on the response, not merely the incoming path. See [Cloudflare rate-limiting best practices](https://developers.cloudflare.com/waf/rate-limiting-rules/best-practices/).

### 17.4 Retries and idempotency

Suppose `POST /orders` reaches the service, commits an order, but its response is lost. The client retries:

- the limiter may charge twice;
- without an idempotency key, the service may create two orders.

Rate limiting does not solve business idempotency. Use a stable idempotency key for the order operation. Decide whether quota means network attempts or unique committed operations; implement dedupe only if the latter is required.

### 17.5 Batch and GraphQL requests

Request count is a poor proxy for cost:

```text
GET /items/1                         one cheap lookup
POST /items/batch with 10,000 IDs   one expensive HTTP request
GraphQL nested query                one request, large fan-out
```

Use weighted cost, maximum batch size/query depth, execution timeouts, result-size limits, and concurrency limits. Cost estimates can be wrong; monitor actual CPU/DB/fan-out and update weights.

### 17.6 WebSockets, streaming, and downloads

Counting only the HTTP handshake misses long-lived work. Consider separate limits for:

- connection attempts/second;
- simultaneous connections per user/IP;
- messages/second and bytes/second per connection/user;
- subscriptions/channels per socket;
- server fan-out cost; and
- media egress bandwidth.

A connection limiter is a concurrency limit; a bytes/second token bucket is a bandwidth limiter; a message limiter is an application rate. They may coexist.

### 17.7 Multiple devices and shared tenants

Choose whether a quota is:

- per device (easy to bypass by adding devices);
- per user across all devices;
- per API key/app;
- per tenant shared by all users; or
- hierarchical: tenant pool plus per-user fairness.

A tenant-wide bucket alone lets one user consume everything. A per-user bucket alone lets a tenant with many users exceed its purchased total. Enforce both, accepting or designing around multi-key atomicity limits.

### 17.8 Priority and reserved capacity

When overload matters, equal first-come-first-served can starve critical work. Use separate buckets/reservations:

```text
total capacity
  +-- reserved: checkout/payment
  +-- reserved: interactive reads
  `-- elastic: background export/reindex
```

Allow unused reserved capacity to be borrowed only with a safe reclaim policy. Priority scheduling plus concurrency/load shedding may be more appropriate than static rates alone.

### 17.9 Clock and boundary bugs

Test:

- exactly at `now - W` and exactly at reset;
- time moving backward;
- very large forward jumps;
- daylight-saving changes (avoid local civil time for duration windows);
- leap/clock synchronization behavior relevant to the chosen clock;
- millisecond collisions; and
- config changing during a window.

Use UTC/epoch for calendar windows and monotonic/server elapsed time for refill calculations where supported.

### 17.10 Limit bypass through route/key variation

Normalize before keying:

- `/users/42`, `/users/042`, encoded path variants;
- query parameter ordering;
- case where semantics are case-insensitive;
- API versions/aliases;
- trailing slashes;
- alternate hostnames; and
- multiple API keys belonging to one tenant.

Key on the matched canonical route template such as `GET:/users/{id}`, not the raw path when a per-endpoint rule is intended. Do not collapse resource IDs if the policy is deliberately per-resource.

### 17.11 Limit evasion and denial attacks

Attackers can:

- rotate identity to evade limits;
- intentionally exhaust a victim's account quota;
- send requests just below thresholds;
- create high-cardinality state/metrics;
- force expensive policy matching; or
- exploit retry/reset synchronization.

Rate limiting is adversarial policy, not only capacity math. Build override/audit tools and combine it with abuse detection.

## 18. Alternatives and complementary mechanisms

### 18.1 Concurrency/adaptive limiters

Use when dependency latency varies or slow work is the risk. Admit only while in-flight work is below a limit; adaptive algorithms may change the limit based on observed latency/errors. This reacts to current health in a way a fixed request rate cannot.

Avoid only using concurrency caps for abuse quotas: a fast attacker can complete and start unlimited operations over time while staying below concurrency.

### 18.2 Bounded queues and backpressure

Use when work can wait and the producer/consumer contract supports it. A bounded queue absorbs a small burst. When full:

- reject;
- block a cooperative producer;
- drop low priority; or
- spill to a durable queue if semantics allow.

An unbounded queue converts overload into memory exhaustion and extreme latency.

### 18.3 Load shedding

Drop work based on live system health/priority rather than a fixed per-user budget. Useful for brownouts and sudden capacity loss. It does not provide predictable tenant entitlement by itself.

### 18.4 Circuit breaker

Stops calls to a dependency that is failing, then probes recovery. A circuit breaker responds to failure state; a rate limiter controls request volume. They often appear together but are not interchangeable.

### 18.5 Autoscaling

Adds capacity after observing/predicting load. It is slower than rejecting one request and cannot solve unlimited abuse or a fixed downstream quota. Rate limiting protects the system during scaling lag and after hard capacity is reached.

### 18.6 Work coalescing, caching, and batching

Sometimes the best limit is less work:

- cache popular reads;
- coalesce concurrent identical requests;
- debounce client refreshes;
- batch small downstream calls;
- precompute results; and
- use conditional requests.

Be careful: poorly bounded batching can create more expensive individual operations.

### 18.7 Probabilistic structures

Count-Min Sketch or sampling can find heavy hitters with bounded memory, but collisions/estimation make them unsuitable as the sole exact paid quota. They are useful for detection, prioritizing which identities receive exact counters, or emergency abuse defense.

## 19. Real-system and product examples

This section distinguishes documented public behavior from illustrative internal design. Public API limits change; follow the linked current documentation rather than memorizing numbers.

### 19.1 WhatsApp-style messaging

**Illustrative design, not a claim about WhatsApp's private architecture.**

A messaging platform needs different limits:

```text
sender user/device
     |
     +-- messages/s per sender -------- spam/fairness
     +-- messages/s sender->recipient - protect one recipient
     +-- group fan-out cost units ----- 1 send may create many deliveries
     `-- media bytes/s ---------------- bandwidth/storage
```

A token bucket suits normal bursty human messages. A pair `(sender, recipient)` limit prevents one sender flooding one victim. A weighted group-message cost or separate fan-out limiter protects delivery work. A worker queue can pace outbound calls to a business messaging provider.

Follow-up: a message accepted into durable storage is not the same as all fan-out deliveries completing. Rate-limit admission before durable acceptance and use backpressure/load shedding on asynchronous delivery consumers.

### 19.2 YouTube Data API: weighted quotas

**Public product behavior, current when checked.**

The YouTube Data API assigns quota cost to API methods rather than treating all requests identically. Its 2026 documentation also describes separate granular quota buckets for some methods and a combined quota for others. This is a real example of:

- weighted operations;
- more than one applicable quota bucket;
- project/application identity;
- long accounting windows; and
- quota-extension workflows.

See [YouTube Data API overview](https://developers.google.com/youtube/v3/getting-started) and [quota calculator](https://developers.google.com/youtube/v3/determine_quota_cost). Exact numbers were changed in 2025-2026 and may change again; do not memorize an old "search costs 100" example without rechecking.

An illustrative architecture:

```text
developer project key + API method
              |
              v
  short burst/rate protection + method-specific/daily quota ledger
              |
          allow or quota error
```

### 19.3 Uber developer APIs: app, user, account, and endpoint dimensions

**Public product behavior, current when checked.**

Uber's developer documentation describes different scopes depending on API/authentication, including app-level, user-token-level, ad-account-level, short-term burst, and endpoint-specific limits, with `429` on excess. This illustrates why "per client" must name the identity and why multiple rules may coexist. See [Uber Rider API rate limiting](https://developer.uber.com/docs/riders/guides/rate-limiting) and [Uber Ads API rate limiting](https://developer.uber.com/docs/ads/overview/rate-limiting).

```text
maps partner:  weighted/leased provider quota
```

### 19.4 Netflix-style streaming

**Illustrative design, not a claim about Netflix's private current implementation.**

Do not rate-limit every video segment like an ordinary metadata API without thinking about playback quality. Separate:

- login/password attempts;
- playback-start/session creation;
- simultaneous streams per account (concurrency/entitlement);
- metadata/search APIs;
- manifest/license requests; and
- CDN bytes/bandwidth shaping.

```text
account --> concurrency entitlement --> playback session
client  --> adaptive bitrate selection --> CDN segments
```

Adaptive bitrate and network congestion control are not the same as API request rate limiting. A harsh per-request limiter on segments can cause rebuffering; bandwidth/token shaping and CDN capacity policies are more relevant.

### 19.5 X/Twitter API

**Public product behavior, current page checked; endpoint limits can change.**

Illustrative consumer-product controls add:

- post/like/follow actions per account;
- account-creation and login rules by IP/device;
- search/read APIs by app/user;
- spam-risk adaptive penalties; and
- fan-out/concurrency protection for hot accounts.

### 19.6 Amazon/AWS API Gateway

**Verified public infrastructure behavior.**

Amazon API Gateway documents token-bucket throttling with a steady rate and burst, applied on a best-effort basis. It also has account/region and route-level relationships. This is a clean example of a gateway being a natural enforcement point while acknowledging that a distributed managed service does not promise a mathematically hard ceiling. See [API Gateway HTTP API throttling](https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-throttling.html).

```text
edge/WAF:       scraper and bot/IP rules
gateway:        customer/seller API quotas
inventory:      per-customer/product purchase-attempt rules
checkout:       concurrency + idempotency + fraud controls
notifications:  queue-paced provider calls
```

### 19.7 Cloudflare edge rate limiting

**Dated first-party production architecture plus current product documentation.**

This is a strong example of deliberately trading perfect global accuracy for edge latency and attack resilience. ## 20. Back-of-the-envelope estimation

### 20.1 Clarify what QPS is being estimated

Track at least:

```text
incoming request QPS
rate-limit decision QPS
state operations/second
allowed QPS
rejected QPS
fallback-decision QPS
```

If one request evaluates a local edge rule plus one global RLS call containing two same-key rules, decision and state-operation counts differ. Say what one "operation" represents.

### 20.2 Illustrative worked design

Assumptions only:

- 2 million peak incoming requests/s;
- 80% require an authenticated global decision;
- 20% are satisfied/rejected by edge/local-only policy;
- global decision batches all same-owner rules in one call;
- 50 million active state keys within the TTL horizon.

Then:

```text
global RLS decision QPS = 2,000,000 * 0.80
                        = 1,600,000 decisions/s
```

This does **not** say one Redis node or one RLS instance can handle that rate. Benchmark, shard, replicate, and leave failover/headroom. If a design target uses 50% peak utilization to survive a zone loss, size from the post-failure topology, not the normal average.

### 20.3 Exact-log memory pressure

Illustrative only: if 5 million active users may each have up to 100 admitted timestamps in a long window:

```text
maximum live timestamp entries = 5,000,000 * 100
                               = 500,000,000 entries
```

Even before actual sorted-set/key/allocator/replica overhead, this is a warning sign. A bounded counter or token bucket may be materially cheaper if approximate/different semantics are acceptable.

### 20.4 Token leasing call reduction

If every active gateway leases 100 tokens at a time and consumes most leases:

```text
global allocation calls ~= admitted token cost / 100
```

The actual reduction is lower when leases strand, costs vary, or hot keys need smaller chunks. Calculate worst-case unused budget:

```text
stranded tokens <= live leases * (chunk_size - 1)
```

This is a rough upper bound for unit-cost requests before expiry, not a guarantee for every lease algorithm.

### 20.5 Queue latency estimate

If an async provider allows 1,000 messages/s and the queue has 200,000 ready messages:

```text
drain time with no new arrivals = 200,000 / 1,000 = 200 seconds
```

If new arrival rate is 900/s, net drain is only 100/s:

```text
drain time ~= 200,000 / (1,000 - 900) = 2,000 seconds
```

This illustrates why queue depth alone is misleading; monitor age and net drain rate.

### 20.6 Capacity checklist

- Peak and attack QPS, not only average.
- Active identities within TTL, not total registered users.
- Rules checked per request and whether operations combine.
- Hot-key share and largest tenant.
- Exact-log entries per key/window.
- Key/value plus real store overhead and replication factor.
- Network bandwidth and connection pools to RLS/Redis.
- Script p95/p99 latency and failure timeout.
- Zone/region failure headroom.
- Denial-cache and metric label cardinality.
- Queue age/drain time where pacing is used.

## 21. Observability, testing, and operations

### 21.1 Metrics

Measure by bounded-cardinality dimensions such as rule, route template, tier, region, and result:

- decisions: allow/reject/shadow-reject/fallback/error;
- remaining-budget distribution and near-limit count;
- RLS and Redis p50/p95/p99 latency;
- timeouts, circuit state, fail-open and fail-closed count;
- Redis errors, memory, evictions, replication/failover, hot shards;
- config version/age and propagation lag;
- local denial-cache hit rate;
- token lease allocation, utilization, expiry, and stranded budget;
- 429/503 response rate and client retry rate;
- downstream latency/errors/concurrency; and
- queue depth, oldest age, arrival rate, and drain rate.

Do **not** label metrics with raw user ID, API key, IP, or arbitrary path. That creates cardinality explosions and privacy risk. Log/sample selected hashed subjects securely for investigation with retention controls.

### 21.2 The success metric is not "more 429s"

Correlate policies with:

- reduced downstream saturation/errors;
- fairness across tenants;
- legitimate-user false positives/support tickets;
- attack mitigation;
- cost avoided;
- latency added to admitted requests; and
- unused/stranded paid capacity.

A misconfigured limiter can keep the database healthy by rejecting every customer. That is not success.

### 21.3 Algorithm unit tests

For every algorithm test:

1. First request/new key.
2. Exactly at limit and one above.
3. Exactly at window/refill boundary.
4. Burst after long idle.
5. Partial token refill and weighted cost.
6. Time moves backward/forward.
7. Key expiry and re-creation.
8. Config/capacity change.
9. `cost = 0`, negative, too large, and integer overflow.
10. Many requests with the same timestamp.

### 21.4 Concurrency tests

- Send hundreds of simultaneous requests with one token/count remaining; allowed count must match the contract.
- Kill a caller after Redis executes but before it receives the result; verify retry behavior.
- Rotate a subwindow while concurrent increments occur.
- Run one hot key alongside ordinary keys; measure noisy-neighbor impact.
- Reshard Redis slots during traffic.

### 21.5 Failure tests

- RLS timeout/unavailability: verify each rule's fail policy.
- Redis primary failure before/after replication.
- Zone loss at peak load.
- Control-plane outage and stale config.
- Region partition and budget/failover behavior.
- Denial-cache full/high-cardinality attack.
- Clock skew and NTP adjustment.
- Queue consumer slowdown and caller timeout interaction.
- Thundering herd at reset/recovery.

### 21.6 Operational controls

- Shadow mode and percentage rollout.
- Per-tenant emergency override with audit/expiry.
- Global kill switch that falls back safely.
- Versioned config and rollback.
- Dashboards for false positives and fallback decisions.
- Runbook for hot shard, Redis failover, and runaway 429.
- Load test with realistic key distribution, not only uniform random keys.

## 22. Interview angle

### 22.1 What the interviewer is testing

The algorithm list is only the opening. The interviewer usually wants to see whether you can:

- turn a vague `N/time` rule into precise semantics;
- select identity and stop easy bypass/false positives;
- compare burst, accuracy, memory, and latency;
- prevent races;
- enforce one quota across many servers;
- partition state and address hot keys;
- explain local versus global/multi-region accuracy;
- define Redis/network failure behavior;
- return usable client signals;
- operate and roll out the policy safely; and
- recognize when concurrency, queuing, or load shedding is the real need.

### 22.2 A 35-45 minute structure

| Time | Focus |
|---:|---|
| 0-5 min | Clarify subject, scope, semantics, action, scale, accuracy, failure mode |
| 5-10 min | Compare algorithms; choose one baseline and state why |
| 10-15 min | API/policy model and end-to-end request path |
| 15-23 min | Gateway/RLS/Redis architecture, key/state, atomic operation |
| 23-30 min | 1,000 servers, sharding, hot keys, local/global hybrid |
| 30-36 min | Redis failure, retry ambiguity, multi-region, fail open/closed |
| 36-41 min | Edge cases: identity, batch cost, concurrency, retries, headers |
| 41-45 min | Monitoring, rollout, trade-offs, recap |

Follow the interviewer: an LLD/coding prompt may need class design and thread safety; an HLD prompt needs fleet/state/failure reasoning.

### 22.4 Answer spine: `L-I-M-I-T`

Use cues, not a memorized paragraph.

#### L - Lock down the contract

- subject/key: IP, user, API key, tenant, route;
- fixed, rolling, average+burst, or paced;
- reject/delay/degrade;
- weighted cost and what events count;
- regional/global accuracy and fail policy.

#### I - Inspect algorithms and identity

- compare fixed, exact sliding, approximate sliding, token, leaky;
- pick based on burst/accuracy/state;
- explain NAT, spoofing, authenticated identity, and hierarchical rules.

#### M - Make one decision atomic

- define state/key/TTL/time source;
- Lua/Function or suitable Redis atomic operation;
- check-plus-update in one owner;
- handle concurrent calls and ambiguous retry.

#### I - Implement at scale

- layered edge + gateway + service/worker placement;
- stateless RLS fleet and sharded Redis;
- 1,000 servers share/own/lease budget;
- hot keys, high cardinality, regions, config plane.

#### T - Tolerate and track failure

- fail open/closed/local fallback by rule;
- replication/failover and bounded overshoot;
- 429/Retry-After/backoff+jitter;
- shadow rollout, metrics, tests, and alternatives such as concurrency limits.

### 22.5 Example opening answer

> "Before selecting an algorithm, I want to define the contract: which authenticated subject and route share the quota, whether `N/minute` means a fixed or rolling window, the tolerated burst and overshoot, whether excess is rejected or queued, and whether we fail open or closed. I will assume a regional per-user-and-route API limit, low-latency `429` rejection, small controlled bursts, and slight failover overshoot is acceptable. I would start with a token bucket at the API gateway, with rules cached locally and the token state updated atomically through a sharded rate-limit service/Redis. A local coarse bucket protects the gateway and Redis; then I will cover 1,000 servers, hot keys, multi-region budgets, and failure behavior."

### 22.8 Follow-up answer cues

| Follow-up | Answer cues |
|---|---|
| Fixed boundary | N at end + N at start; nearly 2N instantly; sliding/token/leaky alternative |
| Token vs log | Average+burst with O(1) state versus exact rolling semantics and O(events) state |
| Token smoothing | It permits B burst; leaky queue/pacer or smaller short bucket for smoothness |
| Leaky latency | Bounded queue; wait roughly queue depth/drain rate; align timeout, reject overflow |
| 1,000 local counters | Limit multiplies; share state, key owner, or centrally lease budgets |
| Divide quota | Uneven load, autoscale, discrete small limits, stranded capacity |
| One token race | Atomic read-refill-compare-decrement at one owner |
| `INCR` + expiry | Increment atomic; workflow can leak key; script/transaction semantics |
| `MULTI/EXEC` | No convenient branch on queued result; Lua/Function or WATCH retry |
| Lost reply | Stable decision ID/dedupe if contract needs it; otherwise accept conservative double-charge |
| Redis failover | Async replication can lose count; WAIT reduces not eliminates; state acceptable overshoot/fail policy |
| Hot tenant | Local cap/leasing/dedicated shard/hierarchical approximate global; isolate noisy neighbor |
| IP+user+tenant atomicity | Different slots/dimensions; cannot trivially atomically aggregate all; conservative layering/coordinator |
| Global regions | Home owner, regional allocation, consensus, or async approximation; choose latency/availability/accuracy |
| Fail policy | Per rule based on abuse/cost versus availability; local fallback and circuit breaker |
| NAT | Auth user/session plus IP; higher coarse IP threshold; risk signals/challenge |
| GraphQL | Server-computed complexity units, depth/batch/result caps, concurrency, observe actual cost |
| Slow dependency | Rate is insufficient; concurrency/adaptive limit, circuit breaker, load shedding |
| Cardinality attack | Validate/bound keys, TTL/memory, coarse edge rule, no raw metrics labels |
| Reset herd | Jitter, progressive refill, backoff, avoid precise synchronized reset where possible |
| Rejected attempts | Separate admission quota from abuse penalty; define retry recovery |
| Live config change | Versioned rules, preserve/migrate/reset state explicitly, shadow rollout |
| WebSocket | Connections, in-flight, messages/s, bytes/s, fan-out cost as separate limits |
| Rollout | Validate, shadow, metrics, cohort/percentage, overrides, rollback |

### 22.9 Red flags in an interview answer

- Starts with "Redis + token bucket" before defining identity and semantics.
- Says a rate limiter prevents DDoS completely.
- Calls token and leaky bucket the same.
- Claims sliding log smooths traffic.
- Puts the full quota independently on every server.
- Divides the quota by server count without addressing skew/autoscaling.
- Uses `GET` then `SET` and calls each command atomic.
- Says a pipeline provides atomicity.
- Uses application-server clocks without discussing skew.
- Calls Redis a disposable cache while depending on counters for a hard limit.
- Claims Lua makes cross-region/multi-slot/application DB work one transaction.
- Ignores Redis timeout after commit and failover state loss.
- Uses IP as perfect user identity.
- Counts a huge batch as one cheap request.
- Has only a request-rate cap for a slowing dependency and ignores concurrency.
- Fails open or closed for every rule without business reasoning.
- Returns `429` but gives no retry/backoff story.
- Has no TTL/cardinality/hot-key defense.
- Deploys directly in enforce mode with no observability or rollback.

## 23. Compact revision sheet

### 23.1 Six facts to retain

1. **Define subject, scope, time/burst semantics, counted event, action, and failure mode before choosing an algorithm.**
2. **Token bucket allows a bounded burst; a queue-based leaky bucket paces output and adds wait.**
3. **Fixed window is cheap but has a near-2N boundary burst; sliding logs are exact but expensive; counters approximate with bounded state.**
4. **Across 1,000 servers, share/own/lease the logical budget; independent full local buckets multiply the limit.**
5. **Make each read-decide-update atomic at one state owner, while separately reasoning about retries, cluster slots, replication, and regions.**
6. **Rate limiting does not replace concurrency limits, idempotency, DDoS defense, load shedding, or transactional business correctness.**

### 23.2 Formula card

```text
fixed window id
    floor(epoch_time / window_size)

weighted two-window estimate
    previous * ((W - elapsed_in_current) / W) + current

token refill
    tokens = min(capacity, tokens + elapsed * refill_rate)

token-bucket traffic envelope from full state
    admitted cost over T <= capacity + refill_rate * T

leaky-queue maximum rough wait
    queue depth / drain rate

steady-state concurrency estimate
    request rate * average latency

independent-local worst-case fleet limit
    number of servers * per-server limit

decision QPS
    incoming QPS * remote checks/request

active state keys
    active subjects in TTL horizon * rules/subject
```

### 23.3 Algorithm choice card

```text
calendar quota / cheapest cap        -> fixed window
true rolling N in W                  -> exact sliding log
bounded memory, tunable rolling error-> subwindow counter
O(1) approximate rolling             -> weighted two-window counter
average + explicit burst             -> token bucket
smooth/paced downstream output       -> bounded leaky queue
slow dependency                      -> concurrency/adaptive limit too
```

### 23.4 Architecture card

```text
Client
  -> edge local abuse cap
  -> gateway auth + cached rules
  -> stateless RLS
  -> Redis shard owning atomic state
  -> application business/concurrency limit
  -> paced async worker where needed

Control plane: Admin -> Rules API -> durable DB -> versioned publish -> caches
```

### 23.5 Design-review checklist

- Is the identity trusted and hard enough to rotate?
- Is `N/W` fixed, rolling, average+burst, or paced?
- Are burst, cost, counted outcome, and rejection behavior explicit?
- Does placement reject early enough while retaining needed context?
- Does one global quota stay one quota across every instance?
- Does the shard key distribute ordinary users and address hot keys?
- Is the read-decide-update atomic?
- Are multi-key, retry, failover, and multi-region limits stated honestly?
- Are TTL and cardinality bounded?
- Is fail-open/closed/fallback per rule?
- Do clients get a useful 429/backoff contract?
- Are concurrency/load shedding needed in addition?
- Can the rule be shadowed, observed, overridden, and rolled back?
