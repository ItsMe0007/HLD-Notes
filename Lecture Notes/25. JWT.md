# JSON Web Tokens (JWT) for System Design (HLD)

> Beginner-first study notes expanded from the supplied bullets. The goal is to understand what a JWT proves, what it does **not** prove, and why token lifecycle and key trust matter more than memorizing three Base64-looking strings.

## Scope, assumptions, and important corrections

These notes assume no prior JWT or identity background. The minimum OAuth, encryption, HTTP, cookie, and SSO concepts needed here are introduced inline. The deeper companion chapters are [OAuth](./23.%20OAuth.md) and [Encryption](./24.%20Encryption.md).

Your bullets identify the important areas, with these corrections:

1. **JWT is a token format, not an authentication, authorization, or SSO protocol.** A system can use a JWT to carry an authentication assertion or authorization claims, but credentials/OIDC authenticate, application policy authorizes, and an identity-provider session creates the SSO experience.
2. A normal signed JWT is more precisely a JWT represented as a **JWS**. Its compact form is `base64url(header).base64url(payload).base64url(signature-or-MAC)`.
3. The signature is **not** `base64(base64(header) . base64(payload))`. The signer cryptographically signs or MACs the exact ASCII bytes `base64url(header) + "." + base64url(payload)`, then Base64url-encodes the resulting signature/MAC bytes.
4. The header and payload are normally **encoded, not encrypted**. Anyone holding the token can usually decode them. A JWE provides confidentiality and has five compact segments, not three.
5. `Authorization: Bearer` describes how a possession credential is presented. The bearer token may be a JWT or an opaque string. `Authorization: Basic` normally carries a Base64 encoding of `username:password`; it is not a JWT and Base64 is not encryption.
6. A JWK is not an exploit or an inherently unsafe key. It is a JSON representation of a key. The vulnerability occurs when a verifier trusts an attacker-provided `jwk`, unsafe `jku` URL, or manipulated `kid` instead of binding the key to an already trusted issuer.
7. Normal key rotation does not automatically invalidate all tokens: old verification keys are normally retained until old tokens expire. Emergency key removal can invalidate a large group of tokens, but creates a large blast radius.

### Standards snapshot as of 17 July 2026

- JWT is defined by [RFC 7519](https://www.rfc-editor.org/rfc/rfc7519.html).
- JWS signing/MAC structures are defined by [RFC 7515](https://www.rfc-editor.org/rfc/rfc7515.html).
- JWE encryption structures are defined by [RFC 7516](https://www.rfc-editor.org/rfc/rfc7516.html).
- JWK and JWKS key representations are defined by [RFC 7517](https://www.rfc-editor.org/rfc/rfc7517.html).
- Algorithm identifiers and key parameters are defined by [RFC 7518](https://www.rfc-editor.org/rfc/rfc7518.html).
- Current general JWT security guidance is [RFC 8725, JWT Best Current Practices](https://www.rfc-editor.org/rfc/rfc8725.html).
- A standardized profile for JWT-shaped OAuth access tokens is [RFC 9068](https://www.rfc-editor.org/rfc/rfc9068.html). OAuth itself does not require access tokens to be JWTs.

Algorithms, library defaults, identity-provider behavior, token lifetimes, and product APIs change. These notes avoid universal lifetime numbers and identify product-specific or illustrative examples.

### Evidence labels used in the real-system section

- **Documented:** supported directly by a linked first-party product document.
- **Illustrative:** a reasonable way JWT could be used in that named system; it is not a claim about private internal architecture.
- **Important non-example:** the system teaches a boundary—such as “Bearer does not imply JWT”—without claiming that its token has JWT format.

# Part I — Plain-language foundation

## 1. Start with the problem: proving a prior decision

Suppose Alice signs in to a shopping application. The password check happens once:

~~~text
Alice sends credentials securely
        |
        v
Authentication service checks them
        |
        v
“Yes, this is Alice”
~~~

On Alice's next request, the order service needs to know:

- which caller is making the request;
- whether the login is still valid;
- which application the proof was meant for;
- what broad permissions were granted;
- whether anyone altered that proof.

It would be wasteful and unsafe to resend the password on every request. Instead, the system gives Alice a temporary credential.

A JWT is one possible **envelope for claims** in that credential:

~~~text
Issuer says:
  subject = user-123
  audience = orders-api
  expires = 10:30
  scope = orders.read

Issuer protects those claims with a MAC or digital signature.
Orders API validates the protection and every relevant claim.
~~~

The useful mental model is a tamper-evident event pass:

- The **claims** say whose pass it is, where it works, and when it expires.
- The **signature/MAC** is the issuer's anti-tamper seal.
- A verifier trusts the pass only if it already trusts the issuer's key.
- The seal does not hide the printed information.
- A stolen, still-valid bearer pass can be replayed.
- Venue staff still apply local rules; possessing “general admission” does not let someone enter a staff-only room.

## 2. One-sentence definition

> A JWT is a compact, URL-safe way to carry a JSON set of claims inside a JWS and/or JWE structure so recipients can verify integrity/authenticity and, when JWE is used, confidentiality.

Break that apart:

- **Claim:** a name/value statement such as `"sub": "user-123"`.
- **Compact:** common signed JWTs fit into three dot-separated segments.
- **URL-safe:** Base64url avoids `+` and `/`, and omits padding in JOSE compact encodings.
- **JWS:** protects integrity and authenticity with a digital signature or MAC.
- **JWE:** encrypts content to provide confidentiality plus authenticated-encryption integrity.
- **Recipient:** the application or API that validates and consumes the token.

## 3. Four concepts that must not be collapsed into “JWT auth”

| Concept | Plain question | What actually performs it? | JWT's possible role |
|---|---|---|---|
| Authentication (AuthN) | “Who are you?” | Password/passkey/MFA, client certificate, or an identity protocol | May carry the result as an ID token or other assertion |
| Authorization (AuthZ) | “May you do this?” | API/application policy, including resource and tenant checks | May carry scopes, roles, or attributes used as policy inputs |
| Session management | “Which continuing login is this?” | Server session record, cookies, tokens, refresh state, expiry, logout | May be one credential in a session design |
| SSO | “Can one login enable several apps?” | Shared identity provider plus OIDC/SAML and its own login session | OIDC commonly delivers a signed JWT ID token to each app |

The phrase “JWT authentication” is common shorthand, but a precise answer is:

> Credentials authenticate the user first; a validated JWT can then represent the resulting identity/session context on later requests.

## 4. Minimal prerequisites, explained inline

### 4.1 HTTP request and header

HTTP is the request/response protocol commonly used between a client and an API. Headers carry metadata:

~~~http
GET /orders/42 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJ...
~~~

The header is only transport. It does not validate the token.

### 4.2 HTTPS/TLS

TLS protects the request while it travels over the network. A bearer token is usable by whoever possesses it, so send it only over HTTPS. TLS does not rescue a token leaked through XSS, malware, browser storage, logs, analytics, crash dumps, or an insecure database.

### 4.3 Encoding, hashing, signing, and encryption

| Operation | Purpose | Reversible? | Secret/key? |
|---|---|---:|---|
| Base64url encoding | Safe textual representation of bytes | Yes, by anyone | No |
| Cryptographic hash | Fixed-size fingerprint | Not meaningfully reversible | No |
| HMAC/MAC | Integrity and shared-secret authenticity | No | Shared secret |
| Digital signature | Integrity and signer authenticity | No | Private sign, public verify |
| Encryption | Confidentiality | Yes, with decryption key | Yes |

JWT's visible segments use **Base64url encoding**. The protection is the JWS MAC/signature or JWE encryption, not Base64url itself.

### 4.4 Cookies

A cookie is a browser storage and transport mechanism. The server sends `Set-Cookie`; the browser later sends `Cookie` to matching origins according to its rules. A cookie can hold:

- an opaque session ID;
- a JWT;
- another small value.

Therefore “cookies versus JWT” compares different layers. The meaningful comparison is **server-side opaque session versus self-contained JWT**, and separately **cookie transport versus Authorization-header transport**.

## 5. What JWT is—and is not

JWT **is**:

- a standardized claims container;
- a way to let multiple verifiers validate issuer-produced claims;
- often self-contained enough for the normal validation path;
- usable in OIDC ID tokens and some OAuth access-token profiles;
- capable of being signed/MACed, encrypted, or nested.

JWT **is not**:

- a password hashing system;
- an OAuth grant flow;
- automatically an access token or refresh token;
- automatically a bearer token;
- automatically encrypted;
- automatically stateless;
- a database permission check;
- an immediate-revocation mechanism;
- proof that every decoded claim is true;
- a substitute for TLS.

## 6. Why JWT is useful in system design

The main architectural benefit is **portable, verifiable claims**.

Imagine 100 API replicas and 20 microservices. With an asymmetric signed JWT:

1. the authorization server alone holds the private signing key;
2. services cache public verification keys;
3. each service can validate a normal request locally;
4. adding API replicas does not add one session-database lookup per request;
5. an identity provider can issue one standardized artifact understood by different technologies.

This is useful for:

- horizontally scaled APIs;
- independently deployed services;
- federated identity and multi-application login;
- edge/API-gateway authorization;
- short-lived service assertions;
- disconnected or intermittently connected verification, if the use case tolerates staleness.

The cost is that a self-contained statement can become **stale**. A user may be disabled or lose a role while an already issued token remains cryptographically valid. JWT trades a central lookup on every request for expiry windows, revocation complexity, claim staleness, token size, and key-distribution work.

# Part II — Stateful sessions versus JWTs

## 7. Stateful session ID / `JSESSIONID`

In a classic server-side session, the client receives a random, opaque identifier:

~~~text
Browser                   Application                 Session store
   | POST /login              |                            |
   | credentials ------------>| verify                     |
   |                          | create session S7 -------->|
   |<-- Set-Cookie:           |                            |
   |    JSESSIONID=random-S7  |                            |
   |                          |                            |
   | GET /account            |                            |
   | Cookie: JSESSIONID=S7 -->| lookup S7 ---------------->|
   |                          |<-- user, roles, expiry -----|
   |<-- response              |                            |
~~~

`JSESSIONID` is the conventional cookie name used by Java servlet containers for an HTTP session. The value should be an unguessable reference; the useful user/session data stays server-side. Exact cookie names and container behavior are configurable.

### Why this is stateful

The identifier has little meaning without the server-side record:

~~~text
S7 -> {
  user_id: "user-123",
  roles: ["customer"],
  expires_at: ...,
  csrf_state: ...,
  revoked: false
}
~~~

Logout can delete or invalidate `S7`, so the next lookup fails immediately.

### Scaling choices

- **Sticky sessions:** route a user's requests to the same app instance. Simple, but failover/rebalancing are harder.
- **Replicated/in-memory sessions:** duplicate session state between instances. Consistency and failover complexity grow.
- **Shared session store:** Redis or a database serves all replicas. Operationally common, but it is a hot dependency.

## 8. Signed JWT request flow

~~~text
Client                    Auth service                  Orders API
  | POST /login                |                             |
  | credentials -------------->| authenticate                |
  |                            | sign short-lived JWT        |
  |<-- access JWT -------------|                             |
  |                                                          |
  | GET /orders/42                                          |
  | Authorization: Bearer JWT ------------------------------>|
  |                                      cached public key   |
  |                                      verify + claims     |
  |<-- response ---------------------------------------------|
~~~

The Orders API may avoid a session lookup because the signed claims contain the subject, audience, time boundary, and coarse permissions it needs. It must still query business data to answer “does this order belong to this user?” and may consult revocation or policy state.

## 9. Session ID versus self-contained JWT

| Dimension | Opaque server session ID | Self-contained signed JWT |
|---|---|---|
| Meaning | Reference to server record | Claims carried in token |
| Per-request identity lookup | Normally yes | Often no on normal path |
| Immediate revocation | Straightforward: delete/disable record | Requires short expiry, denylist/version check, key action, or another state lookup |
| Permission freshness | Read current record/policy | Claims can be stale until expiry or state check |
| Credential size | Usually small | Grows with claims and signature |
| Sensitive data | Kept server-side | Visible to token holder unless JWE; avoid secrets |
| Horizontal scaling | Needs shared/replicated state or affinity | Public-key validation scales well |
| Key management | Random-ID generation and session-store security | Signing keys, JWKS, rotation, algorithm policy |
| Auth service outage | Existing sessions work only if store works | Existing JWTs can work while keys are cached |
| Server control | Strong | Deliberately reduced between issuance and expiry |

Neither is universally better.

## 10. “JWT is stateless” is incomplete

A JWT may make **access-token validation** stateless with respect to a per-token session record. The whole authentication system usually still has state:

- user accounts and password/passkey data;
- refresh-token families;
- consent and device sessions;
- signing keys and versions;
- disabled-user or permission state;
- denylist entries;
- rate limits and audit records.

If every API checks `jti` against Redis on every request, the design has regained central state. That may be the correct security choice, but do not sell it as a completely stateless request path.

## 11. A common hybrid

Many production designs combine the strengths:

~~~text
Browser
  |
  | secure HttpOnly session cookie
  v
Backend for Frontend (BFF) ---- server-side refresh/session state
  |
  | short-lived, audience-specific access JWT
  v
API gateway / microservices ---- cached issuer public keys
~~~

- The browser does not expose a long-lived refresh token to JavaScript.
- The BFF can revoke the user's session.
- Downstream services validate short-lived JWTs without a lookup on every hop.
- Internal tokens can have a narrower audience than the browser-facing session.

Use a simple server session when one backend owns the application and immediate control matters more than distributed local validation. JWT is not a maturity badge.

# Part III — JWT structure and claims

## 12. The common three-part form

A common signed JWT uses JWS Compact Serialization:

~~~text
xxxxx.yyyyy.zzzzz
  |     |     |
  |     |     +-- Base64url(signature or MAC bytes)
  |     +-------- Base64url(JWT claims JSON bytes)
  +-------------- Base64url(protected header JSON bytes)
~~~

The first two segments can normally be decoded by anyone. The third is not “decoded into meaningful JSON”; it represents cryptographic output bytes.

### Base64 versus Base64url

Base64url is a URL/file-name-safe Base64 variant:

- `-` replaces `+`;
- `_` replaces `/`;
- trailing `=` padding is omitted in JOSE compact representations;
- it does not provide secrecy.

## 13. Header: how this object is protected

Example:

~~~json
{
  "typ": "at+jwt",
  "alg": "RS256",
  "kid": "2026-07-key-3"
}
~~~

| Header parameter | Meaning | Critical nuance |
|---|---|---|
| `alg` | JWS signature/MAC algorithm; in JWE, the key-management algorithm | Required by JWS/JWE and must be allowlisted; never let the token choose policy |
| `enc` | JWE content-encryption algorithm | Required for JWE; verifier/decrypter must allowlist it independently of `alg` |
| `typ` | Declared media/type hint for the complete object | Helps separate token kinds, but is not a security decision by itself |
| `kid` | Key identifier used to select among trusted keys | A selector/hint, not evidence that the key is trusted |
| `cty` | Content type, often used for nested objects | For nested JWT, commonly `JWT` |
| `jwk` | Embedded JSON Web Key | Must not become attacker-controlled trust |
| `jku` | URL of a JWK Set | Unsafe if a verifier fetches arbitrary token-provided URLs |

In compact JWS, the entire header segment is protected by the signature/MAC. That prevents undetected modification **after successful validation**, but it does not make the header's requests trustworthy. An attacker can create a new token with a new header; the verifier must impose its own algorithm and key-source policy.

### `typ` versus `alg`

- `typ` says what application object this is intended to be, such as an access token.
- `alg` says which cryptographic operation protects it.
- Neither says which issuer, audience, or business purpose the verifier should trust.

Explicit types such as `at+jwt` can help prevent an ID token being accepted where an access token is expected, when combined with separate validation rules.

## 14. Payload: the claims set

Example access-token-shaped payload:

~~~json
{
  "iss": "https://auth.example.com",
  "sub": "user-123",
  "aud": "https://orders-api.example.com",
  "exp": 1784277000,
  "nbf": 1784276100,
  "iat": 1784276100,
  "jti": "01K0A6...",
  "scope": "orders.read orders.cancel",
  "tenant_id": "tenant-9"
}
~~~

Claims are assertions, not commands. A valid signature means the protected bytes came from a holder of the signing capability and were not altered. It does not mean the application should honor every claim in every context.

## 15. Registered claims

RFC 7519 registers seven short names. The core JWT specification does **not** make all of them mandatory; the profile/application decides what it requires.

| Claim | Name | Plain meaning | Validation/usage nuance |
|---|---|---|---|
| `iss` | Issuer | Who issued the token | Exact-match a configured trusted issuer; bind it to that issuer's keys |
| `sub` | Subject | Who/what the token concerns | Treat identity as at least the pair `(iss, sub)`; do not assume globally unique `sub` |
| `aud` | Audience | Intended recipient(s) | API must find its configured audience; can be a string or array |
| `exp` | Expiration time | Time at/after which it must not be accepted | Reject expired token, with only small configured clock tolerance |
| `nbf` | Not before | Earliest acceptance time | Reject before this time |
| `iat` | Issued at | When issuer created it | Useful for age, incident cutoff, and sanity checks; not expiry |
| `jti` | JWT ID | Unique token identifier | Enables correlation/replay/denylist designs; uniqueness alone does not stop replay |

`exp`, `nbf`, and `iat` use **NumericDate**: seconds since the Unix epoch in UTC, ignoring leap seconds. Do not accidentally compare them with a milliseconds-based clock.

### Claim-by-claim intuition

#### `iss`: “Which authority said this?”

A signature without issuer binding is incomplete. A key trusted for a test tenant, another region, or another identity provider must not validate production identities by accident.

#### `sub`: “Who is the subject under that authority?”

Two issuers may both use `sub = "123"`. Use the issuer-subject pair as the stable external identity and map it to an internal account deliberately.

#### `aud`: “Who was this made for?”

An ID token for `photo-app` must not be accepted by `payments-api`. Validating only the signature and expiration enables token substitution.

#### `exp`, `nbf`, and clock skew

Distributed machines' clocks differ. Configure time synchronization and a small, explicit leeway. Excessive leeway silently lengthens the attack window. There is no universal “correct” leeway.

#### `jti`: “Which particular token is this?”

An API can store a hash of `jti` until `exp` to deny or consume it. Without server-side tracking, a bearer token with a `jti` can still be replayed many times.

## 16. Public and private claim names

The word **public** here is about the **name's namespace**, not payload confidentiality.

| Claim-name class | Meaning | Example |
|---|---|---|
| Registered | Standard name in the JWT claims registry | `iss`, `sub`, `aud`, `exp` |
| Public | Name defined to avoid collisions, by registration or a collision-resistant name | `https://example.com/claims/account_tier` |
| Private | Name agreed between producer and consumer; collision-prone outside that agreement | `tenant_id`, `roles` |

All three are normally readable in an ordinary signed JWT. Never put passwords, private keys, raw payment data, or secrets in a payload. Minimize personal information too: tokens spread into clients, proxies, traces, and logs.

### Common non-registered claims

- `scope`: often a space-delimited OAuth scope string.
- `roles` or `groups`: broad role/group membership.
- `tenant_id`: tenant context.
- `client_id` or `azp`: client/authorized-party context in particular profiles.
- `token_use`: provider-specific token-kind distinction.

The consumer must know the exact profile. A claim name alone does not have universal semantics.

## 17. The signature/MAC, correctly

For a compact JWS, first create the signing input:

~~~text
encodedHeader  = BASE64URL(UTF8(header JSON))
encodedPayload = BASE64URL(UTF8(payload JSON))

signingInput = ASCII(encodedHeader + "." + encodedPayload)
~~~

Then protect it with the configured algorithm:

~~~text
HS256:
  macBytes = HMAC-SHA-256(sharedSecret, signingInput)

RS256:
  signatureBytes = RSA-sign-SHA-256(privateKey, signingInput)

ES256:
  signatureBytes = ECDSA-sign-SHA-256(privateKey, signingInput)

thirdSegment = BASE64URL(macBytes or signatureBytes)
token = encodedHeader + "." + encodedPayload + "." + thirdSegment
~~~

The verifier repeats the algorithm-specific verification over the **original encoded segments**. It should not parse and reserialize the JSON before checking, because whitespace, member ordering, and byte representation affect the signing input.

### What successful validation proves

- The protected header/payload bytes have not changed.
- With a digital signature, they were signed using the private key corresponding to the trusted public key.
- With HMAC, they were MACed by some holder of the shared secret.

It does **not** prove:

- confidentiality;
- that the bearer is the original user;
- freshness beyond validated time/replay policy;
- current account status;
- object-level permission;
- that the selected key or issuer should have been trusted.

## 18. HMAC versus digital signatures

### Symmetric: `HS256`

~~~text
Auth service ---- shared secret K ---- API A
                         |
                         +------------- API B
~~~

Every verifier with `K` can also create a valid MAC. If API B is compromised, the attacker may mint tokens accepted by API A. Strong random secret generation, secure distribution, and rotation are required.

### Asymmetric: `RS256`, `ES256`, or another approved signature algorithm

~~~text
Auth service: private key -> signs
APIs:         public key  -> verify only
~~~

A compromised verifier's public key does not grant signing power. This is usually a better trust model when many independent services verify tokens.

| Dimension | HMAC | Asymmetric signature |
|---|---|---|
| Keys | Same secret signs and verifies | Private signs, public verifies |
| Verifier can mint tokens | Yes | No, if it only has public key |
| Distribution | Secret to every verifier | Public material can be published |
| Rotation | Coordinate secret securely | Publish public keys through JWKS |
| Compute/signature size | Often cheaper/smaller | Often more CPU and/or bytes |
| Good fit | One tightly controlled trust domain | Many services/partners/verifiers |

Do not choose solely from a speed microbenchmark. Trust boundaries and key compromise dominate.

## 19. Decode is not verify

Anyone can perform:

~~~text
split on "."
Base64url-decode header
Base64url-decode payload
parse JSON
~~~

That only **reads attacker-controlled input**. A safe API boundary is:

~~~text
untrusted token
  -> structural/size checks
  -> select configured token profile
  -> find key only inside configured issuer trust
  -> verify allowed algorithm and signature/MAC
  -> validate issuer, audience, type, time, and required claims
  -> only now use claims as authenticated inputs
  -> apply current business authorization
~~~

A UI may decode a token to display non-security information, but the backend must verify it independently.

## 20. A defensive validation pipeline

1. **Accept only the expected transport and size.** Reject malformed or unexpectedly large input early.
2. **Choose the token profile from endpoint configuration.** Do not infer trust solely from untrusted claims.
3. **Parse strictly.** Reject invalid segment count, invalid Base64url, duplicate/ambiguous JSON members according to library/profile policy, and unsupported critical headers.
4. **Require the expected type if the profile uses one.** Access-token and ID-token validators should be separate.
5. **Allowlist algorithms.** For example, configure only `RS256` for this issuer/profile; never accept whatever `alg` asks for.
6. **Resolve a trusted key.** Use `kid` only to select from the issuer's configured JWKS/cache.
7. **Verify the entire cryptographic operation.** Reject on any failure.
8. **Validate exact issuer.** The key must belong to that issuer.
9. **Validate audience/resource.** This service must be an intended recipient.
10. **Validate `exp`, `nbf`, and relevant age policy.** Use controlled clock leeway.
11. **Validate required subject/client/nonce/token-use claims.** Requirements differ for ID, access, logout, and one-time tokens.
12. **Check revocation/session state if the risk model requires it.**
13. **Apply authorization against request data.** Scope/role is only one input; enforce tenant and object ownership.

Fail closed. Return a generic authentication failure to the caller; keep safe reason codes in protected metrics, not raw tokens in logs.

# Part IV — Authentication, authorization, SSO, and HTTP transport

## 21. JWT in authentication

A typical login:

~~~text
1. Client sends password/passkey proof to authentication service over TLS.
2. Service verifies credentials and MFA/account policy.
3. Issuer creates a short-lived token about the authenticated subject.
4. Client presents it later.
5. API validates it and treats the subject claim as authenticated context.
~~~

The JWT did not check the password. It carries a verifiable result of an earlier authentication event.

An OIDC **ID token** is a defined example: it is a JWT containing an authentication assertion for the client. The client must validate signature, issuer, audience, expiration, and flow-specific claims such as `nonce`.

## 22. JWT in authorization

An OAuth **access token** is presented to an API. Some authorization servers issue JWT access tokens; others issue opaque access tokens.

~~~text
Validated token says:
  scope = orders.read
  tenant_id = tenant-9

Request says:
  GET /tenants/tenant-9/orders/42

API still checks:
  - is this token intended for this API?
  - does orders.read cover this operation?
  - does order 42 belong to tenant-9?
  - may this subject read this particular order?
  - is the order/account under a current restriction?
~~~

Putting `"role": "admin"` in a forged payload does nothing if validation is correct. Putting a genuine but overly broad admin role in a long-lived token is still a dangerous authorization design.

## 23. ID token, access token, and refresh token

| Artifact | Intended recipient | Main purpose | JWT required? |
|---|---|---|---|
| ID token | OIDC client application | Authentication result about end user | Yes, OIDC defines it as a JWT |
| Access token | Resource server/API | Authorize API calls | No; can be JWT or opaque |
| Refresh token | Authorization server | Obtain new access tokens/session continuation | No; commonly opaque and strongly protected |

Common mistake:

~~~text
Wrong: send an ID token to every API because it contains user data
Right: send an access token whose audience and scope target that API
~~~

Access and ID tokens may be signed by different keys and require different claims. Do not build one “accept any JWT” validator.

## 24. Bearer versus Basic

### Bearer

~~~http
Authorization: Bearer <access-token>
~~~

“Bearer” means possession is sufficient for use under the scheme. It says nothing about whether the token is JWT-shaped:

~~~text
Bearer eyJ...  -> may be JWT
Bearer A9x...  -> may be opaque
~~~

### Basic

~~~http
Authorization: Basic dXNlcjpwYXNzd29yZA==
~~~

The value is typically `Base64(username + ":" + password)`. Anyone can decode it. Basic must be protected by TLS and is unsuitable for many modern end-user session designs because the long-lived credential is sent repeatedly.

### Cookie

~~~http
Cookie: session=random-opaque-id
~~~

or, less ideally in many browser architectures:

~~~http
Cookie: access_token=<JWT>
~~~

The cookie's browser behavior changes the threat model:

- `HttpOnly` prevents JavaScript from directly reading it;
- `Secure` limits it to HTTPS;
- `SameSite` helps reduce some cross-site request scenarios;
- cookie authentication still needs CSRF analysis;
- XSS may be able to perform actions as the user even when it cannot read an HttpOnly cookie.

## 25. How JWT participates in multi-app SSO

SSO comes from a central identity-provider login session and a federation protocol—not from copying one JWT into every application.

~~~text
                Identity Provider (IdP)
                login session cookie
                  /             \
                 / OIDC          \ OIDC
                v                 v
             App A              App B
          local session      local session
          aud = app-A        aud = app-B
~~~

### First app

1. App A redirects Alice to the IdP.
2. Alice authenticates at the IdP.
3. The IdP creates its own SSO session cookie.
4. App A receives an authorization code and obtains/validates an ID token meant for App A.
5. App A usually creates its own local application session.

### Second app

1. App B redirects Alice to the same IdP.
2. The IdP sees its existing session, so Alice may not need to enter credentials again.
3. App B gets its own code and ID token whose audience is App B.
4. App B creates its own local session.

The IdP session creates “sign in once.” JWT is a common signed assertion delivered by OIDC. Logout is harder: local app sessions, IdP session, refresh grants, and API access tokens have different lifecycles.

## 26. Browser and device storage

There is no storage choice immune to every attacker.

### Server-rendered web app or BFF

Strong default:

- store refresh/access credentials on the server;
- give the browser an opaque, random session cookie;
- use `Secure`, `HttpOnly`, deliberate `SameSite`, narrow domain/path, expiry, rotation, and CSRF defenses;
- never expose signing keys to the browser.

### SPA calling APIs directly

If architecture requires a browser-held access token:

- prefer short-lived tokens;
- keeping access tokens in memory reduces persistent theft compared with local storage;
- page reload/recovery then requires a deliberate reauthentication or protected refresh design;
- a token in `localStorage` is readable by successful same-origin JavaScript/XSS;
- an HttpOnly cookie is unreadable to JavaScript but automatically sent, so CSRF and cookie scoping matter;
- use a mature OIDC client and avoid inventing token refresh concurrency.

### Native/mobile app

- use platform-provided secure storage;
- use Authorization Code with PKCE through the system browser;
- treat the installed app as unable to protect a static client secret;
- rotate/revoke refresh tokens and support device-session management.

### Backend service

- keep credentials in a secret manager/workload identity system;
- do not bake shared HMAC secrets or refresh tokens into images and source code;
- prefer short-lived workload credentials and audience restriction.

### Never put bearer tokens in

- query strings;
- URLs sent to analytics;
- logs or exception messages;
- source control;
- frontend bundles;
- support screenshots/tickets.

URLs leak through history, referrers, proxies, and telemetry.

# Part V — Token lifecycle and invalidation

## 27. Expiration is part of the security boundary

The shorter the access-token lifetime:

- the shorter the natural replay and stale-permission window;
- the more often clients need renewal;
- the more load and availability dependency moves to refresh/token endpoints;
- the more clock and retry edge cases appear.

There is no standards-mandated universal value. “5–15 minutes” is often used as an **illustrative design range, not a rule**. Choose from data sensitivity, expected revocation speed, client reliability, refresh protection, and operational load.

## 28. Access plus refresh tokens

~~~text
short-lived access token -> sent to API frequently
longer-lived refresh token -> sent only to authorization server
~~~

This separates:

- a broadly exposed credential with a small damage window;
- a more powerful credential stored behind stronger controls.

Refresh-token rotation can detect reuse:

~~~text
family F:
  R1 used -> issue R2, invalidate R1
  R1 used again -> possible theft -> revoke family F
~~~

Concurrency matters. Two browser tabs or app replicas may refresh simultaneously. The server needs atomic rotation, a defined grace/idempotency policy, and a safe response to lost replies.

## 29. Invalidation/revocation options

| Technique | How it works | Revocation speed | Cost/trade-off |
|---|---|---|---|
| Short-lived access JWT | Stop renewal; wait for `exp` | Bounded by remaining lifetime | Simplest local validation, but not immediate |
| Refresh-token revocation | Revoke server-side session/family | Prevents future access tokens; current token remains until expiry | Requires refresh state |
| `jti` denylist | Store revoked token ID until expiry | Near immediate after cache propagation | Lookup/storage on requests; size grows with revoked tokens |
| Session/user version | Token carries `ver`; API compares current cached value | Near immediate | Lookup per request or selected high-risk operations |
| “Valid after” timestamp | Reject tokens whose `iat` predates user/tenant incident cutoff | Near immediate | Current-state lookup and careful clocks |
| Opaque reference token + introspection | API asks central service about active state | Immediate centrally | Network latency, availability, caching/staleness |
| Emergency key/secret removal | Stop accepting every token signed by compromised key | Broad and rapid once propagated | Huge blast radius; JWKS caches and in-flight work complicate |
| One-time `jti` consumption | Atomically mark ID used | Immediate after one use | Requires strongly consistent state; loses stateless benefit |

### Memory planning for a denylist

Estimate rather than guess:

~~~text
live denylist entries
  ≈ revocations per second × average remaining token lifetime

storage
  ≈ entries × (hashed jti + expiry + datastore overhead)
~~~

Store a hash or non-sensitive identifier, attach TTL equal to remaining validity, and plan multi-region propagation. A denylist that never expires becomes a database leak and operational burden.

## 30. Why “invalidate the secret key” needs qualification

### Normal rotation

1. Publish the new public key.
2. Begin signing new tokens with it.
3. Continue publishing/accepting the old public key while old tokens can be valid.
4. Remove old key after maximum token lifetime, clock tolerance, and cache behavior permit.

Normal rotation therefore **does not immediately invalidate** old tokens.

### Emergency compromise

1. Stop signing with the compromised key.
2. Remove/distrust its public verification key.
3. Force verifier caches to refresh or distribute a deny decision.
4. Reauthenticate affected sessions.
5. investigate tokens issued during the compromise window.

This invalidates all tokens using that key—legitimate and malicious. With an HMAC secret shared across many services, attribution and containment are worse because any holder could have minted tokens.

## 31. “Use once” token

A JWT can carry a `jti`, but it is not one-time merely because the claim is unique.

Correct consumption:

~~~text
verify token and its one-time purpose/audience/expiry
        |
        v
atomically insert/transition jti from UNUSED -> USED
        |
        +-- success: perform operation exactly once
        |
        +-- already exists/used: reject replay
~~~

Examples include password-reset, email-verification, invitation, and high-risk action tokens. Use a very narrow audience/purpose, short expiry, minimal claims, and atomic server state. Often an opaque random database token is simpler and easier to revoke.

## 32. Logout semantics

“Logout” can mean several independent actions:

- delete the browser's local app cookie/token;
- revoke the app's refresh-token family;
- invalidate the app's server session;
- end the identity provider's SSO session;
- deny current access JWTs;
- sign out one device or all devices.

A local logout that deletes browser storage does not invalidate a copied bearer token. Define the requirement before choosing the mechanism.

## 33. Permission and account changes

Suppose an administrator removes Alice's `payments.approve` role at 10:00, but her JWT expires at 10:15.

Options:

- accept a maximum 15-minute stale window;
- issue shorter tokens;
- check current authorization state for high-risk actions;
- use a permission/session version cache;
- push revocation events to regional caches;
- use reference tokens/introspection for the sensitive API.

A good design often keeps stable identity and coarse scopes in the token while checking volatile, object-specific policy from current data.

## 34. Failure cases around time and retries

### Client receives `401` because access token expired

Refresh at most once through synchronized client logic, then retry only if the API operation is itself safe/idempotent. Do not create an infinite refresh loop.

### Token expires during a long request

Authenticate at request admission under a documented policy. Rechecking midway may be appropriate for streams or very long privileged operations, but can create confusing partial failures.

### Clock moves backward/forward

Use synchronized monotonic-aware infrastructure where applicable, alert on clock drift, and keep leeway bounded. Do not “fix” frequent failures by adding hours of tolerance.

### Authorization service unavailable

Existing access JWTs may continue to validate from cached keys until expiry. New login/refresh fails. This is an availability advantage of local validation, but only if key caches and validation code are healthy.

# Part VI — JWK, JWKS, key trust, and rotation

## 35. Why a public-key distribution format is needed

With asymmetric signing:

~~~text
Authorization server has PRIVATE key -> produces signatures
Resource servers need PUBLIC key      -> verify signatures
~~~

The public key is meant to be distributed. The challenge is representing it consistently, selecting the right version during rotation, and proving it belongs to the expected issuer.

A **JSON Web Key (JWK)** is a JSON object representing one cryptographic key. A **JSON Web Key Set (JWKS)** is an object containing an array of JWKs:

~~~json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "2026-07-key-3",
      "use": "sig",
      "alg": "RS256",
      "n": "<base64url RSA modulus>",
      "e": "AQAB"
    }
  ]
}
~~~

The normal public JWKS endpoint publishes public verification material only.

## 36. JWK fields: `kty`, `kid`, `n`, `e`, and `k`

### Common fields

| Field | Meaning | Security nuance |
|---|---|---|
| `kty` | Key type/family: for example `RSA`, `EC`, `OKP`, or `oct` | Required in a JWK; consumer must support and expect the type |
| `kid` | Key ID | Selects a key during rollover; is not a password or trust proof |
| `use` | Intended broad use, commonly `sig` or `enc` | Optional; if present, enforce consistency |
| `key_ops` | Intended operations such as `verify` | Optional; should not conflict with `use` |
| `alg` | Intended algorithm | Optional; verifier's own allowlist remains authoritative |

### RSA public key

- `n`: RSA modulus, encoded as Base64urlUInt.
- `e`: RSA public exponent, encoded as Base64urlUInt; `AQAB` commonly represents 65537.
- Private RSA JWKs add private parameters such as `d` and must never appear on a public JWKS endpoint.

### Elliptic-curve/OKP public key

- `crv`: named curve.
- `x`, and for traditional EC keys `y`: public coordinates/values.
- A private form adds secret material such as `d`.

### Symmetric key

- `kty: "oct"` indicates an octet-sequence key.
- `k` contains the symmetric key material.
- **This is secret. Never publish a symmetric `k` through a public JWKS endpoint.**

The fields `n` and `e` are public RSA values; `k` is the actual secret for an octet key. Similar-looking Base64url text does not imply similar secrecy.

## 37. Why publishing a JWK can be safe

A digital-signature public key is designed for verification, not signing:

~~~text
public JWKS + token + signature -> verify
public JWKS                     -X-> create a valid new signature
~~~

Publishing it lets:

- many services verify without receiving the private key;
- verifiers discover rotated keys;
- third-party relying parties interoperate;
- the issuer keep signing capability in a KMS/HSM boundary.

The core question is **key provenance**: why does the verifier believe this public key belongs to the intended issuer?

RFC 7517's security guidance is essentially that data should receive no more trust than the method by which the key and its association were obtained. A mathematically valid signature from an attacker's key is still an attacker statement.

## 38. Trusted issuer discovery and “whitelisted domains”

A secure configuration binds values out of band:

~~~text
configured token profile:
  exact issuer     = https://auth.example.com
  expected audience= https://orders-api.example.com
  allowed algorithm= RS256
  JWKS source      = https://auth.example.com/.well-known/jwks.json
~~~

In OIDC/OAuth metadata, the application can retrieve a trusted issuer's discovery document and its `jwks_uri`. Bootstrap trust still comes from administrative configuration and validated HTTPS.

“Whitelist auth-server domains” is directionally right but too vague. Prefer:

- an exact configured issuer;
- an exact or metadata-derived JWKS URI for that issuer;
- HTTPS certificate validation;
- deliberate redirect policy;
- no private/link-local/loopback destinations for remotely fetched header URLs;
- egress controls and response size/time limits;
- separation between development, staging, tenants, and production.

Do not take `iss` from the unverified payload, concatenate `/.well-known/jwks.json`, fetch it, and automatically trust whatever arrives.

## 39. Normal signing-key rotation

Safe rotation uses an overlap:

~~~text
Time ---->

1. JWKS: [K1 public]
   signer: K1 private

2. JWKS: [K1 public, K2 public]   <- publish K2 before using it
   signer: K1 private

3. JWKS: [K1 public, K2 public]
   signer: K2 private             <- new JWT header kid=K2

4. after every K1 token can no longer be valid:
   JWKS: [K2 public]
   destroy/archive K1 private per policy
~~~

### Verification with `kid`

~~~text
JWT header kid=K2
       |
       v
lookup K2 in issuer-scoped local JWKS cache
       |
       +-- found -> verify with K2 public key
       |
       +-- unknown -> controlled refresh from configured JWKS URI
                       |
                       +-- K2 now found -> verify
                       +-- still absent -> reject
~~~

The `kid` is not globally unique. Scope it by issuer and token profile.

## 40. JWKS cache design

Fetching keys for every API request would reintroduce a network dependency and let the JWKS endpoint become a bottleneck. Cache them.

A production cache needs:

- issuer-scoped entries;
- HTTP cache/refresh policy or configured TTL;
- proactive periodic refresh before expiry;
- one controlled refresh on an unknown `kid`;
- request coalescing so 10,000 replicas/threads do not refresh together;
- rate limiting and negative caching for repeated bogus `kid` values;
- last-known-good keys for ordinary issuer outages, under a bounded policy;
- metrics for key age, refresh failure, unknown `kid`, and verification reasons;
- emergency invalidation mechanism that can override stale cache;
- no logging of the full JWT.

### Unknown-`kid` denial of service

An attacker can send tokens with random `kid` values:

~~~text
kid=random-1 -> refresh
kid=random-2 -> refresh
kid=random-3 -> refresh
...
~~~

If each miss triggers remote I/O, the attacker amplifies cheap requests into load on every verifier and the issuer. Coalesce, rate limit, negatively cache, cap accepted token size, and refresh only the issuer's fixed endpoint.

### Availability versus emergency revocation

Keeping last-known-good keys during a JWKS outage improves availability. It can delay response to a compromised key. Define:

- maximum stale-key period;
- how emergency deny events reach every region;
- whether high-risk endpoints fail closed when the cache is too old;
- how the system distinguishes ordinary rotation from incident response.

## 41. JWK/JWKS exploit family

### 41.1 Attacker-provided `jwk`

Vulnerable logic:

~~~text
token header contains jwk = attacker's public key
verifier uses that key merely because it is present
attacker signs token with matching private key
signature is mathematically valid
verifier wrongly accepts attacker's claims
~~~

The crypto works; the trust decision is wrong.

Defense:

- ignore embedded `jwk` unless a specific protocol requires it;
- if required, authenticate/bind it through that protocol's rules;
- normally select only from issuer-configured trusted keys.

### 41.2 Attacker-controlled `jku` or `x5u`

If a verifier fetches a token-provided URL, an attacker may point it to:

- an attacker-hosted key set, enabling forged-token validation;
- a cloud metadata or internal service URL, creating SSRF;
- a huge/slow response, causing resource exhaustion;
- a redirect chain escaping an apparent allowlist.

Defense: prefer a fixed metadata-configured JWKS URI. If a profile truly needs dynamic URLs, use exact allowlists, strong URL parsing, DNS/IP/redirect defenses, restricted egress, HTTPS validation, and strict response limits.

### 41.3 `kid` path, SQL, or command injection

Bad implementation:

~~~text
readFile("/keys/" + header.kid)
SELECT key FROM keys WHERE kid = '<untrusted text>'
shell("get-key " + header.kid)
~~~

An attacker may attempt traversal, injection, or selecting a known weak/empty key.

Defense:

- treat `kid` as an untrusted opaque value;
- compare against keys already parsed into an in-memory map;
- use parameterized queries if a datastore is unavoidable;
- never feed it into a file path or command;
- limit length/character policy as defense in depth, not as the trust proof.

### 41.4 Algorithm confusion

A historical class of bugs changed `RS256` to `HS256` and caused a verifier to use an RSA public key as an HMAC secret. Because the public key is public, an attacker could compute the HMAC.

Defense:

- allowlist algorithms per issuer/token kind;
- bind each key to exactly one algorithm/use;
- use library APIs that distinguish HMAC secrets from asymmetric keys;
- reject type/algorithm mismatches.

### 41.5 Weak HMAC key

An attacker who obtains one JWT can try candidate secrets offline. A human password or short configuration string is not an HMAC key.

Defense: use cryptographically random, sufficiently strong secret material, store it in a secret manager/KMS boundary, rotate it, and prefer asymmetric verification when many services need to verify.

## 42. A safe key-selection rule

Think:

> “Issuer configuration chooses the key set; the token's `kid` may choose one key inside that already trusted set.”

Not:

> “The token tells me where its trusted key is.”

Pseudo-logic:

~~~text
profile = configuration.forEndpoint(request.endpoint)
require token.alg in profile.allowedAlgorithms
unverifiedIssuer = parseForRoutingOnly(token.iss)
require constantExactMatch(unverifiedIssuer, profile.issuer)

jwks = issuerScopedCache.get(profile.fixedJwksUri)
key = jwks.find(token.kid)
if missing:
    jwks = controlledSingleRefresh(profile.fixedJwksUri)
    key = jwks.find(token.kid)
require key exists and matches algorithm/use

verify signature
then validate issuer/audience/type/time/required claims again
~~~

The early issuer parse is only routing to preconfigured policy. No claim becomes trusted until cryptographic and semantic validation succeeds.

# Part VII — JWE, unsecured JWTs, and the threat model

## 43. Signed JWT is encoded, not encrypted

Given an ordinary signed JWT, a token holder can decode:

- user or service identifier;
- issuer and audience;
- roles/scopes;
- timestamps;
- tenant or other custom claims.

The signature prevents undetected modification; it does not hide the data.

Implications:

- minimize claims;
- use stable opaque subject IDs instead of unnecessary email/PII;
- never store secrets in the payload;
- assume clients can inspect it;
- encrypt transport and storage;
- do not treat a JWT as a secure database row delivered to the client.

## 44. JWE: encrypted JOSE object

JWE Compact Serialization has five dot-separated segments:

~~~text
protected-header.encrypted-key.iv.ciphertext.authentication-tag
       1             2       3       4              5
~~~

Conceptually:

- **protected header:** key-management and content-encryption algorithms;
- **encrypted key:** protected content-encryption key, or empty for some direct modes;
- **IV/nonce:** uniqueness input for content encryption;
- **ciphertext:** encrypted payload;
- **authentication tag:** detects tampering for authenticated encryption.

JWE provides confidentiality, but the recipient still needs:

- trusted key ownership;
- allowed algorithms;
- validation of every cryptographic operation;
- issuer/audience/time/application semantics after decryption;
- replay and lifecycle controls.

Encryption is not authorization.

## 45. JWS, JWE, and nested JWT

| Requirement | Structure |
|---|---|
| Recipients may read claims; need integrity/authenticity | Signed/MACed JWT as JWS |
| Claims must be confidential to recipient | Encrypted JWT as JWE |
| Need signer authenticity plus confidentiality across separate trust roles | Often sign, then encrypt as a nested JWT |

Typical nested idea:

~~~text
claims
  -> sign with issuer private key
  -> inner JWS
  -> encrypt for recipient
  -> outer JWE

recipient:
  decrypt and validate outer JWE
  -> verify and validate inner JWS
  -> apply claim policy
~~~

Reject the complete object if any required layer fails. JWE adds CPU, key management, size, debugging difficulty, and intermediary limitations. Often the better answer is “do not put sensitive data in the token,” plus TLS, rather than adding JWE.

## 46. Unsecured JWT and `alg: none`

The JOSE standards define an unsecured JWS using:

~~~json
{
  "alg": "none"
}
~~~

Its compact form has an empty third segment:

~~~text
encodedHeader.encodedPayload.
~~~

This exists for explicitly defined contexts where integrity is provided by another mechanism and the application intentionally permits an unsecured object. RFC 8725 notes that such use can be acceptable only under explicit external protection and policy.

For ordinary authentication, ID-token, or bearer access-token validation:

- do not accept `none`;
- do not fall back to it;
- configure a positive allowlist of expected signed algorithms;
- upgrade libraries with historical `none` or algorithm-confusion flaws.

The vulnerability is not merely that the standard contains `none`; it is that an application lets an untrusted token downgrade a context that requires a signature.

## 47. Threat-to-defense map

| Threat/mistake | What goes wrong | Defense |
|---|---|---|
| Decode without verify | Attacker writes arbitrary claims | Use a verification API and fail closed |
| `alg: none` downgrade | Unsigned token accepted | Algorithm allowlist; require signature |
| RS/HS confusion | Public key misused as HMAC secret | Bind key type and one algorithm |
| Weak HMAC secret | Offline guessing enables minting | Random high-entropy key; secret management |
| Attacker `jwk` | Attacker supplies “trusted” verification key | Select only from configured issuer keys |
| Arbitrary `jku`/`x5u` | Attacker key trust, SSRF, or DoS | Fixed trusted URI and strict egress controls |
| `kid` injection | Key lookup/path/query is manipulated | In-memory map/parameterization; never shell/path concat |
| Skip `iss` | Wrong issuer's token accepted | Exact configured issuer and key binding |
| Skip `aud` | Token for another app/API accepted | Exact audience validation |
| Skip token type | ID token used as access token | Explicit type and mutually exclusive validators |
| Long expiry | Large theft/staleness window | Risk-based short access lifetime + protected refresh |
| No revocation plan | Stolen token works until expiry | Define short life, state check, denylist, or introspection |
| Bearer replay | Stolen token reused | TLS, storage hardening, short life, sender-constrained tokens where needed |
| Sensitive claims | PII/secrets exposed | Minimize claims or use proper encryption/data lookup |
| Token in URL/log | Credential leaks through infrastructure | Header/cookie transport and systematic redaction |
| Over-broad role/scope | Valid token authorizes too much | Least privilege and resource-level policy |
| Stale role | Removed permission remains until expiry | Short life or current-state check for sensitive operations |
| Unknown-`kid` storm | Remote key fetch DoS | Coalesce, rate limit, negative cache |
| Cross-service forwarding | Broad token reaches unintended service | Audience-specific tokens/token exchange |

## 48. Token substitution and cross-JWT confusion

Suppose the same issuer signs:

- an ID token for `web-client`;
- an access token for `orders-api`;
- an email-verification token;
- an internal service assertion.

If all use similar claims and the same key, “signature valid” is not enough. An attacker may present one type at another endpoint.

Defenses:

- different `aud` values;
- explicit `typ` values;
- distinct required claims;
- separate keys where useful;
- mutually exclusive validation code/configuration;
- purpose-specific endpoints;
- never accept “any JWT from our issuer.”

## 49. Replay and sender constraint

A normal bearer JWT proves possession, not that the presenter is the intended device.

~~~text
Alice's valid access token -> stolen -> Mallory replays before exp
signature: still valid
claims:    still valid
~~~

Mitigations:

- TLS and secure storage;
- short access-token lifetime;
- refresh-token rotation and reuse detection;
- device/risk monitoring;
- atomic one-time tracking for one-time artifacts;
- sender-constrained mechanisms such as mTLS-bound tokens or DPoP when the protocol and risk justify them.

Sender constraint reduces simple replay but adds client key lifecycle, proxy compatibility, and recovery complexity.

## 50. Payload injection into downstream systems

Even after signature validation, claims are data. A malicious or compromised issuer, or a permitted tenant-controlled claim, may contain:

- SQL/LDAP special characters;
- path characters;
- log-forging newlines;
- huge arrays/strings;
- URLs that cause SSRF if fetched;
- Unicode lookalikes.

Signature validation never replaces parameterized queries, output encoding, schema/length validation, and safe URL/egress policy.

## 51. Observability without credential leakage

Log/measure:

- issuer identifier;
- token profile/type;
- audience result;
- algorithm and `kid`;
- validation failure reason code;
- hashed/truncated `jti` for correlation if policy permits;
- key-cache age and refresh result;
- latency and status;
- subject pseudonym only where necessary and protected.

Do not log:

- raw Authorization/Cookie headers;
- full token;
- refresh token;
- signature material as a “safe identifier”;
- sensitive payload claims.

Redaction must cover application logs, reverse proxies, API gateways, WAFs, APM agents, traces, crash reports, and support tooling.

# Part VIII — Production HLD for JWT-based API access

## 52. Start with requirements, not token format

Example system: a multi-tenant commerce platform with web/mobile clients and independent APIs.

### Functional

- authenticate users with MFA-capable identity service;
- issue access credentials for Orders and Payments APIs;
- support roles/scopes plus tenant/object checks;
- refresh sessions;
- sign out one device or all devices;
- rotate signing keys;
- disable users and respond to compromise.

### Non-functional

- APIs remain horizontally scalable;
- normal request validation avoids remote auth calls;
- p99 verification overhead stays small relative to API work;
- no private signing key reaches an API service;
- multi-region failover works;
- security changes propagate within a defined bound;
- raw tokens never enter observability systems.

### Questions to clarify in an interview

- Browser, native, third-party, or service clients?
- Is immediate revocation required, and for which actions?
- How stale may roles be?
- What are the trust domains/tenants?
- Which APIs are audiences?
- What is the expected request rate and JWT size?
- Are services allowed to call downstream services as the user?
- Does an external IdP issue the token, or do we operate the issuer?

## 53. High-level architecture

~~~text
                          +-----------------------+
                          | Identity/Auth service |
                          | login, MFA, consent   |
                          +-----+-----------+-----+
                                |           |
                    private sign|           | refresh/session state
                                v           v
                         +----------+   +----------------+
                         | KMS/HSM  |   | Session DB     |
                         +----------+   | refresh family |
                                |       +----------------+
                                |
                    public keys v
                         +-------------+
                         | JWKS/metadata|
                         +------+------+
                                |
                              cache
                                v
Client/BFF -> CDN/WAF -> API Gateway -> Orders service -> Orders DB
                    |        |   |
                    |        |   +------> current object/tenant policy
                    |        |
                    |        +----------> Payments service
                    |
                    +-------------------> rate limit / threat controls

Revocation/event stream -> regional version/deny caches -> gateway/services
~~~

## 54. Component responsibilities

| Component | Responsibilities |
|---|---|
| Identity/auth service | Credential/MFA checks, account policy, token issuance, refresh rotation, session/device management |
| KMS/HSM | Protect private signing keys and restrict signing operations |
| JWKS/metadata service | Publish current/overlap public keys and issuer metadata |
| Session database | Store hashed refresh credentials/families, device sessions, revocation/version state |
| API gateway | TLS termination where appropriate, coarse validation, rate limiting, routing, optional external-to-internal token exchange |
| Resource service | Independently enforce audience/scope and current tenant/object authorization |
| Key cache | Issuer-scoped cached JWKS with controlled refresh |
| Revocation/version cache | Fast lookup for chosen immediate-control policy |
| Audit/event pipeline | Security events with credential redaction and tamper-resistant retention |

Gateway validation does not automatically absolve services. Internal callers may bypass it, routing may change, and services need trustworthy identity propagation. Options are:

- validate the original audience-correct JWT again in each service;
- have the gateway mint a short-lived internal assertion with a narrower audience;
- use authenticated trusted headers only over a strongly protected service identity boundary.

Never trust an Internet client merely because it sent `X-User-Id`.

## 55. Login and issuance flow

~~~text
Client/BFF        Auth service       User/Policy DB       KMS/HSM
    | login/MFA        |                   |                 |
    |----------------->| verify ---------->|                 |
    |                  |<-- user/status ---|                 |
    |                  | create session/refresh family       |
    |                  | sign header.payload --------------->|
    |                  |<-- signature -----------------------|
    |<-- access JWT + protected refresh/session --------------|
~~~

Issuer decisions:

- use a non-PII stable `sub`;
- exact `iss`;
- one or narrowly related `aud`;
- minimal scopes/roles;
- bounded `exp`;
- random `jti` if lifecycle design uses it;
- explicit access-token type;
- active signing `kid`;
- separate profiles/keys for different token purposes where useful.

Private-key export should be prohibited when the KMS/HSM supports in-boundary signing. Grant only the auth issuer permission to sign.

## 56. API request flow

~~~text
Client        Gateway/cache       Orders service        Policy/data
  | Bearer JWT     |                    |                    |
  |-------------->| structure/type     |                    |
  |               | key/crypto/claims  |                    |
  |               |------------------->| trusted context    |
  |               |                    | scope + tenant ---->|
  |               |                    |<-- object decision -|
  |<-------------- response --------------------------------|
~~~

Validation order should reject cheap failures early, but no unverified claim should drive dangerous I/O. For high-value operations, Orders may additionally check the subject/session version or call a current policy engine.

### `401` versus `403`

- **401 Unauthorized** usually means missing, malformed, expired, or otherwise invalid authentication credential. It means “authenticate correctly,” despite the historical label.
- **403 Forbidden** means the credential is understood/authenticated but current policy denies the requested action.

Avoid revealing whether a particular account/resource exists when that information is sensitive.

## 58. Scaling the hot path

For request rate `R` and `N` service hops:

~~~text
local validation operations ≈ R × validating hops
central introspection calls   ≈ R × introspected hops (before caching)
~~~

JWT moves work from network/database I/O to local parsing and cryptography. At large scale:

- validate at the gateway to shed invalid traffic early;
- cache parsed public keys, not validation results for arbitrary bearer tokens unless safe;
- use efficient maintained libraries;
- cap token/header size before expensive parsing;
- avoid putting thousands of entitlements in tokens;
- reuse verified context inside one process/request safely;
- benchmark chosen algorithms and token sizes on actual infrastructure;
- keep business authorization queries indexed/cached separately.

Never skip service authorization merely to save one verification operation.

## 59. Token size and “fat JWT” problems

Large claims cause:

- higher bandwidth on every request;
- proxy/gateway/header-limit failures;
- more parsing and memory allocation;
- stale duplicated policy;
- accidental PII propagation;
- cache-key and logging surprises.

Prefer:

- stable subject/client/tenant identifiers;
- audience;
- coarse scopes;
- essential time/type/session version fields.

Fetch volatile profile data and fine-grained permissions from their owning services. Token-size limits vary by product and configuration, so measure your full request path rather than memorizing one universal maximum.

## 60. Multi-region design

### Signing

Two common models:

1. **Centralized active signer:** simple key ownership; cross-region issuance latency and dependency.
2. **Regional signers with distinct keys:** better regional independence; more JWKS entries, operational policy, incident scope, and issuer/profile decisions.

Do not copy raw private keys casually between regions. Use regional KMS/HSM keys and explicit key-purpose metadata.

### Verification

- distribute/publish the same trusted metadata view;
- cache JWKS regionally;
- prepublish new keys before any region signs with them;
- monitor clock synchronization;
- coalesce refreshes;
- make emergency key-deny events high priority.

### Revocation

Immediate global revocation requires distributed state:

- globally replicated session/version database;
- event stream to regional caches;
- conflict/idempotency handling;
- a stated propagation SLO;
- a fail-open/fail-closed decision during partition.

If the business accepts “valid until short expiry,” no global per-token deny lookup may be necessary.

## 61. Availability and fail-closed boundaries

| Failure | Existing short JWTs | Login/refresh | Design response |
|---|---|---|---|
| Auth service down | Can validate locally | Fails | Cache keys; graceful re-login messaging; restore issuer |
| JWKS endpoint down | Works with trusted cached keys | Issuance may work | Bounded last-known-good cache; alert |
| Unknown new `kid` during outage | Cannot verify | — | Reject; rotation should prepublish and be canaried |
| Revocation cache down | Depends on requirement | May work | High-risk fail closed; low-risk may rely on expiry only if policy allows |
| KMS/HSM signing down | Existing tokens work | New issuance/refresh fails | Multi-AZ key service, quotas, circuit breaking; never fall back to insecure signing |
| Clock skew | False accept/reject | May fail | Time sync, alert, small controlled tolerance |
| Policy DB down | Identity may verify | — | Do not treat authentication as authorization; define endpoint-specific behavior |

Fail-open/fail-closed is not one global switch. Reading a public catalog and approving a payment deserve different decisions.

## 62. Key-management lifecycle

### Generate

- approved algorithm/key size;
- CSPRNG and KMS/HSM;
- unique purpose/environment;
- non-colliding `kid`;
- record owner and activation schedule.

### Store/use

- private key non-exportable where possible;
- least-privilege sign permission;
- public verification keys in JWKS;
- audit every administrative/key operation;
- never share a signing secret with browser/mobile code.

### Rotate

- prepublish;
- canary verifier compatibility;
- activate signer;
- observe new `kid`;
- retain old verifier key for overlap;
- retire after the maximum validity window and caches.

### Revoke/incident

- stop signing;
- globally distrust compromised key;
- revoke affected refresh sessions;
- force reauthentication as needed;
- analyze audit trails and issue window;
- communicate blast radius.

## 63. Downstream service calls

Forwarding one broad user access token through every service creates the **confused deputy** and excessive-audience problem:

~~~text
Client token aud=public-api, broad scopes
        |
        v
Gateway -> Orders -> Inventory -> Payments
           same token everywhere   (dangerous)
~~~

Alternatives:

- exchange it at the gateway for a short-lived internal token with `aud=orders`;
- Orders uses its workload identity for Inventory and propagates a separately protected user context only if needed;
- use “on behalf of” token exchange with narrow delegation;
- avoid user propagation for operations that are truly service-owned.

Each service should know whether it is acting as the user, as itself, or as a constrained combination.

## 64. Security test plan

Test at least:

- missing/extra compact segments;
- invalid Base64url/JSON and oversized token;
- changed payload with original signature;
- `alg: none`;
- wrong allowed algorithm and RS/HS confusion;
- unknown/duplicate `kid`;
- attacker `jwk`, `jku`, and URL/redirect/SSRF cases;
- wrong issuer;
- wrong/multiple audience;
- expired/not-yet-valid/boundary time;
- ID token passed to access-token endpoint;
- token for tenant A used on tenant B object;
- revoked `jti`/session version;
- repeated one-time token;
- stale/rotated JWKS and outage;
- refresh replay/race/lost response;
- token leakage in every observability layer;
- 401/403 behavior without resource enumeration.

Use maintained test libraries and authorized environments. Do not copy real production bearer tokens into third-party JWT-debugging websites.

# Part IX — Real-system examples

## 65. How to read named examples

Public product documentation rarely reveals every internal identity mechanism. The examples below separate documented facts from illustrative design. Do not say “Netflix uses JWT for X” in an interview unless you have a reliable source; say “in a Netflix-like design, I could use JWT here, with these boundaries.”

## 66. WhatsApp: business access and message encryption are separate

**Important non-example:** Meta's WhatsApp Business Platform documentation uses access tokens for API access, but “access token” or “Bearer” alone does not prove JWT format. A client should treat a provider token as opaque unless the provider publishes a JWT validation profile.

An integrator might use its own identity layer:

~~~text
Business operator
      |
      | local web session / short internal access JWT
      v
Integrator backend
      |
      | Meta-issued WhatsApp access token (provider-defined format)
      v
WhatsApp Business Platform API
~~~

Design lessons:

- do not decode a third-party access token and invent authorization semantics;
- keep the Meta credential server-side and use it only for its documented audience;
- use separate tenant/object authorization so business A cannot operate business B's number;
- WhatsApp message end-to-end encryption is a content-confidentiality topic, not solved by JWT;
- never put message contents or long-lived provider credentials in a JWT.

The provider token and the integrator's own session solve different trust boundaries. See Meta's [WhatsApp Business Platform collection](https://www.postman.com/meta/whatsapp-business-platform/overview).

## 67. YouTube/Google: ID token versus API access token

**Documented:** Google OpenID Connect returns an ID token that is a signed JWT. Google instructs clients to validate the signing key from the configured `jwks_uri`, the exact issuer, audience, and expiration. YouTube Data API authorization uses OAuth access tokens.

~~~text
User -> Google OIDC
          |
          +-- ID token JWT, aud=video-scheduler-client
          |      -> scheduler validates login identity
          |
          +-- YouTube OAuth access token
                 -> scheduler sends it to YouTube API
~~~

Do not substitute them:

- the ID token is evidence for the client about the Google authentication;
- the access token is the credential intended for the YouTube resource server;
- the client should not require the YouTube access token to be a JWT unless Google's relevant API contract says so;
- the scheduler should establish its own local session after OIDC validation rather than repeatedly using the ID token as a general API token.

First-party references: [Google OpenID Connect](https://developers.google.com/identity/openid-connect/openid-connect) and [YouTube OAuth 2.0 authorization](https://developers.google.com/youtube/v3/guides/authentication).

## 68. Uber: OIDC ID JWT plus bearer access

**Documented:** Uber's Consumer Identity documentation says OIDC adds an ID token that is a signed JWT. It documents a JWKS endpoint, `kid` selection, and validation of signature algorithm, issuer, audience, expiry/issued time, and `nonce`. It separately returns an OAuth bearer access token.

~~~text
Partner app -> Uber authorization code flow
                    |
                    v
              token endpoint
                /       \
               v         v
       signed ID JWT     access token
       for partner       for Uber APIs
~~~

Interview lesson:

- token format and token purpose are separate;
- `kid` selects a key from Uber's documented JWKS, not an arbitrary token URL;
- `nonce` binds the ID token to the login flow;
- an access token's scope/audience, not the ID token's profile fields, authorizes an Uber API.

See [Uber Identity overview](https://developer.uber.com/docs/consumer-identity/introduction) and [Uber Authorization Code Flow](https://developer.uber.com/docs/consumer-identity/guides/authorization-code-flow).

## 69. Netflix-like streaming: identity is not content protection

**Illustrative:** A streaming platform could use a short-lived signed assertion between an identity edge and a playback-control API:

~~~text
Viewer session
     |
     v
Identity edge -- short JWT: sub, device, aud=playback-control -->
Playback control -- current subscription/device/title policy -->
License/manifest service -- short resource-specific grant -->
CDN / DRM license flow
~~~

JWT might efficiently carry stable identity and device/session context. It should not:

- embed a movie-encryption key;
- replace current entitlement/concurrency/region checks;
- be audience-valid at every backend;
- be confused with DRM or TLS;
- contain a huge catalog entitlement list.

Playback access may change faster than the login identity. Keep the token lean and enforce current title, plan, geography, device, and concurrency policy at the owning service. Netflix's public help documentation describes protected-content/DRM requirements; this does **not** establish its private JWT architecture. See [Netflix protected-content requirements](https://help.netflix.com/en/node/27451/).

## 70. Twitter/X: OAuth bearer does not imply JWT

**Important non-example plus illustrative internal design:** X documents OAuth 2.0 user access tokens and scopes. A consumer must not assume the access token is JWT-shaped merely because it is sent with `Authorization: Bearer`.

~~~text
Third-party posting client
    |
    | X-issued OAuth bearer token, provider-defined format
    v
X API

Inside a Twitter-like system (illustrative):
Gateway -> narrow internal assertion aud=post-service -> post service
~~~

The internal assertion, if JWT, should be:

- very short-lived;
- restricted to the intended service audience;
- clear about user delegation versus workload identity;
- validated independently by the service;
- excluded from URLs and logs.

See X's [OAuth 2.0 Authorization Code Flow with PKCE](https://docs.x.com/fundamentals/authentication/oauth-2-0/user-access-token).

## 71. Amazon Cognito: concrete JWT and JWKS behavior

**Documented:** Amazon Cognito user pools issue JWT ID and access tokens and publish public verification keys through a user-pool JWKS URI. AWS documents:

- matching the token `kid` to a key in that configured JWKS;
- validating issuer, expiry, client/audience, signature, scope, and `token_use` as appropriate;
- caching keys and refreshing on an unknown `kid` because keys can rotate;
- treating refresh tokens separately as opaque credentials in Cognito;
- independently validating ID and access tokens.

~~~text
Cognito user pool
  | private signing keys
  +-- ID JWT ------> client, token_use=id
  +-- access JWT --> API, token_use=access, scopes
  +-- refresh -----> Cognito token/session lifecycle
  |
  +-- public JWKS --> API verifier cache
~~~

This is a strong example of why `kid` exists and why key rotation needs overlap/cache refresh. It also demonstrates that one provider can use different signing keys and claim rules for ID and access tokens.

See AWS [Verifying JSON web tokens](https://docs.aws.amazon.com/cognito/latest/developerguide/amazon-cognito-user-pools-using-tokens-verifying-a-jwt.html) and [Understanding Cognito user-pool JWTs](https://docs.aws.amazon.com/cognito/latest/developerguide/amazon-cognito-user-pools-using-tokens-with-identity-providers.html).

## 72. Product-example comparison

| System | Publicly grounded JWT lesson | Boundary not to cross |
|---|---|---|
| WhatsApp Business | Access credentials authorize provider APIs | Bearer/access-token does not prove JWT; JWT is not message E2EE |
| YouTube/Google | Google OIDC ID token is a JWT; access token targets API | ID token is not a YouTube API access token |
| Uber | OIDC ID JWT uses documented JWKS/claims; bearer access is separate | Do not let `kid`/header invent trust |
| Netflix-like | JWT could carry short-lived playback-control identity | Not DRM, content key, or full entitlement database |
| Twitter/X | OAuth bearer/scopes are documented | Do not assume provider access-token format |
| Amazon Cognito | Concrete access/ID JWTs, `kid`, JWKS rotation and distinct validation | Never accept “any Cognito JWT” without issuer/client/type/audience policy |

# Part X — Trade-offs, alternatives, and selection

## 73. Choose the property before the artifact

| Need | Usually start with |
|---|---|
| Simple first-party web login, one backend, immediate logout | Opaque random server session in secure cookie |
| Many independent APIs need local verification | Short-lived asymmetric signed access JWT |
| Central real-time active/revoked decision | Opaque reference token plus introspection/session lookup |
| Browser should not hold OAuth tokens | BFF plus opaque browser session |
| Third-party delegated API access | OAuth; token format defined by authorization server |
| Cross-app user authentication | OIDC or SAML SSO; app creates local session |
| Claim confidentiality beyond TLS/storage | Minimize claims first; JWE if genuinely required |
| One-time password reset/invite | Often opaque random DB-backed token; narrow one-time JWT is possible |
| Service workload identity | Short-lived workload credential, mTLS/SPIFFE/cloud identity; JWT may be one representation |

## 74. JWT versus opaque access token

| Dimension | JWT access token | Opaque access token |
|---|---|---|
| Validation | Local signature/claims | Introspection or datastore lookup |
| Auth-service hot dependency | Low for normal requests | High unless cached |
| Revocation/freshness | Harder/imperfect until check/expiry | Central and immediate in principle |
| Claim visibility | Token holder can decode signed JWT | Client learns nothing from string |
| Size | Usually larger | Usually small |
| Information propagation | Self-contained | Server controls returned context |
| Key complexity | Signing/JWKS lifecycle | Token generation, secure store/introspection |
| Offline validation | Possible | Not normally |

A hybrid can use a JWT for low-risk APIs and require current introspection/version checks for sensitive operations.

## 75. JWT versus stateful browser session

Choose a server session when:

- one backend/BFF owns the application;
- immediate logout/device control is important;
- permissions change frequently;
- a shared low-latency session store is already acceptable;
- you want minimal information in the browser credential.

Choose access JWTs when:

- many services or external relying parties need independent verification;
- the auth service should not be on every request path;
- short-lived staleness is acceptable;
- you can operate signing keys/JWKS safely;
- audience and purpose boundaries are clear.

These can be combined rather than treated as mutually exclusive.

## 76. HMAC versus asymmetric signing

Use HMAC when:

- issuer and verifier are within one very tight trust boundary;
- every verifier legitimately has equivalent signing authority;
- secret distribution/rotation is well controlled;
- the performance/size difference is materially useful and measured.

Prefer asymmetric signing when:

- many services verify;
- verifiers should not mint;
- partners validate;
- public JWKS rotation/discovery is valuable;
- compromise containment matters.

Avoid describing `HS256` as “less secure” in the abstract. A strong HMAC construction is cryptographically sound; its shared signing authority often creates the architectural problem.

## 77. Local validation at gateway versus every service

| Pattern | Advantage | Risk/cost |
|---|---|---|
| Gateway only | Central config and less repeated crypto | Bypass/header spoofing/internal lateral movement |
| Every service validates external token | Defense in depth and service-owned audience policy | Repeated config/CPU; broad token propagation |
| Gateway exchanges for internal tokens | Narrow audience and trust boundary | Extra issuer/keys/lifecycle; gateway criticality |
| Service mesh identity + protected context | Strong workload channel | User-delegation semantics still need design |

There is no “validated once, trusted everywhere” without an authenticated integrity boundary for the propagated result.

## 78. Fat versus lean JWT

### Fat token

Carries profile, roles, groups, feature flags, entitlements, maybe resource lists.

Benefits:

- fewer reads;
- easy consumer implementation;
- can operate with limited connectivity.

Costs:

- stale and large;
- leaks more information;
- couples issuer to every consumer's model;
- header/proxy limits;
- role explosion;
- every change may require new issuance.

### Lean token

Carries stable subject, issuer, audience, time, client/session, and coarse scopes.

Benefits:

- smaller and less sensitive;
- current policy stays with owning service;
- less coupling.

Costs:

- more policy/profile lookups;
- current-state service becomes a dependency.

A lean JWT is the safer default for HLD unless offline claims are a stated requirement.

## 79. When to avoid JWT

Avoid or reconsider when:

- one monolith can use a simple secure session;
- immediate per-request revocation is mandatory and a state lookup is acceptable;
- claims change constantly;
- the client must not see the claims and JWE complexity is unjustified;
- the team cannot manage signing keys or validation consistently;
- payload would be huge;
- an opaque provider token already has a supported introspection/API contract;
- the only reason is “JWT is modern/stateless.”

## 80. Alternatives and complements

- **Opaque session ID:** browser login/session continuation.
- **Opaque OAuth access token + introspection:** central active-state control.
- **API key:** application/project identification for simpler machine APIs; weak for user delegation and often long-lived.
- **mTLS/workload identity:** authenticates a workload/channel and can sender-constrain access.
- **SAML:** enterprise browser federation; XML assertions rather than JWT.
- **OIDC:** authentication protocol using JWT ID tokens.
- **Macaroons/capability tokens:** caveated delegation; different model and operational complexity.
- **PASETO:** alternative token format with opinionated algorithm versions; ecosystem/protocol compatibility differs.

The surrounding protocol, lifecycle, storage, and authorization policy usually matter more than swapping token syntaxes.

## 81. Common mistakes interviewers listen for

1. “JWT encrypts the payload.”
2. “The signature is just Base64 of header and payload.”
3. “JWT performs authentication and authorization.”
4. “JWT is always stateless.”
5. “Bearer means JWT.”
6. “OAuth token means JWT.”
7. “Decode and read `sub`; then the user is authenticated.”
8. Validate signature but skip `iss` or `aud`.
9. Accept algorithm from the header without a server allowlist.
10. Use one validator for ID, access, and one-time tokens.
11. Publish an HMAC `k` or private JWK.
12. Trust an embedded `jwk` because the signature matches it.
13. Fetch arbitrary `jku` URLs.
14. Concatenate `kid` into a file path/query/command.
15. Put long-lived JWTs in `localStorage` without an XSS/lifecycle discussion.
16. Put tokens in URLs or logs.
17. Use a long token lifetime because “there is a logout button.”
18. Rotate a key without overlap and break every in-flight token.
19. Keep an old compromised key cached for availability without an emergency path.
20. Put every role/permission/resource in the payload.
21. Treat a valid scope as object/tenant authorization.
22. Forward one broad token to every downstream service.
23. Call a `jti` one-time without atomic server tracking.
24. Use a human password as an HMAC secret.
25. Claim a universal token lifetime without requirements.

## 82. Subtle edge cases

### Audience string versus array

Libraries may expose `aud` differently. Validate membership according to the profile, not a naïve string conversion.

### Duplicate JSON members

Different parsers can interpret duplicate claim/header names differently. Use a strict maintained JOSE library and reject ambiguous representations according to the token profile.

### Multiple issuers with the same `kid`

`kid` is only unique inside a key set by convention. Cache by issuer/profile plus `kid`, never globally by `kid` alone.

### Account linking

Do not link users merely by an email claim from arbitrary issuers. Validate issuer and its email-verification semantics, and require an explicit safe linking flow. Use `(iss, sub)` as the external identity.

### Retry after refresh response loss

The server may consume R1 and send R2, but the response can be lost. A strict client retries R1 and looks like an attacker. Define idempotency/grace carefully without making replay easy.

### Key removed too early

A verifier with an old token needs the old public key until that token cannot be accepted. Removal time must cover token validity, leeway, and rollout/caching—not merely signer activation.

### Key compromised but still cached

Normal cache TTL is too slow for emergency response. Distribute an explicit issuer-key deny signal or shorten/actively purge caches for incidents.

### Tenant context changes

A genuine token for tenant A must not select tenant B through a request parameter. Bind route/resource tenant to a validated claim or perform explicit membership checks.

### User deleted and identifier reused

Do not recycle subjects. If internal IDs can be reused, an old token may refer to a different account. Use non-reused stable identifiers and bounded expiry.

### Proxy header normalization

Ensure exactly one trusted Authorization value reaches the application. Reject ambiguous duplicate headers and strip client-supplied internal identity headers at the edge.

## 87. Answer spine: “Explain JWT in two minutes”

1. **Definition:** compact JSON claims container protected as JWS and/or JWE.
2. **Flow:** authenticate first; issuer creates claims; client presents token; API validates.
3. **Structure:** protected header, claims payload, signature/MAC; all Base64url in common JWS compact form.
4. **Signature:** cryptographic operation over exact encoded header plus dot plus encoded payload.
5. **Checks:** allowlisted algorithm, trusted key, `iss`, `aud`, `exp`/`nbf`, type, subject, and policy.
6. **Trade-off:** local validation scales but revocation/role changes are stale; use short access tokens plus protected refresh or choose stateful/introspection.

## 88. Answer spine: “JWT versus session ID?”

Use **reference versus statement**:

- Session ID is an opaque reference; server store is authoritative.
- JWT is a protected statement carrying claims; local validation is possible.
- Session gives easy immediate revocation/current state but requires lookup/replication.
- JWT reduces request-path dependency but brings stale claims, replay, size, signing-key/JWKS, and revocation issues.
- Browser transport is a separate choice; either value could be in a cookie.
- Choose from topology and revocation needs, and mention the BFF hybrid.

## 89. Answer spine: “How do you invalidate a JWT?”

Start by clarifying required scope and speed:

1. One token, one session/device, one user, one tenant, or every token signed by a key?
2. Must it be immediate or is expiry bound acceptable?
3. Use short access expiry and revoke refresh for normal logout.
4. Add `jti` denylist or session/user version for immediate selected invalidation.
5. Use opaque introspection if central real-time control dominates.
6. Remove a signing key only for compromise/large blast radius; normal rotation keeps old verify keys.
7. State cache/region propagation and failure policy.

## 90. Answer spine: “Why is JWKS public, and what is the exploit?”

- Public keys verify but cannot normally sign.
- JWK is JSON key representation; JWKS is a set for discovery/rotation.
- `kid` selects one trusted issuer key.
- Trust comes from exact configured issuer and its fixed/metadata JWKS URI.
- Exploit occurs if verifier trusts attacker `jwk`, fetches arbitrary `jku`, injects `kid`, or confuses algorithms.
- Defense is issuer-key binding, algorithm/key-type allowlists, safe lookup, cache control, and strict claim validation.

## 91. Answer spine: “How does JWT enable SSO?”

Correct the premise gently:

1. Central IdP session enables SSO.
2. Each app redirects using OIDC/SAML.
3. Existing IdP session avoids another credential prompt.
4. OIDC gives each client its own audience-bound ID-token JWT.
5. Each app validates it and normally creates a local session.
6. API access uses access tokens, not the ID token.
7. Single logout requires coordinating local, IdP, refresh, and access-token lifecycles.

## 93. Fast follow-up answers

### Is JWT authentication or authorization?

Neither by itself; it is a token format. It can carry an OIDC authentication assertion or an access token's authorization inputs.

### Is JWT stateless?

Local access-token verification can avoid per-token server state. The complete system normally retains users, refresh sessions, keys, policy, and sometimes revocation state.

### Are JWT header and payload encrypted?

Not in the common signed JWS. They are Base64url-encoded and readable. Use JWE for confidentiality, or preferably keep sensitive data out.

### What is signed?

The exact ASCII `base64url(protected-header) + "." + base64url(payload)` signing input. The resulting MAC/signature bytes form the third segment after Base64url encoding.

### Can I trust a decoded token?

No. Decode only reads attacker-controlled bytes. Verify cryptography and all semantic claims first.

### Does a valid signature mean access is allowed?

No. It establishes protected issuer output under a trusted key. Audience, time, token type, scope, tenant, object ownership, and current policy still decide.

### Is a bearer token always a JWT?

No. Bearer is a presentation/possession scheme; the token can be opaque.

### Basic versus Bearer?

Basic normally sends a Base64-encoded username/password pair; Bearer sends a token whose possession authorizes use. Both require TLS; Base64 is not encryption.

### What does `kid` do?

It selects a key from a trusted set, especially during rotation. It does not make the key trusted.

### What are `n` and `e`?

The RSA public modulus and exponent encoded in an RSA JWK.

### What is `k` in a JWK?

Symmetric key material for `kty: oct`. It is secret and must not be published.

### Why is JWKS public?

It normally contains public verification keys, which can validate signatures but cannot create them.

### What if `kid` is unknown?

Perform one controlled/coalesced refresh from the configured issuer JWKS. If it is still unknown, reject. Rate-limit/negative-cache misses.

### How do keys rotate?

Prepublish new public key, start signing with it, keep old public key through old-token validity, then retire it. Compromise uses an emergency distrust path.

### How do I revoke one JWT?

Store its hashed `jti` in a denylist until expiry, or use session/version/introspection state. Otherwise wait for short expiry and revoke renewal.

### Does `jti` prevent replay?

Only when a recipient tracks and atomically denies/consumes it. A unique untracked value does nothing by itself.

### How does `alg: none` attack work?

A vulnerable verifier lets an untrusted header downgrade a context requiring a signature. Correct validators allowlist required signed algorithms and reject `none`.

### What is algorithm confusion?

A verifier treats one key/algorithm family as another, historically an RSA public key as an HMAC secret. Bind each profile/key to expected algorithms and key types.

### JWS versus JWT?

JWT is the claims object/token definition; JWS is the signing/MAC envelope commonly used to represent it.

### JWE versus JWT?

JWE is the encryption envelope. A JWT can be carried in a JWE, giving the common compact form five segments.

### Should I store JWT in local storage?

It is readable by successful same-origin JavaScript/XSS. Prefer a BFF and secure HttpOnly session cookie when possible; if a SPA must hold a short access token, memory reduces persistence and the full XSS/refresh threat model still matters.

### Why validate audience?

It prevents a valid token intended for one client/API from being substituted at another.

### Can I use an ID token at an API?

Do not. ID tokens target the OIDC client. APIs should receive access tokens intended for that resource server.

### Gateway validates it—should services trust headers?

Only if client-supplied versions are stripped and the gateway-to-service channel has a strong authenticated integrity boundary. Service validation or a narrow signed internal assertion is safer.

### What happens if JWKS is down?

Known keys can continue from a bounded trusted cache. Unknown keys should fail validation; rotation should prepublish to avoid this.

### What is the biggest JWT trade-off?

Local scalable validation versus server control/freshness. Revocation and current permissions become explicit design work.

> “A JWT is a compact claims format, not the authentication or authorization protocol itself.
